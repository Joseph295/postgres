# 03 · 查询优化器：从 Query 到 PlannedStmt 的深度源码分析

> 对应《PostgreSQL 源码学习指南》第 4 章的展开。代码基于 master 分支（版本 20devel）。
> 所有代码引用格式为 `文件路径:行号`，均已在本仓库核实。

## 本章导读

优化器是 PostgreSQL 中理论密度最高的模块。它接收解析器/重写器产出的 `Query` 树（逻辑上就是一棵关系代数表达式），输出可供执行器直接运行的 `PlannedStmt`。这个转换不是简单的翻译，而是一次**在指数级搜索空间中的最优化求解**——其算法框架直接继承自 1979 年 Selinger 等人的 System R 论文，至今仍是所有工业级 CBO（基于代价的优化器）的原型，包括你熟悉的 Spark Catalyst CBO 与 MySQL 优化器。

本章按照一条 SQL 在优化器内的真实旅程展开：

1. **理论对照**：Selinger 1979 的三大思想（动态规划、interesting orders、代价模型）在 PG 源码中的落点；
2. **入口与预处理**：`standard_planner` → `subquery_planner`，重点是三个"逻辑改写"——子链接上拉（IN/EXISTS → 半连接）、子查询展平、常量折叠；
3. **上层规划的阶段化**：`grouping_planner` 与 `UpperRelationKind`（PG 9.6 pathification 重构的产物）；
4. **基表路径生成**：`make_one_rel` → `set_base_rel_pathlists`，顺序扫描/索引扫描/位图扫描/TID 扫描如何被逐一制造出来；
5. **连接搜索**：`standard_join_search` 的逐层动态规划，核心循环逐行走读；表太多时的 GEQO 遗传算法；
6. **代价模型**：`costsize.c` 的公式拆解，包括著名的 Mackert-Lohman 缓存命中模型；
7. **选择率估计**：`selfuncs.c` 如何用 MCV 和直方图把 `WHERE x = 42` 变成一个 0~1 之间的数；
8. **收尾**：`create_plan` 把最优 Path 物化为 Plan，`set_plan_references` 做最后的变量"接线"。

## 前置知识

- **Query 结构体**（`src/include/nodes/parsenodes.h`）：优化器的输入。`rtable` 是范围表（FROM 里的每个表/子查询一项），`jointree` 是连接结构与 WHERE 条件，`targetList` 是投影列表。
- **两个核心概念**（见指南 4.1 节）：
  - `RelOptInfo`：优化过程中"一个或一组关系"的抽象，身上挂着它的全部候选路径（`pathlist`）。基表、子查询、连接中间结果，统统是 RelOptInfo。
  - `Path`：轻量级"计划骨架"，只携带代价（`startup_cost`/`total_cost`）、行数估计（`rows`）和排序信息（`pathkeys`），不带执行细节。**先在便宜的 Path 空间里搜索，胜出后才展开成重量级 Plan**——与 Spark Catalyst 中"先在逻辑/物理计划的轻量表示上做 CBO，最后才 codegen"是同一工程直觉。
- **与 MySQL 的宏观对比**：MySQL（8.0 之前的老优化器）本质上是贪心的左深树搜索 + 启发式剪枝（`greedy_search`），且优化与执行的数据结构耦合较深。PG 则是教科书式的自底向上动态规划，枚举包括 bushy 树在内的完整空间，且 Path/Plan 分离。读完本章你会对这句话有具体的体感。
- 建议先通读 `src/backend/optimizer/README`（1623 行），它是 PG 源码树中质量最高的设计文档之一。本章会反复引用它。

## 1. 理论根基：Selinger 1979 与 PG 的逐概念对照

Selinger 等人的 *Access Path Selection in a Relational Database Management System*（SIGMOD 1979，System R）确立了 CBO 的三根支柱。逐一对照 PG 源码：

### 1.1 动态规划枚举连接顺序

System R 的洞察：n 表连接的顺序有指数种，但**"包含同一组基表的连接中间结果"是等价的子问题**——最优子结构成立。于是自底向上：先求所有 1 表的最优访问路径，再求所有 2 表连接的最优，再 3 表……每层复用下层结果。

PG 的实现就在 `standard_join_search`（`src/backend/optimizer/path/allpaths.c:3952`），注释原文直接写着 "We employ a simple 'dynamic programming' algorithm"（allpaths.c:3964）。`root->join_rel_level[j]` 保存所有恰好包含 j 个基表的 RelOptInfo——这正是 DP 的备忘录表。关键性质（README 第 174-182 行有专门说明）：某一层的所有路径构造完毕后才会构造更高层，因此每层可以放心地调用 `set_cheapest()` 淘汰劣质路径，不必担心上层还引用着它们——这同时简化了路径的内存管理。

### 1.2 Interesting Orders → PathKeys

System R 发现：单看代价淘汰路径会犯错。一条较贵但**输出有序**的路径（比如索引扫描）可能让上层省掉一次排序（merge join 或 ORDER BY），总账反而更便宜。所以每个子问题要按"有趣的顺序"分别保留最优解。

PG 把"顺序"形式化为 `PathKey`（`src/include/nodes/pathnodes.h:1806`）：

```c
typedef struct PathKey
{
    ...
    EquivalenceClass *pk_eclass ...;    /* 排序值所属的等价类 */
    CompareType pk_cmptype;     /* sort direction (ASC or DESC) */
    bool        pk_nulls_first; /* do NULLs come before normal values? */
} PathKey;
```

注意 `pk_eclass` 指向 **EquivalenceClass**（等价类）而非具体列：如果 WHERE 里有 `a.x = b.y`，那么"按 a.x 排序"与"按 b.y 排序"是同一个 PathKey。这是 PG 对 System R 的重要增强——interesting order 的判定经过了等价类的传递闭包（详见 README 的 EquivalenceClasses 一节，README:693 起）。路径的取舍规则在 README:684-690：**只有当一条路径既不更便宜、排序也不更有用时才被丢弃**，具体裁决代码是 `add_path`（`src/backend/optimizer/util/pathnode.c:459`）。

### 1.3 左深树 vs bushy 树：PG 枚举 bushy

System R 只枚举**左深树**（left-deep：每次连接的内表必须是基表），把搜索空间从卡特兰数级压到 n! 级。PG 则枚举完整的 **bushy 空间**——README:163-168 明确说明：

> We consider left-handed plans ...; right-handed plans ...; and bushy plans (both inner and outer can be joins themselves). For example, when building {1 2 3 4} we consider joining {1 2 3} to {4} (left-handed), {4} to {1 2 3} (right-handed), and {1 2} to {3 4} (bushy).

代码落点就是 5.3 节将逐行走读的 `join_search_one_level`（`src/backend/optimizer/path/joinrels.c:78`）：第一段循环产生左深/右深组合，第二段 `for (k = 2;; k++)` 循环显式产生 bushy 组合。PG 靠两个开关控制爆炸：只考虑**有连接条件相连**的组合（避免无谓的笛卡尔积），以及 `from_collapse_limit`/`join_collapse_limit` 限制单个 DP 问题的规模（README:89-102）。

> **对比**：Spark Catalyst 的 CBO join reorder 同样是 DP（`CostBasedJoinReorder` 规则），同样默认只在有连接条件时枚举；MySQL 老优化器则是深度优先贪心 + `optimizer_prune_level` 剪枝，只搜左深空间。PG 在这条谱系上是最"学院派"的。

**小结**：Selinger 三支柱——DP 枚举、interesting orders、代价驱动——分别对应 PG 的 `standard_join_search`、PathKeys + `add_path`、`costsize.c` + `selfuncs.c`。PG 超出 System R 的地方：bushy 枚举、等价类推理、以及超大连接问题上的遗传算法兜底。

## 2. 入口与预处理：standard_planner → subquery_planner

### 2.1 standard_planner：优化器的总装车间

入口 `planner()`（`src/backend/optimizer/plan/planner.c:328`）只做一件事：有 `planner_hook` 就调 hook（pg_hint_plan 等插件的注入点），否则调 `standard_planner`（planner.c:346）。后者的主干流程（省略细节后）：

1. 建立 `PlannerGlobal`（planner.c:366）——跨所有子查询层级共享的全局状态；
2. 判定能否使用并行计划（planner.c:413-429）：只有 SELECT、无修改性 CTE、`max_parallel_workers_per_gather > 0` 等条件全满足才扫描查询树评估并行安全性;
3. 计算 `tuple_fraction`（planner.c:452-477）：游标场景下用户可能只取前几行，用 `cursor_tuple_fraction` 让优化器偏向"快速启动"计划——这是 System R 论文里没有、工程实践逼出来的维度；
4. **递归规划主体**（planner.c:532）：`root = subquery_planner(glob, parse, ...)`；
5. **选出最优路径并物化**（planner.c:536-539）：

```c
    final_rel = fetch_upper_rel(root, UPPERREL_FINAL, NULL);
    best_path = get_cheapest_fractional_path(final_rel, tuple_fraction);

    top_plan = create_plan(root, best_path);
```

逐行解说：`UPPERREL_FINAL` 是上层规划的最后一个阶段关系（见第 3 节），此时它的 pathlist 里躺着整条查询的全部候选完整路径；`get_cheapest_fractional_path` 按 `tuple_fraction` 挑选——取全部行时比 `total_cost`，只取前 N% 时按比例折算 startup/run 代价；`create_plan` 把胜出的 Path 树展开成 Plan 树（第 8 节）。

6. 收尾（planner.c:643）：`top_plan = set_plan_references(root, top_plan);` 做变量扁平化（第 8 节），最后组装 `PlannedStmt`（planner.c:655）。

### 2.2 subquery_planner：预处理流水线

`subquery_planner`（planner.c:770）对**每一层查询**执行一遍，子查询通过递归调用处理。它前半部分是一条"逻辑改写流水线"，与 Spark Catalyst 的 rule-based 优化阶段（谓词下推、子查询去关联等 Rule）职责相当，只是 PG 用手写的顺序调用而非规则引擎。按源码顺序：

```c
    if (parse->cteList)
        SS_process_ctes(root);                  /* planner.c:844 */
    ...
    if (parse->hasSubLinks)
        pull_up_sublinks(root);                 /* planner.c:902 */

    preprocess_function_rtes(root);             /* planner.c:910 */

    pull_up_subqueries(root);                   /* planner.c:916 */
```

随后是对全部表达式统一调用 `preprocess_expression`（targetList 在 planner.c:1044，havingQual 在 planner.c:1065，LIMIT 在 planner.c:1079-1081 等），再做外连接消除、`grouping_planner`。下面拆解三大改写。

### 2.3 pull_up_sublinks：IN/EXISTS 变半连接的机制

`pull_up_sublinks`（`src/backend/optimizer/prep/prepjointree.c:670`）递归遍历 jointree 的 WHERE 和 JOIN/ON 子句，找到**顶层的** `ANY`（即 IN）和 `EXISTS` SubLink，尝试调用：

- `convert_ANY_sublink_to_join`（`src/backend/optimizer/plan/subselect.c:1341`）
- `convert_EXISTS_sublink_to_join`（subselect.c:1593）

以 IN 为例，`WHERE x IN (SELECT y FROM sub ...)` 的改写过程（subselect.c:1341 起的函数注释写得非常清楚）：

1. 检查子查询是**不相关**的（不引用外层变量；相关子查询在特定条件下走 LATERAL 路线）；
2. 把子 SELECT 作为一个新的 `RangeTblEntry` 加入外层查询的 rtable；
3. 构造一个 `JoinExpr`，连接条件是 `x = y`，连接类型为（subselect.c:1450）：

```c
    result->jointype = under_not ? JOIN_ANTI : JOIN_SEMI;
```

即 `IN` → **半连接**（JOIN_SEMI：外表每行最多匹配一次就够），`NOT IN`/`NOT EXISTS` → **反连接**（JOIN_ANTI：只输出无匹配的外表行）。

**为什么这个改写是优化的分水岭？** 不改写时，SubLink 会变成 SubPlan——对外层每一行执行一次子查询（最坏 O(N×M)，即 MySQL 老版本臭名昭著的 DEPENDENT SUBQUERY）。改写成半连接后，它就进入了第 5 节的连接搜索空间：优化器可以自由选择 hash semi join、merge semi join，甚至调换连接顺序。半连接/反连接带来的连接顺序合法性约束由 `SpecialJoinInfo` 记录、`join_is_legal` 检查（README:188-194、README:229-239 详细列出了 SEMI/ANTI 的重结合规则）。

### 2.4 pull_up_subqueries 与 is_simple_subquery：子查询展平的准入条件

`pull_up_subqueries`（prepjointree.c:1220）处理 FROM 里的子查询：`SELECT ... FROM (SELECT ...) sub`。不展平时子查询是"黑盒"——单独规划、外层只见一个 SubqueryScan，跨黑盒的连接顺序优化全部丧失（README:279-290）。展平就是把子查询的表直接并入外层 jointree。

能否展平由 `is_simple_subquery`（prepjointree.c:1954）裁决，条件摘录（prepjointree.c:1969-2074）：

```c
    if (subquery->setOperations)        /* 有 UNION/INTERSECT/EXCEPT：不行 */
        return false;

    if (subquery->hasAggs ||            /* 聚合 */
        subquery->hasWindowFuncs ||     /* 窗口函数 */
        subquery->hasTargetSRFs ||      /* 目标列里的集合返回函数 */
        subquery->groupClause ||        /* GROUP BY */
        subquery->groupingSets ||
        subquery->havingQual ||
        subquery->sortClause ||         /* ORDER BY */
        subquery->distinctClause ||     /* DISTINCT */
        subquery->limitOffset ||        /* OFFSET */
        subquery->limitCount ||         /* LIMIT */
        subquery->hasForUpdate ||       /* FOR UPDATE/SHARE */
        subquery->cteList)              /* WITH */
        return false;

    if (rte->security_barrier)          /* 安全屏障视图：防信息泄漏 */
        return false;
    ...
    if (contain_volatile_functions((Node *) subquery->targetList))
        return false;                   /* volatile 函数：防多次求值 */
```

逐条讲原因：聚合/GROUP BY/LIMIT 等改变了行的"粒度"或数量，拉平会改变语义；`FOR UPDATE` 拉平会把加锁语义提升到错误的层级（prepjointree.c:1976-1980 注释）；安全屏障视图拉平后其内部 Var 散落到外层，选择率估计函数可能被恶意函数利用来探测数据（prepjointree.c:1996-2002）；volatile 函数（如 `random()`）拉平后可能被复制到外层多处、执行多次，结果改变（prepjointree.c:2065-2074）。LATERAL 子查询还有额外的外连接相关限制（prepjointree.c:2007-2063）。

也就是说：**"简单"的标准 = 拉平后语义严格不变**。这份清单是排查"为什么我的视图/子查询没有被展开优化"的第一现场。

### 2.5 preprocess_expression：常量折叠

`preprocess_expression`（planner.c:1429）对每个表达式做一串规范化，核心一步（planner.c:1475）：

```c
    expr = eval_const_expressions(root, expr);
```

`eval_const_expressions`（实现在 `src/backend/optimizer/util/clauses.c`）做：常量子表达式求值（`2+2` → `4`）、只含常量参数的不可变函数内联求值、`CASE`/`AND`/`OR` 的常量短路化简、简单 SQL 函数的内联展开等。随后（planner.c:1480-1520）：`canonicalize_qual` 把 qual 规范化（展平嵌套 AND/OR、提取 OR 各分支的公共项）；`convert_saop_to_hashed_saop` 为大 IN 列表（`x = ANY('{...}')`）标记哈希查找；`SS_process_sublinks` 把没能在 2.3 节变成连接的 SubLink 展开为 SubPlan；最后 `make_ands_implicit` 把 qual 变成隐式 AND 的列表形式——后续所有优化代码都假定这个形式。

**小结**：`subquery_planner` 的预处理是"先做确定性有利的逻辑改写，再进代价搜索"。三大改写的判断标准全部是**语义不变性**：IN/EXISTS 上拉靠半/反连接语义兜住"最多匹配一次"，子查询展平靠 `is_simple_subquery` 的白名单，常量折叠只处理 immutable 的计算。

## 3. grouping_planner 与 upper planner 的阶段化设计

`grouping_planner`（planner.c:1704）负责扫描/连接**之上**的一切：GROUP BY、聚合、窗口函数、DISTINCT、ORDER BY、LIMIT、集合操作、DML 顶层节点。

### 3.1 PG 9.6 的 pathification 重构

历史背景：9.6 之前，扫描/连接部分产出 Path，而上层的 Sort/Agg/Limit 是 `grouping_planner` 直接以 Plan 节点形式"焊死"追加的——上层没有候选之间的代价竞争，也没有扩展点。9.6 的重构（Tom Lane，俗称 upper planner pathification）把上层每个处理阶段也抽象成 RelOptInfo + Path 的竞争场，统一用 `add_path` 裁决。

阶段的枚举是 `UpperRelationKind`（`src/include/nodes/pathnodes.h:142`）：

```c
typedef enum UpperRelationKind
{
    UPPERREL_SETOP,             /* result of UNION/INTERSECT/EXCEPT, if any */
    UPPERREL_PARTIAL_GROUP_AGG, /* result of partial grouping/aggregation */
    UPPERREL_GROUP_AGG,         /* result of grouping/aggregation, if any */
    UPPERREL_WINDOW,            /* result of window functions, if any */
    UPPERREL_PARTIAL_DISTINCT,  /* result of partial "SELECT DISTINCT" */
    UPPERREL_DISTINCT,          /* result of "SELECT DISTINCT", if any */
    UPPERREL_ORDERED,           /* result of ORDER BY, if any */
    UPPERREL_FINAL,             /* result of any remaining top-level actions */
} UpperRelationKind;
```

枚举顺序就是流水线顺序（SQL 语义的执行顺序）：集合操作 → （部分）聚合 → 窗口 → DISTINCT → 排序 → 最终（LIMIT/锁行/DML）。`PlannerInfo` 里有按此枚举下标的数组 `upper_rels[]`（pathnodes.h:547），`fetch_upper_rel(root, kind, ...)` 按需取用。

### 3.2 grouping_planner 的接力流程

主线代码（调用点均已核实）：

1. `current_rel = query_planner(root, standard_qp_callback, &qp_extra);`（planner.c:1924）——拿到扫描/连接部分的最终 joinrel（第 4、5 节的全部内容都发生在这一行内部）；
2. 依次条件接力（planner.c:2069/2087/2107/2122）：

```c
    current_rel = create_grouping_paths(...);   /* 有聚合/GROUP BY 时 */
    current_rel = create_window_paths(...);     /* 有窗口函数时 */
    current_rel = create_distinct_paths(...);   /* 有 DISTINCT 时 */
    current_rel = create_ordered_paths(...);    /* 有 ORDER BY 时 */
```

每个 `create_xxx_paths`（如 `create_grouping_paths`，planner.c:4050）的模式完全一致：`fetch_upper_rel` 拿到本阶段的 RelOptInfo，然后**对输入 rel 的多条候选路径 × 本阶段的多种实现策略**做笛卡尔式组合生成路径。以分组为例：输入若有按 GROUP BY 列有序的路径 → 生成 Group/GroupAggregate 路径（利用 interesting order！）；输入的 cheapest path → 加 Sort 再 GroupAggregate，或直接 HashAggregate；有并行输入 → 部分聚合（`UPPERREL_PARTIAL_GROUP_AGG`）+ Gather + 最终聚合。全部丢进 `add_path` 竞争。

3. 最终把胜者挂到 `UPPERREL_FINAL`（在 grouping_planner 尾部，planner.c:2138 附近 `fetch_upper_rel(root, UPPERREL_FINAL, NULL)`），供 2.1 节第 5 步收割。

**设计价值**：其一，上层操作有了真正的代价竞争（sort-based vs hash-based 聚合的选择就是在这里发生的）；其二，每个阶段都有 `create_upper_paths_hook` 与 FDW 的 `GetForeignUpperPaths` 注入点（如 planner.c:4416-4422），聚合下推到远端（如 postgres_fdw、Citus）才成为可能。这与 Spark 把聚合策略（HashAggregate vs SortAggregate）作为物理计划候选交给策略选择是同构的设计。

**小结**：`grouping_planner` 把"SELECT 语句剩下的部分"组织成 UpperRelationKind 流水线，每一段都是一个"RelOptInfo 竞技场"。理解它的最好实验：对同一个 GROUP BY 查询分别 `SET enable_hashagg = off/on`，观察 EXPLAIN 里 GroupAggregate 与 HashAggregate 的 cost 竞争。

## 4. query_planner → make_one_rel：基表路径生成

### 4.1 query_planner 的准备工作

`query_planner`（`src/backend/optimizer/plan/planmain.c:54`）先处理平凡情形——jointree 里只有一个 RTE_RESULT（`SELECT 1` 这类无表查询）时直接造一条 GroupResultPath 返回（planmain.c:97-160）。否则：建 RelOptInfo 数组（`add_base_rels_to_query`）、`deconstruct_jointree` 把 WHERE 拆成单表限制条件（`a=1`，挂到对应 rel 的 `baserestrictinfo`）与连接条件（`a.x=b.y`，挂 `joininfo`，并在此过程中构建 EquivalenceClass）、标记可用于 merge/hash join 的子句，最后调用 `make_one_rel`。


### 4.4 indxpath.c 主线：索引路径怎么匹配出来

`create_index_paths`（`src/backend/optimizer/path/indxpath.c:239`）对该表的每个索引依次：

1. **谓词检查**（indxpath.c:270）：部分索引（`CREATE INDEX ... WHERE ...`）若其谓词不能被查询条件蕴含（`predOK` 为假）则跳过；
2. **匹配限制子句**（indxpath.c:277）：`match_restriction_clauses_to_index` → 底层是 `match_clauses_to_index`（indxpath.c:2552），判断标准是子句能否写成 `索引列 操作符 表达式` 且操作符属于该索引操作符族——这就是"`WHERE f(x)=1` 用不上 x 上的普通索引"的源码级原因；
3. **生成普通索引路径**（indxpath.c:284）：`get_index_paths`（indxpath.c:715）→ `build_index_paths`（indxpath.c:809），生成 IndexPath（可能是 IndexOnlyScan，如果目标列全在索引里），直接 `add_path`；同时把适合做位图的路径积攒到 `bitindexpaths` 列表；
4. **匹配连接子句**（indxpath.c:294-314）：用连接条件（含等价类推导出的）生成**参数化索引路径**——即"给我一个外层的 b.y 值，我能用索引找 a.x = b.y"，这是高效 nestloop 内表的来源（README 的 Parameterized Paths 一节，README:1059 起）;
5. **位图组合**（indxpath.c:321-353）：`generate_bitmap_or_paths` 处理 OR 条件（每个 OR 分支各用一个索引，位图求并）；`choose_bitmap_and`（indxpath.c:345）贪心挑选多个索引做位图求交；最后包一层 `create_bitmap_heap_path` 进入竞争。位图扫描的本质：先扫索引把命中 TID 收集成位图、按物理页序访问堆——用一次排序换取顺序 I/O，是 SeqScan（纯顺序、全量）与 IndexScan（随机、精准）之间的中间形态。

**小结**：基表路径生成是"广撒网"：SeqScan 保底，TID/Index/IndexOnly/Bitmap/并行各显神通，参数化路径为将来的 nestloop 预埋伏笔；`add_path` 按"代价 × 排序 × 参数化"多维标准修剪，网住的路径进入下一节的连接搜索。

### 5.1 三岔口：make_rel_from_joinlist

### 5.2 standard_join_search 逐行走读

逐行解说：`join_rel_level[j]` 是 DP 备忘录——所有恰含 j 个基表的 joinrel；第 1 层直接用上一节做好的基表们初始化。主循环从 2 到 n：先由 `join_search_one_level` 造出本层全部 joinrel 及其路径，然后对每个 joinrel 补分区连接与 Gather 路径，最后 `set_cheapest` 冻结。**正因为下层已冻结，上层组合时只需在下层留存的（每种有用排序/参数化下的最优）路径里挑**——这就是 1.1 节说的最优子结构。最顶层（levels_needed 层）必然只剩一个 rel（包含全部基表），断言检查后返回（allpaths.c:4051-4053）。

### 5.3 join_search_one_level 逐行走读：一层内怎么组合

（`src/backend/optimizer/path/joinrels.c:78`，这是 PG bushy 枚举的心脏）

**第一段：左深 + 右深**（joinrels.c:96-143）

```c
    foreach(r, joinrels[level - 1])                 /* 遍历 level-1 层的 joinrel */
    {
        RelOptInfo *old_rel = (RelOptInfo *) lfirst(r);

        if (old_rel->joininfo != NIL || old_rel->has_eclass_joins ||
            has_join_restriction(root, old_rel))
        {

            if (level == 2)     /* consider remaining initial rels */
                first_rel = foreach_current_index(r) + 1;
            else
                first_rel = 0;

        }
        else
        {
            make_rels_by_clauseless_joins(root, old_rel, joinrels[1]);
        }
    }
```

逐行解说：把每个 (level-1)-表的 joinrel 与**单表**（`joinrels[1]`）配对——这同时覆盖了左深（joinrel 当外表）与右深（单表当外表），因为 `make_join_rel`（joinrels.c:699）内部会双向尝试。if 条件是关键剪枝：只有当 old_rel 身上有连接子句（`joininfo`）、等价类连接（`has_eclass_joins`）或连接顺序约束（外连接/半连接产生的 SpecialJoinInfo）时，才与"有子句相连的"单表组合（`make_rels_by_clause_joins`，joinrels.c:284，内部检查 `have_relevant_joinclause`）；否则说明这个 rel 与谁都没有连接条件，只能笛卡尔积，那就与所有未含单表逐一笛卡尔（joinrels.c:126-141 注释）。`level == 2` 时的 `first_rel` 技巧：单表 × 单表是对称的，从当前位置之后开始扫即可避免 {A,B} 与 {B,A} 重复枚举一遍。

**第二段：bushy**（joinrels.c:153-203）

```c
    for (k = 2;; k++)
    {
        int         other_level = level - k;

        if (k > other_level)            /* 只需枚举到中点，(k, level-k) 对称 */
            break;

        foreach(r, joinrels[k])
        {
            RelOptInfo *old_rel = (RelOptInfo *) lfirst(r);
            ...
            if (old_rel->joininfo == NIL && !old_rel->has_eclass_joins &&
                !has_join_restriction(root, old_rel))
                continue;               /* 无连接子句的不参与 bushy */
            ...
            for_each_from(r2, joinrels[other_level], first_rel)
            {
                RelOptInfo *new_rel = (RelOptInfo *) lfirst(r2);

                if (!bms_overlap(old_rel->relids, new_rel->relids))
                {
                    if (have_relevant_joinclause(root, old_rel, new_rel) ||
                        have_join_order_restriction(root, old_rel, new_rel))
                    {
                        (void) make_join_rel(root, old_rel, new_rel);
                    }
                }
            }
        }
    }
```

逐行解说：k-表 joinrel × (level-k)-表 joinrel，k 从 2 到 level/2——两边都是连接结果，这正是 bushy 树。三重防爆条款：k 只到中点（组合对称）；`bms_overlap` 用位图集判定两 rel 不含相同基表（含同表的组合不合法）；**必须有相关连接子句或顺序约束**才调 `make_join_rel`（joinrels.c:149-151 注释明说这是为了 avoid unreasonable growth of planning time——bushy 空间的笛卡尔组合被整个砍掉）。

**第三段：兜底**（joinrels.c:224-260）：若本层一个 joinrel 都没造出来（全部 rel 的连接子句都指向子问题外部的表，README 里的 flatten 受限场景），强制做一轮笛卡尔积；若仍失败且并无特殊连接约束，则 `elog(ERROR, "failed to build any %d-way joins", ...)`——注释（joinrels.c:239-255）给出了一个 5 表半连接的例子说明为什么第 4 层可以合法地空着、到第 5 层才出现可行的 bushy 计划。

**路径层面的竞争**：每次 `make_join_rel` 会对该基表集合找到（或创建）唯一的 joinrel RelOptInfo，然后由 `add_paths_to_joinrel`（`src/backend/optimizer/path/joinpath.c:123`）为这对输入生成 nestloop/mergejoin/hashjoin 各种实现路径。不同组合方式（{A,B}+{C} 与 {B,C}+{A}）产出的路径**汇入同一个 RelOptInfo，在 add_path 里同台竞技**（README:168-172）——"连接顺序的枚举"与"连接方法的选择"就这样被优雅地解耦了。

### 5.4 GEQO 遗传算法概述

`geqo`（`src/backend/optimizer/geqo/geqo_main.c:75`）把连接顺序问题当作带约束的旅行商问题（TSP，geqo_main.c:69-71 注释）：一个"染色体"= 基表的一个排列（即一棵左深树的连接顺序）；适应度 = 按该顺序真实调用标准路径生成代码算出的最优代价（`geqo_eval`，所以**每个候选顺序内部的连接方法选择仍是精确的**，README:12-14）；用 Genitor 式稳态遗传（geqo_main.c:24）+ 默认 ERX 边重组交叉算子（geqo_main.c:148）迭代若干代，输出最优个体。代价是放弃全局最优保证与（早期版本的）计划可复现性——PG 用固定种子 `geqo_seed` 缓解后者。实践中更常见的做法是调大 `join_collapse_limit`/`from_collapse_limit` 或用 pg_hint_plan 固定顺序，而非依赖 GEQO。

**小结**：`standard_join_search` 逐层填 DP 表；`join_search_one_level` 在一层内按"左深+右深 → bushy → 笛卡尔兜底"三段枚举，以"必须有连接子句"为主剪枝；连接方法选择下沉到 `add_paths_to_joinrel`，所有候选在共享的 RelOptInfo 中由 `add_path` 按代价+排序+参数化裁决。超过 `geqo_threshold`(12) 个 FROM 项则整体切换为遗传算法。

## 6. 代价模型：costsize.c

### 6.1 GUC 参数就是公式系数

代价的单位是"一次顺序页读取 = 1.0"的相对值。全部系数（`src/include/optimizer/cost.h:24-34`）：

```c
#define DEFAULT_SEQ_PAGE_COST  1.0
#define DEFAULT_RANDOM_PAGE_COST  4.0
#define DEFAULT_CPU_TUPLE_COST  0.01
#define DEFAULT_CPU_INDEX_TUPLE_COST 0.005
#define DEFAULT_CPU_OPERATOR_COST  0.0025
#define DEFAULT_PARALLEL_TUPLE_COST 0.1
#define DEFAULT_PARALLEL_SETUP_COST  1000.0
#define DEFAULT_EFFECTIVE_CACHE_SIZE  524288    /* measured in pages */
```

这些就是同名 GUC 的默认值。`random_page_cost=4` 是机械盘时代的假设，SSD/云盘上普遍应调到 1.1~2——它直接控制第 6.3 节公式中随机 I/O 的惩罚，是"为什么 PG 不走我的索引"的头号嫌疑人。每条路径的代价分两段：`startup_cost`（吐出第一行前的开销）与 `total_cost`（全部行）——LIMIT 场景按比例只取前段，这就是 2.1 节 `tuple_fraction` 的消费方式。

### 6.2 cost_seqscan 逐行分析

（`src/backend/optimizer/path/costsize.c:271`）

```c
    if (param_info)
        path->rows = param_info->ppi_rows;
    else
        path->rows = baserel->rows;                 /* 行数估计来自选择率 */

    get_tablespace_page_costs(baserel->reltablespace,
                              NULL, &spc_seq_page_cost);

    disk_run_cost = spc_seq_page_cost * baserel->pages;     /* :300 */

    cpu_run_cost = cpu_per_tuple * baserel->tuples;         /* :307 */

    cpu_run_cost += path->pathtarget->cost.per_tuple * path->rows;
```

逐行解说：I/O 代价 = 页数 × 顺序页成本（表空间可覆写全局 GUC）；CPU 代价 = **全表元组数**（不是过滤后的行数！每行都要被摸一遍并计算 WHERE）×（每行基础成本 + qual 求值成本），其中 qual 成本由 `cost_qual_eval` 按每个操作符 `cpu_operator_cost` 累加；目标列表达式的求值则按**输出行数**计——被 WHERE 滤掉的行不必计算投影。若并行（costsize.c:313-332），CPU 部分除以 worker 折算系数而 I/O 不除（注释：操作系统预读已让顺序 I/O 难以再摊薄）。EXPLAIN 里 `Seq Scan cost=0.00..N` 的 N 你现在可以手算：`pages * 1.0 + tuples * (0.01 + 0.0025 * qual操作符数)`。

### 6.3 cost_index 与 Mackert-Lohman 公式

`cost_index`（costsize.c:546）分四步：

**第一步：问索引自己**（costsize.c:617-621）：`amcostestimate`（B-tree 是 `btcostestimate`）估出遍历索引本身的代价、**索引选择率**（要回表取的堆元组占比）与**相关性** `indexCorrelation`（索引序与堆物理序的相关系数，来自 pg_statistic）。

**第二步：估回表要摸多少堆页**——这是最精彩的部分。若把命中的 `tuples_fetched` 行按索引序逐个回表，同一堆页可能被反复访问；访问是否算"重复读"取决于缓存。PG 采用 Mackert-Lohman 论文（*Index Scans Using a Finite LRU Buffer*, TODS 1989）的验证模型，`index_pages_fetched`（costsize.c:898）注释给出公式（costsize.c:869-877）：

```
 PF =
     min(2TNs/(2T+Ns), T)            when T <= b
     2TNs/(2T+Ns)                    when T > b and Ns <= 2Tb/(2T-b)
     b + (Ns - 2Tb/(2T-b))*(T-b)/T   when T > b and Ns > 2Tb/(2T-b)
 T = 表页数, N = 表行数, s = 选择率, b = 可用缓冲页数
```

实现逐行（costsize.c:907-949）：

```c
    T = (pages > 1) ? (double) pages : 1.0;

    total_pages = root->total_table_pages + index_pages;    /* 4.2 节存的总页数 */
    b = (double) effective_cache_size * T / total_pages;    /* :915 */
    ...
    if (T <= b)     /* 整表放得进缓存：页数由生日悖论式碰撞公式给出，封顶 T */
    {
        pages_fetched = (2.0 * T * tuples_fetched) / (2.0 * T + tuples_fetched);
        if (pages_fetched >= T)
            pages_fetched = T;
        ...
    }
    else            /* 表大于缓存：取行数超过 lim 后,每行开始产生额外真实 I/O */
    {
        lim = (2.0 * T * b) / (2.0 * T - b);
        if (tuples_fetched <= lim)
            pages_fetched = (2.0 * T * tuples_fetched) / (2.0 * T + tuples_fetched);
        else
            pages_fetched = b + (tuples_fetched - lim) * (T - b) / T;
        ...
    }
```

注意 b 的算法（costsize.c:915）：`effective_cache_size` 被**按页数比例分摊**给查询涉及的所有表——这就是 `effective_cache_size` 影响计划的机制：它不分配任何内存，只改变这条公式里的 b，从而改变索引扫描的 I/O 估价。

**第三步：相关性插值**（costsize.c:730-746、785-787）：完全无关时（最坏）每页都是随机读：`max_IO_cost = pages_fetched * random_page_cost`；完全相关时（CLUSTER 过的表）回表近乎顺序：`min_IO_cost = random_page_cost + (pages_fetched - 1) * seq_page_cost`（且页数直接用 `selectivity × 表页数`）。真实代价按相关性平方插值：

```c
    csquared = indexCorrelation * indexCorrelation;
    run_cost += max_IO_cost + csquared * (min_IO_cost - max_IO_cost);
```

这解释了一个常见现场问题：同样的查询，按插入序自然聚簇的时间序列表走索引很便宜，乱序表却弃用索引——差别全在 `indexCorrelation`。

**第四步：CPU**（costsize.c:795-804）：与 seqscan 同构，但只对 `tuples_fetched` 行计算，且索引内元组另按 `cpu_index_tuple_cost` 计（在 amcostestimate 内）。Index-Only Scan 的优惠在 costsize.c:686/726：回表页数乘以 `(1 - allvisfrac)`——可见性地图中全可见页占比越高，回表越少，这就是 IOS 依赖 VACUUM 的代价模型解释。

`loop_count > 1` 分支（costsize.c:670-713）处理"作为 nestloop 内表被反复执行"的情形：把 M-L 公式套在**全部循环的总取行数**上再摊回单次——跨循环的缓存复用被显式建模。

### 6.4 三种连接的代价要点

连接代价都是两段式：`initial_cost_xxx` 快速粗估（供 `add_path_precheck` 提前淘汰，省去精算），幸存者再 `final_cost_xxx` 精算。

**Nestloop**（`initial_cost_nestloop` costsize.c:3396，`final_cost_nestloop` costsize.c:3478）骨架（costsize.c:3428-3456）：

```c
    startup_cost += outer_path->startup_cost + inner_path->startup_cost;
    run_cost += outer_path->total_cost - outer_path->startup_cost;
    if (outer_path_rows > 1)
        run_cost += (outer_path_rows - 1) * inner_rescan_start_cost;
    ...
    run_cost += inner_run_cost;
    if (outer_path_rows > 1)
        run_cost += (outer_path_rows - 1) * inner_rescan_run_cost;
```

即：外表一遍 + 内表 (N_outer) 遍，但第 2 遍起用 `cost_rescan` 的**重扫代价**（Material 化的内表重扫远便宜于首扫，这是优化器主动在内表上加 Material 节点的账本依据）。SEMI/ANTI 或内表唯一时（costsize.c:3436-3450）匹配即停，期望只扫内表的一部分，留给 final 阶段结合选择率精算。参数化内表路径（4.4 节）在这里收获：内表每遍只扫匹配部分，nestloop + inner index scan 因此常是 OLTP 点连接的冠军。

**Merge join**（costsize.c:3681/3978）：两输入按连接键有序，各扫一遍；代价大头是**为无序输入补 Sort**——若输入路径的 PathKeys 已匹配（interesting orders 的回报！）则 Sort 免掉。final 阶段还会用直方图估计两侧键值范围的重叠度，不重叠的尾部可以提前停扫。

**Hash join**（costsize.c:4320/4439）：startup = 扫完内表 + 建哈希表（每行付 `cpu_operator_cost` 的哈希计算）；run = 扫外表逐行探测。内表超过 `work_mem` 时分批（batch），外内表都要落盘重读，页数进 I/O 账。final 阶段的重点是**桶倾斜**：`estimate_hash_bucket_stats` 用 MCV 估出最常见值占比，探测代价按最忙桶的期望链长放大——这就是"倾斜键上 hash join 被估得很贵"的出处。

**小结**：`costsize.c` 是一组把统计信息（页数、行数、选择率、相关性、MCV）与硬件假设（GUC 系数）折算成两段式标量代价的纯函数。所有 EXPLAIN 数字都能在这里找到出处；所有 `enable_*` 开关并不删除路径，只是给对应路径记一个 `disabled_nodes` 计数使其在比较中垫底（如 costsize.c:336-337）。

## 7. 选择率估计：selfuncs.c

代价公式里的 `rows` 与选择率从哪来？每个操作符在 `pg_operator` 里登记了限制选择率函数（`oprrest`，如 `=` → `eqsel`）与连接选择率函数（`oprjoin`，如 `=` → `eqjoinsel`），优化器按 OID 回调。统计信息由 ANALYZE 写入 `pg_statistic`：每列的 NULL 比例（stanullfrac）、不同值个数（stadistinct）、**MCV**（最常见值及其频率数组）、**等高直方图**（histogram bounds）、物理相关性等。

### 7.1 eqsel / var_eq_const：等值选择率逐段走读

`eqsel`（`src/backend/utils/adt/selfuncs.c:302`）→ `eqsel_internal`（selfuncs.c:311）：先 `get_restriction_variable` 把子句规整为 `Var = 常量` 形态，不满足则返回缺省值 `DEFAULT_EQ_SEL`（0.005）；常量侧已知则进入 `var_eq_const`（selfuncs.c:370）：

```c
    if (constisnull)
        return 0.0;                     /* x = NULL 恒假（操作符 strict） */
    ...
    if (vardata->isunique && vardata->rel && vardata->rel->tuples >= 1.0)
    {
        selec = 1.0 / vardata->rel->tuples;     /* :405 唯一列：恰好 1 行 */
    }
    else if (HeapTupleIsValid(vardata->statsTuple) && ...)
    {
        ...
        if (get_attstatsslot(&sslot, vardata->statsTuple,
                             STATISTIC_KIND_MCV, ...))      /* :424 取 MCV */
        {
            ...
            for (i = 0; i < sslot.nvalues; i++)             /* :449 逐个比对 */
            { ... if (相等) { match = true; break; } }
        }

        if (match)
            selec = sslot.numbers[i];   /* :478 命中 MCV：直接用其精确频率 */
        else
        {
            /* 未命中：剩余概率均摊给其余 distinct 值 */
            for (i = 0; i < sslot.nnumbers; i++)
                sumcommon += sslot.numbers[i];
            selec = 1.0 - sumcommon - nullfrac;             /* :492 */
            ...
            otherdistinct = get_variable_numdistinct(vardata, &isdefault) -
                sslot.nnumbers;
            if (otherdistinct > 1)
                selec /= otherdistinct;                     /* :503 均摊 */
            ...
            if (sslot.nnumbers > 0 && selec > sslot.numbers[sslot.nnumbers - 1])
                selec = sslot.numbers[sslot.nnumbers - 1];  /* :510 封顶 */
        }
    }
```

三层瀑布：唯一索引/DISTINCT 保证 → `1/行数`；常量命中 MCV → 用 ANALYZE 实测频率（这就是 MCV 的意义：**头部值不猜，直接查表**）；未命中 → `(1 - MCV总频率 - NULL率) / 其余distinct数`，即"均匀分布"假设只施加于长尾，并且用最冷门 MCV 的频率封顶（未上榜的值不可能比上榜的更常见）。无统计信息时最终落到 `DEFAULT_EQ_SEL`。

### 7.2 scalarineqsel：范围条件 = MCV 精算 + 直方图插值

`scalarineqsel`（selfuncs.c:655）处理 `<`/`<=`/`>`/`>=`。无统计时有个趣味特例：CTID 上的范围比较按块号在表中的相对位置直接算（selfuncs.c:673-741）；一般列则（selfuncs.c:756-784）：

```c
    mcv_selec = mcv_selectivity(vardata, &opproc, collation, constval, true,
                                &sumcommon);            /* MCV 部分:逐个真算 */

    hist_selec = ineq_histogram_selectivity(root, vardata, ...);  /* :763 */

    selec = 1.0 - stats->stanullfrac - sumcommon;   /* 非NULL非MCV的占比 */
    if (hist_selec >= 0.0)
        selec *= hist_selec;
    else
        selec *= 0.5;       /* 没有直方图：认命,一半一半 */
    /* 之后再加回 mcv_selec（满足条件的 MCV 频率之和） */
```

结构清晰：总体 = MCV 部分（逐值代入操作符实算，频率求和）+ 直方图部分（只覆盖非 NULL 非 MCV 的长尾）。`ineq_histogram_selectivity`（selfuncs.c:1116）在等高直方图的边界数组上**二分**定位常量落在第几个桶，桶内再线性插值（数值类型可换算比例，其他类型取桶中点）——等高直方图保证每桶行数相同，所以"落在第 k 个桶的位置 x 处" ≈ 选择率 `(k-1+x)/桶数`。这与 Spark CBO 的 equi-height histogram、MySQL 8.0 的直方图原理一致，但 PG 的 ANALYZE 默认自动收集（`default_statistics_target=100` → 100 个 MCV + 100 个桶），MySQL 需手动 `ANALYZE TABLE ... UPDATE HISTOGRAM`。

### 7.3 eqjoinsel：连接选择率

`eqjoinsel`（selfuncs.c:2387）估计 `t1.x = t2.y` 的连接选择率，核心在 `eqjoinsel_inner`（selfuncs.c:2592）：

- **两侧都有 MCV 时**（selfuncs.c:2470-2484 准备，inner 内实施）：把两侧 MCV 列表做一次"迷你连接"——逐对（列表很长时先建哈希，selfuncs.c:2480）用真实等值函数比对，匹配上的频率乘积累加得到 MCV×MCV 部分的精确选择率；MCV×非MCV、非MCV×非MCV 部分按剩余频率与剩余 distinct 数均摊，最后从两侧视角各估一个总值取较小者（selfuncs.c:2703 附近，`selec = (totalsel1 < totalsel2) ? totalsel1 : totalsel2`）。
- **缺 MCV 时**退化为教科书公式（selfuncs.c:2730-2735）：

```c
        selec = (1.0 - nullfrac1) * (1.0 - nullfrac2);
        if (nd1 > nd2)
            selec /= nd1;
        else
            selec /= nd2;
```

即 `(1-null1)(1-null2) / max(nd1, nd2)`——System R 论文原公式的直接后裔（注释在 selfuncs.c:2707-2717 给出推导：R1 每行至多连上 `N2/nd2` 行，故选择率不超过 `1/nd2`，对称地也不超过 `1/nd1`）。半连接走 `eqjoinsel_semi`（selfuncs.c:2751），估的是"外表行至少有一个匹配的概率"，口径不同。

**小结**：选择率 = 统计字典（MCV 头部精确值 + 直方图长尾插值 + nullfrac/ndistinct）驱动的概率计算，层层回退到硬编码缺省值（`DEFAULT_EQ_SEL` 等）。工程启示：估计离谱时优先怀疑统计过期（`ANALYZE`）、`default_statistics_target` 太小、以及多列相关性（这是本模型的独立性盲区，PG 用 `CREATE STATISTICS` 扩展统计补救）。

## 8. Path → Plan：create_plan 与 set_plan_references

### 8.1 create_plan：把骨架长成肌肉

`create_plan`（`src/backend/optimizer/plan/createplan.c:345`）接过最优 Path 树，调 `create_plan_recurse`（createplan.c:396）自顶向下按 `pathtype` 分发（`T_SeqScan` → `create_seqscan_plan`，`T_HashJoin` → `create_hashjoin_plan`……），做 Path 阶段刻意省略的脏活：为每个节点构造真正的目标列表；把 RestrictInfo 蜕壳成裸表达式挂到 `plan->qual`，**并剔除已被索引条件蕴含的冗余 qual**（cost 阶段为省时间没做的证明这里补上，costsize.c:826-834 注释有交代）；为 nestloop 的参数化内表分配运行时 Param 编号（`curOuterParams`）；在需要处插入 Sort/Material 等胶水节点。函数注释（createplan.c:336-338）特别强调：此时的 Var 仍是"planner 视角"的编号——留给下一步。

### 8.2 set_plan_references：最后的接线工

`set_plan_references`（`src/backend/optimizer/plan/setrefs.c:288`）做发车前检查，职责清单在 setrefs.c:230-255 的注释里，要点：

1. **压平范围表**：所有子查询层级的 rtable 合并成一张全局表（`glob->finalrtable`），各节点的 `scanrelid` 统一加偏移（setrefs.c:292 `rtoffset`）；
2. **上层 Var 改指下层输出槽**：扫描节点的 Var 保持指向表列，而 join/agg/sort 等上层节点的 Var 全部改写为 `OUTER_VAR/INNER_VAR + 子节点 targetlist 序号`——执行器由此才能用"取第 k 个输出列"的方式高速取值，不必按表列号查找；
3. 决断 `AlternativeSubPlan`（按估计执行次数二选一）、预解析操作符的实现函数 OID（省得执行期查 syscache）、登记计划依赖的对象 OID 列表（`relationOids`/`invalItems`，plancache 靠它做计划失效）；
4. 顺手删除无用的 SubqueryScan/Append/MergeAppend 中间节点（setrefs.c:257-268：rangetable 压平后它们的"变量命名空间缓冲"作用消失了）。

产物回到 2.1 节第 6 步，装进 `PlannedStmt`——优化器旅程结束，交棒执行器。

**小结**：Path→Plan 是"补全执行细节"，setrefs 是"全局编号统一 + 变量接线 + 依赖登记"。两者共同完成了从"搜索友好表示"到"执行友好表示"的转换——再次呼应本章开头：Path/Plan 分离是 PG 优化器最重要的工程设计。

## 全章总结

一条 SQL 在优化器中的完整路线图（括号内为本章小节）：

```
standard_planner (2.1)
 └─ subquery_planner (2.2)
     ├─ pull_up_sublinks: IN/EXISTS → SEMI/ANTI join (2.3)
     ├─ pull_up_subqueries + is_simple_subquery (2.4)
     ├─ preprocess_expression: 常量折叠 (2.5)
     └─ grouping_planner: UpperRelationKind 流水线 (3)
         └─ query_planner → make_one_rel (4)
             ├─ set_base_rel_pathlists: seq/index/bitmap/tid (4.3, 4.4)
             └─ make_rel_from_joinlist (5.1)
                 ├─ standard_join_search: DP 逐层 (5.2, 5.3)
                 │    代价: costsize.c (6)  选择率: selfuncs.c (7)
                 └─ geqo: ≥12 表时的遗传算法 (5.4)
 ├─ create_plan: Path → Plan (8.1)
 └─ set_plan_references: 扁平化与接线 (8.2)  → PlannedStmt
```

推荐动手实验：打开 `debug_print_plan`，选一条三表连接 SQL，对照 6.2/6.3 节手算 EXPLAIN 里每个节点的 cost；再 `SET enable_hashjoin = off` 观察 `disabled_nodes` 机制如何让次优路径上位。

## 自测问题

**1. PG 的连接搜索与 System R 原始算法有哪两个主要区别？各自的源码落点在哪里？**

答：（a）System R 只枚举左深树，PG 枚举含 bushy 树的完整空间——`join_search_one_level`（joinrels.c:78）的第二段 `for (k = 2;; k++)` 循环显式组合两个 joinrel，剪枝条件是必须存在相关连接子句；（b）System R 的 interesting order 只看排序列本身，PG 的 PathKey 经过 EquivalenceClass（pathnodes.h:1806 的 `pk_eclass`），`a.x = b.y` 时按 a.x 有序等价于按 b.y 有序。此外 PG 还有 System R 没有的 GEQO 兜底（allpaths.c:3915，≥ geqo_threshold=12 表时）。

**2. `WHERE x IN (SELECT ...)` 什么条件下能变成连接？变成什么连接？为什么这很重要？**

答：`pull_up_sublinks`（prepjointree.c:670）在 WHERE/JOIN ON 顶层发现 ANY SubLink 且子查询不相关（不引用外层变量）时，调 `convert_ANY_sublink_to_join`（subselect.c:1341）将其变为 **JOIN_SEMI**（NOT IN/NOT EXISTS 变 JOIN_ANTI，subselect.c:1450）。重要性：不转换则成为逐外层行执行的 SubPlan（O(N×M)）；转换后进入连接搜索空间，可选 hash/merge 半连接并参与连接顺序优化。

**3. `effective_cache_size` 并不分配任何内存，它是如何影响计划选择的？**

答：它只作为 Mackert-Lohman 公式的参数 b。`index_pages_fetched`（costsize.c:898）中 `b = effective_cache_size * T / total_pages`（costsize.c:915，按查询涉及的总表页数比例分摊），b 越大，公式估出的重复堆页命中缓存越多、实际 I/O 页数越少，索引扫描的 `cost_index` 估价越低，索引路径越容易胜出。它是纯粹的"优化器对外部缓存规模的假设"。

**4. `var_eq_const`（selfuncs.c:370）估计 `x = 常量` 的选择率分哪三层？常量不在 MCV 里时的公式是什么？**

答：第一层：x 有唯一约束 → `1/tuples`；第二层：常量与 MCV 列表逐个真实比对，命中 → 直接用该值的实测频率 `sslot.numbers[i]`；第三层：未命中 → `selec = (1 - ΣMCV频率 - nullfrac) / (ndistinct - MCV个数)`，即剩余概率均摊给长尾 distinct 值，并以最小 MCV 频率封顶。完全无统计时用 `DEFAULT_EQ_SEL`（0.005）。

**5. 为什么 Path 和 Plan 要分成两套结构？各自缺了什么、多了什么？`set_plan_references` 又补了什么？**

答：搜索空间指数级大，候选表示必须廉价：Path 只带 pathtype、代价、行数、pathkeys、参数化信息，无目标列表和可执行 qual（README:23-26）；胜出者才由 `create_plan`（createplan.c:345）展开为带完整 targetlist/qual 的 Plan。Plan 多了执行细节但变量仍是 planner 编号；`set_plan_references`（setrefs.c:288）最后压平多层 rangetable、把上层节点的 Var 改写为指向子节点输出槽位（OUTER_VAR/INNER_VAR + 序号）、预解析函数 OID 并登记计划失效依赖，使 Plan 成为执行器可直接高速运行、plancache 可管理的最终形态。
