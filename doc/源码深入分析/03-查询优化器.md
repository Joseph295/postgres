# 03 · 查询优化器：从 Query 到 PlannedStmt 的深度源码分析

> 对应《PostgreSQL 源码学习指南》第 4 章的展开。代码基于 master 分支（版本 20devel）。
> 所有代码引用格式为 `文件路径:行号`，均已在本仓库逐一现场核实；代码摘录与源文件逐字一致。

## 本章导读

优化器是 PostgreSQL 中理论密度最高的模块。它接收解析器/重写器产出的 `Query` 树（逻辑上就是一棵关系代数表达式），输出可供执行器直接运行的 `PlannedStmt`。这个转换不是简单的翻译，而是一次**在指数级搜索空间中的最优化求解**——其算法框架直接继承自 1979 年 Selinger 等人的 System R 论文，至今仍是所有工业级 CBO（基于代价的优化器）的原型，包括 Spark Catalyst CBO 与 MySQL 优化器。

v2 版在 v1 骨架上做了三类深化，读之前先知道"深"在哪里：

1. **完整走读而非摘录**：`standard_join_search`、`join_search_one_level`、`add_path`、`cost_seqscan`、`cost_index`、`eqjoinsel_inner`、`create_index_paths → build_index_paths` 全部逐段走完，省略的分支明确注明省略原因；
2. **"所以然"三问**：每个机制显式回答——它解决什么问题（没有它会怎样，附组合数估算与反例构造）；为什么选这个方案而不是替代方案（DP vs Volcano/Cascades、左深 vs bushy、GEQO vs 随机化贪心）；它的正确性/最优性依赖什么假设（独立性假设、误差沿 join 树的指数放大，给数值推演）；
3. **手算闭环与 what-if 推演**：两个可以对着 EXPLAIN 逐位核对的手算例子（seqscan 代价、两表 join 行数），三个"参数/统计变化导致计划翻车"的定量推演，以及从 git 历史挖出的五个关键设计决策（附 commit hash）。

本章按照一条 SQL 在优化器内的真实旅程展开：

1. **理论对照**（§1）：Selinger 1979 的三大思想在 PG 源码中的落点；框架式优化器（Volcano/Cascades）对照，回答"PG 为什么不需要 memo"；
2. **入口与预处理**（§2）：`standard_planner` → `subquery_planner`，三个"逻辑改写"——子链接上拉、子查询展平、常量折叠；
3. **上层规划的阶段化**（§3）：`grouping_planner` 与 `UpperRelationKind`（PG 9.6 pathification 重构的产物）；
4. **基表路径生成**（§4）：`make_one_rel` → `set_base_rel_pathlists`，重点完整走读 `create_index_paths → build_index_paths` 主线；
5. **add_path 支配剪枝**（§5）：搜索空间可控的真正关键——四维支配比较的全函数走读，v1 只提了一句，这里升格为独立一节；
6. **连接搜索**（§6）：`standard_join_search` + `join_search_one_level` 全函数逐段；GEQO；
7. **代价模型**（§7）：`costsize.c` 公式逐项拆解（量纲分析 + 数值例子），含 Mackert-Lohman 缓存模型与 seqscan 手算闭环；
8. **选择率估计**（§8）：`selfuncs.c`，含 `eqjoinsel_inner` 全函数走读与两表 join 行数手算闭环、误差放大推演；
9. **what-if 推演**（§9）：统计过期、相关列、work_mem 三个定量案例；
10. **收尾**（§10）：`create_plan` 与 `set_plan_references`；
11. **设计历史**（§11）：五个关键 commit 的来龙去脉。

## 前置知识

- **Query 结构体**（`src/include/nodes/parsenodes.h`）：优化器的输入。`rtable` 是范围表（FROM 里的每个表/子查询一项），`jointree` 是连接结构与 WHERE 条件，`targetList` 是投影列表。
- **两个核心概念**（见指南 4.1 节）：
  - `RelOptInfo`：优化过程中"一个或一组关系"的抽象，身上挂着它的全部候选路径（`pathlist`）。基表、子查询、连接中间结果，统统是 RelOptInfo。
  - `Path`：轻量级"计划骨架"，只携带代价（`startup_cost`/`total_cost`）、行数估计（`rows`）和排序信息（`pathkeys`），不带执行细节。**先在便宜的 Path 空间里搜索，胜出后才展开成重量级 Plan**——与 Spark Catalyst 中"先在逻辑/物理计划的轻量表示上做 CBO，最后才 codegen"是同一工程直觉。
- **与 MySQL 的宏观对比**：MySQL（8.0 之前的老优化器）本质上是贪心的左深树搜索 + 启发式剪枝（`greedy_search`），且优化与执行的数据结构耦合较深。PG 则是教科书式的自底向上动态规划，枚举包括 bushy 树在内的完整空间，且 Path/Plan 分离。读完本章你会对这句话有具体的体感。
- 建议先通读 `src/backend/optimizer/README`（1623 行），它是 PG 源码树中质量最高的设计文档之一。本章会反复引用它。

## 1. 理论根基：Selinger 1979、搜索空间的规模，与框架式优化器对照

### 1.1 问题有多大：先把组合数算清楚

在讲任何算法之前，先量化"不加控制的连接顺序搜索空间到底有多大"，这是理解后面所有剪枝设计的前提。

n 个表的连接计划 = 一棵二叉树的形状 × 叶子上表的排列。有 n 个带标号叶子的二叉树棵数为：

```
  n! × Cat(n-1)，其中 Cat(k) = C(2k,k)/(k+1) 是卡特兰数
```

代入几个值（只算连接顺序，还没乘每个连接节点 3 种实现方法）：

| n | n! | Cat(n-1) | 连接顺序总数 | ×3^(n-1) 种方法组合 |
|---|-----|----------|--------------|---------------------|
| 4 | 24 | 5 | 120 | ≈ 3.2 千 |
| 8 | 40320 | 429 | ≈ 1.7 × 10^7 | ≈ 3.8 × 10^10 |
| 12 | 4.79 × 10^8 | 58786 | ≈ 2.8 × 10^13 | ≈ 5.0 × 10^18 |

12 表全枚举是 10^18 量级——按每秒评估 10 亿个计划算也要一百多年。PG 用三层手段把它压回可行范围：

1. **动态规划**（§6）：利用"包含同一组基表的中间结果等价"这一最优子结构，把"枚举计划"变成"枚举子问题对"。子问题对总数的上界是 3^n（每个基表要么在左输入、要么在右输入、要么不在本子问题——即对每个 relids 集合枚举其二划分），n=12 时是 531441，直接从 10^18 掉到 10^5 量级；
2. **连接图剪枝**（§6.3）：只组合有连接条件相连的子问题对。对链式连接图（a₁⋈a₂⋈…⋈aₙ 只有相邻表有条件），合法子问题只有连续区间，共 n(n+1)/2 个，DP 配对数是 O(n³)——n=12 时只有几百个组合；
3. **add_path 支配剪枝**（§5）：即便子问题数可控，每个子问题里的候选路径仍会随连接方法、排序、参数化组合而膨胀。add_path 保证每个 RelOptInfo 只留下"帕累托前沿"上的少数路径（实践中通常个位数到十几条），使上层组合的乘法基数保持很小。

这三层是乘法关系：DP 管"子问题个数"，连接图剪枝管"配对个数"，add_path 管"每个子问题里的路径条数"。任何一层失守，规划时间都会失控——这也是为什么 §5 和 §6 是本章篇幅最重的两节。

### 1.2 Selinger 三支柱在 PG 的落点

Selinger 等人的 *Access Path Selection in a Relational Database Management System*（SIGMOD 1979，System R）确立了 CBO 的三根支柱。逐一对照 PG 源码：

**支柱一：动态规划枚举连接顺序。** PG 的实现就在 `standard_join_search`（`src/backend/optimizer/path/allpaths.c:3952`），注释原文直接写着 "We employ a simple 'dynamic programming' algorithm"（allpaths.c:3964）。`root->join_rel_level[j]` 保存所有恰好包含 j 个基表的 RelOptInfo——这正是 DP 的备忘录表。关键性质（README:174-182 有专门说明）：某一层的所有路径构造完毕后才会构造更高层，因此每层可以放心地调用 `set_cheapest()` 淘汰劣质路径，不必担心上层还引用着它们——这同时简化了路径的内存管理（README 原文："Without this, memory management for paths would be much more complicated."）。

**支柱二：Interesting Orders → PathKeys。** System R 发现：单看代价淘汰路径会犯错。一条较贵但**输出有序**的路径（比如索引扫描）可能让上层省掉一次排序（merge join 或 ORDER BY），总账反而更便宜。所以每个子问题要按"有趣的顺序"分别保留最优解。PG 把"顺序"形式化为 `PathKey`（`src/include/nodes/pathnodes.h:1806`）：

```c
typedef struct PathKey
{
    ...
    /* the value that is ordered */
    EquivalenceClass *pk_eclass pg_node_attr(copy_as_scalar, equal_as_scalar);
    Oid			pk_opfamily;	/* index opfamily defining the ordering */
    CompareType pk_cmptype;		/* sort direction (ASC or DESC) */
    bool		pk_nulls_first; /* do NULLs come before normal values? */
} PathKey;
```

注意 `pk_eclass` 指向 **EquivalenceClass**（等价类）而非具体列：如果 WHERE 里有 `a.x = b.y`，那么"按 a.x 排序"与"按 b.y 排序"是同一个 PathKey。这是 PG 对 System R 的重要增强——interesting order 的判定经过了等价类的传递闭包（详见 README 的 EquivalenceClasses 一节，README:693 起）。路径的取舍规则在 README:684-692：**只有当一条路径既不更便宜、排序也不更有用时才被丢弃**，具体裁决代码就是 §5 的 `add_path`。

**没有 PathKeys 会错过什么？构造一个三表反例。** 设 `t1(a)` 上有 B-tree 索引，查询：

```sql
SELECT * FROM t1 JOIN t2 ON t1.a = t2.a JOIN t3 ON t2.b = t3.b
ORDER BY t1.a LIMIT 10;
```

设 t1 有 100 万行。在 t1 这个基表 RelOptInfo 里有两条候选：SeqScan（total_cost ≈ 22500）和沿 a 的 IndexScan（total_cost ≈ 45000，随机 I/O 贵一倍，但 pathkeys = `(t1.a)`）。若只按代价取优，IndexScan 在基表层就被淘汰；三表连接完成后为满足 ORDER BY 必须对整个连接结果（可能还是百万行级）做全量 Sort，startup_cost 巨大，LIMIT 10 毫无用武之地。保留 IndexScan 后，优化器可以构造 `NestLoop(NestLoop(IndexScan(t1), t2), t3)`——每层都保序传递 `(t1.a)`（等价类使它同时也是 `(t2.a)` 序），最终计划 startup 近乎为 0，取满 10 行即停。两个计划的实际执行时间可以差三四个数量级。**"较贵路径的排序属性是对未来的期权"**，这就是 interesting orders 的全部含义；PathKeys 是这份期权的形式化载体。

**支柱三：代价驱动。** 对应 §7 的 `costsize.c` 与 §8 的 `selfuncs.c`：把统计信息与硬件假设折算成可比的标量，使前两根支柱的"优"有定义。

### 1.3 为什么是自底向上 DP，而不是 Volcano/Cascades？

工业界另一大流派是自顶向下的**框架式优化器**：Volcano（Graefe & McKenna, 1993）及其后继 Cascades（SQL Server、CockroachDB、Greenplum Orca 都属此系）。理解 PG 为什么不走这条路，比记住 PG 走了哪条路更重要。

Cascades 的核心构件是 **memo**：一张"等价类表"，每个 group 存一组逻辑等价的表达式（logical/physical expression），优化过程是不断对 group 应用**变换规则**（transformation rules：连接交换律、结合律、聚合下推……）并按需（on demand、带 promise 排序和代价上界剪枝）向下递归求子 group 的最优实现。它的强项是**可扩展性**：新增一个逻辑算子或一条等价变换，只要写一条规则插进去。

PG 的回答分三点：

1. **PG 事实上有"穷人版 memo"**：以 relids 位图集为 key 的 `root->join_rel_hash` / `join_rel_level[]`——每个基表集合恰好对应一个 RelOptInfo，不同连接顺序产出的路径汇入同一个 RelOptInfo 竞争（README:168-173："all the ways to produce the same set of joined base rels will share the same RelOptInfo, so the paths produced from different join combinations that produce equivalent joinrels will compete in add_path()"）。Cascades memo 的"group"概念，在"逻辑自由度只剩连接顺序"的世界里就退化成"基表集合"，一个哈希表足矣。
2. **PG 把逻辑变换从搜索中剥离了**：Cascades 需要在搜索过程中反复应用变换规则，是因为它把"逻辑改写"和"物理实现选择"统一进了一个搜索框架。PG 则把确定性有利的逻辑改写（子链接上拉、子查询展平、外连接消除、常量折叠，见 §2）全部**前置为一次性的预处理**，搜索阶段的逻辑自由度只剩连接顺序与实现方法——这正是 DP 能高效覆盖的形状。代价是：不能表达"改写是否有利依赖于代价"的探索（例如 group-by 该放在 join 之前还是之后）。有趣的是，PG 正在逐步为这类问题打补丁：本版本（20devel）合入的 Eager Aggregation（commit `8e11859102`，见 §11）就是给 joinrel 挂一个平行的 `grouped_rel`，让"提前部分聚合"作为额外候选参与竞争——用"多养一套 RelOptInfo"的办法在 DP 框架内模拟了一条 Cascades 规则。
3. **自底向上对"全量最优"更划算**：自顶向下的优势在于分支上界剪枝（branch-and-bound）和按需求解——当上层只需要特定 order 的解时可以少算。但 SQL 查询最终几乎总是要"全部子问题的最优解"（每层每种 interesting order 都可能被上层用到），按需求解的节省有限，而自底向上一遍扫过、内存管理简单（下层冻结后可安全 pfree 被支配路径，见 §5 引用的 README:174-182）。对一个手写 C、追求单次规划毫秒级完成的系统，这是更务实的选择。

**左深 vs bushy 的权衡也在这里做了与 System R 相反的决定。** System R 只枚举左深树（内表必须是基表），把空间压到 n! 级；PG 枚举完整 bushy 空间（README:163-173）：

> We consider left-handed plans (the outer rel of an upper join is a joinrel, but the inner is always a single list item); right-handed plans (outer rel is always a single item); and bushy plans (both inner and outer can be joins themselves).  For example, when building {1 2 3 4} we consider joining {1 2 3} to {4} (left-handed), {4} to {1 2 3} (right-handed), and {1 2} to {3 4} (bushy), among other choices.

为什么值得？bushy 树的收益场景很实际：`(A⋈B)⋈(C⋈D)` 中若 A⋈B 与 C⋈D 各自都是高选择率的小结果，先分别做小连接再 hash join 两个小中间结果，可能远优于任何左深顺序（左深必须让某个大中间结果一路当外表滚下去）。雪花模型多维表连接、以及 §6.3 会看到的"半连接展平后某些层级根本不存在合法左深计划"的场景（joinrels.c:239-255 的注释例子），都依赖 bushy。代价是子问题配对数从 O(n²) 升到最坏 3^n——PG 用"必须有连接子句相连"的剪枝（§6.3）与 `from_collapse_limit`/`join_collapse_limit`（README:89-102，默认都是 8，把一个大问题切成多个小 DP 问题递归求解）兜住规划时间。

> **对比**：Spark Catalyst 的 CBO join reorder 同样是 DP（`CostBasedJoinReorder` 规则），同样默认只在有连接条件时枚举；MySQL 老优化器则是深度优先贪心 + `optimizer_prune_level` 剪枝，只搜左深空间。PG 在这条谱系上是最"学院派"的。

### 1.4 正确性与最优性依赖什么假设

DP 给出的是"在代价模型下"的最优计划。这句话有两个前提，都会在后文被量化检验：

1. **最优子结构假设**：子问题的最优解组合出全局最优解。PathKeys 机制（每种有用排序各留最优）+ 参数化路径（每种参数化各留最优）+ startup/total 双指标，就是为了让"最优"是一个**向量**而非标量——只要向量维度覆盖了所有"上层可能在意的属性"，最优子结构就成立。如果某天上层开始在意一种新属性（例如 9.6 之前上层不参与竞争、并行化引入 partial path），保留策略就必须扩维，否则会系统性丢最优解——§5 的 add_path 签名里 `parallel_safe`、`disabled_nodes` 的历次加入就是这个逻辑的化石层。
2. **代价模型与选择率的独立性假设**：多个谓词的联合选择率 = 各自选择率之乘积；连接键在两表间均匀分布。这两个假设错起来不是线性的——§8.4 会给出误差沿 join 树指数放大的数值推演，§9.2 给出相关列导致计划灾难的完整链条。

**小结**：不加控制的搜索空间是 10^18 量级（12 表）；PG 用 DP（子问题数 ≤3^n）、连接图剪枝（链式图降到 O(n³) 配对）、add_path 帕累托剪枝（每子问题只留前沿路径）三层乘法压缩。PG 选择自底向上 DP 而非 Cascades，因为逻辑改写被前置后搜索自由度只剩连接顺序，relids 哈希就是天然 memo；选择 bushy 而非左深，用"有连接条件才组合"控制爆炸。所有"最优"都以最优子结构与独立性假设为前提。

## 2. 入口与预处理：standard_planner → subquery_planner

### 2.1 standard_planner：优化器的总装车间

入口 `planner()`（`src/backend/optimizer/plan/planner.c:328`）只做一件事：有 `planner_hook` 就调 hook（pg_hint_plan 等插件的注入点），否则调 `standard_planner`（planner.c:346）。后者的主干流程（省略细节后）：

1. 建立 `PlannerGlobal`（planner.c:366）——跨所有子查询层级共享的全局状态；
2. 判定能否使用并行计划（planner.c:413-429）：只有 SELECT、无修改性 CTE、`max_parallel_workers_per_gather > 0` 等条件全满足才扫描查询树评估并行安全性；
3. 计算 `tuple_fraction`（planner.c:452-477）：游标场景下用户可能只取前几行，用 `cursor_tuple_fraction` 让优化器偏向"快速启动"计划——这是 System R 论文里没有、工程实践逼出来的维度；
4. **递归规划主体**（planner.c:532）：`root = subquery_planner(glob, parse, ...)`；
5. **选出最优路径并物化**（planner.c:536-539）：

```c
    final_rel = fetch_upper_rel(root, UPPERREL_FINAL, NULL);
    best_path = get_cheapest_fractional_path(final_rel, tuple_fraction);

    top_plan = create_plan(root, best_path);
```

逐行解说：`UPPERREL_FINAL` 是上层规划的最后一个阶段关系（见 §3），此时它的 pathlist 里躺着整条查询的全部候选完整路径；`get_cheapest_fractional_path` 按 `tuple_fraction` 挑选——取全部行时比 `total_cost`，只取前 N% 时按比例折算 startup/run 代价；`create_plan` 把胜出的 Path 树展开成 Plan 树（§10）。

6. 收尾（planner.c:643）：`top_plan = set_plan_references(root, top_plan);` 做变量扁平化（§10），最后组装 `PlannedStmt`。

### 2.2 subquery_planner：预处理流水线

`subquery_planner`（planner.c:770）对**每一层查询**执行一遍，子查询通过递归调用处理。它前半部分是一条"逻辑改写流水线"，与 Spark Catalyst 的 rule-based 优化阶段（谓词下推、子查询去关联等 Rule）职责相当，只是 PG 用手写的顺序调用而非规则引擎。按源码顺序：

```c
    if (parse->cteList)
        SS_process_ctes(root);                  /* planner.c:844 */
    ...
    if (parse->hasSubLinks)
        pull_up_sublinks(root);                 /* planner.c:902 */

    preprocess_function_rtes(root);             /* planner.c:910 */

    pull_up_subqueries(root);                   /* planner.c:916 */
```

随后是对全部表达式统一调用 `preprocess_expression`（planner.c:1429 定义，targetList/havingQual/LIMIT 等逐一过一遍），再做外连接消除、`grouping_planner`。下面拆解三大改写。

**为什么这些改写要前置、而且是无条件执行？** 回到 §1.3 的框架对比：这些改写都满足"改写后计划空间严格是改写前的超集或语义等价的更优空间"——上拉后的半连接**仍然可以**被规划成逐行执行的样子（nestloop semi join 内表放子查询计划），但反过来不行。既然改写单向有利，就不需要代价裁决，一次性做完即可。真正"是否有利取决于代价"的改写（比如聚合提前）才需要进搜索空间——这条分界线划得干净，是 PG 优化器结构清晰的根源。

### 2.3 pull_up_sublinks：IN/EXISTS 变半连接的机制

`pull_up_sublinks`（`src/backend/optimizer/prep/prepjointree.c:670`）递归遍历 jointree 的 WHERE 和 JOIN/ON 子句，找到**顶层的** `ANY`（即 IN）和 `EXISTS` SubLink，尝试调用：

- `convert_ANY_sublink_to_join`（`src/backend/optimizer/plan/subselect.c:1341`）
- `convert_EXISTS_sublink_to_join`（subselect.c:1593）

以 IN 为例，`WHERE x IN (SELECT y FROM sub ...)` 的改写过程（subselect.c:1341 起的函数注释写得非常清楚）：

1. 检查子查询是**不相关**的（不引用外层变量；相关子查询在特定条件下走 LATERAL 路线）；
2. 把子 SELECT 作为一个新的 `RangeTblEntry` 加入外层查询的 rtable；
3. 构造一个 `JoinExpr`，连接条件是 `x = y`，连接类型为（subselect.c:1450）：

```c
	result->jointype = under_not ? JOIN_ANTI : JOIN_SEMI;
```

即 `IN` → **半连接**（JOIN_SEMI：外表每行最多匹配一次就够），`NOT EXISTS` → **反连接**（JOIN_ANTI：只输出无匹配的外表行；`convert_EXISTS_sublink_to_join` 内对应的是 subselect.c:1758 的同一行代码）。

**为什么这个改写是优化的分水岭？** 不改写时，SubLink 会变成 SubPlan——对外层每一行执行一次子查询（最坏 O(N×M)，即 MySQL 老版本臭名昭著的 DEPENDENT SUBQUERY）。改写成半连接后，它就进入了 §6 的连接搜索空间：优化器可以自由选择 hash semi join、merge semi join，甚至调换连接顺序。半连接/反连接带来的连接顺序合法性约束由 `SpecialJoinInfo` 记录、`join_is_legal` 检查（README:188-194、README:229-239 详细列出了 SEMI/ANTI 的重结合规则）。

### 2.4 pull_up_subqueries 与 is_simple_subquery：子查询展平的准入条件

`pull_up_subqueries`（prepjointree.c:1220）处理 FROM 里的子查询：`SELECT ... FROM (SELECT ...) sub`。不展平时子查询是"黑盒"——单独规划、外层只见一个 SubqueryScan，跨黑盒的连接顺序优化全部丧失（README:279-290）。展平就是把子查询的表直接并入外层 jointree。

能否展平由 `is_simple_subquery`（prepjointree.c:1954）裁决，条件摘录（prepjointree.c:1969-2074）：

```c
    if (subquery->setOperations)        /* 有 UNION/INTERSECT/EXCEPT：不行 */
        return false;

    if (subquery->hasAggs ||            /* 聚合 */
        subquery->hasWindowFuncs ||     /* 窗口函数 */
        subquery->hasTargetSRFs ||      /* 目标列里的集合返回函数 */
        subquery->groupClause ||        /* GROUP BY */
        subquery->groupingSets ||
        subquery->havingQual ||
        subquery->sortClause ||         /* ORDER BY */
        subquery->distinctClause ||     /* DISTINCT */
        subquery->limitOffset ||        /* OFFSET */
        subquery->limitCount ||         /* LIMIT */
        subquery->hasForUpdate ||       /* FOR UPDATE/SHARE */
        subquery->cteList)              /* WITH */
        return false;

    if (rte->security_barrier)          /* 安全屏障视图：防信息泄漏 */
        return false;
    ...
    if (contain_volatile_functions((Node *) subquery->targetList))
        return false;                   /* volatile 函数：防多次求值 */
```

逐条讲原因：聚合/GROUP BY/LIMIT 等改变了行的"粒度"或数量，拉平会改变语义；`FOR UPDATE` 拉平会把加锁语义提升到错误的层级（prepjointree.c:1976-1980 注释）；安全屏障视图拉平后其内部 Var 散落到外层，选择率估计函数可能被恶意函数利用来探测数据（prepjointree.c:1996-2002）；volatile 函数（如 `random()`）拉平后可能被复制到外层多处、执行多次，结果改变（prepjointree.c:2065-2074）。LATERAL 子查询还有额外的外连接相关限制（prepjointree.c:2007-2063）。

也就是说：**"简单"的标准 = 拉平后语义严格不变**。这份清单是排查"为什么我的视图/子查询没有被展开优化"的第一现场。

### 2.5 preprocess_expression：常量折叠

`preprocess_expression`（planner.c:1429）对每个表达式做一串规范化，核心一步（planner.c:1475）：

```c
    expr = eval_const_expressions(root, expr);
```

`eval_const_expressions`（实现在 `src/backend/optimizer/util/clauses.c`）做：常量子表达式求值（`2+2` → `4`）、只含常量参数的不可变函数内联求值、`CASE`/`AND`/`OR` 的常量短路化简、简单 SQL 函数的内联展开等。随后（planner.c:1480-1520）：`canonicalize_qual` 把 qual 规范化（展平嵌套 AND/OR、提取 OR 各分支的公共项）；`convert_saop_to_hashed_saop` 为大 IN 列表（`x = ANY('{...}')`）标记哈希查找；`SS_process_sublinks` 把没能在 2.3 节变成连接的 SubLink 展开为 SubPlan；最后 `make_ands_implicit` 把 qual 变成隐式 AND 的列表形式——后续所有优化代码都假定这个形式。

**小结**：`subquery_planner` 的预处理是"先做确定性有利的逻辑改写，再进代价搜索"。三大改写的判断标准全部是**语义不变性**：IN/EXISTS 上拉靠半/反连接语义兜住"最多匹配一次"，子查询展平靠 `is_simple_subquery` 的白名单，常量折叠只处理 immutable 的计算。这条"确定性改写前置、代价性选择进搜索"的分界线，正是 PG 不需要 Cascades 式统一框架的结构性原因（§1.3）。

## 3. grouping_planner 与 upper planner 的阶段化设计

`grouping_planner`（planner.c:1704）负责扫描/连接**之上**的一切：GROUP BY、聚合、窗口函数、DISTINCT、ORDER BY、LIMIT、集合操作、DML 顶层节点。

### 3.1 PG 9.6 的 pathification 重构

历史背景：9.6 之前，扫描/连接部分产出 Path，而上层的 Sort/Agg/Limit 是 `grouping_planner` 直接以 Plan 节点形式"焊死"追加的——上层没有候选之间的代价竞争，也没有扩展点。9.6 的重构（Tom Lane，commit `3fc6e2d7f5`，2016-03-07，俗称 upper planner pathification，详见 §11.1）把上层每个处理阶段也抽象成 RelOptInfo + Path 的竞争场，统一用 `add_path` 裁决。

阶段的枚举是 `UpperRelationKind`（`src/include/nodes/pathnodes.h:142`）：

```c
typedef enum UpperRelationKind
{
	UPPERREL_SETOP,				/* result of UNION/INTERSECT/EXCEPT, if any */
	UPPERREL_PARTIAL_GROUP_AGG, /* result of partial grouping/aggregation */
	UPPERREL_GROUP_AGG,			/* result of grouping/aggregation, if any */
	UPPERREL_WINDOW,			/* result of window functions, if any */
	UPPERREL_PARTIAL_DISTINCT,	/* result of partial "SELECT DISTINCT" */
	UPPERREL_DISTINCT,			/* result of "SELECT DISTINCT", if any */
	UPPERREL_ORDERED,			/* result of ORDER BY, if any */
	UPPERREL_FINAL,				/* result of any remaining top-level actions */
} UpperRelationKind;
```

枚举顺序就是流水线顺序（SQL 语义的执行顺序）：集合操作 → （部分）聚合 → 窗口 → DISTINCT → 排序 → 最终（LIMIT/锁行/DML）。`PlannerInfo` 里有按此枚举下标的数组 `upper_rels[]`（pathnodes.h:547），`fetch_upper_rel(root, kind, ...)` 按需取用。

### 3.2 grouping_planner 的接力流程

主线代码（调用点均已核实）：

1. `current_rel = query_planner(root, standard_qp_callback, &qp_extra);`（planner.c:1924）——拿到扫描/连接部分的最终 joinrel（§4、§5、§6 的全部内容都发生在这一行内部）；
2. 依次条件接力（planner.c:2069/2087/2107/2122）：

```c
    current_rel = create_grouping_paths(...);   /* 有聚合/GROUP BY 时 */
    current_rel = create_window_paths(...);     /* 有窗口函数时 */
    current_rel = create_distinct_paths(...);   /* 有 DISTINCT 时 */
    current_rel = create_ordered_paths(...);    /* 有 ORDER BY 时 */
```

每个 `create_xxx_paths`（如 `create_grouping_paths`，planner.c:4050）的模式完全一致：`fetch_upper_rel` 拿到本阶段的 RelOptInfo，然后**对输入 rel 的多条候选路径 × 本阶段的多种实现策略**做笛卡尔式组合生成路径。以分组为例：输入若有按 GROUP BY 列有序的路径 → 生成 Group/GroupAggregate 路径（interesting order 的又一次兑现！）；输入的 cheapest path → 加 Sort 再 GroupAggregate，或直接 HashAggregate；有并行输入 → 部分聚合（`UPPERREL_PARTIAL_GROUP_AGG`）+ Gather + 最终聚合。全部丢进 `add_path` 竞争。PG 13 之后这里还多了一类候选：对**前缀有序**的输入路径生成 Incremental Sort + GroupAggregate（commit `d2d8a229bc`，§11.3）——前缀有序信息正是 PathKeys 基础设施顺手提供的。

3. 最终把胜者挂到 `UPPERREL_FINAL`（planner.c:2138 `fetch_upper_rel(root, UPPERREL_FINAL, NULL)`），供 §2.1 第 5 步收割。

**设计价值**：其一，上层操作有了真正的代价竞争（sort-based vs hash-based 聚合的选择就是在这里发生的）；其二，每个阶段都有 `create_upper_paths_hook`（planner.c:83 声明，各阶段调用如 planner.c:4421-4422）与 FDW 的 `GetForeignUpperPaths` 注入点（如 planner.c:4415-4417），聚合下推到远端（postgres_fdw、Citus）才成为可能；其三，子查询不再只能返回单一 Plan，其多条路径（含有序路径）可作为外层 SubqueryScanPath 的基础，跨层的 interesting order 得以贯通（这是 pathification commit message 里点名的直接收益，见 §11.1）。

**小结**：`grouping_planner` 把"SELECT 语句剩下的部分"组织成 UpperRelationKind 流水线，每一段都是一个"RelOptInfo 竞技场"。理解它的最好实验：对同一个 GROUP BY 查询分别 `SET enable_hashagg = off/on`，观察 EXPLAIN 里 GroupAggregate 与 HashAggregate 的 cost 竞争。

## 4. query_planner → make_one_rel：基表路径生成

### 4.1 query_planner 的准备工作

`query_planner`（`src/backend/optimizer/plan/planmain.c:54`）先处理平凡情形——jointree 里只有一个 RTE_RESULT（`SELECT 1` 这类无表查询）时直接造一条 GroupResultPath 返回（planmain.c:97-160）。否则：建 RelOptInfo 数组（`add_base_rels_to_query`）、`deconstruct_jointree` 把 WHERE 拆成单表限制条件（`a=1`，挂到对应 rel 的 `baserestrictinfo`）与连接条件（`a.x=b.y`，挂 `joininfo`，并在此过程中构建 EquivalenceClass）、标记可用于 merge/hash join 的子句，最后调用 `make_one_rel`。

### 4.2 make_one_rel：两步走

`make_one_rel`（`src/backend/optimizer/path/allpaths.c:183`）：

```c

	set_base_rel_sizes(root);               /* allpaths.c:195 行数/宽度估计 */
	setup_simple_grouped_rels(root);        /* allpaths.c:201 eager agg 的 grouped rel */

```

两点注解：其一，allpaths.c:205-234 那段先累加查询涉及的**全部表的总页数**存入 `root->total_table_pages`——§7.3 的 Mackert-Lohman 公式要用它来按比例瓜分 `effective_cache_size`；其二，`setup_simple_grouped_rels` 是 20devel 新增的 Eager Aggregation 基础设施（§11.5），为可行的基表建出平行的 `grouped_rel`。

### 4.3 set_base_rel_pathlists 的分支

`set_base_rel_pathlists`（allpaths.c:384）遍历每个基表调 `set_rel_pathlist`（allpaths.c:520），后者按 RTE 类型分发：普通表 → `set_plain_rel_pathlist`；外部表 → FDW 的 `GetForeignPaths`；子查询/CTE/VALUES/函数各有专门分支。之后统一：插件 hook、为并行候选生成 Gather 路径、`set_cheapest(rel)` 选出该表的最便宜路径。

普通堆表的路径工厂 `set_plain_rel_pathlist`（allpaths.c:838），全函数：

```c
static void
set_plain_rel_pathlist(PlannerInfo *root, RelOptInfo *rel, RangeTblEntry *rte)
{
	Relids		required_outer;

	/*
	 * We don't support pushing join clauses into the quals of a seqscan, but
	 * it could still have required parameterization due to LATERAL refs in
	 * its tlist.
	 */
	required_outer = rel->lateral_relids;

	/*
	 * Consider TID scans.
	 *
	 * If create_tidscan_paths returns true, then a TID scan path is forced.
	 * This happens when rel->baserestrictinfo contains CurrentOfExpr, because
	 * the executor can't handle any other type of path for such queries.
	 * Hence, we return without adding any other paths.
	 */
	if (create_tidscan_paths(root, rel))
		return;

	/* Consider sequential scan */
	add_path(rel, create_seqscan_path(root, rel, required_outer, 0));

	/* If appropriate, consider parallel sequential scan */
	if (rel->consider_parallel && required_outer == NULL)
		create_plain_partial_paths(root, rel);

	/* Consider index scans */
	create_index_paths(root, rel);
}
```

逐段解说：TID 扫描（`WHERE ctid = ...` 或游标的 `WHERE CURRENT OF`）优先考虑，且 `CURRENT OF` 场景下是**强制**的——此时直接 return，别的路径一概不造；顺序扫描永远是保底候选；满足并行条件时加一条 partial SeqScan（后续可被 Gather 包装）；最后进入索引路径的主战场。

### 4.4 create_index_paths 全函数主线走读

`create_index_paths`（`src/backend/optimizer/path/indxpath.c:239`）的主循环（indxpath.c:250-315）：

```c
	/* Skip the whole mess if no indexes */

		return;

	/* Bitmap paths are collected and then dealt with at the end */

		 * bitmap paths are added to bitindexpaths to be handled below.

						&bitindexpaths);

```

逐段解说：

1. **部分索引谓词检查**（indxpath.c:270-271）：`CREATE INDEX ... WHERE ...` 的索引若其谓词不能被查询条件蕴含（`predOK` 为假，由更早的 `check_index_predicates` 用定理证明器 `predicate_implied_by` 判定）则整个跳过——查不全数据的索引当然不能当普通索引用；
2. **匹配限制子句**（indxpath.c:276-277）：底层是 `match_clauses_to_index`（indxpath.c:2552），判断标准是子句能否写成 `索引列 操作符 表达式` 且操作符属于该索引的操作符族——这就是"`WHERE f(x)=1` 用不上 x 上的普通索引、但用得上 `f(x)` 上的表达式索引"的源码级原因；
3. **生成普通（非参数化）索引路径**（indxpath.c:284-285）：走 `get_index_paths → build_index_paths`（下详），普通路径直接 `add_path`，适合做位图的积攒进 `bitindexpaths`；
4. **匹配连接子句与等价类子句**（indxpath.c:293-314）：用 `t1.x = t2.y` 这类连接条件（含从 EquivalenceClass 推导出的）生成**参数化索引路径**——"给我一个外层的 t2.y 值，我能用索引找 t1.x = 该值"，这是高效 nestloop 内表的来源（README 的 Parameterized Paths 一节，README:1059 起；引入历史见 §11.2）；
5. **位图组合**（indxpath.c:321-353）：`generate_bitmap_or_paths` 处理 OR 条件（每个 OR 分支各用一个索引，位图求并）；`choose_bitmap_and`（indxpath.c:345）贪心挑选多个索引做位图求交；最后 `create_bitmap_heap_path`（indxpath.c:346-348）包装后进入竞争，并行场景再加 partial 位图路径（indxpath.c:351-352）。位图扫描的本质：先扫索引把命中 TID 收集成位图、按物理页序访问堆——用一次排序换取顺序 I/O，是 SeqScan（纯顺序、全量）与 IndexScan（随机、精准）之间的中间形态。此后（indxpath.c:365 起）还会为 `bitjoinpaths` 按不同参数化分组生成参数化位图路径——逻辑与前述同构，本文从略。

### 4.5 build_index_paths 全函数走读

中转站 `get_index_paths`（indxpath.c:715）的存在理由只有一个（函数注释 indxpath.c:708-712 言明）：处理 `ScalarArrayOpExpr`（`x = ANY(数组)`）——索引 AM 原生支持（`amsearcharray`，如 B-tree）就并入普通路径；不支持就先剔除生成普通路径，再单独以 `ST_BITMAPSCAN` 模式重跑一遍 `build_index_paths`（indxpath.c:763-771）收集位图路径。真正的构造在 `build_index_paths`（indxpath.c:809），函数体五步，源码注释自己编了号：

**第 1 步：收集本索引可用的子句，并确定参数化集合**（indxpath.c:861-908）：

```c
										   rinfo->clause_relids);
```

**第 2 步：计算索引序对本查询是否有用**（indxpath.c:915-949）：

准入条件是个四路 OR，恰好对应索引路径可能创造价值的四种方式：有可用条件（少读数据）、有有用排序（  免 Sort）、部分索引谓词有用（索引本身就小）、可以 index-only（免回表）。**全都不满足才不生成**——即使没有任何 WHERE 条件，"全索引扫描但输出有序"或"全索引扫描但不回表"依然可能是好路径。其后（indxpath.c:984-1007）在 `amcanparallel` 且无参数化时再生成一条 parallel index scan 的 partial path，若 costing 后发现分不到 worker 则丢弃。

## 5. add_path：支配剪枝的裁决庭

v1 对 `add_path` 只有一句话，但它是整个搜索空间可控的第三根支柱（§1.1）：DP 控制子问题个数，而 **add_path 控制每个子问题里存活的路径条数**。本节完整走读。

### 5.1 没有它会怎样：一笔账

考虑 4 表连接的顶层 joinrel {A,B,C,D}。它的路径来自 7 种基表集合二划分（{A}+{BCD}、{AB}+{CD}……），每种划分两个方向、3 种连接方法；若下层每个 joinrel 各留 p 条路径，顶层候选数量级是 `7 × 2 × 3 × p²`。若没有任何剪枝，p 自身逐层膨胀：1 层若有 4 条（seq/index×2/bitmap），2 层就有 `2×2×3×4×4 ≈ 200` 条，3 层 `~10⁵` 条，4 层 `~10⁹` 条——**路径数量按平方逐层复合**，DP 白做。add_path 把每层 p 压回 O(有用排序数 × 参数化数 × 2) 的小常数（实践中多为 1~10），乘法才不爆炸。这就是函数注释（pathnode.c:408-412）说的：

```
 *	  In addition to possibly adding new_path, we also remove from the rel's
 *    pathlist any old paths that are dominated by new_path --- that is,
 *    new_path is cheaper, at least as well ordered, generates no more rows,
 *    requires no outer rels not required by the old path, and is no less
 *    parallel-safe.
```

一条路径要被淘汰，必须在**所有维度**上都不优于对手——这是严格的帕累托支配（Pareto dominance）。维度清单：① 代价（startup 与 total 两个分量 + disabled_nodes 高阶分量）；② 排序（pathkeys）；③ 参数化（required_outer 集合）；④ 行数（rows）与并行安全（parallel_safe）。四个维度对应四种"上层可能在意的属性"，正是 §1.4 说的"最优是向量"的具体化。

### 5.2 前置：compare_path_costs_fuzzily

代价维度的比较不是直接比大小，而是走 `compare_path_costs_fuzzily`（pathnode.c:181）：

```c
static PathCostComparison
compare_path_costs_fuzzily(Path *path1, Path *path2, double fuzz_factor)
{
#define CONSIDER_PATH_STARTUP_COST(p)  \
	((p)->param_info == NULL ? (p)->parent->consider_startup : (p)->parent->consider_param_startup)

	/* Number of disabled nodes, if different, trumps all else. */
	if (unlikely(path1->disabled_nodes != path2->disabled_nodes))
	{
		if (path1->disabled_nodes < path2->disabled_nodes)
			return COSTS_BETTER1;
		else
			return COSTS_BETTER2;
	}

	/*
	 * Check total cost first since it's more likely to be different; many
	 * paths have zero startup cost.
	 */
	if (path1->total_cost > path2->total_cost * fuzz_factor)
	{
		/* path1 fuzzily worse on total cost */
		if (CONSIDER_PATH_STARTUP_COST(path1) &&
			path2->startup_cost > path1->startup_cost * fuzz_factor)
		{
			/* ... but path2 fuzzily worse on startup, so DIFFERENT */
			return COSTS_DIFFERENT;
		}
		/* else path2 dominates */
		return COSTS_BETTER2;
	}
	if (path2->total_cost > path1->total_cost * fuzz_factor)
	{
		...（对称分支）
	}
	/* fuzzily the same on total cost ... */
	if (path1->startup_cost > path2->startup_cost * fuzz_factor)
	{
		/* ... but path1 fuzzily worse on startup, so path2 wins */
		return COSTS_BETTER2;
	}
	if (path2->startup_cost > path1->startup_cost * fuzz_factor)
	{
		/* ... but path2 fuzzily worse on startup, so path1 wins */
		return COSTS_BETTER1;
	}
	/* fuzzily the same on both costs */
	return COSTS_EQUAL;
```

四个设计点：

1. **disabled_nodes 是高于一切的高阶代价分量**（pathnode.c:186-193）。`enable_seqscan=off` 并不删除 seqscan 路径，只是给它计 `disabled_nodes+1`（见 §7.2 cost_seqscan 尾部）；比较时被禁次数少者无条件胜出，被禁次数相同才比代价。这保证"全部路径都被禁"时仍能出计划（老实现给被禁路径加 1e10 的惩罚代价，会因浮点精度吞掉真实代价差，导致被禁路径之间选择错乱——2024 年才换成整数分量，commit `e222534679`，§11.4）；
2. **模糊比较**：`fuzz_factor` 标准值 1.01（`STD_FUZZ_FACTOR`）——1% 以内的代价差视为相等。理由（pathnode.c:544-553 的注释）：代价是估出来的，1% 的差异毫无置信度；模糊化还能避免不同平台浮点舍入差异导致回归测试计划抖动；
3. **startup/total 双指标**：一条 total 更优、另一条 startup 更优 → `COSTS_DIFFERENT`，两条都留。这是 LIMIT/游标场景（§2.1 的 `tuple_fraction`）的基础——"快出第一行"和"快出全部行"是两种都值得持有的期权；
4. **consider_startup 门控**：多数 rel 根本不会被 LIMIT 消费（`parent->consider_startup` 为假），此时 startup 优势不足以保命——又一层"别留没人要的期权"的剪枝。

### 5.3 add_path 全函数走读

（pathnode.c:459，完整逻辑；两处 `...` 为省略的对称分支与不可达分支，均注明）

```c
void

{

	int			insert_at = 0;	/* where to insert new item */
	List	   *new_path_pathkeys;
	ListCell   *p1;

	 * This is a convenient place to check for query cancel --- no part of the
	 * planner goes very long without calling add_path().

	new_path_pathkeys = new_path->param_info ? NIL : new_path->pathkeys;

	/*
	 * Loop to check proposed new path against old paths.  Note it is possible
	 * for more than one old path to be tossed out because new_path dominates
	 * it.

	foreach(p1, parent_rel->pathlist)
	{
		Path	   *old_path = (Path *) lfirst(p1);
		bool		remove_old = false; /* unless new proves superior */
		PathCostComparison costcmp;
		PathKeysComparison keyscmp;
		BMS_Comparison outercmp;

		/*
		 * Do a fuzzy cost comparison with standard fuzziness limit.

											 STD_FUZZ_FACTOR);
```

开场三个细节都值得停留：`CHECK_FOR_INTERRUPTS()` 放在这里，是因为 add_path 是全 planner 调用最密的函数——规划一条大查询期间的 Ctrl-C 响应就系在这一行上。**参数化路径被强行视为无 pathkeys**（pathnode.c:472-473；政策声明在注释 pathnode.c:420-424）：参数化路径的用途几乎只有"当 nestloop 内表"，nestloop 每循环一次 rescan 一次内表，内表自身的输出序对上层几乎无用；若允许它按 pathkeys 保命，参数化×排序两维叉乘会让存活路径数失控。这是一个**故意放弃部分最优性换取搜索规模**的显式政策决定。

主循环把新路径与每条既有路径做四维比较：

比较逻辑的完整语义，用一句话概括：**新路径要淘汰旧路径，必须代价不更差（COSTS_BETTER1 或 EQUAL）、排序不更差（keyscmp 非 BETTER2）、参数化集合是旧的子集或相等（要求的外部依赖更少 = 适用面更广）、行数不更多、并行安全性不更差——五个条件同时成立；任何一个维度互有胜负（COSTS_DIFFERENT / PATHKEYS_DIFFERENT / 参数化集合互不包含），两条都保留。** 比较顺序（先代价、再 pathkeys、最后 rows）纯粹是短路效率考量（注释 pathnode.c:498-503：代价最可能"不同"，rows 几乎从不起裁决作用）。

为什么"行数"也是一个维度？同一 relids、同一参数化的两条路径行数居然可以不同——因为参数化路径把不同数量的 join 子句吃进了扫描（pathnode.c:414-418 的注释）：吃了更多子句的路径输出更少行，能让上层 nestloop 的每次循环更便宜，这是独立于代价的价值。COSTS_EQUAL + 全同分支里那串"先并行安全、再行数、再 1e-10 模糊比较、仍平则留旧"的瀑布，是为了**决定性**（determinism）：同一查询在任何平台任何编译器下淘汰同一条路径，避免计划抖动。

### 5.4 add_path_precheck 与 set_cheapest

`add_path_precheck`（pathnode.c:686）是 add_path 的"免精算预审"：连接代价计算分两段（§7.4 的 `initial_cost_XXX` / `final_cost_XXX`），initial 阶段只花几次加法得到代价下界，就先来问一句"这个下界 + 这组 pathkeys + 这个参数化，有没有可能存活？"。它利用 pathlist 的排序线性扫描：一旦遇到 total_cost 已超过新路径下界的旧路径就 break 放行（pathnode.c:705-717）；若在此之前找到一条同参数化、pathkeys 不更差、startup 也不更差的旧路径，直接判死（pathnode.c:727-746），`final_cost_XXX` 的精算（含选择率计算、直方图访问）整个省掉。注意它假设"同参数化才可比、异参数化互不支配"（函数注释 pathnode.c:674-680）——放行错了也无妨，真正的 add_path 会再裁一次；这是典型的"允许假阳性、不允许假阴性"的两级过滤器设计。

一个 rel 的所有路径生成完毕后，`set_cheapest`（pathnode.c:268）扫一遍 pathlist 填充四个快捷字段：`cheapest_total_path`、`cheapest_startup_path`（仅非参数化路径有资格）、`cheapest_unique_path`、`cheapest_parameterized_paths` 列表（每种参数化的幸存者 + cheapest_total，函数注释 pathnode.c:245-262）。上层构造连接时直接取这几个字段，而不必反复线性扫 pathlist。

**小结**：add_path 以 `(disabled_nodes, total, startup)` × pathkeys × required_outer × (rows, parallel_safe) 的帕累托支配裁决每条新路径的生死，把每个 RelOptInfo 的存活路径数压到小常数，使 §1.1 的三层乘法压缩闭环。两个显式的政策性放弃（参数化路径不按序保命、startup 期权仅在 consider_startup 时有效）和一个高阶分量（disabled_nodes）是读懂它行为的三把钥匙。

## 6. 连接搜索：动态规划逐层枚举与 GEQO

### 6.1 三岔口：make_rel_from_joinlist

### 6.2 standard_join_search 全函数走读

（allpaths.c:3952，函数体完整呈现，仅省略函数头注释）

```c
RelOptInfo *
standard_join_search(PlannerInfo *root, int levels_needed, List *initial_rels)
{
	int			lev;
	RelOptInfo *rel;

	/*
	 * This function cannot be invoked recursively within any one planning
	 * problem, so join_rel_level[] can't be in use already.
	 */
	Assert(root->join_rel_level == NULL);

	/*
	 * We employ a simple "dynamic programming" algorithm: we first find all
	 * ways to build joins of two jointree items, then all ways to build joins
	 * of three items (from two-item joins and single items), then four-item
	 * joins, and so on until we have considered all ways to join all the
	 * items into one rel.
	 *
	 * root->join_rel_level[j] is a list of all the j-item rels.  Initially we
	 * set root->join_rel_level[1] to represent all the single-jointree-item
	 * relations.
	 */
	root->join_rel_level = (List **) palloc0((levels_needed + 1) * sizeof(List *));

	root->join_rel_level[1] = initial_rels;

	for (lev = 2; lev <= levels_needed; lev++)
	{
		ListCell   *lc;

		/*
		 * Determine all possible pairs of relations to be joined at this
		 * level, and build paths for making each one from every available
		 * pair of lower-level relations.
		 */
		join_search_one_level(root, lev);

		/*
		 * Run generate_partitionwise_join_paths() and
		 * generate_useful_gather_paths() for each just-processed joinrel.  We
		 * could not do this earlier because both regular and partial paths
		 * can get added to a particular joinrel at multiple times within
		 * join_search_one_level.
		 *
		 * After that, we're done creating paths for the joinrel, so run
		 * set_cheapest().
		 *
		 * In addition, we also run generate_grouped_paths() for the grouped
		 * relation of each just-processed joinrel, and run set_cheapest() for
		 * the grouped relation afterwards.
		 */
		foreach(lc, root->join_rel_level[lev])
		{
			bool		is_top_rel;

			rel = (RelOptInfo *) lfirst(lc);

			is_top_rel = bms_equal(rel->relids, root->all_query_rels);

			/* Create paths for partitionwise joins. */
			generate_partitionwise_join_paths(root, rel);

			/*
			 * Except for the topmost scan/join rel, consider gathering
			 * partial paths.  We'll do the same for the topmost scan/join rel
			 * once we know the final targetlist (see grouping_planner's and
			 * its call to apply_scanjoin_target_to_paths).
			 */
			if (!is_top_rel)
				generate_useful_gather_paths(root, rel, false);

			/* Find and save the cheapest paths for this rel */
			set_cheapest(rel);

			/*
			 * Except for the topmost scan/join rel, consider generating
			 * partial aggregation paths for the grouped relation on top of
			 * the paths of this rel.  After that, we're done creating paths
			 * for the grouped relation, so run set_cheapest().
			 */
			if (rel->grouped_rel != NULL && !is_top_rel)
			{
				RelOptInfo *grouped_rel = rel->grouped_rel;

				Assert(IS_GROUPED_REL(grouped_rel));

				generate_grouped_paths(root, grouped_rel, rel);
				set_cheapest(grouped_rel);
			}

#ifdef OPTIMIZER_DEBUG
			pprint(rel);
#endif
		}
	}

	/*
	 * We should have a single rel at the final level.
	 */
	if (root->join_rel_level[levels_needed] == NIL)
		elog(ERROR, "failed to build any %d-way joins", levels_needed);
	Assert(list_length(root->join_rel_level[levels_needed]) == 1);

	rel = (RelOptInfo *) linitial(root->join_rel_level[levels_needed]);

	root->join_rel_level = NULL;

	return rel;
}
```

逐段解说：

- **备忘录初始化**（3974-3976）：`join_rel_level[j]` 是所有恰含 j 个基表的 joinrel 列表，第 1 层直接用 §4 做好的基表们初始化。这个 per-level 列表与 `join_rel_hash`（以 relids 为 key）共同构成 §1.3 说的"穷人版 memo"——`build_join_rel` 在 `make_join_rel` 内部先查哈希，同一基表集合永远只有一个 RelOptInfo；
- **主循环**（3978-4046）：每层先由 `join_search_one_level` 造出本层全部 joinrel 及其常规路径，**然后**才做三件收尾事。为什么收尾不能放进 join_search_one_level？注释（3990-3994）讲得很清楚：同一个 joinrel 会在一层内被多种划分反复添加路径（{A,B}+{C} 与 {B,C}+{A} 汇入同一 rel），必须等本层全部划分处理完，Gather/分区连接这类"包装型"路径才不会漏掉后来者；
- **三件收尾事**：`generate_partitionwise_join_paths` 为分区表连接生成逐分区连接的 Append 路径；`generate_useful_gather_paths` 把 partial path 用 Gather/GatherMerge 封顶（顶层 joinrel 除外——它要等最终 targetlist 确定后在 grouping_planner 里做）；`set_cheapest` 冻结本层。**正因为下层已冻结，上层组合时只需在下层留存的帕累托前沿里挑**——这就是 §1.2 说的最优子结构在代码里的兑现点；
- **grouped_rel 分支**（4032-4040）：20devel 的 Eager Aggregation（§11.5）——若这个 joinrel 有平行的"已部分聚合"版本，也为它生成路径并冻结。这是在 DP 框架里增加一个"逻辑属性维度"（聚合与否）的教科书式做法：不改搜索算法，只是让每个子问题带两个 RelOptInfo；
- **收官检查**（4048-4055）：最顶层必然恰有一个 rel（包含全部基表），空了就报 `failed to build any %d-way joins`（正常查询不可能触发；触发即 planner bug 或极端约束场景），多于一个则断言失败。

**第二段：bushy**（joinrels.c:145-203）

```c
	/*
	 * Now, consider "bushy plans" in which relations of k initial rels are
	 * joined to relations of level-k initial rels, for 2 <= k <= level-2.
	 *
	 * We only consider bushy-plan joins for pairs of rels where there is a
	 * suitable join clause (or join order restriction), in order to avoid
	 * unreasonable growth of planning time.
	 */
	for (k = 2;; k++)
	{
		int			other_level = level - k;

		/*
		 * Since make_join_rel(x, y) handles both x,y and y,x cases, we only
		 * need to go as far as the halfway point.
		 */
		if (k > other_level)
			break;

		foreach(r, joinrels[k])
		{
			RelOptInfo *old_rel = (RelOptInfo *) lfirst(r);
			int			first_rel;
			ListCell   *r2;

			/*
			 * We can ignore relations without join clauses here, unless they
			 * participate in join-order restrictions --- then we might have
			 * to force a bushy join plan.
			 */
			if (old_rel->joininfo == NIL && !old_rel->has_eclass_joins &&
				!has_join_restriction(root, old_rel))
				continue;

			if (k == other_level)	/* only consider remaining rels */
				first_rel = foreach_current_index(r) + 1;
			else
				first_rel = 0;

			for_each_from(r2, joinrels[other_level], first_rel)
			{
				RelOptInfo *new_rel = (RelOptInfo *) lfirst(r2);

				if (!bms_overlap(old_rel->relids, new_rel->relids))
				{
					/*
					 * OK, we can build a rel of the right level from this
					 * pair of rels.  Do so if there is at least one relevant
					 * join clause or join order restriction.
					 */
					if (have_relevant_joinclause(root, old_rel, new_rel) ||
						have_join_order_restriction(root, old_rel, new_rel))
					{
						(void) make_join_rel(root, old_rel, new_rel);
					}
				}
			}
		}
	}
```

逐段解说：k-表 joinrel × (level-k)-表 joinrel，k 从 2 到 level/2——两边都是连接结果，这正是 bushy 树。四重防爆条款：k 只到中点（`k > other_level` 即 break，组合对称，make_join_rel 双向尝试）；k == other_level 时同层配对也从当前位置向后扫（161-182 行，同一层内 {AB}×{CD} 与 {CD}×{AB} 去重）；`bms_overlap` 用位图集判定两 rel 不含相同基表（含同表的组合不是合法的 level 层划分）；**必须有相关连接子句或顺序约束**才调 `make_join_rel`（149-151 行注释明说这是为了 avoid unreasonable growth of planning time——bushy 空间里的纯笛卡尔组合被整体砍掉，这正是 §1.1 第 2 层压缩在 bushy 段的执行处）。


```
 *	 x IN (SELECT ... FROM t2,t3 WHERE ...) AND
 *	 y IN (SELECT ... FROM t4,t5 WHERE ...)
```

两个 IN 上拉成半连接后是 5 表问题，但 `join_is_legal` 之下**不存在任何合法的 4 表连接**（任何 4 表组合都会拆散某个半连接的原子性），第 4 层必须容忍空层，到第 5 层才能拼出合法的 bushy 计划——这同时也是"为什么必须支持 bushy"的最硬理由：这里连"次优的左深替代"都不存在。无特殊连接时空层则直接 `elog(ERROR)` 揪出 planner bug。

**路径层面的竞争**：每次 `make_join_rel` 对该基表集合找到（或创建）唯一的 joinrel RelOptInfo，检查 `join_is_legal`，然后由 `add_paths_to_joinrel`（`src/backend/optimizer/path/joinpath.c:123`）为这对输入生成路径：sort-based merge join（枚举有用的排序）、已有序输入的 merge join、nestloop（普通/物化内表/参数化内表索引）、hash join，每类内部再区分普通与 partial（并行）版本。不同划分产出的路径**汇入同一个 RelOptInfo，在 add_path 里同台竞技**——"连接顺序的枚举"（本节）与"连接方法的选择"（joinpath.c + §7.4 的代价）就这样被优雅地解耦了。

### 6.4 GEQO：遗传算法兜底，以及"为什么不是贪心"

`geqo`（`src/backend/optimizer/geqo/geqo_main.c:75`）把连接顺序问题当作带约束的旅行商问题（geqo_main.c:68-71 注释："solution of the query optimization problem / similar to a constrained Traveling Salesman Problem (TSP)"）：一个"染色体" = 基表的一个排列（即一棵左深树的连接顺序）；适应度 = 按该顺序真实调用标准路径生成代码算出的最优代价（`geqo_eval` 内部照常跑 `make_join_rel`/`add_paths_to_joinrel`/`add_path`，所以**每个候选顺序内部的连接方法选择仍然是精确的**，也因此 geqo_eval 要保存/恢复 `join_rel_list`，见 standard_join_search 头注释 allpaths.c:3946-3949）；用 Genitor 式稳态遗传（geqo_main.c:24 注释 "adapted from D. Whitley's Genitor algorithm"）+ 默认 ERX 边重组交叉算子（geqo_main.c:146-151）迭代若干代，输出最优个体。

**为什么选遗传算法而不是随机化贪心/模拟退火？** 平心而论，这更多是历史巧合而非充分论证——GEQO 是 1997 年前后的贡献（作者 Martin Utesch，geqo_main.c 文件头），当时 join ordering 的随机化算法（II、SA、AB、GA）在学术上均有拥趸。GEQO 的实际形态其实已经很接近"带记忆的随机化贪心"：染色体只编码左深顺序（搜索空间 n!，放弃 bushy），真正的物理选择全部委托给确定性的标准代码。它的短处明确：无全局最优保证、对染色体交叉不友好的约束（外连接顺序限制）要靠合法性修补、以及早期版本计划不可复现——PG 用固定种子 `geqo_seed` 缓解最后一条。实践中更常见的做法是调大 `join_collapse_limit`/`from_collapse_limit` 把问题切小、或用 pg_hint_plan 固定顺序，而不是依赖 GEQO；把 `geqo_threshold` 调到很大等于宣布"我宁愿多花规划时间也要确定性计划"，对报表型长查询往往是对的。

**小结**：`standard_join_search` 逐层填 DP 表，每层收尾三连（分区连接/Gather/set_cheapest 冻结）+ eager agg 的 grouped_rel；`join_search_one_level` 在一层内按"左深+右深 → bushy → 笛卡尔兜底"三段枚举，以"必须有连接子句"为主剪枝，兜底段的注释例子说明空层的合法性与 bushy 的必要性。连接方法选择下沉到 `add_paths_to_joinrel`，所有候选在共享的 RelOptInfo 中由 add_path 裁决。≥12 个 FROM 项时切换 GEQO——左深随机化 + 精确物理选择的混血。

### 7.1 GUC 参数就是公式系数——先做量纲分析

代价的单位是"一次顺序页读取的耗时 = 1.0"的**相对值**，不是秒。全部系数（`src/include/optimizer/cost.h:24-34`）：

```c

#define DEFAULT_RANDOM_PAGE_COST  4.0
#define DEFAULT_CPU_TUPLE_COST	0.01
#define DEFAULT_CPU_INDEX_TUPLE_COST 0.005
#define DEFAULT_CPU_OPERATOR_COST  0.0025

...

```

量纲上它们分三类，任何代价公式都是三类项的线性组合：

| 系数 | 量纲 | 默认值的物理含义 |
|------|------|------------------|
| `seq_page_cost` / `random_page_cost` | 代价/页 | 随机读一页 = 4 次顺序读。机械盘寻道时代的比值；SSD/云盘上真实比值接近 1，普遍应调到 1.1~2 |
| `cpu_tuple_cost` / `cpu_index_tuple_cost` | 代价/行 | 处理一行堆元组 = 1% 次顺序页读；索引元组减半（更小、无可见性判断） |
| `cpu_operator_cost` | 代价/次 | 执行一次操作符/函数调用 = 0.25% 次顺序页读 |

所以"页数 × 页成本 + 行数 × 行成本 + 行数 × 操作符数 × 操作符成本"每一项的量纲都是纯"代价"，可加。`random_page_cost=4` 直接控制 §7.3 公式中随机 I/O 的惩罚，是"为什么 PG 不走我的索引"的头号嫌疑人。每条路径的代价分两段：`startup_cost`（吐出第一行前的开销）与 `total_cost`（全部行）——LIMIT 场景按比例只取前段，这就是 §2.1 `tuple_fraction` 的消费方式。

### 7.2 cost_seqscan 全函数走读 + 手算闭环例 1

（costsize.c:271，函数体完整）

```c
void
cost_seqscan(Path *path, PlannerInfo *root,
			 RelOptInfo *baserel, ParamPathInfo *param_info)
{
	Cost		startup_cost = 0;
	Cost		cpu_run_cost;
	Cost		disk_run_cost;
	double		spc_seq_page_cost;
	QualCost	qpqual_cost;
	Cost		cpu_per_tuple;
	uint64		enable_mask = PGS_SEQSCAN;

	/* Should only be applied to base relations */
	Assert(baserel->relid > 0);
	Assert(baserel->rtekind == RTE_RELATION);

	/* Mark the path with the correct row estimate */
	if (param_info)
		path->rows = param_info->ppi_rows;
	else
		path->rows = baserel->rows;

	/* fetch estimated page cost for tablespace containing table */
	get_tablespace_page_costs(baserel->reltablespace,
							  NULL,
							  &spc_seq_page_cost);

	/*
	 * disk costs
	 */
	disk_run_cost = spc_seq_page_cost * baserel->pages;

	/* CPU costs */
	get_restriction_qual_cost(root, baserel, param_info, &qpqual_cost);

	startup_cost += qpqual_cost.startup;
	cpu_per_tuple = cpu_tuple_cost + qpqual_cost.per_tuple;
	cpu_run_cost = cpu_per_tuple * baserel->tuples;
	/* tlist eval costs are paid per output row, not per tuple scanned */
	startup_cost += path->pathtarget->cost.startup;
	cpu_run_cost += path->pathtarget->cost.per_tuple * path->rows;

	/* Adjust costing for parallelism, if used. */
	if (path->parallel_workers > 0)
	{
		double		parallel_divisor = get_parallel_divisor(path);

		/* The CPU cost is divided among all the workers. */
		cpu_run_cost /= parallel_divisor;

		/*
		 * It may be possible to amortize some of the I/O cost, but probably
		 * not very much, because most operating systems already do aggressive
		 * prefetching.  For now, we assume that the disk run cost can't be
		 * amortized at all.
		 */

		/*
		 * In the case of a parallel plan, the row count needs to represent
		 * the number of tuples processed per worker.
		 */
		path->rows = clamp_row_est(path->rows / parallel_divisor);
	}
	else
		enable_mask |= PGS_CONSIDER_NONPARTIAL;

	path->disabled_nodes =
		(baserel->pgs_mask & enable_mask) == enable_mask ? 0 : 1;
	path->startup_cost = startup_cost;
	path->total_cost = startup_cost + cpu_run_cost + disk_run_cost;
}
```

逐项解说（每项标注量纲）：

1. **行数**：来自 §8 的选择率乘 `reltuples`（参数化路径则用参数化后的更小估计 `ppi_rows`）。注意行数与代价是两个独立输出——过滤掉 99% 的行**不减少**扫描代价，只减少输出行数；
2. **I/O 项**（costsize.c:300）：`disk_run_cost = spc_seq_page_cost × baserel->pages`，[代价/页]×[页]。表空间可覆写全局 GUC（把热表放在 SSD 表空间并设更低的 page cost，就是通过这个函数生效的）。全部计入 run 而非 startup：第一页也要等扫描开始才读；
3. **CPU 项**（costsize.c:303-307）：`(cpu_tuple_cost + qual 每行求值成本) × baserel->tuples`。乘的是**全表元组数**而非过滤后行数——每一行都要被取出并对它算一遍 WHERE 才知道要不要它。`qpqual_cost` 由 `cost_qual_eval`（costsize.c:4923）对每个操作符按 `cpu_operator_cost` 累加，含子计划、函数等的开销；其 startup 分量（如 qual 里含初始化一次的子计划）进 startup_cost；
4. **投影项**（costsize.c:308-310）：目标列表达式按**输出行数**计——被 WHERE 滤掉的行不必计算投影。`SELECT expensive_func(x) FROM t WHERE y=1` 中 expensive_func 只对幸存行付费，就是这两行代码；
5. **并行折算**（costsize.c:313-332）：CPU 除以 `parallel_divisor`（worker 数 + leader 折扣），I/O **不除**——注释明说操作系统的预读已经让顺序 I/O 无法靠多进程摊薄。行数也除，因为 partial path 的 rows 语义是"每 worker 处理行数"；
6. **disabled_nodes**（costsize.c:336-337）：`enable_seqscan=off` 时 `pgs_mask` 里对应位缺失，此路径记 1 个 disabled 节点，在 §5.2 的比较中垫底但**不会消失**——这就是关全部 enable_* 也能出计划的原因（机制来历见 §11.4）。

**手算闭环例 1**：取回归测试标准表 `tenk1`：`pg_class` 中 `reltuples = 10000`，`relpages = 345`。

*无条件全表扫：*

```
disk = 1.0 × 345            = 345.0
cpu  = 0.01 × 10000         = 100.0
total = 345 + 100           = 445.0
```

EXPLAIN 输出 `Seq Scan on tenk1  (cost=0.00..445.00 rows=10000 width=...)`——445.00 分毫不差。

*加一个过滤条件 `WHERE unique1 < 1000`（1 个操作符）：*

```
disk = 1.0 × 345                       = 345.0
cpu  = (0.01 + 0.0025 × 1) × 10000     = 125.0
total                                  = 470.0
rows = 10000 × selectivity(unique1 < 1000)
     = 10000 × 0.1 = 1000    （§8.2 直方图插值：unique1 均匀分布于 0..9999）
```

EXPLAIN 输出 `Seq Scan on tenk1  (cost=0.00..470.00 rows=1000 width=...)`。对照公式可见 EXPLAIN 里每一位数字的出处：445→470 的差恰是 10000 行 × 1 个操作符 × 0.0025。**建议实际执行一次并改动 `cpu_operator_cost` 复核**——手算能闭环，代价模型就不再是黑箱。

### 7.3 cost_index 全函数走读与 Mackert-Lohman 公式

`cost_index`（costsize.c:546）比 seqscan 多出的全部复杂性来自一个问题：**回表的随机 I/O 到底有多少页、其中多少被缓存挡掉了？** 按执行顺序分五步（`partial_path` 并行分支 costsize.c:749-779 仅计算 worker 数并对 CPU 折算，与主线逻辑正交，本文注明后省略）。

**第一步：确定行数与 qpquals**（costsize.c:587-602）：参数化路径用 `ppi_rows` 并把参数化子句并入待过滤集合；`extract_nonindex_conditions`（costsize.c:840）把不能被索引本身检查的条件分离为 qpquals（回表后逐行过滤的部分）——已被索引子句覆盖的不重复计费。

**第二步：问索引 AM 自己**（costsize.c:617-621）：

```c
	amcostestimate = (amcostestimate_function) index->amcostestimate;
	amcostestimate(root, path, loop_count,
				   &indexStartupCost, &indexTotalCost,
				   &indexSelectivity, &indexCorrelation,
				   &index_pages);
```

B-tree 的 `btcostestimate` 估出：遍历索引本身的代价（下降 + 沿叶链扫描 + 每索引元组 `cpu_index_tuple_cost`）、**索引选择率**（要回表的堆元组占比）与**相关性** `indexCorrelation`（索引序与堆物理序的相关系数，ANALYZE 存入 pg_statistic）。这是 AM 扩展点：代价模型的"索引内部分"随 AM 可插拔，"回表部分"由本函数统一建模。随后 `tuples_fetched = clamp_row_est(indexSelectivity * baserel->tuples)`（costsize.c:636）。

**第三步：估回表页数——Mackert-Lohman 模型**。若把命中的行按索引序逐个回表，同一堆页会被反复访问；重访问算不算重复 I/O 取决于缓存。PG 采用 Mackert & Lohman 的验证模型（*Index Scans Using a Finite LRU Buffer: A Validated I/O Model*, ACM TODS 1989——出处注明在 costsize.c:864-866），`index_pages_fetched`（costsize.c:898）头注释给出公式（costsize.c:869-877）：

```
 *	PF =
 *		min(2TNs/(2T+Ns), T)			when T <= b
 *		2TNs/(2T+Ns)					when T > b and Ns <= 2Tb/(2T-b)
 *		b + (Ns - 2Tb/(2T-b))*(T-b)/T	when T > b and Ns > 2Tb/(2T-b)
 * where
 *		T = # pages in table
 *		N = # tuples in table
 *		s = selectivity = fraction of table to be scanned
 *		b = # buffer pages available (we include kernel space here)
```

实现（costsize.c:907-949）：

```c
	/* T is # pages in table, but don't allow it to be zero */
	T = (pages > 1) ? (double) pages : 1.0;

	/* Compute number of pages assumed to be competing for cache space */
	total_pages = root->total_table_pages + index_pages;
	total_pages = Max(total_pages, 1.0);
	Assert(T <= total_pages);

	/* b is pro-rated share of effective_cache_size */
	b = (double) effective_cache_size * T / total_pages;
	...
	/* This part is the Mackert and Lohman formula */
	if (T <= b)
	{
		pages_fetched =
			(2.0 * T * tuples_fetched) / (2.0 * T + tuples_fetched);
		if (pages_fetched >= T)
			pages_fetched = T;
		else
			pages_fetched = ceil(pages_fetched);
	}
	else
	{
		double		lim;

		lim = (2.0 * T * b) / (2.0 * T - b);
		if (tuples_fetched <= lim)
		{
			pages_fetched =
				(2.0 * T * tuples_fetched) / (2.0 * T + tuples_fetched);
		}
		else
		{
			pages_fetched =
				b + (tuples_fetched - lim) * (T - b) / T;
		}
		pages_fetched = ceil(pages_fetched);
	}
```

直觉解读：`2TNs/(2T+Ns)` 是"Ns 次随机取行落在 T 页上产生多少个不同页"的近似（小 Ns 时 ≈ Ns，即每行一页；大 Ns 时 → 2T，被封顶到 T）——本质是生日悖论式的碰撞折扣。表放得进缓存份额（T ≤ b）时重访问全部免费；放不进时，取行数超过临界值 `lim` 后每行开始产生真实的额外驱逐+重读，进入线性段。**b 的算法（costsize.c:915）是 `effective_cache_size` 按页数比例在查询涉及的所有表+当前索引间分摊**——这就是 `effective_cache_size` 影响计划的全部机制：它不分配一字节内存，只改这一行里的 b，从而改变索引扫描的 I/O 估价（§4.2 存的 `total_table_pages` 在此消费）。

*数值例子*：T = 10000 页，N = 100 万行，s = 0.001（取 1000 行），`effective_cache_size` 默认 524288 页，本查询只涉及这一张表和它的索引（2745 页）：

```
b  = 524288 × 10000 / 12745 ≈ 411367  > T   → 走 T ≤ b 分支
PF = 2×10000×1000 / (2×10000 + 1000) = 2×10⁷ / 21000 ≈ 952.4 → ceil = 953 页
```

1000 次取行只需 953 页 I/O——碰撞折扣约 5%。若 s = 0.05（5 万行）：`PF = 10⁹/70000 ≈ 14286 ≥ T`，封顶 10000 页——**全表的页都要摸一遍，但每页只算一次**。

**第四步：相关性插值**（costsize.c:714-747 的 else 分支 + 785-787）：

```c
		/* max_IO_cost is for the perfectly uncorrelated case (csquared=0) */
		max_IO_cost = pages_fetched * spc_random_page_cost;

		/* min_IO_cost is for the perfectly correlated case (csquared=1) */
		pages_fetched = ceil(indexSelectivity * (double) baserel->pages);
		...
		if (pages_fetched > 0)
		{
			min_IO_cost = spc_random_page_cost;
			if (pages_fetched > 1)
				min_IO_cost += (pages_fetched - 1) * spc_seq_page_cost;
		}
		...
	csquared = indexCorrelation * indexCorrelation;

	run_cost += max_IO_cost + csquared * (min_IO_cost - max_IO_cost);
```

完全无关（乱序表）：每页都是随机读，M-L 页数 × `random_page_cost`；完全相关（刚 CLUSTER 过/自增主键顺序插入）：命中的行挤在连续的 `s × T` 页里，第一页随机、其余顺序。真实代价按相关性**平方**插值（注释自己都标了 "XXX is that appropriate?"——平方意味着中等相关被重罚，是偏保守的工程选择）。

*续上面 s=0.05 的例子算两个极端*：

```
max_IO = 10000 × 4.0            = 40000
min 分支页数 = ceil(0.05×10000) = 500 → min_IO = 4 + 499×1 = 503
corr = 0.95: csq=0.9025 → IO = 40000 + 0.9025×(503-40000) ≈ 4356
corr = 0.30: csq=0.09   → IO = 40000 + 0.09×(503-40000)   ≈ 36445
```

同表同查询，seqscan 是 `10000 + 10⁶×0.0125 = 22500`：相关性 0.95 时索引扫描（4356 + CPU 项）大胜，相关性 0.3 时（36445 + CPU）惨败弃用。**这就是"按插入序自然聚簇的时间序列表走索引、乱序 UUID 主键表弃用索引"的完整定量解释**——差别全在 `pg_stats.correlation` 一个数字上。

**第五步：CPU 与收尾**（costsize.c:795-820）：与 seqscan 同构——`(cpu_tuple_cost + qpqual成本) × tuples_fetched`（只对取回的行算），投影按输出行数。Index-Only Scan 的优惠在 costsize.c:725-737：回表页数乘 `(1 - baserel->allvisfrac)`——可见性地图中全可见页占比越高回表越少，这就是 IOS 依赖 VACUUM 的代价模型解释。`loop_count > 1` 分支（costsize.c:670-713）处理"作为 nestloop 内表被反复执行"：把 M-L 公式套在**全部循环的总取行数**上再摊回单次（`/ loop_count`），跨循环的缓存复用被显式建模——参数化路径（§4.5 第 1 步的 `get_loop_count`）在代价上的闭环就在这里。

### 7.4 三种连接的代价要点

连接代价都是两段式：`initial_cost_XXX` 快速粗估（供 §5.4 的 `add_path_precheck` 提前淘汰），幸存者再 `final_cost_XXX` 精算。

**Nestloop**（`initial_cost_nestloop` costsize.c:3396，`final_cost_nestloop` costsize.c:3478）骨架（costsize.c:3428-3456）：

```c
	startup_cost += outer_path->startup_cost + inner_path->startup_cost;
	run_cost += outer_path->total_cost - outer_path->startup_cost;
	if (outer_path_rows > 1)
		run_cost += (outer_path_rows - 1) * inner_rescan_start_cost;

	inner_run_cost = inner_path->total_cost - inner_path->startup_cost;
	inner_rescan_run_cost = inner_rescan_total_cost - inner_rescan_start_cost;

	if (jointype == JOIN_SEMI || jointype == JOIN_ANTI ||
		extra->inner_unique)
	{
		/*
		 * With a SEMI or ANTI join, or if the innerrel is known unique, the
		 * executor will stop after the first match.
		 * ...
		 */
		...
	}
	else
	{
		/* Normal case; we'll scan whole input rel for each outer row */
		run_cost += inner_run_cost;
		if (outer_path_rows > 1)
			run_cost += (outer_path_rows - 1) * inner_rescan_run_cost;
	}
```

即：外表一遍 + 内表 N_outer 遍，但第 2 遍起用 `cost_rescan` 的**重扫代价**（Material 化的内表重扫远便宜于首扫，这是优化器主动在内表上加 Material 节点的账本依据）。SEMI/ANTI 或内表唯一时匹配即停，期望只扫内表的一部分，留给 final 阶段结合选择率精算。参数化内表路径在这里收获：内表每遍只扫匹配部分（`ppi_rows` 行），nestloop + inner index scan 因此常是 OLTP 点连接的冠军。

**Merge join**（costsize.c:3681/3978）：两输入按连接键有序，各扫一遍；代价大头是**为无序输入补 Sort**——若输入路径的 PathKeys 已匹配（§1.2 interesting orders 的回报）则 Sort 免掉。initial 阶段还用直方图估两侧键值范围的重叠度，不重叠的尾部提前停扫。

**Hash join**（`initial_cost_hashjoin` costsize.c:4320，`final_cost_hashjoin` costsize.c:4439）：startup = 扫完整个内表 + 建哈希表（每行付哈希函数 + `cpu_tuple_cost` 的插入费，costsize.c:4363-4365）；run = 扫外表逐行探测。内表超过 work_mem 时分批（`ExecChooseHashTableSize` 算出 `numbatches`），批 I/O 的记账原文（costsize.c:4396-4412）：

```c
	if (numbatches > 1)
	{
		double		outerpages = page_size(outer_path_rows,
										   outer_path->pathtarget->width);
		double		innerpages = page_size(inner_path_rows,
										   inner_path->pathtarget->width);

		startup_cost += seq_page_cost * innerpages;
		run_cost += seq_page_cost * (innerpages + 2 * outerpages);
	}
```

即内表写一遍读一遍、外表写一遍读一遍，共 `2×(innerpages + outerpages)` 页的顺序 I/O——§9.3 将用这个公式算出 hash→merge 翻转的临界点。final 阶段的重点是**桶倾斜**：`estimate_hash_bucket_stats` 用 MCV 估出最忙桶的期望链长，探测代价按其放大——"倾斜键上 hash join 被估得很贵"的出处。

**小结**：`costsize.c` 是一组把统计信息（页数、行数、选择率、相关性、MCV）与硬件假设（GUC 系数）折算成两段式标量代价的纯函数；三类量纲项（页、行、操作符）线性叠加。seqscan 手算能与 EXPLAIN 逐位对上；cost_index 的全部附加复杂度在"回表随机 I/O × 缓存折扣 × 物理相关性"三个因子上。`enable_*` 开关不删路径，只计 disabled_nodes 垫底。

## 8. 选择率估计：selfuncs.c

代价公式里的 `rows` 与选择率从哪来？每个操作符在 `pg_operator` 里登记了限制选择率函数（`oprrest`，如 `=` → `eqsel`）与连接选择率函数（`oprjoin`，如 `=` → `eqjoinsel`），优化器按 OID 回调。统计信息由 ANALYZE 写入 `pg_statistic`：每列的 NULL 比例（stanullfrac）、不同值个数（stadistinct）、**MCV**（最常见值及其频率数组）、**等高直方图**（histogram bounds）、物理相关性等。

### 8.1 eqsel / var_eq_const：等值选择率逐段走读

`eqsel`（`src/backend/utils/adt/selfuncs.c:302`）→ `eqsel_internal`（selfuncs.c:311）：先 `get_restriction_variable` 把子句规整为 `Var = 常量` 形态，不满足则返回缺省值 `DEFAULT_EQ_SEL`（0.005，selfuncs.c:343）；常量侧已知则进入 `var_eq_const`（selfuncs.c:370）：

```c
							 STATISTIC_KIND_MCV, InvalidOid,

		{
```

### 8.2 scalarineqsel：范围条件 = MCV 精算 + 直方图插值

`scalarineqsel`（selfuncs.c:655）处理 `<`/`<=`/`>`/`>=`。无统计时有个趣味特例：CTID 上的范围比较按块号在表中的相对位置直接算（selfuncs.c:673-741）；一般列则：总体 = MCV 部分（`mcv_selectivity`，selfuncs.c:807，逐值代入操作符实算、频率求和）+ 直方图部分（`ineq_histogram_selectivity`，selfuncs.c:1116，只覆盖非 NULL 非 MCV 的长尾；没有直方图则按 0.5 认命）。`ineq_histogram_selectivity` 在等高直方图的边界数组上**二分**定位常量落在第几个桶，桶内再线性插值（数值类型可换算比例，其他类型取桶中点）——等高直方图保证每桶行数相同，所以"落在第 k 个桶的位置 x 处" ≈ 选择率 `(k-1+x)/桶数`。§7.2 手算例中 `unique1 < 1000` 得 0.1 就是这条路：1000 落在 0..9999 的 10% 分位。这与 Spark CBO 的 equi-height histogram、MySQL 8.0 的直方图原理一致，但 PG 的 ANALYZE 默认自动收集（`default_statistics_target=100` → 100 个 MCV + 100 个桶），MySQL 需手动 `ANALYZE TABLE ... UPDATE HISTOGRAM`。

### 8.3 eqjoinsel_inner 全函数走读 + 手算闭环例 2

`eqjoinsel`（selfuncs.c:2387）估计 `t1.x = t2.y` 的连接选择率。主体先取两侧 MCV（仅当两侧都有统计才费这个劲，selfuncs.c:2434-2446），当两侧 MCV 总条数 ≥ `EQJOINSEL_MCV_HASH_THRESHOLD`（200，断言构建下 20，selfuncs.c:153-158）时预先построbuild 哈希表加速 MCV 匹配（selfuncs.c:2480-2483，20devel 的新优化——把 O(n²) 的 MCV 两两比对降为哈希探测）。核心在 `eqjoinsel_inner`（selfuncs.c:2592），函数体两大分支完整走读：

**分支一：两侧都有 MCV**（selfuncs.c:2605-2704）：

```c
	if (have_mcvs1 && have_mcvs2)
	{
		/*
		 * We have most-common-value lists for both relations.  Run through
		 * the lists to see which MCVs actually join to each other with the
		 * given operator.  This allows us to determine the exact join
		 * selectivity for the portion of the relations represented by the MCV
		 * lists. ...
		 * For motivation see the analysis in Y. Ioannidis and S.
		 * Christodoulakis, "On the propagation of errors in the size of join
		 * results", ... 1991 ...
		 */
		...
		/* Fill the match arrays */
		eqjoinsel_find_matches(eqproc, collation,
							   hashLeft, hashRight,
							   false,
							   sslot1, sslot2,
							   sslot1->nvalues, sslot2->nvalues,
							   hasmatch1, hasmatch2,
							   p_nmatches, &matchprodfreq);
		nmatches = *p_nmatches;
		CLAMP_PROBABILITY(matchprodfreq);

		/* Sum up frequencies of matched and unmatched MCVs */
		matchfreq1 = unmatchfreq1 = 0.0;
		for (i = 0; i < sslot1->nvalues; i++)
		{
			if (hasmatch1[i])
				matchfreq1 += sslot1->numbers[i];
			else
				unmatchfreq1 += sslot1->numbers[i];
		}
		...（第二侧对称累加 matchfreq2/unmatchfreq2）

		/*
		 * Compute total frequency of non-null values that are not in the MCV
		 * lists.
		 */
		otherfreq1 = 1.0 - nullfrac1 - matchfreq1 - unmatchfreq1;
		otherfreq2 = 1.0 - nullfrac2 - matchfreq2 - unmatchfreq2;
		...
		/*
		 * We can estimate the total selectivity from the point of view of
		 * relation 1 as: the known selectivity for matched MCVs, plus
		 * unmatched MCVs that are assumed to match against random members of
		 * relation 2's non-MCV population, plus non-MCV values that are
		 * assumed to match against random members of relation 2's unmatched
		 * MCVs plus non-MCV values.
		 */
		totalsel1 = matchprodfreq;
		if (nd2 > sslot2->nvalues)
			totalsel1 += unmatchfreq1 * otherfreq2 / (nd2 - sslot2->nvalues);
		if (nd2 > nmatches)
			totalsel1 += otherfreq1 * (otherfreq2 + unmatchfreq2) /
				(nd2 - nmatches);
		/* Same estimate from the point of view of relation 2. */
		totalsel2 = matchprodfreq;
		if (nd1 > sslot1->nvalues)
			totalsel2 += unmatchfreq2 * otherfreq1 / (nd1 - sslot1->nvalues);
		if (nd1 > nmatches)
			totalsel2 += otherfreq2 * (otherfreq1 + unmatchfreq1) /
				(nd1 - nmatches);

		/*
		 * Use the smaller of the two estimates. ...
		 */
		selec = (totalsel1 < totalsel2) ? totalsel1 : totalsel2;
	}
```

结构：先对两侧 MCV 列表做一次"迷你连接"（真实等值函数比对/哈希匹配），`matchprodfreq = Σ 匹配对的频率乘积`——**头部×头部的贡献是精确计算的**；然后把总选择率分解为三块概率会计：匹配 MCV 对（精确）+ 本侧未匹配 MCV × 对侧非 MCV 长尾（均匀假设）+ 本侧非 MCV × 对侧(长尾+未匹配 MCV)（均匀假设），从两侧视角各算一遍取较小者。为什么头部要精确算？函数注释点名的 Ioannidis & Christodoulakis 1991 论文正是 §8.4 误差放大的理论出处——join 估计误差主要由分布倾斜的头部值贡献，MCV×MCV 精算就是对症下药。

**分支二：缺 MCV 时退化为教科书公式**（selfuncs.c:2705-2735）：

```c
	else
	{
		/*
		 * We do not have MCV lists for both sides.  Estimate the join
		 * selectivity as MIN(1/nd1,1/nd2)*(1-nullfrac1)*(1-nullfrac2). This
		 * is plausible if we assume that the join operator is strict and the
		 * non-null values are about equally distributed: a given non-null
		 * tuple of rel1 will join to either zero or N2*(1-nullfrac2)/nd2 rows
		 * of rel2, so total join rows are at most
		 * N1*(1-nullfrac1)*N2*(1-nullfrac2)/nd2 giving a join selectivity of
		 * not more than (1-nullfrac1)*(1-nullfrac2)/nd2. By the same logic it
		 * is not more than (1-nullfrac1)*(1-nullfrac2)/nd1, so the expression
		 * with MIN() is an upper bound. ...
		 */
		double		nullfrac1 = stats1 ? stats1->stanullfrac : 0.0;
		double		nullfrac2 = stats2 ? stats2->stanullfrac : 0.0;

		selec = (1.0 - nullfrac1) * (1.0 - nullfrac2);
		if (nd1 > nd2)
			selec /= nd1;
		else
			selec /= nd2;
	}

	return selec;
}
```

即 `(1-null1)(1-null2) / max(nd1, nd2)`——System R 原公式的直接后裔，注释里给了完整推导：R1 的每个非空值至多连上 R2 的 `N2/nd2` 行，故选择率上界 `1/nd2`；对称地上界 `1/nd1`；取 MIN 相当于"从 distinct 数较小的一侧看世界"，并假设该侧的值几乎都能找到配偶——外键连接恰好完美满足这个假设。半连接则走 `eqjoinsel_semi`（selfuncs.c:2751），估的是"外表行**至少有一个**匹配的概率"，口径不同，且会把 nd2 钳到内层 rel 的行数估计（selfuncs.c:2785-2797，这是内层过滤条件影响半连接大小估计的唯一通道）。

**手算闭环例 2**：`orders`（100 万行，`customer_id` 非空，`pg_stats.n_distinct = 100000`）JOIN `customers`（10 万行，主键 `id`，nd = 100000）ON `orders.customer_id = customers.id`，假设无可用 MCV（均匀分布列 ANALYZE 不会存 MCV）：

```
selec = (1-0)×(1-0) / max(100000, 100000) = 1/100000 = 1e-5
join_rows = N1 × N2 × selec = 10⁶ × 10⁵ × 10⁻⁵ = 1,000,000
```

EXPLAIN 会显示 `Hash Join ... rows=1000000`——每张订单恰好配上一个客户，完全符合外键直觉。再加一个 `WHERE customers.region = 'EU'`（选择率 0.2）：customers 侧先滤到 2 万行，join_rows = 10⁶ × (10⁵×0.2) × 10⁻⁵ = 200,000——**连接选择率不变、输入变小**，这就是"选择率与基数分离记账"的好处，手算依然闭环。

### 8.4 独立性假设与误差的指数放大：数值推演

以上所有公式都建立在两个假设上：**谓词独立**（多条件选择率相乘）与**连接键均匀/包含**（非 MCV 部分均匀分布、小 nd 侧的值都有配偶）。它们错起来的方式很有规律：

设每一层 join 的行数估计有一个乘性误差因子 ε（估计/真实）。k 层 join 树上，若每层误差同向（现实中很常见——同一批相关列、同一套过期统计会让所有层偏同一方向），**最终行数误差是 Π εᵢ，按层数指数复合**。数值推演：4 表链式连接，每层选择率都被低估 10 倍（ε = 0.1）：

| 层级 | 估计行数 | 真实行数 | 累计误差 |
|------|----------|----------|----------|
| 基表（过滤后） | 1 000 | 10 000 | 10× |
| 2 表 join | 100 | 100 000 | 1 000× |
| 3 表 join | 10 | 1 000 000 | 100 000× |
| 4 表 join | 1 | 10 000 000 | 10 000 000× |

到第 4 层，优化器以为结果是 1 行，实际是一千万行。灾难不在数字本身，而在**这个数字喂给了上层决策**：1 行的中间结果会诱导优化器（a）选 nestloop 把它当外表（实际循环一千万次）；（b）不并行（1 行不值得 1000.0 的 `parallel_setup_cost`）；（c）上层聚合选普通 HashAgg 且 work_mem 按 1 行规划。这就是 Ioannidis & Christodoulakis 1991（`eqjoinsel_inner` 注释引用的那篇）的中心结论：**join 大小估计误差随 join 数指数传播，且实践中偏向低估**。工程对策：MCV×MCV 精算削头部误差；`CREATE STATISTICS`（扩展统计：ndistinct/依赖度/多列 MCV）修谓词相关性；PG 不做的（运行期反馈重优化、adaptive execution）则是 Spark AQE 与商业库选择的另一条路——PG 的计划一旦选定执行期绝不回头，这使得"估计准"在 PG 里比在有自适应兜底的系统里更加生死攸关。

**小结**：选择率 = 统计字典（MCV 头部精确值 + 直方图长尾插值 + nullfrac/ndistinct）驱动的概率会计，层层回退到硬编码缺省值。等值三层瀑布、范围二段合成、连接的 MCV 迷你连接 + `1/max(nd)` 退化公式，全部可手算闭环。模型的命门是独立性与均匀性假设——误差沿 join 树指数放大，而 PG 没有执行期自适应，估计质量直接决定计划质量。

## 9. what-if 推演：三个定量案例

本节把前两节的公式反过来用：**改变一个输入，推演决策在哪里翻车**。三个案例都建议在真实实例上复现。

### 9.1 统计过期一个数量级：nestloop 惨案

时间序列表 `events`，上次 ANALYZE 时 1000 万行，之后持续写入现有 1 亿行（过期 10 倍），且新行的 `created_at` 全部大于直方图上界（"moving max"问题）。查询：

```sql
SELECT ... FROM events e JOIN devices d ON e.device_id = d.id
WHERE e.created_at > now() - interval '1 hour';
```

推演链条：① `scalarineqsel` 在直方图上二分，`now()-1h` 落在最后一个桶的边界之外 → 直方图部分选择率 ≈ 0，最终估计被钳到接近 0 行（实际最近一小时可能有 50 万行）；② `set_baserel_size_estimates` 给 events 过滤后估出个位数行；③ §7.4 的 nestloop 公式里 `outer_path_rows ≈ 1`，内表 rescan 项 `(outer_rows-1)×rescan_cost` 几乎为 0 → nestloop + devices 上的 inner index scan 代价被估得极低，完胜 hash join；④ 实际执行：50 万次索引探测，每次 3~4 次随机页访问，且 `LOG: duration` 里出现分钟级耗时；⑤ 若上面还有一层 join，§8.4 的指数放大继续接力。**翻车点的定量位置**：nestloop 的真实代价 ≈ `outer_rows × (索引下降+回表)`，与 hash join 的 `两表各扫一遍` 在 outer_rows ≈ 数千行处交叉——estimate 只要差出这个交叉点（这里差了 5 个数量级），决策必错。对策：autovacuum 的 `autovacuum_analyze_scale_factor` 对亿行表意味着 1000 万行变更才触发 ANALYZE，时序表应显式调小或定时 ANALYZE；PG 对"超出直方图上界"的常量有插值补偿（`ineq_histogram_selectivity` 会参考实际观测到的最大值），但补偿依赖统计新鲜度本身，救不了 10 倍过期。

### 9.2 相关列：选择率相乘失真的完整灾难链

表 `addr`（100 万行）：`city` 有 1000 个 distinct 值（`city='SF'` 选择率 0.001），`zip` 有 10000 个 distinct 值（`zip='94107'` 选择率 1e-4）。但 zip функционально决定 city——`zip='94107'` 的行**全部** `city='SF'`。查询 `WHERE city='SF' AND zip='94107'` 的真实选择率是 1e-4（100 行）。

推演链条：① `clauselist_selectivity` 按独立性相乘：`0.001 × 1e-4 = 1e-7` → 估 0.1 行，clamp 到 1 行（真实 100 行，低估 100 倍）；② 这个"1 行"让该表成为 join 树里人人争抢的完美外表——优化器把它排到最前面，为它选 nestloop 链；③ 三表查询 `addr ⋈ persons ⋈ orders` 就变成 `NestLoop(NestLoop(IndexScan(addr), persons), orders)`：addr 实际吐出 100 行、persons 每行匹配 50 人、orders 每人 100 单——第二层 nestloop 实际循环 5000 次、第三层 50 万次，而计划是按 1×50×... 的账本批准的；④ 内表若还是普通索引扫描，50 万次 × 随机 I/O，且**没有任何执行期逃生舱**（对比：Spark AQE 会在 shuffle 后按真实行数改 join 策略）。修复的定量验证：`CREATE STATISTICS st (dependencies) ON city, zip FROM addr; ANALYZE addr;` 之后，`dependencies` 统计记录 zip→city 的函数依赖度 ≈ 1.0，`clauselist_selectivity` 改用 `P(zip) × (dep + (1-dep)×P(city)) ≈ 1e-4`——估计回到 100 行，nestloop 链让位于 hash join。**这个案例是"为什么 add_path 的一切精巧都救不了坏输入"的示范：优化器精确地找到了错误代价模型下的最优解。**

### 9.3 work_mem 与 hash→merge 的临界点计算

两表各 100 万行、10000 页、行宽 100 字节，连接键上两侧都有 B-tree 索引（即 merge join 可以免 Sort 读取有序输入）。内表建哈希需要约 `10⁶ × (100 + 元组头开销) ≈ 120MB` 内存。

用 §7.4 的公式算 hash join 的 work_mem 敏感项。单批（work_mem ≥ 120MB）时无批 I/O；work_mem = 4MB 时 `numbatches = 32`，批 I/O 附加项：

```
Δcost = seq_page_cost × (innerpages + innerpages + 2×outerpages)
      = 1.0 × (12500 + 12500 + 2×12500) = 50000
      （page_size 按行数×行宽折页：10⁶×100B / 8KB ≈ 12500 页）
```

各方案的量级账本（cpu 项两方案近似相同，略）：

```
hash 单批:   扫两表 I/O 20000 + 哈希建/探 ≈ 10⁶×(0.0025+0.01) + 10⁶×0.0025 ≈ 35000 总量级
hash 32 批:  上式 + 50000 ≈ 85000
merge(两侧索引序读): 两侧 cost_index ≈ 相关性好时各 ~15000~25000 + 归并 cpu ≈ 55000 总量级
```

于是出现清晰的翻转区间：**work_mem 足够 → hash 胜（35000 < 55000）；work_mem 掉到引发多批 → hash 变 85000 > merge 的 55000 → 计划翻转为 merge join**（或在相关性差、索引读贵时翻向"显式 Sort + merge"乃至维持 hash 多批）。临界点就是 `Δcost = 2×seq_page_cost×(innerpages+outerpages)` 与两侧免 Sort 收益的交点，全部可由 §7 公式手算。工程含义：调大 work_mem 不只是"少溢盘"，它直接改变计划形态；反过来，一个在开发库（work_mem=64MB）表现良好的查询上生产（4MB）后计划面目全非，root cause 常在这一项。复现方法：`SET work_mem = '4MB' / '256MB'` 两次 EXPLAIN，观察 Hash Join ↔ Merge Join 的切换与 cost 构成。

**小结**：三个案例共用一个模式——**代价模型是正确的放大器，输入错了就精确地放大错误**。统计新鲜度决定行数、行数决定 join 方法与顺序、内存参数决定同一方法的常数项；EXPLAIN (ANALYZE) 里"估计行数 vs 实际行数"一列的巨大偏差，几乎总能沿着本节三条链之一追到根。

## 10. Path → Plan：create_plan 与 set_plan_references

### 10.1 create_plan：把骨架长成肌肉

`create_plan`（`src/backend/optimizer/plan/createplan.c:345`）接过最优 Path 树，调 `create_plan_recurse`（createplan.c:396）自顶向下按 `pathtype` 分发（`T_SeqScan` → `create_seqscan_plan`，`T_HashJoin` → `create_hashjoin_plan`……），做 Path 阶段刻意省略的脏活：为每个节点构造真正的目标列表；把 RestrictInfo 蜕壳成裸表达式挂到 `plan->qual`，**并剔除已被索引条件蕴含的冗余 qual**（cost 阶段为省时间只查"直接重复"，真正的蕴含证明留到这里补做——`extract_nonindex_conditions` 的注释 costsize.c:826-834 有交代）；为 nestloop 的参数化内表分配运行时 Param 编号（`curOuterParams`）；在需要处插入 Sort/Material 等胶水节点。函数注释特别强调：此时的 Var 仍是"planner 视角"的编号——留给下一步。

### 10.2 set_plan_references：最后的接线工

`set_plan_references`（`src/backend/optimizer/plan/setrefs.c:288`）做发车前检查，职责清单在 setrefs.c:230-255 的注释里，要点：

1. **压平范围表**：所有子查询层级的 rtable 合并成一张全局表（`glob->finalrtable`），各节点的 `scanrelid` 统一加偏移（setrefs.c:292 `rtoffset`）；
2. **上层 Var 改指下层输出槽**：扫描节点的 Var 保持指向表列，而 join/agg/sort 等上层节点的 Var 全部改写为 `OUTER_VAR/INNER_VAR + 子节点 targetlist 序号`——执行器由此才能用"取第 k 个输出列"的方式高速取值，不必按表列号查找；
3. 决断 `AlternativeSubPlan`（按估计执行次数二选一）、预解析操作符的实现函数 OID（省得执行期查 syscache）、登记计划依赖的对象 OID 列表（`relationOids`/`invalItems`，plancache 靠它做计划失效）；
4. 顺手删除无用的 SubqueryScan/Append/MergeAppend 中间节点（setrefs.c:257-268：rangetable 压平后它们的"变量命名空间缓冲"作用消失了）。

产物回到 §2.1 第 6 步，装进 `PlannedStmt`——优化器旅程结束，交棒执行器。

**小结**：Path→Plan 是"补全执行细节"，setrefs 是"全局编号统一 + 变量接线 + 依赖登记"。两者共同完成了从"搜索友好表示"到"执行友好表示"的转换——再次呼应本章开头：Path/Plan 分离是 PG 优化器最重要的工程设计。

## 11. 设计权衡与历史：五个 commit 的来龙去脉

用 `git log` 挖出的五个决定性节点，按时间排列。它们共同讲一个故事：**Path 这个抽象一旦立住，后续二十年的演进都是在给它扩维**。

### 11.1 `3fc6e2d7f5`（2016-03-07，PG 9.6）：upper planner pathification

> "Make the upper part of the planner work by generating and comparing Paths."（Tom Lane）

commit message 开头一句罕见地感性："I've been saying we needed to do this for more than five years, and here it finally is." 它拆掉了 `grouping_planner` 里"ever-growing tangle of spaghetti logic"，让 GROUP BY/DISTINCT/ORDER BY/LIMIT 每一步都变成 Path 竞争（§3.1）；同时废除"子查询只能返回单一 Plan"的限制——子查询的有序路径可以向上传播，跨层免 Sort 成为可能（message 原文点名此优化）。这次重构没带来立竿见影的计划改进（"very little change in visible planner behavior overall"），它的价值是**解锁**：FDW 聚合下推、并行聚合、后来的 Incremental Sort 与 Eager Aggregation 全部以它为地基。教训是结构性的：扩展点必须建在"多候选竞争"的抽象上，焊死的单一产物没法长东西。

### 11.2 `e2fa76d80b`（2012-01-27，PG 9.2）：参数化路径

> "Use parameterized paths to generate inner indexscans more flexibly."（Tom Lane）

9.2 之前，nestloop 内表用外表值做索引扫描是**硬编码在 nestloop 构造流程里的特例**，只支持"内表直接是基表"。这个 commit 把"我需要外部提供某些 rel 的当前行"抽象成路径的一等属性（`required_outer`/`param_info`），使参数化可以穿过 join 组合传播——`A ⋈ (B ⋈ C[param: A])` 这类"参数化的整棵子连接当内表"成为可能，OLTP 多表点查询的最优形态自此可达。代价是 add_path 支配比较凭空多出一维（§5.3 的 outercmp），以及"参数化路径不按 pathkeys 保命"的配套政策裁决。同年 `5b7b5518d0` 又大修一轮机制——一等抽象的引入总是牵一发动全身。

### 11.3 `d2d8a229bc`（2020-04-06，PG 13）：Incremental Sort

> "Implement Incremental Sort"（Tomas Vondra 提交，作者 James Coleman & Alexander Korotkov）

在"完全有序"与"完全无序"之间开辟第三态：输入已按 `(a)` 有序、需要 `(a,b)` 时，逐组只排 b。PathKeys 基础设施天然表达"前缀匹配"（§4.5 第 2 步的 `match_pathkeys_to_index` 注释就在为它铺路），所以这项功能的主要工作量在执行器和代价模型（两种模式：逐组排 vs 攒批排），planner 侧只是"在已有前缀有序路径上加一种新 Path"。它是 interesting orders 思想 40 年后的一次增量兑现：**部分有序也是资产**。

### 11.4 `e222534679`（2024-08-21，PG 18）：disabled_nodes 成为独立代价分量

> "Treat number of disabled nodes in a path as a separate cost metric."（Robert Haas）

在此之前，`enable_seqscan=off` 的实现是给路径加 `disable_cost = 1.0e10` 的惩罚。两个老毛病：巨数吞掉真实代价差（浮点精度下 1e10 + 445 与 1e10 + 470 难分伯仲，被禁路径之间的选择随机化）；且 1e10 不是无穷大，偶尔会出现"被禁路径居然赢了"的怪计划。改为整数计数、比较时最优先（§5.2 开头那段 `unlikely` 分支）后，语义变成干净的字典序：先比"违禁次数"，再比真实代价。EXPLAIN 也随之能显示 Disabled 标记（后续 `161320b4b9`/`84b8fccbe5`）。这是"用正确的数据结构表达语义，而不是用魔法常数近似语义"的典型案例。

### 11.5 `8e11859102`（2025-10，20devel）：Eager Aggregation

> "Implement Eager Aggregation"（Richard Guo）

允许把部分聚合下推到 join 之下（先按 join 键部分聚合、缩小 join 输入，join 后再 final 聚合）。实现方式呼应 §1.3 的框架讨论：不引入变换规则引擎，而是给每个可行的 RelOptInfo 挂一个平行的 `grouped_rel`（§4.2 的 `setup_simple_grouped_rels`、§6.2 的 `generate_grouped_paths`），让"聚合位置"作为一个新的候选维度进入既有的 DP + add_path 竞争。这是"是否有利取决于代价"的逻辑改写终于进入搜索空间的第一例——PG 用扩 RelOptInfo 维度的方式，在不背上 memo 框架的前提下吃到了 Cascades 式探索的一小口红利。

## 全章总结

一条 SQL 在优化器中的完整路线图（括号内为本章小节）：

```
standard_planner (2.1)
 └─ subquery_planner (2.2)
     ├─ pull_up_sublinks: IN/EXISTS → SEMI/ANTI join (2.3)
     ├─ pull_up_subqueries + is_simple_subquery (2.4)
     ├─ preprocess_expression: 常量折叠 (2.5)
     └─ grouping_planner: UpperRelationKind 流水线 (3)
         └─ query_planner → make_one_rel (4)
             ├─ set_base_rel_pathlists: seq/tid/index/bitmap (4.3)
             │    └─ create_index_paths → build_index_paths 五步 (4.4, 4.5)
             ├─ 一切路径 → add_path 四维支配裁决 (5)
             └─ make_rel_from_joinlist (6.1)
                 ├─ standard_join_search: DP 逐层 (6.2)
                 │    └─ join_search_one_level: 左深/右深→bushy→兜底 (6.3)
                 │        代价: costsize.c (7)   选择率: selfuncs.c (8)
                 └─ geqo: ≥12 表时的遗传算法 (6.4)
 ├─ create_plan: Path → Plan (10.1)
 └─ set_plan_references: 扁平化与接线 (10.2)  → PlannedStmt
```

三条贯穿全章的主线，合上书还应该记得：

1. **搜索空间的三层乘法压缩**（§1.1）：DP 管子问题个数（≤3^n）、连接图剪枝管配对个数（链式 O(n³)）、add_path 管每子问题的存活路径数（帕累托前沿）。任何一层的存在性论证都能用组合数算出来；
2. **"最优"是向量不是标量**：startup/total、pathkeys、参数化、行数、并行安全、disabled_nodes——每一维都对应上层的一种可能需求；PG 二十年的演进史（§11）就是给这个向量逐次扩维的历史；
3. **代价模型是正确的放大器**：公式全部可手算闭环（§7.2、§8.3），但独立性/均匀性假设失效时误差沿 join 树指数放大（§8.4、§9），而 PG 没有执行期自适应兜底——统计信息的质量就是计划的质量。

推荐动手实验：打开 `debug_print_plan`，选一条三表连接 SQL，对照 §7.2/§7.3 手算 EXPLAIN 里每个节点的 cost；`SET enable_hashjoin = off` 观察 disabled_nodes 机制让次优路径上位；`SET work_mem` 复现 §9.3 的计划翻转。

## 自测问题

**1. 12 表连接不加任何剪枝的计划空间约是多大量级？PG 用哪三层机制把它压缩到什么量级？各自的源码落点在哪里？**

答：连接顺序 = 12! × Cat(11) ≈ 2.8×10¹³，再乘每内部节点 3 种连接方法（3¹¹）≈ 5×10¹⁸。三层压缩：① 动态规划利用最优子结构把"枚举计划"变为"枚举子问题二划分"，上界 3¹² ≈ 53 万对（`standard_join_search`，allpaths.c:3952，逐层填 `join_rel_level[]`）；② 连接图剪枝——只组合有连接子句或顺序约束的对（`join_search_one_level` 中 `have_relevant_joinclause`/`have_join_order_restriction` 检查，joinrels.c:195-196；链式连接图下降到 O(n³) 配对）；③ `add_path`（pathnode.c:459）帕累托支配剪枝把每个 RelOptInfo 的存活路径压到小常数，防止逐层平方复合。`geqo_threshold=12` 正是 DP 开销水位线（guc_parameters.dat:1187-1194）。

**2. add_path 的支配比较涉及哪些维度？一条新路径要淘汰旧路径必须同时满足什么？为什么参数化路径被"假装没有 pathkeys"？**

答：维度：disabled_nodes（最高优先，e222534679 引入的整数高阶分量）、total_cost 与 startup_cost（1% 模糊比较，`compare_path_costs_fuzzily`，pathnode.c:181）、pathkeys（`compare_pathkeys`）、参数化集合（`bms_subset_compare(PATH_REQ_OUTER(...))`）、rows、parallel_safe。淘汰旧路径需同时：代价不更差、排序不更差、required_outer 是旧的子集或相等、行数不更多、并行安全不更差（pathnode.c:515-607）；任一维互有胜负则两条都留。参数化路径视为无 pathkeys（pathnode.c:472-473）是显式政策：它们几乎只作 nestloop 内表、每循环被 rescan，输出序对上层无用，若按序保命会让参数化×排序两维叉乘导致路径数失控——牺牲少量最优性换搜索规模。

**3. 没有 PathKeys 机制（每个 rel 只留最便宜路径），构造一个会错过更优计划的具体查询，并说明差距来源。**

答：`SELECT * FROM t1 JOIN t2 ON t1.a=t2.a JOIN t3 ON t2.b=t3.b ORDER BY t1.a LIMIT 10`，t1 上有 a 的 B-tree 索引。t1 层 SeqScan（如 22500）比 IndexScan 全扫（如 45000）便宜，若按标量代价淘汰后者，最终计划只能对整个三表连接结果做全量 Sort 再取 10 行——startup 巨大、LIMIT 无法生效。保留带 `(t1.a)` pathkeys 的 IndexScan（add_path 的排序维使其存活），可得逐层保序的 nestloop 链，startup≈0、取满 10 行即停，实际执行差三四个数量级。等价类（PathKey 的 `pk_eclass`，pathnodes.h:1806）还使 `t1.a` 的序同时是 `t2.a` 的序，上层判定免排序。

**4. 手算：某表 `relpages=345`、`reltuples=10000`，查询带一个单操作符 WHERE 条件、选择率 0.1。写出 seqscan 的 total_cost 和 rows，并注明每项出处。**

答：由 `cost_seqscan`（costsize.c:271）：I/O = `seq_page_cost × pages` = 1.0×345 = 345（costsize.c:300）；CPU = `(cpu_tuple_cost + cpu_operator_cost×1) × tuples` = (0.01+0.0025)×10000 = 125（costsize.c:306-307，注意乘全表行数，因为每行都要检查 WHERE）；total = 470.0，startup = 0。rows = 10000×0.1 = 1000（选择率来自 §8，与代价独立记账；投影按输出 1000 行计费，简单 Var 投影成本为 0）。与 EXPLAIN 的 `cost=0.00..470.00 rows=1000` 一致。

**5. `effective_cache_size` 不分配任何内存，它如何影响计划？写出关键公式并说明 `total_table_pages` 的角色。**

答：它只是 Mackert-Lohman 公式的参数 b：`index_pages_fetched`（costsize.c:898）中 `b = effective_cache_size × T / total_pages`（costsize.c:915），其中 `total_pages = root->total_table_pages + index_pages`——`make_one_rel` 预先累加的查询全部表页数（allpaths.c:234），把缓存假设按页数比例分摊给各表。b 越大，公式估出的重复堆页越可能免费（T≤b 时重访问全免），`cost_index` 的 I/O 项越低，索引路径越易胜出。它是"优化器对操作系统+共享缓冲总缓存规模的假设"，纯参数、零分配。

**6. `eqjoinsel_inner`（selfuncs.c:2592）在两侧都有 MCV 时如何计算？为什么头部值要精确计算，理论依据是哪篇论文？缺 MCV 时的退化公式及其推导？**

答：有 MCV 时做"迷你连接"：`eqjoinsel_find_matches` 用真实等值函数（MCV 条数多时建哈希表，阈值 `EQJOINSEL_MCV_HASH_THRESHOLD=200`）标出两侧匹配的 MCV，`matchprodfreq=Σ匹配对频率乘积` 为精确部分；再按"未匹配 MCV × 对侧长尾 / (nd-已见)"与"长尾 × 对侧(长尾+未匹配 MCV) / (nd-匹配数)"两项均匀假设补全，从两侧视角各估一个总值取较小（selfuncs.c:2683-2703）。精确算头部是因为 join 估计误差主要由倾斜的头部值贡献——注释引用 Ioannidis & Christodoulakis 1991《On the propagation of errors in the size of join results》，该文同时证明误差沿 join 树指数传播。缺 MCV 时：`selec=(1-nullfrac1)(1-nullfrac2)/max(nd1,nd2)`（selfuncs.c:2730-2735）——R1 每非空行至多配 R2 的 N2/nd2 行给出上界 1/nd2，对称得 1/nd1，取 MIN 即"从小 nd 侧看世界、假设其值都有配偶"。

**7. `join_search_one_level`（joinrels.c:78）分哪三段？为什么会存在"某一层合法 joinrel 为空"的正常情况？举出源码注释中的例子。**

答：三段：① 左深+右深——(level-1) 层 joinrel × 单表，仅当 old_rel 有连接子句/等价类连接/顺序约束时按子句配对，否则笛卡尔（joinrels.c:96-143）；② bushy——joinrels[k] × joinrels[level-k]（k 到中点），要求 relids 不重叠且有相关子句或约束（joinrels.c:153-203）；③ last-ditch——本层为空时强制左深/右深笛卡尔兜底（joinrels.c:205-261）。空层合法的例子（joinrels.c:239-255）：`SELECT ... FROM t1 WHERE x IN (SELECT ... FROM t2,t3 ...) AND y IN (SELECT ... FROM t4,t5 ...)`——两个半连接上拉后成 5 表问题，任何 4 表组合都拆散某个半连接的原子性而被 `join_is_legal` 拒绝，第 4 层必然为空，第 5 层才出现合法 bushy 计划；这同时证明 bushy 枚举不可省。仅当无 SpecialJoinInfo 且无 LATERAL 时空层才 `elog(ERROR)`。

**8. `enable_seqscan=off` 在 PG 18 前后分别是如何实现的？新机制解决了旧机制的什么问题？**

答：18 之前：给被禁路径加 `disable_cost=1.0e10` 惩罚。问题：巨数吞掉真实代价差（1e10+445 与 1e10+470 在浮点比较中难以稳定区分，多条被禁路径间选择近乎随机），且 1e10 非无穷大，极端场景被禁路径可能反超。commit `e222534679`（2024-08）改为路径记整数 `disabled_nodes`，比较时最高优先（`compare_path_costs_fuzzily` 开头的 unlikely 分支，pathnode.c:186-193；pathlist 也按它排序，costsize 各函数如 cost_seqscan 尾部 costsize.c:336-337 负责计数）——语义变为字典序：先比违禁次数、再比真实代价，被禁路径之间的比较恢复精确，且"全部被禁仍能出计划"天然成立。

**9. 相关列 `WHERE city='SF' AND zip='94107'`（zip 函数决定 city）会让优化器犯什么错？错误如何沿计划放大？PG 提供的修复机制是什么？**

答：`clauselist_selectivity` 按独立性假设相乘：0.001×1e-4=1e-7，百万行表估约 1 行（clamp），真实 100 行，低估 100 倍。放大链：该表被当作完美小外表排到 join 链最前 → 选 nestloop 链且不并行（1 行不值 parallel_setup_cost=1000）→ 每层实际行数比账本大百倍并按 §8.4 的规律逐层指数复合 → 内表索引被探测数十万次。且 PG 无执行期重优化，无逃生舱。修复：`CREATE STATISTICS (dependencies) ON city, zip FROM addr` + ANALYZE——扩展统计记录 zip→city 依赖度≈1，选择率改按 `P(zip)×(dep+(1-dep)×P(city))≈1e-4` 计算，估计回到 100 行，计划回归 hash join。

**10. 为什么 PG 不采用 Volcano/Cascades 式的 memo 优化器框架？Eager Aggregation（8e11859102）如何在不引入该框架的情况下获得类似能力？**

答：三个理由：① PG 已把确定性有利的逻辑改写（子链接上拉、子查询展平、常量折叠、外连接消除）前置为一次性预处理，搜索阶段逻辑自由度只剩连接顺序——此时 memo 的 group 概念退化为"基表集合"，以 relids 为 key 的 `join_rel_hash`/`join_rel_level[]` 就是够用的"穷人版 memo"（README:168-173：等价 joinrel 共享 RelOptInfo、路径同台竞争）；② 自顶向下按需求解的优势（分支上界剪枝）在"每层每种 interesting order 都可能被上层消费"的 SQL 场景收益有限，而自底向上冻结下层的性质使内存管理简单（README:174-182，可安全 pfree 被支配路径）；③ 工程上单次规划要求毫秒级。Eager Aggregation 的做法：给每个可行 RelOptInfo 挂平行的 `grouped_rel`（`setup_simple_grouped_rels`，allpaths.c:201；`generate_grouped_paths`，standard_join_search 内 allpaths.c:4032-4040），让"聚合放在 join 前还是后"作为新候选维度进入既有 DP + add_path 竞争——用扩展状态维度而非变换规则引擎，在 DP 框架内模拟了一条 Cascades 规则。
