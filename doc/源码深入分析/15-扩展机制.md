# 第 15 篇 · 扩展机制：PostgreSQL 的"平台性"（v2）

> 对应《PostgreSQL 源码学习指南》第 17 章。源码基线：PostgreSQL 20devel（master 分支）。
> v2 变更：pg_stat_statements 六个 hook 函数与两级存储全走读；`internal_load_library` 全函数走读与 magic block 逐字段比对的所以然；`CreateExtension` 主线走读；postgres_fdw 计划/执行主线与 `foreign_expr_walker` 白名单判定（含 collation 陷阱三个反例）；SPI_execute 主线；"从零写最小扩展"实战闭环；4 组设计历史 commit 考证；3 个 what-if 事故推演。

## 本章导读

如果只许用一句话概括 PG 与 MySQL 的生态差异，那就是：**MySQL 是一个数据库，PostgreSQL 是一个做数据库的平台**。PostGIS（地理）、TimescaleDB（时序）、Citus（分布式）、pgvector（向量检索）都不是 fork，而是运行在原版 PG 内核之上的扩展——这在 MySQL 世界里没有对应物。

本篇自底向上拆解这个"平台"的六层机制，并在每一层追问"为什么这样设计"：

1. **Hook 体系**——内核在关键路径上留的函数指针。为什么是"裸函数指针 + 调用者自觉链式转发"而不是注册表/事件总线？极简主义换来了什么、牺牲了什么？
2. **动态库加载**——`internal_load_library` 全函数走读；magic block 为什么要逐字段比对 ABI；`shared_preload_libraries` 与 `LOAD` 的能力分界为什么恰好画在"共享内存"上，以及 20devel 的 `RegisterShmemCallbacks` 如何放宽这条线；
3. **扩展框架**——`CREATE EXTENSION` 主线走读：控制文件解析 → 脚本执行 → 依赖记录。没有它的 PG 9.1 之前，dump/restore 是什么样的灾难；
4. **FDW**——把任何数据源伪装成表。重点是下推的**正确性边界**：为什么 collation、易变函数、本地自定义算子不能下推，各给一个"下推后结果就是错的"的具体反例；
5. **自定义类型/操作符/操作符类**——PostGIS 模式；
6. **过程语言框架与 SPI**——`SPI_execute` 主线。

贯穿全篇的案例是 `contrib/pg_stat_statements`（hook 教科书）和 `contrib/postgres_fdw`（FDW 参考实现），两者的关键函数全部逐段走读。最后给出一个几十行的"从零写最小扩展"完整闭环——这是动手阶段的直接教材。

## 15.1 Hook 体系全景

### 15.1.1 Hook 是什么：一个函数指针 + 一行 if

PG 的 hook 模式极其朴素。以 planner 为例（`src/backend/optimizer/plan/planner.c`）：

```c
/* Hook for plugins to get control in planner() */
planner_hook_type planner_hook = NULL;          /* planner.c:74 */
```

调用点在 `planner()` 本体（planner.c:328-339）：

```c
PlannedStmt *
planner(Query *parse, const char *query_string, int cursorOptions,
		ParamListInfo boundParams, ExplainState *es)
{
	PlannedStmt *result;

	if (planner_hook)
		result = (*planner_hook) (parse, query_string, cursorOptions,
								  boundParams, es);
	else
		result = standard_planner(parse, query_string, cursorOptions,
								  boundParams, es);
```

全局函数指针默认 NULL；扩展在 `_PG_init` 里把它指向自己的函数；内核在调用点检查非空则转交控制权。扩展函数内部通常再调用 `standard_planner()`（或链式调用上一个 hook），形成"洋葱式"包裹。注意 20devel 的签名比老版本多了一个 `ExplainState *es` 参数——hook 签名与 `standard_planner` 始终保持完全同构，这是"hook 函数可以透明转发"的前提。

### 15.1.2 主要 hook 的位置与签名

| Hook | 定义位置 | 时机 |
|---|---|---|
| `post_parse_analyze_hook` | `parser/analyze.c:74`（调用点 150-151） | 语义分析完成、拿到 Query 树之后 |
| `planner_hook` | `optimizer/plan/planner.c:74`（调用点 333） | 整体替换/包裹优化器 |
| `ExecutorStart_hook` | `executor/execMain.c:70` | 执行器初始化 |
| `ExecutorRun_hook` | `executor/execMain.c:71` | 执行主循环 |
| `ExecutorFinish_hook` | `executor/execMain.c:72` | AFTER 触发器等收尾 |
| `ExecutorEnd_hook` | `executor/execMain.c:73` | 执行器销毁前（统计的黄金采集点） |
| `ExecutorCheckPerms_hook` | `executor/execMain.c:76` | 表级权限检查（行级安全类扩展用） |
| `ProcessUtility_hook` | `tcop/utility.c:72`（调用点 523-524） | 所有 DDL/工具语句 |
| `shmem_startup_hook` | `storage/ipc/ipci.c:31`（调用点 110-111/158-159） | 共享内存创建后初始化自己的份额（legacy 路径） |

其他常用 hook 散布各处：`emit_log_hook`（elog.c，日志外送）、`check_password_hook`、`object_access_hook`（DDL 审计，sepgsql 用）、`set_rel_pathlist_hook` / `create_upper_paths_hook`（往优化器塞自定义路径，Citus/列存的入口）等。用 `grep -rn "_hook_type.*= NULL" src/backend/` 可以数出内核几十个 hook 点。

共享内存申请侧在 20devel 有一处重要演变：PG 15~18 使用 `shmem_request_hook`（commit 4f2400cb3f，2022-05-13），而当前 master 已把它重构为**回调注册表**——`RegisterShmemCallbacks()`（`storage/ipc/shmem.c:873`，commit 283e823f9d）。扩展提交一个 `ShmemCallbacks` 结构（`src/include/storage/shmem.h:123-153`，含 `request_fn`/`init_fn`/`attach_fn` 三个回调与 `flags`），默认仍要求在 postmaster 启动期注册，但新增 `SHMEM_CALLBACKS_ALLOW_AFTER_STARTUP` 标志（shmem.h:167）允许启动后注册。细节见 15.3.4。

### 15.1.3 所以然①：为什么是裸函数指针，而不是注册表/事件总线

一个"正经"的插件框架通常会做成：`RegisterPlannerInterceptor(callback, priority)`，框架内部维护回调列表，按优先级依次调用，支持注销。PG 偏偏没有——每个 hook 就是一个全局变量，谁都可以读写。这个选择值得认真推敲。

**得到的**：

1. **零框架成本**。内核侧每个 hook 点只有两行代码（一个全局变量、一个 if），没有列表管理、没有调用开销（一次判空 + 间接调用）。planner 这样每条语句都过的热路径上，这不是小事；
2. **签名同构 = 全能力**。hook 函数拿到的参数与 `standard_planner` 一模一样，它可以**替换**（完全不调 standard）、**包裹**（前后加逻辑）、**旁观**（记录后透传）——事件总线模式往往只能旁观，替换语义很难表达；
3. **组合语义由扩展自己定义**。链式转发时，后装的扩展在外层，先装的在内层，天然形成洋葱模型，每层自己决定要不要继续往里传。

**付出的**：

1. **没有优先级仲裁**。两个扩展都挂 `planner_hook` 时，调用顺序完全由加载顺序决定：`shared_preload_libraries = 'a, b'` 时，a 先加载（`planner_hook = a_planner`），b 后加载（保存 `prev = a_planner`，再覆盖 `planner_hook = b_planner`）。运行时 b 在外层先执行——**后加载者包裹先加载者（LIFO）**。如果 b 是"改写计划"型（如 Citus）而 a 是"记账"型（如 pgss），这个顺序是合理的（记账在内层看到改写后的结果？不——恰好相反，内层 a 先于 standard_planner 拿到控制权……实际语义要逐层推演）。管理员只能靠调整 GUC 里的顺序来调整语义，且没有任何机制阻止两个扩展的语义组合根本不兼容；
2. **约定无强制力**。"保存 prev 并转发"只是惯例（见 15.1.4 的反面推演），编译器和内核都不检查；
3. **卸载困难**。装 hook 容易，卸载几乎不可能：你把 `planner_hook` 恢复成自己保存的 `prev`，但如果你之后还有别的扩展装载过，它保存的 prev 正是你的函数指针——你一恢复，链条就断了。这就是 `dfmgr.c:183-186` 那条注释的由来：

```c
 * NB: There is presently no way to unload a dynamically loaded file.  We might
 * add one someday if we can convince ourselves we have safe protocols for un-
 * hooking from hook function pointers, releasing custom GUC variables, and
 * perhaps other things that are definitely unsafe currently.
```

所以 PG 没有 `UNLOAD` 命令，`.so` 一旦进入进程就活到进程结束。**极简主义把复杂性从框架转移给了约定**——这是 PG 一贯的工程品味：宁要一个人人看得懂、出了问题可推理的裸机制，不要一个功能完备但语义黑箱的框架。

### 15.1.4 what-if 推演：某个扩展不转发 prev 会怎样

设 `shared_preload_libraries = 'pg_stat_statements, myrewriter'`。pgss 先加载，把 `planner_hook` 指向 `pgss_planner`；myrewriter 后加载，但它的作者**忘了保存并转调 prev**，直接：

```c
planner_hook = myrewriter_planner;   /* 反面教材：没有 prev_planner_hook = planner_hook */
```

后果链：

1. `pgss_planner` 从此永远不会被调用——没有任何报错，没有任何日志；
2. 但 pgss 的其他 hook（ExecutorEnd 等）还在正常工作，所以 `pg_stat_statements` 视图**看起来一切正常**，只是 `plans` 列恒为 0、`total_plan_time` 恒为 0（`pg_stat_statements.track_planning = on` 时）；
3. 更隐蔽的：`pgss_planner` 还负责维护 `nesting_level`（见 15.2.5）。它被绕过后，planner 期间执行的 SQL 函数会被误判为顶层语句，统计口径悄悄错乱；
4. 排查这种问题只能靠 gdb 打印 `planner_hook` 的值再反查符号，或者二分 `shared_preload_libraries`。

这就是"约定无强制力"的代价：**hook 链的完整性依赖链上每一环的自觉**，一环失德，故障静默且归因困难。所以读任何扩展源码，`_PG_init` 里 `prev_xxx = xxx_hook; xxx_hook = my_xxx;` 的配对是第一个要检查的东西。

### 小结

Hook = 全局函数指针 + 调用点判空；组合 = 各扩展自觉保存 prev 并链式转发，后装者在外层。零框架成本、全能力签名，换来无优先级仲裁、无强制约定、无法卸载。这是一个把复杂性推给约定的极简设计，理解它的代价模型（15.1.4）比背 hook 清单更重要。

## 15.2 案例全走读：pg_stat_statements 如何串接全链路

`contrib/pg_stat_statements/pg_stat_statements.c`（约 3000 行）是 hook 教科书：它要回答"每类 SQL 执行了多少次、总耗时多少"，为此几乎挂满了整条管线。本节按数据流完整走读它的每个 hook。

### 15.2.1 总体架构：一张共享哈希表 + 一个外部文本文件

先看数据结构（均在 pg_stat_statements.c）：

```c
typedef struct pgssHashKey          /* :143 */
{
	Oid			userid;			/* user OID */
	Oid			dbid;			/* database OID */
	int64		queryid;		/* query identifier */
	bool		toplevel;		/* query executed at top level */
} pgssHashKey;
```

```c
typedef struct pgssEntry            /* :233 */
{
	pgssHashKey key;			/* hash key of entry - MUST BE FIRST */
	Counters	counters;		/* the statistics for this query */
	Size		query_offset;	/* query text offset in external file */
	int			query_len;		/* # of valid bytes in query string, or -1 */
	int			encoding;		/* query text encoding */
	TimestampTz stats_since;	/* timestamp of entry allocation */
	TimestampTz minmax_stats_since; /* timestamp of last min/max values reset */
	slock_t		mutex;			/* protects the counters only */
} pgssEntry;
```

关键设计：**两级存储**。

- **第一级**：共享内存哈希表 `pgss_hash`，key 是（用户, 库, queryId, 是否顶层）四元组，value 是定长的 `pgssEntry`。`Counters`（:154-215）是纯数字：调用次数、planning/execution 各自的 min/max/mean/方差（Welford 算法）、行数、buffer 命中/读写、WAL 记录数/字节数、JIT 各阶段耗时、并行 worker 数、generic/custom plan 次数——全部定长，所以哈希表可以在启动时精确定容；
- **第二级**：**查询文本不进共享内存**，而是追加写入外部文件 `PGSS_TEXT_FILE`（`pg_stat_tmp/pgss_query_texts.stat`），条目里只存 `(query_offset, query_len)`。读视图时才 `qtext_load_file()`（:2311）把文件整个读进本地内存、按偏移取文本（`qtext_fetch`，:2404）。

为什么拆两级？这是 9.4 的一次重要重构（commit f0d6f20278，2014-01-27，"Keep pg_stat_statements' query texts in a file, not in shared memory"）。此前查询文本按 `track_activity_query_size`（默认 1024 字节）截断后塞在共享内存条目里：长 SQL 被截断导致无法复原，想存完整文本又会让每个条目大得离谱（共享内存启动定容，不能按需伸缩）。移到外部文件后：文本不限长、共享内存占用锐减、代价是读视图时多一次文件 IO——对一个"写多读少"的统计设施，这是正确的权衡方向。文件会积累死文本（条目被淘汰后残留），所以有配套的垃圾回收 `need_gc_qtexts()`/`gc_qtexts()`（:2427/:2476）：当文件大小超过"条目数 × 平均文本长度"的若干倍时，持排他锁重写文件、更新所有条目的偏移。

锁协议也是两级：`pgss->lock`（LWLock，pgssSharedState :248-258）保护哈希表结构（查找共享锁、增删排他锁）；每个条目自己的 `mutex`（自旋锁）只保护计数器——高频的"累加一次统计"只需共享锁 + 条目自旋锁，不同 queryId 的更新互不阻塞。

### 15.2.2 第一步：_PG_init 注册（:390）

```c
void
_PG_init(void)
{
	/*
	 * In order to create our shared memory area, we have to be loaded via
	 * shared_preload_libraries.  If not, fall out without hooking into any of
	 * the main system.  ...
	 */
	if (!process_shared_preload_libraries_in_progress)
		return;

	/*
	 * Inform the postmaster that we want to enable query_id calculation if
	 * compute_query_id is set to auto.
	 */
	EnableQueryId();
```

逐段解说：

- 开头的 `process_shared_preload_libraries_in_progress` 检查是**能力门槛**：不经 preload 加载则静默退出（不装 hook），因为共享内存申请不到（见 15.3.4）。注释特意说明为什么不报错：允许 `CREATE EXTENSION pg_stat_statements`（建 SQL 函数）在未 preload 的库上执行，只是函数调用时会提示未加载；
- `EnableQueryId()`（`nodes/queryjumblefuncs.c:171`）：把全局 `query_id_enabled` 置真（当 `compute_query_id = auto` 时）。queryId 计算自 PG 14 起在内核里（见 15.2.4），扩展只需"举手"声明需要它；
- 接着是五个 `DefineCustomXxxVariable`（:412-468）定义 `pg_stat_statements.max`（条目上限，默认 5000，PGC_POSTMASTER——因为它决定共享内存尺寸，改了必须重启）、`.track`（top/all/none）、`.track_utility`、`.track_planning`、`.save`；然后 `MarkGUCPrefixReserved("pg_stat_statements")`（:470）把整个前缀占住，防止拼错的 GUC 名静默生效；
- 共享内存申报（:475）：`RegisterShmemCallbacks(&pgss_shmem_callbacks);`——注册结构见下节；
- 最后是六组 hook 安装（:480-493），全部遵循"保存 prev 再覆盖"的礼仪：

```c
	prev_post_parse_analyze_hook = post_parse_analyze_hook;
	post_parse_analyze_hook = pgss_post_parse_analyze;
	prev_planner_hook = planner_hook;
	planner_hook = pgss_planner;
	prev_ExecutorStart = ExecutorStart_hook;
	ExecutorStart_hook = pgss_ExecutorStart;
	prev_ExecutorRun = ExecutorRun_hook;
	ExecutorRun_hook = pgss_ExecutorRun;
	prev_ExecutorFinish = ExecutorFinish_hook;
	ExecutorFinish_hook = pgss_ExecutorFinish;
	prev_ExecutorEnd = ExecutorEnd_hook;
	ExecutorEnd_hook = pgss_ExecutorEnd;
	prev_ProcessUtility = ProcessUtility_hook;
	ProcessUtility_hook = pgss_ProcessUtility;
```

### 15.2.3 共享内存：request 与 init 两个回调

注册的回调结构（:267）：

```c
static const ShmemCallbacks pgss_shmem_callbacks = {
	.request_fn = pgss_shmem_request,
	.init_fn = pgss_shmem_init,
};
```

`pgss_shmem_request`（:506）在共享内存定型**前**申报需求——注意这是 20devel 的新 API，用 designated initializer 声明一张哈希表和一个结构体：

```c
static void
pgss_shmem_request(void *arg)
{
	ShmemRequestHash(.name = "pg_stat_statements hash",
					 .nelems = pgss_max,
					 .hash_info.keysize = sizeof(pgssHashKey),
					 .hash_info.entrysize = sizeof(pgssEntry),
					 .hash_flags = HASH_ELEM | HASH_BLOBS,
					 .ptr = &pgss_hash,
		);
	ShmemRequestStruct(.name = "pg_stat_statements",
					   .size = sizeof(pgssSharedState),
					   .ptr = (void **) &pgss,
		);
}
```

`.ptr` 传出指针地址，框架在 init/attach 阶段把分配好的共享内存地址填回来。在 PG ≤ 18 里这两步分别对应 `shmem_request_hook` 里的 `RequestAddinShmemSpace()` 和 `shmem_startup_hook` 里的 `ShmemInitHash()`/`ShmemInitStruct()`——语义相同，新 API 把"申报"与"初始化"的配对关系显式化了，还顺手解决了 EXEC_BACKEND（Windows）模式下每个 backend 重新 attach 的样板代码问题（`attach_fn` 由框架统一调度）。

`pgss_shmem_init`（:530）在共享内存分配后执行初始化。主线（:546-608）：

```c
	Assert(!IsUnderPostmaster);

	/* Initialize the shmem area with no statistics. */
	tranche_id = LWLockNewTrancheId("pg_stat_statements");
	LWLockInitialize(&pgss->lock.lock, tranche_id);
	...
	SpinLockInit(&pgss->mutex);
	pgss->extent = 0;
	...
	/* Set up a shmem exit hook to dump the statistics to disk ... */
	on_shmem_exit(pgss_shmem_shutdown, (Datum) 0);
	...
	/* Unlink query text file possibly left over from crash */
	unlink(PGSS_TEXT_FILE);

	/* Allocate new query text temp file */
	qfile = AllocateFile(PGSS_TEXT_FILE, PG_BINARY_W);
```

之后（若 `pgss_save = on`）从上次正常关机时 `pgss_shmem_shutdown`（:734）落盘的 `pgss_dump_file` 中恢复统计与文本——所以 pgss 的统计**跨重启保留、跨崩溃不保留**（崩溃后文件头校验不过就丢弃重来，统计数据不值得做 WAL 级持久化）。

### 15.2.4 第二步：解析后拿指纹——queryId 的计算与规范化

先看内核侧。语义分析完成后（`parser/analyze.c:147-151`）：

```c
	if (IsQueryIdEnabled())
		jstate = JumbleQuery(query);

	if (post_parse_analyze_hook)
		(*post_parse_analyze_hook) (pstate, query, jstate);
```

`JumbleQuery`（`nodes/queryjumblefuncs.c:139`）遍历 Query 树，把"结构性"字段（节点类型、表 OID、操作符 OID……）喂进哈希、**忽略常量的具体值但记录其位置**，得到 64 位指纹写入 `query->queryId`；返回的 `JumbleState` 里带着 `clocations` 数组——每个被忽略常量在原 SQL 文本中的 (location, length)。于是 `SELECT * FROM t WHERE id = 42` 和 `... WHERE id = 99` 指纹相同。哪些字段参与 jumble 由节点定义上的 `query_jumble_ignore` 标注决定，代码由 `gen_node_support.pl` 自动生成（commit 3db72ebcbe）——这避免了新增节点字段时忘改 jumble 逻辑的经典 bug。

历史脉络（设计历史 D）：queryId 机制经历了三步演化——

1. **8.4**（commit 7466eeac61，2009-01-04）：pg_stat_statements 初版，按**文本完全相同**归组，`WHERE id = 42` 与 `WHERE id = 99` 是两条；
2. **9.2**（commit 7313cc0163，2012-03-28，"lump similar queries together"）：Peter Geoghegan 把 jumble 逻辑放进 pgss 自己——扩展里维护一棵和内核 Query 结构平行的遍历代码，每个大版本都要跟着内核改节点定义，维护成本极高；
3. **PG 14**（commit 5fd9dfa5f5，2021-04-07，"Move pg_stat_statements query jumbling to core"）：jumble 上收内核，新增 `compute_query_id` GUC，queryId 同时供 pgss、`pg_stat_activity.query_id`、`EXPLAIN VERBOSE` 共用。这是"扩展孵化 → 内核收编"路径的典型标本：扩展先证明需求，内核再提供正式设施。

回到 pgss 侧，`pgss_post_parse_analyze`（:834）全文：

```c
static void
pgss_post_parse_analyze(ParseState *pstate, Query *query, const JumbleState *jstate)
{
	if (prev_post_parse_analyze_hook)
		prev_post_parse_analyze_hook(pstate, query, jstate);

	/* Safety check... */
	if (!pgss || !pgss_hash || !pgss_enabled(nesting_level))
		return;

	/*
	 * If it's EXECUTE, clear the queryId so that stats will accumulate for
	 * the underlying PREPARE.  ...
	 */
	if (query->utilityStmt)
	{
		if (pgss_track_utility && IsA(query->utilityStmt, ExecuteStmt))
		{
			query->queryId = INT64CONST(0);
			return;
		}
	}

	/*
	 * If query jumbling were able to identify any ignorable constants, we
	 * immediately create a hash table entry for the query, so that we can
	 * record the normalized form of the query string.  ...
	 */
	if (jstate && jstate->clocations_count > 0)
		pgss_store(pstate->p_sourcetext,
				   query->queryId,
				   query->stmt_location,
				   query->stmt_len,
				   PGSS_INVALID,
				   0, 0, NULL, NULL, NULL,
				   jstate, 0, 0,
				   PLAN_STMT_UNKNOWN);
}
```

三个要点：

- 第一行就转调 prev——礼仪的标准写法；
- `EXECUTE` 语句清零 queryId：让预备语句的执行统计归到 PREPARE 那条的指纹下，避免同一逻辑查询分裂成两条记录；
- 若 jumble 发现了可忽略常量，**提前**调用 `pgss_store` 建条目（kind = `PGSS_INVALID`，不累计数字）——这一步存在的唯一目的，是趁 `jstate` 还活着（它只在解析后这一刻可用），用 `generate_normalized_query()`（:2813）按常量位置把原文替换成 `$1`/`$2` 占位符，生成视图里那条 `SELECT * FROM t WHERE id = $1`。这种"只占坑不计数"的条目叫 **sticky entry**：它的 usage 权重被调高，避免在真正的执行统计到来之前就被 LRU 淘汰——这是"两阶段落账"（解析时存文本、执行完存数字）之间的保活机制。

### 15.2.6 第四步：落账——pgss_ExecutorEnd 与 pgss_store 全走读

`pgss_ExecutorEnd`（:1059）全文：

```c
static void
pgss_ExecutorEnd(QueryDesc *queryDesc)
{
	int64		queryId = queryDesc->plannedstmt->queryId;

	if (queryId != INT64CONST(0) && queryDesc->query_instr &&
		pgss_enabled(nesting_level))
	{
		pgss_store(queryDesc->sourceText,
				   queryId,
				   queryDesc->plannedstmt->stmt_location,
				   queryDesc->plannedstmt->stmt_len,
				   PGSS_EXEC,
				   INSTR_TIME_GET_MILLISEC(queryDesc->query_instr->total),
				   queryDesc->estate->es_total_processed,
				   &queryDesc->query_instr->bufusage,
				   &queryDesc->query_instr->walusage,
				   queryDesc->estate->es_jit ? &queryDesc->estate->es_jit->instr : NULL,
				   NULL,
				   queryDesc->estate->es_parallel_workers_to_launch,
				   queryDesc->estate->es_parallel_workers_launched,
				   queryDesc->plannedstmt->planOrigin);
	}

	if (prev_ExecutorEnd)
		prev_ExecutorEnd(queryDesc);
	else
		standard_ExecutorEnd(queryDesc);
}
```

为什么选 ExecutorEnd 而不是 ExecutorRun 之后？因为此刻 `QueryDesc` 还活着而一切已尘埃落定：总耗时、行数（`es_total_processed`）、buffer/WAL 用量、JIT 计时、并行 worker 实际启动数、计划来源（generic/custom plan）全部就绪，而 `standard_ExecutorEnd` 一执行这些就被销毁。**先落账、再转调销毁**——顺序不能反。

`pgss_store`（:1269）是全模块的汇聚点，其并发协议值得逐段走读：

```c
	query = CleanQuerytext(query, &query_location, &query_len);

	/* clear padding */
	memset(&key, 0, sizeof(pgssHashKey));

	key.userid = GetUserId();
	key.dbid = MyDatabaseId;
	key.queryid = queryId;
	key.toplevel = (nesting_level == 0);

	/* Lookup the hash table entry with shared lock. */
	LWLockAcquire(&pgss->lock.lock, LW_SHARED);

	entry = (pgssEntry *) hash_search(pgss_hash, &key, HASH_FIND, NULL);
```

- `memset` 清 key 的 padding 字节：因为哈希表用 `HASH_BLOBS`（对 key 整块内存做 tag_hash），padding 里的脏数据会让相同逻辑 key 散到不同桶——结构体注释（:139-141）专门警告了这一点；
- `toplevel` 进 key：同一条 SQL 顶层执行和在函数里执行分开统计（`track = all` 时两者都记）。

条目不存在时的**锁升级协议**（:1322-1382）：

```c
	if (!entry)
	{
		...
		if (jstate)
		{
			LWLockRelease(&pgss->lock.lock);
			norm_query = generate_normalized_query(jstate, query,
												   query_location,
												   &query_len);
			LWLockAcquire(&pgss->lock.lock, LW_SHARED);
		}

		/* Append new query text to file with only shared lock held */
		stored = qtext_store(norm_query ? norm_query : query, query_len,
							 &query_offset, &gc_count);
		...
		/* Need exclusive lock to make a new hashtable entry - promote */
		LWLockRelease(&pgss->lock.lock);
		LWLockAcquire(&pgss->lock.lock, LW_EXCLUSIVE);

		if (!stored || pgss->gc_count != gc_count)
			stored = qtext_store(norm_query ? norm_query : query, query_len,
								 &query_offset, NULL);
		...
		entry = entry_alloc(&key, query_offset, query_len, encoding,
							jstate != NULL);

		if (do_gc)
			gc_qtexts();
	}
```

三处精细的并发处理：

1. 规范化文本的生成放在**锁外**（先释放共享锁再做字符串加工，做完重新拿锁）——纯 CPU 工作不占锁；
2. 文本追加 `qtext_store`（:2231）只需**共享锁**就能并发执行：它内部用自旋锁原子推进 `pgss->extent`（文件写入偏移）来"预订"区间，然后各进程用 `pg_pwrite` 写各自互不重叠的区间——**用偏移预订把文件追加变成了无锁并发写**；`n_writers` 计数用于让 GC 探测"有人正在写"；
3. 共享锁升级到排他锁**不是原子的**（LWLock 不支持原地升级，必须先放再拿），中间窗口里可能有别人做了 GC 把刚写的文本搬走了——所以用 `gc_count` 版本号检测，发现变了就重写一次文本。条目创建本身的竞态（两个进程同时为同一 key 建条目）由 `entry_alloc`（:2081，内部 `HASH_ENTER` 幂等）兜底。

条目就绪后的累加（:1385-1494）在**条目自旋锁**内完成：calls、total_time、min/max，以及 mean/方差用 Welford 在线算法（:1410-1419，一次遍历同时维护均值与方差、无精度灾难），然后是十几行 buffer/WAL/JIT/并行/plan-cache 计数器的搬运。自旋锁内只有纯算术——这是自旋锁的使用纪律（不做任何可能阻塞的事）。

`pgss_ProcessUtility`（:1092）对 DDL 走同样的"计时 + 落账"逻辑（略，结构与 pgss_planner 相同：包住 `standard_ProcessUtility`，测时后 `pgss_store`）。

一张图总结 pgss 的寄生结构：

```
post_parse_analyze  →  planner        →  ExecutorStart     →  Run/Finish   →  ExecutorEnd
建 sticky 条目          记 planning 耗时   申请 instrumentation   维护嵌套深度     落账（EXEC）
存规范化文本            维护嵌套深度
        ProcessUtility 覆盖 DDL │ ShmemCallbacks 提供存储 │ qtext 文件存文本
```

TimescaleDB、Citus 的 hook 用法只是把"记账"换成了"改写计划"。

### 小结

pgss 用六个挂点覆盖 SQL 全生命周期：解析后借 `JumbleState` 生成规范化文本（sticky entry 保活），执行毕在 ExecutorEnd 一次性落账。存储上"定长计数进共享内存哈希、变长文本进外部文件"，锁上"LWLock 管结构、自旋锁管计数、偏移预订实现并发追加"。queryId 从"文本相同"到"扩展内 jumble"再到"内核 jumble"的三步演化（7466eeac61 → 7313cc0163 → 5fd9dfa5f5）是"扩展孵化、内核收编"的标准路径。

### 15.3.3 所以然⑤：magic block 到底防什么，为什么逐字段比对

先看 magic block 是什么（`src/include/fmgr.h:468-497`）：

```c
/* Definition of the values we check to verify ABI compatibility */
typedef struct
{
	int			version;		/* PostgreSQL major version */
	int			funcmaxargs;	/* FUNC_MAX_ARGS */
	int			indexmaxkeys;	/* INDEX_MAX_KEYS */
	int			namedatalen;	/* NAMEDATALEN */
	int			float8byval;	/* FLOAT8PASSBYVAL (now vestigial) */
	char		abi_extra[32];	/* see pg_config_manual.h */
} Pg_abi_values;
```

`PG_MODULE_MAGIC` 宏（fmgr.h:522-530）展开成一个导出函数 `Pg_magic_func()`，返回**编译该 .so 时**这些常量的值。为什么是函数不是数据？注释说得明白（fmgr.h:514-516）："dlsym in the backend is only guaranteed to work on functions, not data"——某些平台 dlsym 取数据符号不可靠。

**没有它会怎样？** 推演一个具体事故：你用 PG 16 的头文件编译了扩展，装进了 PG 18 的 `$libdir`。dlopen 成功（ELF 层面没有版本概念），`_PG_init` 成功（它只调用一些跨版本存在的函数），`CREATE FUNCTION ... LANGUAGE C` 成功。然后第一次调用函数：PG 18 传给它的 `FunctionCallInfo` 结构布局与 PG 16 头文件里的定义不同（字段增删导致偏移移位），扩展代码按旧偏移读参数——读到的是**错位的垃圾**。结果不是干净的报错，而是随机的：可能立刻 SIGSEGV，可能返回错误数据，最坏的情况是拿垃圾指针去写共享缓冲区，**把数据页写坏**。ABI 不匹配的本质是"双方对同一块内存的布局有不同理解"，它不产生任何可诊断的错误信号，只产生 undefined behavior。magic block 把这场灾难变成一条干净的错误：

```
ERROR: incompatible library "...": version mismatch
DETAIL: Server is version 18, library is version 16.
```

**为什么逐字段而不只比主版本号？** 因为 ABI 不止由版本决定。`NAMEDATALEN`（默认 64）是编译期可改的：有人为超长表名把它调成 128 重编内核。此时同版本号的官方扩展二进制装进来，所有含 `NameData` 字段的结构（几乎全部系统目录结构）偏移全错——版本号完全相同，灾难完全相同。`FUNC_MAX_ARGS`、`INDEX_MAX_KEYS` 同理（决定若干数组的编译期大小）。`abi_extra`（默认 `"PostgreSQL"`）则留给 fork：EDB/Greenplum 这类改了内核结构的分支把自己的名字写进去，防止"版本号相同但根本是另一个产品"的 .so 混进来。

比对的实现是 `memcmp` 整个 `Pg_abi_values`（dfmgr.c:265-266），加一道 `len != sizeof(Pg_magic_struct)` 防结构本身变过；而**报错**函数 `incompatible_module_error`（dfmgr.c:316）却是逐字段的，而且有讲究的分层：先比 version（版本都不对，其他字段的语义都不可信，只报版本）；再比 `abi_extra`（产品都不对，报字段值没有意义）；最后才逐个报 `FUNC_MAX_ARGS`/`INDEX_MAX_KEYS`/`NAMEDATALEN`/`FLOAT8PASSBYVAL` 中不一致的项（:366-405）。**校验用 memcmp 求快求全，报错逐字段求可诊断**——两个代码路径服务两个目标。

历史（设计历史 D）：magic block 于 2006 年进入 8.2（commit e60cb3a35c，2006-05-30，"Code review for magic-block patch"，Tom Lane）——在此之前"装错版本的 .so"就是上面推演的随机灾难，只能靠 DBA 自觉。2025 年又加了 `PG_MODULE_MAGIC_EXT`（commit 9324c8c580，fmgr.h:540）：允许扩展在 magic block 里附带自己的 name/version 字符串，供 `pg_get_loaded_modules()` 观测——解决"服务器里到底加载了哪些扩展的哪个版本"长期无法自省的问题。

### 15.3.4 所以然②：为什么 shared_preload_libraries 才能申请共享内存

**两种加载时机的能力差异**：

| | `shared_preload_libraries` | `LOAD` / 首次调用 C 函数隐式加载 |
|---|---|---|
| 执行者与时机 | postmaster 启动时（miscinit.c:1855） | 单个 backend 会话内（dfmgr.c:149） |
| 影响范围 | 所有后续 backend（fork 继承） | 仅当前会话 |
| 申请主共享内存 | 可以（此时尚未定型） | 传统上不可以；20devel 起可用 `SHMEM_CALLBACKS_ALLOW_AFTER_STARTUP`（shmem.h:167），但无容量保证 |
| 注册命名 LWLock tranche | 可以（`RequestNamedLWLockTranche()`，lwlock.c:202 注释——仅 postmaster 内有效） | 不可以（用运行时 `LWLockNewTrancheId()` 替代） |
| 注册静态 bgworker | 可以（`RegisterBackgroundWorker()`，bgworker.c:962） | 只能动态启动（`RegisterDynamicBackgroundWorker()`，bgworker.c:1068） |
| 挂 planner/executor hook | 可以，全局生效 | 也可以，但只影响本会话 |
| 变更生效 | 改 GUC 后须重启 | 立即 |

这条能力分界线不是策略选择，而是**架构约束的投影**。PG 的主共享内存是一个在 postmaster 启动时一次性创建的段：启动流程先跑所有子系统（以及 preload 库）的 request 回调收集需求，`CalculateShmemSize` 汇总求和，然后一把 `mmap`/`shmget` 出这么大的段，此后**段不能长大**。子进程通过 fork 继承同一映射地址，所以段里可以放裸指针。这个模型下，"申请共享内存"只在"汇总求和之前"有意义——也就是 preload 窗口。错过窗口的库面对的是一个已经定容、已经分配殆尽的段。

pg_stat_statements、TimescaleDB、Citus 都**必须**写进 `shared_preload_libraries`，因为它们要么要共享内存，要么要全局生效的 hook；而 PostGIS 这类"纯函数 + 类型"扩展不需要 preload——它的 .so 在你第一次调用其函数时按 `pg_proc.probin` 自动加载。

**20devel 的放宽（如实描述）**：`RegisterShmemCallbacks()`（shmem.c:873-891）现在允许带 `SHMEM_CALLBACKS_ALLOW_AFTER_STARTUP` 标志在启动后注册：

```c
	if (shmem_request_state == SRS_DONE && IsUnderPostmaster)
	{
		if ((callbacks->flags & SHMEM_CALLBACKS_ALLOW_AFTER_STARTUP) == 0)
			elog(ERROR, "cannot request shared memory at this time");

		CallShmemCallbacksAfterStartup(callbacks);
	}
```

`CallShmemCallbacksAfterStartup`（shmem.c:897）立刻执行 request 回调、持 `ShmemIndexLock` 检查这批区域是否已被别的 backend 初始化过（要求"全都初始化过"或"全都没有"，混合状态报错），然后相应地走 attach 或 init。但注意函数头注释的限定（shmem.c:860-862）："after startup there's no guarantee that there's enough shared memory available"——启动后的分配来自主段的剩余空间（含预留富余量），**可能失败**。所以这只是给"会话级、小块、能容忍失败"的场景开的口子（20devel 内核自身的子系统都不用它），"大块共享内存必须 preload"的基本格局没有变。真正的运行时大块共享内存另有其路：DSM（动态共享内存段，9.4 引入，并行查询的基础设施）——但 DSM 段没有稳定映射地址、不能存裸指针，编程模型完全不同。

### 小结

加载 = 查重（路径 + inode）→ dlopen(RTLD_NOW|RTLD_GLOBAL) → magic 校验 → `_PG_init` → 入链不卸载。magic block 把"ABI 不匹配的 undefined behavior"变成干净报错，逐字段记录是因为 ABI 由版本号之外的多个编译期参数共同决定。preload 与会话加载的分水岭是"postmaster 级资源"：主共享内存启动定容的架构决定了申请窗口，20devel 的 after-startup 注册只是有限放宽（无容量保证），不改变基本格局。

## 15.4 扩展框架：CREATE EXTENSION 主线走读

`.so` 只解决了"代码进来"，`CREATE EXTENSION` 解决的是"SQL 对象成组管理"。实现在 `src/backend/commands/extension.c`。先看一个真实的控制文件（`contrib/postgres_fdw/postgres_fdw.control`，全文五行）：

```
# postgres_fdw extension
comment = 'foreign-data wrapper for remote PostgreSQL servers'
default_version = '1.3'
module_pathname = '$libdir/postgres_fdw'
relocatable = true
```

一个扩展 = 三种文件：控制文件 `extname.control`（元数据）、安装脚本 `extname--1.0.sql`（对象）、升级脚本 `extname--1.0--1.1.sql`（增量迁移）。翻开 `contrib/pg_stat_statements/Makefile` 就能看到完整链条：`DATA = pg_stat_statements--1.4.sql pg_stat_statements--1.12--1.13.sql ... pg_stat_statements--1.0--1.1.sql`——基线脚本只有一个（1.4），更老版本的安装靠"装 1.4 再沿升级链走"，每个大版本给视图加列就加一个增量脚本。

### 15.4.1 控制文件解析

`parse_extension_control_file()`（extension.c:693）逐行解析 `name = value` 对，填充 `ExtensionControlFile` 结构（extension.c:86-106）：

```c
typedef struct ExtensionControlFile
{
	char	   *name;			/* name of the extension */
	...
	char	   *default_version;	/* default install target version, if any */
	char	   *module_pathname;	/* string to substitute for
									 * MODULE_PATHNAME */
	char	   *comment;		/* comment, if any */
	char	   *schema;			/* target schema (allowed if !relocatable) */
	bool		relocatable;	/* is ALTER EXTENSION SET SCHEMA supported? */
	bool		superuser;		/* must be superuser to install? */
	bool		trusted;		/* allow becoming superuser on the fly? */
	int			encoding;		/* encoding of the script file, or -1 */
	List	   *requires;		/* names of prerequisite extensions */
	...
} ExtensionControlFile;
```

关键字段：`relocatable`（能否 `ALTER EXTENSION SET SCHEMA`）、`trusted`（允许非超级用户安装，脚本临时切换为超级用户执行——PG 13 起 PL/pgSQL 等靠它做到"普通用户可装"）、`requires`（依赖的其他扩展）。还支持版本特定的辅助控制文件 `extname--1.1.control` 覆盖个别字段（`read_extension_aux_control_file`）。

### 15.4.2 CreateExtension 主线（extension.c:2149 → 1836）

`CreateExtension`（:2149）解析 `SCHEMA`/`VERSION`/`CASCADE` 选项、查重后，进入真正的主线 `CreateExtensionInternal`（:1836）。主线七步：

1. **读控制文件**（:1862）：`pcontrol = read_extension_control_file(extensionName);` 确定目标版本（未指定则用 `default_version`，:1867-1875）；
2. **规划安装路径**（:1884-1916）：若 `extname--<ver>.sql` 直接存在则一步到位；否则构建版本图找"从某个可直接安装的版本升上来"的路径：

```c
		/* Extract the version update graph from the script directory */
		evi_list = get_ext_ver_list(pcontrol);

		/* Identify the target version */
		evi_target = get_ext_ver_info(versionName, &evi_list);

		/* Identify best path to reach target */
		evi_start = find_install_path(evi_list, evi_target,
									  &updateVersions);
```

3. **确定 schema**（:1926-1997）：非 relocatable 且控制文件给了 `schema` 的，强制用之（必要时自动 `CREATE SCHEMA`）；否则用 `search_path` 第一个显式条目；
4. **递归安装依赖**（:2019-2036）：遍历 `control->requires`，对每个前置扩展调 `get_required_extension`（:2077）——未安装时若带 `CASCADE` 则递归 `CreateExtensionInternal`（带环检测，:2101-2110），否则报错提示 `Use CREATE EXTENSION ... CASCADE`；
5. **登记目录**（:2041-2047）：`InsertExtensionTuple` 向 `pg_extension` 插入一行（名称、owner、schema、版本），并对 `requires` 里的每个扩展记一条 `DEPENDENCY_NORMAL` 依赖；
6. **执行安装脚本**（:2058）：`execute_extension_script(...)`——见下节；
7. **补升级链**（:2067）：若第 2 步规划了多段路径，`ApplyExtensionUpdates` 把剩余段当作一串 `ALTER EXTENSION UPDATE` 逐段执行。

### 15.4.3 execute_extension_script：脚本执行与依赖记录（:1248）

这个函数是"对象归属扩展"魔法发生的地方。走读要点：

**权限与环境准备**（:1268-1364）：`trusted` 扩展 + 非超级用户时切换到 bootstrap superuser（:1302-1307，`SetUserIdAndSecContext`，事务失败自动还原）；把 `client_min_messages`/`log_min_messages` 临时抬到 WARNING、关 `check_function_bodies`（用 GUC 的 `GUC_ACTION_SAVE` 机制，脚本结束或出错自动还原）；把 `search_path` 设为"目标 schema, 依赖扩展的 schema..., pg_temp"（:1350-1364）——脚本里不带限定名的 `CREATE FUNCTION` 于是落进正确的 schema，结尾的 `pg_temp` 防止临时对象劫持解析。

**核心机制**（:1366-1498）：

```c
	/*
	 * Set creating_extension and related variables so that
	 * recordDependencyOnCurrentExtension and other functions do the right
	 * things.  On failure, ensure we reset these variables.
	 */
	creating_extension = true;
	CurrentExtensionObject = extensionOid;
	PG_TRY();
	{
		char	   *c_sql = read_extension_script_file(control, filename);
		...
		execute_sql_string(c_sql, filename);
	}
	PG_FINALLY();
	{
		creating_extension = false;
		CurrentExtensionObject = InvalidOid;
	}
	PG_END_TRY();
```

执行脚本前置起两个全局标志（extension.c:80-81 定义）。此后脚本里每一条 DDL 在登记依赖时都会路过 `recordDependencyOnCurrentExtension()`（`catalog/pg_depend.c:206`）：

```c
	if (creating_extension)
	{
		ObjectAddress extension;
		...
		recordDependencyOn(object, &extension, DEPENDENCY_EXTENSION);
```

发现 `creating_extension` 为真，就给新对象追加一条指向扩展的 `DEPENDENCY_EXTENSION` 依赖。**扩展成员资格不是显式登记表，而是"创建时环境"的副产品**——这个设计让扩展脚本就是普通 SQL，任何 DDL 天然可用，无须为"扩展里能建什么对象"逐一开洞。这就是 `DROP EXTENSION ... CASCADE` 能级联删干净、pg_dump 对成员对象只输出一行 `CREATE EXTENSION` 的机制根源。

脚本执行前还有几处文本替换（:1396-1489）：`\echo` 行清空（防止有人用 psql 直接跑脚本）、`@extowner@`/`@extschema@`/`@extschema:extname@` 替换（都过 `quote_identifier`，且拒绝含引号字符的名字——防注入）、`MODULE_PATHNAME` 替换成控制文件里的 .so 路径（:1479-1486）——这就是脚本里 `AS 'MODULE_PATHNAME'` 的来历。

### 15.4.4 版本升级：有向图上的 Dijkstra

`get_ext_ver_list()`（extension.c:1582）扫描脚本目录，把每个 `A--B.sql` 变成版本图里的一条边 A→B。`find_update_path()`（:1688）在图上做标准 Dijkstra：

```c
	evi_start->distance = 0;

	while ((evi = get_nearest_unprocessed_vertex(evi_list)) != NULL)
	{
		if (evi->distance == INT_MAX)
			break;				/* all remaining vertices are unreachable */
		evi->distance_known = true;
		if (evi == evi_target)
			break;				/* found shortest path to target */
		foreach(lc, evi->reachable)
		{
			ExtensionVersionInfo *evi2 = (ExtensionVersionInfo *) lfirst(lc);
			int			newdist;
			...
			newdist = evi->distance + 1;
			if (newdist < evi2->distance)
			{
				evi2->distance = newdist;
				evi2->previous = evi;
			}
			else if (newdist == evi2->distance &&
					 evi2->previous != NULL &&
					 strcmp(evi->name, evi2->previous->name) < 0)
			{
				/* Break ties in favor of the version name that comes first ... */
				evi2->previous = evi;
			}
		}
	}
```

细节见品味：等距路径按版本名 strcmp 打破平局（:1737-1750）——不是为了语义，纯粹为了**确定性**（否则升级路径取决于 readdir 返回目录项的顺序，不同机器可能走不同脚本链）。从 1.0 升到 2.0 不需要 `1.0--2.0` 脚本存在，只要 `1.0--1.1`、`1.1--2.0` 能连通即可；`ALTER EXTENSION UPDATE` 沿路径逐段执行。对比 MySQL 世界：模式迁移完全是 Flyway/Liquibase 这类外部工具的事，PG 把"带版本的对象包"做进了内核。

### 15.4.5 所以然③：extension 框架到底解决什么——回到 PG 9.1 之前

extension 于 9.1 引入（commit d9572c4e3b，2011-02-08，"Core support for 'extensions', which are packages of SQL objects"，Tom Lane；多版本升级紧随其后，commit 1214749901）。推演一下没有它的世界（8.4/9.0 时代装 PostGIS 的真实体验）：

1. **安装**：`psql -f postgis.sql`——一个几千行的裸 SQL 脚本，创建上千个函数/类型/操作符。装到哪个 schema 取决于当时的 search_path，装歪了没有任何撤销手段；
2. **删除**：没有"这一千个对象属于 PostGIS"的记录，卸载靠作者提供的 `uninstall_postgis.sql`——一个手工维护的 DROP 清单，漏一个就留垃圾，多删一个就伤及无辜；
3. **dump/restore——最痛的一刀**：`pg_dump` 分不清"用户的表"和"PostGIS 的函数"，把上千个 `CREATE FUNCTION ... AS '$libdir/postgis...'` 全部录进 dump 文件。恢复到新机器时：如果新机器装的是新版本 PostGIS 的 .so，dump 里的旧版函数定义可能引用已改名的 C 符号——restore 一半失败一半成功，库处于半死状态。**dump 文件把"应该由目标机器的软件版本决定的东西"固化了**；
4. **升级**：跑作者提供的 `postgis_upgrade.sql`，它必须自己处理"从任意旧版本升上来"的所有分支——每个作者重新发明一遍版本管理。

extension 框架对这四刀的回答恰好对应它的四个机制：脚本执行环境受控（search_path 强制）解决 1；`pg_depend` 的 `DEPENDENCY_EXTENSION` 解决 2（`DROP EXTENSION` 精确级联）；pg_dump 看到成员对象只输出 `CREATE EXTENSION postgis`——恢复时由**目标机器**上安装的脚本重建对象，版本自动匹配当地 .so，解决 3；版本图 + 增量脚本解决 4。一句话：**extension 把"SQL 对象的集合"提升成了有身份、有版本、有依赖的一等公民**。

### 15.4.6 what-if 推演：.so 升级了，SQL 版本没升

这是生产中最常见的扩展事故形态。设扩展 foo 1.0 装着，你把包管理器里的 foo 升到 2.0——`$libdir/foo.so` 被**原地覆盖**（文件系统层面无版本），但数据库里 `pg_extension.extversion` 还是 1.0，`ALTER EXTENSION foo UPDATE` 没人跑。现在：

- **SQL 层是 1.0 的函数集**：2.0 新增的 SQL 函数不存在（还好，只是没有新功能）；
- **C 层是 2.0 的符号集**：1.0 的 SQL 函数定义引用的 C 符号如果在 2.0 里被改名/删除——调用时 `dlsym` 失败，报 `could not find function "foo_bar" in file ...`（dfmgr.c:130-134）。这还是幸运的情形；
- **不幸的情形**：符号还在但语义变了。比如 2.0 改了某聚合的转移函数的内部状态结构，而 1.0 的 SQL 定义还把旧的 serialfunc/deserialfunc 组合在一起——运行不报错，结果悄悄错误。更隐蔽的是索引类扩展：opclass 支持函数的排序语义变了，**已建索引的序与新函数不一致**，查询开始漏行；
- magic block 帮不上忙——它只校验内核 ABI，不校验"扩展 SQL 版本 ↔ 扩展 .so 版本"的配套关系。

这正是 `PG_MODULE_MAGIC_EXT`（9324c8c580）想缓解的观测盲区：至少能查到加载的 .so 自称什么版本，与 `pg_extension.extversion` 对比。而根本的防御是扩展作者侧的纪律：C 函数入口保持向后兼容（老符号保留为兼容 shim），或在 `_PG_init`/函数入口校验"SQL 层版本"并给出明确报错（pgss 的做法：视图函数有多个版本化入口 `pg_stat_statements_1_9`、`_1_10`……每个 SQL 版本绑定各自的 C 入口，老入口永不删除）。**运维铁律：升 .so 与 `ALTER EXTENSION UPDATE` 必须成对出现在同一个变更窗口里。**

### 小结

扩展 = 控制文件（元数据）+ 版本化 SQL 脚本（对象）+ 可选 .so（代码）。`CreateExtensionInternal` 主线：控制文件 → 版本路径规划 → schema 确定 → 依赖递归 → pg_extension 登记 → 脚本执行 → 升级链补齐。"对象归属"靠 `creating_extension` 全局标志让每条 DDL 自动挂上 `DEPENDENCY_EXTENSION` 依赖；升级靠版本图最短路。它把 9.1 前 dump/restore/卸载/升级的四类混乱一次收编；但"SQL 版本 ↔ .so 版本"的配套仍是运维责任。

## 15.5 FDW：把一切伪装成表

### 15.5.1 接口：FdwRoutine 回调表

FDW 的内核侧在 `src/backend/foreign/`，接口定义在 `src/include/foreign/fdwapi.h`。优化器遇到外部表时，通过 `GetFdwRoutine()`（`foreign/foreign.c:357`）调用该 FDW 的 handler 函数，拿回一张回调表：


`postgres_fdw_handler`（postgres_fdw.c:648）就是填表返回：

```c
Datum

{
	FdwRoutine *routine = makeNode(FdwRoutine);

	/* Functions for scanning foreign tables */
	routine->GetForeignRelSize = postgresGetForeignRelSize;

	routine->IterateForeignScan = postgresIterateForeignScan;
```

`FdwRoutine`（fdwapi.h:208 起）的核心回调按调用顺序（扫描五件套是唯一强制实现的）：

- **`GetForeignRelSize`**（fdwapi.h:27）：计划期第一站，估算行数/宽度——替远端做 ANALYZE 的活；
- **`GetForeignPaths`**（fdwapi.h:31）：生成 `ForeignPath` 并 `add_path` 进候选集，带 cost 参与正常的代价竞争——**FDW 不绕过优化器，而是参赛**；
- **`GetForeignPlan`**（fdwapi.h:35）：被选中后把 Path 转成 `ForeignScan` 计划节点，决定哪些 qual 下推、哪些本地复查，私有信息（拼好的远端 SQL）塞进 `fdw_private`；
- **`BeginForeignScan` / `IterateForeignScan` / `ReScanForeignScan` / `EndForeignScan`**（fdwapi.h:43/46/51/53）：执行期四件套。`IterateForeignScan` 是 Volcano 接口——每次返回一个元组槽，空槽表示结束。**FDW 的本质就是把任何东西适配成这个迭代器**；
- 写路径：`ExecForeignInsert`（fdwapi.h:84）等一族；批量映射：`ImportForeignSchema`（fdwapi.h:164）；
- 进阶下推：`GetForeignJoinPaths`（fdwapi.h:55）、`GetForeignUpperPaths`（fdwapi.h:62）。

### 15.5.4 执行期走读：Begin / Iterate / fetch_more_data

**`postgresBeginForeignScan`**（:1593）：EXPLAIN（无 ANALYZE）直接返回（:1608-1609，别为了打印计划真去连远端）；然后建执行态 `PgFdwScanState`：确定以哪个用户身份联远端（user mapping，:1621-1630）、`GetConnection()` 拿连接（:1636，连接按 user mapping 缓存复用，见 15.5.7）、从 `fdw_private` 取出计划期拼好的 SQL 与 fetch_size（:1643-1648）、准备批量元组的内存上下文与类型转换元数据（attinmeta，文本 → Datum 的入口）。注意此刻**尚未**向远端发任何查询——惰性到第一次要数据。

**`postgresIterateForeignScan`**（:1698）全文：

```c
static TupleTableSlot *
postgresIterateForeignScan(ForeignScanState *node)
{
	PgFdwScanState *fsstate = (PgFdwScanState *) node->fdw_state;
	TupleTableSlot *slot = node->ss.ss_ScanTupleSlot;

	/*
	 * In sync mode, if this is the first call after Begin or ReScan, we need
	 * to create the cursor on the remote side.  ...
	 */
	if (!fsstate->cursor_exists)
		create_cursor(node);

	/*
	 * Get some more tuples, if we've run out.
	 */
	if (fsstate->next_tuple >= fsstate->num_tuples)
	{
		/* In async mode, just clear tuple slot. */
		if (fsstate->async_capable)
			return ExecClearTuple(slot);
		/* No point in another fetch if we already detected EOF, though. */
		if (!fsstate->eof_reached)
			fetch_more_data(node);
		/* If we didn't get any tuples, must be end of data. */
		if (fsstate->next_tuple >= fsstate->num_tuples)
			return ExecClearTuple(slot);
	}

	/*
	 * Return the next tuple.
	 */
	ExecStoreHeapTuple(fsstate->tuples[fsstate->next_tuple++],
					   slot,
					   false);

	return slot;
}
```

首次调用时 `create_cursor()`（:3819）向远端发 `DECLARE c1 CURSOR FOR SELECT ...`（参数化路径的参数值此刻本地求值后代入）；本地缓冲耗尽时 `fetch_more_data()`（:3892）：

```c
		char		sql[64];

		/* This is a regular synchronous fetch. */
		snprintf(sql, sizeof(sql), "FETCH %d FROM c%u",
				 fsstate->fetch_size, fsstate->cursor_number);

		res = pgfdw_exec_query(conn, sql, fsstate->conn_state);
```

每次 `FETCH 100`（fetch_size 默认 100）行进本地数组，`make_tuple_from_result_row` 把文本结果转成 HeapTuple，逐行喂给执行器。**用游标 + 批量 FETCH 把"逐 tuple 的 Volcano 接口"和"网络批量传输"解耦**，与 JDBC fetchSize 同思路；用游标而非直接 SELECT 全量，是为了不把远端结果集一次性物化到本地内存。

两个配套工程细节：连接管理在 `connection.c`——同一 user mapping 的连接跨语句缓存复用，远端事务与本地事务对齐（详见 15.5.7）；PG 14 起的**异步执行**回调（`postgresForeignAsyncRequest`，postgres_fdw.c:8081，commit 27e1f14563）使 `Append` 下多个外部分区能并发等待各自远端的结果（上面 Iterate 代码里 async 分支返回空槽就是"数据没到，先让别的分区跑"），分区表 + postgres_fdw 组成的简易 sharding 因此有了并行取数的味道。

**JOIN/聚合下推**：`postgresGetForeignJoinPaths`（:7143，9.6，commit e4106b2528）在 join 两侧同 server、join 条件整体可下推时，为 joinrel 生成"整个 join 在远端执行"的 ForeignPath；`postgresGetForeignUpperPaths`（:7528，PG 10 聚合下推，commit 7012b132d0）同理处理 GROUP BY/ORDER BY/LIMIT。下推决策始终复用主优化器的代价框架——远端 join 路径和"拉回来本地 join"的路径同台比价，输了照样不选。

### 15.5.5 下推判定走读：foreign_expr_walker 的白名单主义

`is_foreign_expr()`（deparse.c:244）是下推的守门人，三道闸：

```c
	if (!foreign_expr_walker((Node *) expr, &glob_cxt, &loc_cxt, NULL))
		return false;

	/*
	 * If the expression has a valid collation that does not arise from a
	 * foreign var, the expression can not be sent over.
	 */
	if (loc_cxt.state == FDW_COLLATE_UNSAFE)
		return false;

	/*
	 * An expression which includes any mutable functions can't be sent over
	 * because its result is not stable.  ...
	 */
	if (contain_mutable_functions((Node *) expr))
		return false;
```

`foreign_expr_walker()`（deparse.c:312）递归遍历表达式树，节点类型本身就是白名单——switch 里没有列出的节点类型直接返回 false（新节点类型默认不可下推，必须有人论证安全后才加分支）。对列出的类型再查两件事：

**第一件：对象可移植性**。以操作符为例（deparse.c:604 起）：

```c
		case T_OpExpr:
		case T_DistinctExpr:	/* struct-equivalent to OpExpr */
			{
				OpExpr	   *oe = (OpExpr *) node;

				/*
				 * Similarly, only shippable operators can be sent to remote.
				 * (If the operator is shippable, we assume its underlying
				 * function is too.)
				 */
				if (!is_shippable(oe->opno, OperatorRelationId, fpinfo))
					return false;
```

`is_shippable()`（shippable.c:163）= `is_builtin()`（shippable.c:153，OID 小于 `FirstGenbkiObjectId`，即编译期分配 OID 的内核内建对象）或属于 server 选项 `extensions` 里显式声明的扩展。**为什么白名单而不是黑名单？** 因为下推的正确性前提是"同一个表达式在两端语义一致"，而本地 OID 与远端 OID 没有任何对应关系——deparse 只能按**名字**拼 SQL，名字相同不代表语义相同。内建对象两端语义一致由 PG 版本兼容性保证；扩展对象则只有管理员声明"两边都装了同版本 postgis"（`ALTER SERVER ... OPTIONS (extensions 'postgis')`）后才可信。其余一律拒推——宁可慢，不可错。

**第二件：collation 溯源**。walker 里每个节点都在维护一个 `(collation, state)` 状态机（`FDWCollateState`，deparse.c:79-87）：

```c
	FDW_COLLATE_NONE,			/* expression is of a noncollatable type, or
								 * it has default collation that is not
								 * traceable to a foreign Var */
	FDW_COLLATE_SAFE,			/* collation derives from a foreign Var */
	FDW_COLLATE_UNSAFE,			/* collation is non-default and derives from
								 * something other than a foreign Var */
```

规则（Var 分支，deparse.c:336-388）：外部表自己列的 collation 标记 SAFE（远端列本来就是这个 collation，比较行为两端一致）；**其他任何来源**的非默认 collation——常量上的 `COLLATE` 子句（Const 分支 :468-479）、本地表的列、参数——标记 UNSAFE。函数/操作符分支（:557-645）还要检查 `inputcollid` 必须恰好继承自 SAFE 来源，否则整棵树拒推。

### 15.5.6 所以然④：三类不可下推的"结果错误"反例

判定规则的每一条背后都有一个"下推了结果就错"的具体场景。给三个反例：

**反例 1（collation）**：本地外部表列 `name text`（映射远端同名列），查询：

```sql
SELECT * FROM ft WHERE name < 'b' COLLATE "C";
```

用户显式要求按 "C"（字节序）比较：`'B'` < `'b'`，所以名字为 `Boston` 的行**应该**入选。若把 `name < 'b'` 直接拼给远端，远端按其列的实际 collation（比如 `en_US.utf8`，字典序不区分大小写地插排）比较——`'Boston'` 排在 `'b'` 之后，该行**丢失**。不报错、不告警，安静地少一批行。这就是 `COLLATE` 折叠进常量后 `constcollid` 非默认 → `FDW_COLLATE_UNSAFE` → 拒推的原因。更阴险的变体不需要用户写 COLLATE：两台服务器 glibc 版本不同，同名 locale 的排序规则本身有差异——这层风险 walker 管不了（它假设"远端同名列的 collation 行为 = 本地声明"），所以 postgres_fdw 文档要求两端 collation 环境一致，而 walker 至少保证"本地引入的 collation 语义绝不外泄"。

**反例 2（易变/不稳定函数）**：

```sql
SELECT * FROM ft WHERE created_at > now() - interval '5 min';
```

`now()` 是 stable：本地事务内恒定，但**远端的 now() 是远端的时钟**。两机时钟差 30 秒，下推后过滤窗口整体平移 30 秒——结果集与语义（"以本地事务时间为基准"）不符。volatile 函数（`random()`）更明显：`random() < 0.5` 本意是本地对每行掷币，下推后变成远端掷币——采样源都换了。所以 `contain_mutable_functions` 一刀切拒推（deparse.c:281-288 注释以 now() 的 clock offset 为例）。注意补偿机制：这类表达式虽不能进远端 SQL **文本**，但可以本地求值后作为**参数**发送（参数化路径的 `fdw_exprs` 机制）——`created_at > $1`，$1 由本地算出，语义就对了。

**反例 3（本地自定义操作符）**：你在本地定义了 `<->`（编辑距离小于 3 视为"近似等于"），远端没装；或者更糟，远端**恰好也有**一个 `<->` 但是 pgvector 的向量距离。若按名字下推 `name <-> 'foo'`：前者远端报 `operator does not exist`（还算幸运——至少是报错）；后者远端用**语义完全不同的同名算子**算出一个结果返回——不报错，结果是另一道题的答案。`is_shippable` 的白名单（内建 + 显式声明的扩展）掐死的正是"名字巧合"这条路。

三个反例共同指向 FDW 下推的第一性原理：**下推 = 断言"该表达式在远端求值与在本地求值结果恒等"**。collation、时钟、对象语义是三个最容易破坏这条恒等式的自由变量，walker 的全部复杂度都在保守地钉死它们。

### 15.5.7 what-if 推演：远端事务与本地事务的原子性缺口

postgres_fdw 把远端事务对齐到本地事务：首次触碰某连接时发 `START TRANSACTION ISOLATION LEVEL REPEATABLE READ`（保证一条本地查询多次取数看到一致快照），本地提交时在 `pgfdw_xact_callback`（connection.c:1173）的 **PRE_COMMIT** 阶段驱动远端提交：

```c
				case XACT_EVENT_PARALLEL_PRE_COMMIT:
				case XACT_EVENT_PRE_COMMIT:
					...
					/* Commit all remote transactions during pre-commit */
					entry->changing_xact_state = true;
					if (entry->parallel_commit)
					{
						do_sql_command_begin(entry->conn, "COMMIT TRANSACTION");
						pending_entries = lappend(pending_entries, entry);
						continue;
					}
					do_sql_command(entry->conn, "COMMIT TRANSACTION");
```

推演事故：一个本地事务写了**两个** server 的外部表再 COMMIT。流程是：远端 A `COMMIT` → 远端 B `COMMIT` → 本地 COMMIT。在"A 已提交、B 提交失败（网络断/远端崩溃）"的窗口：`do_sql_command` 报错 → 本地事务转入 ABORT——**但 A 上的修改已经永久生效，撤不回来了**。最终状态：A 提交、B 回滚、本地回滚。同理"两个远端都提交成功、本地随后提交失败"（比如本地 WAL 写盘失败）也会裂开：远端全提交、本地回滚。

这就是**没有两阶段提交协调者时的原子性缺口**：postgres_fdw 的提交协议本质是"按顺序逐个单边提交，祈祷都成功"。PRE_COMMIT 时机的选择已经是能做到的最优——把远端提交放在本地提交**之前**，保证"本地已提交但远端没提交"这种更难收拾的状态不出现（本地提交后连报错的机会都没有了）；但它消除不了部分提交。内核明确知道这个缺口，所以干脆禁止把这种事务变成 prepared transaction（connection.c:1260-1262）：

```c
					ereport(ERROR,
							(errcode(ERRCODE_FEATURE_NOT_SUPPORTED),
							 errmsg("cannot PREPARE a transaction that has operated on postgres_fdw foreign tables")));
```

——因为 PREPARE 承诺"之后 COMMIT PREPARED 必定成功"，而远端事务无法陪着一起进入 prepared 状态。完整解法是把远端也做 2PC（远端 `PREPARE TRANSACTION`，本地提交后再逐个 `COMMIT PREPARED`，配全局事务管理器做崩溃后 resolve）——社区多年讨论未合入内核；Citus 等分布式扩展自带了这套协调逻辑。使用 postgres_fdw 的正确姿势因此是：**跨 server 写事务要么避免，要么业务层自己幂等/对账**。`parallel_commit` 选项（多连接并发发 COMMIT，:1216-1221）只是缩短提交窗口，不改变语义。

### 15.5.8 FDW 设计历史

- **9.1**：FDW API 落地（commit bb74240794，2011-02-20，"Implement an API to let foreign-data wrappers actually be functional"）——SQL/MED 的目录语法（SERVER/USER MAPPING）在 8.4 就有，但到 9.1 才真的能扫描；
- **9.3**：postgres_fdw 入 contrib（commit d0d75c4022，2013-02-21），带 WHERE 下推与写能力；
- **9.6**：join 下推（commit e4106b2528，2016-02-09）；
- **10**：聚合下推（commit 7012b132d0，2016-10-21）;
- **14**：异步执行（commit 27e1f14563，2021-03-31）。

十年间的方向一以贯之：**把越来越大的计划子树整体搬到远端**，而每一步都要在 deparse.c 里回答同一个问题——"这个节点在远端求值还恒等吗"。

### 小结

FDW = 一张回调表把外部数据源适配成 Volcano 迭代器 + 让远端路径参与代价竞争。计划期：RelSize 估行数 → Paths 加路径（裸扫/有序/参数化）→ Plan 期 deparse 拼 SQL；执行期：惰性建游标 + 批量 FETCH。下推判定是白名单主义：节点类型白名单、对象 shippable 白名单、collation 溯源状态机、mutable 一票否决——全部为了守住"两端求值恒等"这条正确性底线。原子性上远端提交挂在本地 PRE_COMMIT，只有顺序单边提交，跨 server 写事务存在部分提交缺口。

### 15.6.1 CREATE TYPE：in/out 函数

`DefineType()`（`commands/typecmds.c:154`）要求的最小集合是一对文本转换函数：`input`（C 字符串 → 内部表示）和 `output`（反向），通过 `findTypeInputFunction` / `findTypeOutputFunction`（typecmds.c:481-482 调用）解析并校验签名后写入 `pg_type.typinput/typoutput`。可选 `receive/send` 提供二进制协议版本。典型的注册序列（DDL 层面）：

```sql
CREATE FUNCTION complex_in(cstring) RETURNS complex
    AS 'MODULE_PATHNAME' LANGUAGE C IMMUTABLE STRICT;
CREATE FUNCTION complex_out(complex) RETURNS cstring
    AS 'MODULE_PATHNAME' LANGUAGE C IMMUTABLE STRICT;
CREATE TYPE complex (
    internallength = 16,
    input = complex_in,
    output = complex_out,
    alignment = double
);
```

C 侧函数遵循 fmgr 的 **V1 调用约定**：函数体写成 `Datum complex_in(PG_FUNCTION_ARGS)` 并配一行 `PG_FUNCTION_INFO_V1(complex_in);` 导出元数据（与 `pl_handler.c:221` 的 `PG_FUNCTION_INFO_V1(plpgsql_call_handler)` 是同一个宏），参数经 `PG_GETARG_*` 宏取出——这层统一调用约定就是"SQL 函数可以指向任何 .so 里的符号"的机制基础。一旦注册，新类型就是一等公民：可以做表列、进索引、被统计（还可指定自定义 `typanalyze`，接上第 14 篇的统计槽位机制）。

### 15.6.2 操作符与 opclass：类型接入索引的桥

光有类型还查不快。三层桥接：

1. `CREATE OPERATOR` 把 `&&`（相交）这类符号绑定到函数，并声明 commutator/restrict 选择率估算器等优化器元数据；
2. `CREATE OPERATOR CLASS`（`DefineOpClass`，`commands/opclasscmds.c:333`）声明"某类型在某索引 AM 上如何工作"：一组策略号（操作符）+ 一组支持函数，写入 pg_opclass/pg_amop/pg_amproc；
3. 索引 AM 本身是 `IndexAmRoutine` 回调表（`src/include/access/amapi.h:233`），PG 9.6 起 AM 也可由扩展提供（如 bloom）。

**PostGIS 模式**就是这套机制的极致运用：`geometry` 类型（in/out 解析 WKT）+ 空间操作符（`&&` 等）+ **GiST opclass**。GiST 是"骨架树"：树的平衡、并发、WAL 由内核负责，扩展只需实现语义回调——`GIST_CONSISTENT_PROC`（查询是否可能命中子树，`access/gist.h:32`）、`GIST_UNION_PROC`（合并 bounding box，gist.h:33）、`GIST_PENALTY_PROC`（插入代价，gist.h:36）、`GIST_PICKSPLIT_PROC`（页分裂策略，gist.h:37）。PostGIS 用 bounding box 实现这四个函数，就免费获得了带并发控制和崩溃恢复的 R-Tree。**不改内核一行代码，得到一个空间数据库**——这是"平台性"最有说服力的证据。

### 15.6.3 过程语言（PL）框架与 SPI_execute 主线

PL 框架同样是"handler 模式"：`CREATE LANGUAGE` 注册一个 call handler；调用该语言写的函数时，fmgr 把控制交给 handler。PL/pgSQL 的 handler 是 `plpgsql_call_handler()`（`src/pl/plpgsql/src/pl_handler.c:224`）：编译函数体为内部指令树，逐语句解释执行。`src/pl/` 下同一模式还实现了 plperl/plpython/pltcl——理论上任何语言的解释器都能嵌进来。

PL 里的 SQL 语句靠 **SPI**（Server Programming Interface，`src/backend/executor/spi.c`）执行——"在服务端代码里执行 SQL"的官方 API。`SPI_execute`（spi.c:597）主线：

```c
int
SPI_execute(const char *src, bool read_only, long tcount)
{
	_SPI_plan	plan;
	SPIExecuteOptions options;
	int			res;

	if (src == NULL || tcount < 0)
		return SPI_ERROR_ARGUMENT;

	res = _SPI_begin_call(true);
	if (res < 0)
		return res;

	memset(&plan, 0, sizeof(_SPI_plan));
	plan.magic = _SPI_PLAN_MAGIC;
	plan.parse_mode = RAW_PARSE_DEFAULT;
	plan.cursor_options = CURSOR_OPT_PARALLEL_OK;

	_SPI_prepare_oneshot_plan(src, &plan);

	memset(&options, 0, sizeof(options));
	options.read_only = read_only;
	options.tcount = tcount;

	res = _SPI_execute_plan(&plan, &options,
							InvalidSnapshot, InvalidSnapshot,
							true);

	_SPI_end_call(true);
	return res;
}
```

逐段解说：`_SPI_begin_call` 校验当前处于某个 `SPI_connect()`（spi.c:95）打开的会话中并切到 SPI 的执行内存上下文；`_SPI_prepare_oneshot_plan` 只做 raw parse（一次性计划，不进 plan cache——与 `SPI_prepare`（spi.c:861）的可复用计划相对）；`_SPI_execute_plan` 内部对每条语句完整地走 analyze → rewrite → plan → 创建 Portal → 执行——**与客户端 SQL 完全同一条管线**，只是发起者从协议层换成了服务端 C 代码，结果落在 `SPI_tuptable`，行数落在 `SPI_processed`。SPI 还负责嵌套会话的内存上下文栈和快照管理（read_only 时复用当前快照、读写时推进 CommandCounter）——触发器、PL 函数、以及很多扩展（如 pg_cron）都构建在它之上。

### 小结

类型（in/out）→ 操作符（语义 + 优化器元数据）→ opclass（接入索引 AM 的支持函数）三级注册，让 PostGIS 无需改内核就获得工业级空间索引；PL handler + SPI 则把"用任意语言写存储过程、在服务端跑 SQL"也变成插件问题，SPI 内部就是标准 SQL 管线的服务端入口。

## 15.7 实战：从零写一个最小扩展

读者阶段 4 的直接教材。目标：一个 `my_ext` 扩展，提供 SQL 函数 `add_one(int) → int`（C 实现），并演示 hook 骨架。四个文件放在任意目录即可构建。

**文件 1：`my_ext.control`**——扩展的身份证：

```
# my_ext extension
comment = 'minimal example extension'
default_version = '1.0'
module_pathname = '$libdir/my_ext'
relocatable = true
```

逐行：`default_version` 决定 `CREATE EXTENSION my_ext` 不带 VERSION 时装哪个脚本（找 `my_ext--1.0.sql`）；`module_pathname` 是脚本里 `MODULE_PATHNAME` 的替换值（15.4.3 走读过替换点 extension.c:1479-1486）；`relocatable = true` 表示对象不写死 schema，可整体搬移。

**文件 2：`my_ext--1.0.sql`**——版本 1.0 的对象清单：

```sql
-- complain if script is sourced in psql, rather than via CREATE EXTENSION
\echo Use "CREATE EXTENSION my_ext" to load this file. \quit

CREATE FUNCTION add_one(integer) RETURNS integer
AS 'MODULE_PATHNAME', 'add_one'
LANGUAGE C STRICT IMMUTABLE;
```

逐行：第一行 `\echo ... \quit` 是惯例护栏——经 `CREATE EXTENSION` 执行时会被 extension.c:1396-1401 的正则清空，直接用 psql 跑脚本时则打印提示并退出（防止对象绕过扩展机制被裸建，那样就没有 `DEPENDENCY_EXTENSION` 依赖了）；`AS 'MODULE_PATHNAME', 'add_one'` 指明 .so 路径与符号名；`STRICT` 表示任一参数为 NULL 直接返回 NULL（C 代码因此不用防 NULL 参数）；`IMMUTABLE` 是优化器承诺（同输入恒同输出，可被常量折叠——乱标 volatile 函数为 IMMUTABLE 会造成错误的预计算）。

**文件 3：`my_ext.c`**——C 实现：

```c
#include "postgres.h"
#include "fmgr.h"

PG_MODULE_MAGIC;

PG_FUNCTION_INFO_V1(add_one);

Datum
add_one(PG_FUNCTION_ARGS)
{
    int32 arg = PG_GETARG_INT32(0);

    PG_RETURN_INT32(arg + 1);
}
```

逐行：`PG_MODULE_MAGIC;` 展开成 15.3.3 走读过的 `Pg_magic_func()`——没有它 `internal_load_library` 直接拒载（dfmgr.c:288-291）；`PG_FUNCTION_INFO_V1(add_one)` 导出该函数遵循 V1 调用约定的元数据；函数签名统一是 `Datum f(PG_FUNCTION_ARGS)`（即 `FunctionCallInfo fcinfo`），参数用 `PG_GETARG_INT32(0)` 按位置取（宏内部处理 Datum → C 类型的解包），返回用 `PG_RETURN_INT32`（反向打包）。变长类型（text 等）会涉及 detoast，用 `PG_GETARG_TEXT_PP` 一族。

若要做 hook 型扩展，再加一段（骨架，注意三条纪律：preload 检查、保存 prev、转发 prev）：

```c
#include "optimizer/planner.h"

static planner_hook_type prev_planner_hook = NULL;

static PlannedStmt *
my_planner(Query *parse, const char *query_string, int cursorOptions,
           ParamListInfo boundParams, ExplainState *es)
{
    elog(LOG, "my_ext: planning a query");
    if (prev_planner_hook)
        return prev_planner_hook(parse, query_string, cursorOptions,
                                 boundParams, es);
    return standard_planner(parse, query_string, cursorOptions,
                            boundParams, es);
}

void
_PG_init(void)
{
    prev_planner_hook = planner_hook;
    planner_hook = my_planner;
}
```

（纯 hook 库甚至不需要 control/SQL 文件——`LOAD 'my_ext'` 或 preload 即可生效；带 SQL 对象才需要扩展框架。）

**文件 4：`Makefile`**——PGXS 构建（与 `contrib/pg_stat_statements/Makefile:49-50` 的独立构建路径同款）：

```makefile
MODULES = my_ext
EXTENSION = my_ext
DATA = my_ext--1.0.sql

PG_CONFIG = pg_config
PGXS := $(shell $(PG_CONFIG) --pgxs)
include $(PGXS)
```

逐行：`MODULES = my_ext` 表示把 `my_ext.c` 编成 `my_ext.so`（多文件用 `MODULE_big + OBJS`）；`EXTENSION`/`DATA` 声明控制文件与脚本，`make install` 时装进 `SHAREDIR/extension/`；PGXS 的精髓在最后三行——`pg_config --pgxs` 返回已安装 PG 的 `src/makefiles/pgxs.mk` 路径，include 之后就继承了目标 PG 的全部编译参数（头文件路径、编译器选项、安装路径），**扩展因此可以脱离内核源码树、只依赖二进制安装包构建**。

**闭环验证**：

```
$ make && make install        # .so → $libdir, control/sql → SHAREDIR/extension/
$ psql -c "CREATE EXTENSION my_ext"
$ psql -c "SELECT add_one(41)"   # → 42
$ psql -c "DROP EXTENSION my_ext"  # add_one 随之级联删除（pg_depend 的功劳）
```

之后的演进路径：加 `my_ext--1.0--1.1.sql` 做升级、`REGRESS = ...` 接 pg_regress 回归测试、需要共享内存/bgworker 时按 15.2.2/15.3.4 的模式补 `RegisterShmemCallbacks` 并要求 preload。

### 小结

最小扩展 = control（身份）+ SQL 脚本（对象）+ C 文件（PG_MODULE_MAGIC + V1 约定）+ PGXS Makefile（脱离源码树构建）。逐行理解这四个文件后，contrib 下任何扩展的骨架都是它的放大版。

## 15.8 为什么可扩展性是 PG 与 MySQL 生态分野的根源

回到 1986 年。Stonebraker 与 Rowe 的《The Design of POSTGRES》开篇就把目标定为支持"非传统应用"——CAD、地理、复杂对象——并给出了实现路线：**抽象数据类型（ADT）、用户定义的操作符和函数、可扩展的访问方法**（同期论文《Inclusion of New Types in Relational Data Base Systems》专门论证了这一点）。也就是说，"用户能定义类型并让优化器和索引理解它"不是 PG 后来长出的特性，而是**立项动机本身**。本篇讲的所有机制都是这个思想的直系后代：pg_type/pg_proc/pg_am 这些目录表从 Berkeley 时代就是"可写的注册表"而非只读的系统内部结构。后来的每一层——magic block（2006）、extension 框架（2011）、FDW（2011）、queryId 内核化（2021）、shmem 回调注册表（20devel）——都是在同一根系上把"可插拔"做得更安全、更可运维。

MySQL 走的是另一条路：它的扩展点是**存储引擎接口**（handler API）——在"表怎么存"这一层可插拔（MyISAM/InnoDB/RocksDB），但类型系统、优化器、执行器基本封闭；UDF 只是标量函数，进不了索引和统计。结果是两种生态形态：

- MySQL 的创新以 **fork 和魔改**形态出现（Percona、MariaDB、各大厂内核团队），与上游分叉、合并困难；
- PG 的创新以**扩展**形态出现（PostGIS、TimescaleDB、Citus、pgvector），跟随上游大版本平滑升级，一个内核团队的产出被整个生态复用。

工程上的因果链值得反复体会：**接口开放度决定创新成本，创新成本决定生态形态**。pgvector 从想法到可用只需要实现"类型 + 操作符 + 索引 AM"，赶上向量检索风口只用了很短时间——同样的事在封闭内核上意味着维护一个 fork。对想深入内核的工程师，这也是最佳练手场：写一个扩展（15.7 就是起点），你就在真实内核 API 上写了生产级代码，而不用等社区 review 你对内核本体的改动。

## 自测问题

1. **一个扩展想同时统计所有会话的查询，为什么必须进 `shared_preload_libraries`？只用 `LOAD` 会缺什么？**
   详解：跨会话统计需要共享内存，而主共享内存段在 postmaster 启动时按各子系统的申报总量一次性定容、之后不能长大，所以申请窗口只在启动期（`RegisterShmemCallbacks`，shmem.c:873；启动后注册默认报 "cannot request shared memory at this time"，shmem.c:881-882）。`LOAD` 只影响当前会话：hook 装上了也只有本会话的查询会经过，其他 backend 根本不知道你存在。pg_stat_statements 因此在 `_PG_init` 开头检查 `process_shared_preload_libraries_in_progress`，不满足即静默放弃安装（pg_stat_statements.c:400-401）。20devel 的 `SHMEM_CALLBACKS_ALLOW_AFTER_STARTUP`（shmem.h:167）允许启动后注册小块区域，但"无容量保证"，不改变大格局。

2. **写 hook 函数时为什么要保存并转调 `prev_xxx`？两个扩展都挂 `planner_hook` 时谁先执行？**
   详解：hook 是单个全局函数指针，不保存前值直接覆盖会把先加载扩展的函数指针"挤掉"，且故障是静默的（15.1.4：pgss 的 planner 统计悄悄归零、嵌套计数错乱，无任何报错）。保存 prev 并转调后形成链：`shared_preload_libraries = 'a, b'` 时 b 后加载、保存的 prev 是 a 的函数，运行时 **b 在外层先执行**（LIFO/洋葱模型），逐层转发直到某层调 `standard_planner`。顺序只由加载顺序决定，没有优先级仲裁——这是裸函数指针设计的固有代价。

3. **`pgss_ExecutorRun` 里的 PG_TRY/PG_FINALLY 防的是什么？删掉会怎样？**
   详解：查询出错时 elog(ERROR) 走 longjmp 非局部跳转，不会执行 try 块之后的普通语句。没有 PG_FINALLY，`nesting_level--` 被跳过，计数永久偏大：此后所有顶层语句都被误判为嵌套语句，`track = top` 时 pgss 从此**不再记录任何语句**——一个错误查询导致统计静默失效。PG_FINALLY 保证 longjmp 路径上也执行递减（pg_stat_statements.c:1019-1031）。这是"hook 中修改进程级状态必须考虑非局部跳转"纪律的标准示例。

4. **pg_stat_statements 为什么把查询文本放外部文件而不是共享内存？条目里存什么？**
   详解：共享内存启动定容，按最大文本长度预留会巨大浪费，按小限额截断则长 SQL 无法复原（9.4 前正是如此，按 `track_activity_query_size` 截断）。commit f0d6f20278（2014-01-27）改为：条目只存 `(query_offset, query_len)`（pgssEntry，:237-238），文本追加写入 `pgss_query_texts.stat`；追加用自旋锁原子推进 `pgss->extent` 预订区间、各进程 `pg_pwrite` 并发写互不重叠的区间（qtext_store，:2231）；读视图时整文件载入再按偏移取（qtext_load_file/qtext_fetch）。死文本靠 gc_qtexts 定期重写文件回收，`gc_count` 版本号用于检测"锁升级窗口里发生过 GC"的竞态（pgss_store :1367-1369）。

5. **magic block 逐字段记录 `FUNC_MAX_ARGS`/`NAMEDATALEN` 等值的意义是什么？只比 PG 主版本号为什么不够？**
   详解：ABI 兼容性不只由版本决定。`NAMEDATALEN` 等是编译期可调参数，改了它的自编译内核与官方扩展二进制版本号完全相同，但所有含 NameData 的结构布局都错位——加载后不是干净报错而是 undefined behavior（错位读参数、垃圾指针，最坏写坏数据页）。所以 `Pg_abi_values`（fmgr.h:468-476）把 version、funcmaxargs、indexmaxkeys、namedatalen、float8byval、abi_extra 全部记录，校验时 `memcmp` 整块比对（dfmgr.c:264-266），报错时分层逐字段输出（incompatible_module_error，dfmgr.c:316：版本不对只报版本 → abi_extra 不对只报产品 → 否则逐项列出差异）。`abi_extra` 专防"版本号相同的 fork 产品"。

6. **`ALTER EXTENSION foo UPDATE TO '2.0'` 时不存在 `foo--1.0--2.0.sql`，内核怎么办？平局时选哪条路径？**
   详解：`get_ext_ver_list`（extension.c:1582）把目录里所有 `A--B.sql` 建成版本有向图，`find_update_path`（extension.c:1688）做 Dijkstra 最短路，找到如 1.0→1.1→2.0 的链后逐段执行。无连通路径才报错。等长路径按前驱版本名 strcmp 打破平局（:1737-1750）——不为语义，只为确定性（避免升级路径取决于目录项遍历顺序）。安装时同样用这套图：目标版本没有直装脚本时，从某个可直装版本升上去（CreateExtensionInternal :1890-1916）。

7. **postgres_fdw 判断一个 WHERE 条件能否下推要过哪几道闸？`WHERE ts > now() - interval '1 day'` 会被下推吗？**
   详解：三道闸（is_foreign_expr，deparse.c:244-292）：①`foreign_expr_walker` 递归检查——节点类型在白名单内、函数/操作符 `is_shippable`（内建或 server 声明的扩展）、collation 状态机不落入 UNSAFE；②整树 collation 终态检查；③`contain_mutable_functions` 一票否决。`now()` 是 stable（属 mutable 范畴）→ 不能进远端 SQL 文本（远端时钟是另一个时钟）；但参数化机制可以本地求值后作为 Param 发送，实际效果是"本地算成常量再下推"。换成 `random()` 这类 volatile 则彻底只能本地过滤。

8. **给出一个"collation 条件下推后结果错误"的具体场景，并指出 walker 里拦住它的状态。**
   详解：`SELECT * FROM ft WHERE name < 'b' COLLATE "C"`。本地语义按字节序，`'Boston'`（B < b）应入选；下推后远端按其列的 en_US 字典序比较，Boston 排在 b 后，该行丢失——静默少行。机制上：COLLATE 折叠使 Const 的 `constcollid` 为非默认 collation，walker 的 Const 分支置 `FDW_COLLATE_UNSAFE`（deparse.c:474-479），is_foreign_expr 末端检查 `loc_cxt.state == FDW_COLLATE_UNSAFE` 即拒推（deparse.c:277-278）。只有"collation 来自外部表自己的 Var"（FDW_COLLATE_SAFE，:363-364）才被认为两端行为一致。

9. **一个本地事务写了两个不同 server 的外部表然后 COMMIT，远端 B 在提交时网络中断。最终三方各处于什么状态？为什么内核禁止对这种事务做 PREPARE TRANSACTION？**
   详解：提交顺序是"本地 PRE_COMMIT 阶段逐个向远端发 COMMIT，全部成功后本地才提交"（pgfdw_xact_callback，connection.c:1205-1222）。A 先提交成功、B 提交失败 → 本地事务转 ABORT。终态：**A 已永久提交、B 回滚、本地回滚**——部分提交，无协调者可挽回。这是"顺序单边提交"协议的固有原子性缺口；PRE_COMMIT 时机只保证不出现更糟的"本地已提交而远端未提交"。PREPARE 被禁止（connection.c:1260-1262）是因为 prepared 事务承诺后续 COMMIT PREPARED 必然成功，而远端普通事务无法陪同进入 prepared 状态，承诺兑现不了。完整解需要远端 2PC + 全局事务管理，内核未内置（Citus 等自带）。

10. **PostGIS 没有修改任何内核代码，它的 `&&` 操作符为什么能用上索引？请说出登记链条。**
    详解：`CREATE TYPE geometry`（in/out 函数入 pg_type，DefineType，typecmds.c:154）→ `CREATE OPERATOR &&`（入 pg_operator，带选择率估算器）→ `CREATE OPERATOR CLASS ... USING gist`（DefineOpClass，opclasscmds.c:333，把 `&&` 映射为策略号入 pg_amop、把 consistent/union/penalty/picksplit 支持函数入 pg_amproc）。优化器看到 `col && box` 时查 pg_amop 发现该操作符属于索引列 opclass 的策略集合，即可生成 IndexPath 参与比价；执行时 GiST 骨架（页面组织、并发、WAL）由内核提供，回调号 `GIST_CONSISTENT_PROC` 等（gist.h:32-37）转到 PostGIS 的 C 函数。整条链全部经由"可写的系统目录"完成——这正是 Stonebraker 把目录设计成注册表的初衷。
