# 第 15 篇 · 扩展机制：PostgreSQL 的"平台性"

> 对应《PostgreSQL 源码学习指南》第 17 章。源码基线：PostgreSQL 20devel（master 分支）。

## 本章导读

如果只许用一句话概括 PG 与 MySQL 的生态差异，那就是：**MySQL 是一个数据库，PostgreSQL 是一个做数据库的平台**。PostGIS（地理）、TimescaleDB（时序）、Citus（分布式）、pgvector（向量检索）都不是 fork，而是运行在原版 PG 内核之上的扩展——这在 MySQL 世界里没有对应物。

本篇自底向上拆解这个"平台"的六层机制：

1. **Hook 体系**——内核在关键路径上留的函数指针，扩展的寄生点（以 `pg_stat_statements` 为例逐段走读）；
2. **动态库加载**——`_PG_init`、`shared_preload_libraries` 与 `LOAD` 的能力差异；
3. **扩展框架**——`CREATE EXTENSION` 的控制文件、SQL 脚本与版本升级路径；
4. **FDW**——把任何数据源伪装成表，以及 `postgres_fdw` 的下推实现；
5. **自定义类型/操作符/操作符类**——PostGIS 模式；
6. **过程语言框架与 SPI**。

最后回到源头：为什么说可扩展性是 Stonebraker 在 Berkeley 时代就写进 POSTGRES 基因的第一设计目标。

## 前置知识

- 一条 SQL 的处理管线：parse → analyze → rewrite → plan → execute（tcop 主循环）；
- 优化器的 Path/RelOptInfo 概念（FDW 一节会往 `add_path` 里塞外部路径）；
- 系统目录（pg_proc、pg_type、pg_am 等）与 syscache；
- C 语言函数指针与动态链接（dlopen/dlsym）的基本概念。

## 15.1 Hook 体系全景

### 15.1.1 Hook 是什么：一个函数指针 + 一行 if

PG 的 hook 模式极其朴素。以 planner 为例（`src/backend/optimizer/plan/planner.c`）：

```c
/* Hook for plugins to get control in planner() */
planner_hook_type planner_hook = NULL;          /* planner.c:74 */
...
	if (planner_hook)
		result = (*planner_hook) (parse, query_string, cursorOptions,
								  boundParams);  /* planner.c:333 */
	else
		result = standard_planner(parse, query_string, cursorOptions,
								  boundParams);
```

全局函数指针默认 NULL；扩展在 `_PG_init` 里把它指向自己的函数；内核在调用点检查非空则转交控制权。扩展函数内部通常再调用 `standard_planner()`（或链式调用上一个 hook），形成"洋葱式"包裹。没有注册表、没有优先级框架——**约定优于机制**，这是 PG 一贯的工程品味。

### 15.1.2 主要 hook 的位置与签名

| Hook | 定义位置 | 时机 |
|---|---|---|
| `post_parse_analyze_hook` | `parser/analyze.c:74`（调用点 149-151） | 语义分析完成、拿到 Query 树之后 |
| `planner_hook` | `optimizer/plan/planner.c:74` | 整体替换/包裹优化器 |
| `ExecutorStart_hook` | `executor/execMain.c:70` | 执行器初始化 |
| `ExecutorRun_hook` | `executor/execMain.c:71` | 执行主循环 |
| `ExecutorFinish_hook` | `executor/execMain.c:72` | AFTER 触发器等收尾 |
| `ExecutorEnd_hook` | `executor/execMain.c:73` | 执行器销毁前（统计的黄金采集点） |
| `ExecutorCheckPerms_hook` | `executor/execMain.c:76` | 表级权限检查（行级安全类扩展用） |
| `ProcessUtility_hook` | `tcop/utility.c:72`（调用点 523） | 所有 DDL/工具语句 |
| `shmem_startup_hook` | `storage/ipc/ipci.c:31`（调用点 110/158） | 共享内存创建后初始化自己的份额 |

签名以 `ExecutorRun_hook_type` 为例：`void (*)(QueryDesc *queryDesc, ScanDirection direction, uint64 count)`——与 `standard_ExecutorRun` 完全同构，保证 hook 函数可以透明转发。

共享内存申请侧在 20devel 有一处值得注意的演变：PG 15~18 使用 `shmem_request_hook`，而当前 master 已把它重构为**回调注册表**——`RegisterShmemCallbacks()`（`storage/ipc/shmem.c:873`），扩展提交一个 `ShmemCallbacks` 结构（`src/include/storage/shmem.h`，含 `request_fn`/`init_fn`/`attach_fn` 三个回调），默认仍要求在 postmaster 启动期注册，但新增 `SHMEM_CALLBACKS_ALLOW_AFTER_STARTUP` 标志（shmem.h:168）允许扩展在启动后注册。读 PG ≤ 18 的扩展代码时看到的 `shmem_request_hook = ...` 就是这套机制的前身。

其他常用 hook 散布各处：`emit_log_hook`（elog.c，日志外送）、`check_password_hook`、`object_access_hook`（DDL 审计，sepgsql 用）、`set_rel_pathlist_hook` / `create_upper_paths_hook`（往优化器塞自定义路径，Citus/columnar 存储的入口）、`explain_get_index_name_hook` 等。用 `grep -rn "_hook_type.*= NULL" src/backend/ | wc -l` 可以数出内核几十个 hook 点。

### 15.1.3 案例走读：pg_stat_statements 如何串接全链路

`contrib/pg_stat_statements/pg_stat_statements.c` 是 hook 教科书：它要回答"每类 SQL 执行了多少次、总耗时多少"，为此几乎挂满了整条管线。

**第一步：_PG_init 注册（pg_stat_statements.c:390）**：

```c
	if (!process_shared_preload_libraries_in_progress)
		return;                                  /* 只在 preload 时生效 */
	...
	EnableQueryId();                             /* 要求内核计算 queryId */
	DefineCustomIntVariable("pg_stat_statements.max", ...);   /* 自定义 GUC */
	...
	MarkGUCPrefixReserved("pg_stat_statements");
	RegisterShmemCallbacks(&pgss_shmem_callbacks);            /* 申请共享内存 */

	/* Install hooks. */
	prev_post_parse_analyze_hook = post_parse_analyze_hook;
	post_parse_analyze_hook = pgss_post_parse_analyze;
	prev_planner_hook = planner_hook;
	planner_hook = pgss_planner;
	prev_ExecutorStart = ExecutorStart_hook;
	ExecutorStart_hook = pgss_ExecutorStart;
	...
	ProcessUtility_hook = pgss_ProcessUtility;   /* :480-500 附近 */
```

逐行解说：

- 开头的 `process_shared_preload_libraries_in_progress` 检查是**能力门槛**：不经 preload 加载则静默退出（不装 hook），因为共享内存申请不到（见 15.2）；
- 保存旧值 `prev_xxx` 再覆盖——这是 hook 链的礼仪：自己的处理函数里必须转调 `prev_xxx`（若非空），多个扩展才能共存；
- `pgss_shmem_callbacks`（pg_stat_statements.c:267）注册两个回调：`request_fn = pgss_shmem_request`（:506）在共享内存定型前申报需求，`init_fn = pgss_shmem_init`（:530 附近）在共享内存分配后创建哈希表并从磁盘文件恢复上次的统计。

看一眼 `pgss_shmem_request`（pg_stat_statements.c:506），感受 20devel 的新申报 API：

```c
static void
pgss_shmem_request(void *arg)
{
	ShmemRequestHash(.name = "pg_stat_statements hash",
					 .nelems = pgss_max,
					 .hash_info.keysize = sizeof(pgssHashKey),
					 .hash_info.entrysize = sizeof(pgssEntry),
					 .hash_flags = HASH_ELEM | HASH_BLOBS,
					 .ptr = &pgss_hash,
		);
	ShmemRequestStruct(.name = "pg_stat_statements",
					   .size = sizeof(pgssSharedState),
					   .ptr = (void **) &pgss,
		);
}
```

逐行解说：申报一张共享哈希表（key 是 `(userid, dbid, queryId, toplevel)` 组成的 `pgssHashKey`，条目数上限 `pgss_max`）和一个全局状态结构体；`.ptr` 传出指针地址，框架会在 init/attach 阶段把分配好的共享内存地址填回来。在 PG ≤ 18 里这两步分别对应 `shmem_request_hook` 里的 `RequestAddinShmemSpace()` 和 `shmem_startup_hook` 里的 `ShmemInitHash()`/`ShmemInitStruct()`——语义相同，新 API 把"申报"与"初始化"的配对关系显式化了。

**第二步：解析后拿指纹（pgss_post_parse_analyze，:834）**：

```c
	if (prev_post_parse_analyze_hook)
		prev_post_parse_analyze_hook(pstate, query, jstate);

	if (!pgss || !pgss_hash || !pgss_enabled(nesting_level))
		return;
	...
	if (jstate && jstate->clocations_count > 0)
		pgss_store(pstate->p_sourcetext, query->queryId, ...);
```

`jstate` 是内核 query jumbling（`nodes/queryjumblefuncs.c`）的产物：把 Query 树遍历一遍、忽略常量算出 64 位 `queryId`，并记录每个常量在原文中的位置。pgss 在这里提前建条目，用常量位置把 SQL 文本规范化成 `SELECT * FROM t WHERE id = $1` 的形式——这就是视图里看到的归一化语句。

**第三步：计划与执行计时**。`pgss_planner`（:887）包住真正的 planner 测量计划耗时；`pgss_ExecutorRun`（:1017）本身几乎什么都不做：

```c
	nesting_level++;
	PG_TRY();
	{
		if (prev_ExecutorRun)
			prev_ExecutorRun(queryDesc, direction, count);
		else
			standard_ExecutorRun(queryDesc, direction, count);
	}
	PG_FINALLY();
	{
		nesting_level--;
	}
	PG_END_TRY();
```

它只维护 `nesting_level`——用于区分顶层语句和被函数/触发器引发的内层语句（`pg_stat_statements.track = top` 时只记顶层）。`PG_TRY/PG_FINALLY` 保证出错时嵌套深度也能复原，这是写 hook 的必修防御姿势。

**第四步：落账（pgss_ExecutorEnd，:1059）**：

```c
	int64		queryId = queryDesc->plannedstmt->queryId;

	if (queryId != INT64CONST(0) && queryDesc->query_instr && ...)
	{
		pgss_store(queryDesc->sourceText, queryId, ...,
				   PGSS_EXEC,
				   INSTR_TIME_GET_MILLISEC(queryDesc->query_instr->total),
				   queryDesc->estate->es_total_processed,
				   &queryDesc->query_instr->bufusage,
				   &queryDesc->query_instr->walusage, ...);
	}
	if (prev_ExecutorEnd)
		prev_ExecutorEnd(queryDesc);
	else
		standard_ExecutorEnd(queryDesc);
```

执行时长、行数、buffer 命中/读盘、WAL 生成量在此刻全部就绪，按 queryId 累加进共享哈希表（分区锁 + 每条目自旋锁，控制并发开销）。`pgss_ProcessUtility`（:1092）对 DDL 走同样的"计时 + 落账"逻辑。

一张图总结 pgss 的寄生结构：**post_parse_analyze 建指纹 → planner 记计划耗时 → ExecutorRun 记嵌套 → ExecutorEnd 落账 → ProcessUtility 覆盖 DDL → shmem callbacks 提供存储**。TimescaleDB、Citus 的 hook 用法只是把"记账"换成了"改写计划"。

### 小结

Hook = 全局函数指针 + 调用点判空，扩展负责保存并链式转调前一个值。pg_stat_statements 用六个挂点覆盖了 SQL 全生命周期，是理解"寄生式扩展"的最小完整样本。

## 15.2 动态库加载：_PG_init 与两种加载时机

加载机制在 `src/backend/utils/fmgr/dfmgr.c`。核心函数 `internal_load_library()`（dfmgr.c:189）做三件事：

1. `dlopen` 后查找 magic block（dfmgr.c:256-289）：每个扩展 .so 必须包含 `PG_MODULE_MAGIC` 宏（`src/include/fmgr.h:443`）展开出的版本信息函数，记录编译时的主版本号、`FUNC_MAX_ARGS`、对齐等 ABI 参数，与服务器逐项比对，不符则拒载——防止"用 PG 16 头文件编的 .so 装进 PG 18"这类事故；
2. 若库导出了 `_PG_init()` 则调用之（dfmgr.c:295 附近）——这是扩展唯一的初始化入口，装 hook、定义 GUC 都在这里；
3. 记入已加载列表，避免重复加载（并行 worker 靠 leader 序列化的库列表重放同样的加载）。

**两种加载时机的能力差异**：

| | `shared_preload_libraries` | `LOAD` / 首次调用 C 函数时隐式加载 |
|---|---|---|
| 执行者与时机 | postmaster 启动时（`process_shared_preload_libraries()`，`utils/init/miscinit.c:1855`） | 单个 backend 会话内（`load_file()`，dfmgr.c:149） |
| 影响范围 | 所有后续 backend（fork 继承） | 仅当前会话 |
| 申请共享内存 | 可以（此时共享内存尚未定型） | 传统上不可以；20devel 起可用 `SHMEM_CALLBACKS_ALLOW_AFTER_STARTUP` 走 DSM 补注册（shmem.h:168） |
| 注册命名 LWLock tranche | 可以（`RequestNamedLWLockTranche()`，注释见 `storage/lmgr/lwlock.c:202`——仅 postmaster 内有效） | 不可以（可用运行时的 `LWLockNewTrancheId` 替代） |
| 注册静态 bgworker | 可以（`RegisterBackgroundWorker()`，`postmaster/bgworker.c:962`） | 只能动态启动（`RegisterDynamicBackgroundWorker()`，bgworker.c:1068） |
| 挂 planner/executor hook | 可以 | 也可以，但只影响本会话 |
| 修改 GUC 需重启？ | 是（改 shared_preload_libraries 要重启） | 否 |

这就是为什么 pg_stat_statements、TimescaleDB、Citus 都**必须**写进 `shared_preload_libraries`：它们要么要共享内存，要么要全局生效的 hook；而 PostGIS 这类"纯函数 + 类型"扩展不需要 preload——它的 .so 在你第一次调用其函数时按 `pg_proc.probin` 自动加载。

### 小结

加载 = dlopen + magic 校验 + `_PG_init`。preload 与会话加载的分水岭是"postmaster 级资源"（共享内存、命名 tranche、静态 bgworker）：这些只在启动窗口开放，错过就只能用 DSM/动态 API 的降级方案。

## 15.3 扩展框架：CREATE EXTENSION 的打包机制

`.so` 只解决了"代码进来"，`CREATE EXTENSION` 解决的是"SQL 对象成组管理"。实现在 `src/backend/commands/extension.c`（入口 `CreateExtension`，extension.c:2149 → `CreateExtensionInternal`，extension.c:1836）。先看一个真实的控制文件（`contrib/postgres_fdw/postgres_fdw.control`，全文只有五行）：

```
# postgres_fdw extension
comment = 'foreign-data wrapper for remote PostgreSQL servers'
default_version = '1.3'
module_pathname = '$libdir/postgres_fdw'
relocatable = true
```

一个扩展 = 三种文件：

1. **控制文件** `extname.control`（`SHAREDIR/extension/` 下），由 `parse_extension_control_file()` 解析（extension.c:693），字段对应 `ExtensionControlFile` 结构（extension.c:89-102）：`default_version`、`module_pathname`（脚本里 `MODULE_PATHNAME` 的替换值，指向 .so）、`schema`、`relocatable`（能否 `ALTER EXTENSION SET SCHEMA`）、`trusted`（允许非超级用户安装时临时切换为超级用户执行脚本）、`requires`（依赖的其他扩展）；
2. **SQL 脚本** `extname--1.0.sql`：普通 SQL，创建函数/类型/视图。由 `execute_extension_script()` 执行（extension.c:1248）；
3. **升级脚本** `extname--1.0--1.1.sql`：版本迁移的增量 SQL。翻开 `contrib/pg_stat_statements/` 就能看到从 `--1.0--1.1.sql` 一路到 `--1.12--1.13.sql` 的完整链条——每个 PG 大版本给视图加列，就加一个增量脚本。

"对象归属扩展"的实现机制值得一看（extension.c:1367-1372）：

```c
	creating_extension = true;
	CurrentExtensionObject = extensionOid;
	PG_TRY();
	{
		char	   *c_sql = read_extension_script_file(control, filename);
```

执行脚本前置起两个全局标志，此后所有 DDL 在登记依赖时都会经过 `recordDependencyOnCurrentExtension()`（`catalog/pg_depend.c:206`）：发现 `creating_extension` 为真，就给新对象追加一条指向扩展的 `DEPENDENCY_EXTENSION` 依赖。这就是 `DROP EXTENSION ... CASCADE` 能级联删干净、pg_dump 对扩展成员对象只输出一行 `CREATE EXTENSION` 的原因。扩展本身登记在系统目录 `pg_extension` 里（记录名称、schema、已安装版本）。

**版本升级路径**是这套框架最聪明的部分：`get_ext_ver_list()`（extension.c:1582）扫描目录里所有 `A--B.sql` 文件构建**版本有向图**，`find_update_path()`（extension.c:1688）在图上做 Dijkstra 最短路——从 1.0 升到 2.0 不需要 `1.0--2.0` 脚本存在，只要 `1.0--1.1`、`1.1--2.0` 能连通即可。`ALTER EXTENSION UPDATE` 沿路径逐段执行。对比 MySQL 世界：模式迁移完全是 Flyway/Liquibase 这类外部工具的事，PG 把"带版本的对象包"做进了内核。

### 小结

扩展 = 控制文件（元数据）+ 版本化 SQL 脚本（对象）+ 可选 .so（代码），靠 pg_depend 实现整体生命周期管理，靠版本图最短路实现任意版本间升级。

## 15.4 FDW：把一切伪装成表

### 15.4.1 接口：FdwRoutine 回调表

FDW 的内核侧在 `src/backend/foreign/`，接口定义在 `src/include/foreign/fdwapi.h`。优化器遇到外部表时，通过 `GetFdwRoutine()`（`foreign/foreign.c:357`）调用该 FDW 的 handler 函数，拿回一张回调表：

```c
	datum = OidFunctionCall0(fdwhandler);
	routine = (FdwRoutine *) DatumGetPointer(datum);
```

`FdwRoutine`（fdwapi.h:213 起）的核心回调按调用顺序解说（扫描路径五件套是唯一强制实现的）：

- **`GetForeignRelSize`**（fdwapi.h:27）：计划期第一站。估算外部表行数/宽度，填 `baserel->rows`——相当于替远端做 ANALYZE 的活；
- **`GetForeignPaths`**（fdwapi.h:31）：生成一条或多条 `ForeignPath` 并 `add_path` 进候选集，带上 startup/total cost 参与正常的代价竞争——FDW 不绕过优化器，而是**参赛**；
- **`GetForeignPlan`**（fdwapi.h:35）：被选中后把 Path 转成 `ForeignScan` 计划节点，决定哪些 qual 留给远端（下推）、哪些本地复查，私有信息（如拼好的远端 SQL）塞进 `fdw_private`；
- **`BeginForeignScan`**（fdwapi.h:43）：执行器初始化，建连接、准备游标；
- **`IterateForeignScan`**（fdwapi.h:46）：Volcano 接口——每次返回一个元组槽，空槽表示结束。**FDW 的本质就是把任何东西适配成这个迭代器**；
- **`ReScanForeignScan` / `EndForeignScan`**（fdwapi.h:51/53）：重扫（nestloop 内侧需要）与清理；
- 写路径：`ExecForeignInsert`（fdwapi.h:84）等一族；批量导入：`ImportForeignSchema`（fdwapi.h:164，`IMPORT FOREIGN SCHEMA` 一键映射远端全部表）；
- 进阶下推：`GetForeignJoinPaths`（fdwapi.h:55，两个外部表的 join 整体下推）、`GetForeignUpperPaths`（fdwapi.h:62，聚合/排序/LIMIT 下推）。

### 15.4.2 postgres_fdw：下推的参考实现

`contrib/postgres_fdw/` 是功能最全的参考实现，handler 在 postgres_fdw.c:648。三类下推的要点：

**WHERE 下推**。`postgresGetForeignRelSize`（postgres_fdw.c:719）里第一步就是分类：

```c
	classifyConditions(root, baserel, baserel->baserestrictinfo,
					   &fpinfo->remote_conds, &fpinfo->local_conds);
```

判定标准在 `deparse.c` 的 `is_foreign_expr()`（deparse.c:244）→ `foreign_expr_walker()`（deparse.c:312）：递归遍历表达式树，只有当所有操作符/函数都是**内建的或白名单扩展的**（保证远端语义一致）、collation 可安全外推时才可下推。`now()` 这类 volatile 函数、本地自定义函数都会被留在本地——这是"为什么我的条件没下推"的排查入口。

**执行**。`postgresBeginForeignScan`（postgres_fdw.c:1593）建连接；`postgresIterateForeignScan`（postgres_fdw.c:1698）是标准的"批量拉取 + 本地缓冲"：

```c
	if (!fsstate->cursor_exists)
		create_cursor(node);                 /* 远端 DECLARE CURSOR */

	if (fsstate->next_tuple >= fsstate->num_tuples)
	{
		...
		if (!fsstate->eof_reached)
			fetch_more_data(node);           /* FETCH 100 FROM c1 */
		...
	}
	ExecStoreHeapTuple(fsstate->tuples[fsstate->next_tuple++], slot, false);
```

远端开游标、每次 `FETCH fetch_size`（默认 100）行进本地数组，逐行喂给执行器——把"逐 tuple 的 Volcano 接口"和"网络批量传输"解耦，与 JDBC fetchSize 的思路一致。

两个配套工程细节：连接管理在 `contrib/postgres_fdw/connection.c`——同一 user mapping 的连接跨语句缓存复用，并把远端事务与本地事务对齐（本地 COMMIT 时驱动远端 COMMIT）；PG 14 起还实现了**异步执行**回调（`postgresForeignAsyncRequest` 等，postgres_fdw.c:8077 附近），使 `Append` 下的多个外部分区可以并发地等待各自远端的结果，而不是串行逐个拉取——分区表 + postgres_fdw 组成的简易 sharding 因此有了并行取数的味道。

**JOIN / 聚合下推**。`postgresGetForeignJoinPaths`（postgres_fdw.c:7143）：当 join 两侧是**同一 server** 的外部表、join 条件整体可下推时，为 joinrel 生成一条"整个 join 在远端执行"的 ForeignPath。`postgresGetForeignUpperPaths`（postgres_fdw.c:7528）同理处理 GROUP BY/聚合/ORDER BY/LIMIT。真正生成 SQL 的是 **`deparse.c` 的反解析器**：`deparseSelectStmtForRel()`（deparse.c:1286）从计划结构反向拼出 `SELECT ... FROM a INNER JOIN b ON ... WHERE ... GROUP BY ...` 文本，`deparseFromExprForRel()`（deparse.c:1812）递归处理 join 树，`appendWhereClause()`（deparse.c:1659）拼谓词。这个"计划树 → SQL 文本"的反向旅程，和 Presto/Trino connector 的 pushdown、Spark DataSource V2 的 `SupportsPushDownFilters` 是同一个问题的不同实现——PG 的特殊之处在于下推决策完全复用主优化器的代价框架（可选 `use_remote_estimate` 甚至向远端发 EXPLAIN 拿真实代价）。

### 小结

FDW = 一张回调表把外部数据源适配成 Volcano 迭代器 + 一组可选回调让远端参与代价竞争。postgres_fdw 展示了完整的下推谱系：表达式白名单判定（is_foreign_expr）→ 计划反解析成 SQL（deparse.c）→ 游标批量拉取。

## 15.5 自定义类型、操作符与操作符类

### 15.5.1 CREATE TYPE：in/out 函数

`DefineType()`（`commands/typecmds.c:154`）要求的最小集合是一对文本转换函数：`input`（C 字符串 → 内部表示）和 `output`（反向），通过 `findTypeInputFunction` / `findTypeOutputFunction`（typecmds.c:120-121）解析并校验签名后写入 `pg_type.typinput/typoutput`。可选 `receive/send` 提供二进制协议版本。典型的注册序列（DDL 层面）：

```sql
CREATE FUNCTION complex_in(cstring) RETURNS complex
    AS 'MODULE_PATHNAME' LANGUAGE C IMMUTABLE STRICT;
CREATE FUNCTION complex_out(complex) RETURNS cstring
    AS 'MODULE_PATHNAME' LANGUAGE C IMMUTABLE STRICT;
CREATE TYPE complex (
    internallength = 16,
    input = complex_in,
    output = complex_out,
    alignment = double
);
```

C 侧函数遵循 fmgr 的 **V1 调用约定**：函数体写成 `Datum complex_in(PG_FUNCTION_ARGS)` 并配一行 `PG_FUNCTION_INFO_V1(complex_in);` 导出元数据（与 `pl_handler.c:221` 的 `PG_FUNCTION_INFO_V1(plpgsql_call_handler)` 是同一个宏），参数经 `PG_GETARG_*` 宏取出——这层统一调用约定就是"SQL 函数可以指向任何 .so 里的符号"的机制基础。一旦注册，新类型就是一等公民：可以做表列、进索引、被统计（还可指定自定义 `typanalyze`，接上第 14 篇的统计槽位机制）。

### 15.5.2 操作符与 opclass：类型接入索引的桥

光有类型还查不快。三层桥接：

1. `CREATE OPERATOR` 把 `&&`（相交）这类符号绑定到函数，并声明 commutator/restrict 选择率估算器等优化器元数据；
2. `CREATE OPERATOR CLASS`（`DefineOpClass`，`commands/opclasscmds.c:333`）声明"某类型在某索引 AM 上如何工作"：一组策略号（操作符）+ 一组支持函数，写入 pg_opclass/pg_amop/pg_amproc；
3. 索引 AM 本身是 `IndexAmRoutine` 回调表（`src/include/access/amapi.h:233`），PG 9.6 起 AM 也可由扩展提供（如 bloom）。

**PostGIS 模式**就是这套机制的极致运用：`geometry` 类型（in/out 解析 WKT）+ 空间操作符（`&&` 等）+ **GiST opclass**。GiST 是"骨架树"：树的平衡、并发、WAL 由内核负责，扩展只需实现语义回调——`GIST_CONSISTENT_PROC`（查询是否可能命中子树，`access/gist.h:32`）、`GIST_UNION_PROC`（合并 bounding box，gist.h:33）、`GIST_PENALTY_PROC`（插入代价，gist.h:36）、`GIST_PICKSPLIT_PROC`（页分裂策略，gist.h:37）。PostGIS 用 bounding box 实现这四个函数，就免费获得了带并发控制和崩溃恢复的 R-Tree。**不改内核一行代码，得到一个空间数据库**——这是"平台性"最有说服力的证据。

### 15.5.3 过程语言（PL）框架与 SPI

PL 框架同样是"handler 模式"：`CREATE LANGUAGE` 注册一个 call handler；调用该语言写的函数时，fmgr 把控制交给 handler。PL/pgSQL 的 handler 是 `plpgsql_call_handler()`（`src/pl/plpgsql/src/pl_handler.c:224`）：编译函数体为内部指令树，逐语句解释执行。`src/pl/` 下同一模式还实现了 plperl/plpython/pltcl——理论上任何语言的解释器都能嵌进来。

PL 里的 SQL 语句靠 **SPI**（Server Programming Interface，`src/backend/executor/spi.c`）执行——它是"在服务端代码里执行 SQL"的官方 API：`SPI_connect()`（spi.c:95）建立会话、`SPI_execute()`（spi.c:597）跑一条 SQL、`SPI_prepare()`（spi.c:861）预编译、`SPI_finish()`（spi.c:183）清理。看 `SPI_execute` 的骨架：

```c
	memset(&plan, 0, sizeof(_SPI_plan));
	plan.magic = _SPI_PLAN_MAGIC;
	plan.parse_mode = RAW_PARSE_DEFAULT;
	plan.cursor_options = CURSOR_OPT_PARALLEL_OK;

	_SPI_prepare_oneshot_plan(src, &plan);
	...
	res = _SPI_execute_plan(&plan, &options, InvalidSnapshot, InvalidSnapshot, true);
```

内部就是完整地走一遍 parse → plan → execute 管线，只是发起者从客户端换成了服务端代码，结果落在 `SPI_tuptable` 里。SPI 还管理嵌套的内存上下文与快照——触发器、PL 函数、以及很多扩展（如定时任务扩展 pg_cron）都构建在它之上。

### 小结

类型（in/out）→ 操作符（语义）→ opclass（接入索引 AM 的支持函数）三级注册，让 PostGIS 无需改内核就获得工业级空间索引；PL handler + SPI 则把"用任意语言写存储过程、在服务端跑 SQL"也变成了插件问题。

## 15.6 为什么可扩展性是 PG 与 MySQL 生态分野的根源

回到 1986 年。Stonebraker 与 Rowe 的《The Design of POSTGRES》开篇就把目标定为支持"非传统应用"——CAD、地理、复杂对象——并给出了实现路线：**抽象数据类型（ADT）、用户定义的操作符和函数、可扩展的访问方法**（同期论文《Inclusion of New Types in Relational Data Base Systems》专门论证了这一点）。也就是说，"用户能定义类型并让优化器和索引理解它"不是 PG 后来长出的特性，而是**立项动机本身**。本篇讲的所有机制都是这个思想的直系后代：pg_type/pg_proc/pg_am 这些目录表从 Berkeley 时代就是"可写的注册表"而非只读的系统内部结构。

MySQL 走的是另一条路：它的扩展点是**存储引擎接口**（handler API）——在"表怎么存"这一层可插拔（MyISAM/InnoDB/RocksDB），但类型系统、优化器、执行器基本封闭；UDF 只是标量函数，进不了索引和统计。结果是两种生态形态：

- MySQL 的创新以 **fork 和魔改**形态出现（Percona、MariaDB、各大厂内核团队），与上游分叉、合并困难；
- PG 的创新以**扩展**形态出现（PostGIS、TimescaleDB、Citus、pgvector），跟随上游大版本平滑升级，一个内核团队的产出被整个生态复用。

工程上的因果链值得反复体会：**接口开放度决定创新成本，创新成本决定生态形态**。pgvector 从想法到可用只需要实现"类型 + 操作符 + 索引 AM"，赶上向量检索风口只用了很短时间——同样的事在封闭内核上意味着维护一个 fork。对想深入内核的工程师，这也是最佳练手场：写一个扩展，你就在真实内核 API 上写了生产级代码，而不用等社区 review 你对内核本体的改动。

## 自测问题

1. **一个扩展想同时统计所有会话的查询，为什么必须进 `shared_preload_libraries`？只用 `LOAD` 会缺什么？**
   简答：跨会话统计需要共享内存，而共享内存注册默认只在 postmaster 启动窗口开放（`RegisterShmemCallbacks`，shmem.c:873）；`LOAD` 只影响当前会话，hook 也只在本会话生效，其他 backend 的查询根本不会经过你的代码。pg_stat_statements 在 `_PG_init` 里检查 `process_shared_preload_libraries_in_progress`，不满足即放弃安装 hook（pg_stat_statements.c:400）。

2. **写 hook 函数时为什么要保存并转调 `prev_xxx`？pgss_ExecutorRun 里的 PG_TRY/PG_FINALLY 防的是什么？**
   简答：hook 是单个全局函数指针，不保存前值直接覆盖会"挤掉"先加载的扩展，链式转调才能共存。PG_FINALLY 保证查询出错（longjmp 到错误处理）时 `nesting_level` 计数也能正确回退，否则后续语句的顶层/嵌套判断全部错乱（pg_stat_statements.c:1017-1032）。

3. **`ALTER EXTENSION foo UPDATE TO '2.0'` 时不存在 `foo--1.0--2.0.sql`，内核怎么办？**
   简答：`get_ext_ver_list`（extension.c:1582）把所有 `A--B.sql` 建成版本有向图，`find_update_path`（extension.c:1688）做 Dijkstra 最短路，找到如 1.0→1.1→2.0 的链后逐段执行升级脚本。无连通路径才报错。

4. **postgres_fdw 判断一个 WHERE 条件能否下推的标准是什么？`WHERE ts > now() - interval '1 day'` 会被下推吗？**
   简答：`foreign_expr_walker`（deparse.c:312）要求表达式中所有操作符/函数是内建或白名单扩展的、稳定性足够、collation 可安全外推。`now()` 是 stable 函数——不能直接进远端 SQL 文本，但可以在本地求值后作为参数发送，实际以"本地先算成常量再下推"的方式处理；换成 volatile 函数（如 `random()`）则只能留在本地过滤。

5. **PostGIS 没有修改任何内核代码，它的 `&&` 操作符为什么能用上索引？请说出登记链条。**
   简答：`CREATE TYPE geometry`（in/out 函数入 pg_type）→ `CREATE OPERATOR &&`（入 pg_operator，带选择率估算器）→ `CREATE OPERATOR CLASS ... USING gist`（DefineOpClass，opclasscmds.c:333，把 `&&` 映射为策略号、注册 consistent/union/penalty/picksplit 支持函数入 pg_amop/pg_amproc）。优化器看到 `col && box` 时查 pg_amop 发现该操作符属于索引列的 opclass，即可生成 IndexPath；GiST 骨架（平衡/并发/WAL）由内核提供，PostGIS 只出语义回调。
