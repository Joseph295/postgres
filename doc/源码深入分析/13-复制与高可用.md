# 13 · 复制与高可用：从物理流复制到逻辑解码

> 对应《PostgreSQL 源码学习指南》第 15 章。代码基于本仓库 master 分支（20devel）。
> 所有 `文件:行号` 引用均在本仓库核实过；行号随版本漂移，以函数名为准。

## 本章导读

如果你做过 Kafka + Debezium 的 CDC 管道，这一章会让你产生强烈的既视感：

- **物理流复制** = 把整个 broker 的 log segment 逐字节镜像到另一台机器——备库不理解内容，只管"重放存储层"；
- **复制槽（replication slot）** = consumer group 的 offset 提交 + 一个"只要消费者还在，retention 就不许删我未消费数据"的强保证；
- **逻辑解码（logical decoding）** = 内核里内置的 Debezium：把 WAL（物理 redo 日志）解码成一条条 `INSERT/UPDATE/DELETE` 事件，按事务分组、按提交序输出——Debezium 的 PostgreSQL connector 底下就是直接消费这套机制的输出。

本章按数据流向走完两条链路：

1. 物理链路：`walsender → 复制协议 → walreceiver → 磁盘 → startup 进程 redo`，以及主库提交时的同步等待（syncrep）和备库查询与 redo 的冲突（Hot Standby）；
2. 逻辑链路：`WAL → decode.c → reorderbuffer.c（按事务重组）→ snapbuild.c（历史快照）→ 输出插件 → 网络 → 订阅端 apply worker`。逻辑解码是本章重点，占篇幅约一半。

## 前置知识

- **WAL 基础**（指南第 10 章）：LSN 是 WAL 的字节偏移（64 位，打印成 `%X/%08X`）；WAL 记录按 rmgr（resource manager）分类，如 Heap、Transaction、XLOG；
- **MVCC 与快照**（指南第 9 章）：xmin/xmax、快照决定元组可见性；`vacuum` 清理死元组的下限由"系统里最老的 xmin"决定；
- **进程模型**（指南第 1、14 章）：walsender 是一种特殊 backend；walreceiver、startup、logical replication worker 都是辅助进程；
- 大数据侧默认你熟悉：Kafka 的 log/offset/retention/consumer lag、Debezium 的 snapshot + streaming 两阶段、exactly-once 与幂等写。

## 13.1 物理流复制：把 WAL 字节流搬到另一台机器

### 13.1.1 主库侧：walsender 与 XLogSendPhysical

备库（或 `pg_receivewal` 等工具）连接主库时在连接串里带 `replication=true`，该 backend 就变身 walsender，进入复制命令循环。收到 `START_REPLICATION` 命令后由 `StartReplication()`（`src/backend/replication/walsender.c:860`）校验时间线并进入发送主循环 `WalSndLoop()`（`src/backend/replication/walsender.c:3046`）。物理复制时，循环体反复调用的发送回调是 `XLogSendPhysical()`（`src/backend/replication/walsender.c:3360`）：

```c
/* src/backend/replication/walsender.c:3359 起，节选 */
static void
XLogSendPhysical(void)
{
    ...
    /* Figure out how far we can safely send the WAL. */
    if (sendTimeLineIsHistoric)
    {
        /* 正在发送一条历史时间线，只能发到它被切走的那个点 */
        SendRqstPtr = sendTimeLineValidUpto;
    }
    else if (am_cascading_walsender)
    {
        /* 级联备库：只能发已经回放过（确认有效）的 WAL */
        SendRqstPtr = GetStandbyFlushRecPtr(&SendRqstTLI);
        ...
    }
    else
    {
        /* 主库：只发已经 fsync 到磁盘的 WAL */
        SendRqstPtr = GetFlushRecPtr(NULL);
    }
```

逐段解说：

- `sendTimeLineIsHistoric` 分支：备库请求的可能是一条**历史时间线**（见 13.1.4），只允许发送到时间线分叉点 `sendTimeLineValidUpto`，之后 walsender 会通知客户端切到下一条时间线；
- `am_cascading_walsender` 分支：级联复制（备库再带备库）时，上游只敢发**自己已经回放确认过**的 WAL——收到但未回放的字节还没被校验过；
- 主库分支的注释（walsender.c:3454 附近）值得背下来："**不能发送尚未安全落盘的 WAL**——如果主库随后崩溃重启，这些 WAL 会丢失，备库却已经应用了它们"。这就是 Kafka 里"consumer 只能读到 high watermark（已提交副本确认位置）"的同款约束：**发送上限 = 本地持久化下限**。

之后函数从 `startptr`（上次发送位置）读到 `SendRqstPtr`，每次最多 `MAX_SEND_SIZE` 字节，用 `WALRead()` 直接读 WAL 段文件塞进 libpq 输出缓冲。注意：**物理复制不解析 WAL 内容**，就是搬字节，因此开销极低——类比 Kafka 的 zero-copy sendfile 转发。

### 13.1.2 复制协议：'w'、'k'、'r'、'h' 四种消息

复制流上的消息类型定义在 `src/include/libpq/protocol.h:75-84`：

| 字节 | 常量 | 方向 | 含义 |
|---|---|---|---|
| `'w'` | `PqReplMsg_WALData` | 主→备 | WAL 数据（startLSN + endLSN + 时间戳 + 字节流） |
| `'k'` | `PqReplMsg_Keepalive` | 主→备 | 心跳，可要求立即回复 |
| `'r'` | `PqReplMsg_StandbyStatusUpdate` | 备→主 | feedback：write/flush/apply 三个 LSN |
| `'h'` | `PqReplMsg_HotStandbyFeedback` | 备→主 | 反馈备库 xmin（见 13.4） |

备库定期上报的 `'r'` 消息由主库 `ProcessStandbyReplyMessage()`（`src/backend/replication/walsender.c:2543`）处理：

```c
/* walsender.c:2561 起，节选 */
    writePtr = pq_getmsgint64(&reply_message);   /* 备库已写入内核缓冲的位置 */
    flushPtr = pq_getmsgint64(&reply_message);   /* 备库已 fsync 的位置 */
    applyPtr = pq_getmsgint64(&reply_message);   /* 备库已回放(redo)的位置 */
    ...
    SpinLockAcquire(&walsnd->mutex);
    walsnd->write = writePtr;                    /* 写进共享内存，供 syncrep/监控读取 */
    walsnd->flush = flushPtr;
    walsnd->apply = applyPtr;
```

这三个 LSN 就是 `pg_stat_replication` 里 `write_lsn/flush_lsn/replay_lsn` 的来源，也是 13.3 同步复制放行等待者的依据。物理槽存在时，该函数还会调用 `PhysicalConfirmReceivedLocation()`（walsender.c:2510，调用点 2645）用 flushPtr 推进槽的 `restart_lsn`——**feedback 消息同时就是 offset commit**。

### 13.1.3 备库侧：walreceiver 落盘 + startup 进程永续 redo

备库两个进程分工严格：

- **walreceiver**（`src/backend/replication/walreceiver.c`，主入口 `WalReceiverMain()`，walreceiver.c:154）：只管收字节、写文件、fsync、发 feedback。收到消息后由 `XLogWalRcvProcessMsg()`（walreceiver.c:842）分发：`'w'` 消息交给 `XLogWalRcvWrite()`（walreceiver.c:913）按 LSN 计算出目标 WAL 段文件和偏移写下去，段文件写满或消息处理完后 `XLogWalRcvFlush()`（walreceiver.c:1014）fsync 并更新共享内存中的 flush 位置、随即发送 reply（walreceiver.c:1060）；
- **startup 进程**：就是崩溃恢复用的那个 redo 进程，只是"永不结束"——回放完已有 WAL 就等 walreceiver 通知有新数据。备库的一致性完全由"单进程按 LSN 顺序 redo"保证，与崩溃恢复共用同一套回放代码。

feedback 的构造在 `XLogWalRcvSendReply()`（walreceiver.c:1137）：

```c
/* walreceiver.c:1181 起 */
    resetStringInfo(&reply_message);
    pq_sendbyte(&reply_message, PqReplMsg_StandbyStatusUpdate);  /* 'r' */
    pq_sendint64(&reply_message, writePtr);
    pq_sendint64(&reply_message, flushPtr);
    pq_sendint64(&reply_message, applyPtr);
    pq_sendint64(&reply_message, GetCurrentTimestamp());
    pq_sendbyte(&reply_message, requestReply ? 1 : 0);
```

apply 位置来自 `GetXLogReplayRecPtr()`——即 startup 进程回放到哪了。**写盘的人（walreceiver）和回放的人（startup）解耦**，所以 flushPtr 通常领先 applyPtr，这个差值就是备库的 replay lag。

### 13.1.4 时间线（timeline）：防止"脑裂历史"的世代号

每次备库被提升（promote）为主库，时间线 ID 加一，并写出一个 `.history` 文件记录"我从 TLI=n 的哪个 LSN 分叉"。作用类比 Kafka controller epoch / Raft term：**旧主复活后产生的 WAL 与新主的 WAL 在同一 LSN 区间可能内容不同，时间线号使两段历史可区分、可拒绝**。

`XLogSendPhysical()` 里能看到时间线切换的处理（walsender.c:3432 起）：当发送中的时间线变为历史（比如级联上游被提升），walsender 读取新时间线的 history 文件，用 `tliSwitchPoint()` 算出分叉点，只把旧时间线发到分叉点为止，之后客户端需要用新 TLI 重新发起 `START_REPLICATION`。备库端 `recovery_target_timeline=latest`（默认）会自动跟随切换。

### 小结

- walsender 是"不解析内容的字节搬运工"，发送上限被钉死在主库 fsync 位置——持久化前的数据绝不外流；
- 复制协议核心是 `'w'` 数据 + `'r'` feedback；feedback 的 write/flush/apply 三元组同时服务于监控、同步复制和槽位点推进；
- 备库 = walreceiver（收+落盘）+ startup（永续 redo），一致性由单线程按序回放保证；
- 时间线是复制拓扑的"世代号"，用于在故障切换后区分分叉的历史。

## 13.2 复制槽（slot.c）：带强保证的 consumer offset

### 13.2.1 槽的持久化状态：四个关键字段

槽的核心状态在 `ReplicationSlotPersistentData`（`src/include/replication/slot.h:95`）：

```c
/* src/include/replication/slot.h:95 起，节选 */
typedef struct ReplicationSlotPersistentData
{
    NameData    name;
    Oid         database;          /* 逻辑槽绑定单个数据库；物理槽为 InvalidOid */
    ...
    TransactionId xmin;            /* 数据的 xmin 下限（物理槽+hs_feedback 用） */
    TransactionId catalog_xmin;    /* 目录元组的 xmin 下限（逻辑槽用） */
    XLogRecPtr  restart_lsn;       /* 本槽可能还需要的最老 WAL 位置 */
    ...
    XLogRecPtr  confirmed_flush;   /* 客户端已确认收到的最老 LSN（仅逻辑槽） */
```

对照 Kafka 语义：

| PostgreSQL | Kafka/CDC 对应物 | 语义 |
|---|---|---|
| `restart_lsn` | "为该 consumer 保留的最老 segment 偏移" | **重启解码/复制必须从这里重读 WAL**。逻辑槽上它通常落后于 confirmed_flush——因为重组进行中的长事务需要回到其起点之前的快照位置（见 13.5.3 的 `restart_decoding_lsn`） |
| `confirmed_flush` | committed offset | 客户端确认"这之前的变更我已安全接收"，`START_REPLICATION` 不指定起点时从这里续传（slot.h:131 注释） |
| `xmin` / `catalog_xmin` | 无直接对应（Kafka 没有 MVCC） | 阻止 vacuum 清掉本槽还需要的（目录）元组版本 |
| 槽本身 | consumer group + retention 豁免 | 区别在于：**Kafka retention 到期照删不误，PG 的槽默认无限期扣住 WAL** |

物理槽/逻辑槽的分野：`ReplicationSlotCreate()`（`src/backend/replication/slot.c:378`）的 `db_specific` 参数——逻辑槽必须绑定数据库（因为解码要读该库的系统目录），物理槽全局有效且只用 `restart_lsn`/`xmin`，不用 `confirmed_flush`/`catalog_xmin`。使用前需 `ReplicationSlotAcquire()`（slot.c:629）独占获取，同一槽同一时刻只能有一个消费者——等价于 Kafka 一个分区同组内只给一个 consumer。

### 13.2.2 槽如何扣住 WAL：KeepLogSeg

checkpoint 决定回收哪些 WAL 段时调用 `KeepLogSeg()`（`src/backend/access/transam/xlog.c:8498`）：

```c
/* xlog.c:8498 起，节选 */
static void
KeepLogSeg(XLogRecPtr recptr, XLogSegNo *logSegNo)
{
    ...
    /* Calculate how many segments are kept by slots. */
    keep = XLogGetReplicationSlotMinimumLSN();      /* 所有槽 restart_lsn 的最小值 */
    if (XLogRecPtrIsValid(keep) && keep < recptr)
    {
        XLByteToSeg(keep, segno, wal_segment_size); /* 回收线退到该槽所在段 */

        /* 但最多为槽保留 max_slot_wal_keep_size */
        if (max_slot_wal_keep_size_mb >= 0 && !IsBinaryUpgrade)
        {
            slot_keep_segs = ConvertToXSegs(max_slot_wal_keep_size_mb, ...);
            if (currSegNo - segno > slot_keep_segs)
                segno = currSegNo - slot_keep_segs;  /* 超限：不再迁就槽 */
        }
    }
```

- `XLogGetReplicationSlotMinimumLSN()`（xlog.c:2699）读的是 `ReplicationSlotsComputeRequiredLSN()`（slot.c:1304）汇总到共享内存的**全体槽 restart_lsn 最小值**——一个掉队的槽就能拖住全局回收线，木桶效应；
- `max_slot_wal_keep_size`（xlog.c:142，默认 -1 即不设限）是唯一的保险丝：槽落后超过该值时，checkpoint 照常回收 WAL，随后该槽被标记为 invalidated（`slot.h:128` 的 `invalidated` 字段，原因 `RS_INVAL_WAL_REMOVED`）——等价于 Kafka consumer 掉出 retention 窗口后的 `OffsetOutOfRange`：消费者必须重建（对 CDC 意味着重新全量快照）。

### 13.2.3 catalog_xmin：槽还会扣住 vacuum

逻辑槽的另一只手伸向 MVCC：`ReplicationSlotsComputeRequiredXmin()`（slot.c:1222）汇总全体槽的 xmin/catalog_xmin 写入 ProcArray（`src/backend/storage/ipc/procarray.c` 的 `replication_slot_catalog_xmin`，计算可清理视界时在 procarray.c:2358 附近参与取最小值）。效果：**系统目录（pg_class、pg_attribute…）中比 catalog_xmin 新的死元组不会被 vacuum 掉**——因为解码老 WAL 需要"当时"的表结构（13.5.4 详述）。

### 13.2.4 运维故障模式：槽泄漏打爆磁盘

大数据工程师最熟悉的事故在 PG 上的版本：

1. **症状链**：下游（Debezium connector、备库）停止消费但槽没删 → `restart_lsn` 不动 → `pg_wal` 目录只增不减 → 磁盘 100% → 主库 PANIC 拒绝写入。对应 Kafka 里"消费者死了但 retention 是永久"——Kafka 默认不会这样，**PG 槽默认会**；
2. 附带伤害：逻辑槽的 `catalog_xmin` 不动 → 系统目录膨胀、全库 vacuum 无法推进老化视界，长期还会逼近 wraparound；
3. 监控口径：`pg_replication_slots` 里 `active=false` 且 `restart_lsn` 长期不前进的槽；`wal_status` 列（reserved/extended/unreserved/lost）直接告诉你槽的 WAL 是否已被 `max_slot_wal_keep_size` 牺牲；
4. 处方：生产必设 `max_slot_wal_keep_size`（宁可让 CDC 重新快照，不可让主库停写）；下线订阅方时**必须** `pg_drop_replication_slot()`——删 connector 不删槽是 Debezium 使用者的经典事故。

### 小结

- 槽 = 持久化的消费位点（restart_lsn/confirmed_flush）+ 对 WAL 回收（KeepLogSeg）和 vacuum（catalog_xmin）的双重反压；
- 与 Kafka 的本质差异：PG 默认"数据为消费者无限等待"，Kafka 默认"retention 到期数据必删"；`max_slot_wal_keep_size` 把 PG 拉回 Kafka 模式;
- 槽泄漏 = 磁盘打爆 + 目录膨胀，是物理/逻辑复制共同的头号运维风险。

## 13.3 同步复制（syncrep.c）：提交路径上的分布式 ack

### 13.3.1 synchronous_commit 五档在代码中的分叉点

GUC 值到等待级别的映射在 `assign_synchronous_commit()`（`src/backend/replication/syncrep.c:1130`），级别常量在 `src/include/access/xact.h:71-81`：

| GUC 值 | 枚举 | 本地行为 | 等待备库？ |
|---|---|---|---|
| `off` | `SYNCHRONOUS_COMMIT_OFF` | 提交不等本地 WAL fsync（由 walwriter 异步刷） | 否 |
| `local` | `..._LOCAL_FLUSH` | 等本地 fsync | 否 |
| `remote_write` | `..._REMOTE_WRITE` | 等本地 fsync | 等备库 write（进 OS，未 fsync）→ `SYNC_REP_WAIT_WRITE` |
| `on`（默认） | `..._REMOTE_FLUSH` | 等本地 fsync | 等备库 flush（fsync）→ `SYNC_REP_WAIT_FLUSH` |
| `remote_apply` | `..._REMOTE_APPLY` | 等本地 fsync | 等备库 apply（回放可见）→ `SYNC_REP_WAIT_APPLY` |

注意 `on` 只有在配置了 `synchronous_standby_names` 时才产生远程等待；否则五档中前四档只影响**本地 fsync 时机**。分叉点一：`off` 在 `RecordTransactionCommit()`（`src/backend/access/transam/xact.c:1345`）里走异步提交路径（不立刻 XLogFlush）。分叉点二：提交末尾调用 `SyncRepWaitForLSN(XactLastRecEnd, true)`（xact.c:1599），远程等待从这里开始。

### 13.3.2 SyncRepWaitForLSN：把自己挂进按 LSN 排序的等待队列

`SyncRepWaitForLSN()`（syncrep.c:149）核心片段：

```c
/* syncrep.c:184 起，节选 */
    /* Cap the level for anything other than commit to remote flush only. */
    if (commit)
        mode = SyncRepWaitMode;
    else
        mode = Min(SyncRepWaitMode, SYNC_REP_WAIT_FLUSH);
    ...
    if (WalSndCtl->sync_standbys_status & SYNC_STANDBY_INIT)
    {
        if ((WalSndCtl->sync_standbys_status & SYNC_STANDBY_DEFINED) == 0 ||
            lsn <= WalSndCtl->lsn[mode])
        {   /* 备库已确认到该 LSN，或根本没配同步备库：直接放行 */
            LWLockRelease(SyncRepLock);
            return;
        }
    }
    ...
    MyProc->waitLSN = lsn;
    MyProc->syncRepState = SYNC_REP_WAITING;
    SyncRepQueueInsert(mode);                     /* 按 LSN 有序插入等待队列 */
    Assert(SyncRepQueueIsOrderedByLSN(mode));
    LWLockRelease(SyncRepLock);
    for (;;) { ResetLatch(MyLatch); ... /* latch 等待被唤醒 */ }
```

- 非 commit 记录（如 prepare）最多等到 flush 级——只有 commit 记录才有 apply feedback（syncrep.c:143 注释）；
- 三个等待模式各有一条队列（`WalSndCtl->SyncRepQueue[mode]`），`SyncRepQueueInsert()`（syncrep.c:382）按 waitLSN 从小到大插入。**队列有序**使得放行是 O(被放行者)：从队头摘到第一个 waitLSN 超过确认位置的元素为止；
- 唤醒方是 walsender：收到 feedback 后 `SyncRepReleaseWaiters()`（syncrep.c:484）重算三个模式的"已确认 LSN"，推进 `WalSndCtl->lsn[mode]` 并 set latch 唤醒达标的 backend。

这就是两阶段的 ack 管道：backend（生产者）阻塞在 latch 上，walsender（ack 收集器）根据备库 feedback 批量放行——和 Kafka producer `acks=all` 等 ISR 确认同构。

### 13.3.3 quorum（ANY n）语义

`synchronous_standby_names` 两种语法对应两种策略（常量在 `src/include/replication/syncrep.h:35-36`）：

- `FIRST n (s1, s2, ...)`（`SYNC_REP_PRIORITY`）：按列表顺序取优先级最高的 n 个，确认 LSN = 这 n 个的**最小值**；
- `ANY n (s1, s2, ...)`（`SYNC_REP_QUORUM`）：任意 n 个确认即可，确认 LSN = 全体候选中**第 n 大**的值，由 `SyncRepGetNthLatestSyncRecPtr()`（syncrep.c:702）计算：

```c
/* syncrep.c:721 起，节选 */
    for (i = 0; i < num_standbys; i++)
    {
        write_array[i] = sync_standbys[i].write;
        flush_array[i] = sync_standbys[i].flush;
        apply_array[i] = sync_standbys[i].apply;
    }
    /* Sort each array in descending order */
    qsort(write_array, num_standbys, sizeof(XLogRecPtr), cmp_lsn);
    ...
    *writePtr = write_array[nth - 1];   /* 降序第 n 个 = 至少 n 个备库已达到的位置 */
```

降序排序后取第 `n` 个元素——"至少有 n 个备库到达了这个 LSN"。这就是教科书 quorum ack。但务必记住指南的提醒：**syncrep 只做 quorum ack，不做选主、不做日志truncate 协商**——故障切换的正确性（选最新的备库、fence 旧主）要靠 Patroni 等外部编排，PG 内核不是 Raft。

### 小结

- `synchronous_commit` 的五档分成两类：本地 fsync 时机（off/local）与远程等待级别（write/flush/apply），远程等待入口是 `SyncRepWaitForLSN`；
- 等待队列按 LSN 有序，walsender 依据 feedback 推进水位并批量唤醒；
- `ANY n` 用"降序第 n 个 LSN"实现 quorum；同步复制不等于自动故障切换。

## 13.4 Hot Standby：redo 与只读查询抢地盘

### 13.4.1 冲突从哪来

备库查询持有快照/锁/buffer pin，而 redo 必须原样重放主库的动作，两者会顶牛。`src/backend/storage/ipc/standby.c` 按冲突类型提供一组 `ResolveRecoveryConflictWith*` 函数：

| 冲突类型 | 触发场景 | 处理函数（standby.c） |
|---|---|---|
| Snapshot | 主库 vacuum/HOT 清理了备库查询还"看得见"的死元组 | `ResolveRecoveryConflictWithSnapshot`（470） |
| Lock | redo 要拿 AccessExclusiveLock（如主库 DROP TABLE），备库查询持有表锁 | `ResolveRecoveryConflictWithLock`（626） |
| BufferPin | redo 要清理的页被备库查询 pin 住 | `ResolveRecoveryConflictWithBufferPin`（795） |
| Tablespace/Database | 主库删了表空间/数据库 | 542 / 572 |

最常见的是 snapshot 冲突：

```c
/* standby.c:469 起，节选 */
void
ResolveRecoveryConflictWithSnapshot(TransactionId snapshotConflictHorizon,
                                    bool isCatalogRel,
                                    RelFileLocator locator)
{
    ...
    if (!TransactionIdIsValid(snapshotConflictHorizon))
        return;                              /* 该记录不影响可见性，无冲突 */

    backends = GetConflictingVirtualXIDs(snapshotConflictHorizon,
                                         locator.dbOid);
    ResolveRecoveryConflictWithVirtualXIDs(backends,
                                           RECOVERY_CONFLICT_SNAPSHOT, ...);
```

主库在清理类 WAL 记录里附带一个 `snapshotConflictHorizon`（"我删掉的元组对 xmin ≤ 此值的快照可能仍可见"）；备库回放时找出所有 xmin 冲突的会话，先等、等不起就杀。standby.c:505 还有一个容易忽略的分支：若被清理的是**目录表**且备库上开着逻辑解码，会直接把 catalog_xmin 落后的逻辑槽标记失效（`RS_INVAL_HORIZON`）——目录版本没了，解码无法继续。

### 13.4.2 等多久才动手：max_standby_streaming_delay

"等 vs 杀"的时限由 `GetStandbyLimitTime()`（standby.c:202）决定：

```c
/* standby.c:202 起，节选 */
static TimestampTz
GetStandbyLimitTime(void)
{
    TimestampTz rtime;
    bool        fromStream;
    GetXLogReceiptTime(&rtime, &fromStream);
    if (fromStream)
    {
        if (max_standby_streaming_delay < 0)
            return 0;                                /* -1：无限等，永不杀查询 */
        return TimestampTzPlusMilliseconds(rtime, max_standby_streaming_delay);
    }
```

`max_standby_streaming_delay`（standby.c:42，默认 30 秒）限定的是**redo 允许积累的总延迟**而非单个查询的宽限：基准是"最近一批 WAL 的接收时间"，redo 已经落后很多时，新冲突的查询会立刻被取消。取消通过给目标 backend 发 `PROCSIG_RECOVERY_CONFLICT_*` 信号，对方在下一个中断点抛出 `ERROR: canceling statement due to conflict with recovery`。权衡是经典的两难：调大 → 备库长查询存活但复制延迟涨；调小 → 延迟稳定但报表查询频繁被杀。

### 13.4.3 hot_standby_feedback：把 xmin 反向传播回主库

治本方案之一是让主库"别清那么快"。开启 `hot_standby_feedback`（walreceiver.c:92）后，walreceiver 周期性调用 `XLogWalRcvSendHSFeedback()`（walreceiver.c:1210）发送 `'h'` 消息，携带备库当前的全局 xmin（和 catalog_xmin）。主库 walsender 把它写进自己 PGPROC 或物理槽的 `xmin`（`ProcessStandbyHSFeedbackMessage()`，walsender.c 内），从此主库 vacuum 的可清理视界要**同时迁就备库上最老的查询**。

副作用必须讲清楚：这等于把备库长事务的膨胀效应传导回主库——备库跑 8 小时的分析查询，主库 8 小时清不掉死元组，表和索引一起膨胀。它防的是 snapshot 冲突，防不了 lock/bufferpin 冲突（DROP TABLE 该杀还是杀）。大数据类比：consumer 反压直接冻结上游 compaction。

### 小结

- Hot Standby 冲突本质是"redo 必须重放主库的清理动作"与"备库快照还需要旧版本"的矛盾，按 snapshot/lock/bufferpin 等分类处理；
- `max_standby_streaming_delay` 限制 redo 总延迟，超时按信号取消冲突查询；
- `hot_standby_feedback` 用"反向传播 xmin"换取备库查询稳定，代价是主库膨胀。

## 13.5 逻辑解码：内核里的 Debezium（本章重点）

### 13.5.0 全景与 CDC 对照

```
WAL 段文件
  │  XLogReaderState 逐条读取
  ▼
decode.c        LogicalDecodingProcessRecord: 按 rmgr 分发，物理记录 → ReorderBufferChange
  ▼
reorderbuffer.c 按 xid 聚合交错的变更；超内存落盘/流式下发；commit 时按 LSN 序回放
  ▼               ▲
snapbuild.c     历史目录快照（告诉解码器"当时"的表结构长什么样）
  ▼
logical.c       回调框架 → 输出插件（pgoutput / test_decoding / wal2json）
  ▼
walsender       逻辑 START_REPLICATION，同一条复制协议
  ▼
worker.c        订阅端 apply worker：回放到本地表
```

与 Debezium/Kafka Connect 管道逐层对照：

| PostgreSQL 逻辑复制 | Debezium/Kafka 管道 |
|---|---|
| WAL | MySQL binlog / 源库事务日志 |
| decode.c + reorderbuffer.c + snapbuild.c | Debezium connector 内部的 log parser + transaction buffer |
| 输出插件（pgoutput）| SourceRecord 序列化（Avro/JSON converter） |
| 逻辑槽 confirmed_flush | Kafka Connect offset topic 里的 source offset |
| walsender 逻辑流 | Kafka topic（但 PG 是拉起点由槽记、推模式单订阅者） |
| tablesync.c 初始全量 + 增量续传 | Debezium initial snapshot + streaming 无缝衔接 |
| worker.c apply（含并行 apply）| sink connector / consumer |
| origin（复制来源标记，防回环） | Kafka header 里的 source 标记防循环复制 |

关键差异：Debezium 把"解析日志"放在外部进程、位点存 Kafka；PG 把整条解码管道放进内核、位点存槽。Debezium PG connector 本质上只是一个逻辑复制客户端：创建逻辑槽 + `START_REPLICATION` 消费 pgoutput 流。

### 13.5.2 reorderbuffer.c：把交错的 WAL 重组成"整事务、按提交序"

WAL 里各并发事务的记录是交错的（append-only 全局日志），而下游想要的是 Debezium 那样"事务边界清晰、按提交顺序"的事件流。reorderbuffer 就是那层 transaction buffer。

#### 核心数据结构 ReorderBufferTXN 走读

`src/include/replication/reorderbuffer.h:293-468`，按功能分组读：

#### 内存超限：落盘（spill）还是流式（streaming）

每次入队末尾都检查 `ReorderBufferCheckMemoryLimit()`（reorderbuffer.c:3889）：

- 总预算 `logical_decoding_work_mem`（reorderbuffer.c:226，默认 64MB）覆盖该解码会话全部在途事务；
- **spill 路径** `ReorderBufferSerializeTXN()`（reorderbuffer.c:3990）：把最大事务的变更写到 `pg_replslot/<slot>/` 下的临时文件（按 xid+LSN 段命名），内存清零；commit 时再读回。对照 Kafka Streams 状态超限落 RocksDB；监控指标 `pg_stat_replication_slots` 的 `spill_txns/spill_bytes`；
- **streaming 路径** `ReorderBufferStreamTXN()`（reorderbuffer.c:4335）：若输出插件支持 stream 回调且订阅端声明 `streaming=on`，**未提交的大事务**被切成块提前发给下游（`stream_start/stream_change/stream_stop`），提交时发 `stream_commit`、中止发 `stream_abort` 让下游回滚——这相当于 CDC 管道里"发未提交数据 + 补偿消息"，下游必须能暂存或回滚（PG 订阅端用临时文件缓存或 parallel apply worker 直接应用）。

#### 提交回放：按 LSN 归并多路有序流

`DecodeCommit` → `ReorderBufferCommit()`（reorderbuffer.c:2880）→ `ReorderBufferReplay()`（reorderbuffer.c:2819）→ `ReorderBufferProcessTXN()`（reorderbuffer.c:2204）。此时顶层事务与各子事务各有一条按 LSN 有序的 changes 链（部分可能在磁盘上），`ReorderBufferIterTXNInit()`（reorderbuffer.c:1285）为每条链建游标，用**二叉堆按当前变更 LSN 做 k 路归并**，逐条回调输出插件的 `change_cb`——所以下游看到的单事务内部顺序 = 主库执行顺序，事务之间 = 提交顺序（单槽全局有序，比 Kafka 分区内有序更强，但也因此**单槽吞吐没有并行度**——这是逻辑复制延迟的结构性上限，对策是按库/表拆多个 publication+槽，等价于 Kafka 加分区）。

### 13.5.3 snapbuild.c：为什么解码需要"历史目录快照"

解码 WAL 里的元组字节需要知道**写这条 WAL 时**表长什么样（列数、类型、名字都在系统目录里）。但目录是活的：今天解上周的 WAL 时表可能已经 `ALTER` 过。解法是 MVCC 自举：目录的旧版本元组还在（被槽的 `catalog_xmin` 保护着，见 13.2.3），只要构造出"当时"的快照就能读到"当时"的目录。snapbuild.c 负责从 WAL 流里增量构建这种历史快照。

难点：从任意 LSN 开始读 WAL，看不到已在途事务的开头，不知道谁在跑。状态机（`src/include/replication/snapbuild.h:35-58`）四阶段解决：

```
START ──(遇到 xl_running_xacts)──▶ BUILDING_SNAPSHOT ──(初始在途事务全部结束)──▶
FULL_SNAPSHOT ──(FULL 时刻的在途事务也全部结束)──▶ CONSISTENT
```

转移逻辑集中在 `SnapBuildFindSnapshot()`（`src/backend/replication/logical/snapbuild.c:1243`）：

```c
/* snapbuild.c:1299 与 1349/1383/1407 起，节选 */

    {   /* 捷径 a：此刻无任何在途事务，直接 CONSISTENT */
        builder->start_decoding_at = lsn + 1;
        builder->xmin = builder->xmax = running->nextXid;
        builder->state = SNAPBUILD_CONSISTENT;
        ...
    }
    else if (... SnapBuildRestore(builder, lsn)) { ... }  /* 捷径 b：磁盘上有序列化快照 */
    else if (builder->state == SNAPBUILD_START)
    {   /* c1: START → BUILDING_SNAPSHOT，记下 nextXid 作为毕业线 */
        builder->state = SNAPBUILD_BUILDING_SNAPSHOT;
        builder->next_phase_at = running->nextXid;
        ...
    }

             TransactionIdPrecedesOrEquals(builder->next_phase_at,
                                           running->oldestRunningXid))
    {   /* c2: 初始在途事务全结束 → FULL_SNAPSHOT */
        builder->state = SNAPBUILD_FULL_SNAPSHOT;
        builder->next_phase_at = running->nextXid;
        ...
    }
    else if (builder->state == SNAPBUILD_FULL_SNAPSHOT && ...)
    {   /* c3: FULL 时刻的在途事务也全结束 → CONSISTENT */
        builder->state = SNAPBUILD_CONSISTENT;
```

为什么要两级等待？

- **BUILDING → FULL**：从第一个 `xl_running_xacts` 记录起跟踪所有目录修改事务的提交。等"当时在途"的事务全部结束后，**此后新启动的事务**都能被完整观察，可以正确解码——但更老的在途事务的变更没头没尾，仍不能解；
- **FULL → CONSISTENT**：再等一代，让 FULL 时刻的在途事务也全部终结，此后**任何提交的事务**的全部变更都已被收集，输出流不会出现"半个事务"。`start_decoding_at` 之前解出的变更只积累不下发。

这正是 Debezium "snapshot 阶段结束后才切 streaming、保证不丢不重"的镜像问题：Debezium 靠锁/一致性读点位对齐，PG 靠等两代在途事务自然死亡。等待可能很久（有长事务时），所以创建逻辑槽卡在 `catchup` 状态的第一排查项就是长事务。CONSISTENT 后快照会周期性序列化到 `pg_logical/snapshots/`（`SnapBuildSerializationPoint`，见 decode.c:143 在 shutdown checkpoint 时触发），重启时经捷径 b `SnapBuildRestore` 直接恢复，免去重新等待。

```c
    LogicalDecodeShutdownCB shutdown_cb;
    /* two-phase 支持 */
    LogicalDecodeFilterPrepareCB filter_prepare_cb;
    ... prepare_cb / commit_prepared_cb / rollback_prepared_cb ...
    /* 大事务流式解码支持 */
    ... stream_start_cb / stream_change_cb / stream_commit_cb / stream_abort_cb ...
} OutputPluginCallbacks;
```

框架侧 `logical.c` 负责组装解码上下文：`CreateInitDecodingContext()`（`src/backend/replication/logical/logical.c:322`，建槽时）/`CreateDecodingContext()`（logical.c:491，续传时）把 XLogReader、reorderbuffer、snapbuild 串起来，并把 reorderbuffer 的函数指针接到 `*_cb_wrapper` 系列（如 logical.c:213 的 `ctx->reorder->apply_change = change_cb_wrapper`）——wrapper 里设置错误上下文、检查快照状态后才进插件。客户端确认位点时 `LogicalConfirmReceivedLocation()`（logical.c:1813）推进 `confirmed_flush` 并择机推进 `restart_lsn`/`catalog_xmin`——**逻辑槽的 offset commit 最终落在这里**。

内置插件 `pgoutput`（`src/backend/replication/pgoutput/pgoutput.c`）在 `_PG_output_plugin_init()`（pgoutput.c:261）填表，`pgoutput_change()`（pgoutput.c:1484）按 publication 过滤表/行/列后，用 `src/backend/replication/logical/proto.c` 的二进制格式编码成逻辑复制协议消息。第三方插件（wal2json、test_decoding、Debezium 早年用的 decoderbufs）只是换一种序列化格式，机制完全相同；Debezium 现在默认直接消费 pgoutput。

### 13.5.5 订阅端：worker.c 的 apply 循环

订阅端 `CREATE SUBSCRIPTION` 后，launcher 启动 apply worker（入口 `ApplyWorkerMain()`，`src/backend/replication/logical/worker.c:6003`），连上发布端 walsender 开逻辑流，进入 `LogicalRepApplyLoop()`（worker.c:4003）：

```c
/* worker.c:4053 起，节选 */
        len = walrcv_receive(LogRepWorkerWalRcvConn, &buf, &fd);
        if (len != 0)
        {
            for (;;)        /* 处理完本轮所有已到数据 */
            {
                ...
                c = pq_getmsgbyte(&s);
                if (c == PqReplMsg_WALData)          /* 'w'：内嵌逻辑消息 */
                {
                    start_lsn = pq_getmsgint64(&s);
                    end_lsn = pq_getmsgint64(&s);
                    send_time = pq_getmsgint64(&s);
                    if (last_received < end_lsn)
                        last_received = end_lsn;     /* 推进本地位点 */
                    apply_dispatch(&s);              /* 按消息类型分发应用 */
                }
```

- 传输层复用物理复制协议（`'w'`/`'k'` 外壳），payload 是 pgoutput 编码的逻辑消息；`apply_dispatch()`（worker.c:3797）按首字节分发：BEGIN/COMMIT/INSERT/UPDATE/DELETE/TRUNCATE/RELATION（表结构元数据）/STREAM_* 等；
- `apply_handle_insert()`（worker.c:2650）走**执行器**写本地表：构造 slot、找 ResultRelInfo、触发本地索引/约束/（默认不触发的）触发器——所以订阅端是"重放 SQL 级效果"而不是"重放字节"，目标表可以有不同索引、不同分区方式；UPDATE/DELETE 按 REPLICA IDENTITY（默认主键）定位行；
- apply worker 也给发布端回 feedback 推进槽的 `confirmed_flush`；应用位置还写入本地 `replication origin`，崩溃重启后从 origin 记录的位点续订——**位点的最终真相在订阅端 origin，槽的 confirmed_flush 是它的影子**，这套"消费端记 offset + 服务端留底"与 Kafka Connect 的 offset 管理同构；
- 初始同步：`tablesync.c` 为每张表起 tablesync worker，用同一槽机制导出一致性快照 `COPY` 存量数据，再追增量到与主 apply 流汇合——Debezium 的 snapshot→streaming 切换在 PG 里是内核原生的。

**限制**（大数据同学做管道设计必须知道）：

1. **DDL 不复制**：逻辑流里只有行变更，发布端 `ALTER TABLE ADD COLUMN` 不会传播，需先在订阅端手工执行（Debezium 处理 schema change 也同样是最痛的部分）；
2. **序列**：`nextval` 的推进不进逻辑流。本版本（20devel）新增了 `sequencesync.c`（`src/backend/replication/logical/sequencesync.c`）：序列被登记进 `pg_subscription_rel`，由专门的 sequencesync worker **周期性同步当前值**（`REFRESH SEQUENCES` 触发重同步）——是"定期对账"而非"逐次增量"，切换前仍建议校验序列值；
3. 大对象（large object）不复制；TRUNCATE 需 publication 显式包含；双向复制要靠 origin 过滤防回环。

### 小结

- decode.c 是"翻译层"：按 rmgr 把物理 WAL 记录翻成 `ReorderBufferChange`，不关心事务边界与表结构；
- reorderbuffer.c 是"事务缓冲层"：`ReorderBufferTXN` 按 xid 聚合有序变更链，超过 `logical_decoding_work_mem` 时挑最大事务流式下发或落盘，提交时 k 路归并按 LSN 回放，输出流"事务完整、按提交序、默认无 abort"；
- snapbuild.c 用 START→BUILDING→FULL→CONSISTENT 四阶段等出一个能做目录时间旅行的一致快照——这是"今天解昨天的 WAL"的前提，也是建槽慢的常见原因；
- 输出插件是可插拔的序列化层，pgoutput 是内置实现；订阅端 worker.c 用执行器重放并以 origin 记录位点；
- 整条链路即内核态 Debezium：槽=offset，snapbuild=snapshot 一致性点，streaming=未提交事务提前下发，限制集中在 DDL 与序列。

## 自测问题

**1. 为什么 `XLogSendPhysical()` 在主库上只发送已 fsync 的 WAL？如果发送了未落盘的字节会怎样？**

答：见 walsender.c:3454 附近注释。若主库把未 fsync 的 WAL 发给备库后崩溃，重启后主库会从最后落盘点继续写出**内容不同**的 WAL，而备库已回放了丢失版本——两边历史分叉且无法察觉。因此发送上限被钉在 `GetFlushRecPtr()`。这等价于 Kafka consumer 只能读到 high watermark 的原因。

**2. 一个逻辑槽的 `confirmed_flush` 一直在推进，但 `restart_lsn` 停住不动，最可能的原因是什么？会有什么后果？**

答：发布端存在长事务。`restart_lsn` 由"最老在途事务的 `restart_decoding_lsn`"决定（snapbuild.c:1212-1219）——只要该事务不结束，重启解码就必须回到它开始前的快照点重读。后果：WAL 回收线（`KeepLogSeg`，xlog.c:8498）被扣住，`pg_wal` 膨胀；`catalog_xmin` 同样停滞导致目录膨胀。超过 `max_slot_wal_keep_size` 后槽会被判 invalidated。

**3. `synchronous_commit = remote_write` 和 `on` 在丢数据风险上差在哪？代码上对应哪两个等待模式？**

答：`remote_write`（`SYNC_REP_WAIT_WRITE`）只等备库把 WAL 写进内核页缓存；`on`（`SYNC_REP_WAIT_FLUSH`）等备库 fsync。前者在"主库故障 + 备库同时断电"的双故障下会丢已确认事务，后者只要备库盘还在就不丢。映射见 `assign_synchronous_commit()`（syncrep.c:1130），等待入口 `SyncRepWaitForLSN()`（syncrep.c:149）。

**4. 开启 `hot_standby_feedback` 能否避免备库上所有查询取消？为什么？**

答：不能。它只消除 **snapshot 冲突**（主库 vacuum 会迁就备库 xmin，见 walreceiver.c:1210 的 `'h'` 消息）；lock 冲突（主库 DROP/ALTER 需要排他锁）和 bufferpin 冲突仍会按 `max_standby_streaming_delay` 超时取消查询（standby.c:626/795）。代价是备库长查询把膨胀反传到主库。

**5. snapbuild 为什么不能在遇到第一个 `xl_running_xacts` 后直接开始向下游输出变更？FULL_SNAPSHOT 和 CONSISTENT 的区别是什么？**

答：从任意 LSN 开始读 WAL 时，已在途事务的前半段变更没读到，且缺少能看清"当时目录"的快照。BUILDING 阶段等初始在途事务全部结束后进入 FULL_SNAPSHOT：**此后新启动的事务**可被正确解码，但 FULL 时刻仍在途的老事务的变更依然残缺；再等这批事务全部终结才到 CONSISTENT（snapbuild.c:1383/1407 两个转移），从 `start_decoding_at` 起下游才不会收到半个事务。这对应 Debezium snapshot→streaming 交接必须找到的无重叠一致点。
