## 本章导读

如果你做过 Kafka + Debezium 的 CDC 管道，这一章会让你产生强烈的既视感：

- **物理流复制** = 把整个 broker 的 log segment 逐字节镜像到另一台机器——备库不理解内容，只管"重放存储层"；
- **复制槽（replication slot）** = consumer group 的 offset 提交 + 一个"只要消费者还在，retention 就不许删我未消费数据"的强保证；
- **逻辑解码（logical decoding）** = 内核里内置的 Debezium：把 WAL（物理 redo 日志）解码成一条条 `INSERT/UPDATE/DELETE` 事件，按事务分组、按提交序输出——Debezium 的 PostgreSQL connector 底下就是直接消费这套机制的输出。

本章按数据流向走完两条链路：

1. **物理链路**：`walsender → 复制协议 → walreceiver → 磁盘 → startup 进程 redo`，加上主库提交路径上的同步等待（syncrep）和备库查询与 redo 的冲突（Hot Standby）；
2. **逻辑链路**：`WAL → decode.c → reorderbuffer.c（按事务重组）→ snapbuild.c（历史快照）→ 输出插件 → 网络 → 订阅端 apply worker`。逻辑解码是本章重点。

阅读方法建议：每一节先看"要解决什么问题、不这样做会怎样"，再进代码。本章所有
全函数走读都会先给出问题背景，代码片段之后逐段解说；跳过的代码段会注明跳过原因
（通常是错误处理、统计上报或与主线无关的调试路径）。

## 前置知识

- **WAL 基础**（指南第 10 章）：LSN 是 WAL 的字节偏移（64 位，打印成 `%X/%08X`）；WAL 记录按 rmgr（resource manager）分类，如 Heap、Transaction、XLOG；
- **MVCC 与快照**（指南第 9 章）：xmin/xmax、快照决定元组可见性；vacuum 清理死元组的下限由"系统里最老的 xmin"决定；
- **进程模型**（指南第 1、14 章）：walsender 是一种特殊 backend；walreceiver、startup、logical replication worker 都是辅助进程；
- 大数据侧默认你熟悉：Kafka 的 log/offset/retention/consumer lag、ISR 与 high watermark、Debezium 的 snapshot + streaming 两阶段、exactly-once 与幂等写。

## 13.0 为什么物理复制天然一致、逻辑复制天然灵活

进入代码之前先回答本章最根本的"所以然"：同一份 WAL，为什么衍生出两套完全不同
的复制机制？答案在于**重放的抽象层次**不同。

**物理复制是字节级重放**。备库拿到的是 WAL 的原始字节，用与崩溃恢复完全相同的
redo 代码重放到自己的数据文件上。备库不需要理解"这是哪张表的哪一行"——它只知道
"把这些字节写到 relfilenode X 的第 N 页偏移 M 处"。因此：

- 一致性是**免费**的：备库的磁盘状态和主库在同一 LSN 时的磁盘状态逐位相同（hint bit
  等少数例外不影响逻辑内容）。不存在"复制了一半的行"或"索引和堆不一致"——因为
  索引更新、TOAST、FSM 全都在 WAL 字节流里，一起被重放；
- 约束也是刚性的：主备**必须同大版本、同平台、同编译选项**（页格式绑定）；只能整库
  复制，不能只复制部分表；备库只读，不能有自己的写入。

**逻辑复制是语义级重放**。decode.c 把 WAL 字节翻译回"某表 INSERT 了某行"这样的
语义事件，订阅端用**执行器**重新执行这些事件。因此：

- 灵活性是结构性的：目标库可以是不同大版本（跨版本升级的标准路径）、可以只订阅
  部分表/部分行/部分列、目标表可以有自己的索引和分区方式、目标库可以同时接受
  本地写入（双活/汇聚场景）；
- 代价同样是结构性的：订阅端要真的"执行" SQL 级效果——写放大（一次远端 INSERT
  = 订阅端一次完整的执行器插入 + 索引维护 + 自己的 WAL）；DDL 不在语义事件流里
  （见 13.5.7）；一致性要靠机制去挣（事务重组、按提交序 apply、origin 位点）。

一张对照表总结：

| 维度 | 物理复制（字节级） | 逻辑复制（语义级） |
|---|---|---|
| 重放单位 | 页面修改（redo 记录） | 行级变更（INSERT/UPDATE/DELETE） |
| 版本要求 | 主备同大版本、同平台 | 可跨大版本（升级利器） |
| 复制范围 | 整个实例（全部数据库） | 按 publication 选表/行/列 |
| 备端可写 | 否（只读 Hot Standby） | 是（本地表可写，需防冲突） |
| DDL | 天然复制（DDL 也是 WAL 字节） | 不复制，需人工同步 |
| 写放大 | 低（只搬字节+重放页改） | 高（完整执行器路径+订阅端自己的 WAL） |
| 一致性保证 | 逐字节等同，免费 | 事务级等同，靠 reorderbuffer/origin 挣来 |
| Kafka 类比 | broker 间 replica fetch | Debezium CDC 管道 |

这个分野决定了本章结构：物理链路的复杂度集中在**水位管理**（发多少、等多少、删多少），
逻辑链路的复杂度集中在**语义重建**（事务边界、历史表结构、位点语义）。

## 13.1 物理流复制：把 WAL 字节流搬到另一台机器

### 13.1.1 主库侧：walsender 的变身与发送循环

备库（或 `pg_receivewal` 等工具）连接主库时在连接串里带 `replication=true`，该
backend 就变身 walsender，进入复制命令循环（`exec_replication_command()`）。收到
`START_REPLICATION` 命令后由 `StartReplication()`（`src/backend/replication/walsender.c:860`）
校验时间线并进入发送主循环 `WalSndLoop()`（walsender.c:3046），物理复制时传入的
发送回调是 `XLogSendPhysical`（调用点 walsender.c:1025）；逻辑复制则由
`StartLogicalReplication()`（walsender.c:1530）传入 `XLogSendLogical`（调用点
walsender.c:1599）。`WalSndLoop` 本身只是"能发就发、没活睡 latch、处理 feedback"
的调度壳，真正的水位决策在发送回调里。

### 13.1.2 XLogSendPhysical 全函数走读

这是物理复制主库侧的心脏。全函数约 300 行，下面按五个阶段完整走读（只省略
级联备库文件重载的 retry 细节和 PS 显示更新，二者不影响主线语义）。

**阶段一：状态检查与发送上限决策**（walsender.c:3360 起）：

```c
/* walsender.c:3359 起 */
static void
XLogSendPhysical(void)
{
    XLogRecPtr  SendRqstPtr;
    XLogRecPtr  startptr;
    XLogRecPtr  endptr;
    Size        nbytes;
    ...

    if (streamingDoneSending)

    if (sendTimeLineIsHistoric)

         * 新时间线 history 文件，用 tliSwitchPoint() 求分叉点并把
         * SendRqstPtr 截断到 sendTimeLineValidUpto（3411-3450，见下文解说） */
    }
    else

         *
         * Attempt to send all data that's already been written out and
         * fsync'd to disk.  We cannot go further than what's been written out

         * it's unsafe to send WAL that is not securely down to disk on the

```

三个分支决定了发送上限 `SendRqstPtr`，各有各的"安全"定义：

- **历史时间线分支**：备库请求的是一条已被切换掉的时间线（见 13.1.5），只允许发送
  到分叉点 `sendTimeLineValidUpto`——分叉点之后旧时间线上的字节在本服务器的历史里
  已经"不算数"了；
- **级联分支**：级联复制（备库再带备库）时，上游备库只敢发**自己已经回放确认过**
  的 WAL（`GetStandbyFlushRecPtr`）——收到但未回放的字节还没经过校验，而且上游
  自己也可能被提升或跟随新时间线，这里同时检测两种时间线变化并转入历史时间线模式；
- **主库分支**（walsender.c:3454-3464 的注释是本函数最重要的一段）：只发已经
  fsync 到磁盘的 WAL（`GetFlushRecPtr()`，`src/backend/access/transam/xlog.c:7000`）。

**为什么不能发未 flush 的 WAL？** 把注释里的场景推演完整：假设主库把 WAL 缓冲区里
尚未 fsync 的记录 R 发给了备库，随后主库掉电。重启后主库从最后一次 fsync 的位置
继续——R 在主库上**从未存在过**，同一个 LSN 区间会被新的记录 R' 覆写。但备库已经
回放了 R。现在主备在同一 LSN 处持有不同内容，且双方都不知道：复制协议按 LSN 续传，
备库会从 R 之后继续接收，把 R' 之后的记录接在 R 之后回放——数据悄无声息地分叉。
所以**发送上限 = 本地持久化下限**是硬性不变式（见 13.6 I1）。这就是 Kafka 里
"consumer 只能读到 high watermark"的同款约束：未被持久化确认的数据对外不可见，
否则 leader 崩溃重启后会产生不可检测的历史分歧。

**阶段二：lag 追踪采样**（walsender.c:3491）：`LagTrackerWrite(SendRqstPtr,
GetCurrentTimestamp())` 把"这个 LSN 大约是此刻写出的"记入环形缓冲；将来备库
feedback 报告 write/flush/apply 到达该 LSN 时，用当前时间减去记录时间就得到
`pg_stat_replication` 的 `write_lag/flush_lag/replay_lag`。省略其上方 20 行注释：
它解释为什么在这里采样（而不是在 XLogFlush 热路径里记时间戳）足够准确。

**阶段三：历史时间线的终止**（walsender.c:3507-3523）：如果正在发送历史时间线且
`sentPtr` 已到分叉点，发送 `CopyDone`、置 `streamingDoneSending`，客户端随后会用
新时间线重新发起 `START_REPLICATION`。注释（3497-3505）解释了一个精妙的边界：
分叉点可能落在一条 WAL 记录中间（页边界切断），此时可能已经把半条记录发给了下游，
这没关系——下游同样无法回放半条记录，会在时间线切换时正确处理。

**阶段四：切片与回退**（walsender.c:3525-3565）：


发到 `SendRqstPtr` 即置 `WalSndCaughtUp`（追平，历史时间线除外——还差一步

### 13.1.3 复制协议：'w'、'k'、'r'、'h'、's' 五种消息

复制流上的消息类型定义在 `src/include/libpq/protocol.h:75-84`：

| 字节 | 常量 | 方向 | 含义 |
|---|---|---|---|
| `'w'` | `PqReplMsg_WALData` | 主→备 | WAL 数据（startLSN + endLSN + 时间戳 + 字节流） |
| `'k'` | `PqReplMsg_Keepalive` | 主→备 | 心跳，可要求立即回复 |
| `'s'` | `PqReplMsg_PrimaryStatusUpdate` | 主→备 | 主库状态（LSN + xid 视界，供订阅端 retain_dead_tuples 用，20devel 新增） |
| `'r'` | `PqReplMsg_StandbyStatusUpdate` | 备→主 | feedback：write/flush/apply 三个 LSN |
| `'h'` | `PqReplMsg_HotStandbyFeedback` | 备→主 | 反馈备库 xmin（见 13.4.3） |

备库定期上报的 `'r'` 消息由主库 `ProcessStandbyReplyMessage()`
（walsender.c:2543）处理，主线四步：

```c
/* walsender.c:2560 起，节选 */

    replyRequested = pq_getmsgbyte(&reply_message);

    walsnd->flush = flushPtr;
    walsnd->apply = applyPtr;
    ...
```

1. 解析三个 LSN，写入共享内存 `WalSnd` 结构——这就是 `pg_stat_replication` 里
   `write_lsn/flush_lsn/replay_lsn` 的来源；
2. 用 `LagTrackerRead` 结算三种 lag（与 13.1.2 阶段二的采样配对）；
3. 调 `SyncRepReleaseWaiters()`（walsender.c:2635）——**同步复制的放行入口就挂在
   feedback 处理路径上**，见 13.3.4；
4. 若本连接占用槽，用 flushPtr 推进槽位点（物理槽 `PhysicalConfirmReceivedLocation`
   walsender.c:2510，逻辑槽 `LogicalConfirmReceivedLocation`）——**feedback 消息
   同时就是 offset commit**。

一条 `'r'` 消息同时喂饱监控、同步复制、槽回收三个子系统，这是 PG 复制协议里
最"一鱼三吃"的设计。

### 13.1.4 备库侧：walreceiver 落盘 + startup 进程永续 redo

备库两个进程分工严格：

- **walreceiver**（`src/backend/replication/walreceiver.c`，主入口 `WalReceiverMain()`，
  walreceiver.c:154）：只管收字节、写文件、fsync、发 feedback。收到消息后由
  `XLogWalRcvProcessMsg()`（walreceiver.c:842）分发：`'w'` 消息交给
  `XLogWalRcvWrite()`（walreceiver.c:913）按 LSN 算出目标 WAL 段文件和偏移写下去；
  段文件写满或本批消息处理完后 `XLogWalRcvFlush()`（walreceiver.c:1014）fsync 并
  更新共享内存 flush 位置、随即发 reply；
- **startup 进程**：就是崩溃恢复用的那个 redo 进程，只是"永不结束"——回放完已有
  WAL 就等 walreceiver 通知有新数据。备库的一致性完全由"单进程按 LSN 顺序 redo"
  保证，与崩溃恢复共用同一套回放代码——这是"物理复制天然一致"的工程根基：
  复制没有专用的数据路径，就没有专用的 bug。

feedback 的构造在 `XLogWalRcvSendReply()`（walreceiver.c:1137）：

```c
/* walreceiver.c:1181 起 */
    resetStringInfo(&reply_message);
    pq_sendbyte(&reply_message, PqReplMsg_StandbyStatusUpdate);
    pq_sendint64(&reply_message, writePtr);
    pq_sendint64(&reply_message, flushPtr);
    pq_sendint64(&reply_message, applyPtr);
    pq_sendint64(&reply_message, GetCurrentTimestamp());
    pq_sendbyte(&reply_message, requestReply ? 1 : 0);
```

apply 位置来自 `GetXLogReplayRecPtr()`——startup 进程回放到哪了。**写盘的人
（walreceiver）和回放的人（startup）解耦**，所以 flushPtr 通常领先 applyPtr，
差值就是备库 replay lag。这也解释了 13.3 中 `remote_apply` 为什么比 `on` 慢：
前者要等第二个进程的进度。

### 13.1.5 时间线：没有它，故障切换后必然数据分叉

**问题推演（所以然之六）**：假设没有时间线机制。主库 A 在 LSN 0/5000 处崩溃，
备库 B 回放到 0/4800 时被提升为新主，开始从 0/4800 继续写自己的 WAL。运维修好了
A，把它作为备库重新指向 B。A 请求"从我本地 WAL 的末尾 0/5000 续传"——但 B 上
0/4800 之后的内容和 A 本地 0/4800–0/5000 的内容是**两段不同的历史**（A 崩溃前
写的最后 200 字节从未到达 B）。没有任何标识能区分这两段历史：A 会把 B 的新 WAL
直接接在自己旧 WAL 后面回放，得到一个页级别损坏的数据库；更隐蔽的场景是 A 重启后
没人把它指向 B，应用把写发给了 A——脑裂完成，且事后无法从 WAL 判断哪些提交只
存在于哪一边。

时间线（timeline）就是给"历史"编号：每次提升，TLI 加一，并写出 `.history` 文件
记录"我从 TLI=n 的哪个 LSN 分叉"。作用类比 Kafka 的 controller epoch / Raft 的
term：**同一 LSN 区间的两段不同历史因 TLI 不同而可区分、可拒绝**。旧主 A 以备库
身份连上新主 B 时，B 发现 A 请求的 (TLI, LSN) 落在分叉点之后的旧时间线上，A 需要
回退到分叉点（`pg_rewind` 的工作，用 WAL 把 A 上分叉之后修改过的页反向回滚）或
重做基础备份。

`XLogSendPhysical` 里的历史时间线处理（13.1.2 阶段一、三）就是该机制的发送侧：
只把旧时间线发到 `tliSwitchPoint()` 算出的分叉点，之后客户端换新 TLI 续传。备库
端 `recovery_target_timeline=latest`（默认）自动跟随切换。注意时间线**只提供
可检测性，不提供防脑裂**：它保证分叉能被发现，不保证分叉不发生（见 13.7.3）。

### 小结

- walsender 是"不解析内容的字节搬运工"；发送上限的三分支各自钉死在"本地已持久化
  /已回放/已切换"的边界上，其中主库分支的"只发已 fsync"是防止主备无声分叉的
  硬性不变式；
- 复制协议核心是 `'w'` 数据 + `'r'` feedback；一条 feedback 同时驱动监控、
  syncrep 放行、槽位点推进；
- 备库 = walreceiver（收+落盘）+ startup（永续 redo），复用崩溃恢复代码；
- 时间线给历史编号，让故障切换后的 WAL 分叉从"无声损坏"变成"可检测错误"。

## 13.2 复制槽（slot.c）：带强保证的 consumer offset

### 13.2.1 槽解决什么：无槽时代的赌博式保留

**问题推演（所以然之二）**。复制槽是 9.4 引入的（commit `858ec11858`，2014-01，
"Introduce replication slots"）。在那之前，主库靠 `wal_keep_segments`（PG 13 起
更名为 `wal_keep_size`，commit `c3fe108c02`，2020-07）为断连的备库保留 WAL——
一个纯粹的静态猜测："我赌备库最多落后 N 个段"。

把一次典型事故推演完整：设 `wal_keep_segments = 64`（约 1GB），备库因机房网络
故障断连。主库照常写入，checkpoint 照常回收 WAL——它**不知道**有个备库还需要
这些段，只机械地保留最近 64 个。若主库写入速率 100GB/天，断连 15 分钟后备库需要
的段就被回收了。一天后网络恢复，备库请求续传，主库答复 "requested WAL segment
has already been removed"。此时唯一出路是重做基础备份——对 10TB 的库意味着数小时
到数天的全量拷贝，期间主库还要承受备份读放大。运维的"对策"是把
`wal_keep_segments` 调到夸张的大——于是 99% 的时间在浪费磁盘，剩下 1% 的时间
仍然不够。**静态保留在"消费者进度未知"的前提下必然两头不讨好。**

槽把猜测换成了合同：消费者的进度被持久化在服务端（`pg_replslot/<slot>/state`），
WAL 回收线以**所有槽中最落后的 restart_lsn** 为界（见 13.2.3）。Kafka 视角：
`wal_keep_segments` 是 `log.retention.bytes`（到期照删，消费者自求多福），槽是
"retention 对该 consumer group 永不过期"——一个 Kafka 故意不提供的强保证，
因为它的代价同样惊人（13.2.5）。

### 13.2.2 槽的持久化状态：四个关键字段

槽的核心状态在 `ReplicationSlotPersistentData`（`src/include/replication/slot.h:95`）：

```c
/* src/include/replication/slot.h:95 起，节选 */
typedef struct ReplicationSlotPersistentData
{
    NameData    name;
    Oid         database;          /* 逻辑槽绑定单个数据库；物理槽为 InvalidOid */
    ...
    TransactionId xmin;            /* 数据的 xmin 下限（物理槽+hs_feedback 用） */
    TransactionId catalog_xmin;    /* 目录元组的 xmin 下限（逻辑槽用） */

    /* oldest LSN that might be required by this replication slot */
    XLogRecPtr  restart_lsn;

    /* RS_INVAL_NONE if valid, or the reason for having been invalidated */
    ReplicationSlotInvalidationCause invalidated;

    /*
     * Oldest LSN that the client has acked receipt for.  This is used as the
     * start_lsn point in case the client doesn't specify one, ...
     */
    XLogRecPtr  confirmed_flush;
```

（字段行号：xmin=114、catalog_xmin=122、restart_lsn=125、invalidated=128、
confirmed_flush=136。）对照 Kafka 语义：

| PostgreSQL | Kafka/CDC 对应物 | 语义 |
|---|---|---|
| `restart_lsn` | "为该 consumer 保留的最老 segment 偏移" | **重启解码/复制必须从这里重读 WAL**。逻辑槽上它通常落后于 confirmed_flush——重组在途长事务需要回到其起点之前的快照位置（13.5.4 详解其推进机制） |
| `confirmed_flush` | committed offset | 客户端确认"这之前的变更我已安全接收"，`START_REPLICATION` 不指定起点时从这里续传（slot.h:130-135 注释） |
| `xmin` / `catalog_xmin` | 无直接对应（Kafka 没有 MVCC） | 阻止 vacuum 清掉本槽还需要的（目录）元组版本 |
| 槽本身 | consumer group + retention 豁免 | 区别：**Kafka retention 到期照删不误，PG 的槽默认无限期扣住 WAL** |

物理槽/逻辑槽的分野在 `ReplicationSlotCreate()`（`src/backend/replication/slot.c:378`）
的 `db_specific` 参数——逻辑槽必须绑定数据库（解码要读该库的系统目录），物理槽
全局有效且只用 `restart_lsn`/`xmin`。使用前需 `ReplicationSlotAcquire()`
（slot.c:629）独占获取：同一槽同一时刻只能有一个消费者——等价于 Kafka 同组内
一个分区只分给一个 consumer。

### 13.2.3 槽如何扣住 WAL：KeepLogSeg

checkpoint 决定回收哪些 WAL 段时调用 `KeepLogSeg()`
（`src/backend/access/transam/xlog.c:8498`），全函数逻辑就是"四方压价"：

```c
/* xlog.c:8497 起，节选 */
static void
KeepLogSeg(XLogRecPtr recptr, XLogSegNo *logSegNo)
{
    XLogSegNo   currSegNo;
    XLogSegNo   segno;
    XLogRecPtr  keep;

    XLByteToSeg(recptr, currSegNo, wal_segment_size);
    segno = currSegNo;

    /* Calculate how many segments are kept by slots. */
    keep = XLogGetReplicationSlotMinimumLSN();
    if (XLogRecPtrIsValid(keep) && keep < recptr)
    {
        XLByteToSeg(keep, segno, wal_segment_size);

        /* Account for max_slot_wal_keep_size to avoid keeping more than
         * configured. ... */
        if (max_slot_wal_keep_size_mb >= 0 && !IsBinaryUpgrade)
        {
            uint64      slot_keep_segs;

            slot_keep_segs =
                ConvertToXSegs(max_slot_wal_keep_size_mb, wal_segment_size);

            if (currSegNo - segno > slot_keep_segs)
                segno = currSegNo - slot_keep_segs;
        }
    }
    ...
    /* but, keep at least wal_keep_size if that's set */
    if (wal_keep_size_mb > 0)
    { ... }
```

- `XLogGetReplicationSlotMinimumLSN()`（xlog.c:2699）读的是
  `ReplicationSlotsComputeRequiredLSN()`（slot.c:1304）汇总到共享内存的**全体槽
  restart_lsn 最小值**——一个掉队的槽就能拖住全局回收线，木桶效应；
- `max_slot_wal_keep_size`（xlog.c:142，默认 -1 不设限；PG 13 引入，commit
  `c655077639`，2020-04）是唯一的保险丝：槽落后超过该值时 checkpoint 照常回收，
  随后该槽被标记 invalidated（slot.h:128，原因 `RS_INVAL_WAL_REMOVED`）——等价于
  Kafka consumer 掉出 retention 后的 `OffsetOutOfRange`：消费者必须重建（对 CDC
  意味着重新全量快照）。这个 GUC 的引入本身就是一次设计反省：9.4 给了槽无限
  强保证，六年的生产事故证明"必须有人能对失控的消费者说不"；
- 中段还有 WAL summarizer 的保留需求（增量备份，此处省略），末段 `wal_keep_size`
  兜底——静态保留如今只是槽机制的下限补充。

### 13.2.4 catalog_xmin：槽的另一只手伸向 vacuum

逻辑槽还会扣住 MVCC：`ReplicationSlotsComputeRequiredXmin()`（slot.c:1222）汇总
全体槽的 xmin/catalog_xmin 写入 ProcArray，参与全局可清理视界计算。效果：**系统
目录（pg_class、pg_attribute…）中比 catalog_xmin 新的死元组不会被 vacuum 掉**——
因为解码历史 WAL 需要"当时"的表结构（13.5.4 详述这个依赖为什么存在）。注意它只
保护目录表，不保护用户表——逻辑解码从不读用户表的旧版本，行数据全部来自 WAL 本身。

### 13.2.5 槽的代价反噬：一条完整的故障时间线

强保证的另一面。以下时间线是 Debezium/PG 生产环境最经典的事故，逐步推演：

- **T+0**：下游 Kafka Connect 集群升级，Debezium connector 停止，**但没人删除
  它在 PG 上创建的逻辑槽**（删 connector 不删槽——Debezium 官方文档专门警告过）。
  槽 `active=false`，`restart_lsn`、`catalog_xmin`、`confirmed_flush` 全部冻结；
- **T+2h**：`pg_wal` 目录开始线性增长——KeepLogSeg 的回收线被冻结的 restart_lsn
  钉住。监控若只看复制延迟（没有活跃连接就没有延迟指标）则毫无告警；
- **T+12h**：autovacuum 的日志开始出现目录表膨胀迹象：catalog_xmin 冻结导致
  pg_class/pg_attribute 上频繁 DDL/临时表产生的死元组无法回收。有大量分区滚动或
  临时表的业务，系统目录可以膨胀到数十 GB，所有 SQL 的目录缓存构建变慢；
- **T+1d**：`pg_wal` 吃满文件系统。PG 对 WAL 写入失败的反应是 **PANIC**——主库
  直接崩溃且无法重启（重启也要写 WAL）。此时只能手工删段文件腾地方？不行，会
  破坏一致性；正确做法是紧急扩容文件系统或删槽后等 checkpoint 回收；
- **更深的余波**：若冻结的是 `xmin`/`catalog_xmin` 且持续数周，最老 xid 距
  `nextXid` 逼近 20 亿，触发 wraparound 保护，全库进入只读——这是比磁盘打爆更难
  恢复的状态。

**监控口径**：`pg_replication_slots` 里 `active=false` 且 `restart_lsn` 长期不动
的槽；`wal_status` 列（reserved/extended/unreserved/lost）直接告诉你该槽的 WAL
是否已被 `max_slot_wal_keep_size` 牺牲。**处方**：生产必设 `max_slot_wal_keep_size`
（宁可让 CDC 重新快照，不可让主库停写）；下线订阅方时必须
`pg_drop_replication_slot()`。

### 小结

- 槽 = 持久化在服务端的消费位点（restart_lsn/confirmed_flush）+ 对 WAL 回收
  （KeepLogSeg）和 vacuum（catalog_xmin）的双重反压；
- 它取代的是 `wal_keep_segments` 的静态赌博：保留量从"猜的常数"变成"真实消费
  进度的函数"；
- 强保证的代价是把下游的故障转化为主库的磁盘/膨胀/wraparound 危机；
  `max_slot_wal_keep_size` 是把 PG 拉回 Kafka "retention 有限"哲学的保险丝。

## 13.3 同步复制（syncrep.c）：提交路径上的分布式 ack

### 13.3.1 确认级别谱系：每一档在什么故障下丢数据

GUC 值到等待级别的映射在 `assign_synchronous_commit()`（syncrep.c:1130），级别
枚举在 `src/include/access/xact.h:71-81`，等待模式常量在
`src/include/replication/syncrep.h:23-25`：

| GUC 值 | 枚举 | 本地行为 | 远程等待 |
|---|---|---|---|
| `off` | `SYNCHRONOUS_COMMIT_OFF` | 提交不等本地 WAL fsync | 无 |
| `local` | `..._LOCAL_FLUSH` | 等本地 fsync | 无 |
| `remote_write` | `..._REMOTE_WRITE` | 等本地 fsync | 备库 write（进 OS 页缓存）→ `SYNC_REP_WAIT_WRITE` |
| `on`（默认） | `..._REMOTE_FLUSH` | 等本地 fsync | 备库 flush（fsync）→ `SYNC_REP_WAIT_FLUSH` |
| `remote_apply` | `..._REMOTE_APPLY` | 等本地 fsync | 备库 apply（回放可见）→ `SYNC_REP_WAIT_APPLY` |

**逐级故障推演（所以然之五）**——对每一档问同一个问题："client 收到 commit OK
之后，什么样的故障组合会让这笔事务消失或不可见？"

- **off**：主库单机崩溃即可丢——WAL 还在 wal_buffers 里没 fsync（walwriter 最多
  延迟 `wal_writer_delay*3` 刷出）。丢的是"最近几百毫秒"的已确认事务。数据库
  自身不会损坏（WAL 顺序不破坏），只是**确认过的事务蒸发**。适合可重放的埋点类
  写入；
- **local**：主库崩溃不丢（本地已 fsync）；主库**磁盘整体损毁**丢——WAL 只有
  一份。这是无备库或异步备库时的默认底线；
- **remote_write**：主库磁盘损毁不丢（备库内存里有）；但"主库损毁 **且** 备库在
  尚未把页缓存刷盘时同时掉电"丢——双故障窗口。备库进程崩溃（OS 还在）不丢，
  因为字节已进内核。它买的是比 `on` 少一次远端 fsync 的延迟，赌的是"两台机器
  不同时掉电"；
- **on**：主库损毁 + 备库掉电都不丢——WAL 已在两份持久介质上。会丢的场景只剩
  "主库损毁且备库磁盘也损毁"（真正的双盘灾难）。注意 `on` **不保证备库可见性**：
  事务在备库已 fsync 但可能尚未回放，failover 后数据都在，但切换前的瞬间从备库
  读不到；
- **remote_apply**：在 `on` 的持久性之上追加**读一致性**：确认返回时事务在备库
  已回放、对备库查询可见。用于"主库写、备库读"的负载分离场景，代价是提交延迟
  再加一个 replay lag（等的是 startup 进程，见 13.1.4）。它防的不是丢数据，
  是"写后读旧"。

注意 `on` 只有配置了 `synchronous_standby_names` 才产生远程等待，否则前四档只
影响本地 fsync 时机。分叉点一：`off` 在 `RecordTransactionCommit()`
（`src/backend/access/transam/xact.c:1345`）走异步提交路径（不立刻 XLogFlush）。
分叉点二：提交末尾 `SyncRepWaitForLSN(XactLastRecEnd, true)`（xact.c:1599），
远程等待从这里开始。

### 13.3.2 SyncRepWaitForLSN 全函数走读

`SyncRepWaitForLSN()`（syncrep.c:149）是每次提交都要过的路径，函数结构 =
快速退出 + 入队 + latch 循环 + 清理。逐段走读：

**第一段：无锁快速退出**（syncrep.c:179-188）：

```c
/* syncrep.c:179 起 */
    if (!SyncRepRequested() ||
        ((((volatile WalSndCtlData *) WalSndCtl)->sync_standbys_status) &
         (SYNC_STANDBY_INIT | SYNC_STANDBY_DEFINED)) == SYNC_STANDBY_INIT)
        return;

    /* Cap the level for anything other than commit to remote flush only. */
    if (commit)
        mode = SyncRepWaitMode;
    else
        mode = Min(SyncRepWaitMode, SYNC_REP_WAIT_FLUSH);
```

- 这个函数**每次提交都会执行**，所以第一件事是不加锁地判断"根本没配同步复制"
  （checkpointer 已初始化状态位但没定义同步备库）直接返回——热路径零开销；
- 非 commit 记录（如 PREPARE）最多等到 flush 级：备库 feedback 里的 apply 位置
  只对 commit 记录有意义（函数头注释 143-146 行）。

**第二段：持锁再判 + 入队**（syncrep.c:193-255）：

```c
/* syncrep.c:208 起，节选 */
    if (WalSndCtl->sync_standbys_status & SYNC_STANDBY_INIT)
    {
        if ((WalSndCtl->sync_standbys_status & SYNC_STANDBY_DEFINED) == 0 ||
            lsn <= WalSndCtl->lsn[mode])
        {
            LWLockRelease(SyncRepLock);
            return;
        }
    }
    ...
    /*
     * Set our waitLSN so WALSender will know when to wake us, and add
     * ourselves to the queue.
     */
    MyProc->waitLSN = lsn;
    MyProc->syncRepState = SYNC_REP_WAITING;
    SyncRepQueueInsert(mode);
    Assert(SyncRepQueueIsOrderedByLSN(mode));
    LWLockRelease(SyncRepLock);
```

持 `SyncRepLock` 后重查两个条件：同步备库定义可能刚被管理员清空（避免永久卡死），
以及**备库可能已经确认过本 LSN**（`lsn <= WalSndCtl->lsn[mode]`——三个模式各维护
一个"已确认水位"）。都不满足才把自己挂进等待队列。省略 217-245 行：状态位尚未
初始化时的退化判断（启动初期的边界情形）。

**第三段：latch 等待循环**（syncrep.c:272-355）：

```c
/* syncrep.c:272 起，节选 */
    for (;;)
    {
        /* Must reset the latch before testing state. */
        ResetLatch(MyLatch);

        if (MyProc->syncRepState == SYNC_REP_WAIT_COMPLETE)
            break;

        if (ProcDiePending)
        {
            ...
            ereport(WARNING, ...
                     errdetail("The transaction has already committed locally, "
                               "but might not have been replicated to the standby.")...);
            whereToSendOutput = DestNone;
            SyncRepCancelWait();
            break;
        }

        if (QueryCancelPending)
        { /* 同上：WARNING + SyncRepCancelWait + break */ }

        rc = WaitLatch(MyLatch, WL_LATCH_SET | WL_POSTMASTER_DEATH, -1,
                       WAIT_EVENT_SYNC_REP);

        if (rc & WL_POSTMASTER_DEATH)
        { ... SyncRepCancelWait(); break; }
    }
```

这一段藏着同步复制语义上最微妙的一点：**事务此刻已经在本地提交完成**（WAL 已
flush、CLOG 已写），等待只是推迟向客户端报告。因此：

- 收到终止信号（ProcDiePending）**不能报 ERROR**——那会让客户端以为事务回滚了，
  而它实际已提交；也不能直接 ack——客户端有权假设被确认的提交已复制。折中是发
  WARNING（"已本地提交但可能未复制"）并切断输出后退出等待。这就是运维圈常说的
  "同步复制卡住时 Ctrl-C 掉的事务其实提交了"的出处；
- 查询取消（QueryCancelPending）同理：取消的是**等待**，不是事务；
- postmaster 死亡则永远等不到 ack（walsender 都会退出），直接放弃。

**队列为什么必须按 LSN 有序**——`SyncRepQueueInsert()`（syncrep.c:382）从队尾
反向找第一个 `waitLSN < 自己` 的元素插到它后面（提交顺序与 LSN 基本一致，所以
通常一步就停）。有序的意义在放行端：唤醒者只需从队头开始摘除
`waitLSN <= 确认水位` 的进程，**碰到第一个未达标的就停**——放行成本 O(被放行数)，
与队列总长无关。若无序，每次 feedback 都要全队列扫描，在高并发提交 + 频繁
feedback 的场景下是 O(n·m) 的灾难。Kafka 对照：这就是 producer 按 offset 顺序
等 ack，broker 的 high watermark 一推进就能按序批量确认的同款结构。

### 13.3.3 放行端：SyncRepReleaseWaiters 全函数走读

唤醒方是 walsender：收到 `'r'` feedback 后调用 `SyncRepReleaseWaiters()`
（syncrep.c:484，调用点 walsender.c:2635）：

```c
/* syncrep.c:502 起，节选 */
```

- 先排除自己不是候选同步备库、还在 backup/startup 阶段等情况（省略中段

- `SyncRepGetSyncRecPtr()`（syncrep.c:595）计算"全体同步备库的集体确认位置"：
  拿到当前候选列表后，优先模式走 `SyncRepGetOldestSyncRecPtr()`（syncrep.c:669，
  前 num_sync 名的**最小值**——最慢的同步备库说了算），quorum 模式走
  `SyncRepGetNthLatestSyncRecPtr()`；
- 三个模式的水位各自只前进不后退（`<` 判断），推进了才调 `SyncRepWakeQueue()`
  （syncrep.c:914）从队头按序摘人、置 `SYNC_REP_WAIT_COMPLETE`、set latch。

整体是两阶段 ack 管道：backend（生产者）阻塞在自己的 latch 上，walsender（ack
收集器）根据备库 feedback 推进水位并**批量**放行——一次 feedback 可能唤醒成百上
千个等待中的提交。与 Kafka producer `acks=all` 等 ISR 确认同构。

### 13.3.4 quorum（ANY n）语义

`synchronous_standby_names` 两种语法对应两种策略（`src/include/replication/syncrep.h:35-36`）：
`FIRST n (...)`（`SYNC_REP_PRIORITY`）按列表顺序取优先级最高的 n 个，确认位置取
其**最小**；`ANY n (...)`（`SYNC_REP_QUORUM`）任意 n 个确认即可，确认位置 =
全体候选中**第 n 大**，由 `SyncRepGetNthLatestSyncRecPtr()`（syncrep.c:702）计算：

```c
/* syncrep.c:721 起，节选 */
    for (i = 0; i < num_standbys; i++)
    {
        write_array[i] = sync_standbys[i].write;
        flush_array[i] = sync_standbys[i].flush;
        apply_array[i] = sync_standbys[i].apply;
    }

    /* Sort each array in descending order */
    qsort(write_array, num_standbys, sizeof(XLogRecPtr), cmp_lsn);
    qsort(flush_array, num_standbys, sizeof(XLogRecPtr), cmp_lsn);
    qsort(apply_array, num_standbys, sizeof(XLogRecPtr), cmp_lsn);

    /* Get Nth latest Write, Flush, Apply positions */
    *writePtr = write_array[nth - 1];
    *flushPtr = flush_array[nth - 1];
    *applyPtr = apply_array[nth - 1];
```

降序第 n 个元素 = "至少有 n 个备库到达了这个 LSN"——教科书 quorum ack。

但务必记住：**syncrep 只做 quorum ack，不做选主、不做日志截断协商**。PG 不是
Raft——它没有任期选举、没有多数派日志匹配、没有旧主自动降级。故障切换的正确性
（选最新备库、fence 旧主）完全靠 Patroni/pg_auto_failover 等外部编排。这个设计
边界的完整讨论见 13.7.3。

### 小结

- `synchronous_commit` 五档 = 本地 fsync 时机（off/local）× 远程确认深度
  （write/flush/apply）；每深一档消掉一类故障：进程崩溃 → 单机掉电 → 单盘损毁
  → 双机同时掉电 → 备库读旧；
- `SyncRepWaitForLSN` 的核心语义坑：等待期间事务已本地提交，取消等待 ≠ 回滚；
- 等待队列按 LSN 有序换来 O(被放行数) 的批量唤醒；`ANY n` 用降序第 n 个 LSN
  实现 quorum；同步复制不等于自动故障切换。

## 13.4 Hot Standby：redo 与只读查询抢地盘

### 13.4.1 冲突从哪来

备库查询持有快照/锁/buffer pin，而 redo 必须原样重放主库的动作，两者会顶牛。
`src/backend/storage/ipc/standby.c` 按冲突类型提供一组
`ResolveRecoveryConflictWith*` 函数：

| 冲突类型 | 触发场景 | 处理函数（standby.c） |
|---|---|---|
| Snapshot | 主库 vacuum/HOT 清理了备库查询还"看得见"的死元组 | `ResolveRecoveryConflictWithSnapshot`（470） |
| Lock | redo 要拿 AccessExclusiveLock（主库 DROP/ALTER），备库查询持有表锁 | `ResolveRecoveryConflictWithLock`（626） |
| BufferPin | redo 要改的页被备库查询 pin 住 | `ResolveRecoveryConflictWithBufferPin`（795） |
| Tablespace / Database | 主库删了表空间/数据库 | 542 / 572 |

最常见的是 snapshot 冲突：主库在清理类 WAL 记录里附带
`snapshotConflictHorizon`（"我删掉的元组对 xmin ≤ 此值的快照可能仍可见"）；备库
回放时 `GetConflictingVirtualXIDs()` 找出所有 xmin 冲突的会话，先等、等不起就杀。
standby.c 里还有一个容易忽略的分支：若被清理的是**目录表**且备库上开着逻辑解码
（PG 16 起允许，commit `0fdab27ad6`），会直接把 catalog_xmin 落后的逻辑槽标记
失效（`RS_INVAL_HORIZON`）——目录旧版本没了，解码无法继续，杀槽比杀查询更彻底。

### 13.4.2 等多久才动手：max_standby_streaming_delay

"等 vs 杀"的时限由 `GetStandbyLimitTime()`（standby.c:202）决定：基准是
`GetXLogReceiptTime()`——**最近一批 WAL 的接收时间**，允许的宽限是
`max_standby_streaming_delay`（standby.c:42，默认 30 秒；-1 表示无限等）。注意
它限定的是 **redo 允许积累的总延迟**而非单个查询的宽限：redo 已经落后很多时，
新冲突的查询会被立刻取消。取消通过 `PROCSIG_RECOVERY_CONFLICT_*` 信号，对方在
下一个中断点抛 `ERROR: canceling statement due to conflict with recovery`。
权衡是经典两难：调大 → 备库长查询存活但复制延迟涨（failover RTO 变差）；调小 →
延迟稳定但报表查询频繁被杀。

### 13.4.3 hot_standby_feedback：把 xmin 反向传播回主库

治本方案之一是让主库"别清那么快"。开启 `hot_standby_feedback`
（walreceiver.c:92）后，walreceiver 周期性调用 `XLogWalRcvSendHSFeedback()`
（walreceiver.c:1210）发 `'h'` 消息，携带备库当前的全局 xmin 与 catalog_xmin
（各带 epoch 防回卷误判）。主库侧 `ProcessStandbyHSFeedbackMessage()`
（walsender.c:2731）的关键落点：

```c
/* walsender.c:2835 起 */
    if (MyReplicationSlot != NULL)  /* XXX: persistency configurable? */
        PhysicalReplicationSlotNewXmin(feedbackXmin, feedbackCatalogXmin);
    else
    {
        if (TransactionIdIsNormal(feedbackCatalogXmin)
            && TransactionIdPrecedes(feedbackCatalogXmin, feedbackXmin))
            MyProc->xmin = feedbackCatalogXmin;
        else
            MyProc->xmin = feedbackXmin;
    }
```

有物理槽时 xmin 记在槽里（断连后依然生效！），无槽时记在 walsender 自己的
PGPROC.xmin 上（断连即失效）。之后主库 `ComputeXidHorizons()` 计算 vacuum 视界
时会把它当成"一个还在跑的本地事务"——主库的清理从此**迁就备库上最老的查询**。

副作用的完整机制见 13.7.4：这等于把备库长事务的膨胀效应传导回主库。它只防
snapshot 冲突，防不了 lock/bufferpin 冲突（DROP TABLE 该杀还是杀）。大数据类比：
consumer 反压直接冻结上游 compaction。

### 小结

- 冲突本质是"redo 必须重放主库的清理/排他动作"与"备库快照还需要旧版本"的矛盾；
- `max_standby_streaming_delay` 限制 redo 总延迟，超时按信号杀查询；
- `hot_standby_feedback` 用反向传播 xmin 换备库查询稳定，代价是主库膨胀
  （槽 + feedback 的组合会让该效应在断连期间仍持续）。

## 13.5 逻辑解码：内核里的 Debezium（本章重点）

### 13.5.0 全景、CDC 对照与"为什么必须重组事务"

```
WAL 段文件
  │  XLogReaderState 逐条读取
  ▼
decode.c        LogicalDecodingProcessRecord: 按 rmgr 分发，物理记录 → ReorderBufferChange
  ▼
reorderbuffer.c 按 xid 聚合交错的变更；超内存落盘/流式下发；commit 时按 LSN 序回放
  ▼               ▲
snapbuild.c     历史目录快照（告诉解码器"当时"的表结构长什么样）
  ▼
logical.c       回调框架 → 输出插件（pgoutput / test_decoding / wal2json）
  ▼
walsender       逻辑 START_REPLICATION，同一条复制协议
  ▼
worker.c        订阅端 apply worker：用执行器回放到本地表
```

与 Debezium/Kafka Connect 管道逐层对照：

| PostgreSQL 逻辑复制 | Debezium/Kafka 管道 |
|---|---|
| WAL | MySQL binlog / 源库事务日志 |
| decode.c + reorderbuffer.c + snapbuild.c | Debezium connector 内部的 log parser + transaction buffer |
| 输出插件（pgoutput） | SourceRecord 序列化（Avro/JSON converter） |
| 逻辑槽 confirmed_flush | Kafka Connect offset topic 里的 source offset |
| walsender 逻辑流 | Kafka topic（但 PG 是单订阅者推模式，位点由槽记） |
| tablesync/sequencesync 初始同步 | Debezium initial snapshot + streaming 无缝衔接 |
| worker.c apply（含 parallel apply） | sink connector / consumer |
| origin（复制来源标记） | Kafka header 里的 source 标记防循环复制 |

关键差异：Debezium 把"解析日志"放在外部进程、位点存 Kafka；PG 把整条解码管道
放进内核、位点存槽。Debezium PG connector 本质上只是一个逻辑复制客户端：建逻辑
槽 + `START_REPLICATION` 消费 pgoutput 流。

**为什么必须重组事务、且只在提交时输出（所以然之三）**。WAL 是全实例唯一的
append-only 日志，所有并发事务的记录**物理交错**：`T1.insert, T2.insert,
T1.update, T3.delete, T2.commit, T1.abort, ...`。假如解码器按 WAL 顺序直接把行
变更吐给下游，会发生什么？

1. **下游看到未提交数据**。T1 的 insert 被下游 apply 后，T1 在源库 abort 了。
   源库靠 MVCC 天然"看不见"未提交行，下游的普通表没有这层保护——脏数据已经
   落地，还可能已被下游的下游消费；
2. **撤回没有廉价的表达**。要撤回就得给下游发"补偿事件"（把 insert 反做成
   delete），这要求解码器为每个在途事务记住全部已发变更以便反向生成，还要求
   下游 sink 全链路支持补偿语义——绝大多数 sink（ES、数仓、缓存）不支持；
3. **事务原子性碎裂**。同一事务的 5 行修改和别的事务的修改交错到达，下游任何
   时刻的状态都可能是"半个事务"，做不了一致性读，也无法实现"按事务分组提交到
   目标库"。

所以 reorderbuffer 的合同是：**按 xid 攒够整个事务，等看到 COMMIT 记录才按序
回放给插件；看到 ABORT 就整段丢弃**。aborted 事务默认根本不出现在逻辑流里——
这是与"裸 binlog 消费"最大的语义差异。下游因此拿到的是"事务完整、按提交序、
无回滚"的干净流，代价是大事务的内存/延迟问题——而这正是 streaming 模式
（13.5.3）要在"下游可回滚"的新前提下打破的限制。

### 13.5.1 decode.c：物理记录 → 逻辑变更

  得回到它开始前**，这就是 13.2.2 里 restart_lsn 落后于 confirmed_flush 的原因；
- **`changes` 链表按 LSN 追加**：单个（子）事务内部天然有序，回放时无需排序，
  只需在多条链之间归并；
- **`tuplecids`**：同事务"先改目录再用新结构写数据"（`CREATE TABLE + COPY`、
  `ALTER TABLE + INSERT`）需要 cmin/cmax 级的命令内可见性，该映射配合历史快照
  实现目录的**命令级**时间旅行；
- **size/total_size**：内存记账（PG 13 起精确记账），是驱逐策略的输入。

代价是失去跨槽的事务原子性。

### 13.5.3 流式解码小结：把"提交时输出"的限制安全地打破

综合 13.5.0 与 13.5.2：streaming 模式的本质是把 reorderbuffer 的"合同"从
"我保证只给你已提交的完整事务"降级为"我可能给你未提交的块，但保证最终告诉你
commit 还是 abort"。安全性前提有三：

1. 下游必须**可回滚**（临时文件缓存 or parallel apply 的真实事务）；
2. 上游探测到并发 abort 能停流（`SetupCheckXidLive`）并补发 `stream_abort`；
3. 半个语义单元不能流（TOAST chunk 与主行的 partial change 约束）。

监控上，`pg_stat_replication_slots` 的 `stream_txns/stream_count/stream_bytes`
与 `spill_*` 对照，能直接看出大事务是被流走了还是落盘了。

### 13.5.4 snapbuild.c：历史目录快照与自举悖论

**问题（所以然之四）**：解码 WAL 里的元组字节需要知道**写这条 WAL 时**表长什么
样——列数、类型、名字全在系统目录里。但目录是活的：今天解上周的 WAL 时表可能
已经 `ALTER` 过。解法是 MVCC 自举：目录旧版本元组还在（被槽的 catalog_xmin
保护），只要构造出"当时"的快照就能读到"当时"的目录。

**自举悖论的完整推演**：想从任意 LSN 开始解码，你需要一个描述"该 LSN 时刻哪些
事务已提交"的快照来读目录；而想构造这个快照，你得知道该时刻有哪些事务在途——
这个信息只有从更早的 WAL 里追踪而来；而"更早"要早到什么程度？必须早于所有
在该 LSN 时仍在途的事务的**起点**——但不解码你又不知道它们的起点在哪。死循环。
PG 的破法：不从任意点开始，而是**从一个 `xl_running_xacts` 记录开始向前爬**，
用两代事务的自然死亡换取完整信息（bgwriter/checkpointer 周期性写入该记录，
列出当刻所有在途 xid）。这正是 Debezium "必须先做 snapshot 才能 streaming、
且两阶段必须无缝对齐"的镜像问题：Debezium 靠一致性读点位对齐，PG 靠等待。

状态机四阶段（`src/include/replication/snapbuild.h:35-58`）：

```
START ──(遇到 xl_running_xacts)──▶ BUILDING_SNAPSHOT ──(初始在途事务全部结束)──▶
FULL_SNAPSHOT ──(FULL 时刻的在途事务也全部结束)──▶ CONSISTENT
```

#### 入口：SnapBuildProcessRunningXacts——槽位点也从这里推进

每条 `xl_running_xacts` 记录都会流到 `SnapBuildProcessRunningXacts()`
（`src/backend/replication/logical/snapbuild.c:1141`）：未一致时调
`SnapBuildFindSnapshot()` 推状态机；已一致则把快照序列化到磁盘。随后是**槽的
两个核心位点的推进现场**：

```c
/* snapbuild.c:1186 起，节选 */
    xmin = ReorderBufferGetOldestXmin(builder->reorder);
    if (xmin == InvalidTransactionId)
        xmin = running->oldestRunningXid;
    ...
    LogicalIncreaseXminForSlot(lsn, xmin);
    ...
    txn = ReorderBufferGetOldestTXN(builder->reorder);

    /*
     * oldest ongoing txn might have started when we didn't yet serialize
     * anything because we hadn't reached a consistent state yet.
     */
    if (txn != NULL && XLogRecPtrIsValid(txn->restart_decoding_lsn))
        LogicalIncreaseRestartDecodingForSlot(lsn, txn->restart_decoding_lsn);

    else if (txn == NULL &&
             XLogRecPtrIsValid(builder->reorder->current_restart_decoding_lsn) &&
             XLogRecPtrIsValid(builder->last_serialized_snapshot))
        LogicalIncreaseRestartDecodingForSlot(lsn,
                                              builder->last_serialized_snapshot);
```

- `catalog_xmin` 的推进以"reorderbuffer 里最老快照的 xmin"为界——在途事务还
  握着老快照，目录就不能清；
- `restart_lsn` 的推进以"最老在途事务的 `restart_decoding_lsn`"为界（该字段 =
  该事务开始前最后一次快照落盘的位置）——**这就是 13.2.2 表里那行注释的出处**：
  长事务在途时 restart_lsn 停在它起点之前，confirmed_flush 却随消费继续前进；
  没有在途事务时可以直接跳到最近一次序列化快照处。之所以只在 running_xacts 点
  推进（而非每次 commit），是因为推进意味着一次槽状态文件的 fsync（1193-1203
  行注释）。

#### SnapBuildFindSnapshot 全函数走读

`SnapBuildFindSnapshot()`（snapbuild.c:1243）。函数头注释给出三条路 a/b/c，
正文六个分支，逐个走：

**分支 0：太老的 running_xacts 直接跳过**（1274-1288）：建槽时记录了
`initial_xmin_horizon`（当时全局 xmin 的下限）；如果这条 running_xacts 里最老
在途 xid 比它还老，说明所需目录版本可能已经被 vacuum 掉（保护是从建槽时刻才
开始的），据此构建会得到一个"以为自己能读到某些目录版本"的坏快照——等待
（`SnapBuildWaitSnapshot`）后跳过。

**分支 a：空载捷径**（1299-1323）：

```c
/* snapbuild.c:1299 起 */
    if (running->oldestRunningXid == running->nextXid)
    {
        if (!XLogRecPtrIsValid(builder->start_decoding_at) ||
            builder->start_decoding_at <= lsn)
            /* can decode everything after this */
            builder->start_decoding_at = lsn + 1;

        /* As no transactions were running xmin/xmax can be trivially set. */
        builder->xmin = running->nextXid;   /* < are finished */
        builder->xmax = running->nextXid;   /* >= are running */
        ...
        builder->state = SNAPBUILD_CONSISTENT;
        ...
        ereport(LogicalDecodingLogLevel(),
                errmsg("logical decoding found consistent point at %X/%08X", ...),
                errdetail("There are no running transactions."));

        return false;
    }
```

`oldestRunningXid == nextXid` 意味着**此刻一个在途事务都没有**——不存在"看不到
开头"的事务，快照平凡成立，直接跳到 CONSISTENT。这也是"在业务低谷建逻辑槽"
这条运维经验的原理：半夜没有长事务，一条 running_xacts 就够。注意这个捷径在
任何状态下都可触发（哪怕已经苦等在 c 的某个子阶段）。

**分支 b：磁盘上有现成快照**（1329-1335）：`SnapBuildRestore()`
（snapbuild.c:1846）尝试从 `pg_logical/snapshots/`（`PG_LOGICAL_SNAPSHOTS_DIR`，
reorderbuffer.h:24）加载**本槽或其他槽**之前在该 LSN 序列化的快照——多个逻辑槽
共享快照文件，一人построил全员受益。但两种情形禁用：`building_full_snapshot`
（要导出完整快照给初始同步用——序列化快照只含目录修改事务，不够）和
`in_slot_creation`（建槽必须自己找一个"无半个事务"的起点，落在别人快照上会导致
从 confirmed_flush 起吐出半个事务，函数头注释 1253-1259 详述）。

**分支 c1：START → BUILDING_SNAPSHOT**（1349-1373）：

```c
/* snapbuild.c:1349 起，节选 */
    else if (builder->state == SNAPBUILD_START)
    {
        builder->state = SNAPBUILD_BUILDING_SNAPSHOT;
        builder->next_phase_at = running->nextXid;

        /*
         * Start with an xmin/xmax that's correct for future, when all the
         * currently running transactions have finished. ...
         */
        builder->xmin = running->nextXid;   /* < are finished */
        builder->xmax = running->nextXid;   /* >= are running */
        ...
        ereport(LOG,
                errmsg("logical decoding found initial starting point at %X/%08X", ...),
                errdetail("Waiting for transactions (approximately %d) older than %u to end.",
                          running->xcnt, running->nextXid));

        SnapBuildWaitSnapshot(running, running->nextXid);
    }
```

记下当前 `nextXid` 作为第一道毕业线。从现在起 snapbuild 开始跟踪每一个**目录
修改事务**的提交（`SnapBuildCommitTxn`，snapbuild.c:945，由 DecodeCommit 调用）
——快照的本体就是"已提交的目录修改 xid 集合"。为什么不直接用 running_xacts 里
的 xids 数组抄近路？注释（1344-1347）：列表里标记为 running 的事务可能**实际上
已经把 commit 记录写进 WAL**（快照记录与提交并发），基于它构造会漏提交——所以
只能保守地等它们全部死亡。

**分支 c2：BUILDING_SNAPSHOT → FULL_SNAPSHOT**（1383-1397）：条件
`next_phase_at <= running->oldestRunningXid`——第一道毕业线之前启动的事务全部
结束。从此刻起，**每个新启动的事务**从它 begin 起的全部目录修改提交都被观察到
了，为它们构造快照是完备的：新事务的变更可以正确解码。但 FULL 时刻仍在途的
老事务不行——它们改的目录（若有）我们没看全，它们的行变更也可能引用我们无法
重建的目录状态。记下第二道毕业线 `next_phase_at = running->nextXid`。

**分支 c3：FULL_SNAPSHOT → CONSISTENT**（1407-1418）：第二道毕业线之前的事务
也全部结束。现在**任何将要提交的事务**都开始于 FULL 之后、其整个生命周期都被
完整观察——不会再输出半个事务。`start_decoding_at` 之前解出的变更只积累不下发
（reorderbuffer 照常攒，回放时被 `DecodeTXNNeedSkip` 拦掉）。

**为什么必须两级等待、一级不够？** 一级等待（BUILDING→FULL）解决的是"目录快照
完备性"：等初始在途事务死光，才能保证此后新事务的快照不缺目录提交。二级等待
（FULL→CONSISTENT）解决的是"变更流完备性"：FULL 时刻还在途的事务，其**前半段
行变更**发生在我们开始收集之前，输出它们就是输出半个事务；而它们又不能被简单
丢弃（万一提交了呢），只能等这一代也自然终结。两级毕业线都用
`TransactionIdPrecedesOrEquals` 对 `oldestRunningXid` 比较——xid 单调分配使
"代际"有明确定义。

`SnapBuildWaitSnapshot()`（snapbuild.c:1440）对列表里每个老 xid 做
`XactLockTableWait`（等待其事务锁释放 = 等它结束），然后主动
`LogStandbySnapshot()` 再打一条 running_xacts——不然就得干等 bgwriter 的下一个
周期。这不是正确性需要（哪怕不等，状态机也会在未来的 running_xacts 上转移），
纯粹为了缩短建槽耗时。

**运维映射**：`CREATE SUBSCRIPTION`/`pg_create_logical_replication_slot` 卡在
"waiting for transactions to end"日志上 → 第一排查项就是长事务（含 prepared
transaction——它们不结束，状态机永远毕不了业）。CONSISTENT 后快照周期性序列化
（`SnapBuildSerialize`，snapbuild.c:1502；shutdown checkpoint 时经
`SnapBuildSerializationPoint`，decode.c:143 触发），重启经分支 b 直接恢复，
免去重新等待。PG 17 起 shutdown 时还会把 confirmed_flush 落盘（commit
`e0b2eed047`，2023-09），减少重启后的重复解码。

### 13.5.6 订阅端：ApplyWorkerMain → LogicalRepApplyLoop 主线

订阅端 `CREATE SUBSCRIPTION` 后，launcher 启动 leader apply worker。入口链非常
短（`src/backend/replication/logical/worker.c:6003`）：


`SetupApplyOrSyncWorker` → `InitializeLogRepWorker()`（worker.c:5795）：以
`session_replication_role=replica` 连库（**订阅端默认不触发普通触发器**的开关
就在这）、锁住并加载 `pg_subscription` 元组。`run_apply_worker()`
（worker.c:5682）做四件事：

```c

    else
```

- **replication origin 是订阅端的本地位点**："本订阅已 apply 到发布端的哪个
  LSN"，随每个 apply 事务**原子地**记入本地（提交 apply 事务 = 同时提交位点
  推进，天然 exactly-once）。崩溃重启后从 origin 位点续订——**位点的最终真相
  在订阅端 origin，发布端槽的 confirmed_flush 只是它的影子**。与 Kafka Connect
  把 source offset 存 offset topic 同构；
- two_phase 选项要等全部 tablesync 就绪才从 PENDING 转 ENABLED——初始同步期间
  PREPARE 无法与表拷贝流正确交织；
- `start_apply()`（worker.c:5642）用 PG_TRY 包住主循环：出错时清掉 origin 的
  未提交推进，按 `disable_on_error` 决定是禁订阅退出还是重抛（重启后无限重试）。

主循环 `LogicalRepApplyLoop()`（worker.c:4003），骨架：

### 13.5.7 冲突：逻辑复制从"停摆"到"可观测"

物理复制没有冲突（备库只读），逻辑复制有——订阅端表可以被本地写。推演一次经典
停摆（what-if 之二的前半）：订阅端有人手工插入了 `id=42`，随后发布端也插入
`id=42` 并进入复制流。apply worker 执行 INSERT 撞唯一键 → ERROR → `start_apply`
的 PG_CATCH → worker 退出 → launcher 重启 worker → 从 origin 位点重放**同一个
事务** → 再撞 → 无限循环。此时订阅停止前进，发布端槽的 confirmed_flush 冻结，
`pg_wal` 开始堆积（13.2.5 的剧本再次上演）——**一行冲突数据可以最终打爆发布端
磁盘**，这是逻辑复制最反直觉的故障传导链。

处理手段的演进（也是一部设计史）：

- **PG 15**：`disable_on_error`（出错自动禁用订阅，`DisableSubscriptionAndExit`，
  worker.c:6023——至少不再空转重试、不再钉住发布端）；
  `ALTER SUBSCRIPTION ... SKIP (lsn = ...)`（commit `208c5d65bb`，2022-03）——
  跳过整个出错事务（`is_skipping_changes()` 那道闸），代价是该事务在订阅端
  永久缺失；
- **PG 18**：冲突不再只是裸 ERROR，而是分类上报（commit `9758174e2e`，
  2024-08）：`src/include/replication/conflict.h:34-55` 定义了
  `insert_exists`、`update_origin_differs`、`update_missing`、`delete_missing`
  等类型，统计进 `pg_stat_subscription_stats`；
- **19/20devel**：`multiple_unique_conflicts`（commit `73eba5004a`）、
  `update_deleted`（commit `fd5a1a0c3e`，需 `retain_dead_tuples`——订阅端保留
  死元组以便区分"行被删了"和"行从未存在"，13.5.6 里 `'s'` 消息和
  `RetainDeadTuplesData` 就是为它服务的基础设施）。

但**自动解决策略（last-write-wins 之类）内核仍不提供**——PG 的立场是：冲突检测
可以内置，冲突语义属于应用。做多活的团队要么保证写集不相交（按业务键分片），
要么用 BDR/pglogical 这类带解决策略的扩展。

### 13.5.8 逻辑复制的边界清单

大数据同学做管道设计必须背下来的三条：

1. **DDL 不复制**：逻辑流里只有行变更。发布端 `ALTER TABLE ADD COLUMN` 不传播，
   需先在订阅端执行（先加列订阅端、再加列发布端是安全顺序，反之 apply 会因列
   缺失报错）。Debezium 的 schema evolution 之痛在这里同样存在；
2. **序列**：`nextval` 的推进不进逻辑流（它甚至不是事务性的）。本版本
  （20devel）已含 `sequencesync.c`（843 行，commit `5509055d69`，2025-11）：
   序列登记进 `pg_subscription_rel`，由专门的 sequencesync worker **周期性对账
   同步当前值**，`ALTER SUBSCRIPTION ... REFRESH SEQUENCES` 触发重同步——注意
   是"定期对账"而非"逐次增量"，故障切换前仍建议校验；
3. 大对象（large object）不复制；TRUNCATE 需 publication 显式包含；双向复制
   必须配 `origin = none` 过滤防回环。

### 小结

- decode.c 是翻译层：按 rmgr 把 WAL 记录翻成 `ReorderBufferChange`，三道过滤
  （无元组/非本库/origin），不碰事务边界与表结构；
- reorderbuffer.c 是事务缓冲层：按 xid 聚合有序变更链，超过
  `logical_decoding_work_mem` 时优先流式（需下游可回滚）否则落盘
  （`pg_replslot`），提交时 k 路归并按 LSN 回放——输出流"事务完整、按提交序、
  默认无 abort"；
- snapbuild.c 用两级等待破解"解历史 WAL 需要历史目录快照"的自举悖论，四状态机
  的每次毕业都以一代在途事务的自然死亡为条件；它同时是槽 restart_lsn/
  catalog_xmin 的推进者；
- 订阅端 = walreceiver 式主循环 + 执行器级重放 + origin 原子位点；冲突会让
  订阅停摆并反噬发布端，PG 15+ 提供跳过/禁用，PG 18+ 提供分类观测，解决策略
  仍留给应用层。

## 13.6 不变式清单

每条给出"是什么—谁维护—违反了会怎样"。

**I1：主库 walsender 绝不发送未 fsync 的 WAL。**
维护者：`XLogSendPhysical` 的 `SendRqstPtr = GetFlushRecPtr(NULL)`
（walsender.c:3464）。违反后果：主库崩溃重启后覆写未落盘区间，备库已回放的
字节与主库新历史在同一 LSN 分叉，且协议层无法检测——静默数据分歧（13.1.2）。

**I2：restart_lsn 之前的 WAL 不可删除。**
维护者：`KeepLogSeg`（xlog.c:8498）以全槽最小 restart_lsn 压回收线；唯一合法
豁免是 `max_slot_wal_keep_size`，且必须同步把槽标记 invalidated。违反后果：
消费者重启后 `WALRead` 报 "segment already removed"，物理备库只能重做基础备份、
逻辑订阅只能重新全量同步；若只删文件不标槽，故障还会被推迟到下次重启才爆发。

**I3：catalog_xmin 之前的目录元组版本不可清理。**
维护者：`ReplicationSlotsComputeRequiredXmin`（slot.c:1222）汇入 vacuum 视界；
备库场景由 hot_standby_feedback 的 catalog_xmin 延伸。违反后果：解码历史 WAL
时 `RelidByRelfilenumber`/目录扫描读到已清理的版本，轻则解码报错、槽失效
（`RS_INVAL_HORIZON`，13.4.1），重则输出用错误表结构解释的行。

**I4：confirmed_flush 单调递增，且只反映客户端明确确认过的位置。**
维护者：`LogicalConfirmReceivedLocation`（logical.c:1813）。违反后果：回退会让
下游收到重复事务（下游若无幂等则脏写）；虚增（确认了未真正持久化的位置）则
崩溃后从 confirmed_flush 续传直接**丢事务**——这正是 Debezium 要求"先把消息
落 Kafka 再 ack LSN"的原因。

**I5：同步复制等待队列按 waitLSN 有序。**
维护者：`SyncRepQueueInsert`（syncrep.c:382），断言
`SyncRepQueueIsOrderedByLSN`。违反后果：`SyncRepWakeQueue` 从队头摘到第一个
未达标者就停，乱序队列会让已达标的 backend 永远等不到唤醒——提交无限挂起。

**I6：单事务内变更按 LSN 序回放。**
维护者：changes 链按 LSN 追加 + 迭代器二叉堆归并；断言
`prev_lsn <= change->lsn`（reorderbuffer.c:2287）。违反后果：同一行的
update/delete 乱序到达下游，最终状态错误（先删后插变成先插后删）。

**I7：事务只有在 CONSISTENT 点之后、且整个生命周期被观察的情况下才可输出。**
维护者：snapbuild 状态机 + `start_decoding_at` 拦截。违反后果：下游收到半个
事务，原子性破坏且无法靠重放修复（缺失的前半段永远不会再来）。

**I8：订阅端 apply 按发布端提交序、origin 位点与数据同事务推进。**
维护者：单一 leader apply worker 串行 apply（parallel 场景由 leader 协调提交
顺序）+ `replorigin_session_setup` 的事务性位点。违反后果：乱序 apply 打破
事务间因果（先见转账入账后见开户）；位点与数据不同步则崩溃后重复或丢失 apply。

## 13.7 what-if 推演实验室

### 13.7.1 备库 apply 慢，主库 WAL 堆积

**链条**：备库回放慢（单进程 redo 撞上大量随机页读，或备库盘差）→ 若备库带
物理槽，其 feedback 的 flushPtr 虽正常推进（walreceiver 落盘不慢），槽
restart_lsn 也随之推进——**慢 apply 本身不扣 WAL**；真正堆积发生在
walreceiver 也跟不上（网络/落盘慢）或断连时：restart_lsn 冻结 → KeepLogSeg
让步 → `pg_wal` 膨胀 → 磁盘满 PANIC。若同时配了 `remote_apply` 同步等待，
慢 apply 还会直接把主库提交延迟拉高——反压前移到业务层。

**三种解法对比**：
1. `max_slot_wal_keep_size` 设上限：主库自保，代价是备库可能被牺牲（invalidated
   后重建）。适合"主库可用性 > 备库存在"的场景，是生产默认推荐；
2. 无槽 + `wal_keep_size` 静态保留：回到 13.2.1 的赌博模式，只适合备库短暂
   抖动可接受重建的边缘环境；
3. 治本：提升备库回放能力（`recovery_prefetch` 预取、更好的盘）或把只读流量
   挪走让 redo 独占 IO。前两种是止血，第三种才是治疗。

### 13.7.2 逻辑订阅端唯一键冲突停摆

完整推演见 13.5.7。补充操作侧结论:发现订阅 error 循环时，止血顺序是
`disable_on_error`/手工 DISABLE（解除对发布端槽的钉扎）→ 定位冲突行（PG 18 的
`pg_stat_subscription_stats` + 日志里的冲突详情）→ 决策：删订阅端脏行重启
apply，或 `ALTER SUBSCRIPTION ... SKIP (lsn=...)` 跳过整个事务（记录审计！）。
预防上必须把"订阅端表禁止业务写"当成纪律（除非做了写集分离的多活设计）。

### 13.7.3 主备切换脑裂：PG 为什么不防

**如何发生**：网络分区把主库 A 与监控/备库 B 隔开。运维（或不带 fencing 的
脚本）判定 A 死亡，promote B。但 A 还活着且还有应用能连上它——现在两个"主库"
各自接受写入，各自推进自己的时间线历史。分区恢复后，A 的 TLI=1 历史与 B 的
TLI=2 历史自分叉点起不可调和：时间线机制（13.1.5）能**检测**分叉（A 试图作为
备库接入 B 时会被拒绝），但分叉期间两边写入的数据必须人工裁决——通常意味着
丢弃一边或痛苦的手工合并。

**PG 为什么不防**：防脑裂 = 共识问题，需要 quorum 成员管理、租约/任期、旧主
自动降级（fencing）。PG 内核的设计边界是**单主 + 日志复制**，明确不做选主：
(1) 共识协议与"单机数据库 + 可选复制"的部署谱系冲突——两节点部署根本凑不出
多数派；(2) fencing 需要控制 PG 之外的资源（VIP、存储、连接路由），内核做不到
也不该做；(3) 生态已用外部编排解决：Patroni 把成员关系放进 etcd/consul 的
共识存储，用租约到期自动降级旧主 + `synchronous_commit` 保证新主不丢已确认
事务——PG 提供的是"确认语义"这块积木（13.3），选主留给拥有全局视野的层。
对照 Kafka：KRaft 把共识内置进了 broker——那是"分布式系统内置存储"与
"数据库外挂复制"两种出身的分野。

### 13.7.4 hot_standby_feedback 反噬主库的机制

打开后（13.4.3）：备库全局 xmin 经 `'h'` 消息写入主库 walsender 的
PGPROC.xmin 或物理槽 xmin → 参与 `ComputeXidHorizons()` → 主库上
vacuum/HOT-prune 的死元组回收线被钉在备库最老查询的 xmin 上。推演最坏链条：
备库跑 8 小时分析查询 → 主库 8 小时内所有表的 UPDATE/DELETE 死元组一个都清
不掉 → 高频更新表膨胀数倍、索引随之膨胀 → 更糟的是 heap prune 停摆让 HOT 链
变长，主库查询自身变慢；关掉 feedback 的瞬间视界跳变，vacuum 又会集中爆发一轮
IO。若 feedback 经物理槽传递，**备库断连期间 xmin 仍被槽记住**——事故常见形态：
备库下线一周没删槽，主库膨胀 + xid 视界冻结一周。监控：`pg_replication_slots.xmin`
与 `age(xmin)`；对策：报表类长查询挪到逻辑副本或快照导出，物理备库只留短查询。

## 自测问题（附详解）

**1. 为什么 `XLogSendPhysical()` 在主库上只发送已 fsync 的 WAL？如果发送了未落盘的字节会怎样？**

答：见 walsender.c:3454-3464 注释与 13.1.2 推演。主库把未 fsync 的记录 R 发出
后崩溃，重启后同一 LSN 区间被新记录 R' 覆写——R 在主库历史中从未存在，备库却
已回放。协议按 LSN 续传无法发现分歧，主备静默分叉。所以发送上限钉死在
`GetFlushRecPtr()`（不变式 I1），等价于 Kafka consumer 只能读到 high watermark。

**2. 一个逻辑槽的 `confirmed_flush` 一直推进，但 `restart_lsn` 停住不动，最可能的原因与后果？**

答：发布端有长事务在途。`restart_lsn` 由"最老在途事务的 `restart_decoding_lsn`"
决定（`SnapBuildProcessRunningXacts`，snapbuild.c:1218-1219）——重启解码必须回
到该事务开始前的快照点重读 WAL。后果：`KeepLogSeg`（xlog.c:8498）的回收线被
钉住，`pg_wal` 膨胀；catalog_xmin 同步停滞导致目录膨胀；超过
`max_slot_wal_keep_size` 后槽被判 `RS_INVAL_WAL_REMOVED`。

**3. `synchronous_commit` 的 `remote_write`、`on`、`remote_apply` 各自在什么故障下丢数据或读到旧数据？**

答（13.3.1 逐级推演）：`remote_write` 只等备库写进 OS 页缓存——主库损毁 + 备库
同时掉电的双故障丢已确认事务；`on` 等备库 fsync——只有主备两边磁盘都损毁才丢，
但 failover 瞬间备库可能尚未回放（数据在、不可见）；`remote_apply` 再等备库
回放——不丢且备库立即可读，代价是提交延迟加上 replay lag。等待模式常量
`SYNC_REP_WAIT_WRITE/FLUSH/APPLY`（syncrep.h:23-25）。

**4. `SyncRepWaitForLSN` 等待期间收到 Ctrl-C 或 pg_terminate_backend，事务是什么状态？**

答：**已提交**。等待发生在本地 WAL flush 与 CLOG 更新之后（xact.c:1599），
取消的只是"向客户端报告前的复制确认等待"。代码对两种中断都只发 WARNING
（"The transaction has already committed locally, but might not have been
replicated to the standby."，syncrep.c:301-335）并退出等待，绝不报 ERROR——
报 ERROR 会让客户端误以为回滚。运维含义：同步复制卡住时杀 backend 得到的是
"提交了但可能没复制"的事务。

**5. snapbuild 为什么需要两级等待（BUILDING→FULL→CONSISTENT），一级不够吗？**

答：一级等待换来目录快照完备（初始在途事务死光后，新事务的快照不缺目录提交），
但 FULL 时刻仍在途的老事务，其前半段行变更发生在收集开始之前——输出即半个
事务；又不能丢弃（可能提交）。只能再等这一代终结（snapbuild.c:1383/1407 两个
转移条件都是 `next_phase_at <= running->oldestRunningXid`）。捷径：无在途事务
时直接 CONSISTENT（1299），磁盘有序列化快照时 `SnapBuildRestore`（1329，建槽
时禁用）。运维推论：建槽卡 catchup 先查长事务与 prepared transaction。

**6. reorderbuffer 为什么用二叉堆做 k 路归并，而不是把整个事务的变更排序？**

答：各（子）事务的 changes 链在入队时就按 LSN 追加（WAL 读取本身有序），排序
是白费——需要的只是多条有序流的归并。堆里每个元素是一条链的游标
（`ReorderBufferIterTXNInit`，reorderbuffer.c:1285），每次取全局最小 LSN 再压
入该链下一条，复杂度 O(N log k)（k=含变更的子事务数）且支持"链尾在磁盘上分批
恢复"（`ReorderBufferIterTXNNext`，1458-1493）——全量排序则要求全部变更同时
在内存，与 spill 机制矛盾。

**7. 逻辑复制的 streaming 模式打破了"只在提交时输出"，它靠什么保住正确性？**

答：三个前提（13.5.3）：下游可回滚（订阅端临时文件缓存到 stream_commit 才
apply，或 parallel apply worker 用真实事务，abort 时回滚）；上游能探测并发
中止（`SetupCheckXidLive`，reorderbuffer.c:2299，探测到即停流，之后补
`stream_abort`）；不完整语义单元不流（TOAST chunk 与主行绑定，
`rbtxn_has_partial_change` 的事务不进流式候选，reorderbuffer.c:3858）。

**8. 订阅端崩溃重启后，为什么既不会丢事务也不会重复 apply 事务？**

答：位点（replication origin）与数据在**同一个本地事务**里推进：apply 事务
提交 = 数据 + "已应用到发布端 LSN X" 原子落盘。重启后
`replorigin_session_get_progress()`（worker.c:5714）取回位点，从 X 续订——
X 之前的事务不会再收（发布端从 confirmed_flush/请求位点续传），X 之后的事务
本地没有任何痕迹（连位点都没推进）所以整体重放。失败路径上
`replorigin_xact_clear`（start_apply 的 PG_CATCH，worker.c:5656）防止"位点
推了数据没提交"。这就是内核原生的 exactly-once sink。

**9. 发布端删除了一个大表的分区（DROP TABLE 子分区），物理备库和逻辑订阅端各会发生什么？**

答：物理备库：redo 要拿该表的 AccessExclusiveLock，与备库正在查它的会话冲突
（lock conflict，`ResolveRecoveryConflictWithLock`，standby.c:626），按
`max_standby_streaming_delay` 等待后取消对方查询——hot_standby_feedback 帮不上
忙（它只消 snapshot 冲突）。逻辑订阅端：**什么都不发生**——DROP 是 DDL，不进
逻辑流；订阅端该表继续存在，后续发布端若复用同名表还可能因结构不一致 apply
报错。两条链路对 DDL 的处理差异正是"字节级 vs 语义级"（13.0）的直接推论。

**10. 为什么逻辑槽必须绑定单个数据库，而物理槽不用？**

答：解码要用**历史快照读该库的系统目录**（表结构、类型信息，13.5.4），目录是
库级的，跨库读目录在 PG 进程模型里不可能（一个 backend 只连一个库）；
`DecodeInsert` 也据此过滤非本库记录（decode.c:936）。物理槽只搬字节、不解释
内容，WAL 是实例级的，自然全局有效（`ReplicationSlotCreate` 的 `db_specific`
参数，slot.c:378）。
