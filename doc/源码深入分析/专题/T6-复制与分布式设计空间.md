## 1. 问题定义：复制四问与 CAP 的工程化表述

### 1.1 复制四问

任何数据复制系统，无论叫流复制、binlog、Raft 还是 quorum，都必须回答四个问题。
这四问是本篇的坐标系，后文每个方案都会被放回这个坐标系里：

**Q1 复制什么（What）——抽象层次决定灵活性与写放大。**

```
物理字节（页面修改）── PG 物理复制、Oracle Data Guard
行级逻辑变更        ── PG 逻辑复制、MySQL row binlog、Debezium CDC
语句               ── MySQL statement binlog（史前方案，非确定性函数即翻车）
日志记录本身        ── Aurora/Neon："只复制 redo，页面由存储层自己长出来"
```

层次越低，一致性越"免费"（字节相同则语义必然相同），但异构灵活性越差；
层次越高越灵活（跨版本、部分表、可写副本），但一致性要靠机制去挣——这是
第 13 篇 §13.0 的核心结论，本篇把它推广到 PG 之外：**MySQL 从语句级走到行级、
Aurora 从页面级退到日志级，全行业在这条轴上反复搬家**。

**Q2 何时确认（When）——确认点的位置就是 RPO 的位置。**

commit 返回客户端之前，复制进行到了哪一步？这条谱系每一档对应一种故障下的
数据丢失量（RPO）：

```
不等（异步）→ 等对端收到（内存）→ 等对端落盘 → 等对端回放可见 → 等多数派落盘
PG:  off      remote_write         on            remote_apply     ANY n(...)
```

PG 把整条谱系做成了每事务可调的 `synchronous_commit`（13.3.1），这是 PG 复制
设计中最被低估的一件事：**确认级别是事务属性而非实例属性**。Kafka 用户会立刻
认出这是 producer 的 `acks=0/1/all` ——同样是把 RPO 决策下放到每条消息。

**Q3 谁能写（Who）——写拓扑决定冲突是否存在。**

```
单主（primary-replica）：冲突不存在，代价是写扩展性为零        ── PG、MySQL 默认
多主（active-active）  ：任意节点可写，冲突必然存在，必须裁决  ── BDR/pgEdge、MySQL GR 多主
无主（leaderless）     ：客户端写 quorum，读时修复            ── Dynamo/Cassandra 系
分片单主               ：全局多写、每分片单写，冲突被划分掉    ── CockroachDB/TiKV/Citus
```

第四种是当代分布式 SQL 的主流答案：不解决冲突，而是用分片把"多主"退化为
"很多个单主"——冲突问题变成跨分片事务问题（2PC/共识）。

**Q4 谁来仲裁（Arbiter）——故障时谁有权宣布"现在谁是主"。**

这是四问中最深的一问，也是 PG 与几乎所有后来者分道扬镳的地方：

```
人肉仲裁          ── 原生 PG：pg_ctl promote 由 DBA/脚本执行
外部共识（DCS）   ── Patroni：选主逻辑放在 etcd/consul/k8s，PG 只当执行者
内置共识（数据面）── Raft 系：数据复制协议本身就是选主协议
存储层 quorum     ── Aurora/Neon：计算层无状态，"谁是主"退化为"谁持有存储租约/任期"
```

Kafka 的历史恰好把这条轴走了一遍：早年靠 ZooKeeper（外部 DCS，对应 Patroni
形态），KRaft 之后共识内置（对应 Raft 系形态）。PG 停在第一档，把第二档留给
生态——这是否正确是第 3 节的主题。

### 1.2 CAP 的工程化表述：从定理到 RPO/RTO

CAP 的常见误读是"三选二"。工程上有用的表述是两条：

1. **分区时（P 发生）**：要么拒绝写入保一致（CP），要么继续写入保可用（AP）——
   这是二选一，没有第三条路。对单主复制系统，这个选择具体化为：**主库联系不上
   足够的副本时，提交是挂起还是继续？** PG 的 `synchronous_standby_names` 配满
   而备库全挂时提交无限等待（CP 行为）；MySQL 半同步超时
   （`rpl_semi_sync_master_timeout`，默认 10 秒）后**静默退化为异步**（AP 行为）——
   同一个问题，两个内核给了相反的默认答案，各自成为一类事故的来源（§6.8）。
2. **无分区时（PACELC 的 ELC 部分）**：延迟与一致性的连续交换。每多等一个副本
   ack，RPO 减小、提交延迟增加。Abadi 的 PACELC（2012）把这半边补上后，
   "复制配置"才成为一个完整的工程决策：**P 时选 A 还是 C，E（else，平时）选
   L 还是 C**。

对运维语言的翻译：

| 理论量 | 运维量 | 决定因素 |
|---|---|---|
| 分区时选 C | RPO = 0 | 确认点在"多数派/指定副本已持久化"之后 |
| 分区时选 A | RPO > 0（丢失窗口 = 复制滞后） | 异步复制或超时退化 |
| 恢复速度 | RTO | 仲裁机制：人肉（分钟~小时）、DCS 租约（~TTL）、内置共识（~选举超时，秒级） |

一个容易忽略的推论：**RPO=0 与自动 failover 必须同时设计，缺一个则另一个的
承诺作废**。异步复制 + 自动切换 = 自动化的数据丢失；同步复制 + 人肉切换 =
RPO 为零但 RTO 以小时计。Patroni 文档反复强调 `synchronous_mode` 与自动
failover 的联动（见 §3.3），根源即此。

## 2. 方案谱系与优劣矩阵（核心）

### 2.1 谱系总览

按"共识（仲裁）位于系统的哪一层"排列，六个代表方案：

```
仲裁在系统外（人/脚本/外部DCS）              仲裁内置于数据路径             仲裁下沉到存储层
├── PG 物理流复制（+Patroni 补仲裁）         ├── MySQL 组复制（XCom/Paxos）  ├── Aurora（6副本quorum）
├── PG 逻辑复制                             ├── Raft 系（etcd/TiKV/CRDB）   └── Neon（safekeeper Paxos）
└── MySQL binlog（异步/半同步）
```

### 2.2 六个方案逐个剖析

#### A. PG 物理流复制（详见第 13 篇，此处只放坐标）

四问答案：字节级 / `synchronous_commit` 全谱系可调 / 严格单主 / **不仲裁**。
机制内核：walsender 只发已 fsync 的 WAL（不变式 I1），复制槽提供"未消费不删"
的强保证（slot.c），timeline 提供分叉**检测**（不是分叉预防，13.1.5）。
一句话定位：**复制是 WAL 的外挂消费者，HA 是别人的事**。

#### B. PG 逻辑复制

四问答案：行级 / 同样可挂到 syncrep（逻辑 walsender 也是 walsender）/ 订阅端
可写（多活的原料）/ 不仲裁。写放大显著：订阅端走完整执行器路径并产生自己的
WAL。截至 20devel，DDL 仍不复制（§5.2），冲突从"检测并停摆"演化到"分类上报
+ 保留死元组 + 可入表记录"（§5.3），但**内核拒绝自动裁决冲突**——这个立场
在 §4.5 分析。

#### C. MySQL binlog：异步、半同步与组复制

MySQL 是与 PG 平行演化的最佳对照组，三个阶段各有教训：

- **异步 binlog**：与 PG 异步流复制同构，但多一层结构性弱点——binlog 是
  **独立于 InnoDB redo log 的第二份日志**，两者靠内部 XA（两阶段提交）保持
  一致，fsync 次数与崩溃恢复复杂度都翻倍。PG 只有一份 WAL，物理与逻辑复制
  共用同一日志源（逻辑解码直接吃 WAL），这是 PG 架构干净得多的地方。
  MySQL 8.0 的 clone plugin + binlog 与 PG 的 basebackup + WAL 大体同构。
- **半同步（semi-sync）**：等一个副本 ack 后才向客户端返回。5.7 起默认
  `AFTER_SYNC`（"lossless semi-sync"）：**主库先把 binlog 刷盘并等 ack，
  之后才做引擎层 commit（事务才可见）**；旧的 `AFTER_COMMIT` 则先本地提交
  可见再等 ack——主库崩溃时可能出现"客户端没收到成功、其他会话却已读到、
  而副本上没有"的幻影事务（来源：MySQL 5.7 Reference Manual §16.3.9；
  oneuptime.com 半同步机制分析，2026-03）。**PG 的 syncrep 在这个坐标上是
  一个混合体**：commit record 先本地 fsync（持久化次序上像 AFTER_COMMIT），
  但 `SyncRepWaitForLSN` 在 `ProcArrayEndTransaction` 之前等待，事务对
  其他会话尚不可见（可见性次序上像 AFTER_SYNC）；崩溃重启后该事务**会**
  变得可见——这是 §7 自测第 2 题的完整展开。
- **组复制（Group Replication，2016，MySQL 5.7.17+）**：把 Paxos 变体 XCom
  内置进 server。提交时把写集（write set）经 XCom 广播，**多数派对"事务的
  全局顺序"达成共识后**，各成员独立做冲突认证（certification，乐观并发检查），
  通过则按全序应用。支持单主/多主模式，内置成员管理与故障检测——**仲裁在
  数据路径里**，脑裂由多数派原则防住（少数派分区自动降为只读）。代价：提交
  延迟至少一轮组内多数派往返；吞吐受最慢多数派成员牵制；认证冲突在多主模式
  下表现为提交时回滚（对应 PG 逻辑复制 apply 时冲突，只是时点提前到了提交）。
  （来源：MySQL 官方博客 "The king is dead, long live the king: our homegrown
  Paxos-based consensus"，2016；dev.mysql.com Group Replication 文档。）

#### D. 共识复制：Raft 系（etcd / TiKV / CockroachDB）

Raft（Ongaro & Ousterhout，2014）把复制日志与选主合并成一个协议：日志复制
到多数派才 commit，任期（term）单调递增防旧主，选举超时自动产生新主。三种
工程用法体现规模差异：

- **etcd**：整个数据集一个 Raft 组。简单，但写吞吐 = 单组吞吐，只适合元数据。
- **TiKV / CockroachDB**：**multi-raft**——数据按 range 切分（CRDB 默认
  512MB），每个 range 是独立 Raft 组，一台节点承载成千上万组（心跳合并、
  quiescence 等工程优化）。写扩展性来自组的数量；跨 range 事务再叠一层
  2PC + 时间戳排序（CRDB 的 HLC，TiDB 的 TSO/Percolator）。
- **Patroni 用法**（PG 生态的答案）：**共识只管元数据（谁是主），不在数据
  路径上**。leader key 带 TTL 存于 etcd/consul/ZooKeeper/k8s API；只有能
  续租 leader key 的节点才允许以主库身份运行 PG，续租失败立即自我降级为只读
  （来源：Patroni 4.1.x 文档，readthedocs，2026-05）。数据路径仍是 PG 原生
  流复制——**Raft 的选主与 PG 的日志复制拼装成完整系统**。RPO=0 需要再叠
  `synchronous_mode`（让 PG 的确认语义与 Patroni 的候选人资格联动）。

与 Kafka 对照记忆：etcd ≈ KRaft controller quorum；TiKV 的 multi-raft ≈
每个 partition 一个 ISR 组；Patroni ≈ ZooKeeper 时代的 Kafka（controller
选举在外部，数据复制在内部）。

#### E. 存储层复制之一：Aurora 的 6/4/3 quorum

Aurora（SIGMOD'17 "Design Considerations for High Throughput Cloud-Native
Relational Databases"）把复制从数据库层整个搬进定制存储服务：

- 数据 6 副本，横跨 3 个 AZ（每 AZ 2 份）；**写 quorum 4/6，读 quorum 3/6**
  （W+R > N，且 W > N/2 防两个写 quorum 并存）。6/4/3 的设计目标是"AZ+1"
  容错：整个 AZ 失联再加一台机器故障，仍不丢写（读 quorum 仍可凑齐重建）。
- **"the log is the database"**：计算节点只把 redo log record 发给存储节点，
  永不写整页；存储节点自己持续物化页面。checkpoint 消失、崩溃恢复几乎为零
  （恢复 = 存储层继续 apply，计算节点重启即用）；跨网络字节数骤降（对比 PG
  物理复制还要传 FPI 整页镜像）。
- 微妙之处（SIGMOD'18 续篇 "On Avoiding Distributed Consensus for I/Os,
  Commits, and Membership Changes"）：Aurora **刻意避免运行共识协议**——
  因为**只有一个写者**（单主计算节点），LSN 由单点单调分配，正常路径上
  quorum 只是"计数到 4"，不需要 Paxos 的提案编号竞争；成员变更用 epoch
  标记 + 存储元数据完成。**单写者假设让 quorum 退化为廉价的多数派计数**——
  这解释了为什么 Aurora 读副本仍是"一写多读"，写扩展要另起炉灶（Aurora
  Multi-Master 长期只是边缘产品）。
- 读路径实况：正常读**不走读 quorum**——计算节点知道每个 segment 的完成度，
  直接找一个足够新的副本读；读 quorum 只在崩溃恢复重建状态时使用（来源：
  AWS Database Blog "Amazon Aurora under the hood: quorums and correlated
  failure"，2017）。

#### F. 存储层复制之二：Neon 的 safekeeper Paxos

Neon（开源，2025-05 被 Databricks 收购）把 Aurora 思路拆得更细，并且角色
分配可以直接用 Paxos 术语描述（来源：neon 仓库 `docs/walservice.md`、
`docs/safekeeper-protocol.md`；Jack Vanlightly 的分析 2023-11 与
"Log Replication Disaggregation Survey: Neon and MultiPaxos"，2025-02-19）：

- **compute（无状态 PG）= proposer**：WAL 不落本地盘，直接流向 safekeeper；
- **safekeeper × 3 = acceptor**：多数派落盘即事务持久（commit ack 点）；
  运行 Paxos 式协议管理任期（term），防两个 compute 同时自认主库——**PG
  内核不做的 fencing，Neon 在存储协议层补上了**；
- **pageserver = learner**：消费已提交 WAL，切成 per-page 增量，按
  `GetPage@LSN` 服务计算层，材料化页面 + 分层存储（S3）。

与 Aurora 的关键差异：Aurora 存储节点既存日志又材料化页面（一体）；Neon 把
"日志持久化"（safekeeper，延迟敏感、小集群）与"页面服务"（pageserver，
吞吐敏感、可单独伸缩、可重建）分开——代价是多一跳，收益是 branching/PITR/
scale-to-zero 都变成 pageserver 层的纯读操作。**PG 内核在其中几乎未改**：
Neon 修改点集中在 smgr 与 WAL 写出路径，复用了 PG 复制协议的大部分——这是
"PG 复制机制作为可拆积木"的最有力证据（§3 会引用这一点）。

### 2.3 统一维度优劣矩阵

一致性保证指"故障切换后已确认事务是否幸存"（≈RPO）；RTO 指无人工介入时
恢复写入的时间；写放大指同一逻辑写在全系统落盘的份数/路径数；异构灵活性指
跨版本/部分复制/副本可写；脑裂防护指旧主能否被机制性阻止继续接受写。

| 方案 | 一致性保证（RPO） | RTO | 写放大 | 异构灵活性 | 脑裂防护 |
|---|---|---|---|---|---|
| PG 异步流复制 | 丢复制滞后窗口 | 无自动切换：人肉分钟级 | 低：字节搬运+重放 | 无（同版本同平台整实例） | **无**（timeline 仅事后检测分叉） |
| PG 同步流复制（FIRST/ANY） | RPO=0（指定/多数副本落盘） | 同上；配 Patroni 后 ~TTL（默认 30s 量级） | 低 | 无 | 无（内核层）；Patroni 层租约 fencing |
| PG 逻辑复制 | 取决于挂不挂 syncrep；位点由槽持久 | 不适用（非 HA 机制） | **高**：订阅端完整执行器+自身 WAL+索引 | **最高**：跨大版本/选表选行选列/订阅端可写 | 无；多活时靠 origin 防回环，冲突另论 |
| MySQL 半同步（AFTER_SYNC） | 名义 RPO=0，**超时静默降级为异步** | 外部工具（Orchestrator 等）秒~分钟 | 中：binlog+redo 双日志 | 中：行级 binlog 可异构消费 | 无（外部工具负责，同 PG） |
| MySQL 组复制 | RPO=0（多数派共识排序） | 内置：秒级（故障检测+视图变更） | 中偏高：双日志+组内广播 | 低-中（组内同构；多主受限模式） | **内置**：少数派自动只读 |
| Raft 系（etcd/TiKV/CRDB） | RPO=0（日志多数派提交） | **内置：选举超时，亚秒~秒级** | 中：3 副本日志+各自状态机 | 低（协议内同构副本） | **内置**：term+多数派，教科书级 |
| Aurora 6/4/3 | RPO=0（4/6 落盘才 ack） | 计算节点重启/切换 ~30s；存储层自愈 | 特殊：**6 份日志**但 0 份整页跨网 | 无（AWS 专有；只读副本共享存储） | 存储层租约/epoch fencing 单写者 |
| Neon safekeeper | RPO=0（safekeeper 多数派落盘） | compute 无状态，重新挂载秒级 | 3 份 WAL + pageserver 材料化 | 低（但 branch/PITR 极廉价） | **协议内置**：term 防双主写入 WAL |

读矩阵的三个结论：

1. **脑裂防护一列与"共识在哪一层"完全同构**：仲裁在系统外的方案（PG、MySQL
   半同步）该列为"无"，必须靠外部租约；仲裁在数据路径或存储协议里的方案
   天然拥有 fencing。这不是巧合：fencing 的本质是"旧主的写必须被多数派拒绝"，
   而只有参与每次写确认的组件才有资格拒绝。
2. **写放大与异构灵活性正相关**（Q1 的层次定律）：逻辑复制写放大最高、灵活性
   也最高；Aurora 用"只传日志"把写放大的跨网部分打到理论下限，但换来最低的
   灵活性（连自建都不行）。
3. **RPO=0 有三种价格**：等一个指定副本（PG sync：最脆，单点依赖备库存活）、
   等多数派（Raft/GR/Aurora：抗单点，但延迟=组内次慢者）、等且可退化
   （MySQL 半同步：默认价格最低，故障时账单最贵）。

## 3. PG 的设计边界论："数据库不做选主"是哲学还是包袱

第 13 篇 §13.7.3 已给出机制层答案（timeline 只检测不预防、fencing 需要控制
PG 之外的资源）。本节把它上升为设计论辩，正反各三条，然后用生态现状裁决。

### 3.1 正方：这是哲学——"机制 vs 策略"的正确切分

**论点一：部署谱系论。** PG 的部署形态从树莓派单机到 128 核专用机到 k8s
StatefulSet，跨五个数量级。共识需要 ≥3 个投票成员才有意义，而 PG 最常见的
HA 形态恰恰是**两节点主备**——内置共识对多数用户是无法启用的死代码，或者
更糟：诱导用户在两节点上跑"共识"，得到虚假的安全感。MySQL 组复制的最低
三成员要求使它至今未成为 MySQL HA 的多数派方案，反而是外部仲裁的
Orchestrator/MHA 长期占主流——这佐证了部署谱系论不是 PG 的借口。

**论点二：fencing 完备性论。** 选主协议只能保证"新主合法"，不能保证"旧主
无害"。旧主的危害面包括：仍持有 VIP/DNS、连接池仍指向它、应用仍持有旧连接、
异步从库仍在从它拉 WAL。这些资源全部在数据库进程的控制范围之外——**内核内置
选主只能完成 fencing 的一半，另一半必须由掌握基础设施的层完成**（STONITH、
VIP 漂移、k8s 删 Pod）。既然无论如何都需要外部编排层，内核内置的那一半就是
重复建设。Patroni 的实现精确对应这个分析：它不发明共识（复用 etcd），只做
"续租失败 → 立即降级本地 PG"这个胶水动作，因为这正是内核做不了、编排层
擅长的部分。

**论点三：积木质量论。** PG 提供的是被验证过的确认语义积木：
`synchronous_commit` 全谱系、`pg_last_wal_replay_lsn()` 可比较位点、
`pg_ctl promote` 干净的时间线切换、`pg_rewind` 旧主回收。Neon 能把 PG 拆开
只换存储层、Patroni 能用 200ms 的循环把 PG 变成准 CP 系统，恰恰因为这些
积木边界清晰、语义可组合。对照 MySQL：组复制与半同步与异步是三套并行机制，
组合语义（如组复制+异步灾备链路）的坑位远多于 PG 的单一 WAL 源。

### 3.2 反方：这是包袱——默认体验的债务在复利

**论点一：默认不安全论。** 开箱即用的 PG 是"异步复制 + 无仲裁 + 无 fencing"，
每个生产用户都必须自己完成从这里到可用 HA 的最后十公里。行业数据是残酷的：
每一代 PG 用户都重演过"脚本误 promote 双主 → timeline 分叉 → 人工裁决丢
一边数据"的事故。而 20 年来内核的回应始终是文档而非机制。"哲学正确"不能
抵消"多数用户在最重要的问题上得不到默认保护"。

**论点二：事实标准论（自我反驳的哲学）。** 如果选主真的不该数据库生态管，
那 Patroni 不会成为事实标准——**需求是刚性的，内核不做只是把它推给了生态**，
而生态方案的总复杂度（PG + Patroni + etcd 三个分布式系统的运维面）高于
内置方案。用户实际部署的"PG HA"平均包含 5+ 个组件（PG、Patroni、etcd×3、
HAProxy/vip-manager、pgBackRest），每个组件间接口都是事故面。MySQL 组复制
+ MySQL Router 两个组件就闭环。

**论点三：语义裂缝论。** 外挂仲裁与内核语义之间存在永久性裂缝。例证：
(a) Patroni 必须小心翼翼地动态改写 `synchronous_standby_names` 才能保证
"新主一定拥有全部已确认事务"，而这个改写与 PG 的 GUC reload 时序之间存在
已知的竞态窗口，Patroni 用额外的状态机规避；(b) syncrep 等待可被取消/超时
打断（syncrep.c:331 的 WARNING，"transaction has already committed locally"），
外部工具**无法检测**哪些事务处于这种"本地已提交未复制"状态——failover 后
这些事务是否幸存成为薛定谔态（§6.8）；(c) PG17 之前逻辑复制槽不随 failover
转移，任何外部工具都无法弥补（订阅者位点在新主上不存在），最终还是要内核
出手（failover slots，§5.4）——**证明"内核不管 HA"的边界一直在被现实往回推**。

### 3.3 生态现状裁决（2026-07 查证）

- **Patroni**：活跃开发中，当前 4.1.x（readthedocs 显示 4.1.3 文档 PDF 日期
  2026-05-05，最新分支 4.1.4）。关键演化方向值得细读，因为每一步都在填
  "外部仲裁"固有的坑：
  - `failsafe_mode`（3.0 引入，4.x 完善）：**DCS 整体不可用时**（etcd 挂了），
    旧行为是主库自我降级（把 DCS 可用性变成了数据库可用性——外部仲裁方案的
    经典自伤）；failsafe 模式下主库改为轮询全体成员，全员可达则维持主身份。
    这本质上是**在 DCS 失效时退化出一个临时的成员制仲裁**——Patroni 被迫
    长出了半个共识协议。
  - `synchronous_mode: quorum`（4.0 起）：Patroni 接管
    `synchronous_standby_names` 写成 `ANY k(...)` 并把"参选新主资格"限制为
    quorum 成员——把 PG 的 ANY 语义（syncrep_gram.y:70，SYNC_REP_QUORUM，
    syncrep.h:36）与选主资格闭环，堵上 §3.2 论点三(a) 的裂缝。
    （来源：patroni.readthedocs.io，dcs_failsafe_mode 与 dynamic
    configuration 页，2026-05。）
- **pg_auto_failover**：另一种哲学——不用 DCS，用**单一 monitor 节点 + 状态机**
  仲裁（monitor 本身用 PG 存状态，可再套 HA，递归尴尬）。项目已从 citusdata
  组织移至 hapostgres 社区组织，版本长期停留在 2.2 系列，以维护性修复为主
  （Debian 打包 2.2-3 于 2025-11、2.2-4 于 2026-05 入库；来源：GitHub
  hapostgres/pg_auto_failover、tracker.debian.org，2026-07 检索）。事实上
  已让出主流位置。二者的市场分布本身是个裁决：**放弃了外部共识的
  pg_auto_failover 衰落，全面拥抱外部共识的 Patroni 统治**——需求侧要的
  从来不是"简单"，是"仲裁的正确性"。
- **对照 MySQL 组复制**：内置仲裁换来了两个 PG 生态没有的能力（分区少数派
  自动只读、无外部依赖的秒级切换），但也付出了 PG 用户看得见的代价：组内
  流控（flow control）使吞吐受最慢成员牵制、大事务在认证阶段的限制、多主
  模式的一长串使用禁区（外键+多主的已知问题等）。两条路线在 2026 年谁也
  没有吃掉谁。

### 3.4 裁决

本篇的立场：**在 2014 年这是哲学，在 2026 年它正在变成可管理的包袱**。
正方三论在"内核不内置共识"上依然成立——部署谱系与 fencing 完备性没有变化；
但反方论点三揭示的趋势是真实的：failover slots（PG17）、slot 同步 worker、
`synchronized_standby_slots`（walsender.c:1836 一带，发送逻辑变更前等待
指定物理槽确认，防止订阅者超前于候选新主）——内核这十年一直在**为外部
failover 工具铺路而不越界选主**。边界没有消失，但在精确地内移：从"HA 全部
是外面的事"移到"**切换的决定是外面的事，切换的可行性是内核的事**"。这条
新边界（decision vs feasibility）比旧口号（"PG 不做 HA"）更能预测未来的
内核演化——§5 的每个新特性都落在这条线上。

## 4. 分布式 PG 路线图：每条路保留了什么、放弃了什么

"把 PG 变成分布式数据库"有三条已被验证的路线，加一条永远在路上的（多主）。
评价每条路线用同一把尺：**保留了 PG 的什么（兼容性资产），放弃了 PG 的什么
（单机语义），新增了什么复制机制**。

### 4.1 路线总览

| 路线 | 代表 | 保留 | 放弃 | 复制机制 |
|---|---|---|---|---|
| 扩展内嵌 | Citus | 内核 100%（是个 extension） | 全局快照一致性、部分跨分片 SQL | 沿用 PG 流复制（每 worker 各自 HA）+ 2PC |
| 协议仿制 | CockroachDB | 仅 wire protocol 与方言子集 | 整个 PG 内核 | per-range Raft |
| 上层移植 | YugabyteDB | 真实 PG 查询层（fork） | 堆表/WAL/缓冲池等整个存储层 | per-tablet Raft |
| 存算分离 | Aurora / AlloyDB / Neon | 内核 95%+（改 smgr/WAL 写出） | 本地存储路径、（部分）checkpoint | 存储层 quorum/Paxos |
| 多主 | BDR→PGD / pgEdge | 内核全部（extension）| **单主的无冲突世界** | 逻辑复制网状拓扑 |

### 4.2 Citus：留下全部内核，放弃"单一系统"错觉

Citus 是纯 extension（planner/executor hook + 元数据表），coordinator 把
分布式表按 shard 路由到 worker。当前状态：Citus 13.0（2025-02-06 发布）
支持 PG 17，团队属 Microsoft，持续演进（来源：citusdata.com 博客与
PostgreSQL 官网新闻，2025-02）。

- **保留**：每个 shard 内是原汁 PG——索引类型、触发器、扩展、单 shard 事务
  语义全部保真。升级随内核大版本走。
- **放弃**：(1) **全局快照**——跨 worker 没有统一的 MVCC 快照，跨分片读
  不保证看到一致性切面（各 worker 各自拍快照，之间有偏斜）；多分片写用 2PC
  保原子，但 2PC 的 prepare/commit 窗口对并发读暴露中间态；(2) 跨分片唯一
  约束/外键仅限亲和（co-located）分布列；(3) 复制与 HA 完全没有新机制——
  每个 worker 自己配流复制 + Patroni，**分布式层不管副本**（早年的
  statement-based shard replication 已被弃用）。
- 对本篇主题的启示：Citus 证明了"分布式 SQL 层"与"复制层"可以完全解耦——
  但代价是故障单元乘法：N 个 worker × 各自的 HA 拓扑，任何一个 worker 的
  failover 都是一次局部事故。

### 4.3 CockroachDB 与 YugabyteDB：两种"PG 兼容"的诚实度

两者同为 "PG 接口 + 自研分布式存储 + per-shard Raft"，但"PG 兼容"的含义
天差地别，恰是"保留什么"的两个极端：

- **CockroachDB**：只保留 **wire protocol**（pgwire）与 SQL 方言近似，查询层
  用 Go 从零重写，存储用自研 Pebble（RocksDB 系）。数据按 range（默认 512MB）
  切分，每 range 一个 Raft 组，leaseholder 承接读；跨 range 事务用 HLC 时间戳
  + 意向写实现 serializable（后补了 read committed）。放弃内核的后果是永久的
  兼容性追赶：存储过程、扩展生态、pg_catalog 边角、类型细节都要逐个重造。
- **YugabyteDB**：fork **真实的 PG 查询层**（当前基于 PG 15 的查询层，2025
  状态），把 heap/索引/WAL 之下整个换成 DocDB（RocksDB 系），表切成 tablet，
  每 tablet 一个 Raft 组。保留查询层意味着 PL/pgSQL、触发器、大部分扩展语义
  免费正确；代价是**跟随内核大版本的移植成本巨大**（PG 15 → 每次 rebase
  都是外科手术，第 13 篇读者应能想象把执行器焊在异质存储上的胶水厚度）。
- 兼容度量化参考：第三方 PG 兼容性指数 YugabyteDB 85.08% vs CockroachDB
  53.66%——**注意该数字由 YugabyteDB 方发布，有立场**，但方向与两者架构
  选择的先验预期一致（来源：yugabyte.com 对比页，2025；检索 2026-07）。
- 复制视角的共同点：**它们都用"分片单主 + Raft"回答了复制四问的 Q3/Q4**，
  从而根除了 PG 生态的两大痛点（外部仲裁、脑裂）；共同代价是每次写至少一轮
  多数派网络往返（PACELC 的 EL 端被吃掉），单机点查延迟对 PG 原生没有优势。

### 4.4 存算分离三变体：Aurora、AlloyDB、Neon

三者共享同一个洞察（WAL 已经是"变更的唯一真理"，那就让存储层直接消费 WAL），
分歧在"存储层长什么样"：

- **Aurora**（2014 上线，SIGMOD'17）：§2.2E 已析。保留：接口层面近乎完整的
  PG（计算节点就是改造的 PG）。放弃：checkpoint、全页写、本地存储路径；
  用户放弃自托管。复制角色：**quorum 属于日志段**，页面是日志的物化视图。
- **AlloyDB**（Google，2022 GA）：同构思路——计算层 PG + 分离的日志处理
  服务（LPS，log processing service）在存储层持续 apply WAL 生成块，区域级
  多副本；叠加列式内存加速与 ML 调优（来源：Google Cloud "AlloyDB under
  the hood: intelligent, database-aware storage"，2022-05；此段架构信息
  为作者背景知识 + 该文，未在本次检索中重新核对细节）。定位与 Aurora 几乎
  正面对撞，差异化押在 HTAP（列存加速）上。
- **Neon**（§2.2F 已析）：唯一开源的一个，也是唯一把"日志共识"（safekeeper）
  与"页面服务"（pageserver）拆开的一个。放弃得最少（内核补丁面小、可自托管）、
  新增能力最激进（copy-on-write branch、scale-to-zero、PITR 即开分支）。
  2025-05 被 Databricks 收购后向 "lakebase" 叙事靠拢（来源：neon.com 架构
  文档，2026 检索）。
- 三变体对复制四问的共同回答值得铭刻：**Q2（何时确认）被移交给存储层 quorum，
  Q4（仲裁）退化为存储租约**——计算层单写者由存储层的 epoch/term 机制 fence，
  数据库内核里那个"谁是主"的千古难题，被"计算节点根本没有状态"消解了。
  这是对 §3 边界论最优雅的回答：PG 不内置选主，反而使它成为唯一能被这样
  拆解的主流内核（MySQL 的双日志结构让同样的手术难做得多——binlog 与 redo
  谁是真理源？Aurora MySQL 版为此付出了巨大的工程代价）。

### 4.5 active-active：冲突解决的本质困难

多主逻辑复制的谱系：2ndQuadrant BDR →（商业化）EDB Postgres Distributed
（PGD，v5/v6，含 commit scopes：group commit / CAMO / lag control 分级确认）；
pgLogical 血统 → pgEdge 的 **Spock** 扩展（开源多主，主打分布式边缘部署）。
（来源：enterprisedb.com PGD 文档；pgedge.com 博客 "Introducing pgEdge
Distributed PostgreSQL (and Spock)" 及冲突管理篇，2023–2025；检索 2026-07。）

工程现状：默认冲突策略几乎都是 **last-update-wins（LWW，按提交时间戳）**+
可配置 resolver；pgEdge 补充 "delta apply / conflict-free delta columns"——
对计数器类列不复制新值而复制增量（轻量 CRDT），从根上避免"两边同时 +1 只
留一个"的丢失。冲突事件入表可审计（Spock 的 conflict tracking，与 PG 内核
2026-07 刚合入的冲突日志表 a5918fddf1 思路趋同，见 §5.3）。

**为什么说冲突解决存在本质困难，而不只是工程未竟**：

1. **LWW 在语义上就是"静默丢写"**：两个并发 UPDATE，时间戳晚者胜，早者的
   写连同它所基于的业务决策整体蒸发，且应用无从感知。时钟偏斜进一步让
   "晚"失去物理意义。LWW 不是解决冲突，是给丢数据一个确定性的仪式。
2. **行级 resolver 无法保护跨行不变式**：转账（A 扣 B 加）在两个节点各自
   满足约束，合并后可以双双成立而总额不守恒；唯一约束在全局层面根本无法由
   行级裁决保证（两节点各插一行 email 相同的用户，resolver 留哪个都违反了
   "当时另一节点上的读已经看不到对方"这一事实）。**真正的解药只有两种：
   全局共识排序（回到 §2.2D，放弃多主的低延迟卖点）或 CRDT（把数据类型
   限制在可交换半格上，放弃通用 SQL 语义）**。多主产品的"冲突避免"文档
   （按地域切分写、delta 列）本质上是在引导用户把负载改造成"其实没有冲突"
   的形状——即退化回分片单主。
3. **delete 是幽灵**：行删了，后到的针对它的 update 如何判定"这是冲突"而
   不是"本来就没有这行"？必须保留死元组的墓碑。PG 内核在 20devel 对此的
   回应是 `retain_dead_tuples`（pg_subscription.h:166，fd5a1a0c3e，
   2025-08-04）：订阅可声明保留冲突检测所需死元组，从而可靠检出
   `update_deleted`（conflict.h:43 `CT_UPDATE_DELETED`）——注意内核只做到
   **可靠检测**，裁决仍然拒绝内置。这个克制与 §3 的边界论一脉相承：
   检测是机制，裁决是策略。

## 5. 最新进展（截至 2026-07，git log 与公开资料双查证）

以下按"复制能力補齐"的四条线组织。版本归属按本仓库分支时间推定
（PG19 于 a281a3e6db 2026-06-29 从 master 分出，故 2025-07 至 2026-06 的
commit 属 PG19，之后属 20devel/PG20）。

### 5.1 逻辑复制补齐线之一：sequence 同步（PG19）

- `5509055d69`（2025-11-05）"Add sequence synchronization for logical
  replication"：`CREATE PUBLICATION ... FOR ALL SEQUENCES`（语法见
  gram.y:11432 一带）后订阅端可同步序列值——填掉了"逻辑复制切换后序列
  回跳、新主插入撞主键"这个十年老坑（此前标准操作是切换脚本手动
  setval 一遍）。注意语义是**周期性/按需快照式同步**（sequence 变更不走
  逐条解码，实现上同步的是当前值），不是精确逐增量复制。
- 后续修复流显示这条路径的复杂度：`f6a4c498dc`（2025-11-07）加
  `seq_sync_error_count` 统计；`1ba3eee89a`（2026-01-20）与 `55f5186848`
  （2026-07-17）两次处理"同步期间序列被并发 DROP"的竞态。

### 5.2 逻辑复制补齐线之二：DDL 复制（仍未入核）

- Commitfest patch #3595 "Support logical replication of DDL commands"
  自 2022 年起历经多轮（deparse 成 JSON 中间表示再在订阅端重建），截至
  2026-07 **仍未提交入核**（来源：commitfest.postgresql.org/patch/3595、
  pgsql-hackers 线程、PG wiki 2023-06 演示稿；检索 2026-07）。
- 难点本质：DDL 不是行变更，无法走 reorderbuffer 的语义事件模型；deparse
  要在**解码时点的历史目录快照**下重建语句（snapbuild 的老问题换了主场）；
  且 DDL 与 DML 的交错顺序必须严格保序。生态绕行方案：pgroll/sqitch 类
  外部 schema 管理、PGD 的 DDL 复制（商业实现存在多年，恰说明技术可行、
  入核门槛在于通用性与维护面）。
- 与之互补的运维工具已入核：`pg_createsubscriber`（`d44032d014`，
  2024-03-25，PG17，src/bin/pg_basebackup/pg_createsubscriber.c）把物理
  备库原地转换为逻辑订阅者——初始 schema 与数据走物理复制（免 COPY、免
  DDL 问题），之后切逻辑通道；PG18 加 `-a/--all`（为除模板库外所有库建
  订阅，pg_createsubscriber.c:290）。跨大版本升级的标准路径自此从
  "pg_dump schema + 逻辑全量"变为"物理备库 + 原地转换"。

### 5.3 逻辑复制补齐线之三：冲突处理演化（PG15 → PG20 五步）

内核对冲突的态度演化是一条清晰的"检测强化、裁决克制"路线：

| 步 | commit | 版本 | 能力 |
|---|---|---|---|
| 停摆可控 | `705e20f855`（2022-03-14）、`208c5d65bb`（2022-03-22） | 15 | `disable_on_error`、`ALTER SUBSCRIPTION ... SKIP (lsn)` |
| 分类上报 | `9758174e2e`（2024-08-20） | 18 | 冲突按类型记日志与统计 |
| 墓碑保留 | `fd5a1a0c3e`（2025-08-04） | 19 | `retain_dead_tuples` → 可靠检出 `update_deleted` |
| 类型完备 | conflict.h:34-62 | 19 | 8 类：insert_exists / update_origin_differs / update_exists / update_deleted / update_missing / delete_origin_differs / delete_missing / multiple_unique_conflicts |
| 冲突入表 | `a5918fddf1`（2026-07-02） | 20devel | `subconflictlogdest`（pg_subscription.h:105，log/table/all 三值）+ 冲突日志表（subconflictlogrelid） |

至今没有一步是"自动按策略改写数据"——对照 §4.5 pgEdge/PGD 的 LWW
resolver，内核把裁决权明确留在外面。

### 5.4 failover slots 线：让逻辑消费者在切换后幸存（PG17 完成闭环）

- `ddd5f4f54a`（2024-02-14）slot 同步函数 + `93db6cbda0`（2024-02-22）
  slotsync worker：备库按 `sync_replication_slots` 自动镜像主库的逻辑槽
  （槽建 `failover=true`）；
- `synchronized_standby_slots` GUC（slot.c:176）：主库逻辑 walsender 发送
  变更前等待指定物理槽的确认位点（walsender.c:1836 一带）——**防止订阅者
  消费进度超过候选新主的接收进度**，否则切换后新主上的槽位点在订阅者身后，
  变更重复还好说，超前则永久丢失。
- 这组特性是 §3.4 "可行性归内核、决定权归外部"边界的教科书案例：内核不
  决定何时切换，但保证"切过去之后逻辑复制还能接上"。

### 5.5 槽治理与门槛下调线

- `c655077639`（2020-04-07，PG13）`max_slot_wal_keep_size` 之后，槽失效
  原因已枚举为四种（slot.h:60-68）：`RS_INVAL_WAL_REMOVED` / `HORIZON` /
  `WAL_LEVEL` / `IDLE_TIMEOUT`；最后一种来自 PG18 的
  `idle_replication_slot_timeout`（slot.c:170，`05dedf43d3` 2025-07-11 改
  单位为秒）——闲置槽自动失效，槽泄漏（§6.3）从"事故"降格为"可配置的
  止损"。
- `04ff636cbc`（2025-03-21，PG18）`max_active_replication_origins`
  （origin.c:106，默认 10）从 `max_replication_slots` 中独立出来——订阅密集
  部署的资源模型理顺。
- **`67c20979ce`（2025-12-23，PG19）"Toggle logical decoding dynamically
  based on logical slot presence"**：`wal_level=replica` 下创建第一个逻辑槽
  即自动升格 effective WAL level 至 logical，**不再要求重启**（测试
  src/test/recovery/t/051_effective_wal_level.pl）。这是逻辑复制采用门槛
  的历史性下调——十年来"想上 CDC？先停库改 wal_level"这个第一拦路虎消失。
  后续竞态修复（`93a3e6839b` 2026-06-09、`c209931bd4` 2026-07-16）显示
  动态切换与恢复端交互的微妙。

一句话总结趋势：**逻辑复制正在从"复制机制"长成"平台"**——位点可随 failover
迁移（5.4）、序列可同步（5.1）、冲突可审计（5.3）、启用不停机（5.5），只剩
DDL（5.2）一块最硬的骨头。

## 6. trick 与 bug 考古（8 例，附证据）

### 6.1 timeline：防 WAL 分叉的设计史，以及检测器自己的 bug

- `66ec2db728`（2004-07-19）PITR 入核两天后，`2042b3428d`（2004-07-21，
  Tom Lane）"Invent WAL timelines"——动机写在 commit message 里：让
  point-in-time recovery "more manageable"。原始问题：恢复到过去某点后
  继续写，新历史会覆写归档里旧历史的同名 WAL 段——**同一 LSN 两种内容，
  这是最早被认识到的"WAL 分叉"**，failover 双主只是它的分布式变体。
  解法：历史 ID（TLI）进段文件名 + `.history` 文件记血缘。
- 检测器自身的 bug 史：`3f0ab05233`（2013-01-31）修复"promotion 后立刻
  崩溃则时间线切换 fumble"——end-of-recovery 不再强制 checkpoint 的优化
  引入了"crash recover 到错误时间线"的窗口，补救是把 `minRecoveryPointTLI`
  记进控制文件。教训：timeline 是**元数据层的一致性机制，它自己的持久化
  时序同样要过崩溃一致性审判**。
- trick 视角：TLI 与 Raft term 的形似神异见自测第 7 题。

### 6.2 hot standby 冲突取消与 hot_standby_feedback：双刃剑全史

- `bca8b7f16a`（2011-02-16，Simon Riggs，PG9.1）引入 feedback：备库把自己
  最老查询的 xmin 经复制协议回传，钉住主库 vacuum 视界，换取备库查询不再
  被 redo 冲突取消（机制细节第 13 篇 §13.4.3）。
- 它的前任更惨烈：`vacuum_defer_cleanup_age`（盲目让主库延迟 N 个 xid 清理）
  在 PG16 被处决（`1118cd37eb`，2023-04-24），处决前一个月刚修完它的杀身
  之祸——`be504a3e97`（2023-03-07）"Fix corruption due to
  vacuum_defer_cleanup_age underflowing 64bit xids"：这个参数参与视界计算
  时可使 64 位 xid 下溢，**造成真实数据损坏**。一个为"减少备库查询取消"
  而生的参数，最终以"能腐化主库数据"收场。
- feedback 的双刃剑本质（第 13 篇 §13.7.4 已推演）加一条考古注脚：经物理槽
  传递的 feedback xmin 在**备库断连后仍被槽记住**，故"备库下线一周"="主库
  vacuum 冻结一周"——这正是 §5.5 `idle_replication_slot_timeout` 十几年后
  才补上的止损阀所针对的事故形态。

### 6.3 复制槽泄漏与 max_slot_wal_keep_size：强保证的反噬

- `858ec11858`（2014-01，PG9.4）引入槽时的设计选择是**无限强保证**：消费者
  不来，WAL 永不删（对比 Kafka：retention 到期就删，consumer 落后是 consumer
  的事）。六年事故积累（pg_wal 打满 → 主库 PANIC 停写，比消费者丢数据更糟）
  之后，`c655077639`（2020-04-07，PG13）补上 `max_slot_wal_keep_size`，
  `pg_replication_slots.wal_status` 同时出现——**从"槽保护消费者"改为
  "槽保护消费者，但主库保命优先"**。这是分布式系统里经典的
  producer-consumer 背压问题在数据库里的重演：PG 最初选了"无限背压上游"，
  被现实教育成"有限背压 + 消费者熔断"（invalidated 槽）。
- 20devel 里失效原因已是四元枚举（slot.h:60-68），治理面覆盖 WAL 保留、
  xmin 视界、wal_level 降级、闲置超时——每一个枚举值背后都是一类真实事故。

### 6.4 逻辑解码 spill 打爆磁盘：内存限制的三段史

- 史前（9.4–12）：reorderbuffer 对**单事务**超过 4096 条变更即 spill 到
  `pg_replslot/<slot>/*.spill`（这个常量今天还在：reorderbuffer.c:227
  `max_changes_in_memory = 4096`，已退居 restore 路径），但**总内存无上限、
  spill 量无上限**。生产事故形态：长事务/批量 UPDATE + 多个复制槽（每槽
  独立解码独立 spill）→ 磁盘打满 + 数百万小文件。公开案例：pgsql-hackers
  线程 "logical decoding : exceeded maxAllocatedDescs for .spill files"
  （2019–2020，postgrespro 存档）记录了单目录 600 万+ spill 文件、清理阶段
  长时间停顿的现场，甚至触发文件描述符上限 bug。
- `cec2edfa78`（2019-11-16，PG13）`logical_decoding_work_mem`：预算制取代
  每事务计数；`45fdc9738b`（2020，PG14）streaming：超预算的事务改为**边执行
  边发**给订阅端，从"spill 到发送端磁盘"变为"流到接收端"（第 13 篇
  §13.5.3）。后续还有 spill 文件 LZ4 压缩提案（pgsql-hackers，2024）。
- 教训与 Kafka 对照：这是 consumer 端 rebalance/堆积问题的镜像——PG 逻辑
  解码在发送端重组事务（因为要按提交序交付完整事务），相当于把 Kafka
  Streams 的 state store 塞进了 broker；PG14 streaming 相当于把重组责任
  推给消费端。**"在哪一端缓冲"是所有按序交付系统的永恒选择题**。

### 6.5 长期数据丢失 bug：失效消息不分发（2023 报告，2025 修复）

- `4909b38af0`（2025-04-10，Amit Kapila）"Fix data loss in logical
  replication"。病灶：`ALTER PUBLICATION ... ADD TABLE` 等**不取强锁**的
  DDL 与并发 DML 交错时，逻辑解码**不向并发进行中的事务分发目录失效消息**，
  这些事务（乃至其后的事务）继续用陈旧的目录缓存判断"该表不在 publication
  里"，**静默跳过本应发给订阅端的变更**。即：往 publication 里加一张表，
  加表瞬间的并发写可能永远不被复制。
- 考古价值三重：(1) 潜伏期极长——commit message 明言 "The bug exists in
  all supported branches"，回补到 14（13 因改动过大另行处理），即从逻辑
  复制诞生起潜伏近十年；报告线程标题就叫 "long-standing data loss bug in
  initial sync of logical replication"（pgsql-hackers，2023-11 起，Tomas
  Vondra 等）；(2) 修复的代价权衡写进了 commit message——曾考虑给
  publication DDL 取强锁，因 FOR ALL TABLES 需要锁全库表而否决，改为把
  失效消息广播给所有 in-progress 解码事务，接受 publication DDL 频繁场景
  的性能回退；(3) 它属于最危险的 bug 类别：**不崩溃、不报错、数据默默
  不见**——正是逻辑复制"一致性靠机制去挣"命题的反面教材：挣的机制
  （snapbuild 历史快照）有洞，一致性就真的没有。

### 6.6 contrecord bug：跨段记录中断导致的备库卡死（2021）

- `ff9f111bce`（2021-09-29）"Fix WAL replay in presence of an incomplete
  record"。病灶：一条 WAL 记录跨段写到一半时主库崩溃，重启后主库从记录
  **起点**之前重新写——但备库/归档可能已经收到了前半段。此前代码假设
  "见到 contrecord 缺失 = WAL 结束"，于是备库在此死等后半段，而主库已在
  同一 LSN 写了完全不同的新记录——**同 LSN 双历史，timeline 机制管不到
  （TLI 没变！）**。修复引入 `XLOG_OVERWRITE_CONTRECORD`：主库显式写一条
  "我要覆写未完成记录"的标记，备库读到即知旧前半段作废。
- 考古价值：这是"WAL 分叉"家族里最隐蔽的成员——分叉不需要 promote，
  **一次普通崩溃就能造出来**；timeline 只防"历史分支"级分叉，防不了
  "同历史内的部分写"分叉。与 §6.1 合读可得完整的分叉威胁模型。

### 6.7 promotion 后引用无效页：minRecoveryPoint 家族（2018）

- `3c64dcb1e3`（2018-07-05）"Prevent references to invalid relation pages
  after fresh promotion"。病灶链：备库回放先于一致点（minRecoveryPoint）
  推进磁盘状态 → 刚 promote 且尚未完成 end-of-recovery checkpoint 时崩溃
  → 二次恢复只回放到旧终点，而部分页面已带着"未来"的修改落盘 → 查询触及
  这些页面读到不一致状态。修复将"忽略无效页"的宽容窗口与恢复一致点判定
  收紧。同族还有 `2b4f313038`（2020-12-03）修 pg_rewind 对备库 rewind 的
  多个 bug、`a00c138b78`（2021-07-29）abort 记录回放截断时更新
  minRecoveryPoint。
- 教训：**"备库随时可被 promote"意味着恢复路径上每一个中间态都必须是
  合法的潜在起点**——这是物理复制"一致性免费"论断的边界条款：字节级重放
  免费，但"重放到一半"的状态管理一点都不免费。

### 6.8 synchronous_standby_names 配置陷阱集

全部现场核实于 src/backend/replication/：

1. **幽灵名字挂死提交**：配置的名字没有任何备库匹配（备库以
   `application_name` 报名，默认值不匹配是常态），则每个 commit 都在
   `SyncRepWaitForLSN` 无限等待——**主库看似健康，写全挂**。新手事故率
   最高的一条。
2. **取消等待 ≠ 回滚**：Ctrl-C/超时打断等待时事务**已本地提交**，仅发
   WARNING（syncrep.c:331 起："canceling wait for synchronous replication
   due to user request"，errdetail 明言 "The transaction has already
   committed locally, but might not have been replicated to the
   standby."）。此后 failover，新主可能没有这笔"客户端被告知取消、旧主上
   却已生效"的事务——PG 同步复制**不是原子广播，是提交后的确认门**。任何
   基于"synchronous_commit=on 则 failover 绝不丢已确认事务"的架构论证，
   都必须补上"且没有人取消过等待、没有配置过 statement_timeout 打断它"
   的脚注。
3. **`'*'` 通配符**（syncrep.c:887）：匹配任意备库名——包括临时接上来的
   pg_basebackup、pg_receivewal。"随便哪个连上来的都算同步备库"在混布
   环境里等于把 RPO 交给运气。
4. **FIRST vs ANY 的语义陷阱**（syncrep_gram.y:70-71；syncrep.h:35-36）：
   `FIRST 2 (a,b,c)` 是**按列表优先级**取前两个活跃者（priority 模式），
   `ANY 2 (a,b,c)` 才是真 quorum。裸写 `'a,b'` 等价 `FIRST 1 (a,b)`——
   即**只等一个**，很多自以为"双同步"的部署实际是单确认。
5. **与 Patroni 的互动面**：Patroni 之所以要接管这个 GUC（§3.3），正因
   上述语义只有与"选主候选资格"联动才闭环——内核语义单独存在时，2/3/4
   条陷阱都无解。

## 7. 自测 7 题（附详解）

**1. 用"复制四问"给 Aurora 定位，并指出它与 PG 物理流复制在 Q1 上的本质差异。**

答：Q1 复制什么——仅 redo log 记录（比 PG 的"WAL 字节 + 备库全量重放"更进
一步：连页面材料化都下沉给存储层，计算节点间零页面传输）；Q2 何时确认——
4/6 副本落盘；Q3 谁能写——单写者（计算层单主）；Q4 谁仲裁——存储层
epoch/租约 fence 单写者，计算节点无状态使选主退化为"挂载权转移"。Q1 差异
的本质：PG 备库既是日志消费者又是页面材料化者（startup 进程 redo），Aurora
把两个角色拆开且后者多租户共享——所以 PG 备库必须与主库同构，Aurora 读
副本只是"另一个挂到相同存储的计算头"。

**2. MySQL 半同步的 AFTER_SYNC 为何号称 lossless？PG 的 syncrep 在同一坐标系里是什么？**

答：AFTER_SYNC 把"等副本 ack"插在 binlog 刷盘之后、**引擎层提交（可见）之前**：
主库崩溃时未获 ack 的事务在主库上也未提交，failover 到副本不会出现"旧主可见
而新主没有"的事务，故 lossless。PG 的等待点（SyncRepWaitForLSN）在 commit
record 本地 fsync + CLOG 更新之后、`ProcArrayEndTransaction` 之前：持久化
次序上像 AFTER_COMMIT（本地先落盘），可见性次序上像 AFTER_SYNC（等待期间
其他会话看不见它）。裂缝在崩溃恢复：重启后 CLOG 说它已提交，它**变得可见**
——若此时反而 failover 走了，新主上没有它。加上等待可被取消（§6.8.2），
PG 的"RPO=0"承诺比 MySQL AFTER_SYNC 多两个脚注。这不是实现粗糙，是
"复制外挂于单机提交路径"的架构必然：PG 无法把"本地提交"悬置到 ack 之后，
因为 WAL flush 就是它的提交定义。

**3. 为什么不能"把 Raft 搬进 PG 内核"直接替换流复制？至少给三条结构性理由。**

答：(1) 日志语义不匹配：Raft 要求 leader 崩溃后**未提交尾部可被截断覆写**，
而 PG 的 WAL 一旦本地 fsync 即被本地提交定义引用（见第 2 题）——引入 Raft
意味着"本地 flush ≠ 提交"，重写整个提交路径与恢复路径（Neon 正是做了这个
手术，但把它移出了内核，safekeeper 层持有"可截断日志"）；(2) 单 Raft 组吞吐
= 单点吞吐，等于给整个实例的写路径加一轮强制多数派 RTT，而 PG 用户基数的
大头是单机/主备，不能为少数派需求向所有人收延迟税——分片 multi-raft 又
要求先解决分片（那是 Citus/CRDB 的整条路线，不是复制子系统的补丁）；
(3) fencing 完备性（§3.1 论点二）：内核内共识仍管不住 VIP/连接池/旧连接，
外部编排层无论如何省不掉，收益不抵成本。

**4. Aurora 写 quorum 4/6、读 quorum 3/6，为什么论文说它"避免了分布式共识"？**

答：quorum ≠ 共识。共识协议（Paxos/Raft）解决的是**多个提案者竞争同一日志位置**
时的安全性（提案编号、两阶段、多数派否决旧提案者）。Aurora 只有一个写者
（单主计算节点），LSN 单调由单点分配，不存在竞争——写路径退化为"广播 + 数
ack 到 4"；副本间不需要互相协商，修复读也只在崩溃恢复时用（正常读凭计算层
维护的段完成度直读单副本）。成员变更用 epoch 版本化的元数据处理。换言之
**Aurora 用"单写者"这个体制假设买掉了共识的全部复杂度**；对照 Neon：同样
单写者，但 safekeeper 协议仍保留 term 投票以 fence 前任 compute——因为
Neon 的 compute 是易失的、可被平台随意替换，"谁是当前唯一写者"本身需要
仲裁。两者对比精确展示了：**共识的最小必要剂量 = 写者身份的不确定度**。

**5. 逻辑复制多主为什么不能靠"更好的 resolver"达到强一致？给出不可绕过的反例。**

答：resolver 是行级、事后、局部的，而一致性约束是跨行、事前、全局的。
反例一（唯一约束）：节点 A、B 并发插入相同 email 的两行，各自本地约束检查
通过（对方的行尚未到达）；复制相遇后无论 resolver 保留哪行，都已经发生过
"两个事务都成功返回"——违反约束的观察已泄漏给客户端，任何裁决都只是善后。
反例二（跨行不变式）：转账在 A 节点扣款、并发地在 B 节点也扣款，各自余额
检查通过，合并后余额为负——resolver 逐行看没有任何冲突（不同行！），
不变式已破。根治只有两条路：写前全局排序（共识，放弃多主延迟优势）或把
数据类型限制为可交换操作（CRDT，放弃通用 SQL）。所以 §4.5 说产品文档里的
"冲突避免"实为引导用户退回分片单主。PG 内核止步于"完备检测 +
retain_dead_tuples + 冲突入表"（§5.3）而不做 resolver，是对这个不可能性
的清醒，而非能力不足。

**6. PG17 的 failover slots 由哪两半组成？只启用其中一半各会发生什么？**

答：两半是"槽随备库同步"（slotsync worker，`93db6cbda0`：备库镜像主库上
`failover=true` 的逻辑槽）与"发送门限"（`synchronized_standby_slots`，
slot.c:176：主库逻辑 walsender 发给订阅者之前，等待指定物理槽确认到该位点）。
只有前者：订阅者可能已消费到 LSN=X，而候选备库只收到 X-Δ；切换后镜像槽
的位点停在备库收到的位置，订阅者位点反而**超前于**新主——中间 Δ 的变更
新主永远无法再生成，静默丢失。只有后者：订阅者被限速在备库进度之内，但
新主上根本没有这个槽，订阅者无处续接（PG16 及以前的现实，运维靠切换后
重建槽 + 全量重同步）。两半合璧才成立"逻辑消费者在 failover 后无损续读"
——注意决策（何时切）仍在外部工具手里，内核只保证可行性（§3.4 边界）。

**7. timeline（TLI）与 Raft term 有何形似与神异？pg_rewind 在其中扮演什么角色？**

答：形似——都是单调递增的"历史纪元号"，用于识别并拒绝旧纪元的写者/日志
（备库拒绝接入分叉的 TLI；Raft 节点拒绝旧 term 的 AppendEntries）。神异有三：
(1) 产生方式：term 由**多数派投票**授予，天然全局唯一；TLI 由 promote 动作
**自封**，两个备库同时 promote 会产生两个"合法"的相同新 TLI——TLI 的唯一性
靠人/外部工具保证；(2) 作用时点：term 在**写入前**拦截旧 leader（旧 leader
的日志根本无法提交）；TLI 在**事后**检测分叉（分叉期间两边都写成功了）；
(3) 处置手段：Raft 自动截断落败者的未提交尾部；PG 的对应操作是 pg_rewind
——把旧主自分叉点起的 WAL 效果**回滚到分叉前**（依据新主的 WAL 与数据目录
差异回退页面），使其可作为备库重新接入。即 pg_rewind 是"Raft 日志截断"的
人工触发、粗粒度、事后版——再次印证 PG 把共识协议里自动化的环节留给了
外部编排（Patroni 正是自动调用 pg_rewind 的最大主顾）。
