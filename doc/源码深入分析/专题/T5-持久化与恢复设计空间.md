## 1. 问题定义：两对根本矛盾

### 1.1 持久化矛盾：每事务 fsync 的延迟 vs 缓冲聚合的丢失窗口

"提交即持久"（durability）的字面实现只有一种：commit 返回前，把本事务的日志推到
非易失介质上——在 POSIX 世界里就是一次 `fsync()`/`fdatasync()`。矛盾在于：

- **fsync 是整个提交路径上唯一无法用 CPU 换取的开销**。SATA SSD 一次 flush 约
  0.1–1ms，云盘（EBS/云硬盘）常在 0.5–2ms，机械盘 5–10ms。单连接串行小事务的
  TPS 上限就是 `1/fsync延迟`——机械盘上约 100 TPS，这与 CPU 无关。
- **聚合是唯一的出路，而聚合必然打开丢失窗口**。把 N 个事务的日志攒成一次 fsync，
  延迟摊薄 N 倍；但在"攒"的窗口里宕机，这 N 个事务全部或部分丢失。

所有数据库的持久化设计都是在这条谱系上选点或让用户选点：

```
每事务 fsync ←────────────────────────────→ 定期 fsync（如每秒）
  延迟最高、零丢失窗口                        延迟最低、丢失窗口 = 刷盘间隔
```

关键洞察是：**这条谱系上存在"免费午餐"区间**——组提交（group commit）。当并发提交者
足够多时，一次 fsync 天然服务多个事务，聚合不再需要牺牲持久性，只需要牺牲一点排队
延迟。PG 的 `XLogFlush` 等待/接力协议（09 篇 §4.1）就是隐式组提交：不需要任何配置，
并发越高聚合率越高。显式的时间换聚合（`commit_delay`/`commit_siblings`，
xlog.c:139-140，生效条件见 xlog.c:2899-2902：必须 `enableFsync` 且活跃后端数达到
`CommitSiblings`）只是在并发不足时人为制造聚合机会。第 4 节把 PG / InnoDB / Redis
在这条谱系上的所有档位排成一张"每档丢什么"的表。

### 1.2 恢复矛盾：恢复时间 vs 运行时开销

崩溃恢复的时间大致等于"重放自某个一致性起点以来的所有日志"。起点越新，恢复越快；
但把起点推新（checkpoint）本身要把脏页刷盘，是运行时的 I/O 开销。于是第二对矛盾：

- **checkpoint 越频繁** → 恢复越快，但运行时刷脏 I/O 越多，且（对 PG）每轮
  checkpoint 后的首次页修改要写整页镜像（FPW），WAL 体积膨胀（§3.5 量化）；
- **checkpoint 越稀疏** → 运行时 I/O 平滑，但恢复窗口拉长，且 WAL 堆积占盘。

这对矛盾的极端解发人深省：

- 极端 A：**没有 checkpoint**——Aurora 把页物化下放到存储层持续进行，计算节点
  永远不刷脏页，恢复近乎即时（§2.7）；
- 极端 B：**没有恢复**——shadow paging / COW B-tree 每次提交都产生一个完整一致的
  树根，崩溃后翻开最后一个有效根即可，恢复时间 O(1)，代价全部预付在运行时的
  写路径上（§2.4/§2.5）。

传统 WAL 体系（PG、InnoDB）选择在两个极端之间连续可调（`max_wal_size` /
`checkpoint_timeout` / `checkpoint_completion_target`），把选点权交给 DBA。

### 1.3 隐藏的第三维：复制把单机问题变成分布式问题

单机视角下上面两对矛盾已经完备；但现代部署几乎必有复制，而**日志天然是最好的复制
载体**——它是全序的、幂等可重放的、增量的。于是持久化格式的选择会连带决定复制的
形态与成本：

- PG 的 WAL 一份日志同时承担崩溃恢复、物理复制、逻辑解码三份职责，FPW 的体积
  膨胀因此直接变成复制带宽膨胀（§3.5）；
- `synchronous_commit` 的五档里有三档（`remote_write`/`on`/`remote_apply`）实际上
  是把"持久性"重定义为跨节点属性（§4.1）；
- Aurora/Neon 则干脆把"日志即复制流"推到极致：网络上只跑日志，页面在远端物化
  （§2.7/§2.8）。

评价任何持久化方案时都要问一句：**它的日志能不能直接当复制流用？** 这就是第 2 节
矩阵中"复制友好度"一列的由来。

## 2. 方案谱系与优劣矩阵

### 2.1 谱系总览

按"崩溃后靠什么恢复一致性"分类，设计空间大致是一棵二叉树：

```
恢复靠什么？
├── 靠日志重放（in-place update + log）
│   ├── redo + undo：ARIES 家族（InnoDB、SQL Server、Db2、Oracle）
│   ├── redo-only：PostgreSQL（no-overwrite 存储替代 undo）
│   └── 日志下放远端：Aurora（计算节点只写日志）、Neon（PG 的同构改造）
└── 靠原子切换根指针（never overwrite，天然无恢复）
    ├── shadow paging：System R（后被作者本人否定）、SQLite 的 rollback journal 变体
    ├── COW B-tree：LMDB、BTRFS、ZFS
    └── log-structured：LFS、LSM-tree（RocksDB/LevelDB —— 日志"就是"数据的雏形）
```

下面逐一分析，每个方案回答同样五个问题：写放大多少？恢复多快？checkpoint（或等价
物）多贵？实现多复杂？对复制多友好？最后汇总成 §2.9 的矩阵。

### 2.2 ARIES 全家：redo + undo（InnoDB / SQL Server / Db2）

ARIES（Mohan et al., "ARIES: A Transaction Recovery Method Supporting Fine-
Granularity Locking and Partial Rollbacks Using Write-Ahead Logging", ACM TODS
1992）是教科书标准答案，核心三原则：WAL、重放历史（repeating history）、undo 产生
CLR（补偿日志记录）。它支持 **steal**（未提交脏页可以刷盘）+ **no-force**（提交时
不必刷数据页），代价是恢复分三阶段：analysis → redo（重放到崩溃点，包括未提交事务
的修改）→ undo（回滚未提交事务）。

以 InnoDB 为例落地形态：

- redo log（`#innodb_redo` 循环文件）记录物理-逻辑（physiological）改动；
- undo 不写在 redo 里，而是存在 undo tablespace 的回滚段中——undo 记录本身也是
  受 redo 保护的数据页，所谓 "undo is data"；
- MVCC 的旧版本靠 undo 链重建（读旧版本 = 当前行 + 反向应用 undo）。

**优势**：数据页原地更新、堆是紧凑的（没有死元组占位）；回滚段天然给 MVCC 提供
旧版本，长事务不会像 PG 那样把表膨胀固化在堆里；恢复语义完备（任意时刻崩溃、
恢复中再崩溃都被 CLR 的"undo 也只做一次"性质覆盖）。

**代价**：
- 实现复杂度是全谱系最高档——undo 链、回滚段、purge 线程、CLR，每一个都是
  独立子系统；InnoDB 的 history list length 失控是经典事故模式（等价于 PG 的
  VACUUM 落后，问题守恒，只是搬了地方）；
- 恢复要做三阶段，undo 阶段在长事务场景可能非常长（MySQL 崩溃恢复回滚巨型
  事务的等待是 DBA 熟知的痛点）；
- 写放大：redo + undo（undo 页本身还要 redo 保护）+ 数据页 + doublewrite（§3.3）。

### 2.3 redo-only：PostgreSQL 的"用存储格式换掉 undo"

PG 的选择在 09 篇 §6 已详细论证，这里只放进设计空间坐标：PG 的堆是
**no-overwrite** 的——UPDATE 产生新元组而非覆写旧元组，旧版本留在原地由 MVCC
可见性规则（CLOG + 快照）判定死活。于是"崩溃后清除未提交修改"根本不需要 undo：
未提交元组的 xmin 在 CLOG 里不是 COMMITTED，对所有人不可见，物理清除推迟给
VACUUM。恢复 = 单纯 redo 到日志尽头，一个阶段。

**优势**：
- 恢复逻辑显著简化（无 undo 阶段、无 CLR），巨型事务回滚是 O(1)（只写一条
  abort 记录）；
- WAL 是纯 redo 流，物理复制的实现被简化为"把恢复无限进行下去"（standby 就是
  一个不退出的 recovery），这是 PG 复制体系 2010 年后快速成熟的结构性原因；
- 逻辑解码也直接寄生在同一份 WAL 上（一份日志三份职责，09 篇前置知识节）。

**代价**：问题守恒——undo 的活儿变成了 VACUUM 的活儿：死元组占堆空间、
索引膨胀、freeze 债务（XID 回卷）；hint bit 写、HOT 优化、fillfactor 调优全是
这一选择的下游补丁。另一个隐性代价：no-overwrite 意味着每个 UPDATE 都可能触碰
新页，checkpoint 后首次触碰即 FPW，写放大比原地更新更敏感于 checkpoint 频率。

### 2.4 shadow paging：System R 的弃案与它的现代平反

Shadow paging（Lorie 1977）：维护页表的两份映射（current / shadow），事务修改
一律写新页，提交 = 原子切换页表根。崩溃后 shadow 版本天然一致，**恢复时间 O(1)，
没有日志、没有 checkpoint**。

System R 真的用了它，然后其作者们在复盘论文（Gray, McJones, Blasgen 等, "The
Recovery Manager of the System R Database Manager", ACM Computing Surveys 1981）
里给出了数据库史上著名的自我否定，理由至今成立：

1. **破坏物理聚簇**：每次写都换新页，逻辑相邻的数据物理上被打散，顺序扫描退化
   为随机读（机械盘时代这是致命的）；
2. **页表本身成为新的持久化问题**：大库的页表是多级结构，提交要级联写页表页，
   写放大和提交延迟都上去了；
3. **并发瓶颈**：页表根切换是全局原子点，细粒度并发事务难以叠加（System R 实际
   上是 shadow paging + 日志混合，两边的成本都付了）；
4. 恢复快的收益在当时被 checkpoint + 日志的可调性完全覆盖。

**但注意这是"在机械盘 + 单机 OLTP + 有并发写"前提下的否定**。前提变了结论就变：
LMDB（见 §2.5）在读多写少 + 内存映射 + 单写者的场景里把 shadow paging 做成了
教科书级的成功；ZFS/BTRFS 在文件系统层（天然大页、天然要 COW 快照）也复活了它。
设计空间分析最重要的一课：**没有被淘汰的方案，只有被淘汰的前提组合**。

### 2.5 COW B-tree：LMDB / BTRFS

Copy-on-write B-tree 是 shadow paging 的树形精化：修改叶子页 → 复制该页 → 父页
指针改指新页 → 父页也 COW → 级联到根；提交 = 原子写根。LMDB 的落地非常极致：

- 双 meta page 轮换（页 0/页 1 交替当根），提交 = 写数据页 + fsync + 写 meta +
  fsync，**没有 WAL、没有 checkpoint、没有恢复过程**——打开数据库时选两个 meta
  中事务号大且校验有效的那个即可；
- 单写者模型（全库一把写锁），彻底绕开 System R 撞上的并发瓶颈；读者靠 MVCC
  快照（旧根）完全无锁；
- 代价书写得很清楚：路径级联 COW 的写放大（改 1 个叶子 = 写 O(树高) 个页）、
  单写者吞吐上限、以及空间回收需要追踪旧读者（LMDB 的 freelist 与"长读事务
  导致库文件膨胀"问题——又一次问题守恒，等价于 PG 的长事务阻碍 VACUUM）。

BTRFS/ZFS 在文件系统层做同样的事，还顺手获得了免费的快照/克隆——COW 家族对
"时间旅行"类功能的友好度是全谱系最高的（Neon 的分支功能同样来自"版本历史天然
保留"，见 §2.8）。但数据库压在 COW 文件系统上会付双重 COW 税，这也是 PG 在
ZFS 上建议 `wal_init_zero=off`（xlog.c:134，GUC 由 commit 475861b261，2019-04-02
引入）的背景之一：COW 文件系统上预写零既不能预留物理空间也没有性能意义。

### 2.6 log-structured：日志即数据的雏形（LFS / LSM）

LFS（Rosenblum & Ousterhout 1991）的激进命题：既然日志必须写，数据也必须写，
为什么不**只写日志**？数据就存在日志里，靠索引在日志中定位最新版本。LSM-tree
（O'Neil 1996；RocksDB/LevelDB/Cassandra）是它的现代形态：

- 写路径 = WAL（保护 memtable）+ memtable，崩溃恢复只需重放很短的 WAL 重建
  memtable——**恢复时间由 memtable 尺寸决定，典型秒级**；
- 没有"把脏页写回原位"的 checkpoint，等价物是 memtable flush（顺序写新
  SSTable，天然增量）；
- SSTable 不可变，天然对复制/备份友好（rsync 级别的文件拷贝即增量备份）；
- 全部代价集中到 **compaction**：后台不断重写数据以维持读性能，总写放大常见
  10–30 倍（RocksDB leveled compaction 的公开典型值），且 compaction 与前台写
  抢 I/O 造成延迟毛刺。

把 LSM 放进本谱系的意义在于看清一个守恒律：**B-tree 家族把写放大付在"更新时"
（FPW/doublewrite/COW 级联），LSM 把写放大付在"整理时"（compaction）**。谁优
取决于读写比与介质：SSD 顺序写便宜、随机写伤寿命，把随机写变顺序写的 LSM 因此
在写密集场景占优；读路径则要付多层查找 + bloom filter 的复杂度税。

### 2.7 Aurora："日志即数据库"（The log is the database）

Amazon Aurora（Verbitski et al., "Amazon Aurora: Design Considerations for High
Throughput Cloud-Native Relational Databases", SIGMOD 2017）对 MySQL/PG 做了同一
个手术：**计算节点只把 redo 日志发给存储层，永远不写数据页**。

- 存储层 6 副本跨 3 AZ，写 quorum 4/6，读 quorum 3/6；提交 = 日志达到 4/6 quorum
  持久；
- 存储节点各自持续地把日志物化成页（continuous background materialization），
  计算节点按需 `(page, LSN)` 读取；
- 于是：**没有 checkpoint**（物化是存储层的持续后台行为，不占计算节点 I/O）、
  **没有 FPW/doublewrite**（页面物化不是原地覆写，不存在撕裂窗口）、**恢复近乎
  即时**（崩溃后计算节点只需与存储层对齐 volume durable LSN，不重放日志）；
- 论文给出的动机数据：传统 MySQL 主备部署下一次写要放大成多次网络 I/O（redo、
  binlog、数据页、doublewrite、frm），Aurora 把跨网络的量压缩到只有 redo，
  声称同硬件下写吞吐提升数倍（论文 §3，2017）。

代价同样清楚：存储层要实现日志物化引擎（等于把恢复逻辑做成 7×24 在线服务）、
读路径引入网络往返（靠计算节点大缓冲池遮蔽）、以及深度绑定专有存储——这是
一条只有云厂商走得通的路。**Aurora 在本谱系里的坐标：把 §1.2 的恢复矛盾整体
外包给存储层，用网络和副本数付账。**

### 2.8 Neon：对 PG 的同构改造

Neon（开源，github.com/neondatabase/neon）证明了 Aurora 思路可以不魔改内核地套
在 PG 上，因为 PG 的 WAL 本来就是纯 redo 物理日志（§2.3 的红利再次兑现）：

- 计算节点是近乎原生的 PG，`XLogFlush` 的落盘目标被换成流向 **safekeeper** 集群
  （Paxos 风格 quorum，WAL 的持久性由多数派确认定义——Neon 文档《Architecture
  overview》，2025）；
- **pageserver** 消费 WAL，按 (relation, block) 重组进分层存储（L0/L1 layer
  文件，内存里像一个"重排序缓冲"——Neon 博客《Deep dive into Neon storage
  engine》，2022），对计算节点提供 `GetPage@LSN` 接口：要哪一页的哪个历史版本，
  就地重放到那个 LSN 给你；
- 恢复、PITR、分支（branch）全部退化为"以某个 LSN 为起点读页"——历史版本天然
  保留，PITR 不再需要"基础备份 + 重放归档"（Neon 博客《Point in time recovery
  under the hood》，2023）；
- 页面从不被计算节点原地写回磁盘，撕裂页问题在结构上消失，FPW 的必要性随之
  消失（其对 WAL 体积的意义见 §3.5）。

与 Aurora 的差异值得记录：Aurora 存储层只物化"最新页"，Neon pageserver 保存
**全历史版本**（本质是一个以 (page, LSN) 为键的 LSM），所以 Neon 的分支/时间
旅行是一等公民，代价是存储层写放大与 GC（层文件 compaction）更重——Neon 在
存储层重新遇到了 §2.6 的 LSM 守恒律。

### 2.9 统一优劣矩阵

评分说明：▲=优/低成本，■=中等，●=劣/高成本。所有评分是"结构决定的量级"而非
实测数（同一格内不同实现可差数倍）。

| 方案 | 写放大 | 恢复时间 | checkpoint 开销 | 实现复杂度 | 复制友好度 |
|---|---|---|---|---|---|
| ARIES redo+undo（InnoDB） | ●（redo+undo+页+doublewrite） | ■（三阶段，undo 可拖长） | ■（模糊 checkpoint，无 FPI 但有 doublewrite 常量税） | ●（undo/CLR/purge 三套子系统） | ■（redo 物理日志偏页面局部，跨版本复制难；MySQL 实际另造 binlog 承担复制，双日志税） |
| redo-only（PG） | ■●（redo+页+FPW 尖峰；VACUUM 二次写） | ■（单阶段 redo；窗口由 max_wal_size 调） | ●（刷全部脏页 + 下轮 FPI 债，见 §3.5） | ■（无 undo，但换来 VACUUM/freeze 体系） | ▲（WAL 直接当物理复制流 + 逻辑解码源；FPW 吃带宽是唯一减分项） |
| shadow paging（System R 弃案） | ●（页表级联 + 聚簇破坏的隐性读放大） | ▲（O(1)，翻根即可） | ▲（无 checkpoint 概念） | ■（机制简单，难在并发与空间回收） | ●（没有天然的增量流可发布） |
| COW B-tree（LMDB/BTRFS） | ●（O(树高) 页/次提交） | ▲（O(1)，双 meta 择新） | ▲（无） | ▲（LMDB 全库约 1 万行 C 的量级，单写者换来的） | ■（快照天然可拷贝，但无记录级流；BTRFS send/receive 是补做的） |
| log-structured（LSM） | ●（compaction 累计 10–30×） | ▲（重放 memtable WAL，秒级） | ▲（flush 即天然增量 checkpoint） | ■（compaction 调度是无底洞） | ▲（不可变 SSTable + 短 WAL，文件级复制极友好） |
| Aurora（日志即数据库） | ▲（网络上只有 redo；物化在存储层内部消化） | ▲（近即时，与存储层对齐即可） | ▲（无：物化持续后台化） | ●（把恢复做成在线服务 + quorum 存储） | ▲（6 副本即复制本体；跨 region 也走日志） |
| Neon（PG 同构改造） | ■（计算侧只写 WAL；pageserver 内部付 LSM 税） | ▲（GetPage@LSN，无重放等待） | ▲（计算节点无 checkpoint 刷脏） | ●（safekeeper+pageserver 两个分布式系统） | ▲（WAL 流即一切；分支/PITR 一等公民） |

从矩阵读出的三个结构性结论：

1. **"恢复时间 ▲"的方案全部把成本移到了写路径或后台整理**（COW 级联、
   compaction、存储层物化）——§1.2 的矛盾从不消失，只换付款人；
2. **复制友好度与"日志是否纯 redo 物理流"强相关**：PG、LSM、Aurora/Neon 在此
   列全优；InnoDB 因 redo 不适合跨版本/逻辑消费而被迫维护 binlog 双日志，是
   MySQL 生态大量复杂性（XA 两阶段刷盘、组提交耦合）的根源；
3. **云改变了矩阵的权重**：单机时代 checkpoint 开销与恢复时间是核心列，云上
   "复制友好度"和"存储可分离性"权重陡增——这解释了为什么 2.7/2.8 两行的设计
   在 2017 年后才出现：不是想不到，是网络带宽/延迟到 2015 年前后才让"页面走
   网络"变得可行。

### 2.10 PG 为什么在这条路上：路径依赖的注脚

PG 的 redo-only 不是从设计空间里冷静选出的，而是历史耦合的产物：Berkeley
POSTGRES 最初的宏愿是**彻底不要日志**（Stonebraker "The Design of the POSTGRES
Storage System", VLDB 1987——no-overwrite 存储 + 时间旅行，靠强制刷盘保证一致），
时间旅行后来被砍掉，但 no-overwrite 存储留了下来；2001 年 7.1 引入 WAL
（commit 4d14fe0048 时代）时，undo 就自然而然地不需要了——旧版本本来就在堆里。
换句话说：**PG 是先有了"不覆写"的存储，才反推出 redo-only 的日志**；InnoDB 是
先承诺了"原地更新的紧凑堆"，才不得不造 undo。两者都是自洽的整体，单抄任何一半
都会崩坏——这也是"给 PG 加 undo"的 zheap 项目（2018–2021，EDB 主导，未合入）
举步维艰的结构原因：它要同时换掉存储格式、恢复逻辑和 VACUUM 三根柱子。

## 3. 页撕裂问题的三种解法深比

撕裂成因与 PG 无 FPW 时静默损坏的完整推演见 09 篇 §5.1–5.2，此处直接进入三种
解法的横向对比。问题一句话重述：数据库页（8K/16K）大于介质原子写单元（512B/4K），
断电可能留下"半新半旧"的页；而 redo 日志大多记录页内增量，**基底页坏了，增量
重放无从谈起**。

### 3.1 解法一：PG full-page write——把解药混进日志流

机制（细节见 09 篇 §2.3）：每轮 checkpoint 后第一次修改某页时，WAL 里不记增量
而记整页镜像（FPI）；恢复时无条件用镜像盖掉可能撕裂的页。判定逻辑在
`XLogRecordAssemble`（xloginsert.c:622 起，依据 `RedoRecPtr` 与 `doPageWrites`，
xloginsert.c:514-529）。

结构特点：
- **解药与日志同流**：不需要额外的持久化结构，恢复路径单一；
- **代价按"checkpoint 周期内首次触碰的页数"计费**，是尖峰型的：checkpoint 刚过
  的那段时间 WAL 写入速率可以翻数倍；
- 顺带红利：FPI 让恢复/standby 重放时对这些页**不需要读旧页**（09 篇 §8.4 的
  `BLK_RESTORED` 路径），把恢复的随机读变成了顺序日志读——Aurora 论文批评
  FPW 浪费，但单机恢复速度反而受益于它。

### 3.2 解法二：InnoDB doublewrite——把解药放在日志外面

机制：脏页刷盘前，先把整批页顺序写入 doublewrite 区（MySQL 8.0.20 起为独立的
`#ib_16384_N.dblwr` 文件，此前在系统表空间内）并 fsync，然后再写数据文件原位。
崩溃后若发现数据文件里某页撕裂（校验和不符），从 doublewrite 区找到完好副本修复
（MySQL 8.4 Reference Manual §17.6.4，2024）。

与 FPW 的深层差异：

| 维度 | PG FPW | InnoDB doublewrite |
|---|---|---|
| 付费时点 | WAL 写入时（前台，影响提交路径的 WAL 量） | 脏页刷盘时（后台 flush，不在提交路径上） |
| 付费金额 | 每 checkpoint 周期每脏页 1 次整页进 WAL | 每次页刷盘都双写（但 doublewrite 区是顺序批量写） |
| 与 checkpoint 频率的耦合 | 强（周期越短 FPI 越多） | 无（按刷脏次数计费，与 checkpoint 无关） |
| 对日志体积/复制带宽 | 直接膨胀（§3.5） | 零影响（redo 保持纯增量） |
| 恢复路径 | 单一：镜像随日志重放 | 双路径：先修页，再重放 redo（多一个持久化结构要管理） |
| 为什么对方不能抄 | InnoDB redo 是 physiological 的且 checkpoint 是模糊/连续的，没有"周期首次触碰"这个天然计费点 | PG 若引入 doublewrite 需给 checkpointer/bgwriter/backend 三个刷脏路径都加双写逻辑，且丢掉 FPI 加速恢复的红利；社区多次讨论未有人做成 |

量化直觉：doublewrite 把页写放大常数固定为 ~2×（顺序部分很便宜），FPW 的放大
是 `首次触碰页数 × 8KB / 原始 WAL 量`，在"热点页反复改"的负载里 FPW 更便宜
（一个周期只付一次），在"均匀大范围写"的负载里 doublewrite 更便宜。**两者都是
在为同一个硬件缺陷付保险费，只是一个按日志计费、一个按刷盘计费。**

### 3.3 解法三：让硬件承诺原子写——untorn writes

真正的釜底抽薪是让"页写"本身原子化，保险费降为零。这条线 2024–2026 年进展很快：

- **硬件早已就位**：NVMe 规范有 AWUPF（Atomic Write Unit Power Fail）等字段，
  企业级 SSD 普遍能原子写 ≥16KB；SCSI 有 WRITE ATOMIC(16)。缺的是操作系统到
  文件系统的语义贯通——直到 Linux 补齐：
- **Linux 6.11（2024）**：块层 + 直接 I/O 的 `RWF_ATOMIC`（`pwritev2` 新标志，
  io_uring 同步支持）落地，语义是"要么全写要么全不写"，注意**不包含持久化**，
  fsync 仍需自己调（LWN《Atomic writes without tears》，2024-05）；
- **Linux 6.13（2025-01）**：ext4/XFS 支持单文件系统块的 RWF_ATOMIC（DIO 限定；
  XFS 借 LBS 可在 x86_64 上做 16K 块，ext4 需 16K PAGE_SIZE 平台）（LWN
  《Support for atomic block writes in 6.13》2025-02；Phoronix 2024-11）；
- **Linux 6.16（2025-06）**：XFS 多文件系统块原子写；后续 LWN 报道显示 buffered
  atomic writes 也在推进（LWN《Buffered atomic writes, writethrough, and more》，
  2025）；`statx` 可查询文件的原子写能力窗口。
- **MySQL 已经吃上**：`innodb_doublewrite=0` 配合支持原子写的存储是官方认可的
  配置（"当使用原子写时……doublewrite buffer 不是必需的"，MySQL 8.4 Manual
  §17.6.4；更早在 Fusion-io/NVDIMM 时代就有 `innodb_doublewrite=OFF` 的自动
  检测传统）。
- **PG 的跟进状态（截至本仓库 2026-07 的 master）**：`grep -rn RWF_ATOMIC src/`
  为空——**核心尚无任何原子写支持**。pgsql-hackers 上的相关讨论挂在《Large
  block sizes support in Linux》线程（2024 起，Pankaj Raghav 等，LBS/untorn
  writes 与 `full_page_writes` 的关系）与 pgconf 走廊议题里；结构性障碍在于
  PG 的数据文件 I/O 仍是 buffered（DIO 只有 debug 选项，见 §5.2），而内核的
  原子写目前以 DIO 为先。**FPW 的退休要等 PG 的 DIO/AIO 写路径先落地**——这
  是 §5 的路线图交汇点。

三方总结：

| | FPW | doublewrite | 原子写硬件 |
|---|---|---|---|
| 保险费付给谁 | WAL/复制/归档带宽 | 本地刷盘 I/O | 一次性：硬件采购 + 内核版本 + DIO 改造 |
| 可否运行时关闭 | 可（`full_page_writes=off`，xlog.c:6550 附近 `XLOG_FPW_CHANGE` 记录切换），但硬件不保证时=玩火（09 篇 §12.5） | 可（`innodb_doublewrite=0`） | 不适用（能力由 statx 探测） |
| 谁已受益 | ZFS/COW 文件系统用户可安全关 FPW（文件系统层已原子） | 同左 + 原子写设备 | MySQL（官方支持）；PG 待跟进 |

### 3.4 FPW 对 WAL 体积与复制带宽的量化

FPI 的账本可以精确地记：

- **观测工具**：`pg_stat_wal` 的 `wal_fpi` 列（FPI 条数）；本开发周期新增
  `wal_fpi_bytes`（commit f9a09aa295，2025-10-28）直接给出字节数，此前只能用
  `pg_waldump --stats` 离线统计。`pg_waldump --stats=record` 输出中 "FPI size"
  一列即是；
- **量级**：一条普通 heap INSERT 的 WAL 记录约几十到一二百字节，而一个 FPI 约
  8KB（可压缩，见下）。checkpoint 刚结束时几乎每次页修改都带 FPI，**放大可达
  两个数量级**；OLTP 负载整体统计中 FPI 占 WAL 总量 50%–90% 是常见区间（可用
  上述工具在任意负载上直接验证，比例随 checkpoint 周期与热点集中度波动）；
- **checkpoint 周期的耦合**（§1.2 矛盾的具体化）：`max_wal_size` 调小一半 ≈
  checkpoint 频率翻倍 ≈ 每页在单位时间里多付一次 FPI——所以"WAL 写太多导致
  checkpoint 提前触发导致 FPI 更多"存在正反馈，`max_wal_size` 过小的系统会
  陷入 FPI 风暴；
- **复制与归档是连带受害者**：物理复制按字节传 WAL，FPI 尖峰直接变成复制带宽
  尖峰与 standby 回放延迟尖峰；`archive_command`/`pg_receivewal` 的归档体积
  同理。跨 region 复制按流量计费时，FPI 是真金白银；
- **缓解手段**：`wal_compression`（只压 FPI，不压普通记录）——pglz（9.5，
  commit 57aa5b2bb1，2015-03-11）→ lz4（commit 4035cd5d4e，2021-06-29，PG 15）
  → zstd（commit e9537321a7，2022-03-11，PG 15），枚举见 xlog.h:84-87。zstd
  对 FPI 常有 2–4× 压缩比，代价是每条 FPI 的 CPU；lz4 是"几乎免费"档。注意
  该参数是 PGC_SUSET（commit c2e5f4d1c1）：普通用户不可自行开关。

### 3.5 一个反直觉结论

把 §3.1–3.4 放在一起看：**FPW 是三种解法里唯一把保险费付进"复制/归档"公共
账户的**，也因此是唯一随架构演进可以整体消失的——Aurora/Neon 页面不原地写回，
FPW 在结构上不再需要；原子写硬件普及后单机 PG 也能摘掉它。而 doublewrite 的
本地刷盘税与存储架构无关，只能靠硬件原子写豁免。设计空间的时间维度上，FPW 是
一个"过渡期方案"的完美样本：丑、贵，但退出路径清晰。

## 4. 组提交与持久性谱系："每档丢什么"

### 4.1 PG：synchronous_commit 五档

实现锚点：枚举定义 xact.h:71-81（`SYNCHRONOUS_COMMIT_OFF / LOCAL_FLUSH /
REMOTE_WRITE / REMOTE_FLUSH / REMOTE_APPLY`，且 `ON` 就是 `REMOTE_FLUSH` 的别名，
xact.h:81）；GUC 面只暴露 `off/local/remote_write/on/remote_apply` 五个词
（guc_tables.c:341-357）。提交路径：`RecordTransactionCommit` 中先
`XLogFlush(XactLastRecEnd)`（本地持久），再 `SyncRepWaitForLSN`（xact.c:1599，
等远端确认）；`off` 档则跳过等待，WAL 交给 walwriter 背景刷。

| 档位 | commit 返回时已保证 | 单机崩溃丢什么 | 主库彻底损毁丢什么 |
|---|---|---|---|
| off | 记录已进 WAL 缓冲 | 最近未刷盘的已"提交"事务（窗口约至多 3×`wal_writer_delay`），**无损坏，只丢尾巴**（数据库仍一致） | 同左 + 未复制的尾巴 |
| local | WAL 已 fsync 到本地 | 零 | 自上次复制以来的全部 |
| remote_write | 本地 fsync + 同步备已 write（进备机内核页缓存，未 fsync） | 零 | 零（除非主库损毁的同时备机 OS 也崩溃——两个独立故障叠加） |
| on（=remote_flush） | 本地 fsync + 同步备已 fsync | 零 | 零（备机磁盘在即安全） |
| remote_apply | 上述全部 + 备机已重放可见 | 零 | 零，且故障切换后**读不到旧数据**（读扩展场景的因果一致性） |

三个常被忽视的细节：

1. `synchronous_commit` 是**可以按事务设置的**（普通 GUC，`SET LOCAL` 即可）——
   同一个库里资金表用 `on`、点击日志用 `off`，这是 PG 谱系设计的独到之处：丢失
   窗口的选择权细到单事务；
2. `off` 不等于 `fsync=off`：前者只是异步提交，崩溃后数据库一致、只丢尾巴；
   后者关掉所有刷盘约束，崩溃后可能**损坏**（页与 WAL 的先后不再有保证）。
3. 没有同步备时 `on/remote_*` 自动降级为 `local` 语义（`synchronous_standby_names`
   为空则不等待）——"配置了 on 就有零 RPO"是错觉，必须配同步复制才成立。

### 4.2 InnoDB：innodb_flush_log_at_trx_commit 三档

| 值 | commit 时动作 | mysqld 崩溃（OS 活着）丢什么 | OS/断电丢什么 |
|---|---|---|---|
| 1（默认） | redo write + fsync | 零 | 零 |
| 2 | redo write（进 OS 缓存），每秒后台 fsync | 零（OS 页缓存还在） | 约最近 1 秒 |
| 0 | 什么都不做，每秒后台 write+fsync | 约最近 1 秒 | 约最近 1 秒 |

（MySQL 8.4 Reference Manual, `innodb_flush_log_at_trx_commit`，2024。）
对照 PG：InnoDB 的 `2` 档在 PG 里没有直接对应——PG 的 `off` 连 write 都不等
（比 2 更松、接近 0 档但窗口更短且可按事务选）；PG 没有"每秒定时器"语义，
walwriter 的 `wal_writer_delay` 循环是近似物。另注意 MySQL 实际部署还要叠加
binlog 的 `sync_binlog` 参数，两份日志的组提交耦合（binlog group commit /
LOGICAL_CLOCK）是 MySQL 特有的复杂度——再次印证 §2.9 结论 2。

### 4.3 Redis：appendfsync 三档（作为谱系远端参照）

| 档位 | 语义 | 丢什么 |
|---|---|---|
| always | 每条命令 fsync AOF | 理论零（实际实现为每事件循环批次） |
| everysec（默认） | 每秒 fsync | 约最近 1 秒 |
| no | 交给 OS（约 30 秒） | 数十秒 |

Redis 的价值在于展示谱系的另一端生态：它的默认档就是"丢 1 秒"，因为其定位
（缓存/近似状态）允许把持久性从"属性"降为"概率"。同一张谱系图上，PG 默认档
（on，无同步备时=local）与 InnoDB 默认档（1）都钉在"单机零丢失"，这是关系型
数据库与数据结构服务器的立场差，而非技术差。

### 4.4 统一谱系图

```
零窗口 ←──────────────────────────────────────────→ 大窗口
PG remote_apply > PG on ≈ InnoDB 1 > PG remote_write > PG local
                                        > InnoDB 2 ≈ Redis always(批次)
                                        > InnoDB 0 ≈ Redis everysec ≈ PG off
                                        > Redis no
纵轴（每事务延迟成本）与横轴严格负相关——没有任何实现打破这条曲线，
能改变的只有曲线的"斜率"：组提交（并发聚合）、更快的介质（NVMe/PMEM）、
更近的确认点（remote_write 不等备机 fsync）。
```

**统一结论：所有档位设计都在回答同一个问题——"你愿意为最后 N 毫秒的历史付
多少每事务延迟？"** 谱系设计的好坏不在于默认值多安全，而在于（a）语义是否
可精确陈述（"丢什么"能不能一句话说清），（b）选择粒度多细（PG 的每事务粒度 >
MySQL 的每实例粒度），（c）是否把"丢失"与"损坏"严格分离（PG 的 off 与
fsync=off 之别是教科书示范）。

## 5. 最新进展（截至 2026-07，git log 与公开资料双查证）

### 5.1 AIO：PG 18 落地，PG 19 打磨

git 考古（本仓库）：

- commit da7226993f（2025-03-17）"aio: Add core asynchronous I/O infrastructure"
  ——核心基础设施进入 PG 18；
- commit 55b454d0e1 / 247ce06b88（2025-03-18）`io_method=worker`（I/O 工人进程
  池代执行）；commit c325a7633f（2025-03-18）`io_method=io_uring`（Linux 限定，
  依赖 liburing，commit 8eadd5c73c）；
- 默认值即 worker：`DEFAULT_IO_METHOD = IOMETHOD_WORKER`（aio.h:42）；
- PG 18 的覆盖范围是**读路径**：顺序扫描、bitmap heap scan、VACUUM 等
  read_stream 用户（commit 12ce89fd07 / 047cba7fa0 bufmgr AIO 读支持）；
- PG 19 开发周期继续打磨：commit d1c01b79d4（2026-04-08）**I/O worker 池自动
  调节**（不再需要手工定 `io_workers` 数量）、io_uring 大 I/O 异步处理
  （a9ee668817）、更多 read_stream 化的 AM（如 bloom，4c910f3bbe）、fast path
  同步发射优化（cceb1bf45e）。

外部性能数据（均为 2025 年发布的 PG 18 测评）：冷缓存顺序扫描/VACUUM 常见
**2–3× 提升**，云盘（高单请求延迟）收益最大（pganalyze《Waiting for Postgres
18: Accelerating Disk Reads with Asynchronous I/O》2025；CYBERTEC《PostgreSQL 18:
Better I/O performance with AIO》2025）。有趣的反直觉点：高带宽本地盘上
`worker` 有时**快于** `io_uring`——因为校验和计算与内存拷贝可以摊给 worker
进程的 CPU，而 io_uring 路径把这些留在 backend 单线程里（PlanetScale
《Benchmarking Postgres 17 vs 18》2025 亦观察到 io_method 间的负载相关差异）。
这提醒我们：AIO 的收益结构是"隐藏延迟 + 卸载 CPU"两部分，谁占主导取决于介质。

### 5.2 DIO 路线图：AIO 的下一站

现状：数据文件 I/O 仍是 buffered；`debug_io_direct`（guc_tables.c:625，PG 16
引入）仅供开发测试，WAL 段初始化已有专门的 DIO 位（`IO_DIRECT_WAL_INIT`，
xlog.c:3285）。路线图（Andres Freund，"AIO in Postgres 18 and beyond"，
pgconf.eu 2025-10-23 幻灯片）：**写路径 AIO → DIO 成为可选项 → 绕过页缓存的
双缓冲**。DIO 落地才能接上 §3.3 的 RWF_ATOMIC（内核原子写以 DIO 为先），
所以 "FPW 退休"的依赖链是：写路径 AIO（PG 19+ 开发中）→ DIO → untorn writes
→ `full_page_writes` 默认关闭。每一环都是多年期工程。

### 5.3 wal_compression 演化

时间线（均已在 §3.4 引用 commit）：9.5 pglz（2015）→ PG 15 lz4 + zstd
（2021/2022）→ 本周期的观测增强 `wal_fpi_bytes`（f9a09aa295，2025-10-28）。
方向清晰：先给算法选择，再给量化观测。尚未发生但被讨论过的下一步是压缩普通

### 5.4 增量备份：PG 17 的 walsummarizer 体系

commit dc21234005（2023-12-20）"Add support for incremental backup"（Robert
Haas，PG 17）：新增 **walsummarizer** 后台进程（postmaster/walsummarizer.c）
持续把 WAL 蒸馏成"每个 relation 哪些块被改过"的摘要（block reference table），
`pg_basebackup --incremental` 据此只传变更块，恢复端用 **pg_combinebackup**
（src/bin/pg_combinebackup/）把全量+若干增量合成可用数据目录。

设计空间视角的看点：这是**用 WAL 的第四份职责换掉"块级脏位图"**——Oracle/
InnoDB 系的增量备份要么扫全库比对，要么维护在线变更位图（额外运行时开销）；
PG 复用"反正都要写的 WAL"离线蒸馏，运行时近零开销，代价是摘要滞后于 WAL 与
一批边角 bug 的修复期：`wal_level=minimal` 生成的 WAL 拒绝摘要（402b586d0a）、
增量块 BLCKSZ 对齐（10e3226ba1）、VM fork 膨胀（ffc226ab64，2025）、
INCREMENTAL.* 文件名劫持限制（9e72f6bfae）等——一个新持久化特性的"修坟期"
通常要跨两个大版本，这个案例是量化样本。

### 5.5 TDE：核心仍然没有，生态先行

截至本仓库 master（20devel，2026-07），核心**没有** TDE：全库加密补丁系
（cluster file encryption，2019–2023 多轮提交未遂）因密钥管理与 WAL/CRC 交互
的复杂度停滞。生态侧：Percona `pg_tde` 扩展 2025-07-01 GA，2.1 版（2025-11-27）
宣称 WAL 加密生产可用，且适配了 PG 18 的 AIO 默认路径，PG 18 起独立打包
（Percona 文档与发布公告，2025-11）。设计空间注脚：TDE 与本篇主题的交点在
**FPW/CRC/压缩与加密的顺序问题**（先压后加密、CRC 算在密文还是明文、FPI 是
否泄露页结构），这正是核心补丁多年难产的技术根源之一——持久化格式的每个字节
都被太多子系统共享（复制、备份、pg_rewind、pg_waldump），加密是对"一份日志
多份职责"红利的反向压力测试。

## 6. trick 与 bug 考古

### 6.1 fsyncgate 2018 全案复盘（重头戏）

**现象与引爆**：2018-03，Craig Ringer 在 pgsql-hackers 发帖《PostgreSQL's
handling of fsync() errors is unsafe and risks data loss at least on XFS》：
生产环境（薄配置存储短暂 ENOSPC/EIO）出现 checkpoint 报错重试后"成功"、
实则数据已丢的案例。LWN 随后以《PostgreSQL's fsync() surprise》（2018-04-23）
把它变成跨社区事件，LSF/MM 2018 峰会专门讨论；全案原始邮件被 danluu.com/
fsyncgate/ 存档，PG wiki 建立了 Fsync Errors 页持续追踪各内核行为差异。

**Linux fsync 语义陷阱**（问题的另一半在内核）：

1. **错误只报告一次**：writeback 失败后，内核把错误记在 file/inode 上，第一个
   fsync 调用者拿到 EIO，错误标志随即清除——**下一次 fsync 返回成功**；
2. **脏页被标记为干净**：更致命的是，多数文件系统在 writeback 失败后把脏页
   标成干净（避免坏设备把内存钉死），数据在内核里已经没了——"成功"的重试
   fsync 刷的是空气；
3. **错误可能报给错的人**：4.13 之前，错误挂在 file 描述符上，如果错误发生在
   checkpointer 打开该文件**之前**，checkpointer 的 fsync 永远看不到它（PG
   的 fd 缓存 + 多进程各自 open 使这个窗口真实存在）。内核侧 Jeff Layton 的
   errseq_t（Linux 4.13，2017）把错误挂到 inode 的 address_space 并带序列号，
   保证之后每个打开者的首次 fsync 都能看到——但这只修了"报告丢失"，报告一个
   已丢数据的错误，数据还是丢了。

**PG 的错误假设**：PG 二十年来的隐含模型是"fsync 失败 → 稍后重试 → 重试成功
= 数据落盘"。checkpoint 代码正是这么写的：fsync 报错则本轮 checkpoint 失败
（ERROR），脏页留着，下轮重试——在上述内核语义下，下轮"成功"是假的，
checkpoint 推进 redo 起点，**WAL 被回收，最后一份完好数据被亲手删掉**。
静默丢数据的完整链条就此闭合。

**修复**：commit 9ccdd7f66e（2018-11-19，Thomas Munro 提交，Craig Ringer 署名，
回补所有支持分支）"PANIC on fsync() failure"——放弃重试幻想：数据文件 fsync
一旦失败立即 PANIC，强制走崩溃恢复，从 WAL（数据的唯一存活副本）重放。提交
信息明说：WAL 的 fsync 失败早已 PANIC，此次把同样待遇扩展到数据文件。同时给
出 `data_sync_retry` GUC（fd.c:163，默认 false=PANIC）供"不丢弃脏页的 OS"
（如 FreeBSD/ZFS 系）保留旧行为；`data_sync_elevel()`（fd.c:3967）把散落各处
的错误级别统一升格。提交信息还诚实记录了残余窗口：文件关闭后 inode 被内存
压力驱逐再重开，错误仍可能漏报——后续 PG 13 的 fsync 请求转发改用文件描述符
传递（keep the fd open）进一步压缩该窗口。

**跨系统影响**：danluu 的存档与后续报道显示 MySQL/InnoDB、MongoDB（WiredTiger）
存在同构假设并跟进修复（fsync 失败即视为致命）；内核侧则催生了对 sync_file_range
语义的重新审视（它从不报告 writeback 错误，PG 至今在 `checkpoint_flush_after`
里使用它仅作为提示而非持久性保证）。

**设计空间教训**（本篇视角的三条）：

1. **持久化协议的正确性依赖于对下层错误模型的假设，而这个假设必须显式成文**。
   PG 的 WAL 协议在数学上无懈可击（09 篇 §10 不变式清单），被击穿的是"fsync
   返回 0 ⇒ 之前所有 write 已持久"这条未写进任何论文的公理；
2. **PANIC 是正确的工程答案**：当下层失去可信度时，唯一安全的动作是回到自己
   可信的持久化结构（WAL）重建世界——这与 §2 矩阵无关，任何方案（doublewrite、
   COW、LSM）遇到"介质谎报成功"都只能如此；
3. **跨层契约要靠社区间碰撞才能暴露**：数据库假设内核语义、内核假设应用会
   立刻检查错误，两边各自"合理"了二十年。fsyncgate 之后 PG wiki 维护各 OS
   fsync 行为对照表，这种"显式化下层契约"的做法本身就是最大遗产。

### 6.2 WAL 段文件复用与"新段为什么不用怕旧内容"的三重校验 trick

**trick**：WAL 段写满后不删除，而是改名复用为未来的段（`InstallXLogFileSegment`，
xlog.c:525 附近状态机；回收判定 xlog.c:4076 `wal_recycle`）。收益：稳态下
pg_wal 里的段文件全是"二手"的，避免了每 16MB 一次的创建+清零+元数据 fsync。

**问题**：二手段里满是上个周期的合法 WAL 记录——CRC 完全正确。恢复读到段尾
之后，怎么知道"接下来的记录是旧租户的遗物而不是新日志"？答案是三重校验，
任何一重失配即判定"WAL 到此为止"：

1. **页头地址**：每个 8K WAL 页的头部带 `xlp_pageaddr`（本页应有的 LSN 地址），
   读取器校验失配即停（xlogreader.c:1325）。这正是 commit 7d4d5c00f0
   （2001-07-19）引入复用机制时**同一提交**加上的——提交信息明说：为了让
   改名复用安全，必须能把旧化身的页可靠区分出来；
2. **反向链**：每条记录的 `xl_prev` 必须指向前一条记录的起点（09 篇 §1.2 表），
   旧记录的 xl_prev 指向旧时间线的位置，对不上即停；
3. **CRC**：覆盖记录全文，防半条记录/介质损坏。

**为什么新段仍然默认清零**（`wal_init_zero=true`，xlog.c:134）：不是为了正确性
——上面三重校验不依赖零页——而是为了**空间预留**：xlog.c:3300 起的注释写明，
逐字节写零是确保文件空间真实分配（稀疏文件的 hole 会在未来写入时才分配，
届时 ENOSPC 会打在提交路径上）且间接块已落盘，之后 `fdatasync` 才够用。
在 COW 文件系统（ZFS）上预写零两头落空，所以 commit 475861b261（2019-04-02）
才把 `wal_init_zero`/`wal_recycle` 做成 GUC 让 ZFS 用户双关。**一个"清零"
动作里叠着三层不同目的（空间、性能、无关正确性），是读源码时厘清"机制为谁
服务"的绝佳练习。**

### 6.3 XLP_FIRST_IS_CONTRECORD 与 overwrite contrecord bug（PG 14 时代著名恢复 bug）

**背景机制**：WAL 记录跨页/跨段续写时，后续页头打 `XLP_FIRST_IS_CONTRECORD`
标志（xlog_internal.h:75），表示"本页开头是上一条记录的续体"。

**bug（commit ff9f111bce 提交信息完整记录了案发过程）**：物理复制/归档按
**整段**发货。一条记录横跨段 A/B，段 A 写满已被发给 standby，主库在写段 B
之前崩溃。主库崩溃恢复读到段 A 尾部的半条记录，正确地判定"WAL 到此为止"，
**从半条记录的起点继续写新日志**——本地完全正确。但 standby 手里那份段 A
是旧版本（含半条记录头），它苦等段 B 开头的续体，等来的却是主库新写的、
毫不相干的字节：`LOG: invalid contrecord length 7262 at A8/D9FFFBC8`，复制
永久卡死，运维只能删段重拉或干脆重建 standby。**根因是一个分布式一致性缺口：
主库假设"我没写完的记录可以就地覆写"，但该记录的前半已经发布给了不会倒带的
订阅者。**

**第一次修复尝试失败**：commit 515e3d84a0（提交信息中提及）只覆盖归档场景且
有扩展性问题，被 revert——恢复协议的修补自身也要走弯路。

**最终修复**（ff9f111bce，2021-09-29，Álvaro Herrera，方案出自 Andres Freund，
回补至当时所有支持分支，PG 14.1/13.5 等）：主库**不再覆写**断头记录的已发布
部分，而是从下一个正常位置开始写，并写入一条专用记录
`XLOG_OVERWRITE_CONTRECORD`（插入点 xlog.c:8028，`CreateOverwriteContrecordRecord`），
同时在续体本应出现的页头打 `XLP_FIRST_IS_OVERWRITE_CONTRECORD`（
xlog_internal.h:79）——向所有下游宣告："那半条记录作废，跳过它继续"。standby
重放时据此优雅跨过断点。注意提交信息里的运维告诫：因为引入了新 WAL 记录
类型，**必须先升 standby 后升 primary**，否则旧 standby 读到新记录直接不认。

**余波不断**（同一机制后续修复链，git log 可查）：6672d79139（2022-08-29）
promotion 时 `abortedRecPtr` 处理不当仍可造成 WAL 损坏；44bd3ed332 修
OVERWRITTEN_CONTRECORD 的 LSN 判定；40c516bba8 修逻辑解码对该记录的处理；
直至 d3917d8f13（2025-07-19）还在修"读半条记录无限等待"。**一条 0x0004 的
标志位牵出跨越四年的修复链——"日志的末尾在哪里"这个看似平凡的问题，在
'日志同时是复制流'的约束下是全系统最难的边界条件**（呼应 §1.3）。

### 6.4 restore_command 注入风险与 PITR 误用

**机制**：归档恢复时，PG 把 `restore_command` 字符串做 `%f`/`%p` 等替换后直接
交给 `system()` 执行（`RestoreArchivedFile`，xlogarchive.c:55，执行点
xlogarchive.c:178；`recovery_end_command` 同样，xlogarchive.c:331）。

**风险面一：配置即代码**。`restore_command` 以 postgres OS 用户的 shell 权限
执行任意命令。攻击路径不是注入参数（%f/%p 的值来自 PG 自身生成的文件名），
而是**谁能写这个配置**：`ALTER SYSTEM SET restore_command`（superuser）+ 触发
恢复 = 数据库超级用户到 OS 用户的权限升级台阶。云厂商（RDS 等）因此一律禁用
ALTER SYSTEM 或以受限代理执行归档命令。PG 自身的立场记录在文档：archive/
restore_command 的安全边界等同于"能编辑 postgresql.conf 的人"，这是设计决定
（灵活性优先），不是疏忽——但任何做托管/多租户的人必须知道这道门。

**风险面二：命令语义误用**（更常见的事故来源）：

- restore_command 对"文件不存在"**必须返回非零**且不能把半个文件留在 %p——
  用 `cp` 天然满足，用自写脚本/对象存储 CLI 时"重试到超时再返回 0"或留下
  截断文件会让恢复把垃圾当日志（所幸 §6.2 的三重校验多数时候能兜住截断，
  但兜不住"完整但过老"的文件放错位置）；
- `system()` 等待期间忽略 SIGINT/SIGQUIT 的语义（xlogarchive.c:252 注释）
  意味着挂死的 restore_command 会让恢复无法用 Ctrl-C 打断——脚本必须自带
  超时；
- PITR 目标误用：`recovery_target_time` 落在目标事务提交**之前**一毫秒、
  忘记 `recovery_target_inclusive`、或到达目标后默认动作（`recovery_target_action`）
  不是 pause 导致直接 promote——时间线随即分叉，归档里出现 `.history` 文件，
  第二次尝试若不指定 `recovery_target_timeline` 会沿新时间线走错路。PITR 的
  操作协议本质上是在手工执行一个分布式系统的"时间旅行 + 分叉管理"，这也是
  Neon 把 PITR 做成 `GetPage@LSN` 一等公民（§2.8）的产品动机：把最容易误用的
  运维协议变成 API。

### 6.5 checkpoint_segments → max_wal_size：一个 GUC 的用户心智重构

commit 88e9823026（2015-02-23，Heikki Linnakangas，PG 9.5）用
`min_wal_size`/`max_wal_size` 取代了服役 14 年的 `checkpoint_segments`。这不是
改名，是**计费模型反转**：

- 旧参数回答"两次 checkpoint 之间允许多少段"——但用户真正关心的磁盘占用是
  它的隐函数（稳态约 `(2 + checkpoint_completion_target) × checkpoint_segments
  + 1` 段，还得自己乘 16MB），设小了 checkpoint 风暴（§3.4 正反馈），设大了
  磁盘意外爆；
- 新参数直接回答"pg_wal 最多占多少字节"（软上限），checkpoint 触发点反推
  得出；`min_wal_size` 则控制低谷期回收多少段（复用 vs 删除，接 §6.2）。
  后续 d661532e27 把 standby 的 restartpoint 也纳入 max_wal_size 约束。

考古的价值在于它示范了 GUC 设计的一条原则：**参数应当用用户的资源语言
（字节、时间）而非实现的内部语言（段数、周期）命名**。同一原则的后续执行：
`wal_keep_segments → wal_keep_size`（c3fe108c02，PG 13）。反例警示则是
`max_wal_size` 是"软"限制（恢复期、`wal_keep_size`、卡住的复制槽都能突破），
用户以为是硬上限而磁盘打爆的工单至今常见——改名只能修一半心智模型。

### 6.6 wal_level 演化与"提升等级"的兼容性坑

演化线（git 查证）：9.0 时代 `minimal/archive/hot_standby` → commit b555ed8102
（2016-02-29，PG 9.6）合并 archive+hot_standby 为 `replica`（两者成本差已
微不足道，双选项只剩认知负担）→ commit f6d6d2920d（2017-01-14，PG 10）默认值
从 minimal 升为 replica（同时抬高 max_wal_senders/max_replication_slots 默认，
"默认可复制"成为产品立场）。

坑的清单（全部源于"等级越低、WAL 信息越少"这一单向性）：

1. **minimal 的 WAL 不完整**：`wal_level=minimal` 下，同事务内新建的表批量
   载入可以**跳过 WAL**（靠提交前 fsync 关系文件保证持久性）——这些数据在
   WAL 流里根本不存在。因此从 minimal 期间的 WAL 做任何下游消费都是不完整的：
   PG 17 增量备份明确拒绝（commit 402b586d0a"Do not summarize WAL if generated
   with wal_level=minimal"）；物理复制/归档同理被 GUC 联锁直接禁止；
2. **提级需要重启**（wal_level 是 postmaster 级 GUC），且 standby 的
   `hot_standby` 能力取决于**主库**写 WAL 时的等级——备份自 minimal 时期的
   基础备份无法用于建 standby；
3. **降级更危险**：replica→minimal 会使既有复制槽/归档链条静默失效，PG 用
   启动检查（max_wal_senders>0 与 minimal 冲突报错）挡住一部分，挡不住
   "归档链在 minimal 窗口出现空洞"这种事后才发现的问题；
4. **logical 的成本误区**：replica→logical 增加的 WAL 量通常不大（主要是
   UPDATE/DELETE 的旧行标识与 origin 信息），真正的坑在恢复末端的边角——
   如 c209931bd4（2025）还在修"恢复结束时刻 minimal 等级下逻辑解码状态"的
   正确性。等级参数的每一档都是一份跨子系统契约，改默认值（PG 10）比加新档
   （9.4 logical）的溢出效应小得多——GUC 演化的又一条经验。

### 6.7 番外：commit_delay 的"名不副实"考古

`commit_delay`（xlog.c:139）字面像"延迟提交换聚合"，但它的现代实现（PG 9.3
重写后）只在**即将成为 flush 领导者的进程**上睡眠（xlog.c:2899-2902：仅当
`CommitDelay > 0 && enableFsync && MinimumActiveBackends(CommitSiblings)`），
跟随者本来就在等领导者，不额外睡——所以它放大的是每次 fsync 的**批规模**而
非每个事务的延迟。这个参数从 7.x 时代活到今天，语义已经历三代实现，文档里
"几乎没人需要设置它"的注记是对 §1.1"隐式组提交已经够好"的官方盖章。低速
介质（机械盘/高延迟云盘）+ 高并发小事务仍是它罕见的用武之地。

## 7. 自测（附详解）

**Q1. 为什么说"每事务 fsync vs 缓冲聚合"的矛盾存在免费午餐区间？PG 靠什么
机制吃到它？**

详解：矛盾的原始形态是"聚合必须扩大丢失窗口"，但当并发提交者众多时，一次
fsync 天然覆盖多个事务——聚合率随并发自动上升而窗口不变（每个事务仍等到
自己的 LSN 落盘才返回）。PG 的 `XLogFlush` 等待/接力协议（09 篇 §4.1）让
后到者搭前者的刷盘便车，即隐式组提交；`commit_delay` 只是并发不足时人为
攒批的补充（§6.7）。免费午餐的边界：单连接串行负载没有可聚合的同伴，TPS
仍被 fsync 延迟锁死——此时唯一的杠杆是 `synchronous_commit=off` 付窗口。

**Q2. 在 §2.9 矩阵中，为什么"恢复时间 ▲"的方案没有一个同时做到"写放大 ▲"？
这是巧合吗？**

详解：不是巧合，是守恒律。恢复时间 O(1) 意味着崩溃瞬间磁盘上必须已存在一个
完整一致的状态（COW 的旧根、LSM 的 SSTable+短 WAL、Aurora 的持续物化），
维持"随时一致"必然要在写路径或后台整理上持续付费（级联 COW、compaction、
存储层重放）。反之 WAL 家族允许磁盘上是"不一致基底 + 修复日志"，写路径最省
（顺序追加），把整理工作推迟到崩溃后一次性支付。恢复矛盾（§1.2）只能移动
付款时点，不能消灭账单。

**Q3. FPW、doublewrite、RWF_ATOMIC 三种撕裂解法，各自把保险费记在哪个账户？
为什么说 FPW 是唯一"随架构演进可整体消失"的？**

详解：FPW 记在 WAL 体积 → 连带复制带宽与归档账户（§3.4）；doublewrite 记在
本地刷盘 I/O 账户（顺序双写，与日志无关）；RWF_ATOMIC 把费用一次性转为硬件
能力 + 内核版本 + DIO 改造。FPW 消失的两条路径都已可见：架构上（Aurora/Neon
页面不原地写回，撕裂窗口在结构上不存在）与硬件上（原子写贯通后 `full_page_
writes=off` 合法化）；doublewrite 与存储架构无关，只有硬件豁免一条路。附加
记忆点：PG 侧 RWF_ATOMIC 的前置依赖是写路径 AIO+DIO（§5.2），截至 20devel
源码中无任何 RWF_ATOMIC 引用。

**Q4. `synchronous_commit=off` 与 `fsync=off`、InnoDB 的 0/2 档各丢什么？
哪个可能导致损坏而非丢失？**

详解：`synchronous_commit=off`——WAL 顺序与页/WAL 先后关系全部保留，只是
commit 不等刷盘：崩溃丢最近的尾巴（至多约 3×wal_writer_delay），库仍一致。
`fsync=off`——放弃一切刷盘顺序约束，页可能先于其 WAL 落盘，崩溃后 redo 基底
被破坏：**损坏**而非丢失，两者有本质区别。InnoDB 0 档：commit 什么都不做，
每秒后台刷，mysqld 崩溃或断电都丢约 1 秒；2 档：commit 时 write 进 OS 缓存，
mysqld 崩溃不丢（OS 还在），断电丢约 1 秒。四者中只有 `fsync=off` 属于
"损坏"象限（§4.1 细节 2）。

**Q5. fsyncgate 中，即使内核 4.13 的 errseq_t 修复完全部署，PG 2018 年前的
重试逻辑仍然丢数据。为什么？**

详解：errseq_t 只修"错误报告给谁"（之后每个打开者首次 fsync 必见错误），
不修"数据去哪了"——writeback 失败时内核已把脏页标干净丢弃，数据在内核缓存
里不存在了。PG 旧逻辑收到 EIO 后保留自己 shared buffer 里的脏页下轮重刷吗？
不——PG 的 checkpoint 完成 fsync 时，脏页早已 write() 交给内核并从 PG 侧
视为已写，PG 手里没有副本可重刷；重试的 fsync 即使"如实"再报错，PG 能做的
也只有放弃 checkpoint，而唯一存活副本在 WAL 里。所以正确动作只有 PANIC 走
恢复（9ccdd7f66e），与报告机制是否完善无关（§6.1 教训 2）。

**Q6. WAL 段复用后，恢复如何区分"日志正常结束"与"读到了旧租户的遗物"？
如果三重校验只剩 CRC 一重，构造一个会被误判的场景。**

详解：三重校验=页头 `xlp_pageaddr`（xlogreader.c:1325）+ 记录 `xl_prev` 反链
+ CRC（§6.2）。只剩 CRC 时的误判场景：段 000...0A 被复用改名为 000...2A 但
尚未被新日志写到第 3 页，恢复重放推进到该段第 3 页——页里是旧化身的完整
合法记录，CRC 全部通过，恢复会把上个周期的旧事务当新日志重放，造成时间线
错乱的静默损坏。`xlp_pageaddr` 一比对（页自称地址是 0A 段的偏移，与当前读
位置 2A 不符）即识破。这就是 7d4d5c00f0 必须在引入复用的同一提交里加页地址
字段的原因。

**Q7. overwrite contrecord bug 的根因为什么被归为"分布式一致性缺口"而不是
单机恢复 bug？修复方案为什么选择"写一条新类型记录"而不是"standby 主动丢弃
半条记录"？**

详解：单机视角下崩溃恢复的行为完全正确——半条记录无效，从其起点覆写天经
地义。错误出在**发布语义**：段 A 一旦整段发给 standby/归档，其内容就成了
不可收回的承诺，主库覆写等于单方面篡改已发布历史，而订阅者不会倒带（§6.3）。
修复不能靠 standby 侧丢弃，因为 standby 无法区分"这半条记录永远不会有续体"
与"续体还在网络路上"——只有主库知道真相，所以必须由主库显式广播
`XLOG_OVERWRITE_CONTRECORD`"作废声明"，让所有下游（含逻辑解码，40c516bba8）
获得确定性判据。这与 §1.3 的主题闭环：日志一旦兼任复制流，"日志的结尾"
就从本地判断升格为需要协议保证的分布式共识。
