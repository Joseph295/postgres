- 几个贯穿全篇的术语：
  - **CBO / RBO**：基于代价 / 基于规则的优化器。RBO 按固定优先级选计划（如"有索引就走索引"），不看数据分布。

  - **Q-error**：估计值与真实值之比取大于 1 的方向，`max(est/act, act/est)`。学术界衡量基数估计质量的标准指标。

## 1. 问题定义：三大难题与"优化器悖论"

### 1.1 难题一：搜索空间爆炸——已被基本驯服

n 表连接的计划数按 `n! × Cat(n-1) × 3^(n-1)` 增长，12 表就是 10^18 量级（组合数表见 03 篇 §1.1）。但要说清楚一个常被误解的事实：**这是三大难题里唯一被基本解决的一个**。动态规划（利用最优子结构把"枚举计划"变成"枚举子问题"）、连接图剪枝（只组合有连接条件相连的子问题）、支配剪枝（每个子问题只留帕累托前沿）三层手段叠加后，几十个表的查询在毫秒到几十毫秒内可以规划完；实在不行还有随机化兜底（GEQO）和问题切分（`from_collapse_limit`/`join_collapse_limit`）。搜索算法的边际收益在 1990 年代后期就已经很小了。

### 1.2 难题二：基数估计——真正的"万恶之源"

优化器对每个中间结果的行数估计建立在三个假设上（详见 §4.1）：

- **均匀性**：直方图桶内、MCV 之外的值均匀分布；
- **独立性**：不同谓词的选择率相乘；
- **包含性**：join 时小 distinct 集是大 distinct 集的子集。

这三个假设在真实数据上系统性失效，而且**误差不是加性的，是乘性的**：join 树每高一层，下层的行数误差乘进上层的选择率计算。03 篇 §8.4 给过数值推演：单层 3 倍误差，五层连接后就是 3^5 ≈ 243 倍。Leis 等人 2015 年的实证研究（"How Good Are Query Optimizers, Really?"，JOB 基准，用 IMDB 真实数据）测量了主流数据库的中间结果估计：**在真实相关数据上，6 表以上连接的中间基数估计 Q-error 中位数就达到 10^2，尾部达到 10^4–10^8**，且所有被测系统（PG、主流商业库）无一幸免。

### 1.3 难题三：代价模型失真——重要性常被高估

即使行数全对，代价公式仍可能失真：`random_page_cost = 4` 是机械盘时代的常数（NVMe 上实测应接近 1.1–1.5）；缓存命中建模只有 Mackert-Lohman 公式 + `effective_cache_size` 的粗糙近似（03 篇 §7.3）；CPU 代价三常数（`cpu_tuple_cost` 等）与真实指令数的关系从未标定过。但 Leis 2015 的同一组实验给了一个重要定量结论：**给定真实基数，即使把代价模型整个换成"只数中间结果行数"的玩具模型，计划质量损失也很小；反之给定默认代价模型，把估计基数换成真实基数，计划质量大幅提升**。代价模型误差通常在 2–10 倍内，且是系统性偏差（各计划同向偏移，排序常常不变）；基数误差是 10^2–10^8 且方向随机。

### 1.4 优化器悖论：Lohman 之问

Guy Lohman（IBM，System R 谱系的直系传人，DB2 优化器负责人之一）2014 年在 ACM SIGMOD Blog 发表 *Is Query Optimization a "Solved" Problem?*，论点可以概括为三句话：

1. **"The root of all evil, the Achilles Heel of query optimization, is the estimation of the size of intermediate results."**——代价模型误差最多造成 30% 量级的偏差，而基数估计误差"can quickly grow to many orders of magnitude"；
2. 学术界与工业界几十年的投入结构是倒置的：论文绝大多数在改进搜索算法（join ordering、新框架、剪枝策略），而搜索算法只是在"对一堆错误的数字做精确的极值运算"；
3. 独立性假设、均匀性假设造成的误差**沿计划自底向上按乘法累积**，因此越复杂的查询，优化器"精确搜索"的意义越接近于在噪声里选最小值——这就是**优化器悖论**：搜索越彻底，越有能力找到那个"因为估计错误而看起来最便宜"的极端计划。一个只搜左深树的笨优化器反而因为空间小、计划形态保守，翻车更少。

这个悖论解释了本篇后面几乎所有的设计分歧：hints 之争（§3）本质是"当估计不可信时，人要不要能接管"；自适应执行（§2.6）本质是"把决策推迟到真实基数可观测之后"；learned CE（§4.3）本质是"直接学联合分布，绕开独立性假设"。**三大难题里，搜索是已解的，代价模型是次要的，基数估计是核心的——记住这个权重排序，是评价一切优化器方案的前提。**

## 2. 方案谱系与优劣矩阵

### 2.0 统一比较维度

先定义四个维度，避免"各说各话"：

- **搜索完备性**：在给定的逻辑等价空间里，能否保证找到代价模型意义下的最优计划？（注意限定语：完备 ≠ 实际最优，见 §1.4）
- **扩展性**：加一个新算子/新等价变换（比如"聚合下推""新 join 算法"）要改多少地方、侵入多深？
- **可预测性**：同一查询在统计微小扰动下计划是否稳定？规划结果是否可解释、可复现？
- **规划耗时**：典型 OLTP 查询（<8 表）与极端查询（>20 表）的规划开销量级。

### 2.1 流派一：自底向上动态规划（System R → PG、DB2）

**代表**：System R（1979）、DB2、PostgreSQL（`standard_join_search`，`src/backend/optimizer/path/allpaths.c`）、MySQL 8.0 的新 hypergraph 优化器（DPhyp 算法，同属此族）。

**机制**：按基表集合大小逐层构造，`join_rel_level[j]` 存所有 j 表子问题的最优路径集，天然利用最优子结构；interesting orders（PG 的 PathKeys）保证"每种有用排序各留一个最优"。

**四维评价**：
- 完备性：★★★★★。在"逻辑变换已前置完成"的前提下，对连接顺序 × 实现方法 × interesting orders 的空间是穷尽的（PG 还枚举 bushy 树，比 System R 只搜左深更完备）；
- 扩展性：★★☆。加一种新扫描/join 方法容易（多一个 `create_xxx_path` 调用点）；但加一条**代价相关的逻辑变换**（如"聚合提前"）很痛——DP 的分层结构假设逻辑形状固定，要打破就得像 eager aggregation 那样给每个 joinrel 平行地多养一套 `grouped_rel`（commit `8e11859102`，见 §5.1），侵入面大；
- 可预测性：★★★★。确定性算法，无随机性（GEQO 除外）；但"代价几乎相等的两个计划因统计微扰翻转"的问题与所有 CBO 共享；
- 规划耗时：小查询极快（无 memo 管理开销，C 结构直接堆砌）；大查询指数增长，需 GEQO/collapse limit 兜底。

### 2.2 流派二：自顶向下变换规则框架（Volcano/Cascades）

**代表**：Volcano（Graefe & McKenna 1993）→ Cascades（Graefe 1995）→ **SQL Server**（Cascades 的第一个工业实现，Graefe 亲自操刀）、**Greenplum ORCA**（开源的独立优化器进程，服务 Greenplum/曾计划服务 HAWQ）、**Apache Calcite**（Java 生态的优化器框架，Hive/Flink/Druid/Phoenix 都在用）、**CockroachDB**（2018 年重写，纯 Go 的 Cascades，规则用 DSL（Optgen）生成代码）。

**机制**：一切皆规则。memo 里每个 group 是一个逻辑等价类，优化 = 不断对 group 应用变换规则（transformation rules 产生新逻辑表达式，implementation rules 产生物理算子），自顶向下按需（on demand）递归，配 promise 排序与代价上界（branch-and-bound）剪枝。required properties（排序、分布）沿递归下传，代替 interesting orders。

**四维评价**：
- 完备性：★★★★☆。理论上规则闭包覆盖整个等价空间；实践中为控规划时间会分 stage/timeout（SQL Server 的 search 0/1/2 阶段，超时即取当前最优——所以实际是"预算内完备"）；
- 扩展性：★★★★★。这是它存在的理由：新算子 = 新规则文件，不碰搜索引擎。CockroachDB 用 Optgen DSL 写了几百条规则；Calcite 作为"优化器即库"被十几个系统复用——这种复用在 DP 流派里从未发生过；
- 可预测性：★★★。规则触发顺序、timeout 位置都影响结果；SQL Server 的计划回归（升级后变慢）是其生态里专门产业（Query Store 就是为此而生）；
- 规划耗时：memo 管理、规则匹配有固定开销，小查询比 PG 慢一个常数因子；大查询靠上界剪枝反而可能更好。

**与 PG 的深层分歧**在 03 篇 §1.3 已展开，这里只补一句结构性判断：Cascades 把"逻辑改写"与"物理选择"统一进搜索，赢得扩展性；PG 把逻辑改写前置为一次性预处理（子链接上拉、子查询展平、外连接消除），把搜索压缩成"只剩连接顺序与实现方法"的纯 DP 形状，赢得速度与简单性。**两者的 memo 之差不是有无，而是维度**：PG 的 `join_rel_hash`（`src/backend/optimizer/util/relnode.c:616-679`）是以 relids 位图为 key 的"穷人版 memo"，它只能给"基表集合"建等价类；Cascades 的 memo 能给"任意逻辑表达式"建等价类，所以才装得下"聚合在 join 前还是 join 后"这类问题。

### 2.3 流派三：启发式 / 规则优先（早期 MySQL、Oracle RBO）

**代表**：MySQL 8.0 之前的老优化器（贪心左深搜索 `greedy_search` + `optimizer_prune_level` 启发剪枝 + 大量硬编码规则）、Oracle 10g 之前的 RBO 模式、多数嵌入式数据库（SQLite 的优化器也以启发式为主，带一个小型代价搜索）。

**机制**：固定优先级规则（可用索引 > 全表扫）+ 贪心构造连接顺序（每步选"当前看起来最便宜"的下一个表），不维护子问题最优解集合。

**四维评价**：
- 完备性：★。贪心无回溯，错过最优是常态；只搜左深树；
- 扩展性：★★。规则是硬编码的 if-else，加算子要通读并小心排序所有既有规则的优先级；
- 可预测性：★★★★★。这是被普遍低估的优点——规则式计划**不依赖统计**，数据涨十倍计划不变，OLTP 场景（点查/两三表 join）里"永远走主键索引"恰恰就是最优解。MySQL 撑起互联网 OLTP 半壁江山，靠的正是"在简单查询空间里，启发式与最优解重合"；
- 规划耗时：最快，近似 O(n²)。

**历史评注**：这条路线的失败之处在复杂分析查询——这也是 MySQL 8.0.31+ 引入 hypergraph join optimizer（回到流派一的 DP）的原因。设计空间的教训：**启发式不是"劣质 CBO"，而是"对查询复杂度分布的一种押注"**；押对了（OLTP）就是最优工程，押错了（OLAP）就没救。

### 2.4 流派四：随机化搜索（GEQO、模拟退火、迭代改进）

**代表**：PG 的 GEQO（`src/backend/optimizer/geqo/`，遗传算法，`geqo_threshold` 默认 12 表触发）；学术上 Ioannidis & Kang 1990 系统比较过模拟退火（SA）、迭代改进（II）、两阶段优化（2PO）；一些系统用随机化 DP 变体（如 iterative DP）。

**机制**：把连接顺序编码为个体（GEQO 编码为"基表访问序列"即 tour），在计划空间上做启发式随机游走（遗传算法的选择/交叉，或退火的邻域跳转），代价函数复用既有的 path 构建与 add_path。

**四维评价**：
- 完备性：★★。无任何保证，只有"概率上不太差"；GEQO 实际是遗传算法皮、Edge Recombination 核，而且每个 tour 还是要走一遍受限的 DP（`gimme_tree`）来变成计划；
- 扩展性：★★★★（搭便车：代价评估复用主线代码，改主线它自动受益）；
- 可预测性：本质是抽签。PG 为此专门把它改成确定性抽签——`geqo_seed` 默认 0.0、每次规划重置私有随机序列（commit `f5bc74192d`，2009；细节见 §6.8），这才让"同一查询两次规划得到不同计划"的投诉消失；
- 规划耗时：多项式且可调（种群大小 × 代数），这正是它存在的全部意义：**给 DP 的指数尾部兜底**。

**设计空间评注**：随机化流派在学术上已基本让位——2015 年后大 join 问题的主流答案是更好的确定性算法（DPhyp、线性化 DP、IKKBZ 启发式做种子）而不是随机游走。PG 社区多次讨论过用其他算法替换 GEQO（惯性 + "12 表以上的查询本来就少"让它活到今天）。

### 2.5 流派五：自适应执行（把决策推迟到运行时）

**代表**：
- **SQL Server**（Intelligent Query Processing 家族）：adaptive join（运行时看实际行数决定 hash 还是 nested loop）、memory grant feedback（下次执行修正内存申请）、interleaved execution（多语句表值函数先执行一段拿到真实基数再优化其余部分）、Query Store 自动强制历史好计划；
- **Spark AQE**（3.0+）：以 shuffle 物化点为界，运行时用真实统计重新优化剩余计划——动态合并分区、sort-merge join 降级为 broadcast join、倾斜 join 自动切分；
- **Oracle**：adaptive plans（计划里内置 statistics collector 算子与备选子计划，运行时二选一）、cardinality feedback / SQL plan directives；
- 学术源头：Kabra & DeWitt 1998（mid-query re-optimization）、Eddies（Avnur & Hellerstein 2000，每个元组动态路由，最激进形态）。

**机制共性**：承认 §1.4 的悖论——既然规划期估计不可信，就在计划里留"观测点 + 决策点"，等真实基数流过再拍板。

**四维评价**：
- 完备性：不适用（它不是搜索算法，是对搜索结果的运行时保险）；
- 扩展性：★★。每个"自适应点"都要执行器配合（物化、算子切换、状态迁移），是跨优化器/执行器的侵入式改造；
- 可预测性：★★。同一查询不同输入走不同物理路径，EXPLAIN 与实际执行脱节，调试难度上升（Spark 的 EXPLAIN 为此专门标注 AQE 改写）；
- 规划/执行耗时：把成本从规划期挪到运行期（观测、物化、可能的重规划）。**对短查询是纯开销，对长查询是保险**——所以它在批处理/OLAP 系统（Spark）里最激进，在 OLTP 系统里最保守。

PG 在这个流派里几乎完全缺席（只有 hash join 运行时增加 batch 数、Memoize 缓存这类算子内微调，计划形状一经选定绝不改变）。为什么？§4.4 专门回答。

### 2.6 优劣矩阵总表

| 维度 \ 流派 | 自底向上 DP（PG/DB2） | Cascades（SQL Server/CRDB/Calcite/ORCA） | 启发式优先（老 MySQL） | 随机化（GEQO/SA） | 自适应执行（AQE/IQP） |
|---|---|---|---|---|---|
| 搜索完备性 | 高（预处理后的空间内穷尽，含 bushy） | 高（规则闭包，但受 stage/timeout 预算截断） | 低（贪心+左深） | 无保证 | 不适用（运行时纠错） |
| 扩展性（加新算子/变换） | 中：加物理算子易，加代价相关逻辑变换难（需侵入 DP 结构） | 高：新规则即插即用，框架可整体复用（Calcite） | 低：硬编码规则互相纠缠 | 高（搭代价评估的便车） | 低：每个自适应点都要改执行器 |
| 可预测性/可解释性 | 高（确定性；但共享"代价近似相等即随机翻转"问题） | 中（规则顺序、timeout 影响结果，计划回归成产业） | 极高（不依赖统计） | 低→中（PG 靠固定 geqo_seed 换回可复现） | 低（EXPLAIN 与实际执行脱节） |
| 规划耗时 | 小查询极快；>12 表指数爆炸需兜底 | 常数开销较大；大查询靠上界剪枝 | 最快 O(n²) | 多项式可调 | 规划快，代价挪到运行期 |
| 隐含哲学 | 逻辑改写前置，搜索只管物理 | 逻辑与物理统一进规则搜索 | 押注查询简单，规则=最优 | 承认指数尾部，概率保底 | 承认估计不可信，观测后再决策 |

**读矩阵的三个要点**：

1. 没有全优列。每一列的强项恰是另一列的弱项：Cascades 的扩展性来自 memo 与规则引擎的固定开销；启发式的可预测性来自放弃搜索；自适应的鲁棒性来自放弃"计划即契约"。
2. **流派正在融合**。MySQL 8.0 从流派三迁到流派一；Spark Catalyst 是"Cascades 式规则框架 + DP 式 join reorder + AQE"三合一；PG 自己也在长出流派二的器官（pgs_mask 钩子把"路径生成策略"变成了可由插件控制的位掩码，commit `4020b370f2`，见 §5.2）和流派五的萌芽（memory feedback 类的讨论）。谱系图正确的读法不是"选哪派"，而是"每个系统按自己的负载分布，从五个流派各抓多少"。
3. 按 §1.4 的权重排序重读此表：五个流派里只有"自适应执行"正面攻击基数估计难题，其余四个都在搜索/架构维度上竞争——这正是 Lohman 批评的投入倒置在设计空间里的投影。

## 3. PG 的独特争论：为什么不做 query hints

### 3.1 论战复盘

PG 官方 TODO 长期把 optimizer hints 列在 **"Features We Do Not Want"** 下，官方 wiki 有专页 [OptimizerHintsDiscussion](https://wiki.postgresql.org/wiki/OptimizerHintsDiscussion) 归档论点。2010 年末 pgsql-performance 邮件列表爆发过延烧数月的著名长帖（起因是用户抱怨"别的库都有 hints"），此后每隔几年复燃一次。双方论点整理如下。

**反方（核心开发者主流，Tom Lane 为代表）**：

1. **hints 是把优化器 bug 固化进应用代码**。用户加 hint 解决了自己的问题就不再报 bug，社区失去改进优化器的信号——"人们用了 hints 就很少向项目报告查询问题，这妨碍了优化器本身的改进"（wiki 原文大意）。这是最有 PG 特色的论点：一个靠社区报告驱动的项目，hints 会切断反馈回路；
2. **hints 不随数据演化**。表小的时候正确的 hint，表大十倍后就是错的；但它藏在千百条 SQL 里没人回头改。优化器的存在意义恰恰是"同一 SQL 在不同数据规模下自动换计划"；
3. **升级灾难**。大版本升级后优化器变强，旧 hints 从"救命"变"拖后腿"，而删 hints 要全量回归测试；
4. **助长坏习惯**：DBA 拍一个 hint 上去，而不是去找真正的问题（缺统计、缺索引、写法不当）；
5. 语法侵入 SQL 文本（Oracle 式注释 hint），使同一 SQL 无法在不同环境间移植。

**正方（用户/运维视角，2010 长帖及历次复燃中反复出现）**：

1. **生产事故没有时间等社区修优化器**。凌晨三点计划翻车，你需要的是十分钟内止血的手段，而不是一个未来版本的修复承诺；
2. 基数估计有理论上无法消除的失效场景（§4.1）——既然优化器**必然**会在某些查询上永远估错（跨表相关就是例子），拒绝提供覆盖手段等于把这些查询判死刑；
3. 用户实际上已经在用"穷人 hints"：`enable_seqscan=off`、`set join_collapse_limit=1`（用 JOIN 语法顺序钉死连接顺序）、拆 CTE 制造优化屏障、`OFFSET 0` 防子查询上拉——**官方拒绝体面的 hints，结果是逼出一堆不体面的 hints**，它们同样不随数据演化、同样藏在应用代码里，还更难读；
4. 每个主要商业库（Oracle、SQL Server、DB2）和多数 PG 兼容云产品都提供了计划控制手段，说明这是真实需求不是坏品味。

### 3.2 pg_hint_plan：体制外的答案

NTT 开发的 [pg_hint_plan](https://github.com/ossc-db/pg_hint_plan) 扩展用注释语法（`/*+ SeqScan(a) HashJoin(a b) Leading((a b) c) */`）通过 planner hook 强制扫描方法/连接方法/连接顺序/行数修正。它的存在本身就是论战的一个论据：**Amazon Aurora PostgreSQL、YugabyteDB 等把它作为默认自带扩展**——市场用脚投票补上了核心拒绝提供的东西。它的实现质量也反证了反方的一个技术论点："hints 难以在 PG 的 hook 体系外实现"并不成立，但**hook 式实现的脆弱性**（每个大版本都要追着 planner 内部结构改）恰恰展示了没有官方接口的代价。

### 3.3 深层分歧：计划稳定性 vs 最优性

把论战再往下挖一层，真正的哲学分歧不是"要不要 hints"，而是**"计划是应该每次重新求最优，还是应该稳定"**：

- **Oracle 路线**：承认计划回归是最大生产风险，提供全套稳定性设施——Stored Outlines（9i，把 hints 集持久化）→ SQL Plan Baselines（11g+，新计划先验证再启用，known-good 计划兜底）。哲学：**可预测性 > 最优性**，宁可稳定在次优计划上；
- **SQL Server 路线**：Query Store 记录历史计划与性能，回归时可一键 force plan，2017+ 还能自动强制（Automatic Plan Correction）。哲学与 Oracle 同向，且更自动化；
- **PG 传统路线**：每次规划都是无记忆的全新求解，没有任何"上次计划"的概念（plancache 的 generic plan 是性能优化不是稳定性设施，而且它本身就是翻车源，见 §6.6）。哲学：**最优性 > 可预测性**，相信统计与代价模型，翻车视为需要修复的 bug。

这个分歧的实质是对 §1.4 悖论的两种回应：Oracle/SQL Server 承认估计不可信，所以给计划上保险；PG 坚持修根因。二十年的运维实践基本裁决了这一局——**大型生产系统需要计划稳定性设施**，PG 生态里 pg_hint_plan + 各云厂商的 plan management 私有实现（Aurora 的 Query Plan Management）就是裁决书。

### 3.4 2026 收尾：pg_plan_advice——第三条道路

论战在本仓库所在的开发周期里发生了戏剧性收尾。Robert Haas 主导的 **pg_plan_advice** 已作为 contrib 模块进入 PG 19（本仓库 `contrib/pg_plan_advice/`，配套 `contrib/pg_stash_advice/`），其 README 开宗明义（`contrib/pg_plan_advice/README`）：

> Goals include (1) enforcing plan stability (my previous plan was good and I would like to keep getting a similar one) and (2) allowing users to experiment with plans other than the one preferred by the optimizer.

机制上它精确地绕开了反方 20 年来的每一条批评：

1. **不是凭空写 hints，而是"从已有好计划提取建议"**：`EXPLAIN (PLAN_ADVICE)` 从既有计划生成文本形式的 advice（`SEQ_SCAN(foo) HASH_JOIN(bar@ss)`…），回灌给未来的规划周期强制复现相同决策——README 强调 **round-trip safe**：生成的每条 advice 保证无歧义地应用回查询的同一部分。这把"hint 不随数据演化"问题转化为"计划固定是用户显式选择的持续决策"；
2. **不侵入 SQL 文本**：advice 通过 GUC（`pg_plan_advice.advice`）或 `pg_stash_advice`（按 query ID 自动供给）注入，应用代码零改动——直接回应反方论点 5；
3. **只钉决策、不改代价**：强制的是"这里用 hash join"这类离散决策，不允许伪造行数（这与 pg_hint_plan 的 `Rows()` 修正 hint 形成对照）；
4. 底座是同期进入核心的通用基础设施：`disabled_nodes` 代价分量（03 篇 §11.4）与 **path generation strategies 位掩码**——每个 RelOptInfo 新增 `pgs_mask`，插件可通过 `joinrel_setup_hook`/`join_path_setup_hook` 按 rel 粒度关闭特定路径生成策略（commit `4020b370f2`，2026-01-28）。这等于给"计划控制"修了官方公路，pg_hint_plan 式的 hook 杂技不再必要。

命名本身就是政治智慧："advice" 而非 "hint"。这场二十年论战的最终答案不是正方或反方获胜，而是重构问题：**PG 拒绝的从来是"内嵌在 SQL 里、伪造估计值的 Oracle 式 hints"，最终接受的是"外置的、以复现既有计划为主要用途的计划稳定性设施"**——恰好落在 Oracle SQL Plan Baseline 与 SQL Server Query Store forced plan 所定义的同一象限，只是晚到了十五年，且做得更"round-trip safe"。

## 4. 基数估计的前沿

### 4.1 默认假设的系统性失效场景清单

PG 的估计器（03 篇 §8）建立在四个假设上，每个都有明确的失效模式：

| 假设 | 内容 | 系统性失效场景 | 典型误差方向 |
|---|---|---|---|
| 属性内均匀 | 直方图桶内均匀、非 MCV 值均匀分摊剩余频率 | 重尾分布的长尾值、桶边界附近的范围查询 | 双向，通常 <10× |
| **属性间独立** | 多谓词选择率相乘：`P(a∧b)=P(a)·P(b)` | 函数依赖（city→country、zip→state）、语义相关（品牌→品类）| **低估**，随谓词数指数恶化 |
| **join 键包含性** | 小 ndistinct 集是大集的子集，match 率按大集算 | 部分匹配的 join（历史表 join 活跃表、外键有大量悬空）| 高估；反向场景（键空间错开）则严重低估 |
| 跨表独立 | join 选择率与两侧过滤谓词无关 | **过滤谓词与 join 结果相关**（"只看 2024 年订单" join "只看新客户"）| 双向，无统计可救 |
| 无信息兜底 | 估不了就取常数：`DEFAULT_NUM_DISTINCT = 200`（`src/include/utils/selfuncs.h:52`）等一组 DEFAULT_*_SEL | 函数结果、复杂表达式、无统计的外部表 | 任意 |

第 2、4 行是重灾区。经典再现：`WHERE city = 'Beijing' AND country = 'China'`——两个谓词几乎完全冗余，真实选择率 ≈ P(city)，PG 算出 P(city)×P(country)，在此基础上每多一个相关谓词再低估一个数量级；低估的行数使上层选择 nested loop，实际行数到来时执行时间爆炸（03 篇 §9.2 有完整灾难链推演）。

### 4.2 扩展统计：能力边界在哪里

PG 10 起的 `CREATE STATISTICS`（`src/backend/statistics/`，基础设施 commit `7b504eb282`，2017）提供三种多列统计：

- **ndistinct**（`mvdistinct.c`）：多列组合的 distinct 数，救 GROUP BY 多相关列的组数估计；
- **functional dependencies**（`dependencies.c`）：列间函数依赖度，救等值谓词的相关性（把 P(a)·P(b) 修正为 P(a)·[f + (1-f)·P(b)]）；
- **MCV lists**（`mcv.c`）：多列联合 MCV，救"组合值分布极不均匀"的场景，且支持不等值。

能力边界（每条都是结构性的，不是实现没做完）：

1. **只覆盖单表**。所有扩展统计定义在一个表上——上表第 4 行"跨表相关"完全无解。这是理论困境：跨表相关统计的候选空间是所有表对×列对，无法预先枚举收集（学术方案是 join synopses / 按 workload 收集，PG 均无）；
2. **要 DBA 显式创建**。默认不存在，等于把"发现相关列"的难题推给人。诊断路径通常是：EXPLAIN ANALYZE 发现行数差数量级 → 猜相关列 → 建统计 → ANALYZE → 复查；
3. 组合爆炸：k 列的候选统计对象是 2^k 级，MCV 列表维度升高后覆盖率骤降（联合分布的稀疏性）；
4. 表达式统计 PG 14 才支持；本仓库刚扩到虚拟生成列（commit `f7f4052a4e`，2026-04）——边界仍在一张表内推进，没有跨表的迹象。

### 4.3 Learned cardinality estimation：学术热与工业冷

2018 年起学术界爆发式产出：**MSCN**（多集卷积网络，query-driven：从历史查询+真实基数学回归）、**Naru/NeuroCard**（深度自回归模型直接学数据联合分布，data-driven）、**DeepDB**（Sum-Product Network 学联合分布）等。在基准上它们把相关列场景的 Q-error 从 10^3 压到个位数——直接消灭独立性假设。

但工业落地截至 2026 年仍近乎为零（主流引擎默认路径无一采用）。两篇系统评测给出了原因（[Are We Ready For Learned Cardinality Estimation?, VLDB 2021](https://www.vldb.org/pvldb/vol14/p1640-wang.pdf)；Learned Cardinality Estimation: An In-depth Study, SIGMOD 2022）：

1. **更新是死穴**：数据一变模型即陈旧，重训练分钟到小时级，而 ANALYZE 采样是秒级；增量更新（学术上的 DDUp 等）尚不成熟；
2. **尾部行为不可控**：平均 Q-error 漂亮，但个别查询上会产生传统方法不会犯的离谱错误，且无法解释、无法预测在哪儿犯——数据库运维恰恰只在乎尾部；
3. 推理延迟（毫秒级 GPU/大模型）挤占 OLTP 规划预算；
4. VLDB 2021 那篇的结论很克制：在动态环境下，学习式方法**并不总是**优于精心调优的传统方法（直方图+采样），而工程成本高一个量级。

落地形态因此收缩为两类务实变体：**query-driven 反馈修正**（不学联合分布，只记住"这个谓词模式上次估错了多少"——微软 SQL Server 的 CE feedback、Postgres Pro 的 **AQO 扩展**（用 hook 记录 plan 节点真实基数、按查询模式做机器学习修正，正是 PG hook 体系承载此类实验的例证）），以及云厂商在 workload 分析离线侧（索引推荐、物化视图推荐）使用学习模型——那里没有毫秒预算与尾部风险。**格局判断：learned CE 攻击的确实是 §1.4 认定的核心难题，方向无争议；卡住的是数据库工程的三条硬约束——更新代价、尾部可控、可解释。**

### 4.4 为什么 PG 没做自适应重优化

对照 §2.5，PG 缺席自适应流派的原因是结构性的，不是没想到（邮件列表多次讨论过）：

1. **执行器是纯拉动流水线**（03/04 篇）：Volcano 式 one-tuple-at-a-time，除 Sort/Hash/Materialize 外没有天然物化点。Spark AQE 的立足点是 shuffle 边界——每个 stage 结束数据落盘，天然的"观测+重规划"栅栏；PG 计划树中间没有这种栅栏，插入观测点=强制物化=对估计正确的查询是纯损耗；
2. **计划与执行状态深度耦合**：PlannedStmt 展开成 PlanState 树后，运行中途换计划意味着迁移扫描位置、hash 表、外层已消费的行——"部分结果已返回给客户端"时甚至语义上不可行（Oracle adaptive plan 只敢在第一批行返回前二选一，就是这个原因）；
3. **哲学一致性**：与 §3.3 同源——核心社区的回应始终是"把估计修对"（扩展统计、更好的 selfuncs）而不是"估错了再补救"。自适应执行在他们看来同样是"把优化器 bug 藏起来"的设施；
4. 现有的微自适应仅在算子内：hash join 运行时增加 batch 数应对低估（不改计划形状）、Memoize 的缓存命中自调节、（本仓库）partial path 并行度相关的运行期调整。**计划形状一经 create_plan 固化，绝不再变**——这在五大流派里让 PG 成为最"计划即契约"的系统，讽刺的是它同时又拒绝提供契约的稳定性设施（§3.3），两头都占了"相信优化器"的哲学极端。pg_plan_advice 的出现标志着这个极端立场松动了一半。

## 6. trick 与 bug 考古（9 条，附证据）

### 6.1 trick：add_path 的模糊支配比较与 1% 门槛

支配剪枝本身 03 篇 §5 已走读，这里提炼设计空间视角的 trick：`compare_path_costs_fuzzily` 用 `STD_FUZZ_FACTOR = 1.01`（`src/backend/optimizer/util/pathnode.c:43-47`）把"代价差 1% 以内"视为相等。**目的不是精度而是剪枝力度**：若严格比较，两条代价差 0.001% 的路径会因"互不支配"双双存活，路径列表随 join 层级膨胀；模糊化让"几乎一样好"的路径按后到先杀，pathlist 稳定在个位数。深层含义：**优化器自己承认代价模型没有 1% 的分辨率**（§1.3），在噪声分辨率以下强行排序毫无意义——这是把统计谦逊编码进比较函数的罕见例子。

### 6.2 bug 考古：模糊比较器 + qsort = 未定义行为（`ae0c8d09fb`，2006）

6.1 的 trick 有一个致命的使用禁区，PG 自己踩过：`choose_bitmap_and()` 曾用"1% 模糊比较"做 qsort 的比较函数，且实现还写错了——注释说 1% 相对差，代码写成 0.01 **绝对**差，导致选择率 0.001 和 0.000000000001 被判相等。修复 commit（`ae0c8d09fb`）的分析直指要害：**任何模糊比较都不满足传递性**（a≈b、b≈c 推不出 a≈c），把非传递比较器交给 qsort，结果是完全未定义的排序输出。最终修法是放弃模糊、改精确比较。教训成对记忆：模糊比较用于**支配裁决**（两两独立判断）是 trick，用于**排序**是 bug。

### 6.3 trick：PathKeys 规范化——指针相等替代结构比较

`make_canonical_pathkey`（`src/backend/optimizer/path/pathkeys.c:47-100`）维护 `root->canon_pathkeys` 全局池：先追平等价类合并链（`while (eclass->ec_merged) eclass = eclass->ec_merged`），再查池复用，不存在才新建。效果是**同一（等价类,opfamily,方向,nulls 位置）四元组全查询唯一实例**，于是"两条路径排序是否兼容"这种搜索热路径上亿万次调用的判断退化为**指针比较**（pathkeys.c:156 注释原文："pointer comparison is enough to decide whether canonical ECs are the same"）。这是编译器 interning/hash-consing 技术在优化器里的教科书应用。函数内还有个 GEQO 伏笔：规范化对象必须分配在 `planner_cxt` 而非 GEQO 的临时 context（pathkeys.c:83-86 注释），否则 GEQO 轮次结束释放临时 context 后池里全是悬垂指针——§6.8 那类 bug 的预防性设计。

### 6.4 trick：relids 位图集合——DP 备忘录的 O(1) 钥匙

DP 的子问题身份是"基表集合"，PG 把它编码为 `Bitmapset`（relids）：子问题查重是位图哈希（`find_join_rel`，`src/backend/optimizer/util/relnode.c:656-680`）；连接合法性判断（两个子问题不相交、满足外连接顺序约束）全部是 `bms_overlap`/`bms_is_subset` 位运算；参数化路径的"依赖哪些外层表"（`param_source_rels`）同样是位图。两个工程细节见功力：哈希表**懒建**——joinrel 少于 32 个时线性扫链表更快（relnode.c:663-664：`if (!root->join_rel_hash && list_length(root->join_rel_list) > 32)`）；等价性依赖一个不变式——**relids 相同 ⇒ 逻辑等价**，这恰是 §2.2 所说"穷人版 memo"的成立条件，也解释了为什么 SJE/join removal 这类"改变 relids 集合"的变换 bug 率奇高（§6.7）：它们动的是整个 DP 的主键。

### 6.5 bug 模式考古：相关列低估的完整病理（§4.1 的仓库端证据）

三个仓库内锚点串起这条最著名的翻车链：① 独立性相乘的实现就在 `clauselist_selectivity` 系（03 篇 §8），无统计时相关谓词逐个乘小；② 兜底常数 `DEFAULT_NUM_DISTINCT = 200`（`src/include/utils/selfuncs.h:52`）——对函数表达式等"估不了"的场景，一律假设 200 个 distinct 值，真实值差几个数量级是常态；③ `clamp_row_est`（`src/backend/optimizer/path/costsize.c:211-215`）把行数下限钳到 1——本意是防 0 行导致除零与代价失真，副作用是**低估会饱和在 1**："估 1 行"是 EXPLAIN 里最危险的数字，它驱动上层选 nested loop + 参数化内侧索引扫，真实来了 10 万行就是 10 万次内侧重扫（03 篇 §9.1 的 nestloop 惨案）。社区为 ② ③ 打的补丁是扩展统计（`7b504eb282` 起的整个 `src/backend/statistics/`），但 §4.2 的边界（单表、手工创建）决定了这条病理链在跨表场景至今活着。

### 6.6 bug 考古：generic plan 悬崖与"五次规则"的由来

**现象**：PREPARE 语句前 5 次执行都快，第 6 次突然慢几个数量级——运维圈的著名"悬崖"。**根因**在 `choose_custom_plan`（`src/backend/utils/cache/plancache.c:1170-1225`）：前 5 次强制 custom plan（`if (plansource->num_custom_plans < 5) return true;`——注释自己承认 5 是 "arbitrary"），之后若 generic plan 的估计代价低于 custom plan 的**平均**代价就永久切换 generic。悬崖成因：generic plan 对参数无知，选择率按"平均参数"估（如 `var_eq_non_const` 用 1/ndistinct）；若数据倾斜——99% 的参数值命中小结果、1% 命中大结果——generic plan 的估计代价反而可能**低于**custom 平均（它看不见倾斜），于是第 6 次起所有参数共用一个对热值灾难性的计划。**史**：`e6faf910d7`（2011，PG 9.2）重设计 plancache 引入 custom/generic 竞争与五次规则；`2aac3399ae`（2013）把规划成本计入 custom 侧成本（否则简单查询永远选 custom，缓存失去意义）——但这反而加重了向 generic 倾斜的力；投诉积累到 PG 12 才给出逃生门 `plan_cache_mode`（plancache.c:1191-1193 可见 FORCE_GENERIC/FORCE_CUSTOM 分支）。设计空间点评：这是 PG 体内唯一一个"参数嗅探"式机制，它同时展示了缓存计划的收益与"以 5 次样本的平均值做永久决策"的粗糙——SQL Server 对同一问题的回答是 parameter sensitive plan optimization（按参数值分桶缓存多个计划），PG 的回答至今是那个 arbitrary 的 5。

### 6.7 bug 考古：Self-Join Elimination——一个变换的两次生命

优化器错误计划 bug 的完整标本。**第一次生命**：SJE 在 PG 17 周期合入（`d3d55ce5713`），随后连环爆雷，仅 revert 记录里列出的后续修复就有 16 个 commit；典型如 `5ef34a8fc3`（2024-01）："SJE mistakenly omits qual clauses"——消除自连接时把两份 RestrictInfo 判为重复而只保留一份，**过滤条件凭空丢失，直接产出错误查询结果**（不是慢，是错）。2024-05-06，发布前夕，Alexander Korotkov 整体 revert（`d1d286d83c`），revert 信息的坦诚值得全文品味："We are quite late in the release cycle and new bugs continue to appear. Even though we have fixes for all known bugs, there is a risk of throwing many bugs to end users."——**已知 bug 全修完仍然撤**，因为持续冒出的 bug 说明理解不完备。**第二次生命**：一年后以更保守的形态重新合入（`fc069a3a63`，2025-02，PG 18，先禁 PHV 参与：`1aa7cf9eb8`），且合入后修复仍在滴答（`c66d6d19eb`、`f0ac6d494b`、`07b7a964d3`、`cfcd571116`……直到 2026 年）。**为什么这类变换如此危险**：SJE 修改 relids 集合本身——§6.4 说过 relids 是整个优化器的主键，删除一个 relid 意味着追杀全部残留引用（Var、PHV、RestrictInfo、EquivalenceClass、RowMark…），漏一处就是错误结果或规划崩溃。这个标本还给 §2 矩阵提供了注脚：同样的变换在 Cascades 里是一条局部规则（memo 天然管理等价性），在 PG 的"前置改写"架构里却要手工维护全局一致性——扩展性维度的差距，代价就是这两次生命。

### 6.8 trick 考古：GEQO 的随机种子——从"不可复现"到确定性抽签

GEQO 早年真的是抽签：同一查询两次 EXPLAIN 可能给出不同计划，"计划不可复现"是当年 GEQO 投诉的主项。更早历史上它曾经重置过**全局** `random()` 序列来求可复现，结果污染整个后端的随机数消费者，于是被拿掉——可复现性随之丢失。最终修法（`f5bc74192d`，2009）：给每次 `geqo()` 执行建立**私有**随机数状态，用 GUC `geqo_seed`（默认 0.0，`src/backend/utils/misc/guc_parameters.dat:1169`）初始化——同一 seed + 同一输入 ⇒ 同一计划，想探索不同随机搜索的人自己调 seed。配套还有 `400e2c9344`（重写 `gimme_tree` 使任意 tour 都能落成合法连接序列——此前随机 tour 撞上外连接顺序约束时靠随机重试，大 join 问题上会耗尽耐心直接 `failed to make a valid plan` 报错）。设计空间教训浓缩成一句：**随机化算法进入生产系统的入场券是"受控的随机"**——私有 RNG 状态 + 显式 seed，这一课如今是所有含随机成分的系统组件（含 learned CE）的标配需求。

### 6.9 bug 考古：OR 子句索引化压过外连接语义（`dcc2334736`，2008）

早于 SJE 十六年的同类型标本，展示"聪明的变换 × 语义边界 = 错误结果"的永恒模式：`create_or_index_quals()` 会从 OR 子句中提取可索引的限制条件下推——但当 OR 子句引用了某个下层外连接的可空侧时，下推提取会让本应在外连接**之后**评估的条件提前生效，可空侧被提前过滤，**错误结果**（真实用户 bug 报告驱动修复）。修法是把 `outerjoin_delayed` 的判定收紧到"引用任何下层外连接可空侧即算 delayed"。它与 6.7 同构：优化器的每条激进变换都必须显式证明与外连接/NULL 语义可交换，而这类证明义务在代码里没有类型系统兜底，只能靠测试与 bug 报告逼近——这也是 §5.3 里 NOT IN → antijoin 拖了二十年、commit 标题要强调 "when safe" 的原因。

## 7. 自测（7 题带详解）

**Q1. Lohman 的"优化器悖论"具体指什么？它如何解释"更彻底的搜索可能产出更差的计划"？**

<details><summary>详解</summary>

悖论：几十年投入集中于搜索算法，但搜索只是"对一堆错误数字做精确极值运算"——基数估计误差（10^2–10^8，方向随机）远大于代价模型误差（<10×，系统性同向），而搜索的完备性只保证找到"估计代价"最小的计划。搜索越彻底、空间越大（如枚举 bushy 树、激进变换），越可能撞上某个"真实很差但因估计误差碰巧显得极便宜"的极端计划——误差是乘性放大的（§1.2），复杂计划形态给了误差更多放大层级。保守的搜索（左深、启发式）反而因计划形态受限而少踩极端。这解释了 §2.6 矩阵里启发式流派"低完备性、高鲁棒性"的组合为何在 OLTP 上成立。
</details>

**Q2. PG 的 `join_rel_hash` 与 Cascades 的 memo，等价性 key 各是什么？这个差异如何决定两者能表达的优化问题范围？**

<details><summary>详解</summary>

PG：key 是 relids 位图（基表集合），成立前提是"relids 相同 ⇒ 逻辑等价"——这要求除连接顺序外的一切逻辑决策已在预处理阶段定死。Cascades：key 是任意逻辑表达式的等价类（group），聚合/去重/投影的位置都可以是 group 内的候选。因此"聚合放 join 前还是 join 后"在 Cascades 里是一条规则，在 PG 里破坏了 relids 不变式（同一 relids 集合出现了聚合过/没聚合过两种语义不同的结果），必须像 eager aggregation（`8e11859102`）那样给每个 joinrel 平行挂 `grouped_rel`——本质是把 key 手工扩展成 (relids, 是否已部分聚合) 二元组。
</details>

**Q3. 反对 hints 的五条论点中，哪几条被 pg_plan_advice 的设计逐条化解？哪一条它故意不解决？**

<details><summary>详解</summary>

化解：①"固化优化器 bug、切断反馈"——advice 主用途是复现自己产出过的好计划，而非人工臆造，且是外置可清点的对象（可审计谁在钉计划）；②"不随数据演化"——钉住计划成为显式、可撤销的运维决策而非埋在 SQL 里的化石；⑤"侵入 SQL 文本"——GUC/query ID 注入，零代码改动。故意不解决的：它**拒绝提供 Rows() 式的估计值修正**（对照 pg_hint_plan），README 明言 bad advice 得 bad plan 是用户责任、且不强制优化器考虑因非代价原因被拒绝的计划——即它管"决策钉住"，不管"把估计改对"，与③④（升级、坏习惯）相关的风险转由用户的 advice 生命周期管理承担。
</details>

**Q4. "前 5 次 custom plan" 的 5 为什么救不了倾斜参数？给出悬崖发生的代数条件。**

<details><summary>详解</summary>

切换条件是 `generic_cost < avg_custom_cost`（plancache.c:1206-1223）。倾斜场景：99% 的参数命中小结果（custom 代价小 c），1% 命中热值（custom 代价大 C）。avg_custom ≈ 0.99c + 0.01C + 规划成本；而 generic plan 按"平均参数"估计（1/ndistinct 选择率），估出的 generic_cost 常介于 c 与均值之间——只要前 5 次抽样没让 avg_custom 低于它（热值一旦入样即被拉高，规划成本还在给 custom 侧加码），就永久切 generic；此后热值参数执行"按平均行数选出的计划"（多半是本该 hash/seqscan 却选了 nestloop+索引，或反之），执行时间跳变数个量级。根因是**用 5 次样本均值做不可逆决策 + generic 侧对参数分布无知**；逃生门 `plan_cache_mode=force_custom_plan`；SQL Server 的按参数分桶（PSP optimization）是设计空间里的对照答案。
</details>

**Q5. 为什么 6.1 的模糊比较是 trick 而 6.2 的模糊比较是 bug？一句话给出判据。**

<details><summary>详解</summary>

判据：**模糊等价不可传递，因此只能用于两两独立的裁决（支配判断），绝不能作为全序比较器交给排序算法**。add_path 中每次 fuzzily 比较只决定"这两条路径谁杀谁"，无传递性需求；`choose_bitmap_and` 把它塞进 qsort，qsort 的正确性前提是比较器满足严格弱序（传递性是核心公理），违反即未定义行为——输出可能任意错乱。附加教训：那个实现还把 1% 相对差写成 0.01 绝对差（量纲错误），两个 bug 叠加（`ae0c8d09fb`）。
</details>

**Q6. SJE 两次合入的历史（6.7）印证了 §2 矩阵中哪一格的代价？如果 PG 是 Cascades 架构，这类 bug 的形态会如何不同？**

<details><summary>详解</summary>

印证"自底向上 DP 的扩展性：加代价无关但改变 relids 的逻辑变换难"这一格。PG 的改写是前置的破坏性图手术：删除 relid 后须手工追杀 Var/PHV/RestrictInfo/EC/RowMark 里的全部残留引用，遗漏即错误结果（`5ef34a8fc3` 丢 qual）或崩溃。Cascades 下 SJE 是一条产生新等价表达式的规则，原表达式仍留在 memo 中，等价性由框架管理——不会有"全局残留引用"这类 bug，代价转移为规则爆炸与验证规则本身的语义正确性（外连接/NULL 边界照样要证明，见 6.9）。即架构决定 bug 形态：DP+前置改写把风险集中为全局一致性维护，Cascades 把风险分散为规则语义正确性。
</details>

**Q7. Learned CE 直接攻击独立性假设，为什么截至 2026 年主流引擎默认路径仍不用它？三条硬约束各对应传统方法的什么优势？**

<details><summary>详解</summary>

① 更新代价：模型重训练分钟-小时级 vs ANALYZE 采样秒级增量——传统统计天生"可便宜刷新"；② 尾部可控性：学习模型平均 Q-error 优异但会犯传统方法不会犯的、不可预测的离谱错误，而生产只在乎尾部——直方图/MCV 的误差有可推导的界；③ 推理延迟与可解释性：OLTP 规划预算是亚毫秒，且 DBA 无法对着神经网络 debug 一个坏计划——`pg_stats` 一眼可查。因此落地收缩为 query-driven 反馈修正（SQL Server CE feedback、Postgres Pro AQO）与离线推荐场景（VLDB 2021 "Are We Ready" 的结论）。
</details>
