## 前置知识

- 04 篇 §1（Volcano 模型与 pull/push 初步对比）、§3.3（`ExecProcNodeFirst`）、
  §4（TupleTableSlot 与惰性 deform）、§6（表达式虚拟机与 computed goto）是本篇的
  直接前置。本篇引用这些机制时只给结论与行号，不再展开。
- 几个贯穿全篇的术语：
  - **每行开销（per-tuple overhead）**：一行数据流过一个算子所付出的、与"真正的
    计算"无关的固定成本——函数调用、分支、状态读写。执行模型之争本质是摊薄或
    消灭这项开销的三种手法之争。
  - **pipeline / pipeline breaker**：数据能不落地地从一个算子流向下一个算子，称为
    在同一条 pipeline 上；必须攒齐全部输入才能出第一行的算子（Sort、hash join 的
    build 侧、聚合）称为 pipeline breaker。所有执行模型都以 pipeline 为编译/调度单位。
  - **SIMD**：单指令多数据。向量化执行的内层循环（对一列连续值做同一操作）是
    SIMD 的天然形态；逐行解释执行则几乎无法利用。
  - **物化（materialization）**：把中间结果写成实际存在的行/列数据。粒度谱系：
    tuple-at-a-time（不物化，一行穿针引线）→ vector-at-a-time（物化一小批）→
    column-at-a-time（物化整列，MonetDB 老路线）→ full materialization（物化整个
    中间表）。

## 1. 问题定义：三个正交维度

### 1.1 维度一：控制流——pull 还是 push

**pull（消费者驱动）**：父算子调用子算子的 `next()` 索要下一行。PG 的
`ExecProcNode`（`src/include/executor/executor.h`，经由 `PlanState->ExecProcNode`
函数指针分发）是教科书实现。控制流住在消费者手里，于是：

- LIMIT/游标/协议层暂停（JDBC fetchSize）天然免费——不想要就不再调；
- 每行付出"叶到根 N 层间接调用 + N 次返回"的代价；
- 调用栈即数据流，gdb 一个 `bt` 看穿整条管道。

**push（生产者驱动）**：扫描算子驱动循环，把行"推"给父算子注册的回调/内联代码。
控制流住在生产者手里，于是：

- 一条 pipeline 可以折叠成一个紧循环（这是编译执行的前提，见 §2.3）；
- 提前终止需要带外信号（返回值、异常、标志位轮询）；
- DAG 形状的计划（一个结果喂多个消费者）表达起来比 pull 自然。

关键认识：**pull/push 是控制流问题，与性能的相关性常被高估**。Shaikhha、
Dashti、Koch 的论文 *Push versus Pull-Based Loop Fusion in Query Engines*（Journal
of Functional Programming, 2018）系统论证了两者在充分优化后可以互相转换，真正的
性能差异来自下面两个维度。DuckDB 用 push、HyPer 用 push、PG 和 MySQL 8 用
pull、ClickHouse 两者混用——四者性能特征的差异主要不是 pull/push 决定的。

### 1.2 维度二：数据粒度——一行还是一批

`next()`（或 push 的回调）一次移交多少数据：

- **tuple-at-a-time**：一行。每行开销 = 全部固定成本 × 行数。PG 在算子间接口上
  是纯 tuple-at-a-time：`ExecProcNode` 返回一个 `TupleTableSlot *`。
- **vector-at-a-time**：一批（典型 1000～2048 行）的**列式**小块。固定成本被摊薄
  三个数量级；内层循环是"对一个数组做一件事"，编译器可自动向量化（SIMD），
  分支可以被转换成掩码运算；批大小刻意选到能装进 L1/L2，避免 MonetDB 整列物化
  的内存带宽灾难。
- **column-at-a-time**：整列物化（MonetDB BAT 代数）。算子内是完美的紧循环，但
  中间结果大小与数据量成正比，超出 cache 后每个算子都在读写主存。X100 论文
  （§2.2）正是对自家这个模型的修正。

### 1.3 维度三：代码形态——解释还是编译

执行"1 + a > b AND c LIKE '%x'"这类表达式（以及整个算子逻辑）的代码从哪来：

- **树解释**：沿表达式树递归求值，每个节点一次间接调用。PG 9.6 以前的
  `ExecEvalExpr` 就是这样。
- **线性字节码解释**：把树"编译"成一维 opcode 数组，用 dispatch 循环解释。
  PG 10 的 `ExecInterpExpr`（04 篇 §6 已走读）。消灭了树遍历的指针追逐，
  computed goto 进一步帮助分支预测（§6.1）。
- **机器码编译（JIT/codegen）**：查询到达后现场生成机器码。又分两条子路线：
  - *表达式级*：只编译表达式与 tuple deform，算子骨架仍是解释的——PG 11 的
    LLVM JIT；
  - *计划级（data-centric）*：把整条 pipeline 编译成一个函数，中间值全程留在
    寄存器——HyPer/Umbra/Spark whole-stage codegen（§2.3）。

### 1.4 为什么每行开销是微架构问题

以 PG 执行 `SELECT sum(x) FROM t WHERE y > 10` 为例，一行的旅程（04 篇 §3、§6
已详证）：`ExecScan` 循环 → 存储层取行 → qual 表达式解释 → 投影 → 返回上层 →
聚合 transfn 调用。每一步都有间接调用（函数指针：`ExecProcNode`、opcode 分发、
`FunctionCallInvoke`）。间接调用在现代 CPU 上的成本不是"call 指令本身"，而是：

1. **I-cache 抖动**：N 个算子的代码轮流进出一级指令缓存（典型 32-64 KB），
   行与行之间同一算子的代码被反复驱逐；
2. **分支预测失败**：一个中央 dispatch 点要预测几十种去向，间接跳转目标缓冲
   （BTB）命不中就是十几个周期的流水线清空；
3. **乱序窗口浪费**：逐行逻辑的数据依赖链长，CPU 每周期能发射 4-6 条指令的
   能力大部分时间闲置；SIMD 单元完全闲置。

MonetDB/X100 论文（CIDR 2005）对 MySQL 4 执行 TPC-H Q1 的经典测量：**每条
元组每次操作平均花费数百个时钟周期，其中真正做加法/乘法的不到 10%**。这组
数字是整个向量化流派的出发点。反过来，PG 的主场 OLTP 每查询只流过几行到几
千行，每行开销总量微不足道，瓶颈在解析/规划/加锁/IO——这解释了 PG 为什么
可以二十年不动算子间接口（§3.4）。

**三个维度正交**，任何组合都存在实例：

| 系统 | 控制流 | 粒度 | 代码形态 |
|---|---|---|---|
| PostgreSQL | pull | tuple | 字节码解释 + 表达式 JIT 可选 |
| MySQL 8.0 iterator executor | pull | tuple | 解释 |
| MonetDB | 算子级物化 | column | 解释（预编译原语） |
| VectorWise / ClickHouse | pull（CH 部分 push） | vector | 解释（预编译原语）+ 少量运行期 codegen |
| DuckDB（2021 起） | **push** | vector | 解释（预编译原语） |
| HyPer / Umbra | push | tuple（寄存器中） | 全 pipeline 编译 |
| Spark SQL（whole-stage） | push（生成代码内 pull 混合） | tuple + 列式扫描 | 编译（Janino/JVM） |
| Databricks Photon | pull | vector | 解释（手写 SIMD 原语） |

## 2. 方案谱系与优劣矩阵

### 2.1 流派一：Volcano tuple-pull——组合性优先

**论文**：Goetz Graefe, *Volcano — An Extensible and Parallel Query Evaluation
System*, IEEE TKDE 1994（思想成型于 1990 年前后；并行部分见 *Encapsulation of
Parallelism in the Volcano Query Processing System*, SIGMOD 1990）。

核心贡献不是性能而是**架构**：open/next/close 统一接口把"算子组合"变成纯粹的
接口问题，任意算子可以挂在任意算子之上；并行、分区、排序属性都可以封装成
"看起来也是个算子"的东西（exchange，§4.2）。PG 的 43 个 `nodeXxx.c`
（`src/backend/executor/`，2026-07 现场清点）每个独立可读、独立可测，扩展可以
用 CustomScan 塞进自己的算子——这些都是 Volcano 接口纪律的红利。

**每行开销**：最高。一行穿过 N 层算子 = N 次间接调用；表达式再解释一遍。
**cache/分支**：最差（§1.4 的全部问题都有）。
**实现/调试复杂度**：最低。算子间无耦合；栈回溯即数据流；EXPLAIN ANALYZE 的
per-node 计时天然对齐到函数边界。
**OLTP 适配**：最好——每查询行数少时固定开销可忽略，而 pull 的可暂停性正是
游标/扩展协议所需要的。
**OLAP 适配**：差。这是所有后续流派存在的理由。

### 2.2 流派二：向量化——摊薄开销 + 释放 SIMD

**论文**：Peter Boncz, Marcin Zukowski, Niels Nes, *MonetDB/X100:
Hyper-Pipelining Query Execution*, CIDR 2005。工业谱系：X100 →
VectorWise（Actian）→ 深刻影响 ClickHouse（2009 起开发，2016 开源）、
DuckDB（2019 起，CIDR 2019 有系统论文；2021-10 起转为 push-based 调度，见 §5.4）、
Databricks Photon、Meta Velox、Apache Arrow DataFusion。

三板斧：

1. **接口不变、粒度变**：仍是 Volcano 式 next()，但一次返回约 1000 行的列式
   vector——每行的解释/调用开销摊薄 1000 倍；
2. **原语（primitive）预编译**：`select_lt_int32(col, const, sel_vector)` 这类
   "类型 × 操作"组合在**构建期**编译成紧循环，运行期只做原语的粗粒度调度——
   解释开销只发生在每 vector 一次，循环体内零解释；
3. **选择向量（selection vector）**：过滤不复制数据，只维护一个"存活行下标"
   数组，后续原语按下标访问——分支被转换成数据访问模式。

**每行开销**：低（固定成本 / 1000）。**cache**：好——vector 大小刻意选在
L1/L2 量级（X100 论文的实验：过大退化为 MonetDB 整列物化，过小退化为逐行）。
**分支预测**：内层循环几乎无分支，选择率导致的 mispredict 用选择向量/掩码规避。
**SIMD**：天然（列连续 + 定长循环，编译器自动向量化或手写 intrinsics）。
**实现复杂度**：中高——原语组合爆炸（类型 × 操作 × 有无选择向量），需要代码
生成器辅助维护；所有算子都要重写成批接口。
**调试/可观测**：好——原语是普通预编译函数，profiler 直接可见。
**OLTP**：不利——点查只有 1 行也要走 vector 机制；但退化开销可控。

### 2.3 流派三：数据为中心编译——消灭解释本身

**论文**：Thomas Neumann, *Efficiently Compiling Efficient Query Plans for
Modern Hardware*, VLDB 2011（HyPer）。前史：System R 最早（1979）就是编译到
汇编的，后因维护性放弃；IBM 的 Rao 等（ICDE 2006）与 Krikellas 等（ICDE 2010）
重启了这条线，Neumann 2011 给出了成熟范式。

范式要点：

1. **以 pipeline 为编译单位**，不以算子为单位。算子边界在生成的代码里消失：
   `scan → filter → probe hash → agg` 编译成**一个**循环，一行数据从载入到进
   聚合哈希表全程留在**寄存器**里，零函数调用、零物化；
2. **produce/consume 双接口**取代 next()：代码生成期，父算子对子算子说
   produce（"你去生产"），子算子生成扫描循环后回调父算子的 consume（"我把行
   放在寄存器 r1-r3 了，你继续"）——所以数据流是 push 的，但这是**编译期**的
   控制反转，运行期只有一个裸循环；
3. **LLVM 生成**：不生成 C（编译太慢），直接产 LLVM IR，复杂逻辑（spill、
   目录访问）留在预编译的 C++ 里，生成代码只做热路径。

**每行开销**：趋近零（等价于手写 C 程序）。**cache/分支**：最优——一条
pipeline 的代码就是一个小循环，I-cache 常驻。**SIMD**：可做但不天然（逐行
寄存器风格需要额外的循环向量化 pass；Umbra 的经验是收益有限）。
**实现复杂度**：最高——每个算子都是"生成代码的代码"，改一行逻辑要读两层；
调试要 gdb 进 JIT 代码；**编译延迟**成为新的痛点：亚毫秒查询付不起几十毫秒的
LLVM 优化。HyPer 后续为此发展出**自适应执行**（André Kohn, Viktor Leis,
Thomas Neumann, *Adaptive Execution of Compiled Queries*, ICDE 2018）：先用
字节码解释器立刻开跑，后台并行编译，编好后**热切换**；Umbra 进一步做了
*Tidy Tuples and Flying Start*（Kohn/Leis/Neumann, VLDB Journal 2021）：自研
直接出 x86 机器码的轻量后端，编译快两个数量级，LLVM 只留给长查询。
**OLTP**：预编译计划（存储过程）场景极佳，ad-hoc 短查询依赖自适应分层。
**OLAP**：计算密集型（复杂表达式、多列）最优。

**向量化 vs 编译的正面对决**：Timo Kersten 等，*Everything You Always Wanted
to Know About Compiled and Vectorized Queries But Were Afraid to Ask*, VLDB
2018。结论（本篇多处引用，值得记住）：两者总体**同一量级**，各擅胜场——
计算密集（表达式多、能留寄存器）编译胜；**内存密集（hash join probe 这类
cache miss 主导）向量化胜**，因为批循环里多个独立 miss 可以并行在飞
（memory-level parallelism），而编译的逐行依赖链一次只能等一个 miss；
向量化在编译延迟、profiling、增量演化上全面占优。这篇论文实质上为
"混合派"（§2.4）与 PG 的渐进路线（§3）提供了理论背书。

### 2.4 流派四：工程混合派——Photon 与 Velox 的选择

**Photon**：Databricks, *Photon: A Fast Query Engine for Lakehouse Systems*,
SIGMOD 2022（最佳工业论文）。背景耐人寻味：Spark 已有 whole-stage codegen
（2016 年 2.0 引入，把 Neumann 范式搬到 JVM/Janino），Photon 却选择用 C++
**向量化解释**替换它。论文给出的理由：(a) 生成代码难观测、难调试，工程师
定位问题要读生成的 Java；(b) 运行期自适应（根据观测到的数据特征换原语，如
发现列无 NULL 就换免 NULL 检查版本）在解释框架里是换个函数指针，在编译框架
里要重编译；(c) 手写 SIMD 原语已能拿到编译收益的大部分。

**Velox**：Pedro Pedreira 等, *Velox: Meta's Unified Execution Engine*, VLDB
2022。定位是"执行器作为库"：一份 C++ 向量化执行引擎，供 Presto、Spark
（经由 Gluten）、PyTorch 数据加载等复用。设计上与 Photon 同派：向量化解释 +
表达式级小规模 codegen 可选 + 大量运行期自适应（dictionary 编码直通、惰性
向量、自适应过滤器重排——与 PG planner 静态排 qual 形成对照）。

混合派的共同判断：**向量化解释是更好的"默认"，编译留给局部热点**。这与 PG
的路线（解释为主 + 表达式 JIT 可选）在哲学上惊人一致，只是 PG 的"默认"还是
tuple-at-a-time（原因见 §3.4）。

### 2.5 统一优劣矩阵

| 维度 | Volcano tuple-pull（PG/MySQL8） | 向量化（X100/CH/DuckDB） | 数据为中心编译（HyPer/Umbra） | 混合（Photon/Velox） |
|---|---|---|---|---|
| 每行固定开销 | 高（N 层调用 + 解释） | 低（摊薄~1000×） | 趋近零（寄存器直通） | 低 |
| I-cache 友好度 | 差 | 好（原语循环小） | 最优（pipeline=一个循环） | 好 |
| 分支预测 | 差（中央分发点） | 好（无分支内循环+掩码） | 好（分支被编译定死） | 好 |
| SIMD 利用 | 无 | 天然 | 需额外努力 | 天然（常手写 intrinsics） |
| 内存密集负载（join probe） | 差 | **最优**（MLP 在飞） | 中（逐行等 miss） | 最优 |
| 计算密集负载（复杂表达式） | 差 | 好 | **最优** | 好 |
| 首行延迟/编译延迟 | 零 | 零 | 有（需自适应分层缓解） | 零~小 |
| 实现复杂度 | 低 | 中高（原语组合爆炸） | 最高（元编程+两层调试） | 中高 |
| 可观测/可调试 | 最好（栈即管道） | 好 | 差（profile 进生成代码） | 好 |
| 运行期自适应能力 | 弱（但见 §3.3 特化） | 强（换原语=换指针） | 弱（要重编译） | 强 |
| OLTP 点查 | **最优** | 可接受 | 好（配合计划缓存/分层） | 可接受 |
| OLAP 扫描聚合 | 差 | 最优 | 最优 | 最优 |
| 对存储格式的要求 | 无（行存即可） | 希望列式/至少列式解耦 | 无强求 | 列式 |

两点提醒，防止把矩阵读成"新的碾压旧的"：

1. 矩阵里 Volcano 独占两行"最好"——可观测性与 OLTP——恰是 PG 用户基数最大
   的两个需求；
2. Kersten 2018 的实测反复表明向量化与编译差距在几十个百分点内互有胜负，而
   它们对 Volcano 逐行解释的优势是**数量级**的。所以真正的分水岭在"是否摆脱
   逐行解释"，而不在后两派之间——这正是 PG 用表达式虚拟机 + JIT + 存储层批量
   化去追赶的那个数量级（§3）。

## 3. PG 的渐进路线复盘："不动接口"的三层改造

PG 没有做任何一个流派的整体迁移，而是保持 `ExecProcNode` 逐行接口不变，在
三个层面分别收回开销。以下每一项都在本仓库核实。

### 3.1 第一层：表达式虚拟机（PG 10，2017）

commit `b8d7f053c5`（2017-03-14，Andres Freund，*Faster expression evaluation
and targetlist projection*）把表达式从"树递归解释"改为"线性 opcode +
computed-goto 分发"，机制细节 04 篇 §6 已走读。站在设计空间的角度补三点：

- 这是从 §1.3 的"树解释"迁到"字节码解释"，**收益全部来自消灭树遍历与改善
  分发**，与向量化/编译无关；opcode 至今已增至 140 余个（`EEOP_LAST` 在
  `src/include/executor/execExpr.h:296`）；
- 它同时是 JIT 的**地基**：线性 opcode 序列既可以解释（`ExecInterpExpr`），
  也可以逐 opcode 翻译成 LLVM IR（`src/backend/jit/llvm/llvmjit_expr.c`）——
  一份中间表示两种后端，这与 Umbra"字节码解释器 + 编译后端热切换"的分层
  思想同构，只是 PG 的切换粒度是整个查询而非 pipeline；
- 对极简表达式（单个 Var、单个 Const），连字节码循环都省掉，直接换成专用
  C 函数——`ExecJustInnerVar` 等十余个 fast path
  （`src/backend/executor/execExprInterp.c:159-169`）。这是"解释器分层"的
  微型版。

### 3.2 第二层：JIT（PG 11，2018）——得失复盘

**做了什么**：基于 LLVM，编译对象是**表达式求值**与 **tuple deforming**
（`src/backend/jit/llvm/llvmjit_expr.c`、`llvmjit_deform.c`），可选把内置
函数从预制 bitcode 内联进来再整体优化。**不是** HyPer 式计划级编译——算子
骨架、`ExecProcNode` 分发、slot 传递全部原样保留。

**时间线**（本仓库 git log 核实）：

- PG 11 开发期默认打开，发布前夕 commit `8f32bacc00`（2018-09-15）在 v11 分支
  改回默认关闭，HEAD（即将成为 PG 12）保持打开——当时社区已经在犹豫；
- PG 12（2019）起 `jit = on` 随默认发布；
- **commit `7f8c88c2b8`（2026-04-02，作者 Jelte Fennema-Nio，提交者 Tomas
  Vondra，进入 PG 19）：`jit: Change the default to off.`**

后一个 commit 的理由值得全文细读（本仓库 `git show 7f8c88c2b8`），它是一份
罕见的官方"得失结算书"，要点：

1. JIT 对大分析查询有效，但对本来很快的查询可能造成严重回退——编译 +
   优化的成本完全压倒收益；
2. 理想解法是改进代价模型，但"决定是否启用 JIT 的部分相当简单"，且**没有
   任何对 LLVM 编译成本的可靠估计**——`jit_above_cost` 拿查询的
   *执行*代价估计去猜*编译*是否划算，两者本无因果；
3. 于是出现"性能悬崖"：几条无关的 INSERT 让表统计变化、查询代价估计跨过
   `jit_above_cost` 阈值，执行时间突然多出几百毫秒的编译；
4. "大多数大规模部署早已默认关闭 JIT，**包括所有 hyperscaler**"——默认值
   只是向既成事实投降。

外部佐证（WebSearch，检索 2026-07）：提案线程由 Jelte Fennema-Nio 于
2026-01-30 发起，两周讨论后被接受（[pgsql-hackers 线程](https://www.mail-archive.com/pgsql-committers@lists.postgresql.org/msg45372.html)、
[Medium: PostgreSQL 19 Disables JIT by Default](https://medium.com/@lexkrstn/postgresql-19-disables-jit-by-default-70c7ae5c748a)）；
此前 pgsql-hackers 已有多轮同题讨论（如 Laurenz Albe 参与的
"[Change default of jit to off](https://www.postgresql.org/message-id/173b59adb0c7b70dc63ac6a63b0008c50ed6d822.camel@cybertec.at)" 线程）。

**设计空间视角的教训**：PG 的 JIT 困境不是 LLVM 的锅，而是**分层缺失**——
HyPer/Umbra 用"解释器先跑 + 后台编译 + 热切换"（§2.3）把编译成本从关键路径
上摘掉，PG 的 JIT 是同步的、全有全无的、按查询粒度的，于是只能靠一个先天
失准的代价阈值来赌。JIT 本身在正确场景下收益是真实的（TPC-H 大查询 20-50%
的公开测试很多），默认关闭意味着它退化为"专家显式开启的 OLAP 加速器"。

### 3.3 第三层：批量化点状改造——"渐进式向量化"

PG 没有改 `ExecProcNode` 的逐行接口，而是把批量化**下沉到接口以下**的热点：
存储层按页批处理、执行器入口按形态特化、deform 循环按列布局特化。逐项核实：

**(a) 页级批量可见性判断（PG 19，2026）**。commit `0b96e734c5`（2026-01-12，
Andres Freund，*heapam: Add batch mode mvcc check and use it in page mode*）：
顺序扫描本来就是"一次锁页、收集全页可见元组下标到 `rs_vistuples[]`"
（page-at-a-time 模式），现在把逐元组的 `HeapTupleSatisfiesMVCC` 调用改为一次
`HeapTupleSatisfiesMVCCBatch`（`src/backend/access/heap/heapam_visibility.c:1690`，
调用点 `heapam.c:589`）批量判定整页。注释里有一个微妙的 ABI 细节：批量
状态打包进 `BatchMVCCState`，是因为参数太多会导致 x86-64 上栈传参
（`src/include/access/heapam.h:494-505`）——批量化连参数传递的寄存器分配都
算计到了。这是教科书级的"接口以下向量化"：执行器仍然逐行拿元组，但 MVCC
判定的分支、快照访问、hint-bit 检查已经在一个紧循环里摊薄。

**(b) SeqScan 运行期特化（PG 18，2025）**。commit `fb9f955025`（2025-01-21，
提交者 Amit Langote，作者 David Rowley 等，*Refactor ExecScan() to allow
inlining of its core logic*）：按"有无 qual × 有无投影"预选四个变体
`ExecSeqScan / WithQual / WithProject / WithQualProject`（外加 EPQ 变体），
`ExecInitSeqScan` 在初始化时一次性选定函数指针
（`src/backend/executor/nodeSeqscan.c:277-290`）。变体内核心逻辑全内联，
把"每行判断一次要不要过滤/投影"的分支提升到了初始化期。后续
`b227b0bb4e`（2025-08-11，用 `pg_assume()` 压缩变体代码尺寸）、
`83a53572a6`（2026-01-26，强制内联 `SeqNext`）继续打磨。这是**运行期特化**
（partial evaluation）——编译执行的思想，用 C 的常量传播 + 函数指针实现，
零编译延迟。

**(c) tuple deform 重写（PG 19，2026）**。commit `c456e39113`（2026-03-16，
David Rowley，*Optimize tuple deformation*）：`attcacheoff` 从"deform 时顺手
填"改为 `TupleDescFinalize()` **预计算**，TupleDesc 记录"最后一个有固定偏移
的列"，deform 拆成两个循环——固定偏移列用无分支的专用循环直取，之后才进
逐列判 NULL/变长的慢循环；同时把 NULL 位图处理改为按字处理。配合更小的
`CompactAttribute` 结构（PG 18 引入）降低 D-cache 压力。04 篇 §4.4 走读的
惰性 deform 解决"少取列"，这一波解决"取列本身的循环质量"——同样没动
`slot_getattr` 接口。

**(d) 早已存在的批量化**：bitmap heap scan 按页取元组集合；
`heap_prepare_pagescan` 的 page-at-a-time 本身就是 1990 年代末的批量化遗产；
COPY 的多行插入缓冲；`tuplesort` 的批量归并等。

**路线总结**：PG 的"渐进式向量化" = 接口逐行不变 + 热点循环批量化/特化。
收益上限受限于接口（算子间每行一次间接调用的地板还在），但每一步都零风险、
可独立回退、不破坏 43 个算子和无数扩展的既有契约。对照 §2.5 矩阵：PG 选择
在"实现复杂度/兼容性"列拿满分，在"每行开销"列拿部分分。

### 3.4 为什么 PG 不整体转向量化/编译——约束清单

1. **Table AM 接口是逐行的**：`table_scan_getnextslot` 一次一个 slot；改成批
   接口意味着所有存储扩展跟着改（PG 19 已为此埋线，见 §5.2 的讨论；社区确有
   `table_scan_getnextslot_tidrange` 之外增加批量 API 的长期讨论）；
2. **表达式与函数生态**：几千个内置函数 + 扩展函数全部是 `Datum f(FCINFO)`
   逐值签名，向量化要求"类型 × 操作"原语重写，PG 的可扩展类型系统让原语
   集合不可枚举——这是 PG 与 ClickHouse（封闭类型系统）最本质的差别；
3. **OLTP 主场 + 混合负载**：默认配置必须对点查无回退，向量化的收益人群
   （纯 OLAP）在 PG 生态里本就有 citus columnar、TimescaleDB、DuckDB
   （pg_duckdb）等出口；
4. **社区演化模型**：无法接受"两套执行器并存多年"的维护成本（对照：
   ClickHouse/DuckDB 是公司/小团队集中决策，Photon 是全新闭源引擎）。

## 4. 并行执行的设计空间

单机并行是执行模型的第四个维度（严格说是控制流维度在多核上的延伸），三种
经典方案：

### 4.1 PG：scatter-gather（1990 年代 exchange 的单收敛点特例）

PG 9.6（2016）引入并行查询，形态是 **Gather/GatherMerge 单收敛点**：

- planner 生成 *partial path*（可被多 worker 各跑一份的子计划），顶上盖一个
  Gather（`src/backend/executor/nodeGather.c`）或保序的 GatherMerge；
- `ExecGather` 首次执行时启动 worker（`nodeGather.c:184-198`，
  `nworkers_launched` 处），每个 worker 执行整棵 partial 子树，结果行经
  `tqueue.c` 的 shm_mq（共享内存单向队列）发给 leader；
- 数据分片不靠重分布，而靠**扫描层的动态自助**：并行 SeqScan 的页分配器让
  worker 逐段领块（`ParallelBlockTableScanDesc`），天然负载均衡；
- worker 间**不存在任何横向数据通道**：只有 worker→leader 一条路。

关键里程碑（git log 核实）：Parallel Hash `1804284042`（2017-12-20，PG 11，
build 侧多 worker 共建一张共享哈希表）、Parallel Append `ab72716778`
（2017-12-05，PG 11）。

### 4.2 exchange 算子：Volcano/Greenplum 路线

Graefe 1990 的 exchange 算子把并行完全封装成"又一个算子"：计划树插入
exchange 节点，节点内做进程/线程边界 + 数据重分布（按 hash 重分区、广播、
汇聚），上下游算子对并行无感。工业化的极致是 Greenplum/MPP 系：Motion 节点
分 **Redistribute Motion**（按 join 键 hash 重分布）、**Broadcast Motion**、
**Gather Motion** 三种，计划被 Motion 切成若干 slice，每个 slice 在每个
segment 上一个进程。

对照 PG：**PG 只有 Gather Motion 这一种**。后果：

- **join**：Greenplum 可以把两张大表都按 join 键重分布后做本地 join；PG 的
  并行 join 要求内表要么每 worker 全量重建（PG 11 前）、要么共建一张共享表
  （Parallel Hash）——probe 侧永远不能按键分片，哈希表必须全量可见。内表
  远大于内存时，重分布方案每 worker 只处理 1/N 的数据，PG 方案的批次数由
  全量数据决定；
- **聚合**：PG 必须走 Partial Aggregate（worker 各自算部分聚合）→ Gather →
  Finalize Aggregate（leader 单线程合并）。当 group 数极大时，finalize 阶段
  的 leader 成为串行瓶颈；重分布方案可以按 group 键分片，各 worker 直接产出
  最终组，无 finalize 瓶颈。`string_agg`/`array_agg` 这类无 combine 函数的
  聚合在 PG 里直接丧失并行资格；
- **DISTINCT/窗口函数**同理受限。

PG 不做 repartition 的原因是诚实的工程账：重分布需要 worker 间 N×N 的队列
矩阵、流控、死锁避免（Greenplum 为 Motion 死锁专门有 interconnect 层），
且 PG 的并行 worker 是**按查询临时 fork 的进程**而非常驻线程池，搭建横向
通道的固定成本更高。

### 4.3 morsel-driven：HyPer 路线

**论文**：Viktor Leis, Peter Boncz, Alfons Kemper, Thomas Neumann,
*Morsel-Driven Parallelism: A NUMA-Aware Query Evaluation Framework for the
Many-Core Age*, SIGMOD 2014。

与前两者的根本区别：**并行度不进计划**。计划只是 pipeline 的集合；运行期把
输入切成 morsel（~10 万行的小段），固定线程池（每核一线程）从全局队列领
morsel 执行当前 pipeline，做完领下一块：

- **弹性**：并行度逐 morsel 可变，查询中途可增减线程（PG 的 worker 数在
  ExecutorStart 时定死，跑到一半既不能加也不能减）;
- **NUMA 亲和**：morsel 记住自己在哪个 NUMA 节点，优先由本节点线程处理；
- **抗倾斜**：天然 work stealing，无 straggler 问题（PG 的并行扫描按块自助
  已部分做到这点，但 hash build/probe 粒度的倾斜仍会让个别 worker 拖尾）；
- 依赖共享数据结构（无锁哈希表等）与线程模型——PG 的进程模型 + 私有内存
  传统使这条路线的移植成本极高。

DuckDB 采用了 morsel-driven（push-based pipeline 调度 + row-group 粒度任务），
是该论文最广泛部署的实现。

### 4.4 PG 选择的代价小结

| | PG scatter-gather | exchange/MPP | morsel-driven |
|---|---|---|---|
| 并行度决定时机 | plan+ExecutorStart，固定 | plan，固定 | 运行期，逐 morsel |
| 数据重分布 | 无（仅收敛） | 任意（hash/broadcast） | 共享结构内隐式 |
| 大-大 join | 受限（共享 build 全量） | 好 | 好 |
| 大基数聚合 | finalize 串行瓶颈 | 好 | 好 |
| 倾斜鲁棒性 | 扫描层好，算子层差 | 差（静态分片） | 最好 |
| NUMA | 无感知 | 无感知（跨机版无此问题） | 感知 |
| 实现复杂度/PG 兼容性 | 低/完美 | 高（interconnect） | 高（要线程池+共享结构） |

PG 的选择逻辑与 §3 完全同构：**选兼容性与增量性，放弃上限**。partial path
框架把并行塞进了既有的 path/cost 体系，一个 Gather 节点 + shm_mq 就复用了
全部既有算子——代价是上面表格里的三个"受限"。

## 5. 最新进展（截至 2026-07，仓库 + WebSearch 核实）

### 5.1 PG 18（2025-09 发布）：AIO 落地，执行器开始"面向未来的 IO"

- **read_stream 抽象（PG 17 铺垫）**：`b7b0f3f272`（*Use streaming I/O in
  sequential scans*，PG 17）与 `2b73a8cd33`（bitmap heap scan）把"算子决定
  下一个要读的块"抽象成回调（`ReadStreamBlockNumberCB`，
  `src/include/storage/read_stream.h:89`），流层负责预读距离控制与合并——
  执行器/AM 与 IO 调度解耦；
- **AIO 子系统（PG 18）**：`c325a7633f`（2025-03-18，*aio: Add
  io_method=io_uring*）等系列，`src/backend/storage/aio/` 整个目录。
  `io_method = worker`（默认，IO 工人进程代读）/`io_uring`（Linux 5.1+ 内核
  直提）/`sync`。顺序扫描、bitmap heap scan、vacuum 已走 AIO；
- 外部实测（[pganalyze, 2025](https://pganalyze.com/blog/postgres-18-async-io)、
  [PostgreSQL 18 发布公告, 2025-09](https://www.postgresql.org/about/news/postgresql-18-released-3142/)）：
  冷缓存顺序扫描 2-3×，OLTP 场景约 2%——收益精确落在"IO 延迟可被预取隐藏"
  的负载上。

**执行模型视角**：AIO + read_stream 是把 §2 矩阵之外的第四个开销来源——
**IO 等待**——从执行流水线上摘掉。pull 模型每行同步阻塞的老毛病（读到缺页
就停整条管道）由"预读距离自动伸缩的流"缓解；这也是 §5.3 异步执行器讨论的
物质基础。

  线程，2024-2026 持续），目标是让 index scan 的 heap 访问也走 read_stream；
  仓库里 2026-03/04 的 `2d3490dd99`（跨 index scan reset 保持 buffer pin）、
  `c7d09595e4` 等 heapam 改动是其铺垫；
- **执行器节点间预取**：Vondra 的 PoC（pgsql-hackers
  "[PoC: prefetching data between executor nodes (e.g. nestloop + indexscan)](https://www.postgresql.org/message-id/c660f36c-f648-47f1-ad97-a362432451e4@vondra.me)"），
  试图让 nestloop 提前把内表 index scan 的 IO 发出去——这实质是**pull 接口
  上的有限异步化**：不改逐行契约,但允许"预告下一次 pull"；
- **异步执行框架现状**：PG 14 的 `27e1f1456`（2021-03-31，*Add support for
  asynchronous execution*）只覆盖 Append 下的 postgres_fdw ForeignScan
  （`src/backend/executor/execAsync.c` 全文仅 155 行，仅处理
  `T_ForeignScanState`；`nodeAppend.c:207` 的 `async_capable` 判定）——
  五年过去仍未推广到本地算子，说明在纯 pull 契约上做异步的阻力；AIO 时代
  它可能被 read_stream 路线取代而非扩展。

### 5.4 DuckDB/Umbra 对 PG 的启示

- **DuckDB**（向量化 + push 调度 + morsel 并行）证明了：嵌入式/单机场景下，
  向量化解释执行不需要 JIT 就能打满内存带宽；其 2021 年从 pull 改 push 的
  动机（官方博客与 Mark Raasveldt 的 "DuckDB Internals" 讲座，2021-2022）
  主要是**调度与并行**（pipeline 显式化、算子内并行、async IO 挂接容易），
  而非单线程速度——印证 §1.1"pull/push 本身不是性能分水岭"；
- **Umbra**（编译 + 自适应分层 + morsel）证明了：编译执行的两大痛点（编译
  延迟、调试）可以用"极快的下层后端 + 按需升级"驯服；
- 对 PG 的现实启示不是"重写执行器"，而是三个可移植的局部思想，且 PG 正在
  逐一兑现：批量化下沉（§3.3，对应向量化内核）、运行期特化（SeqScan 变体，
  对应编译的常量折叠）、IO 与执行解耦（read_stream，对应 push 引擎的异步
  调度）。生态位分工也在成型：pg_duckdb / pg_analytics 类扩展把 OLAP 批量
  负载直接桥接给向量化引擎，PG 核心守 OLTP 与混合负载。

## 6. trick 与 bug 考古（8 条，附证据）

### 6.1 trick：computed goto / direct threading——把分支预测器当资源分配

`src/backend/executor/execExprInterp.c:91` 起：`#define EEO_USE_COMPUTED_GOTO`
（GCC/Clang 下启用），`dispatch_table`（:114）存每个 opcode 的 label 地址，
`EEO_NEXT()`（:133）展开为 `goto *((void *) op->opcode)`。机制 04 篇 §6.2 已
讲，这里补设计空间的谱系：这是 1973 年 James Bell 提出的 threaded code 技术
（*Threaded Code*, CACM 1973），CPython/Ruby/Lua/JVM 解释器全在用。关键收益
是**每个 opcode 结尾各有一个独立的间接跳转指令**，CPU 的 BTB 可以按"当前
opcode → 常见后继 opcode"学习成对转移模式；而 switch 循环只有一个中央跳转，
预测目标彼此践踏。另一细节：`reverse_dispatch_table`（:117）做 label 地址 →
opcode 的反查——因为 `ExprEvalStep.opcode` 字段被**就地替换成了 label 地址**
（省一次数组下标），JIT 与调试代码要还原 opcode 时得反查回来。一个字段两种
解读，省内存省间接层，代价是核心数据结构离开注释就不可读。

### 6.2 trick：ExecProcNodeFirst——函数指针自替换实现零成本插桩

`src/backend/executor/execProcnode.c:439`：`ExecInitNode` 统一把
`node->ExecProcNode` 设为包装函数 `ExecProcNodeFirst`（:448），首次调用时做
一次性工作（栈深检查 :457；判断是否需要 instrument），然后把指针**改写**为
真正的执行函数或 `ExecProcNodeInstr`（:465），此后每行调用零额外分支。
这是解释执行世界里的"自修改代码"：用一次间接层的自摘除，换来"EXPLAIN
ANALYZE 不拖累普通查询"。2026-04 的 `544000288e`（*Move ExecProcNodeInstr to
allow inlining*）还在继续压榨这条路径。同族 trick：§3.3b 的 SeqScan 四变体
选择，同样是"把每行分支提升到初始化期"。

### 6.3 trick：slot 惰性 deform 的偏移缓存及其 2026 重构

04 篇 §4.4 已走读 `slot_deform_heap_tuple` 的 `attcacheoff` 机制。考古点在于
这个 1990 年代的优化在 2026 年被推翻重来：`c456e39113`（2026-03-16）发现
"deform 时顺手缓存偏移"的老设计让热循环背着"写缓存 + 判缓存"的包袱，改为
`TupleDescFinalize()` 一次性预计算 + 记录"可快速 deform 的前缀列数"，热循环
分裂成快慢两段。教训：**缓存放在热循环里自增量维护，不如离线一次算清**——
同一原理的另一次胜利是 planner 的 `CompactAttribute`（PG 18）。批量化不一定
要改接口，把循环体做纯也是向量化（编译器自动 SIMD 的前提）。

### 6.4 bug 史：hash join 批次爆炸与 growEnabled 白旗

`ExecHashIncreaseNumBatches`（`src/backend/executor/nodeHash.c:1055`）在内存
超限时倍增批次数并重分当前批。但若数据在同一 hash 值上极度倾斜，重分后
"全留下"或"全搬走"（`nfreed == 0 || nfreed == ninmemory`，:1207），说明
再分多少批都切不开这坨相同键值——代码就地**永久缴械**：
`hashtable->growEnabled = false`（:1209），注释原文"just gut it out and hope
the server has enough RAM"。这是罕见的"官方摆烂"：承认在 hybrid hash join
的设计空间里，同键倾斜无解（除非换 skew-aware 算法或落盘链表）。04 篇 §8.1
推演过批次爆炸的过程；本条的考古价值在于：**每个批次翻倍点、每个缴械点都是
估行误差的下游买单现场**——对照 §2 各流派，向量化/编译执行都不解决这个问题，
它属于优化器与算法维度（T4 篇），执行模型之争与鲁棒性之争是两条正交战线。

### 6.5 bug 史：PG 13 前 hash agg 不设防的 OOM

2020-03-18 的 `1f39bce021`（Jeff Davis, *Disk-based Hash Aggregation*）之前，
hash 聚合**没有任何落盘路径**：planner 估计 group 数装得下 `work_mem` 才选
hash agg，但执行期若估错（distinct 数被低估 10 倍是家常便饭，见 T4 篇基数
估计章），哈希表照样长——直到 OOM killer 出面。这是"planner 假设 = 执行器
契约"最恶劣的案例：估计错误不降级、不报错，直接威胁整机。修复引入溢出分区
落盘（与 hash join 同构的 hybrid 方案），随后 `d6c08e29e7` 增加
`hash_mem_multiplier` 缓解"落盘太积极反而回退"的次生问题，`05c0258966` 修
了落盘路径自己的 bug #16784。完整故事线：**先有正确性（不 OOM），再调性能
（少落盘），再修新路径的 bug**——一个执行器功能的典型三段式生命周期。

### 6.6 bug 史：JIT crash 考古四连

本仓库 `git log -- src/backend/jit` 采样，按时间：

- `fe2a16d8b3`（2020-10-15）：LLVM 3.9 的 bug 使 `72559438f9`（复制函数参数
  attribute）后 JIT 编译 crash——**依赖库的版本矩阵**成为执行器稳定性的一部分；
- `0207d5fbeb`（2021-07-22）：内联访问线程局部变量（TLS）的函数会 crash——
  LLVM 内联器与 PG 进程模型的隐秘冲突；
- `53c8d6c9f1`（2024-04-10）：*Fix illegal attribute propagation in LLVM
  JIT*——错误传播函数 attribute 导致生成错误代码；
- `9044fc1d45`（2024-11-06，Thomas Munro）：*Monkey-patch LLVM code to fix ARM
  relocation bug*——为绕过 LLVM 在 ARM 上的重定位 bug，PG 源码里**复制并
  打补丁了一段 LLVM 的 C++ 代码**（`src/backend/jit/llvm/`），等上游修复。
  一个数据库发行版里躺着别人家编译器的 hotfix，是编译执行路线维护成本的
  最好写照——对照 §2.4 Photon 论文"不用 codegen"的理由清单，条条命中。

### 6.7 bug 史：abbreviated keys 因 strxfrm 不一致被禁（2015-2016）

事件线（仓库 + WebSearch 核实）：

- `4ea51cdfe8`（2015-01-19，Robert Haas/Peter Geoghegan）：PG 9.5 引入
  abbreviated keys——排序时把每个 text 的 `strxfrm()` 结果前 8/16 字节塞进
  SortTuple 的 datum 槽，比较先比这个前缀，等值才付出完整 `strcoll()`。
  排序快 40-200% 的明星特性；
- 2016-03：用户报告非 C locale 下 B-tree 索引"找不到已存在的行"
  （[Missing rows with index scan 线程](https://postgrespro.com/list/thread-id/1236493)）。
  根因：**多数版本的 glibc 的 `strxfrm()` 与 `strcoll()` 结果不一致**
  （违反 ISO C90/POSIX 对二者等价性的要求），某些 locale（如 de_DE）受影响
  ——前缀比较说 a<b、完整比较说 a>b，排序不再自洽，用它建的索引物理序错乱；
- `3df9c374e2`（2016-03-23，Robert Haas，进入 9.5.2）：*Disable abbreviated
  keys for string-sorting in non-C locales*。9.5.2 发布说明要求受影响索引
  REINDEX（[官方公告 2016-03-31](https://www.postgresql.org/about/news/1656/)、
  [wiki: Abbreviated keys glibc issue](https://wiki.postgresql.org/wiki/Abbreviated_keys_glibc_issue)）；
- 后续：ICU collation（PG 10）提供版本化、行为一致的变换，abbreviated keys
  对 ICU 与 builtin C.UTF-8（`f69319f2f1`，PG 17）重新可用；glibc 路线至今
  仍禁用。教训与 T1/T5 篇的 collation 惨案同源：**把正确性押在 libc 未版本化
  的语义上，等于把索引一致性交给发行版升级节奏**。执行器视角的教训更普适：
  一切"先比摘要、再比全值"的加速（abbreviated keys、hash、simhash）都隐含
  "摘要序与全值序一致"这条不变式，而不变式跨越了模块边界（tuplesort ↔ libc），
  没有任何一方能单独测出来。

### 6.8 trick：ExecJust* 快径与"解释器分层"的微缩版

`execExprInterp.c:159-169`：`ExecReadyInterpretedExpr` 识别"整个表达式只是
取一个 inner/outer/scan Var""只是一个 Const""赋值单列"等十余种一步模式，
直接把 `ExprState->evalfunc` 换成专用小函数，绕过整个 opcode 循环。这与
§2.3 Umbra 的分层（字节码 → Flying Start → LLVM）是同一思想在两个数量级上
的表达：**按表达式复杂度选执行形态**。PG 的三层是：ExecJust*（一步模式）→
ExecInterpExpr（通用解释）→ JIT（大查询可选）。缺的正是层间的**运行期自动
升级**——这是 §3.2 结论的另一种说法。

## 7. 自测（7 题，附详解）

**Q1. LIMIT 10 盖在一棵深树上，pull 与 push 模型各付出什么代价实现提前终止？**

<details><summary>详解</summary>

pull：Limit 节点取满 10 行后不再调用 `ExecProcNode`，控制流天然回收，代价为
零（04 篇 §8.3 走读过传播路径）。push：生产者的扫描循环不知道消费者已满足，
必须有带外机制——Spark 生成代码里散布 `shouldStop()` 轮询；DuckDB 的 sink
返回 NEED_MORE_INPUT/FINISHED 枚举让调度器撤 pipeline；异常/长跳转也是一种
实现。这不是 push 的致命伤（工程上都解决了），但说明"控制流住在谁家"决定了
哪一方的哪些功能是免费的：pull 免费得到暂停/游标/LIMIT，push 免费得到
pipeline 显式化与算子内并行调度。</details>

**Q2. 向量化和编译执行分别消灭了 Volcano 的哪部分开销？为什么 Kersten 2018
说 hash join probe 场景向量化反而更快？**

<details><summary>详解</summary>

Volcano 每行开销 = 算子间接调用 + 表达式解释 + cache/分支损耗。向量化
**摊薄**前两者（÷批大小）并用列式紧循环修复第三者；编译执行**消灭**前两者
（pipeline 熔成一个函数，数据留寄存器）。probe 场景的反转：probe 的瓶颈是
哈希桶随机访存 miss（百级周期），编译代码逐行推进，一次只暴露一个 miss，
流水线空转；向量化把"算 hash""查桶""比键"拆成三个批循环，一个循环内十几个
独立 miss 同时在飞（memory-level parallelism），总延迟被重叠。结论：计算
密集看编译，访存密集看向量化——"每行开销"之外，**miss 并行度**是第二个
隐藏维度。</details>

**Q3. PG 的 JIT 编译的是什么、不是什么？为什么 `jit_above_cost` 从原理上
就难以准确？**

<details><summary>详解</summary>

编译的是：表达式求值（opcode 序列 → LLVM IR）与 tuple deforming，可选内联
内置函数 bitcode。不是：算子逻辑、`ExecProcNode` 分发、slot 传递——PG JIT
不是 HyPer 式计划编译，算子间每行一次间接调用原封不动。`jit_above_cost` 拿
**执行代价估计**（行数与页数的函数）当开关，但要权衡的是**编译成本 vs 编译
节省**：编译成本取决于表达式数量/复杂度与 LLVM 优化级别，与执行代价没有
因果；编译节省取决于表达式实际执行次数（行数估计，本身就是最不可靠的输入，
见 T4 篇）。两头都测不准，中间的阈值只能靠猜——`7f8c88c2b8` 的 commit
message 把这一点写成了官方结论，并因"性能悬崖 + hyperscaler 都已关闭"在
PG 19 改默认为 off。</details>

**Q4. `HeapTupleSatisfiesMVCCBatch` 为什么放在 heapam 层而不是给执行器加
批量接口？这体现了 PG 批量化的什么原则？**

<details><summary>详解</summary>

顺扫的 page-at-a-time 模式本来就在"锁一次页→收集全页可见元组→逐个吐行"
（`heapam.c:589` 调用点），可见性判定天然有一个页内循环——把逐元组调用
`HeapTupleSatisfiesMVCC` 改成一次批量调用（`heapam_visibility.c:1690`），
**接口以上的世界完全无感**：`table_scan_getnextslot` 仍逐行返回。原则：批量
化下沉到"已经存在批循环"的位置，而不是自顶向下改契约。同族：SeqScan 变体
特化（分支提升到 init 期）、deform 快慢环分裂。收益上限受接口地板限制，但
每步零兼容性风险——这就是 §3.3 说的"渐进式向量化"。附带一个细节：批量参数
打包成 `BatchMVCCState` 是为避免 x86-64 栈传参（`heapam.h:494-505` 注释），
说明这类改造的收益尺度已经细到寄存器分配。</details>

**Q5. `growEnabled = false` 在什么条件下触发？为什么"放弃增批"反而是正确
决策？**

<details><summary>详解</summary>

`ExecHashIncreaseNumBatches` 重分当前批后发现 `nfreed == 0 || nfreed ==
ninmemory`（`nodeHash.c:1207`）：所有元组重分后仍落在同一批——它们 hash 值
相同（同键重复值海量）。批次数翻倍只按 hash 高位切分，同 hash 值永远切不
开；继续翻倍只会让批文件数指数增长（每个批一个临时文件+缓冲），内存反而
更紧张，而那坨数据一行都赶不走。所以正确决策是停止分批、任由该批超限，
"hope the server has enough RAM"。根治需要算法级改变（同键链表落盘、或转
排序归并），hybrid hash join 框架内无解。它与 §6.5 的 hashagg OOM 史共同展示：执行器的
很多"丑陋"是把优化器估计误差兜在运行期的疤痕。</details>

**Q6. PG 并行没有 repartition（重分布）算子，推演它如何限制大基数分组聚合，
以及 morsel-driven 为什么不需要 repartition 也能避开这个瓶颈。**

<details><summary>详解</summary>

PG 的路径：worker 各扫一片数据 → Partial HashAggregate（各自对**所有出现的
group** 维护部分状态）→ Gather（行全部汇到 leader）→ Finalize HashAggregate
（leader 单线程 combine）。当 group 基数接近行数（如按几乎唯一的键分组）时：
(a) 每个 worker 的部分聚合几乎不缩减行数，Gather 队列（`tqueue.c` 的
shm_mq）传输量 ≈ 全表；(b) leader 的 finalize 要单线程处理全部 group——
worker 越多，串行段占比越高，Amdahl 定律封顶。若有 Redistribute（Greenplum
Motion），可按 group 键 hash 重分布，各 worker 独占一部分键做**最终**聚合，
无 finalize 段。morsel-driven（HyPer）的解法不同：所有线程写**一张共享的
无锁聚合哈希表**（或先线程本地预聚合、再按哈希分区并行合并分区），合并阶段
本身也是按分区并行的——用共享内存数据结构替代了跨进程重分布。PG 做不了后者
的原因：进程模型 + 私有内存传统，共享哈希表要走 DSA（Parallel Hash 已付过
一次这个成本，`1804284042`），对聚合尚未有人完成同等工程。另注意 PG 的并行
度在 ExecutorStart 就固定（`nodeGather.c:184-198` 启动 worker），倾斜时无法
中途增援——morsel 的运行期弹性正是冲着这两点来的。</details>

**Q7. abbreviated keys 事件中，被破坏的不变式精确地说是什么？为什么这个
不变式无法靠 PG 自身的测试发现？**

<details><summary>详解</summary>

不变式：对任意 a、b，`sign(cmp(abbrev(a), abbrev(b)))` 要么等于
`sign(strcoll(a, b))`，要么为 0（此时回退完整比较）。即**摘要序必须是全值序
的粗化，不能与之矛盾**。glibc 的 `strxfrm()` 在部分 locale 下产生与
`strcoll()` 矛盾的序（违反 POSIX 对二者等价的要求），于是同一组数据在
"走快径"与"走慢径"时排出不同的顺序，B-tree 物理序不再自洽，二分查找在
"正确"的比较函数下找不到按"错误"的序插入的行。PG 测不出来的原因：不变式
横跨 PG 与 libc 的模块边界——PG 的回归测试跑在少数几个 locale/libc 版本上，
且排序结果"看起来有序"（按 strxfrm 序自洽），只有用 strcoll 语义去检验
strxfrm 序、且恰好在受影响 locale + glibc 版本组合上才能暴露。修复
（`3df9c374e2`，9.5.2）只能一刀切禁用非 C locale 的快径，直到出现**版本化、
可自证一致**的 collation 提供者（ICU、builtin C.UTF-8）才逐步恢复。普适
教训：任何"先比摘要"的优化，其正确性论证必须覆盖摘要函数的全部实现来源。
</details>
