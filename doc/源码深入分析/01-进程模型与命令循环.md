## 本章导读

一条 SQL 从客户端发出，到进入解析器之前，中间发生了什么？本章回答这个问题的前半段：

1. **postmaster 是怎么启动的**——GUC 加载、共享内存创建、监听 socket、信号处理的谨慎设计；
2. **连接是怎么变成进程的**——ServerLoop → BackendStartup → postmaster_child_launch 的 fork 模型，以及子进程崩溃后为什么要全库重启；
3. **backend 进程是怎么完成自我初始化的**——InitPostgres 的初始化序列；
4. **backend 的一生**——PostgresMain 主循环、协议报文分发、`sigsetjmp` 错误恢复（理解 PG 错误处理的钥匙）；
5. **一条 SQL 的总调度**——exec_simple_query 逐段走读；
6. **简单协议与扩展协议的对比**——对应 MySQL 的普通查询与 prepared statement 协议。

读完本章，你应当能独立回答："psql 里敲下一条 SELECT 后，服务端哪几个函数以什么顺序被调用？出错时控制流跳到了哪里？"

## 前置知识

- **Hellerstein 等人在《Architecture of a Database System》(2007) 中的进程模型分类**：DBMS 有三种经典模型——process-per-worker（每连接一个进程）、thread-per-worker（每连接一个线程）、进程/线程池。PostgreSQL 采用第一种；MySQL 采用第二种（每连接一个线程，8.0 之前长期如此，企业版另有线程池插件）。多进程的好处是**地址空间隔离**：一个连接的野指针几乎不可能直接踩坏另一个连接的私有内存；代价是连接创建昂贵、跨连接共享状态必须显式放进共享内存。
- **fork 语义**：`fork()` 之后子进程获得父进程地址空间的写时复制（COW）副本，并且**继承所有已打开的文件描述符和已建立的共享内存映射**。这是 PG 进程模型的物理基础。
- **setjmp/longjmp**：C 语言的非局部跳转。`sigsetjmp(buf, 1)` 保存当前执行上下文（含信号掩码）并返回 0；此后任意深度的调用栈里执行 `siglongjmp(buf, 1)`，控制流会"瞬移"回 `sigsetjmp` 处并返回 1。PG 用它实现了 C 语言里的"异常机制"。
- 建议先浏览 `src/backend/tcop/`、`src/backend/postmaster/` 目录结构（见指南 0.3 节）。

## 1. 启动：main() → PostmasterMain()

### 1.1 万物起点 main()

所有 PostgreSQL 服务端进程（postmaster、backend、checkpointer……）的入口都是同一个 `main()`（`src/backend/main/main.c:71`）。它做的事情极少但极关键：

```c
/* src/backend/main/main.c:113 */
MyProcPid = getpid();
MemoryContextInit();          /* 建立 TopMemoryContext / ErrorContext */
(void) set_stack_base();      /* 栈深检查的基准点 */
```

- `MemoryContextInit()` 建立内存上下文体系的根。注意顺序：**错误处理依赖 ErrorContext，所以内存管理必须先于一切启动**——这行之后代码才被允许使用 `elog/ereport`（见 main.c:106-114 的注释）。
- 之后设置 locale、检查不能以 root 运行（`check_root`，main.c:198），最后根据第一个命令行参数分发（main.c:205-236）：`--boot` 走 initdb 的引导模式，`--single` 走单用户模式，`--forkchild` 是 EXEC_BACKEND 模式下子进程的入口（后述），**默认落入 `PostmasterMain()`**。

### 1.2 PostmasterMain 逐段分析

`PostmasterMain()`（`src/backend/postmaster/postmaster.c:497`）长达 900 余行，但结构非常线性，可以切成六段：

**① 进程环境与内存上下文**（postmaster.c:506-538）

```c
/* src/backend/postmaster/postmaster.c:535 */
PostmasterContext = AllocSetContextCreate(TopMemoryContext,
                                          "Postmaster",
                                          ALLOCSET_DEFAULT_SIZES);
MemoryContextSwitchTo(PostmasterContext);
```

postmaster 自己的 palloc 都分配在 `PostmasterContext` 里——注释明确说这是"backends 可以回收的空间"：子进程完成初始化后会整体删掉这个上下文（见 `postgres.c:4478-4482`），fork 继承下来的这部分内存就还给了系统。这是"内存上下文 = 生命周期管理"思想的第一次亮相。

**② 信号处理设置**（postmaster.c:552-589，详见 1.5 节）

**③ GUC 加载**（postmaster.c:594-831）

顺序是：`InitializeGUCOptions()`（postmaster.c:594，把所有 GUC 设为编译期默认值）→ 解析命令行（postmaster.c:601 起，`-B`/`-c`/`-D` 等都翻译成 `SetConfigOption(..., PGC_S_ARGV)`）→ `SelectConfigFiles()`（postmaster.c:790，定位数据目录并**第一次读取 postgresql.conf**）。GUC 有严格的来源优先级（default < 配置文件 < 命令行 < ……），这里的三步正是按优先级从低到高叠加。

**④ 数据目录检查与上锁**（postmaster.c:834-920）

`checkDataDir()`（postmaster.c:834）→ `checkControlFile()`（postmaster.c:837）→ `ChangeToDataDir()`（postmaster.c:840）→ `CreateDataDirLockFile(true)`（postmaster.c:909，创建 `postmaster.pid`，防止两个 postmaster 操作同一数据目录——对应 MySQL 数据目录里的锁机制，但 PG 的实现基于 pid 文件+共享内存双重互锁）→ `LocalProcessControlFile(false)`（postmaster.c:920，读 pg_control 并校验 CRC）。

**⑤ 共享内存与信号量创建**（详见 1.3 节）

**⑥ 监听 socket、认证配置、启动后台进程**（详见 1.4 节及第 2 节）

最后一行有效代码是 `status = ServerLoop();`（postmaster.c:1415）——postmaster 的余生都在这个循环里。

### 1.3 共享内存创建调用链

这是"多进程模型决定一切"的第一个具体体现。所有跨连接共享的状态——缓冲池、锁表、ProcArray、WAL 缓冲——都必须活在一整块 System V/mmap 共享内存里。创建流程分"登记需求"和"实际分配"两个阶段：

```
PostmasterMain
 ├─ RegisterBuiltinShmemCallbacks()      postmaster.c:931  内建子系统登记回调
 ├─ process_shared_preload_libraries()   postmaster.c:936  预加载扩展（它们也可申请共享内存）
 ├─ InitializeMaxBackends()              postmaster.c:954  算出 MaxBackends（尺寸的关键输入）
 ├─ ShmemCallRequestCallbacks()          postmaster.c:976  逐个回调，累加各子系统的尺寸需求
 └─ CreateSharedMemoryAndSemaphores()    postmaster.c:1018 → src/backend/storage/ipc/ipci.c:120
     ├─ CalculateShmemSize()             ipci.c:57   汇总尺寸（"big hogs"精确估 + 100K 零头）
     ├─ PGSharedMemoryCreate()           ipci.c:135  真正向 OS 要一整块共享内存段
     ├─ InitShmemAllocator()             ipci.c:147  在段内建立 shmem 分配器（一个简单 hash 索引）
     ├─ ShmemInitRequested()             ipci.c:150  逐个初始化各子系统的共享结构
     └─ shmem_startup_hook               ipci.c:158  给 pg_stat_statements 之类的扩展留的钩子
```

两个值得注意的设计：

- **一次性分配，永不扩容**。`CalculateShmemSize()`（ipci.c:57-80）在启动时把所有需求加总、按 8KB 圆整，之后共享内存段大小终生不变。这就是为什么改 `shared_buffers` 必须重启——对比 MySQL 8.0 已支持在线调整 `innodb_buffer_pool_size`，PG 的取舍是换来极简的共享内存管理（没有在线伸缩就没有伸缩期间的并发难题）。
- **扩展的共享内存也在此刻定形**，所以 `shared_preload_libraries` 里的库必须在共享内存创建**之前**加载（postmaster.c:936 早于 1018 的顺序不可交换）。

### 1.4 监听 socket

postmaster.c:1126-1288：先 `on_proc_exit(CloseServerPorts, 0)` 注册退出清理，然后按 `listen_addresses` 逐个调 `ListenServerPort()` 建 TCP socket（postmaster.c:1154），再按 `unix_socket_directories` 建 Unix 域 socket（postmaster.c:1255）。一个都建不起来则 FATAL（postmaster.c:1286-1288）。注释（postmaster.c:895-908）解释了顺序讲究：**先拿数据目录锁、先建 TCP 后建 Unix socket**，因为 `/tmp` 里的 socket 锁文件不如数据目录锁可靠。

随后加载认证配置 `load_hba()` / `load_ident()`（postmaster.c:1343/1353）——注意 pg_hba.conf 是 postmaster 读的，而认证本身发生在子进程里（第 3 节）。最后启动后台进程：checkpointer、bgwriter（postmaster.c:1400-1403），再启动 startup 进程做崩溃恢复（postmaster.c:1408），状态机置为 `PM_STARTUP`（postmaster.c:1394）。**注意：postmaster 自己从不做恢复，恢复也是委托给子进程（startup process）完成的。**

### 1.5 信号处理：postmaster 为什么如此谨慎

看 postmaster.c:552-563 这段：

```c
/* src/backend/postmaster/postmaster.c:552 */
pqinitmask();
sigprocmask(SIG_SETMASK, &BlockSig, NULL);          /* 先屏蔽所有信号 */

pqsignal(SIGHUP,  handle_pm_reload_request_signal); /* 重读配置 */
pqsignal(SIGINT,  handle_pm_shutdown_request_signal);
pqsignal(SIGQUIT, handle_pm_shutdown_request_signal);
pqsignal(SIGTERM, handle_pm_shutdown_request_signal);
pqsignal(SIGALRM, PG_SIG_IGN);                      /* 忽略 */
pqsignal(SIGPIPE, PG_SIG_IGN);                      /* 忽略 */
pqsignal(SIGUSR1, handle_pm_pmsignal_signal);       /* 子进程给 postmaster 发消息的通道 */
pqsignal(SIGUSR2, dummy_handler);                   /* 不用，但要给子进程"占位" */
pqsignal(SIGCHLD, handle_pm_child_exit_signal);     /* 子进程退出，收尸 */
```

谨慎体现在四层：

1. **handler 里几乎什么都不做**。以 `handle_pm_child_exit_signal` 为例，它只置一个 `pending_pm_child_exit` 标志并 `SetLatch()`，真正的处理（`process_pm_child_exit()`，postmaster.c:2261）延迟到 ServerLoop 的主流程里执行。原因：信号 handler 里只能调用 async-signal-safe 函数，而 postmaster 要做的收尸逻辑涉及 malloc、ereport、状态机迁移，这些都不安全。这是教科书级的 "self-pipe/latch 模式"。
2. **先全屏蔽、装好 handler 再放开**（postmaster.c:553 与 589），杜绝"handler 还没装好信号就来了"的竞态。fork 时机同理：postmaster fork 前处于确定的信号掩码状态，子进程因此不会在设置好自己的 handler 之前收到信号（见 postgres.c:4382-4384 的注释）。
3. **SIGUSR2 明明不用，却注册了 dummy_handler**（postmaster.c:562）。postgres.c:4386-4391 的注释解释了原因：若 postmaster 对某信号设为 SIG_IGN，fork 出的子进程在来得及改 handler 之前收到该信号会被直接丢弃。注册一个空 handler 就是替子进程"占住"这个信号。
4. **postmaster 绝不能死**。它是整棵进程树的看门人，持有监听 socket 和 postmaster.pid。它自己不碰用户数据、不进临界区、代码路径极短，因此崩溃概率极低；任何可能失败的重活（恢复、检查点、连接服务）都委托给子进程。postmaster.c:1363-1381 甚至有一段专门检查"postmaster 意外变成多线程"（macOS 上 setlocale 可能引入线程）并直接 FATAL——因为**多线程进程里 fork 后调用非 async-signal-safe 函数是未定义行为**。这条检查是"进程模型纯洁性"的活化石。

> **与 MySQL 对比**：mysqld 是单进程，信号处理由专门的 signal handler 线程集中处理，其余线程屏蔽信号。PG 的 postmaster 因为要 fork，反而必须保证自己是单线程 + 最小 handler，两者对"信号是危险品"的敬畏是一致的，手段则由各自的进程/线程模型决定。

**小结**：PostmasterMain 是一条直线：内存上下文 → 信号 → GUC → 数据目录锁 → 共享内存 → 监听 socket → 后台进程 → ServerLoop。所有设计怪癖都可归因于两条铁律：共享内存一次性定形；postmaster 必须简单到不会死。

## 2. ServerLoop：从连接到进程

### 2.1 ServerLoop 主体

`ServerLoop()`（postmaster.c:1678）是一个 `WaitEventSetWait()` 驱动的事件循环：

```c
/* src/backend/postmaster/postmaster.c:1692（有删节） */
nevents = WaitEventSetWait(pm_wait_set, DetermineSleepTime(), events, ...);
for (int i = 0; i < nevents; i++)
{
    if (events[i].events & WL_LATCH_SET)
        ResetLatch(MyLatch);
    /* 信号 handler 置的标志在这里统一处理（高优先级，无条件检查） */
    if (pending_pm_shutdown_request)  process_pm_shutdown_request();
    if (pending_pm_reload_request)    process_pm_reload_request();
    if (pending_pm_child_exit)        process_pm_child_exit();
    if (pending_pm_pmsignal)          process_pm_pmsignal();

    if (events[i].events & WL_SOCKET_ACCEPT)
    {
        ClientSocket s;
        if (AcceptConnection(events[i].fd, &s) == STATUS_OK)
            BackendStartup(&s);                      /* fork！ */
        if (s.sock != PGINVALID_SOCKET)
            closesocket(s.sock);   /* 父进程立即关掉自己那份客户端 socket */
    }
}
```

要点：`accept()` 得到的客户端 socket 在 fork 之后**父进程立刻 close**（postmaster.c:1731-1735）——fd 引用计数减到只剩子进程一份，连接的生死从此与 postmaster 无关。循环尾部还有低频任务：给赖着不退出的子进程补发 SIGKILL（postmaster.c:1781-1793）、每分钟校验 postmaster.pid 没被人删掉（postmaster.c:1805-1814）、每 58 分钟 touch 一次 socket 文件防止被 /tmp 清理任务误删（postmaster.c:1821-1826）——这些注释非常值得一读，全是生产环境血泪史。

### 2.2 BackendStartup

`BackendStartup()`（postmaster.c:3576）在 fork 之前做完所有可能失败的资源分配：

1. `canAcceptConnections(B_BACKEND)`（postmaster.c:3594）：检查状态机——恢复中、关闭中都会拒绝；
2. `AssignPostmasterChildSlot()`（postmaster.c:3598）：领取子进程槽位。**槽位耗尽时并不直接拒绝，而是启动一个"dead-end child"**（postmaster.c:3610）——一个只为向客户端报错"too many clients"就退出的一次性进程。为什么不在 postmaster 里直接回错？因为回错要走协议、可能阻塞，postmaster 不做任何可能阻塞的事；
3. `postmaster_child_launch(...)`（postmaster.c:3627）：真正 fork。失败则释放槽位并尽力给客户端发一条裸协议错误（`report_fork_failure_to_client`，postmaster.c:3666）。

### 2.3 postmaster_child_launch：fork 模型的核心

`src/backend/postmaster/launch_backend.c:205`。这是 PG 17 起统一所有子进程（backend、checkpointer、walwriter……）启动方式的入口，值得完整摘录关键分支：

```c
/* src/backend/postmaster/launch_backend.c:217（有删节） */
#ifdef EXEC_BACKEND
    pid = internal_forkexec(child_type, child_slot,
                            startup_data, startup_data_len, client_sock);
    /* 子进程将从 SubPostmasterMain 进入 */
#else                           /* !EXEC_BACKEND */
    pid = fork_process();
    if (pid == 0)               /* 子进程 */
    {
        MyBackendType = child_type;
        /* 关闭从 postmaster 继承的监听 socket 等 */
        ClosePostmasterPorts(child_type == B_LOGGER);
        /* 与 postmaster"脱钩"：重置信号、latch、随机种子…… */
        InitPostmasterChild();
        /* 不需要共享内存的子进程（如 syslogger）主动解除映射 */
        if (!child_process_kinds[child_type].shmem_attach)
        {
            dsm_detach_all();
            PGSharedMemoryDetach();
        }
        MemoryContextSwitchTo(TopMemoryContext);
        MyPMChildSlot = child_slot;
        ...
        /* 按进程类型分发到各自的 Main 函数（backend → BackendMain） */
        child_process_kinds[child_type].main_fn(startup_data, startup_data_len);
        pg_unreachable();
    }
#endif
```

**子进程如何继承共享内存？答案是"什么都不用做"。** fork 语义保证子进程天生带着父进程的全部内存映射——包括 `PGSharedMemoryCreate()` 建立的那块共享内存段以及指向它的所有全局指针变量。所以代码里只有反向操作：**不需要**共享内存的子进程显式 detach（launch_backend.c:244-248）。这就是指南里那句"PG 不需要复杂的启动同步逻辑"的出处：缓冲池指针、锁表指针在 fork 的一瞬间就已经在子进程里就位了。

**EXEC_BACKEND 模式的意义**：Windows 没有 fork，只能 `CreateProcess`（等价于 fork+exec），新进程地址空间从零开始，什么都继承不到。于是 `internal_forkexec()`（launch_backend.c:285）把所有需要"继承"的状态——GUC 非默认值、共享内存段的 ID 和期望映射地址、客户端 socket——序列化写进临时参数文件（launch_backend.c:313），然后 fork+exec 出 `postgres --forkchild=<type> <参数文件>`（launch_backend.c:360-370）；新进程从 main() 的 DISPATCH_FORKCHILD 分支进入 `SubPostmasterMain`，读回参数文件、**在相同虚拟地址重新 attach 共享内存**（`AttachSharedMemoryStructs`，ipci.c:92）。在 Unix 上定义 `EXEC_BACKEND` 宏也能强制走这条路，用途是让开发者在 Linux 上调试 Windows 的进程启动路径。它的存在提醒我们：**PG 把"fork 继承"这个 Unix 特性做成了可替换的抽象**，代价是维护一份状态序列化清单（save/restore_backend_variables）。

### 2.4 HandleChildCrash：一个孩子崩溃，为什么全家重启？

postmaster 通过 SIGCHLD → `process_pm_child_exit()`（postmaster.c:2261）为子进程收尸。如果发现某个子进程是**非正常退出**（被信号杀死、或 exit code 非 0/1），就进入：

```c
/* src/backend/postmaster/postmaster.c:2818 */
static void
HandleChildCrash(int pid, int exitstatus, const char *procname)
{
    if (FatalError || Shutdown == ImmediateShutdown)
        return;                          /* 已在善后，不重复 */
    LogChildExit(LOG, procname, pid, exitstatus);
    ereport(LOG, (errmsg("terminating any other active server processes")));
    HandleFatalError(PMQUIT_FOR_CRASH, true);
}
```

`HandleFatalError()`（postmaster.c:2732）对**所有其他子进程**广播 SIGQUIT（子进程侧的 handler 是 `quickdie`，见第 4 节），置 `FatalError = true`，状态机转入 `PM_WAIT_BACKENDS` 等大家死绝。之后 `PostmasterStateMachine()`（postmaster.c:2911）走到关键一步：

```c
/* src/backend/postmaster/postmaster.c:3235（有删节） */
if (FatalError && pmState == PM_NO_CHILDREN)
{
    ereport(LOG, (errmsg("all server processes terminated; reinitializing")));
    shmem_exit(1);
    LocalProcessControlFile(true);
    ResetShmemAllocator();
    ShmemCallRequestCallbacks();
    CreateSharedMemoryAndSemaphores();   /* 共享内存推倒重建！ */
    UpdatePMState(PM_STARTUP);
    StartupPMChild = StartChildProcess(B_STARTUP);  /* 重新走崩溃恢复 */
}
```

**为什么必须如此激进？** 因为异常死亡的进程可能死在任意时刻——比如正持有缓冲区锁、改到一半 LWLock 的等待队列、写到一半共享哈希表。共享内存此刻处于**不可知状态**，任何还映射着它的进程继续跑都可能把损坏扩散到磁盘。唯一安全的做法：假定共享内存已污染，杀光所有映射者，重建共享内存，用 WAL 恢复到一致状态。这是多进程隔离模型的"自我保护条款"——隔离保护的是**私有**内存，共享内存一旦可疑就只能整体作废。运维上你看到的 `WARNING: terminating connection because of crash of another server process` 就来自这条链路（`restart_after_crash=off` 时则直接停库，postmaster.c:3224-3229）。

> **与 MySQL 对比**：mysqld 里一个线程崩溃（段错误）会直接放倒整个进程，连"只重启一部分"的选项都没有——所有线程共享全部地址空间，没有任何隔离可言。PG 的差别在于：普通的 `ereport(ERROR)`（比如约束冲突、除零）只会回滚当前事务、连接继续存活；只有**进程级异常死亡**才触发全库 restart。隔离把"灾难半径"分成了两档。

**小结**：ServerLoop 是"等事件 → 处理信号标志 → accept → fork"的极简循环。fork 让子进程零成本继承共享内存与 socket；EXEC_BACKEND 用参数文件+重新 attach 模拟这种继承以支持 Windows。子进程异常死亡时，postmaster 假定共享内存已污染，广播 SIGQUIT、重建共享内存、重走 WAL 恢复。

## 3. 新生进程的自我修养：从 BackendMain 到 InitPostgres

fork 出的 backend 子进程从 `BackendMain()`（`src/backend/tcop/backend_startup.c:76`）开始，到进入主循环前的路径是：

```
BackendMain()                        backend_startup.c:76
 ├─ BackendInitialize()              backend_startup.c:110 → 141
 │   ├─ pq_init()                    backend_startup.c:177  初始化前后端通讯
 │   └─ ProcessStartupPacket()       backend_startup.c:295 → 486  读启动包（库名、用户名、协议版本）
 ├─ InitProcess()                    backend_startup.c:116  在共享内存领取 PGPROC 槽位
 └─ PostgresMain(dbname, username)   backend_startup.c:124 → postgres.c:4364
     └─ InitPostgres(...)            postgres.c:4469 → postinit.c:722
```

注意分工：**读启动包在 InitProcess 之前**——此时进程还没占用宝贵的 PGPROC 槽位，恶意的"连上就不说话"的客户端只能耗掉一个廉价进程，卡不住共享资源（backend_startup.c 对此设有 `authentication_timeout`）。

`InitPostgres()`（`src/backend/utils/init/postinit.c:722`）是所有"要访问数据库"的进程（backend、autovacuum worker、walsender、bgworker）共用的初始化总管，源码注释特别警告"Be very careful with the order of calls"（postinit.c:718）。骨架顺序如下，每一步都有依赖上一步的理由：

| 步骤 | 位置 | 作用 |
|---|---|---|
| `InitProcessPhase2()` | postinit.c:740 | 把 PGPROC 挂进 ProcArray——**从此对其他 backend 可见**，MVCC 快照会把我算进去 |

## 4. PostgresMain：backend 的一生

`PostgresMain()`（`src/backend/tcop/postgres.c:4364`）是 backend 的"主函数"，永不返回。先做一次性设置，然后进入无限循环。

### 4.1 一次性设置（postgres.c:4393-4549）

- **安装信号 handler**（postgres.c:4397-4431）：`SIGHUP→重读配置`、`SIGINT→StatementCancelHandler`（取消当前查询，postgres.c:3155）、`SIGTERM→die`（退出，postgres.c:3114）、`SIGQUIT→quickdie`（postgres.c:4410 → 3017）。`quickdie` 就是上一节 crash restart 时 postmaster 广播的那颗子弹：它不走任何清理（共享内存不可信！），报一条错就 `_exit(2)`。
- `BaseInit()`（postgres.c:4435）、生成取消密钥（postgres.c:4451）、调用 `InitPostgres()`（postgres.c:4469）、删除继承来的 `PostmasterContext`（postgres.c:4478-4482）。
- 创建 `MessageContext`（postgres.c:4531）：**每处理完一条客户端消息就整体 reset** 的内存上下文——解析树、计划树都分配在这里，"用完即弃、永不逐个 free"，这是 PG 没有内存泄漏焦虑的秘密。

### 4.2 sigsetjmp：理解 PG 错误处理的钥匙

主循环之前的这段代码是全 PG 最重要的 30 行之一：

```c
/* src/backend/tcop/postgres.c:4573（有删节） */
if (sigsetjmp(local_sigjmp_buf, 1) != 0)
{
    /* —— 只有发生 ERROR 并 longjmp 回来时才进入这里 —— */
    error_context_stack = NULL;        /* 手工清错误上下文栈（没用 PG_TRY） */
    HOLD_INTERRUPTS();                 /* 清理期间禁止再被打断 */
    disable_all_timeouts(false);
    QueryCancelPending = false;
    DoingCommandRead = false;
    pq_comm_reset();                   /* 恢复 libpq 输出状态 */
    EmitErrorReport();                 /* 把错误发给客户端 + 写日志 */
    debug_query_string = NULL;
    AbortCurrentTransaction();         /* 回滚当前事务：释放锁、pin、内存…… */
    PortalErrorCleanup();
    if (MyReplicationSlot != NULL)
        ReplicationSlotRelease();
    jit_reset_after_error();
    MemoryContextSwitchTo(MessageContext);
    FlushErrorState();
    if (doing_extended_query_message)
        ignore_till_sync = true;       /* 扩展协议：跳过后续报文直到 Sync */
    xact_started = false;
    if (pq_is_reading_msg())
        ereport(FATAL, ... "protocol synchronization was lost" ...);
    RESUME_INTERRUPTS();
}

/* We can now handle ereport(ERROR) */
PG_exception_stack = &local_sigjmp_buf;    /* postgres.c:4687 */
```

把整条机制串起来：

1. **抛出**：代码里任何地方执行 `ereport(ERROR, ...)`，最终走到 `errfinish()`（`src/backend/utils/error/elog.c:486`）。对 `elevel == ERROR` 的情况，errfinish 做最小清理后执行 `PG_RE_THROW()`（elog.c:551）→ `pg_re_throw()`（elog.c:2200）→ **`siglongjmp(*PG_exception_stack, 1)`**（elog.c:2204）。`PG_exception_stack`（elog.c:102）是一个全局的 `sigjmp_buf *`，此刻指向 PostgresMain 的 `local_sigjmp_buf`——于是不管错误发生在解析器、优化器还是执行器的第 20 层调用栈里，控制流都一步跳回 postgres.c:4573，`sigsetjmp` 这次返回非 0，进入上面的清理块。
2. **接住之后靠什么清理？** longjmp 只恢复寄存器，不会替你 free 内存、放锁、关文件。PG 的答案是：**所有资源都登记在事务粒度的管理器里**——内存挂在事务内存上下文、锁记在锁表、缓冲区 pin 记在 ResourceOwner。所以清理块的核心就一句 `AbortCurrentTransaction()`（postgres.c:4629），它级联释放一切。这是"longjmp 式异常"能在 C 里安全工作的前提：**错误恢复不依赖栈展开，而依赖集中式资源记账**。C++ 用 RAII 在栈展开时逐个析构，PG 用 ResourceOwner 在事务中止时批量清算——殊途同归。
3. **中途可以拦截**：`PG_TRY()/PG_CATCH()/PG_END_TRY()` 宏（`src/include/utils/elog.h:388-421`）本质就是把 `PG_exception_stack` 暂时指向本地新建的 `sigjmp_buf`，形成一条**异常处理器链**；PG_CATCH 里可以处理后 `PG_RE_THROW()` 继续外抛。PostgresMain 的 sigsetjmp 是这条链的**链底**。
4. **为什么链底不用 PG_TRY 而手写 sigsetjmp？** postgres.c:4557-4563 的注释给出答案：PG_TRY 的 CATCH 块执行期间异常栈是空的，若清理代码自己又出错就没人接了；而手写方式让这个 setjmp **永远在位**——清理中再出错，就再跳回来一次（真的死循环会被 elog 的递归深度检查拦下）。
5. **两个细节**：`sigsetjmp(buf, 1)` 的第二参数 1 表示同时保存/恢复**信号掩码**——万一是从信号 handler 里 longjmp 出来的，落地时掩码能回到 UnBlockSig（postgres.c:4565-4570 注释）；`volatile` 修饰的三个局部变量（postgres.c:4369-4371）是因为 longjmp 后非 volatile 局部变量的值是未定义的——这是 C 标准的规定，也是阅读所有含 sigsetjmp 代码时的必备常识。

> **与 MySQL 对比**：MySQL 内核的错误处理主要靠返回值层层上传 + THD 上的 Diagnostics_area，每层调用都要写 `if (error) goto err;`。PG 用 longjmp 把"错误路径"从主代码里整体抽走，代价是必须发明 MemoryContext + ResourceOwner 这套记账体系。两者是同一问题的两种系统解。

### 4.3 主循环逐行走读（postgres.c:4696-5201）

```c
/* src/backend/tcop/postgres.c:4696（骨架，有删节） */
for (;;)
{
    doing_extended_query_message = false;
    MemoryContextSwitchTo(MessageContext);
    MemoryContextReset(MessageContext);        /* ① 上一条消息的所有内存一键蒸发 */
    initStringInfo(&input_message);

    if (send_ready_for_query)                  /* ② 告诉客户端"我闲了" */
    {
        /* 按事务状态上报 idle / idle in transaction，启动 idle 超时…… */
        ReadyForQuery(whereToSendOutput);      /* postgres.c:4863，附带 flush 输出 */
        send_ready_for_query = false;
    }

    DoingCommandRead = true;                   /* ③ 读消息期间允许异步中断立即生效 */
    firstchar = ReadCommand(&input_message);   /* ④ postgres.c:4878 → 494，阻塞点 */
    CHECK_FOR_INTERRUPTS();
    DoingCommandRead = false;

    if (ConfigReloadPending) { ... ProcessConfigFile(PGC_SIGHUP); }  /* ⑤ */
    if (ignore_till_sync && firstchar != EOF)
        continue;                              /* ⑥ 出错后跳过报文直到 Sync */

    switch (firstchar)                         /* ⑦ 按报文类型分发 */
    { ... }
}
```

- ①（postgres.c:4718-4719）：MessageContext 整体 reset——上一条 SQL 的解析树、计划树、字符串全部回收，无需任何显式 free。
- ②（postgres.c:4745-4865）：`ReadyForQuery` 是协议里的 'Z' 报文，携带事务状态字节（idle/in transaction/failed）。发送前顺手做低频杂务：flush 统计、处理 NOTIFY、设置 idle_in_transaction_session_timeout。psql 提示符能亮出来，就是收到了这个报文。
- ③④：`DoingCommandRead = true` 期间，SIGTERM 之类的中断可以在读 socket 的等待中被立即处理（`ProcessClientReadInterrupt`，postgres.c:515）——所以空闲连接收到 `pg_terminate_backend()` 能马上退出，而不是等下一条 SQL。`ReadCommand`（postgres.c:494）按目的地分发到 `SocketBackend`（真客户端）或 `InteractiveBackend`（单用户模式）；协议 3.0 的报文是"1 字节类型 + 4 字节长度 + 载荷"，所以 firstchar 就是消息类型字符。
- ⑦ switch 的主要分支（`PqMsg_*` 常量定义于 `src/include/libpq/protocol.h`，其值就是协议字符）：

| case | 字符 | 位置 | 动作 |
|---|---|---|---|
| `PqMsg_Query` | 'Q' | postgres.c:4930 | 简单查询 → `exec_simple_query()`（postgres.c:4946），完毕置 `send_ready_for_query = true` |
| `PqMsg_Parse` | 'P' | postgres.c:4954 | 解出语句名/SQL/参数类型 → `exec_parse_message()`（postgres.c:4977） |
| `PqMsg_Bind` | 'B' | postgres.c:4984 | → `exec_bind_message()`（postgres.c:4994） |
| `PqMsg_Execute` | 'E' | postgres.c:4999 | 解出 portal 名/行数上限 → `exec_execute_message()`（postgres.c:5013） |
| `PqMsg_Describe`/`PqMsg_Close`/`PqMsg_Flush` | 'D'/'C'/'H' | postgres.c:5100/5054/5134 | 扩展协议的辅助报文 |
| `PqMsg_Sync` | 'S' | postgres.c:5140 | 扩展协议的"段落句号"：`finish_xact_command()` 并重新发 ReadyForQuery |
| `PqMsg_Terminate` / EOF | 'X' | postgres.c:5159-5182 | 客户端正常告别/断线 → `proc_exit(0)`。注释警告：别在这加清理代码，一切清理必须注册成 on_proc_exit 回调，否则其他退出路径会漏掉 |
| default | | postgres.c:5196 | 未知报文 → FATAL（协议失步，宁死不猜） |

注意 'Q' 处理完就置 `send_ready_for_query = true`，而 'P'/'B'/'E' **不置**——扩展协议要等 'S'（Sync）才发 ReadyForQuery，这正是 pipelining 的实现基础：客户端可以连发多组 Parse/Bind/Execute 再统一 Sync。出错时 `ignore_till_sync`（postgres.c:4663-4664 置位，4925 生效）保证错误后整个 pipeline 剩余报文被丢弃，事务状态与客户端预期不失步。

**小结**：PostgresMain = 一次性初始化 + sigsetjmp 链底 + "reset 内存 → ReadyForQuery → 读报文 → switch 分发"的循环。错误处理三件套：`ereport(ERROR)` 抛、`siglongjmp` 跳、`AbortCurrentTransaction()` 清。

## 5. exec_simple_query 逐段走读

`exec_simple_query()`（`src/backend/tcop/postgres.c:1030`）是简单协议下一条 SQL 的完整生命周期，也是全书最重要的"总目录"函数。逐段拆解（行号均已核实）：

**① 登记与开事务**（postgres.c:1044-1072）

```c
debug_query_string = query_string;                 /* 1044：errcontext 与 core dump 里能看到 SQL */
pgstat_report_activity(STATE_RUNNING, query_string); /* 1046：pg_stat_activity.query 的来源 */
start_xact_command();                              /* 1064 → 2874：确保处于事务命令中 */
drop_unnamed_stmt();                               /* 1072：清掉旧的无名预编译语句 */
```

`start_xact_command()` 是 PG "一切皆事务"的入口：即使你没写 BEGIN，单条 SQL 也会被包进一个事务命令（等价于 MySQL 的 autocommit，但 PG 是在 tcop 层显式编排的）。

**② 词法+语法分析**（postgres.c:1077-1083）

```c
oldcontext = MemoryContextSwitchTo(MessageContext);   /* 1077：解析树放 MessageContext */
parsetree_list = pg_parse_query(query_string);        /* 1083 → 617：flex/bison，输出 RawStmt 链表 */
```

注意此时**只做语法分析、不查任何系统目录**——注释强调"即使在已中止的事务里也安全"（postgres.c:1080-1081）。返回的是 List：一条 "Q" 报文可以携带分号分隔的多条 SQL（psql 的 `\;` 就是这么玩的），所以下面是一个 foreach。

**③ 多语句的隐式事务块**（postgres.c:1114）

```c
use_implicit_block = (list_length(parsetree_list) > 1);
```

历史行为：一条简单查询报文里的多条语句默认作为**一个事务**执行，靠 `BeginImplicitTransactionBlock()`（postgres.c:1176）实现。

**④ 循环体：对每条原始解析树**（postgres.c:1119 起）

```c
commandTag = CreateCommandTag(parsetree->stmt);        /* 1143：如 "SELECT"，用于状态与完成报文 */

if (IsAbortedTransactionBlockState() &&                /* 1158：事务已中止？ */
    !IsTransactionExitStmt(parsetree->stmt))
    ereport(ERROR, ... "current transaction is aborted, "
            "commands ignored until end of transaction block");
```

这就是 PG 用户最熟悉的报错之一：事务内一条语句失败后，除 COMMIT/ROLLBACK 外一律拒绝。为什么放在语义分析**之前**检查？因为分析/重写/规划都要访问系统目录，而目录访问在中止事务里会失败（postgres.c:1151-1157 注释）。这与 MySQL "语句失败后事务照常继续"是行为级差异，根源是 PG 的错误恢复以事务为单位。

**⑤ 快照 → 分析/重写 → 规划**（postgres.c:1184-1233）

```c
if (analyze_requires_snapshot(parsetree))
{
    PushActiveSnapshot(GetTransactionSnapshot());       /* 1186：分析/规划需要看数据（如常量求值） */
    snapshot_set = true;
}
querytree_list = pg_analyze_and_rewrite_fixedparams(parsetree, query_string,
                                                    NULL, 0, NULL);  /* 1213 → 683 */
plantree_list = pg_plan_queries(querytree_list, query_string,
                                CURSOR_OPT_PARALLEL_OK, NULL);       /* 1216 → 988 */
if (snapshot_set)
    PopActiveSnapshot();                                /* 1229 */
```

- `pg_analyze_and_rewrite_fixedparams`：语义分析（查目录，把表名解析成 OID，输出 `Query` 树）+ 规则/视图重写（一条 Query 可能重写成 0..N 条）。
- `pg_plan_queries`：优化器入口，`Query` → `PlannedStmt`。
- 规划用的快照**用完即弃**（postgres.c:1219-1230 注释）：执行阶段会另取快照——复用规划期快照会导致"取快照早于加表锁"的可见性异常，注释里附了邮件列表链接，是"读注释即读设计史"的好例子。
- 内存策略（postgres.c:1190-1211）：多语句时每条语句用独立子上下文，跑完一条删一条；最后一条直接用 MessageContext——反正马上会整体 reset。

**⑥ Portal：执行的容器**（postgres.c:1239-1306）

```c
portal = CreatePortal("", true, true);              /* 1239：无名 portal */

                  commandTag, plantree_list, NULL); /* 1248：装入计划 */

(void) PortalRun(portal, FETCH_ALL, true,
                 receiver, receiver, &qc);          /* 1297：真正执行，直到取完所有行 */
receiver->rDestroy(receiver);
PortalDrop(portal, false);                          /* 1306 */
```

Portal 是"一次查询执行"的抽象（游标是它的显式形态）。`DestReceiver` 则把"执行"与"结果去哪"解耦——同一执行器代码，结果可以发协议报文、存 SPI、进物化元组集。简单协议里 `FETCH_ALL` 一口气跑完；下一节会看到扩展协议正是在这里改成了"可以只取 N 行"。

**⑦ 收尾**（postgres.c:1308-1378）

```c
if (lnext(parsetree_list, parsetree_item) == NULL)  /* 最后一条语句 */
{
    if (use_implicit_block)
        EndImplicitTransactionBlock();              /* 1320 */
    finish_xact_command();                          /* 1321 → 2913：提交（隐式 autocommit）*/
}
else if (IsA(parsetree->stmt, TransactionStmt))
    finish_xact_command();                          /* 1329：显式 BEGIN/COMMIT 语句 */
else
{
    CommandCounterIncrement();                      /* 1344：同事务内后续语句能看见前面的写 */
    disable_statement_timeout();                    /* 1351：statement_timeout 按语句计 */
}
EndCommand(&qc, dest, false);                       /* 1360：发 CommandComplete（如 "SELECT 5"）*/
```

`finish_xact_command` 放在 `EndCommand` **之前**是刻意的：提交若失败，客户端应收到错误而非"完成+错误"两条消息（postgres.c:1310-1317 注释）。`CommandCounterIncrement` 是 PG MVCC 的一个精妙细节：事务内可见性由 (XID, CommandId) 二元组决定，CCI 递增命令计数器，使同事务中后一条语句能看到前一条的修改。循环结束后，空语句发 `NullCommand`（postgres.c:1378），再按 `log_duration/log_min_duration_statement` 打日志（postgres.c:1383-1405）。

**小结**：exec_simple_query = 开事务 → parse → (对每条语句) 检查中止状态 → 快照 → analyze+rewrite → plan → Portal 执行 → 提交/CCI → CommandComplete。五个阶段的输入输出都是 `src/include/nodes/` 里定义的不可变树，这条流水线就是后续所有章节（解析、优化、执行）的目录页。

## 6. 简单协议 vs 扩展协议

扩展协议把 exec_simple_query 一体化的流水线**拆成了三个可独立寻址的阶段**，每个阶段对应一个报文和一个函数：

### 6.1 exec_parse_message（'P'，postgres.c:1421）

- 只做**parse + analyze + rewrite**，产物包成 `CachedPlanSource`（`CreateCachedPlan`，postgres.c:1533）；**不做 plan**——规划推迟到 Bind 时，因为好的计划可能依赖参数值。
- SQL 里可以带 `$1, $2...` 占位符，参数类型可由客户端指定或由分析器推断。
- 有名语句 `StorePreparedStatement()`（postgres.c:1597）存进会话的哈希表（≈ MySQL 的 `PREPARE stmt FROM ...`）；无名语句存 `unnamed_stmt_psrc`（postgres.c:1605），会被下一个无名 Parse 覆盖——驱动（如 JDBC）单次参数化查询用的就是无名语句，省去命名管理。
- 回 `ParseComplete`（postgres.c:1621）。

### 6.2 exec_bind_message（'B'，postgres.c:1662）

- 找到语句（`FetchPreparedStatement`/`unnamed_stmt_psrc`，postgres.c:1699/1705）→ 按声明类型解码参数值 → `CreatePortal()`（postgres.c:1792）→ **`GetCachedPlan(psrc, params, ...)`**（postgres.c:2054）→ `PortalDefineQuery` + `PortalStart`（postgres.c:2062/2088）→ 回 `BindComplete`（postgres.c:2105）。
- `GetCachedPlan` 是计划缓存的心脏：它决定返回**通用计划**（generic plan，忽略具体参数、可跨执行复用）还是**定制计划**（custom plan，按本次参数值优化）。默认策略：前 5 次用定制计划，若通用计划的估算代价不明显更差则切换为通用计划（逻辑在 `src/backend/utils/cache/plancache.c`）。这是 PG 面对"参数敏感计划"问题的答案——MySQL 的 prepared statement 每次执行都重新优化，没有这层缓存权衡。

### 6.3 exec_execute_message（'E'，postgres.c:2151）

- `GetPortalByName()`（postgres.c:2177）→ `PortalRun(portal, max_rows, ...)`（postgres.c:2320）。
- 与简单协议最大的执行期差异：**max_rows 可以非零**——只取 N 行后挂起 portal，回 `PortalSuspended`（postgres.c:2381）；客户端可对同一 portal 再发 'E' 继续取。这是 JDBC `setFetchSize()` 的服务端机制，等价于一个协议级游标。
- 取完则回 `CommandComplete`；若语句要求立即提交（如 CREATE DATABASE），调 `finish_xact_command()`（postgres.c:2344 附近）。

### 6.4 对比总表

| 维度 | 简单协议 'Q' | 扩展协议 'P'/'B'/'E' |
|---|---|---|
| 入口 | `exec_simple_query`（postgres.c:1030） | `exec_parse/bind/execute_message`（postgres.c:1421/1662/2151） |
| 解析/规划 | 每次全量执行 | Parse 一次，Plan 在 Bind 期（可复用 generic plan） |
| 参数 | 无，只能字符串拼接 | 类型化参数、可二进制传输（无注入风险、免转义开销） |
| 结果集 | 一次取完（FETCH_ALL） | 可分批（max_rows + PortalSuspended） |
| 一报文多语句 | 允许，隐式事务块 | Parse 只接受单条语句 |
| ReadyForQuery | 每条 'Q' 后立即发 | 直到 'S'（Sync）才发 → 支持 pipelining |
| 错误恢复 | 下一条 'Q' 照常 | `ignore_till_sync` 丢弃报文直到 Sync（postgres.c:4663/4925） |
| MySQL 对应 | COM_QUERY | COM_STMT_PREPARE / COM_STMT_EXECUTE |

**小结**：扩展协议 = 把 exec_simple_query 的流水线切成 Parse（缓存分析产物）、Bind（参数化+择机规划）、Execute（可分批执行）三段，用 Sync 定界事务与错误恢复。它换来的三样东西——计划复用、二进制参数、流水线——正是 OLTP 高并发场景的刚需。

## 自测问题

**1. postmaster 收到 SIGCHLD 后，为什么不在信号 handler 里直接收尸，而只是置一个标志？**

因为收尸涉及 ereport、内存分配、状态机迁移，这些函数不是 async-signal-safe，在 handler 里调用是未定义行为。handler（`handle_pm_child_exit_signal`）只置 `pending_pm_child_exit` 并 SetLatch 唤醒主循环，实际处理延迟到 ServerLoop 里的 `process_pm_child_exit()`（postmaster.c:2261）。这是"最小信号 handler + 主循环轮询标志"的经典模式。

**2. 一个 backend 因段错误被 SIGSEGV 杀死，另一个 backend 只是执行了违反唯一约束的 INSERT，两者后果有何不同？为什么？**

前者是进程异常死亡：postmaster 在 `HandleChildCrash`（postmaster.c:2818）中假定共享内存可能已被死者污染，向所有子进程广播 SIGQUIT（quickdie），随后重建共享内存并重走 WAL 崩溃恢复——全库瞬断。后者只是 `ereport(ERROR)`：siglongjmp 回 PostgresMain，`AbortCurrentTransaction()` 回滚本事务，连接与其他会话毫发无损。区别的根源：前者破坏的可能是**共享**状态，后者的破坏被事务机制完全兜住。

**3. `ereport(ERROR, ...)` 之后控制流是怎样一步步回到主循环的？清理靠什么完成？**

`ereport` → `errfinish()`（elog.c:486）→ elevel==ERROR 时 `PG_RE_THROW()` → `pg_re_throw()`（elog.c:2200）→ `siglongjmp(*PG_exception_stack, 1)`。`PG_exception_stack` 若未被 PG_TRY 覆盖，则指向 PostgresMain 的 `local_sigjmp_buf`（postgres.c:4687），控制流落到 postgres.c:4573 的 sigsetjmp 清理块。清理不靠栈展开，而靠集中记账：`AbortCurrentTransaction()` 通过事务内存上下文、锁表、ResourceOwner 批量释放错误发生时持有的一切资源。

**4. 为什么 exec_simple_query 在语义分析之前就要检查 `IsAbortedTransactionBlockState()`？纯语法分析为什么不用检查？**

语义分析、重写、规划都要读系统目录，而目录访问需要正常的事务环境——在已中止的事务里会再次失败（postgres.c:1151-1157）。所以除 COMMIT/ROLLBACK 外的语句必须在进入这些阶段前被拒绝。而 `pg_parse_query` 是纯 flex/bison 语法处理，不碰目录、不需要事务，因此放在检查之前也安全（postgres.c:1080-1081 注释）。

**5. JDBC 设置 `setFetchSize(100)` 后，服务端是靠什么机制实现"每次只返回 100 行"的？简单协议为什么做不到？**

扩展协议的 Execute 报文携带 max_rows：`exec_execute_message`（postgres.c:2151）调 `PortalRun(portal, max_rows, ...)` 只取 100 行，portal 挂起并回 `PortalSuspended`（postgres.c:2381），客户端再发 'E' 从断点继续。简单协议里 `exec_simple_query` 对无名 portal 固定 `FETCH_ALL`（postgres.c:1297-1298），且执行完 portal 即被 drop，没有可供二次寻址的执行状态；要分批只能用 SQL 层的 DECLARE CURSOR / FETCH。
