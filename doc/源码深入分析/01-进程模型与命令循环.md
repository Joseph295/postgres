## 前置知识

- **进程模型分类**：Hellerstein 等人在《Architecture of a Database System》(2007) 中把 DBMS 分成三种经典模型——process-per-worker（每连接一个进程）、thread-per-worker（每连接一个线程）、进程/线程池。PostgreSQL 采用第一种；MySQL 采用第二种（每连接一个线程，企业版另有线程池插件）。多进程的好处是**地址空间隔离**：一个连接的野指针几乎不可能直接踩坏另一个连接的私有内存；代价是连接创建昂贵（生产上普遍前置 pgbouncer 一类连接池）、跨连接共享状态必须显式放进共享内存。
- **fork 语义**：`fork()` 之后子进程获得父进程地址空间的写时复制（COW）副本，并且**继承所有已打开的文件描述符和已建立的共享内存映射**。这是 PG 进程模型的物理基础，第 2.3 节反复用到。
- **setjmp/longjmp**：C 语言的非局部跳转。`sigsetjmp(buf, 1)` 保存当前执行上下文（含信号掩码）并返回 0；此后任意深度的调用栈里执行 `siglongjmp(buf, 1)`，控制流会"瞬移"回 `sigsetjmp` 处并返回 1。PG 用它实现了 C 语言里的"异常机制"（第 5 节）。
- **async-signal-safe**：POSIX 规定信号 handler 里只能安全调用一小撮函数（`write`、`_exit`、`sigprocmask` 等）；`malloc`、`printf`、任何可能拿锁的函数都不行——handler 可能恰好打断这些函数自身的执行，造成死锁或堆损坏。这个概念是理解 postmaster 与 backend 全部信号设计的钥匙。
- 建议先浏览 `src/backend/tcop/`、`src/backend/postmaster/` 目录结构；`src/backend/tcop/` 的名字来自 "traffic cop"（交通警察），正是"总调度"之意。

## 第 1 节 · 启动：main() → PostmasterMain()

### 1.1 万物起点 main()

所有 PostgreSQL 服务端进程（postmaster、backend、checkpointer……）的入口都是同一个 `main()`（`src/backend/main/main.c:71`）。它做的事情极少但顺序极关键：

```c
/* src/backend/main/main.c:106（节选，省略 WIN32 与 ps display 部分） */

	 * Fire up essential subsystems: error and memory management
	 *
	 * Code after this point is allowed to use elog/ereport, though
	 * localization of messages may not work right away, and messages won't go
	 * anywhere but stderr until GUC settings get loaded.
	 */
	MyProcPid = getpid();
	MemoryContextInit();

	/*
	 * Set reference point for stack-depth checking.  (There's no point in
	 * enabling this before error reporting works.)
	 */
	(void) set_stack_base();
```

- `MemoryContextInit()`（main.c:114）建立内存上下文体系的根：TopMemoryContext 与 ErrorContext。注意注释里那句 "Code after this point is allowed to use elog/ereport"——**错误处理依赖 ErrorContext，所以内存管理必须先于一切启动**。这是本文第一个"顺序即依赖"的例子。
- `set_stack_base()`（main.c:120）记录栈基址，供后续 `check_stack_depth()` 防御无限递归（解析超深表达式时你会见到 "stack depth limit exceeded"，源头在此）。
- 之后设置 locale（main.c:125-159）、检查不能以 root 运行（`check_root`，main.c:198-199）。

最后是按第一个命令行参数分发（main.c:205-233），这个 switch 值得全文摘录，因为它解释了同一个可执行文件的所有身份：

```c

**⑤ 共享内存与信号量创建（postmaster.c:926-1030）**

**三问：为什么共享内存一次性分配、永不扩容？**

1. *解决什么问题*：多进程模型下，所有进程必须在**同一虚拟地址**（或至少可换算的地址）看到同一块内存，里面还布满了指针。若允许运行时扩容，就要么搬迁（所有进程里的指针全部失效），要么追加新段（每个进程都要同步 attach，且已有数据结构无法利用新空间）。
2. *为何选此方案而非替代*：MySQL 8.0 支持在线调整 `innodb_buffer_pool_size`，那是因为 mysqld 单进程，malloc/munmap 自家地址空间即可。PG 的取舍是：启动时 `CalculateShmemSize()` 把所有需求加总、按需求圆整，之后段大小终生不变——换来的是**零同步成本的共享内存管理**：没有在线伸缩，就没有伸缩期间的并发难题。代价就是运维熟知的"改 `shared_buffers` 必须重启"。
3. *正确性不变式*：任何申请共享内存的子系统必须在 `CreateSharedMemoryAndSemaphores()` 之前登记完毕。所以 `shared_preload_libraries` 必须在共享内存创建**之前**加载（postmaster.c:936 早于 1018，顺序不可交换）；运行时 `LOAD` 的扩展永远拿不到自己的共享内存分区。

### 1.3 三问：postmaster 的信号处理为什么如此谨慎

**① 解决什么问题？没有它会怎样？**

信号可以在**任意两条指令之间**打断进程。假设 postmaster 在 SIGCHLD handler 里直接做收尸工作——收尸涉及 `waitpid`、ereport（内部 malloc）、状态机迁移、启动新进程。推演一个具体故障：主流程正在 `palloc()` 内部、堆的空闲链表处于半更新状态，此刻 SIGCHLD 到来，handler 里的 ereport 又调 `malloc` ——同一把堆数据结构被重入，轻则死锁，重则堆损坏、postmaster 崩溃。而 postmaster 一死，整棵进程树失去看门人：监听 socket 无人持有、postmaster.pid 变成孤儿、子进程崩溃无人善后。所以现行设计是：**handler 只置标志 + SetLatch，一行都不多做**。以 SIGCHLD 为例，`handle_pm_child_exit_signal` 只置 `pending_pm_child_exit`，真正的收尸（`process_pm_child_exit()`，postmaster.c:2261）延迟到 ServerLoop 主流程执行。

**② 为何选此方案而非替代？**

- *替代一：MySQL 式专职信号线程*。mysqld 单进程多线程，把信号统一 sigwait 到一个专门线程里同步处理，其余线程屏蔽信号。PG 不能用：postmaster 必须保持单线程（见 1.2 段⑦的 FATAL 检查），因为它要 fork，多线程 + fork 是未定义行为的雷区。
- *替代二：postmaster 历史上的做法*。2023 年之前，postmaster 的信号 handler 里真的做复杂工作（甚至直接 fork），靠"处理期间屏蔽所有信号"保平安。commit **7389aad636**（2023-01-12，Thomas Munro，"Use WaitEventSet API for postmaster's event loop"）把它重构成今天的样子：每个信号 handler 切成 `handle_pm_XXX()`（置标志+latch）与 `process_pm_XXX()`（主循环里做事）两半，postmaster 从此和普通 backend 一样用 latch 驱动。这是"教科书式 self-pipe/latch 模式"迟到二十多年的落地。
- *为什么 SIGUSR2 明明不用还注册 dummy_handler*（postmaster.c:562 `pqsignal(SIGUSR2, dummy_handler);	/* unused, reserve for children */`）？postgres.c:4386-4391 的注释给出答案：若 postmaster 对某信号设 SIG_IGN，fork 出的子进程在来得及改 handler 之前收到该信号会被**直接丢弃**。注册一个空 handler 就是替子进程"占住"这个信号。

**③ 正确性不变式**

- *先屏蔽、装好 handler、再放开*（postmaster.c:553 与 589）：杜绝"handler 还没装好信号就来了"的竞态。
- *fork 时机的掩码确定性*：postmaster fork 前处于确定的信号掩码状态（BlockSig），子进程因此不会在设置好自己的 handler 之前收到信号——postgres.c:4382-4384 的注释显式依赖这一点："postmaster blocked all signals before forking child process, so there is no race condition"。
- *postmaster 绝不能死*：它不碰用户数据、不进临界区、代码路径极短；任何可能失败的重活（恢复、检查点、连接服务、甚至给客户端回错误）都委托给子进程。这条不变式将在 2.2 节（dead-end child）再次出现。

**小结**：PostmasterMain 是一条直线：内存上下文 → 信号 → GUC → 数据目录锁 → 共享内存 → 监听 socket → 后台进程 → ServerLoop。所有设计怪癖都可归因于两条铁律：**共享内存一次性定形；postmaster 必须简单到不会死**。

### 2.2 BackendStartup 全函数走读

`BackendStartup()`（postmaster.c:3576-3655）在 fork 之前做完所有可能失败的资源分配：

```c
/* src/backend/postmaster/postmaster.c:3575 */
static int
BackendStartup(ClientSocket *client_sock)
{
	PMChild    *bn = NULL;
	pid_t		pid;
	BackendStartupData startup_data;
	CAC_state	cac;

	/*
	 * Capture time that Postmaster got a socket from accept (for logging
	 * connection establishment and setup total duration).
	 */
	startup_data.socket_created = GetCurrentTimestamp();

	/*
	 * Allocate and assign the child slot.  Note we must do this before
	 * forking, so that we can handle failures (out of memory or child-process
	 * slots) cleanly.
	 */
	cac = canAcceptConnections(B_BACKEND);
	if (cac == CAC_OK)
	{
		/* Can change later to B_WAL_SENDER */
		bn = AssignPostmasterChildSlot(B_BACKEND);
		if (!bn)
		{
			/*
			 * Too many regular child processes; launch a dead-end child
			 * process instead.
			 */
			cac = CAC_TOOMANY;
		}
	}
	if (!bn)
	{
		bn = AllocDeadEndChild();
		if (!bn)
		{
			ereport(LOG,
					(errcode(ERRCODE_OUT_OF_MEMORY),
					 errmsg("out of memory")));
			return STATUS_ERROR;
		}
	}

	/* Pass down canAcceptConnections state */
	startup_data.canAcceptConnections = cac;
	bn->rw = NULL;

	/* Hasn't asked to be notified about any bgworkers yet */
	bn->bgworker_notify = false;

	pid = postmaster_child_launch(bn->bkend_type, bn->child_slot,
								  &startup_data, sizeof(startup_data),
								  client_sock);
	if (pid < 0)
	{
		/* in parent, fork failed */
		int			save_errno = errno;

		(void) ReleasePostmasterChildSlot(bn);
		errno = save_errno;
		ereport(LOG,
				(errmsg("could not fork new process for connection: %m")));
		report_fork_failure_to_client(client_sock, save_errno);
		return STATUS_ERROR;
	}

	/* in parent, successful fork */
	ereport(DEBUG2,
			(errmsg_internal("forked new %s, pid=%d socket=%d",
							 GetBackendTypeDesc(bn->bkend_type),
							 (int) pid, (int) client_sock->sock)));

	/*
	 * Everything's been successful, it's safe to add this backend to our list
	 * of backends.
	 */
	bn->pid = pid;
	return STATUS_OK;
}
```

逐步解说：

1. `canAcceptConnections(B_BACKEND)`（postmaster.c:1837）检查状态机：恢复中返回 `CAC_STARTUP`/`CAC_RECOVERY`、关闭中返回 `CAC_SHUTDOWN`。注意**拒绝理由此刻只是记下来**（`startup_data.canAcceptConnections = cac`），拒绝动作留给子进程做（3.2 节）；
2. `AssignPostmasterChildSlot()` 领取子进程槽位。槽位耗尽时**并不直接拒绝，而是转为启动一个 "dead-end child"**——一个只为向客户端报错就退出的一次性进程；
3. `postmaster_child_launch(...)` 真正 fork。失败则释放槽位并尽力给客户端发一条裸协议错误——`report_fork_failure_to_client()`（postmaster.c:3665-3685）先把 socket 设为非阻塞、只 send 一次，注释明确说 "we do not care to risk blocking the postmaster on this connection"。

**三问：dead-end child 为什么存在？**

1. *解决什么问题*：连接数打满时，总得告诉客户端 "sorry, too many clients already" 而不是默默挂断。但回这句话要走 FE/BE 协议、可能 TLS 握手、可能阻塞——postmaster 一旦在慢客户端上阻塞，整个实例就不再 accept 新连接、不再给子进程收尸。推演：没有 dead-end child 且 postmaster 亲自回错，一个恶意客户端只要 connect 后不读数据，就能把 postmaster 卡在 send() 上，整库瘫痪。
2. *为何选此方案*：fork 一个廉价进程去送死（它不 attach 共享内存槽位机制中的正规资源，发完错误就退出），把"可能阻塞的工作"外包出去，postmaster 保持零阻塞。这符合 1.3 节的不变式"postmaster 绝不能死（也绝不能停）"。
3. *设计史*：dead-end children 由 commit **bbe3c02d38**（2007-08-09，Tom Lane）引入，起因是一个真实故障：持续涌入的新连接请求会阻止 postmaster 完成 shutdown/crash-restart。2024 年 commit **a78af04270**（2024-11-14，Heikki Linnakangas，"Assign a child slot to every postmaster child process"）进一步统一了槽位管理——现在连 dead-end child 也有自己的 PMChild 结构（`AllocDeadEndChild()`），postmaster 对每个孩子的记账方式终于一致。

```c
/* src/backend/postmaster/launch_backend.c:204 */
pid_t
postmaster_child_launch(BackendType child_type, int child_slot,
						void *startup_data, size_t startup_data_len,
						const ClientSocket *client_sock)
{
	pid_t		pid;

	Assert(IsPostmasterEnvironment && !IsUnderPostmaster);

	/* Capture time Postmaster initiates process creation for logging */
	if (IsExternalConnectionBackend(child_type))
		((BackendStartupData *) startup_data)->fork_started = GetCurrentTimestamp();

#ifdef EXEC_BACKEND
	pid = internal_forkexec(child_type, child_slot,
							startup_data, startup_data_len, client_sock);
	/* the child process will arrive in SubPostmasterMain */
#else							/* !EXEC_BACKEND */
	pid = fork_process();
	if (pid == 0)				/* child */
	{
		MyBackendType = child_type;

		/* Capture and transfer timings that may be needed for logging */
		if (IsExternalConnectionBackend(child_type))
		{
			conn_timing.socket_create =
				((BackendStartupData *) startup_data)->socket_created;
			conn_timing.fork_start =
				((BackendStartupData *) startup_data)->fork_started;
			conn_timing.fork_end = GetCurrentTimestamp();
		}

		/* Close the postmaster's sockets */
		ClosePostmasterPorts(child_type == B_LOGGER);

		/* Detangle from postmaster */
		InitPostmasterChild();

		/* Detach shared memory if not needed. */
		if (!child_process_kinds[child_type].shmem_attach)
		{
			dsm_detach_all();
			PGSharedMemoryDetach();
		}

		/*
		 * Enter the Main function with TopMemoryContext.  The startup data is
		 * allocated in PostmasterContext, so we cannot release it here yet.
		 * The Main function will do it after it's done handling the startup
		 * data.
		 */
		MemoryContextSwitchTo(TopMemoryContext);

		MyPMChildSlot = child_slot;
		if (client_sock)
		{
			MyClientSocket = palloc_object(ClientSocket);
			memcpy(MyClientSocket, client_sock, sizeof(ClientSocket));
		}

		/*
		 * Run the appropriate Main function
		 */
		child_process_kinds[child_type].main_fn(startup_data, startup_data_len);
		pg_unreachable();		/* main_fn never returns */
	}
#endif							/* EXEC_BACKEND */
	return pid;
}
```

**子进程如何继承共享内存？答案是"什么都不用做"。** fork 语义保证子进程天生带着父进程的全部内存映射——包括 `PGSharedMemoryCreate()` 建立的共享内存段以及指向它的所有全局指针变量。所以代码里只有**反向**操作：不需要共享内存的子进程（如 syslogger）显式 detach（launch_backend.c:243-248）。缓冲池指针、锁表指针在 fork 的一瞬间就已经在子进程里就位，这就是 PG 不需要任何"启动同步协议"的原因。

**三问：EXEC_BACKEND 为什么值得维护这么一大套？**

1. *解决什么问题*：Windows 移植。没有它就没有 Windows 版 PostgreSQL。
2. *为何选此方案*：替代方案是在 Windows 上模拟 fork（Cygwin 走过这条路，慢且脆），或者放弃多进程改多线程（等于重写整个内核）。序列化-重放虽然啰嗦，但把平台差异收敛到一份"状态清单"（save/restore_backend_variables）里。在 Unix 上手动定义 `EXEC_BACKEND` 宏也能强制走这条路，让开发者在 Linux 上就能调试 Windows 的启动路径。
3. *设计史*：EXEC_BACKEND 随 2003-2004 年的 Windows 移植进入源码树；commit **1a321f26d8**（2004-05-28，Tom Lane，"Code review for EXEC_BACKEND changes"）把所有 fork 子进程的路径统一收拢到 `SubPostmasterMain` 一个入口。二十年后，commit **f1baed18bc** + **aafc05de1b**（2024-03-18，Heikki Linnakangas，"Refactor postmaster child process launching"）再进一步：把散落在 postmaster.c 里五花八门的 `StartXXX` 启动代码统一成 `postmaster_child_launch()` + `child_process_kinds[]` 注册表，launch_backend.c 这个文件由此诞生。你今天读到的整洁结构是两轮大重构的成果——**"fork 继承"这个 Unix 特性被做成了可替换的抽象**。

		return;

	ereport(LOG,
			(errmsg("terminating any other active server processes")));

	/*
	 * Switch into error state. The crashed process has already been removed
	 * from ActiveChildList.
	 */
	HandleFatalError(PMQUIT_FOR_CRASH, true);
}
```

`HandleFatalError()`（postmaster.c:2731-2806）做四件事：`SetQuitSignalReason(reason)` 把"为什么让你死"写进共享内存（子进程的 quickdie 会读它来决定给客户端的措辞，见 5.7 节）；`TerminateChildren(SIGQUIT)` 对**所有其他子进程**广播 SIGQUIT（若 GUC `send_abort_for_crash` 打开则改发 SIGABRT 以收集 core）；置 `FatalError = true`；按当前状态机位置转入 `PM_WAIT_BACKENDS` 等大家死绝。这个函数是 commit **463a2ebd9f**（2025-01-24，Andres Freund，"postmaster: Commonalize FatalError paths"）从多处重复代码里提炼出来的。

大家死绝后，`PostmasterStateMachine()`（postmaster.c:2911）走到关键一步（全文摘录）：

```c
/* src/backend/postmaster/postmaster.c:3208 */

	 * If the startup process failed, or the user does not want an automatic
	 * restart after backend crashes, wait for all non-syslogger children to
	 * exit, and then exit postmaster.  We don't try to reinitialize when the
	 * startup process fails, because more than likely it will just fail again
	 * and we will keep trying forever.

	{
		if (StartupStatus == STARTUP_CRASHED)
		{

					(errmsg("shutting down due to startup process failure")));

		{
			ereport(LOG,
					(errmsg("shutting down because \"restart_after_crash\" is off")));
			ExitPostmaster(1);
		}
	}

	/*
	 * If we need to recover from a crash, wait for all non-syslogger children
	 * to exit, then reset shmem and start the startup process.
	 */
	if (FatalError && pmState == PM_NO_CHILDREN)
	{
		ereport(LOG,
				(errmsg("all server processes terminated; reinitializing")));

		/* remove leftover temporary files after a crash */
		if (remove_temp_files_after_crash)
			RemovePgTempFiles();

		/* allow background workers to immediately restart */
		ResetBackgroundWorkerCrashTimes();

		/* re-read control file into local memory */
		LocalProcessControlFile(true);

		 * Re-initialize shared memory and semaphores.  Note: We don't call
		 * RegisterBuiltinShmemCallbacks(), we keep the old registrations.  In
		 * order to re-register structs in extensions, we'd need to reload
		 * shared preload libraries, and we don't want to do that.
		 */
		ResetShmemAllocator();
		ShmemCallRequestCallbacks();
		CreateSharedMemoryAndSemaphores();

**共享内存推倒重建 + 重新走 WAL 崩溃恢复**——和实例断电重启完全同构。`restart_after_crash` GUC（commit **5ffaa9005c**，2010-07-20，Robert Haas）允许关闭自动重启：托管环境（如 Patroni/K8s）往往宁可让实例死透、由外部编排器接管故障切换决策。

**三问：为什么必须如此激进？**

1. *解决什么问题*：异常死亡的进程可能死在**任意时刻**——正持有 LWLock、改到一半共享哈希表、写到一半缓冲区头。共享内存此刻处于不可知状态。推演：若 postmaster 只重启死者、放过其他人，一个死在"已把 buffer 标记为脏但还没挂进检查点队列"瞬间的 backend，会让这个脏页永远不被写盘——数据丢失且无人报错。
2. *为何选此方案而非替代*：理论替代是"精细审计"——追踪死者持有过哪些锁、碰过哪些结构，逐一修复。但共享内存里没有 undo 信息，无法知道死者把哪个字节改到一半；修复代码本身也要读这块可疑内存。唯一可证明正确的做法：**假定共享内存已污染，杀光所有映射者，重建，用 WAL 恢复到一致状态**。WAL 本来就是为断电准备的，crash-restart 只是复用了它。
3. *正确性不变式*：「普通 `ereport(ERROR)` 的破坏被事务机制完全兜住（第 5 节论证），因此只回滚不重启；**进程级异常死亡**破坏的可能是共享状态，因此必须全体重启」。两档灾难半径的分界线，恰好就是"错误是否经由 elog 框架受控抛出"。运维上你看到的 `WARNING: terminating connection because of crash of another server process` 就来自这条链路。

> **与 MySQL 对比**：mysqld 里一个线程段错误会直接放倒整个进程——所有连接、缓冲池一起消失，连"只重启一部分"的选项都没有；恢复同样靠 InnoDB redo log 重放。PG 的多进程隔离把"灾难半径"分成了两档：SQL 级错误只伤本事务；进程级崩溃才伤全库。MySQL 只有后一档。

> **What-if 推演 ①：backend 在持有 LWLock 时被 `kill -9`，会发生什么？**
>
> 1. 该 backend 立即消失，它在共享内存里持有的 LWLock **不会被释放**（kill -9 没有任何清理机会）；等待该锁的其他 backend 全部挂起。
> 2. postmaster 收到 SIGCHLD，`process_pm_child_exit()` 发现退出状态是"被信号 9 杀死"→ `HandleChildCrash` → `HandleFatalError(PMQUIT_FOR_CRASH, true)`：所有子进程收到 SIGQUIT，执行 `quickdie()`（postgres.c:3017）——不做清理直接 `_exit(2)`，那些等锁的进程也随之消失，死锁自动"解开"。
> 3. 共享内存重建、WAL 恢复、重新开门。未提交事务回滚，已提交事务由 WAL 保证不丢。
> 4. 对比：同一个 backend 若是 `ereport(ERROR)`，`AbortTransaction()` 第一件事就是 `LWLockReleaseAll()`（xact.c:2878，见 5.5 节）——受控错误路径下锁一定会还，这正是"两档灾难半径"的微观机理。
> 5. 顺带回答"为什么 `pg_terminate_backend()` 用 SIGTERM 而 DBA 手册严禁对 backend 用 `kill -9`"：SIGTERM 走 die() → 受控 FATAL → 正常清理（4.5 节推演 ④）；kill -9 则等于亲手触发一次全库 crash-restart。

> **What-if 推演 ②：crash-restart 进行中，又一个进程异常退出，会不会无限套娃？**
>
> 不会。`HandleChildCrash` 开头就检查 `if (FatalError || Shutdown == ImmediateShutdown) return;`——善后期间的第二、三个异常退出（多半正是被我们 SIGQUIT 杀掉的孩子）只做记账不再触发新一轮广播。而如果有孩子连 SIGQUIT 都不理（比如卡在不可中断的内核态 I/O），ServerLoop 段三的定时器在 `SIGKILL_CHILDREN_AFTER_SECS` 后补刀 SIGKILL（postmaster.c:1781-1793）。状态机要求 `PM_NO_CHILDREN`（一个不剩）才重建共享内存——旧进程还映射着旧段时就重建，等于制造两个平行世界。

**小结**：ServerLoop 是"等事件 → 消化信号标志 → accept → fork"的极简循环。fork 让子进程零成本继承共享内存与 socket；EXEC_BACKEND 用参数文件+重 attach 模拟这种继承以支持 Windows。子进程异常死亡时，postmaster 假定共享内存已污染：广播 SIGQUIT、重建共享内存、重走 WAL 恢复——与断电重启同构。

**三问：为什么读启动包在 InitProcess 之前？**

1. *解决什么问题*：PGPROC 槽位是稀缺共享资源（`max_connections` 决定总量）。若先领槽位再读包，恶意客户端"连上就不说话"即可用 N 条空连接占满槽位，配额型 DoS。
2. *为何选此方案*：现行顺序下，没发启动包的客户端只占着一个廉价进程 + 一条 socket，共享资源分文未取；`authentication_timeout`（默认 60s）到点即 `_exit(1)` 回收。替代方案（占着槽位限时回收）会让"槽位清理"依赖复杂的跨进程超时协议。
3. *正确性不变式*：**碰共享内存之前，进程必须可以被无条件粗暴杀死**。段二的 `_exit(1)` 策略、段四之前的 `check_on_shmem_exit_lists_are_empty()` 都在守护这条不变式。

> **What-if 推演 ③：认证阶段挂死（比如 LDAP 服务器无响应），会怎样？**
>
> 分两个阶段看，安全机制不同：
> - *启动包阶段*（BackendInitialize）：`STARTUP_PACKET_TIMEOUT` 到点触发 `StartupPacketTimeoutHandler`（backend_startup.c:999-1001）→ `_exit(1)`。因为没碰共享内存，直接蒸发即可，postmaster 收尸时视为正常退出（exit code 1 不算 crash）。
> - *真正的认证阶段*（`PerformAuthentication`，postinit.c:948，此时已在 InitPostgres 里、**已领 PGPROC**）：这里重新武装了 `authentication_timeout`，但到点后不能再 `_exit`，而是走 `ereport(FATAL)` ——受控退出，proc_exit 回调会把 PGPROC 还给 ProcArray、断开 sinval 网。注释（backend_startup.c:275-278）还提醒：两段超时是分别计时的，恶意客户端最多可以拖住一个进程约 2×authentication_timeout。
> - 若认证代码卡在不可中断的系统调用里连 SIGALRM 都进不来：FAST shutdown 的 SIGTERM 同样无效，最后由 immediate shutdown / crash 路径的 SIGKILL 兜底（ServerLoop 段三）。防御是分层的：超时 → 信号 → SIGKILL。

## 第 4 节 · PostgresMain：backend 的一生

`PostgresMain()`（`src/backend/tcop/postgres.c:4364`）是 backend 的"主函数"，永不返回。结构：一次性设置 → sigsetjmp 恢复块 → 无限循环。

### 4.1 一次性设置走读（postgres.c:4364-4549）

**函数头与三个 volatile（postgres.c:4364-4371）**

```c

`volatile` 不是装饰：C 标准规定 longjmp 之后**非 volatile 局部变量若在 setjmp 后被修改过，其值是未定义的**（编译器可能把它们缓存在寄存器里，而 longjmp 只恢复寄存器现场到 setjmp 时刻）。这是阅读一切含 sigsetjmp 代码的必备常识。

### 4.2 sigsetjmp 恢复块全文走读

主循环之前的这段代码是全 PG 最重要的百来行。先看官方给出的"为什么不用 PG_TRY"（postgres.c:4551-4570，注释全文）：

```c
/* src/backend/tcop/postgres.c:4551 */
	/*
	 * POSTGRES main processing loop begins here
	 *
	 * If an exception is encountered, processing resumes here so we abort the
	 * current transaction and start a new one.
	 *
	 * You might wonder why this isn't coded as an infinite loop around a
	 * PG_TRY construct.  The reason is that this is the bottom of the
	 * exception stack, and so with PG_TRY there would be no exception handler
	 * in force at all during the CATCH part.  By leaving the outermost setjmp
	 * always active, we have at least some chance of recovering from an error
	 * during error recovery.  (If we get into an infinite loop thereby, it
	 * will soon be stopped by overflow of elog.c's internal state stack.)
	 *
	 * Note that we use sigsetjmp(..., 1), so that this function's signal mask
	 * (to wit, UnBlockSig) will be restored when longjmp'ing to here.  This
	 * is essential in case we longjmp'd out of a signal handler on a platform
	 * where that leaves the signal blocked.  It's not redundant with the
	 * unblock in AbortTransaction() because the latter is only called if we
	 * were inside a transaction.
	 */
```

两条信息：**链底的 setjmp 永远在位**——清理过程中再出错，就再跳回来一次（真的死循环会被 elog.c 的递归深度检查拦下，5.2 节）；`sigsetjmp(buf, 1)` 的第二参数 1 表示同时保存/恢复**信号掩码**——万一是从信号 handler 里 longjmp 出来的（如 FloatExceptionHandler 直接 ereport(ERROR)），落地时掩码能回到 UnBlockSig。

恢复块本体全文：

```c
/* src/backend/tcop/postgres.c:4573 */

	{
		/*
		 * NOTE: if you are tempted to add more code in this if-block,
		 * consider the high probability that it should be in
		 * AbortTransaction() instead.  The only stuff done directly here
		 * should be stuff that is guaranteed to apply *only* for outer-level
		 * error recovery, such as adjusting the FE/BE protocol status.
		 */

		/* Since not using PG_TRY, must reset error stack by hand */
		error_context_stack = NULL;

		/* Prevent interrupts while cleaning up */
		HOLD_INTERRUPTS();

		/*
		 * Forget any pending QueryCancel request, since we're returning to
		 * the idle loop anyway, and cancel any active timeout requests.  (In
		 * future we might want to allow some timeout requests to survive, but
		 * at minimum it'd be necessary to do reschedule_timeouts(), in case
		 * we got here because of a query cancel interrupting the SIGALRM
		 * interrupt handler.)	Note in particular that we must clear the
		 * statement and lock timeout indicators, to prevent any future plain
		 * query cancels from being misreported as timeouts in case we're
		 * forgetting a timeout cancel.
		 */
		disable_all_timeouts(false);	/* do first to avoid race condition */
		QueryCancelPending = false;
		idle_in_transaction_timeout_enabled = false;
		idle_session_timeout_enabled = false;

		/* Not reading from the client anymore. */
		DoingCommandRead = false;

		/* Make sure libpq is in a good state */
		pq_comm_reset();

		/* Report the error to the client and/or server log */
		EmitErrorReport();

		/*
		 * If Valgrind noticed something during the erroneous query, print the
		 * query string, assuming we have one.
		 */
		valgrind_report_error_query(debug_query_string);

		/*
		 * Make sure debug_query_string gets reset before we possibly clobber
		 * the storage it points at.
		 */
		debug_query_string = NULL;

		/*
		 * Abort the current transaction in order to recover.
		 */
		AbortCurrentTransaction();

		if (am_walsender)
			WalSndErrorCleanup();

		PortalErrorCleanup();

		/*
		 * We can't release replication slots inside AbortTransaction() as we
		 * need to be able to start and abort transactions while having a slot
		 * acquired. But we never need to hold them across top level errors,
		 * so releasing here is fine. There also is a before_shmem_exit()
		 * callback ensuring correct cleanup on FATAL errors.
		 */
		if (MyReplicationSlot != NULL)
			ReplicationSlotRelease();

		/* We also want to cleanup temporary slots on error. */
		ReplicationSlotCleanup(false);

		jit_reset_after_error();

		/*
		 * Now return to normal top-level context and clear ErrorContext for
		 * next time.
		 */
		MemoryContextSwitchTo(MessageContext);
		FlushErrorState();

		/*
		 * If we were handling an extended-query-protocol message, initiate
		 * skip till next Sync.  This also causes us not to issue
		 * ReadyForQuery (until we get Sync).
		 */
		if (doing_extended_query_message)
			ignore_till_sync = true;

		/* We don't have a transaction command open anymore */
		xact_started = false;

		/*
		 * If an error occurred while we were reading a message from the
		 * client, we have potentially lost track of where the previous
		 * message ends and the next one begins.  Even though we have
		 * otherwise recovered from the error, we cannot safely read any more
		 * messages from the client, so there isn't much we can do with the
		 * connection anymore.
		 */
		if (pq_is_reading_msg())
			ereport(FATAL,
					(errcode(ERRCODE_PROTOCOL_VIOLATION),
					 errmsg("terminating connection because protocol synchronization was lost")));

		/* Now we can allow interrupts again */
		RESUME_INTERRUPTS();
	}

	/* We can now handle ereport(ERROR) */
	PG_exception_stack = &local_sigjmp_buf;
```

清理清单可以归为五组，注意**每一项都是幂等的、且不依赖"错误发生在哪"**：

1. *中断与超时归零*：`HOLD_INTERRUPTS()` 保护清理过程；`disable_all_timeouts` 抢在清标志前执行（注释点名"race condition"：先清 QueryCancelPending 再关定时器的话，定时器可能在两步之间又把标志置上）；
2. *协议层复位*：`pq_comm_reset()` 恢复 libpq 输出状态（错误可能发生在写一半消息时）；`EmitErrorReport()` 把 errstart 以来积累的错误数据真正发给客户端+写日志——注意**报错先于回滚**，所以客户端看到错误时事务可能还没回滚完，但这无关紧要，因为客户端在收到 ReadyForQuery 之前本来就不能发下一条命令；
3. *资源大清算*：`AbortCurrentTransaction()`（一行顶一万行，5.5 节详解）、`PortalErrorCleanup()`、复制槽释放、`jit_reset_after_error()`；
4. *协议状态机*：扩展协议出错要 `ignore_till_sync = true`（后续报文跳过直到 Sync，第 7 节）；若错误发生在**读消息读到一半**时（`pq_is_reading_msg()`），消息边界已丢，唯一安全的选择是 FATAL 断连——"宁死不猜"；
5. *落地*：`MemoryContextSwitchTo(MessageContext)`（errfinish 把上下文留在 ErrorContext，见 5.3 节）、`FlushErrorState()` 清空错误栈。

最后 `PG_exception_stack = &local_sigjmp_buf;`（postgres.c:4687）——把全局异常栈指针指向这里，从此**任意深度的 `ereport(ERROR)` 都会跳回上面这个 if 块**。

**一切清理必须注册成回调**——因为退出路径不止这一条（FATAL、die、超时……），只有回调链是所有路径的必经之地。

> **What-if 推演 ④：backend 阻塞在 ReadCommand 等客户端输入时，收到 `pg_terminate_backend()` 的 SIGTERM，会怎样？**
>
> 1. handler `die()`（postgres.c:3114）只做三件事：`InterruptPending = true; ProcDiePending = true;` + `SetLatch(MyLatch)`。不 exit、不 longjmp——此刻可能正在 malloc 内部。
> 2. 底层读 socket 的等待是 latch + socket 的联合等待，SetLatch 把它唤醒；重试前调用 `ProcessClientReadInterrupt(true)`（postgres.c:514）：
>    ```c
>    /* src/backend/tcop/postgres.c:519 */
>    	if (DoingCommandRead)
>    	{
>    		/* Check for general interrupts that arrived before/while reading */
>    		CHECK_FOR_INTERRUPTS();
>    ```
>    正因为主循环在 (2) 步置了 `DoingCommandRead = true`，这里才敢直接 CHECK——空闲期没有任何半成品状态，随时可死。
> 3. `ProcessInterrupts()`（postgres.c:3452）看到 ProcDiePending，`ereport(FATAL, "terminating connection due to administrator command")` → errfinish 的 FATAL 路径 → `proc_exit(1)` → 回调链归还 PGPROC、断开 sinval、关 socket。**空闲连接因此能瞬间响应终止**。
> 4. 对照组一：若正在**执行查询**（DoingCommandRead=false），SIGTERM 只置标志，等查询代码里下一个 `CHECK_FOR_INTERRUPTS()` 采样点才生效——所以卡在不含采样点的第三方 C 函数（或失控的 PL 扩展）里的 backend 对 SIGTERM 无反应，DBA 最后只能 kill -9（代价见推演 ①）。
> 5. 对照组二：若阻塞在**写**（客户端不收数据），`ProcessClientWriteInterrupt`（postgres.c:561）额外处理：先把 `whereToSendOutput` 改成 DestNone 再死——因为可能已经写了半条消息，再往 socket 里塞错误报文只会火上浇油（注释原话：会阻塞、且会丢失协议同步）。

**小结**：PostgresMain = 一次性初始化 + sigsetjmp 链底 + "reset 内存 → ReadyForQuery → 读报文 → switch 分发"的循环。空闲期的可中断性由 `DoingCommandRead` 窗口显式声明；一切退出路径汇聚到 proc_exit 回调链。错误处理三件套——`ereport(ERROR)` 抛、`siglongjmp` 跳、`AbortCurrentTransaction()` 清——下一节逐环拆解。

## 第 5 节 · 错误处理主线：ereport → errfinish → siglongjmp → AbortCurrentTransaction

这一节回答本章最深的一个问题：**为什么 PG 可以在任意调用深度、任意子系统里 `ereport(ERROR)`，而不泄漏内存、不悬挂锁、不留下半开文件？** 全链条：

```
ereport(ERROR, ...)                          宏，src/include/utils/elog.h:166
 ├─ errstart(ERROR, domain)                  elog.c:355   决定报不报、升不升级，压错误栈
 ├─ errmsg()/errcode()/...                   填充当前错误栈帧（分配在 ErrorContext）
 └─ errfinish(__FILE__, __LINE__, __func__)  elog.c:486
     └─ elevel==ERROR → PG_RE_THROW()        elog.c:551
         └─ pg_re_throw()                    elog.c:2200
             └─ siglongjmp(*PG_exception_stack, 1)
                 └─ 落回 postgres.c:4573 的恢复块（或某个 PG_TRY 的 CATCH）
                     ├─ EmitErrorReport()    elog.c:1883  真正输出错误
                     └─ AbortCurrentTransaction()  xact.c:3504  级联释放一切资源
```

### 5.1 抛出端：ereport 宏的真面目

```c
/* src/include/utils/elog.h:144（HAVE_PG_INTEGER_CONSTANT_P 分支） */
#define ereport_domain(elevel, domain, ...)	\
	do { \
		pg_prevent_errno_in_scope(); \
		if (pg_integer_constant_p(elevel) && (elevel) >= ERROR ? \
			errstart_cold(elevel, domain) : \
			errstart(elevel, domain)) \
			__VA_ARGS__, errfinish(__FILE__, __LINE__, __func__); \
		if (pg_integer_constant_p(elevel) && (elevel) >= ERROR) \
			pg_unreachable(); \
	} while(0)

#define ereport(elevel, ...)	\
	ereport_domain(elevel, TEXTDOMAIN, __VA_ARGS__)
```

三个细节：`errstart` 返回 false 时（比如 DEBUG 消息且日志级别不够），`errmsg()` 等参数**根本不求值**——这是 ereport 能撒遍热路径而无性能负担的原因；`errstart_cold` 带 `pg_attribute_cold`，提示编译器把 ERROR 分支布局到冷区；`pg_unreachable()` 告诉编译器 `ereport(ERROR)` 之后的代码不可达（所以你能看到 `ereport(ERROR, ...)` 后面不写 return 的代码）。

### 5.2 errstart 走读：升级规则与递归防御

`errstart()`（`src/backend/utils/error/elog.c:355`）最重要的是开头的**级别升级规则**：

```c
/* src/backend/utils/error/elog.c:362 */
	/*
	 * Check some cases in which we want to promote an error into a more
	 * severe error.  None of this logic applies for non-error messages.
	 */
	if (elevel >= ERROR)
	{
		/*
		 * If we are inside a critical section, all errors become PANIC
		 * errors.  See miscadmin.h.
		 */
		if (CritSectionCount > 0)
			elevel = PANIC;

		/*
		 * Check reasons for treating ERROR as FATAL:
		 *
		 * 1. we have no handler to pass the error to (implies we are in the
		 * postmaster or in backend startup).
		 *
		 * 2. ExitOnAnyError mode switch is set (initdb uses this).
		 *
		 * 3. the error occurred after proc_exit has begun to run.  (It's
		 * proc_exit's responsibility to see that this doesn't turn into
		 * infinite recursion!)
		 */
		if (elevel == ERROR)
		{
			if (PG_exception_stack == NULL ||
				ExitOnAnyError ||
				proc_exit_inprogress)
				elevel = FATAL;
		}
```

- **临界区里 ERROR 升 PANIC**：`START_CRIT_SECTION()` 的语义就是"接下来这几行若失败，共享状态将不一致"（典型：WAL 记录已写、数据页改一半）。此时回滚已不可能，唯一诚实的做法是 PANIC → abort() → 让 postmaster 走 crash-restart、用 WAL 修复。这条规则把"什么情况必须全库重启"从人的自觉变成了机器强制。
- **没有 handler 时 ERROR 升 FATAL**：`PG_exception_stack == NULL` 意味着还没执行到 postgres.c:4687（比如 postmaster 里、或 backend 启动早期）——没人接的异常自动变成"退出本进程"。这就是为什么 PostmasterMain 里的 `ereport(ERROR, ...)`（如 GUC 冲突检查）实际效果是启动失败退出。

随后压栈（elog.c:432-452）：`recursion_depth` 防错误处理自身出错的无限递归——递归时直接 `MemoryContextReset(ErrorContext)` 弃掉旧错误，深度过头还会放弃 error_context_stack；4.2 节注释里"死循环会被 elog.c 拦下"指的就是这里。错误信息的所有分配都挂在 `edata->assoc_context = ErrorContext`（elog.c:471）——ErrorContext 在 `MemoryContextInit()` 时创建并**预留了保底内存**，所以"内存耗尽时报告内存耗尽错误"也能工作。

### 5.3 errfinish 走读：三条出路

`errfinish()`（elog.c:486-630）先补齐位置信息、切到 ErrorContext、逐个调用 `error_context_stack` 上的回调（这是错误消息里 `CONTEXT: SQL statement ...` 行的来源——各子系统通过挂回调给错误"贴现场标签"）。然后按 elevel 走三条出路：

**出路一：elevel == ERROR → 抛给 handler（elog.c:528-551）**

```c
/* src/backend/utils/error/elog.c:524 */
	/*
	 * If ERROR (not more nor less) we pass it off to the current handler.
	 * Printing it and popping the stack is the responsibility of the handler.
	 */
	if (elevel == ERROR)
	{
		/*
		 * We do some minimal cleanup before longjmp'ing so that handlers can
		 * execute in a reasonably sane state.
		 *
		 * Reset InterruptHoldoffCount in case we ereport'd from inside an
		 * interrupt holdoff section.  (We assume here that no handler will
		 * itself be inside a holdoff section.  If necessary, such a handler
		 * could save and restore InterruptHoldoffCount for itself, but this
		 * should make life easier for most.)
		 */
		InterruptHoldoffCount = 0;
		QueryCancelHoldoffCount = 0;

		CritSectionCount = 0;	/* should be unnecessary, but... */

		/*
		 * Note that we leave CurrentMemoryContext set to ErrorContext. The
		 * handler should reset it to something else soon.
		 */

		recursion_depth--;
		PG_RE_THROW();
	}
```

注意此路**不打印错误**——打印被推迟到 handler（PostgresMain 恢复块里的 `EmitErrorReport()`）。为什么？因为中途可能有 PG_TRY 把这个错误整个吃掉（比如 PL/pgSQL 的 `EXCEPTION WHEN` 子句），被吃掉的错误不该出现在日志里。

**出路二：FATAL → proc_exit(1)（elog.c:577-609）**。先 `EmitErrorReport()`（elog.c:555，这条路会立即打印），然后：

```c
/* src/backend/utils/error/elog.c:603 */
		/*
		 * Do normal process-exit cleanup, then return exit code 1 to indicate
		 * FATAL termination.  The postmaster may or may not consider this
		 * worthy of panic, depending on which subprocess returns it.
		 */
		proc_exit(1);
```

FATAL 的本质是"**受控地**退出本进程"：proc_exit 会依次执行 on_shmem_exit / on_proc_exit 回调链——归还 PGPROC、放锁、断 sinval、写统计。所以 backend 的 FATAL 只终结一个连接，postmaster 视 exit(1) 为正常退出（2.4 节的判据）。也因此 `PG_TRY` **接不住 FATAL**（elog.h:365-368 的注释明确警告）——它压根不走 longjmp。

**出路三：PANIC → abort()（elog.c:611-622）**。产生 core、退出状态为被 SIGABRT 杀死 → postmaster 认定 crash → 全库重启。三条出路对应三档灾难半径：**ERROR 伤一个事务，FATAL 伤一条连接，PANIC 伤整个实例**。

### 5.4 pg_re_throw：最后一跳

### 5.5 落点清理：为什么 longjmp 不泄漏资源

longjmp 只恢复寄存器，不会替你 free 内存、放锁、关文件。PG 的答案是：**所有资源从一开始就不登记在栈上，而登记在事务粒度的集中账本里**——

- 内存：palloc 挂在内存上下文树上，事务/消息级上下文随 Abort/循环顶整体销毁；
- 重量级锁：记在共享内存锁表里，按事务释放；
- 缓冲区 pin、文件引用、快照、catcache 引用……：记在 **ResourceOwner**（`src/backend/utils/resowner/resowner.c`）里。

于是恢复块的核心就一行 `AbortCurrentTransaction()`（xact.c:3504，内部循环调 `AbortCurrentTransactionInternal` 处理子事务栈）。真正干活的 `AbortTransaction()`（xact.c:2855）值得走读开头一段：

```c

**三问：集中记账 vs 栈展开（RAII）**

1. *解决什么问题*：C 没有析构函数。若用"错误码层层上传"的替代方案，每一层都要写 `if (ret != OK) { cleanup; return ret; }`——PG 有几十万个可能失败的点，主逻辑会被错误路径淹没，且漏写一处就是泄漏。
2. *为何选此方案*：C++ 的 RAII 在栈展开时逐个析构（谁分配谁负责）；PG 把"谁分配"改成"记在谁的账上"，中止时**批量清算**。两者殊途同归，但集中记账有个额外好处：资源生命周期与**事务**对齐而不是与**函数栈帧**对齐——一个跨越多次函数调用的缓冲区 pin 在 RAII 里很难表达，在 ResourceOwner 里天然成立。MySQL 走的是第三条路：错误码上传 + THD 上的 Diagnostics_area + 各子系统手工清理，代价正是无处不在的 `goto err`。
3. *正确性不变式*：**任何资源在拿到手的那一刻就必须已经登记入账**（先记账后使用，ResourceOwner API 甚至提供 `EnlargeXXX` 先预留账本空间再获取资源的两段式，保证记账本身不会因 OOM 失败）。只要不变式成立，"任意点抛错"就等价于"账本清算"，与错误发生位置无关。

### 5.6 中途拦截：PG_TRY 宏全文

```c
/* src/include/utils/elog.h:388 */
#define PG_TRY(...)  \
	do { \
		sigjmp_buf *_save_exception_stack##__VA_ARGS__ = PG_exception_stack; \
		ErrorContextCallback *_save_context_stack##__VA_ARGS__ = error_context_stack; \
		sigjmp_buf _local_sigjmp_buf##__VA_ARGS__; \
		bool _do_rethrow##__VA_ARGS__ = false; \
		if (sigsetjmp(_local_sigjmp_buf##__VA_ARGS__, 0) == 0) \
		{ \
			PG_exception_stack = &_local_sigjmp_buf##__VA_ARGS__

#define PG_CATCH(...)	\
		} \
		else \
		{ \
			PG_exception_stack = _save_exception_stack##__VA_ARGS__; \
			error_context_stack = _save_context_stack##__VA_ARGS__

#define PG_END_TRY(...)  \
		} \
		if (_do_rethrow##__VA_ARGS__) \
				PG_RE_THROW(); \
		PG_exception_stack = _save_exception_stack##__VA_ARGS__; \
		error_context_stack = _save_context_stack##__VA_ARGS__; \
	} while (0)
```

（省略 PG_FINALLY，elog.h:405-411。）本质一目了然：**把 `PG_exception_stack` 暂时指向本地新 buf，退出时还原**——异常处理器链就是这个全局指针的保存/恢复链，PostgresMain 的 sigsetjmp 是链底。两处对比链底手写版（postgres.c:4573）：PG_TRY 用 `sigsetjmp(..., 0)`（不存掩码，快；因为 PG_TRY 不用于从信号 handler 跳出的场景）而链底用 1；PG_CATCH 一进来就把异常栈**还原成外层**，所以 CATCH 里再抛错会外抛——而链底若也这么写，CATCH 期间就没有任何 handler 了，这正是官方注释解释的"链底必须手写、让 setjmp 永远在位"的原因（4.2 节）。

### 5.7 FATAL 之外的死法：quickdie

第 2.4 节 postmaster 广播的 SIGQUIT 在 backend 侧落到 `quickdie()`（postgres.c:3017-3107）。骨架：

```c
/* src/backend/tcop/postgres.c:3016（节选） */
void
quickdie(SIGNAL_ARGS)
{
	sigaddset(&BlockSig, SIGQUIT);	/* prevent nested calls */
	sigprocmask(SIG_SETMASK, &BlockSig, NULL);
	...
	HOLD_INTERRUPTS();

	/*
	 * If we're aborting out of client auth, don't risk trying to send
	 * anything to the client; we will likely violate the protocol, not to
	 * mention that we may have interrupted the guts of OpenSSL or some
	 * authentication library.
	 */
	if (ClientAuthInProgress && whereToSendOutput == DestRemote)
		whereToSendOutput = DestNone;
	...
	error_context_stack = NULL;

	switch (GetQuitSignalReason())
	{
		...
		case PMQUIT_FOR_CRASH:
			/* A crash-and-restart cycle is in progress */
			ereport(WARNING_CLIENT_ONLY,
					(errcode(ERRCODE_CRASH_SHUTDOWN),
					 errmsg("terminating connection because of crash of another server process"),
					 errdetail("The postmaster has commanded this server process to roll back"
							   " the current transaction and exit, because another"
							   " server process exited abnormally and possibly corrupted"
							   " shared memory."),
					 errhint("In a moment you should be able to reconnect to the"
							 " database and repeat your command.")));
			break;
		...
	}

	/*
	 * We DO NOT want to run proc_exit() or atexit() callbacks -- we're here
	 * because shared memory may be corrupted, so we don't want to try to
	 * clean up our transaction.  Just nail the windows shut and get out of
	 * town.  The callbacks wouldn't be safe to run from a signal handler,
	 * anyway.
	 *
	 * Note we do _exit(2) not _exit(0).  This is to force the postmaster into
	 * a system reset cycle if someone sends a manual SIGQUIT to a random
	 * backend.  This is necessary precisely because we don't clean up our
	 * shared memory state.  (The "dead man switch" mechanism in pmsignal.c
	 * should ensure the postmaster sees this as a crash, too, but no harm in
	 * being doubly sure.)
	 */
	_exit(2);
}
```

与 FATAL 处处相反：**不走 proc_exit 回调**（共享内存不可信，清理动作本身可能踩雷）、在信号 handler 里直接 ereport（注释坦承 "It's dubious ... It is certainly not async-signal safe. But it seems better to try"——对客户端厚道优先于教条，反正 postmaster 兜底 SIGKILL）、`_exit(2)` 而非 `_exit(1)`（非 0/1 的退出码让 postmaster 把"有人手滑对 backend 发 SIGQUIT"也当作 crash 处理——因为我们确实没清理共享状态）。这段代码是十几年打磨出来的：commit **d6de43099a**（2009-12-16）修复了在 handler 里重开 SIGQUIT 导致的重入；commit **8e19a82640**（2018-08-08，Heikki Linnakangas，"Don't run atexit callbacks in quickdie signal handlers"）堵上了 exit() 会跑 atexit 回调的洞（改 _exit）；commit **44fc6e259b**（2020-09-16，Tom Lane）把所有子进程类型的 SIGQUIT 处理集中到 `InitPostmasterChild` 统一装配。

**小结**：错误处理主线 = 三档灾难半径（ERROR/FATAL/PANIC）× 两个机制（longjmp 到链底 + 集中记账清算）。ERROR 在 errstart 里可能被强制升级（临界区→PANIC，无 handler→FATAL）；errfinish 对 ERROR 不打印、直接 PG_RE_THROW 给 handler；恢复块用 AbortCurrentTransaction 批量清算所有入账资源。"任意深度抛错不泄漏"的充要条件是"先记账后用资源"这条全局纪律。

		commandTag = CreateCommandTag(parsetree->stmt);
		cmdtagname = GetCommandTagNameAndLen(commandTag, &cmdtaglen);

		set_ps_display_with_len(cmdtagname, cmdtaglen);

		BeginCommand(commandTag, dest);

		/*
		 * If we are in an aborted transaction, reject all commands except
		 * COMMIT/ABORT.  It is important that this test occur before we try
		 * to do parse analysis, rewrite, or planning, since all those phases
		 * try to do database accesses, which may fail in abort state. (It
		 * might be safe to allow some additional utility commands in this
		 * state, but not many...)
		 */
		if (IsAbortedTransactionBlockState() &&
			!IsTransactionExitStmt(parsetree->stmt))
			ereport(ERROR,
					(errcode(ERRCODE_IN_FAILED_SQL_TRANSACTION),
					 errmsg("current transaction is aborted, "
							"commands ignored until end of transaction block")));

		/* Make sure we are in a transaction command */
		start_xact_command();

		/*
		 * If using an implicit transaction block, and we're not already in a
		 * transaction block, start an implicit block to force this statement
		 * to be grouped together with any following ones.  (We must do this
		 * each time through the loop; otherwise, a COMMIT/ROLLBACK in the
		 * list would cause later statements to not be grouped.)
		 */
		if (use_implicit_block)
			BeginImplicitTransactionBlock();

		/* If we got a cancel signal in parsing or prior command, quit */
		CHECK_FOR_INTERRUPTS();
```

这就是 PG 用户最熟悉的报错之一："current transaction is aborted"——事务内一条语句失败后，除 COMMIT/ROLLBACK 外一律拒绝。为什么放在语义分析**之前**检查？注释说得明白：分析/重写/规划都要访问系统目录，而目录访问在中止事务里会再次失败。②的"纯语法分析在中止事务里也安全"正是这个检查得以安放于此的前提。这与 MySQL "语句失败后事务照常继续"是行为级差异，根源是 PG 的错误恢复以事务为单位（第 5 节）。

```c

		{
			/*
			 * If this is the last parsetree of the query string, close down
			 * transaction statement before reporting command-complete.  This
			 * is so that any end-of-transaction errors are reported before
			 * the command-complete message is issued, to avoid confusing
			 * clients who will expect either a command-complete message or an
			 * error, not one and then the other.  Also, if we're using an
			 * implicit transaction block, we must close that out first.
			 */
			if (use_implicit_block)
				EndImplicitTransactionBlock();
			finish_xact_command();
		}
		else if (IsA(parsetree->stmt, TransactionStmt))
		{
			/*
			 * If this was a transaction control statement, commit it. We will
			 * start a new xact command for the next command.
			 */
			finish_xact_command();
		}
		else
		{
			/*

			/*

			 * those that start or end a transaction block.

			/*

			 * string, so that the timeout applies separately to each query.
			 * (Our next loop iteration will start a fresh timeout.)

			disable_statement_timeout();
		}

		 * one EndCommand report for each raw parsetree, thus one for each SQL
		 * command the client sent, regardless of rewriting. (But a command
		 * aborted by error will not send an EndCommand report at all.)
		 */
		EndCommand(&qc, dest, false);

		/* Now we may drop the per-parsetree context, if one was created. */
		if (per_parsetree_context)
			MemoryContextDelete(per_parsetree_context);

## 第 7 节 · 简单协议 vs 扩展协议

### 7.1 报文是怎么读进来的：SocketBackend 走读

`SocketBackend()`（postgres.c:366-484）负责从 socket 读一条完整报文。协议 3.0 的报文格式是"1 字节类型 + 4 字节长度 + 载荷"：

```c
/* src/backend/tcop/postgres.c:371（节选） */
	/*
	 * Get message type code from the frontend.
	 */
	HOLD_CANCEL_INTERRUPTS();
	pq_startmsgread();
	qtype = pq_getbyte();
	...
	switch (qtype)
	{
		case PqMsg_Query:
			maxmsglen = PQ_LARGE_MESSAGE_LIMIT;
			doing_extended_query_message = false;
			break;
		...
		case PqMsg_Bind:
		case PqMsg_Parse:
			maxmsglen = PQ_LARGE_MESSAGE_LIMIT;
			doing_extended_query_message = true;
			break;
		...
		case PqMsg_Sync:
			maxmsglen = PQ_SMALL_MESSAGE_LIMIT;
			/* stop any active skip-till-Sync */
			ignore_till_sync = false;
			/* mark not-extended, so that a new error doesn't begin skip */
			doing_extended_query_message = false;
			break;
		...
		default:

			/*
			 * Otherwise we got garbage from the frontend.  We treat this as
			 * fatal because we have probably lost message boundary sync, and
			 * there's no good way to recover.
			 */
			ereport(FATAL,
					(errcode(ERRCODE_PROTOCOL_VIOLATION),
					 errmsg("invalid frontend message type %d", qtype)));
			maxmsglen = 0;		/* keep compiler quiet */
			break;
	}

	/*
	 * In protocol version 3, all frontend messages have a length word next
	 * after the type code; we can read the message contents independently of
	 * the type.
	 */
	if (pq_getmessage(inBuf, maxmsglen))
		return EOF;				/* suitable message already logged */
	RESUME_CANCEL_INTERRUPTS();

	return qtype;
```

三个设计点：**先验证类型再信任长度**（注释：失步时"说命令未知"好过"拿垃圾当长度字把内存耗尽"，并按类型选择合理的长度上限）；`doing_extended_query_message` 与 `ignore_till_sync` 这两个协议状态标志**在读报文时就地维护**——4.2 节恢复块里 `if (doing_extended_query_message) ignore_till_sync = true` 的另一半就在这里（Sync 到来时解除跳过）；读消息期间 HOLD_CANCEL_INTERRUPTS（取消不能把读了一半的报文丢在缓冲区里，否则失步）。

### 7.2 扩展协议的三个阶段

扩展协议把 exec_simple_query 一体化的流水线**拆成了三个可独立寻址的阶段**：

**exec_parse_message（'P'，postgres.c:1421）**

- 只做 parse + analyze + rewrite，产物包成 `CachedPlanSource`（`CreateCachedPlan` 调用在 postgres.c:1533/1564）；**不做 plan**——规划推迟到 Bind，因为好的计划可能依赖参数值。
- SQL 里可带 `$1, $2...` 占位符，参数类型可由客户端指定或由分析器推断。
- 有名语句 `StorePreparedStatement()`（postgres.c:1597）存进会话哈希表（≈ MySQL 的 `PREPARE stmt FROM ...`）；无名语句存 `unnamed_stmt_psrc`（postgres.c:1605），被下一个无名 Parse 覆盖——驱动（如 JDBC）单次参数化查询用的就是无名语句。
- 回 `ParseComplete`（postgres.c:1621）。

**exec_bind_message（'B'，postgres.c:1662）**

- 找到语句（`FetchPreparedStatement`，postgres.c:1699）→ 按声明类型解码参数值 → **`GetCachedPlan(psrc, params, NULL, NULL)`**（postgres.c:2054）→ `PortalDefineQuery` + `PortalStart` → 回 `BindComplete`（postgres.c:2105）。
- `GetCachedPlan` 是计划缓存的心脏：它决定返回**通用计划**（generic plan，忽略具体参数、可跨执行复用）还是**定制计划**（custom plan，按本次参数值优化）。默认策略：前 5 次用定制计划，若通用计划的估算代价不明显更差则切换（逻辑在 `src/backend/utils/cache/plancache.c`，`plan_cache_mode` GUC 可强制）。这是 PG 对"参数敏感计划"问题的答案——MySQL 的 prepared statement 每次执行都重新优化，没有这层缓存权衡。

**exec_execute_message（'E'，postgres.c:2151）**

- `GetPortalByName()`（postgres.c:2177）→ `PortalRun(portal, max_rows, ...)`（postgres.c:2320）。
- 与简单协议最大的执行期差异：**max_rows 可以非零**——只取 N 行后挂起 portal，回 `PortalSuspended`（postgres.c:2381）；客户端可对同一 portal 再发 'E' 从断点继续。这是 JDBC `setFetchSize()` 的服务端机制，等价于协议级游标。

### 7.3 对比总表

| 维度 | 简单协议 'Q' | 扩展协议 'P'/'B'/'E' |
|---|---|---|
| 入口 | `exec_simple_query`（postgres.c:1030） | `exec_parse/bind/execute_message`（postgres.c:1421/1662/2151） |
| 解析/规划 | 每次全量执行 | Parse 一次，Plan 在 Bind 期（可复用 generic plan） |
| 参数 | 无，只能字符串拼接 | 类型化参数、可二进制传输（无注入风险、免转义开销） |
| 结果集 | 一次取完（FETCH_ALL） | 可分批（max_rows + PortalSuspended） |
| 一报文多语句 | 允许，隐式事务块 | Parse 只接受单条语句 |
| ReadyForQuery | 每条 'Q' 后立即发 | 直到 'S'（Sync）才发 → 支持 pipelining |
| 错误恢复 | 下一条 'Q' 照常 | `ignore_till_sync` 丢弃报文直到 Sync（postgres.c:4663/4925） |
| MySQL 对应 | COM_QUERY | COM_STMT_PREPARE / COM_STMT_EXECUTE |

**三问：为什么错误后要 skip-till-Sync？** ①解决什么问题：pipelining 允许客户端连发多组 Parse/Bind/Execute 不等回应；若第 2 组的 Bind 出错而服务端继续执行第 3 组，客户端对"哪些成功了"的认知将与服务端失步。②为何选此方案：Sync 是客户端亲手画的"错误恢复边界"——出错后丢弃一切直到 Sync，然后 ReadyForQuery 重新对表，简单且无歧义；替代方案（给每条报文编号+选择性重传）是把 TCP 重新发明一遍。③不变式：`doing_extended_query_message` 必须在读到报文的第一时间被正确设置（SocketBackend 里做，7.1 节），否则恢复块无从判断该不该进入跳过模式。

**小结**：扩展协议 = 把流水线切成 Parse（缓存分析产物）、Bind（参数化+择机规划）、Execute（可分批执行）三段，用 Sync 定界事务与错误恢复。换来的三样东西——计划复用、二进制参数、流水线——正是 OLTP 高并发场景的刚需。

## 自测问题（附详解）

**1. postmaster 收到 SIGCHLD 后，为什么不在信号 handler 里直接收尸，而只是置一个标志？**

因为收尸涉及 waitpid 循环、ereport（内部 malloc）、状态机迁移、甚至启动新进程，这些函数不是 async-signal-safe——handler 可能恰好打断主流程的 malloc，重入堆分配器轻则死锁重则堆损坏，而 postmaster 崩溃意味着整棵进程树失去看门人。所以 `handle_pm_child_exit_signal` 只置 `pending_pm_child_exit` 并 SetLatch，实际处理延迟到 ServerLoop 里的 `process_pm_child_exit()`（postmaster.c:2261）。这套 handle_/process_ 两段式是 commit 7389aad636（2023）重构的成果（此前 postmaster 真的在 handler 里 fork）。

**2. 一个 backend 因段错误被 SIGSEGV 杀死，另一个 backend 只是执行了违反唯一约束的 INSERT，两者后果有何不同？为什么？**

前者是进程异常死亡：postmaster 在 `HandleChildCrash`（postmaster.c:2818）里假定共享内存可能已被死者污染，经 `HandleFatalError` 向所有子进程广播 SIGQUIT（quickdie 直接 _exit(2)），随后在 PostmasterStateMachine 里重建共享内存、重走 WAL 崩溃恢复——全库瞬断。后者只是 `ereport(ERROR)`：siglongjmp 回 PostgresMain 恢复块，`AbortCurrentTransaction()` 回滚本事务，连接与其他会话毫发无损。区别的根源：前者可能破坏**共享**状态且无从审计；后者的破坏被事务机制完全兜住。这就是"三档灾难半径"里 ERROR 与 PANIC/crash 的分界。

**3. `ereport(ERROR, ...)` 之后控制流是怎样一步步回到主循环的？清理靠什么完成？**

`ereport` 宏先 `errstart(ERROR, ...)`（elog.c:355，可能升级为 FATAL/PANIC），参数求值填充错误栈帧，然后 `errfinish()`（elog.c:486）。对 elevel==ERROR，errfinish 做最小清理（清 InterruptHoldoffCount 等）后 `PG_RE_THROW()` → `pg_re_throw()`（elog.c:2200）→ `siglongjmp(*PG_exception_stack, 1)`。`PG_exception_stack` 若未被 PG_TRY 覆盖，则指向 PostgresMain 的 `local_sigjmp_buf`（postgres.c:4687），控制流落到 postgres.c:4573 的恢复块：EmitErrorReport 输出错误，`AbortCurrentTransaction()` 批量清算。清理不靠栈展开，而靠集中记账：内存挂上下文树、锁记锁表、pin 等记 ResourceOwner——"先记账后用资源"的纪律保证任意位置抛错都无泄漏。

**4. 为什么 PostgresMain 的链底错误处理不用 PG_TRY 宏，而手写 sigsetjmp？**

看 PG_CATCH 的展开（elog.h:398）：一进 CATCH 就把 `PG_exception_stack` 还原成**外层**保存值——对链底来说外层是 NULL，于是 CATCH（即清理代码）执行期间任何 ereport(ERROR) 都会因无 handler 被升级为 FATAL，"错误恢复中的错误"直接杀连接。手写方式让链底 setjmp **永远在位**：清理中再出错就再跳回来一次，真的死循环由 elog.c 的 recursion_depth 检查拦下（elog.c:432）。官方注释（postgres.c:4557-4563）原文解释了这一点。

**5. 为什么 exec_simple_query 在语义分析之前就要检查 `IsAbortedTransactionBlockState()`？纯语法分析为什么不用检查？**

语义分析、重写、规划都要读系统目录，而目录访问需要正常的事务环境——在已中止的事务里会再次失败（postgres.c:1150-1157 注释）。所以除 COMMIT/ROLLBACK 外的语句必须在进入这些阶段前被拒绝。而 `pg_parse_query` 是纯 flex/bison 语法处理，不碰目录、不需要事务（postgres.c:1079-1081 注释明确说 "safe even if we are in aborted transaction state"），因此放在检查之前安全。

**6. JDBC 设置 `setFetchSize(100)` 后，服务端靠什么机制实现"每次只返回 100 行"？简单协议为什么做不到？**

扩展协议的 Execute 报文携带 max_rows：`exec_execute_message`（postgres.c:2151）调 `PortalRun(portal, max_rows, ...)`（postgres.c:2320）只取 100 行，portal 挂起并回 `PortalSuspended`（postgres.c:2381），客户端再发 'E' 从断点继续。简单协议里 `exec_simple_query` 对无名 portal 固定 `FETCH_ALL`（postgres.c:1297-1298），且执行完 portal 即 drop，没有可供二次寻址的执行状态；要分批只能用 SQL 层的 DECLARE CURSOR / FETCH。

**7. 空闲的 backend 为什么能瞬间响应 `pg_terminate_backend()`，而卡在某个第三方 C 函数里的 backend 却杀不动？**

SIGTERM 的 handler `die()`（postgres.c:3114）只置 ProcDiePending + SetLatch。空闲时 backend 阻塞在 ReadCommand，且主循环已置 `DoingCommandRead = true`：latch 唤醒等待后 `ProcessClientReadInterrupt(true)`（postgres.c:514）看到 DoingCommandRead 就直接 `CHECK_FOR_INTERRUPTS()` → FATAL 退出——空闲态没有半成品状态，随时可死。执行查询期间则必须等代码跑到下一个 `CHECK_FOR_INTERRUPTS()` 采样点；不含采样点的第三方函数（或死循环的 C 扩展）永远采不到样，SIGTERM 就"失效"了，最后手段只剩 kill -9——代价是触发全库 crash-restart（推演 ①）。

**8. quickdie 为什么用 `_exit(2)` 而不是 `proc_exit(0)`？为什么敢在信号 handler 里 ereport？**

`_exit(2)`：一不跑 proc_exit/atexit 回调——共享内存可能已损坏，任何"清理"（比如把 PGPROC 从 ProcArray 摘除，这要拿 LWLock）都可能踩雷或死锁（commit 8e19a82640 正是为此把 exit 改成 _exit）；二退出码 2 不在 0/1 的"正常退出"白名单里，postmaster 会把它当 crash——这是自洽的：既然没清理共享状态，就必须让 postmaster 走重置流程。handler 里 ereport 确实不是 async-signal-safe，注释坦承 "It's dubious ... But it seems better to try"：给客户端一句"另一个进程崩了，马上可以重连"的解释值得冒险，卡死了也有 postmaster 的 SIGKILL 兜底（ServerLoop 段三）。

**9. 为什么 `shared_preload_libraries` 里的扩展能拥有自己的共享内存分区，而 `LOAD` 进来的不能？**

共享内存在 `CreateSharedMemoryAndSemaphores()`（postmaster.c:1018）一次性定形，之后永不扩容（1.2 节⑤的三问）。`process_shared_preload_libraries()`（postmaster.c:936）在此**之前**执行，扩展由此得以通过 shmem request 回调登记尺寸需求（postmaster.c:976 统一收集）。`LOAD` 发生在某个 backend 的运行期，共享内存早已定形，无处安放新分区——这也是 pg_stat_statements 必须重启才能启用的原因。

**10. FAST shutdown（SIGINT）时，一个 backend 正在读启动包、另一个正在跑长查询，它们分别怎么退出？**

正在读启动包的：BackendInitialize 阶段的 SIGTERM handler 是 `process_startup_packet_die`（backend_startup.c:196/989），直接 `_exit(1)`——合法是因为此阶段有"未碰共享内存"的不变式（backend_startup.c:186-195 注释），进程外部无任何需要清理的状态，还有 `check_on_shmem_exit_lists_are_empty()`（backend_startup.c:365）在守护。正在跑长查询的：PostgresMain 阶段 SIGTERM→`die()` 只置标志，查询代码在下一个 CHECK_FOR_INTERRUPTS 采样点走 `ereport(FATAL)` → errfinish → `proc_exit(1)` 回调链完成全套清理（还 PGPROC、放锁、写统计）。同一信号、两个阶段、两种退出方式，分界线就是"是否已触碰共享内存"。
