## 本章导读

一条 `SELECT * FROM t WHERE id = 42;` 到达 backend 进程后，在被优化器接手之前，要经历三次"变形"：

```
SQL 字符串
  └─ pg_parse_query()                        src/backend/tcop/postgres.c:617
      └─ raw_parser()                        src/backend/parser/parser.c:42
          ├─ 词法分析  scan.l   (Flex)   → token 流（中间隔着 base_yylex 过滤器）
          └─ 语法分析  gram.y   (Bison)  → 原始解析树（SelectStmt 等）
  └─ pg_analyze_and_rewrite_fixedparams()    src/backend/tcop/postgres.c:683
      ├─ parse_analyze_fixedparams()         src/backend/parser/analyze.c:128
      │   └─ transformStmt()                 analyze.c:335  → Query（查询树）
      └─ pg_rewrite_query()                  postgres.c:816
          └─ QueryRewrite()                  src/backend/rewrite/rewriteHandler.c:4789
              → 展开视图、应用规则、注入 RLS → List<Query>
```

三个阶段的分工要先记牢：

| 阶段 | 输入 → 输出 | 是否查系统目录 | 教科书对应 |
|---|---|---|---|
| 词法+语法分析 | 字符串 → raw parse tree | **否**（纯语法，不碰任何表） | lexing / parsing |
| 语义分析 | raw parse tree → `Query` | **是**（表名→OID、类型推导、重载消解） | semantic analysis / binding |
| 重写 | `Query` → `List<Query>` | 是（读 pg_rewrite、pg_policy） | view inlining / query rewrite |

**为什么 raw parse tree 与 Query 必须分成两棵树？** 这是本章最重要的一条架构不变式，值得用三问的方式回答：

1. **解决什么问题**：PREPARE 语句、PL/pgSQL 函数体、扩展查询协议中的命名语句都需要长期缓存"这条 SQL 的解析结果"。缓存什么？如果缓存 `Query`，那么其中的表 OID、列号、操作符 OID 在任何一次 DDL（`ALTER TABLE ... ADD COLUMN`、`DROP INDEX`、甚至 `ANALYZE` 触发的重建）之后都可能失效——缓存命中率将惨不忍睹，且失效检测本身就要遍历全树。
2. **为何选此方案而非替代**：MySQL 8.0 之前的做法接近"一棵树走到底"（解析与 resolve 交织），代价是 prepared statement 在表结构变化后必须整体重新 prepare，且解析代码与目录访问深度耦合。PG 的两棵树方案让 raw tree 成为**与目录状态无关的纯语法产物**——表名还是字符串、类型还是名字——因此可以无限期缓存；目录一变，扔掉 Query、拿缓存的 raw tree 重跑一遍语义分析即可（plancache 正是这样做的）。
3. **正确性不变式**：raw tree 中不允许出现任何 OID（gram.y 的语义动作不查目录，第 2 节验证）；Query 中不允许出现任何未解析的名字字符串（表→`relid`，列→`Var(varno,varattno)`，第 3 节验证）。两个世界的边界就是 `transformStmt()`。

另外注意 DDL 的特殊待遇：DDL 语句只有 raw tree 形态，`transformStmt()` 的 default 分支直接把它包进 `CMD_UTILITY` 的 Query（analyze.c:435-444），不做语义分析、不重写、不优化，直送 `ProcessUtility`。"PG 的查询处理管线只服务 DML"这句话的出处就在这里。

## 前置知识

- **编译原理**：了解正则表达式驱动的词法分析（Flex）、LALR(1) 语法分析（Bison）的基本概念即可，本文会在用到处解释。关键概念只有一个：LALR(1) 的 "1" 意味着解析器在每一步**只能偷看一个 token** 就必须决定动作；
- **关系代数**：选择 σ、投影 π、连接 ⋈——`Query` 结构体就是这三个算子的具体化，第 4 节详述；
- **MySQL 对照**：MySQL 8.0 起逐步用手写递归下降替换 Bison 解析器，PG 则坚持 Flex+Bison 生成，第 2.5 节讨论这一取舍；本文多处以 MySQL 行为作参照系（隐式类型转换、WHERE 中引用别名等）；
- **阅读工具**：打开 GUC `debug_print_parse = on`，日志里会打印语义分析后的 Query 树；调试时对任意节点指针执行 `call pprint(ptr)` 同样有效。想看重写后的树用 `debug_print_rewritten = on`。

## 第 1 节 · 词法分析：scan.l

`src/backend/parser/scan.l`（1430 行）由 Flex 编译成 `scan.c`，导出核心函数 `core_yylex()`。它把输入字符串切成 token：关键字、标识符、常量、操作符、标点。

### 1.1 Flex 规则结构：排他状态机

Flex 的匹配规则是"最长匹配优先，等长时先定义者优先"（scan.l:148-150 的注释原话：*The longest pattern which matches an input string is always chosen. For equal-length patterns, the first occurring in the rules list is chosen.*）。scan.l 的组织方式是：**用排他状态（exclusive state）处理所有"内部语法与外面不同"的片段**。状态声明在 scan.l:174-184：

```
%x xb        位串字面量 B'0101'
%x xc        扩展 C 风格注释 /* ... */（支持嵌套！）
%x xd        双引号定界标识符 "MyTable"
%x xh        十六进制串 X'FF'
%x xq        标准单引号字符串
%x xqs       引号结束后的"续接检测"状态
%x xe        扩展字符串 E'\n'（支持反斜杠转义）
%x xdolq     美元引用 $tag$ ... $tag$
%x xui/xus   带 Unicode 转义的标识符/字符串 U&"..." / U&'...'
%x xeu       扩展字符串中的 Unicode 代理对
```

**三问**：①解决什么问题——若不用状态机，就得在一条正则里同时描述"字符串里的引号转义、注释里的注释、美元引用里的任意字符"，正则会组合爆炸且极难验证；②为何选此方案——Flex 原生支持 `%x` 排他状态，进入 `xq` 后 `INITIAL` 的全部规则失效，直到 `BEGIN(INITIAL)`，等价于"每种子语言一台独立的小状态机"；③不变式——任何状态退出时必须恢复 `INITIAL` 或报"unterminated string"类错误，不存在"带着半开状态返回 token"的路径。

三个值得走读的细节：

**(1) 嵌套注释。** SQL 标准不要求 `/* /* */ */` 合法，但 PG 支持。实现是在 `xc` 状态里维护深度计数器（scan.l:447-466）：

```c
{xcstart}		{                       /* 在 INITIAL 状态遇到斜杠星 */
        SET_YYLLOC();
        yyextra->xcdepth = 0;           /* 嵌套深度清零 */
        BEGIN(xc);                      /* 切入注释状态 */
        yyless(2);                      /* 只消费两个字符，防止 "/*/" 之类误吞 */
    }
<xc>{
{xcstart}		{ (yyextra->xcdepth)++; yyless(2); }   /* 注释里又见注释：深度+1 */
{xcstop}		{
        if (yyextra->xcdepth <= 0)
            BEGIN(INITIAL);             /* 最外层注释结束，回到正常状态 */
        else
            (yyextra->xcdepth)--;       /* 只是内层结束 */
    }
```

**(2) 美元引用（dollar quoting）**。`$func$ ... $func$` 是 PG 独有语法，专为"字符串里还有字符串"的场景（写函数体）设计：`xdolq` 状态记住开头的 tag，只在遇到**完全相同**的 tag 时才结束，中间的引号、反斜杠一概不解释。没有它，PL/pgSQL 函数体里每层字符串都要引号翻倍（`''''`），四层嵌套就没法读了。

**(3) 位置追踪**。每条规则开头都调用 `SET_YYLLOC()`（scan.l:98）：

```c
#define SET_YYLLOC()  (*(yylloc) = yytext - yyextra->scanbuf)
```

它把 token 在原始字符串中的**字节偏移**存入 `yylloc`，一路带进 raw tree 各节点的 `location` 字段、再进 `Query`。你在 psql 里看到报错信息下方那个精确指向出错位置的 `^`，就靠这条从词法层贯穿到语义层的位置链（`parser_errposition()`）。注意存的是偏移而不是行列号：行列号在报错时才按需计算，正常路径零开销。

### 1.2 关键字处理：kwlist.h 的单一数据源 + 完美哈希

PG 有近 500 个关键字。它们**只在一个文件里定义一次**：`src/include/parser/kwlist.h`。该文件通篇是这样的宏调用（按 ASCII 排序）：

```c
/* name, value, category, is-bare-label */
PG_KEYWORD("abort", ABORT_P, UNRESERVED_KEYWORD, BARE_LABEL)        /* kwlist.h:28 */
PG_KEYWORD("between", BETWEEN, COL_NAME_KEYWORD, BARE_LABEL)        /* kwlist.h:60 */
PG_KEYWORD("like", LIKE, TYPE_FUNC_NAME_KEYWORD, BARE_LABEL)        /* kwlist.h:266 */
PG_KEYWORD("select", SELECT, RESERVED_KEYWORD, BARE_LABEL)          /* kwlist.h:418 */
```

`PG_KEYWORD` 宏本身**不在此文件定义**——每个使用方自带定义再 `#include`，同一份数据被展开成不同形态：词法器要"名字→token 号"的查找表，gram.y 要 token 声明，`pg_get_keywords()` 系统函数要展示列表。这是 C 语言的 "X-Macro" 惯用法，保证关键字表永不失步。

**设计史①：完美哈希的引入。** 关键字查找曾长期是二分查找。2019 年 1 月，Tom Lane 先以 commit `afb0d0712f`（2019-01-06, *Replace the data structure used for keyword lookup*）把"字符串指针数组"压缩为"单一字符串池+偏移数组"（省内存、省启动重定位），三天后以 commit `c64d0cd5ce`（2019-01-09, *Use perfect hashing, instead of binary search, for keyword lookup*）改为构建期生成**完美哈希函数**。提交信息记录了动机与收益：社区多年猜测哈希应比二分快但一直没有合适的生成工具，Joerg Sonnenberger 提供了灵感与样例代码；代价是每关键字约 4 字节的查找表（比前一个 commit 省下的还少），收益是**raw parsing（flex+bison 阶段）总耗时下降约 20%**。生成器是 `src/tools/gen_keywordlist.pl`（内部用 PerfectHash.pm），查找入口在 `ScanKeywordLookup()`（src/common/kwlookup.c:38）：

```c
len = strlen(str);
if (len > keywords->max_kw_len)         /* 比最长关键字还长？直接排除 */
    return -1;
h = keywords->hash(str, len);           /* 完美哈希：无冲突，一次命中 */
if (h < 0 || h >= keywords->num_keywords)
    return -1;
kw = GetScanKeyword(h, keywords);
while (*str != '\0')                    /* 逐字符比对，同时做 ASCII-only 小写化 */
{
    char  ch = *str++;
    if (ch >= 'A' && ch <= 'Z')
        ch += 'a' - 'A';                /* 故意不用 tolower()：土耳其语等 locale 里
                                         * 'I' 的小写不是 'i'，会破坏 SQL99 语义 */
    if (ch != *kw++)
        return -1;
}
```

词法器的标识符规则（scan.l:1054 起）先查这张表，查不到才降级为普通标识符：是关键字则返回 gram.y 对应的 token 号；不是则 `downcase_truncate_identifier()` 转小写、截断到 `NAMEDATALEN-1`，返回 `IDENT`。**PG "标识符不加双引号即折叠为小写"的行为源头就在这里**（MySQL 在多数平台上表名大小写敏感与否取决于文件系统，是完全不同的设计）。

关键字分为四个**保留级别**（kwlist.h 第三列），对应 gram.y 末尾的四个产生式：`unreserved_keyword`（gram.y:18898，可直接当表名列名用）、`col_name_keyword`（gram.y:19257，可当列名不可当函数名）、`type_func_name_keyword`（gram.y:19334）、`reserved_keyword`（gram.y:19366，完全保留）。这就是为什么 `CREATE TABLE abort(...)` 合法而 `CREATE TABLE select(...)` 报错——PG 尽量把关键字降级为非保留，减少与用户命名的冲突；"关键字当标识符用"由这些产生式变成**语法的一部分**，而非词法特判。第四列 `BARE_LABEL` 是 PG 14 起的补充维度：标记该关键字能否作为 `SELECT expr keyword` 形式的裸列别名（不写 AS）。

### 1.3 小结

- scan.l 用**排他状态**隔离字符串/注释/位串等子语言，嵌套注释靠深度计数器；
- 关键字是"单一数据源 kwlist.h + X-Macro + 构建期完美哈希"，查找 O(1) 且各组件永不失步；完美哈希由 `c64d0cd5ce` 引入，raw parsing 提速约 20%；
- 大小写折叠是 ASCII-only 的（SQL99 要求），标识符统一转小写；
- 每个 token 携带字节偏移 `location`，支撑贯穿全程的精确错误定位。

## 第 2 节 · 语法分析：gram.y 与 raw parse tree

`src/backend/parser/gram.y` 有 21016 行，是 PG 最大的单个源文件，由 Bison 编译成 LALR(1) 状态机。语义动作只做一件事：**把 token 组装成 C 结构体**（raw parse tree），不查目录、不做类型检查。

### 2.1 入口：raw_parser 与 RawParseMode

`raw_parser()`（parser.c:42-86）全函数只有五步：初始化 Flex 扫描器（把 `ScanKeywords`/`ScanKeywordTokens` 传进去）、设置前瞻状态、初始化 Bison、`base_yyparse()`、清理返回。值得走读的是第二步（parser.c:52-71）：

```c
/* base_yylex() only needs us to initialize the lookahead token, if any */
if (mode == RAW_PARSE_DEFAULT)
    yyextra.have_lookahead = false;
else
{
    /* this array is indexed by RawParseMode enum */
    static const int mode_token[] = {
        [RAW_PARSE_DEFAULT] = 0,
        [RAW_PARSE_TYPE_NAME] = MODE_TYPE_NAME,
        [RAW_PARSE_PLPGSQL_EXPR] = MODE_PLPGSQL_EXPR,
        [RAW_PARSE_PLPGSQL_ASSIGN1] = MODE_PLPGSQL_ASSIGN1,
        [RAW_PARSE_PLPGSQL_ASSIGN2] = MODE_PLPGSQL_ASSIGN2,
        [RAW_PARSE_PLPGSQL_ASSIGN3] = MODE_PLPGSQL_ASSIGN3,
    };

    yyextra.have_lookahead = true;
    yyextra.lookahead_token = mode_token[mode];
    ...
}
```

`RawParseMode` 把一个**虚构 token 注入 token 流头部**，让 gram.y 的起始产生式据此分派到"只解析一个类型名/一个表达式/一个赋值语句"的入口。PL/pgSQL 因此得以**复用主语法**解析函数体里的表达式，而不需要维护第二份表达式文法——这是"一份文法、多个入口"的典型手法。

### 2.2 base_yylex：token 过滤器完整走读

Bison 拿到的 token 并非直接来自 `core_yylex()`，中间隔着过滤器 `base_yylex()`（parser.c:110）。它存在的原因写在注释里（parser.c:92-94）：

> *This filter is needed because in some cases the standard SQL grammar requires more than one token lookahead. We reduce these cases to one-token lookahead by replacing tokens here, in order to keep the grammar LALR(1).*

**三问·①没有它会怎样？** 举两个具体 SQL：

- `x NOT BETWEEN 1 AND 2`：解析器读到 `NOT` 时必须决定——它是逻辑非（`NOT (x)` 前缀，%right 高结合）还是 `NOT BETWEEN` 复合操作符的一部分（应取 BETWEEN 的低优先级）？答案取决于**下一个** token 是不是 BETWEEN/IN/LIKE/ILIKE/SIMILAR，而 LALR(1) 只许看一个。若强行在文法里写 `a_expr: a_expr NOT BETWEEN ...`，Bison 会在 `a_expr . NOT` 这一状态上报 shift/reduce 冲突：`SELECT a NOT` 后面既可能是 `BETWEEN 1 AND 2` 也可能是……什么都不可能（裸 NOT 不能后置），但 `NOT b` 作为独立表达式的产生式使状态机无法区分两种前缀。
- `SELECT nulls FROM t` vs `ORDER BY x NULLS FIRST`：`nulls` 是非保留关键字，可以当列名；读到 `NULLS` 时，只有看到下一个 token 是否 FIRST/LAST 才知道它是排序修饰还是标识符。同理 `WITH TIME ZONE` vs `WITH cte AS (...)`、`AT TIME ZONE`、`FORMAT JSON`。

**②为何选过滤器而非其他方案？** 备选有三：(a) 让 Flex 直接识别多词 token（`"NOT"[ \t]+"BETWEEN"`）——注释（parser.c:96-100）明确否决：两词之间可以有**注释**，正则写不下；且会重新引入 scanner 回溯，性能代价比过滤层更高。(b) 换 GLR/LR(2) 解析器——Bison 的 GLR 模式在歧义时性能不可控，且丢失"零冲突"的静态保证。(c) 忍受冲突用 %prec 硬压——会掩盖其他真实歧义。过滤器把复杂度收敛在一个 160 行的函数里，是局部性最好的方案。

**③正确性不变式**：过滤器合成的 `NOT_LA`、`WITH_LA`、`NULLS_LA`、`FORMAT_LA`、`WITHOUT_LA`（gram.y:864 声明）**词法器永远不会自然产出**，因此文法里这些 token 的出现点完全受控；且过滤器必须精确维护 `yylloc` 与 scanbuf 中被临时写入的 `'\0'`（parser.c:163-190 的 save/restore 舞蹈），保证报错位置指向当前 token 而非偷看的下一个。

主干走读（parser.c:110-251，省略 UIDENT/USCONST 的三 token 前瞻分支——那是 U&'...' UESCAPE 语法专用，逻辑同构）：

```c
/* 第 1 步：取当前 token——可能上次已偷看过存着 */
if (yyextra->have_lookahead)
    { cur_token = yyextra->lookahead_token; ... yyextra->have_lookahead = false; }
else
    cur_token = core_yylex(&(lvalp->core_yystype), llocp, yyscanner);

/* 第 2 步：不是"需要前瞻"的 token 就直接返回；是则硬编码其长度 */
switch (cur_token)
{
    case FORMAT:  cur_token_length = 6; break;
    case NOT:     cur_token_length = 3; break;
    case NULLS_P: cur_token_length = 5; break;
    case WITH:    cur_token_length = 4; break;
    ...
    default: return cur_token;          /* 绝大多数 token 走这条快路 */
}

/* 第 3 步：偷看下一个 token（处理好 yylloc 与 '\0' 的保存恢复） */
next_token = core_yylex(&(yyextra->lookahead_yylval), llocp, yyscanner);
...
yyextra->have_lookahead = true;

/* 第 4 步：按组合替换当前 token */
switch (cur_token)
{
    case NOT:      /* NOT + BETWEEN/IN/LIKE/ILIKE/SIMILAR → NOT_LA */
    case NULLS_P:  /* NULLS + FIRST/LAST → NULLS_LA */
    case WITH:     /* WITH + TIME/ORDINALITY → WITH_LA */
    ...
}
```

`NOT_LA` 的用途很精妙：让 `a NOT LIKE b` 中的 NOT 拿到 LIKE 的优先级而不是逻辑 NOT 的优先级（gram.y:16009 的 `a_expr NOT_LA LIKE a_expr %prec NOT_LA`）。

### 2.3 SelectStmt 产生式走读

SELECT 语法被拆成三层，是阅读 gram.y 的最佳入口：

```
SelectStmt        (gram.y:13639)  顶层
 ├─ select_no_parens  (gram.y:13659)  不带外层括号：负责挂 ORDER BY/LIMIT/WITH/FOR UPDATE
 └─ select_with_parens (gram.y:13643) 带括号：'(' select_no_parens ')'
      select_no_parens → simple_select | select_clause sort_clause | ...
       └─ simple_select (gram.y:13751) 核心：SELECT ... FROM ... WHERE ...，
                                       以及 UNION/INTERSECT/EXCEPT、VALUES、TABLE t
```

`simple_select` 第一条产生式的语义动作（gram.y:13751-13767，逐字）：

```c
simple_select:
			SELECT opt_all_clause opt_target_list
			into_clause from_clause where_clause
			group_clause having_clause window_clause
				{
					SelectStmt *n = makeNode(SelectStmt);

					n->targetList = $3;
					n->intoClause = $4;
					n->fromClause = $5;
					n->whereClause = $6;
					n->groupClause = ($7)->list;
					n->groupDistinct = ($7)->distinct;
					n->havingClause = $8;
					n->windowClause = $9;
					$$ = (Node *) n;
				}
```

可以看到语义动作纯粹是"抄写员"：`$3`、`$5` 这些子产生式的结果被原样塞进字段，没有任何目录访问。集合操作也在这一层（gram.y:13805）：

```c
			| select_clause UNION set_quantifier select_clause
				{
					$$ = makeSetOp(SETOP_UNION, $3 == SET_QUANTIFIER_ALL, $1, $4);
				}
```

`SelectStmt` 结构体（parsenodes.h:2328）因此身兼两职：叶子节点表示一个普通 SELECT（用 `targetList`/`fromClause` 等字段），内部节点表示 UNION/INTERSECT/EXCEPT（用 `op`/`larg`/`rarg` 字段，parsenodes.h:2370-2373）。同一节点类型让 gram.y 挂 ORDER BY/LIMIT 时无需区分简单/复合 SELECT。

为什么 `ORDER BY` 只出现在 `select_no_parens` 而不在 `simple_select`？gram.y:13737-13742 的注释给出 SQL 标准语义（逐字）：

```
 * Note that sort clauses cannot be included at this level --- SQL requires
 *		SELECT foo UNION SELECT bar ORDER BY baz
 * to be parsed as
 *		(SELECT foo UNION SELECT bar) ORDER BY baz
```

把排序类子句提升到不带括号的最外层，正是**用语法结构编码这条语义规则**——想让 ORDER BY 只作用于 UNION 的一支？加括号即可，`select_with_parens` 分支会接住它。

### 2.4 %prec 与优先级：用声明消灭歧义

表达式语法 `a_expr`（gram.y:15913）如果朴素地写 `a_expr: a_expr '+' a_expr | a_expr '*' a_expr | ...`，会产生大量 shift/reduce 冲突。Bison 的解法是**优先级声明**（gram.y:881-954，从低到高，节选）：

```
%left    UNION EXCEPT          ← 集合操作优先级最低
%left    INTERSECT
%left    OR
%left    AND
%right   NOT
%nonassoc IS ISNULL NOTNULL
%nonassoc '<' '>' '=' LESS_EQUALS GREATER_EQUALS NOT_EQUALS
%nonassoc BETWEEN IN_P LIKE ILIKE SIMILAR NOT_LA
%left    '+' '-'
%left    '*' '/' '%'
%left    '^'
%right   UMINUS                ← 一元负号：并无这个 token，纯粹是"优先级占位符"
%left    '[' ']'
%left    '(' ')'
%left    TYPECAST              ← :: 转换优先级极高
%left    '.'
```

`UMINUS` 展示了 `%prec` 的经典用法（gram.y:15950）：

```c
			| '-' a_expr					%prec UMINUS
				{ $$ = doNegate($2, @1); }
```

`-` 作为二元减号优先级较低，但一元负号必须更高（`-a * b` 应解析为 `(-a) * b`）。`%prec UMINUS` 把**这一条产生式**的优先级改成 UMINUS 的，而不影响 token `-` 本身。gram.y 在优先级块附近的注释是一段方法论：给关键字全局设优先级"有全局副作用，可能掩盖别处的歧义"，因此应优先使用局部的 `%prec`；如果必须给关键字设优先级，尽量设成与 IDENT 相同。连 `SelectStmt: select_no_parens %prec UMINUS`（gram.y:13639）也用了 `%prec`——用于解决与圆括号（行构造器 vs 子查询）的歧义。

**设计史②：操作符优先级向标准看齐。** commit `c6b3c939b7`（2015-03-11, Tom Lane, *Make operator precedence follow the SQL standard more closely*，进入 9.5）调整了 `<=`/`>=`/`<>` 与 IS 测试的优先级——此前 `a < b IS TRUE` 之类表达式的解析结果与 SQL 标准相悖。这次调整是**用户可见的不兼容变更**，因此同时引入了 GUC `operator_precedence_warning` 帮用户找出受影响的查询（该 GUC 已在 14 移除）。这个案例说明：优先级表不是一次写死的数学真理，而是带着历史包袱、需要迁移策略的公共接口。

### 2.5 为什么 PG 坚持 LALR，而 MySQL 走向递归下降？

这是一个真实的工程取舍，两边都有充分理由：

| 维度 | PG（Bison/LALR） | MySQL（8.0 起逐步手写递归下降） |
|---|---|---|
| **歧义检测** | Bison 在构建期**证明**语法无冲突（PG 要求 0 shift/reduce 冲突）；改语法引入歧义会立即编译失败 | 手写解析器的歧义只能靠测试和 code review 发现 |
| **可维护性** | 语法即文档：gram.y 的产生式几乎能与 SQL 标准 BNF 对照 | 解析逻辑散落在函数调用里，与语义代码交织 |
| **错误恢复/提示** | 较弱：LALR 出错时只知道"此处不期望该 token"；PG 用 location + `^` 弥补 | 强项：可以在任意上下文写定制的错误提示与恢复 |
| **上下文相关处理** | 别扭：需要 base_yylex 过滤器、四级关键字、%prec 等技巧维持 LALR(1) | 天然容易：解析函数里随手 if 判断上下文 |
| **性能** | 表驱动状态机，缓存友好，常数小 | 调用栈深，但可针对热点手工优化 |

PG 的选择本质上是**把正确性证明交给工具**：SQL 语法极其庞大（近 500 关键字、21000 行产生式），社区每个版本都在加语法，"Bison 报冲突就不许合并"这条硬约束让几百人协作改同一个语法文件成为可能。具体的冲突处理成本对比：在 LALR 侧，一个上下文相关点的成本是"一段 base_yylex 分支 + 一个 _LA token + 若干 %prec 标注"，一次付清、静态验证；在递归下降侧，同样的点成本是"在恰当的解析函数里加 if"，写起来快，但每个后续改动者都要**重新理解**这个 if 与全局文法的交互，歧义回归只能靠测试兜底。MySQL 更看重报错体验与灵活性，愿意承担手写解析器的维护成本。没有对错，只有权衡——但如果你要向一个几十年生命周期、松散协作的项目学架构，PG 的选择更值得体会。

### 2.6 小结

- gram.y 产出的 raw parse tree 是"SQL 语句的忠实转写"：`SelectStmt` 的字段与 SQL 子句一一对应，不含任何目录信息；
- 一个 token 的前瞻不够时用 `base_yylex` 过滤器合成 `NOT_LA` 等虚构 token，维持 LALR(1)；备选方案（词法多词 token、GLR）分别因注释穿插与失去静态保证被否决；
- 歧义靠优先级声明与 `%prec` 局部标注解决，社区纪律是零冲突；优先级表本身也有版本演化（`c6b3c939b7`）；
- `RawParseMode` 通过注入首 token 让同一份文法服务 PL/pgSQL 等多个入口。

## 第 3 节 · 语义分析：analyze.c 与 parse_*.c 家族

语义分析是解析器里**真正有数据库味道**的部分：查目录、绑定名字、推导类型、消解重载。总入口 `parse_analyze_fixedparams()`（analyze.c:128）建立 `ParseState`（解析期的"工作台"，挂着范围表 `p_rtable`、名字空间 `p_namespace`、连接列表 `p_joinlist`、聚合标志等状态），然后调 `transformStmt()`。

### 3.1 transformStmt 走读

`transformStmt()`（analyze.c:335-452）是按语句类型分发的 switch（省略开头 `DEBUG_NODE_TESTS_ENABLED` 的覆盖率测试段，与语义无关）：

```c
switch (nodeTag(parseTree))
{
        /* Optimizable statements */
    case T_InsertStmt:  result = transformInsertStmt(pstate, (InsertStmt *) parseTree); break;
    case T_DeleteStmt:  result = transformDeleteStmt(pstate, (DeleteStmt *) parseTree); break;
    case T_UpdateStmt:  result = transformUpdateStmt(pstate, (UpdateStmt *) parseTree); break;
    case T_MergeStmt:   result = transformMergeStmt(pstate, (MergeStmt *) parseTree);   break;
    case T_SelectStmt:
        {
            SelectStmt *n = (SelectStmt *) parseTree;
            if (n->valuesLists)                 /* VALUES (...),(...) */
                result = transformValuesClause(pstate, n);
            else if (n->op == SETOP_NONE)       /* 普通 SELECT */
                result = transformSelectStmt(pstate, n, NULL);
            else                                /* UNION/INTERSECT/EXCEPT 树 */
                result = transformSetOperationStmt(pstate, n);
        }
        break;
    case T_ReturnStmt: ...      /* SQL 函数体的 RETURN */
    case T_PLAssignStmt: ...    /* PL/pgSQL 赋值语句 */
        /* Special cases：游标、EXPLAIN、CTAS、CALL——它们内嵌一条需分析的查询 */
    case T_DeclareCursorStmt: ... case T_ExplainStmt: ...
    case T_CreateTableAsStmt: ... case T_CallStmt: ...
    default:
        /* other statements don't require any transformation; just return
         * the original parsetree with a Query node plastered on top. */
        result = makeNode(Query);
        result->commandType = CMD_UTILITY;
        result->utilityStmt = parseTree;
        break;
}
/* Mark as original query until we learn differently */
result->querySource = QSRC_ORIGINAL;
result->canSetTag = true;
```

尾部两行（analyze.c:448-449）为重写器埋下伏笔：每条 Query 出生时都自认"原始查询、负责设置命令标签"，之后若被规则替换，`fireRules()` 会改写这两个字段（见 5.5 节）。函数上方的注释（analyze.c:364-367）还提醒：改这个 switch 时必须同步 `stmt_requires_parse_analysis()` 和 `analyze_requires_snapshot()`——语义分析要查目录，因此**需要快照**；纯 utility 不需要。

### 3.2 transformSelectStmt 全函数走读

`transformSelectStmt()`（analyze.c:1743-1885）是主战场。它的处理顺序本身就是 SQL 的名字可见性规则，下面按源码顺序分十步走读（代码为逐字摘录，穿插解说；仅省略 PLAssignStmt 专用的 `passthru` 分支内部——那是 PL/pgSQL 赋值语句借道 SELECT 语法的适配层，主线不受影响）。

**第①步：WITH 子句独立先行（analyze.c:1750-1758）**

```c
	qry->commandType = CMD_SELECT;

	/* process the WITH clause independently of all else */
	if (stmt->withClause)
	{
		qry->hasRecursive = stmt->withClause->recursive;
		qry->cteList = transformWithClause(pstate, stmt->withClause);
		qry->hasModifyingCTE = pstate->p_hasModifyingCTE;
	}
```

CTE 最先处理，因为 FROM 里可能引用它们；`transformWithClause` 会把每个 CTE 的子查询递归 `transformStmt`，并登记进 `pstate` 的 CTE 名字空间。

**第②步：拦截非法 INTO（analyze.c:1760-1766）**

```c
	/* Complain if we get called from someplace where INTO is not allowed */
	if (stmt->intoClause)
		ereport(ERROR,
				(errcode(ERRCODE_SYNTAX_ERROR),
				 errmsg("SELECT ... INTO is not allowed here"),
				 parser_errposition(pstate,
									exprLocation((Node *) stmt->intoClause))));
```

`SELECT INTO` 只在 CTAS 包装路径合法（transformStmt 之前已被 `transformOptionalSelectInto` 剥走，analyze.c:320 附近），走到这里还带着 INTO 就是语法位置错误。

**第③步：暂存上下文，处理 FROM（analyze.c:1768-1775）**

```c
	/* make FOR UPDATE/FOR SHARE info available to addRangeTableEntry */
	pstate->p_locking_clause = stmt->lockingClause;

	/* make WINDOW info available for window functions, too */
	pstate->p_windowdefs = stmt->windowClause;

	/* process the FROM clause */
	transformFromClause(pstate, stmt->fromClause);
```

注意 `p_locking_clause` 必须在 FROM 之前挂好：`addRangeTableEntry` 要据此决定对表加 `RowShareLock` 还是 `AccessShareLock`（3.4 节）。FROM 处理完后，`pstate->p_rtable`（范围表）与 `p_namespace`（可见名字）就绪，后续所有列名解析都在这张台面上进行。

**第④步：投影列表（analyze.c:1777-1790）**

```c
	/* transform targetlist */
	qry->targetList = transformTargetList(pstate, stmt->targetList,
										  EXPR_KIND_SELECT_TARGET);
	...
	else
		markTargetListOrigins(pstate, qry->targetList);
```

`*` 在这里展开成具体列（靠 RTE 的 `eref` 列名表）；`markTargetListOrigins` 给每个输出列标注来源表列，供 JDBC 驱动之类的元数据 API 使用。

**第⑤步：WHERE 与 HAVING（analyze.c:1792-1798）**

```c
	/* transform WHERE */
	qual = transformWhereClause(pstate, stmt->whereClause,
								EXPR_KIND_WHERE, "WHERE");

	/* initial processing of HAVING clause is much like WHERE clause */
	qry->havingQual = transformWhereClause(pstate, stmt->havingClause,
										   EXPR_KIND_HAVING, "HAVING");
```

`transformWhereClause` = `transformExpr` + `coerce_to_boolean`。注意 `qual` 是局部变量——WHERE 条件在 Query 里**没有自己的字段**，第⑨步它将与 FROM 合体。`EXPR_KIND_*` 枚举让同一套表达式变换代码能按语境拒绝非法结构（如 WHERE 里出现聚合会报 "aggregate functions are not allowed in WHERE"）。

**为什么 WHERE 不能引用 SELECT 别名？** `transformWhereClause` 解析列名只查 FROM 建立的 `p_namespace`，不查 targetList。你在 MySQL 里习惯的 `HAVING 别名` 在 PG 里同样不行（MySQL 对 HAVING 有非标准扩展）。这不是实现懒惰而是语义决定：`SELECT count(*) AS c ... WHERE c > 1` 若合法，就意味着先聚合后过滤行，因果倒置。

**第⑥步：ORDER BY 先于 GROUP BY（analyze.c:1800-1819）**

```c
	/*
	 * Transform sorting/grouping stuff.  Do ORDER BY first because both
	 * transformGroupClause and transformDistinctClause need the results. Note
	 * that these functions can also change the targetList, so it's passed to
	 * them by reference.
	 */
	qry->sortClause = transformSortClause(pstate,
										  stmt->sortClause,
										  &qry->targetList,
										  EXPR_KIND_ORDER_BY,
										  false /* allow SQL92 rules */ );

	qry->groupClause = transformGroupClause(pstate,
											stmt->groupClause,
											&qry->groupingSets,
											&qry->targetList,
											qry->sortClause,
											EXPR_KIND_GROUP_BY,
											false /* allow SQL92 rules */ );
	qry->groupDistinct = stmt->groupDistinct;
```

两个反直觉点：(a) 处理顺序与执行顺序相反——先 ORDER BY 后 GROUP BY，因为 GROUP BY 想尽量**复用** ORDER BY 已生成的 SortGroupClause（相同表达式共享 `tleSortGroupRef` 编号，优化器便可用一次排序同时满足两者）；(b) 两者都以 `&qry->targetList` 传引用——`ORDER BY 不在输出里的表达式` 会作为 `resjunk=true` 的隐藏列**追加**进 targetList（排序需要该值在元组里，但最终不输出）。

**第⑦步：DISTINCT 三分支（analyze.c:1821-1843）**——`NIL`（无）、`linitial == NULL`（普通 DISTINCT）、否则 DISTINCT ON。DISTINCT ON 是 PG 扩展，要求 ON 表达式必须是 sortClause 前缀，`transformDistinctOnClause` 在此强制。

**第⑧步：LIMIT 与窗口（analyze.c:1845-1861）**

```c
	qry->limitOffset = transformLimitClause(pstate, stmt->limitOffset,
											EXPR_KIND_OFFSET, "OFFSET",
											stmt->limitOption);
	qry->limitCount = transformLimitClause(pstate, stmt->limitCount,
										   EXPR_KIND_LIMIT, "LIMIT",
										   stmt->limitOption);
	qry->limitOption = stmt->limitOption;

	/* transform window clauses after we have seen all window functions */
	qry->windowClause = transformWindowDefinitions(pstate,
												   pstate->p_windowdefs,
												   &qry->targetList);

	/* resolve any still-unresolved output columns as being type text */
	if (pstate->p_resolve_unknowns)
		resolveTargetListUnknowns(pstate, qry->targetList);
```

窗口定义**最后**处理：`OVER w` 引用的命名窗口、targetList 里的窗口函数都要先收集齐。`resolveTargetListUnknowns` 是 unknown 类型的最终兜底：`SELECT 'abc'` 的输出列若到此仍无人认领类型，强制定为 text（3.7 节详述 unknown 的一生）。

**第⑨步：收尾装配——FROM 与 WHERE 合体（analyze.c:1863-1870）**

```c
	qry->rtable = pstate->p_rtable;
	qry->rteperminfos = pstate->p_rteperminfos;
	qry->jointree = makeFromExpr(pstate->p_joinlist, qual);

	qry->hasSubLinks = pstate->p_hasSubLinks;
	qry->hasWindowFuncs = pstate->p_hasWindowFuncs;
	qry->hasTargetSRFs = pstate->p_hasTargetSRFs;
	qry->hasAggs = pstate->p_hasAggs;
```

`makeFromExpr(p_joinlist, qual)` 这一行值得停一秒：**WHERE 条件作为 quals 并入 jointree**——这正是关系代数的视角：选择 σ 与连接 ⋈ 同属"从关系集合筛出元组"的操作（第 4 节展开）。一批 `has*` 标志从工作台抄进 Query，供后续阶段免于全树扫描。

**第⑩步：锁定子句、排序规则、聚合检查（analyze.c:1872-1884）**

```c
	foreach(l, stmt->lockingClause)
	{
		transformLockingClause(pstate, qry,
							   (LockingClause *) lfirst(l), false);
	}

	assign_query_collations(pstate, qry);

	/* this must be done after collations, for reliable comparison of exprs */
	if (pstate->p_hasAggs || qry->groupClause || qry->groupingSets || qry->havingQual)
		parseCheckAggregates(pstate, qry);
```

`parseCheckAggregates` 执行你熟悉的那条规则："SELECT 里的裸列必须出现在 GROUP BY 中"（MySQL 的 `ONLY_FULL_GROUP_BY` 对应物，PG 从不妥协）。它必须排在 collation 指派之后——判断"targetList 表达式是否与某 GROUP BY 表达式相同"要求两者的 collation 标注已一致。另外，自 commit `247dea89f7` 起，transformGroupClause 会为分组步骤建立一个 `RTE_GROUP` 范围表项并把分组表达式替换为指向它的 Var（见 4.3 节设计史）。

### 3.3 FROM 子句：transformFromClauseItem 全走读

`transformFromClause()`（src/backend/parser/parse_clause.c:116）遍历 FROM 项，每项交给 `transformFromClauseItem()`（parse_clause.c:1190）。这是解析器里最典型的递归函数：输入一个 FROM 项的 raw 节点，输出应挂进 jointree 的节点（`RangeTblRef` 或 `JoinExpr`），同时产出两个名字空间输出参数（函数头注释 parse_clause.c:1171-1188）。走读其全部分支：

**入口防线（parse_clause.c:1194-1195）**：`check_stack_depth();`——FROM 可以嵌套 JOIN 表达式任意深，先防栈溢出。

**分支 1：裸表/CTE 引用（1197-1216）**

```c
	if (IsA(n, RangeVar))
	{
		/* Plain relation reference, or perhaps a CTE reference */
		RangeVar   *rv = (RangeVar *) n;
		RangeTblRef *rtr;
		ParseNamespaceItem *nsitem;

		/* Check if it's a CTE or tuplestore reference */
		nsitem = getNSItemForSpecialRelationTypes(pstate, rv);

		/* if not found above, must be a table reference */
		if (!nsitem)
			nsitem = transformTableEntry(pstate, rv);

		*top_nsitem = nsitem;
		*namespace = list_make1(nsitem);
		rtr = makeNode(RangeTblRef);
		rtr->rtindex = nsitem->p_rtindex;
		return (Node *) rtr;
	}
```

名字解析的优先级藏在这里：**先查 CTE 与 ENR**（AFTER 触发器的过渡表等"临时命名元组存储"），查不到才当真实表处理（`getNSItemForSpecialRelationTypes`，parse_clause.c:1156-1157 还有个细节：带 schema 限定名的引用直接跳过 CTE 检查——`myschema.t` 不可能是 CTE）。这就是"同名 CTE 遮蔽真实表"行为的实现点。返回给 jointree 的只是 `RangeTblRef{rtindex}`——**jointree 从不直接持有 RTE，只持有下标**。

**分支 2-5：子查询、函数、表函数（XMLTABLE/JSON_TABLE）、图查询表（1217-1271）**：结构与分支 1 同构——各自调用 `transformRangeSubselect` / `transformRangeFunction` / `transformRangeTableFunc` / `transformRangeGraphTable` 建 RTE，包一个 RangeTblRef 返回。其中子查询分支值得点一句：FROM 里的子查询在这里就被**递归完整分析**成 Query 挂进 `RTE_SUBQUERY`，LATERAL 与否只影响它能看到的名字空间。（`RangeGraphTable` 是本源码树 20devel 携带的 SQL/PGQ 属性图特性，主线思想不变。）

**分支 6：TABLESAMPLE 包装（1272-1296）**：递归处理被包的关系，然后检查其 RTE 必须是普通表/物化视图/分区表，把采样信息挂到 `rte->tablesample`。省略细节，因为它只是"先递归再贴标签"。

**分支 7：JOIN 表达式——主线（1297 起）**。这是递归的核心，分八幕：

*第一幕：先左后右，中间开 LATERAL 窗口（1321-1354）*

```c

										  &l_nsitem,
```

处理右子树前把左侧名字**临时压入** pstate 名字空间，处理完再弹出——这就是 `t1 JOIN LATERAL (SELECT ... WHERE x = t1.id) s` 里子查询能看到 t1 的机制。源码注释（1332-1336）还记录了一条标准的怪癖：SQL:2008 要求非 INNER/LEFT 连接时左侧名字**仍然可见但引用即报错**（"Stupid design, but that's what it says"——注释原话），所以名字照样压入、只是 `lateral_ok=false`。

*第二幕：冲突检查与 NATURAL 展开（1357-1421）*：`checkNameSpaceConflicts` 保证左右子树暴露的表别名不重名（SQL 要求同层 FROM 别名唯一）；NATURAL JOIN 则扫描两侧列名表，把同名列生成一个等价的 USING 列表（跳过已删除列——它们在 colnames 里是空字符串占位）。

*第三幕：USING 列表变换（1444-1549）*：对每个 USING 列名做四件事：查重（`USING(foo,foo)` 报错）、在左右两侧各自定位（找不到或找到两次都报错，1489-1527）、用 `buildVarFromNSColumn` 造出左右 Var、最后 `transformJoinUsingClause` 把它们拼成 `l.c1 = r.c1 AND ...` 的 ON 条件。**USING 不是语法糖那么简单**——它还改变输出列集（合并列只出现一次）与列的语义（见第五幕）。用户写的 ON 条件则走 `transformJoinOnClause`（1553），在合并名字空间 `my_namespace` 里解析。

*第四幕：预测 rtindex 并标记 NULL 化（1560-1597）*

```c

		switch (j->jointype)
```

JOIN 自己也是一个 RTE（`RTE_JOIN`），但此刻还没建——代码**预测**它将得到的下标（注释 1569-1574 解释：从这里到 `addRangeTableEntryForJoin` 之间不可能再建其他 RTE，所以预测安全，事后有 Assert 验证）。`markRelsAsNulledBy` 把"外连接可能补 NULL"记进相应 RTE 的 nulling 集合，此后构造的 Var 会带上 `varnullingrels` 标注——这是 PG 16 起外连接语义在解析层的表达，优化器据此判断谓词能否越过外连接下推。

*第五幕：构造合并列的 join alias Var（1602-1660）*：对 USING 列重新 build 左右 Var（注释 1617-1620：此时它们的 `varnullingrels` 可能与第三幕造的不同！），`buildMergedJoinVar` 决定合并列的表现——INNER/LEFT JOIN 时合并列就是左列；FULL JOIN 时是 `COALESCE(l.c, r.c)`，谁都不等价，只能作为 join 输出列引用（1644-1658 的三分支）。

*第六幕：拼装输出列集（1662-1684）*：`extractRemainingColumns` 把两侧未参与 USING 的列追加进输出；若 JOIN 带别名（`... JOIN ... AS j(a,b)`），别名语法上**遮蔽**全部输入（1675-1684 把每列的 `p_varnosyn` 改指 join 自身）。

*第七幕：建 JOIN RTE 并验证预测（1689-1704）*：`addRangeTableEntryForJoin(...)` 真正建 RTE；随后 `Assert(j->rtindex == nsitem->p_rtindex);` 验证第四幕的预测。这类"先预测、后断言"的写法在 PG 源码里很常见——比"先建占位、后回填"更少状态残留。

*第八幕：登记 p_joinexprs、处理 USING 别名、返回名字空间（1709-1745+）*：`pstate->p_joinexprs` 是"rtindex → JoinExpr"的稀疏映射（懒惰填充 NULL 占位）；PG 14 的 `JOIN ... USING (...) AS x` 别名在此生成独立 nsitem；最后返回的名字空间遵循规则——JOIN 有别名则只暴露别名，否则输入表名仍可限定引用、但合并列以 join 输出列身份参与非限定解析。

**小结这个函数的不变式**：递归返回时，(a) 本子树涉及的所有 RTE 已按遇到顺序 append 进 `p_rtable`，且**任何已分配下标永不改变**；(b) 返回的 jointree 节点只含 rtindex 引用；(c) 名字空间列表精确反映 SQL 作用域规则。整个语义分析的正确性都押在 (a) 上——Var 一旦生成，`varno` 就是永久地址。

### 3.4 addRangeTableEntry：名字→OID 与加锁

`parserOpenTable()`（parse_relation.c:1459-1498）内部走 `table_openrv_extended` 完成"名字 → OID"（其中的 search_path 解析、relcache 查找在第 11 篇展开）——**这是整条 SQL 处理路径上第一次接触系统目录，也是第一次加锁**。它还有个贴心细节（1475-1488）：不带 schema 的表名找不到时，会检查是不是引用了"排在后面的 CTE"，若是则给出 "Use WITH RECURSIVE, or re-order the WITH items" 的定向提示——这是从大量用户困惑中沉淀出来的错误信息工程。

**加锁与失效的正确性不变式**（三问之③，也是 6.1 节 what-if 的基础）：

1. 锁在 `table_close(rel, NoLock)` 后**保持到事务结束**，因此 Query 里的 `relid` 及所有列号在本事务内始终有效——`AccessShareLock` 挡住任何 `DROP/ALTER TABLE`（它们要 `AccessExclusiveLock`）；
2. 加锁顺序 = FROM 子句从左到右的 RTE 建立顺序 = 重写器 `AcquireRewriteLocks()`（rewriteHandler.c:148）对缓存查询重放的顺序——**同一条语句无论初次解析还是从 plancache 复活，加锁序一致**，这是避免自死锁与保证失效检测有明确同步点的前提；
3. `rellockmode` 被记进 RTE（parsenodes.h:1207），执行器按它重新加锁（AcquireExecutorLocks），保证"计划复用时的锁 ≥ 解析时的锁"。

列名解析发生在表达式变换时（3.5 节调回来）：`refnameNamespaceItem()`（parse_relation.c:130）按"表别名.列名"定位；`colNameToVar()`（parse_relation.c:926）处理不带限定的列名——遍历当前名字空间的所有 nsitem 调 `scanRTEForColumn()`（parse_relation.c:843），**若多个 RTE 都有该列则报 ambiguous 错误**。解析成功的产物是 `Var` 节点：`(varno, varattno)` = "范围表第几项的第几列"，从此表达式里再无字符串名字。

### 3.6 重载消解全路径：oper() / func_get_detail / func_select_candidate

PG 的操作符和函数都可重载（同名不同参数类型），且用户可任意扩展。`=` 在系统目录里就有几十个版本。给定 `int4 = numeric` 这样的调用选哪个？这套算法是 PG 类型系统的精华（官方手册 "Type Conversion" 章的规范文本，就是对下面代码的散文化转写）。

**先回答三问之②：为什么不用"最先匹配"？构造反例。** 设想候选按目录扫描顺序返回、取第一个隐式可达者：`SELECT 2 + 3.5`（int4 + numeric）的隐式可达候选包括 `float4+float4`、`float8+float8`、`numeric+numeric` 等（int4→float4、numeric→float8 均是 pg_cast 里 castcontext='i' 的隐式转换）。若"最先匹配"撞上 `float4+float4`，`3.5` 这个精确的 numeric 会被降成单精度浮点，`2 + 3.5` 算出 `5.5000005` 之类的结果——**且结果取决于目录行的物理顺序，不同版本、不同 initdb 可能不同**。PG 的打分算法（下述）确定性地选出 `numeric+numeric`。另一个方向的反例：若"最先匹配"选了 `float8+float8`，`SELECT 9007199254740993::int8 + 1` 类的整数运算会静默丢失精度。结论：重载消解必须是**全候选集上的确定性评分**，而非扫描序敏感的短路。

**STRING 类别一票胜出**——注释（1290-1291）的理由：*this bias towards STRING is appropriate since unknown-type literals look like strings*。非 STRING 的类别冲突则整位失败。归类成功后淘汰"类别不合"或"该位存在优先类型候选却给了非优先类型"的候选（1363-1421）；注释还有个兜底：若这条规则会杀光所有候选，就当它没发生过（保留全部）。

*第 4 步（1423-1478）：最后一搏。* 若已知类型的输入全是同一个类型，就**假定 unknown 输入也是该类型**，看是否恰好唯一命中：

```c
	/*
	 * Last gasp: if there are both known- and unknown-type inputs, and all
	 * the known types are the same, assume the unknown inputs are also that
	 * type, and see if that gives us a unique match.  If so, use that match.
	 */
```

注释（1428-1431）指明它与 `binary_oper_exact` 的分工：那边只找精确匹配，这里允许"假定后再经转换/多态匹配"。走完仍剩多个，`return NULL`——上层随即报 `operator is not unique` / `function is not unique`，要求用户显式加 cast。**宁可报错，绝不猜错。**

**函数版全径：func_get_detail()（parse_func.c:1522-1855）**。比操作符多出的四个环节按源码顺序走读：

1. *候选收集（1553-1556）*：`FuncnameGetCandidates(funcname, nargs, fargnames, expand_variadic, expand_defaults, include_out_arguments, false, fgc_flags)`（实现在 src/backend/catalog/namespace.c:1199）不只按名字与参数个数过滤，还要展开 VARIADIC（可变参数摊平/收拢两种形态）、补参数默认值、匹配命名参数（`f(b => 1, a => 2)`）——同一个 pg_proc 行可能以多种"形态"进候选集。
2. *精确匹配快路（1562-1570）*：对候选链逐个 `memcmp(argtypes, best_candidate->args, nargs * sizeof(Oid))`——注释：*so that the normal case (exact match) is as quick as possible*。
3. *"函数名即类型名"的转换解释（1572-1667）*：精确匹配失败后，若是单参数、无命名参数的调用且函数名对应一个类型名，则考虑把 `text(varchar_col)`、`int4('42')` 解释为**类型转换**而非函数调用，返回 `FUNCDETAIL_COERCION`。注释（1581-1586）给出优先级理由：`text(varchar)` 必须解释为直接转换，否则下面的启发式完全有能力选出 `text(name(varchar))` 这种"先转 name 再调 text(name)"的诡异组合。只接受 RELABELTYPE（免费重贴标签）与 COERCEVIAIO（走 I/O 函数）两种路径；拒绝 COERCION_PATH_FUNC 的理由（1591-1593）很优雅：真有转换函数的话，它本来就叫目标类型的名字，正常函数查找自然会找到它。1607-1611 的 NB 注释警告这里的判断不得超出 `coerce_type` 的能力，否则两个模块会**无限互递归**。
4. *启发式消解（1669-1702）*：与操作符完全同一套 `func_match_argtypes` + `func_select_candidate`（parse_func.c:1084-1085 的注释记录：parse_oper.c 原有的重复代码已被合并于此）。
5. *收尾装配（1705-1852）*：命名参数时把实际参数位置回填进 `NamedArgExpr.argnumber`（1740-1753）；从 pg_proc 取默认值表达式并裁掉本次调用已提供的（1764-1828，命名记法用 Bitmapset 挑选，位置记法直接砍头）；最后按 `prokind` 分类返回 NORMAL/AGGREGATE/PROCEDURE/WINDOWFUNC（1830-1848）——聚合与窗口函数与普通函数**共用整条消解路径**，只在最后一步分家。

另一处消解发生在 ORDER BY/GROUP BY：`get_sort_group_operators()`（parse_oper.c:183）通过类型的**默认 btree 操作符类**找 `<`/`=`/`>`，这就是"某类型能否排序"的判据。

### 3.7 parse_coerce.c：coerce_type 主干走读

消解算法反复问"能否转换"，回答者是 parse_coerce.c。转换强度分三级（`CoercionContext`，src/include/nodes/primnodes.h）：

- `COERCION_IMPLICIT`：表达式里可自动发生（最严格）；
- `COERCION_ASSIGNMENT`：赋值语境（INSERT/UPDATE 的目标列）额外允许，比如 `int4 → text` 存储时允许，表达式里不允许；
- `COERCION_EXPLICIT`：仅当用户写 `CAST` / `::` 才允许。

哪条转换在哪级可用，记录在系统目录 **pg_cast** 里（`castcontext` 列），查询入口 `find_coercion_pathway()`（parse_coerce.c:3157）。也就是说：**隐式转换规则本身是目录数据，可由扩展增删**——Berkeley "可扩展数据库"血统的又一体现（手册同时严厉警告：往 pg_cast 加隐式转换是自找歧义——你已经在 3.6 节看到隐式路径集合如何直接决定候选集大小）。

总入口 `coerce_to_target_type()`（parse_coerce.c:79-132）负责组合拳：先 `can_coerce_type` 预检，再剥掉顶层 CollateExpr、调 `coerce_type` 做类型转换、调 `coerce_type_typmod()`（parse_coerce.c:752）补长度约束（`varchar(10)` 的 10）、最后把 CollateExpr 装回去。

**coerce_type()（parse_coerce.c:158 起）主干按分支走读**：

*分支 0（166-171）：同型或 NULL 节点直接返回。*

*分支 1（172-191）：目标是 ANY/ANYELEMENT 等"无形态"多态伪类型*——原样返回节点，注释强调**不要**包 RelabelType：函数参数的暴露类型必须是实际类型。unknown 常量也允许直接喂给这些伪类型（unknown 值本身是合法 Datum）。

*分支 2（192-232）：目标是 ANYARRAY/ANYENUM/ANYRANGE 等"有形态"多态伪类型*——与分支 1 不同，实参必须真的是数组/枚举/区间，**不能**是 unknown 常量（那样会落到下面调用伪类型输入函数而报错）；domain-over-array 则重贴为基类型。

*分支 3（233-372）：unknown 字面量的落地——全文最重要的分支。*

```c
	if (inputTypeId == UNKNOWNOID && IsA(node, Const))
	{
		/*
		 * Input is a string constant with previously undetermined type. Apply
		 * the target type's typinput function to it to produce a constant of
		 * the target type.
		 *
		 * NOTE: this case cannot be folded together with the other
		 * constant-input case, since the typinput function does not
		 * necessarily behave the same as a type conversion function. For
		 * example, int4's typinput function will reject "1.2", whereas
		 * float-to-int type conversion will round to integer.
		 */
```

要点逐条：目标是 domain 时取**基类型**的输入函数（260-269，domain 的约束延后由 `coerce_to_domain` 统一校验）；typmod 一般传 -1、长度检查延后——唯 INTERVAL 例外必须现在传（271-283，注释原话感叹标准之丑：*Ugly as sin, but so is this part of the spec...*）；新 Const 的 `location` 沿用原字面量位置（294-299，9.2 起为 pg_stat_statements 而改）；调 `stringTypeDatum` 真正执行输入函数（311-318），前后包着 parser 错误位置回调，所以 `select '12x'::int4` 的报错会精确指向字面量；varlena 值强制 detoast 成扁平形态（320-327）保证 Const 可按字节比较。

这解释了 PG 与 MySQL 的一个根本分歧：`'abc'` 这种字面量**没有固有类型**，它是什么完全由上下文决定。`WHERE id = '42'` 里它在解析期就折叠成 int4 常量 42（索引照用）；`WHERE name = 42` 则因无 `text = int4` 隐式路径而报错。**三问之②：延迟决议 vs 立即绑定**——若像 MySQL 那样把字面量立即绑为字符串类型，就必须内建"字符串与一切数值可比"的弱类型规则来救场，代价是 `WHERE int_col = '42abc'` 静默截断、`WHERE varchar_col = 42` 让索引悄悄失效（MySQL 按数值语义比较字符串列，无法走索引）。PG 用 unknown 这个"薄命类型"（活不过语义分析，第⑧步的 `resolveTargetListUnknowns` 是它最后的归宿）换来了强类型与索引可用性的两全。

*分支 4（373-388）：Param 钩子*——PREPARE 语句的 `$1` 类型待定时，`p_coerce_param_hook` 允许协议层把"参数类型由首次使用的上下文反推"接进来。

*分支 5（389-413）：CollateExpr 下潜*——转换必须钻到 COLLATE 下面做（否则分支 3/4 对 Const/Param 的特判够不着），注释自嘲 "This is really ugly"。

*分支 6（414-480）：查 pg_cast 走正路*——`find_coercion_pathway` 命中后两种形态：有转换函数则 `build_coercion_expression` 生成运行期函数调用（FuncExpr）；`COERCION_PATH_RELABELTYPE` 则零成本包一层 `RelabelType`（二进制兼容，如 varchar→text）。目标是 domain 时统一再套 `coerce_to_domain`。

*分支 7（481 起，本文从略的收尾）*：RECORD↔具体行类型互转、继承表的行类型上转（`ConvertRowtypeExpr`）等——省略原因：属于复合类型的旁路，不影响标量主线。全部落空则 `elog(ERROR, "failed to find conversion function ...")`（能走到这说明 can_coerce_type 与本函数失步，属内部错误）。

多分支共同类型的选择：`select_common_type()`（parse_coerce.c:1342）为 UNION/CASE/COALESCE/VALUES 的多个分支选统一输出类型，规则：全 unknown 则 text；否则取第一个非 unknown 类型所在**类别**，类别内优先类型优先，且必须能从所有分支隐式转到——3.2 节 transformValuesClause 的逐列处理（analyze.c:2007）正是调它。

### 3.8 小结

- 语义分析的输入是"字符串写法"，输出是"OID 写法"：表→`relid`、列→`Var(varno,varattno)`、操作符→`OpExpr.opno`、函数→`FuncExpr.funcid`；
- transformSelectStmt 的十步顺序编码了名字可见性规则：FROM → targetList → WHERE/HAVING → ORDER BY → GROUP BY → DISTINCT → LIMIT → 窗口 → 装配 → 检查；WHERE 并入 jointree 是关系代数视角的落地；
- transformFromClauseItem 以"先左后右 + LATERAL 窗口 + rtindex 预测"递归处理 JOIN；不变式：RTE 下标一经分配永不变；
- 名字解析伴随**加锁**（parserOpenTable），锁保持到事务结束，加锁顺序与重写器/执行器的重放顺序一致；
- 重载消解 = lookaside 缓存 → 精确匹配 → 隐式可达过滤 → 精确位次计分 → 同类别优先类型加分 → unknown 归类（STRING 偏置）→ 同型假定 → 仍歧义则报错；"最先匹配"会引入目录序依赖与精度事故；
- 三级转换语境 + pg_cast 目录 = 可扩展但克制的类型转换体系；unknown 字面量在解析期由目标类型的输入函数折叠成 Const，是 PG 强类型与索引友好的关键设计。

## 第 4 节 · Query：关系代数表达式的具体化

`Query`（src/include/nodes/parsenodes.h:120-262）是语义分析的输出、重写器和优化器的输入，是 PG 最重要的结构体之一。

### 4.1 关系代数视角

一条 `SELECT a.x, b.y FROM a JOIN b ON a.id=b.id WHERE a.x > 0` 的代数形式：

```
π_{a.x, b.y} ( σ_{a.x > 0} ( a ⋈_{a.id=b.id} b ) )
     │               │              │
 targetList     jointree 的 quals   jointree 的连接结构，叶子是 rtable 的下标
```

- **rtable**：查询涉及的所有"关系"的清单（真实表、子查询、函数、CTE……），即代数表达式的**操作数集合**。表达式里的 `Var.varno` 都是它的 1-based 下标；
- **jointree**：一个 `FromExpr`，`fromlist` 是连接结构（叶子为 `RangeTblRef`，内部节点为 `JoinExpr`），`quals` 是 WHERE 条件——**σ 与 ⋈ 合并存放**，因为对优化器而言两者都是"限制元组的谓词"，内连接的 ON 条件和 WHERE 条件可以自由互换位置；外连接的 ON 条件则必须留在对应 JoinExpr 上（影响补 NULL 语义）；
- **targetList**：`TargetEntry` 列表，即投影 π。`resjunk=true` 的隐藏列（供 ORDER BY/FOR UPDATE 用，最终不输出）也在其中。

rtable 与 jointree 的分离是刻意的：**rtable 是"提到了哪些关系"，jointree 是"它们怎样组合"**。UPDATE 的目标表在 rtable 里但可以不在 jointree 里；视图展开后废弃的 RTE 仍留在 rtable（保序，Var 编号不变）只是从 jointree 摘除。

### 4.3 RangeTblEntry 与两处设计史

RTE 的种类由 `RTEKind`（parsenodes.h:1122-1137）枚举：

```c
typedef enum RTEKind
{
	RTE_RELATION,				/* ordinary relation reference */
	RTE_SUBQUERY,				/* subquery in FROM */
	RTE_JOIN,					/* join */
	RTE_FUNCTION,				/* function in FROM */
	RTE_TABLEFUNC,				/* TableFunc(.., column list) */
	RTE_VALUES,					/* VALUES (<exprlist>), (<exprlist>), ... */
	RTE_CTE,					/* common table expr (WITH list element) */
	RTE_NAMEDTUPLESTORE,		/* tuplestore, e.g. for AFTER triggers */
	RTE_GRAPH_TABLE,			/* GRAPH_TABLE clause */
	RTE_RESULT,					/* RTE represents an empty FROM clause; such
								 * RTEs are added by the planner, they're not
								 * present during parsing or rewriting */
	RTE_GROUP,					/* the grouping step */
} RTEKind;
```

`RangeTblEntry` 结构体在 parsenodes.h:1139 起；关键字段：`relid/inh/relkind/rellockmode/perminfoindex`（关系 RTE，1201-1209），`subquery/security_barrier`（子查询 RTE，1217-1219），以及注释（1187-1193）特别说明的混合态——视图展开后的 RTE_SUBQUERY **保留** relid/relkind/rellockmode/perminfoindex，因为视图本身仍要加锁鉴权（5.4 节）。

**设计史③：rteperminfos 的拆分。** commit `a61b1f7482`（2022-12-06, Álvaro Herrera 提交，Amit Langote 主写，进入 PG 16）把权限检查信息（requiredPerms/checkAsUser/selectedCols/…）从 RangeTblEntry 中拆出为独立的 `RTEPermissionInfo` 节点列表（parsenodes.h:1404）。提交信息给出的动机：此前执行器初始化时必须**扫描整张范围表**找需要鉴权的 RTE；分区表有几千个子分区时，子分区本身无需逐个鉴权（继承根已查过），却仍要被遍历过滤一遍。拆分后只有继承根 RTE 建 perminfo 条目，执行器只扫这张短表；RTE 以 `perminfoindex` 反向指入。对本章的意义：**解析器负责建表（addRTEPermissionInfo，3.4 节），重写器展开视图时追加（5.4 节），planner 把各子查询层的表拍平成一张总表交执行器**——一条贯穿四阶段的数据结构。

**设计史④：RTE_GROUP 的新增。** commit `247dea89f7`（2024-09-10, Richard Guo）引入"分组步骤"RTE。动机（提交信息摘译）：分组表达式若含子查询，targetList 与 HAVING 里的**每一份**都会各自展开成独立 SubPlan——只有一份会被替换为 Agg 输出列的引用，其余被重复求值；配合 GROUPING SETS 时，重复求值还可能读到"本应为 NULL 的分组列"，产生**错误结果**。修法是在解析器里把分组表达式替换为指向 RTE_GROUP 的 Var，让"分组输出必须取自分组步骤"这一语义要求**直接体现在解析器输出里**；planner 再把这些 Var 换回处理后的表达式，保证每个子查询只有一个 SubPlan 实例。这是"用数据结构表达语义约束，而非靠各阶段自觉遵守"的漂亮案例——也提醒我们：rtable 里的项未必对应任何"表"，它是广义的"可被 Var 引用的元组源"。

### 4.4 小结

- `rtable`/`jointree`/`targetList` 分别对应关系代数的操作数集合、σ+⋈、π；`groupClause`+`havingQual` 是 γ+σ，`setOperations` 是 ∪∩−；
- 每个字段都有明确的"填充者→消费者"闭环，改内核时两端必须同步（尤其 `has*` 标志——它们是给后续阶段的性能承诺）；
- RTE 是"可被 Var 引用的元组源"的广义抽象：从真实表到分组步骤（RTE_GROUP）都是它；
- 权限信息已拆到 rteperminfos（PG 16），执行器鉴权从 O(rtable) 降到 O(需鉴权表数)。

## 第 5 节 · 重写器：规则系统、视图与行级安全

### 5.1 历史背景：Stonebraker 的规则系统

重写器（`src/backend/rewrite/`）是 Berkeley POSTGRES 时代的直系遗产。Stonebraker 在 1980 年代提出：与其为触发器、视图、完整性约束各造一套机制，不如提供统一的**声明式规则系统**——"当对表 R 执行事件 E 时，改做（INSTEAD）或附加做（ALSO）动作 A"，且规则在**查询树层面**改写（一次改写服务整条语句），而非像触发器那样逐行触发。这套思想沉淀为 `CREATE RULE` 语法和 pg_rewrite 目录，规则的内存形态是 `RewriteRule`（src/include/rewrite/prs2lock.h:24——文件名里的 "prs2" 即 POSTGRES Rule System v2）。

四十年后的今天，用户级 `CREATE RULE` 已不推荐使用（语义陷阱多，触发器更直观），但规则系统的两大核心应用活得很好：**视图**（每个视图 = 一条系统自动创建的 `ON SELECT DO INSTEAD` 规则 `_RETURN`，动作就是视图定义查询，见 `get_view_query()`，rewriteHandler.c:2574）和**可更新视图/RLS 的实现骨架**。

### 5.2 QueryRewrite 总入口全走读

`QueryRewrite()`（rewriteHandler.c:4789-4878）结构极清晰，三步：

```c
	int64		input_query_id = parsetree->queryId;
	...
	/* This function is only applied to top-level original queries */
	Assert(parsetree->querySource == QSRC_ORIGINAL);
	Assert(parsetree->canSetTag);

	/* Step 1: Apply all non-SELECT rules possibly getting 0 or many queries */
	querylist = RewriteQuery(parsetree, NIL, 0, 0);

	/* Step 2: Apply all the RIR rules on each query
	 * This is also a handy place to mark each query with the original queryId */
	results = NIL;
	foreach(l, querylist)
	{
		Query	   *query = (Query *) lfirst(l);

		query = fireRIRrules(query, NIL);
		query->queryId = input_query_id;
		results = lappend(results, query);
	}
```

Step 3（4847-4876）裁决命令标签：遍历结果，若原始查询（QSRC_ORIGINAL）仍在列表里，它设标签（断言它 canSetTag 且唯一）；否则挑**最后一条**与原命令同类型的 INSTEAD 规则查询顶替（`lastInstead->canSetTag = true;`）；两者皆无则由 tcop 按原始语句造默认标签。这就是"对着带规则的表 UPDATE，返回的 'UPDATE n' 到底数谁的行数"这一冷知识的答案。

输出是 `List<Query>`——**一进多出**是重写器最反直觉的接口：一条 `INSERT INTO v ...` 在规则作用下可能变成"往 A 表插一条 + 往日志表插一条"两条 Query，随后被逐条优化、逐条执行。

两步的分工：Step 1 处理**数据修改类规则**（会增删查询）；Step 2 处理 **RIR 规则**（Retrieve-Instead-Retrieve，即视图展开，只改写单条查询内部）+ RLS 注入。顺序不可反：先裂变，再对每条产物统一展开视图。

### 5.3 fireRIRrules 全走读

`fireRIRrules()`（rewriteHandler.c:2042-2364）对一条 Query 做"就地展开"，五个阶段：

**阶段 0（2054-2063）**：顺手展开 CTE 的 SEARCH/CYCLE 子句（`rewriteSearchAndCycle`）——注释坦言这只是"既然路过每条 Query，就在这做了"。

**阶段 1（2065-2212）：主循环，扫描 rtable 展开视图。** 循环头就有讲究（2066-2067 注释原文）：

```c
	/*
	 * don't try to convert this into a foreach loop, because rtable list can
	 * get changed each time through...
	 */
	rt_index = 0;
	while (rt_index < list_length(parsetree->rtable))
```

展开视图会往 rtable **追加**元素（如 INSTEAD OF 触发器场景复制目标 RTE），foreach 的快照式遍历会漏掉或错乱。循环体内六连筛（每条都注明为什么跳过）：

1. `RTE_SUBQUERY` → 递归 `fireRIRrules(rte->subquery, activeRIRs)` 后 continue（子查询自身无规则可言，2094-2109）；顺手上传 hasRowSecurity；
2. 非 `RTE_RELATION` → continue（JOIN 等无规则，2112-2115）；
3. 物化视图 → continue（2118-2128：查询里引用的 matview **不展开**——它的意义就是"已物化"；注释还展望了未来"不可扫描时按需展开"的可能）；
4. `ON CONFLICT` 的 EXCLUDED 伪关系 → continue（2131-2139：它指向视图也不能展开，否则 RTE 变 SUBQUERY 类型进入未测试领域）；
5. 未被引用的 RTE → continue（2141-2150 注释原话：*This prevents infinite expansion loop due to new rtable entries inserted by expansion of a rule*——规则展开新增的表项若无人引用就别再展开）；
6. ApplyRetrieveRule 刚复制出的新结果关系 → continue（2153-2158，配合 5.4 节的"偷梁换柱"）。

过筛的关系表打开（**NoLock**！2160-2163 注释：*We can use NoLock here since either the parser or AcquireRewriteLocks should have locked the rel already*——锁纪律的又一体现），收集其 `rd_rules` 里 event==CMD_SELECT 的规则，然后是**防递归的关键三行**（2189-2207）：

```c

					ereport(ERROR,
							(errcode(ERRCODE_INVALID_OBJECT_DEFINITION),
							 errmsg("infinite recursion detected in rules for relation \"%s\"",
									RelationGetRelationName(rel))));
				activeRIRs = lappend_oid(activeRIRs, RelationGetRelid(rel));

				{
```

**阶段 2（2214-2227）**：递归处理 WITH 里的子查询。**阶段 3（2229-2248）**：`hasSubLinks` 时用 `fireRIRonSubLink` walker 钻进表达式中的子查询（rewriteHandler.c:2005-2031，注意 `QTW_IGNORE_RC_SUBQUERIES`——rtable 和 cteList 里的已经处理过，不能重复）。

**阶段 4（2250-2361）：RLS 注入。** 时机刻意选在**一切视图展开之后**（2250-2254 注释：否则带子查询的策略会被 query_tree_walker 二次遍历）。对每个普通表/分区表 RTE：

```c

								  &securityQuals, &withCheckOptions,
```

`ApplyRetrieveRule()`（rewriteHandler.c:1762-1927）执行"视图 RTE → 子查询 RTE"的偷梁换柱。分五段：

**第 0 段（1773-1784）：防御与安全阀。** ON SELECT 规则必须单动作、无条件（`elog(ERROR)` 级内部一致性）；然后是一个 2024 年加入的安全阀：


GUC `restrict_nonsystem_relation_kind` 由安全修复 commit `66e94448ab`（2024-08-05, *Restrict accesses to non-system views and foreign tables during pg_dump*，即 CVE-2024-7348 的修复）引入：pg_dump 运行期间攻击者可抢先重建同名视图/外表，让 dump 进程展开攻击者控制的查询——修复思路不是给 pg_dump 加锁，而是**在重写器里给"会执行任意用户查询定义"的展开点装开关**。设计史⑤：重写器是安全边界的一部分，这一点在原始设计里并不显眼，是被漏洞"发现"的。

**第 1 段（1786-1853）：视图作为 DML 目标。** 查询目标表是视图且没被规则改写时（说明依赖 INSTEAD OF 触发器）：INSERT 什么都不用做（未展开的 RTE 直接当结果关系，触发器兜底）；UPDATE/DELETE/MERGE 则需要"数据源+目标"两个身份——


**设计史⑥：OLD/NEW 表项的未遂清理。** 与视图 rtable 相关还有一桩一日游公案：commit `1b4d280ea1`（2023-01-11, Tom Lane, *Get rid of the "new" and "old" entries in a view's rangetable*）试图删掉视图定义查询里历史遗留的 OLD/NEW 两个哑元 RTE（它们占着 rtable 前两席、纯属 prs2 时代的化石），**当天**就被 `f0e6d6d3c9` 撤销——revert 信息记录了原因：跨版本升级测试的 buildfarm 动物无法处理新旧服务器 `CREATE VIEW` 输出的"外观差异"。教训与 4.3 的 RTE_GROUP 相映成趣：rtable 布局是被 pg_dump 文本、存储的规则、跨版本工具共同咬死的**准 on-disk 格式**，动它的成本远超代码本身。（该清理后来在 PG 16 周期以更完备的形态完成。）

**第 2 段（1855-1885）：取出并预处理视图定义。**


RTE 原地从"表"变成"子查询"，但 relid/relkind/rellockmode/perminfoindex **保留**（1904-1907 注释）——视图虽不再被扫描，仍要以视图 owner 的身份做权限检查、仍要加锁。展开后 `SELECT * FROM v` 等价于 `SELECT * FROM (视图定义) v`；优化器的子查询上拉（pull_up_subqueries）通常还会把这层展平，最终计划与手写内联 SQL 无异——**"PG 视图零开销"的完整机制**。`security_barrier` 视图是例外：该标志阻止谓词穿越屏障（与 RLS 的 securityQuals 同源的防泄露语义）。

**第 4 段（1912-1926）：列数修补。** `CREATE OR REPLACE VIEW` 允许给视图**加列**，缓存的原查询可能按旧列数解析——这里对 `rte->eref->colnames` 补 `?column?` 哑名对齐（6.4 节的 what-if 演示这为什么必要且充分）。

### 5.5 RewriteQuery 全走读：INSERT/UPDATE/DELETE 规则

`RewriteQuery()`（rewriteHandler.c:4044-4649）处理数据修改类规则。签名里的两个防重入参数先记住：`rewrite_events` 是"正在处理的 (表, 事件) 栈"，`orig_rt_length`/`num_ctes_processed` 防止产品查询里**来自原查询的部分**被二次处理。分七段：

**第 1 段（4067-4141）：先递归重写 WITH 里的 DML。** 注释（4056-4058）：必须最先做，因为 CTE 可能被拷进规则动作。对每个非 SELECT 的 CTE 递归 `RewriteQuery`，结果必须**恰好一条**可优化 Query 才能塞回 CTE 节点；0 条（INSTEAD NOTHING）、多条（多语句 INSTEAD/DO ALSO/条件 INSTEAD）各报专门错误（4102-4138）——`WITH (INSERT ...)` 与规则系统的组合空间被显式封死，因为 CTE 语义要求"单次求值"，与"一进多出"根本冲突（尾段 4631-4646 还有一道同源保险：原查询带 CTE 且重写产出多条可优化查询时直接报错）。

**第 2 段（4151-4172）：定位目标表。** SELECT/UTILITY 跳过整段；否则取 `resultRelation` 的 RTE（Assert 必为 RTE_RELATION——视图此刻还是"表"！），`relation_open(rt_entry->relid, NoLock)`（又是 NoLock+注释确认锁已在手）。

**第 3 段（4182-4357）：按命令类型重排 targetList。** INSERT 分支先在 fromlist 里找 VALUES RTE（多行 INSERT 的载体；`rtr->rtindex > orig_rt_length` 跳过产品查询里来自原查询的旧 VALUES，4196-4215），然后：

```c

	 * not do anything if they are invoked after the update or delete is
```

### 5.6 小结

- 重写器是 Stonebraker 规则系统的现代残部：视图 = `_RETURN` ON SELECT DO INSTEAD 规则，展开 = 把 RTE 原地改造成 RTE_SUBQUERY（保留 relid/rellockmode/perminfoindex 以便锁与鉴权），随后靠优化器展平，故视图无运行期开销；security_barrier 视图与 RLS 的 securityQuals 是同一套防泄露屏障机制；
- `QueryRewrite` 一进多出：DML 规则可让一条语句裂变为多条 Query；缺省值填充、OLD/NEW 替换、条件 INSTEAD 的 `IS NOT TRUE` 补集保留、原查询的 INSERT 先行/UPDATE-DELETE 殿后，都由命令间的可见性语义决定；
- 两套防递归栈：activeRIRs（视图/策略展开链，键=OID）与 rewrite_events（DML 规则链，键=OID+事件）；
- 读这一模块时刻记住不变式：**改写必须保持 Var 编号有效**——rtable 只增不删、只改不动下标；一切新语义靠追加表项与 ChangeVarNodes 表达；
- 重写器也是安全边界：CVE-2024-7348 的修复（`66e94448ab`）直接落在 ApplyRetrieveRule 的入口。

## 第 6 节 · what-if 推演

### 6.1 语义分析进行到一半，另一个会话对表执行 DDL，会怎样？

分时刻推演 `SELECT * FROM t JOIN s ...`：

- **t 的 RTE 建立之前**：并发 `DROP TABLE t` 畅通无阻。本会话随后 `parserOpenTable` 查不到表，报 "relation t does not exist"——干净失败，无一致性问题。
- **t 已锁、s 未锁**：`addRangeTableEntry(t)` 已拿 AccessShareLock 并读取了 t 的列表（buildRelationAliases 用的 `rel->rd_att`）。此刻并发 `ALTER TABLE t ADD COLUMN` 需要 AccessExclusiveLock，被我们的 AccessShareLock **阻塞**——所以"已读取的表结构在语句内失步"这件事被锁直接排除。并发 `ALTER TABLE s ...` 则可能抢在我们锁 s 之前完成：没关系，我们稍后打开 s 时会通过 relcache 失效机制读到**新**结构（锁的获取点就是失效消息的同步点：`LockRelationOid` 内部在锁到手后处理 sinval 队列，保证 relcache 条目新鲜——这正是"先锁后读"顺序的意义）。
- **整条语句分析完、执行前**（扩展协议下 parse/bind/execute 分离，中间可以隔很久）：锁仍在（持至事务末），DDL 依旧进不来。**跨事务**的缓存计划才是真正的暴露窗口：plancache 对每个依赖的 relid 注册失效回调，DDL 提交时广播 sinval，缓存计划被标脏，下次执行用保存的 **raw tree** 重做语义分析——两棵树设计（导读）在此闭环。
- **重写阶段**：视图定义引用的表在解析期没锁过，`AcquireRewriteLocks`（ApplyRetrieveRule 第 2 段）补锁；补锁瞬间同样吞下失效消息。因此 fireRIRrules 中随处可见的 `relation_open(..., NoLock)` 并非侥幸，而是"上游必已加锁"的显式契约（源码注释反复申明这一点）。

**兜底不变式**：任何"读表结构"的代码路径，之前必有一次"以适当锁模式打开该表"；锁的生命周期 ≥ 依赖该结构的数据结构（Query/Plan）的使用周期，或者该数据结构挂上失效回调。两条满足其一，DDL 并发就是安全的。

### 6.2 视图循环引用是如何被挡住的？

正常途径造不出环：`CREATE VIEW v1 AS SELECT * FROM v2` 要求 v2 已存在。但 `CREATE OR REPLACE VIEW` 可以事后造环：

```sql
CREATE VIEW v2 AS SELECT 1 AS x;
CREATE VIEW v1 AS SELECT x FROM v2;
CREATE OR REPLACE VIEW v2 AS SELECT x FROM v1;   -- 环成立，DDL 本身不报错！
SELECT * FROM v1;
-- ERROR:  infinite recursion detected in rules for relation "v1"
```

推演展开过程：fireRIRrules(查询) 遇 v1 → activeRIRs=[v1] → ApplyRetrieveRule 递归 fireRIRrules(v1 定义) 遇 v2 → activeRIRs=[v1,v2] → 递归 fireRIRrules(v2 定义) 又遇 v1 → `list_member_oid(activeRIRs, v1)` 命中 → 报错（rewriteHandler.c:2189-2193）。注意检测在**查询时**而非 DDL 时——DDL 时检测需要全图遍历且挡不住并发交错的两个 REPLACE，查询时检测则天然精确（只拦真正被走到的环）。同款栈还保护 RLS 策略引用视图、策略引用带策略的表等变体（2294-2298）。递归深度本身另有 `check_stack_depth()` 兜底（深而无环的视图链合法，只受栈深限制）。

### 6.3 规则互相触发的无限递归如何防护？

```sql
CREATE TABLE a(i int); CREATE TABLE b(i int);
CREATE RULE r1 AS ON INSERT TO a DO ALSO INSERT INTO b VALUES (new.i);
CREATE RULE r2 AS ON INSERT TO b DO ALSO INSERT INTO a VALUES (new.i);
INSERT INTO a VALUES (1);
-- ERROR:  infinite recursion detected in rules for relation "a"
```

推演：RewriteQuery(INSERT a) → rewrite_events=[(a,INSERT)] → 产品查询 INSERT b → 递归 RewriteQuery(INSERT b) → rewrite_events=[(a,INSERT),(b,INSERT)] → 产品查询 INSERT a → 第三层递归在 4497-4505 的 foreach 中命中 (a,INSERT) → 报错。对比 6.2：这里键是 **(表, 事件)** 而非裸 OID，所以 `ON INSERT TO a DO ALSO UPDATE a ...` 这类同表异事件的规则链合法（INSERT→UPDATE 不构成同键回边）；也因此 `ON UPDATE TO a DO ALSO UPDATE a` 会被拦——它是货真价实的自环。与 activeRIRs 一样，这是"沿当前展开链查重"而非全局计数，链上无环的深规则树不受深度以外的限制。

### 6.4 CREATE OR REPLACE VIEW 给视图加列，旧计划/旧查询会怎样？

场景：会话 A PREPARE 了 `SELECT * FROM v`（v 有 2 列），会话 B `CREATE OR REPLACE VIEW v AS SELECT ... 3 列 ...` 并提交。推演：B 的 REPLACE 更新 pg_rewrite 里的 `_RETURN` 规则并广播失效；A 的缓存计划被标脏，重新用 raw tree 做语义分析——`*` 重新展开，拿到 3 列，一切如新。真正微妙的是**存储在别处的旧查询**（比如另一个视图 w 的定义引用了 v，且 w 是在 v 只有 2 列时创建的）：w 的定义查询里 `v` 的 RTE 只带 2 个列别名，而展开时塞进来的新 `_RETURN` 动作产出 3 列。这正是 ApplyRetrieveRule 第 4 段（rewriteHandler.c:1912-1924）存在的理由：按 `ExecCleanTargetListLength(rule_action->targetList)` 补齐 `?column?` 哑名。为什么补名字就够了？因为 w 定义里的 Var 最多引用到 varattno=2——旧查询**不可能**引用它出生后才添加的列（解析期就不存在）；新列只需"占位可寻址"，不需被引用。反过来 REPLACE VIEW **不允许**删列/改列型（DDL 层直接拒绝），正是为了保住"已存 Var 永远有效"这条不变式——又一次，所有设计殊途同归于同一条军规。

## 自测问题（附详解）

**1. 为什么 PG 要区分 raw parse tree（SelectStmt）与 Query 两种树？只用一种行不行？**

不行。raw tree 与系统目录状态无关（表名还是字符串），可长期缓存：PREPARE、PL/pgSQL 都缓存 raw tree，DDL 使目录变化后重做语义分析即可（6.1/6.4 节推演了完整失效链）。Query 里全是 OID 和下标，目录一变即作废。另外 DDL 语句只有 raw tree 形态（transformStmt 的 default 分支直接包成 CMD_UTILITY，analyze.c:435-444），根本不生成真正的查询树。

**2. 没有 base_yylex 过滤器，`x NOT BETWEEN 1 AND 2` 会让文法出什么问题？**

LALR(1) 状态机读到 `NOT` 时只能看一个 token 就必须决定 shift 进"前缀逻辑非"还是等待归约进 `NOT BETWEEN` 复合结构，而这取决于再下一个 token 是否 BETWEEN/IN/LIKE 等——两个 token 的前瞻超出 LALR(1) 能力，直接在 gram.y 里写会产生 shift/reduce 冲突（违反 PG 的零冲突纪律）。过滤器（parser.c:207-218）在词法与语法之间偷看一个 token，把 `NOT`+`BETWEEN` 合成虚构 token `NOT_LA`，文法便只需一个前瞻。同理 `NULLS FIRST`（NULLS 是可作列名的非保留字）、`WITH TIME ZONE`、`FORMAT JSON`。备选方案（Flex 识别多词 token）因两词间可有注释而不可行（parser.c:96-100 注释）。

**3. 类型消解若改成"扫描候选、取第一个隐式可达者"，`SELECT 2 + 3.5` 会错在哪？**

`int4 + numeric` 无精确匹配；隐式可达候选含 `float4+float4`、`float8+float8`、`numeric+numeric` 等（int4、numeric 到浮点都有隐式 cast）。"最先匹配"的结果取决于目录扫描顺序——可能选中 float4 版本，`3.5` 被降为单精度出现 `5.5000005` 类误差，且行为随版本/initdb 漂移。现行算法（func_select_candidate 第 1 步，parse_func.c:1185-1227）按"精确位次"计分：`numeric+numeric` 在第二参数位精确命中（nmatch=1）胜出，随后 coerce_type 给 2 包上到 numeric 的转换——结果精确且确定。设计底线见 parse_func.c:1095 历史注释：*while _never_ doing the wrong thing*。

**4. `WHERE name = 42`（name 为 text 列）在 PG 报错而 MySQL 能执行，从源码角度解释；`name = '42'` 为什么又合法？**

`oper()`（parse_oper.c:376）为 `=` 寻找 `(text, int4)` 解释：精确匹配失败；`func_match_argtypes()`（parse_func.c:1043）要求 COERCION_IMPLICIT 路径，而 pg_cast 中 text/int4 间没有隐式转换，候选全灭，报 "operator does not exist"。`name = '42'` 则不同：'42' 是 unknown 字面量，`binary_oper_exact()`（parse_oper.c:272-281）直接假定它与另一侧同型，命中 `text = text`，随后 coerce_type 在解析期把它折叠成 text 常量。MySQL 的做法是立即绑字面量为字符串再靠弱类型比较规则救场，代价是隐式转换让索引失效与静默截断（3.7 节展开）。

**5. unknown 类型字面量从出生到消亡经历哪些站点？**

出生：词法/语法层的字符串常量 → transformExprRecurse 造出 `consttype = UNKNOWNOID` 的 Const。中途：参与操作符/函数消解时被 binary_oper_exact 同型假定（parse_oper.c:272），或被 func_select_candidate 第 3 步按类别归类（STRING 偏置，parse_func.c:1288-1294 注释）、第 4 步同型假定。消亡（三选一）：①被 coerce_type 的 unknown-Const 分支（parse_coerce.c:233-372）用目标类型的 typinput 折叠成具体类型 Const；②作为输出列活到 transformSelectStmt 第⑧步，被 `resolveTargetListUnknowns`（analyze.c:1860-1861）强制定为 text；③在 select_common_type（parse_coerce.c:1342）的"全 unknown 则 text"规则下定型。不变式：unknown 不进 Query 的最终 targetList（p_resolve_unknowns 语境下）。

**6. "PG 的视图没有性能开销"这句话的机制依据是什么？什么时候不成立？**

`ApplyRetrieveRule()`（rewriteHandler.c:1762）把视图 RTE 原地改为 RTE_SUBQUERY、塞入视图定义查询的深拷贝（1899-1900），效果与手写子查询完全相同；优化器的子查询上拉进一步把它展平参与全局优化（谓词下推、连接重排）。不成立的情形：`security_barrier` 视图（rte->security_barrier，1901）禁止谓词穿越屏障；含聚合/DISTINCT/窗口函数/LIMIT 等无法上拉的视图作为独立子查询规划（这与手写子查询的限制一致，不是视图额外的税）。

**7. Query.jointree 为什么把 WHERE 条件和 FROM 连接结构放在一起，而不给 WHERE 单独一个字段？**

关系代数中 σ 与内连接 ⋈ 的条件可交换可合并：`σ_p(A ⋈_q B) ≡ A ⋈_{p∧q} B`。把 quals 挂在 FromExpr 上（`makeFromExpr(pstate->p_joinlist, qual)`，analyze.c:1865），优化器可统一收集全部谓词、按引用的关系集合自由决定求值位置，不必关心语法上写在 ON 还是 WHERE。外连接的 ON 条件必须留在对应 JoinExpr 上——它影响补 NULL 语义（transformFromClauseItem 第四幕的 markRelsAsNulledBy 与 varnullingrels 正是为此建立的解析期标注）。

**8. 一条 `INSERT INTO t ...`，t 上有条件 INSTEAD 规则 `WHERE new.i > 10 DO INSTEAD INSERT INTO big ...`，重写后会产出什么？为什么补集条件用 `IS NOT TRUE` 而不是 `NOT`？**

产出两条 Query：①规则动作（INSERT INTO big，带上 `new.i > 10` 条件，querySource=QSRC_QUAL_INSTEAD_RULE，canSetTag=false）；②原查询加上 `AND (new.i > 10) IS NOT TRUE`（fireRules 的 qual_product 路径，rewriteHandler.c:2517-2539）。若用 `NOT (new.i > 10)`：当 i 为 NULL 时条件是 NULL，`NOT NULL` 还是 NULL，WHERE 按假处理——该行既不满足规则条件（不进 big）也进不了原查询（被 NOT 滤掉），**行凭空丢失**。`IS NOT TRUE` 把 NULL 判为真，行正确落入原查询。三值逻辑陷阱的教科书案例（CopyAndAddInvertedQual 头注释，rewriteHandler.c:2368-2372）。

**9. fireRIRrules 与 RewriteQuery 各自的防递归栈，键为什么一个是 OID、一个是 (OID, 事件)？**

fireRIRrules 的 activeRIRs 防的是 **SELECT 规则（视图展开）** 的环：一个视图只有一条 `_RETURN` 规则、一种展开方式，OID 即足够（rewriteHandler.c:2189）。RewriteQuery 的 rewrite_events 防的是 **DML 规则**的环：同一张表可以同时有 ON INSERT / ON UPDATE / ON DELETE 规则，`ON INSERT TO a DO ALSO UPDATE a` 是合法且常见的（插入后更新汇总列），若按裸 OID 查重会被误杀；只有 **(同表, 同事件)** 再次出现才是真环（rewriteHandler.c:4497-4510）。两个栈都是"沿当前展开链查重"，允许无环的深嵌套。
