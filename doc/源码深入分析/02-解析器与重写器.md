# 02 · 解析器与重写器：从 SQL 字符串到规范化查询树

> 本文是《PostgreSQL 源码学习指南》第 3 章的深度展开，基于本地源码树（版本 20devel，master 分支）撰写。
> 文中所有 `文件:行号` 引用均已在这份源码中逐一核实，可直接打开对照。

## 本章导读

一条 `SELECT * FROM t WHERE id = 42;` 到达 backend 进程后，在被优化器接手之前，要经历三次"变形"：

```
SQL 字符串
  └─ pg_parse_query()                        src/backend/tcop/postgres.c:617
      └─ raw_parser()                        src/backend/parser/parser.c:42
          ├─ 词法分析  scan.l   (Flex)   → token 流
          └─ 语法分析  gram.y   (Bison)  → 原始解析树（SelectStmt 等）
  └─ pg_analyze_and_rewrite_fixedparams()    src/backend/tcop/postgres.c:683
      ├─ parse_analyze_fixedparams()         src/backend/parser/analyze.c:128
      │   └─ transformStmt()                 analyze.c:335  → Query（查询树）
      └─ pg_rewrite_query()                  postgres.c:816
          └─ QueryRewrite()                  src/backend/rewrite/rewriteHandler.c:4789
              → 展开视图、应用规则、注入 RLS → List<Query>
```

三个阶段的分工要先记牢：

| 阶段 | 输入 → 输出 | 是否查系统目录 | 教科书对应 |
|---|---|---|---|
| 词法+语法分析 | 字符串 → raw parse tree | **否**（纯语法，不碰任何表） | lexing / parsing |
| 语义分析 | raw parse tree → `Query` | **是**（表名→OID、类型推导、重载消解） | semantic analysis / binding |
| 重写 | `Query` → `List<Query>` | 是（读 pg_rewrite、pg_policy） | view inlining / query rewrite |

为什么"raw parse tree 与 Query 必须分成两棵树"？因为 PREPARE 语句、PL/pgSQL 缓存的语句需要一个**与目录状态无关**的表示——表被重建、列被改名后，raw tree 仍然有效，重新做一次语义分析即可；而 `Query` 里全是 OID，目录一变它就作废。这条边界是 PG 解析器架构里最重要的设计决策。

## 前置知识

- **编译原理**：了解正则表达式驱动的词法分析（Flex）、LALR(1) 语法分析（Bison）的基本概念即可，本文会在用到处解释；
- **关系代数**：选择 σ、投影 π、连接 ⋈——`Query` 结构体就是这三个算子的具体化，第 4 节详述；
- **MySQL 对照**：MySQL 使用手写递归下降解析器（8.0 之前是 Bison，之后逐步重写），PG 则坚持 Flex+Bison 生成，第 2.4 节讨论这一取舍；
- **阅读工具**：打开 GUC `debug_print_parse = on`，日志里会打印语义分析后的 Query 树；调试时对任意节点指针执行 `call pprint(ptr)` 同样有效。

## 第 1 节 · 词法分析：scan.l

`src/backend/parser/scan.l`（1430 行）由 Flex 编译成 `scan.c`，导出核心函数 `core_yylex()`。它把输入字符串切成 token：关键字、标识符、常量、操作符、标点。

### 1.1 Flex 规则结构：排他状态机

Flex 的匹配规则是"最长匹配优先，等长时先定义者优先"。scan.l 开头的注释（scan.l:148-171）说明了它的组织方式：**用排他状态（exclusive state）处理所有"内部语法与外面不同"的片段**。状态声明在 scan.l:174-184：

```
%x xb        位串字面量 B'0101'
%x xc        扩展 C 风格注释 /* ... */（支持嵌套！）
%x xd        双引号定界标识符 "MyTable"
%x xh        十六进制串 X'FF'
%x xq        标准单引号字符串
%x xqs       引号结束后的"续接检测"状态
%x xe        扩展字符串 E'\n'（支持反斜杠转义）
%x xdolq     美元引用 $tag$ ... $tag$
%x xui/xus   带 Unicode 转义的标识符/字符串 U&"..." / U&'...'
%x xeu       扩展字符串中的 Unicode 代理对
```

进入 `xq` 状态后，`INITIAL` 状态的所有规则全部失效，直到读到收尾引号 `BEGIN(INITIAL)` 才回来——这就是词法层面的"模式切换"，避免了在一条正则里描述整个字符串语法的组合爆炸。

三个值得走读的细节：

**(1) 嵌套注释。** SQL 标准不要求 `/* /* */ */` 合法，但 PG 支持。实现是在 `xc` 状态里维护一个深度计数器（scan.l:444 附近）：

```c
{xcstart}		{                       /* 在 INITIAL 状态遇到斜杠星 */
        SET_YYLLOC();
        yyextra->xcdepth = 0;           /* 嵌套深度清零 */
        BEGIN(xc);                      /* 切入注释状态 */
        yyless(2);                      /* 只消费两个字符，防止 "/*/" 之类误吞 */
    }
<xc>{
{xcstart}		{ (yyextra->xcdepth)++; yyless(2); }   /* 注释里又见注释：深度+1 */
{xcstop}		{
        if (yyextra->xcdepth <= 0)
            BEGIN(INITIAL);             /* 最外层注释结束，回到正常状态 */
        else
            (yyextra->xcdepth)--;       /* 只是内层结束 */
    }
```

**(2) 美元引用（dollar quoting）**。`$func$ ... $func$` 是 PG 独有语法，专为"字符串里还有字符串"的场景（写函数体）设计：`xdolq` 状态记住开头的 tag，只在遇到**完全相同**的 tag 时才结束，中间的引号、反斜杠一概不解释。

**(3) 位置追踪**。每条规则开头都调用 `SET_YYLLOC()`（scan.l:98）：

```c
#define SET_YYLLOC()  (*(yylloc) = yytext - yyextra->scanbuf)
```

它把 token 在原始字符串中的字节偏移存入 `yylloc`，一路带进 raw tree 的 `location` 字段、再进 `Query`。你在 psql 里看到报错信息下方那个精确指向出错位置的 `^`，就靠这条从词法层贯穿到语义层的位置链（`parser_errposition()`）。

### 1.2 关键字处理：kwlist.h 的单一数据源技巧

PG 有 500 多个关键字。它们**只在一个文件里定义一次**：`src/include/parser/kwlist.h`。该文件通篇是这样的宏调用（按 ASCII 排序）：

```c
/* name, value, category, is-bare-label */
PG_KEYWORD("abort", ABORT_P, UNRESERVED_KEYWORD, BARE_LABEL)        /* kwlist.h:28 */
PG_KEYWORD("between", BETWEEN, COL_NAME_KEYWORD, BARE_LABEL)        /* kwlist.h:60 */
PG_KEYWORD("like", LIKE, TYPE_FUNC_NAME_KEYWORD, BARE_LABEL)        /* kwlist.h:266 */
PG_KEYWORD("select", SELECT, RESERVED_KEYWORD, BARE_LABEL)          /* kwlist.h:418 */
```

`PG_KEYWORD` 宏本身**不在此文件定义**——每个使用方自带定义再 `#include`，同一份数据被展开成不同的形态：词法器要"名字→token 号"的查找表，gram.y 要 token 声明，`pg_get_keywords()` 系统函数要展示列表。这是 C 语言的 "X-Macro" 惯用法，保证关键字表永不失步。构建期 `gen_keywordlist.pl` 还会从它生成**完美哈希函数**（perfect hash），查找入口在 `ScanKeywordLookup()`（src/common/kwlookup.c:38）：

```c
len = strlen(str);
if (len > keywords->max_kw_len)         /* 比最长关键字还长？直接排除 */
    return -1;
h = keywords->hash(str, len);           /* 完美哈希：无冲突，一次命中 */
if (h < 0 || h >= keywords->num_keywords)
    return -1;
kw = GetScanKeyword(h, keywords);
while (*str != '\0')                    /* 逐字符比对，同时做 ASCII-only 小写化 */
{
    char  ch = *str++;
    if (ch >= 'A' && ch <= 'Z')
        ch += 'a' - 'A';                /* 故意不用 tolower()：土耳其语等 locale 里
                                         * 'I' 的小写不是 'i'，会破坏 SQL99 语义 */
    if (ch != *kw++)
        return -1;
}
```

词法器的标识符规则（scan.l:1056）先查这张表，查不到才降级为普通标识符：

```c
{identifier}	{
        kwnum = ScanKeywordLookup(yytext, yyextra->keywordlist);
        if (kwnum >= 0)
        {   /* 是关键字：返回 gram.y 里对应的 token 号 */
            yylval->keyword = GetScanKeyword(kwnum, yyextra->keywordlist);
            return yyextra->keyword_tokens[kwnum];
        }
        /* 不是关键字：转小写、必要时截断到 NAMEDATALEN-1，返回 IDENT */
        ident = downcase_truncate_identifier(yytext, yyleng, true);
        yylval->str = ident;
        return IDENT;
    }
```

关键字分为四个**保留级别**（kwlist.h 第三列），对应 gram.y 末尾的四个产生式：`unreserved_keyword`（gram.y:18898，可以直接当表名列名用）、`col_name_keyword`（gram.y:19257，可当列名不可当函数名）、`type_func_name_keyword`（gram.y:19334）、`reserved_keyword`（gram.y:19366，完全保留）。这就是为什么 `CREATE TABLE abort(...)` 合法而 `CREATE TABLE select(...)` 报错——PG 尽量把关键字降级为非保留，减少与用户命名的冲突，这些产生式让"关键字当标识符用"成为语法的一部分而非词法特判。

### 1.3 小结

- scan.l 用**排他状态**隔离字符串/注释/位串等子语言，嵌套注释靠深度计数器；
- 关键字是"单一数据源 kwlist.h + X-Macro + 构建期完美哈希"，查找 O(1) 且各组件永不失步；
- 大小写折叠是 ASCII-only 的（SQL99 要求），标识符统一转小写——这就是 PG "不加双引号即小写"行为的源头；
- 每个 token 携带字节偏移 `location`，支撑贯穿全程的精确错误定位。

## 第 2 节 · 语法分析：gram.y 与 raw parse tree

`src/backend/parser/gram.y` 有 21016 行，是 PG 最大的单个源文件，由 Bison 编译成 LALR(1) 状态机。语义动作只做一件事：**把 token 组装成 C 结构体**（raw parse tree），不查目录、不做类型检查。

### 2.1 入口与 token 过滤器

`raw_parser()`（parser.c:42）先初始化 Flex 扫描器，再调 `base_yyparse()`。但 Bison 拿到的 token 并非直接来自 `core_yylex()`，中间隔着一层过滤器 `base_yylex()`（parser.c:111）。它存在的原因写在注释里（parser.c:92-94）：**标准 SQL 语法有些地方需要两个 token 的前瞻，而 LALR(1) 只允许一个**。解法是在过滤器里偷看下一个 token，把某些组合合并成"虚构 token"：

```c
/* parser.c:208 —— NOT 后面跟着 BETWEEN/IN/LIKE 等时，替换成 NOT_LA */
case NOT:
    switch (next_token) {
        case BETWEEN: case IN_P: case LIKE: ...
            cur_token = NOT_LA;
```

`NOT_LA`、`WITH_LA`、`FORMAT_LA` 等在 gram.y:864 声明，词法器永远不会产出它们。`NOT_LA` 的用途很精妙：让 `a NOT LIKE b` 中的 NOT 拿到 LIKE 的优先级而不是逻辑 NOT 的优先级（见 gram.y:16009 的 `%prec NOT_LA`）。此外 `raw_parser()` 的 `RawParseMode` 参数可以向 token 流头部注入 `MODE_PLPGSQL_EXPR` 等特殊 token（parser.c:57-70），让同一份语法从"表达式"等非常规起点开始解析——PL/pgSQL 就是这样复用主语法的。

### 2.2 SelectStmt 产生式走读

SELECT 语法被拆成三层，是阅读 gram.y 的最佳入口：

```
SelectStmt        (gram.y:13639)  顶层
 ├─ select_no_parens  (gram.y:13659)  不带外层括号：负责挂 ORDER BY/LIMIT/WITH/FOR UPDATE
 └─ select_with_parens (gram.y:13643) 带括号：'(' select_no_parens ')'
      select_no_parens → simple_select | select_clause sort_clause | ...
       └─ simple_select (gram.y:13751) 核心：SELECT ... FROM ... WHERE ...，
                                       以及 UNION/INTERSECT/EXCEPT、VALUES、TABLE t
```

`simple_select` 第一条产生式的语义动作（gram.y:13751）：

```c
simple_select:
    SELECT opt_all_clause opt_target_list
    into_clause from_clause where_clause
    group_clause having_clause window_clause
        {
            SelectStmt *n = makeNode(SelectStmt);   /* 分配节点并打上 NodeTag */
            n->targetList = $3;      /* 投影列表：List<ResTarget> */
            n->intoClause = $4;      /* SELECT INTO 目标（罕用） */
            n->fromClause = $5;      /* FROM 项列表，元素是 RangeVar/JoinExpr 等 */
            n->whereClause = $6;     /* WHERE 表达式树（A_Expr 等原始节点） */
            n->groupClause = ($7)->list;
            n->havingClause = $8;
            n->windowClause = $9;
            $$ = (Node *) n;
        }
```

可以看到语义动作纯粹是"抄写员"：`$3`、`$5` 这些子产生式的结果被原样塞进字段。集合操作也在这一层（gram.y:13805）：

```c
| select_clause UNION set_quantifier select_clause
    { $$ = makeSetOp(SETOP_UNION, $3 == SET_QUANTIFIER_ALL, $1, $4); }
```

`SelectStmt` 结构体（src/include/nodes/parsenodes.h:2328）因此身兼两职：叶子节点表示一个普通 SELECT（用 `targetList`/`fromClause` 等字段），内部节点表示 UNION/INTERSECT/EXCEPT（用 `op`/`larg`/`rarg` 字段，parsenodes.h:2370-2373）。注释（parsenodes.h:2310-2317）解释了动机：同一节点类型让 gram.y 挂 ORDER BY/LIMIT 时无需区分简单/复合 SELECT。

为什么 `ORDER BY` 只出现在 `select_no_parens` 而不在 `simple_select`？gram.y:13737-13746 的注释给出 SQL 标准语义：`SELECT foo UNION SELECT bar ORDER BY baz` 必须解析成 `(SELECT foo UNION SELECT bar) ORDER BY baz`——把排序类子句提升到不带括号的最外层，正是用语法结构编码这条规则。

### 2.3 %prec 与优先级：用声明消灭歧义

表达式语法 `a_expr`（gram.y:15913）如果朴素地写 `a_expr: a_expr '+' a_expr | a_expr '*' a_expr | ...`，会产生大量 shift/reduce 冲突。Bison 的解法是**优先级声明**（gram.y:881-953，从低到高）：

```
%left    UNION EXCEPT          ← 集合操作优先级最低
%left    OR
%left    AND
%right   NOT
%nonassoc IS ISNULL NOTNULL
%nonassoc '<' '>' '=' LESS_EQUALS GREATER_EQUALS NOT_EQUALS
%nonassoc BETWEEN IN_P LIKE ILIKE SIMILAR NOT_LA
%left    '+' '-'
%left    '*' '/' '%'
%left    '^'
%right   UMINUS                ← 一元负号：并无这个 token，纯粹是"优先级占位符"
%left    TYPECAST              ← :: 转换优先级极高
```

`UMINUS` 展示了 `%prec` 的经典用法（gram.y:15950）：

```c
| '-' a_expr        %prec UMINUS     { $$ = doNegate($2, @1); }
```

`-` 作为二元减号优先级较低，但一元负号必须更高（`-a * b` 应解析为 `(-a) * b`）。`%prec UMINUS` 把**这一条产生式**的优先级改成 UMINUS 的，而不影响 token `-` 本身。gram.y:890-900 的注释是一段方法论：给关键字全局设优先级"有全局副作用，可能掩盖别处的歧义"，因此应优先使用局部的 `%prec`；如果必须给关键字设优先级，尽量设成与 IDENT 相同。连 `SelectStmt: select_no_parens %prec UMINUS`（gram.y:13639）也用了 `%prec`——用于解决与圆括号（行构造器 vs 子查询）的歧义。

### 2.4 为什么 PG 坚持 LALR，而 MySQL 走向递归下降？

这是一个真实的工程取舍，两边都有充分理由：

| 维度 | PG（Bison/LALR） | MySQL（8.0 起逐步手写递归下降） |
|---|---|---|
| **歧义检测** | Bison 在构建期**证明**语法无冲突（PG 要求 0 shift/reduce 冲突）；改语法引入歧义会立即编译失败 | 手写解析器的歧义只能靠测试和 code review 发现 |
| **可维护性** | 语法即文档：gram.y 的产生式几乎能与 SQL 标准 BNF 对照 | 解析逻辑散落在函数调用里，与语义代码交织 |
| **错误恢复/提示** | 较弱：LALR 出错时只知道"此处不期望该 token"；PG 用 location + `^` 弥补 | 强项：可以在任意上下文写定制的错误提示与恢复 |
| **上下文相关处理** | 别扭：需要 base_yylex 过滤器、四级关键字、%prec 等技巧维持 LALR(1) | 天然容易：解析函数里随手 if 判断上下文 |
| **性能** | 表驱动状态机，缓存友好，常数小 | 调用栈深，但可针对热点手工优化 |

PG 的选择本质上是**把正确性证明交给工具**：SQL 语法极其庞大（500+ 关键字、20000 行产生式），社区每个版本都在加语法，"Bison 报冲突就不许合并"这条硬约束让几百人协作改同一个语法文件成为可能。代价则是所有上下文相关性都要用第 2.1 节的 token 过滤器和第 1.2 节的关键字分级去"驯化"成 LALR(1) 可解的形态。MySQL 则更看重报错体验与灵活性，愿意承担手写解析器的维护成本。没有对错，只有权衡——但如果你要向一个几十年生命周期、松散协作的项目学架构，PG 的选择更值得体会。

### 2.5 小结

- gram.y 产出的 raw parse tree 是"SQL 语句的忠实转写"：`SelectStmt` 的字段与 SQL 子句一一对应，不含任何目录信息；
- 一个 token 的前瞻不够时用 `base_yylex` 过滤器合成 `NOT_LA` 等虚构 token，维持 LALR(1)；
- 歧义靠优先级声明与 `%prec` 局部标注解决，社区纪律是零冲突；
- LALR vs 递归下降是"工具保证正确性"与"手工换取灵活性"的取舍。

## 第 3 节 · 语义分析：analyze.c 与 parse_*.c 家族

语义分析是解析器里**真正有数据库味道**的部分：查目录、绑定名字、推导类型、消解重载。总入口 `parse_analyze_fixedparams()`（analyze.c:128）建立 `ParseState`（解析期的"工作台"，挂着范围表、名字空间、聚合标志等状态），然后调 `transformStmt()`。

### 3.1 transformStmt 与 transformSelectStmt

`transformStmt()`（analyze.c:335）是按语句类型分发的 switch：

```c
switch (nodeTag(parseTree))
{
    case T_InsertStmt:  result = transformInsertStmt(...); break;
    case T_DeleteStmt:  result = transformDeleteStmt(...); break;
    case T_UpdateStmt:  result = transformUpdateStmt(...); break;
    case T_MergeStmt:   result = transformMergeStmt(...);  break;
    case T_SelectStmt:
        {
            SelectStmt *n = (SelectStmt *) parseTree;
            if (n->valuesLists)                 /* VALUES (...),(...) */
                result = transformValuesClause(pstate, n);
            else if (n->op == SETOP_NONE)       /* 普通 SELECT */
                result = transformSelectStmt(pstate, n, NULL);
            else                                /* UNION/INTERSECT/EXCEPT 树 */
                result = transformSetOperationStmt(pstate, n);
        }
        break;
    ...
    default:            /* DDL 等工具语句：不做变换，包一层 CMD_UTILITY 的 Query */
        result = makeNode(Query);
        result->commandType = CMD_UTILITY;
        result->utilityStmt = parseTree;
}
```

注意 default 分支：**DDL 不经过语义分析**（也不经过重写和优化），原始树直接被塞进 `Query.utilityStmt`，一路送到 `ProcessUtility`。这就是"PG 的查询处理管线只服务 DML"的出处。

`transformSelectStmt()`（analyze.c:1743）是主战场，处理顺序本身就是语义规则（逐段解说）：

```c
qry->commandType = CMD_SELECT;
if (stmt->withClause)                            /* ① WITH 最先、独立处理 */
    qry->cteList = transformWithClause(pstate, stmt->withClause);

pstate->p_locking_clause = stmt->lockingClause;  /* 供 FROM 处理时决定锁级别 */
pstate->p_windowdefs = stmt->windowClause;

transformFromClause(pstate, stmt->fromClause);   /* ② FROM：建范围表+名字空间（3.2 节） */

qry->targetList = transformTargetList(pstate,    /* ③ SELECT 列表：此时 FROM 的列名 */
        stmt->targetList, EXPR_KIND_SELECT_TARGET);  /*   已可见，'*' 在这里展开 */

qual = transformWhereClause(pstate, stmt->whereClause,   /* ④ WHERE：表达式变换 */
        EXPR_KIND_WHERE, "WHERE");                       /*   并强制结果为 boolean */
qry->havingQual = transformWhereClause(pstate, stmt->havingClause,
        EXPR_KIND_HAVING, "HAVING");             /* HAVING 的初步处理与 WHERE 相同 */

qry->sortClause = transformSortClause(...);      /* ⑤ 先 ORDER BY 后 GROUP BY——  */
qry->groupClause = transformGroupClause(...);    /*   两者都可能往 targetList 里加   */
                                                 /*   隐藏列（resjunk），且 GROUP BY  */
                                                 /*   需要参照 ORDER BY 的结果        */
...                                              /* ⑥ DISTINCT [ON] / LIMIT / WINDOW */

qry->rtable = pstate->p_rtable;                  /* ⑦ 收尾：把工作台上的成果装入 Query */
qry->rteperminfos = pstate->p_rteperminfos;
qry->jointree = makeFromExpr(pstate->p_joinlist, qual);  /* FROM 与 WHERE 合体！ */

qry->hasSubLinks = pstate->p_hasSubLinks;        /* ⑧ 各种 has* 标志，供后续阶段快速判断 */
qry->hasAggs = pstate->p_hasAggs;
...
assign_query_collations(pstate, qry);            /* ⑨ 排序规则（collation）统一指派 */
if (pstate->p_hasAggs || qry->groupClause || ...)
    parseCheckAggregates(pstate, qry);           /* ⑩ 聚合合法性检查（裸列必须在 GROUP BY 里）*/
```

第⑦步值得停一秒：**WHERE 条件没有自己的字段，它作为 qual 被并入 `jointree`**——这正是关系代数的视角：选择 σ 与连接 ⋈ 同属"从关系集合筛出元组"的操作（第 4 节展开）。

处理顺序即名字可见性规则：FROM 先于 targetList（所以 SELECT 里能引用表列），targetList 先于 ORDER BY（所以 ORDER BY 能引用输出列别名），而 WHERE 在 targetList 之后处理却**不能**引用别名——因为 `transformWhereClause` 查名字时只查 FROM 建立的名字空间，不查 targetList。你在 MySQL 里习惯的 `WHERE 别名` 报错，在 PG 里同样成立，根源就在这段代码的名字空间设计。

### 3.2 parse_relation.c：名字解析与 RangeTblEntry

`transformFromClause()`（src/backend/parser/parse_clause.c:116）遍历 FROM 项，每项交给 `transformFromClauseItem()`（parse_clause.c:1190）递归处理（裸表、子查询、JOIN 树、函数、LATERAL……）。裸表最终到达 `addRangeTableEntry()`（src/backend/parser/parse_relation.c:1511）：

```c
RangeTblEntry *rte = makeNode(RangeTblEntry);
rte->rtekind = RTE_RELATION;
rte->alias = alias;
/* 决定锁级别：FOR UPDATE 目标拿 RowShareLock，否则 AccessShareLock */
lockmode = isLockedRefname(pstate, refname) ? RowShareLock : AccessShareLock;

rel = parserOpenTable(pstate, relation, lockmode);   /* ★ 查目录并加锁！        */
rte->relid = RelationGetRelid(rel);                  /* 从此表以 OID 存在        */
rte->relkind = rel->rd_rel->relkind;                 /* 普通表/视图/分区表...    */
rte->rellockmode = lockmode;

rte->eref = makeAlias(refname, NIL);                 /* 有效列名表：用户别名与    */
buildRelationAliases(rel->rd_att, alias, rte->eref); /* 真实列名合成            */

perminfo = addRTEPermissionInfo(&pstate->p_rteperminfos, rte);
perminfo->requiredPerms = ACL_SELECT;                /* 记下"执行期需检查 SELECT 权限" */

pstate->p_rtable = lappend(pstate->p_rtable, rte);   /* 挂入范围表，下标即 varno  */
...
table_close(rel, NoLock);   /* 释放引用计数，但锁保留到事务结束——      */
                            /* 防止解析和执行之间表被 DROP/ALTER        */
```

`parserOpenTable()`（parse_relation.c:1459）内部走 relcache 完成"名字 → OID"——**这是整条 SQL 处理路径上第一次接触系统目录**，也是第一次加锁。锁在语句结束后仍持有到事务结束，保证 `Query` 里的 OID 在本事务内始终有效。

列名解析在表达式变换时发生（3.3 节会调回来）：`refnameNamespaceItem()`（parse_relation.c:130）按"表别名.列名"定位 RTE；`colNameToVar()`（parse_relation.c:926）处理不带限定的列名——遍历当前名字空间的所有 RTE 调 `scanRTEForColumn()`（parse_relation.c:843），**若多个 RTE 都有该列则报 ambiguous 错误**。解析成功的产物是 `Var` 节点：`(varno, varattno)` = "范围表第几项的第几列"，从此表达式里再无字符串名字。

RTE 的种类由 `RTEKind`（parsenodes.h:1122）枚举：`RTE_RELATION`（真实表）、`RTE_SUBQUERY`（FROM 里的子查询，**视图展开后也是它**）、`RTE_JOIN`、`RTE_FUNCTION`、`RTE_VALUES`、`RTE_CTE`、`RTE_GROUP` 等。`RangeTblEntry` 结构体定义在 parsenodes.h:1139。

### 3.4 parse_oper.c / parse_func.c：操作符与函数的重载消解

PG 的操作符和函数都可重载（同名不同参数类型），且用户可任意扩展。`=` 在系统目录里就有几十个版本。给定 `int4 = numeric` 这样的调用，选哪个？这套算法是 PG 类型系统的精华，文档见官方手册 "Type Conversion" 章，实现分两级。

**第一级：精确匹配。** `oper()`（parse_oper.c:376）先调 `binary_oper_exact()`（parse_oper.c:266）：

```c
/* 一侧是 unknown 字面量？假定它与另一侧同类型 */
if ((arg1 == UNKNOWNOID) && (arg2 != InvalidOid))
    { arg1 = arg2; was_unknown = true; }
...
result = OpernameGetOprid(opname, arg1, arg2);   /* 直查 pg_operator，命中即返回 */
...
if (was_unknown)          /* 还试一次：把域类型（domain）退化到基类型再查 */
    basetype = getBaseType(arg1);
```

所以 `where id = '42'`（id 是 int4，'42' 是 unknown）直接命中 `int4 = int4`，零开销。精确匹配失败才进入第二级。

**第二级：候选集筛选。** `oper_select_candidate()`（parse_oper.c:317）先取出所有同名二元操作符，然后：

```c
/* 第 1 步：淘汰"隐式转换也到不了"的候选 */
ncandidates = func_match_argtypes(nargs, input_typeids, candidates, &candidates);
if (ncandidates == 0)  return FUNCDETAIL_NOTFOUND;   /* 没有：报"操作符不存在" */
if (ncandidates == 1)  { *operOid = candidates->oid; return FUNCDETAIL_NORMAL; }
/* 第 2 步：还剩多个 → 与函数共用同一套启发式打分 */
candidates = func_select_candidate(nargs, input_typeids, candidates);
```

`func_match_argtypes()`（src/backend/parser/parse_func.c:1043）的判据是 `can_coerce_type(..., COERCION_IMPLICIT)`——只考虑**隐式**转换路径（3.5 节解释三个转换等级）。

真正的裁决者是 `func_select_candidate()`（parse_func.c:1129），一套按顺序应用、每步"保留得分最高者"的启发式（源码中 1089-1127 行的历史注释保留了 1993-1998 年间这套规则的演化争论，值得一读）：

1. **精确位次最多者胜**（parse_func.c:1185-1227）：逐参数位统计"候选参数类型 == 输入类型（域退化到基类型后）"的个数 `nmatch`，只保留 nmatch 最大的候选。剩 1 个即返回。
2. **优先类型（preferred type）加分**（parse_func.c:1229-1276）：仍多于 1 个时，重新打分——参数位上"精确匹配 **或** 候选类型是输入类型所属类别（category）中的优先类型"都算分。类别与优先类型来自 `pg_type.typcategory/typispreferred`：数值类别的优先类型是 `float8`，字符串类别是 `text`。这一步解释了"整数与浮点混算为什么倒向 float8"。注意 7.4 加入的限制（源码注释）：只认**同类别**内的优先类型，杜绝跨类别乱升格。
3. **unknown 参数归类**（parse_func.c:1278-1361）：若仍未定且存在 unknown 输入，对每个 unknown 位收集所有候选在该位的类型类别——**若有任一候选是 STRING 类别，则该位定为 STRING**（"unknown 字面量长得就像字符串"，parse_func.c:1290 注释）；否则要求所有候选同类别，有冲突即失败。然后淘汰类别不合、或（当该位存在优先类型候选时）非优先类型的候选。
4. **最后一搏**（函数后段）：若所有已知参数同为一个类型，尝试假定 unknown 参数也是该类型，看是否恰好唯一命中。

走完仍剩多个候选，报 `operator is not unique` / `function is not unique`，要求用户显式加 cast——**宁可报错，绝不猜错**，这是整套算法的设计底线（parse_func.c:1094-1097 的历史注释原话："_never_ doing the wrong thing"）。

函数调用的消解入口是 `func_get_detail()`（parse_func.c:1522），流程同构：`FuncnameGetCandidates()` 取候选（还要处理 VARIADIC、参数默认值、命名参数），先精确匹配，再 `func_match_argtypes` + `func_select_candidate`。操作符消解复用函数的启发式（parse_func.c:1084-1085 注释：parse_oper.c 原有的重复代码已被合并）。

另一处消解发生在 ORDER BY/GROUP BY：`get_sort_group_operators()`（parse_oper.c:183）通过类型的**默认 btree 操作符类**找 `<`/`=`/`>`，这就是"某类型能否排序"的判据。

### 3.5 parse_coerce.c：隐式类型转换

消解算法反复问"能否转换"，回答者是 parse_coerce.c。转换强度分三级（src/include/nodes/primnodes.h:735 的 `CoercionContext`）：

- `COERCION_IMPLICIT`：表达式里可自动发生（最严格）；
- `COERCION_ASSIGNMENT`：赋值语境（INSERT/UPDATE 的目标列）额外允许，比如 `int4 → text` 存储时允许，表达式里不允许；
- `COERCION_EXPLICIT`：仅当用户写 `CAST` / `::` 才允许。

哪条转换在哪级可用，记录在系统目录 **pg_cast** 里（`castcontext` 列），查询入口 `find_coercion_pathway()`（parse_coerce.c:3157）。也就是说：**隐式转换规则本身是目录数据，可由扩展增删**——这是 Berkeley "可扩展数据库"血统的又一体现（当然，手册严厉警告往 pg_cast 里加隐式转换是自找歧义）。

执行转换的是 `coerce_type()`（parse_coerce.c:158），最重要的分支是 unknown 字面量的落地（parse_coerce.c:233）：

```c
if (inputTypeId == UNKNOWNOID && IsA(node, Const))
{
    /* 之前悬而未决的字符串常量：直接调用目标类型的输入函数（typinput），
     * 在解析期就折叠成目标类型的 Const。
     * 注意不能与一般转换合并：int4 的输入函数会拒绝 "1.2"，
     * 而 float→int 的转换函数会四舍五入——两者语义不同。 */
    baseTypeId = getBaseTypeAndTypmod(targetTypeId, &baseTypeMod);
    ...
    newcon->constvalue = stringTypeDatum(baseType, con->constvalue ...);
```

这解释了 PG 一个与 MySQL 截然不同的行为：`'abc'` 这种字面量**没有固有类型**，它是什么类型完全由上下文决定；`WHERE id = '42'` 里它变成 int4 常量（于是索引能用），`WHERE name = 42` 则会因为找不到 `text = int4` 的隐式路径而报错——PG 拒绝 MySQL 式的"字符串和数字随便比"（那会让索引悄悄失效）。

其他入口：`coerce_to_target_type()`（parse_coerce.c:79）是带 typmod 处理的总入口；`coerce_type_typmod()`（parse_coerce.c:752）负责长度约束（`varchar(10)` 的 10）；`select_common_type()`（parse_coerce.c:1342）为 UNION/CASE/COALESCE 的多个分支选统一输出类型（规则：全 unknown 则 text；否则取第一个非 unknown 类型所在类别，类别内优先类型优先，且必须能从所有分支隐式转到）。

### 3.6 小结

- 语义分析的输入是"字符串写法"，输出是"OID 写法"：表→`relid`、列→`Var(varno,varattno)`、操作符→`OpExpr.opno`、函数→`FuncExpr.funcid`；
- 名字解析伴随**加锁**（parserOpenTable），锁保持到事务结束，这是 Query 中 OID 有效性的保证；
- 重载消解 = 精确匹配 → 隐式可达性过滤 → 精确位次计分 → 同类别优先类型加分 → unknown 归类（STRING 偏置）→ 仍歧义则报错；
- 三级转换语境（implicit/assignment/explicit）+ pg_cast 目录 = 可扩展但克制的类型转换体系；unknown 字面量在解析期由目标类型的输入函数折叠成 Const。

## 第 4 节 · Query：关系代数表达式的具体化

`Query`（src/include/nodes/parsenodes.h:120）是语义分析的输出、重写器和优化器的输入，是 PG 最重要的结构体之一。先看它与关系代数的对应——一条 `SELECT a.x, b.y FROM a JOIN b ON a.id=b.id WHERE a.x > 0` 的代数形式是：

```
π_{a.x, b.y} ( σ_{a.x > 0} ( a ⋈_{a.id=b.id} b ) )
     │               │              │
 targetList     jointree 的 quals   jointree 的连接结构，叶子是 rtable 的下标
```

- **rtable**（parsenodes.h:180）：范围表，查询涉及的所有"关系"的清单（真实表、子查询、函数、CTE……），即代数表达式的**操作数集合**。表达式里的 `Var.varno` 都是它的 1-based 下标；
- **jointree**（parsenodes.h:187）：一个 `FromExpr`，`fromlist` 是连接结构（叶子为 `RangeTblRef`，内部节点为 `JoinExpr`），`quals` 是 WHERE 条件——**σ 与 ⋈ 合并存放**，因为对优化器而言两者都是"限制元组的谓词"，内连接的 ON 条件和 WHERE 条件可以自由互换位置；
- **targetList**（parsenodes.h:203）：`TargetEntry` 列表，即投影 π。`resjunk=true` 的隐藏列（供 ORDER BY/FOR UPDATE 用，最终不输出）也在其中。

rtable 与 jointree 的分离是刻意的：**rtable 是"提到了哪些关系"，jointree 是"它们怎样组合"**。UPDATE 的目标表在 rtable 里但可以不在 jointree 里；视图展开后废弃的 RTE 仍留在 rtable（保序，Var 编号不变）只是从 jointree 摘除。

其余字段逐组解说（行号均为 parsenodes.h）：

| 字段 | 行号 | 含义 |
|---|---|---|
| `commandType` | 124 | SELECT/INSERT/UPDATE/DELETE/MERGE/UTILITY |
| `querySource` | 127 | 原始语句还是规则产生（QSRC_ORIGINAL/QSRC_INSTEAD_RULE…），重写器用 |
| `queryId` | 139 | query jumbling 的输出，pg_stat_statements 的主键 |

| `hasAggs`…`hasGroupRTE` | 156-174 | 一组标志位缓存："有聚合/窗口函数/子链接/递归 CTE/RLS…"，让后续阶段免于全树扫描 |
| `cteList` | 178 | WITH 子句的 CommonTableExpr 列表 |
| `rteperminfos` | 186 | 与 rtable 平行的权限检查信息（PG 16 起从 RTE 中拆出） |

一句话总结：**Query 就是"SQL 语义的完备清单"**——SELECT 的每个语法成分在这里都有一个规范化的、以 OID 和下标表达的席位；优化器全部工作就是对这棵树做保语义变换。

### 小结

- `rtable`/`jointree`/`targetList` 分别对应关系代数的操作数集合、σ+⋈、π；`groupClause`+`havingQual` 是扩展代数的 γ+σ，`setOperations` 是 ∪∩−；
- WHERE 与 JOIN 条件同住 jointree，为优化器的谓词下推、连接重排铺路;
- 一批 `has*` 标志位是给后续阶段的性能提示，修改内核时必须维护其一致性。

## 第 5 节 · 重写器：规则系统、视图与行级安全

### 5.1 历史背景：Stonebraker 的规则系统

重写器（`src/backend/rewrite/`）是 Berkeley POSTGRES 时代的直系遗产。Stonebraker 在 1980 年代提出：与其为触发器、视图、完整性约束各造一套机制，不如提供统一的**声明式规则系统**——"当对表 R 执行事件 E 时，改做（INSTEAD）或附加做（ALSO）动作 A"，且规则在**查询树层面**改写（一次改写服务整条语句），而非像触发器那样逐行触发。这套思想沉淀为 `CREATE RULE` 语法和 pg_rewrite 目录，规则的内存形态是 `RewriteRule`（src/include/rewrite/prs2lock.h:24——文件名里的 "prs2" 即 POSTGRES Rule System v2）。

四十年后的今天，用户级 `CREATE RULE` 已不推荐使用（语义陷阱多，触发器更直观），但规则系统的两大核心应用活得很好：**视图**（每个视图 = 一条系统自动创建的 `ON SELECT DO INSTEAD` 规则，动作就是视图定义查询）和作为**可更新视图/RLS 的实现骨架**。

### 5.2 QueryRewrite 总入口

`QueryRewrite()`（src/backend/rewrite/rewriteHandler.c:4789）分三步，源码结构极清晰：

```c
/* Step 1: 应用所有非 SELECT 规则（INSERT/UPDATE/DELETE 上的规则），
 *         一条 Query 可能变成 0 条（无条件 INSTEAD NOTHING）或多条 */
querylist = RewriteQuery(parsetree, NIL, 0, 0);

/* Step 2: 对每条结果应用 RIR 规则（"Retrieve-Instead-Retrieve"，即视图展开），
 *         RLS 也在 fireRIRrules 里注入 */
foreach(l, querylist)
    query = fireRIRrules(query, NIL);

/* Step 3: 决定谁设置命令标签：原始查询还在就是它；
 *         否则由最后一条同类 INSTEAD 规则查询顶替（rewriteHandler.c:4847-4875） */
```

输出是 `List<Query>`——一进多出。这是重写器最反直觉的接口：一条 `INSERT INTO v ...` 在规则作用下可能变成"往 A 表插一条 + 往日志表插一条"两条 Query，随后被逐条优化、逐条执行。

### 5.3 视图展开：ApplyRetrieveRule

`fireRIRrules()`（rewriteHandler.c:2042）用 while 循环遍历 rtable（rewriteHandler.c:2066 的注释提醒：不能用 foreach，因为展开过程会往 rtable 追加元素）：子查询 RTE 递归处理；`RTE_RELATION` 若其 relcache 里挂着 ON SELECT 规则（即它是视图），调用 `ApplyRetrieveRule()`（rewriteHandler.c:1762）。核心动作在 rewriteHandler.c:1869-1900：

```c
rule_action = copyObject(linitial(rule->actions));   /* 取出视图定义查询（存于
                                                      * pg_rewrite，深拷贝一份） */
AcquireRewriteLocks(rule_action, true, (rc != NULL)); /* 给视图引用的表加锁——
                                                      * 这些表不在原查询里，解析期没锁过 */
rule_action = fireRIRrules(rule_action, activeRIRs); /* 递归：视图套视图逐层展开，
                                                      * activeRIRs 记录展开链防无限递归 */
parsetree->hasRowSecurity |= rule_action->hasRowSecurity;

/* 偷梁换柱：视图的 RTE 原地从"表"变成"子查询" */
rte = rt_fetch(rt_index, parsetree->rtable);
rte->rtekind = RTE_SUBQUERY;
rte->subquery = rule_action;
rte->security_barrier = RelationIsSecurityView(relation);
/* 注意保留 relid/relkind/rellockmode/perminfoindex（rewriteHandler.c:1903-1907 注释）：
 * 视图本身仍要加锁、仍要做权限检查，虽然它已不再作为"表"被扫描 */
```

展开后 `SELECT * FROM v` 就等价于 `SELECT * FROM (视图定义) v`——后续优化器的子查询上拉（pull_up_subqueries）通常还会把这层子查询展平，最终计划与手写内联 SQL 无异。这就是"PG 的视图零开销"的完整机制。`security_barrier` 视图是例外：该标志阻止优化器把外层谓词推进子查询，防止恶意用户用低成本的自定义函数先于安全过滤器执行而窃取行内容——这正是 RLS 的前身。

对视图执行 UPDATE/DELETE 时（rewriteHandler.c:1786-1849）：若视图有 INSTEAD OF 触发器，展开视图作为数据来源、同时保留一份未展开的 RTE 作为目标关系，并追加一个 resjunk 的整行 Var（"wholerow"）供触发器拿到旧行——自动可更新视图（无触发器、无规则的简单视图）则由更早的 `rewriteTargetView` 直接把目标换成底层表。

### 5.4 RewriteQuery：INSERT/UPDATE/DELETE 规则

`RewriteQuery()`（rewriteHandler.c:4044）处理数据修改类规则，要点：

- 先递归重写 WITH 里的 DML（rewriteHandler.c:4055-4066），因为 CTE 可能被拷进规则动作；
- 对 INSERT 目标做缺省值填充：`rewriteTargetListIU()`（rewriteHandler.c:823）把 targetList 重排成与表列序一致，未提供的列补上 DEFAULT 表达式——你以为"补默认值"是执行器的事，其实发生在重写器；
- 取出目标表上匹配事件类型的规则，生成"产品查询"（product queries）：`DO ALSO` 规则把动作查询追加到列表；`DO INSTEAD` 替换原查询；带条件（WHERE）的 INSTEAD 规则最微妙——原查询并不消失，而是被加上 `AND (rule_qual IS NOT TRUE)` 保留（rewriteHandler.c:2368-2372 注释解释了为什么必须用 `IS NOT TRUE` 而非 `NOT`：qual 为 NULL 时语义才正确）；
- 规则动作里的伪关系 OLD/NEW 被替换：OLD → 目标表的引用，NEW → 原查询 targetList 中对应表达式的拷贝；
- 递归上限：展开链超过限制（规则互相触发）时报"infinite recursion detected"。

### 5.5 行级安全（RLS）注入

RLS 也在 `fireRIRrules()` 的收尾阶段处理（rewriteHandler.c:2250-2361），时机选在所有视图展开之后（rewriteHandler.c:2250-2254 注释：否则带子查询的策略会被 query_tree_walker 二次遍历）：

```c
foreach(lc, parsetree->rtable)
{
    /* 只有普通表和分区表可能有策略 */
    if (rte->rtekind != RTE_RELATION || ...) continue;

    get_row_security_policies(parsetree, rte, rt_index,       /* rowsecurity.c:98 */
                              &securityQuals, &withCheckOptions,
                              &hasRowSecurity, &hasSubLinks);
    /* 读 pg_policy：SELECT/UPDATE 等适用 USING 表达式 → securityQuals；
     * INSERT/UPDATE 的 WITH CHECK 表达式 → withCheckOptions */

    if (securityQuals != NIL || withCheckOptions != NIL)
    {
        if (hasSubLinks) { /* 策略含子查询：为其加锁、递归展开其中的视图、防递归 */ }

        /* 注入点：挂到 RTE 的 securityQuals 链表头部——
         * 比 security barrier 视图的既有屏障更优先执行（rewriteHandler.c:2338-2343 注释） */
        rte->securityQuals = list_concat(securityQuals, rte->securityQuals);
        parsetree->withCheckOptions = list_concat(withCheckOptions, ...);
    }
    if (hasRowSecurity) parsetree->hasRowSecurity = true;
}
```

注意策略条件不是简单 AND 进 WHERE，而是放进 `rte->securityQuals`：优化器会为其维持"先于一切用户谓词求值"的屏障语义（leakproof 的函数除外），防止侧信道泄露。写方向的 `WITH CHECK` 则进入 `Query.withCheckOptions`，由执行器在插入/更新每行时校验。`hasRowSecurity` 标志还参与计划缓存失效：同一条语句在不同角色下必须使用不同计划。

### 5.6 小结

- 重写器是 Stonebraker 规则系统的现代残部：视图 = ON SELECT DO INSTEAD 规则，展开 = 把 RTE 原地改造成 RTE_SUBQUERY，随后靠优化器展平，故视图无运行期开销；
- `QueryRewrite` 一进多出：DML 规则可让一条语句裂变为多条 Query；缺省值填充、OLD/NEW 替换、条件 INSTEAD 的补集保留都在这里完成；
- RLS 在视图展开后注入 `rte->securityQuals` 与 `withCheckOptions`，带屏障语义防泄露；
- 读这一模块时刻记住不变式：**改写必须保持 Var 编号有效**——rtable 只增不删、只改不动下标。

## 自测问题

**1. 为什么 PG 要区分 raw parse tree（SelectStmt）与 Query 两种树？只用一种行不行？**

不行。raw tree 与系统目录状态无关（表名还是字符串），可长期缓存：PREPARE 语句、PL/pgSQL 都缓存 raw tree，DDL 使目录变化后重做语义分析即可。Query 里全是 OID 和下标，目录一变即失效。另外 DDL 语句只有 raw tree 形态（transformStmt 的 default 分支直接包成 CMD_UTILITY，analyze.c:435-444），根本不生成真正的查询树。

**2. `WHERE name = 42`（name 为 text 列）在 PG 报错而 MySQL 能执行，从源码角度解释原因。**

`oper()`（parse_oper.c:376）为 `=` 寻找 `(text, int4)` 的解释：精确匹配失败；候选筛选时 `func_match_argtypes()` 要求存在 COERCION_IMPLICIT 级别的转换路径，而 pg_cast 中 text 与 int4 之间没有任何隐式转换（仅 assignment/explicit 级），全部候选被淘汰，报 "operator does not exist"。MySQL 则内建了字符串与数字的自动互转（代价是隐式转换让索引失效等陷阱）。注意 `name = '42'` 反而合法：'42' 是 unknown 字面量，`binary_oper_exact()` 直接假定它为 text。

**3. `SELECT 2 + 3.5` 中 `+` 是如何被消解到 `numeric + numeric` 的？**

2 是 int4，3.5 是 numeric。精确匹配 `(int4, numeric)` 在 pg_operator 中不存在；`oper_select_candidate()` 过滤出隐式可达的候选（int4+int4、int8+int8、float8+float8、numeric+numeric 等）；`func_select_candidate()` 第一轮按精确位次计分：numeric+numeric 在第二参数位精确命中（nmatch=1），int4+int4 在第一位命中（nmatch=1），并列；第二轮加入优先类型判断仍需继续；实际上 numeric→int4 无隐式路径，所以 int4+int4 早在可达性过滤就被淘汰，numeric+numeric（int4 可隐式升 numeric）最终以最高分胜出，随后 `coerce_type()` 给 2 包上到 numeric 的转换。

**4. "PG 的视图没有性能开销"这句话的机制依据是什么？什么时候不成立？**

`ApplyRetrieveRule()`（rewriteHandler.c:1762）把视图 RTE 原地改为 RTE_SUBQUERY、塞入视图定义查询（rewriteHandler.c:1899-1900），效果与手写子查询完全相同；优化器的子查询上拉会进一步把它展平参与全局优化（谓词下推、连接重排）。不成立的情形：`security_barrier` 视图（rte->security_barrier = true，rewriteHandler.c:1901）禁止谓词穿越屏障；含聚合/DISTINCT/窗口函数等无法上拉的视图会作为独立子查询计划。

**5. Query.jointree 为什么把 WHERE 条件和 FROM 连接结构放在一起，而不给 WHERE 单独一个字段？**

关系代数中 σ（选择）与内连接 ⋈ 的条件是可交换、可合并的：`σ_p(A ⋈_q B) ≡ A ⋈_{p∧q} B`（p 只引用单侧时还可下推）。把 quals 挂在 FromExpr 上（transformSelectStmt 的 `makeFromExpr(pstate->p_joinlist, qual)`，analyze.c:1865），优化器可以统一收集全部谓词、按引用的关系集合自由决定在计划树的哪一层求值，而不必关心它语法上写在 ON 还是 WHERE 里。外连接的 ON 条件则必须留在对应 JoinExpr 上，因为它影响补 NULL 语义，不能与 WHERE 混同。
