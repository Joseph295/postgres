# 07 · 访问方法与索引：从 Table AM 到五种索引的源码走读

> 对应《PostgreSQL源码学习指南》第 8 章。所有 `文件:行号` 引用基于本仓库（20devel，master 分支）核实；行号会随版本漂移，函数名与文件路径长期稳定，漂移时用 `grep -n "^函数名" 文件路径` 重新定位。

## 本章导读

上一篇我们看清了元组如何躺在 8KB 的页里、页如何进出缓冲池。本篇回答的问题是：**数据以什么结构组织、怎么高效地增删改查**——这正是 `src/backend/access/` 目录的职责。

PostgreSQL 在这里做了一个漂亮的**双抽象**：

- **Table AM**（表访问方法，`src/include/access/tableam.h`）：表的物理存储可插拔，heap 是默认且唯一的内建实现；
- **Index AM**（索引访问方法，`src/include/access/amapi.h`）：索引可插拔，内建了 btree、hash、gist、gin、spgist、brin 六种（本文按惯例把 gist/spgist 视作一族"树框架"，合称五类）。

本篇的重心是 **B-tree 深度走读**（约占一半篇幅）：Lehman & Yao 1981 算法如何让"读不阻塞写"，`_bt_search` 的下降循环，`_bt_doinsert` 的插入与页分裂，以及 PG 12+ 的 suffix truncation 与 deduplication。随后依次拆解 GIN（倒排索引）、GiST（广义搜索树）、BRIN（块范围摘要）、Hash 与 SP-GiST，最后回到索引与堆的三种协作方式（回表、Index Only Scan、Bitmap Scan）。

如果你来自 InnoDB 世界，请先记住本篇最大的思维转换：**PG 的表是堆表，所有索引都是二级索引**。InnoDB 的主键 B+ 树叶子节点就是数据行（聚簇索引），二级索引叶子存主键值；而 PG 的 B-tree 叶子节点存的是 `(索引键, ctid)`，ctid 直接指向堆页的物理位置——没有"聚簇索引"这个概念（`CLUSTER` 命令只做一次性物理重排，不维持）。这个差异会解释后文的大量设计取舍。

## 前置知识

1. **页与元组布局**（本系列第 06 篇）：PageHeader / ItemId / special space；索引页把 special space 用作各自的元数据（如 B-tree 的左右兄弟指针）。
2. **缓冲池与锁**：`ReadBuffer` 拿页、`LockBuffer` 加页级共享/排他锁（内部是 LWLock）；pin（引用计数）与 lock（内容锁）是两回事。
3. **ctid / ItemPointer**：`(块号, 页内偏移)` 二元组，堆元组的物理地址，也是所有索引指向堆的"外键"。
4. **WAL 基本概念**：一条 WAL 记录是一个原子重做单元；跨越多条 WAL 记录的多页修改需要额外机制保证崩溃一致性——B-tree 页分裂是最好的教材（详见第 09 篇）。
5. 教科书 B+ 树及其经典加锁协议（lock coupling / crabbing）：下降时先锁父再锁子，确认子节点"安全"后才放父锁。

## 一、双抽象接口：Table AM 与 Index AM

### 1.1 Table AM：为什么 PG 12 才把表存储抽象出来

索引从 Berkeley 时代起就是可插拔的（系统目录 `pg_am` 一直存在），但**表存储直到 PG 12（2019 年）才抽象出接口**。原因值得玩味：

- 历史上执行器、VACUUM、COPY 等模块直接调用 `heap_insert`/`heap_getnext` 等函数，heap 的假设（元组不可变、xmin/xmax 可见性、ctid 指针、HOT 链）渗透进了几十个调用点；
- 真正的驱动力是社区想引入 **zheap**（undo 型存储，解决表膨胀）与列存等替代引擎，被迫先做"接口手术"：把所有 `heap_*` 直接调用改为经由回调表分发；
- 抽象的代价是接口形状仍带着浓重的 heap 色彩——比如 TID 必须是"块号+行号"的形式，MVCC 可见性检查仍由表 AM 自己负责。接口是"从唯一实现中提取的接口"，而不是先验设计。这也是 OrioleDB 等项目仍需要补丁核心代码的原因。

接口本体是一张巨大的回调函数表 `TableAmRoutine`（`src/include/access/tableam.h:321`，一直到 :881 结束）。heap 的实现注册在 `src/backend/access/heap/heapam_handler.c:2659`：

### 1.2 Index AM：一张更老练的回调表

`IndexAmRoutine`（`src/include/access/amapi.h:233`）分两部分：一组**能力标志位**和一组**接口函数**。摘录（amapi.h:233 起）：

```c

    amgetbitmap_function amgetbitmap; /* 一次吐出全部 TID 到位图（可为 NULL） */
    amendscan_function amendscan;
    ...

```

几个设计要点：

- **amgettuple 与 amgetbitmap 二选一或全都有**：btree/hash/gist/spgist 两者都提供；GIN 只提供 `amgetbitmap`（倒排索引天然按键聚簇、无法高效按堆序逐个吐 TID）；BRIN 也只有位图接口（它根本不存 TID，只能给出"候选页范围"）。优化器据此决定该索引能走 IndexScan 还是只能走 BitmapScan。
- **策略号（strategy）与支持函数（support function）**：AM 本身不懂数据类型。btree 定义了 5 个策略号（< <= = >= >）和一个三态比较支持函数；GiST 定义了 consistent/union/penalty/picksplit 等支持函数号。操作符类（opclass）负责把具体类型的函数填进这些"插槽"——这是 PG "AM 框架 + opclass 插件"两层可扩展性的核心。
- 每种索引的 handler：`bthandler`（nbtree/nbtree.c:118）、`hashhandler`（hash/hash.c:70）、`gisthandler`（gist/gist.c:59）、`ginhandler`（gin/ginutil.c:39）、`spghandler`（spgist/spgutils.c:44）、`brinhandler`（brin/brin.c:254）。

**与 InnoDB 对比**：InnoDB 只有 B+ 树一种索引结构（自适应哈希是它之上的内存缓存，全文索引是独立子系统），存储引擎接口（handler API）在 server 与引擎之间切一刀；PG 则在"表怎么存"与"索引怎么建"各切一刀，索引 AM 还进一步把"类型语义"externalize 给 opclass——所以 PostGIS 不改一行内核代码就能让几何类型用上 GiST 索引。

**小结**：Table AM 是 PG 12 为引擎可插拔补交的作业，接口仍带 heap 胎记；Index AM 则是 Berkeley 血统的老抽象，能力标志 + 回调 + opclass 三层结构让"新数据类型自带索引能力"成为常态。

## 二、B-tree（一）：Lehman & Yao 算法——为什么读不需要锁耦合

代码在 `src/backend/access/nbtree/`（"nbt" = new btree）。**先通读 `src/backend/access/nbtree/README`，它是全内核质量最高的 README**（1082 行），本节相当于它前 200 行的导读。

### 2.1 教科书协议的问题

经典 B+ 树并发控制用 **lock coupling（蟹行）**：下降时持有父节点锁、再拿子节点锁、确认子节点不会分裂/合并波及父节点后才放父锁。写操作最坏情况要从叶子往上锁住一整条路径。问题：

- 根节点是所有路径的必经之地，锁耦合让根成为热点；
- 读操作也要参与锁协议，读写互相阻塞。

### 2.2 L&Y 的两个附加物：right-link 与 high key

Lehman & Yao (1981) 给每个节点加了两样东西（README:17-30）：

1. **right-link**：指向右兄弟页的指针；
2. **high key**：本页允许的键上界（最右页没有 high key，隐含 +∞）。

于是产生一个不变式：**页分裂永远"向右分裂"**——原页保留左半部分，新页放右半部分挂在 right-link 上，原页的 high key 降为分裂点。任何时刻，一个页上"曾经有过"的键，要么还在本页，要么在它右边的某个页上，绝不会向左跑。

由此，下降过程中的竞态有了廉价的自我修复方式：

> 沿父页 downlink 到达子页后，比较搜索键与子页 high key。**如果搜索键 > high key，说明这个页在我读 downlink 之后分裂过，目标键被搬去了右边——沿 right-link 追即可**（可能追多次）。

因此：

- **读操作全程只锁一个页**（防止读到一半页被改写的短暂共享锁），层与层之间不持任何锁，无锁耦合；
- **写操作也只在叶子层拿一个排他锁**；分裂时才需要同时锁原页+新右页+老右兄弟（更新其左链）。

PG 相对论文的改动（README:60-120）：论文假设无共享内存、读可以完全无锁，PG 因共享缓冲池必须加页级读锁；PG 增加了左兄弟指针以支持反向扫描；根分裂由 metapage（0 号页，记录真根与 fast root）统一处理；PG 用"子页块号"而非分离键来在父层重定位 downlink，因此上升时也不需要论文中三锁并持的锁耦合。

另一个关键工程决策（README:44-51）：L&Y 要求同层键严格唯一，PG 12 起把**堆 TID 作为隐式的最后一个键列（tiebreaker）**，逻辑重复键按 TID 排序后全局唯一。这个"heapkeyspace"设计是 suffix truncation 与 deduplication 的前提。

**与 InnoDB 对比**：InnoDB 的 B+ 树用 index 级 S/X latch + 页 latch 的两级方案，乐观下降失败后升级为悲观加整棵子树锁；PG 的 L&Y 方案任何路径上最多持 1-3 个页锁，代价是读路径可能多走几次 right-link。两者都规避了教科书式全程锁耦合，思路完全不同。

**小结**：right-link + high key 把"分裂对读者不可见"变成"分裂对读者可检测、可恢复"，从而把并发协议从"预防"（锁耦合）变成"补偿"（向右追）。这是本章最重要的一个算法思想。

## 三、B-tree（二）：_bt_search 下降循环逐行走读

搜索入口 `_bt_search`（`src/backend/access/nbtree/nbtsearch.c:100`）。它拿一个"插入型 scankey"（BTScanInsert，带三态比较函数），从根下降到第一个可能包含该键的叶页。逐段走读（nbtsearch.c:100-208）：

```c
BTStack
_bt_search(Relation rel, Relation heaprel, BTScanInsert key, Buffer *bufP,
           int access, bool returnstack)
{
    BTStack     stack_in = NULL;
    int         page_access = BT_READ;
    ...
    /* Get the root page to start with */
    *bufP = _bt_getroot(rel, heaprel, access);
    ...
    /* Loop iterates once per level descended in the tree */
    for (;;)
    {
        ...
        *bufP = _bt_moveright(rel, heaprel, key, *bufP, (access == BT_WRITE),
                              stack_in, page_access);

        /* if this is a leaf page, we're done */
        page = BufferGetPage(*bufP);
        opaque = BTPageGetOpaque(page);
        if (P_ISLEAF(opaque))
            break;

        offnum = _bt_binsrch(rel, key, *bufP);
        itemid = PageGetItemId(page, offnum);
        itup = (IndexTuple) PageGetItem(page, itemid);
        child = BTreeTupleGetDownLink(itup);

        if (returnstack)
        {
            new_stack = (BTStack) palloc_object(BTStackData);
            new_stack->bts_blkno = BufferGetBlockNumber(*bufP);
            new_stack->bts_offset = offnum;
            new_stack->bts_parent = stack_in;
            stack_in = new_stack;
        }

        if (opaque->btpo_level == 1 && access == BT_WRITE)
            page_access = BT_WRITE;

        /* drop the read lock on the page, then acquire one on its child */
        *bufP = _bt_relandgetbuf(rel, *bufP, child, page_access);
    }
```

- `_bt_getroot`（:111）：从 metapage 缓存（relcache 的 `rd_amcache`）拿 fast root，避免每次搜索都读 0 号页；缓存可能过期，但读到后可以自我验证（README "Other Things That Are Handy to Know" 一节）。
- 循环体第一件事永远是 `_bt_moveright`（:140）：**修补"读 downlink 与到达子页之间发生了分裂"的竞态**。这是 L&Y 的灵魂调用点。
- `P_ISLEAF` 命中即跳出（:146）——到叶子了。
- `_bt_binsrch`（:153）在内部页做二分，返回"最后一个 < 搜索键的 pivot 元组"，取其 downlink 得子页块号（:157）。
- `bts_blkno/bts_offset` 压入 **BTStack**（:165-172）：记录"我从哪个父页哪个位置下来的"。将来叶子分裂要往父页插 downlink 时，从这个 stack 出发重定位父页（而不是像教科书那样一路持有父锁）。stack 只记块号不留锁——父页此刻就可以被别人改。
- level==1 的小优化（:179）：写模式下降到倒数第二层时，提前把下一层（叶层）的锁模式切成 BT_WRITE，避免"先拿读锁、再升级"的额外一轮。
- `_bt_relandgetbuf`（:183）：**先放本页锁、再锁子页**——层间从不持两把锁，这正是无锁耦合的代码体现。
- 函数尾部（:193-205）：如果整棵树只有一个页（根即叶），前面按读锁拿的，需要释放后重新拿写锁，再 `_bt_moveright` 补一次竞态检查。

### 3.1 _bt_moveright：并发分裂的修复者

`_bt_moveright`（nbtsearch.c:242），核心循环（:275-315）：

```c
    cmpval = key->nextkey ? 0 : 1;

    for (;;)
    {
        page = BufferGetPage(buf);
        opaque = BTPageGetOpaque(page);

        if (P_RIGHTMOST(opaque))
            break;                       /* 最右页，high key = +∞，不用动 */

        /* Finish any incomplete splits we encounter along the way. */
        if (forupdate && P_INCOMPLETE_SPLIT(opaque))
        {
            ...
            _bt_finish_split(rel, heaprel, buf, stack);
            ...
            continue;
        }

        if (P_IGNORE(opaque) || _bt_compare(rel, key, page, P_HIKEY) >= cmpval)
        {
            /* step right one page */
            buf = _bt_relandgetbuf(rel, buf, opaque->btpo_next, access);
            continue;
        }
        else
            break;
    }
```

- 判定规则一行写尽：**搜索键 ≥/> 本页 high key（P_HIKEY 恒为页内第 1 项）就向右走一页**，循环直到落在正确的键区间里。`nextkey` 决定用 > 还是 ≥（范围扫描定位起点时用）。
- `P_IGNORE`：页处于 half-dead 或 deleted 状态（并发页删除的墓碑，见第五节）也向右走——对读者而言，"页被删了"与"页分裂了"用同一套恢复动作。
- `P_INCOMPLETE_SPLIT` 分支（:286-305）：写模式下遇到"分裂做了一半"的页，先替前人把分裂善后（`_bt_finish_split`，往父页补 downlink），再继续——这是崩溃恢复策略的一部分，第四节详述。

### 3.2 _bt_binsrch：页内二分

`_bt_binsrch`（nbtsearch.c:344）是标准二分，但边界语义精确到苛刻（:388-449）：

```c
    high++;                         /* establish the loop invariant for high */
    cmpval = key->nextkey ? 0 : 1;  /* select comparison value */

    while (high > low)
    {
        OffsetNumber mid = low + ((high - low) / 2);

        result = _bt_compare(rel, key, page, mid);
        if (result >= cmpval)
            low = mid + 1;
        else
            high = mid;
    }

    if (P_ISLEAF(opaque))
        ...
        return low;      /* 叶页：第一个 >= 搜索键的位置 */

    Assert(low > P_FIRSTDATAKEY(opaque));
    return OffsetNumberPrev(low);   /* 内部页：最后一个 < 搜索键的位置 */
```

- 不变式：`low` 之前的槽全部 < 键，`high` 及之后全部 ≥ 键；收敛到 `high == low`。
- **叶页返回 low**（第一个 ≥ 键的项，即扫描起点）；**内部页返回 low-1**（最后一个 < 键的 pivot，它的 downlink 覆盖搜索键所在的子树）。内部页第一个数据项被 `_bt_compare` 视作 -∞，保证 low-1 一定存在（:430-447 的注释解释了这一点）。
- 插入路径用的是变体 `_bt_binsrch_insert`（nbtsearch.c:454 附近），支持把二分边界缓存在 `BTInsertState` 里——唯一性检查刚做过的二分，插入定位时直接复用，省一次 O(log n)。

**小结**：`_bt_search` = "每层：修右移 → 二分 → 记栈 → 放锁下潜"。全部并发防御浓缩在 `_bt_moveright` 一个调用里；BTStack 用"记路标不持锁"替代了教科书的锁耦合。

## 四、B-tree（三）：插入与页分裂的两阶段

### 4.1 _bt_doinsert 主流程

入口 `_bt_doinsert`（`src/backend/access/nbtree/nbtinsert.c:105`），骨架（:163-266）：

```c
search:
    stack = _bt_search_insert(rel, heapRel, &insertstate);   /* ① 找到并写锁叶页 */

    if (checkingunique)
    {
        xwait = _bt_check_unique(rel, &insertstate, heapRel, checkUnique,
                                 &is_unique, &speculativeToken);   /* ② 唯一性检查 */
        if (unlikely(TransactionIdIsValid(xwait)))
        {
            /* Have to wait for the other guy ... */
            _bt_relbuf(rel, insertstate.buf);
            ...
            XactLockTableWait(xwait, rel, &itup->t_tid, XLTW_InsertIndex);
            goto search;                                     /* 等完从头再来 */
        }
        if (itup_key->heapkeyspace)
            itup_key->scantid = &itup->t_tid;                /* 唯一性确立，恢复 TID 键 */
    }

    if (checkUnique != UNIQUE_CHECK_EXISTING)
    {
        newitemoff = _bt_findinsertloc(rel, &insertstate, checkingunique,
                                       indexUnchanged, stack, heapRel);   /* ③ 页内定位 */
        _bt_insertonpg(rel, heapRel, itup_key, insertstate.buf, InvalidBuffer,
                       stack, itup, insertstate.itemsz, newitemoff,
                       insertstate.postingoff, false);        /* ④ 真正插入（可能分裂） */
    }
```

逐步解说：

- **① `_bt_search_insert`**（nbtinsert.c:320）：`_bt_search` 的插入包装，带**最右页快速路径**（README "Fastpath For Index Insertion"）：单调递增插入（自增主键、时间戳）时，缓存上次插入的最右叶页块号，下次直接锁它验证"仍是最右页且键仍属于它"，命中则完全跳过树下降。这对 append-only 负载是巨大优化——InnoDB 对应的缓解手段是右侧分裂点优化，思路同源。
- **② `_bt_check_unique`**（nbtinsert.c:411）唯一性检查的并发正确性依据（:190-199 的注释讲得极清楚）：检查期间**持有"该键可能落在的第一个叶页"的写锁**，任何想插同一键的对手也必须先拿同一把锁，故同一时刻只有一个插入者在做检查。检查中若发现同键元组仍"可能存活"，需要回表确认；若创建它的事务还在跑，返回其 XID，调用方释放锁后等待（:216-236），醒来 `goto search` 整个重来。注意技巧（:122-124）：检查阶段把 scantid（TID 尾键）临时置 NULL——这样才能定位到"该键的第一个页"而不是"该键+本 TID 的位置"；确立唯一后再恢复（:239-240）。键含 NULL 时直接跳过检查（:126-140，SQL 语义 NULL ≠ NULL）。
- **③ `_bt_findinsertloc`**（nbtinsert.c:829）：在叶页内定位插入偏移。若页满，它会依次尝试三条免分裂退路（:918/:961/:1012 调 `_bt_delete_or_dedup_one_page`，nbtinsert.c:2730）：删 LP_DEAD 项（simple deletion）→ bottom-up deletion（版本膨胀启发式）→ **deduplication**（见第五节）；都救不回来才认命分裂。
- **④ `_bt_insertonpg`**（nbtinsert.c:1119）：空间够则页内插入 + 单条 WAL（:1297-1421 的临界区）；不够则进入分裂。

### 4.2 _bt_split：一次分裂 = 两个原子动作

页满时 `_bt_insertonpg` 调用（nbtinsert.c:1232、:1263）：

```c
        rbuf = _bt_split(rel, heaprel, itup_key, buf, cbuf, newitemoff, itemsz,
                         itup, origitup, nposting, postingoff);
        ...
        _bt_insert_parent(rel, heaprel, buf, rbuf, stack, isroot, isonly);
```

**阶段一：`_bt_split`**（nbtinsert.c:1489）——本层的分裂，一条 WAL 记录内原子完成：

1. `_bt_findsplitloc`（nbtsplitloc.c:130）选分裂点：以"把新元组假想放进去后的页"为基准，在均衡空间与 suffix truncation 收益之间选最优的项间分割点（README "Notes about suffix truncation"末段：重复键成堆时会故意选极不均衡的分裂点，把重复键"困"在左页）；
2. 在本地内存组装左页镜像（新 high key = 对分裂点做 suffix truncation 的结果，`_bt_truncate`，nbtutils.c:692），再申请新右页；
3. 临界区内（:1952-2095）：左页整体覆写、右页写入、老右兄弟的左链改指新右页、**左页打上 `BTP_INCOMPLETE_SPLIT` 标志**（:1582 设置），一条 `XLogInsert(RM_BTREE_ID, ...)`（:2083）覆盖全部三个页的修改。

崩溃语义：这条 WAL 要么全放要么全不放，本层永远一致；但**父层还没有新右页的 downlink**——这正是 INCOMPLETE_SPLIT 标志存在的意义。

**阶段二：`_bt_insert_parent`**（nbtinsert.c:2130）——把 downlink 插入父页，第二条 WAL：

```c
        /* get high key from left, a strict lower bound for new right page */
        ritem = (IndexTuple) PageGetItem(page, PageGetItemId(page, P_HIKEY));

        /* form an index tuple that points at the new right page */
        new_item = CopyIndexTuple(ritem);
        BTreeTupleSetDownLink(new_item, rbknum);

        /* Re-find and write lock the parent of buf. */
        ...
```

- 新 downlink 的键就是左页刚生成的 high key（:2208-2214）；
- 用 BTStack 重定位父页：从 `bts_blkno` 出发，必要时向右追（父页自己也可能分裂过了）——`_bt_getstackbuf` 按**子页块号**匹配 downlink，这是 PG 能免去 L&Y 三锁耦合的原因（README:139-155）；
- 父页插入与**清除子页 INCOMPLETE_SPLIT 标志**在同一条 WAL 里原子完成（:1327-1328 清标志）；父页若也满，则递归分裂——每层都是独立的"两阶段"；
- 根分裂（:2153-2165）：调 `_bt_newlevel` 造新根 + 更新 metapage，对应 WAL 类型 `XLOG_BTREE_NEWROOT`（:2639）。

**incomplete split 的善后**：若两条 WAL 之间崩溃（或父页插入因 OOM/磁盘满失败），恢复后树里留下一个有 INCOMPLETE_SPLIT 标志的左页。PG 9.4 起的策略是**懒惰修复**（README "WAL Considerations"）：搜索仍然正确（右页可经左页 right-link 到达，只是慢）；下一个走到这里的**写入者**在 `_bt_moveright`（nbtsearch.c:286-298）或 `_bt_findinsertloc`（nbtinsert.c:1067-1069）里发现标志，调用 `_bt_finish_split`（nbtinsert.c:2272）补插 downlink 后再干自己的活。不让 VACUUM 或恢复流程来修，是因为补 downlink 可能引发新分裂、需要新磁盘空间——磁盘满时 VACUUM 必须还能跑。

**与 InnoDB 对比**：InnoDB 用 mini-transaction（mtr）把"分裂+父插入"包成一个多页原子单元，持有整条路径的 latch 直到 mtr 提交；PG 则把它拆成两个各自原子的动作，中间状态合法化（INCOMPLETE_SPLIT），用"下一个人顺手修"来收敛。前者靠更强的锁，后者靠更聪明的不变式。

**小结**：插入 = 搜索定位 + 页内插入；分裂 = "本层分裂"与"父层插 downlink"两个独立原子动作，中间态由 INCOMPLETE_SPLIT 标记并被后来的写入者惰性修复。唯一性检查的本质是"第一可能页的写锁"充当了该键的互斥量。

## 五、B-tree（四）：页删除、suffix truncation 与 deduplication

### 5.1 页删除：比分裂难得多（nbtpage.c）

分裂是"无中生有加一页"，删除是"从并发读者眼皮底下抽走一页"，难度不对称。PG 的约束（README "Deleting entire pages during VACUUM"）：只删**完全变空**的页；从不合并半满页（相邻合并会让反向扫描漏项）；从不删除任何一层的最右页。

删除入口 `_bt_pagedel`（nbtpage.c:1812），同样是**两阶段**，每阶段一条 WAL：

1. **阶段一 `_bt_mark_page_halfdead`**（nbtpage.c:2102）：找到父页，把指向目标页的 downlink 删掉、让原本指向左邻 downlink 的键区间顺延给右兄弟，同时给目标叶页打上 **half-dead** 标志。此刻页从父层"消失"了，但左右兄弟链还连着它——和分裂的中间态镜像对称。搜索遇到 half-dead 页按 `P_IGNORE` 处理，向右走即可。
2. **阶段二 `_bt_unlink_halfdead_page`**（nbtpage.c:2329）：按 左邻 → 自己 → 右邻 的顺序加锁，改双向链摘除该页，标成 fully dead。

若两阶段之间崩溃，half-dead 页留在树里，对读写无害，**下一次 VACUUM** 撞见时继续完成阶段二（README "WAL Considerations" 末段）。

删除"某父页的最后一个孩子"时更复杂：要把整条"瘦子树"（一串每层单页的链）一起删，从祖父层摘 downlink，README:285-330 有完整推演。

**墓碑与延迟回收**：fully dead 的页不能立刻进 FSM 复用——可能还有并发读者正拿着指向它的旧指针飞行途中。页上会记一个 safexid，等到该 XID 对所有人可见（即当时的所有快照都结束）才可回收（README "Placing deleted pages in the FSM"，Lanin & Shasha 的 "drain technique"）。这与堆元组的死亡→VACUUM 回收是同构的两阶段模式。

### 5.2 Suffix truncation（PG 12）

叶页分裂时，新 high key（随后成为父层 downlink）只需**足以区分左右两页**，多余的后缀键列可以整列截掉（`_bt_truncate`，nbtutils.c:692；理论出处 Bayer & Unterauer 的 Prefix B-Trees，README "Notes about suffix truncation"）。被截掉的列语义上是 "-∞"。例如 `(date, id)` 复合索引在 date 边界处分裂时，pivot 只需保留 date 列。收益：pivot 变小 → 内部页扇出变大 → 树更矮；且 pivot 不再"过度描述"键空间边界。INCLUDE 索引的非键列正是靠同一机制在分裂时被必然截掉。分裂点选择逻辑（nbtsplitloc.c）会在空间均衡附近专门挑"能截得最多"的分割点。

### 5.3 Deduplication（PG 13，nbtdedup.c）

`_bt_dedup_pass`（`src/backend/access/nbtree/nbtdedup.c:59`）把页内键值相同的多条索引元组合并为一条 **posting list 元组**（键 + 排序的 TID 数组，与 GIN 的 posting list 几乎同构，README "Notes about deduplication"）。设计要点：

- **完全惰性**：只在"页要分裂了、且 LP_DEAD 项已删完仍不够"这一最后关头才做（`_bt_delete_or_dedup_one_page`，nbtinsert.c:2730 → :2827 调用 `_bt_dedup_pass`），平时不付任何维护成本；
- 新插入的 TID 落在既有 posting list 区间内时做 **posting list split**——在同一原子动作里改写 posting list 并完成插入（README "Posting list splits"）；
- **唯一索引也做 dedup**（README "Deduplication in unique indexes"）：唯一索引里的重复全是 MVCC 版本垃圾，为它们分裂页是"永久结构损伤换临时空间"，dedup 纯粹是给 VACUUM/bottom-up deletion 争取时间；
- 不做 GIN 式 varbyte 压缩：会破坏页空间核算的简单性，且删除单个 TID 变慢。

**小结**：页删除以 half-dead 中间态+两条 WAL 复刻了分裂的"两阶段+惰性收尾"哲学，外加墓碑延迟回收；suffix truncation 让树更矮，deduplication 让版本膨胀不再必然转化为页分裂——三者共同回应同一主题：**B-tree 的结构退化必须比业务增长慢**。

## 六、GIN：倒排索引与 LSM 直觉（access/gin/）

做过搜索引擎/大数据的读者最熟悉这个结构。GIN（Generalized Inverted Index）存 `(key, posting list)`：key 是被索引项的**元素**（数组元素、tsvector 词素、jsonb 路径），posting list 是包含该元素的堆行 TID 集合（`src/backend/access/gin/README` 开头 20 行）。一行文档产出多个 key——这是它与 btree "一行一键" 的本质差别。

### 6.1 两层树结构

- **Entry tree**：对 key 建的一棵 B-tree（复用类似 nbtree 的结构，代码 ginbtree.c/ginentrypage.c）。叶元组两种形态：
  - key 的 TID 少 → 直接在叶元组里内联一个 **posting list**（varbyte 差分压缩，ginpostinglist.c）；
  - TID 多到放不下 → 叶元组存一个指针（`GinGetPostingTree`，ginblock.h:234），指向一棵独立的 **posting tree**——按 TID 排序的 B-tree（gindatapage.c，`createPostingTree` 在 gindatapage.c:1775）。
- 于是一次 `tsvector @@ query` 查询 = 对每个查询词在 entry tree 定位 → 遍历其 posting list/tree → 多个 TID 集合按查询逻辑做交并（ginget.c，`gingetbitmap` 在 ginget.c:1928）。GIN 只提供 amgetbitmap，结果必然走 Bitmap Heap Scan。
- Entry tree **没有删除操作**（README:28-31）：大语料的词表几乎只增不减，砍掉 entry 删除让并发协议大幅简化——死 TID 由 VACUUM 从 posting 中清除，空 entry 留着无害。

### 6.2 fastupdate pending list：GIN 的 memtable

GIN 的写放大天生严重：插一行文档 = 更新 N 个 key 的 posting。`fastupdate=on`（默认）时走 `ginHeapTupleFastInsert`（`src/backend/access/gin/ginfast.c:219`）：

```c
    metabuffer = ReadBuffer(index, GIN_METAPAGE_BLKNO);
    ...
    if (collector->sumsize + collector->ntuples * sizeof(ItemIdData) > GinListPageSize)
    {
        /* Total size is greater than one page => make sublist */
        separateList = true;
    }
    else
    {
        LockBuffer(metabuffer, GIN_EXCLUSIVE);
        metadata = GinPageGetMeta(metapage);
        if (metadata->head == InvalidBlockNumber ||
            collector->sumsize + ... > metadata->tailFreeSize)
        {
            separateList = true;
            LockBuffer(metabuffer, GIN_UNLOCK);
        }
    }
```

新文档的所有 `(key, TID)` 对**不进树**，而是原样追加到 metapage 串起的**pending list 链表**尾部（能塞进尾页就塞，塞不下就先建好子链再一次挂上，保持高并发）。积累超过 `gin_pending_list_limit`（变量定义 ginfast.c:39，默认 4MB）或 VACUUM 到来时，`ginInsertCleanup`（ginfast.c:780）把 pending 项**批量归并**进 entry/posting tree——同一 key 的多个 TID 合并成一次树操作，把 N 次随机写摊销成排序后的批量写。

这就是 **LSM 树 memtable + compaction 的直觉**：写先进无序小结构，后台合并进有序大结构。代价也同样经典：查询必须额外顺序扫一遍 pending list（对每个查询都是 O(pending 大小)），所以 pending list 上限不能开太大；对查询延迟敏感的场景常直接 `fastupdate=off`。

**小结**：GIN = "key 的 B-tree" 套 "TID 的 B-tree"，加一截作为写缓冲的 pending list。它把 btree 的"一行一键"改成"一行多键"，用 no-delete entry tree 和批量归并换全文检索场景的规模。

## 七、GiST：把"树"做成框架（access/gist/）

GiST（Generalized Search Tree，Hellerstein et al. 1995）回答的问题是：B-tree 里哪些逻辑是"树的骨架"、哪些是"键的语义"？答案：平衡树维护、页分裂、并发（GiST 用 NSN+rightlink，思路借自 L&Y）是骨架；而"什么样的键该下降到哪个子树"是语义。GiST 把语义抽成 opclass 的支持函数（编号见 `src/include/access/gist.h:32-37`）：

```c
#define GIST_CONSISTENT_PROC            1
#define GIST_UNION_PROC                 2
#define GIST_COMPRESS_PROC              3
...
#define GIST_PENALTY_PROC               5
#define GIST_PICKSPLIT_PROC             6
```

四个核心回调的意义（内部页的每个 entry 是一个"谓词"，为其子树内所有键的**并集/上界**）：

| 回调 | 搜索/插入中的角色 | R-tree 实例（矩形） |
|---|---|---|
| `consistent(entry, query)` | **搜索下降判据**：该子树是否可能含匹配项？false 才能剪枝，允许假阳性（返回 recheck） | 子树包围盒与查询矩形相交吗？ |
| `union(entries)` | 生成/更新父 entry：子集合的上界 | 多个矩形的最小包围盒 |
| `penalty(entry, new)` | **插入下降判据**：新键插进该子树的"代价"，选 penalty 最小的孩子 | 包围盒面积增量 |
| `picksplit(entries)` | 页满时把 entry 分成两组 | 让两个新包围盒面积和最小（Guttman 二次分裂等） |

对照源码：搜索走 `gistindex_keytest`（gistget.c:134），对每个 scan key 调 consistent（gistget.c:230-236 的 `FunctionCall5Coll`，第 5 个参数回传 recheck 标志——GiST 允许有损判定，靠回表 recheck 兜底）；插入走 `gistdoinsert`（gist.c:639），逐层用 `gistchoose`（gistutil.c:374，内部调 penaltyFn，gistutil.c:730-733）选子树，页满时 `gistSplitByKey`（gistsplit.c:623）调 picksplit，再用 unionFn（gistutil.c:204）重算父 entry。

**R-tree 是 GiST 的一个特例**：把上表右列的四个函数填进去就是 R-tree（PG 内建的 `box_ops` 等即是，gistproc.c）。同理，填"区间包含"得到 range 类型索引，填"集合签名"得到全文检索的 RD-tree。**与 B-tree 的根本差异**：GiST 不要求键有全序，子树谓词可以互相重叠——所以搜索可能要下降多个分支，也因此它能索引"相交""包含""距离"这类 B-tree 无能为力的谓词；`amcanorderbyop=true` 还让它支持 KNN（`ORDER BY point <-> ?`，按 distance 支持函数做最佳优先遍历）。

**小结**：GiST 用 consistent/union/penalty/picksplit 四个回调把"平衡树"与"键语义"解耦，B-tree 式全序索引之外的几乎一切树形索引（R-tree、RD-tree、区间树）都成了它的填空题。PostGIS 的空间索引就是最著名的一份答卷。

## 八、BRIN：块范围摘要（access/brin/）

BRIN（Block Range Index）根本不是"树"，也不存任何 TID。它把表按物理顺序切成**块范围**（默认 128 个堆页一段，`BRIN_DEFAULT_PAGES_PER_RANGE`，`src/include/access/brin.h:40`），每段只存一条**摘要元组**——minmax opclass 下就是该段内此列的 min/max（还有 bloom、minmax-multi 等 opclass，brin_bloom.c / brin_minmax_multi.c）。

**大数据类比**：这就是 Parquet/ORC 的 footer 里 row group 级 min/max 统计、Snowflake 的 zone map——**用于跳过数据（data skipping），而非定位数据**。查询 `WHERE ts BETWEEN a AND b` 时，`bringetbitmap`（brin.c:572）遍历所有摘要元组，把"区间与 [min,max] 相交"的整段堆页**全部**加入位图（有损，页内逐行 recheck），不相交的段整段跳过。所以：

- BRIN 只在**物理顺序与逻辑顺序强相关**的列上有效（append-only 时序表的时间戳列近乎完美；被随机更新打乱后急剧退化，可用 `pg_stat` 相关性列判断）;
- 换来的是近乎可以忽略的体积：10GB 表的 BRIN 常常只有几十 KB，而 btree 会有数百 MB。

**revmap 机制**：摘要元组本身存在普通索引页里，位置会变（摘要更新变大时可能搬家），于是需要一层稳定的定位结构——**revmap（range map，反向映射）**，占据索引文件开头的固定页链，按块范围号数组式地存"该范围的摘要元组现在在哪"的 ItemPointer。代码在 brin_revmap.c：`brinGetTupleForHeapBlock`（brin_revmap.c:194）由堆块号算出 revmap 页与槽位、取出指针再读摘要；`brinSetHeapBlockItemptr`（brin_revmap.c:155）在摘要搬家/新建时回填；`brinRevmapExtend`（brin_revmap.c:112）随表增长扩展 revmap。插入新行时 `brininsert`（brin.c:349）查 revmap 找到所属范围的摘要，若新值越界则原地（或搬家后）放宽 min/max——**BRIN 的更新只放宽、不收紧**，收紧要靠重新 summarize。

**小结**：BRIN 用"每 128 页一条 min/max"把索引做成了 zone map，靠 revmap 间接层解决摘要元组的寻址。它是 PG 里最"大数据"的索引：不回答"行在哪"，只回答"哪些整段肯定没有"。

## 九、Hash 与 SP-GiST 简述

**Hash**（access/hash/）：桶号 = hash 值的低位，桶满挂 overflow 页链；扩容用**渐进式桶分裂**——每次只分裂一个桶（`src/backend/access/hash/README` 开头：源自 Seltzer & Yigit 1991 的线性哈希族方案），不像内存哈希表那样整体 rehash。搜索入口 `_hash_first`（hashsearch.c:289）。只支持 `=`（`amstrategies` 只有等值一种），不支持排序、范围、唯一约束、多列首列跳过；PG 10 之前甚至不写 WAL（崩溃后必须重建），名声不佳。如今它的合理场景很窄：超长键（如 URL）的等值查询，索引项只存 4 字节 hash 值，比 btree 小。**与 InnoDB 自适应哈希对比**：InnoDB 的 AHI 是 B+ 树之上自动构建的内存加速层，PG 的 hash 是持久化的磁盘索引、需要显式创建，两者除了都叫 hash 几乎没有共同点——PG 没有 AHI 的对应物。

**SP-GiST**（access/spgist/）：Space-Partitioning GiST——GiST 的兄弟框架，但支持的是**非平衡的空间划分树**：四叉树（spgquadtreeproc.c）、kd 树（spgkdtreeproc.c）、基数树/trie（spgtextproc.c）。与 GiST 的区别：GiST 的子树谓词可重叠、树保持平衡；SP-GiST 的划分**不相交**、树可以很深且不平衡。磁盘页与树节点不再一一对应（很多小节点打包进一页），插入主逻辑 `spgdoinsert`（spgdoinsert.c:1909）是全内核最烧脑的函数之一。典型用途：文本前缀匹配（trie）、点数据的四叉树、IP 网段（inet_ops）。

**小结**：hash 是"只做等值"的窄工具；SP-GiST 把"不相交划分、不平衡深树"这一族结构（trie/四叉树/kd 树）也纳入了框架化版图。两者共同说明：Index AM 接口的表达力足以容纳形态迥异的结构。

## 十、索引与堆的协作：三种扫描形态

PG 的索引一律不存行数据（且**索引不含可见性信息**——这是与 InnoDB 最重要的运维级差异：InnoDB 二级索引可以在索引层判断部分可见性，PG 的索引项没有 xmin/xmax，可见性必须问堆）。由此派生三种扫描形态。

### 10.1 IndexScan：逐个回表

执行器的 IndexScan 节点循环调用 `index_getnext_slot`（indexam.c:698）= `index_getnext_tid`（indexam.c:599，进入 AM 的 amgettuple 拿下一个 TID）+ `index_fetch_heap`（indexam.c:657）：

```c
bool
index_fetch_heap(IndexScanDesc scan, TupleTableSlot *slot)
{
    bool        all_dead = false;
    bool        found;

    found = table_index_fetch_tuple(scan->xs_heapfetch, &scan->xs_heaptid,
                                    scan->xs_snapshot, slot,
                                    &scan->xs_heap_continue, &all_dead);

    if (found)
        pgstat_count_heap_fetch(scan->indexRelation);

    /* 整条 HOT 链都死了？告诉索引 AM 给这个 TID 打 LP_DEAD（kill_prior_tuple） */
    if (!scan->xactStartedInRecovery)
        scan->kill_prior_tuple = all_dead;

    return found;
}
```

- 回表走的是 Table AM 的 `index_fetch_tuple` 回调（heap 实现 `heapam_index_fetch_tuple`），在堆页内沿 **HOT 链**找到对本快照可见的版本；
- **kill_prior_tuple 反馈回路**（:676-677）：回表发现整条链对所有人都死了，就反手把索引项标 LP_DEAD（nbtree README "Simple deletion"）——下次扫描直接跳过，未来页满时可直接删除。索引与堆的"垃圾情报"就是通过这条通道流动的；
- 随机回表是 IndexScan 的成本大头，这正是 InnoDB 聚簇索引免掉主键查询回表、而 PG 必须靠下面两种形态补偿的地方。

### 10.2 IndexOnlyScan：依赖 Visibility Map

若查询列全在索引里，理论上可以不回表——但 PG 索引无可见性信息，怎么办？答案是**页粒度的可见性缓存**：Visibility Map（每堆页 2 bit，由 VACUUM 维护 all-visible 位）。nodeIndexonlyscan.c:165：

```c
        if (!VM_ALL_VISIBLE(scandesc->heapRelation,
                            ItemPointerGetBlockNumber(tid),
                            &node->ioss_VMBuffer))
        {
            /* Rats, we have to visit the heap to check visibility. */
            InstrCountTuples2(node, 1);
            if (!index_fetch_heap(scandesc, node->ioss_TableSlot))
                continue;       /* no visible tuple, try next index entry */
            ...
        }
        /* Fill the scan tuple slot with data from the index */
        StoreIndexTuple(node, slot, scandesc);
```

TID 所在堆页 all-visible → 直接用索引元组当结果，零回表；否则退化为普通回表。EXPLAIN 里的 `Heap Fetches` 就是这个退化次数——**长期不 VACUUM 的表，Index Only Scan 会退化成比 IndexScan 还贵的东西**。读不加 VM 缓冲锁的正确性论证见 :136-163 的大段注释（写侧先清 VM 位再改索引，配合索引页锁的内存屏障语义）。

### 10.3 BitmapScan：两阶段与 recheck

两阶段执行：

1. **建位图**：BitmapIndexScan 节点调 AM 的 `amgetbitmap`，把所有匹配 TID 一次性倒进 TIDBitmap；多个索引的位图可做 BitmapAnd/BitmapOr。内存不够时位图自动**退化为有损（lossy）**：不再记"哪些行"，只记"哪些页"；
2. **按物理顺序扫堆**：BitmapHeapScan 节点（`BitmapHeapNext`，nodeBitmapHeapscan.c:174）把位图页号排序后顺序访问堆——把随机 I/O 变成准顺序 I/O，这是它对大选择率区间查询优于 IndexScan 的根源。对有损页和 AM 自报有损的项（`recheck` 标志），取出的每行要**重算原始谓词**（nodeBitmapHeapscan.c:197-205："If we are using lossy info, we have to recheck the qual conditions"）。

所谓 "precheck + recheck" 的两阶段语义贯穿三类 AM：GIN/BRIN/GiST 的 consistent 判定都允许假阳性（precheck，宁滥勿缺），由堆侧 recheck 精确化——正因为有这个兜底，前面各种"有损但紧凑"的索引设计（BRIN 的 min/max、GiST 的签名、位图的 lossy 页）才得以成立。

**小结**：三种形态是"回表成本"这同一约束的三种解法：IndexScan 逐行付费，IndexOnlyScan 靠 VACUUM 预付（VM），BitmapScan 批量排序后付。索引无可见性信息是 PG 与 InnoDB 分岔的总根源：它换来了索引项的紧凑与 HOT 优化的可能，代价是 VACUUM 成为索引性能的隐形参与者。

## 自测问题

**1. Lehman & Yao 算法靠什么免除锁耦合？读操作如何发现并从并发分裂中恢复？**

每页增加 right-link（右兄弟指针）与 high key（本页键上界），且分裂只向右。读者到达子页后比较搜索键与 high key（`_bt_moveright`，nbtsearch.c:242）：搜索键超过 high key 说明页在"读 downlink → 到达"窗口内分裂过，沿 right-link 向右追一页或多页即可。因此层间不需要同时持有父子两把锁，读全程只锁当前页。

**2. B-tree 页分裂为什么写两条 WAL？崩溃在两条之间会怎样？**

`_bt_split`（nbtinsert.c:1489）用第一条 WAL 原子完成本层的三页修改（左页覆写、新右页、老右兄弟左链），并给左页打 INCOMPLETE_SPLIT 标志；`_bt_insert_parent`（nbtinsert.c:2130）用第二条 WAL 往父页插 downlink 并清标志。之间崩溃则树处于合法的中间态：搜索经左页 right-link 仍可达新右页；下一个写入者路过时由 `_bt_finish_split`（nbtinsert.c:2272）补插 downlink。刻意不让 VACUUM 修复，因为补 downlink 可能引发分裂、需要磁盘空间。

**3. 唯一索引的并发唯一性检查如何避免两个事务同时插入相同键？**

`_bt_doinsert` 检查期间持有"该键（不含 TID 尾键）可能落在的第一个叶页"的排他锁，任何同键插入者必须先获得同一把锁，故检查互斥；发现未决的同键元组则记下对方 XID，放锁等待其结束后 `goto search` 重来（nbtinsert.c:213-236）。检查通过后到插入完成前持续持有写锁，后来者必能看到本次插入。

**4. GIN 的 pending list 与 LSM memtable 的类比中，哪些点对应、哪些点不对应?**

对应：写入先追加到小的无序结构（pending list / memtable），延迟批量归并进大的有序结构（entry+posting tree / SSTable），把 N 次随机写摊销为批量写；查询必须额外查小结构。不对应：pending list 是磁盘页链而非内存结构（崩溃不丢、也不靠 WAL 重建）；合并（`ginInsertCleanup`，ginfast.c:780）是就地插树而非生成新文件层；没有多层 compaction 与层间合并策略。

**5. 为什么 PG 的 Index Only Scan 需要 Visibility Map，而 InnoDB 的覆盖索引扫描不需要类似机制？**

PG 索引项不携带任何可见性信息（无 xmin/xmax），判断某索引项对本快照是否可见本来必须回堆；VM 提供页粒度的"本页全部可见"缓存（由 VACUUM 置位），命中则跳过回表（nodeIndexonlyscan.c:165）。InnoDB 的二级索引页携带 PAGE_MAX_TRX_ID，多数情况下能在索引层判定可见性，不可判定时回聚簇索引并借 undo 构造旧版本——它的"旧版本"在 undo 段里，而 PG 的旧版本以完整元组留在堆中、索引对其一无所知。
