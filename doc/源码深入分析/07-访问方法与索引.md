## 本章导读

上一篇我们看清了元组如何躺在 8KB 的页里、页如何进出缓冲池。本篇回答的问题是：**数据以什么结构组织、怎么高效地增删改查**——这正是 `src/backend/access/` 目录的职责。

PostgreSQL 在这里做了一个漂亮的**双抽象**：

- **Table AM**（表访问方法，`src/include/access/tableam.h`）：表的物理存储可插拔，heap 是默认且唯一的内建实现；
- **Index AM**（索引访问方法，`src/include/access/amapi.h`）：索引可插拔，内建了 btree、hash、gist、gin、spgist、brin 六种（本文按惯例把 gist/spgist 视作一族"树框架"，合称五类）。

本篇的重心是 **B-tree 深度走读**（超过一半篇幅），并且这次我们不满足于"是什么"，而是把几组"所以然"追到底：

1. **Lehman & Yao 到底解决了什么问题**——先给出教科书的锁耦合方案及其并发瓶颈，再证明 right-link + high key 如何让读端只持一把锁，并完整推演"读者追赶并发分裂"的时序（第二节）；
2. **为什么 PG 的页分裂必须拆成"先分裂、后插父"两个原子动作**——崩溃在两步之间会怎样、谁负责补完、为什么这比"一条 WAL 搞定一切"更好（第五节）；
3. **suffix truncation 和 deduplication 各解决什么问题**——用量化例子看索引膨胀（第六节）；
4. **为什么所有二级索引指向 TID 而不是主键**——与 InnoDB 聚簇索引的三角权衡完整论证（第九节）。

围绕这些"所以然"，本篇给出四条完整的函数级走读主线：搜索（`_bt_search` + `_bt_moveright` 全函数）、插入（`_bt_doinsert` → `_bt_check_unique` 全函数 → `_bt_findinsertloc` → `_bt_insertonpg`）、分裂（`_bt_split` 全函数——多页原子修改的典范教材）、以及 GIN 的写路径（`ginHeapTupleFastInsert` + `ginInsertCleanup` 主线）。另附 B-tree 不变式清单（每条给违反后果与 amcheck 检测手段，第七节）、git 历史决策点（第八节）、三个 what-if 崩溃/并发推演（散布在 2.3、4.4、6.4 节）。

如果你来自 InnoDB 世界，请先记住本篇最大的思维转换：**PG 的表是堆表，所有索引都是二级索引**。InnoDB 的主键 B+ 树叶子节点就是数据行（聚簇索引），二级索引叶子存主键值；而 PG 的 B-tree 叶子节点存的是 `(索引键, ctid)`，ctid 直接指向堆页的物理位置——没有"聚簇索引"这个概念（`CLUSTER` 命令只做一次性物理重排，不维持）。这个差异会在第九节得到完整论证。

## 前置知识

1. **页与元组布局**（本系列第 06 篇）：PageHeader / ItemId / special space；索引页把 special space 用作各自的元数据（如 B-tree 的左右兄弟指针 `btpo_prev`/`btpo_next`、层号 `btpo_level`、标志位 `btpo_flags`）。
2. **缓冲池与锁**：`ReadBuffer` 拿页、`LockBuffer` 加页级共享/排他锁（内部是 LWLock）；pin（引用计数，防页被驱逐）与 lock（内容锁，防并发读写）是两回事。nbtree 里 `_bt_getbuf`（nbtpage.c:830）= pin + lock，`_bt_relbuf` = unlock + unpin，`_bt_relandgetbuf`（nbtpage.c:988）= 放旧页锁再锁新页（顺序是关键，后文反复出现）。
3. **ctid / ItemPointer**：`(块号, 页内偏移)` 二元组，堆元组的物理地址，也是所有索引指向堆的"外键"。
4. **WAL 基本概念**：一条 WAL 记录是一个原子重做单元，可以覆盖多个页（都在临界区内修改并注册）；但**跨越多条 WAL 记录的多页修改**没有原子性保证，需要额外机制保证崩溃一致性——B-tree 页分裂是最好的教材（WAL 机制详见第 09 篇）。
5. **CRIT_SECTION（临界区）**：`START_CRIT_SECTION()` 到 `END_CRIT_SECTION()` 之间任何 ERROR 都会升级为 PANIC——因为共享缓冲区里的页已被修改、WAL 还没写完，进程级恢复已不可能。所以代码总是"先做完一切可能失败的事（分配内存、读页、组装镜像），再进临界区"。`_bt_split` 是这个纪律最好的示范。
6. 教科书 B+ 树及其经典加锁协议（lock coupling / crabbing）：下降时先锁父再锁子，确认子节点"安全"后才放父锁。

## 一、双抽象接口：Table AM 与 Index AM

### 1.1 Table AM：为什么 PG 12 才把表存储抽象出来

索引从 Berkeley 时代起就是可插拔的（系统目录 `pg_am` 一直存在），但**表存储直到 PG 12（2019 年）才抽象出接口**。原因值得玩味：

- 历史上执行器、VACUUM、COPY 等模块直接调用 `heap_insert`/`heap_getnext` 等函数，heap 的假设（元组不可变、xmin/xmax 可见性、ctid 指针、HOT 链）渗透进了几十个调用点；
- 真正的驱动力是社区想引入 **zheap**（undo 型存储，解决表膨胀）与列存等替代引擎，被迫先做"接口手术"：把所有 `heap_*` 直接调用改为经由回调表分发；
- 抽象的代价是接口形状仍带着浓重的 heap 色彩——比如 TID 必须是"块号+行号"的形式，MVCC 可见性检查仍由表 AM 自己负责。接口是"从唯一实现中提取的接口"，而不是先验设计。这也是 OrioleDB 等项目仍需要补丁核心代码的原因。

接口本体是一张巨大的回调函数表 `TableAmRoutine`（`src/include/access/tableam.h:321`，一直到 :881 结束）。heap 的实现注册在 `src/backend/access/heap/heapam_handler.c:2659`：

```c

    .type = T_TableAmRoutine,

    .slot_callbacks = heapam_slot_callbacks,

    .scan_begin = heap_beginscan,
    .scan_end = heap_endscan,
    .scan_rescan = heap_rescan,
    .scan_getnextslot = heap_getnextslot,
    ...
    .index_fetch_begin = heapam_index_fetch_begin,
    ...
    .index_fetch_tuple = heapam_index_fetch_tuple,

    .tuple_insert = heapam_tuple_insert,
    ...
    .tuple_delete = heapam_tuple_delete,
    .tuple_update = heapam_tuple_update,
    .tuple_lock = heapam_tuple_lock,
    ...
```

主要回调按职责分组解说（行号为 tableam.h 中的声明位置）：

### 1.2 Index AM：一张更老练的回调表

`IndexAmRoutine`（`src/include/access/amapi.h:233`）分两部分：一组**能力标志位**和一组**接口函数**。摘录（amapi.h:233 起）：

```c
typedef struct IndexAmRoutine
{
    NodeTag     type;
    uint16      amstrategies;    /* 策略号（操作符类别）总数，0 表示不固定 */
    uint16      amsupport;       /* 支持函数个数 */
    ...
    bool        amcanorder;      /* 能否支持 ORDER BY 索引列（btree=true） */
    bool        amcanorderbyop;  /* 能否按操作符结果排序（gist 的 KNN 距离排序） */
    ...
    bool        amcanunique;     /* 支持唯一索引？（只有 btree 为 true） */
    bool        amcanmulticol;   /* 支持多列？ */
    bool        amoptionalkey;   /* 第一列没有约束条件时也能扫？ */
    ...
    /* interface functions */
    ambuild_function ambuild;            /* 建索引 */
    aminsert_function aminsert;          /* 插入一条索引项 */
    ambulkdelete_function ambulkdelete;  /* VACUUM 批量删除 */
    amvacuumcleanup_function amvacuumcleanup;
    amcostestimate_function amcostestimate; /* 供优化器估代价 */
    ...
    ambeginscan_function ambeginscan;
    amrescan_function amrescan;
    amgettuple_function amgettuple;   /* 逐个返回 TID（可为 NULL） */
    amgetbitmap_function amgetbitmap; /* 一次吐出全部 TID 到位图（可为 NULL） */
    amendscan_function amendscan;
    ...
} IndexAmRoutine;
```

几个设计要点：

- **amgettuple 与 amgetbitmap 二选一或全都有**：btree/hash/gist/spgist 两者都提供；GIN 只提供 `amgetbitmap`（倒排索引天然按键聚簇、无法高效按堆序逐个吐 TID）；BRIN 也只有位图接口（它根本不存 TID，只能给出"候选页范围"）。优化器据此决定该索引能走 IndexScan 还是只能走 BitmapScan。
- **策略号（strategy）与支持函数（support function）**：AM 本身不懂数据类型。btree 定义了 5 个策略号（< <= = >= >）和一个三态比较支持函数；GiST 定义了 consistent/union/penalty/picksplit 等支持函数号。操作符类（opclass）负责把具体类型的函数填进这些"插槽"——这是 PG "AM 框架 + opclass 插件"两层可扩展性的核心。
- 每种索引的 handler：`bthandler`（nbtree/nbtree.c:118）、`hashhandler`（hash/hash.c:70）、`gisthandler`（gist/gist.c:59）、`ginhandler`（gin/ginutil.c:39）、`spghandler`（spgist/spgutils.c:44）、`brinhandler`（brin/brin.c:254）。

**与 InnoDB 对比**：InnoDB 只有 B+ 树一种索引结构（自适应哈希是它之上的内存缓存，全文索引是独立子系统），存储引擎接口（handler API）在 server 与引擎之间切一刀；PG 则在"表怎么存"与"索引怎么建"各切一刀，索引 AM 还进一步把"类型语义"externalize 给 opclass——所以 PostGIS 不改一行内核代码就能让几何类型用上 GiST 索引。

**小结**：Table AM 是 PG 12 为引擎可插拔补交的作业，接口仍带 heap 胎记；Index AM 则是 Berkeley 血统的老抽象，能力标志 + 回调 + opclass 三层结构让"新数据类型自带索引能力"成为常态。

## 二、B-tree（一）：Lehman & Yao——从锁耦合的困境到"向右追赶"

代码在 `src/backend/access/nbtree/`（"nbt" = new btree）。**先通读 `src/backend/access/nbtree/README`，它是全内核质量最高的 README**（1082 行）；本节相当于它前两章（"The basic Lehman & Yao Algorithm"、"Differences to the Lehman & Yao algorithm"）的展开推导。

### 2.1 教科书方案：lock coupling 及其瓶颈

先把问题摆清楚。B+ 树下降搜索的每一步是：在父页做二分 → 读出 downlink（子页块号）→ 去锁子页。危险窗口在**读 downlink 与锁住子页之间**：

- 若此窗口内子页发生了**分裂**，一半键搬去了新兄弟页，而父页里还没有新兄弟的 downlink——搜索者按旧 downlink 到达子页后，目标键可能已经不在这个页上；
- 若发生了**页删除/合并**，downlink 甚至可能指向一个已经报废的页。

教科书（Bayer & Schkolnick 风格）的答案是 **lock coupling（蟹行，crabbing）**：

1. 读操作：持有父页共享锁的状态下才去拿子页共享锁，拿到后再放父锁——保证"我读到的 downlink 在我到达子页前不会失效"，任何时刻持两把锁；
2. 写操作：下降时持父页排他锁拿子页排他锁，检查子页是否"安全"（插入不满、删除不半空），安全才放父锁；不安全就继续持有——最坏情况从根到叶持有一整条路径的排他锁。

这个方案**正确但昂贵**：

- **根页是所有路径的必经之地**。每个读者都要在根上拿一次锁（哪怕是共享锁，也要经过一次缓存行独占的原子操作），根的锁字成为全局竞争点；写者更是可能把根排他锁住很久；
- **读写互相阻塞**：一个正在分裂叶页的写者持有路径锁时，所有经过该路径的读者排队；
- **持锁跨 I/O**：子页可能不在缓冲池里，锁耦合意味着持着父锁等磁盘。

InnoDB 的现实版本缓解了它（index 级 S/X latch + 乐观下降 + 失败后悲观重来并锁住受影响子树），但"结构修改操作（SMO）期间读者排队"的本质仍在。

### 2.2 L&Y 的两个附加物：right-link 与 high key

Lehman & Yao (1981) 的洞察是：**与其阻止读者看到中间态，不如把中间态做成读者可以廉价识别并自行恢复的合法状态**。每页加两样东西（README "The basic Lehman & Yao Algorithm"）：

1. **right-link**：指向右兄弟页的指针（PG 里是 special space 的 `btpo_next`）；
2. **high key**：本页允许的键上界，即"本页所有键 ≤ high key"（每层最右页没有 high key，隐含 +∞）。

再加一条铁律：**页分裂永远"向右分裂"**——原页原地保留左半部分（块号不变！），新页接收右半部分并挂在原页 right-link 上，原页的 high key 降为分裂点。于是有了一个全局不变式：

> 一个页上"曾经有过"的键，任何时刻要么还在本页，要么在沿 right-link 可达的某个右侧页上——绝不会向左跑，也绝不会凭空消失。

由此，"读 downlink 与到达子页之间发生了分裂"这个竞态有了 O(1) 的检测手段和 O(分裂次数) 的恢复手段：

> 到达子页后，比较搜索键与该页 high key。**若搜索键 > high key，说明目标键区间已经不归本页管——本页在窗口期内分裂过，沿 right-link 向右追一页，重复检测**。因为分裂只向右、追赶严格右移，追赶必然终止。

代价收益一目了然：

- **读操作全程只持一把页锁**（防止读页读到一半被改写的短暂共享锁），层与层之间零锁——根页锁的持有时间缩短为"一次页内二分"；
- **写操作在下降阶段与读者完全一样**，只在最终叶页拿排他锁；分裂时才需要同时锁 2~3 个页（原页 + 新右页 + 老右兄弟，见第五节）；
- 换来的代价：读路径偶尔多走几次 right-link（只在恰好撞上并发分裂时发生），以及**每页永久多存一个 high key**。

### 2.3 what-if 推演 ①：分裂进行中，读者到达会看到什么

把时序拆到最细。设叶页 P 原本存键 [10..50]，high key = 60，读者 R 要找键 45，写者 W 正在分裂 P：

```
时刻   写者 W                                    读者 R
t1                                              在父页读到 downlink → P（P 覆盖 45）
t2    锁住 P，发现放不下，执行分裂：
      新页 P' 接收 [30..50]，P 保留 [10..29]，
      P.highkey=30，P.rightlink=P'
      （一条 WAL，一个临界区内完成）
t3    释放 P、P' 的锁（downlink 还没插父页！）
t4                                              锁住 P，_bt_moveright 比较：45 > highkey(30)
t5                                              放 P 锁，沿 rightlink 锁 P'
t6                                              45 ≤ P'.highkey(60)，停下，页内二分找到 45
```

几个关键点：

- t2 整个分裂在 P 的排他锁 + 一个临界区内完成，读者 R 在 t4 拿到锁时看到的必然是"分裂前"或"分裂后"的完整状态，绝无半成品页；
- t4 的检测只花一次 `_bt_compare`——这是"把并发协议的成本摊给冲突路径而不是所有路径"的典型设计；
- 若 R 在 t2 之前就锁住了 P（读锁），W 会等 R 放锁——页级锁保证页内容的物理一致性，L&Y 保证跨页的逻辑一致性，两层分工清晰；
- 如果 P 在窗口内分裂了两次（P → P' → P''），R 就追两次，每次判据相同；
- 最右页 high key 是 +∞（`P_RIGHTMOST` 判断），任何键都不会触发右移，递归有底。

反向扫描怎么办？L&Y 论文只有 right-link。PG 增加了 left-link（`btpo_prev`）支持反向扫描，但左移无法用 high key 自我验证（分裂是向右的，左移可能"跳过"分裂出的中间页），所以反向扫描用"先记住当前页块号，左移后检查其 right-link 是否仍指回来，不是则向右找回"的重试协议（README "Differences to the Lehman & Yao algorithm" 一节）。README 明确说放锁再移动 "is safe when moving right or up, but not when moving left or down"——向左与向下天然逆着分裂方向，需要额外验证。

### 2.4 PG 相对论文的改动清单

README "Differences to the Lehman & Yao algorithm" 逐条列了改动，摘其中影响主线理解的五条：

1. **读锁不能省**：论文假设每个进程有独立内存副本、读可以完全无锁；PG 是共享缓冲池，读页期间必须持共享锁防止页被并发改写。
2. **left-link 与反向扫描**：见上。
3. **metapage 与 fast root**：0 号页是元数据页，记录真根（`btm_root`）与 fast root（`btm_fastroot`——当上层只剩单页链时，直接从更低层开始搜索）。根的位置变化不必发 invalidation，读者用过期缓存也能自我修正（`_bt_getroot`，nbtpage.c:347）。
4. **父层重定位不需要锁耦合**：论文里插入者上升插 downlink 时要三锁并持地在父层右移；PG 用"按**子页块号**匹配 downlink"（`_bt_getstackbuf`，nbtinsert.c:2351）代替"按分离键匹配"——块号是唯一且不变的，所以上升时也是一次一锁（README "Lehman and Yao must couple/chain locks ..." 段）。
5. **键唯一性靠堆 TID**（README:42-48）：L&Y 要求同层键严格唯一（子树键区间 Ki < v ≤ Ki+1 的严格性依赖它）。PG 12 起把**堆 TID 作为隐式的最后一个键列（tiebreaker）**，逻辑重复键按 TID 排序后全局唯一。这个"heapkeyspace"设计（BTREE_VERSION 4）是 suffix truncation 与 deduplication 的前提，第六、八节详述。

**与 InnoDB 对比**：InnoDB 的 B+ 树在 SMO 期间仍需要把受影响子树对写者隔离（乐观失败 → SX/X latch 子树）；PG 的 L&Y 方案任何路径上最多同时持 1~3 把页锁，代价是读路径可能多走 right-link、且父层 downlink 允许暂时缺失（第五节的 incomplete split）。两者都规避了教科书式全程锁耦合，思路完全不同：InnoDB 靠"更粗粒度但短暂的隔离"，PG 靠"把中间态合法化"。

**小结**：lock coupling 的本质是**预防**——不让读者看到不一致；L&Y 的本质是**补偿**——允许读者看到"downlink 落后于分裂"的状态，但保证该状态可用一次比较检测、沿 right-link 恢复。并发协议从悲观预防转为乐观补偿，根锁热点随之消失。这是本章最重要的算法思想，后面 incomplete split（写者版的补偿）与页删除（half-dead 中间态）都是同一哲学的复用。

## 三、B-tree（二）：_bt_search 与 _bt_moveright 全函数走读

搜索入口 `_bt_search`（`src/backend/access/nbtree/nbtsearch.c:100`）。它拿一个"插入型 scankey"（`BTScanInsert`，由 `_bt_mkscankey`（nbtutils.c:61）从索引元组或扫描条件构造，带三态比较语义，允许省略右侧键列），从根下降到**第一个可能包含该键的叶页**。函数不长，除头部注释外全文如下（nbtsearch.c:99-208）：

```c
BTStack
_bt_search(Relation rel, Relation heaprel, BTScanInsert key, Buffer *bufP,
           int access, bool returnstack)
{
    BTStack     stack_in = NULL;
    int         page_access = BT_READ;

    /* heaprel must be set whenever _bt_allocbuf is reachable */
    Assert(access == BT_READ || access == BT_WRITE);
    Assert(access == BT_READ || heaprel != NULL);

    /* Get the root page to start with */
    *bufP = _bt_getroot(rel, heaprel, access);

    /* If index is empty and access = BT_READ, no root page is created. */
    if (!BufferIsValid(*bufP))
        return (BTStack) NULL;

    /* Loop iterates once per level descended in the tree */
    for (;;)
    {
        Page        page;
        BTPageOpaque opaque;
        OffsetNumber offnum;
        ItemId      itemid;
        IndexTuple  itup;
        BlockNumber child;
        BTStack     new_stack;

        /*
         * Race -- the page we just grabbed may have split since we read its
         * downlink in its parent page (or the metapage).  If it has, we may
         * need to move right to its new sibling.  Do that.
         *
         * In write-mode, allow _bt_moveright to finish any incomplete splits
         * along the way.  Strictly speaking, we'd only need to finish an
         * incomplete split on the leaf page we're about to insert to, not on
         * any of the upper levels (internal pages with incomplete splits are
         * also taken care of in _bt_getstackbuf).  But this is a good
         * opportunity to finish splits of internal pages too.
         */
        *bufP = _bt_moveright(rel, heaprel, key, *bufP, (access == BT_WRITE),
                              stack_in, page_access);

        /* if this is a leaf page, we're done */
        page = BufferGetPage(*bufP);
        opaque = BTPageGetOpaque(page);
        if (P_ISLEAF(opaque))
            break;

        /*
         * Find the appropriate pivot tuple on this page.  Its downlink points
         * to the child page that we're about to descend to.
         */
        offnum = _bt_binsrch(rel, key, *bufP);
        itemid = PageGetItemId(page, offnum);
        itup = (IndexTuple) PageGetItem(page, itemid);
        Assert(BTreeTupleIsPivot(itup) || !key->heapkeyspace);
        child = BTreeTupleGetDownLink(itup);

        /*
         * We need to save the location of the pivot tuple we chose in a new
         * stack entry for this page/level.  If caller ends up splitting a
         * page one level down, it usually ends up inserting a new pivot
         * tuple/downlink immediately after the location recorded here.
         */
        if (returnstack)
        {
            new_stack = (BTStack) palloc_object(BTStackData);
            new_stack->bts_blkno = BufferGetBlockNumber(*bufP);
            new_stack->bts_offset = offnum;
            new_stack->bts_parent = stack_in;
            stack_in = new_stack;
        }

        /*
         * Page level 1 is lowest non-leaf page level prior to leaves.  So, if
         * we're on the level 1 and asked to lock leaf page in write mode,
         * then lock next page in write mode, because it must be a leaf.
         */
        if (opaque->btpo_level == 1 && access == BT_WRITE)
            page_access = BT_WRITE;

        /* drop the read lock on the page, then acquire one on its child */
        *bufP = _bt_relandgetbuf(rel, *bufP, child, page_access);

        /* okay, all set to move down a level */
    }

    /*
     * If we're asked to lock leaf in write mode, but didn't manage to, then
     * relock.  This should only happen when the root page is a leaf page (and
     * the only page in the index other than the metapage).
     */
    if (access == BT_WRITE && page_access == BT_READ)
    {
        /* trade in our read lock for a write lock */
        _bt_unlockbuf(rel, *bufP);
        _bt_lockbuf(rel, *bufP, BT_WRITE);

        /*
         * Race -- the leaf page may have split after we dropped the read lock
         * but before we acquired a write lock.  If it has, we may need to
         * move right to its new sibling.  Do that.
         */
        *bufP = _bt_moveright(rel, heaprel, key, *bufP, true, stack_in, BT_WRITE);
    }

    return stack_in;
}
```

逐段解说：

- **入口断言**（:107-108）：写模式必须提供 heaprel——因为写下降可能触发 `_bt_finish_split`，后者可能引发父页分裂、需要 `_bt_allocbuf`（nbtpage.c:854）分配新页，而分配新页可能要回收旧的已删除页，回收判断需要问堆表的可见性视界（见 6.4 节）。一个参数的存在性背后是一整条因果链。
- **`_bt_getroot`**（:111，实现在 nbtpage.c:347）：优先从 relcache 的 `rd_amcache` 拿缓存的 fast root 块号，避免每次搜索都读 metapage；缓存可能过期，但拿到的页自带层号与标志位，可以自我验证并修正——又是"允许过期 + 廉价验证"的 L&Y 式思路。
- **主循环每层三步**：修右移（`_bt_moveright`，:140）→ 页内二分（`_bt_binsrch`，:153）→ 取 downlink 下潜（:157、:183）。注意循环体第一件事**永远**是 `_bt_moveright`——竞态修复不是异常路径，而是每层的标准动作（不冲突时它只花一次比较）。
- **BTStack**（:165-172）：记录"我从哪个父页哪个偏移下来的"。将来叶子分裂要往父页插 downlink 时，从这个栈出发重定位父页（`_bt_getstackbuf`）。**栈只记块号和偏移，不留锁也不留 pin**——父页此刻起就可以被别人任意分裂/插入，重定位时按子页块号匹配即可。这是教科书"持父锁上升"的 PG 替代品。`returnstack=false`（纯读）时干脆不分配，省 palloc。
- **level==1 优化**（:179-180）：写模式下降到倒数第二层（level 1）时，提前把下一层的锁模式切成 BT_WRITE。若不这么做，就得先拿叶页读锁、再放掉换写锁、再 moveright 一轮——多一次锁往返。内部页永远只拿读锁（修改内部页的是"上升"路径，不是"下降"路径）。
- **`_bt_relandgetbuf`**（:183，nbtpage.c:988）：**先放本页锁，再锁子页**——层间从不同时持有两把锁，这正是"无锁耦合"的代码落点。对比 2.1 节：锁耦合方案在此处是"先锁子页、再放父锁"，一词之差，并发行为天壤之别。
- **尾部的根即叶特例**（:193-205）：整棵树只有一个页时，循环第一轮就 break 了，锁还是读锁；换成写锁的窗口里页可能又分裂了，所以换锁后必须再 `_bt_moveright` 一次。**任何"放锁-再拿锁"的缝隙后面都跟着一次 moveright**——这个模式在 nbtree 里至少出现四次，可当作校验代码正确性的心智规则。

### 3.1 _bt_moveright：并发分裂的修复者（全函数）

`_bt_moveright`（nbtsearch.c:242-322），除头部大注释外全文：

```c
static Buffer
_bt_moveright(Relation rel,
              Relation heaprel,
              BTScanInsert key,
              Buffer buf,
              bool forupdate,
              BTStack stack,
              int access)
{
    Page        page;
    BTPageOpaque opaque;
    int32       cmpval;

    Assert(!forupdate || heaprel != NULL);

    cmpval = key->nextkey ? 0 : 1;

    for (;;)
    {
        page = BufferGetPage(buf);
        opaque = BTPageGetOpaque(page);

        if (P_RIGHTMOST(opaque))
            break;

        /*
         * Finish any incomplete splits we encounter along the way.
         */
        if (forupdate && P_INCOMPLETE_SPLIT(opaque))
        {
            BlockNumber blkno = BufferGetBlockNumber(buf);

            /* upgrade our lock if necessary */
            if (access == BT_READ)
            {
                _bt_unlockbuf(rel, buf);
                _bt_lockbuf(rel, buf, BT_WRITE);
            }

            if (P_INCOMPLETE_SPLIT(opaque))
                _bt_finish_split(rel, heaprel, buf, stack);
            else
                _bt_relbuf(rel, buf);

            /* re-acquire the lock in the right mode, and re-check */
            buf = _bt_getbuf(rel, blkno, access);
            continue;
        }

        if (P_IGNORE(opaque) || _bt_compare(rel, key, page, P_HIKEY) >= cmpval)
        {
            /* step right one page */
            buf = _bt_relandgetbuf(rel, buf, opaque->btpo_next, access);
            continue;
        }
        else
            break;
    }

    if (P_IGNORE(opaque))
        elog(ERROR, "fell off the end of index \"%s\"",
             RelationGetRelationName(rel));

    return buf;
}
```

- **判定规则一行写尽**（:307）：搜索键 ≥（nextkey=true 时）或 >（常规）本页 high key（`P_HIKEY` 恒为页内第 1 项）就向右走一页，循环直到落入正确键区间。`nextkey` 用于范围扫描定位"严格大于某键的第一项"。头部注释还提醒："The page could even have split more than once, so scan as far as needed."
- **`P_RIGHTMOST` 先判**（:280）：最右页 high key = +∞，无条件停下。也保证了下面 `_bt_compare(page, P_HIKEY)` 永远有 high key 可比。
- **`P_IGNORE`**（:307）：页处于 half-dead 或 deleted 状态（并发页删除的墓碑，见 6.1 节）也向右走——对读者而言，"页被删了"与"页分裂了"共用同一套恢复动作：它的键区间已并入右兄弟。设计的对称性极佳。
- **`P_INCOMPLETE_SPLIT` 分支**（:286-305）：只有写下降（forupdate）才处理。遇到"分裂做了一半"的页，先把锁升级成写锁（注意 :297 升级后**重查标志**——升级锁的窗口里别人可能已经替你修完了），调 `_bt_finish_split` 替前人补插父层 downlink（该函数会消耗掉锁与 pin），再从 :303 重新拿页、`continue` 重新检查。读下降不管它——incomplete split 对读者无害（右页经 right-link 可达）。为什么写者必须管，第五节给出完整论证。
- **尾部防御**（:317-319）：走到 `P_IGNORE` 且最右——树坏了，报 "fell off the end of index"。amcheck 出现之前，这是索引损坏最常见的用户可见信号之一。

### 3.2 _bt_binsrch：页内二分的精确边界

`_bt_binsrch`（nbtsearch.c:344-450）是标准二分，但边界语义精确到苛刻。核心循环：

```c
    high++;                     /* establish the loop invariant for high */

    cmpval = key->nextkey ? 0 : 1;  /* select comparison value */

    while (high > low)
    {
        OffsetNumber mid = low + ((high - low) / 2);

        /* We have low <= mid < high, so mid points at a real slot */

        result = _bt_compare(rel, key, page, mid);

        if (result >= cmpval)
            low = mid + 1;
        else
            high = mid;
    }
```

- 不变式（:380-384 注释原文）：`low` 之前的槽全部 < 搜索键，`high` 及之后全部 ≥ 搜索键（nextkey=true 时把 </≥ 换成 ≤/>）；收敛到 `high == low`。
- **叶页返回 `low`**（第一个 ≥ 键的项，即正向扫描起点；`key->backward` 反向扫描时返回 `OffsetNumberPrev(low)`，:424-427——反向扫描需要"最后一个匹配项"）；**内部页返回 `low-1`**（:449，最后一个 < 键的 pivot，其 downlink 覆盖搜索键所在子树——子树区间是 Ki < v ≤ Ki+1，TID tiebreaker 保证 pivot 上几乎不会发生真正的相等比较）。
- 内部页第一个数据项被 `_bt_compare`（nbtsearch.c:689）视作 **-∞**（不做实际比较直接判小），保证 `low-1 ≥ P_FIRSTDATAKEY` 恒成立（:447 的 Assert）。这个 -∞ 项就是每个内部页的第一个 pivot——只有 downlink 没有键，第五节分裂代码里会看到它的诞生。
- 插入路径用变体 `_bt_binsrch_insert`（nbtsearch.c:475），把二分收敛后的 `low`/`stricthigh` 缓存进 `BTInsertState`——唯一性检查刚做过的二分，插入定位时直接复用，省一次 O(log n)；新 TID 落在既有 posting list 区间内时还会经 `_bt_binsrch_posting`（nbtsearch.c:603）算出 postingoff（第六节 dedup 用）。

**小结**：`_bt_search` = "每层：修右移 → 二分 → 记栈 → 放锁下潜"。全部并发防御浓缩在 `_bt_moveright` 一个调用里：high key 判右移、P_IGNORE 判死页、INCOMPLETE_SPLIT 判半程分裂，三种异常一个循环收敛。BTStack 用"记路标不持锁"替代教科书的锁耦合，代价是使用路标时要按块号重新找 downlink——这笔交易的另一半在 `_bt_getstackbuf`（第五节）。

## 四、B-tree（三）：插入主线——_bt_doinsert 到 _bt_insertonpg

### 4.1 _bt_doinsert：主流程走读

入口 `_bt_doinsert`（`src/backend/access/nbtree/nbtinsert.c:105`），由 AM 接口 `btinsert` 调用。逐段走读（除注释外基本全文）。

**第一段：构造 scankey 与 NULL 旁路**（:109-141）：

```c
    bool        is_unique = false;

             * checkingunique steps.  They are unnecessary because core code
             * considers NULL unequal to every value, including NULL.
```

两个关键决定都在这里做出：

- **scantid 临时置 NULL**（:123）：完整的插入键是"用户键列 + 堆 TID"。唯一性检查阶段必须刻意**抹掉 TID 尾键**——带着 TID 搜索会直接定位到"该键 + 本 TID"的精确位置，而检查需要的是"该键的**第一个**可能位置"（所有同键元组从这里开始）。检查通过后再把 scantid 装回去（:239-240）。
- **键含 NULL 直接跳过检查**（:125-140）：SQL 语义 NULL ≠ NULL（默认 NULLS DISTINCT），含 NULL 的键永远"唯一"。注释还点出一个性能动机：不跳过的话，大量 NULL "重复键"会让 `_bt_findinsertloc` 的定位退化成 O(N²)。

**第二段：搜索 + 唯一性检查 + 等待循环**（:163-241）：

```c

    {
        TransactionId xwait;
        uint32      speculativeToken;

        xwait = _bt_check_unique(rel, &insertstate, heapRel, checkUnique,
                                 &is_unique, &speculativeToken);

        if (unlikely(TransactionIdIsValid(xwait)))
        {
            /* Have to wait for the other guy ... */
            _bt_relbuf(rel, insertstate.buf);
            insertstate.buf = InvalidBuffer;

            /*
             * If it's a speculative insertion, wait for it to finish (ie. to
             * go ahead with the insertion, or kill the tuple).  Otherwise
             * wait for the transaction to finish as usual.
             */
            if (speculativeToken)
                SpeculativeInsertionWait(xwait, speculativeToken);
            else

            if (stack)
                _bt_freestack(stack);
            goto search;

            itup_key->scantid = &itup->t_tid;
```

`_bt_doinsert` 头部有一段极重要的注释（:190-199），值得原文引用——它是唯一索引并发正确性的全部依据：

```c
     * NOTE: obviously, _bt_check_unique can only detect keys that are already
     * in the index; so it cannot defend against concurrent insertions of the
     * same key.  We protect against that by means of holding a write lock on
     * the first page the value could be on, with omitted/-inf value for the
     * implicit heap TID tiebreaker attribute.  Any other would-be inserter of
     * the same key must acquire a write lock on the same page, so only one
     * would-be inserter can be making the check at one time.  Furthermore,
     * once we are past the check we hold write locks continuously until we
     * have performed our insertion, so no later inserter can fail to see our
     * insertion.
```

翻译成不变式：**"某键可能落在的第一个叶页"的写锁，就是该键的互斥量**。任何两个想插同一键的会话必然在同一个页上抢同一把写锁；抢到锁的检查完成后到插入完成前锁不间断（`_bt_findinsertloc` 右移时也是先锁新页再放旧页，见 4.5），所以后来者必能看见先行者的插入。等待逻辑见 4.4 的 what-if 推演。

```c
                PageGetMaxOffsetNumber(page) >= P_HIKEY &&
                _bt_compare(rel, insertstate->itup_key, page, P_HIKEY) > 0)
            {
                ...
                return NULL;
            }
```

单调递增插入（自增主键、时间戳）时，缓存上次插入的最右叶页块号（`RelationGetTargetBlock`）；下次插入直接**条件加锁**（拿不到锁立即放弃而不是排队——高并发下缓存很快失效，排队不值得），验证五个条件：仍是最右叶页、未被删除、装得下新元组、页上有数据、新键大于页内第一项。全部通过则跳过整个树下降，返回 NULL 栈。注意两个细节：

- 页放不下时**不走**快速路径（:308-312 注释）：因为分裂需要真实的下降栈，NULL 栈的分裂要靠重新搜索定位父页，很慢；
- 该优化对 append-only 负载把 O(log N) 的下降变成 O(1)。InnoDB 对应的缓解是最右页的 "split at insert point" 启发式，思路同源但 PG 直接省掉了下降本身。

### 4.3 _bt_check_unique 全函数走读：唯一约束与 MVCC 的微妙互动

`_bt_check_unique`（nbtinsert.c:411-786）可能是全内核里 MVCC 语义最密集的 300 行。签名与返回值约定：

```c
static TransactionId
_bt_check_unique(Relation rel, BTInsertState insertstate, Relation heapRel,
                 IndexUniqueCheck checkUnique, bool *is_unique,
                 uint32 *speculativeToken)
```

返回 `InvalidTransactionId` 表示无冲突；返回某个 XID 表示"撞上了一个**未决**事务插入的同键元组，等它"；发现**已提交**的存活同键元组则直接 `ereport(ERROR)` 不返回。

**准备段**（:415-446）：

```c
    SnapshotData SnapshotDirty;
    ...
    InitDirtySnapshot(SnapshotDirty);

    page = BufferGetPage(insertstate->buf);
    opaque = BTPageGetOpaque(page);
    maxoff = PageGetMaxOffsetNumber(page);

    /*
     * Find the first tuple with the same key.
     *
     * This also saves the binary search bounds in insertstate.  We use them
     * in the fastpath below, but also in the _bt_findinsertloc() call later.
     */
    Assert(!insertstate->bounds_valid);
    offset = _bt_binsrch_insert(rel, insertstate);
```

第一个 MVCC 微妙点：**用 DirtySnapshot 而不是事务快照**。普通快照只能看见已提交数据；但唯一性检查必须把"未提交的插入"也当作潜在冲突——否则两个并发事务都看不见对方、双双插入成功，提交后唯一约束就被击穿了。DirtySnapshot 的语义是"已提交的 + 正在进行中的都可见"，并把碰到的进行中事务的 xmin/xmax 回填到快照结构里供调用方决定等谁。

**主循环**（:454-766）——注意循环的迭代单位是**堆 TID 而不是索引元组**（因为 dedup 之后一条 posting list 元组内含多个 TID，`inposting`/`curposti` 状态机负责在 posting list 内逐 TID 步进）：

```c
    for (;;)
    {
        ...
        if (offset <= maxoff)
        {
            /*
             * Fastpath: In most cases, we can use cached search bounds ...
             */
            if (nbuf == InvalidBuffer && offset == insertstate->stricthigh)
            {
                Assert(insertstate->bounds_valid);
                ...
                break;
            }

            /*
             * We can skip items that are already marked killed.
             * ...
             */

            {

                bool        all_dead = false;

                if (!inposting)
                {

                        break;  /* we're past all the equal tuples */
                    ...
                }
```

逐个要点：

- **fastpath**（:486-493）：二分时已缓存 `stricthigh`（第一个严格大于插入键的偏移）。扫到它就说明同键元组已看完，直接 break。没有任何重复键的常见情形下，一次 `_bt_compare` 都不用做。
- **跳过 LP_DEAD 项**（:496-508）：被先前扫描标记为"全链已死"的项直接跳过，不做比较也不回表。热更新负载下同键死项可能成百上千，这个跳过是硬性能需求。

**命中同键后的三岔路**（:546-710），这是与 MVCC 互动的核心：

```c
                if (checkUnique == UNIQUE_CHECK_EXISTING &&
                    ItemPointerCompare(&htid, &itup->t_tid) == 0)
                {
                    found = true;
                }
                else if (table_index_fetch_tuple_check(heapRel, &htid,
                                                       &SnapshotDirty,
                                                       &all_dead))
                {
                    TransactionId xwait;
                    ...
                    if (checkUnique == UNIQUE_CHECK_PARTIAL)
                    {
                        if (nbuf != InvalidBuffer)
                            _bt_relbuf(rel, nbuf);
                        *is_unique = false;
                        return InvalidTransactionId;
                    }

                    /*
                     * If this tuple is being updated by other transaction
                     * then we have to wait for its commit/abort.
                     */
                    xwait = (TransactionIdIsValid(SnapshotDirty.xmin)) ?
                        SnapshotDirty.xmin : SnapshotDirty.xmax;

                    if (TransactionIdIsValid(xwait))
                    {
                        if (nbuf != InvalidBuffer)
                            _bt_relbuf(rel, nbuf);
                        /* Tell _bt_doinsert to wait... */
                        *speculativeToken = SnapshotDirty.speculativeToken;
                        /* Caller releases lock on buf immediately */
                        insertstate->bounds_valid = false;
                        return xwait;
                    }
```

- **回表是必须的**：索引项不带可见性信息，索引里存在同键项完全不说明冲突——它很可能是一个已删除行的残影。`table_index_fetch_tuple_check` 按 DirtySnapshot 回堆验证（内部会追 HOT 链——一条索引项代表整条 HOT 链，链上任何成员满足 DirtySnapshot 都算数）。
- **三种结局**：(a) 堆里确有 DirtySnapshot 可见的同键行，且创建/删除它的事务已结束 → 真冲突；(b) 该行由未决事务插入（`SnapshotDirty.xmin` 有效）或正被未决事务删除（xmax 有效）→ 返回 xwait 让调用方等待；(c) 回表发现全链皆死（all_dead）→ 顺手把索引项标成 LP_DEAD（:679-710，用 hint-bit 基础设施在共享锁下安全写标志），下次没人再为它回表。
- **真冲突还有最后一道检查**（:603-633）：报错前先看**自己要插的行**是否还活着（用 SnapshotSelf 回表）。这不是多余——CREATE INDEX CONCURRENTLY 期间，插入索引项的事务可能同时在删这行；如果自己的行已死，直接 break 不报错。注释解释了为何必须追 HOT 链。
- **报错前先放锁**（:644-677）：`BuildIndexValueDescription` 要访问系统目录，最坏情况会递归摸到当前索引造成死锁，所以先 `_bt_relbuf` 再 ereport。错误消息就是用户熟悉的 `duplicate key value violates unique constraint`。

**跨页推进段**（:721-765）：同键元组可能跨页（一页装不下那么多重复键时）。当前页扫完后比较插入键与 high key：严格小于则结束；**等于**则右邻可能还有同键项，沿 right-link 拿右页**读锁**（`nbuf`）继续扫。注意此时**左页（insertstate->buf）的写锁始终不放**——互斥量不能丢；右页只需读锁，因为想插该键的对手会先卡在左页写锁上。这也解释了 `_bt_doinsert` 注释里说的 "In practice the first page the value could be on ... is almost always also the only page"——`_bt_findsplitloc` 刻意不让重复键跨页（见第五节），跨页扫描极少发生。

**收尾**（:768-785）：`UNIQUE_CHECK_EXISTING` 模式下没找到自己的 TID 则报 "failed to re-find tuple"（典型原因：索引表达式不是 immutable，两次求值结果不同）；释放 nbuf，返回无冲突。

### 4.4 what-if 推演 ②：两个事务同时插入相同唯一键的完整仲裁

设事务 A、B 同时向唯一索引插入键 k（对应不同堆行），完整时序：

```
t1  A: _bt_search_insert 拿到"k 可能所在第一叶页" P 的写锁
t2  B: _bt_search_insert 到达同一页 P，排队等写锁
t3  A: _bt_check_unique 扫 P：索引里没有 k（或只有死项）→ 无冲突
t4  A: scantid 装回，_bt_findinsertloc / _bt_insertonpg 完成插入，
       期间写锁连续持有；插入完成才放锁
t5  B: 拿到 P 的写锁，_bt_check_unique 扫到 A 刚插入的 k 项
t6  B: 回表（DirtySnapshot）：行存在，但 xmin = A 的 XID，仍在进行中
       → SnapshotDirty.xmin 有效 → 返回 xwait = A
t7  B: _bt_doinsert 放掉 P 的锁（不能持锁等，否则阻塞无关插入且可能死锁），
       XactLockTableWait(A) 睡在 A 的事务锁上
t8a A 提交 → B 醒来，goto search 从头再来：
       重新搜索、重新拿 P 写锁、重新 check_unique
       → 回表见到已提交的 k 行 → ereport "duplicate key value"
t8b A 回滚 → B 醒来重来：回表时 k 行的 xmin 已 abort，DirtySnapshot
       不可见 → all_dead 大概率成立，B 顺手把 A 的索引项标 LP_DEAD，
       然后自己正常插入成功
```

细节问答：

- **为什么 B 醒来后必须 `goto search` 整个重做，而不是接着刚才的位置继续？** 放锁等待期间页可能分裂、目标位置可能右移、甚至可能有第三个事务 C 又插了 k。所有缓存状态（bounds、offset）都不可信，只有重来。
- **speculativeToken 是什么？** `INSERT ... ON CONFLICT` 的投机插入：行先写进堆但标记"投机"，冲突时可以无痕撤销单行而不必回滚整个事务。等待方用 `SpeculativeInsertionWait` 只等"这一行的裁决"而不是整个事务结束——避免 ON CONFLICT 场景下的活锁（两个投机插入互等对方事务结束会死锁，等"单行裁决"则不会）。
- **InnoDB 对比**：InnoDB 靠插入意向锁 + 记录锁/间隙锁仲裁——B 会在 k 的记录上挂 S 锁等 A，A 回滚则 B 获准插入。原理同构（都是"等未决事务"），但 InnoDB 的等待队列挂在行锁管理器上，PG 挂在"事务锁"（每个事务启动时锁自己 XID，别人等它 = 等事务结束）上；PG 的索引层不维护任何锁表，页写锁 + 事务锁两个原语拼出了唯一约束。

### 4.5 _bt_findinsertloc：页内定位与三条免分裂退路

`_bt_findinsertloc`（nbtinsert.c:829-1026）在拿到"第一个可能页"后精确定位插入偏移。heapkeyspace 主路径（:854-921）：

```c
        if (checkingunique)
        {
            if (insertstate->low < insertstate->stricthigh)
            {
                /* Encountered a duplicate in _bt_check_unique() */
                Assert(insertstate->bounds_valid);
                uniquedup = true;
            }

            for (;;)
            {
                /*
                 * Does the new tuple belong on this page? ...
                 */
                if (insertstate->bounds_valid &&
                    insertstate->low <= insertstate->stricthigh &&
                    insertstate->stricthigh <= PageGetMaxOffsetNumber(page))
                    break;

                /* Test '<=', not '!=', since scantid is set now */
                if (P_RIGHTMOST(opaque) ||
                    _bt_compare(rel, itup_key, page, P_HIKEY) <= 0)
                    break;

                _bt_stepright(rel, heapRel, insertstate, stack);
                /* Update local state after stepping right */
                page = BufferGetPage(insertstate->buf);
                opaque = BTPageGetOpaque(page);
                /* Assume duplicates (if checkingunique) */
                uniquedup = true;
            }
        }

        /*
         * If the target page cannot fit newitem, try to avoid splitting the
         * page on insert by performing deletion or deduplication now
         */
        if (PageGetFreeSpace(page) < insertstate->itemsz)
            _bt_delete_or_dedup_one_page(rel, heapRel, insertstate, false,
                                         checkingunique, uniquedup,
                                         indexUnchanged);
```

- **为什么可能右移**：唯一性检查是不带 scantid 搜的，落在"该键第一页"；现在 scantid 装回来了，本 TID 的精确归属可能在右兄弟（同键跨页时）。右移判据是完整键（含 TID）对 high key 的比较。
- **`_bt_stepright`**（nbtinsert.c:1041-1086）与搜索时右移的差别是锁协议：**先写锁右页、再放左页**——4.1 节引用的注释说了，检查通过后写锁必须连续，中途出现无锁窗口会让并发的 `_bt_check_unique` 看不到我们即将进行的插入。这是全 nbtree 少数持双页锁前进的地方。它也顺手处理路上的 incomplete split（:1067-1072）。
- **三条免分裂退路**：页满时调 `_bt_delete_or_dedup_one_page`（nbtinsert.c:2730-2829），依次尝试——
  1. **simple deletion**（:2768-2780）：删掉页上所有 LP_DEAD 项（`_bt_simpledel_pass`，nbtinsert.c:2859；还会顺带检查同一批堆块上的其他项能否搭车删除）。这是 `kill_prior_tuple` 反馈回路（第十节）攒下的"垃圾情报"变现的时刻；
  2. **bottom-up deletion**（:2821-2823，`_bt_bottomupdel_pass`，nbtdedup.c:309）：executor 传来 `indexUnchanged` 提示（UPDATE 没改本索引的列，纯粹为新版本添项）或唯一索引发现重复时，主动回表探测一批"很可能已死"的 TID 并删除——针对版本膨胀的定向反击（PG 14，见第八节）；
  3. **deduplication**（:2826-2828，`_bt_dedup_pass`，nbtdedup.c:59）：把同键元组压成 posting list（6.3 节）。
  三条都救不回来，才由 `_bt_insertonpg` 认命分裂。语义顺序值得体会：**先删真垃圾，再删可能的垃圾，再压缩活数据，最后才扩张结构**。
- 尾部 `postingoff == -1` 的特例（:1004-1023）：定位命中了一个 LP_DEAD 的 posting list 元组——不值得为分裂它而清 LP_DEAD 位，先做 simple deletion 把它删掉再重新二分。

要点：

- **posting list split 融入同一临界区**（:1299-1300）：新 TID 落在既有 posting list 的 TID 区间内时，覆写旧 posting（`nposting` 是分裂后的版本）+ 插入新元组，两个动作一条 WAL（`XLOG_BTREE_INSERT_POST`）——保证"从 posting 里挤出来的 TID"不会在崩溃后丢失或重复；
- **清子页 INCOMPLETE_SPLIT 标志与父页插入同条 WAL**（:1322-1330 与 :1362-1363）：当本次调用是"往父页插 downlink"（`!isleaf`，`cbuf` 是分裂的左子页）时，父页加项与子页清标志在同一临界区、同一条 WAL（`XLOG_BTREE_INSERT_UPPER`）内原子完成。**"分裂完成"这个状态转换是原子的**——第五节的 incomplete split 机制依赖这一点；
- 若这次父页插入本身又放不下，则走 :1225-1263 的分裂分支：`_bt_split` + `_bt_insert_parent` 递归上升，`cbuf` 的清标志动作转由 `_bt_split` 在它的临界区里完成（:1987-1994）；
- **fast root / metapage**：`split_only_page` 场景顺带更新 metapage 的 fastroot（:1279-1294、:1308-1316）；
- 尾部维护最右页缓存（:1433-1449）：本次插入落在最右叶页且树高 ≥ `BTREE_FASTPATH_MIN_LEVEL` 才缓存——小索引搜索本来就快，缓存反而添乱。注意 `_bt_getrootheight` 在放锁**之后**才调（:1444-1445 注释），持锁调用可能与 metapage 锁形成环。

**小结**：插入主线 = "搜到第一可能页并写锁（可走最右页 O(1) 快速路径）→ DirtySnapshot 回表式唯一检查（锁即互斥量，未决则放锁等待重来）→ 右移到精确页（锁连续）→ 三条退路避免分裂 → 临界区内页内插入 + 单条 WAL"。每一步的锁纪律都可以从 4.1 节那段注释推导出来。

## 五、B-tree（四）：_bt_split 全函数走读——多页原子修改的典范

页满时 `_bt_insertonpg` 执行（nbtinsert.c:1232、:1263）：

```c
        /* split the buffer into left and right halves */
        rbuf = _bt_split(rel, heaprel, itup_key, buf, cbuf, newitemoff, itemsz,
                         itup, origitup, nposting, postingoff);
        ...
        _bt_insert_parent(rel, heaprel, buf, rbuf, stack, isroot, isonly);
```

一次分裂 = 两个各自原子的动作：**本层分裂**（`_bt_split`，一条 WAL）+ **父层插 downlink**（`_bt_insert_parent` → `_bt_insertonpg`，另一条 WAL）。先走读第一个。

### 5.1 _bt_split 走读（nbtinsert.c:1489-2111）

`_bt_split` 要在一个临界区里改**三个页**（原页、新右页、老右兄弟），还要为可能的失败留好退路。它的结构是教科书级的"先备份、后动手"：

**第一段：选分裂点**（:1536-1568）：

```c
    origpage = BufferGetPage(buf);
    oopaque = BTPageGetOpaque(origpage);
    isleaf = P_ISLEAF(oopaque);
    isrightmost = P_RIGHTMOST(oopaque);
    maxoff = PageGetMaxOffsetNumber(origpage);
    origpagenumber = BufferGetBlockNumber(buf);

    ...
    firstrightoff = _bt_findsplitloc(rel, origpage, newitemoff, newitemsz,
                                     newitem, &newitemonleft);
```

`_bt_findsplitloc`（nbtsplitloc.c:130）以"把新元组假想放进去后的页"为基准枚举候选分割点，在三个目标间加权：左右空间均衡、suffix truncation 收益最大（挑相邻元组差异最靠前的分割点，见 6.2）、以及两个特殊模式——最右页分裂按 90:10 不对称分（顺序插入负载下右页马上又会被填满，均分是浪费）；重复键成堆时刻意选极不均衡的点把重复键"困"在左页（配合 `_bt_check_unique` 的"第一页几乎总是唯一页"假设）。分裂点表示为"firstright 偏移 + newitemonleft 标志"，兼容新元组恰为分界的两个边界情形（:1543-1565 的注释详细讨论了 `firstrightoff == maxoff+1` 的合法性）。

**第二段：本地组装左页镜像**（:1570-1593）：

```c

    lopaque->btpo_flags |= BTP_INCOMPLETE_SPLIT;
```

左页不是原地改，而是在**栈上的临时缓冲**（`PGAlignedBlock leftpage_buf`）里从零组装，成功后一次 memcpy 覆盖原页。函数头注释（:1523-1534）解释了顺序安排：leftpage 只是工作区，"Failing before acquiring new buffer won't have lasting consequences, since origpage won't have been modified"——**新右页缓冲要等左页镜像连 high key 都装好之后才申请**，此前任何失败（内存不足、truncation 报错）都零副作用地 ERROR 掉。注意 :1582：**`BTP_INCOMPLETE_SPLIT` 标志在镜像组装阶段就打在左页上**——它将随分裂 WAL 一起落盘，是"父层 downlink 还没插"的持久化声明。

**第三段：确定 firstright / lastleft，生成左页新 high key**（:1613-1714）：

- 叶页分裂：新 high key = `_bt_truncate(lastleft, firstright)`（nbtutils.c:692）——一个**介于 lastleft 与 firstright 之间、且尽可能短**的分隔键（6.2 节详述）。:1623-1641 的注释澄清了与 L&Y 原文的关系：论文用 lastleft 的副本当新 high key；PG 的 truncation 结果 ≥ lastleft 且 < firstright，同样满足子树不变式。
- **内部页分裂不做 truncation**（:1687-1713 注释）：每个分隔键都诞生于某次叶分裂，然后被逐层向上复制。祖父页的分隔键必须与其左子树最右父页的 high key**逐字节一致**、后者又与其最右子页的 high key 一致——各层必须存在一条"相同分隔键的接缝（seam）"引导搜索。内部页再截一刀会撕破接缝。
- 同段注释还预告：内部页分裂时 firstright 元组进右页后会被截成只剩 downlink 的 **-∞ 项**（"minus infinity" item）——这就是 3.2 节里 `_bt_compare` 直接判小的那个页首项的来历，注释特意强调 "Do not confuse this mechanism with suffix truncation"。
- 若分裂恰逢 posting list split，`origpagepostingoff` 让所有取数逻辑用"假想已完成 posting 分裂的页"作基准（:1613-1621）。

页布局在此定型——**右页整体继承原页的键上界，左页收缩**：

- 右页：`[原页的旧 high key | 原页右半数据（+新元组，如果它属于右半）]`，right-link → 原页的老右兄弟，left-link → 原页；内部页分裂时右页第一个数据项在 `_bt_pgaddtup` 时被截成 -∞ 项（:1808-1810 的 `minusinfoff` 与 :1877-1878）；

### 5.2 _bt_insert_parent 与递归上升

`_bt_insert_parent`（nbtinsert.c:2130-2256）完成第二阶段：

```c
        /* get high key from left, a strict lower bound for new right page */
        ritem = (IndexTuple) PageGetItem(page,
                                         PageGetItemId(page, P_HIKEY));

        /* form an index tuple that points at the new right page */
        new_item = CopyIndexTuple(ritem);
        BTreeTupleSetDownLink(new_item, rbknum);

        /*
         * Re-find and write lock the parent of buf.
         * ...
         */
        pbuf = _bt_getstackbuf(rel, heaprel, stack, bknum);

        /*
         * Unlock the right child.  The left child will be unlocked in
         * _bt_insertonpg().
         * ...
         */
        _bt_relbuf(rel, rbuf);

        if (pbuf == InvalidBuffer)
            ereport(ERROR, (errcode(ERRCODE_INDEX_CORRUPTED), ...));

        /* Recursively insert into the parent */
        _bt_insertonpg(rel, heaprel, NULL, pbuf, buf, stack->bts_parent,
                       new_item, MAXALIGN(IndexTupleSize(new_item)),
                       stack->bts_offset + 1, 0, isonly);
```

- **新 downlink 的键就是左页刚生成的 high key**（:2208-2214）——分隔键的"接缝"由此建立：叶层 high key → 父层 downlink 键 → 未来祖父层，逐层逐字节复制；
- **`_bt_getstackbuf`**（nbtinsert.c:2351）从栈记录的 `bts_blkno` 出发写锁父页，按**子页块号**先右扫后左扫地找 downlink；找不到就沿 right-link 追（父页自己也可能分裂过了）——L&Y 式补偿在上升方向的重演。路上遇到父页自己的 INCOMPLETE_SPLIT 也先修（:2370-2374）；
- 随后递归进 `_bt_insertonpg`：父页够放则 4.6 节的主路径完成"加 downlink + 清子页标志"；父页也满则再来一轮 `_bt_split` + `_bt_insert_parent`——**每一层都是独立的两阶段**；
- **根分裂**（:2153-2165）：栈空且确是根时，`_bt_newlevel`（nbtinsert.c:2492）造新根（含指向左右页的两个 downlink，左 downlink 是 -∞ 项）并更新 metapage，一条 `XLOG_BTREE_NEWROOT`。栈空但根已被别人升级过（"concurrent ROOT page split"，:2180）则从新根层的最左页开始右找（:2199-2205 的 fakestack）。

### 5.3 所以然 ②：为什么必须"先分裂后插父"，且允许 incomplete split 存在

现在回答本章第二个"所以然"。设想两种反面方案：

**反面方案 A：分裂 + 插父打包成一条 WAL**。看似干净，实际做不到普遍成立——插父可能触发父页分裂，父页分裂又可能触发祖父分裂，直到根。一条 WAL 记录将要覆盖**不定层数、最多整条路径**的页修改；WAL 记录必须在所有页都锁定并修改后才能写出，意味着写者要同时持有整条上升路径的写锁——**这就把 L&Y 省掉的路径锁又请回来了**，且 WAL 记录大小无上界（每层还携带整页数据）。PG 9.4 之前的方案接近它的变体（分裂时记录"未完成分裂"到 WAL，恢复末尾统一补），带来的是恢复逻辑的复杂与 standby 上无法处理等问题。

**反面方案 B：崩溃后由恢复流程补插父**。9.4 之前正是如此（见第八节 commit 40dae7ec537）：REDO 结束后扫描"未完成分裂"清单补作业。问题一，补插父可能需要分裂父页 → 需要**分配新页** → 恢复流程被迫能做常规空间分配，而磁盘满时恢复不允许失败；问题二，逻辑只在恢复路径存在，平时永远不跑，是测试盲区（"rarely exercised code path" 是 bug 温床）。

**PG 9.4+ 的现行方案**：把"父层 downlink 缺失"定义为**合法的树状态**，用 `BTP_INCOMPLETE_SPLIT` 标志持久化声明，修复责任交给**下一个恰好路过的写入者**：

- 正确性不受影响：搜索到达左页后发现键超出 high key，沿 right-link 照样能到右页——incomplete split 状态下右页只是"少一条快捷方式"，L&Y 不变式毫发无损。读者完全无感；
- 修复触发点有三处，全在写路径：`_bt_moveright`（nbtsearch.c:286-305，写下降路过）、`_bt_stepright`（nbtinsert.c:1067-1072，插入右移路过）、`_bt_getstackbuf`（nbtinsert.c:2370-2374，上升定位父页时路过）。修复者调 `_bt_finish_split`（nbtinsert.c:2272-2317）：锁右兄弟、判断是否根分裂（查 metapage）、然后调用与正常路径**完全相同**的 `_bt_insert_parent`——修复代码就是正常代码，天天被并发场景锤炼，没有测试盲区；
- 为什么修复者限定为写入者、而不让 VACUUM 或恢复流程做？`_bt_finish_split` 头注释（:2268-2269）点明："finishing a page split requires allocating a new page if and when the parent page splits in turn"——补 downlink 可能级联分裂、需要新磁盘空间。**磁盘满的时候 VACUUM 必须还能跑**（它是释放空间的唯一希望），所以 VACUUM 绝不能承担"可能需要分配空间"的义务；恢复流程同理。写入者本来就在申请空间，多申请一点无妨；
- 崩溃窗口分析：`_bt_split` 的 WAL 落盘后、`_bt_insertonpg`（父层）的 WAL 落盘前崩溃 → 恢复重放出一个带 INCOMPLETE_SPLIT 标志的左页 → 树立即可用，读写照常，直到某个写者路过顺手补完。两条 WAL 各自原子，中间态自描述、自恢复。

一句话：**方案 A 用锁换原子性，方案 B 用恢复复杂度换原子性，PG 用"中间态合法化 + 惰性修复"把原子性需求本身消解了**。这与 L&Y 处理读者竞态的哲学（2.2 节）完全同构——它是写者版的 right-link。

**与 InnoDB 对比**：InnoDB 用 mini-transaction（mtr）把"分裂 + 插父（可级联）"整体作为一个多页原子单元，持有 index latch（或受影响子树的 latch）直到 mtr 提交，redo log 里这些页改动作为一组原子应用。这正是"反面方案 A"的工程化实现——InnoDB 承担了路径级 latch 与 SMO 期间的并发抑制，换来"树里永远没有中间态"的简单不变式；PG 反之。两个成熟系统在同一岔路口选了相反方向，是理解两者并发模型差异的最佳标本。

**小结**：`_bt_split` 是"多页原子修改"的模板：临时缓冲组装 → 失败零副作用 → 从左到右锁兄弟 → 临界区内 memcpy + 多页注册 + 单条 WAL → 只留必要的锁。而"分裂"与"插父"之间刻意**不**原子：INCOMPLETE_SPLIT 把中间态变成合法状态，三处写路径触发点 + `_bt_finish_split` 复用正常代码惰性修复。

## 六、B-tree（五）：页删除、suffix truncation 与 deduplication

### 6.1 页删除：比分裂难得多（nbtpage.c）

分裂是"无中生有加一页"，删除是"从并发读者眼皮底下抽走一页"，难度不对称。PG 的约束（README "Deleting entire pages during VACUUM"）：只删**完全变空**的页；从不合并半满页；从不删除任何一层的最右页。

删除入口 `_bt_pagedel`（nbtpage.c:1812），同样是**两阶段**，每阶段一条 WAL：

1. **阶段一 `_bt_mark_page_halfdead`**（nbtpage.c:2102）：把父页中指向目标页的 downlink 删掉、让键区间顺延给右兄弟，同时给目标叶页打上 **half-dead** 标志。此刻页从父层"消失"了，但同层左右兄弟链还连着它——与分裂的中间态（"兄弟链有、父层无"）镜像对称。搜索遇到 half-dead 页按 `P_IGNORE` 向右走即可。
2. **阶段二 `_bt_unlink_halfdead_page`**（nbtpage.c:2329）：按 左邻 → 自己 → 右邻 的顺序加锁（仍是从左到右防死锁），改双向链摘除该页，标成 fully dead（`BTPageSetDeleted`，nbtree.h:240，写入 safexid）。

若两阶段之间崩溃，half-dead 页留在树里，对读写无害，**下一次 VACUUM** 撞见时继续完成阶段二——又一个"中间态合法化 + 惰性修复"。删除"某父页最后一个孩子"时更复杂：要把整条"瘦子树"（每层单页的链）从顶部开始摘，README "Deleting entire pages during VACUUM" 有完整推演。

### 6.2 Suffix truncation（PG 12）：让树更矮

叶页分裂时，新 high key（随后成为父层 downlink 的键）只需**足以区分左右两页**，多余的后缀键列可以整列截掉（`_bt_truncate`，nbtutils.c:692；理论出处 Bayer & Unterauer 的 Prefix B-Trees）。被截掉的列语义上是 **-∞**。

**量化例子**：复合索引 `(order_date date, order_id bigint, line_no int)`，行宽约 32 字节（8+8+4+头部对齐）。若分裂点两侧的元组 date 不同（比如 lastleft 是 `('2026-07-17', 900001, 3)`、firstright 是 `('2026-07-18', 900002, 1)`），pivot 只需保留 `('2026-07-18', -∞, -∞)`——date 一列就足够分界。内部页元组从 32 字节缩到 ~16 字节（还省了 TID tiebreaker 列），**扇出翻倍**；对一个 10 亿行的索引，内部页扇出从 ~250 提到 ~500 意味着树高从 5 层降到 4 层——每次点查少一次页访问（4~8KB 的 I/O 或一次缓冲池命中）。此外 pivot 不再"过度描述"边界：`_bt_findsplitloc` 会在均衡点附近专门挑"能截得最多"的分割点，让键空间的划分落在自然边界（date 变化处）而不是任意行上，后续同 date 的插入不会跨页乱窜。INCLUDE 索引的非键列正是靠同一机制在分裂时被必然截掉（4b42a89938 起 assert 保证）。

### 6.3 Deduplication（PG 13，nbtdedup.c）：让重复键不再膨胀

`_bt_dedup_pass`（nbtdedup.c:59）把页内键值相同的多条索引元组合并为一条 **posting list 元组**（键 + 排序的 TID 数组，与 GIN 的 posting list 同构但不压缩）。

**量化例子**：`status text` 列上的索引，值域只有 5 种。1000 万行、每行索引元组 ~24 字节（键 + TID + 头部），朴素存储 240MB、约 3 万个叶页，且每个键的百万级重复项各占几千页。dedup 后一条 posting list 元组摊到每个 TID 只有 6 字节（TID 本体）+ 键与头部被整页摊薄，索引缩到 ~70MB。更重要的隐性收益：**唯一索引里的"重复"全是 MVCC 版本垃圾**（同键不同版本的堆行各有一条索引项），dedup 把"版本膨胀 → 页分裂"的必然链条切断——页分裂是永久性结构损伤（分裂出的页不会自动合并回来），而 posting list 是可逆的临时压缩，给 VACUUM / bottom-up deletion 争取时间窗口。所以**唯一索引也默认做 dedup**（README "Deduplication in unique indexes"）。

设计要点：

- **完全惰性**：只在"页要分裂了、simple deletion 和 bottom-up deletion 都没救回来"这一最后关头才做（4.5 节的三条退路顺序），平时零维护成本；
- 新插入的 TID 落在既有 posting list 区间内时做 **posting list split**：拆开旧 posting、挤出一个 TID 与新元组一起重排，与页内插入同临界区同 WAL（4.6 节）；
- 不做 GIN 式 varbyte 压缩：会破坏"页空间核算必须精确可预算"的简单性（分裂点选择、1/3 页上限检查都依赖元组大小可直接计算），且删除单个 TID 要重解码整条链。

### 6.4 what-if 推演 ③：VACUUM 删页时，飞行中的并发扫描为何不迷路

页删除最险的窗口：扫描者 S 已从某处读到指向页 P 的指针（downlink 或兄弟链），尚未到达；VACUUM 正在删 P。分三层防线：

1. **删除进行中（half-dead / deleted 标志已打）**：S 到达 P，`P_IGNORE` 命中，沿 right-link 右移（3.1 节）。P 的兄弟链在 unlink 前保持完好，unlink 的三页原子修改保证 S 要么看到"P 还在链上"、要么看到"左右已互连"，无第三种状态；
2. **删除已完成但页还没被复用**：deleted 页仍保留 `btpo_next`，右移仍然可达正确页；
3. **页被回收复用**——这是真正的危险：P 变成了树中完全无关位置的新页，S 按旧指针到达后 high key 判断彻底失效，扫描会**静默返回错误结果**。防线是**延迟回收**：`_bt_unlink_halfdead_page` 把页标记 deleted 时写入 **safexid**（当时的 `nextXid`，全量 64 位 FullTransactionId，nbtree.h:236）；页能否进 FSM 复用由 `BTPageIsRecyclable`（nbtree.h:292）判定：

   ```c
       return GlobalVisCheckRemovableFullXid(heaprel, safexid);
   ```

   语义：**当 safexid 对所有活跃快照都"已过去"时**，意味着"删页时刻之前开始的所有扫描都已结束"——不可能再有谁持有指向 P 的旧指针，回收安全。这与堆元组"死亡 → 越过 xmin 视界 → 回收"是同一个两阶段模式（Lanin & Shasha 称为 drain technique）。README "Placing deleted pages in the FSM" 一节还讨论了残余风险：FSM 本身不是崩溃安全的，极端情形下一个未到回收点的页可能被交出——所以取页方 `_bt_allocbuf`（nbtpage.c:854）拿到 FSM 给的页后还要**再验一次** deleted 状态与 safexid。
4. 另一侧的对称问题——**索引扫描暂停期间 TID 被 VACUUM 回收再复用**：扫描停在某叶页（只留 pin 或连 pin 都不留）期间，VACUUM 可能删掉它已读出的 TID 对应的索引项并回收堆行。防线是 `_bt_killitems`（nbtutils.c:191）回标 LP_DEAD 前用**页 LSN 是否变化**判断"离开期间页被改过没有"（nbtsearch.c:67-70：`so->currPos.lsn = BufferGetLSNAtomic(...)`），变过就放弃回标——宁可少标不可错标。

### 6.5 页内空间的四级回收阶梯

把 4.5 与本节合起来，nbtree 对"空间不够"的响应是一个精心排序的阶梯，每级成本递增：

| 级别 | 动作 | 触发 | 成本 |
|---|---|---|---|
| 1 | simple deletion：删 LP_DEAD 项 | 页满且存在 LP_DEAD | 页内操作 + 一条 WAL |
| 2 | bottom-up deletion：回表探测疑似死项 | indexUnchanged 提示 / 唯一索引重复 | 少量堆页访问 |
| 3 | deduplication：同键压 posting list | allequalimage 且 dedup 未禁用 | 页内重写 |
| 4 | page split：结构扩张 | 前三级都失败 | 三页修改 + 两条 WAL + 永久占用新页 |
| （5） | VACUUM 空页 → half-dead → deleted → safexid 后进 FSM | 后台 | 全局扫描 |

**小结**：删除侧的一切设计都在回答同一个问题——**如何在不阻塞读者的前提下收缩结构**。答案与分裂侧完全对称：中间态合法化（half-dead）、惰性收尾（下次 VACUUM）、墓碑延迟回收（safexid 视界）。suffix truncation 与 deduplication 则从源头减缓结构增长：前者压内部页（树高），后者压叶页（版本膨胀）。

## 七、B-tree 不变式清单与 amcheck

把散落全篇的不变式集中成表。检测工具是 contrib/amcheck（`verify_nbtree.c`）：`bt_index_check`（verify_nbtree.c:252，只拿 AccessShareLock，逐页独立检查）与 `bt_index_parent_check`（verify_nbtree.c:284，拿 ShareLock 排除并发写，能做跨层检查）。

| # | 不变式 | 精确表述 | 违反后果 | 检测手段 |
|---|---|---|---|---|
| 1 | 页内有序 | 每页内元组按（键, TID）严格升序 | 二分定位错位：查不到存在的键、插入位置错乱、唯一约束漏判 | `bt_target_page_check`（verify_nbtree.c:1241）逐对相邻元组比较；报错 "item order invariant violated" |
| 2 | high key 上界 | 页内所有元组 ≤ 本页 high key（最右页除外）；叶页 high key 可与末项相等，内部页严格 | `_bt_moveright` 该右移时不右移（漏结果）或不该右移时右移（死循环/漏结果） | 同上：每个元组与 high key 比较（invariant_leq_offset，verify_nbtree.c:3171 一族） |
| 3 | 跨页有序 / 兄弟链一致 | 左页所有键 ≤ 左页 high key < 右页第一个键；left-link/right-link 互逆 | 正向/反向扫描漏行或重复行；分裂时 "right sibling's left-link doesn't match" ERROR（nbtinsert.c:1917-1925 的运行时防御） | `bt_right_page_check_scankey`（verify_nbtree.c:1867）跨页比较；`bt_recheck_sibling_links`（verify_nbtree.c:1101） |
| 4 | 层间分隔键一致（接缝） | 父页中划定子页键区间的 pivot 与子页 high key 逐字节一致；downlink 指向的子页确实覆盖对应区间 | 搜索下降进错子树——键"存在但找不到"；插入落错页后连锁破坏 #1 | `bt_child_check`（verify_nbtree.c:2394，需 parent_check）：对每个 downlink 下潜验证子页边界 |
| 5 | right-link 可达性 / downlink 完备性 | 同层从最左页沿 right-link 可遍历全层；每个页要么有父层 downlink，要么其左兄弟带 INCOMPLETE_SPLIT 标志，要么是 half-dead 的目标 | 孤儿页：数据在树里但任何下降都到不了（表现为索引比表"少行"） | `bt_downlink_missing_check`（verify_nbtree.c:2559）；全层遍历对账 |
| 6 | 删除页回收门槛 | deleted 页在 safexid 越过全局可见性视界前不得复用 | 飞行中的扫描落在被复用的页上，静默错误结果（6.4 节） | 难以事后检测——靠 `BTPageIsRecyclable`（nbtree.h:292）+ `_bt_allocbuf` 复验的双保险 |
| 7 | 键与堆一致 | 每条堆行在索引中恰有一条对应项（heapallindexed） | 索引漏行：走索引的查询比顺序扫描少结果 | `bt_index_check(..., heapallindexed => true)`：全索引装进布隆过滤器，再全表扫描逐行探测 |

使用建议：例行体检用 `bt_index_check`（不挡业务写）；怀疑损坏时升级 `bt_index_parent_check(index, heapallindexed => true, rootdescend => true)`（rootdescend 对每条叶元组从根重新下降验证可达性，最彻底也最贵）。生产中索引损坏的常见外因：glibc/ICU collation 版本变化导致文本比较规则漂移（等价于全体键换了序），这会同时违反 #1-#4 而页面本身无任何字节损坏——amcheck 是唯一能发现它的工具。

## 九、所以然 ④：为什么所有二级索引指向 TID 而不是主键

InnoDB 二级索引的叶子存主键值，回表 = 再走一次主键 B+ 树；PG 所有索引的叶子存 TID，回表 = 直接按物理地址读堆页。这不是随意的选择，而是一个**三角权衡**——回表成本、索引写放大、更新时的索引维护成本——三者不可兼得：

**回表成本**：PG 的 TID 回表是 O(1) 物理寻址（一次缓冲池查找）；InnoDB 是 O(log N) 的主键树下降（通常 3~4 层，上层大多命中缓存但仍有比较与 latch 成本）。且 InnoDB 二级索引项宽度随主键膨胀——UUID 主键（16B）意味着每个二级索引项都背着 16B 尾巴；PG 的 TID 恒定 6 字节。这一局 PG 胜。

**更新时的索引维护（写放大的一面）**：代价在这里反转。堆表更新 = 新版本可能落在新物理位置 = **TID 变了** = **所有**索引（哪怕被更新的列与该索引无关）都要插入指向新 TID 的新项；InnoDB 更新非索引列时，行在主键树里**原地**改（配合 undo 构造旧版本），二级索引完全不动——因为二级索引指向的主键值没变。这一局 InnoDB 胜，而且是大胜：这正是 Uber 2016 年那篇著名的迁移博文抱怨的"PG 写放大"。

**PG 的补偿——HOT（Heap-Only Tuple）**：若更新没改任何索引列、且新版本能塞进同一个堆页，则新版本挂在页内 HOT 链上，**不生成任何索引项**——索引仍指向链头 TID，回表时沿链找可见版本（tableam 的 `index_fetch_tuple` 实现，第十节）。HOT 命中时 PG 的更新成本反超 InnoDB（连主键树都不用动，InnoDB 好歹要写 undo + 改聚簇页）；HOT 不命中（页满或改了索引列）时退回全索引写放大。于是 `fillfactor`、"避免索引热更新列"成为 PG 特有的调优项。PG 14 的 bottom-up deletion（第八节）进一步兜底 HOT 不命中时的索引膨胀。

**第三个角——可见性与空间回收**：InnoDB 的旧版本在 undo 段，主表与二级索引"干净"，但长事务会让 undo 堆积、且二级索引判可见性要靠回表+undo 重建（它的二级索引页只有一个 PAGE_MAX_TRX_ID 做粗判）；PG 的旧版本以完整元组躺在堆里 + 每个非 HOT 更新都在索引里留一条旧项，换来**读路径永不重建旧版本**（版本就躺在那里），代价是 VACUUM 必须打扫堆和所有索引。**索引指向 TID 还是主键，本质是"版本链藏在哪"的下游决策**：undo 型存储天然适合"索引指向逻辑键"（版本移动只发生在主表），堆型存储天然适合"索引指向物理地址"（否则每次回表都要多走一层间接）。

还有两个常被忽略的连带收益/损失：

- PG 免疫"主键树倾斜"问题：InnoDB 随机主键（UUID）导致聚簇树本身随机分裂、二级索引回表也随机化；PG 的堆是追加友好的，任何主键形态都不影响堆布局；
- PG 的 CREATE INDEX 不依赖其他索引存在（各索引独立指向堆），可以并行构建互不干扰；InnoDB 二级索引逻辑上依附主键。

**结论**：PG 选择 TID 指向 = 押注"读多写少 + HOT 能吃掉大部分更新"；InnoDB 选择主键指向 = 押注"主键点查为王 + 非索引列更新频繁"。两者各自的名场面（PG 的 Index Only Scan 需要 VM 兜底、InnoDB 的 UUID 主键灾难）都是同一枚硬币的两面。

## 十、GIN：倒排索引与 LSM 直觉（access/gin/）

做过搜索引擎/大数据的读者最熟悉这个结构。GIN（Generalized Inverted Index）存 `(key, posting list)`：key 是被索引项的**元素**（数组元素、tsvector 词素、jsonb 路径），posting list 是包含该元素的堆行 TID 集合（`src/backend/access/gin/README` 开头即讲）。一行文档产出多个 key——这是它与 btree "一行一键"的本质差别。

### 10.1 两层树结构

- **Entry tree**：对 key 建的一棵 B-tree（复用类似 nbtree 的并发结构，代码 ginbtree.c/ginentrypage.c）。叶元组两种形态：
  - key 的 TID 少 → 直接在叶元组里内联一个 **posting list**（varbyte 差分压缩，ginpostinglist.c）；
  - TID 多到放不下 → 叶元组存一个指针（`GinGetPostingTree`，ginblock.h:234），指向一棵独立的 **posting tree**——按 TID 排序的 B-tree（gindatapage.c，`createPostingTree` 在 gindatapage.c:1775）。
- 一次 `tsvector @@ query` 查询 = 对每个查询词在 entry tree 定位 → 遍历其 posting list/tree → 多个 TID 集合按查询逻辑做交并（`gingetbitmap`，ginget.c:1928）。GIN 只提供 amgetbitmap，结果必然走 Bitmap Heap Scan。
- Entry tree **没有删除操作**：大语料的词表几乎只增不减，砍掉 entry 删除让并发协议大幅简化——死 TID 由 VACUUM 从 posting 中清除，空 entry 留着无害。

### 10.2 ginHeapTupleFastInsert 走读：GIN 的 memtable

### 10.3 ginInsertCleanup 走读：compaction

`ginInsertCleanup`（ginfast.c:780-1025）把 pending list 批量归并进主结构。主线：

**并发互斥**（:807-828）：

**主循环**（:877-1009）：沿 pending 链逐页把 `(key, TID)` 读进内存累加器（`processPendingPage` → BuildAccumulator，按 key 聚合 TID），直到链尾或"攒够一批"（:902-904：`GinPageHasFullRow(page) && accum.allocatedMemory >= workMemory * 1024`）就 flush：

要点：

- **`ginEntryInsert` 是批量红利的来源**：同一 key 的成百上千个 TID 合并成一次 entry tree 定位 + posting 批量插入——这正是 commit ff301d6e690 说的 "bulk insert is significantly faster than retail insert"；
- **插树期间不持任何 pending 页锁**（:918-933）：树插入可能级联分裂、耗时任意长。放锁期间尾页可能又被追加，flush 完回来用 `maxoff` 对比检测增量，补处理（:951-961）；
- **崩溃安全性靠幂等**（函数头注释 :768-771 原文）："if we crash after posting entries to the main index and before removing them from the pending list, it's okay because when we redo the posting later on, nothing bad will happen"——重复把同一 `(key, TID)` 插进 posting 是无害的（集合语义去重）。所以"插主树"与"摘 pending 页"（`shiftList`）不需要跨 WAL 原子性。又一次，**用幂等性代替原子性**；
- 内层的 `vacuum_delay_point` 说明清理受 VACUUM 成本延迟节流控制；
- 尾部（:1019-1020）普通后端清完立即 `IndexFreeSpaceMapVacuum` 归还页——pending 页周转极快，攒着不还会让索引虚胖。

### 10.4 LSM 类比与代价

这就是 **LSM 树 memtable + compaction 的直觉**：写先进无序小结构，后台批量合并进有序大结构，把 N 次随机写摊销成排序后的批量写。对应与不对应之处：

- 对应：追加式写缓冲 / 批量合并 / 查询需要额外查小结构（每个 GIN 查询都要**顺序全扫 pending list**，故 `gin_pending_list_limit` 不能开大）；
- 不对应：pending list 是磁盘页链而非内存表（崩溃不丢，不需要 WAL 重放重建）；合并是就地插一棵树而非生成新 SSTable 文件层；没有多层 level 与层间归并策略。
- 运维口诀由此自然导出：批量装载时 `fastupdate=on` + 大 limit + 事后 `gin_clean_pending_list()`；查询延迟敏感的在线服务常干脆 `fastupdate=off`，把合并成本摊回每次插入。

**小结**：GIN = "key 的 B-tree"套"TID 的 B-tree"，加一截作为写缓冲的 pending list。写路径的三个层次——追加（快速插入）、聚合（BuildAccumulator）、归并（ginEntryInsert）——用"幂等重放"替代跨结构原子性，是 nbtree"中间态合法化"之外另一种崩溃一致性风格。

## 十一、GiST：把"树"做成框架（access/gist/）

GiST（Generalized Search Tree，Hellerstein et al. 1995）回答的问题是：B-tree 里哪些逻辑是"树的骨架"、哪些是"键的语义"？答案：平衡树维护、页分裂、并发（GiST 用 NSN + rightlink，思路借自 L&Y——NSN 是分裂序列号，读者比较页 NSN 与自己出发时的全局计数器来检测"路过期间发生过分裂"，等价于 nbtree 的 high key 判据）是骨架；而"什么样的键该下降到哪个子树"是语义。GiST 把语义抽成 opclass 的支持函数（编号见 `src/include/access/gist.h:32-37`）：

```c
#define GIST_CONSISTENT_PROC            1
#define GIST_UNION_PROC                 2
#define GIST_COMPRESS_PROC              3
...
#define GIST_PENALTY_PROC               5
#define GIST_PICKSPLIT_PROC             6
```

四个核心回调的意义（内部页的每个 entry 是一个"谓词"，为其子树内所有键的**并集/上界**）：

| 回调 | 搜索/插入中的角色 | R-tree 实例（矩形） |
|---|---|---|
| `consistent(entry, query)` | **搜索下降判据**：该子树是否可能含匹配项？false 才能剪枝，允许假阳性（返回 recheck） | 子树包围盒与查询矩形相交吗？ |
| `union(entries)` | 生成/更新父 entry：子集合的上界 | 多个矩形的最小包围盒 |
| `penalty(entry, new)` | **插入下降判据**：新键插进该子树的"代价"，选 penalty 最小的孩子 | 包围盒面积增量 |
| `picksplit(entries)` | 页满时把 entry 分成两组 | 让两个新包围盒面积和最小（Guttman 二次分裂等） |

对照源码：搜索走 `gistindex_keytest`（gistget.c:134），对每个 scan key 调 consistent（gistget.c:230-236 的 `FunctionCall5Coll`，第 5 个参数回传 recheck 标志——GiST 允许有损判定，靠回表 recheck 兜底）；插入走 `gistdoinsert`（gist.c:639），逐层用 `gistchoose`（gistutil.c:374，内部调 penaltyFn）选子树，页满时 `gistSplitByKey`（gistsplit.c:623）调 picksplit，再用 unionFn 重算父 entry。

与 B-tree 的根本差异，以及由此而来的能力边界：

- GiST 不要求键有全序，**子树谓词可以互相重叠**——所以一次搜索可能要下降多个分支（搜索是"栈上多路 DFS"而不是单路径下降）；也因此它能索引"相交""包含""距离"这类 B-tree 无能为力的谓词；
- 插入没有唯一正确位置（penalty 只是启发式），同一数据集不同插入顺序长出不同的树——树质量依赖 picksplit/penalty 的好坏，这是 GiST 索引"重建后性能变好"现象的根源；
- `amcanorderbyop=true` 让它支持 KNN（`ORDER BY point <-> ?`）：用 distance 支持函数做最佳优先遍历（优先队列按下界排序），R-tree 变身近邻搜索结构。

**R-tree 是 GiST 的一个特例**：把上表右列的四个函数填进去就是 R-tree（PG 内建的 `box_ops` 等即是，gistproc.c）。同理，填"区间包含"得到 range 类型索引，填"集合签名"得到 RD-tree。PostGIS 的空间索引就是最著名的一份 GiST 答卷。

**小结**：GiST 用 consistent/union/penalty/picksplit 四个回调把"平衡树"与"键语义"解耦；并发骨架（NSN + rightlink）是 L&Y 思想的移植。B-tree 式全序索引之外的几乎一切树形索引都成了它的填空题。

## 十二、BRIN：块范围摘要（access/brin/）

BRIN（Block Range Index）根本不是"树"，也不存任何 TID。它把表按物理顺序切成**块范围**（默认 128 个堆页一段，`BRIN_DEFAULT_PAGES_PER_RANGE`，`src/include/access/brin.h:40`），每段只存一条**摘要元组**——minmax opclass 下就是该段内此列的 min/max（还有 bloom、minmax-multi 等 opclass，brin_bloom.c / brin_minmax_multi.c）。

**大数据类比**：这就是 Parquet/ORC footer 里 row group 级的 min/max 统计、Snowflake 的 zone map——**用于跳过数据（data skipping），而非定位数据**。查询 `WHERE ts BETWEEN a AND b` 时，`bringetbitmap`（brin.c:572）遍历所有摘要元组，把"区间与 [min,max] 相交"的整段堆页**全部**加入位图（有损，页内逐行 recheck），不相交的段整段跳过。所以：

- BRIN 只在**物理顺序与逻辑顺序强相关**的列上有效（append-only 时序表的时间戳列近乎完美；被随机更新打乱后急剧退化，可用 `pg_stats.correlation` 判断）；
- 换来的是近乎可忽略的体积：10GB 表的 BRIN 常常只有几十 KB，而 btree 会有数百 MB。

**revmap 机制**：摘要元组存在普通索引页里，位置会变（摘要变大时可能搬家），于是需要一层稳定的定位结构——**revmap（range map）**，占据索引文件开头的固定页链，按块范围号数组式地存"该范围的摘要元组现在在哪"的 ItemPointer。代码在 brin_revmap.c：`brinGetTupleForHeapBlock`（brin_revmap.c:194）由堆块号算出 revmap 页与槽位、取出指针再读摘要；`brinSetHeapBlockItemptr`（brin_revmap.c:155）在摘要搬家/新建时回填；`brinRevmapExtend`（brin_revmap.c:112）随表增长扩展。插入新行时 `brininsert`（brin.c:349）查 revmap 找到所属范围的摘要，若新值越界则放宽 min/max——**BRIN 的更新只放宽、不收紧**，收紧要靠重新 summarize（VACUUM 或 `brin_summarize_new_values`）。

**小结**：BRIN 用"每 128 页一条 min/max"把索引做成了 zone map，靠 revmap 间接层解决摘要元组的寻址。它是 PG 里最"大数据"的索引：不回答"行在哪"，只回答"哪些整段肯定没有"。

## 十三、Hash 与 SP-GiST 简述

**Hash**（access/hash/）：桶号 = hash 值的低位，桶满挂 overflow 页链；扩容用**渐进式桶分裂**——每次只分裂一个桶（`src/backend/access/hash/README` 开头：源自 Seltzer & Yigit 1991 的线性哈希族方案），不像内存哈希表那样整体 rehash。搜索入口 `_hash_first`（hashsearch.c:289）。只支持 `=`（amstrategies 只有等值一种），不支持排序、范围、唯一约束；PG 10 之前甚至不写 WAL（崩溃后必须重建），名声不佳。如今合理场景很窄：超长键（如 URL）的等值查询——索引项只存 4 字节 hash 值，比 btree 小。**与 InnoDB 自适应哈希（AHI）对比**：AHI 是 B+ 树之上自动构建的内存缓存，PG 的 hash 是持久化磁盘索引、需显式创建，两者除了名字几乎没有共同点——PG 没有 AHI 的对应物。

**SP-GiST**（access/spgist/）：Space-Partitioning GiST——GiST 的兄弟框架，但支持的是**非平衡的空间划分树**：四叉树（spgquadtreeproc.c）、kd 树（spgkdtreeproc.c）、基数树/trie（spgtextproc.c）。与 GiST 的区别：GiST 的子树谓词可重叠、树保持平衡；SP-GiST 的划分**不相交**、树可以很深且不平衡。磁盘页与树节点不再一一对应（很多小节点打包进一页），插入主逻辑 `spgdoinsert`（spgdoinsert.c:1909）是全内核最烧脑的函数之一。典型用途：文本前缀（trie）、点数据（四叉树）、IP 网段（inet_ops）。

**小结**：hash 是"只做等值"的窄工具；SP-GiST 把"不相交划分、不平衡深树"这一族结构也纳入框架化版图。两者共同说明：Index AM 接口的表达力足以容纳形态迥异的结构。

## 十四、索引与堆的协作：三种扫描形态

PG 的索引一律不存行数据，且**索引不含可见性信息**——索引项没有 xmin/xmax，可见性必须问堆。这是与 InnoDB 最重要的运维级差异，由此派生三种扫描形态。

### 14.1 IndexScan：逐个回表

执行器的 IndexScan 节点循环调用 `index_getnext_slot`（indexam.c:698）= `index_getnext_tid`（indexam.c:599，进入 AM 的 amgettuple 拿下一个 TID）+ `index_fetch_heap`（indexam.c:657）：

```c
bool
index_fetch_heap(IndexScanDesc scan, TupleTableSlot *slot)
{
    bool        all_dead = false;
    bool        found;

    found = table_index_fetch_tuple(scan->xs_heapfetch, &scan->xs_heaptid,
                                    scan->xs_snapshot, slot,
                                    &scan->xs_heap_continue, &all_dead);

    if (found)
        pgstat_count_heap_fetch(scan->indexRelation);

    /*
     * If we scanned a whole HOT chain and found only dead tuples, tell index
     * AM to kill its entry for that TID (this will take effect in the next
     * amgettuple call, in index_getnext_tid).  We do not do this when in
     * recovery because it may violate MVCC to do so.  See comments in
     * RelationGetIndexScan().
     */
    if (!scan->xactStartedInRecovery)
        scan->kill_prior_tuple = all_dead;

    return found;
}
```

- 回表走 Table AM 的 `index_fetch_tuple` 回调（heap 实现 `heapam_index_fetch_tuple`），在堆页内沿 **HOT 链**找对本快照可见的版本；
- **kill_prior_tuple 反馈回路**：回表发现整条链对所有人都死了，就反手让索引 AM 把该项标 LP_DEAD（btree 侧落在 `_bt_killitems`，nbtutils.c:191，带 6.4 节说的 LSN 防御）——下次扫描直接跳过，页满时被 simple deletion 收割（4.5 节退路 1）。索引与堆之间的"垃圾情报"就靠这条通道流动；
- 随机回表是 IndexScan 的成本大头，这正是 InnoDB 聚簇索引免掉主键查询回表、而 PG 必须靠下面两种形态补偿的地方。

```c
        }
```

TID 所在堆页 all-visible → 直接用索引元组当结果，零回表；否则退化为普通回表。EXPLAIN 里的 `Heap Fetches` 就是退化次数——**长期不 VACUUM 的表，Index Only Scan 会退化成比 IndexScan 还贵的东西**。读 VM 不加锁的正确性论证见该文件 :136-163 的大段注释（写侧先清 VM 位再改堆，配合锁的内存屏障语义）。

### 14.3 BitmapScan：两阶段与 recheck

1. **建位图**：BitmapIndexScan 节点调 AM 的 `amgetbitmap`，把所有匹配 TID 一次性倒进 TIDBitmap；多个索引的位图可做 BitmapAnd/BitmapOr。内存不够时位图自动**退化为有损（lossy）**：不再记"哪些行"，只记"哪些页"；
2. **按物理顺序扫堆**：BitmapHeapScan 节点（nodeBitmapHeapscan.c）把位图页号排序后顺序访问堆——把随机 I/O 变成准顺序 I/O，这是它对大选择率区间查询优于 IndexScan 的根源。对有损页和 AM 自报有损的项（recheck 标志），取出的每行要**重算原始谓词**。

"precheck + recheck"的两阶段语义贯穿多个 AM：GIN/BRIN/GiST 的判定都允许假阳性（宁滥勿缺），由堆侧 recheck 精确化——正因为有这个兜底，前面各种"有损但紧凑"的索引设计（BRIN 的 min/max、GiST 的签名、位图的 lossy 页）才得以成立。

**小结**：三种形态是"回表成本"这同一约束的三种解法：IndexScan 逐行付费，IndexOnlyScan 靠 VACUUM 预付（VM），BitmapScan 批量排序后付。索引无可见性信息是 PG 与 InnoDB 分岔的总根源（第九节的三角权衡）：它换来索引项的紧凑与 HOT 的可能，代价是 VACUUM 成为索引性能的隐形参与者。

## 自测问题（附详解）

**1. 教科书 lock coupling 为什么在高并发下不可接受？L&Y 用什么替换了它，代价是什么？**

lock coupling 要求任何时刻持父子两把锁下降，根页成为所有路径共享的锁热点，写者最坏持有整条路径的排他锁，且可能持锁等 I/O。L&Y 给每页加 right-link 与 high key，并规定分裂只向右：于是"downlink 过期"可以在到达子页后用一次 high key 比较检测、沿 right-link 追赶恢复（`_bt_moveright`，nbtsearch.c:242），层间零锁、读全程单锁。代价：每页多存一个 high key；撞上并发分裂的读者要多走 right-link；以及树中允许"downlink 暂缺"等中间态，代码必须处处容忍它们（P_IGNORE、INCOMPLETE_SPLIT 检查）。

**2. 写下降与读下降在 `_bt_search`/`_bt_moveright` 里有哪几处行为差异？各自的原因是什么？**

四处：(a) 写下降到 level 1 时提前把叶层锁模式切为 BT_WRITE（nbtsearch.c:179-180），省一次"读锁-放-写锁-再 moveright"的往返；(b) `_bt_moveright` 只在 forupdate 时处理 INCOMPLETE_SPLIT（:286-305）——写者有义务在插入前完成前人的分裂（插入不允许发生在半分裂页上，且修复可能需要分配页，读者不该背这个成本）；(c) 写下降必须传 heaprel（页分配/回收要查堆的可见性视界）；(d) 写者用 `_bt_search_insert` 可走最右页快速路径，读者永远全程下降。

**3. B-tree 页分裂为什么写两条 WAL？崩溃在两条之间会怎样？为什么修复者是"下一个写入者"而不是 VACUUM 或恢复流程？**

`_bt_split`（nbtinsert.c:1489）用第一条 WAL 原子完成本层最多四页的修改（左页覆写、新右页整页、老右兄弟 left-link、可能的下层子页清标志），并给左页打上 BTP_INCOMPLETE_SPLIT（:1582）；`_bt_insert_parent`（nbtinsert.c:2130）经 `_bt_insertonpg` 用第二条 WAL 往父页插 downlink 并同时清标志（:1322-1330）。打包成一条不可行：插父可能级联分裂到根，WAL 记录尺寸与持锁范围都无上界（等于请回路径锁）。之间崩溃则树处于合法中间态：右页经左页 right-link 可达，读者无感；下一个路过的写入者在 `_bt_moveright`/`_bt_stepright`/`_bt_getstackbuf` 三处触发点之一调用 `_bt_finish_split`（nbtinsert.c:2272）补完。不让 VACUUM/恢复流程修，是因为补 downlink 可能引发新分裂、需要分配磁盘空间——磁盘满时 VACUUM 必须还能跑、恢复不允许失败（commit 40dae7ec537 的设计动机）。

**4. `_bt_check_unique` 为什么必须用 DirtySnapshot 而不是事务自己的 MVCC 快照？为什么检查前要把 scantid 置 NULL？**

用事务快照时，两个并发事务互相看不见对方未提交的同键插入，会双双通过检查、提交后唯一约束被击穿。DirtySnapshot 把"进行中"的元组也视为存在，并回传其 xmin/xmax 供等待（nbtinsert.c:589-601）。scantid 置 NULL（nbtinsert.c:123）是因为完整插入键含 TID 尾键，带着它搜索会定位到"该键+本 TID"的精确位置；而唯一检查需要从"该键的第一个可能位置"开始扫过**所有**同键项——抹掉 TID 才能落到同键区的最左端。唯一性确立后再恢复 scantid（:239-240）用于精确定位插入点。

**5. 两个事务同时 INSERT 相同唯一键，从锁的角度完整描述谁先谁后、等待与醒来后的行为。**

互斥量是"该键（不含 TID）可能落在的第一个叶页"的写锁（nbtinsert.c:190-199 注释）。先到者 A 检查无冲突后连续持有写锁直到插入完成；后到者 B 卡在页锁上，拿到锁后扫到 A 的新项，DirtySnapshot 回表发现 xmin 未决，得到 xwait=A，**放掉页锁**后 `XactLockTableWait` 睡在 A 的事务锁上（持锁等待会阻塞无关插入并可能死锁）。A 提交 → B 重来（goto search）后报 duplicate key；A 回滚 → B 重来后发现该项已死（顺手标 LP_DEAD）并成功插入。ON CONFLICT 的投机插入用 speculativeToken 只等"单行裁决"避免互等死锁。

**6. 什么是 suffix truncation 与 deduplication？各解决什么维度的膨胀？为什么 dedup 对唯一索引也有意义？**

suffix truncation（PG 12，`_bt_truncate`，nbtutils.c:692）：叶分裂时新 high key 只保留足以分隔左右页的键前缀，其余列记为 -∞——pivot 变小、内部页扇出变大、树更矮，压的是**内部页/树高**维度。deduplication（PG 13，`_bt_dedup_pass`，nbtdedup.c:59）：页将分裂时把同键元组合并为 posting list，压的是**叶页/重复键**维度。唯一索引的重复项全是 MVCC 版本垃圾：为版本垃圾分裂页是用永久结构损伤换临时空间，dedup 把它压成可逆的 posting list，给 VACUUM/bottom-up deletion 争取时间——所以唯一索引默认启用 dedup。

**7. VACUUM 删除了一个空叶页。为什么这个页不能立即被复用？safexid 门槛的语义是什么？**

可能仍有"飞行中"的扫描早已读到指向该页的指针（downlink 或兄弟链）。页保持 deleted 状态时这些扫描靠 P_IGNORE + right-link 自愈；但页一旦被复用成别处的新页，旧指针到达后的 high key 判断完全失效，产生静默错误结果。因此 `_bt_unlink_halfdead_page` 在页上记录 safexid（当时的 nextXid，64 位，nbtree.h:236），`BTPageIsRecyclable`（nbtree.h:292）只在 `GlobalVisCheckRemovableFullXid(heaprel, safexid)` 成立——即删页时刻之前开始的所有快照都已结束——时才允许进 FSM。这与堆元组"死亡→越过视界→回收"同构（Lanin & Shasha 的 drain technique）。FSM 非崩溃安全，`_bt_allocbuf` 取页后还会复验。

**8. GIN 的 pending list 与 LSM memtable 的类比，哪些点对应、哪些点不对应？它的崩溃一致性靠什么？**

对应：写入先追加到小的无序结构（pending list / memtable），延迟批量归并进大的有序结构（entry+posting tree / SSTable），同 key 的 TID 聚合后一次插树，把 N 次随机写摊销成批量写（`ginHeapTupleFastInsert`，ginfast.c:219；`ginInsertCleanup`，ginfast.c:780）；查询必须额外扫小结构。不对应：pending list 是 WAL 保护的磁盘页链（崩溃不丢）；合并是就地插树而非产生新文件层；无多层 compaction。崩溃一致性：清理的"插主树"与"摘 pending 页"两步间崩溃后重放是**幂等**的——同一 (key, TID) 重复插入 posting 无害（ginfast.c:768-771 注释），以幂等性替代跨步原子性。

**9. 为什么 PG 的 Index Only Scan 需要 Visibility Map，而 InnoDB 的覆盖索引扫描不需要类似机制？**

PG 索引项不携带可见性信息（无 xmin/xmax），判断索引项对本快照是否可见本须回堆；VM 提供页粒度"全部可见"缓存（VACUUM 置位），命中则跳过回表（nodeIndexonlyscan.c:165），未命中退化回表，次数即 EXPLAIN 的 Heap Fetches。InnoDB 二级索引页有 PAGE_MAX_TRX_ID 可做粗粒度可见性判断，不可判定时回聚簇索引借 undo 构造旧版本——它的旧版本在 undo 段，而 PG 的旧版本以完整元组留在堆中、索引对其一无所知。根源是第九节的三角权衡：版本链藏在哪，决定了索引能不能"自证可见"。

**10. 一条 UPDATE 语句改了表中一个非索引列，PG 与 InnoDB 各自的索引维护成本是什么？PG 在什么条件下能做到零索引写？**

InnoDB：行在聚簇索引原地更新 + 写 undo，二级索引零改动（指向的主键值未变）。PG：新版本元组写入堆，若命中 HOT——未修改任何索引列**且**新版本放得进同一堆页——则新版本挂在页内 HOT 链上，所有索引零改动（索引仍指向链头 TID，回表沿链找可见版本）；HOT 不命中则**每个**索引都要插一条指向新 TID 的项（哪怕该索引的列没变），这是 PG 写放大的主源，PG 14 bottom-up deletion（commit d168b66682）与 dedup 都是它的止血带。调优抓手：预留 fillfactor 保 HOT 命中率、避免索引热更新列。
