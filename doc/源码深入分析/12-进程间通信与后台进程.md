## 本章导读

PG 选择了多进程模型（每连接一个 backend + 一群后台常驻进程），于是三个问题必须回答：

1. **进程间怎么共享数据？** ——启动时一次性划好的静态共享内存，加上按需伸缩的动态共享内存（DSM/DSA）；
2. **进程间怎么互相等待与唤醒？** ——Latch + WaitEventSet 把"等 socket、等定时器、等同伴戳我一下"统一成一个事件循环；信号 + CHECK_FOR_INTERRUPTS + ProcSignal 解决"异步打断一个正在干活的进程"；
3. **谁在后台干活？** ——checkpointer、bgwriter、walwriter、autovacuum launcher、archiver、统计系统各司其职；background worker 框架则把"写一个数据库内的常驻进程"开放给扩展。

本章 v2 在 v1 骨架上大幅深化，按以下线索展开：

- **完整走读**六条主线：`ShmemInitStruct`（附着 vs 首建双路径）、`SetLatch`/`WaitEventSetWait`（含唤醒竞态的完整论证）、`ProcessInterrupts`（逐中断分发）、`CheckpointerMain` 主循环、`ForwardSyncRequest`/`AbsorbSyncRequests`（fsync 转发协议）、`RegisterBackgroundWorker → StartBackgroundWorker → BackgroundWorkerMain` 生命周期；
- **所以然三问**：为什么要发明 Latch？为什么取消是协作式而非抢占式？为什么 fsync 集中给 checkpointer？外加 bgwriter 分家（9.2）与统计系统重构（PG 15）两桩历史公案；
- **不变式清单**与**what-if 推演**（checkpointer 死掉、bgworker 崩溃、信号在错误时刻到达）；
- 结尾自测 10 题带详解。

## 12.1 共享内存布局

### 12.1.1 静态共享内存：启动时一次划定

postmaster 启动时调用 `CreateSharedMemoryAndSemaphores()`（`src/backend/storage/ipc/ipci.c:120`），全函数如下（逐字）：

```c
void
CreateSharedMemoryAndSemaphores(void)
{
	PGShmemHeader *shim;
	PGShmemHeader *seghdr;
	Size		size;

	Assert(!IsUnderPostmaster);

	/* Compute the size of the shared-memory block */
	size = CalculateShmemSize();
	elog(DEBUG3, "invoking IpcMemoryCreate(size=%zu)", size);

	/*
	 * Create the shmem segment
	 */
	seghdr = PGSharedMemoryCreate(size, &shim);

	/*
	 * Make sure that huge pages are never reported as "unknown" while the
	 * server is running.
	 */
	Assert(strcmp("unknown",
				  GetConfigOption("huge_pages_status", false, false)) != 0);

	/*
	 * Set up shared memory allocation mechanism
	 */
	InitShmemAllocator(seghdr);

	/* Initialize all shmem areas */
	ShmemInitRequested();

	/* Initialize dynamic shared memory facilities. */
	dsm_postmaster_startup(shim);

	/*
	 * Now give loadable modules a chance to set up their shmem allocations
	 */
	if (shmem_startup_hook)
		shmem_startup_hook();
}
```

五步依次是：算总账（`CalculateShmemSize()`，`ipci.c:57`：各子系统需求之和 + 100KB 零头 + 扩展经 `RequestAddinShmemSpace()`（`ipci.c:45`）申报的量，最后向上取整到 8KB 页）→ 建段（`PGSharedMemoryCreate()`，mmap/SysV，可用 huge pages）→ 装分配器（`InitShmemAllocator()`，`shmem.c:637`）→ 按注册清单初始化各子系统结构（`ShmemInitRequested()`，`shmem.c:424`）→ DSM 设施自举 + 扩展钩子。

初始化清单是什么？本版本（20devel）刚完成一次注册机制重构（commit `1fc2e9fbc0` "Introduce a registry of built-in shmem subsystems"）：各内核子系统以回调三元组（request_fn / init_fn / attach_fn）注册，名单集中在 `src/include/storage/subsystemlist.h`，由 `RegisterBuiltinShmemCallbacks()`（`ipci.c:168`）用 X-macro 技巧逐个展开注册。名单读一遍就是共享内存的"户口本"（节选，逐项见 subsystemlist.h）：LWLock、DSM/DSM registry、XLOG（WAL 缓冲）、CLOG/CommitTs/SUBTRANS/MultiXact（SLRU 们）、**BufferManager（大头：shared_buffers）**、锁表与谓词锁、ProcGlobal/ProcArray/BackendStatus、TwoPhase、BackgroundWorker、**SharedInval（第 11 篇的失效队列）**、PMSignal/ProcSignal、**Checkpointer（12.4 的请求队列就在这）**、AutoVacuum、复制槽/Origin/WalSnd/WalRcv/WalSummarizer、PgArch、ApplyLauncher/SlotSync、BTree/SyncScan/Async（NOTIFY 队列）、**Stats（PG 15 起的统计共享区）**、自定义等待事件、注入点、WaitLSN、逻辑解码控制、数据校验和、AIO。

关键性质：**尺寸在启动时定死**。所以改 `shared_buffers`、`max_connections`、`max_locks_per_transaction` 都要重启——每一项都进了 `CalculateShmemSize()` 的公式。运行前想知道会分配多少，可看运行时计算 GUC `shared_memory_size` / `shared_memory_size_in_huge_pages`（`InitializeShmemGUCs()`，`ipci.c:189`）。

`ShmemInitRequested()`（`shmem.c:424`）在 postmaster 单进程阶段执行，两轮循环：先对每个申请项调 `InitShmemIndexEntry()` 划地并登记；再统一调用各子系统的 `init_fn` 回调完成结构内部初始化。函数开头的 `Assert(!IsUnderPostmaster)` 和注释 "There are no concurrent processes yet, so no need for locking" 点明了它的执行环境——**此刻天下无人，无锁即正确**。

### 12.1.2 ShmemIndex：按名字认领地盘，附着与首建的双路径

一块裸内存怎么让几十个子系统、几百个进程都能找到自己的结构？答案是一个**本身就放在共享内存里**的哈希表 **ShmemIndex**：名字（字符串）→（位置指针，尺寸）。围绕它有一对底层函数，"首建"与"附着"各走一条：

**首建路径 `InitShmemIndexEntry()`**（`shmem.c:511`，主干逐字）：

```c
static void
InitShmemIndexEntry(ShmemRequest *request)
{
	const char *name = request->options->name;
	ShmemIndexEnt *index_entry;
	bool		found;
	size_t		allocated_size;
	void	   *structPtr;

	/* look it up in the shmem index */
	index_entry = (ShmemIndexEnt *)
		hash_search(ShmemIndex, name, HASH_ENTER_NULL, &found);
	if (found)
		elog(ERROR, "shared memory struct \"%s\" is already initialized", name);
	...
	structPtr = ShmemAllocRaw(request->options->size,
							  request->options->alignment,
							  &allocated_size);
	if (structPtr == NULL)
	{
		/* out of memory; remove the failed ShmemIndex entry */
		hash_search(ShmemIndex, name, HASH_REMOVE, NULL);
		ereport(ERROR, ...);
	}
	index_entry->size = request->options->size;
	index_entry->allocated_size = allocated_size;
	index_entry->location = structPtr;

	/* Initialize depending on the kind of shmem area it is */
	switch (request->kind)
	{
		case SHMEM_KIND_STRUCT:
			if (request->options->ptr)
				*(request->options->ptr) = index_entry->location;
			break;
		case SHMEM_KIND_HASH:
			shmem_hash_init(structPtr, request->options);
			break;
		case SHMEM_KIND_SLRU:
			shmem_slru_init(structPtr, request->options);
			break;
	}
}
```

三个细节：① `HASH_ENTER_NULL` + `found` 检查——同名重复初始化直接 `elog(ERROR)`，这是"**单次初始化**"不变式的执行者；② 分配失败会**回滚哈希表项**再报错，不留半成品；③ 除裸结构（STRUCT）外还支持共享哈希表（HASH）和 SLRU 两种"品类"，各有专门的 init。

**附着路径 `AttachShmemIndexEntry()`**（`shmem.c:580`，主干逐字）：

```c
static bool
AttachShmemIndexEntry(ShmemRequest *request, bool missing_ok)
{
	const char *name = request->options->name;
	ShmemIndexEnt *index_entry;

	/* Look it up in the shmem index */
	index_entry = (ShmemIndexEnt *)
		hash_search(ShmemIndex, name, HASH_FIND, NULL);
	if (!index_entry)
	{
		if (!missing_ok)
			ereport(ERROR, ...);
		return false;
	}

	/* Check that the size in the index matches the request */
	if (index_entry->size != request->options->size &&
		request->options->size != SHMEM_ATTACH_UNKNOWN_SIZE)
	{
		ereport(ERROR,
				(errmsg("shared memory struct \"%s\" was created with"
						" different size: existing %zu, requested %zd",
						name, index_entry->size, request->options->size)));
	}
	...
	switch (request->kind)
	{
		case SHMEM_KIND_STRUCT:
			if (request->options->ptr)
				*(request->options->ptr) = index_entry->location;
			break;
		...
	}
	return true;
}
```

附着只做三件事：查名字、**核对尺寸**（防止扩展升级后新旧代码对同一块内存的结构定义不一致——这是共享内存版本安全的最后防线）、把调用者的全局指针变量指向已存在的位置。它**不写**共享内存本体。

### 12.1.3 ShmemInitStruct 全函数走读：get-or-create 的一站式封装

上面那对函数在新注册机制中由框架调用；而扩展（pg_stat_statements 等）至今用的是老接口 `ShmemInitStruct()`（`shmem.c:1013`）。文件头注释（`shmem.c:105-119`）明说它是 "Legacy"、新内核代码应改用 `ShmemRequestStruct()`，但 "as of this writing it is still widely used in extensions"。全函数逐字：

```c
void *
ShmemInitStruct(const char *name, Size size, bool *foundPtr)
{
	void	   *ptr = NULL;
	ShmemStructOpts options = {
		.name = name,
		.size = size,
		.ptr = &ptr,
	};
	ShmemRequest request = {&options, SHMEM_KIND_STRUCT};

	Assert(shmem_request_state == SRS_DONE ||
		   shmem_request_state == SRS_INITIALIZING ||
		   shmem_request_state == SRS_REQUESTING);

	LWLockAcquire(ShmemIndexLock, LW_EXCLUSIVE);

	/*
	 * During postmaster startup, look up the existing entry if any.
	 */
	*foundPtr = false;
	if (IsUnderPostmaster)
		*foundPtr = AttachShmemIndexEntry(&request, true);

	/* Initialize it if not found */
	if (!*foundPtr)
		InitShmemIndexEntry(&request);

	LWLockRelease(ShmemIndexLock);

	Assert(ptr != NULL);
	return ptr;
}
```

逐行拆解：

1. **把参数打包成新机制的请求结构**——旧接口如今是新机制的薄封装（`SHMEM_KIND_STRUCT`），一套底层双走；
2. **状态机断言**：进程私有变量 `shmem_request_state` 追踪"申报 → 初始化 → 完成"的阶段（`shmem.c:171` 起的注释），保证扩展不会在错误阶段（比如 shmem_request_hook 之外）偷偷分配；
3. **`ShmemIndexLock` 排他锁**：多个 backend 可能并发首次调用（典型场景：扩展的 `shmem_startup_hook` 在每个进程启动时都会跑）。锁保证"查表 + 建表"是原子的——两个进程同时进来，只有一个走 `InitShmemIndexEntry` 首建，另一个必然 `found = true` 走附着；
4. **`IsUnderPostmaster` 分流**：postmaster/单用户模式自己就是首建者，不用查（查了也没有）；子进程先尝试附着（`missing_ok = true`），查不到再首建——这正是"**附着 vs 首建的双路径**"在一个函数里的汇合点；
5. 返回语义是 "**get or create**"：调用者必须检查 `*foundPtr`——`false` 表示这块内存刚分出来还是生的，得自己初始化内容；`true` 表示别人已初始化过，直接用。框架不提供 init 回调（注释：`the caller of ShmemInitStruct() must check the return status ... and initialize the struct if it was not previously initialized`），这也是它比新接口"危险"的地方——忘记检查 found 就重复初始化，等于把共享状态清零（见 12.6 不变式 I1）。

为什么子进程附着后指针能直接用？因为共享内存映射地址随 fork 继承，**同一结构在所有进程中的虚拟地址相同**，可以直接存裸指针。（EXEC_BACKEND 平台（Windows）没有 fork 继承，才需要 `ShmemAttachRequested()`（`shmem.c:463`）/`AttachSharedMemoryStructs()`（`ipci.c:92`）在每个子进程重放一遍附着。）

账本可查：`pg_get_shmem_allocations()`（`shmem.c:1048`，即 `pg_shmem_allocations` 视图）遍历 ShmemIndex 打印每项的名字/偏移/申请尺寸/实际尺寸，是排查"共享内存花在哪"的利器。

分配器本身（`ShmemAllocRaw`）是**只进不出**的指针递增分配（无 free）——静态区的结构与实例共存亡，不需要回收，简单即正确。

### 12.1.4 动态共享内存：dsm.c 与 DSA

静态区应付不了"并行查询临时要 512MB 哈希表"这类需求，于是有第二层：

- **dsm.c**（`src/backend/storage/ipc/dsm.c`）：运行期创建/销毁整段共享内存。`dsm_create()`（`dsm.c:524`）创建一段并返回句柄（handle，一个可跨进程传递的整数），其他进程 `dsm_attach()`（`dsm.c:673`）映射进来。底层实现可选 POSIX shm / SysV / Windows / mmap 文件（`src/include/storage/dsm_impl.h:17-20`，Linux 默认 POSIX）。注意：**各进程 attach 到的虚拟地址可能不同**，段内不能存裸指针，只能存相对偏移；
- **DSA**（dynamic shared area，`src/backend/utils/mmgr/dsa.c`）：在若干 DSM 段之上做的**跨进程 malloc**。`dsa_create_ext()`（`dsa.c:421`）建区，`dsa_attach()`（`dsa.c:510`）加入，`dsa_allocate_extended()`（`dsa.c:683`）/`dsa_free()` 分配释放。因为地址不统一，它发明了 `dsa_pointer`（段号+段内偏移的编码），解引用要走 `dsa_get_address()`。区会随需求自动创建更大的新段（指数增长），有上限可控。

典型消费者：并行查询（leader 建 DSM 段传计划与参数，worker attach；元组经 `shm_mq` 共享内存队列回流）、并行哈希连接的共享哈希表、以及 12.4.8 的统计系统（dshash——DSA 上的跨进程哈希表）。

> 对照：静态区像 Spark executor 启动时的固定 memory pool，DSA 像 task 运行期找 MemoryManager 动态要的执行内存；`dsa_pointer` 的"句柄代替指针"就是共享内存编程的通用税。静态区能用裸指针纯粹是吃了 fork 继承的红利——动态段没这红利，就得交税。

### 小结

- 静态共享内存启动时按 `CalculateShmemSize()`（ipci.c:57）一次划定（故相关 GUC 要重启），子系统清单见 subsystemlist.h（20devel 新注册机制，commit `1fc2e9fbc0`）；
- 认领机制是共享哈希表 ShmemIndex：首建走 `InitShmemIndexEntry()`（分配+登记+失败回滚），附着走 `AttachShmemIndexEntry()`（查名+**核对尺寸**+接指针）；`ShmemInitStruct()` 在 `ShmemIndexLock` 排他锁下把两条路径合成 get-or-create，`*foundPtr` 区分生熟；
- 指针可直接共享（地址继承自 fork，EXEC_BACKEND 除外）；静态分配器只进不出；
- 动态层 dsm.c 管段的生命周期（attach 地址不保证一致 → 偏移代指针），DSA 在其上提供跨进程 malloc 与 dsa_pointer。

## 12.2 Latch 与 WaitEventSet：统一的等待/唤醒原语

### 12.2.1 所以然之一：为什么需要 Latch 抽象，而不是直接用信号或条件变量

一个 backend/后台进程经常要**同时**等多件事：

- socket 可读（客户端发来请求 / walreceiver 等主库数据）；
- 超时（该做周期工作了：checkpoint_timeout、bgwriter_delay）；
- 别的进程唤醒（"有 fsync 请求给你了"、"配置改了"）；
- postmaster 死亡（父进程没了要自杀，否则变孤儿继续写共享内存）。

在多**进程**模型下把这四件事组合起来，经典 Unix 工具箱里每一样都差一口气，推演如下：

1. **条件变量（pthread_cond_t）**？POSIX 条件变量默认只在线程间工作；进程间版本（PTHREAD_PROCESS_SHARED）可移植性差，而且**根本等不了 socket**——你在 cond_wait 里睡着时客户端断开了，你不知道。凡是"既要等同伴又要等 fd"的场合，条件变量都出局。（PG 内部的 ConditionVariable 其实就是在 Latch 之上封装的——先有 Latch 才有它。）
2. **纯信号（pause/sigsuspend）**？同样等不了 socket，且信号会丢：两次 SIGUSR1 挤在一起只算一次。语义上信号是"戳一下"，没有"有活没干完"的持久状态。
3. **select/poll + 信号**？这是最接近的组合，但有教科书级竞态——**检查与睡眠之间的窗口**：

```
   目标进程                              发送进程
   ─────────                            ─────────
   if (!work_pending)     ←── ①检查：没活
                                        work_pending = true;
                                        kill(pid, SIGUSR1);   ←── ②信号到达，
                                            handler 置了个全局 flag，返回
   select(..., timeout=∞);  ←── ③带着"没活"的旧结论睡死
```

   信号在 ① 和 ③ 之间到达：handler 跑过了、flag 置了，但 select 还没进内核，**信号打断不了一个还没开始的 select**；进了内核之后，这个早已递送完毕的信号也不会再打断它。目标进程带着一条待办事项睡到天荒地老——这就是 **lost wakeup（丢失唤醒）**。`waiteventset.c` 文件头注释（`waiteventset.c:20-28`）原话："A signal by itself doesn't interrupt poll() on all platforms, and even on platforms where it does, a signal that arrives just before the poll() call does not prevent poll() from entering sleep."

PG 的解法是 2010 年引入的 **Latch**（commit `2746e5f21d`，Heikki Linnakangas，提交信息即点题："Latches can be used to reliably wait until a signal arrives, which is hard otherwise because signals don't interrupt select() on some platforms, and even when they do, there's race conditions"）。Latch 是一个**有状态的一位门闩**：

- `SetLatch()` 置位——异步信号安全，可以在 signal handler 里调；置过再置是幂等的（"这是廉价的"）；
- `ResetLatch()` 复位——只有 owner 自己调；
- `WaitLatch()` 等待——**如果位已经置起，立刻返回，根本不睡**。

"状态位"解决语义问题（唤醒不再是转瞬即逝的事件，而是持久的待办标记）；剩下的问题是**怎么让一个置位动作打断正在内核里睡的 select/poll/epoll**——这就是 **self-pipe trick**：进程自建一根管道并把读端加进 poll 集合；SIGURG 的 handler 往写端写一个字节。于是"信号"被转换成"fd 可读"，与 socket 事件同质化了。关键在于它**天然免疫上面那个竞态窗口**：信号如果在 poll 之前一瞬到达，字节已经躺在管道里，poll 一进内核就发现读端就绪、立即返回——管道里的字节不会像信号那样"过期"。文件头注释（`waiteventset.c:26-28`）："An incoming byte on a pipe however reliably interrupts the sleep, and causes poll() to return immediately even if the signal arrives before poll() begins."

后续演进（都能在 git 里挖到）：

- **2016**：commit `98a64d0bd7`（Andres Freund）"Introduce WaitEventSet API"——把"每次 WaitLatch 都临时搭一套 poll 集合"改成**长生命周期的事件集**，Linux 上用 epoll，免去每次等待重建内核对象的开销；latch、socket、postmaster 死亡统一为 WaitEvent；
- **2021**：commit `83709a0d5a`（Thomas Munro）改用 **SIGURG** 专职做 latch 信号（腾出 SIGUSR1 给 ProcSignal 专用）；同日 commit `6a2a70a020` 在 epoll 平台用 **signalfd** 直接把 SIGURG 变成 fd，可省掉 self-pipe 和 handler；kqueue 平台（macOS/BSD）则用 `EVFILT_SIGNAL`（commit `6148656a0b`）；
- **2025**：commit `393e0d2314` 把 WaitEventSet 从 latch.c 拆到独立的 `waiteventset.c`。

平台选择在编译期决定（`waiteventset.c:88-101`）：epoll（Linux）> kqueue（macOS/BSD）> poll > Windows WaitForMultipleObjects；唤醒通道选择在 `waiteventset.c:104-115`：epoll+signalfd，poll+self-pipe。所以今天在 Linux 上 self-pipe 只是后备实现，但**它的语义模型仍是理解这套机制的钥匙**。

### 12.2.2 SetLatch 全函数走读

`SetLatch()`（`src/backend/storage/ipc/latch.c:290`），Unix 路径逐字：

```c
void
SetLatch(Latch *latch)
{
#ifndef WIN32
	pid_t		owner_pid;
#else
	HANDLE		handle;
#endif

	/*
	 * The memory barrier has to be placed here to ensure that any flag
	 * variables possibly changed by this process have been flushed to main
	 * memory, before we check/set is_set.
	 */
	pg_memory_barrier();

	/* Quick exit if already set */
	if (latch->is_set)
		return;

	latch->is_set = true;

	pg_memory_barrier();
	if (!latch->maybe_sleeping)
		return;

#ifndef WIN32
	/* （注释省略：见 12.2.4 的竞态论证，原文 latch.c:317-337） */
	owner_pid = latch->owner_pid;
	if (owner_pid == 0)
		return;
	else if (owner_pid == MyProcPid)
		WakeupMyProc();
	else
		WakeupOtherProc(owner_pid);
#else
	...  /* Windows: SetEvent(latch->event)，此处省略 */
#endif
}
```

四段逻辑，每段都有讲究：

1. **第一道内存屏障**：调用者的惯用法是"先写共享标志（如 `CheckpointerShmem->ckpt_flags` 或 `QueryCancelPending`），再 SetLatch"。屏障保证标志的写入先于 `is_set` 的写入对其他 CPU 可见——否则等待者被唤醒后去看标志，可能看到旧值，空转一圈又睡回去，而唤醒不会再来（is_set 已被这次消费掉）。
2. **快速退出**：已置位就直接返回。这使 SetLatch 可以被高频滥调而无系统调用开销——"This is cheap if the latch is already set, otherwise not so much"（`latch.c:280`）。
3. **`maybe_sleeping` 检查**：这是 WaitEventSetWait 睡前立的牌子（见下）。牌子没立 = 对方根本没在睡（正在干活，迟早会走到循环头检查标志）= **不用发信号/写管道**。这个优化把"唤醒一个醒着的进程"的成本降到两次内存屏障，无系统调用。
4. **发内核唤醒**：owner 是自己（说明在自己的 signal handler 里 SetLatch 打断自己的等待）走 `WakeupMyProc()`（写 self-pipe / 在 signalfd 模式下 `kill(MyProcPid, SIGURG)`），是别人则 `WakeupOtherProc(pid)`（`kill(pid, SIGURG)`）。`latch.c:332-337` 的注释还承认一个残余竞态：读完 `owner_pid` 之后可能刚好有新进程接管了这个 latch，于是信号没发给它——"This is okay so long as all callers of ResetLatch/WaitLatch follow the standard coding convention of waiting at the bottom of their loops, not the top"——**机制的正确性最终托付给了编码惯例**，这正是 12.6 不变式 I2 的出处。

`ResetLatch()`（`latch.c:374`）短小但屏障同样致命：

```c
void
ResetLatch(Latch *latch)
{
	/* Only the owner should reset the latch */
	Assert(latch->owner_pid == MyProcPid);
	Assert(latch->maybe_sleeping == false);

	latch->is_set = false;

	/*
	 * Ensure that the write to is_set gets flushed to main memory before we
	 * examine any flag variables.  Otherwise a concurrent SetLatch might
	 * falsely conclude that it needn't signal us, even though we have missed
	 * seeing some flag updates that SetLatch was supposed to inform us of.
	 */
	pg_memory_barrier();
}
```

注意它与 SetLatch 的屏障是**成对**的：SetLatch 保证"标志先于 is_set 写入"，ResetLatch 保证"is_set 清零先于随后的标志读取"。两边合起来才构成无锁握手（详见下节推演）。

### 12.2.3 WaitLatch 与 WaitEventSetWait 主线走读

`WaitLatch()`（`latch.c:172`）本体已退化为 WaitEventSet 的薄封装（逐字）：

```c
int
WaitLatch(Latch *latch, int wakeEvents, long timeout,
		  uint32 wait_event_info)
{
	WaitEvent	event;

	/* Postmaster-managed callers must handle postmaster death somehow. */
	Assert(!IsUnderPostmaster ||
		   (wakeEvents & WL_EXIT_ON_PM_DEATH) ||
		   (wakeEvents & WL_POSTMASTER_DEATH));

	/*
	 * Some callers may have a latch other than MyLatch, or no latch at all,
	 * or want to handle postmaster death differently.  It's cheap to assign
	 * those, so just do it every time.
	 */
	if (!(wakeEvents & WL_LATCH_SET))
		latch = NULL;
	ModifyWaitEvent(LatchWaitSet, LatchWaitSetLatchPos, WL_LATCH_SET, latch);

	if (IsUnderPostmaster)
		ModifyWaitEvent(LatchWaitSet, LatchWaitSetPostmasterDeathPos,
						(wakeEvents & (WL_EXIT_ON_PM_DEATH | WL_POSTMASTER_DEATH)),
						NULL);

	if (WaitEventSetWait(LatchWaitSet,
						 (wakeEvents & WL_TIMEOUT) ? timeout : -1,
						 &event, 1,
						 wait_event_info) == 0)
		return WL_TIMEOUT;
	else
		return event.events;
}
```

要点：开头的 Assert 强制每个受 postmaster 管的等待者声明对"父亲死了"的态度（自动退出或拿到事件自己处理）——**任何一个等待点漏了这条，postmaster 崩溃后就会留下僵尸子进程继续写共享内存**；`LatchWaitSet` 是进程常驻的 WaitEventSet，避免每次等待重建 epoll 实例（正是 commit `98a64d0bd7` 的初衷）。

真正的等待在 `WaitEventSetWait()`（`waiteventset.c:1040`）。主线逐字（省略 Windows 分支与计时细节，均注明）：

```c
int
WaitEventSetWait(WaitEventSet *set, long timeout,
				 WaitEvent *occurred_events, int nevents,
				 uint32 wait_event_info)
{
	int			returned_events = 0;
	...
	/* 计时初始化：记录起点以便被打断后算剩余超时（waiteventset.c:1051-1062，省略） */

	pgstat_report_wait_start(wait_event_info);

#ifndef WIN32
	waiting = true;
#endif
	while (returned_events == 0)
	{
		int			rc;

		/* （长注释见原文 waiteventset.c:1076-1102，核心句译注于下文） */
		if (set->latch && !set->latch->is_set)
		{
			/* about to sleep on a latch */
			set->latch->maybe_sleeping = true;
			pg_memory_barrier();
			/* and recheck */
		}

		if (set->latch && set->latch->is_set)
		{
			occurred_events->fd = PGINVALID_SOCKET;
			occurred_events->pos = set->latch_pos;
			occurred_events->user_data =
				set->events[set->latch_pos].user_data;
			occurred_events->events = WL_LATCH_SET;
			occurred_events++;
			returned_events++;

			/* could have been set above */
			set->latch->maybe_sleeping = false;

			if (returned_events == nevents)
				break;			/* output buffer full already */

			/* 还想顺手收集非 latch 事件：改成零超时探询一次（1127-1133，省略） */
		}

		rc = WaitEventSetWaitBlock(set, cur_timeout,
								   occurred_events, nevents - returned_events);

		if (set->latch &&
			set->latch->maybe_sleeping)
			set->latch->maybe_sleeping = false;

		if (rc == -1)
			break;				/* timeout occurred */
		else
			returned_events += rc;

		/* If we're not done, update cur_timeout for next iteration（1153-1161，省略） */
	}
#ifndef WIN32
	waiting = false;
#endif

	pgstat_report_wait_end();

	return returned_events;
}
```

主线三步：

1. **睡前立牌**：latch 未置位 → 写 `maybe_sleeping = true` → 屏障 → **复查** `is_set`。复查命中说明"就在立牌前后有人 Set 了"，直接返回 WL_LATCH_SET 事件，不进内核；
2. **进内核**：`WaitEventSetWaitBlock()` 是平台分发点（同名四个实现：epoll 版 `waiteventset.c:1184`、kqueue 版 :1320、poll 版 :1473、Win32 版 :1607）。以 epoll 版为例：`epoll_wait()` 返回后逐个事件翻译——latch 位置的 fd（signalfd/self-pipe 读端）就绪则 `drain()` 排空它，再**核实 `is_set` 仍为真**才上报 WL_LATCH_SET（`waiteventset.c:1239`，防杂音）；`WL_POSTMASTER_DEATH` 位置的管道 HUP 则再调 `PostmasterIsAliveInternal()` 复核（注释：历史上 select 出过假死报告，"a spurious WL_POSTMASTER_DEATH event would be painful"），确认后若调用者选了 `WL_EXIT_ON_PM_DEATH` 直接 `proc_exit(1)`；socket 事件按 EPOLLIN/OUT/RDHUP 翻译成 WL_SOCKET_*；
3. **醒后摘牌**：无论因何返回都把 `maybe_sleeping` 清掉，然后按超时/事件数决定重试或返回。

`EINTR` 的处理值得一提（epoll 版 `waiteventset.c:1197-1209`）：信号打断返回 0（"要重试"），外层循环回到第 1 步——于是任何在 handler 里 SetLatch 的信号都会在重试的复查里被看到。**信号无论何时到达，最终都收敛到 is_set 这个持久状态上**。

### 12.2.4 唤醒竞态的完整论证：为什么顺序不可颠倒

现在把两边拼起来，证明"永不丢唤醒"。约定生产者-消费者协议：

- **生产者（如 ForwardSyncRequest）**：`①写共享数据/标志 → ②SetLatch(&target->procLatch)`；
- **消费者（所有后台进程主循环）**：`③ResetLatch(MyLatch) → ④检查标志/干活 → ⑤WaitLatch(MyLatch, ...)`。

**正确性论证**（穷举 SetLatch 相对消费者循环的落点）：

- Set 落在 ③ 之前：is_set 为真，⑤ 的 WaitEventSetWait 开头复查直接命中，立即返回。③ 虽把位清了，但 ③ 之后的 ④ 会读到 ① 写的标志（ResetLatch 的屏障保证 is_set 清零先于标志读取；SetLatch 的第一道屏障保证标志写入先于 is_set 置位——所以"我清掉的那个 set"所携带的数据，④ 必能看见）；
- Set 落在 ③ 与 ⑤ 之间（最险的窗口）：is_set 置起后没人睡（`maybe_sleeping` 为假，生产者省掉信号），但 ⑤ 进来第一件事就是查 `is_set`——为真，**不睡，立即返回**，回到 ③④ 把活干掉。位就是留言条，人不在也不怕；
- Set 落在 ⑤ 睡前立牌与进内核之间：立牌后的**复查**兜住（`maybe_sleeping = true; barrier; recheck`）；即使复查也错过（Set 恰在复查之后），此时生产者读到 `maybe_sleeping == true`，一定会补发 SIGURG/写管道，signalfd/self-pipe 的字节让 epoll_wait 立即返回；
- Set 落在睡眠中：内核唤醒，天经地义。

四种落点全覆盖，**唤醒最多造成一次空转，绝不丢失**。

**反证一（消费者颠倒 ③④）**：改成"④检查标志 → ③ResetLatch → ⑤WaitLatch"。生产者在 ④ 之后、③ 之前完成 ①②：标志你已经检查过了（当时还没置），latch 置位却被紧随其后的 Reset 无辜清掉——⑤ 睡死，活躺在共享内存里没人管。这正是裸 select+flag 竞态的还魂。**Reset 必须先于检查，宁可多空转一圈。**

**反证二（生产者颠倒 ①②）**：改成"②SetLatch → ①写标志"。消费者被唤醒，④ 读标志——可能还没写进来（唤醒跑得比数据快，尤其弱内存序机器上没有屏障保证顺序），空转一圈睡回去；而 ① 随后完成，**不再有第二次 SetLatch**。丢活。所以 SetLatch 文档的隐含契约是"**你想让对方看到的东西必须在 SetLatch 之前写完**"，由 SetLatch 开头的 `pg_memory_barrier()` 把这个契约落到硬件上。

于是所有后台进程的主循环都长一个样（**醒来先复位，睡前是最后一次检查**）：

```c
for (;;)
{
    ResetLatch(MyLatch);          /* 复位门闩 */
    处理各种挂起标志 / CHECK_FOR_INTERRUPTS();
    做本职工作;
    WaitLatch(MyLatch, WL_LATCH_SET | WL_TIMEOUT | WL_EXIT_ON_PM_DEATH,
              超时毫秒, WAIT_EVENT_XXX);
}
```

（个别循环故意反着写并注明理由，如 autovacuum launcher"先睡后干"——见 12.4.6，那是因为它想启动后先歇满一个 naptime；即便如此 Reset 依然紧跟在 WaitLatch 之后、检查之前，不变式没破。）

### 12.2.5 wait_event 名字如何进 pg_stat_activity

`WaitLatch`/`WaitEventSetWait` 的 `wait_event_info` 参数就是 `pg_stat_activity.wait_event` 里名字的来源。全部等待事件集中登记在数据文件 `src/backend/utils/activity/wait_event_names.txt`，构建期由 `generate-wait_event_types.pl`（同目录）生成枚举、名字表和 SGML 文档——加一个等待点只需在 txt 里添一行。运行期：进入等待前 `pgstat_report_wait_start(wait_event_info)`（`src/include/utils/wait_event.h:67`，inline，仅一次 4 字节原子写）把 32 位编码（高 8 位类别 + 低位具体事件）写进本进程的 BackendStatus 共享内存槽；`pg_stat_activity` 读到编码再查名字表翻译。注释还留了一句考古："Historically we used to make this reporting conditional on pgstat_track_activities, but the check for that seems to add more cost than it saves"——开销小到连开关都不值得设。这就是 PG 可观测性"无处不在却几乎免费"的实现套路。

### 小结

- 多路等待（socket+定时器+同伴唤醒+父进程死亡）在多进程模型下无现成原语：条件变量等不了 fd，信号无状态且有 check-then-sleep 竞态；Latch = 持久状态位 + self-pipe/signalfd 把唤醒转成 fd 事件，一举两得（commit `2746e5f21d` → `98a64d0bd7` → `83709a0d5a`/`6a2a70a020`）；
- SetLatch：屏障 → 幂等快出 → `maybe_sleeping` 免信号优化 → SIGURG/self-pipe；ResetLatch 的屏障与之成对；
- WaitEventSetWait：睡前立牌复查、平台化 WaitBlock（epoll 版对 latch/postmaster 死亡都有"复核再上报"的防杂音逻辑）、醒后摘牌、EINTR 重试收敛到 is_set；
- 顺序铁律及其反证：消费者 Reset→检查→Wait，生产者写数据→SetLatch；任一颠倒都能构造出丢唤醒时序；
- wait_event 名字源于 wait_event_names.txt（脚本生成代码），等待前一次原子写进 BackendStatus。

## 12.3 信号与 CHECK_FOR_INTERRUPTS：安全地打断一个进程

### 12.3.1 两段式中断

Unix 信号随时可能到达——可能正在 palloc 一半、持有自旋锁。信号处理器里几乎什么都不能做（async-signal-safe 函数表短得可怜，malloc/elog 都不在内）。PG 因此把中断拆成两段：

**第一段：信号处理器只置标志。** SIGINT 的处理器 `StatementCancelHandler()`（`src/backend/tcop/postgres.c:3155`，注册于 `postgres.c:4398`），逐字：

```c
void
StatementCancelHandler(SIGNAL_ARGS)
{
	/*
	 * Don't joggle the elbow of proc_exit
	 */
	if (!proc_exit_inprogress)
	{
		InterruptPending = true;
		QueryCancelPending = true;
	}

	/* If we're still here, waken anything waiting on the process latch */
	SetLatch(MyLatch);
}
```

SIGTERM 的处理器 `die()`（`postgres.c:3114`）同构：置 `ProcDiePending + InterruptPending`、记录发送者（`pg_siginfo->pid/uid`，只记第一次以便报告始作俑者）、SetLatch。**SetLatch 是点睛之笔**：如果进程正睡在 WaitLatch 里（空闲等命令、等锁），标志置了没人看也没用——SetLatch 把它从内核等待里拽出来，醒来的代码路径里自然有 CHECK_FOR_INTERRUPTS。两段式与 12.2 的 Latch 在这里合流。

**第二段：主流程在安全点自查。** `CHECK_FOR_INTERRUPTS()` 宏（`src/include/miscadmin.h:125`，逐字）：

```c
/* Service interrupt, if one is pending and it's safe to service it now */
#define CHECK_FOR_INTERRUPTS() \
do { \
	if (INTERRUPTS_PENDING_CONDITION()) \
		ProcessInterrupts(); \
} while(0)
```

其中 `INTERRUPTS_PENDING_CONDITION()`（`miscadmin.h:115`）在 Unix 上就是 `unlikely(InterruptPending)`——读一个进程本地 volatile 变量，纳秒级。所以它被撒遍所有循环：执行器每处理若干行、每次拿锁等待、每次缓冲区 IO、排序归并的内层循环……这就是"取消一条正在跑的 SQL 通常一瞬间生效"的原因——最近的一个 CFI 点不会太远。

### 12.3.2 所以然之二：为什么是协作式取消，而不是抢占式

"取消一个查询"最朴素的做法是抢占式：像 `kill -9` 那样直接终止，或者在信号处理器里当场 `siglongjmp` 跳走。PG 偏偏选了"置标志、等对方自己走到安全点"的协作式。推演一下抢占式在共享内存架构下会发生什么：

设想 backend 正在执行 `hash_search(HASH_ENTER)` 往**共享内存哈希表**（比如锁表）插入一项：已经从 freelist 摘下一个 entry、改了桶链表头、还没写 entry 内容——此刻信号到达，如果当场跳走：

1. **数据结构半更新**：桶链上挂着一个内容未初始化的 entry，下一个遍历这条链的进程读到垃圾指针，段错误或（更糟）静默错读；
2. **锁永久悬挂**：它当时持有该分区的 LWLock/自旋锁。跳走的代码不知道自己持有什么锁（锁的登记本身也可能正写到一半），锁永远不释放——所有等这把锁的进程死等，故障从一个连接扩散成全库瘫痪；
3. **不变式全面失守**：pin 住的 buffer 计数、LWLock 持有列表、事务状态机……每一处"先改 A 再改 B"的两步操作都可能停在中间。

关键在于：**这些结构在共享内存里，进程死了它们不死**。单进程多线程系统同样怕这个（所以 pthread_cancel 声名狼藉），但 PG 连"进程退出自动回收"的兜底都没有——共享内存的烂摊子会留给所有活着的进程。所以：

- 协作式是共享内存架构的**必然**：只有执行流自己知道"现在数据结构是完整的"，也只有它能选择在完整时刻响应取消；
- 响应手段是 `ereport(ERROR)`——借第 05 篇讲的 sigsetjmp（`postgres.c:4573`）+ `AbortCurrentTransaction()` 机制，把"取消"完全复用错误恢复基础设施：释放锁、unpin buffer、重置内存上下文，全部走已被千锤百炼的清理路径，不需要单独的"取消清理"代码；
- 对于实在不能协作的场景（进程失联、真的想立刻杀），PG 的答案是 SIGQUIT/`quickdie`：**不做任何清理**直接 `_exit(2)`，然后由 postmaster 判定共享内存不可信，杀光所有子进程重放 WAL（crash-restart）。要么优雅协作，要么全体重来，**没有中间态**——中间态就是数据损坏。

这也解释了用户常见的困惑"**为什么 pg_cancel_backend 有时不生效**"：

- 目标正在**不含 CFI 的长循环**里（某些扩展的 C 代码、某些紧密计算），标志置了没人查；
- 目标处在 `HOLD_INTERRUPTS()` 区间（`miscadmin.h:136`，`InterruptHoldoffCount++`）或 WAL **临界区**（`START_CRIT_SECTION()`，`miscadmin.h:152`）——ProcessInterrupts 第一行就检查这两个计数，非零直接 return（此时抛 ERROR 会被升级成 PANIC，绝不能取消）；
- 目标卡在**不可中断的内核态**（如 NFS 上的 D 状态 read），信号根本递送不进去；
- 目标正在读客户端消息（`QueryCancelHoldoffCount != 0`），取消被刻意推迟以免 FE/BE 协议失步（见下节）。

前两种情况下标志并没有丢——只是延迟到下一个允许的 CFI 点；后两种要么等内核返回，要么就只剩 pg_terminate_backend/SIGQUIT。

### 12.3.3 ProcessInterrupts 全函数走读：中断的总分发

`ProcessInterrupts()`（`postgres.c:3452`）按**严格优先级**处理十几种挂起标志，逐段走读（代码均逐字，段间省略号处注明）：

**第 0 段：安全门。**

```c
	/* OK to accept any interrupts now? */
	if (InterruptHoldoffCount != 0 || CritSectionCount != 0)
		return;
	InterruptPending = false;
```

持锁清理期/临界区内直接返回，**标志原样保留**（InterruptPending 也不清），出了禁区的下一个 CFI 再处理——中断从不丢，只延迟。这是 what-if E3 的机制核心。

**第 1 段：ProcDiePending（pg_terminate_backend / 快速关库）——死刑优先。**

```c

	{

		int			sender_uid = ProcDieSenderUid;

			ereport(FATAL, ... "terminating logical replication worker ..." ...);
```

（上段为压缩排版，完整分支见 `postgres.c:3459-3522`。）看点：`ProcDie trumps QueryCancel`——都要死了，取消不取消无意义；按进程身份定制退出文案与方式（FATAL 抛错 vs 直接 proc_exit），错误详情里带上**是谁杀的我**（sender pid/uid）；认证阶段特意切断对客户端的输出（防止给未认证连接泄露信息）。

**第 2 段：客户端连接检查**（`postgres.c:3524-3553`）：`CheckClientConnectionPending` 由 `client_connection_check_interval` 定时器周期置位，此处真正做 `pq_check_connection()` 探活并重新武装定时器；`ClientConnectionLost` 则 FATAL 断开（`whereToSendOutput = DestNone`——连接都死了，别再往里写）。这是"客户端 Ctrl-C 后取消长查询"的第二保险：即使没人发 cancel，查询也会在探测到断连时终止。

**第 3 段：QueryCancelPending——本函数的戏眼**（`postgres.c:3563-3633`）：


三个精妙处：① `QueryCancelHoldoffCount`（由 `HOLD_CANCEL_INTERRUPTS()` 控制）保护"正在读客户端消息"的区间——此时抛错会让 FE/BE 协议失步（半条消息躺在缓冲区里），于是**重新武装 InterruptPending**，读完消息立刻补处理；② 取消的"罪名认定"：同一个 QueryCancelPending 可能来自用户 cancel、lock_timeout 或 statement_timeout（超时处理器置的就是这同一个标志），此处用 `get_timeout_indicator()` 查明真凶、报不同错误码/文案——**一切异步打断殊途同归于标志位，罪名在处理时才判**；③ 空闲时（DoingCommandRead）的用户取消被静默吞掉——没有正在跑的语句，报错徒增噪音。

**第 4-7 段：会话级超时**（`postgres.c:3635-3691`）：恢复冲突（`pendingRecoveryConflicts`，standby 上的查询与回放冲突，细类处理在 `ProcessRecoveryConflictInterrupts`）；`IdleInTransactionSessionTimeoutPending`/`TransactionTimeoutPending`/`IdleSessionTimeoutPending` 三兄弟全是 FATAL 级（直接断连接），且都先复核 GUC 仍非零才动手——注释解释：GUC 改回 0 并不会撤销已挂起的中断，不复核就会误杀；`IdleStatsUpdateTimeoutPending` 则趁真正空闲时把攒着的统计刷出去（12.4.8）。

**第 8 段：杂项批发**（`postgres.c:3693-3709`）：`ProcSignalBarrierPending`（全局屏障，如 DROP DATABASE 要求所有进程放弃对该库文件的引用）、`ParallelMessagePending`（并行 worker 经 shm_mq 转发上来的错误/通知）、`LogMemoryContextPending`（`pg_log_backend_memory_contexts()` 的执行现场——注意：**是被观察进程自己在安全点打印自己的内存上下文**，而不是观察者去读它的内存，这个设计完全规避了跨进程读内存的同步问题）、并行 apply 消息、槽同步、repack 消息。

### 12.3.4 超时的实现路径

`statement_timeout` 走同一条路：backend 启动时 `RegisterTimeout(STATEMENT_TIMEOUT, StatementTimeoutHandler)`（`src/backend/utils/init/postinit.c:799`）。timeout.c 用**一个 SIGALRM 复用多个定时器**（维护按到期时间排序的堆，每次只武装最近的一个）；到点后 `StatementTimeoutHandler()`（`postinit.c:1414`）做的事和收到 SIGINT 一样——置 QueryCancelPending + InterruptPending + SetLatch。lock_timeout、idle_in_transaction_session_timeout、transaction_timeout 同理，只是置的标志不同。**处理器越傻越好，聪明留给 ProcessInterrupts。**

### 12.3.5 ProcSignal：跨进程"多路复用信号"

Unix 实时信号之外可自定义的信号编号极少，而 PG 需要十几种进程间通知；且 kill 只认 pid，pid 会被 OS 复用。`src/backend/storage/ipc/procsignal.c` 把 SIGUSR1 做成多路复用通道：共享内存里每进程一个槽（`ProcSignalInit()`，`procsignal.c:170`），槽里是一排 per-reason 的 `sig_atomic_t` 标志。发送方 `SendProcSignal()`（`procsignal.c:296`，known-procNumber 分支逐字）：

```c
	if (procNumber != INVALID_PROC_NUMBER)
	{
		Assert(procNumber < NumProcSignalSlots);
		slot = &ProcSignal->psh_slot[procNumber];

		SpinLockAcquire(&slot->pss_mutex);
		if (pg_atomic_read_u32(&slot->pss_pid) == pid)
		{
			/* Atomically set the proper flag */
			slot->pss_signalFlags[reason] = true;
			SpinLockRelease(&slot->pss_mutex);
			/* Send signal */
			return kill(pid, SIGUSR1);
		}
		SpinLockRelease(&slot->pss_mutex);
	}
```

先自旋锁下**复核槽仍属于该 pid**（防槽已易主——进程退出后槽会被复用），写 reason 标志，再补一发真实 SIGUSR1 促其尽快查看。接收方 `procsignal_sigusr1_handler()`（`procsignal.c:696`）对每种 reason 调 `CheckProcSignal()`（`procsignal.c:671`，读标志并清零——注释指出 `volatile sig_atomic_t` 让它无锁可读）然后转手调用各 Handle 函数——它们照例只置本进程标志 + 最后统一 `SetLatch(MyLatch)`，重活留给 CFI。原因清单（`ProcSignalReason`）：catchup（第 11 篇失效追赶）、NOTIFY、并行消息、walsender 停止、barrier、日志内存上下文、并行 apply、repack、槽同步、恢复冲突。`pg_cancel_backend()`/`pg_terminate_backend()` 则直接发 SIGINT/SIGTERM，两套并存：**语义粗、历史久的走裸信号；语义细、种类多的走 ProcSignal**。

### what-if E3：信号在"错误的时刻"到达，如何保证不丢？

把三种极端时刻各推一遍：

1. **到达时进程在临界区/HOLD 区间**：handler 照常置标志（handler 从不管这些计数），CFI 走到 ProcessInterrupts 第一行被弹回，**标志保留**；`RESUME_INTERRUPTS()`/`END_CRIT_SECTION()` 之后的下一个 CFI 处理之。唯一的时间上限是"下一个 CFI 有多远"；
2. **到达时进程正处理上一个中断（重入）**：ereport(ERROR) 展开过程中 sigsetjmp 恢复块以 `sigsetjmp(..., 1)` 保存并恢复信号屏蔽字——`postgres.c` 主循环的恢复块（`postgres.c:4565` 注释）说明 longjmp 回来时信号被重新阻塞（BlockSig），错误清理全程免打扰，清理完 `RESUME_INTERRUPTS()` + 解除屏蔽，新到的标志在下一轮循环的 CFI 被处理。checkpointer 的恢复块注释（`checkpointer.c:279-284`）同理并特意说明"这不使 HOLD_INTERRUPTS 冗余，因为 InterruptPending 可能早已置位"；
3. **到达时进程睡在 WaitLatch 里**：handler 末尾的 SetLatch 把它拽醒（12.2 已论证不丢）；即使信号恰在"检查完标志、还没进内核"的窗口到达，is_set 位也会让 WaitEventSetWait 立即返回。

三条路的公共结论：**信号本身允许丢（同种信号合并递送也没关系），因为它只是"去看标志"的催促；真正的信息全在持久的标志位里，而标志只增不丢，处理最多延迟、从不遗漏。**

### 小结

- 中断两段式：信号处理器只置标志 + SetLatch；主流程在海量 CFI 安全点调 ProcessInterrupts 统一分发；
- 协作式是共享内存架构的必然：抢占式终止会留下半更新结构与悬挂锁，且没有"进程死了状态自动回收"的兜底；取消复用 ereport(ERROR)+sigsetjmp 错误恢复全套设施；不能协作时的替代品是 quickdie + 全库 crash-restart，没有中间态；
- ProcessInterrupts 的优先级链：Holdoff/临界区安全门 → ProcDie（trumps cancel）→ 连接检查/断连 → QueryCancel（Holdoff 重挂、超时罪名认定、空闲吞掉）→ 恢复冲突 → 三种会话超时（复核 GUC）→ barrier/并行消息/日志内存上下文等杂项；
- ProcSignal 用共享内存标志槽（自旋锁下复核 pid）+ SIGUSR1 把一个信号复用成十几种跨进程通知；
- 信号在任何时刻到达都不丢：标志持久、Latch 持久、sigsetjmp 恢复期屏蔽、出禁区后补处理。

## 12.4 后台进程逐个解剖

所有后台进程由 postmaster 启动并监管，主循环全部是 12.2 的标准模式。先深挖 checkpointer（机制最全），其余按职责走读。

### 12.4.1 checkpointer：CheckpointerMain 全走读

**职责**：执行 checkpoint（把某时刻前的全部脏页刷盘并推进恢复起点），以及**代全体进程执行 fsync**。

`CheckpointerMain()`（`src/backend/postmaster/checkpointer.c:206`）分四幕。

**第一幕：出生仪式**（`checkpointer.c:213-360`）。`AuxiliaryProcessMainCommon()` 完成辅助进程共同初始化后，先把自己的 pid 写进共享内存（`CheckpointerShmem->checkpointer_pid = MyProcPid`——ForwardSyncRequest 靠它判断"checkpointer 在不在"）；然后配置信号表：

```c
	pqsignal(SIGHUP, SignalHandlerForConfigReload);
	pqsignal(SIGINT, ReqShutdownXLOG);
	pqsignal(SIGTERM, PG_SIG_IGN);	/* ignore SIGTERM */
	/* SIGQUIT handler was already set up by InitPostmasterChild */
	pqsignal(SIGALRM, PG_SIG_IGN);
	pqsignal(SIGPIPE, PG_SIG_IGN);
	pqsignal(SIGUSR1, procsignal_sigusr1_handler);
	pqsignal(SIGUSR2, SignalHandlerForShutdownRequest);
```

注意 **SIGTERM 被无视**（注释 `checkpointer.c:220-223`：系统关机时 init 会对所有进程群发 SIGTERM，checkpointer 必须活到最后写 shutdown checkpoint，postmaster 会用自己的协议——SIGINT=写 shutdown checkpoint，SIGUSR2=可以退了——指挥它）；`before_shmem_exit(pgstat_before_server_shutdown, 0)`（`checkpointer.c:255`）登记"退出前刷统计落盘"——因为 checkpointer 是正常关库时**最后一个**能产生统计的退出者。随后建私有内存上下文并设 sigsetjmp 恢复块（`checkpointer.c:286-346`）：出错时释放 LWLock/buffer pin/临时文件（"a minimal subset of AbortTransaction()"），**若当时正做 checkpoint（ckpt_active）则把失败计数 ckpt_failed++ 写回共享内存并广播 done_cv**——等在 `RequestCheckpoint()` 里的 backend 靠比对 failed 计数得知"这轮失败了"；最后 `pg_usleep(1000000L)`：错误后至少睡 1 秒，防止写错误风暴刷爆日志。

**第二幕：主循环**（`checkpointer.c:366-607`）。骨架逐字（统计与告警支线压缩，注明）：

```c
	for (;;)
	{

		int			flags = 0;
		...

		/* Clear any already-pending wakeups */
		ResetLatch(MyLatch);

		{

				chkpt_or_rstpt_timed = true;
			do_checkpoint = true;

		}

			CheckpointerShmem->ckpt_flags = 0;
			CheckpointerShmem->ckpt_started++;
			SpinLockRelease(&CheckpointerShmem->ckpt_lck);

			ConditionVariableBroadcast(&CheckpointerShmem->start_cv);

			if (flags & CHECKPOINT_END_OF_RECOVERY)
				do_restartpoint = false;

			/* 统计计数与 "checkpoints are occurring too frequently" 告警

			else
				ckpt_start_recptr = GetInsertRecPtr();

			ckpt_cached_elapsed = 0;

				ckpt_performed = CreateCheckPoint(flags);
			else

			smgrdestroyall();

			SpinLockAcquire(&CheckpointerShmem->ckpt_lck);
			CheckpointerShmem->ckpt_done = CheckpointerShmem->ckpt_started;
			SpinLockRelease(&CheckpointerShmem->ckpt_lck);

			ConditionVariableBroadcast(&CheckpointerShmem->done_cv);

			/* last_checkpoint_time 记 *开始* 时刻（注释：使时间驱动的

			ProcessCheckpointerInterrupts();

				break;
		}

		/* 逻辑解码禁用检查 + archive_timeout 检查 + 上报统计（566-577，省略） */

		/* 又有新请求就不睡，直接下一轮 */
		if (((volatile CheckpointerShmemStruct *) CheckpointerShmem)->ckpt_flags)
			continue;

		/* 计算到下一次 checkpoint / xlog switch 的剩余时间作为超时 */
		now = (pg_time_t) time(NULL);
		elapsed_secs = now - last_checkpoint_time;
		if (elapsed_secs >= CheckPointTimeout)
			continue;			/* no sleep for us ... */
		cur_timeout = CheckPointTimeout - elapsed_secs;
		/* archive_timeout 参与取 Min（595-601，省略） */

		(void) WaitLatch(MyLatch,
						 WL_LATCH_SET | WL_TIMEOUT | WL_EXIT_ON_PM_DEATH,
						 cur_timeout * 1000L /* convert to ms */ ,

```

走读要点：

- **标准循环范式的教科书**：ResetLatch → AbsorbSyncRequests（消化攒下的 fsync 请求）→ 处理中断 → 判定触发条件 → 干活 → 算准超时再睡。两个触发源一目了然：**请求驱动**（backend 经 `RequestCheckpoint()`（`checkpointer.c:1064`）写 ckpt_flags + SetLatch：手动 CHECKPOINT、WAL 量超 max_wal_size（CAUSE_XLOG）、建库删库等）与**时间驱动**（checkpoint_timeout）；
- **请求握手协议**：`ckpt_started`/`ckpt_done`/`ckpt_failed` 三个计数 + start_cv/done_cv 两个条件变量构成与等待者（RequestCheckpoint 里想等完成的 backend）的完整握手——backend 记下请求前的 started，等 done 追上它，用 failed 差值判断成败。自旋锁只包住计数操作，广播在锁外；
- **checkpoint 期间的世界没有停**：写脏页由 `BufferSync()` 执行，其每写一页调一次 `CheckpointWriteDelay()`（`checkpointer.c:795`）——若进度超前于 `checkpoint_completion_target` 折算的日程（`IsCheckpointOnSchedule()`，`checkpointer.c:865`）就睡 100ms 摊平 I/O，**睡前照常 AbsorbSyncRequests + 处理 SIGHUP + 上报统计**；即使进度落后不睡，也每 `WRITES_PER_ABSORB` 次写吸一轮请求（`checkpointer.c:840-849`）——否则一场大 checkpoint 写几百万页，队列早溢出了；
- **restartpoint 分支**：恢复中（standby）执行 `CreateRestartPoint`，失败（常因自上次以来没收到新的 checkpoint WAL 记录）不算错误，把 `last_checkpoint_time` 拨到"15 秒后重试"的位置。

**第三、四幕：关库两阶段**（`checkpointer.c:613-658`）。收到 SIGINT（ShutdownXLOGPending）→ 跳出主循环，`ExitOnAnyError = true`（此后任何 ERROR 直接 exit(1)，不再回 sigsetjmp——关库路上没有重试）→ `ShutdownXLOG()` 写 shutdown checkpoint → `SendPostmasterSignal(PMSIGNAL_XLOG_IS_SHUTDOWN)` 报告 → 进入第二个小循环干等 SIGUSR2（期间还能做"越晚越好"的收尾如刷统计）→ `proc_exit(0)`。**把"写完 shutdown checkpoint"与"退出"拆成两步**，是为了让 postmaster 在两者之间还能安排别的收尾次序。

### 12.4.2 fsync 请求转发协议：ForwardSyncRequest / Compact / AbsorbSyncRequests 全走读

背景：脏页可能被任何进程写出（bgwriter、驱逐页面的普通 backend），但 write() 只进页缓存，真正的持久化要 fsync；而 fsync 由 checkpointer 在 checkpoint 时集中执行（为什么？见 12.4.3）。于是谁写了页就要把"这个文件将来要 fsync"记账给 checkpointer。账本是共享内存里的环形队列（`CheckpointerShmem->requests[]`，容量 `Min(NBuffers, MAX_CHECKPOINT_REQUESTS)`，`checkpointer.c:988`——**每个 shared buffer 摊一个槽**，所以正常情况下装得下）。

**生产者 `ForwardSyncRequest()`**（`checkpointer.c:1213`，全函数逐字）：

```c
bool
ForwardSyncRequest(const FileTag *ftag, SyncRequestType type)
{
	CheckpointerRequest *request;
	bool		too_full;
	int			insert_pos;

	if (!IsUnderPostmaster)
		return false;			/* probably shouldn't even get here */

	if (AmCheckpointerProcess())
		elog(ERROR, "ForwardSyncRequest must not be called in checkpointer");

	LWLockAcquire(CheckpointerCommLock, LW_EXCLUSIVE);

	/*
	 * If the checkpointer isn't running or the request queue is full, the
	 * backend will have to perform its own fsync request.  But before forcing
	 * that to happen, we can try to compact the request queue.
	 */
	if (CheckpointerShmem->checkpointer_pid == 0 ||
		(CheckpointerShmem->num_requests >= CheckpointerShmem->max_requests &&
		 !CompactCheckpointerRequestQueue()))
	{
		LWLockRelease(CheckpointerCommLock);
		return false;
	}

	/* OK, insert request */
	insert_pos = CheckpointerShmem->tail;
	request = &CheckpointerShmem->requests[insert_pos];
	request->ftag = *ftag;
	request->type = type;

	CheckpointerShmem->tail = (CheckpointerShmem->tail + 1) % CheckpointerShmem->max_requests;
	CheckpointerShmem->num_requests++;

	/* If queue is more than half full, nudge the checkpointer to empty it */
	too_full = (CheckpointerShmem->num_requests >=
				CheckpointerShmem->max_requests / 2);

	LWLockRelease(CheckpointerCommLock);

	/* ... but not till after we release the lock */
	if (too_full)
	{
		ProcNumber	checkpointerProc = pg_atomic_read_u32(&ProcGlobal->checkpointerProc);

		if (checkpointerProc != INVALID_PROC_NUMBER)
			SetLatch(&GetPGProcByNumber(checkpointerProc)->procLatch);
	}

	return true;
}
```

协议逐条：

1. **入队不去重**（函数头注释 `checkpointer.c:1202-1210`）：为缩短持锁时间，正常路径直接怼进队尾，重复请求留给消费者去重（sync.c 的哈希表天然合并同 key）——**把去重成本从热路径挪到冷路径**；
2. **队满两级降级**：先 `CompactCheckpointerRequestQueue()` 原地压缩；压不出空位（队里全是不重复项——注释说这"vanishingly rare"，因为槽数=NBuffers）才返回 false，调用者（`RegisterSyncRequest`，sync.c）退化为**自己 fsync 这个文件**——慢（可能重复 fsync 同一文件几百次）但正确。注释的警句：队满时"each backend begins doing an fsync for every block written, which gets very expensive and can slow down the whole system"（`checkpointer.c:1272-1277`）——这正是 commit `7f242d880b`（2011，Robert Haas，"Try to avoid running with a full fsync request queue"）引入压缩的动机，配套的 `3134d8863e` 还加了 `buffers_backend_fsync` 统计列让你在监控里看见这种降级；
3. **checkpointer 不在也返回 false**（checkpointer_pid == 0）：crash-restart 的空窗期里 backend 自己兜底 fsync——见 what-if E1；
4. **半满催醒**：阈值设在 1/2 而非全满，给消费者留提前量；**SetLatch 放在锁外**（注释特意标出"but not till after we release the lock"）——被唤醒的 checkpointer 第一件事就是抢这把锁，锁内唤醒等于叫人起床又把门堵住。

**压缩 `CompactCheckpointerRequestQueue()`**（`checkpointer.c:1284`）：持锁遍历环形队列，用进程本地哈希表（key 是整个 request 结构——所以要求结构无随机填充字节，注释 `checkpointer.c:1347-1352`）标记"后面还会再出现的重复项"，再一趟两指针原地压实。方向讲究：**保留最后一次出现、删除前面的**——因为中间可能有 SYNC_FORGET_REQUEST/SYNC_FILTER_REQUEST（撤销类请求，DROP TABLE 时发），倒着保留第一次会改变撤销语义（`checkpointer.c:1328-1338`）。临界区内不做（要 palloc，`checkpointer.c:1306-1308`）。

**消费者 `AbsorbSyncRequests()`**（`checkpointer.c:1424`，全函数逐字）：

```c
void
AbsorbSyncRequests(void)
{
	CheckpointerRequest *requests = NULL;
	CheckpointerRequest *request;
	int			n,
				i;
	bool		loop;

	if (!AmCheckpointerProcess())
		return;

	do
	{
		LWLockAcquire(CheckpointerCommLock, LW_EXCLUSIVE);

		/*---
		 * We try to avoid holding the lock for a long time by:
		 * 1. Copying the request array and processing the requests after
		 *    releasing the lock;
		 * 2. Processing not the whole queue, but only batches of
		 *    CKPT_REQ_BATCH_SIZE at once.
		 *
		 * Once we have cleared the requests from shared memory, we must
		 * PANIC if we then fail to absorb them (e.g., because our hashtable
		 * runs out of memory).  This is because the system cannot run safely
		 * if we are unable to fsync what we have been told to fsync.
		 * ...
		 */
		n = Min(CheckpointerShmem->num_requests, CKPT_REQ_BATCH_SIZE);
		if (n > 0)
		{
			if (!requests)
				requests = (CheckpointerRequest *) palloc(n * sizeof(CheckpointerRequest));

			for (i = 0; i < n; i++)
			{
				requests[i] = CheckpointerShmem->requests[CheckpointerShmem->head];
				CheckpointerShmem->head = (CheckpointerShmem->head + 1) % CheckpointerShmem->max_requests;
			}

			CheckpointerShmem->num_requests -= n;

		}

		START_CRIT_SECTION();

		/* Are there any requests in the queue? If so, keep going. */
		loop = CheckpointerShmem->num_requests != 0;

		LWLockRelease(CheckpointerCommLock);

		for (request = requests; n > 0; request++, n--)
			RememberSyncRequest(&request->ftag, request->type);

		END_CRIT_SECTION();
	} while (loop);

	if (requests)
		pfree(requests);
}
```

三个设计点：① **批处理 + 锁外消化**：每批最多 `CKPT_REQ_BATCH_SIZE`（10000，`checkpointer.c:160`）条拷到本地数组，持锁时间与队长解耦，生产者不会被一次超长 Absorb 饿死；② **临界区包住 RememberSyncRequest**：请求一旦从共享队列摘下，就只存在于本进程内存——此刻若因 OOM 抛 ERROR，这些"必须 fsync"的记账凭空蒸发，系统将来可能宣称"checkpoint 完成"却没同步某文件 → **静默数据丢失**。所以用 START_CRIT_SECTION 把失败升级成 PANIC（crash-restart 重放 WAL，宁可停机不可丢账）；③ `RememberSyncRequest()`（sync.c）把请求归入 per-file 哈希表，**同一文件的一万条块写请求合并成一个"待 fsync"表项**——去重在此免费发生。

`CreateCheckPoint` 在开始 fsync 前也会调 AbsorbSyncRequests（函数头注释 `checkpointer.c:1418-1421`），保证"本 checkpoint 覆盖的写"全部记账在案后才动手 sync——这是恢复正确性的一环。

### 12.4.3 所以然之三：为什么 fsync 集中到 checkpointer，backend 只转发请求

三个角度论证：

1. **fd 与系统调用经济学**：fsync 作用于整个文件。一轮 checkpoint 可能写出某文件的十万个脏页，来自几十个进程——若谁写谁 fsync，同一文件被 fsync 几十次，每次都是全文件页缓存回写的代价；集中到 checkpointer，per-file 哈希表把它们合并成**每文件一次**。同时 fsync 需要打开 fd，checkpointer 的 fd 缓存（smgr/vfd 层）可以长期持有这些文件句柄，backend 驱逐一页就为 fsync 开关一次 fd 是纯浪费；
2. **崩溃语义只需要"checkpoint 前全部落盘"**：WAL 协议的要求是——checkpoint 记录写入前，其覆盖的脏页必须已持久化（否则恢复起点推进了，页却可能丢）。**平时写出的页根本不需要立刻 fsync**：崩溃了自有 WAL 从上一个 checkpoint 重放。所以"何时 fsync"天然就该绑定在"谁推进恢复起点"上——那正是 checkpointer。把 fsync 平摊到每次页写出（像教科书朴素设计那样）纯属过度持久化；
3. **延迟隔离**：fsync 是数十毫秒级的抖动大户（尤其机械盘/繁忙的云盘）。让处理用户查询的 backend 亲自 fsync，等于把存储抖动直接注入查询尾延迟。集中到后台进程，用户路径只剩内存操作（写队列），抖动被 checkpoint 的慢节奏（completion_target 摊平）吸收。

代价与兜底也要看见：集中化引入了队列这个共享瓶颈（锁竞争、队满），于是有 12.4.2 的三件套——不去重入队（快）、满时压缩（省）、实在不行 backend 自己 fsync（正确性永不妥协）。**性能优化全部可降级，正确性路径永远存在**——这是 PG 工程风格的典型样本。

### 12.4.4 bgwriter：与 checkpointer 的分工，及 9.2 分家史

**职责**：在两次 checkpoint 之间**提前**清洗一部分脏页，让普通 backend 驱逐页面时尽量拿到干净页，不必自己写盘。分工表：

| | checkpointer | bgwriter |
|---|---|---|
| 目标 | 正确性:推进恢复起点，保证可恢复 | 性能:降低 backend 亲自写脏页的概率 |
| 写哪些页 | checkpoint 时刻前的**全部**脏页 | 时钟指针**将要扫到**的少量脏页(跟着驱逐压力走) |
| fsync | 执行(消费请求队列) | 不执行(写完转发请求给 checkpointer) |
| 节奏 | checkpoint_timeout / max_wal_size | bgwriter_delay(默认 200ms)+ 反馈控制 |

**所以然之四：为什么要拆成两个进程？** 9.1 及以前它们是同一个进程（就叫 bgwriter）。commit `806a2aee37`（2011-11-01，Simon Riggs，进 9.2）把它劈开，提交信息给了两个理由：一是"beneficial effect on performance at high write rates"，二是（主因）"refactoring to more easily allow changes for power reduction by simplifying **previously tortuous code** required to allow page cleaning and checkpointing to **time slice in the same process**"。展开推演：

- **节奏根本合不到一起**：页清洗要求 200ms 级的细碎心跳（跟住驱逐速率），checkpoint 是持续几分钟、还要按 completion_target 精确掐节奏的长任务。单进程只能在 checkpoint 写页的间隙"顺便"清洗——于是 `CheckpointWriteDelay` 的前身要在一个睡眠点上同时服务两套调度目标，代码被拧成麻花（"tortuous"）；更糟的是 checkpoint 高峰期恰是驱逐压力高峰期，清洗却被 checkpoint 挤占，**最需要 bgwriter 的时刻它最缺席**；
- **分家后各自变简单**：checkpointer 的睡眠改成固定 100ms 的"Big Sleep"（`CheckpointWriteDelay` 注释 `checkpointer.c:830-834` 原话："This sleep used to be connected to bgwriter_delay, typically 200ms... Checkpointer and bgwriter are no longer related so take the Big Sleep"——代码里至今留着分家的墨迹）；bgwriter 则能独立休眠省电（见下）。副产品：关库次序也清晰了——bgwriter 可以早早退，checkpointer 必须活到最后。

`BackgroundWriterMain()`（`src/backend/postmaster/bgwriter.c:89`）主循环：ResetLatch → `ProcessMainLoopInterrupts()`（`interrupt.c:34`，SIGHUP/barrier/shutdown 三件套，简单后台进程共用）→ `BgBufferSync(&wb_context)`（`bgwriter.c:236`，真正的策略在 bufmgr.c：沿 clock-sweep 指针**向前看**，按最近缓冲区分配速率的移动平均估算该预清多少页——一个反馈控制环）→ 上报统计 → `WaitLatch(..., BgWriterDelay, WAIT_EVENT_BGWRITER_MAIN)`（`bgwriter.c:307`）。系统空闲时（BgBufferSync 返回 can_hibernate 且连续两轮无事，`bgwriter.c:329`）睡眠拉长 `HIBERNATE_FACTOR`（50，`bgwriter.c:65`）倍，第一个要驱逐页面的 backend 会 SetLatch 叫醒它——省电与响应两全（这正是分家提交信息里预告的 "power reduction"）。

### 12.4.5 walwriter

**职责**：周期性把 WAL 缓冲刷盘，主要服务两类人：`synchronous_commit = off` 的事务（提交不等 fsync，靠 walwriter 在 wal_writer_delay 内补上持久化，故最多丢一小窗口、不坏一致性）；以及减少高并发下同步提交者亲自 flush 的次数（顺手把别人的也刷了 → 组提交效果）。`WalWriterMain()`（`src/backend/postmaster/walwriter.c:89`）主循环核心一句 `XLogBackgroundFlush()`（`walwriter.c:241`），配熟悉的休眠机制：连续 `LOOPS_UNTIL_HIBERNATE`（50）轮无事可做后睡眠拉长 `HIBERNATE_FACTOR`（25）倍（`walwriter.c:79-80`），异步提交的 backend 发现它在冬眠会 SetLatch 叫醒。

### 12.4.6 autovacuum launcher / worker

两级架构：**launcher**（常驻一个）只做调度，**worker**（动态，≤ autovacuum_max_workers）干活。`AutoVacLauncherMain()`（`src/backend/postmaster/autovacuum.c:411`）主循环片段（`autovacuum.c:626` 起，逐字）：

```c
		launcher_determine_sleep(av_worker_available(), false, &nap);

		/*
		 * Wait until naptime expires or we get some type of signal (all the
		 * signal handlers will wake us by calling SetLatch).
		 */
		(void) WaitLatch(MyLatch,
						 WL_LATCH_SET | WL_TIMEOUT | WL_EXIT_ON_PM_DEATH,
						 (nap.tv_sec * 1000L) + (nap.tv_usec / 1000L),
						 WAIT_EVENT_AUTOVACUUM_MAIN);

		ResetLatch(MyLatch);

		ProcessAutoVacLauncherInterrupts();
```

循环上方注释（`autovacuum.c:619-624`）特意声明它与标准模式**相反**——"we'd like to sleep before the first launch of a child process"，先睡后干，因为希望启动后先歇满一个 naptime 再派 worker（Reset 依然紧贴在 Wait 之后、检查之前，防丢唤醒的不变式未破）。launcher 维护各数据库的上次处理时间表（`rebuild_database_list()`），按 autovacuum_naptime 均匀轮转；到点挑库后经 `do_start_worker()` 请 postmaster fork 一个 worker——**launcher 自己不能 fork**（fork 出的孩子只会继承 launcher 而非 postmaster 的状态，且信号/收尸关系会乱），流程是写共享内存槽 + 发 PMSIGNAL 给 postmaster 代孕。`AutoVacWorkerMain()`（`autovacuum.c:1418`）连上目标库，按"死元组数 > threshold + scale_factor × 表行数"公式挑表执行 VACUUM/ANALYZE，全程受 vacuum_cost 参数限流。worker 是正经的"能跑事务的 backend"。这个"launcher 调度 + postmaster 代孕 + worker 连库"的三角，就是后来 background worker 框架的原型。

### 12.4.7 archiver

**职责**：把写满的 WAL 段安全送去归档（archive_command 或 archive library 回调）。`PgArchiverMain()`（`src/backend/postmaster/pgarch.c:221`）→ `pgarch_MainLoop()`（`pgarch.c:313`）：等 latch（walwriter/backend 切换 WAL 段时通知）或 60 秒自醒 → `pgarch_ArchiverCopyLoop()` 扫 `archive_status` 目录里的 `.ready` 文件逐个归档、成功改名 `.done`。失败重试有退避；`pgarch.c:336` 附近注释的"收到 SIGTERM 后最多再等 60 秒把手头归档完"逻辑体现了归档进程的谨慎——它守护的是备份链完整性。

### 12.4.8 统计系统：PG 15 大重构（所以然之五）

PG 14 及以前有个独立的 **stats collector** 进程：各 backend 把计数打包成 **UDP** 报文发给它，它聚合后周期性写临时文件，读统计的 backend 再去读文件。三宗罪，正是 commit `5891c7a8ed`（2022-04-06，Andres Freund，"pgstat: store statistics in shared memory"，进 PG 15）的动机：

1. **丢包**：UDP 无确认，collector 忙时内核 socket 缓冲一满就丢——统计天生不准，`n_dead_tup` 偏低会直接误导 autovacuum 调度；
2. **延迟与 I/O**：读路径经过"collector 攒批 → 写临时文件 → backend 读文件"，`pg_stat_*` 视图的数据可以陈旧数百毫秒到秒级，statsfile 的反复重写在大库上是可观的 I/O（很多 DBA 被迫把 stats_temp_directory 挪到 tmpfs）；
3. **进程税**：collector 不 attach 共享内存（当年正因如此才选 UDP），于是它是个"体制外"进程——crash 恢复、关库次序、EXEC_BACKEND 都要为它单独立规矩；fork 一个进程只为收计数，在共享内存已经成熟（DSA/dshash 就绪）之后纯属历史包袱。

PG 15 后：**collector 进程消失**，累计统计直接放共享内存——

- 固定对象（bgwriter、checkpointer、WAL、SLRU…）各占静态共享区一块；可变对象（每表、每函数、每库…）放 **dshash**——DSA 上的跨进程哈希表（`src/backend/utils/activity/pgstat_shmem.c:70` 的 dshash_parameters，`StatsShmemInit()` 在 `pgstat_shmem.c:175`）；
- backend 先在本地攒增量，`pgstat_report_stat()`（`pgstat.c:723`）在事务边界且距上次至少 `PGSTAT_MIN_INTERVAL`（1000ms，`pgstat.c:127`）时批量刷入共享内存（拿各条目的 LWLock 累加）——写路径不丢，读路径直接读共享内存，实时且零文件 I/O；
- 分模块文件 `pgstat_relation.c`、`pgstat_database.c`、`pgstat_io.c` 等各管一类；`backend_status.c` 是另一套更轻的"当前状态"设施（pg_stat_activity 的数据源，含 12.2.5 的 wait_event），它一直都在共享内存里，别与累计统计混淆。

对做过监控管道的人，这就是"UDP 上报 + 单点聚合"迁移到"共享内存多写者直更"的经典架构升级：代价是引入并发控制（每条目锁 + 本地攒批限频来摊薄），收益是精确性、实时性，还顺手裁掉一个进程。

### what-if E1：checkpointer 死掉会怎样？

- **谁发现、谁重启**：postmaster 在 `process_pm_child_exit` 里对 checkpointer 有专门分支（`postmaster.c:2389-2419`）：只有"正常退出（exit 0）且正处于 PM_WAIT_CHECKPOINTER 关库阶段"才算善终；**其余任何退出——包括 FATAL（exit 1）——一律 `HandleChildCrash()`**（对比：bgwriter/walwriter 的 exit 0 随时可接受，`postmaster.c:2376-2384`）。HandleChildCrash → `HandleFatalError()`（`postmaster.c:2732`）：对所有孩子群发 SIGQUIT/SIGABRT、置 FatalError、走 PM_WAIT_BACKENDS——**全库 crash-restart**。为什么这么狠？checkpointer 深度读写共享内存（请求队列、buffer 头、WAL 状态），它异常死了共享内存即不可信；且它是唯一的 fsync 记账人，账本状态不明时唯一安全的动作就是推倒重放；
- **in-flight fsync 请求的命运**：三个去向全部安全。(a) 已被 Absorb 进 checkpointer 私有哈希表的：随进程死亡蒸发——没关系，这些文件对应的写在 WAL 里都有记录，且上一个 checkpoint 没把它们算作已持久化，crash-restart 从上一个 checkpoint 重放会重写并重新记账；(b) 还躺在共享内存队列里的：随共享内存重建蒸发——同理由 WAL 重放兜底；(c) 崩溃瞬间正要入队的 backend：它们自己也马上被 SIGQUIT 杀掉了。**正确性从不依赖请求队列不丢，队列只是性能优化；正确性的锚永远是"WAL + 上一个 checkpoint"**；
- **重启空窗**：恢复期间新的 checkpointer 起来前（以及单用户模式），`ForwardSyncRequest` 见 `checkpointer_pid == 0` 返回 false，写页者自己 fsync——慢，但没有窗口期漏账。

### 小结

- CheckpointerMain 四幕：出生（写 pid、无视 SIGTERM、登记退出刷统计、sigsetjmp 里含 ckpt_failed 握手）→ 主循环（请求/时间双触发、started/done/failed+两 CV 的完成握手、CheckpointWriteDelay 掐节奏且睡里偷吸请求）→ 写 shutdown checkpoint → 等 SIGUSR2 退场；
- fsync 转发协议三件套：入队不去重（热路径快）、队满压缩保留末次出现（撤销语义）、再满降级 backend 自己 fsync（正确性不妥协）；Absorb 批处理 + 锁外消化 + 临界区保"账不能丢"（丢账宁可 PANIC）；
- fsync 集中的三角度：per-file 合并 + fd 缓存、崩溃语义只要求 checkpoint 前落盘、把存储抖动隔离出用户路径；
- bgwriter 管性能（预清洗 + 反馈控制 + 休眠），checkpointer 管正确性；9.2 分家（`806a2aee37`）因为两种节奏在单进程里时间切片把代码拧成麻花；
- walwriter 兜底异步提交并助攻组提交；archiver 守护归档链；autovacuum 是 launcher 调度 + postmaster 代孕 + worker 连库的三角；
- PG 15（`5891c7a8ed`）统计进共享内存：治 UDP 丢包、文件中转延迟、多养一个进程三宗罪，代价是并发控制（本地攒批 + 事务边界限频刷入摊薄）。

## 12.5 background worker 框架：把"常驻进程"开放给扩展

### 12.5.1 声明与注册：RegisterBackgroundWorker 走读

`src/backend/postmaster/bgworker.c` 把 12.4 那套"postmaster 管生死"的待遇开放给扩展。声明一个 worker 就是填一个结构（`src/include/postmaster/bgworker.h:96`，逐字）：

```c
typedef struct BackgroundWorker
{
	char		bgw_name[BGW_MAXLEN];
	char		bgw_type[BGW_MAXLEN];
	int			bgw_flags;
	BgWorkerStartTime bgw_start_time;
	int			bgw_restart_time;	/* in seconds, or BGW_NEVER_RESTART */
	char		bgw_library_name[MAXPGPATH];
	char		bgw_function_name[BGW_MAXLEN];
	Datum		bgw_main_arg;
	char		bgw_extra[BGW_EXTRALEN];
	pid_t		bgw_notify_pid; /* SIGUSR1 this backend on start/stop */

```

要点：`bgw_flags` 常填 `BGWORKER_SHMEM_ACCESS | BGWORKER_BACKEND_DATABASE_CONNECTION`（`bgworker.h:53,60`，后者依赖前者）；`bgw_start_time` 声明在启动流程的哪个阶段拉起（PostmasterStart / ConsistentState / RecoveryFinished，`bgworker.h:88`）；入口以"**库名+函数名字符串**"表达而非函数指针——因为注册发生在 postmaster，而函数解析必须发生在 fork 出的 worker 进程里（postmaster 甚至可能没加载那个 .so；EXEC_BACKEND 下两个进程地址空间根本不同），`LookupBackgroundWorkerFunction()`（`bgworker.c:1360`）在 worker 侧完成 dlopen+dlsym。

**静态注册 `RegisterBackgroundWorker()`**（`bgworker.c:962`）主干走读（关键片段逐字）：

看点：**只能在 postmaster 处理 `shared_preload_libraries` 时调用**（即扩展 `_PG_init()` 里），过时不候（shmem init 之后 elog(ERROR)）——因为静态 worker 的花名册 `BackgroundWorkerList` 住在 **PostmasterContext（postmaster 私有内存）**，postmaster 独裁管理，不需要锁；注册失败一律 LOG 后静默返回而非报错——启动阶段一个扩展注册失败不该拖死整个实例。

**动态注册 `RegisterDynamicBackgroundWorker()`**（`bgworker.c:1068`）——普通 backend 运行期申请，机制完全不同：花名册换成**共享内存槽数组** `BackgroundWorkerData->slot[]`（`BackgroundWorkerShmemInit()`，`bgworker.c:198`，槽数 = max_worker_processes）。主干：`BackgroundWorkerLock` 排他锁下找空槽 → 拷入 worker 描述、`slot->generation++`（**代数**：槽会复用，handle=槽号+代数才能唯一指认"我那次注册"，防止查询到复用后的别人）→ `pg_write_barrier()` **然后才置 `in_use = true`**（注释：postmaster 读槽不加锁——它不能加锁，见下——屏障保证它看到 in_use 时内容必已就绪）→ 释放锁 → `SendPostmasterSignal(PMSIGNAL_BACKGROUND_WORKER_CHANGE)` 通知 postmaster 来抄名单。并行查询的 worker 就走这条路（`BGWORKER_CLASS_PARALLEL` 另有 max_parallel_workers 配额检查，`bgworker.c:1103-1112`）。

调用者可持 `BackgroundWorkerHandle`（槽号+代数）跟踪：`GetBackgroundWorkerPid()`（`bgworker.c:1180`）查状态；`WaitForBackgroundWorkerStartup()`（`bgworker.c:1235`）等它起来——内部就是标准 latch 循环（CHECK_FOR_INTERRUPTS → 查槽 → WaitLatch(WL_LATCH_SET | WL_POSTMASTER_DEATH) → ResetLatch），前提是注册时把 `bgw_notify_pid` 设为自己（postmaster 在 worker 启停时会 kill(bgw_notify_pid, SIGUSR1)，经 12.3 的机制 SetLatch 唤醒等待者）。

### 12.5.2 postmaster 侧：从名单到 fork

postmaster 收到 PMSIGNAL_BACKGROUND_WORKER_CHANGE 后由 `BackgroundWorkerStateChange()`（`bgworker.c:272`）把共享内存槽里的新条目抄进自己的 BackgroundWorkerList（**postmaster 不信任共享内存**——它要在共享内存损坏时也能活，所以只在信号驱动下小心翼翼地读一次，拷贝到私有内存再用）；随后 ServerLoop 调 `maybe_start_bgworkers()`（`postmaster.c:4280`）。走读要点（片段逐字）：

- 崩溃恢复期（FatalError）直接清旗返回——重生之后再说；
- 遍历名单，跳过在跑的（rw_pid != 0）、收殓被标记终结的（rw_terminate → `ForgetBackgroundWorker()`，`bgworker.c:455`）；
- **崩溃退避**：

```c
		if (rw->rw_crashed_at != 0)
		{
			if (rw->rw_worker.bgw_restart_time == BGW_NEVER_RESTART)
			{
				int			notify_pid;

				notify_pid = rw->rw_worker.bgw_notify_pid;

				ForgetBackgroundWorker(rw);

				/* Report worker is gone now. */
				if (notify_pid != 0)
					kill(notify_pid, SIGUSR1);

				continue;
			}

			/* read system time only when needed */
			if (now == 0)
				now = GetCurrentTimestamp();

			if (!TimestampDifferenceExceeds(rw->rw_crashed_at, now,
											rw->rw_worker.bgw_restart_time * 1000))
			{
				/* Set flag to remember that we have workers to start later */
				HaveCrashedWorker = true;
				continue;
			}
		}
```

  BGW_NEVER_RESTART 的直接除名并通知关心者；否则距上次崩溃不足 `bgw_restart_time` 秒就先记账（HaveCrashedWorker），postmaster 主循环会按最近的到期时间安排下次唤醒；
- 阶段闸门 `bgworker_should_start_now()`（`postmaster.c:4234`）：一个利用 **fallthrough** 的 switch——pmState 越"晚"（PM_RUN > PM_HOT_STANDBY > PM_RECOVERY/STARTUP），允许的 start_time 集合越大：`BgWorkerStart_PostmasterStart` 几乎任何早期状态都放行，`BgWorkerStart_RecoveryFinished` 只在 PM_RUN 放行——想连库跑事务的 worker 必须等库真正开门；
- 每次至多启动 `MAX_BGWORKERS_TO_LAUNCH`（100）个，超了置 StartWorkerNeeded 下轮再来——**postmaster 的注意力是稀缺资源**，它还要 accept 新连接。

`StartBackgroundWorker()`（`postmaster.c:4173`）：`AssignPostmasterChildSlot(B_BG_WORKER)` 占子进程槽（失败按崩溃对待、退避后重试——"if we tried again right away, most likely we'd find ourselves hitting the same resource-exhaustion condition"）→ `postmaster_child_launch(B_BG_WORKER, ..., &rw->rw_worker, sizeof(BackgroundWorker), ...)` fork 并把 BackgroundWorker 结构作为 startup_data 传给孩子 → 记 pid、`ReportBackgroundWorkerPID()` 通知 notify_pid。

fork 落地后进入 `BackgroundWorkerMain()`（`bgworker.c:741`），六步（关键片段逐字）：


② **信号表按权限定制**：会连库的 worker 拿到与普通 backend 相同的取消/procsignal 待遇；不连库的对 SIGINT/SIGUSR1 装聋。SIGTERM 一律接 `die`（12.3 的协作式退出）——所以 `pg_terminate_backend()` 对 bgworker 同样有效，前提是它的循环里有 CFI 或 WaitLatch。

③ **sigsetjmp 恢复块**（`bgworker.c:807-833`）：与 backend 不同，worker 的块**不清理后重来，而是报告错误后 `proc_exit(1)`**——"We just need to clean up, report the error, and go away"。恢复期先 HOLD_INTERRUPTS 再解除信号屏蔽（注释：可能还要与并行 leader 通信）。exit(1) 接到 12.5.4 的重启规则——**扩展代码里未捕获的 ERROR = 死给 postmaster 看，按 bgw_restart_time 重生**，框架替扩展作者兜住了"异常处理忘写"的底。

④ **`InitProcess()`**：领 PGPROC（从此有 MyLatch 的共享版、能用 LWLock）；`BaseInit()` 完成基础设施。

⑤ **解析入口**：`LookupBackgroundWorkerFunction(bgw_library_name, bgw_function_name)`——此刻才 dlopen/dlsym，呼应 12.5.1 的"字符串表达入口"。

⑥ **移交控制权**：`entrypt(worker->bgw_main_arg); proc_exit(0);`——入口函数正常返回视同"功成身退"（exit 0 = 不再重启）。注释（`bgworker.c:855-860`）点明框架与 backend 的最后一点差别：InitPostgres 没有替你调——**连不连库、连哪个库是 worker 自己的事**，用 `BackgroundWorkerInitializeConnection()`（`bgworker.c:875`，校验 bgw_flags 后调 InitPostgres，一进程一次、不可换库）。入口函数的标准开场因此是：

```c
BackgroundWorkerUnblockSignals();                 /* bgworker.c:949，解除 fork 时的屏蔽 */
BackgroundWorkerInitializeConnection("mydb", NULL, 0);  /* 需要连库时 */
for (;;)                                          /* 12.2 的标准主循环 */
{
    ResetLatch(MyLatch);
    CHECK_FOR_INTERRUPTS();
    ... 干活，可用 SPI 跑 SQL（自己负责 StartTransactionCommand/CommitTransactionCommand）...
    WaitLatch(MyLatch, WL_LATCH_SET | WL_TIMEOUT | WL_EXIT_ON_PM_DEATH, ms, 事件);
}
```

### 12.5.4 what-if E2：一个 bgworker 崩溃，波及范围有多大？

规则全在 `CleanupBackend()`（`postmaster.c:2588`）里，按退出码三分：

- **exit(0)**：善终。`rw->rw_terminate = true`，下轮 maybe_start_bgworkers 除名——**不再重启**（哪怕设了 restart_time）；
- **exit(1)**（含任何 ereport(FATAL/ERROR) 经 sigsetjmp 块的退出）：视同"可预期失败"。`rw->rw_crashed_at = GetCurrentTimestamp()`、置 HaveCrashedWorker，按 `bgw_restart_time` 退避后重启（BGW_NEVER_RESTART 则除名）。**波及面：零**——其他进程完全无感（除了 bgw_notify_pid 收到一发 SIGUSR1）；
- **其他退出码或死于信号**（段错误、abort、OOM killer…）：判定 `crashed = true` → `HandleChildCrash()` → **全库 crash-restart**——所有 backend/后台进程吃 SIGQUIT，共享内存重建，WAL 重放。判据的道理与 12.3.2 同源：exit(1) 是走完 proc_exit 回调链（释放锁、detach 共享内存）的**有序**退出，共享内存可信；死于 SIGSEGV 的进程可能在任何共享结构上留下半更新，**没有任何办法证明共享内存仍然完好**，唯一安全动作是全体重来。此外 `ReleasePostmasterChildSlot()` 返回 false（子进程没有干净地登出共享内存记账）时同样按 crash 处理（`postmaster.c:2646-2653`）——一道独立的"死后验尸"。

推论：**写 bgworker 时"挂"也要挂得体面**——用 ereport(ERROR/FATAL) 让 sigsetjmp 块送你走 exit(1)，波及为零；任何真正的 C 层崩溃则是拉全库陪葬。这也是"判断一个扩展是否'重'，看它有没有 worker"的深层含义：有 worker 的扩展拿到了与内核进程同等的权利，也背上了同等的义务（bgw_flags 里的 SHMEM_ACCESS 不是白给的）。

崩溃重启之后：`ResetBackgroundWorkerCrashTimes()`（CleanupBackend 注释提到）会重置静态 worker 的状态，restart_after_crash 流程走完后它们随新世界一起重生。

### 12.5.5 案例：pg_cron 类调度器怎么用它

社区扩展 pg_cron（数据库内的 crontab）是教科书用法：

1. `shared_preload_libraries = 'pg_cron'` → `_PG_init()` 里 `RegisterBackgroundWorker()` 注册一个 launcher worker，`bgw_start_time = BgWorkerStart_RecoveryFinished`（主库恢复完成才启动）；
2. launcher 主循环：连上元数据库读 `cron.job` 表 → `WaitLatch` 睡到最近的触发点 → 到点后为每个任务 `RegisterDynamicBackgroundWorker()` 派生执行 worker（或走 libpq 自连，两种模式它都支持）；
3. 执行 worker 连上目标库，在事务里跑任务 SQL，把结果写回 `cron.job_run_details`，退出（exit 0）；
4. 配置变更无需轮询：普通 backend 里执行 `cron.schedule()` 的 SQL 函数改完表后，对 launcher `SetLatch`/发信号即可即时生效——又回到了 12.2 的原语。

同一框架的其他住户：逻辑复制的 launcher/apply worker（`replication/logical/launcher.c`，内核自己也用这套 API——12.3.3 里 ProcessInterrupts 对 IsLogicalLauncher 的特判即为此）、pg_prewarm 的 autoprewarm、TimescaleDB 的 job scheduler、Citus 的 maintenance daemon。

### 小结

- BackgroundWorker 结构声明名字/权限/启动阶段/重启策略/入口（**库名+函数名字符串**，因为解析必须在 worker 进程内发生）；
- 静态注册限于 shared_preload_libraries 阶段，名册在 postmaster 私有内存；动态注册走共享内存槽（代数防复用混淆、写屏障配合 postmaster 的无锁读）+ PMSIGNAL 通知，handle 可等启动/停止；
- postmaster 侧：抄名单 → 崩溃退避 → pmState 阶段闸门（fallthrough switch）→ 限量 fork → 传 startup_data；worker 侧：接户口 → 按权限装信号表 → sigsetjmp（错误即 exit(1)）→ InitProcess → dlsym 入口 → 移交；连库自理；
- 崩溃波及三分法：exit 0 除名 / exit 1 按 restart_time 独自重启（波及零）/ 其他 = 共享内存不可信 = 全库 crash-restart；
- pg_cron 模式 = 静态 launcher + 动态执行 worker + SetLatch 即时唤醒，是编写"数据库内守护程序"的标准范式。

## 12.6 不变式清单

每条给出"是什么、代码在哪守护、违反后果"。

**I1. 共享内存结构单次初始化**：每个命名结构只能被初始化一次；后来者必须走附着路径。守护者：ShmemIndex 上的 get-or-create（`ShmemInitStruct` 的 `*foundPtr`）、`InitShmemIndexEntry` 对重名的 `elog(ERROR)`（`shmem.c:522-523`）、附着时的尺寸核对（`shmem.c:598-605`）、`shmem_request_state` 状态机断言。违反后果：忽略 `found` 重复初始化 = 把其他进程正在用的共享状态清零（锁表被清 = 所有已持有的锁凭空消失，数据结构互踩）；尺寸不核对 = 新旧版本扩展对同一块内存按不同布局读写，越界踩到邻居结构，产生与本模块毫无关系的"灵异"崩溃。

**I2. Latch 等待循环的标准顺序**：消费者必须 `ResetLatch → 检查条件 → WaitLatch`（先重置再检查）；生产者必须"先写共享数据、后 SetLatch"。守护者：无法由编译器强制，靠惯例 + `SetLatch` 注释（`latch.c:332-337`）明文托付 + 两侧成对的内存屏障。违反后果：12.2.4 的两个反证——消费者把检查放在 Reset 前，或生产者先 Set 后写数据，都能构造出"活躺在共享内存里、进程睡死"的丢唤醒时序；这类 bug 只在特定线程交错下出现，复现概率千万分之一，是最难调试的一类。

**I3. 每个受管等待点必须声明对 postmaster 死亡的态度**：守护者：`WaitLatch` 开头的 Assert（`latch.c:178-180`）。违反后果：postmaster 死后该进程成为孤儿继续读写共享内存；此时如果有人手工重启一个新实例（新 postmaster 判定无人使用后重建共享内存/或接管数据目录），新旧两个世界同时写同一份数据文件——数据损坏的最快捷径。

**I4. 信号处理器只置标志（+SetLatch），绝不做实事**：守护者：全体 handler 的代码纪律（`StatementCancelHandler`/`die`/procsignal handler 无一例外）。违反后果：handler 里调用任何非 async-signal-safe 函数（palloc/elog/LWLockAcquire）都可能撞上"主流程正在同一结构上操作到一半"——malloc 元数据损坏、自旋锁自死锁（handler 等主流程持有的锁，主流程等 handler 返回）。

**I5. 中断处理的可重入与延迟安全**：ProcessInterrupts 必须可以在任何 CFI 点安全调用，且在 Holdoff/临界区内绝对不动作、标志不丢。守护者：函数第一行的双计数检查（`postgres.c:3455-3456`，此时不清 InterruptPending）；`QueryCancelHoldoffCount` 下的重新武装（`postgres.c:3573`）；sigsetjmp 恢复期的信号屏蔽 + HOLD_INTERRUPTS。违反后果：临界区内抛 ERROR 被升级 PANIC（整库重启）；读客户端消息中途抛错 = FE/BE 协议失步，此后该连接收发全部错位；恢复期被再打断 = 错误恢复的恢复，elog 状态栈溢出。

**I6. fsync 账本只可靠转移、不可丢失**：请求从共享队列摘下到进入 sync.c 哈希表之间不允许失败逃逸。守护者：`AbsorbSyncRequests` 的 START_CRIT_SECTION（失败即 PANIC，`checkpointer.c:1446-1451` 注释）；压缩只删"后面还有同款"的项；队满降级路径保证请求要么入队要么当场自己 fsync。违反后果：某文件的"待 fsync"记账丢失 → checkpoint 宣称完成但该文件从未 sync → 掉电后页面丢失而恢复起点已推进 → **静默数据损坏**（最恶性的一类：不崩溃、不报错，只是数据悄悄没了）。

**I7. 共享内存写坏 = 全体重来，无中间态**：任何 attach 共享内存的进程异常死亡（非 exit 0/1）都触发全库 crash-restart。守护者：`CleanupBackend` 的退出码三分法、checkpointer 的更严规则（连 exit 1 也算 crash）、`ReleasePostmasterChildSlot` 的死后验尸。违反后果（假如不这样做）：带着可能半更新的锁表/缓冲区头继续运行，错误会以无法归因的方式在任意其他进程爆发。

## 自测问题

**1. 为什么修改 shared_buffers 必须重启实例，而并行查询却能运行时拿到几百 MB 共享内存？**

静态共享内存段在 postmaster 启动时由 `CalculateShmemSize()`（ipci.c:57）汇总各子系统需求后一次性创建（`CreateSharedMemoryAndSemaphores()`，ipci.c:120），分配器只进不出，尺寸不可变——shared_buffers 是其中最大的一项（还连带决定 fsync 请求队列等派生结构的大小，checkpointer.c:988）。并行查询用第二层机制：运行期 `dsm_create()`（dsm.c:524）创建独立动态段，用完销毁；跨进程 malloc 语义由 DSA（dsa.c）在 DSM 段之上提供，以 dsa_pointer（段+偏移）代替裸指针，因为各进程 attach 地址可能不同。

**2. 两个进程并发首次调用 `ShmemInitStruct("X", size, &found)`，为什么不会重复分配？调用者拿到 found=false/true 各该做什么？**

函数全程持 `ShmemIndexLock` 排他锁（shmem.c:1027），"查表（AttachShmemIndexEntry）+ 建表（InitShmemIndexEntry）"是原子的：先进锁的走首建路径（分配 + 登记 ShmemIndex），后进的必然 HASH_FIND 命中走附着路径拿到同一地址。found=false 表示内存是生的，调用者**必须**随即初始化其内容（老接口没有 init 回调，shmem.c:117-119 注释明言）；found=true 表示已有人初始化，直接用——若忽略 found 无条件重新初始化，等于把别人正在用的共享状态清零（不变式 I1）。附着路径还会核对尺寸（shmem.c:598），新旧代码结构定义不一致时报错而不是静默越界。

**3. 后台进程主循环为什么必须"先 ResetLatch 再检查再 WaitLatch"？分别构造消费者侧和生产者侧顺序颠倒时的丢唤醒时序。**

Latch 的防丢失语义依赖顺序。消费者颠倒（先检查后 Reset）：生产者在"检查完、Reset 前"完成"写标志+SetLatch"——标志错过了本轮检查，置位又被紧随的 Reset 清掉，WaitLatch 睡死，活无人干。生产者颠倒（先 SetLatch 后写数据）：消费者被唤醒时数据未到（无屏障保证顺序），检查落空睡回去，而数据写完后不再有第二次 Set。正确顺序下四种 Set 落点（Reset 前/检查窗口/睡前立牌窗口/睡眠中）分别由 is_set 持久位、WaitEventSetWait 开头复查、maybe_sleeping 立牌后复查+补信号、内核唤醒兜住（waiteventset.c:1103-1134），最多空转一圈绝不丢（12.2.4 完整论证）。

**4. `SetLatch` 里两个 `pg_memory_barrier()` 各防什么？`maybe_sleeping` 是什么优化？**

第一道屏障（latch.c:303）保证调用者在 SetLatch 之前写的共享标志先于 is_set 对其他 CPU 可见——否则等待者醒来读标志可能读到旧值，空转睡回后唤醒已被消费，活丢了；它与 ResetLatch 里的屏障（latch.c:388，保证 is_set 清零先于后续标志读取）构成成对握手。第二道屏障（latch.c:311）隔开"置 is_set"与"读 maybe_sleeping"，防止乱序导致双方互相错过（我先读了你的旧 maybe_sleeping=false，你随后立牌睡下，而我的 is_set 你立牌前的复查又没看到）。maybe_sleeping 是 WaitEventSetWait 睡前立的牌（waiteventset.c:1106）：没立牌说明对方醒着、迟早自查标志，SetLatch 就省掉 kill/写管道的系统调用，把"唤醒醒着的人"降为纯内存操作。

**5. 描述 `pg_cancel_backend(pid)` 之后，目标 backend 上一条长查询被取消的完整路径；再列举三种"取消不生效"的原因。**

发送方对目标发 SIGINT；目标的 `StatementCancelHandler()`（postgres.c:3155）在 handler 里仅置 QueryCancelPending+InterruptPending 并 SetLatch。目标执行到下一个 `CHECK_FOR_INTERRUPTS()`（miscadmin.h:125）进入 `ProcessInterrupts()`（postgres.c:3452）：过 Holdoff/临界区安全门后，对 QueryCancelPending 先用 get_timeout_indicator 认定罪名（用户取消/lock_timeout/statement_timeout 报不同文案），然后 `ereport(ERROR)` 长跳转到 postgres.c:4573 的 sigsetjmp 块，经 AbortCurrentTransaction 释放锁与资源、重置内存上下文，回主循环。不生效的原因：①目标在无 CFI 的 C 长循环里，标志无人查；②处在 HOLD_INTERRUPTS/临界区（安全门直接 return，延迟处理）；③正读客户端消息（QueryCancelHoldoffCount≠0，防协议失步，重挂 InterruptPending 稍后处理）；④卡在不可中断内核态（D 状态 I/O）。前三种标志未丢只是延迟，最后一种要等内核返回。

**6. bgwriter 和 checkpointer 都写脏页，为什么 fsync 只让 checkpointer 做？请求怎么到它手里，队列满了怎么办？**

三角度：fsync 按文件生效，集中后 per-file 哈希表把万条块请求合并为每文件一次，且 checkpointer 长期缓存这些 fd；恢复正确性只要求"checkpoint 完成前其覆盖的脏页已持久化"，平时写出的页由 WAL 兜底、无需立刻 fsync，所以 fsync 天然绑定在推进恢复起点的进程上；集中还把存储抖动隔离出用户查询路径。转发：写页者调 `ForwardSyncRequest()`（checkpointer.c:1213）把 (FileTag, type) 怼进共享内存环形队列（容量=NBuffers，不去重以缩短持锁），半满时锁外 SetLatch 催醒；checkpointer 在主循环和 CheckpointWriteDelay 里反复 `AbsorbSyncRequests()`（checkpointer.c:1424）批量取走、临界区内记入 sync.c 哈希表（此处失败即 PANIC——账不能丢，不变式 I6）。队满先 `CompactCheckpointerRequestQueue()`（checkpointer.c:1284）去重（保留末次出现以保 FORGET/FILTER 撤销语义）；仍满则返回 false，调用者自己 fsync 该文件——性能降级、正确性不降（commit 7f242d880b 的由来）。

**7. checkpointer 进程被 OOM killer 杀掉（SIGKILL）会发生什么？环形队列里没消化的 fsync 请求会丢吗？**

postmaster 收尸时发现 checkpointer 非"关库阶段 exit 0"退出（postmaster.c:2389-2416：checkpointer 连 FATAL/exit 1 都算 crash），走 HandleChildCrash→HandleFatalError：全体子进程吃 SIGQUIT，共享内存重建，从上一个 checkpoint 起重放 WAL——全库 crash-restart，然后新 checkpointer 随新世界启动。队列里的请求"丢了但无所谓"：这些写在 WAL 里都有记录，且未被算进任何已完成的 checkpoint，重放会重写这些页并重新记账——正确性锚在"WAL+上一个 checkpoint"，请求队列只是性能优化。另外空窗期里（checkpointer_pid==0）ForwardSyncRequest 返回 false，写页的进程自己 fsync 兜底。

**8. 一个 bgworker 分别以 exit(0)、ereport(FATAL)、段错误三种方式结束，postmaster 各作何反应？为什么判据是退出码而不是 restart 配置？**

exit(0)：善终，rw_terminate=true 除名，**永不重启**（哪怕配置了 restart_time）。ereport(FATAL)（或任何未捕获 ERROR 经 BackgroundWorkerMain 的 sigsetjmp 块）→ proc_exit(1)：记 rw_crashed_at，按 bgw_restart_time 退避后由 maybe_start_bgworkers 重启（BGW_NEVER_RESTART 则除名），**其他进程零波及**。段错误（非 0/1 退出码或死于信号）：CleanupBackend 判 crashed（postmaster.c:2616-2617）→ HandleChildCrash → 全库 crash-restart。判据是退出码而非配置，因为它区分的不是"要不要重启这个 worker"而是"**共享内存还可不可信**"：exit(1) 走完了 proc_exit 回调链（释放锁、干净 detach），死于信号的进程可能在任何共享结构上留下半更新，无法证明无害，只能全体重来（不变式 I7）。

**9. PG 15 为什么废除 stats collector 进程改用共享内存？各解决什么问题，代价是什么？**

三宗罪（commit 5891c7a8ed）：UDP 无确认，collector 忙时丢包，统计天生不准（n_dead_tup 偏低误导 autovacuum）；"collector 攒批→写临时文件→backend 读文件"链路使 pg_stat_* 陈旧且产生持续文件 I/O；collector 不 attach 共享内存，是关库/崩溃流程里的体制外特例，多养一个进程纯为收计数。新方案：固定对象进静态共享区，可变对象进 dshash（DSA 上的跨进程哈希表，pgstat_shmem.c:70/175）；backend 本地攒增量、事务边界且距上次 ≥1000ms（PGSTAT_MIN_INTERVAL，pgstat.c:127）才拿条目锁批量刷入（pgstat_report_stat，pgstat.c:723）。代价：引入多写者并发控制（每条目 LWLock），靠攒批限频摊薄；收益：不丢、实时、零中转文件、少一个进程。

**10. 一个扩展想要"每 10 分钟在库里跑一段 SQL"的常驻能力，需要用到本章哪些机制？各自负责什么？**

background worker 框架管生命周期：`_PG_init()` 里 `RegisterBackgroundWorker()`（bgworker.c:962，仅限 shared_preload_libraries 阶段），bgw_flags 填 SHMEM_ACCESS|DATABASE_CONNECTION，bgw_start_time 选 RecoveryFinished，bgw_restart_time 定崩溃重启退避；postmaster 经 maybe_start_bgworkers/StartBackgroundWorker（postmaster.c:4280/4173）fork 并监管；`BackgroundWorkerMain()`（bgworker.c:741）装信号表/sigsetjmp/解析入口后移交。worker 内：`BackgroundWorkerUnblockSignals()` + `BackgroundWorkerInitializeConnection()`（bgworker.c:875）建会话，SPI 在自管事务里跑 SQL。节拍由 Latch 管：WaitLatch(WL_LATCH_SET|WL_TIMEOUT|WL_EXIT_ON_PM_DEATH, 600000, 自定义事件)——TIMEOUT 定周期、LATCH_SET 支持外部 SetLatch 即时触发、EXIT_ON_PM_DEATH 防孤儿（不变式 I3）；循环遵守 Reset→检查→Wait（I2）。CHECK_FOR_INTERRUPTS 使其可被 pg_terminate_backend 优雅终止（SIGTERM→die→ProcDiePending→FATAL→exit(1)）；等待事件在 wait_event_names.txt 体系登记（或 WaitEventExtensionNew）即可在 pg_stat_activity 里观测。
