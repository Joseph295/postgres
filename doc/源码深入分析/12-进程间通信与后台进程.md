# 12 · 进程间通信与后台进程

> 对应《PostgreSQL 源码学习指南》第 14 章。
> 所有 `文件:行号` 引用均已在本地源码树（`/Users/xujunhong/postgres`，20devel）核实。

## 本章导读

PG 选择了多进程模型（每连接一个 backend + 一群后台常驻进程），于是三个问题必须回答：

1. **进程间怎么共享数据？** ——启动时一次性划好的静态共享内存，加上按需伸缩的动态共享内存（DSM/DSA）；
2. **进程间怎么互相等待与唤醒？** ——Latch + WaitEventSet 把"等 socket、等定时器、等同伴戳我一下"统一成一个事件循环；信号 + CHECK_FOR_INTERRUPTS + ProcSignal 解决"异步打断一个正在干活的进程"；
3. **谁在后台干活？** ——checkpointer、bgwriter、walwriter、autovacuum launcher、archiver、统计系统各司其职；background worker 框架则把"写一个数据库内的常驻进程"开放给扩展。

读完本章，`ps aux | grep postgres` 里每一行、`pg_stat_activity.wait_event` 里每个名字，你都能指出它对应的源码位置。

## 前置知识

- 多进程模型基础：postmaster fork 出所有子进程，子进程天然继承共享内存映射（指南第 1 章）；
- 上一篇文档的 MemoryContext（本章的共享内存明确**不**归它管）与失效队列（sinvaladt 正是共享内存环形队列的一个实例）；
- 对照物：MySQL 是单进程多线程，"共享"是默认状态，难点在加锁；PG 反过来，隔离是默认状态，难点在**显式共享**。你做过的多进程大数据组件（如 Presto 的进程间 shuffle、共享内存的本地缓存）经验在这里非常对口。

## 12.1 共享内存布局

### 12.1.1 静态共享内存：启动时一次划定

postmaster 启动时调用 `CreateSharedMemoryAndSemaphores()`（`src/backend/storage/ipc/ipci.c:120`）：

```c
void
CreateSharedMemoryAndSemaphores(void)
{
	...
	Assert(!IsUnderPostmaster);          /* 只有 postmaster 才能创建 */

	/* 1. 汇总所有子系统的内存需求 */
	size = CalculateShmemSize();         /* ipci.c:57 */

	/* 2. 一次性创建整个共享内存段（mmap/SysV，可用 huge pages） */
	seghdr = PGSharedMemoryCreate(size, &shim);

	/* 3. 在这块内存上初始化简易分配器 */
	InitShmemAllocator(seghdr);

	/* 4. 按注册清单初始化各子系统的共享结构 */
	ShmemInitRequested();

	/* 5. 动态共享内存设施自身的准备 */
	dsm_postmaster_startup(shim);

	/* 6. 给扩展留的钩子（pg_stat_statements 等在此分配自己的共享区） */
	if (shmem_startup_hook)
		shmem_startup_hook();
}
```

初始化清单是什么？本版本（20devel）刚完成一次重构：各子系统通过回调注册需求，清单集中在 `src/include/storage/subsystemlist.h`（`RegisterBuiltinShmemCallbacks()`，`ipci.c:167`）。名单读一遍就是共享内存的"户口本"（节选，subsystemlist.h:28-81）：LWLock、DSM/DSM registry、XLOG（WAL 缓冲）、CLOG/CommitTs/SUBTRANS/MultiXact（SLRU 们）、**BufferManager（大头：shared_buffers）**、锁表与谓词锁、ProcGlobal/ProcArray、BackendStatus、TwoPhase、BackgroundWorker、**SharedInval（上一篇的失效队列）**、PMSignal/ProcSignal、Checkpointer、AutoVacuum、复制槽/Origin/WalSnd/WalRcv、PgArch、ApplyLauncher、BTree/SyncScan/Async（NOTIFY 队列）、**Stats（PG 15 起的统计共享区）**、自定义等待事件。

关键性质：**尺寸在启动时定死**。所以改 `shared_buffers`、`max_connections`、`max_locks_per_transaction` 都要重启——每一项都进了 `CalculateShmemSize()` 的公式。

### 12.1.2 ShmemInitStruct 与 ShmemIndex：按名字认领地盘

一块裸内存怎么让几十个子系统、几百个进程都能找到自己的结构？答案是一个存放在共享内存里的哈希表 **ShmemIndex**：名字（字符串）→（偏移量，尺寸）。每个子系统用 `ShmemInitStruct()`（`src/backend/storage/ipc/shmem.c:1013`）认领：

```c
void *
ShmemInitStruct(const char *name, Size size, bool *foundPtr)
{
	...
	LWLockAcquire(ShmemIndexLock, LW_EXCLUSIVE);

	*foundPtr = false;
	if (IsUnderPostmaster)
		*foundPtr = AttachShmemIndexEntry(&request, true);  /* 子进程：查表附着 */

	if (!*foundPtr)
		InitShmemIndexEntry(&request);       /* 首次（postmaster）：分配并登记 */

	LWLockRelease(ShmemIndexLock);
	...
	return ptr;
}
```

语义是"**get or create**"：postmaster 首次调用真正分配（`InitShmemIndexEntry()`，`shmem.c:511`），fork 出的子进程再调用同名会拿到 `found = true` 和同一个指针——因为共享内存映射地址被继承，指针在所有进程中数值相同，可以直接存指针而不必存偏移。`pg_get_shmem_allocations()`（同文件，SQL 可查 `pg_shmem_allocations` 视图）能把这张账本打印出来，运维排查"共享内存花在哪"的利器。

分配器本身是**只进不出**的指针递增分配（无 free）——静态区的结构与实例共存亡，不需要回收，简单即正确。

### 12.1.3 动态共享内存：dsm.c 与 DSA

静态区应付不了"并行查询临时要 512MB 哈希表"这类需求，于是有第二层：

- **dsm.c**（`src/backend/storage/ipc/dsm.c`）：运行期创建/销毁整段共享内存。`dsm_create()`（`dsm.c:524`）创建一段并返回句柄（handle，一个可跨进程传递的整数），其他进程 `dsm_attach()`（`dsm.c:673`）映射进来。底层实现可选 POSIX shm / SysV / Windows / mmap 文件（`src/include/storage/dsm_impl.h:17-20`，Linux 默认 POSIX）。注意：**各进程 attach 到的虚拟地址可能不同**，段内不能存裸指针，只能存相对偏移；
- **DSA**（dynamic shared area，`src/backend/utils/mmgr/dsa.c`）：在若干 DSM 段之上做的**跨进程 malloc**。`dsa_create_ext()`（`dsa.c:421`）建区，`dsa_attach()`（`dsa.c:510`）加入，`dsa_allocate_extended()`（`dsa.c:683`）/`dsa_free()` 分配释放。因为地址不统一，它发明了 `dsa_pointer`（段号+段内偏移的编码），解引用要走 `dsa_get_address()`。区会随需求自动创建更大的新段（也是指数增长），有上限可控。

典型消费者：并行查询（leader 建 DSM 段传计划与参数，worker attach；元组经 `shm_mq` 共享内存队列回流）、并行哈希连接的共享哈希表、以及 12.6 的统计系统（dshash——DSA 上的跨进程哈希表）。

> 对照：静态区像 Spark executor 启动时的固定 memory pool，DSA 像 task 运行期找 MemoryManager 动态要的执行内存；`dsa_pointer` 的"句柄代替指针"就是共享内存编程的通用税。

### 小结

- 静态共享内存启动时按 `CalculateShmemSize()` 一次划定（故相关 GUC 要重启），子系统清单见 subsystemlist.h，认领机制是共享哈希表 ShmemIndex 上的 `ShmemInitStruct(name)`（get-or-create）；
- 指针可直接共享（地址继承自 fork）；静态分配器只进不出；
- 动态层 dsm.c 管段的生命周期（attach 地址不保证一致→偏移代指针），DSA 在其上提供跨进程 malloc 与 dsa_pointer。

## 12.2 Latch 与 WaitEventSet：统一的等待/唤醒原语

### 12.2.1 要解决的问题

一个 backend/后台进程经常要**同时**等多件事：socket 可读（客户端发来请求）、超时（该做周期工作了）、别的进程唤醒（"有 fsync 请求给你了"）、postmaster 死亡（父进程没了要自杀）。经典 Unix 里这几件事机制各异（select、alarm、signal），而信号打断 select 又有著名的丢失唤醒竞态。PG 把它们统一成两个抽象：

- **Latch**（门闩）：一个可以被其他进程/信号处理器安全地 `SetLatch()` 的一位状态。每个进程在 PGPROC 里有一个共享 Latch（`MyLatch`）；
- **WaitEventSet**（`src/backend/storage/ipc/waiteventset.c`）：一组等待条件（latch、socket、postmaster 死亡）的容器，一次系统调用等待其中任意一个。

平台后端在编译期选定（`waiteventset.c:88-98`）：Linux 用 **epoll**、macOS/BSD 用 **kqueue**、其余用 poll、Windows 用 WaitForMultipleObjects。"latch 被 set"如何变成 fd 事件？两种实现（`waiteventset.c:107-113`）：有 signalfd 的 Linux 把 SIGURG 导进 signalfd；否则用**自唤醒管道（self-pipe）**——信号处理器往管道写一个字节，epoll/kqueue 就能像等 socket 一样等它（kqueue 则可直接用 EVFILT_SIGNAL）。这是教科书级的 self-pipe trick 工业实现。

### 12.2.2 WaitLatch 与典型循环模式

`WaitLatch()`（`src/backend/storage/ipc/latch.c:172`，节选）：

```c
int
WaitLatch(Latch *latch, int wakeEvents, long timeout,
		  uint32 wait_event_info)
{
	WaitEvent	event;

	/* 受 postmaster 管的进程必须声明如何应对 postmaster 死亡 */
	Assert(!IsUnderPostmaster ||
		   (wakeEvents & WL_EXIT_ON_PM_DEATH) ||
		   (wakeEvents & WL_POSTMASTER_DEATH));

	...
	/* 复用进程常驻的 LatchWaitSet，避免每次重建 epoll 实例 */
	ModifyWaitEvent(LatchWaitSet, LatchWaitSetLatchPos, WL_LATCH_SET, latch);
	...
	if (WaitEventSetWait(LatchWaitSet,
						 (wakeEvents & WL_TIMEOUT) ? timeout : -1,
						 &event, 1,
						 wait_event_info) == 0)
		return WL_TIMEOUT;
	else
		return event.events;
}
```

真正的等待在 `WaitEventSetWait()`（`waiteventset.c:1040`）。所有后台进程的主循环都是同一个模式（**先干活后睡觉，醒来先复位**）：

```c
for (;;)
{
    ResetLatch(MyLatch);          /* 复位门闩 */
    处理各种挂起标志 / CHECK_FOR_INTERRUPTS();
    做本职工作;
    WaitLatch(MyLatch, WL_LATCH_SET | WL_TIMEOUT | WL_EXIT_ON_PM_DEATH,
              超时毫秒, WAIT_EVENT_XXX);
}
```

无竞态的关键在 Latch 的语义：如果有人在你 `ResetLatch` 之后、`WaitLatch` 之前 `SetLatch()`，WaitLatch **立即返回**（状态位还在），唤醒永不丢失；反之 Reset 放在循环开头，保证"睡前最后检查一遍工作"，宁可空转一圈也不睡过头。`SetLatch()`（`latch.c:290`）本身是信号安全的：置位 + 内存屏障 + 必要时 `kill(SIGURG)`/写 self-pipe。

### 12.2.3 wait_event 名字如何进 pg_stat_activity

`WaitLatch` 的最后一个参数 `wait_event_info` 就是你在 `pg_stat_activity.wait_event` 里看到的名字的来源。全部等待事件集中登记在一个数据文件 `src/backend/utils/activity/wait_event_names.txt`，构建期由 `generate-wait_event_types.pl`（同目录）生成枚举、名字表和 SGML 文档——加一个等待点只需在 txt 里添一行。运行期：进入等待前 `pgstat_report_wait_start(wait_event_info)`（`src/include/utils/wait_event.h:48`，inline，仅一次原子写）把 32 位编码（高 8 位类别 + 低位具体事件）写进本进程的 BackendStatus 共享内存槽；`pg_stat_activity` 视图读到编码再查生成的名字表翻译成字符串。开销小到可以包在每次等待外面——这就是 PG 可观测性"无处不在却几乎免费"的实现套路。

### 小结

- Latch 解决信号与 poll 的丢失唤醒问题（set-位 + self-pipe/signalfd/kqueue 把唤醒变成 fd 事件）；WaitEventSet 把 latch/socket/超时/postmaster 死亡统一进 epoll 级事件循环；
- 主循环铁律：ResetLatch → 检查中断标志 → 干活 → WaitLatch；SetLatch 在 Reset 之后到来会让下次 Wait 立刻返回；
- wait_event 名字源于 wait_event_names.txt（脚本生成代码），等待前一次原子写进 BackendStatus，pg_stat_activity 翻译展示。

## 12.3 信号与 CHECK_FOR_INTERRUPTS：安全地打断一个进程

### 12.3.1 两段式中断

Unix 信号随时可能到达——可能正在 palloc 一半、持有自旋锁。信号处理器里几乎什么都不能做。PG 因此把中断拆成两段：

- **信号处理器只置标志**：如 SIGINT 的处理器 `StatementCancelHandler()`（`src/backend/tcop/postgres.c:3155`，注册于 4398 行）只做 `QueryCancelPending = true; InterruptPending = true; SetLatch(MyLatch)`；
- **主流程在安全点自查**：`CHECK_FOR_INTERRUPTS()` 宏（`src/include/miscadmin.h:125`）：

```c
#define CHECK_FOR_INTERRUPTS() \
do { \
	if (INTERRUPTS_PENDING_CONDITION()) \
		ProcessInterrupts(); \
} while(0)
```

快路径就是读一个进程本地变量，纳秒级，所以它被撒遍所有循环：执行器每处理若干行、每次拿锁等待、每次缓冲区 IO、排序归并的内层循环……这就是"取消一条正在跑的 SQL 通常一瞬间生效"的原因——最近的一个 CFI 点不会太远。

### 12.3.2 ProcessInterrupts：中断的总分发

`ProcessInterrupts()`（`postgres.c:3452`，节选头部）：

```c
void
ProcessInterrupts(void)
{
	/* 关键防线：持锁清理期/临界区内不处理中断 */
	if (InterruptHoldoffCount != 0 || CritSectionCount != 0)
		return;
	InterruptPending = false;

	if (ProcDiePending)
	{
		...
		QueryCancelPending = false; /* ProcDie trumps QueryCancel */
		LockErrorCleanup();
		...
		ereport(FATAL,
				(errcode(ERRCODE_ADMIN_SHUTDOWN),
				 errmsg("terminating connection due to administrator command", ...)));
	}
	...
```

按优先级依次检查一串标志：`ProcDiePending`（pg_terminate_backend / 快速关库 → FATAL 断开）、客户端连接检查、`QueryCancelPending`（pg_cancel_backend / Ctrl-C / 超时 → `ereport(ERROR)`）、恢复冲突、catchup（上一篇的失效追赶）、通知、并行 worker 消息、屏障……注意实现手段：**抛出 ERROR/FATAL**，借上一篇讲的 sigsetjmp + AbortCurrentTransaction 机制完成取消——查询取消复用了错误恢复全套基础设施，不需要单独的"取消清理"代码路径。

`HOLD_INTERRUPTS()/RESUME_INTERRUPTS()`（增减 `InterruptHoldoffCount`）标出不可打断区间；`CritSectionCount` 则来自 WAL 临界区（此时抛错会升级成 PANIC，所以绝不处理取消）。

### 12.3.3 超时的实现路径

`statement_timeout` 走同一条路：backend 启动时 `RegisterTimeout(STATEMENT_TIMEOUT, StatementTimeoutHandler)`（`src/backend/utils/init/postinit.c:799`）。timeout.c 用一个 SIGALRM 复用多个定时器；到点后 `StatementTimeoutHandler()`（`postinit.c:1414`）做的事和收到 SIGINT 一样——置 QueryCancelPending。ProcessInterrupts 处理取消时用 `get_timeout_indicator(STATEMENT_TIMEOUT, ...)`（`postgres.c:3587`）区分"用户取消"和"超时"，报不同的错误文案。lock_timeout、idle_in_transaction_session_timeout 同理。**一切异步打断，殊途同归于标志位 + CFI。**

### 12.3.4 ProcSignal：跨进程"多路复用信号"

Unix 信号编号不够用（PG 需要十几种进程间通知），且发信号需要知道 pid 还可能有复用竞态。`src/backend/storage/ipc/procsignal.c` 把 SIGUSR1 做成多路复用通道：共享内存里每进程一个槽（`ProcSignalInit()`，`procsignal.c:170`），槽里是一排 per-reason 布尔标志。发送方 `SendProcSignal()`（`procsignal.c:296`，节选）：

```c
		slot = &ProcSignal->psh_slot[procNumber];

		SpinLockAcquire(&slot->pss_mutex);
		if (pg_atomic_read_u32(&slot->pss_pid) == pid)
		{
			/* 置对应原因的标志位 */
			slot->pss_signalFlags[reason] = true;
			SpinLockRelease(&slot->pss_mutex);
			/* 再补一发真实信号促其尽快查看 */
			return kill(pid, SIGUSR1);
		}
```

先在共享槽里写"原因"（并用 pid 复核防止槽已易主），再发 SIGUSR1。接收方 `procsignal_sigusr1_handler()`（`procsignal.c:696`）逐个检查标志，转手调用各 Handle 函数——它们照例只置本进程标志 + SetLatch，重活留给 CFI。原因清单（`ProcSignalReason`）包括：catchup、NOTIFY、并行消息、walsender 唤醒、barrier、日志内存上下文、恢复冲突各细类等。`pg_cancel_backend()` 这类则直接发 SIGINT/SIGTERM，两套并存。

### 小结

- 中断两段式：信号处理器只置标志 + SetLatch；主流程在海量 CHECK_FOR_INTERRUPTS 安全点调 ProcessInterrupts 统一分发；
- 取消/终止/超时都收敛为 QueryCancelPending/ProcDiePending → ereport(ERROR/FATAL) → 复用 sigsetjmp 错误恢复完成清理；HoldoffCount 与临界区保护不可打断区间；
- ProcSignal 用共享内存标志槽 + SIGUSR1 把一个信号复用成十几种跨进程通知。

## 12.4 后台进程逐个解剖

所有后台进程由 postmaster 启动并监管，主循环全部是 12.2 的标准模式。逐个看职责与要点。

### 12.4.1 checkpointer（`src/backend/postmaster/checkpointer.c`）

**职责**：执行 checkpoint（把某时刻前的全部脏页刷盘并推进恢复起点），以及**代全体进程执行 fsync**。

`CheckpointerMain()`（`checkpointer.c:206`）主循环（`checkpointer.c:366`，节选）：

```c
	for (;;)
	{
		...
		ResetLatch(MyLatch);
		AbsorbSyncRequests();               /* 吸收攒下的 fsync 请求（见下） */
		ProcessCheckpointerInterrupts();    /* checkpointer.c:665 */
		if (ShutdownXLOGPending || ShutdownRequestPending)
			break;

		/* 检查共享内存里的请求标志（backend 请求 CHECKPOINT/建库等） */
		if (((volatile CheckpointerShmemStruct *) CheckpointerShmem)->ckpt_flags)
			do_checkpoint = true;

		/* 或者距上次 checkpoint 已超过 checkpoint_timeout */
		now = (pg_time_t) time(NULL);
		elapsed_secs = now - last_checkpoint_time;
		if (elapsed_secs >= CheckPointTimeout)
		{
			do_checkpoint = true;
			flags |= CHECKPOINT_CAUSE_TIME;
		}

		if (do_checkpoint)
			... CreateCheckPoint(flags) / CreateRestartPoint(flags) ...
```

两个触发源一目了然：**请求驱动**（backend 写共享内存标志 + SetLatch：手动 CHECKPOINT、WAL 量超过 max_wal_size 等）与**时间驱动**（checkpoint_timeout）。写脏页过程中周期调用 `CheckpointWriteDelay()`（`checkpointer.c:795`）按 `checkpoint_completion_target` 掐节奏摊平 I/O，睡的每一拍里还继续 `AbsorbSyncRequests()`（`checkpointer.c:1424`）。

**fsync 请求转发（ForwardSyncRequest）** 是本文件最精彩的机制。背景：脏页可能被任何进程写出（bgwriter、驱逐页面的普通 backend），但 fsync 集中由 checkpointer 在 checkpoint 时执行才高效（同一文件攒一次 fsync）。于是谁写了页就要把"这个文件将来要 fsync"记账给 checkpointer——`ForwardSyncRequest()`（`checkpointer.c:1213`，节选）：

```c
	LWLockAcquire(CheckpointerCommLock, LW_EXCLUSIVE);

	/* checkpointer 不在，或队列满且压缩失败 → 返回 false，调用者自己 fsync */
	if (CheckpointerShmem->checkpointer_pid == 0 ||
		(CheckpointerShmem->num_requests >= CheckpointerShmem->max_requests &&
		 !CompactCheckpointerRequestQueue()))
	{
		LWLockRelease(CheckpointerCommLock);
		return false;
	}

	/* 入队：文件标签 + 请求类型写进共享内存环形数组 */
	insert_pos = CheckpointerShmem->tail;
	request = &CheckpointerShmem->requests[insert_pos];
	request->ftag = *ftag;
	request->type = type;
	CheckpointerShmem->tail = (CheckpointerShmem->tail + 1) % CheckpointerShmem->max_requests;
	CheckpointerShmem->num_requests++;

	/* 队列过半：SetLatch 催 checkpointer 来消费 */
	too_full = (CheckpointerShmem->num_requests >=
				CheckpointerShmem->max_requests / 2);
	LWLockRelease(CheckpointerCommLock);
	if (too_full)
		SetLatch(&GetPGProcByNumber(checkpointerProc)->procLatch);
```

三个工程细节：队满先尝试**去重压缩**（`CompactCheckpointerRequestQueue()`，同一文件的重复请求合并——注释指出队满时每个 backend 退化为自己 fsync 每个块会拖垮全系统）；实在不行**优雅降级**（返回 false，调用者自己 fsync，慢但正确）；催醒阈值设在半满而非全满（给消费者留提前量）。这是共享内存生产者-消费者队列的模板级实现。

### 12.4.2 bgwriter（`src/backend/postmaster/bgwriter.c`）——与 checkpointer 的分工

**职责**：在两次 checkpoint 之间**提前**清洗一部分脏页，让普通 backend 驱逐页面时尽量拿到干净页，不必自己写盘。分工表：

| | checkpointer | bgwriter |
|---|---|---|
| 目标 | 正确性：推进恢复起点，保证可恢复 | 性能：降低 backend 亲自写脏页的概率 |
| 写哪些页 | checkpoint 时刻前的**全部**脏页 | 时钟指针**将要扫到**的少量脏页（跟着驱逐压力走） |
| fsync | 执行（消费请求队列） | 不执行（写完转发请求给 checkpointer） |
| 节奏 | checkpoint_timeout / max_wal_size | bgwriter_delay（默认 200ms）+ 反馈控制 |

`BackgroundWriterMain()`（`bgwriter.c:89`）主循环（`bgwriter.c:223`）：

```c
	for (;;)
	{
		ResetLatch(MyLatch);
		ProcessMainLoopInterrupts();

		/* 干一轮活：按驱逐速率的移动平均决定本轮清洗多少页 */
		can_hibernate = BgBufferSync(&wb_context);

		pgstat_report_bgwriter();
		...
		rc = WaitLatch(MyLatch, WL_LATCH_SET | WL_TIMEOUT | WL_EXIT_ON_PM_DEATH,
					   BgWriterDelay, WAIT_EVENT_BGWRITER_MAIN);   /* bgwriter.c:307 */
```

真正的策略在 `BgBufferSync()`（storage/buffer/bufmgr.c）：沿 clock-sweep 指针**向前看**，按最近的缓冲区分配速率（反馈控制环）估算该预清多少页。系统空闲时 `can_hibernate` 触发休眠模式（`bgwriter.c:334`，睡 50 倍长），第一个要驱逐页面的 backend 会 SetLatch 叫醒它——省电与响应两全。

### 12.4.3 walwriter（`src/backend/postmaster/walwriter.c`）

**职责**：周期性把 WAL 缓冲刷到盘，主要服务两类人：`synchronous_commit = off` 的事务（提交不等 fsync，靠 walwriter 在 wal_writer_delay 内补上持久化，故最多丢一小窗口、不坏一致性）；以及减少高并发下同步提交者亲自 flush 的次数（顺手把别人的也刷了→组提交效果）。`WalWriterMain()`（`walwriter.c:89`）主循环（`walwriter.c:212`）核心就一句 `XLogBackgroundFlush()`，配熟悉的休眠机制（连续若干轮无事可做后拉长睡眠，异步提交的 backend 发现它在冬眠会 SetLatch 叫醒）。

### 12.4.4 autovacuum launcher / worker（`src/backend/postmaster/autovacuum.c`）

两级架构：**launcher**（常驻一个）只做调度，**worker**（动态，≤ autovacuum_max_workers）干活。`AutoVacLauncherMain()`（`autovacuum.c:411`）主循环片段（`autovacuum.c:616` 起）：

```c
		launcher_determine_sleep(av_worker_available(), false, &nap);

		(void) WaitLatch(MyLatch,
						 WL_LATCH_SET | WL_TIMEOUT | WL_EXIT_ON_PM_DEATH,
						 (nap.tv_sec * 1000L) + (nap.tv_usec / 1000L),
						 WAIT_EVENT_AUTOVACUUM_MAIN);

		ResetLatch(MyLatch);
		ProcessAutoVacLauncherInterrupts();
```

（注意此循环注释特意说明它与标准模式相反——先睡后干，因为希望启动后先歇满一个 naptime 再派 worker。）launcher 维护各数据库的上次处理时间表（`rebuild_database_list()`），按 autovacuum_naptime 均匀轮转；到点挑库后经 `do_start_worker()` 请 postmaster fork 一个 worker（launcher 自己不能 fork——它不是 postmaster，流程是写共享内存槽 + 发 PMSIGNAL 给 postmaster）。`AutoVacWorkerMain()`（`autovacuum.c:1418`）连上目标库，按"死元组数 > threshold + scale_factor × 表行数"公式挑表执行 VACUUM/ANALYZE，全程受 vacuum_cost 参数限流。worker 是正经的"能跑事务的 backend"，这个"launcher 调度 + postmaster 代孕 + worker 连库"的三角，就是后来 background worker 框架的原型。

### 12.4.5 archiver（`src/backend/postmaster/pgarch.c`）

**职责**：把写满的 WAL 段安全送去归档（archive_command 或 archive library 回调）。`PgArchiverMain()`（`pgarch.c:221`）→ `pgarch_MainLoop()`（`pgarch.c:308` 起）：等 latch（walwriter/backend 切换 WAL 段时会通知）或 60 秒自醒 → `pgarch_ArchiverCopyLoop()` 扫 `archive_status` 目录里的 `.ready` 文件逐个归档、成功改名 `.done`。失败重试有退避；注释里那段"收到 SIGTERM 后最多再等 60 秒"的逻辑（`pgarch.c:340` 附近）体现了归档进程的谨慎——它守护的是备份链完整性。

### 12.4.6 统计系统（`src/backend/utils/activity/`，PG 15 大重构）

PG 14 及以前有个独立的 stats collector 进程，靠 **UDP** 收各 backend 的计数——丢包、延迟、临时文件读写，槽点满满。PG 15 重构（本目录）后：**collector 进程消失**，累计统计直接放共享内存：

- 固定对象（bgwriter、checkpointer、WAL、SLRU…）各占静态共享区一块；可变对象（每表、每函数、每库…）放 **dshash**——DSA 上的跨进程哈希表（`pgstat_shmem.c:70` 的 dshash_parameters，`StatsShmemInit()` 在 `pgstat_shmem.c:175`）；
- backend 先在本地攒增量，`pgstat_report_stat()`（`pgstat.c:723`）在事务边界且距上次至少 `PGSTAT_MIN_INTERVAL`（1000ms，`pgstat.c:127`）时批量刷入共享内存（拿各条目的 LWLock 累加）——读写路径都不再有丢失；
- 分模块文件 `pgstat_relation.c`、`pgstat_database.c`、`pgstat_io.c` 等各管一类；`backend_status.c` 是另一套更轻的"当前状态"设施（pg_stat_activity 的数据源，含 12.2.3 的 wait_event）。

对你这种做过监控管道的人，这就是"从 UDP 上报 + 单点聚合"迁移到"共享内存多写者直更"的经典架构升级，代价是引入了并发控制，收益是精确性与实时性。

### 小结

- checkpointer：请求/超时双触发做 checkpoint，慢节奏摊 I/O；独家执行 fsync，全体进程经 ForwardSyncRequest 环形队列记账（去重压缩、半满催醒、队满降级自己 fsync）；
- bgwriter 管性能（预清洗将被驱逐的脏页，反馈控制 + 休眠），checkpointer 管正确性，二者以"是否执行 fsync、覆盖哪些页"分界；
- walwriter 兜底异步提交的持久化并助攻组提交；archiver 守护 WAL 归档链；
- autovacuum 是 launcher 调度 + postmaster 代孕 + worker 连库干活的三角；
- PG 15 起统计进共享内存（固定块 + dshash），backend 本地攒批、事务边界限频刷入，UDP collector 成为历史。

## 12.5 background worker 框架：把"常驻进程"开放给扩展

### 12.5.1 注册 API

`src/backend/postmaster/bgworker.c` 把 12.4 那套"postmaster 管生死"的待遇开放给扩展。声明一个 worker 就是填一个结构（`src/include/postmaster/bgworker.h:96`）：

```c
typedef struct BackgroundWorker
{
	char		bgw_name[BGW_MAXLEN];        /* ps 与日志里显示的名字 */
	char		bgw_type[BGW_MAXLEN];
	int			bgw_flags;                   /* 要共享内存？要连数据库？ */
	BgWorkerStartTime bgw_start_time;        /* 在启动流程的哪个阶段拉起 */
	int			bgw_restart_time;	/* in seconds, or BGW_NEVER_RESTART */
	char		bgw_library_name[MAXPGPATH]; /* 哪个 .so */
	char		bgw_function_name[BGW_MAXLEN]; /* 入口函数名 */
	Datum		bgw_main_arg;                /* 传给入口的参数 */
	char		bgw_extra[BGW_EXTRALEN];
	pid_t		bgw_notify_pid; /* SIGUSR1 this backend on start/stop */
} BackgroundWorker;
```

要点：`bgw_flags` 常填 `BGWORKER_SHMEM_ACCESS | BGWORKER_BACKEND_DATABASE_CONNECTION`（`bgworker.h:53,60`）；`bgw_start_time` 可选恢复完成后（`BgWorkerStart_RecoveryFinished`）等阶段（`bgworker.h:87-88`）；入口以"库名+函数名"字符串表达——因为注册发生在 postmaster，而函数要在 fork 出的 worker 进程里查找解析。两条注册路径：

- **静态**：`RegisterBackgroundWorker()`（`bgworker.c:962`）——只能在 postmaster 处理 `shared_preload_libraries` 时（扩展 `_PG_init()` 里）调用，函数开头的检查（`bgworker.c:971` 起）明确拒绝其他场合；
- **动态**：`RegisterDynamicBackgroundWorker()`（`bgworker.c:1068`）——普通 backend 运行期申请：写共享内存槽位 + 通知 postmaster fork，返回 `BackgroundWorkerHandle`；可用 `WaitForBackgroundWorkerStartup()`（`bgworker.c:1235`）等它起来（内部即等 SIGUSR1 通知 + 查槽位状态）。并行查询的 worker 就是走这条路。

### 12.5.2 生命周期

postmaster fork → `BackgroundWorkerMain()`（`bgworker.c:741`）：设置默认信号处理（SIGTERM→die、SIGUSR1→procsignal）、建立 sigsetjmp 恢复块、按 `bgw_library_name/bgw_function_name` 动态加载并跳入扩展的入口函数。入口函数的标准开场：

```c
BackgroundWorkerUnblockSignals();                 /* 解除 fork 时的信号屏蔽 */
BackgroundWorkerInitializeConnection("mydb", NULL, 0);  /* 需要连库时；postinit.c 路径 */
for (;;)                                          /* 然后进入 12.2 的标准主循环 */
{
    ResetLatch(MyLatch);
    CHECK_FOR_INTERRUPTS();
    ... 干活，可用 SPI 跑 SQL（自己负责 StartTransactionCommand/CommitTransactionCommand）...
    WaitLatch(MyLatch, WL_LATCH_SET | WL_TIMEOUT | WL_EXIT_ON_PM_DEATH, ms, 事件);
}
```

退出码语义：exit(0) = 不再重启；exit(1) = 按 `bgw_restart_time` 重启（崩溃同理，postmaster 的监管逻辑一视同仁）；`BGW_NEVER_RESTART` 则一次性。worker 崩溃如果发生在共享内存被污染的场景，同样触发全库 crash-restart——权利与义务对等。

### 12.5.3 案例：pg_cron 类调度器怎么用它

社区扩展 pg_cron（数据库内的 crontab）是教科书用法：

1. `shared_preload_libraries = 'pg_cron'` → `_PG_init()` 里 `RegisterBackgroundWorker()` 注册一个 launcher worker，`bgw_start_time = BgWorkerStart_RecoveryFinished`（主库恢复完成才启动）；
2. launcher 主循环：连上元数据库读 `cron.job` 表 → `WaitLatch` 睡到最近的触发点 → 到点后为每个任务 `RegisterDynamicBackgroundWorker()` 派生执行 worker（或走 libpq 自连，两种模式它都支持）；
3. 执行 worker 连上目标库，在事务里跑任务 SQL，把结果写回 `cron.job_run_details`，退出（exit 0）；
4. 配置变更无需轮询：普通 backend 里执行 `cron.schedule()` 的 SQL 函数改完表后，对 launcher `SetLatch`/发信号即可即时生效——又回到了 12.2 的原语。

同一框架的其他住户：逻辑复制的 launcher/apply worker（`replication/logical/launcher.c`，内核自己也用这套 API）、pg_prewarm 的 autoprewarm、TimescaleDB 的 job scheduler、Citus 的 maintenance daemon。**判断一个扩展是否"重"，看它有没有 worker** ——有 worker 就意味着它在数据库里有了自己的心跳。

### 小结

- BackgroundWorker 结构声明名字/权限/启动阶段/重启策略/入口（库名+函数名字符串）；静态注册限于 shared_preload_libraries 阶段，动态注册运行期即可并有 handle 可等；
- 生命周期完全托管：postmaster fork/重启/收尸；worker 内部标配 sigsetjmp + 标准 Latch 主循环 + 可选连库跑 SQL；
- pg_cron 模式 = 静态 launcher + 动态执行 worker + SetLatch 即时唤醒，是编写"数据库内守护程序"的标准范式。

## 自测问题

**1. 为什么修改 shared_buffers 必须重启实例，而并行查询却能运行时拿到几百 MB 共享内存？**

静态共享内存段在 postmaster 启动时由 `CalculateShmemSize()`（ipci.c:57）汇总各子系统需求后一次性创建（`CreateSharedMemoryAndSemaphores()`，ipci.c:120），分配器只进不出，尺寸不可变——shared_buffers 是其中最大的一项。并行查询用的是第二层机制：运行期 `dsm_create()`（dsm.c:524）创建独立的动态段，用完销毁；跨进程的 malloc 语义再由 DSA（dsa.c）在 DSM 段之上提供，以 dsa_pointer（段+偏移）代替裸指针，因为各进程 attach 地址可能不同。

**2. 后台进程主循环为什么必须"先 ResetLatch 再干活再 WaitLatch"？把 ResetLatch 挪到 WaitLatch 之前会怎样？**

Latch 的防丢失语义依赖顺序：Reset 后到 Wait 前若有人 SetLatch，位仍是置起的，Wait 立即返回。标准顺序（Reset → 检查/干活 → Wait）保证任何在"检查工作"之后到达的唤醒都会让下一次 Wait 直接穿透，最多空转一圈、绝不睡过头。若在干活之后、Wait 之前才 Reset，就会把"干活期间到达的唤醒"清掉，随后 Wait 陷入沉睡——这正是裸 select+signal 的经典竞态，Latch 存在的意义就是消灭它（latch.c:172、SetLatch latch.c:290）。

**3. 描述 `pg_cancel_backend(pid)` 之后，目标 backend 上一条长查询被取消的完整路径。**

发送方对目标发 SIGINT；目标的 `StatementCancelHandler()`（postgres.c:3155）在信号处理器里仅置 `QueryCancelPending = InterruptPending = true` 并 SetLatch。目标进程继续执行到下一个 `CHECK_FOR_INTERRUPTS()`（miscadmin.h:125，遍布执行器循环/锁等待/IO 路径），条件成立进入 `ProcessInterrupts()`（postgres.c:3452）；若不在 HoldInterrupts/临界区内，则对 QueryCancelPending 执行 `ereport(ERROR)`，长跳转到 postgres.c:4573 的 sigsetjmp 恢复块，经 AbortCurrentTransaction 释放锁与资源、重置内存上下文，向客户端报"canceling statement"，回到主循环等下一条命令。statement_timeout 只是换了置标志的人（postinit.c:1414 的定时器处理器），后半路完全相同。

**4. bgwriter 和 checkpointer 都写脏页，为什么 fsync 只让 checkpointer 做？请求怎么到它手里，队列满了怎么办？**

fsync 按文件生效，集中到 checkpoint 时每文件做一次最省；且恢复正确性只要求"checkpoint 完成前其覆盖的脏页已持久化"，平时写出的页无需立刻 fsync。因此 bgwriter/backend 写完页后调 `ForwardSyncRequest()`（checkpointer.c:1213）把 (文件标签, 请求类型) 写入共享内存环形队列，半满时 SetLatch 催 checkpointer 用 `AbsorbSyncRequests()`（checkpointer.c:1424）吸收记账，checkpoint 时统一 sync。队满先 `CompactCheckpointerRequestQueue()` 去重；仍满则返回 false，调用者退化为自己 fsync 该文件——性能降级但正确性不降。

**5. 一个扩展想要"每 10 分钟在库里跑一段 SQL"的常驻能力，需要用到本章哪些机制？各自负责什么？**

background worker 框架负责进程生命周期：`_PG_init()` 里 `RegisterBackgroundWorker()`（bgworker.c:962）注册（shared_preload_libraries 阶段），postmaster 负责 fork/崩溃重启，`BackgroundWorkerMain()`（bgworker.c:741）装载入口；`bgw_flags` 声明共享内存与连库权限，`BackgroundWorkerInitializeConnection()` 建立数据库会话后用 SPI 在自管事务里跑 SQL。Latch/WaitLatch 负责节拍（WL_TIMEOUT 10 分钟 + WL_LATCH_SET 支持即时唤醒 + WL_EXIT_ON_PM_DEATH 随 postmaster 退出）；CHECK_FOR_INTERRUPTS 使其可被 pg_terminate_backend 优雅终止；若需要在 pg_stat_activity 里可观测，等待时传入登记于 wait_event_names.txt 体系的自定义等待事件即可。
