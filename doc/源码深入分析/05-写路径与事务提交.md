# 05 · 写路径与事务提交：从 heap_insert 到 COMMIT 落盘

> 对应《PostgreSQL 源码学习指南》第 6 章。所有 `文件:行号` 均已在本地源码树（20devel，master 分支）核实。

## 本章导读

读路径回答的是"数据怎么被找到"，写路径回答的是"数据怎么被**可靠地**改掉"。本章沿着一条 INSERT / UPDATE / DELETE 语句的执行器出口（`nodeModifyTable.c` 调用 `table_tuple_insert()` 等 Table AM 接口）往下，走进 heap 访问方法的三大写函数，再看它们如何组装 WAL 记录，最后跟随 `COMMIT` 走完事务提交的全过程。读完本章你应该能回答：

1. 一条 INSERT 在磁盘上到底动了哪些字节？什么时候动？
2. 为什么 PG 的 UPDATE 是"插入新版本 + 给旧版本打标记"？HOT 优化省掉了什么？
3. 行锁为什么"不占锁表内存"？xmax 和 infomask 是怎么把"删除者"和"加锁者"编码进同一个字段的？
4. WAL 记录是怎么拼出来的？full-page write 在哪一行代码里被决定？
5. COMMIT 时哪一步是唯一必须等磁盘的？CLOG、ProcArray 各自扮演什么角色？为什么回滚几乎免费？

## 前置知识

- **元组头**（`src/include/access/htup_details.h`）：每行数据前有一个 `HeapTupleHeaderData`，其中 `t_xmin`（创建者事务 ID）、`t_xmax`（删除/加锁者事务 ID）、`t_ctid`（版本链指针）、`t_infomask`/`t_infomask2`（标志位）是本章的主角。
- **临界区**（critical section）：`START_CRIT_SECTION()` 到 `END_CRIT_SECTION()` 之间任何 `elog(ERROR)` 都会升级为 PANIC（全库重启）。内核用它保证"共享缓冲区里的页修改"与"对应的 WAL 记录"这两件事要么都完成、要么进程直接死掉靠 WAL 恢复——绝不允许出现"页改了一半但没日志"的中间态。
- **WAL-before-data**：脏数据页刷盘之前，覆盖它的 WAL 必须先落盘。这是 ARIES 的第一原则，PG 靠"页头 LSN + 刷页前 `XLogFlush`"实现（见 06 章 FlushBuffer）。
- **与 InnoDB 的根本差异**：InnoDB 的 UPDATE 原地改行、旧值进 undo 段；PG 没有 undo，新旧版本都躺在堆页里，靠 `xmin`/`xmax` + CLOG 决定谁能看见谁。这一个设计差异决定了本章几乎所有代码的形状。

## 6.1 heap_insert：一次插入的完整解剖

入口在 `src/backend/access/heap/heapam.c:2005`。函数骨架非常清晰，先看全貌（节选真实代码）：

```c
/* heapam.c:2004 */
void
heap_insert(Relation relation, HeapTuple tup, CommandId cid,
            uint32 options, BulkInsertState bistate)
{
    TransactionId xid = GetCurrentTransactionId();          /* ① */
    ...
    heaptup = heap_prepare_insert(relation, tup, xid, cid, options);  /* ② */

    buffer = RelationGetBufferForTuple(relation, heaptup->t_len,
                                       InvalidBuffer, options, bistate,
                                       &vmbuffer, NULL, 0);           /* ③ */
    ...
    CheckForSerializableConflictIn(relation, NULL, InvalidBlockNumber); /* ④ */
    ...
    START_CRIT_SECTION();                                   /* ⑤ */
    RelationPutHeapTuple(relation, buffer, heaptup, ...);   /* ⑥ */
    ...
    MarkBufferDirty(buffer);                                /* ⑦ */
    if (RelationNeedsWAL(relation))
    {
        ...
        recptr = XLogInsert(RM_HEAP_ID, info);              /* ⑧ */
        PageSetLSN(page, recptr);                           /* ⑨ */
    }
    END_CRIT_SECTION();                                     /* ⑩ */
    UnlockReleaseBuffer(buffer);                            /* ⑪ */
```

### ① GetCurrentTransactionId 的时机（heapam.c:2008）

`GetCurrentTransactionId()`（`src/backend/access/transam/xact.c:456`）体现了 PG 的**惰性 XID 分配**：事务开始（BEGIN）时并不分配 XID，只有第一次执行写操作、真正需要在元组上盖"作者签名"时才调用 `AssignTransactionId()`（xact.c:637）向共享计数器要一个新 XID。纯只读事务全程没有 XID——这就是为什么长时间的只读 SELECT 不会消耗 XID、不会加剧回卷（wraparound）压力。对比 InnoDB：只读事务同样有 trx_id 延迟分配优化（5.6+），思想一致。

注意它是 `heap_insert` 的**第一行**：XID 必须在拿页锁之前拿到，因为分配 XID 本身可能等锁（XidGenLock）甚至做 I/O（扩展 CLOG 页），这些都不允许发生在持有缓冲区排他锁期间。

### ② heap_prepare_insert：填充元组头（heapam.c:2228）

```c
/* heapam.c:2242 */
tup->t_data->t_infomask &= ~(HEAP_XACT_MASK);      /* 清掉所有事务相关标志位 */
tup->t_data->t_infomask2 &= ~(HEAP2_XACT_MASK);
tup->t_data->t_infomask |= HEAP_XMAX_INVALID;      /* 提示位：xmax 无效（没人删过它） */
HeapTupleHeaderSetXmin(tup->t_data, xid);          /* xmin = 我的 XID：“我创建的” */
if (options & HEAP_INSERT_FROZEN)
    HeapTupleHeaderSetXminFrozen(tup->t_data);     /* COPY FREEZE 等场景：直接冻结 */
HeapTupleHeaderSetCmin(tup->t_data, cid);          /* 命令号，用于同事务内可见性 */
HeapTupleHeaderSetXmax(tup->t_data, 0);            /* xmax 清零 */
tup->t_tableOid = RelationGetRelid(relation);
```

- `cid`（CommandId）解决"同一事务内，第 3 条语句能否看到第 1 条语句插入的行"的问题（能）以及"游标打开后插入的行游标是否可见"（不可见）。
- 随后（heapam.c:2257 起）判断是否需要 TOAST：元组超过 `TOAST_TUPLE_THRESHOLD`（约 2KB）时，超长字段被切片压缩存入 toast 表，主元组里只留 18 字节指针。对应 InnoDB 的行溢出页（off-page column），但 PG 的 TOAST 是独立的表 + 索引。

### ③ RelationGetBufferForTuple：找一个能放下元组的页（hio.c:500）

`src/backend/access/heap/hio.c:500` 是插入的"选址"逻辑，要点：

1. **先试上次插入的页**（relcache 里的 `smgr_targblock` 或 BulkInsertState 缓存的页），命中则零成本；
2. 不行则查 **FSM**：`GetPageWithFreeSpace()`（`src/backend/storage/freespace/freespace.c:137`）在 FSM 树里找一个空闲空间 ≥ 需求的页（FSM 结构见 06 章 7.7 节）；
3. FSM 也找不到就**扩展文件**（新增页）。扩展时可以批量多页（`num_pages` 参数），减少扩展锁争抢；
4. **fillfactor**：hio.c:537 起计算 `saveFreeSpace = RelationGetTargetPageFreeSpace(...)`——INSERT 主动不把页塞满，给未来同页的 HOT 更新留余地。这与 InnoDB 页的 `MERGE_THRESHOLD`/预留 1/16 空间思路相似，但 PG 的动机主要是 HOT 而不是 B-tree 分裂。
5. 若目标页是 all-visible，还要顺手 pin 住对应的 VM 页（返回给 `vmbuffer`）——因为插入后该页不再"全可见"，稍后要在临界区里清 VM 位，而 VM 页 I/O 绝不能发生在临界区内。

函数返回时，目标页的缓冲区已被 pin 住并加了**排他内容锁**。

### ④ 序列化冲突检查（heapam.c:2056）

`CheckForSerializableConflictIn()` 只在 SERIALIZABLE 隔离级别有实际工作（SSI 的写侧检查）。注释（heapam.c:2042）解释了为什么放在临界区外：万一要报错，回滚成本最低。

### ⑤–⑨ 临界区：页修改与 WAL 的原子配合

这是全章最重要的一段代码模式，值得逐行读（heapam.c:2065–2186，节选）：

```c
/* heapam.c:2065 */
/* NO EREPORT(ERROR) from here till changes are logged */
START_CRIT_SECTION();

RelationPutHeapTuple(relation, buffer, heaptup, ...);   /* 把元组真正写进页：
                                                           分配 line pointer、拷贝数据、
                                                           更新 pd_lower/pd_upper，
                                                           并把分到的 TID 写回 heaptup->t_self */
if (clear_all_visible)
{
    visibilitymap_clear(...);              /* 清 VM 里该页的 all-visible 位 */
    PageClearAllVisible(page);             /* 清页头 PD_ALL_VISIBLE 标志 */
}
if (TransactionIdIsNormal(xid) && !(options & HEAP_INSERT_FROZEN))
    PageSetPrunable(page, xid);            /* 在页头 pd_prune_xid 记个提示：
                                              万一本事务 abort，这行就是死元组，
                                              方便以后的页内清理(prune)发现它 */
MarkBufferDirty(buffer);                   /* 置 BM_DIRTY —— 必须在写 WAL 之前 */

if (RelationNeedsWAL(relation))            /* 临时表/unlogged 表跳过整个 WAL 块 */
{
    xl_heap_insert xlrec;
    xl_heap_header xlhdr;
    ...
    xlrec.offnum = ItemPointerGetOffsetNumber(&heaptup->t_self);
    ...
    XLogBeginInsert();                                     /* 开始组装一条 WAL 记录 */
    XLogRegisterData(&xlrec, SizeOfHeapInsert);            /* 记录主数据：offnum+flags */
    ...
    XLogRegisterBuffer(HEAP_INSERT_BLKREF_HEAP, buffer,
                       REGBUF_STANDARD | bufflags);        /* 登记被修改的页 */
    XLogRegisterBufData(HEAP_INSERT_BLKREF_HEAP, &xlhdr, SizeOfHeapHeader);
    XLogRegisterBufData(HEAP_INSERT_BLKREF_HEAP,           /* 元组体挂在该页名下：
                        (char *) heaptup->t_data + SizeofHeapTupleHeader,
                        heaptup->t_len - SizeofHeapTupleHeader);
                                                              如果这条记录最终带全页镜像，
                                                              这些数据可以省略不写 */
    recptr = XLogInsert(RM_HEAP_ID, info);                 /* 写入 WAL 缓冲区，返回 LSN */
    PageSetLSN(page, recptr);                              /* ★ 把 LSN 盖进页头 pd_lsn */
}
END_CRIT_SECTION();
```

几个关键设计点：

- **先改页、再写 WAL、最后盖 LSN**，全程持排他锁 + 临界区。顺序看似与"日志先行"矛盾，其实不然：WAL-before-data 约束的是**落盘顺序**而非内存中的修改顺序。内存里先改页没关系，因为 `PageSetLSN` 把这条 WAL 记录的结束位置写进了页头 `pd_lsn`（`src/include/storage/bufpage.h:187`），以后任何人想把这个脏页刷盘，都必须先 `XLogFlush(pd_lsn)`（见 06 章 FlushBuffer 走读）。页头 LSN 就是那根"栓住数据页的绳子"。
- **MarkBufferDirty 必须在 XLogInsert 之前**：否则存在一个窗口——WAL 写了、checkpoint 恰好开始、页还没标脏所以没被 checkpoint 刷出，之后崩溃恢复时 redo 又因为"页 LSN ≥ 记录 LSN"被跳过……PG 用严格的调用顺序 + 断言（`XLogRegisterBuffer` 中 `Assert(BufferIsDirty(buffer))`，xloginsert.c:266）杜绝这类错误。
- **XLOG_HEAP_INIT_PAGE 优化**（heapam.c:2121）：如果插入的是页上第一条元组，WAL 里不必带全页镜像，重放时直接重新初始化空页即可（`REGBUF_WILL_INIT`）。
- **元组数据登记在页名下**（`XLogRegisterBufData`，xloginsert.c:413）而不是主数据区：这样当这条记录决定带 full-page image 时，元组内容已经包含在页镜像里，可以自动省略，避免双份存储。

### ⑩⑪ 收尾

`END_CRIT_SECTION()` 后解锁释放缓冲区（heapam.c:2188），然后 `CacheInvalidateHeapTuple()`（heapam.c:2205，只对系统目录表有实际动作）和统计计数。注意**数据页从头到尾没有刷盘**——它只是躺在共享缓冲区里变成了脏页，何时落盘是 checkpointer/bgwriter 的事（06 章）。

### 小结

heap_insert = 拿 XID → 填元组头 → FSM 选址 → 临界区内{写页 + 组装 WAL + 盖 LSN} → 解锁。两条铁律：修改共享页必须"排他锁 + 临界区 + WAL"三件套；数据页永不主动刷盘，靠页头 LSN 与 WAL 挂钩。

## 6.2 heap_update：PG 特色最浓的函数

`src/backend/access/heap/heapam.c:3267`，全函数约 1100 行，是 heapam 里最复杂的函数。核心心智模型：**UPDATE = 在（尽量同一）页里 insert 新版本 + 把旧版本的 xmax 设为我的 XID + 旧版本 ctid 指向新版本**。下面按执行顺序分段走读。

### 6.2.1 准备阶段：四张属性位图（heapam.c:3356）

```c
/* heapam.c:3356 */
hot_attrs = RelationGetIndexAttrBitmap(relation, INDEX_ATTR_BITMAP_HOT_BLOCKING);
sum_attrs = RelationGetIndexAttrBitmap(relation, INDEX_ATTR_BITMAP_SUMMARIZED);
key_attrs = RelationGetIndexAttrBitmap(relation, INDEX_ATTR_BITMAP_KEY);
id_attrs  = RelationGetIndexAttrBitmap(relation, INDEX_ATTR_BITMAP_IDENTITY_KEY);
```

- `hot_attrs`：所有"会阻止 HOT"的列 = 被普通索引（B-tree 等，含部分索引谓词、表达式索引引用的列）覆盖的列；
- `sum_attrs`：只被**摘要型索引**（BRIN）覆盖的列——改了它们不阻止 HOT，但要通知 BRIN 更新摘要；
- `key_attrs`：唯一索引/主键/外键引用的"键"列——决定行锁强度；
- `id_attrs`：replica identity 列——决定逻辑复制要不要记旧值。

这些位图从 relcache 拿缓存，必须在锁页**之前**取好（注释 heapam.c:3345：更新系统目录时后取会死锁）。

### 6.2.2 锁定旧元组所在页（heapam.c:3369–3383）

```c
block = ItemPointerGetBlockNumber(otid);
buffer = ReadBuffer(relation, block);
page = BufferGetPage(buffer);
if (PageIsAllVisible(page))
    visibilitymap_pin(relation, block, &vmbuffer);   /* 先 pin VM 页，避免锁内 I/O */
LockBuffer(buffer, BUFFER_LOCK_EXCLUSIVE);
```

之后通过 line pointer 拿到旧元组 `oldtup`（heapam.c:3436–3439）。

### 6.2.3 HeapDetermineColumnsInfo：判定改了哪些列（heapam.c:4549）

```c
/* heapam.c:3452 */
modified_attrs = HeapDetermineColumnsInfo(relation, interesting_attrs,
                                          id_attrs, &oldtup, newtup, &id_has_external);
```

实现（heapam.c:4549）就是对每个"感兴趣的列"做逐列新旧值比较：`heap_getattr()` 取出两边的 Datum，`heap_attr_equals()` 按类型语义比较（定长类型比字节，变长类型 `datumIsEqual`）。返回"真的变了"的列集合。注意这是**按值比较**，`UPDATE t SET a = a` 这种没改值的更新不会把 a 算进 modified_attrs——所以它仍然可以走 HOT。

### 6.2.4 行锁强度选择（heapam.c:3467）

```c
if (!bms_overlap(modified_attrs, key_attrs))
{
    *lockmode = LockTupleNoKeyExclusive;      /* 没改键列：弱一档的排他锁 */
    mxact_status = MultiXactStatusNoKeyUpdate;
    key_intact = true;
    MultiXactIdSetOldestMember();
}
else
{
    *lockmode = LockTupleExclusive;           /* 改了键列：完全排他 */
    ...
}
```

这是 PG 9.3 引入的 `FOR NO KEY UPDATE` / `FOR KEY SHARE` 机制：普通 UPDATE（不改键）持 NoKeyExclusive 锁，与外键检查用的 KeyShare 锁**不冲突**——父表行被子表引用时，更新父表的非键列不会被外键触发的行锁卡住。这解决了 MySQL 世界里常见的"外键导致的锁等待放大"问题的 PG 版本。

### 6.2.5 可见性/并发检查与 `l2` 重试循环（heapam.c:3498 起）

`l2:` 标签开始的一大段用 `HeapTupleSatisfiesUpdate()` 判断旧元组当前状态：正被别的事务更新/加锁时，按 `wait` 参数等待或返回 `TM_Updated` 等失败码（交给执行器做 READ COMMITTED 的 EvalPlanQual 重检查）。这一段还处理 MultiXact（多个事务同时持有该行锁）的复杂情形，第一遍阅读可以跳过，记住结论：**走出这段代码时，当前事务已确认有权更新该行**。

### 6.2.6 计算新旧元组的 xmax/infomask（heapam.c:3755、3805）

```c
/* heapam.c:3755 —— 旧元组：xmax = 我的 XID（可能与既有锁合并成 MultiXact） */
compute_new_xmax_infomask(HeapTupleHeaderGetRawXmax(oldtup.t_data),
                          oldtup.t_data->t_infomask, oldtup.t_data->t_infomask2,
                          xid, *lockmode, true,
                          &xmax_old_tuple, &infomask_old_tuple, &infomask2_old_tuple);

/* heapam.c:3805 —— 新元组：xmin = 我的 XID */
newtup->t_data->t_infomask &= ~(HEAP_XACT_MASK);
newtup->t_data->t_infomask2 &= ~(HEAP2_XACT_MASK);
HeapTupleHeaderSetXmin(newtup->t_data, xid);
HeapTupleHeaderSetCmin(newtup->t_data, cid);
newtup->t_data->t_infomask |= HEAP_UPDATED | infomask_new_tuple;
HeapTupleHeaderSetXmax(newtup->t_data, xmax_new_tuple);
```

`compute_new_xmax_infomask()`（heapam.c:5503）是把"旧元组上可能存在的锁信息"折叠进新 xmax 的核心逻辑：如果旧元组正被其他事务持有 KeyShare 锁，就得创建一个 MultiXact 把"锁持有者们 + 我这个更新者"打包塞进 xmax（`HEAP_XMAX_IS_MULTI`）。

### 6.2.7 跨页更新：TOAST、放不下与锁定顺序（heapam.c:3826–4076）

如果新版本放不下同页（或需要 TOAST），事情变复杂——必须解开旧页的锁去做 TOAST/找新页，但解锁期间别人可能抢先更新该行。PG 的解法（heapam.c:3848 起）：

1. **先把旧元组临时标成"仅加锁"**：用 `xl_heap_lock` WAL 记录把 `xmax = 我的XID + HEAP_XMAX_LOCK_ONLY` 写上（heapam.c:3888–3946 的小临界区）。这样解开页锁后，并发事务看到该行已被锁定会乖乖等待；而如果我们中途崩溃/出错，这个 xmax 属于 aborted 事务，别人可以照常继续——一个优雅的"意向占位"；
2. 解锁旧页（保留 pin），做 TOAST（heapam.c:3965）；
3. **锁定顺序防死锁**（heapam.c:3983 注释原文）："We use the rule *lock the lower-numbered page of the relation first*"。找新页时调用 `RelationGetBufferForTuple(relation, len, buffer, ...)` 把旧页缓冲区传进去（heapam.c:3999），由 hio.c 内部按**块号从小到大**的固定顺序对两个页加锁——两个 backend 交叉更新同两页时不会死锁。这与 InnoDB 内部按固定顺序拿 page latch 的纪律相同；
4. `for (;;)` 循环（heapam.c:3994）处理"重新拿到锁后发现页里空间又变了/all-visible 位又被设置了"的竞态，必要时解锁重来。

### 6.2.8 HOT 判定：两个条件缺一不可（heapam.c:4063）

```c
/* heapam.c:4063 */
if (newbuf == buffer)                 /* 条件一：新版本落在同一页 */
{
    if (!bms_overlap(modified_attrs, hot_attrs))   /* 条件二：没改任何索引列 */
    {
        use_hot_update = true;
        if (bms_overlap(modified_attrs, sum_attrs))
            summarized_update = true; /* 只需要通知 BRIN 这类摘要索引 */
    }
}
else
{
    PageSetFull(page);                /* 跨页了：给旧页打 PD_PAGE_FULL 提示，
                                         下次有人来更新时先做页内 prune 腾空间 */
}
```

`README.HOT`（`src/backend/access/heap/README.HOT`）的要点浓缩：

- **动机**：没有 HOT 时，每个新版本都要往**每个索引**插一条指向新 TID 的项——这是 PG 写放大的最大来源，也是"单页清理（prune）"无法实施的原因（回收元组前必须先删索引项，而找到索引项要扫索引）；
- **机制**：HOT 链上只有链头有索引项。新版本打 `HEAP_ONLY_TUPLE` 标志（没有索引指向我），旧版本打 `HEAP_HOT_UPDATED` 标志（顺着我的 ctid 能找到后继）。索引扫描到达链头后沿 ctid 链在**同页内**前进，直到找到可见版本；
- **单页回收**：链中间的死版本可以被任何路过的查询顺手清掉（`pruneheap.c`），链头 line pointer 变成 `LP_REDIRECT`（`src/include/storage/itemid.h:40`）直接指向存活版本——空间回收完全不碰索引；
- **链不跨页**：跨页就得新建索引项、结束 HOT 链——否则索引扫描要跳页追链，破坏"一次页访问"的性能承诺。

函数结尾据此设置 `*update_indexes`（heapam.c:4344–4356）：HOT → `TU_None`（一个索引都不插）；HOT 但改了 BRIN 列 → `TU_Summarizing`；非 HOT → `TU_All`。执行器 `ExecUpdate` 按它决定调不调 `ExecInsertIndexTuples`。

对比 InnoDB：InnoDB 二级索引在 UPDATE 未改索引列时本来就不动（原地更新 + undo），HOT 是 PG 为自己的 no-overwrite 存储追回这项性能的补丁；代价是引入 prune、`LP_REDIRECT`、fillfactor 调优等一整套配套机制。

### 6.2.9 主临界区（heapam.c:4167–4297）


## 6.3 heap_delete 与行锁编码

### 6.3.1 heap_delete 要点（heapam.c:2759）

DELETE 是 UPDATE 的简化版：没有新版本，只给旧元组盖 xmax。核心临界区（heapam.c:3036–3067）：

```c
/* heapam.c:3058 */
tp.t_data->t_infomask &= ~(HEAP_XMAX_BITS | HEAP_MOVED);
tp.t_data->t_infomask2 &= ~HEAP_KEYS_UPDATED;
tp.t_data->t_infomask |= new_infomask;          /* compute_new_xmax_infomask 的结果 */
tp.t_data->t_infomask2 |= new_infomask2;
HeapTupleHeaderClearHotUpdated(tp.t_data);      /* 我是链尾，没有后继 */
HeapTupleHeaderSetXmax(tp.t_data, new_xmax);    /* xmax = 我的 XID：“我删的” */
HeapTupleHeaderSetCmax(tp.t_data, cid, iscombo);
tp.t_data->t_ctid = tp.t_self;                  /* ctid 指向自己：链到此为止 */
```

磁盘上**没有任何字节被删除**——元组还在，只是被打上"已被 XID=N 删除"的标记。若事务提交，等所有老快照消亡后由 prune/VACUUM 真正回收；若事务回滚，xmax 属于 aborted 事务，可见性判断自动无视它，行"复活"零成本。分区表间的行迁移（`DELETE + INSERT` 到另一分区）会额外调用 `HeapTupleHeaderSetMovedPartitions()`（heapam.c:3070）把 ctid 设为特殊值，让并发 UPDATE 能报出"行已迁移"错误。

### 6.3.2 行锁 = xmax + infomask 的位编码

PG 行锁**不进共享内存锁表**：锁一行就是把锁持有者的 XID 写进该行 `t_xmax`，锁的种类编码在 infomask 里。`src/include/access/htup_details.h` 的关键标志位：

```c
/* htup_details.h:194 起（节选） */
#define HEAP_XMAX_KEYSHR_LOCK  0x0010  /* xmax 是 KEY SHARE 锁持有者 */
#define HEAP_XMAX_EXCL_LOCK    0x0040  /* xmax 是排他锁持有者 */
#define HEAP_XMAX_LOCK_ONLY    0x0080  /* xmax 只是锁，不是删除 */
#define HEAP_XMAX_SHR_LOCK     (HEAP_XMAX_EXCL_LOCK | HEAP_XMAX_KEYSHR_LOCK)
#define HEAP_XMIN_COMMITTED    0x0100  /* 提示位：xmin 已提交 */
#define HEAP_XMIN_INVALID      0x0200  /* 提示位：xmin 已中止 */
#define HEAP_XMIN_FROZEN       (HEAP_XMIN_COMMITTED|HEAP_XMIN_INVALID)  /* 两位同置=冻结 */
#define HEAP_XMAX_COMMITTED    0x0400  /* 提示位：xmax 已提交 */
#define HEAP_XMAX_INVALID      0x0800  /* 提示位：xmax 已中止/无效 */
#define HEAP_XMAX_IS_MULTI     0x1000  /* xmax 是 MultiXactId 而非普通 XID */
#define HEAP_UPDATED           0x2000  /* 本行是 UPDATE 产生的新版本 */
```

解码规则：

| 行状态 | 编码 |
|---|---|
| `SELECT ... FOR KEY SHARE` | xmax=locker XID，`KEYSHR_LOCK + LOCK_ONLY` |
| `FOR SHARE` | `SHR_LOCK`（EXCL+KEYSHR 两位同置）`+ LOCK_ONLY` |
| `FOR NO KEY UPDATE` | `EXCL_LOCK + LOCK_ONLY`（不带 KEYS_UPDATED） |
| `FOR UPDATE` | `EXCL_LOCK + LOCK_ONLY + HEAP_KEYS_UPDATED`(infomask2) |
| 真删除/更新 | xmax=XID，**无** `LOCK_ONLY` |
| 多个锁持有者 | xmax=MultiXactId，`HEAP_XMAX_IS_MULTI`，成员及各自模式存在 multixact SLRU 里 |

判别函数 `HEAP_XMAX_IS_LOCKED_ONLY()`（htup_details.h:230）。这个设计的意义：**锁百万行不占一字节锁表内存**（InnoDB 的行锁位图虽也很省，但仍随锁行数增长且存在锁表结构）。代价是：等锁的实现变成"等 xmax 那个事务结束"——等待者对持有者的 XID 调常规锁 `XactLockTableWait()`，即等在"事务锁"上；以及**加行锁也是写操作**（要改页、写 WAL），所以 PG 里 `SELECT FOR UPDATE` 会产生脏页。四个提示位（XMIN/XMAX_COMMITTED/INVALID）是 CLOG 查询结果的元组级缓存，可在仅持共享级锁时设置（见 buffer/README 规则 4），这就是"大量只读也会产生写 I/O"的来源。

### 小结

UPDATE/DELETE 都不动旧数据字节，只盖 xmax 章；行锁复用 xmax 字段。HOT 的两个条件（同页 + 未改索引列）决定了写放大的量级——这也是 PG 调优里 fillfactor、"别索引不必要的列"两条经验的源码依据。

## 6.4 WAL 记录的组装与插入

上面各写函数遵循同一套 WAL 构造 API（`src/backend/access/transam/xloginsert.c`）：

```
XLogBeginInsert()                xloginsert.c:153   声明开始组装
XLogRegisterData(data, len)      xloginsert.c:372   登记“主数据”（不依附于页）
XLogRegisterBuffer(id, buf, fl)  xloginsert.c:246   登记第 id 号被修改的页
XLogRegisterBufData(id, ...)     xloginsert.c:413   登记依附于第 id 号页的数据
XLogInsert(rmid, info)           xloginsert.c:482   组装 + 写入 WAL 缓冲区，返回 LSN
```

`XLogInsert` 本体出乎意料地短（xloginsert.c:512–535）：

```c
do
{
    GetFullPageWriteInfo(&RedoRecPtr, &doPageWrites);   /* 取最近 checkpoint 的 Redo 点 */
    rdt = XLogRecordAssemble(rmid, info, RedoRecPtr, doPageWrites,
                             &fpw_lsn, &num_fpi, &fpi_bytes, &topxid_included);
    EndPos = XLogInsertRecord(rdt, fpw_lsn, curinsert_flags, ...);
} while (!XLogRecPtrIsValid(EndPos));   /* 若 RedoRecPtr 在组装期间变了则重组 */
```

**full-page write（FPW）的判定**在 `XLogRecordAssemble()`（xloginsert.c:621）里，对每个登记的页（xloginsert.c:679 起）：

```c
if (regbuf->flags & REGBUF_FORCE_IMAGE)       needs_backup = true;   /* 调用者强制 */
else if (regbuf->flags & REGBUF_NO_IMAGE)     needs_backup = false;  /* 调用者禁止 */
else if (!doPageWrites)                       needs_backup = false;  /* full_page_writes=off */
else
{
    XLogRecPtr page_lsn = PageGetLSN(regbuf->page);
    needs_backup = (page_lsn <= RedoRecPtr);  /* ★ 核心判据 */
    ...
}
```

**`page_lsn <= RedoRecPtr` 即"本页自最近一次 checkpoint 以来还没被 WAL 记录过"**——那么这是 checkpoint 后对它的第一次修改，必须把整页塞进 WAL。原因是"页撕裂"（torn page）：8KB 的页写到 4KB 时断电，磁盘上是半新半旧的垃圾，普通的增量 redo 记录无法在垃圾上重放；有了整页镜像，恢复时直接用镜像覆盖，再重放后续增量。之后同一页的第二次、第三次修改只记增量（页 LSN 已 > RedoRecPtr），直到下一个 checkpoint 重置起点。

组装时还有两个瘦身优化（xloginsert.c:703、728）：`needs_data = !needs_backup`——带了整页镜像就不必再记增量数据；`REGBUF_STANDARD` 页可以挖掉 `pd_lower` 到 `pd_upper` 之间的"空洞"（空闲空间）不写入镜像，还可用 pglz/lz4/zstd 压缩（`wal_compression`）。

> **对比 InnoDB doublewrite**：同一个 torn page 问题，InnoDB 的答案是把脏页先顺序写进 doublewrite buffer 再写到最终位置——付出每页双写的代价，redo 日志保持纯物理短记录；PG 的答案是把整页并入 WAL——WAL 在 checkpoint 后短时间内急剧膨胀（这就是 PG 监控里 checkpoint 之后 WAL 量尖峰的成因），但数据文件只写一次。两者殊途同归，权衡点不同：PG 可通过拉长 checkpoint 间隔摊薄 FPW 开销，InnoDB 则常年背着双写但对日志量友好。

`XLogInsertRecord()`（xlog.c）随后把记录拷入共享 WAL 缓冲区：先在极短的自旋保护下"预留"一段 LSN 区间（只推进指针），然后在 `WALInsertLock` 之外并发拷贝——多 backend 可以同时往 WAL 缓冲区不同位置写，这是 PG 9.4 WAL 并行插入改造的成果。

## 6.5 事务提交：CommitTransaction 全流程

### 6.5.1 CommitTransaction 逐段走读（xact.c:2270）

`src/backend/access/transam/xact.c:2270`。按顺序发生的事：

1. **用户代码收尾**（xact.c:2299–2313）：循环执行 `AfterTriggerFireDeferred()`（延迟触发器）与 `PreCommit_Portals()`（关游标），直到两者都无事可做——触发器可能开游标、关游标可能又激发触发器，所以是循环；
2. **各子系统 pre-commit**（xact.c:2322–2381）：扩展回调、并行 worker 清理、ON COMMIT 动作、`smgrDoPendingSyncs`（本事务新建的未记 WAL 的文件此刻 fsync）、NOTIFY 入队、SSI 的最终串行化检查（此处还可能报错回滚——这是最后的机会）；
3. **状态机切换**：`s->state = TRANS_COMMIT`（xact.c:2393），此后不允许再失败；
4. **`RecordTransactionCommit()`**（xact.c:2407）：写提交记录 + 刷 WAL + 更新 CLOG——持久化的核心，下节详解；
5. **`ProcArrayEndTransaction(MyProc, latestXid)`**（xact.c:2431）：把自己从"运行中事务"里除名——**从这一刻起，其他事务的新快照认为我已提交**，我的所有修改瞬间对新快照可见；
6. **资源释放**（xact.c:2449 起，顺序精心设计）：先释放对外可见资源（缓冲区 pin 等），然后 `AtEOXact_Inval(true)`（xact.c:2475）广播目录失效消息，**然后才释放锁**（xact.c:2479）——保证等我锁的人醒来时一定能看到我的 DDL 变更；最后 `smgrDoPendingDeletes(true)`（xact.c:2495）物理删除本事务 DROP 掉的表文件（放在锁后是因为删文件可能耗时数秒）。

注意第 4、5 步的顺序不可交换（注释 xact.c:2427）：必须先把提交写进 WAL/CLOG，再从 ProcArray 消失；否则会有一瞬间别人认为你"已结束"却在 CLOG 里查到"进行中"。

### 6.5.2 RecordTransactionCommit 的关键分支（xact.c:1345）

```c
TransactionId xid = GetTopTransactionIdIfAny();
bool markXidCommitted = TransactionIdIsValid(xid);   /* 有没有 XID？ */
...
wrote_xlog = (XactLastRecEnd != 0);                  /* 本事务写过 WAL 吗？ */
```

**分支一：无 XID 的只读事务**（xact.c:1384–1436）：不写提交记录、不碰 CLOG。若连 WAL 都没写过，直接 `goto cleanup`——**只读事务的提交是纯内存操作，零 I/O**。（例外：只做了 HOT prune 这类"顺手写 WAL 但没有 XID"的事务，仍要触发 WAL 刷盘。）

**分支二：有 XID 的写事务**（xact.c:1437–1513）：

```c
/* xact.c:1469 */
START_CRIT_SECTION();
MyProc->delayChkptFlags |= DELAY_CHKPT_IN_COMMIT;   /* 挡住并发 checkpoint：
        若 checkpoint 的 REDO 点切在“commit 记录已写、CLOG 未更新”中间，
        崩溃恢复会从 REDO 点开始而错过 CLOG 更新 → 已提交事务丢失 */
pg_write_barrier();
XactLogCommitRecord(...);                            /* 组装并 XLogInsert 提交记录：
        含提交时间戳、子事务 XID 列表、要删除的文件、失效消息等 */
```

接着是**同步/异步提交**的分岔（xact.c:1540）：

```c
if ((wrote_xlog && markXidCommitted &&
     synchronous_commit > SYNCHRONOUS_COMMIT_OFF) || forceSyncCommit || nrels > 0)
{
    XLogFlush(XactLastRecEnd);                       /* ★ 全事务唯一一次同步等磁盘 */
    if (markXidCommitted)
        TransactionIdCommitTree(xid, nchildren, children);   /* 刷完才能写 CLOG */
}
else
{
    XLogSetAsyncXactLSN(XactLastRecEnd);             /* 只是登记：请 walwriter 稍后刷 */
    if (markXidCommitted)
        TransactionIdAsyncCommitTree(xid, nchildren, children, XactLastRecEnd);
}
```

- **同步提交**（默认）：`XLogFlush` 把 WAL 刷到提交记录末尾。这是整条写路径上**唯一**必须等待 fsync 的点——数据页、索引页统统不刷；
- **异步提交**（`synchronous_commit=off`）：不等刷盘，把 LSN 交给 walwriter（xlog.c:2630 `XLogSetAsyncXactLSN`），最多丢最近 ~3 倍 `wal_writer_delay` 的已提交事务，但**绝不会损坏数据**（WAL 仍按序完整，只是尾部丢失）——等价于 InnoDB 的 `innodb_flush_log_at_trx_commit=2` 的风险模型；
- 细节：异步路径的 CLOG 更新（`TransactionIdAsyncCommitTree`，transam.c:252）带上了提交记录 LSN——将来若有人要把携带该 XID 提示位的堆页刷盘，SLRU 层能先补刷 WAL，维持 WAL-before-data；
- `nrels > 0`（事务里 DROP 过表）强制同步：文件删除不可逆，必须确保提交记录先落盘。

### 6.5.3 XLogFlush 与组提交（xlog.c:2800）

`XLogFlush()` 的循环（xlog.c:2848–2926）是教科书级的组提交实现：

```c
for (;;)
{
    RefreshXLogWriteResult(LogwrtResult);
    if (record <= LogwrtResult.Flush) break;      /* 别人已顺手把我刷了：白嫖成功 */
    ...
    insertpos = WaitXLogInsertionsToFinish(WriteRqstPtr);  /* 等在飞的 WAL 拷贝完成 */
    if (!LWLockAcquireOrWait(WALWriteLock, LW_EXCLUSIVE))
        continue;       /* 拿不到写锁就睡到锁释放，醒来先回头检查是否已被代刷 */
    ...
    if (CommitDelay > 0 && enableFsync && MinimumActiveBackends(CommitSiblings))
        pg_usleep(CommitDelay);   /* 可选：故意多睡一会儿，攒更多同伴 */
    WriteRqst.Write = insertpos;  /* 刷到当前最新插入位置，而不只是自己的 record */
    WriteRqst.Flush = insertpos;
    XLogWrite(WriteRqst, insertTLI, false);       /* write + fsync */
    LWLockRelease(WALWriteLock);
    break;
}
```

组提交的三重机制：① `LWLockAcquireOrWait` 让等待者睡在锁上，锁释放时**先检查自己是否已被持锁者顺手刷掉**，避免无谓的重复 fsync；② 拿到锁的人把刷盘目标扩大到**当前所有已插入的 WAL**（`insertpos`），一次 fsync 送走一批事务；③ `commit_delay/commit_siblings` 提供显式的"等一等攒一批"（默认关闭）。与 InnoDB 的 group commit（binlog 三阶段之前的 redo group commit）机制思想完全一致。

### 6.5.4 CLOG 更新：TransactionIdCommitTree（transam.c:240）

CLOG（目录 `pg_xact`，代码 `clog.c`）是每事务 2 bit 的巨型位图：IN_PROGRESS(00)/COMMITTED(01)/ABORTED(10)/SUB_COMMITTED(11)，一个 8KB 页存 32K 个事务的状态。`TransactionIdCommitTree` → `TransactionIdSetTreeStatus()`（`src/backend/access/transam/clog.c:192`）要解决**主事务 + 子事务的原子提交**：全家都在同一 CLOG 页时，持一次页锁全部置位，天然原子；跨页时按"先把异页子事务标 SUB_COMMITTED → 原子地把主事务及同页子事务标 COMMITTED → 再把异页子事务改 COMMITTED"三步走（clog.c:161–182 的注释给出了完整示例）——并发观察者对 SUB_COMMITTED 的解读是"去看我父事务的状态"，因此对外整体表现为原子。

### 6.5.5 ProcArrayEndTransaction 的原子性意义（procarray.c:663）

```c
/* procarray.c:680 */
if (LWLockConditionalAcquire(ProcArrayLock, LW_EXCLUSIVE))
{
    ProcArrayEndTransactionInternal(proc, latestXid);   /* 清 xid/xmin，推进
                                                           latestCompletedXid，
                                                           xactCompletionCount++ */
    LWLockRelease(ProcArrayLock);
}
else
    ProcArrayGroupClearXid(proc, latestXid);            /* 锁忙：组清除 */
```

为什么必须持 `ProcArrayLock` 排他锁清除自己的 XID？注释（procarray.c:668）指向 transam/README：**快照的原子性**。`GetSnapshotData()` 持共享锁扫描 ProcArray；如果提交者不加锁就消失，拍快照的人可能看到"半个提交"——例如快照认为 XID=100 已结束（不在运行列表），但 100 的子事务 101 还在列表里，于是同一事务的修改一半可见一半不可见。持锁清除保证：任何快照要么把我整个算作"运行中"（我的一切不可见），要么整个算作"已完成"（配合 CLOG=committed，我的一切可见）。**这把锁就是 PG 里"提交对外原子生效"的仲裁点。**

高并发下所有提交者抢这一把排他锁会成为瓶颈，于是有 `ProcArrayGroupClearXid()`（procarray.c:784）：拿不到锁的提交者把自己挂进一个无锁链表，第一个成员代表全组拿一次锁、批量清除所有人的 XID——与 XLogFlush 的组提交异曲同工，把 N 次锁交接合并成 1 次。

### 6.5.6 AbortTransaction 对比（xact.c:2855）

回滚路径惊人地"轻"：

1. 先做异常环境清理（xact.c:2862–2920）：释放 LWLock、缓冲区锁、等待状态——因为 abort 可能发生在任何 `elog(ERROR)` 的 longjmp 之后，环境是"脏"的；
2. `RecordTransactionAbort()`（xact.c:1795）：有 XID 才写一条 `XLOG_XACT_ABORT` 记录，然后 `TransactionIdAbortTree()`（调用点 xact.c:1893）把 CLOG 标成 ABORTED。**关键：不需要 XLogFlush**（注释解释：崩溃后查不到提交记录的事务本来就被视为 aborted，abort 记录丢了也无所谓）；也**没有 DELAY_CHKPT 保护**（丢失 abort 标记无害）；
3. `ProcArrayEndTransaction()` 同样调用——从运行列表消失后，别人查 CLOG 得到 aborted，我写的所有版本永久不可见；
4. **不碰任何数据页**。已插入的元组、已盖的 xmax 原样留在页里，成为死元组等 VACUUM 收拾。

这就是指南 0.2 节"回滚免费"的完整源码依据：InnoDB 回滚要逆向应用 undo（大事务回滚可能比执行还慢），PG 回滚是 O(1) 的标记操作——代价延迟支付给了 VACUUM。

### 小结

提交 = 写 commit 记录 → XLogFlush（唯一的等待点，有组提交）→ CLOG 置位（SLRU 上的两个 bit）→ ProcArray 除名（持锁保证快照原子性）→ 放锁清资源。回滚只是 CLOG 标 ABORTED，数据页零改动。同步与异步提交的差别仅在"等不等 XLogFlush"。

## 全景回顾：一条 UPDATE + COMMIT 的 WAL 足迹

```
BEGIN;                          -- 无任何 I/O，连 XID 都没有
UPDATE t SET v=v+1 WHERE id=1;  -- GetCurrentTransactionId() 分配 XID
                                -- heap_update:
                                --   (可能) XLOG_HEAP_LOCK    跨页时的占位锁
                                --   XLOG_HEAP_UPDATE / HOT_UPDATE   旧页+新页(+VM页)
                                --   (非 HOT) 每个索引一条 btree INSERT_LEAF 记录
COMMIT;                         -- XLOG_XACT_COMMIT
                                -- XLogFlush ← 唯一 fsync 等待
                                -- CLOG: xid → COMMITTED
                                -- ProcArray: 除名，全世界可见
```

用 `pg_waldump` 对照验证这张表，是本章最好的收尾实验。

## 自测问题

**1. 为什么 `MarkBufferDirty` 必须在 `XLogInsert` 之前、而 `PageSetLSN` 在其之后？**

简答：若先写 WAL 后标脏，checkpoint 可能恰好在两者之间确定脏页集合，该页既没被 checkpoint 写出、其 WAL 又在 REDO 点之前，恢复时这条修改就丢了。而 `PageSetLSN` 需要 `XLogInsert` 的返回值（记录末尾 LSN），天然在其后；页头 LSN 的作用是让未来刷此页的人先 `XLogFlush(pd_lsn)`，维持 WAL-before-data。

**2. 一条 UPDATE 满足什么条件才是 HOT 更新？HOT 省掉了什么、没省掉什么？**

简答：两个条件（heapam.c:4063）：新版本与旧版本同页；`modified_attrs` 与 `hot_attrs`（所有非摘要索引引用的列）无交集。省掉了所有索引的新条目插入（`update_indexes=TU_None`）以及未来 VACUUM 扫索引删项的成本；没省掉的：新版本元组本身照样占堆空间、照样写 WAL，旧版本照样要被 prune 回收（只是 prune 可以单页完成）。

**3. `SELECT ... FOR UPDATE` 锁 100 万行，共享内存锁表会爆吗？等待这些行锁的人在等什么？**

简答：不会。行锁写在每行的 `t_xmax` + infomask（`HEAP_XMAX_LOCK_ONLY` 等位）里，不占锁表。等待者对 xmax 中的 XID 调 `XactLockTableWait()`——等的是那个"事务锁"（每个活跃事务在锁表里的唯一条目），事务结束释放事务锁，所有等它行锁的人被唤醒重查。锁表里真正的风险点是每行第一次加锁瞬间的 tuple 锁与大量表锁，而非行锁本身。

**4. full-page write 的判定条件是什么？为什么 checkpoint 调频会影响 WAL 量？InnoDB 如何解决同一问题？**

简答：`XLogRecordAssemble`（xloginsert.c:694）中 `page_lsn <= RedoRecPtr` 且 `doPageWrites` 为真——即"最近一次 checkpoint 后第一次修改该页"。checkpoint 越频繁，"第一次"越多，整页镜像越多，WAL 越大；拉长 `checkpoint_timeout` 可显著摊薄。InnoDB 用 doublewrite buffer：脏页先写双写区再写原位，redo 保持短记录，代价是数据页双写。

**5. `synchronous_commit=off` 时，崩溃后可能出现"数据页有新数据但事务标记为未提交"吗？可能丢已提交事务吗？会数据损坏吗？**

简答：三问分别是：可能——脏页可以先于提交刷盘（steal 策略），但恢复后该 XID 在 CLOG 中非 committed，这些行不可见，等 VACUUM 清理，无害；可能——最近未刷盘的 commit 记录丢失，客户端收到过成功应答的事务被回退（这是 off 的代价）；不会——WAL 本身仍严格顺序完整（`TransactionIdAsyncCommitTree` 还确保 CLOG 页刷盘前先刷对应 WAL），数据文件的一致性不受影响。风险模型等价于 InnoDB `innodb_flush_log_at_trx_commit=2`。
