## 前置知识

- **元组头**（`src/include/access/htup_details.h`）：每行数据前有一个 `HeapTupleHeaderData`，其中 `t_xmin`（创建者事务 ID）、`t_xmax`（删除/加锁者事务 ID）、`t_ctid`（版本链指针）、`t_infomask`/`t_infomask2`（标志位）是本章的主角。
- **页布局**（`src/include/storage/bufpage.h`）：页头 `pd_lsn`（bufpage.h:187）记录"最后一次修改本页的 WAL 记录的结束位置"，是 WAL-before-data 的执行抓手；`pd_lower`/`pd_upper` 之间是空闲空间（"空洞"）。
- **临界区**（critical section）：`START_CRIT_SECTION()`（`src/include/miscadmin.h:152`，本体只是 `CritSectionCount++`）到 `END_CRIT_SECTION()` 之间，任何 `elog(ERROR)` 都会被 `errstart` 升级为 PANIC（`src/backend/utils/error/elog.c:372`）。为什么必须如此，见 6.6.1 的完整推演。
- **WAL-before-data**：脏数据页刷盘之前，覆盖它的 WAL 必须先落盘。形式化表述与代码强制点见 6.6.2。
- **与 InnoDB 的根本差异**：InnoDB 的 UPDATE 原地改行、旧值进 undo 段；PG 没有 undo，新旧版本都躺在堆页里，靠 `xmin`/`xmax` + CLOG 决定谁能看见谁。这一个设计差异决定了本章几乎所有代码的形状（正反论证见 6.6.3）。

## 6.0 写路径不变式清单

写路径的每一行代码都在维护下面这些不变式。先立规矩再读代码，你会发现那些"看似啰嗦"的顺序约束全都各有归属。每条给出"违反后果"——这是理解"为什么代码非这么写不可"的钥匙。

| # | 不变式 | 代码依据 | 违反后果 |
|---|---|---|---|
| I1 | **页 LSN 单调不减**：`pd_lsn` 只会被更大的 LSN 覆盖（每次修改后 `PageSetLSN` 盖上本次记录的结束 LSN） | bufpage.h:417 `PageSetLSN`；每个写函数临界区末尾 | FPW 判定 `page_lsn <= RedoRecPtr`（xloginsert.c:694）失真：本该带整页镜像的记录没带，torn page 无法修复 |
| I2 | **WAL-before-data**：数据页落盘前，`XLogFlush(pd_lsn)` 必须先完成 | bufmgr.c:4585（FlushBuffer 中） | 崩溃后磁盘上有"没有日志描述的修改"，redo 无从对齐，见 6.6.2 推演 |
| I3 | **修改共享页必须三件套**：排他内容锁 + 临界区 + WAL（临时表/unlogged 免 WAL） | heap_insert 等所有写函数 | 无锁：读者看到半个元组；无临界区：ERROR 后页改了日志没写，见 6.6.1；无 WAL：崩溃丢修改 |
| I4 | **pin 后才能读/改页内容**：任何对 `Page` 指针的解引用前，缓冲区必须已 pin（否则页可能被换出复用） | bufmgr README；`ReadBuffer` 返回即带 pin | 读到的是别的页的内容——典型的 use-after-free |
| I5 | **临界区内不做内存分配**：`palloc` 在临界区内触发断言 `Assert(CritSectionCount == 0 \|\| context->allowInCritSection)`（mcxt.c:198–199） | 所以 `ExtractReplicaIdentity` 等分配都在 `START_CRIT_SECTION` 之前（heapam.c:4091–4101 注释明说） | 分配失败 → ERROR → 升级 PANIC，把"可恢复的内存不足"变成全库重启 |
| I6 | **临界区内不做 I/O、不等锁**：VM 页、CLOG 页、FSM 页都要在临界区外先 pin/读好 | heap_insert 先 pin vmbuffer（heapam.c:2030–2037）；XID 在函数第一行分配 | I/O 失败 → ERROR → PANIC；等锁 → 持排他页锁睡眠，全系统卡死 |
| I7 | **MarkBufferDirty 先于 XLogInsert** | heapam.c:2098 vs 2178；xloginsert.c:266 `Assert(BufferIsDirty(buffer))` 强制 | checkpoint 可能在"WAL 已写、页未标脏"窗口确定脏页集合：该页不被写出、其 WAL 又在下个 REDO 点之前，恢复时修改丢失 |
| I8 | **同关系多页加锁按块号升序** | hio.c:450–456 注释；heapam.c:3979–3986 注释 | 两个 backend 交叉更新同两页 → 死锁（且缓冲区锁是轻量锁，无死锁检测，是永久卡死） |
| I9 | **提交顺序：WAL flush → CLOG → ProcArray 除名 → 放锁** | xact.c:1544–1550、2426–2431 注释 | 任意交换都产生"半提交可见"或"已答复的提交丢失"，见 6.7 |
| I10 | **任何可能随页落盘的 xid 必须先有 WAL**：跨页 UPDATE 中途的"占位锁"也要写 `xl_heap_lock`（heapam.c:3859–3864 注释） | heapam.c:3888–3946 | 崩溃后页上出现一个 WAL 里从未出现过的 xmax，备库/恢复无法解释 |

> 对照 InnoDB：I2 对应 redo 的 `flush_lsn` 约束（buf_flush 前先 `log_write_up_to`），I7 对应 mtr（mini-transaction）提交时"先加入 flush list 再释放 latch"的纪律，I8 对应 InnoDB 内部按固定顺序拿 page latch。两边都是 ARIES 家族，不变式几乎一一对应；差别在 PG 没有 undo，所以少了"回滚也要写日志"的一组不变式——代价在别处偿还（6.6.3）。

## 6.1 heap_insert：一次插入的完整解剖（全函数走读）

入口在 `src/backend/access/heap/heapam.c:2004–2219`。本节覆盖全函数每一行的逻辑；元组体拷贝的字节细节在 `RelationPutHeapTuple`（hio.c:35）内，属机械操作，仅概述。

### 6.1.1 函数开头：惰性 XID 分配（heapam.c:2004–2020)

```c
/* heapam.c:2004 */
void
heap_insert(Relation relation, HeapTuple tup, CommandId cid,
			uint32 options, BulkInsertState bistate)
{
	TransactionId xid = GetCurrentTransactionId();
	HeapTuple	heaptup;
	Buffer		buffer;
	Page		page;
	Buffer		vmbuffer = InvalidBuffer;
	bool		clear_all_visible = false;
	bool		vmbuffer_modified = false;

	/* Cheap, simplistic check that the tuple matches the rel's rowtype. */
	Assert(HeapTupleHeaderGetNatts(tup->t_data) <=
		   RelationGetNumberOfAttributes(relation));

	AssertHasSnapshotForToast(relation);
```

`GetCurrentTransactionId()`（xact.c:456–463）体现了 PG 的**惰性 XID 分配**：

```c
/* xact.c:456 */
TransactionId
GetCurrentTransactionId(void)
{
	TransactionState s = CurrentTransactionState;

	if (!FullTransactionIdIsValid(s->fullTransactionId))
		AssignTransactionId(s);
	return XidFromFullTransactionId(s->fullTransactionId);
}
```

事务开始（BEGIN）时并不分配 XID，只有第一次真正需要在元组上盖"作者签名"时才调用 `AssignTransactionId()`（xact.c:637）向共享计数器要一个新 XID。纯只读事务全程没有 XID——这就是为什么长时间的只读 SELECT 不会消耗 XID、不会加剧回卷（wraparound）压力。`AssignTransactionId` 还维护一条子事务不变式：**子事务的 XID 必须晚于父事务**，所以它会先迭代地给所有还没有 XID 的祖先分配（xact.c:662–684 用显式栈避免深递归）。对比 InnoDB：只读事务同样有 trx_id 延迟分配优化（5.6+），思想一致。

注意 `xid` 的获取是**函数第一行**：分配 XID 本身可能等锁（XidGenLock）甚至做 I/O（扩展 CLOG 页），这些都不允许发生在持有缓冲区排他锁之后（不变式 I6）。

### 6.1.2 heap_prepare_insert：填充元组头与 TOAST 判断（heapam.c:2028、2227–2268）

```c
	heaptup = heap_prepare_insert(relation, tup, xid, cid, options);
```

`heap_prepare_insert`（heapam.c:2227）先拒绝并行 worker 中的插入（无法广播新 combo CID），然后逐字段填元组头：

```c
/* heapam.c:2242 */
	tup->t_data->t_infomask &= ~(HEAP_XACT_MASK);
	tup->t_data->t_infomask2 &= ~(HEAP2_XACT_MASK);
	tup->t_data->t_infomask |= HEAP_XMAX_INVALID;
	HeapTupleHeaderSetXmin(tup->t_data, xid);
	if (options & HEAP_INSERT_FROZEN)
		HeapTupleHeaderSetXminFrozen(tup->t_data);

	HeapTupleHeaderSetCmin(tup->t_data, cid);
	HeapTupleHeaderSetXmax(tup->t_data, 0); /* for cleanliness */
	tup->t_tableOid = RelationGetRelid(relation);
```

- 先清掉所有事务相关标志位，再置 `HEAP_XMAX_INVALID` 提示位（"没人删过它"）；`xmin = 我的 XID` 即"我创建的"。
- `cid`（CommandId）解决"同一事务内，第 3 条语句能否看到第 1 条语句插入的行"（能）以及"游标打开后插入的行游标是否可见"（不可见）。
- `HEAP_INSERT_FROZEN`（COPY FREEZE 等场景）直接把元组标成冻结，跳过未来的冻结开销。

随后是 **TOAST 判断**（heapam.c:2253–2267）：只有普通表和物化视图才走 toaster；元组含外部 TOAST 指针、或 `t_len > TOAST_TUPLE_THRESHOLD`（约 2KB）时调用 `heap_toast_insert_or_update()` 把超长字段切片压缩存入 toast 表，主元组里只留 18 字节指针，函数返回瘦身后的副本（`heaptup != tup`，函数末尾 heapam.c:2214–2218 负责把 `t_self` 拷回并释放副本）。对应 InnoDB 的行溢出页（off-page column），但 PG 的 TOAST 是独立的表 + 索引，切片粒度约 2KB 且自带 4 种存储策略（PLAIN/EXTENDED/EXTERNAL/MAIN）。

### 6.1.3 选址（heapam.c:2030–2039）

```c
	/*
	 * Find buffer to insert this tuple into.  If the page is all visible,
	 * this will also pin the requisite visibility map page.
	 */
	buffer = RelationGetBufferForTuple(relation, heaptup->t_len,
									   InvalidBuffer, options, bistate,
									   &vmbuffer, NULL,
									   0);

	page = BufferGetPage(buffer);
```

返回时目标页已被 pin 住并加了排他内容锁；若目标页是 all-visible，对应 VM 页也已 pin 好（`vmbuffer`）——因为稍后要在临界区里清 VM 位，而 VM 页 I/O 绝不能发生在临界区内（不变式 I6）。该函数本身是 6.2 的主角。

### 6.1.4 SSI 检查与 VM 页加锁（heapam.c:2041–2063）

```c
	CheckForSerializableConflictIn(relation, NULL, InvalidBlockNumber);

	/* Lock the vmbuffer before the critical section */
	if (PageIsAllVisible(page))
	{
		LockBuffer(vmbuffer, BUFFER_LOCK_EXCLUSIVE);
		clear_all_visible = true;
	}
```

- `CheckForSerializableConflictIn()` 只在 SERIALIZABLE 隔离级别有实际工作（SSI 的写侧检查）。注释（heapam.c:2042–2054）解释了两点：放在临界区外是因为万一要报错，此刻回滚成本最低；对 INSERT 只需查表级 SSI 锁——新元组不可能与既有元组锁冲突，而 heap 页锁只是元组锁的合并形式、不像索引页锁那样锁"间隙"。
- VM 页的**排他锁**也在临界区前拿好（20devel 新行为：VM 修改被纳入更严格的锁协议），`clear_all_visible` 记住"稍后要清位"。

### 6.1.5 临界区：页修改与 WAL 的原子配合（heapam.c:2065–2186）

这是全章最重要的代码模式，逐行看（以下与源码逐字一致，仅省略中间注释块的英文原文）：

```c
	/* NO EREPORT(ERROR) from here till changes are logged */
	START_CRIT_SECTION();

	RelationPutHeapTuple(relation, buffer, heaptup,
						 (options & HEAP_INSERT_SPECULATIVE) != 0);

	if (clear_all_visible)
	{
		/* It's possible the VM bits were already clear */
		if (visibilitymap_clear(relation->rd_locator,
								ItemPointerGetBlockNumber(&(heaptup->t_self)),
								vmbuffer, VISIBILITYMAP_VALID_BITS))
			vmbuffer_modified = true;

		PageClearAllVisible(page);
	}

	if (TransactionIdIsNormal(xid) && !(options & HEAP_INSERT_FROZEN))
		PageSetPrunable(page, xid);

	MarkBufferDirty(buffer);
```

- `RelationPutHeapTuple`（hio.c:35）把元组真正写进页：`PageAddItem` 分配 line pointer、把元组体拷到 `pd_upper` 之下、更新 `pd_lower`/`pd_upper`，并把分到的 TID 写回 `heaptup->t_self`。
- `PageSetPrunable(page, xid)` 在页头 `pd_prune_xid` 记个提示。源码注释（heapam.c:2082–2094）给了两个动机：页满后触发 `heap_page_prune_and_freeze()` 以便下次访问时把页标成 all-visible；以及**万一本事务 abort**，这行就是死元组——若页上没有别的 UPDATE/DELETE，没有这个提示它会一直躺到下次 VACUUM 才被发现。
- `MarkBufferDirty(buffer)` 置 `BM_DIRTY`——必须在写 WAL 之前（不变式 I7，由 xloginsert.c:266 的 `Assert(BufferIsDirty(buffer))` 强制）。

接着是 WAL 块（heapam.c:2100–2184，`RelationNeedsWAL` 为假即临时表/unlogged 表时整块跳过）：

```c
	/* XLOG stuff */
	if (RelationNeedsWAL(relation))
	{
		xl_heap_insert xlrec;
		xl_heap_header xlhdr;
		XLogRecPtr	recptr;
		uint8		info = XLOG_HEAP_INSERT;
		int			bufflags = 0;

		if (RelationIsAccessibleInLogicalDecoding(relation))
			log_heap_new_cid(relation, heaptup);

		if (ItemPointerGetOffsetNumber(&(heaptup->t_self)) == FirstOffsetNumber &&
			PageGetMaxOffsetNumber(page) == FirstOffsetNumber)
		{
			info |= XLOG_HEAP_INIT_PAGE;
			bufflags |= REGBUF_WILL_INIT;
		}

		xlrec.offnum = ItemPointerGetOffsetNumber(&heaptup->t_self);
		xlrec.flags = 0;
		if (clear_all_visible)
			xlrec.flags |= XLH_INSERT_ALL_VISIBLE_CLEARED;
		if (options & HEAP_INSERT_SPECULATIVE)
			xlrec.flags |= XLH_INSERT_IS_SPECULATIVE;
		...
		if (RelationIsLogicallyLogged(relation) &&
			!(options & HEAP_INSERT_NO_LOGICAL))
		{
			xlrec.flags |= XLH_INSERT_CONTAINS_NEW_TUPLE;
			bufflags |= REGBUF_KEEP_DATA;
			...
		}

		XLogBeginInsert();
		XLogRegisterData(&xlrec, SizeOfHeapInsert);

		xlhdr.t_infomask2 = heaptup->t_data->t_infomask2;
		xlhdr.t_infomask = heaptup->t_data->t_infomask;
		xlhdr.t_hoff = heaptup->t_data->t_hoff;

		XLogRegisterBuffer(HEAP_INSERT_BLKREF_HEAP, buffer,
						   REGBUF_STANDARD | bufflags);
		XLogRegisterBufData(HEAP_INSERT_BLKREF_HEAP, &xlhdr,
							SizeOfHeapHeader);
		XLogRegisterBufData(HEAP_INSERT_BLKREF_HEAP,
							(char *) heaptup->t_data + SizeofHeapTupleHeader,
							heaptup->t_len - SizeofHeapTupleHeader);

		XLogSetRecordFlags(XLOG_INCLUDE_ORIGIN);

		if (vmbuffer_modified)
			XLogRegisterBuffer(HEAP_INSERT_BLKREF_VM, vmbuffer, 0);

		recptr = XLogInsert(RM_HEAP_ID, info);

		PageSetLSN(page, recptr);

		if (vmbuffer_modified)
			PageSetLSN(BufferGetPage(vmbuffer), recptr);
	}

	END_CRIT_SECTION();
```

关键设计点：

- **先改页、再写 WAL、最后盖 LSN**，全程持排他锁 + 临界区。顺序看似与"日志先行"矛盾，其实不然：WAL-before-data 约束的是**落盘顺序**而非内存中的修改顺序（形式化见 6.6.2）。内存里先改页没关系，因为 `PageSetLSN` 把这条 WAL 记录的结束位置写进了页头 `pd_lsn`，以后任何人想把这个脏页刷盘，都必须先 `XLogFlush(pd_lsn)`。页头 LSN 就是那根"栓住数据页的绳子"。
- **XLOG_HEAP_INIT_PAGE 优化**（heapam.c:2121–2126）：插入的是页上第一条也是唯一一条元组时，置 `REGBUF_WILL_INIT`——重放时直接重新初始化空页，这条记录也就永远不需要 full-page image（`XLogRecordAssemble` 对 WILL_INIT 页走 `REGBUF_NO_IMAGE` 等价路径的判断见 6.5）。批量装载（COPY）的 WAL 量因此显著减小。
- **元组数据登记在页名下**（`XLogRegisterBufData`）而不是主数据区：这样当这条记录最终决定带 full-page image 时，元组内容已经包含在页镜像里，可以自动省略（`needs_data = !needs_backup`，xloginsert.c:708），避免双份存储。逻辑复制是例外——解码需要元组本身，所以 `REGBUF_KEEP_DATA` 强制保留（heapam.c:2136–2145 注释）。
- **一条记录登记两个页**：堆页 + VM 页。恢复时原子重放，VM 位与页头 `PD_ALL_VISIBLE` 不会脱节。

### 6.1.6 收尾（heapam.c:2188–2218）

```c
	UnlockReleaseBuffer(buffer);

	if (clear_all_visible)
		LockBuffer(vmbuffer, BUFFER_LOCK_UNLOCK);
	if (BufferIsValid(vmbuffer))
		ReleaseBuffer(vmbuffer);

	CacheInvalidateHeapTuple(relation, heaptup, NULL);

	pgstat_count_heap_insert(relation, 1);

	if (heaptup != tup)
	{
		tup->t_self = heaptup->t_self;
		heap_freetuple(heaptup);
	}
}
```

`CacheInvalidateHeapTuple()` 只对系统目录表有实际动作（登记 sinval 消息，abort 时也要能失效缓存）；源码注释指出放在释放缓冲区之后是安全的，因为 `heaptup` 全在本地内存。注意**数据页从头到尾没有刷盘**——它只是躺在共享缓冲区里变成了脏页，何时落盘是 checkpointer/bgwriter 的事（06 章）。

### 小结

heap_insert = 拿 XID → 填元组头（可能 TOAST）→ 选址（连带 pin VM）→ SSI 检查 → 临界区内{写页 + 清 VM + 标脏 + 组装 WAL + 盖 LSN} → 解锁 → 缓存失效与统计。两条铁律贯穿始终：修改共享页必须"排他锁 + 临界区 + WAL"三件套（I3）；数据页永不主动刷盘，靠页头 LSN 与 WAL 挂钩（I1/I2）。

## 6.2 RelationGetBufferForTuple：选址与扩展的并发协议（全函数走读）

`src/backend/access/heap/hio.c:499–884`。这个函数解决"把一条长 `len` 的元组放进哪个页"的问题，难点全在并发：FSM 信息可能过时、两个 backend 可能同时扩展文件、heap_update 还要求同时锁两个页而不死锁。函数头注释（hio.c:434–498）本身就是一份协议说明书，值得先读。

### 6.2.1 入参协议与前置检查（hio.c:499–556）

```c
/* hio.c:499 */
Buffer
RelationGetBufferForTuple(Relation relation, Size len,
						  Buffer otherBuffer, uint32 options,
						  BulkInsertState bistate,
						  Buffer *vmbuffer, Buffer *vmbuffer_other,
						  int num_pages)
```

- `otherBuffer`：heap_update 专用——旧元组所在页的缓冲区（已 pin 未锁）。函数返回时**两个页都持排他锁**。为什么要把加锁工作收进本函数？注释（hio.c:450–456）：两个 backend 交叉更新同两页时若各自按不同顺序加锁会死锁，所以强制"**同一关系的页按块号升序加锁**"（不变式 I8），而实现这条规则最方便的位置就是这里。
- `num_pages`：调用者预估还要多少页（`heap_multi_insert` 传大值），决定批量扩展的下限。
- 开头的检查：`len = MAXALIGN(len)` 保守对齐；超过 `MaxHeapTupleSize`（约 8140B）直接报错——**此刻还允许 ereport(ERROR)**，注释（hio.c:496–497）特别强调本函数必须在缓冲池里出现任何未记日志的修改之前调用。

### 6.2.2 fillfactor 与目标空闲空间（hio.c:536–551）

```c
	saveFreeSpace = RelationGetTargetPageFreeSpace(relation,
												   HEAP_DEFAULT_FILLFACTOR);

	nearlyEmptyFreeSpace = MaxHeapTupleSize -
		(MaxHeapTuplesPerPage / 8 * sizeof(ItemIdData));
	if (len + saveFreeSpace > nearlyEmptyFreeSpace)
		targetFreeSpace = Max(len, nearlyEmptyFreeSpace);
	else
		targetFreeSpace = len + saveFreeSpace;
```

- `saveFreeSpace` 是 fillfactor 换算出的预留字节数：INSERT 主动不把页塞满，给未来同页的 HOT 更新留余地。注释（hio.c:491–494）指出一个微妙点：**同页 UPDATE 不经过本函数**，所以 fillfactor 留出的空间恰好只被 HOT 更新消费，不会被后来的 INSERT 抢走。这与 InnoDB 页预留 1/16 空间思路相似，但 PG 的动机主要是 HOT 而不是 B-tree 分裂。
- `nearlyEmptyFreeSpace` 处理一个边角：低 fillfactor 的表插入大元组时，`len + saveFreeSpace` 可能超过任何页所能提供的空间，导致每插一条就扩展一页。于是"几乎空的页"（只有少量 line pointer 残留）也被视为可用。

### 6.2.3 初始目标页：缓存 → FSM → 最后一页（hio.c:571–596）

```c
	if (bistate && bistate->current_buf != InvalidBuffer)
		targetBlock = BufferGetBlockNumber(bistate->current_buf);
	else
		targetBlock = RelationGetTargetBlock(relation);

	if (targetBlock == InvalidBlockNumber && use_fsm)
	{
		targetBlock = GetPageWithFreeSpace(relation, targetFreeSpace);
	}

	if (targetBlock == InvalidBlockNumber)
	{
		BlockNumber nblocks = RelationGetNumberOfBlocks(relation);

		if (nblocks > 0)
			targetBlock = nblocks - 1;
	}
```

三级 fallback：① 上次插入的页（BulkInsertState 里 pin 着的页，或 relcache 的 `smgr_targblock`），命中则零成本；② 查 FSM——`GetPageWithFreeSpace()`（freespace.c:137）在 FSM 的三层"最大堆"树里找一个空闲空间 ≥ 需求的页；③ FSM 一无所知（比如刚建库、FSM 还没建起来）就试最后一页，避免"每元组一页"综合征。

### 6.2.4 主循环：加锁顺序与 VM pin 的两阶段检查（hio.c:598–680）

```c
loop:
	while (targetBlock != InvalidBlockNumber)
	{
		if (otherBuffer == InvalidBuffer)
		{
			/* easy case */
			buffer = ReadBufferBI(relation, targetBlock, RBM_NORMAL, bistate);
			if (PageIsAllVisible(BufferGetPage(buffer)))
				visibilitymap_pin(relation, targetBlock, vmbuffer);
			...
			LockBuffer(buffer, BUFFER_LOCK_EXCLUSIVE);
		}
		else if (otherBlock == targetBlock)
		{
			/* also easy case */
			buffer = otherBuffer;
			...
		}
		else if (otherBlock < targetBlock)
		{
			/* lock other buffer first */
			buffer = ReadBuffer(relation, targetBlock);
			...
			LockBuffer(otherBuffer, BUFFER_LOCK_EXCLUSIVE);
			LockBuffer(buffer, BUFFER_LOCK_EXCLUSIVE);
		}
		else
		{
			/* lock target buffer first */
			...
		}
```

四分支就是不变式 I8 的直译：单页直接锁；同页只锁一次；两页按块号升序锁。**VM pin 的两阶段检查**是另一个精彩细节：`PageIsAllVisible` 在**加锁前**先看一眼（此时结果可能错），需要就先把 VM 页 pin 好（pin 可能引发 I/O，不能在持锁后做，注释 hio.c:606–612）；加锁后再用 `GetVisibilityMapPins()`（hio.c:678）复查——若锁后发现位变了，它会放锁、补 pin、重新加锁。"先乐观检查、拿锁后复核、必要时退避重试"是 PG 处理"锁内禁 I/O"的标准手法。

### 6.2.5 空间判定与 FSM 反馈循环（hio.c:686–763）

```c
		page = BufferGetPage(buffer);

		if (PageIsNew(page))
		{
			PageInit(page, BufferGetPageSize(buffer), 0);
			MarkBufferDirty(buffer);
		}

		pageFreeSpace = PageGetHeapFreeSpace(page);
		if (targetFreeSpace <= pageFreeSpace)
		{
			/* use this page as future insert target, too */
			RelationSetTargetBlock(relation, targetBlock);
			return buffer;
		}
```

空间够就记住这个页（下次直接命中缓存级）并返回——**这是函数的正常出口**。空间不够则放锁放 pin，然后三选一（hio.c:724–762）：

1. **批量扩展余粮**：上次批量扩展的页还没用完（`bistate->next_free`），直接取下一个，不查 FSM；
2. `!use_fsm`：直接跳出循环去扩展；
3. **FSM 反馈**：`RecordAndGetPageWithFreeSpace()`——把刚发现的"该页其实只有这么多空间"写回 FSM，同时要下一个候选页。注释（hio.c:558–566）指出这一步是**防死循环的关键**：FSM 信息过时会让多个 backend 反复被指到同一个不够大的页，必须边试边纠正 FSM，循环才收敛。

### 6.2.6 扩展文件：RelationAddBlocks 的并发协议（hio.c:765–766 调用，实现在 hio.c:236–432）

```c
	/* Have to extend the relation */
	buffer = RelationAddBlocks(relation, bistate, num_pages, use_fsm,
							   &unlockedTargetBuffer);
```

`RelationAddBlocks` 的要点（20devel 的实现来自 PG16 的批量扩展改造，commit 00d1e02be2）：

- **按等待者数量放大扩展量**（hio.c:261–281）：`extend_by_pages += extend_by_pages * waitcount`，其中 `waitcount` 是正在等扩展锁的 backend 数。10 个人在排队就一次多扩 10 倍——把"锁下的工作量"换成"锁的交接次数"，与组提交同构。
- 若同一 bistate 之前扩展过，至少按上次的量扩（hio.c:283–299，减少 `mdzeroextend` 模式切换、平滑间歇性争抢），上限 `MAX_BUFFERS_TO_EXTEND_BY`（64 页，受"必须同时 pin 住所有新页"约束）。
- `ExtendBufferedRelBy(..., EB_LOCK_FIRST, ...)`（hio.c:339）完成物理扩展并**锁住第一个新页**——保证返回给调用者的页不会被并发者抢先塞满。
- 新页初始化后（hio.c:355–362，`PageInit` + `MarkBufferDirty`，并防御性检查页确实是空的），剩余页的处理体现了"自留 vs 共享"的权衡（hio.c:308–321）：有 bistate 时自己需要的页**不进 FSM**（否则别的 backend 会立刻来抢，制造无谓争抢），其余页 `RecordPageWithFreeSpace` 进 FSM 让大家用；最后 `FreeSpaceMapVacuumRange`（hio.c:404）把新条目向 FSM 上层传播，否则别人查不到。

### 6.2.7 扩展后的重加锁与最终复核（hio.c:769–883）

扩展路径上有两个可能"短暂放过锁"的窗口（为 FSM 登记、为 VM pin），所以返回前必须复核。这段是无死锁协议的最后一环：

```c
	else if (otherBuffer != InvalidBuffer)
	{
		/*
		 * ... It's guaranteed to be of a lower page number than the new
		 * page.  To conform with the deadlock prevent rules, we ought to
		 * lock otherBuffer first, but that would give other backends a
		 * chance to put tuples on our page. To reduce the likelihood of
		 * that, attempt to lock the other buffer conditionally, that's
		 * very likely to work. ...
		 */
		Assert(otherBuffer != buffer);
		Assert(targetBlock > otherBlock);

		if (unlikely(!ConditionalLockBuffer(otherBuffer)))
		{
			unlockedTargetBuffer = true;
			LockBuffer(buffer, BUFFER_LOCK_UNLOCK);
			LockBuffer(otherBuffer, BUFFER_LOCK_EXCLUSIVE);
			LockBuffer(buffer, BUFFER_LOCK_EXCLUSIVE);
		}
		recheckVmPins = true;
	}
```

新页块号必然大于 `otherBlock`，按 I8 应该先锁 otherBuffer——但那样会把刚扩出来的页裸露给并发者。折中：`ConditionalLockBuffer` 试锁（不等待就不可能死锁，大概率成功）；失败才退回"放掉新页锁 → 按升序重锁"的慢路径。最终复核（hio.c:858–870）：

```c
	pageFreeSpace = PageGetHeapFreeSpace(page);
	if (len > pageFreeSpace)
	{
		if (unlockedTargetBuffer)
		{
			if (otherBuffer != InvalidBuffer)
				LockBuffer(otherBuffer, BUFFER_LOCK_UNLOCK);
			UnlockReleaseBuffer(buffer);

			goto loop;
		}
		elog(PANIC, "tuple is too big: size %zu", len);
	}
```

放过锁 → 空间可能被抢 → `goto loop` 从头再来；从没放过锁却空间不够 → 逻辑矛盾，PANIC。注意新页**故意不进 FSM**（hio.c:872–880 注释）：留给本 backend 短期独占，赌它马上还要继续插——这是"顺序插入负载"的经验优化。

### 小结

RelationGetBufferForTuple 是三个协议的交汇点：**FSM 反馈协议**（边试边纠正过时信息，保证循环收敛）、**升序加锁协议**（I8，heap_update 的两页锁全部委托到这里）、**批量扩展协议**（按排队人数放大扩展量 + 自留/入 FSM 的分配策略）。它也是"锁内禁 I/O"纪律的最佳教材：每一次 `visibilitymap_pin` 都安排在放锁窗口，代价是层层"复核 + 重试"。

## 6.3 heap_update：PG 特色最浓的函数（全函数走读，分十四段）

`src/backend/access/heap/heapam.c:3266–4369`，约 1100 行，是 heapam 里最复杂的函数。核心心智模型：**UPDATE = 在（尽量同一）页里 insert 新版本 + 把旧版本的 xmax 设为我的 XID + 旧版本 ctid 指向新版本**。所有复杂度都来自三件事的叠加：并发者可能正拿着这行的锁（xmax 波及检查）、新版本可能放不下同页（跨页协议）、以及尽量别碰索引（HOT）。以下按执行顺序分段，主线全覆盖。

### 6.3.1 签名与入口检查（heapam.c:3266–3338）

```c
/* heapam.c:3266 */
TM_Result
heap_update(Relation relation, const ItemPointerData *otid, HeapTuple newtup,
			CommandId cid, uint32 options pg_attribute_unused(), Snapshot crosscheck, bool wait,
			TM_FailureData *tmfd, LockTupleMode *lockmode,
			TU_UpdateIndexes *update_indexes)
```

返回 `TM_Result`（TM_Ok/TM_Updated/TM_Deleted/TM_SelfModified…），失败时经 `tmfd` 带回后继版本的 ctid/xmax，供执行器做 READ COMMITTED 的 EvalPlanQual 重检查。三个出参：`lockmode`（本次实际用的行锁强度）、`update_indexes`（告诉执行器要不要插索引项——HOT 的成果出口）。入口处：拒绝并行模式（combo CID 无法广播，heapam.c:3331）；断言模式下检查 inplace-update 系统表的锁协议（heapam.c:3336–3338）。

### 6.3.2 四张属性位图（heapam.c:3340–3367）

```c
	hot_attrs = RelationGetIndexAttrBitmap(relation,
										   INDEX_ATTR_BITMAP_HOT_BLOCKING);
	sum_attrs = RelationGetIndexAttrBitmap(relation,
										   INDEX_ATTR_BITMAP_SUMMARIZED);
	key_attrs = RelationGetIndexAttrBitmap(relation, INDEX_ATTR_BITMAP_KEY);
	id_attrs = RelationGetIndexAttrBitmap(relation,
										  INDEX_ATTR_BITMAP_IDENTITY_KEY);
```

- `hot_attrs`：所有"会阻止 HOT"的列 = 被普通索引（B-tree 等，含部分索引谓词、表达式索引引用的列）覆盖的列；
- `sum_attrs`：只被**摘要型索引**（BRIN）覆盖的列——改了它们不阻止 HOT，但要通知 BRIN 更新摘要；
- `key_attrs`：唯一索引/主键/外键引用的"键"列——决定行锁强度；
- `id_attrs`：replica identity 列——决定逻辑复制要不要记旧值。

注释（heapam.c:3343–3348）解释为什么必须在锁页**之前**取好：更新的恰好是相关系统目录时，锁页后再查 relcache 可能死锁；而且拿到的是副本，不怕中途 relcache 失效。

### 6.3.3 读页、pin VM、加锁（heapam.c:3369–3383）

```c
	block = ItemPointerGetBlockNumber(otid);
	INJECTION_POINT("heap_update-before-pin", NULL);
	buffer = ReadBuffer(relation, block);
	page = BufferGetPage(buffer);

	if (PageIsAllVisible(page))
		visibilitymap_pin(relation, block, &vmbuffer);

```

又是 6.2.4 的"锁前乐观 pin VM"手法。`INJECTION_POINT` 是测试注入点（配合 `syscache-update-pruned.spec` 等隔离测试）。

### 6.3.4 line pointer 有效性：syscache 时代的新分支（heapam.c:3385–3430）

```c
	lp = PageGetItemId(page, ItemPointerGetOffsetNumber(otid));
	...
	if (!ItemIdIsNormal(lp))
	{
		Assert(RelationSupportsSysCache(RelationGetRelid(relation)));

		UnlockReleaseBuffer(buffer);
		...
		return TM_Deleted;
	}
```

通常调用者持 pin 或快照，旧元组不可能被 prune 掉、这里必然是 `LP_NORMAL`。例外是 `otid` 来自 syscache（既无 pin 也无快照）——此时可能看到 `LP_UNUSED/LP_DEAD/LP_REDIRECT`，说明并发 prune 已经动过手。长注释（heapam.c:3387–3408）解释了为什么统一返回 `TM_Deleted`：`LP_UNUSED/LP_DEAD` 情形拿不到后继 ctid，无法构造合法的 `TM_Updated`；而真正的 SQL UPDATE/MERGE 持快照保证走不到这里，所以这个选择只影响错误报文的措辞。这是近年为修复系统目录并发缺陷补的防御分支。

### 6.3.5 HeapDetermineColumnsInfo：判定改了哪些列（heapam.c:3444–3454，实现 4549–4630）

```c
	modified_attrs = HeapDetermineColumnsInfo(relation, interesting_attrs,
											  id_attrs, &oldtup,
											  newtup, &id_has_external);
```

实现（heapam.c:4549）对每个"感兴趣的列"做逐列新旧值比较：`heap_getattr()` 取出两边 Datum，`heap_attr_equals()` 按类型语义比较（定长类型比字节，变长走 `datumIsEqual`）。三个细节：整行引用（attrnum==0）与除 tableOid 外的系统列直接判"变了"；这是**按值比较**，`UPDATE t SET a = a` 不会把 a 算进 modified_attrs——所以无效更新仍可走 HOT；顺带检测"未修改的 replica identity 列是否存的是 TOAST 外部指针"（`*has_external`）——那种列的值不在新元组的 WAL 里，逻辑复制必须把它塞进 old_key_tuple（heapam.c:3444–3450 注释）。

### 6.3.6 行锁强度选择（heapam.c:3456–3489）

```c
	if (!bms_overlap(modified_attrs, key_attrs))
	{
		*lockmode = LockTupleNoKeyExclusive;
		mxact_status = MultiXactStatusNoKeyUpdate;
		key_intact = true;
		...
		MultiXactIdSetOldestMember();
	}
	else
	{
		*lockmode = LockTupleExclusive;
		mxact_status = MultiXactStatusUpdate;
		key_intact = false;
	}
```

这是 PG 9.3 引入的四级行锁体系（commit 0ac5ad5134，见 6.9）：普通 UPDATE（不改键列）持 `NoKeyExclusive`，与外键检查用的 `KeyShare` **不冲突**——父表行被子表引用时，更新父表的非键列不会被外键触发的行锁卡住。这解决了 MySQL 世界里常见的"外键导致的锁等待放大"的 PG 版本。`MultiXactIdSetOldestMember()` 提前声明"我可能马上加入某个 MultiXact"，防止 MultiXact 清理线程把我即将加入的组当垃圾回收（heapam.c:3473–3481 注释）。

### 6.3.7 `l2` 重试循环：xmax 波及检查（heapam.c:3498–3731）

这是全函数最烧脑的一段——**确认"我有权更新这行"**。`HeapTupleSatisfiesUpdate()` 给出旧元组当前状态，麻烦的是 `TM_BeingModified`：有人正拿着这行（锁或未决更新）。处理分四类（heapam.c:3558–3680）：

```c
l2:
	checked_lockers = false;
	locker_remains = false;
	result = HeapTupleSatisfiesUpdate(&oldtup, cid, buffer);
	...
	else if (result == TM_BeingModified && wait)
	{
		TransactionId xwait;
		uint16		infomask;
		...
		/* must copy state data before unlocking buffer */
		xwait = HeapTupleHeaderGetRawXmax(oldtup.t_data);
		infomask = oldtup.t_data->t_infomask;
```

1. **xmax 是 MultiXact**（heapam.c:3558–3630）：`DoesMultiXactIdConflict()` 判断成员锁与我的 `*lockmode` 是否冲突。冲突则放页锁 → 拿 tuple 锁（排队优先权，防饿死）→ `MultiXactIdWait()` 睡等所有冲突成员结束 → 重新锁页 → 若 xmax/infomask 已变，`goto l2` 从头再来。即使等完，MultiXact 里也可能还有**幸存者**（比如我只需 NoKeyExclusive，KeyShare 持有者可以继续活着）——他们必须被折叠进新 xmax（6.3.8）。若 Multi 里有一个"更新者"成员，还得查它提交与否：committed → 返回 TM_Updated（更新冲突）；aborted → 我可以继续。
2. **锁持有者就是我自己**（heapam.c:3631–3640）：直接放行（`can_continue`），但要保留锁信息。
3. **只有 KeyShare 锁持有者，且我不改键列**（heapam.c:3641–3651）：**根本不用等**——这正是 9.3 外键锁改造的收益兑现点。
4. **普通事务持有排他性锁/未决更新**（heapam.c:3652–3680）：放页锁 → 拿 tuple 锁 → `XactLockTableWait(xwait, ...)` 等在**对方的事务锁**上 → 重锁页 → xmax 变了就 `goto l2`；没变则 `UpdateXmaxHintBits` 把 CLOG 查询结果沉淀成提示位，`HEAP_XMAX_INVALID`（对方 abort 了）则放行。

走出这段时的判定（heapam.c:3682–3688）很精炼：

```c
		if (can_continue)
			result = TM_Ok;
		else if (!ItemPointerEquals(&oldtup.t_self, &oldtup.t_data->t_ctid))
			result = TM_Updated;
		else
			result = TM_Deleted;
```

`ctid 指向别处` = 被更新（有后继版本），`ctid 指向自己` = 被删除。随后（heapam.c:3702–3731）：SERIALIZABLE 的 RI 交叉检查（`crosscheck` 快照）；`result != TM_Ok` 则填 `tmfd`、清资源、返回，交给执行器决定重试或报错。最后一个竞态补丁（heapam.c:3742–3748）：如果锁页期间页变成了 all-visible 而我们没 pin VM，只能放锁 → pin → 重锁 → `goto l2`。

**这段的结论**：走出 l2 段时，当前事务已确认有权更新该行，且旧元组上还活着的锁持有者名单已拿到。

### 6.3.8 计算新旧元组的 xmax/infomask（heapam.c:3750–3812）


`compute_new_xmax_infomask()`（heapam.c:5503）是把"旧元组上可能存在的锁信息"折叠进新 xmax 的核心逻辑：若旧元组正被其他事务持有 KeyShare 锁，就创建一个 MultiXact 把"锁持有者们 + 我这个更新者"打包塞进 xmax（`HEAP_XMAX_IS_MULTI`）。**新元组**也可能要继承 xmax（heapam.c:3763–3800）：幸存的 KeyShare 锁必须跟着行走到新版本上，否则锁就"丢"了——新版本的 `xmax = 幸存锁持有者`，带 `HEAP_XMAX_KEYSHR_LOCK | HEAP_XMAX_LOCK_ONLY`。然后填新元组头（heapam.c:3806–3812，`xmin = 我`，`HEAP_UPDATED` 标志），并用 `HeapTupleHeaderAdjustCmax` 处理同事务内既插又删的 combo CID（heapam.c:3818）。

### 6.3.9 TOAST 与跨页判断：先占位再放锁（heapam.c:3820–3969）


```c

		oldtup.t_data->t_infomask2 &= ~HEAP_KEYS_UPDATED;
```

这是一个优雅的"意向占位"：并发事务看到该行已被锁定会乖乖等待；而如果我们中途崩溃/出错，这个 xmax 属于 aborted 事务，可见性判断自动无视它，别人照常继续（what-if 推演见 6.8.4）。之后放锁做 TOAST（heapam.c:3962–3969，`heap_toast_insert_or_update` 可能返回瘦身副本 `heaptup`）。

### 6.3.11 SSI 与 HOT 判定（heapam.c:4040–4088）


HOT 的两个条件缺一不可：**新版本落在同一页** 且 **没改任何被普通索引引用的列**。跨页则给旧页打 `PD_PAGE_FULL` 提示——下次有人来更新时先做页内 prune 腾空间，争取下回能 HOT。`README.HOT`（`src/backend/access/heap/README.HOT`）的要点浓缩：

- **动机**：没有 HOT 时，每个新版本都要往**每个索引**插一条指向新 TID 的项——这是 PG 写放大的最大来源，也是"单页清理（prune）"无法实施的原因（回收元组前必须先删索引项，而找到索引项要扫索引）；
- **机制**：HOT 链上只有链头有索引项。新版本打 `HEAP_ONLY_TUPLE` 标志（没有索引指向我），旧版本打 `HEAP_HOT_UPDATED` 标志（顺着我的 ctid 能找到后继）。索引扫描到达链头后沿 ctid 链在**同页内**前进，直到找到可见版本；
- **单页回收**：链中间的死版本可以被任何路过的查询顺手清掉（`pruneheap.c`），链头 line pointer 变成 `LP_REDIRECT`（`src/include/storage/itemid.h:40`）直接指向存活版本——空间回收完全不碰索引；
- **链不跨页**：跨页就得新建索引项、结束 HOT 链——否则索引扫描要跳页追链，破坏"一次页访问"的性能承诺。

对比 InnoDB：InnoDB 二级索引在 UPDATE 未改索引列时本来就不动（原地更新 + undo），HOT 是 PG 为自己的 no-overwrite 存储追回这项性能的补丁；代价是引入 prune、`LP_REDIRECT`、fillfactor 调优等一整套配套机制。

### 6.3.12 临界区前的最后准备：replica identity 与 VM 锁排序（heapam.c:4091–4164）


放 VM 锁、放页锁（先 newbuf 后 buffer）、`CacheInvalidateHeapTuple`（必须在放 pin 前——oldtup 指向页内！）、放 pin、放 tuple 锁、统计计数，最后：


### 小结

heap_update 的十四段可以压缩成五幕：**确权**（l2 循环把 xmax 上的并发者全部检查/等待/折叠）→ **选址**（同页直插；否则先占位加锁、放锁 TOAST、按升序锁协议拿两页）→ **判 HOT**（同页 + 不碰索引列）→ **临界区**（插新版本、旧版本盖 xmax、ctid 链接、一条 WAL 覆盖至多四页）→ **汇报**（update_indexes 决定索引写放大）。fillfactor、"别索引不必要的列"这两条调优经验的源码依据全在 6.3.11 的两个条件里。

## 6.4 heap_delete 与行锁编码

### 6.4.1 heap_delete 要点（heapam.c:2759）

DELETE 是 UPDATE 的简化版：没有新版本，只给旧元组盖 xmax。同样有 l1 重试循环（波及检查）、同样的 VM 处理。核心临界区（heapam.c:3036–3072）：

```c
	/* store transaction information of xact deleting the tuple */
	tp.t_data->t_infomask &= ~(HEAP_XMAX_BITS | HEAP_MOVED);
	tp.t_data->t_infomask2 &= ~HEAP_KEYS_UPDATED;
	tp.t_data->t_infomask |= new_infomask;
	tp.t_data->t_infomask2 |= new_infomask2;
	HeapTupleHeaderClearHotUpdated(tp.t_data);
	HeapTupleHeaderSetXmax(tp.t_data, new_xmax);
	HeapTupleHeaderSetCmax(tp.t_data, cid, iscombo);
	/* Make sure there is no forward chain link in t_ctid */
	tp.t_data->t_ctid = tp.t_self;

	/* Signal that this is actually a move into another partition */
	if (changingPart)
		HeapTupleHeaderSetMovedPartitions(tp.t_data);
```

磁盘上**没有任何字节被删除**——元组还在，只是被打上"已被 XID=N 删除"的标记。若事务提交，等所有老快照消亡后由 prune/VACUUM 真正回收；若事务回滚，xmax 属于 aborted 事务，可见性判断自动无视它，行"复活"零成本。`ClearHotUpdated + ctid 指向自己` 表示"我是链尾，没有后继"。分区表间的行迁移（DELETE + INSERT 到另一分区）用 `HeapTupleHeaderSetMovedPartitions()` 把 ctid 设为特殊值，让并发 UPDATE 能报出"行已迁移"而不是安静地跟丢。

### 6.4.2 行锁 = xmax + infomask 的位编码

PG 行锁**不进共享内存锁表**：锁一行就是把锁持有者的 XID 写进该行 `t_xmax`，锁的种类编码在 infomask 里。`src/include/access/htup_details.h:194–210`（逐字摘录）：

```c
#define HEAP_XMAX_KEYSHR_LOCK	0x0010	/* xmax is a key-shared locker */
#define HEAP_XMAX_EXCL_LOCK		0x0040	/* xmax is exclusive locker */
#define HEAP_XMAX_LOCK_ONLY		0x0080	/* xmax, if valid, is only a locker */
#define HEAP_XMAX_SHR_LOCK	(HEAP_XMAX_EXCL_LOCK | HEAP_XMAX_KEYSHR_LOCK)
#define HEAP_XMIN_COMMITTED		0x0100	/* t_xmin committed */
#define HEAP_XMIN_INVALID		0x0200	/* t_xmin invalid/aborted */
#define HEAP_XMIN_FROZEN		(HEAP_XMIN_COMMITTED|HEAP_XMIN_INVALID)
#define HEAP_XMAX_COMMITTED		0x0400	/* t_xmax committed */
#define HEAP_XMAX_INVALID		0x0800	/* t_xmax invalid/aborted */
#define HEAP_XMAX_IS_MULTI		0x1000	/* t_xmax is a MultiXactId */
#define HEAP_UPDATED			0x2000	/* this is UPDATEd version of row */
```

解码规则：

| 行状态 | 编码 |
|---|---|
| `SELECT ... FOR KEY SHARE` | xmax=locker XID，`KEYSHR_LOCK + LOCK_ONLY` |
| `FOR SHARE` | `SHR_LOCK`（EXCL+KEYSHR 两位同置）`+ LOCK_ONLY` |
| `FOR NO KEY UPDATE` | `EXCL_LOCK + LOCK_ONLY`（不带 KEYS_UPDATED） |
| `FOR UPDATE` | `EXCL_LOCK + LOCK_ONLY + HEAP_KEYS_UPDATED`(infomask2) |
| 真删除/更新 | xmax=XID，**无** `LOCK_ONLY` |
| 多个锁持有者 | xmax=MultiXactId，`HEAP_XMAX_IS_MULTI`，成员及各自模式存在 multixact SLRU 里 |

判别函数 `HEAP_XMAX_IS_LOCKED_ONLY()`（htup_details.h:230）。这个设计的意义与代价：

- **锁百万行不占一字节锁表内存**（InnoDB 的行锁位图虽也很省，但仍随锁行数增长且存在锁表结构）；
- 等锁的实现变成"等 xmax 那个事务结束"——等待者对持有者的 XID 调常规锁 `XactLockTableWait()`，即等在"事务锁"上（每个活跃事务在锁表里的唯一条目，事务结束即释放并唤醒所有人）；
- **加行锁也是写操作**（要改页、写 WAL），所以 PG 里 `SELECT FOR UPDATE` 会产生脏页；
- 四个提示位（XMIN/XMAX_COMMITTED/INVALID）是 CLOG 查询结果的元组级缓存，可在仅持共享级锁时设置（buffer/README 规则），这就是"大量只读也会产生写 I/O"的来源；
- 冻结的编码复用：`HEAP_XMIN_FROZEN = COMMITTED|INVALID` 两位同置——一个"逻辑上矛盾"的组合被赋予第三种含义，省下一个标志位。

### 小结

UPDATE/DELETE 都不动旧数据字节，只盖 xmax 章；行锁复用 xmax 字段，代价是行锁语义分散在 infomask 位组合里（MultiXact 又把复杂度外包给了另一个 SLRU）。HOT 的两个条件（同页 + 未改索引列）决定了写放大的量级。

## 6.5 WAL 记录的组装与插入

### 6.5.1 构造 API 与 XLogInsert 主体

各写函数遵循同一套 WAL 构造 API（`src/backend/access/transam/xloginsert.c`）：

`XLogInsert` 本体出乎意料地短（xloginsert.c:512–535，逐字摘录核心循环）：

```c
	do
	{
		XLogRecPtr	RedoRecPtr;
		bool		doPageWrites;
		bool		topxid_included = false;
		XLogRecPtr	fpw_lsn;
		XLogRecData *rdt;
		int			num_fpi = 0;
		uint64		fpi_bytes = 0;

		 * Get values needed to decide whether to do full-page writes. Since
		 * we don't yet have an insertion lock, these could change under us,
		 * but XLogRecordAssemble will recheck them once it has a lock.
		 */
		GetFullPageWriteInfo(&RedoRecPtr, &doPageWrites);

		rdt = XLogRecordAssemble(rmid, info, RedoRecPtr, doPageWrites,
								 &fpw_lsn, &num_fpi, &fpi_bytes,
								 &topxid_included);

								  fpi_bytes, topxid_included);
	} while (!XLogRecPtrIsValid(EndPos));
```

do-while 是一个**乐观并发协议**：组装时用的 `RedoRecPtr`（最近 checkpoint 的重做起点）没加锁，可能过时。`XLogRecordAssemble` 把"没带镜像的页中最小的页 LSN"通过 `fpw_lsn` 报给 `XLogInsertRecord`；后者拿到插入锁后复查——若 RedoRecPtr 已经前移、且 `fpw_lsn <= 新 RedoRecPtr`（说明有页其实需要镜像了），就返回 Invalid，让外层**重新组装**。牺牲极小概率的重做，换来组装阶段完全无锁。


四级判定链，优先级从高到低：调用者强制（`REGBUF_FORCE_IMAGE`，如一致性检查）→ 调用者禁止（`REGBUF_NO_IMAGE`）→ 全局开关（`full_page_writes=off` 或恢复中）→ **核心判据 `page_lsn <= RedoRecPtr`**。

**`page_lsn <= RedoRecPtr` 即"本页自最近一次 checkpoint 以来还没被 WAL 记录过"**——那么这是 checkpoint 后对它的第一次修改，必须把整页塞进 WAL。原因是"页撕裂"（torn page）：8KB 的页写到 4KB 时断电，磁盘上是半新半旧的垃圾，普通的增量 redo 记录无法在垃圾上重放；有了整页镜像，恢复时直接用镜像覆盖，再重放后续增量。之后同一页的第二、三次修改只记增量（页 LSN 已 > RedoRecPtr，这依赖不变式 I1 的 LSN 单调性），直到下一个 checkpoint 重置起点。`needs_data = !needs_backup` 则是 6.1.5 提到的瘦身：带镜像就不必再记增量。

后半段（xloginsert.c:721–862）构造镜像本体：`REGBUF_STANDARD` 页挖掉 `pd_lower`–`pd_upper` 之间的空洞不写（xloginsert.c:732–751）；`wal_compression` 开启时用 pglz/lz4/zstd 压缩（xloginsert.c:762–769）；`BKPIMAGE_APPLY` 位区分"恢复时要真的应用"和"只为一致性校验带上"（xloginsert.c:788–795）。省略未详读的部分：记录头/CRC 的最终拼装（xloginsert.c:900 以后）——纯格式序列化，无决策逻辑。

> **对比 InnoDB doublewrite**：同一个 torn page 问题，InnoDB 的答案是把脏页先顺序写进 doublewrite buffer 再写到最终位置——付出每页双写的代价，redo 日志保持纯物理短记录；PG 的答案是把整页并入 WAL——WAL 在 checkpoint 后短时间内急剧膨胀（这就是 PG 监控里 checkpoint 之后 WAL 量尖峰的成因），但数据文件只写一次。两者殊途同归，权衡点不同：PG 可通过拉长 checkpoint 间隔摊薄 FPW 开销，InnoDB 则常年背着双写但对日志量友好。

`XLogInsertRecord()`（xlog.c）随后把记录拷入共享 WAL 缓冲区：先在极短的自旋保护下"预留"一段 LSN 区间（只推进指针），然后在插入锁之外并发拷贝——多 backend 可以同时往 WAL 缓冲区不同位置写，这是 PG 9.4 WAL 并行插入改造的成果（commit 9a20a9b21b，见 6.9）。

### 小结

WAL 组装的三层智慧：登记与组装分离（调用者只声明"改了哪些页、附了什么数据"，格式决策全在 Assemble）；FPW 判定靠一次 LSN 比较（依赖 I1）；对 RedoRecPtr 的乐观读 + fpw_lsn 复查协议让组装完全无锁。

## 6.6 所以然四问：写路径的"为什么必须如此"

### 6.6.1 临界区为什么必须存在？——每一种交错的推演

问题设定：临界区保护的代码形如"改页 A（可能还有 B）→ MarkBufferDirty → XLogInsert → PageSetLSN"。假设**没有** `START_CRIT_SECTION`，即允许中途 `elog(ERROR)` 走 longjmp 逃回事务清理路径（进程不死、数据库继续跑）。逐一推演每个中断点：

**交错 ①：改了页、还没 XLogInsert 就 ERROR。**
页上已经有新元组/新 xmax，但 WAL 里没有任何记录。事务 abort 了，可见性还能兜底（xmin 属于 aborted 事务）——看似无害？不：
- 对 `heap_insert`，`pd_lower/pd_upper` 已被 `PageAddItem` 推进，页处于"改了一半"的物理状态。此后这个脏页可以被 bgwriter 刷盘（它只认 BM_DIRTY），于是磁盘上出现一个**任何 WAL 记录都没描述过的页状态**。此后崩溃恢复：redo 从 checkpoint 起点重放，重放到该页的下一条记录时，页内容与"重放程序假设的前置状态"不符——比如 offset 已被占用、空间对不上。redo 例程写死了 `elog(PANIC, "failed to add tuple")` 之类的兜底，恢复直接失败，**数据库起不来**。
- 更隐蔽的是 `heap_update` 的多页修改：旧页盖了 xmax、新页还没插入版本时 ERROR，版本链断裂——旧版本声称"我被更新了"（无 LOCK_ONLY 的 xmax），却没有任何后继。若该事务实际又没有 abort（比如 ERROR 出现在子事务里被捕获），跟链的读者会追进虚空。
- 部分修改还破坏 I7 与 FPW 的前提：页被改了但 `pd_lsn` 没变，下次这页被合法修改时 FPW 判定拿到的是旧 LSN——判定本身没错，但镜像里将包含这次"幽灵修改"，把垃圾固化进 WAL。

**交错 ②：XLogInsert 成功、PageSetLSN 之前 ERROR。**
WAL 有记录、页没有 LSN 章。该脏页此后刷盘时 `XLogFlush(pd_lsn)` 刷的是**旧** LSN——本次记录可能还没落盘页就先写出去了，违反 I2。且下次修改该页时 FPW 判据用了旧 LSN，可能多打一次镜像（无害）或在 `wal_consistency_checking` 下直接对不上账。

**交错 ③：多页记录只盖了一个页的 LSN。**
`log_heap_update` 之后要给至多四个页盖同一 LSN，盖到一半 ERROR：两个页对"同一次修改"的 WAL 观点不一致，其中一个页的 I2 保障失效。

结论：**"改页"与"记日志 + 盖章"必须是不可分割的原子段**。但 C 语言没有事务内存，PG 的原子性手段只有一个——**让中间态永不暴露**：要么顺利走完，要么进程立即死掉、共享内存整体作废、靠 WAL 从上一个一致状态重放。这正是 `elog(ERROR)` 升级 PANIC 的全部含义（elog.c:368–373）：

```c
		/*
		 * If we are inside a critical section, all errors become PANIC
		 * errors.  See miscadmin.h.
		 */
		if (CritSectionCount > 0)
			elevel = PANIC;
```

PANIC 杀掉整个实例、丢弃共享缓冲区里所有"改了一半"的页，恢复时 WAL 里要么有完整的这条记录（重放，得到改完的状态），要么没有（保持改之前的状态）——**二值化**了原本的中间态。这也解释了 I5/I6：临界区里做分配/I/O/等锁，等于把"内存不足""磁盘抖动"这类日常小故障都升格成全库重启，所以必须把一切可能失败的操作挪到临界区之前。InnoDB 的对应物是 mtr：mini-transaction 期间持有 page latch、redo 先进私有 buffer，mtr commit 时一次性拷入 log buffer 并释放 latch——同样是"让中间态不可见"，只是靠的是 latch 覆盖而不是"出错即自杀"。

### 6.6.2 WAL-before-data 的形式化与强制点

**形式化表述**：设 `flushed(LSN)` 表示 WAL 已持久化到该位置，对任意数据页 P：

```
写盘(P) 开始之前，必须 flushed(pd_lsn(P)) 成立。
```

配合两条辅助不变式即可证明可恢复性：(a) `pd_lsn(P)` 单调不减（I1）；(b) 每次修改 P 的 WAL 记录 r 满足 `end_lsn(r) = 修改后的 pd_lsn(P)`（临界区里的 `PageSetLSN`）。于是"磁盘上的 P"永远对应 WAL 中某个**前缀**能重放出的状态——恢复 = 找到 checkpoint 起点 + 顺序重放 + 用页 LSN 跳过已包含的修改（`页LSN ≥ 记录LSN` 则 skip）。

**代码强制点**只有一处，`FlushBuffer`（bufmgr.c:4526，调用在 4585）：

```c
	recptr = BufferGetLSN(buf);

	/*
	 * Force XLOG flush up to buffer's LSN.  This implements the basic WAL
	 * rule that log updates must hit disk before any of the data-file changes
	 * they describe do.
	 * ...
	 */
	if (pg_atomic_read_u64(&buf->state) & BM_PERMANENT)
		XLogFlush(recptr);
```

所有脏页出口（checkpointer、bgwriter、backend 自己驱逐页）都必经 FlushBuffer，所以一处卡点即可覆盖全局。unlogged 表跳过（本来就不保证崩溃后存活，其索引页上的"假 LSN"甚至可能超过真实 WAL 位置，刷了反而灾难——注释 bufmgr.c:4571–4582）。

**违反它的具体丢数据场景**：假设去掉这行 XLogFlush。事务 T 更新页 P（新值 v2，WAL 记录 r 还在 WAL 缓冲区里没落盘），bgwriter 把 P（含 v2）写到磁盘，然后断电。重启恢复：WAL 里没有 r（没刷出去），重放结束时数据库自认为处于"r 之前"的一致状态；但磁盘上的 P 已经是 v2。现在两种结局都是坏的：若 T 未提交，P 上留着一个 CLOG 里查不到提交、但也**没有任何 WAL 能解释**的修改——如果这只是元组数据还能靠可见性兜底，但若修改的是页结构（prune 压缩、line pointer 重排），后续 redo 在这个"来自未来"的页上重放旧记录，轻则 PANIC 重则静默错乱；若 r 是索引页分裂的一半（另一半的页没被刷出），索引树从此左右手互相矛盾——这正是"没有日志的部分写"不可恢复的本质：**恢复算法失去了'磁盘状态对应 WAL 某前缀'这个归纳基础**。

注意方向性：反过来"WAL 已落盘、数据页没落盘"永远安全——重放即可补齐。所以 PG 是标准的 **steal + no-force** 缓冲策略（脏页可提前刷、提交不强制刷数据页），全部代价压在一条 XLogFlush 上。

### 6.6.3 为什么选 no-overwrite + 版本链，而不是 InnoDB 的原地更新 + undo？

从三个维度正反论证（PG 立场 vs InnoDB 立场）：

**① 崩溃恢复复杂度。**
PG：恢复 = 单遍 redo。旧版本还在堆里，未提交事务的修改不需要"撤销"——它们的 xmin/xmax 在 CLOG 里不是 committed，天然不可见。恢复时间 ∝ 上个 checkpoint 以来的 WAL 量，与"崩溃时有多少未提交大事务"**无关**。
InnoDB：恢复 = redo 全量重放（含未提交修改）+ 扫描 undo 回滚所有未提交事务。崩溃前若有改了千万行的未提交事务，恢复要把这千万行逐一逆向应用，可用性受长事务人质挟持（尽管后台回滚让数据库能先开门）。
反方（InnoDB 视角）：PG 的"恢复简单"是把复杂度移到了运行期——可见性判断每行都要查 xmin/xmax/CLOG/快照，而 InnoDB 主键读命中最新版本时零额外判断，只有老快照才去 undo 链上重建。

**② 回滚成本。**
PG：ROLLBACK 是 O(1)——CLOG 标 ABORTED 即可（6.7.6），死版本留给 VACUUM 慢慢收。abort 记录甚至不用 fsync。
InnoDB：ROLLBACK 是 O(修改量)，逆向应用 undo，大事务回滚可能比正向执行还慢（还要写 redo）。
反方：PG 的"回滚免费"意味着 abort 掉的垃圾**真实占据堆页与索引**——一个反复失败重试的批量任务能把表撑肥一倍；InnoDB 回滚完，空间即刻归还。

**③ 空间债务。**
PG：每个 UPDATE 都物化一个完整新版本（哪怕只改 1 字节），旧版本 + 非 HOT 时的全套索引项构成持续膨胀压力，必须由 VACUUM/autovacuum 这个"第二写入系统"常态化偿还；长快照会拖住全表的垃圾回收水位。
InnoDB：主表始终只有最新版本，旧版本以**逻辑 diff** 形式存在 undo 段（通常远小于整行），purge 只清 undo 与删除标记，二级索引无版本膨胀。
反方（PG 视角）：undo 不是免费的——ibdata/undo 表空间同样会被长事务撑爆（经典的"undo 不回缩"运维事故）；且 InnoDB 为维护 undo 链，**每次更新都要额外写 undo 页的 redo**，写路径天然比 PG 的"一条 WAL 搞定"多一份日志开销。PG 还白赚了两样：任意版本都可寻址使得**同页多版本 + HOT 单页回收**成为可能；以及"崩溃后零 undo"让 PITR/备库激活都极其干脆。

结论不是谁对谁错，而是负载形状：高频短更新 + 长事务少 → InnoDB 的原地更新占优；混合读写 + 需要廉价回滚/快速恢复 → PG 的选择占优。PG 社区并非没动摇过——zheap 项目（2018 前后）试图给 PG 加 undo 式存储，最终因复杂度搁浅，反向印证了 no-overwrite 与 PG 全体系（可见性、恢复、Table AM 接口）的耦合之深。

### 6.6.4 组提交：在不牺牲持久性的前提下摊薄 fsync

持久性语义要求：**向客户端答复 COMMIT 成功之前，提交记录必须已 fsync**。天真实现是每事务一次 fsync——7200 转磁盘上限约每秒百余次提交。组提交的观察：N 个并发提交者的记录都在 WAL 缓冲区里前后相邻，**一次 fsync 到最大 LSN 即可同时满足所有人**，且不放松任何人的持久性（每个人仍然是"fsync 覆盖我之后才返回"）。

`XLogFlush`（xlog.c:2800）的等待/唤醒时序推演。设三个 backend A、B、C 几乎同时提交，提交记录 LSN 依次 100、200、300：

```
t1  A: RefreshXLogWriteResult → Flush=0 < 100，须刷
      WaitXLogInsertionsToFinish(...) → insertpos=300（B、C 的拷贝已完成）
      LWLockAcquireOrWait(WALWriteLock) → 拿到锁
t2  B: 同样发现 Flush < 200 → LWLockAcquireOrWait → 锁被 A 持有，睡眠
t3  C: 同上，睡眠
t4  A: （可选 commit_delay 小睡，攒更多同伴）
      WriteRqst.Write = WriteRqst.Flush = insertpos = 300
      XLogWrite(...) → write + fsync 到 300
      LWLockRelease(WALWriteLock) → 唤醒 B、C
t5  B: LWLockAcquireOrWait 返回 false（锁释放了但没把锁给我）
      → continue 回循环头 → RefreshXLogWriteResult → Flush=300 ≥ 200 → break，零 fsync 返回
t6  C: 同上，零 fsync 返回
```

三重机制对应源码（xlog.c:2848–2930，逐字摘录关键行）：

```c
		if (!LWLockAcquireOrWait(WALWriteLock, LW_EXCLUSIVE))
		{
			/*
			 * The lock is now free, but we didn't acquire it yet. Before we
			 * do, loop back to check if someone else flushed the record for
			 * us already.
			 */
			continue;
		}
		...
		if (CommitDelay > 0 && enableFsync &&
			MinimumActiveBackends(CommitSiblings))
		{
			pgstat_report_wait_start(WAIT_EVENT_COMMIT_DELAY);
			pg_usleep(CommitDelay);
			pgstat_report_wait_end();
			...
			insertpos = WaitXLogInsertionsToFinish(insertpos);
		}

		/* try to write/flush later additions to XLOG as well */
		WriteRqst.Write = insertpos;
		WriteRqst.Flush = insertpos;

		XLogWrite(WriteRqst, insertTLI, false);
```

① `LWLockAcquireOrWait` 的语义是关键（为组提交定制，commit 9b38d46d9f）：普通 `LWLockAcquire` 醒来时**自动持锁**，B 就会白白执行一次"刷已刷过的区间"的 fsync；OrWait 版本醒来**不拿锁**，先回头查"是否已被代刷"。② 持锁者把刷盘目标扩大到 `insertpos`（当前所有已完成拷贝的 WAL），一次 fsync 送走一批。③ `commit_delay/commit_siblings` 提供显式"等一等攒一批"（默认关，f11e8be3e8 改造后仅在真正要 fsync 的人身上生效，见 6.9）。

时序里还有一个容易忽略的正确性点：`WaitXLogInsertionsToFinish` 必须在写盘前——WAL 缓冲区里 LSN 区间是先"预留"后"拷贝"的（9.4 并行插入），直接把缓冲区写盘可能写出**还没拷完的洞**。等待"在飞的插入"完成后，`[起点, insertpos]` 才是连续实心的。与 InnoDB 对照：redo 的 group commit 同构（log writer 线程 + flush 到最大 LSN + 等待者查 flushed_to_disk_lsn），MySQL 的三阶段（binlog 组提交）额外解决 binlog 与 redo 两份日志的顺序一致，PG 单日志无此问题。ProcArray 的 `ProcArrayGroupClearXid`（6.7.5）是同一思想在锁交接上的复刻。

## 6.7 事务提交：CommitTransaction 全流程

### 6.7.1 CommitTransaction 逐段走读（xact.c:2270）

`src/backend/access/transam/xact.c:2270`。按顺序发生的事：

1. **用户代码收尾**（xact.c:2299–2313）：循环执行 `AfterTriggerFireDeferred()`（延迟触发器）与 `PreCommit_Portals()`（关游标），直到两者都无事可做——触发器可能开游标、关游标可能又激发触发器，所以是循环；
2. **各子系统 pre-commit**（xact.c:2322–2387）：扩展回调（`XACT_EVENT_PRE_COMMIT`）、并行 worker 清理、ON COMMIT 动作、`smgrDoPendingSyncs`（本事务新建的未记 WAL 的文件此刻 fsync，wal_level=minimal 的优化路径）、NOTIFY 入队、SSI 的最终串行化检查 `PreCommit_CheckForSerializationFailure`（此处还可能报错回滚——**最后的失败机会**）；
3. **状态机切换**：`s->state = TRANS_COMMIT`（xact.c:2393），此后不允许再失败；
4. **`RecordTransactionCommit()`**（xact.c:2407）：写提交记录 + 刷 WAL + 更新 CLOG——持久化的核心，下节全函数走读。源码注释就一句："This is where we durably commit."；
5. **`ProcArrayEndTransaction(MyProc, latestXid)`**（xact.c:2431）：把自己从"运行中事务"里除名——**从这一刻起，其他事务的新快照认为我已提交**，我的所有修改瞬间对新快照可见。注释（xact.c:2426–2429）强调它必须在放锁之前、RecordTransactionCommit 之后；
6. **资源释放**（xact.c:2449 起，顺序精心设计，注释 xact.c:2433–2447）：先释放对外可见资源（缓冲区 pin 等，`RESOURCE_RELEASE_BEFORE_LOCKS`），然后 `AtEOXact_Inval(true)`（xact.c:2475）广播目录失效消息，**然后才释放锁**（xact.c:2479）——保证等我锁的人醒来时一定能看到我的 DDL 变更；最后 `smgrDoPendingDeletes(true)`（xact.c:2495）物理删除本事务 DROP 掉的表文件（放在锁后是因为删文件可能耗时数秒，注释明说）。

第 4、5 步的顺序不可交换：必须先把提交写进 WAL/CLOG，再从 ProcArray 消失；否则会有一瞬间别人认为你"已结束"却在 CLOG 里查到"进行中"——半提交状态泄漏（不变式 I9）。

### 6.7.2 RecordTransactionCommit 全函数走读（xact.c:1344–1614）

#### 准备段（xact.c:1347–1378）


两个判定变量贯穿全函数：`markXidCommitted`（有没有 XID = 是不是写事务）和 `wrote_xlog`（本事务写过 WAL 吗）。提交记录的"随行货物"也在此收集：待删文件列表、已提交子事务 XID 列表、待广播的目录失效消息——它们搭提交记录的车持久化，恢复/备库据此重演删文件和缓存失效。

#### 分支一：无 XID 的只读事务（xact.c:1384–1436）

```c

	{
```

不写提交记录、不碰 CLOG。若连 WAL 都没写过，直接 `goto cleanup`——**只读事务的提交是纯内存操作，零 I/O**。两个边角：无 XID 却有失效消息的事务（inplace 系统表更新等）要发一条专门的 standby 失效记录（注释直言"每个已知用例都是缺陷"）；无 XID 但写过 WAL（HOT prune 顺手清理）的事务不能 goto，要继续往下走到"异步登记刷盘"分支——prune 的 WAL 也该在合理时间内落盘。

#### 分支二：写事务的提交记录（xact.c:1437–1513）

```c
		Assert((MyProc->delayChkptFlags & DELAY_CHKPT_IN_COMMIT) == 0);
```

`DELAY_CHKPT_IN_COMMIT` 是"提交小临界区"：挡住并发 checkpoint，防止其 REDO 点切在"commit 记录已写、CLOG 未更新"中间——那样崩溃恢复从 REDO 点开始会**错过 CLOG 更新**，已提交事务在 CLOG 里永远停在 in-progress（注释 xact.c:1448–1454 原文即此推演；本段完整 what-if 见 6.8.2）。改自家 `delayChkptFlags` 不必拿 ProcArrayLock（只有自己写），代价是 checkpoint 端读得"模糊"一点，无碍正确性（注释 1460–1463）。`pg_write_barrier()` 保证标志位先于提交时间戳全局可见（复制冲突判定依赖此序）。随后 `TransactionTreeSetCommitTsData` 记提交时间戳（commit_ts SLRU，可选特性）。

#### 同步/异步分岔（xact.c:1515–1574）

```c

		 synchronous_commit > SYNCHRONOUS_COMMIT_OFF) ||
		forceSyncCommit || nrels > 0)
	{
```

- **同步提交**（默认）：`XLogFlush` 把 WAL 刷到提交记录末尾。这是整条写路径上**唯一**必须等待 fsync 的点——数据页、索引页统统不刷（no-force）；刷完才允许写 CLOG（顺序即 I9 的前半段）；
- **异步提交**（`synchronous_commit=off`）：不等刷盘，把 LSN 交给 walwriter（`XLogSetAsyncXactLSN`，xlog.c:2630），最多丢最近约 3 倍 `wal_writer_delay` 的已提交事务，但**绝不损坏一致性**（推演见 6.8.3）；
- 细节：异步路径的 CLOG 更新带上了提交记录 LSN——SLRU 页头记住"本页依赖的最大 WAL 位置"，将来该 CLOG 页刷盘前会先补刷 WAL，在 CLOG 层面同样维持 I2；
- `nrels > 0`（事务里 DROP 过表）强制同步：文件删除不可逆，必须确保提交记录先落盘，否则可能"文件删了、提交记录丢了"——恢复后事务被视为 aborted，表却已无文件。

### 6.7.3 XLogFlush 与组提交（xlog.c:2800）

全函数结构：恢复中改走 `UpdateMinRecoveryPoint`（备库回放时 XLogFlush 语义变成"推进最小恢复点"）；快速路径 `record <= LogwrtResult.Flush` 直接返回（本地缓存的已刷位置，多数调用零开销命中）；主循环的等待/唤醒时序已在 6.6.4 完整推演。补一个健壮性细节（xlog.c:2943–2968 注释）：刷不到目标位置时只 `elog(ERROR)` 而非 PANIC——坏 LSN 的数据页不该拖垮整库；但从 xact.c 的提交临界区调来时会自动升级 PANIC（注释原文提及），因为提交语义已不容失败。

### 6.7.4 CLOG 更新：TransactionIdCommitTree（transam.c:240）

CLOG（目录 `pg_xact`，代码 `clog.c`）是每事务 2 bit 的巨型位图：IN_PROGRESS(00)/COMMITTED(01)/ABORTED(10)/SUB_COMMITTED(11)，一个 8KB 页存 32K 个事务的状态。`TransactionIdCommitTree` → `TransactionIdSetTreeStatus()`（clog.c:192）要解决**主事务 + 子事务的原子提交**。clog.c:161–182 的注释给出完整算法与示例：全家同页 → 持一次页锁全部置位，天然原子；跨页 → 三步走：

```
1. 把异页子事务标为 SUB_COMMITTED
2. 原子地把主事务及同页子事务标为 COMMITTED
3. 把第 1 步那批改为 COMMITTED
```

并发观察者对 SUB_COMMITTED 的解读是"去看我父事务的状态"，因此第 2 步的一次页内原子操作就是全家的提交瞬间——对外整体表现为原子。

### 6.7.5 ProcArrayEndTransaction 的原子性意义（procarray.c:663）

```c
/* procarray.c:680 */
		if (LWLockConditionalAcquire(ProcArrayLock, LW_EXCLUSIVE))
		{
			ProcArrayEndTransactionInternal(proc, latestXid);
			LWLockRelease(ProcArrayLock);
		}
		else
			ProcArrayGroupClearXid(proc, latestXid);
```

为什么必须持 `ProcArrayLock` 排他锁清除自己的 XID？注释（procarray.c:668–671）指向 `src/backend/access/transam/README`：**快照的原子性**。`GetSnapshotData()` 持共享锁扫描 ProcArray；如果提交者不加锁就消失，拍快照的人可能看到"半个提交"——例如快照认为 XID=100 已结束（不在运行列表），但 100 的子事务 101 还在列表里，同一事务的修改一半可见一半不可见。持锁清除保证：任何快照要么把我整个算作"运行中"（我的一切不可见），要么整个算作"已完成"（配合 CLOG=committed，我的一切可见）。**这把锁就是 PG 里"提交对外原子生效"的仲裁点。** 无 XID 的只读事务则完全不加锁（procarray.c:689–700）——它不影响任何人的快照正确性。

高并发下所有提交者抢这一把排他锁会成为瓶颈，于是有 `ProcArrayGroupClearXid()`（procarray.c:784，commit 0e141c0fbb）：拿不到锁的提交者把自己挂进一个无锁链表（原子 CAS 入队），链表头的第一个成员代表全组拿一次锁、批量清除所有人的 XID、再逐个唤醒——与 XLogFlush 的组提交异曲同工，把 N 次锁交接合并成 1 次。

### 6.7.6 AbortTransaction 对比（xact.c:2855）

回滚路径惊人地"轻"：

1. 先做异常环境清理：释放 LWLock、缓冲区内容锁、待处理 I/O——因为 abort 可能发生在任何 `elog(ERROR)` 的 longjmp 之后，环境是"脏"的；
2. `RecordTransactionAbort()`（xact.c:1796）：有 XID 才写一条 `XLOG_XACT_ABORT` 记录，然后 `TransactionIdAbortTree()`（调用点 xact.c:1893）把 CLOG 标成 ABORTED。**关键：不需要 XLogFlush**——崩溃后查不到提交记录的事务本来就被视为 aborted，abort 记录丢了也无所谓；也**没有 DELAY_CHKPT 保护**（xact.c:1456–1458 注释：丢失 abort 标记无害，"the presumption would be that it aborted, anyway"）；
3. `ProcArrayEndTransaction()` 同样调用——从运行列表消失后，别人查 CLOG 得到 aborted，我写的所有版本永久不可见；
4. **不碰任何数据页**。已插入的元组、已盖的 xmax 原样留在页里，成为死元组等 prune/VACUUM 收拾。

这就是"回滚免费"的完整源码依据（正反面已在 6.6.3 论证）：InnoDB 回滚要逆向应用 undo，PG 回滚是 O(1) 的标记操作——代价延迟支付给了 VACUUM。

### 小结

提交 = 写 commit 记录（带删文件/子事务/失效消息货物）→ XLogFlush（唯一等待点，组提交摊薄）→ CLOG 置位（跨页时靠 SUB_COMMITTED 三步保持原子）→ ProcArray 除名（持锁保证快照原子，抢不到锁就组清除）→ 放锁清资源。回滚只是 CLOG 标 ABORTED + ProcArray 除名，零数据页改动、零强制刷盘。同步与异步提交的差别仅在"等不等 XLogFlush"。

## 6.8 What-if：四场崩溃推演

方法论：kill -9（或断电）意味着共享缓冲区全部蒸发、WAL 只保留已 fsync 的前缀、数据文件保留"任意已被 bgwriter/checkpointer 刷出的页"。恢复 = 从最近 checkpoint 的 REDO 点顺序重放 WAL，按页 LSN 跳过已包含的修改。逐场景验证：**任何交错下，恢复结束的状态都等价于'该事务要么完整提交、要么完整不存在'**。

### 6.8.1 场景一：heap_insert 临界区内，XLogInsert 返回之后、END_CRIT_SECTION 之前 kill -9

时间线：页里已有新元组、`pd_lsn` 已盖成 recptr（或恰好还没盖），WAL 记录已进 **WAL 缓冲区**但未必落盘，事务未提交。

恢复自愈全过程：
1. 共享缓冲区蒸发——内存里那个改过的页**不存在了**。磁盘上的页是什么样？该页在本次修改后从未被刷盘（刷盘要先 `XLogFlush(pd_lsn)`，而进程死前多半没人刷它），所以磁盘上是修改前的旧样子。
2. WAL 两种情况：(a) 这条 `XLOG_HEAP_INSERT` 没来得及被任何人刷出——恢复重放到 WAL 末尾也没有它，页保持旧状态，插入如未发生；(b) 它被顺路刷出了（别的事务提交时 XLogFlush 扫尾带上，见 6.6.4 的"刷到 insertpos"）——恢复时重放它：读页、比较 `页LSN < 记录LSN` → 应用：重新 `PageAddItem` 放入元组、盖 LSN。页上于是有了这个元组。
3. 但事务的 commit 记录必然不在 WAL 里（还没走到提交）。恢复结束后该 XID 在 CLOG 里是 in-progress；PG 恢复完成时不会给这种事务补 abort 记录，而是靠规则"**CLOG 非 committed 即视同 aborted**"——之后任何读者做可见性判断：xmin=该 XID → 查 CLOG → 非提交 → 不可见。这行成为死元组，页头 `pd_prune_xid` 提示（6.1.5 特意设置的！）让未来的 prune 顺手回收它。
4. 两种情况殊途同归：**要么物理上没有这行，要么物理上有但逻辑上永远不可见**——对用户等价。自愈不需要任何"撤销"动作，这正是 no-overwrite 的恢复红利（6.6.3①）。

变体：如果 kill 发生在 `RelationPutHeapTuple` 之后、`XLogInsert` 之前（临界区更早的位置）呢？更简单——修改只存在于内存，WAL 里什么都没有，恢复后世界干干净净。临界区保证的正是"这个窗口里不会发生 ERROR-并-继续运行"，kill -9 反而是临界区语义的"合法出口"。

### 6.8.2 场景二：RecordTransactionCommit 中，XLogFlush 返回之后、TransactionIdCommitTree 完成之前 kill -9

时间线：提交记录**已持久化**，CLOG 还没标 committed（或标了但 CLOG 页没刷盘——CLOG 页本来就极少即时刷盘），客户端还没收到成功答复（答复在更后面）。

恢复自愈全过程：
1. 恢复重放 WAL，遇到这条 `XLOG_XACT_COMMIT` 记录。它的 redo 例程（xact_redo）做的事就是 `TransactionIdCommitTree`——**把崩溃前没来得及做/没持久化的 CLOG 更新重新做一遍**，还顺带重演删文件、发失效消息。CLOG 的最终状态：committed。
2. 结论：该事务**完整提交**。客户端虽没收到答复，但重连后能看到数据——这符合语义：答复前崩溃，结果"可提交可不提交"，只要不自相矛盾即可；这里 WAL 一锤定音。
3. 现在看 `DELAY_CHKPT_IN_COMMIT` 防的反例：假设没有它，checkpoint 恰在此窗口完成并把 REDO 点推到 commit 记录**之后**。下次恢复从新 REDO 点开始，**这条 commit 记录永远不会被重放**；而 CLOG 更新又没持久化——事务在 WAL 里"已提交"、在 CLOG 里永远 in-progress → 视同 aborted，**已持久化的提交丢了**（如果客户端已收到答复，就是持久性违约）。`delayChkptFlags` 让 checkpoint 等待所有处于该窗口的 backend 走完，窗口不会被 REDO 点切开。
4. 对称问题——"CLOG 标了 committed、其所在页先于 commit 记录刷盘、然后 WAL 尾丢失"——由异步路径的 `TransactionIdAsyncCommitTree` 携带 LSN + SLRU 刷页前补刷 WAL 兜底（6.7.2）；同步路径则天然满足（XLogFlush 先行）。

### 6.8.3 场景三：synchronous_commit = off，断电——丢多少？为什么不丢一致性？

**丢多少**：提交记录只进了 WAL 缓冲区，等 walwriter 周期性刷出。`XLogSetAsyncXactLSN` 会按需唤醒 walwriter（commit 4de82f7d7c 之后不再干等周期），官方口径的最坏窗口约 3 × `wal_writer_delay`（默认 200ms → 约 0.6s）+ 缓冲区滞留，实践按"最近一秒内答复过成功的事务可能回退"评估。**每一个丢失的事务都是'答复过成功但结果消失'**——这是 off 的全部代价。

**为什么不丢一致性**（与 `fsync=off` 的本质区别）：
1. WAL 的丢失只发生在**尾部**——fsync 顺序 = LSN 顺序，不可能"丢中间留后面"。恢复重放到 WAL 有效前缀的末尾，得到的是某个时刻 T 的**前缀一致**状态：T 之前提交的全在，T 之后的整体消失。没有事务被"提交一半"。
2. 数据页会不会"超前"于这个截断点？不会有害：脏页刷盘前 `XLogFlush(pd_lsn)`（I2）保证——**任何落盘的数据页，其全部修改都在 WAL 有效前缀里**。也就是说数据文件永远不比 WAL"新"。稍旧没关系，重放补齐。
3. 会不会出现"页上有未提交数据"？会（steal 策略 + 场景一的机制），但 CLOG 里这些 XID 非 committed → 不可见 → prune 收尸。一致性无恙。
4. CLOG 侧由 `TransactionIdAsyncCommitTree` 的 LSN 挂钩堵住最后一个口子（否则 CLOG 页先落盘会宣称一个 WAL 里不存在的提交）。

对比：`fsync=off` 是另一回事——连 I2 都不保证，数据页可以带着"WAL 里没有的修改"落盘，崩溃后是真损坏。风险模型对照 InnoDB：`synchronous_commit=off` ≈ `innodb_flush_log_at_trx_commit=2`（每秒刷、崩溃丢一秒、不坏库）；`fsync=off` ≈ 关掉 redo 的 fsync 校验，属于"仅测试可用"。

### 6.8.4 场景四：跨页 UPDATE 半途崩溃（页放不下 → 走 6.3.9–6.3.10 路径）

跨页 UPDATE 有两个天然的"半途"点，分别推演：

**(a) `xl_heap_lock` 小临界区已完成（占位锁已写 WAL），TOAST/选址进行中，`XLOG_HEAP_UPDATE` 尚未写入时 kill -9。**
1. 磁盘/WAL 可能留下的痕迹：旧元组 `xmax = 我的XID + HEAP_XMAX_LOCK_ONLY`（xl_heap_lock 若被顺路刷出并在恢复中重放，页上就有这个占位；没刷出则连占位都没有）。TOAST 表里可能已插入若干切片（toast 插入走正常 heap_insert + WAL）。
2. 恢复后：我的 XID 无 commit 记录 → 视同 aborted。占位 xmax 属于 aborted 事务 → `HeapTupleSatisfiesUpdate` 判 `HEAP_XMAX_INVALID`，任何后来的更新者当它不存在，行**自动解锁**——这正是 6.3.9 引用的注释所说 "If we crash/error before following through with the actual update, xmax will be of an aborted transaction, allowing other sessions to proceed"。
3. TOAST 切片的 xmin 也是我 → 不可见死元组 → VACUUM 回收。没有悬挂引用：主表新版本从未落地，没人持有指向这些切片的 TOAST 指针。
4. 结论：世界回到更新前，唯一残留是几个待回收的死元组。

**(b) 主临界区完成（`XLOG_HEAP_UPDATE` 已写：旧页 xmax + 新页新版本 + ctid 链接一体成型），COMMIT 之前 kill -9。**
1. 这条多页记录要么整个在 WAL 有效前缀里、要么整个不在（单条记录原子，CRC 校验保证不会重放半条）。不在 → 同 (a)。在 → 恢复重放：旧页盖 xmax、新页插新版本、VM 位清除，一次完成，**版本链在物理上完好**——不存在"旧页改了新页没改"的中间态，这正是 6.3.13 强调的"一条记录登记多页"的价值。
2. 事务无 commit 记录 → aborted。可见性：旧版本 xmax=aborted XID → 删除无效，旧版本仍可见；新版本 xmin=aborted XID → 不可见。**行的逻辑状态 = 更新前**，尽管物理上多了一个死的新版本、旧版本的 ctid 指着它。跟链的读者（EvalPlanQual 等）追到新版本后同样用 xmin 判定，不会误认。
3. 若这是非 HOT 更新且崩溃前索引项还没插（`ExecInsertIndexTuples` 在 heap_update 返回之后）：新版本成了"无索引指向的孤儿"——无碍，它本来就不可见；prune 按 HOT 规则之外的普通死元组回收它，line pointer 变 LP_DEAD 后由 VACUUM 统一清算。反过来"索引项插了、堆事务 aborted"的情形同样靠"索引不存版本信息、回表后按堆判可见性"消化。
4. 结论：任意交错下不变式都成立——**版本链的每一次物理演化都被单条 WAL 记录原子覆盖，逻辑正确性则完全托付给 xmin/xmax + CLOG**。两层分工清晰，这就是 PG 写路径崩溃自愈的总公式。

## 全景回顾：一条 UPDATE + COMMIT 的 WAL 足迹

```
BEGIN;                          -- 无任何 I/O，连 XID 都没有
UPDATE t SET v=v+1 WHERE id=1;  -- GetCurrentTransactionId() 分配 XID
                                -- heap_update:
                                --   (可能) XLOG_HEAP_LOCK    跨页时的占位锁（6.3.9）
                                --   XLOG_HEAP_UPDATE / HOT_UPDATE   旧页+新页(+VM页)，
                                --     checkpoint 后首次触碰的页附带 FPI（6.5.2）
                                --   (非 HOT) 每个索引一条 btree INSERT_LEAF 记录
COMMIT;                         -- XLOG_XACT_COMMIT（带删文件/子事务/失效消息）
                                -- XLogFlush ← 唯一 fsync 等待（组提交摊薄）
                                -- CLOG: xid → COMMITTED
                                -- ProcArray: 除名，全世界可见
```

用 `pg_waldump` 对照验证这张表，是本章最好的收尾实验；配合 `pg_walinspect` 还能直接看到 FPI 的膨胀。

## 自测问题（附详解）

**1. 为什么 `MarkBufferDirty` 必须在 `XLogInsert` 之前、而 `PageSetLSN` 在其之后？**

详解：若先写 WAL 后标脏，存在窗口——checkpoint 恰在其间确定脏页集合并推进 REDO 点：该页没被 checkpoint 写出（不在脏页集合）、其 WAL 又落在新 REDO 点之前，恢复从新 REDO 点开始，这条修改**既不在磁盘也不会被重放**（不变式 I7）。PG 用调用顺序 + `XLogRegisterBuffer` 中的 `Assert(BufferIsDirty(buffer))`（xloginsert.c:266）双保险。而 `PageSetLSN` 需要 `XLogInsert` 的返回值（记录结束 LSN），天然在其后；页头 LSN 的作用是让未来刷此页的人先 `XLogFlush(pd_lsn)`（bufmgr.c:4585），维持 I2。

**2. 一条 UPDATE 满足什么条件才是 HOT 更新？HOT 省掉了什么、没省掉什么？**

详解：两个条件（heapam.c:4063–4084）：新版本与旧版本同页（`newbuf == buffer`）；`modified_attrs` 与 `hot_attrs`（所有非摘要索引引用的列）无交集。注意判定按**值**比较（6.3.5），`SET a=a` 不阻止 HOT；只改 BRIN 列则走 `TU_Summarizing` 半 HOT。省掉的：所有索引的新条目插入（`update_indexes=TU_None`）、未来 VACUUM 扫索引删项的成本、以及死版本可被单页 prune 回收（不碰索引）。没省掉的：新版本元组本身照样占堆空间、照样写一条 XLOG_HEAP(HOT)_UPDATE、旧版本照样要被 prune；且 HOT 依赖页内有空间——fillfactor 预留（6.2.2）正是为它服务。

**3. `SELECT ... FOR UPDATE` 锁 100 万行，共享内存锁表会爆吗？等待这些行锁的人在等什么？**

详解：不会爆。行锁写在每行的 `t_xmax` + infomask（`HEAP_XMAX_LOCK_ONLY` 等位，6.4.2），不占锁表。等待者的路径：发现 xmax 是活跃事务 → （先拿 tuple 锁排队防饿死，heapam.c:3547–3556 注释）→ `XactLockTableWait()` 等在持有者的**事务锁**上——每个活跃事务在锁表里的唯一条目，事务结束即释放并唤醒所有等待者重查。锁表真正的风险点是等待瞬间的 tuple 锁与大量表锁，而非行锁本身。代价的另一面：加行锁是写操作（改页 + `xl_heap_lock` WAL），`SELECT FOR UPDATE` 会产生脏页和 WAL。

**4. full-page write 的判定条件是什么？为什么 checkpoint 调频会影响 WAL 量？InnoDB 如何解决同一问题？**

详解：`XLogRecordAssemble`（xloginsert.c:679–700）四级判定：`REGBUF_FORCE_IMAGE` 强制 → `REGBUF_NO_IMAGE` 禁止 → `doPageWrites` 关闭则免 → 否则 **`page_lsn <= RedoRecPtr`**，即"最近一次 checkpoint 后第一次修改该页"。checkpoint 越频繁，"第一次"越多，8KB 级镜像越多，WAL 越大——这是 PG 监控里 checkpoint 后 WAL 尖峰的成因；拉长 `checkpoint_timeout`/`max_wal_size` 可摊薄，`wal_compression` 可压缩镜像。InnoDB 用 doublewrite buffer：脏页先顺序写双写区再写原位，redo 保持短记录，代价是数据页双写。两者都在解 torn page，权衡不同（6.5.2）。

**5. `synchronous_commit=off` 时，崩溃后可能出现"数据页有新数据但事务标记为未提交"吗？可能丢已提交事务吗？会数据损坏吗？**

详解：三问分别是——可能：脏页可先于提交刷盘（steal），恢复后该 XID 在 CLOG 非 committed，这些行不可见，等 prune/VACUUM 清理，无害（6.8.1 同机制）；可能：最近未刷盘的 commit 记录丢失（约 3×`wal_writer_delay` 窗口），客户端收到过成功应答的事务被整体回退——这是 off 的全部代价（6.8.3）；不会损坏：WAL 只丢尾部、恢复得到前缀一致状态，I2 保证数据文件永不"超前"WAL，`TransactionIdAsyncCommitTree` 的 LSN 挂钩连 CLOG 页也纳入 I2。风险模型 ≈ InnoDB `innodb_flush_log_at_trx_commit=2`；与 `fsync=off`（真损坏）有本质区别。

**6. 临界区内为什么禁止 palloc、禁止 I/O？`elog(ERROR)` 升级 PANIC 的机制在哪一行代码？**

详解：临界区的契约是"改页与记日志之间绝不容忍'出错但继续运行'"（6.6.1 的交错推演）。palloc 失败会 `elog(ERROR)`；I/O 失败同样；而 elog.c:368–373 中 `if (CritSectionCount > 0) elevel = PANIC;` 会把它们全部升级为全库重启——所以"可能失败的操作"必须前置：内存分配（`ExtractReplicaIdentity` 在临界区前，heapam.c:4091 注释；mcxt.c:198 的 `AssertNotInCriticalSection` 直接断言）、VM 页 pin/读（6.2.4 两阶段）、XID 分配（函数第一行）。同时临界区内也不允许等重锁（持排他页锁睡眠会级联卡死系统）。

**7. heap_update 在"新版本放不下同一页"时为什么要先给旧元组写一个 `xl_heap_lock`？如果不写会怎样？**

详解：跨页要放开旧页内容锁去做 TOAST/选址（可能 I/O、可能等锁，都不能持锁做）。放锁期间若不占位，并发事务会把这行更新掉，回来后一切重来甚至丢失更新。所以先在小临界区里把 `xmax = 我的XID + HEAP_XMAX_LOCK_ONLY` 写上（heapam.c:3888–3946）。必须写 WAL 的原因是不变式 I10：这个 xmax 可能随脏页落盘，磁盘上不允许出现 WAL 无法解释的 xid。崩溃安全性靠"aborted 事务的锁自动失效"（6.8.4a）：若中途崩溃，xmax 属于 aborted 事务，别人照常继续——占位锁自然蒸发，无须任何清理代码。

**8. RelationGetBufferForTuple 如何保证两个 backend 交叉更新同两页时不死锁？扩展新页时为什么敢"先锁新页再补锁旧页"？**

详解：规则是"同一关系的缓冲区按块号升序加锁"（I8，hio.c:450–456 注释），heap_update 把旧页作为 `otherBuffer` 传入，由 hio.c:614–655 的四分支统一按序加锁——双方顺序一致则环路不可能形成。扩展路径上新页块号必然最大，按序该先锁旧页，但那样会让新页裸奔被抢；于是 `ConditionalLockBuffer(otherBuffer)`（hio.c:825）试锁——**不等待的加锁永远不会构成死锁环**；失败才放掉新页锁、按升序重锁，并在最后复核空间、必要时 `goto loop` 重试（hio.c:858–869）。

**9. 提交路径上 `XLogFlush → TransactionIdCommitTree → ProcArrayEndTransaction → 放锁` 四步，任意交换相邻两步会出什么错？**

详解：①CLOG 先于 XLogFlush：CLOG 页可能先落盘，崩溃后 CLOG 宣称 committed 而 WAL 里没有提交记录——凭空出现的"提交"，可见性错乱（异步路径正因此要给 CLOG 挂 LSN）。②ProcArray 除名先于 CLOG：窗口内别人拍快照认为我已结束，查 CLOG 却得到 in-progress → 按 aborted 处理我的行；CLOG 随后翻成 committed——同一事务对不同观察者一半可见一半不可见，快照原子性破产（transam/README，6.7.5）。③放锁先于 ProcArray 除名：等我行锁/表锁的人醒来重查，发现我"还在运行"（ProcArray 里还有我）→ 继续等或误判冲突；DDL 场景下对方还可能在收到失效消息前使用旧目录（xact.c:2468–2474 注释）。顺序即 I9，一步都不能换。

**10. 崩溃恢复后，一个 aborted 事务插入的元组是谁、在什么时机清掉的？为什么恢复本身不清？**

详解：恢复只做 redo，不做 undo（6.6.3①）——重放完 WAL，aborted/无 commit 记录的事务的元组原样躺在页里，仅仅是 CLOG 非 committed → 不可见。清理是**机会主义 + 兜底**双轨：任何后续访问该页的查询可触发单页 prune（`pruneheap.c`，靠 heap_insert 当年埋下的 `pd_prune_xid` 提示，heapam.c:2082–2094 注释明说这就是为 abort 场景准备的），把死元组的 line pointer 变 LP_DEAD/回收空间；VACUUM/autovacuum 兜底扫全表并顺带清索引项。恢复不清的原因：清理需要知道"哪些 XID aborted"之外还要重排页内空间、动索引——那是正常运行期的工作，塞进恢复只会拖长不可用时间，而 MVCC 保证不清也绝无正确性问题。
