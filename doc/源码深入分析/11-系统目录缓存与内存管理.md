## 本章导读

前面的章节里，我们反复说"解析器查系统目录把表名变成 OID"、"执行器拿到一个 `Relation` 描述符"。这一章拆开这几个黑盒：

1. **系统目录（system catalog）**：表的定义本身也存在表里——那"第一张表"的定义存在哪？这是数据库的自举（bootstrap）问题；
2. **三层缓存**：syscache / relcache / plancache，它们是 PG 每条 SQL 都要走的热路径。本章会把 syscache 的查找全路径（含负缓存的创建与命中）和 relcache 的构建主线完整走一遍；
3. **缓存失效**：多进程架构下，一个 backend 做了 DDL，其他 backend 的缓存怎么知道自己过期了？我们会走读共享内存环形队列的读写协议全文，并完整论证"锁-失效协议"——为什么拿到表锁之后必须立刻消费失效消息，这是 PG "DDL 也是事务"的正确性支柱；
4. **内存管理 MemoryContext**：PG 用 C 写了 40 年而内存泄漏极少，靠的不是纪律，而是一套"区域分配器 + 错误恢复一键重置"的工程设计。我们会走读 `AllocSetAlloc()` 全函数三条路径、chunk header 的 65-bit 编码技巧，并从"`elog(ERROR)` 可以在任意深度抛出"这个约束出发，反推为什么 region-based 分配是这个架构下唯一能保证不泄漏的方案。

- **Q1**：失效为什么走"提交时广播 + 接收方在安全点懒处理"，而不是写方直接改别人的缓存？事务回滚时那些失效消息怎么办？

## 11.1 系统目录：表的定义也是表

### 11.1.1 四张核心目录表的关系

| 目录表 | 存什么 | 关键列 | 类比 |
|---|---|---|---|
| `pg_class` | 一切"有存储或像表的东西"：表、索引、视图、序列、物化视图 | `oid`、`relname`、`relnamespace`、`relfilenode`、`relkind` | Hive Metastore 的 TBLS |
| `pg_attribute` | 每个表的每一列一行 | `attrelid`（指向 pg_class.oid）、`attname`、`atttypid`、`attnum` | COLUMNS_V2 |
| `pg_type` | 每种数据类型一行（含每个表隐含的行类型） | `oid`、`typname`、`typlen`、`typinput/typoutput` | 无直接对应——PG 类型系统本身可扩展 |
| `pg_proc` | 每个函数/操作符实现一行 | `oid`、`proname`、`prorettype`、`proargtypes` | UDF 注册表 |

关系链：`pg_class` 的一行说"有张表叫 t"，`pg_attribute` 里以 `attrelid = t 的 oid` 的若干行说"它有哪些列"，每列的 `atttypid` 指向 `pg_type`，而 `pg_type` 里类型的输入输出函数 `typinput/typoutput` 又指向 `pg_proc`。**整个数据库的元数据就是一张以 OID 为外键的关系图**，而 OID（Object Identifier，32 位无符号整数）就是这张图里的"指针"。

`src/include/access/transam.h:197` 定义了一条重要分界线：

```c
#define FirstNormalObjectId		16384
```

小于 16384 的 OID 留给 initdb 灌入的内建对象（手工或脚本预分配，保证所有安装实例一致），用户创建的对象从 16384 起分配。这就是为什么在任何 PG 实例上，`pg_class` 里 `pg_class` 自己的 OID 永远是 1259。

### 11.1.2 自举问题：genbki.pl、.dat 文件与 BKI

鸡生蛋问题：要执行 `INSERT INTO pg_class ...` 得先能读懂 pg_class 的列定义，而列定义存在 pg_attribute 里，读 pg_attribute 又要先读懂 pg_attribute 自己……PG 的解法分三层：

**第一层：C 头文件就是表定义。** 每张目录表在 `src/include/catalog/` 下有一个头文件，用 `CATALOG` 宏同时生成 C 结构体和表定义元数据：

```c
CATALOG(pg_class,1259,RelationRelationId) BKI_BOOTSTRAP BKI_ROWTYPE_OID(83,RelationRelation_Rowtype_Id) BKI_SCHEMA_MACRO
```

（`src/include/catalog/pg_class.h:34`）逐段解读：`pg_class` 是表名，`1259` 是硬编码 OID，`RelationRelationId` 是生成的 C 宏名；`BKI_BOOTSTRAP` 标记它是**自举表**——pg_class（1259，`pg_class.h:34`）、pg_attribute（1249，`pg_attribute.h:39`）、pg_proc（1255，`pg_proc.h:32`）、pg_type（1247，`pg_type.h:38`）四张表享此待遇：它们的结构直接编译进二进制（C 结构体 `FormData_pg_class` 等），初始化时不需要"先查目录"就能按结构读写。

**第二层：.dat 文件是初始数据。** `src/include/catalog/pg_proc.dat`、`pg_type.dat` 等 Perl 数据文件记录了全部内建函数、类型的初始行（pg_proc.dat 有几千个条目）。改内核加一个内建函数，就是在 .dat 里加一段。

**第三层：genbki.pl 编译出 BKI 脚本。** `src/backend/catalog/genbki.pl` 在**构建期**读取所有 pg_*.h + pg_*.dat，生成 `postgres.bki`——一种极简的"目录汇编语言"（`create`/`insert`/`open` 等原语）。`initdb` 时，postgres 以特殊的 bootstrap 模式启动，由 `src/backend/bootstrap/bootstrap.c` 解释执行 BKI 脚本，用最原始的堆插入接口把初始目录行写进去。此时没有 SQL、没有优化器、甚至没有完整的缓存失效机制（11.2 会看到 bootstrap 模式禁用负缓存正是因为这个）。

> **对照 MySQL**：8.0 的数据字典初始化用的是硬编码 SQL + InnoDB 内部接口；PG 的方案更"自产自销"——连初始数据都是从与 C 代码同源的声明式文件生成的，保证头文件里的结构体和磁盘上的行永远一致。

### 小结

- 目录表是普通堆表，元数据关系图以 OID 为指针，16384 是内建/用户对象的分界；
- 自举靠三板斧：四张 `BKI_BOOTSTRAP` 表结构编译进二进制、.dat 文件提供初始行、genbki.pl 生成的 BKI 脚本在 initdb 时由 bootstrap 模式解释执行；
- 自举问题还有一个运行期变体——"读 pg_class 需要先有 pg_class 的 relcache"——留给 11.3.3 的 relcache init file。

## 11.2 syscache：按键缓存目录行

每解析一条 SQL 都要查十几次目录（表名→OID、每列类型、每个操作符……）。如果每次都走"打开堆表 + 索引扫描"，解析会比执行还慢。所以 PG 在每个 backend 的本地内存里做了 **syscache**：按键哈希缓存单条目录元组。

分两层实现：

- `src/backend/utils/cache/syscache.c`：定义"有哪些 cache"。每个 cache 对应一张目录表的一个**唯一索引**，例如 RELOID（按 OID 查 pg_class）、RELNAMENSP（按 (relname, relnamespace) 查 pg_class）。cache 清单如今由目录头文件里的声明自动生成（`syscache.c:82` 引入生成的 `catalog/syscache_info.h`）。
- `src/backend/utils/cache/catcache.c`：通用的"目录元组哈希缓存"机制本身。

为什么要按键数拆出 `SearchCatCache1/2/3/4` 这组入口（`catcache.c:1390-1419`），而不是统一用 `SearchCatCache()`（`catcache.c:1374`，从 `cache->cc_nkeys` 读键数）？源码注释（`catcache.c:1384-1388`）说得直白：`nkeys` 变成编译期常量后，`SearchCatCacheInternal` 是 `static inline`，编译器可以内联并展开键比较循环。这是 2017 年 Andres Freund 的性能提交引入的（见 11.7 设计历史 D3）——目录缓存查找热到值得为省几条循环指令专门改 API。

### 11.2.2 全路径走读之一：SearchCatCacheInternal（命中路径）

`catcache.c:1425` 起，函数体完整摘录（仅省略 `#ifdef CATCACHE_STATS` 统计计数与 `CACHE_elog` 调试日志——它们不影响逻辑；其余逐字）：

```c
static inline HeapTuple
SearchCatCacheInternal(CatCache *cache,
					   int nkeys,
					   Datum v1,
					   Datum v2,
					   Datum v3,
					   Datum v4)
{
	Datum		arguments[CATCACHE_MAXKEYS];
	uint32		hashValue;
	Index		hashIndex;
	dlist_iter	iter;
	dlist_head *bucket;
	CatCTup    *ct;

	Assert(cache->cc_nkeys == nkeys);

	/*
	 * one-time startup overhead for each cache
	 */
	ConditionalCatalogCacheInitializeCache(cache);

	/* Initialize local parameter array */
	arguments[0] = v1;
	arguments[1] = v2;
	arguments[2] = v3;
	arguments[3] = v4;

	/*
	 * find the hash bucket in which to look for the tuple
	 */
	hashValue = CatalogCacheComputeHashValue(cache, nkeys, v1, v2, v3, v4);
	hashIndex = HASH_INDEX(hashValue, cache->cc_nbuckets);

	/*
	 * scan the hash bucket until we find a match or exhaust our tuples
	 *
	 * Note: it's okay to use dlist_foreach here, even though we modify the
	 * dlist within the loop, because we don't continue the loop afterwards.
	 */
	bucket = &cache->cc_bucket[hashIndex];
	dlist_foreach(iter, bucket)
	{
		ct = dlist_container(CatCTup, cache_elem, iter.cur);

		if (ct->dead)
			continue;			/* ignore dead entries */

		if (ct->hash_value != hashValue)
			continue;			/* quickly skip entry if wrong hash val */

		if (!CatalogCacheCompareTuple(cache, nkeys, ct->keys, arguments))
			continue;

		/*
		 * We found a match in the cache.  Move it to the front of the list
		 * for its hashbucket, in order to speed subsequent searches.  (The
		 * most frequently accessed elements in any hashbucket will tend to be
		 * near the front of the hashbucket's list.)
		 */
		dlist_move_head(bucket, &ct->cache_elem);

		/*
		 * If it's a positive entry, bump its refcount and return it. If it's
		 * negative, we can report failure to the caller.
		 */
		if (!ct->negative)
		{
			ResourceOwnerEnlarge(CurrentResourceOwner);
			ct->refcount++;
			ResourceOwnerRememberCatCacheRef(CurrentResourceOwner, &ct->tuple);

			return &ct->tuple;
		}
		else
		{
			return NULL;
		}
	}

	return SearchCatCacheMiss(cache, nkeys, hashValue, hashIndex, v1, v2, v3, v4);
}
```

逐步展开：

1. **惰性初始化**：`ConditionalCatalogCacheInitializeCache()` 第一次用到某 cache 时才去读它对应目录表的 TupleDesc、键列信息——避免 backend 启动时把几十个 cache 全部初始化一遍；
2. **三级过滤**：`dead` 标志 → 哈希值 → 精确键比较。`dead` 条目是"已被失效但还有人持引用"的僵尸（见 11.4.5 的 `CatCacheInvalidate`），不能立即释放但必须跳过；先比 32 位哈希再做逐键比较是经典的快速排除；
3. **近似 LRU**：命中即 `dlist_move_head`——桶内链表头部永远是最近命中的条目，零额外元数据的 LRU 近似；
4. **引用计数 + ResourceOwner 双保险**：`refcount++` 保证条目不会在使用中被释放；`ResourceOwnerRememberCatCacheRef` 把这笔引用登记到当前 ResourceOwner——如果后续代码 `elog(ERROR)` 长跳转走人，事务中止清理会按登记自动 `ReleaseCatCache`，引用不会漏（与 11.5 的内存管理是同一套"错误恢复不靠人"哲学）。注意 `ResourceOwnerEnlarge` 在 `refcount++` **之前**调用：先保证登记数组有空位，才做不可逆操作——这个顺序本身就是一条不变式（见 11.6-I3）；
5. **负条目命中返回 NULL**：调用方拿到 NULL 就知道"确定不存在"，不会再去扫目录。注意负条目**不加引用计数**——它不携带元组，调用方不会持有指向它的指针。

### 11.2.3 全路径走读之二：SearchCatCacheMiss（未命中与负缓存的创建）

`catcache.c:1533` 起，主体逐字摘录（省略函数头注释与 `CACHE_elog`/统计宏，理由同上）：

```c
static pg_noinline HeapTuple
SearchCatCacheMiss(CatCache *cache,
				   int nkeys,
				   uint32 hashValue,
				   Index hashIndex,
				   Datum v1,
				   Datum v2,
				   Datum v3,
				   Datum v4)
{
	ScanKeyData cur_skey[CATCACHE_MAXKEYS];
	Relation	relation;
	SysScanDesc scandesc;
	HeapTuple	ntp;
	CatCTup    *ct;
	bool		stale;
	Datum		arguments[CATCACHE_MAXKEYS];

	/* Initialize local parameter array */
	arguments[0] = v1;
	arguments[1] = v2;
	arguments[2] = v3;
	arguments[3] = v4;

	relation = table_open(cache->cc_reloid, AccessShareLock);

	/*
	 * Ok, need to make a lookup in the relation, copy the scankey and fill
	 * out any per-call fields.
	 */
	memcpy(cur_skey, cache->cc_skey, sizeof(ScanKeyData) * nkeys);
	cur_skey[0].sk_argument = v1;
	cur_skey[1].sk_argument = v2;
	cur_skey[2].sk_argument = v3;
	cur_skey[3].sk_argument = v4;

	do
	{
		scandesc = systable_beginscan(relation,
									  cache->cc_indexoid,
									  IndexScanOK(cache),
									  NULL,
									  nkeys,
									  cur_skey);

		ct = NULL;
		stale = false;

		while (HeapTupleIsValid(ntp = systable_getnext(scandesc)))
		{
			ct = CatalogCacheCreateEntry(cache, ntp, NULL,
										 hashValue, hashIndex);
			/* upon failure, we must start the scan over */
			if (ct == NULL)
			{
				stale = true;
				break;
			}
			/* immediately set the refcount to 1 */
			ResourceOwnerEnlarge(CurrentResourceOwner);
			ct->refcount++;
			ResourceOwnerRememberCatCacheRef(CurrentResourceOwner, &ct->tuple);
			break;				/* assume only one match */
		}

		systable_endscan(scandesc);
	} while (stale);

	table_close(relation, AccessShareLock);

	/*
	 * If tuple was not found, we need to build a negative cache entry
	 * containing a fake tuple.  The fake tuple has the correct key columns,
	 * but nulls everywhere else.
	 *
	 * In bootstrap mode, we don't build negative entries, because the cache
	 * invalidation mechanism isn't alive and can't clear them if the tuple
	 * gets created later.  (Bootstrap doesn't do UPDATEs, so it doesn't need
	 * cache inval for that.)
	 */
	if (ct == NULL)
	{
		if (IsBootstrapProcessingMode())
			return NULL;

		ct = CatalogCacheCreateEntry(cache, NULL, arguments,
									 hashValue, hashIndex);

		/* Creating a negative cache entry shouldn't fail */
		Assert(ct != NULL);

		/*
		 * We are not returning the negative entry to the caller, so leave its
		 * refcount zero.
		 */

		return NULL;
	}

	return &ct->tuple;
}
```

逐步展开：

1. **`pg_noinline`**：函数头注释（`catcache.c:1528-1531`）说明这是刻意的——miss 路径体积大，若被内联进 `SearchCatCacheInternal` 会污染热路径的指令缓存，所以显式禁止内联。快慢路径分离到"连编译器优化都要管"的程度；
2. **真正扫目录**：`table_open(目录表, AccessShareLock)` + `systable_beginscan(唯一索引, 键)`——这就是没有缓存时每次查找的代价（见 11.2.4 的量化）；
3. **`stale` 重试环**：`CatalogCacheCreateEntry()` 可能返回 NULL——函数头前的大段注释（`catcache.c:1571-1575`）解释：如果元组带 TOAST 出线字段，创建条目时要去读 TOAST 表，而**读 TOAST 表本身会 `table_open` → 可能触发 `AcceptInvalidationMessages()`**，刚读出来的元组可能当场变陈旧。此时必须重扫。这是"失效可以发生在任何目录访问点"这一事实的第一个现场（不变式 11.6-I1）；
4. **重复条目无害**：注释（`catcache.c:1561-1569`）还坦白了另一种竞态：扫描目录期间递归的缓存查找可能已经把同一个元组塞进了缓存，于是会出现第二份拷贝。PG 的选择是**不检测**——第一份再也不会被命中，迟早被换出，"功能上无问题，罕见到不值得花指令"。这种"允许良性冗余换取快路径干净"的取舍在 PG 里反复出现；
5. **负条目的诞生**：扫不到时用 `CatalogCacheCreateEntry(cache, NULL, arguments, ...)` 造一个"只有键、其余全空"的假元组条目插进同一个桶。refcount 保持 0（没人持有它）——它的生命完全由失效机制管理。

再看 `CatalogCacheCreateEntry()`（`catcache.c:2167`）里负条目分支（逐字）：

### 11.2.4 所以然（Q2）：没有负缓存会怎样

推演一个最普通的场景：`search_path = "$user", public`，用户执行 `SELECT * FROM t`，而 t 在 public 下。

解析器要把 `t` 变成 OID，必须按 search_path 顺序试探：先查 `("$user" 对应 schema, t)`——**不存在**；再查 `(public, t)`——命中。第一步的"不存在"如果不缓存，那么：

- 每条引用 t 的 SQL（哪怕是同一条 prepared statement 的重复解析）都要为这个"不存在"付一次 `SearchCatCacheMiss` 全价：`table_open(pg_class)` + `systable_beginscan`（索引 `pg_class_relname_nsp_index` 的 btree 下降：根→叶 2-3 次 buffer 访问，每次都要 pin/lock 缓冲页、拿 LWLock）+ 扫描收尾。相比之下命中 catcache 只是一次哈希 + 链表比较，两者差一到两个数量级；
- 更糟的是乘法效应：函数重载解析（按名字查 `pg_proc` 的候选集）、操作符查找、隐式类型转换探测都是"大量试探、多数落空"的负载。一条中等复杂度的 SQL 解析期做几十次"查无此物"很正常；
- 负载还会打在**共享资源**上：目录索引的缓冲页、buffer mapping 分区锁——所有 backend 的失败探测都砸向同几个页面，高并发下就是锁竞争热点。

负缓存把"不存在"也变成 O(1) 本地哈希命中，代价是**失效机制必须同样覆盖它**：`CREATE TABLE "$user".t` 提交时，失效消息按键哈希值广播（见 11.4），`CatCacheInvalidate` 按 `hash_value` 匹配删除条目，正负条目一视同仁（`catcache.c:629-631` 注释明说"whether positive or negative"）——否则新表会被负条目"隐身"。这也解释了 bootstrap 模式为什么禁用负缓存（`catcache.c:1628-1631` 注释）：那时失效机制尚未运转，负条目一旦过期无人清除。

负缓存是 2002 年 Tom Lane 的提交 `26ac217173` 引入的（见 11.7-D1），提交信息里还记录了一个连带收获：有了"插入和删除对失效一视同仁"的语义后，inval.c 的逻辑反而大幅简化了。

### 小结

- syscache = 每 backend 本地的、按唯一索引键组织的目录行哈希缓存；syscache.c 管"有哪些缓存"，catcache.c 管通用机制；`SearchCatCache1/2/3/4` 按键数特化以便内联展开；
- 命中路径：哈希→桶链表（dead/哈希值/键三级过滤）→移到链头（近似 LRU）→正条目 refcount+ResourceOwner 登记、负条目直接返回 NULL；
- miss 路径刻意 `pg_noinline`；扫目录期间失效可能到达（TOAST 访问），靠 in-progress 栈 + stale 重试环保证不缓存陈旧数据；
- 负缓存把"查无此物"降为 O(1)，专治 search_path/函数重载这类高频落空探测；其正确性完全依赖"按哈希值失效、正负一视同仁"。

## 11.3 relcache：表的"全息描述符"

syscache 缓存的是**一行**目录元组，但执行器打开一张表需要的是**聚合视图**：列描述、索引列表、触发器、分区信息、物理文件号……把这些每次都从十几张目录表现拼出来太贵，于是有了 relcache（`src/backend/utils/cache/relcache.c`）：以表 OID 为键，缓存 `RelationData` 结构。内核代码里到处可见的 `Relation rel` 参数就是指向它的指针。

### 11.3.1 RelationData 逐要点

结构定义在 `src/include/utils/rel.h:55`，字段可分为六组（节选自源码，注释为原文）：

```c
typedef struct RelationData
{
	RelFileLocator rd_locator;	/* relation physical identifier */
	SMgrRelation rd_smgr;		/* cached file handle, or NULL */
	int			rd_refcnt;		/* reference count */
	ProcNumber	rd_backend;		/* owning backend's proc number, if temp rel */
	bool		rd_islocaltemp; /* rel is a temp rel of this session */
	bool		rd_isnailed;	/* rel is nailed in cache */
	bool		rd_isvalid;		/* relcache entry is valid */
	...
	SubTransactionId rd_createSubid;	/* rel was created in current xact */
	...
	Form_pg_class rd_rel;		/* RELATION tuple */
	TupleDesc	rd_att;			/* tuple descriptor */
	Oid			rd_id;			/* relation's object id */
	LockInfoData rd_lockInfo;	/* lock mgr's info for locking relation */
	RuleLock   *rd_rules;		/* rewrite rules */
	TriggerDesc *trigdesc;		/* Trigger info, or NULL if rel has none */
	...
	PartitionKey rd_partkey;	/* partition key, or NULL */
	PartitionDesc rd_partdesc;	/* partition descriptor, or NULL */
```

- **物理定位**：`rd_locator`（表空间/数据库/文件号三元组，决定磁盘路径）+ `rd_smgr`（缓存的存储管理器文件句柄）。注意逻辑身份 `rd_id`（OID）与物理身份 `rd_locator.relNumber`（relfilenode）是两回事：TRUNCATE、VACUUM FULL 换物理文件而 OID 不变；
- **生命周期管理**：`rd_refcnt` 引用计数（谁 open 谁 close）；`rd_isvalid` 表示条目是否仍然新鲜——**失效消息到达时并不总是删除条目，而是标脏或就地重建**（见 11.3.4），下次使用时再懒重建；`rd_isnailed` 标记"钉死"的条目：pg_class、pg_attribute 等自举表的 relcache 条目不能依赖"查目录重建"（会无限递归），启动早期由 `formrdesc()`（`relcache.c:1889`）用编译进二进制的结构硬造出来，永不驱逐；
- **事务感知**：`rd_createSubid`、`rd_newRelfilelocatorSubid`、`rd_firstRelfilelocatorSubid`、`rd_droppedSubid` 四个 SubTransactionId 字段记录"这张表/它的物理文件是不是本(子)事务新建/新换/已删的"——本事务新建的表 COPY 可跳 WAL 等优化全靠它们精确维护，rel.h:69 起有一大段注释专讲其微妙之处。11.3.4 会看到失效处理必须小心翼翼保住这些字段（它们只存在于本地，重建时无法从目录恢复）；
- **逻辑描述**：`rd_rel`（pg_class 行的拷贝）、`rd_att`（TupleDesc，列描述数组，堆元组的编解码都要它）；
- **行为附件**：`rd_rules`（视图/规则）、`trigdesc`（触发器）、`rd_rsdesc`（行级安全策略）、外键列表、索引列表；
- **分区**：`rd_partkey` / `rd_partdesc` 及其私有内存上下文——注意模式：**大的可变子结构各带一个自己的 MemoryContext**（`rd_partkeycxt`、`rd_pdcxt`、`rd_rulescxt`……），重建条目时整块删除，不必逐字段 pfree（11.5 的思想渗透到每个角落）。

### 11.3.2 RelationIdGetRelation：拿描述符的唯一正门

`relcache.c:2093`（逐字摘录，含函数头前的关键告诫）：

```c
Relation
RelationIdGetRelation(Oid relationId)
{
	Relation	rd;

	AssertCouldGetRelation();

	/*
	 * first try to find reldesc in the cache
	 */
	RelationIdCacheLookup(relationId, rd);

	if (RelationIsValid(rd))
	{
		/* return NULL for dropped relations */
		if (rd->rd_droppedSubid != InvalidSubTransactionId)
		{
			Assert(!rd->rd_isvalid);
			return NULL;
		}

		RelationIncrementReferenceCount(rd);
		/* revalidate cache entry if necessary */
		if (!rd->rd_isvalid)
		{
			RelationRebuildRelation(rd);

			/*
			 * Normally entries need to be valid here, but before the relcache
			 * has been initialized, not enough infrastructure exists to
			 * perform pg_class lookups. The structure of such entries doesn't
			 * change, but we still want to update the rd_rel entry. So
			 * rd_isvalid = false is left in place for a later lookup.
			 */
			Assert(rd->rd_isvalid ||
				   (rd->rd_isnailed && !criticalRelcachesBuilt));
		}
		return rd;
	}

	/*
	 * no reldesc in the cache, so have RelationBuildDesc() build one and add
	 * it.
	 */
	rd = RelationBuildDesc(relationId, true);
	if (RelationIsValid(rd))
		RelationIncrementReferenceCount(rd);
	return rd;
}
```

三个分支道尽了 relcache 的运转方式：**命中即用**（加引用计数）、**脏了懒重建**（失效只标记，使用时才重建——避免失效风暴时做无用功）、**没有就构建**。两处细节：

- 函数头注释（`relcache.c:2085-2086`）："NB: caller should already have at least AccessShareLock on the relation ID, else there are nasty race conditions."——**先锁后拿描述符**是硬性前置条件，为什么，11.4.6 完整论证；
- 懒重建分支里的 Assert 揭示了启动期的自举妥协：`criticalRelcachesBuilt` 为假（关键系统表/索引的 relcache 还没建全）时，nailed 条目允许带着 `rd_isvalid = false` 继续用——它们的**结构**是硬编码的、不会错，只是 `rd_rel` 副本可能旧。这是"正确性分级"：结构性字段必须对，统计性字段允许暂时旧。

### 11.3.3 RelationBuildDesc 主线：拼装流水线与 in-progress 重试协议

`RelationBuildDesc()`（`relcache.c:1059`）是"没有就构建"分支的主力。主线（按源码顺序，行号为现场）：

1. **注册在建名单**（`relcache.c:1095-1108`）：把 `targetRelId` 记入 `in_progress_list`，并设 `invalidated = false`，然后进入 `retry:` 标签。这个名单的作用：**构建期间**若有针对该表的失效消息到达（构建过程要读大量目录，随时可能 `AcceptInvalidationMessages`），`RelationCacheInvalidateEntry()`（`relcache.c:2957-2959`）发现该 OID 尚不在哈希表里，就把在建名单里的对应项标记 `invalidated = true`；
2. **取 pg_class 行**：`ScanPgRelation(targetRelId, true, false)`（`relcache.c:1113`）——底下就是 syscache/systable 扫描，这是两层缓存的层叠点。查不到返回 NULL（"表刚被删了"）；
3. **分配骨架**：`AllocateRelationDesc(relp)` 在 CacheMemoryContext 里分配 `RelationData` 并拷贝 pg_class 行到 `rd_rel`；
4. **初始化事务感知字段**（`relcache.c:1155-1199`）：新建条目的四个 SubTransactionId 全部置 Invalid（"不是本事务新建的"），并按 `relpersistence` 分流——临时表要解出属主 backend（注释详述了"捡到别人崩溃残留的临时表"这个坑：靠 `rd_islocaltemp` 而不是 `rd_backend == MyProcNumber` 判断"是我的"）；
5. **拼列描述**：`RelationBuildTupleDesc(relation)`（`relcache.c:1204`）逐列查 pg_attribute（走 syscache）填 `rd_att`；
6. **按需挂附件**（`relcache.c:1206-1266`）：外键、分区信息一律**先置空、用到再加载**（"not loaded till asked for"）；索引表走 `RelationInitIndexAccessInfo`，堆表/序列走 `RelationInitTableAccessMethod`；有规则才 `RelationBuildRuleLock`（规则住进私有 `rd_rulescxt`）、有触发器才 `RelationBuildTriggers`、有 RLS 才 `RelationBuildRowSecurity`；
7. **锁信息与物理定位**：`RelationInitLockInfo` + `RelationInitPhysicalAddr`（`relcache.c:1271-1276`），然后 `rd_smgr = NULL`（文件句柄惰性打开）；
8. **失效检查与重试**（`relcache.c:1287-1299`，逐字）：

```c
	if (in_progress_list[in_progress_offset].invalidated)
	{
		RelationDestroyRelation(relation, false);
		goto retry;
	}
```

刚拼好的描述符如果在拼装期间已经过期，**当场销毁、整个重来**。注释（`relcache.c:1286-1292`）还立了一条纪律：从这个检查点到函数结束**不得再插入任何可能读目录的代码**——否则又开出新的失效窗口。这与 catcache 的 in-progress 栈是同一个问题的两处解法（catcache 按哈希值标记在建条目，relcache 按 OID 标记在建名单）；

9. **入表**：`RelationCacheInsert` 挂进以 OID 为键的动态哈希表（还要处理递归构建导致"已存在同 OID 条目"的角落情形，`relcache.c:1304` 起注释）。

这条流水线大量消费 syscache——**relcache 是 syscache 的下游**；而 plancache 又是两者的下游（11.4.7）。失效自然也要按这个方向层层传播。

另一处值得注意的取舍写在函数开头（`relcache.c:1067-1081`）：构建期的临时分配（各种中间元组、列表）**通常直接漏在调用方上下文里**——注释坦言"Traditionally we've just leaked that data"，理由是调用方上下文至多事务级、构建又不频繁，回收不值得多建一个上下文；只有 `debug_discard_caches` 压测（每次都重建）时才启用专门的临时上下文。这是 MemoryContext 哲学的直白示范：**"泄漏"是否成立取决于所在上下文的生命周期**（对照 11.8-W3 的反例）。

### 11.3.4 失效到达时：RelationFlushRelation 的三岔口

11.4 会讲消息如何到达；这里先看到达后 relcache 侧做什么。`RelationCacheInvalidateEntry()`（`relcache.c:2942`）查哈希表，找到则调 `RelationFlushRelation()`（`relcache.c:2831`），后者是个三岔口（源码逻辑概括，关键条件逐字）：

- **本事务新建/换过文件的表**（`rd_createSubid != InvalidSubTransactionId || rd_firstRelfilelocatorSubid != InvalidSubTransactionId`）：**必须就地重建而非丢弃**——丢了条目就丢了"新建"状态（那是本地独有信息，目录里没有）。重建时临时 `RelationIncrementReferenceCount` 保护自己；
- **无人引用的普通表**（`RelationHasReferenceCountZero`）：直接 `RelationClearRelation` 从缓存删掉——下次用到再全新构建，最省事；
- **有人正引用的表**：若在事务内，`RelationRebuildRelation`（`relcache.c:2589`）**就地重建**：新拼一个描述符，再把新内容换进旧的 `RelationData` 外壳——因为持引用者手里攥着旧指针，外壳地址不能变；若 TupleDesc 等内容未变还会保留原子结构（`keep_tupdesc`），让持有者手里的 `rd_att` 指针继续有效。不在事务内（无法读目录）就只能 `RelationInvalidateRelation` 标脏等下次。

"就地重建保指针"的策略可以追溯到 2000 年 Tom Lane 的提交 `a152ebeec6`（见 11.7-D6）：SI 队列溢出引发的全量刷缓存曾让"持有旧 tupledesc 指针"的代码崩溃，从那以后 relcache 重建的第一原则就是**外壳与关键子结构尽量原位复用**。

### 11.3.5 relcache init file：启动加速与运行期自举（Q 之 B⑤）

现在回收 11.1 留下的伏笔。新 backend 启动时要查 pg_class——查 pg_class 得先有 pg_class 的 relcache 条目（否则堆访问层连元组都解不开）；扫它的索引还得先有那些索引的 relcache 条目。这就是**运行期的鸡生蛋**。PG 的解法是两级：

- **保底**：`formrdesc()`（`relcache.c:1889`）用编译进二进制的 `FormData_pg_class`/`FormData_pg_attribute` 硬造核心系统表的描述符并钉死（nailed）。硬造的条目没有索引信息、`rd_rel` 也不完整，但足以支撑最初几次目录堆扫描，随后再按部就班升级成完整条目（分阶段由 `criticalRelcachesBuilt` 等标志推进）；
- **加速**：把构建好的系统表 relcache 条目序列化到数据目录下的 **`pg_internal.init`** 文件（文件名定义在 `src/include/utils/relcache.h:25`）。`load_relcache_init_file()`（`relcache.c:6202`）启动时直接反序列化，跳过整套"硬造→升级"的目录扫描；`write_relcache_init_file()`（`relcache.c:6622`）在条目齐全后写出——先写临时文件再 rename 的经典原子写法。

init file 是磁盘上的缓存，于是它也有失效问题，而且更阴险：对系统表做 DDL（如 REINDEX 系统索引）后，**必须防止别的 backend 在窗口期用旧文件启动**。看 `AtEOXact_Inval()` 的编排（11.4.2 摘录）：`RelationCacheInitFilePreInvalidate()`（**先删文件**）→ 广播失效消息 → `RelationCacheInitFilePostInvalidate()`。顺序不能反：若先广播后删文件，某个新 backend 可能在两步之间启动——它读了旧文件，又因为太新而错过了消息，从此带着旧缓存运行。先删文件则新 backend 只能走慢路径现读目录，必然看到新数据。这是"多副本缓存 + 广播失效"里典型的**窗口排序**问题。

### 小结

- relcache 以 OID 为键缓存 `RelationData`——表的全息描述符：物理定位/生命周期/事务感知/逻辑描述/附件/分区六组字段；大附件各住私有 MemoryContext，重建即整块蒸发；
- `RelationIdGetRelation()` 三分支：命中、脏则懒重建、缺则 `RelationBuildDesc()` 现拼；调用前必须已持锁；
- `RelationBuildDesc` 是消费 syscache 的拼装流水线，用 in-progress 名单 + `goto retry` 对抗"拼装期间失效到达"；
- 失效到达时三岔口：本事务新建的必须就地重建（保住本地独有状态）、无人用的直接删、有人用的就地重建保指针；
- 运行期自举 = formrdesc 硬造 + nailed 钉死 + 分阶段升级；`pg_internal.init` 加速启动，其失效靠"先删文件、再广播、后允许重写"的窗口排序。

## 11.4 缓存失效：多进程下的一致性

现在到真正的难题。缓存是**每 backend 私有**的，而目录是共享的。backend A `ALTER TABLE t ADD COLUMN` 提交后，backend B 缓存里的 t 还是旧结构——如果 B 拿着旧 TupleDesc 去解读新格式的元组，就是数据损坏。PG 的解法是三件套：**失效消息 + 共享内存环形队列 + 锁**。

### 11.4.1 所以然（Q1）：为什么是"提交时广播 + 接收方懒处理"

先推演一遍设计空间。"写方直接修改读方的缓存"在多进程下**根本不可行**：缓存住在各 backend 的私有地址空间，A 摸不到 B 的内存。就算搬进共享内存也不行——B 可能正拿着指向某条目的裸指针在栈上，A 并发改写/释放它就是悬垂指针；要避免就得给每次缓存访问加跨进程锁，把最热的路径（每条 SQL 几百次缓存命中）变成锁竞争场。结论：**失效只能由每个进程自己、在自己确认安全的时刻执行**。写方能做的只是"通知"。

通知的送达时机也有讲究。逐条推演：

- **改目录时立即广播？** 不行。事务还没提交，其他 backend 按 MVCC 根本看不见新目录行；让它们现在丢缓存、重读目录，读回来的还是旧行——白干。更糟的是若事务最终回滚，这次广播就是彻头彻尾的误伤；
- **提交时广播**：正确的时机。提交瞬间新目录行对外可见，此刻通知别人"你缓存的那行变了"恰如其分。于是 inval.c 的做法是：DDL 修改目录元组时（堆更新路径调 `CacheInvalidateHeapTuple()`，`inval.c:1568`），把失效消息**先攒在事务本地链表里**（TopTransactionContext 中），提交时一次性广播；
- **回滚呢？** 共享队列**一条都不发**——别的 backend 从未见过被回滚的目录行，它们的缓存从头到尾是对的，无需打扰。但**自己**必须善后：本事务内每个命令结束时（`CommandEndInvalidationMessages()`，`inval.c:1406`，由 `CommandCounterIncrement` 调用，`xact.c:1633`）已经**在本进程内**执行过这些失效（否则同事务内"CREATE TABLE 后立刻 SELECT 它"都做不到——自己得看见自己的 DDL）；也就是说自己的缓存已按"新目录"刷新过。现在新目录作废了，就得把这些缓存再刷一遍让它们回到"旧目录"。所以 `AtEOXact_Inval(false)` 在本地**重放**攒下的消息（消息语义是"哈希值 H 的条目作废"，不含新旧数据，重放即"再失效一次"，下次访问按已回滚后的目录快照重读——天然幂等，方向无关）。这就是回滚路径"本地重放、共享不发"的完整论证；
- **接收方为什么懒处理？** 消息躺在共享队列里，接收方不会被打断去处理（信号只置标志）。原因同开头：失效要动缓存数据结构，而接收方可能正在**使用**这些结构（比如正拿着一个 catcache 元组指针在解析）。只有接收方自己知道什么时候"手上没有裸指针"——这些时刻就是 11.4.4 的安全点。把处理推迟到安全点，等价于把"跨进程并发问题"降级成"单进程内的调用时序问题"。

一句话：**多进程地址空间隔离 ⇒ 失效必须自助；MVCC 可见性 ⇒ 广播必须在提交；裸指针安全 ⇒ 处理必须在安全点**。三个约束叠加，答案唯一。

### 11.4.3 传输层：sinvaladt.c 的共享内存环形队列

`SendSharedInvalidMessages()`（`src/backend/storage/ipc/sinval.c:47`）是薄封装，直接落到 `src/backend/storage/ipc/sinvaladt.c`。先看数据结构（`sinvaladt.c:130-135`、`138-163`、`166-197`）：

```c
#define MAXNUMMESSAGES 4096
#define MSGNUMWRAPAROUND (MAXNUMMESSAGES * 262144)
#define CLEANUP_MIN (MAXNUMMESSAGES / 2)
#define CLEANUP_QUANTUM (MAXNUMMESSAGES / 16)
#define SIG_THRESHOLD (MAXNUMMESSAGES / 2)
#define WRITE_QUANTUM 64
```

`SISeg` 是共享内存段：`minMsgNum`（所有人都读过的水位）/`maxMsgNum`（下一条消息号）/`nextThreshold`（下次清理阈值）+ 4096 槽的 `buffer[]` + 每 backend 一个 `ProcState`（`nextMsgNum` 消费位点、`resetState`、`signaled`、`hasMessages`、`sendOnly` 五个关键字段）。消息号**单调递增**，槽位 = `消息号 % MAXNUMMESSAGES`——Kafka 位点既视感；`MSGNUMWRAPAROUND` 定期把所有序号整体平移防 int 溢出。

**写入端 `SIInsertDataEntries()`（`sinvaladt.c:372`）全走读**，主体逐字：

```c
	while (n > 0)
	{
		int			nthistime = Min(n, WRITE_QUANTUM);
		int			numMsgs;
		int			max;
		int			i;

		n -= nthistime;

		LWLockAcquire(SInvalWriteLock, LW_EXCLUSIVE);

		/*
		 * If the buffer is full, we *must* acquire some space.  Clean the
		 * queue and reset anyone who is preventing space from being freed.
		 * Otherwise, clean the queue only when it's exceeded the next
		 * fullness threshold.  We have to loop and recheck the buffer state
		 * after any call of SICleanupQueue.
		 */
		for (;;)
		{
			numMsgs = segP->maxMsgNum - segP->minMsgNum;
			if (numMsgs + nthistime > MAXNUMMESSAGES ||
				numMsgs >= segP->nextThreshold)
				SICleanupQueue(true, nthistime);
			else
				break;
		}

		/*
		 * Insert new message(s) into proper slot of circular buffer
		 */
		max = segP->maxMsgNum;
		while (nthistime-- > 0)
		{
			segP->buffer[max % MAXNUMMESSAGES] = *data++;
			max++;
		}

		/* Update current value of maxMsgNum using spinlock */
		SpinLockAcquire(&segP->msgnumLock);
		segP->maxMsgNum = max;
		SpinLockRelease(&segP->msgnumLock);

		/*
		 * Now that the maxMsgNum change is globally visible, we give everyone
		 * a swift kick to make sure they read the newly added messages.
		 * Releasing SInvalWriteLock will enforce a full memory barrier, so
		 * these (unlocked) changes will be committed to memory before we exit
		 * the function.
		 */
		for (i = 0; i < segP->numProcs; i++)
		{
			ProcState  *stateP = &segP->procState[segP->pgprocnos[i]];

			stateP->hasMessages = true;
		}

		LWLockRelease(SInvalWriteLock);
	}
```

逐步展开：

1. **分批（WRITE_QUANTUM=64）**：函数头注释（`sinvaladt.c:376-383`）解释——不是怕挡住其他写者，而是怕挡住"刚追上进度、正想做 SICleanupQueue 传递 catchup 信号"的 backend 太久；顺带保证清理检查每 64 条至少做一次；
2. **两个清理触发条件**：队列真装不下（**必须**腾地方，reset 挡路者也在所不惜）或消息数越过 `nextThreshold`（例行清理，摊薄成本）。清理后**循环重查**——`SICleanupQueue` 可能中途放锁（见下），不保证一次腾够；
3. **写缓冲在先，改 `maxMsgNum` 在后，且改动用自旋锁包住**：这是无锁读者协议的另一半——读者以 `maxMsgNum` 为界，只要界后于数据可见，读者永远不会读到写了一半的槽。文件头注释（`sinvaladt.c:100-102`）强调自旋锁自带的内存屏障正是弱内存序平台上让"数据先于序号可见"成立的机制；
4. **广播 `hasMessages` 标志**：给每个在册进程置一位。这个标志是读端零成本快速路径的钥匙。

**读取端 `SIGetDataEntries()`（`sinvaladt.c:475`）全走读**，主体逐字：

```c
	segP = shmInvalBuffer;
	stateP = &segP->procState[MyProcNumber];

	/*
	 * Before starting to take locks, do a quick, unlocked test to see whether
	 * there can possibly be anything to read.  On a multiprocessor system,
	 * it's possible that this load could migrate backwards and occur before
	 * we actually enter this function, so we might miss a sinval message that
	 * was just added by some other processor.  But they can't migrate
	 * backwards over a preceding lock acquisition, so it should be OK.  If we
	 * haven't acquired a lock preventing against further relevant
	 * invalidations, any such occurrence is not much different than if the
	 * invalidation had arrived slightly later in the first place.
	 */
	if (!stateP->hasMessages)
		return 0;

	LWLockAcquire(SInvalReadLock, LW_SHARED);

	/*
	 * We must reset hasMessages before determining how many messages we're
	 * going to read.  That way, if new messages arrive after we have
	 * determined how many we're reading, the flag will get reset and we'll
	 * notice those messages part-way through.
	 *
	 * Note that, if we don't end up reading all of the messages, we had
	 * better be certain to reset this flag before exiting!
	 */
	stateP->hasMessages = false;

	/* Fetch current value of maxMsgNum using spinlock */
	SpinLockAcquire(&segP->msgnumLock);
	max = segP->maxMsgNum;
	SpinLockRelease(&segP->msgnumLock);

	if (stateP->resetState)
	{
		/*
		 * Force reset.  We can say we have dealt with any messages added
		 * since the reset, as well; and that means we should clear the
		 * signaled flag, too.
		 */
		stateP->nextMsgNum = max;
		stateP->resetState = false;
		stateP->signaled = false;
		LWLockRelease(SInvalReadLock);
		return -1;
	}

	n = 0;
	while (n < datasize && stateP->nextMsgNum < max)
	{
		data[n++] = segP->buffer[stateP->nextMsgNum % MAXNUMMESSAGES];
		stateP->nextMsgNum++;
	}

	/*
	 * If we have caught up completely, reset our "signaled" flag so that
	 * we'll get another signal if we fall behind again.
	 *
	 * If we haven't caught up completely, reset the hasMessages flag so that
	 * we see the remaining messages next time.
	 */
	if (stateP->nextMsgNum >= max)
		stateP->signaled = false;
	else
		stateP->hasMessages = true;

	LWLockRelease(SInvalReadLock);
	return n;
```

逐步展开：

1. **无锁快速退出**：`hasMessages` 为假直接返回——保证"没有 DDL 发生时，检查失效几乎零成本"。这是它能被每次拿锁、每次事务开始高频调用的前提。开头那段注释是内存模型论证的范本：这个无锁读可能被 CPU 乱序提前、错过刚写入的消息——但它**不可能被乱序到更早的一次锁获取之前**，而 11.4.6 的协议恰恰保证"需要看见的消息一定在拿锁之前已发出"，所以窗口是无害的（等价于"消息晚到了一点"）；
2. **先清标志、后定界**：顺序精心设计——若在"定界之后、处理期间"又有新消息到达，写者会重新置位 `hasMessages`，下次调用不会漏。若顺序反过来（先读 max 再清标志），"读 max 之后、清标志之前"到达的消息会连同标志一起被清掉，**永久丢失**。这一对读写者靠单个 bool 完成的握手是无锁协议的教科书样例（不变式 11.6-I8）；
3. **reset 路径**：`resetState` 为真说明自己落后太多、位点已被作废（谁作废的？见下面 `SICleanupQueue`）。此时把位点跳到 `max`（承认从此之前的消息都"处理过了"——反正马上要全量重置缓存）并返回 -1，上层将调用核弹级的 `InvalidateSystemCaches()`；
4. **`SInvalReadLock` 共享模式的反常用法**：函数头注释（`sinvaladt.c:459-465`）特别警告——多个读者并发各改各的 `ProcState`，共享锁在这里不是"只读"而是"互不相扰 + 排斥清理者"（`SICleanupQueue` 以排他模式拿它来冻结所有读者位点）。

**清理端 `SICleanupQueue()`（`sinvaladt.c:579`）全走读**，核心循环逐字：

```c
	min = segP->maxMsgNum;
	minsig = min - SIG_THRESHOLD;
	lowbound = min - MAXNUMMESSAGES + minFree;

	for (i = 0; i < segP->numProcs; i++)
	{
		ProcState  *stateP = &segP->procState[segP->pgprocnos[i]];
		int			n = stateP->nextMsgNum;

		/* Ignore if already in reset state */
		Assert(stateP->procPid != 0);
		if (stateP->resetState || stateP->sendOnly)
			continue;

		/*
		 * If we must free some space and this backend is preventing it, force
		 * him into reset state and then ignore until he catches up.
		 */
		if (n < lowbound)
		{
			stateP->resetState = true;
			/* no point in signaling him ... */
			continue;
		}

		/* Track the global minimum nextMsgNum */
		if (n < min)
			min = n;

		/* Also see who's furthest back of the unsignaled backends */
		if (n < minsig && !stateP->signaled)
		{
			minsig = n;
			needSig = stateP;
		}
	}
	segP->minMsgNum = min;
```

逐步展开（含函数其余部分）：

1. **两条落后线**：`lowbound`（= 若不清到这里就腾不出 `minFree` 槽）以下的 backend **直接 reset**——"你的位点作废了，别追了，到时候全量重建吧"；`minsig`（落后超过 SIG_THRESHOLD=2048 条）以下、且尚未被催过（`signaled`）的最落后者记为 `needSig`；
2. **`sendOnly` 豁免**：恢复期的 Startup 进程只发不收（它没有 relcache 可失效，但要替 redo 的 DDL 发消息），不参与水位计算——否则它永远不前进的位点会卡死整个队列；
3. **序号防溢出**：`min >= MSGNUMWRAPAROUND`（约 10.7 亿）时全体序号平移（`sinvaladt.c:644-650`）；
4. **自适应阈值**：`nextThreshold = (numMsgs / CLEANUP_QUANTUM + 1) * CLEANUP_QUANTUM`（`sinvaladt.c:656-660`）——队列越满、清理越勤；
5. **放锁再发信号**（`sinvaladt.c:667-679`）：`SendProcSignal` 不快，绝不持锁做它。每次**只催一个人**（最落后者）——`ReceiveSharedInvalidMessages` 尾部（`sinval.c:135-140`）在自己追平后会再调 `SICleanupQueue(false, 0)` 把信号**接力**给下一个最落后者。注释称之为 "daisy chaining"：把催赶负载摊成链式的后台涓流，避免同时惊醒一群进程造成负载尖峰。

这套"落后者 reset + 单点催赶 + 接力"的组合拳来自 2008 年 Tom Lane 的整体重写 `fad153ec45`（11.7-D2）。设计哲学：**宁可某个空闲连接偶尔付出全量重建缓存的代价，也绝不让 DDL 被最慢的消费者拖死**——DDL 持有的往往是全库瞩目的排他锁，它每多等一秒，代价都在全局放大。

### 11.4.4 接收端：AcceptInvalidationMessages 与安全点

真正的消费入口 `AcceptInvalidationMessages()`（`inval.c:930`）主体逐字：

```c
void
AcceptInvalidationMessages(void)
{
#ifdef USE_ASSERT_CHECKING
	/* message handlers shall access catalogs only during transactions */
	if (IsTransactionState())
		AssertCouldGetRelation();
#endif

	ReceiveSharedInvalidMessages(LocalExecuteInvalidationMessage,
								 InvalidateSystemCaches);
	...
}
```

（省略部分是 `debug_discard_caches` 压测钩子：人为在每个可能失效的点强制全量刷缓存，把罕见竞态变成必现，`inval.c:941-977` 注释值得一读。）第一个回调逐条应用消息，第二个回调处理 reset（-1）情形。

中间层 `ReceiveSharedInvalidMessages()`（`sinval.c:69`）有个乍看古怪的实现：用**静态**数组和**静态** volatile 游标攒消息，先处理"外层递归剩下的"，再循环 `SIGetDataEntries` 批量取（每批 32 条）。为什么静态？函数头注释（`sinval.c:60-66`）：处理一条失效消息（`invalFunction`）本身可能触发目录读取 → 再次进入 `AcceptInvalidationMessages` → **递归**进入本函数。递归层必须能接着消化外层已从共享队列取出但尚未处理的消息（否则那些消息就丢了——共享队列侧的位点已经前进）。静态缓冲让递归层直接看到外层的余货。这个递归危险在 2018 年才被彻底修顺（`f868a8143a`，11.7-D4）。

`LocalExecuteInvalidationMessage()`（`inval.c:823`）按消息类型分发，值得注意的几路：

- **catcache 消息**（`msg->id >= 0`，id 即 cache id）：先 `InvalidateCatalogSnapshot()`（目录快照也是缓存！），再 `SysCacheInvalidate(id, hashValue)` → `CatCacheInvalidate()`（`catcache.c:643`）：按哈希值定位桶，匹配条目**有人引用则标 `dead`、无人引用直接删**；正负条目一视同仁；顺带把 in-progress 栈里同哈希的在建条目标死。注释（`catcache.c:633-638`）记录了一个历史教训：早年按 TID 匹配，结果 VACUUM FULL 移动目录元组后 TID 变了、失效落空——改为按哈希值匹配，接受偶尔误伤（同哈希不同键的条目被无辜清除，代价只是重新加载）；
- **relcache 消息**：`RelationCacheInvalidateEntry(relId)` → 11.3.4 的三岔口；`relId == InvalidOid` 表示全量（来自 reset）；
- **smgr/relmap 消息**：文件句柄缓存与表空间映射缓存——广义上"一切跨进程可变共享状态的本地副本"都挂在这一套失效总线上。

**安全点清单**——`AcceptInvalidationMessages` 在哪些点被调用：

1. **事务开始**：`AtStart_Cache()`（`src/backend/access/transam/xact.c:1197`）就一行 `AcceptInvalidationMessages()`。保证每个事务开局看到的目录不旧于事务开始时刻；
2. **拿表锁时**：`LockRelationOid()`（`lmgr.c:107`，调用点 136 行）——11.4.6 详述，正确性的支柱；
3. **收到 catchup 信号后**：`HandleCatchupInterrupt()`（`sinval.c:154`）在信号处理器里只置 `catchupInterruptPending` 标志（信号处理器里绝不能碰缓存这种复杂结构）；真正的处理由 `ProcessCatchupInterrupt()`（`sinval.c:173`）在读客户端命令的间隙完成——若在事务内直接 `AcceptInvalidationMessages()`，否则**起一个空事务再立即提交**，让事务开始路径顺带消化消息。注释（`sinval.c:185-189`）坦言这是防御式选择："直接调大概也行，但出错路径的收拾没把握"——宁绕远路走已验证的路径；
4. 其他零散点：目录扫描开始（`systable_beginscan` 深处的快照获取）等。

这些点的共性：**当前代码不持有任何"过期了会出事"的缓存裸指针**。这正是不变式 11.6-I1。

### 11.4.5 时序窗口：光有消息队列为什么不够

只靠上述机制有个致命窗口：

```
backend B: 查缓存得到表 t 的旧描述符 ──────────────► 用旧描述符读 t
                              ▲
backend A:        ALTER TABLE t 提交，广播失效消息
```

B 在"查缓存"和"使用"之间没有义务消费消息（它可能正处于两个安全点之间的长执行路径上）。消息队列只保证**最终**知道，不保证**及时**。补洞的是锁。

### 11.4.6 锁-失效协议：LockRelationOid 全函数

`lmgr.c:107` 全函数逐字摘录：

```c
void
LockRelationOid(Oid relid, LOCKMODE lockmode)
{
	LOCKTAG		tag;
	LOCALLOCK  *locallock;
	LockAcquireResult res;

	SetLocktagRelationOid(&tag, relid);

	res = LockAcquireExtended(&tag, lockmode, false, false, true, &locallock,
							  false);

	/*
	 * Now that we have the lock, check for invalidation messages, so that we
	 * will update or flush any stale relcache entry before we try to use it.
	 * RangeVarGetRelid() specifically relies on us for this.  We can skip
	 * this in the not-uncommon case that we already had the same type of lock
	 * being requested, since then no one else could have modified the
	 * relcache entry in an undesirable way.  (In the case where our own xact
	 * modifies the rel, the relcache update happens via
	 * CommandCounterIncrement, not here.)
	 *
	 * However, in corner cases where code acts on tables (usually catalogs)
	 * recursively, we might get here while still processing invalidation
	 * messages in some outer execution of this function or a sibling.  The
	 * "cleared" status of the lock tells us whether we really are done
	 * absorbing relevant inval messages.
	 */
	if (res != LOCKACQUIRE_ALREADY_CLEAR)
	{
		AcceptInvalidationMessages();
		MarkLockClear(locallock);
	}
}
```

逐段展开：

1. `SetLocktagRelationOid`（`lmgr.c:88`）：共享目录（pg_database 等）用 `InvalidOid` 作库号——它们的锁必须全集群互斥；普通表用 `MyDatabaseId`；
2. `LockAcquireExtended(...)`：走常规锁管理器，可能在此**睡眠等待**冲突锁释放；
3. **拿到锁后立刻 `AcceptInvalidationMessages()`**——整个协议的画龙点睛。为什么必须是"拿锁后"？完整论证如下。

**协议双方的义务**：

- **DDL 方**：改表结构必须先拿该表的 `AccessExclusiveLock`（与一切模式冲突）。这保证它动手时**没有任何人**正持有该表的锁（也就没有人正在用它的描述符——因为"用前必拿锁"是另一半义务）。提交流程的顺序是：改目录 → `AtEOXact_Inval` 广播消息 → 事务收尾释放锁。**消息先于锁释放进入共享队列**，这个顺序是协议的根基；
- **DML 方**：使用表之前必须 `relation_open(oid, 某种锁)` → `LockRelationOid()` → **获锁后先消费失效消息，再去碰 relcache**。

**穷举时序**（对任意一对并发的 DDL 方 A 与 DML 方 B）：

- **B 先拿到锁**：A 的排他锁请求排队等待。B 从拿锁到放锁的整个期间，t 的定义都不可能改变（改需要排他锁），B 用缓存的描述符天然正确。A 等到 B 放锁才能动手；
- **A 先拿到锁**：B 排队等待。等 B 被唤醒拿到锁时，A 必已提交（或回滚）并释放了锁，且**A 的失效消息已在队列里**（消息先于放锁）。B 获锁后的第一件事就是 `AcceptInvalidationMessages()`——旧条目被标脏/删除，B 随后 `RelationIdGetRelation()` 走"懒重建"分支拿到新结构。

两种时序都安全，且不存在第三种：锁管理器保证同一时刻不会两方都持有冲突锁。11.4.5 的窗口被焊死——**B 拿到锁的那一刻起到放锁为止，"缓存有效"有锁背书；拿锁瞬间的消费动作把此前的欠账一次结清**。

注释里点名的 `RangeVarGetRelid()` 是这个协议最微妙的用户：按名字找表是"查目录（可能命中过期缓存）→ 锁 OID → **重新查一遍**确认名字还指向这个 OID"的循环——第二次查询之所以能看到新目录，正因为锁后消费了消息。没有这一行，`DROP TABLE t; CREATE TABLE t(...)` 并发下按名开表就可能开到幽灵。

**`LOCKACQUIRE_ALREADY_CLEAR` 优化**：若本进程已持有同级锁且此前已消费过消息（`MarkLockClear` 标记过），则从那时起没人可能改过这张表（改需排他锁，而我一直持锁）——跳过 `AcceptInvalidationMessages`。**锁的持有期成了缓存有效性的租约**。注释后半段解释了为什么需要"cleared"状态而不能简单看"是否已持锁"：递归场景下，外层的 `AcceptInvalidationMessages` 可能正处理到一半（本函数可能被消息处理逻辑递归调用），锁在手不等于消息吃完了——`MarkLockClear` 只在**真正消化完**之后打标。这个精细化是 2018 年 `f868a8143a` 的一部分（11.7-D4）。

> **这回答了指南第 12 章的悬念**："为什么 PG 的 DDL 是事务性的？"因为目录修改走 MVCC（可回滚）、失效消息攒到提交才广播（回滚就地销毁）、并发安全由锁-失效协议兜底。MySQL 8.0 之前 DDL 不能回滚，正是因为老数据字典（.frm 文件）不在事务机制之内。

### 11.4.7 plancache 的失效联动

缓存的计划树引用了表结构与函数——它们同样会过期。`src/backend/utils/cache/plancache.c` 不自己发明机制，而是**挂接**到上面这套：`InitPlanCache()`（`plancache.c:148`）注册回调：

```c
	CacheRegisterRelcacheCallback(PlanCacheRelCallback, (Datum) 0);
	CacheRegisterSyscacheCallback(PROCOID, PlanCacheObjectCallback, (Datum) 0);
```

（`plancache.c:150-151`，其后还有 NAMESPACEOID 等多个 cache 的注册。）当 relcache 收到某表的失效消息时，`PlanCacheRelCallback()`（`plancache.c:2126`）遍历本 backend 的所有缓存计划，把**依赖该表**的计划标记 invalid；下次执行 prepared statement 时发现计划失效，从保存的解析树重新规划。函数（PROCOID）、类型的 syscache 失效同理。这就是"ALTER TABLE 后 prepared statement 不会用旧计划崩掉"的全部秘密——失效是一棵**传播树**：目录元组变更 → catcache/relcache 标脏 → 注册回调把 plancache 等下游一并标脏。11.4.4 里 `InvalidateSystemCachesExtended()`（`inval.c:785`）在 reset 时也要遍历这三张回调注册表，保证核弹波及所有下游。

### 小结

- 设计推演三段论：地址空间隔离 ⇒ 失效自助；MVCC 可见性 ⇒ 提交时才广播；裸指针安全 ⇒ 安全点懒处理。回滚 = 共享队列不发（别人从未看见）+ 本地重放（撤销 CommandEnd 时对自己缓存的新目录刷新）；
- 传输层 4096 槽环形队列：单调序号 + 取模；写端"先写数据后推 maxMsgNum（自旋锁+屏障）再广播 hasMessages"；读端"无锁快查→先清标志后定界→批量取"；清理端"落后越线者 reset、最落后者单点催赶、追平者接力（daisy chain）"；
- 接收端拉模式，安全点：事务开始、**获锁后**、catchup 信号的命令间隙处理；处理递归靠静态缓冲；
- 锁-失效协议：消息先于锁释放入队 + 获锁后先消费再用缓存，两条义务穷举掉所有并发时序；`ALREADY_CLEAR` 把锁持有期变成缓存租约；
- plancache 通过注册回调挂在 relcache/syscache 失效上，失效沿依赖方向形成传播树。

## 11.5 内存管理：MemoryContext 与 AllocSet

### 11.5.1 所以然（Q3）：从 elog(ERROR) 反推出 region-based

先立住约束条件，再看方案空间。PG 后端代码的运行环境有三个铁的事实：

- **事实一：错误可以在任意深度抛出。** `elog(ERROR)` 走 `siglongjmp`，从几十层调用栈深处直接飞回顶层恢复点（`postgres.c:4573`）。被跳过的每一层栈帧里已分配的内存，没有任何"函数返回路径"可以执行清理代码；
- **事实二：进程长命。** 连接池下一个 backend 活几天很正常，每条 SQL 漏几十字节，一天就是不可接受的增长；
- **事实三：分配极端高频。** 执行器处理每一行都可能分配（detoast、函数临时结果），一条扫百万行的查询就是百万次级别的分配。

拿方案空间逐一对照：

- **malloc + 手工 free**：事实一直接判死刑。要在 longjmp 语义下不漏，等价于"每个分配点都要把指针登记到某个错误路径可达的表里"——那张表本身就是一个简化版 MemoryContext。C 语言里凡是认真对待错误路径的大型项目，最终都会长出 region/pool（Apache 的 apr_pool 同理）；
- **C++ RAII / Rust 所有权**：语言机制上确实解决事实一（栈展开自动析构），但 PG 是 C；即便重写，RAII 的析构是**逐对象** O(n) 的，事实三下"百万行 × 每行若干对象"的逐个析构开销可观；而且数据库对象的生命周期与**语义阶段**（per-tuple/per-query/per-transaction/per-session）对齐，与词法作用域并不重合——一个 TupleDesc 从解析活到执行结束，横跨几十个函数，RAII 需要把所有权层层 move，表达远不如"分配在 per-query 上下文"一句话；
- **引用计数**：每次指针赋值都要维护计数，事实三下开销大；C 里没有编译器帮忙，漏加/多减就是泄漏/悬垂，纪律负担比手工 free 更重；且循环引用无解。PG 只在**少数跨模块共享的重对象**上用显式引用计数（catcache 条目、relcache 条目、buffer pin），并且全部配套 ResourceOwner 登记兜底——计数只是所有权协议，泄漏兜底另有机制；
- **追踪式 GC**：不确定性停顿对事务延迟是毒药；C 里精确 GC 不可行，保守 GC 有误保留。且数据库其实**清楚地知道**自己的生命周期阶段（一行、一条 SQL、一个事务），用 GC 等于扔掉这份信息再让运行时猜回来；
- **region-based（arena）**：把"生命周期"实体化为对象（MemoryContext），分配时声明归属，释放按 region 整体进行。事实一的解法：顶层恢复点删掉事务级 region 整棵子树，**不需要知道谁在哪一层分配了什么**；事实二的解法：region 的删除是结构保证的，与代码路径无关；事实三的解法：整块释放 O(块数)，per-tuple region 反复 reset 几乎零成本。三个事实全部命中。

所以结论不是"palloc 碰巧好用"，而是：**在"C + longjmp 错误模型 + 长命进程 + 高频分配"这组约束下，region-based 是唯一同时满足三个事实的方案**。代价（换来上述一切的）：生命周期错配不再是编译错误，而是运行期 bug——分配进太长命的上下文成为"逻辑泄漏"，太短命则悬垂（见 11.8-W3 与不变式 11.6-I5）。

机制框架在 `src/backend/utils/mmgr/mcxt.c`（上下文树、palloc/pfree 分发），默认实现 AllocSet 在 `aset.c`（另有 generation.c、slab.c、bump.c 等特化实现，接口相同，通过 `MemoryContextMethods` 虚表接入）。设计文档 `src/backend/utils/mmgr/README` 值得通读。

### 11.5.2 上下文树的骨干层级

`mcxt.c:167-173` 定义全局上下文指针（逐字）：


典型 backend 的树（括号内为创建位置，全部现场核实）：

```
TopMemoryContext                    进程根，与进程共存亡（mcxt.c:369，MemoryContextInit）
├── ErrorContext                    错误处理专用，8KB 常驻保命钱（mcxt.c:392-397）
├── CacheMemoryContext              relcache/syscache/plancache 的家（catcache.c:726 CreateCacheMemoryContext）
│                                   ——缓存长命，所以住长命上下文，靠 11.4 的失效机制而非生命周期管理
├── MessageContext                  当前客户端命令的原始文本/解析树（postgres.c:4531 创建，每条命令开头重置）
├── TopTransactionContext           事务级数据（xact.c:1235-1239，AtStart_Memory 惰性创建，事务收尾重置）
│   └── CurTransactionContext       当前(子)事务的数据（子事务时在 xact.c:1300 另建子上下文）
│        └── ExecutorState          ★ per-query：一次执行的所有状态（execUtils.c:100）
│             └── ExprContext       ★ per-tuple：表达式求值的临时内存（execUtils.c:265）
└── TransactionAbortContext         中止处理专用救生艇，32KB 常驻（xact.c:1222-1228）
```

`MemoryContextInit()`（`mcxt.c:362`）里创建 ErrorContext 的注释（`mcxt.c:380-387`）把"保命钱"说透了：要求它**任何时刻都保有至少 8KB**——"This is the only case where retained memory in a context is *essential* --- we want to be sure ErrorContext still has some memory even if we've run out elsewhere!" 并且用 `MemoryContextAllowInCriticalSection(ErrorContext, true)`（`mcxt.c:397`）豁免"临界区禁止分配"断言——否则 PANIC 报错会先死在断言上、看不到真正死因。

两个星号层级是性能与正确性的关键惯用法：执行器处理**每一行**时在 per-tuple 上下文里随便 palloc，处理完这行调一次 `MemoryContextReset()`（`mcxt.c:406`）整体清空——**内层循环完全不需要 pfree**。Reset 对 AllocSet 是把非 keeper 块还给 malloc、keeper 块指针复位，O(块数) 而非 O(分配次数)。

`MemoryContextSwitchTo()`（`src/include/utils/palloc.h:138`，两行 inline：换全局变量 `CurrentMemoryContext`，返回旧值）是切换当前上下文的唯一手段。PG 代码里"switch 进去分配、switch 回来"的配对无处不在——看懂它，执行器代码可读性瞬间提升。

### 11.5.3 AllocSetAlloc 全函数走读：三条路径

一个 AllocSet 向 OS（malloc）批发**块（block）**，向调用者零售 **chunk**。参数常量（`aset.c:83-85`、`src/include/utils/memutils.h:157-159`）：

```c
#define ALLOC_MINBITS		3	/* smallest chunk size is 8 bytes */
#define ALLOCSET_NUM_FREELISTS	11
#define ALLOC_CHUNK_LIMIT	(1 << (ALLOCSET_NUM_FREELISTS-1+ALLOC_MINBITS))
```

即 8、16、…、8192 字节共 11 级 freelist，超 8KB 走大块路径；标准参数 `ALLOCSET_DEFAULT_SIZES` = 初始块 8KB、封顶 8MB。

`AllocSetAlloc()`（`aset.c:1012`）**全函数**逐字摘录（省略 Valgrind 标注宏与 `MEMORY_CONTEXT_CHECKING`/`RANDOMIZE_ALLOCATED_MEMORY` 调试段——它们在生产构建为空；其余逐字）：

```c
void *
AllocSetAlloc(MemoryContext context, Size size, int flags)
{
	AllocSet	set = (AllocSet) context;
	AllocBlock	block;
	MemoryChunk *chunk;
	int			fidx;
	Size		chunk_size;
	Size		availspace;

	Assert(AllocSetIsValid(set));

	/* due to the keeper block set->blocks should never be NULL */
	Assert(set->blocks != NULL);

	/*
	 * If requested size exceeds maximum for chunks we hand the request off to
	 * AllocSetAllocLarge().
	 */
	if (size > set->allocChunkLimit)
		return AllocSetAllocLarge(context, size, flags);

	fidx = AllocSetFreeIndex(size);
	chunk = set->freelist[fidx];
	if (chunk != NULL)
	{
		AllocFreeListLink *link = GetFreeListLink(chunk);

		Assert(fidx == MemoryChunkGetValue(chunk));

		/* pop this chunk off the freelist */
		set->freelist[fidx] = link->next;

		return MemoryChunkGetPointer(chunk);
	}

	/*
	 * Choose the actual chunk size to allocate.
	 */
	chunk_size = GetChunkSizeFromFreeListIdx(fidx);
	Assert(chunk_size >= size);

	block = set->blocks;
	availspace = block->endptr - block->freeptr;

	/*
	 * If there is enough room in the active allocation block, we will put the
	 * chunk into that block.  Else must start a new one.
	 */
	if (unlikely(availspace < (chunk_size + ALLOC_CHUNKHDRSZ)))
		return AllocSetAllocFromNewBlock(context, size, flags, fidx);

	/* There's enough space on the current block, so allocate from that */
	return AllocSetAllocChunkFromBlock(context, block, size, chunk_size, fidx);
}
```

**路径一：freelist 命中**（最热）。`AllocSetFreeIndex()`（`aset.c:277`）把尺寸换算成级别：本质是 `ceil(log2(size >> 3))`，有 bitscan 指令的平台一条 `pg_leftmost_one_pos32((uint32) size - 1) - ALLOC_MINBITS + 1` 搞定，没有的平台手工展开字节查表——源码注释直言 "Yes, this function is enough of a hot-spot to make it worth this much trouble."。命中后弹出链表头即返回，全程十来条指令、**无锁**（上下文是进程私有的——多进程架构的隐性红利：最热的分配器不需要任何同步）。空闲链表的 next 指针直接**存在空闲 chunk 的用户数据区里**（`GetFreeListLink`），零元数据开销。

**路径二：活跃块切片 / 新块**。freelist 空则从当前活跃块尾部切一片（`AllocSetAllocChunkFromBlock()`，`aset.c:816`：`block->freeptr` 推进 `chunk_size + ALLOC_CHUNKHDRSZ`，写好 chunk 头返回——指针递增式分配）。活跃块不够则进 `AllocSetAllocFromNewBlock()`（`aset.c:861`），它做三件事：

1. **旧块尾巴一字不浪费**（`aset.c:887-926`）：把旧活跃块的剩余空间切成尽可能大的 2^n 片挂到对应 freelist——旧块即将被压到链表深处、永不再作为分配来源，不切就永久浪费。循环最多 `ALLOCSET_NUM_FREELISTS-1` 次（因为能走到这里剩余空间必小于 `ALLOC_CHUNK_LIMIT`）；
2. **块尺寸指数增长**（逐字，`aset.c:928-935`）：

```c
	/*
	 * The first such block has size initBlockSize, and we double the space in
	 * each succeeding block, but not more than maxBlockSize.
	 */
	blksize = set->nextBlockSize;
	set->nextBlockSize <<= 1;
	if (set->nextBlockSize > set->maxBlockSize)
		set->nextBlockSize = set->maxBlockSize;
```

默认 8KB→16KB→…→8MB 封顶。小上下文（绝大多数）只占一个 keeper 块；大上下文 malloc 次数 O(log n)——与动态数组扩容/Arrow 内存池同一招；

3. **malloc 失败降级**（`aset.c:956-962`）：大块 malloc 失败时逐次减半重试（不低于 1MB 且不小于本次所需）——尽力而为之后才报 OOM。

**路径三：大块直通**。`size > allocChunkLimit`（默认 8KB）走 `AllocSetAllocLarge()`（`aset.c:735`）：malloc 一个"正好装下这一个 chunk"的独立块，chunk 头打上 external 标记（`MemoryChunkSetHdrMaskExternal`），插到活跃块**之下**（`aset.c:782-799`，"Stick the new block underneath the active allocation block"——不能压住活跃块，否则它尾部的可用空间就被埋了）。大 chunk 被 `AllocSetFree()`（`aset.c:1107`）释放时**整块立即 free 还给 OS**——大内存不进 freelist 囤积；而小 chunk 被 pfree 只是挂回 freelist，要等上下文 reset/delete 才归还 OS。

上下文创建本身也被优化到极致：`AllocSetContextCreateInternal()`（`aset.c:347`）维护**上下文级 freelist**（`context_freelists[2]`，`aset.c:257`）——被删除的标准尺寸上下文，其头部连同 keeper 块整体回收复用，因为 PG 每查询要建/毁大量上下文（ExecutorState、ExprContext、各种算子的私有上下文）。

**碎片与浪费的量化（B④）**：

- **chunk 圆整浪费**：请求 s ∈ (2^k, 2^(k+1)] 分配 2^(k+1)。最坏 s = 2^k + 1，浪费 2^k − 1 字节，**趋近 50%**；请求尺寸均匀分布时期望浪费约 25%。换来的是：无相邻块合并、无 best-fit 搜索、free 是 O(1) 头插——分配器逻辑简单到能整体内联进热路径。PG 代码里大量结构体尺寸本来就贴近 2^n，实际浪费远低于最坏值；
- **freelist 囤积（高水位效应）**：pfree 的小 chunk 不归还 OS，而且**只能被同尺寸级的请求复用**。最坏情形：先在某上下文里分配 100 万个 32B 对象再全部 pfree，然后只分配 64B 对象——那 32MB 的 32B freelist 一字节都用不上，直到上下文 reset。这是"上下文该按生命周期划分"的另一个理由：同一上下文内的分配模式通常同质；
- **块翻倍的尾部浪费**：新块 malloc 后只切了一个 chunk 就不再分配的话，最坏闲置接近整块（8MB 封顶）——但这部分空间仍属于该上下文、后续分配会继续用，只有"上下文在分配高峰后长期存活但不再分配"才成为真浪费。`ALLOCSET_SMALL_SIZES`（小初始块）与 generation/slab/bump 等特化分配器就是针对已知分配模式砍这类开销的。

### 11.5.4 chunk header：pfree 怎么知道该找谁——逐行拆解

`pfree(p)` 只有一个裸指针，怎么知道 p 属于哪个上下文、该用哪个实现的 free？`mcxt.c:1619`（逐字，省略 USE_VALGRIND 段）：

```c
void
pfree(void *pointer)
{
	MCXT_METHOD(pointer, free_p) (pointer);
}
```

`MCXT_METHOD`（`mcxt.c:205-206`）：

```c
#define MCXT_METHOD(pointer, method) \
	mcxt_methods[GetMemoryChunkMethodID(pointer)].method
```

`GetMemoryChunkMethodID()`（`mcxt.c:213`）逐字（省略 Valgrind 宏）：

```c
static inline MemoryContextMethodID
GetMemoryChunkMethodID(const void *pointer)
{
	uint64		header;

	/*
	 * Try to detect bogus pointers handed to us, poorly though we can.
	 * Presumably, a pointer that isn't MAXALIGNED isn't pointing at an
	 * allocated chunk.
	 */
	Assert(pointer == (const void *) MAXALIGN(pointer));

	header = *((const uint64 *) ((const char *) pointer - sizeof(uint64)));

	return (MemoryContextMethodID) (header & MEMORY_CONTEXT_METHODID_MASK);
}
```

逐行：① 断言指针 MAXALIGN 对齐——palloc 的返回值必然对齐，不对齐的多半是野指针，尽力检测；② `p - 8` 读出 8 字节头 `hdrmask`；③ 取**最低 4 位**作为分配器 ID，查全局方法表 `mcxt_methods[]`（`mcxt.c:64`）分发到 `AllocSetFree`/`GenerationFree`/…——**C 语言手搓的虚函数表，对象头开销仅 4 bit**。

`hdrmask` 的完整编码在 `src/include/utils/memutils_memorychunk.h`（布局说明 24-51 行，字段宏 96-113 行）。64 位从低到高：

| 位段 | 宽度 | 含义 |
|---|---|---|
| bit 0-3 | 4 | `MemoryContextMethodID`（MCTX_ASET_ID / MCTX_GENERATION_ID / …，enum 在 `memutils_internal.h:123-137`） |
| bit 4 | 1 | external 标志（大块 chunk：块地址不靠偏移，而由各分配器自行推算——AllocSet 里就是"chunk 头减去块头尺寸"） |
| bit 5-34 | 30 | value 字段，分配器自由使用——AllocSet 存 freelist 级别 fidx（`AllocSetAllocChunkFromBlock` 里 `MemoryChunkSetHdrMask(chunk, block, fidx, MCTX_ASET_ID)`，free 时凭它 O(1) 归位） |
| bit 34-63 | 30 | chunk 到所在 block 起点的**字节偏移** |

注意 34 出现了两次——这不是笔误。头文件注释（`memutils_memorychunk.h:37-43`）解释了这个"65 bit 塞进 64 bit"的戏法：value 的最高位与 blockoffset 的最低位**共享同一个物理 bit**。之所以敢共享，是因为 chunk 和 block 都强制 MAXALIGN（至少 8 字节对齐），两者的偏移必然是 MAXALIGN 倍数、最低位恒为 0；读取偏移时用 `MEMORYCHUNK_BLOCKOFFSET_MASK`（`0x3FFFFFFE`，注意末位是 E）把这个共享位屏蔽掉即可。写入侧 `MemoryChunkSetHdrMask()`（`memutils_memorychunk.h:168-182`）就是三段移位取或。

于是 `GetMemoryChunkContext()`（`mcxt.c:759`）的完整链路是：`p - 8` 读头 → 低 4 位查虚表 → 各分配器的 `get_chunk_context` 实现按 blockoffset 找到 block → block 头里存着 `aset` 指针（`AllocBlockData` 的第一个字段）→ 上下文到手。external 大块则由 `AllocSetFree()`（`aset.c:1115-1127`）用 `ExternalChunkGetBlock` 直接定位，还顺手做“`freeptr == endptr`”的完整性校验（单 chunk 块必然写满），校验失败即 `elog(ERROR, "could not find block containing chunk %p", ...)`——对野指针的又一层尽力防御。

推论（不变式 11.6-I6）：palloc 的指针**绝不能** `free()`（malloc 不认这个头），malloc 的指针绝不能 pfree（低 4 位是随机的——`memutils_internal.h:124-125` 甚至专门把 glibc malloc 头常见的 0001/0010 两个码点保留为"故意无效"，让误传 malloc 指针大概率撞上 BOGUS 表项干净报错而不是乱跳）；也不能对指针做偏移后 pfree。

`palloc()` 本体（`mcxt.c:1390`）快到只有：取 `CurrentMemoryContext` → 清 `isReset` 标志 → 一次虚调用 `context->methods->alloc`。注释还点明连"检查返回值"都被挪进了 alloc 实现（`MemoryContextAllocationFailure`），让 palloc 尾调用直返——为了 sibling call 优化连一条比较指令都不肯多留。这一整套 8 字节头 + 4bit 虚表分发的设计出自 2022 年 David Rowley 的 `c6e0fe1f2a`（11.7-D5）：在此之前 chunk 头是 16 字节起步、且 pfree 要带 context 参数间接跳转两次；重构一举把头减半、把"由指针找上下文"变成纯位运算。ID 位宽从 3 扩到 4 则是 2024 年 `0ba8b75e7e`（bump 分配器加入后 8 个码点不够用了）。

### 11.5.5 错误恢复：sigsetjmp + 上下文重置 = 不可能泄漏

这是整套设计的点睛之笔。C 没有异常和 RAII，PG 用 `elog(ERROR)` → `siglongjmp` 模拟异常。问题是：长跳转飞过的几十层栈帧里 palloc 的内存、拿着的锁、打开的文件怎么办？

答案在 `PostgresMain()` 的顶层恢复块（`postgres.c:4573` 起，节选主干，语句均逐字）：

```c
	if (sigsetjmp(local_sigjmp_buf, 1) != 0)
	{
		/* Since not using PG_TRY, must reset error stack by hand */
		error_context_stack = NULL;

		/* Prevent interrupts while cleaning up */
		HOLD_INTERRUPTS();

		disable_all_timeouts(false);	/* do first to avoid race condition */
		QueryCancelPending = false;
		...
		/* Report the error to the client and/or server log */
		EmitErrorReport();
		...
		debug_query_string = NULL;

		/*
		 * Abort the current transaction in order to recover.
		 */
		AbortCurrentTransaction();
		...
		/*
		 * Now return to normal top-level context and clear ErrorContext for
		 * next time.
		 */
		MemoryContextSwitchTo(MessageContext);
		FlushErrorState();
```

`AbortCurrentTransaction()`（`xact.c:3504`）→ `AbortTransaction()`（`xact.c:2855`）的清理序列开头（逐字节选，顺序即源码顺序）：

```c
	/* Prevent cancel/die interrupt while cleaning up */
	HOLD_INTERRUPTS();
	...
	/* Make sure we have a valid memory context and resource owner */
	AtAbort_Memory();
	AtAbort_ResourceOwner();

	/*
	 * Release any LW locks we might be holding as quickly as possible.
	 * (Regular locks, however, must be held till we finish aborting.)
	 * Releasing LW locks is critical since we might try to grab them again
	 * while cleaning up!
	 */
	LWLockReleaseAll();
	...
	pgstat_report_wait_end();
	...
	/* Clean up buffer content locks, too */
	UnlockBuffers();

	/* Reset WAL record construction state */
	XLogResetInsertion();
	...
	LockErrorCleanup();
```

其后：写 abort 记录、按 ResourceOwner 逐一释放 buffer pin/catcache 引用/文件句柄、最后释放常规锁（注意上面注释：**常规锁必须留到中止收尾**才放——放早了别人会在我们清理到一半时闯进来，这正是 11.4.6 协议里"消息先于锁释放"在中止路径的镜像）。随后 `CleanupTransaction()`（`xact.c:3062`）里 `AtCleanup_Memory()`（`xact.c:2016`）重置 `TransactionAbortContext` 并把 **TopTransactionContext 整棵重置**——事务里不管哪层代码 palloc 过什么，无论错误发生在哪一行，全部随树蒸发。

三个支柱缺一不可：

1. **内存按生命周期归池**：事务内分配都在 TopTransactionContext 子树 → 删树即回收，不需要知道"谁分配了什么"；
2. **救生艇上下文**：`AtAbort_Memory()`（`xact.c:1926`）第一时间切进 `TransactionAbortContext`（`xact.c:1222-1228` 惰性创建，32KB 常驻，创建注释明说"insulate AbortTransaction from out-of-memory scenarios"）——**中止处理本身需要内存**，而当前上下文可能正是因 OOM 崩的，绝不能再在里面分配；同理 `EmitErrorReport` 用的是带 8KB 保命钱的 ErrorContext；
3. **非内存资源全部登记在 ResourceOwner**：buffer pin、catcache/relcache 引用（11.2.2 的 `ResourceOwnerRememberCatCacheRef`）、文件句柄、DSM 段……中止时按登记逐一释放，并在 assert 构建下对"忘了释放"的资源报 warning——泄漏在开发期就会尖叫。

> 对照大数据世界：这就是 Spark task 失败时整个 TaskMemoryManager 回收、Arrow 的 per-batch allocator 的思路——**把"正确释放"从每处代码的义务，变成结构的必然**。PG 在 1990 年代就把它做成了 C 代码的全局纪律。

### 11.5.6 与手工 malloc 管理和 RAII 的对比总表

| 维度 | 手工 malloc/free | C++ RAII / Rust 所有权 | PG MemoryContext |
|---|---|---|---|
| 释放的责任方 | 每个调用点（人肉配对） | 编译器按作用域插入析构 | 上下文生命周期的所有者（少数几处框架代码） |
| 错误路径安全 | 极难：每个 return/goto 都要补 free | 栈展开自动析构，天然安全 | longjmp + 删上下文树，天然安全 |
| 单次分配开销 | malloc：全局锁/bins 查找 | 同 malloc（或自定义 allocator） | freelist 命中约十条指令，无锁（进程私有） |
| 批量释放 | O(n) 逐个 free | O(n) 逐个析构 | O(blocks) 整树 reset/delete |
| 生命周期表达 | 无（靠注释） | 绑定作用域/所有者，编译期检查 | 绑定上下文树，**运行期结构**，可跨函数、可动态重挂（MemoryContextSetParent） |
| 典型风险 | 泄漏、double-free、悬垂指针 | 循环引用（shared_ptr）、析构顺序 | 分配到错误的上下文（活得太短→悬垂；太长→"逻辑泄漏"） |

补充两点讨论：

- **它不是 GC**：无追踪、无停顿、确定性释放；也**不是万能**——共享内存里的对象（锁表、缓冲池）不归 MemoryContext 管，跨进程动态内存用 DSA（下一章）另起一套；
- MySQL/InnoDB 的 `mem_heap_t` 是同思想近亲，但没有做成全内核统一的树形层级 + 错误恢复协议。PostgreSQL 把它升格为**所有代码共同遵守的资源模型**（内存归 context、其他资源归 ResourceOwner），这才是"防泄漏"的完整答案。诊断工具也围着它建：`MemoryContextStats()` 打印整棵树占用（可用 `pg_log_backend_memory_contexts()` 触发运行中的 backend 输出）、`pg_backend_memory_contexts` 视图实时可查。

### 小结

- 三个约束（longjmp 错误模型、长命进程、高频分配）反推出 region-based 是唯一解；代价是生命周期错配从编译错误降级为运行期 bug；
- 骨干层级各对应一种生命周期：进程（Top/Cache/Error）、命令（Message）、事务（TopTransaction）、查询（ExecutorState）、行（ExprContext）；救生艇（TransactionAbortContext/ErrorContext）常驻保命内存；
- AllocSetAlloc 三路径：freelist O(1) 无锁命中 → 活跃块指针递增切片/新块（旧块尾巴切碎入 freelist、块尺寸翻倍封顶、malloc 失败减半重试）→ 大块独立 malloc 且 pfree 立即归还；圆整最坏浪费 ~50%（期望 ~25%），freelist 有高水位囤积效应；
- chunk 头 8 字节：4bit 分配器 ID 查虚表 + 1bit external + 30bit value + 30bit 块偏移（借对齐共享 1 bit 塞下 65 bit 信息）；由指针找上下文是纯位运算；
- 错误恢复三支柱：生命周期归池、救生艇上下文、ResourceOwner 登记。内存安全来自结构而非纪律。

## 11.6 不变式清单

每条给出内容、依据现场、违反后果。

- **I1（失效处理点安全性）**：`AcceptInvalidationMessages()` 只允许在"当前代码不持有任何过期即出事的缓存裸指针"的安全点调用；反过来，任何可能触发目录读取的操作（`table_open`、TOAST 访问、syscache miss）都必须假定失效可能在其中发生。依据：catcache 的 in-progress 栈与 stale 重试环（`catcache.c:1571-1575`、`2198-2231`）、relcache 的 in_progress_list + `goto retry`（`relcache.c:1287-1297`）、`RelationBuildDesc` 尾部"此后不得读目录"的纪律注释。违反后果：缓存进陈旧条目且无人再失效它——后续所有查询拿到过期表结构，轻则查不到新列，重则按旧 TupleDesc 解读新元组导致数据错乱。
- **I2（先锁后描述符）**：使用任何表之前必须先 `LockRelationOid`（获锁后自动消费失效消息），再 `RelationIdGetRelation`。依据：`relcache.c:2085-2086` 的 NB 注释、`lmgr.c:118-133`。违反后果：11.4.5 的窗口重现——DDL 并发下拿到旧描述符且永远不知道它旧了。
- **I3（引用先登记后持有）**：对 catcache/relcache 条目 `refcount++` 之前必须先 `ResourceOwnerEnlarge` 保证登记有位置，加完立即 `Remember`。依据：`catcache.c:1496-1498`、`1612-1614`。违反后果：Enlarge 若在 refcount++ 之后 OOM 抛错，这笔引用无人登记、中止清理不会归还——条目永远 `refcount > 0`，失效时只能标 dead 不能释放，缓存里积累僵尸。
- **I4（提交顺序：消息先于放锁；init file 先删后广播）**：`AtEOXact_Inval(true)` 必须在释放常规锁之前执行；涉及系统表时 `RelationCacheInitFilePreInvalidate` 必须在广播之前、`PostInvalidate` 在之后。依据：`inval.c:1209-1227`、11.4.6 的穷举论证。违反后果：前者破坏锁-失效协议的"A 先拿锁"分支（B 获锁后消费不到 A 的消息）；后者留下"新 backend 读旧 init file 又错过消息"的窗口。
- **I5（CacheMemoryContext 生命周期规则）**：住在 CacheMemoryContext（及其子上下文）里的对象只能由缓存自身的失效/替换逻辑释放，事务中止不清理它们；因此**向其中写入必须要么一步到位、要么先在临时上下文构建完再搬入/重挂**。依据：`CatalogCacheCreateEntry` 一次 palloc 成型（`catcache.c:2236-2248`）、relcache 附件先建后挂的私有上下文模式（`rd_rulescxt` 等）。违反后果：构建中途 `elog(ERROR)`，半成品永久滞留——典型的会话级"逻辑泄漏"（见 11.8-W3）。
- **I6（palloc/malloc 不可混用，指针不可偏移释放）**：pfree 依赖 `p-8` 的 hdrmask；free() 依赖 malloc 自己的头。依据：11.5.4 全节、glibc 头码点被特意标为 BOGUS（`memutils_internal.h:124-125`）。违反后果：运气好撞上 BOGUS 表项报 "could not find block containing chunk"；运气差按错误的偏移找 block，读到垃圾指针后崩溃在千里之外。
- **I7（临界区禁止 palloc；清理路径不得在可能耗尽的上下文分配）**：`AssertNotInCriticalSection`（`mcxt.c:198-199`）；中止路径必须先切救生艇（`AtAbort_Memory`，`xact.c:1926`）。违反后果：临界区内 OOM 会把 ERROR 升级为 PANIC 全实例重启；中止路径在耗尽的上下文里再分配则中止本身失败，进入递归错误。
- **I8（sinval 读者协议：先清 hasMessages 再定界读 maxMsgNum）**：依据：`sinvaladt.c:501-510` 注释与代码顺序。违反后果：窗口内到达的消息连同标志一起被清，该 backend 静默错过失效——与 I1 违反同症：缓存悄然过期。
- **I9（消息处理的递归安全）**：失效处理代码若可能再触发 `AcceptInvalidationMessages`，必须保证外层已取出的消息不丢。依据：`ReceiveSharedInvalidMessages` 的静态缓冲设计（`sinval.c:60-66`）。违反后果：递归层重置游标覆盖外层余货，消息丢失——2018 年前这里真有 bug（11.7-D4）。

## 11.7 设计权衡与历史（git log 考古）

- **D1 · `26ac217173`（2002-03-03，Tom Lane）"Catcaches can now store negative entries as well as positive ones"**：负缓存的诞生。动机即 11.2.4 的推演（"speed up repetitive failed searches"）；副产品是 inval.c 大幅简化——有了"插入与删除对失效一视同仁"的语义（都按键哈希清条目），发送端不再需要区分事件类型。提交信息还记录了它揪出的老 bug：`heap_create_with_catalog` 曾在目录行自洽之前就做 CommandCounterIncrement，新失效逻辑一上线立刻 assert 失败——失效机制越精确，对"目录行必须时刻自洽"的纪律要求越苛刻。
- **D2 · `fad153ec45`（2008-06-19，Tom Lane）"Rewrite the sinval messaging mechanism to reduce contention and avoid unnecessary cache resets"**：今日 sinvaladt.c 的骨架来源——读写分锁（SInvalReadLock/SInvalWriteLock）、`hasMessages` 无锁快查、按阈值摊销的清理、每次只催一人的 catchup 接力，全部出自这次重写。此前版本读队列也要抢同一把锁，DDL 风暴下失效检查本身成为瓶颈。更早的 `22fe3d4b04`（1999，Tom Lane）把最初的实现重写减半，提交信息开头是一句著名的自白："I finally understood what sinvaladt.c is doing"。
- **D3 · `141fd1b66c`（2017-10-13，Andres Freund）"Improve sys/catcache performance"**：按键数特化的 `SearchCatCacheN` 家族、miss 路径 `pg_noinline` 拆分等微架构级优化的来源。理由在 11.2.1：目录缓存查找热到值得为内联和循环展开专门改 API 形状。
- **D4 · `f868a8143a`（2018-09-07，Tom Lane）"Fix longstanding recursion hazard in sinval message processing"**：11.4.4 的递归静态缓冲与 11.4.6 的 `MarkLockClear`/`LOCKACQUIRE_ALREADY_CLEAR` 精细化同源于此——失效处理中再触发失效处理的递归曾能造成消息丢失；修复顺带引入"锁已清账"状态，成为今天跳过重复 `AcceptInvalidationMessages` 的依据。
- **D5 · `c6e0fe1f2a`（2022-08-29，David Rowley）"Improve performance of and reduce overheads of memory management"**：MemoryChunk 8 字节头 + 4bit（当时 3bit）方法 ID 虚表分发的来源，chunk 头从 16 字节减半，"由指针找上下文"变成纯位运算。后续 `0ba8b75e7e`（2024-04-07，David Rowley）把方法 ID 从 3 bit 扩到 4 bit——bump 分配器（无 per-chunk 头的极限特化）加入后码点告急，顺带把 glibc malloc 头的常见 bit 模式设为故意无效码点，强化了 I6 的误用检测。
- **D6 · `a152ebeec6`（2000，Tom Lane）"Fix problems seen in parallel regress tests when SI buffer overruns"**：relcache "就地重建保指针"策略（11.3.4）的源头——SI 队列溢出引发全量刷缓存时，持有 tupledesc/规则/触发器指针的代码曾直接崩溃；此后重建原则改为"外壳复用、关键子结构尽量保留，保不住就 elog(ERROR) 也不悬垂"。同一提交还让 catcache 拥有了自己的 TupleDesc 拷贝，砍断"刷 catcache 需要 relcache、刷 relcache 又要查 catcache"的循环依赖——缓存层级间的自举问题在失效路径上同样存在。

纵览这六笔：1999-2002 解决**正确性**（负缓存语义、重建不悬垂），2008 解决**扩展性**（多核下的锁竞争），2017-2024 解决**微架构效率**（内联、缓存行、位运算）。一套机制二十五年沿同一方向持续打磨，接口（SearchSysCache/palloc/AcceptInvalidationMessages）却几乎没变——分层与接口设计对了，优化就可以永远只动实现。

## 11.8 What-if 推演

### W1：DDL 与长查询并发——为什么查询中途表定义"不会变"？

设 backend B 正在跑一条扫 t 一小时的查询，backend A 想 `ALTER TABLE t DROP COLUMN c`。

- B 在查询开始时已持有 t 的 `AccessShareLock`（计划执行期间不放）；A 的 `AccessExclusiveLock` 与之冲突，**A 排队睡觉**。一小时内 t 的定义物理上不可能改变——不是"B 没看见变化"，而是"变化根本没发生"。B 结束事务放锁，A 才动手。这就是"锁持有期 = 定义冻结期"；
- 变体一：A 做的是**低锁级 DDL**（如 `CREATE INDEX CONCURRENTLY` 的某些阶段、`ANALYZE`，只拿不与 AccessShare 冲突的锁）。此时 A 可以在 B 查询中途提交，失效消息进入队列。B 何时看见？可能在查询中途的某个安全点（比如 B 的查询又要 open 另一张表、或收到 catchup 信号后的间隙）就消费了消息——t 的 relcache 条目被 `RelationFlushRelation` **就地重建**（B 持引用，走保指针路径），B 手里的 `Relation` 指针依然有效，`rd_att` 若未变也原样保留。B 的执行计划继续跑，最多统计信息变新了。**能低锁做的 DDL，恰恰是那些"旧描述符继续用也无害"的 DDL**——锁级别的设计与失效协议是一体两面（`relcache.c:2865-2874` 的注释正是这个论证：外层事务持有的锁级别足以排除"有害"的并发变更）；
- 变体二：B 是**空闲**连接（不持任何锁）。A 的 DDL 畅通无阻。B 的缓存过期了没关系——B 下次要用 t 必先拿锁，拿锁即结账。若 B 长期空闲不动、队列逼近水位，`SICleanupQueue` 会给它发 catchup 信号，`ProcessCatchupInterrupt` 在读命令的间隙起空事务消化消息。所以"空闲连接的缓存"最终也不会无限陈旧。

### W2：sinval 队列溢出、backend 被 reset——付出什么代价？

某 backend 落后超过 `lowbound`（队列 4096 槽装不下新消息且它挡道），被标 `resetState = true`。它下次 `SIGetDataEntries` 得到 -1，`ReceiveSharedInvalidMessages` 调用 `resetFunction = InvalidateSystemCaches`（`inval.c:916`）→ `InvalidateSystemCachesExtended`（`inval.c:785`）：目录快照作废、**全部 catcache 清空**（`ResetCatalogCaches`）、**全部 relcache 条目删除或重建**（`RelationCacheInvalidate`，连 smgr 句柄和 relmap 一起）、三类注册回调全部以"全量"参数触发（plancache 把**所有**缓存计划标失效）。

随后的第一条查询要付的账单：每张涉及的表 `RelationBuildDesc` 现拼（每张十几次 syscache 查找，各自 miss 走目录索引扫描）、每个函数/类型/操作符查找全部 miss、所有 prepared statement 重新规划。一个热连接的缓存往往有几百张表、几千条 catcache——这是一次几十到几百毫秒级的延迟尖峰，还伴随对系统目录缓冲页的集中访问（多个 backend 同时被 reset 时尤其明显，这正是 fad153ec45 之前版本的实际故障模式：级联 reset 风暴）。

设计上的缓解链条：queue 大小 4096 → 落后 2048 就催（SIG_THRESHOLD）→ 催赶 daisy chain 摊平负载 → 实在不行才 reset，且 reset 名单尽量小（只有真正挡道者）。工程启示：监控里若见 `standby`/空闲连接引发的周期性慢查询尖峰，`SharedInvalidMessageCounter` 的跳变和 reset 是排查方向之一；连接池的"长期纯空闲连接 + 高频 DDL（含 temp table 创建）"组合是这种尖峰的经典配方——temp table 的建删同样走目录、同样广播失效。

### W3：CacheMemoryContext 泄漏的典型 bug 模式

CacheMemoryContext 与进程同寿、事务中止不清理（这是设计使然：缓存必须跨事务活）。于是它是"逻辑泄漏"的第一案发现场。三种经典模式：

- **模式一：切进去忘出来。** `MemoryContextSwitchTo(CacheMemoryContext)` 后早退分支忘了切回——此后本该短命的分配（解析树、字符串）统统进了永久区。每条 SQL 漏一点，几天后 backend RSS 悄然过 GB。防御惯用法：`oldcxt = MemoryContextSwitchTo(...); ...; MemoryContextSwitchTo(oldcxt);` 尽量紧凑成对，中间不放会提前 return 的逻辑；
- **模式二：构建半途抛错。** 在 CacheMemoryContext 里逐步拼一个复杂缓存对象，中途查目录抛错——已分配的部分成为永久孤儿（I5 的违反后果）。内核的两个标准解法都在本章出现过：catcache 的**一次分配成型**（头+元组单次 palloc，要么全有要么全无）；relcache/plancache 的**先在临时上下文构建、成功后整体搬入或 `MemoryContextSetParent` 重挂**——plancache 保存计划正是"在自己的私有上下文里建好，最后一步把上下文挂到 CacheMemoryContext 名下"，错误路径上删临时上下文即可；
- **模式三：失效回调忘了释放附属结构。** 缓存条目被替换时，只 pfree 了主结构、漏了某个后加字段指向的独立分配。它不会被任何机制兜底——CacheMemoryContext 永不 reset。这类 bug 的表现是"与 DDL 频率成正比的缓慢增长"，诊断靠 `MemoryContextStats(TopMemoryContext)`（gdb 内调用）或 `pg_backend_memory_contexts` 视图看哪个上下文异常肥大，再配合 `debug_discard_caches = 1` 让重建路径高频执行、把增长放大到肉眼可见。

三种模式共同说明：MemoryContext 把泄漏从"忘了 free"转化成了"放错了家"——问题没有消失，但从不可审计（散落在每个调用点）变成了可审计（只需盯住少数长命上下文的写入路径）。

## 自测问题（8 题，附详解）

**1. initdb 之前世界上还没有任何目录表，PG 怎么创建出 pg_class 自己？运行期新 backend 启动时的对应问题又是怎么解决的？**

构建期：四张自举表在头文件里用 `CATALOG(...) BKI_BOOTSTRAP` 声明（`pg_class.h:34` 等），C 结构体编译进二进制；genbki.pl 把 pg_*.h/.dat 生成 postgres.bki；initdb 以 bootstrap 模式解释执行 BKI，按编译期结构直接堆插入，无需查目录。运行期：`formrdesc()`（`relcache.c:1889`）用同一批编译期结构硬造核心表的 relcache 条目并钉死（nailed），支撑最初的目录扫描，再分阶段（`criticalRelcachesBuilt`）升级；`pg_internal.init`（`relcache.h:25`）把建好的条目序列化到磁盘，让后续 backend 跳过整个过程（`load_relcache_init_file`，`relcache.c:6202`）。

**2. 负缓存条目解决什么问题？它为什么在 bootstrap 模式下被禁用？失效时它怎么被清除？**

缓存"某键在目录中不存在"这一事实（`SearchCatCacheMiss` 里 `CatalogCacheCreateEntry(cache, NULL, arguments, ...)` 造只含键的假元组，`catcache.c:1633-1655`），把 search_path 探测、函数重载解析等高频落空查找从"每次索引扫描"降为 O(1) 哈希命中。bootstrap 禁用（`catcache.c:1628-1631` 注释）：失效机制未运转，若先缓存"不存在"、之后对象被创建，无人能清除过期负条目。清除：失效消息携带键哈希值，`CatCacheInvalidate`（`catcache.c:643`）按哈希匹配删除，"whether positive or negative"（`catcache.c:629-631`）——CREATE 与 DROP 对失效一视同仁，这正是 26ac217173 能简化 inval.c 的原因。

**3. backend A 对表 t 做 ALTER TABLE 并提交，backend B 是在哪个精确时刻丢弃 t 的旧 relcache 条目的？**

A 提交时 `AtEOXact_Inval(true)`（`inval.c:1196`）把消息经 `SIInsertDataEntries()`（`sinvaladt.c:372`）写入共享环形队列（写数据→自旋锁内推 `maxMsgNum`→置各进程 `hasMessages`），然后才释放锁。B 不会被打断；它在下一个安全点调用 `AcceptInvalidationMessages()`（`inval.c:930`）——事务开始（`xact.c:1197`）、对任意表**获锁之后**（`lmgr.c:136`）、或 catchup 信号后的命令间隙（`sinval.c:173`）。处理时 `RelationFlushRelation`（`relcache.c:2831`）三岔口：无人用则删、有人用则就地重建保指针、本事务新建的必须重建保状态。真正"用上新结构"发生在 B 下次 `RelationIdGetRelation()`（`relcache.c:2093`）的懒重建/新建分支。

**4. 完整论证：为什么 `LockRelationOid` 拿到锁之后必须立刻 `AcceptInvalidationMessages`？`LOCKACQUIRE_ALREADY_CLEAR` 为什么是安全的优化？**

失效是拉模式，存在"B 已拿到旧描述符但尚未到达安全点"的窗口，消息队列只保证最终性。锁协议补时效性，靠两条义务：DDL 方改结构必持排他锁，且**消息入队先于放锁**（`AtEOXact_Inval` 在事务收尾释放锁之前执行）；DML 方用表必先拿锁，且**获锁后先消费消息**（`lmgr.c:134-138`）。穷举：B 先得锁则定义冻结（A 排队），A 先得锁则 B 醒来时消息必已在队列、消费后必见新结构。两种时序皆安全且无第三种。ALREADY_CLEAR：我已持同级锁且上次持锁后已清账（`MarkLockClear`），期间无人能拿到冲突的排他锁去改表 ⇒ 不可能有新的相关消息 ⇒ 跳过消费是安全的——锁持有期就是缓存有效性租约。需要显式"cleared"状态而非"持锁即可"，是因为递归场景下锁在手时外层消息可能还没吃完（`f868a8143a`）。

**5. 事务回滚时，攒下的失效消息为什么"共享队列一条不发、本地却要重放一遍"？论证两个方向各自的正确性。**

不发：其他 backend 按 MVCC 从未看见被回滚的目录行，它们的缓存自始至终正确，广播只会造成无谓的重建（甚至在极端下引发失效风暴）。本地重放：本事务内每个命令边界 `CommandEndInvalidationMessages`（`inval.c:1406`）已按**新目录**刷过自己的缓存（否则自己看不见自己的 DDL）；回滚后新目录作废，这些缓存条目成了"来自未来"的脏数据，必须再失效一次。消息语义是"键哈希 H 作废"而非"改成某值"，重放天然幂等且方向无关——失效后下次访问按回滚后的快照重读，自动回到旧世界。`CurrentCmdInvalidMsgs`（当前命令还没本地执行过的部分）直接丢弃即可（`inval.c:1185-1189`）。

**6. 走读 `SIGetDataEntries`：为什么必须"先清 hasMessages、再读 maxMsgNum"？顺序反过来会发生什么？**

写者的顺序是：写缓冲→推 maxMsgNum→置 hasMessages。读者先清标志后定界（`sinvaladt.c:501-515`）：若处理期间有新消息，写者会重新置位，读者下次必然再来。若反序（先读 max 再清标志）：写者恰在"读者取 max 之后、清标志之前"完成一整轮插入——新消息序号大于读者的界，本轮不会读到；随后读者一把清掉标志，把写者刚置的位也抹了。该 backend 从此认为"没消息"，直到下一条无关消息碰巧再置位——期间它带着过期缓存运行（I8）。开头的无锁快查同理有精妙的内存序论证（`sinvaladt.c:486-494`）：乱序窗口的危害不超过"消息晚到一点"，而需要及时性的场合（拿锁后）有锁获取做屏障。

**7. `pfree(p)` 只有一个裸指针，完整说出它找到正确释放函数和所属上下文的每一步。为什么 palloc 的指针不能 free()、malloc 的指针不能 pfree()？**

`p - 8` 读出 64 位 `hdrmask`（`GetMemoryChunkMethodID`，`mcxt.c:213`）；低 4 位是 `MemoryContextMethodID`，查 `mcxt_methods[]` 虚表（`MCXT_METHOD`，`mcxt.c:205`）得到 `free_p` 实现（如 `AllocSetFree`，`aset.c:1107`）。找上下文（`GetMemoryChunkContext`，`mcxt.c:759`）：非 external chunk 取 hdrmask 高 30 位块偏移（与 value 段借对齐共享 1 bit，读时掩掉），`chunk - offset` 即 block，block 头首字段就是 `aset` 上下文指针；external 大块直接按块头定位并校验 `freeptr == endptr`。混用为什么崩：free() 期望 malloc 自己的簿记头，拿到 palloc 块等于按垃圾簿记操作堆；pfree 一个 malloc 指针则低 4 位随机——glibc 头常见模式 0001/0010 被特意保留为 BOGUS 码点（`memutils_internal.h:124-125`），大概率干净报错，否则按错误偏移找 block 后果不可预测。偏移后的指针同理：`p-8` 处根本不是 chunk 头。

**8. 一次查询执行中 elog(ERROR) 从执行器深处抛出：内存为什么不泄漏？中止处理自己为什么不会因 OOM 卡死？catcache 的引用计数谁来归还？**

longjmp 到 `postgres.c:4573` 恢复块 → `AbortCurrentTransaction()`（`xact.c:3504`）。内存：查询期一切 palloc 都在 TopTransactionContext 子树（ExecutorState/ExprContext 都是其后代），`CleanupTransaction` 的 `AtCleanup_Memory()`（`xact.c:2016`）整树重置，无需知道谁分配的（支柱 1）。不 OOM 卡死：`AbortTransaction` 第一步 `AtAbort_Memory()`（`xact.c:1926`）切进 32KB 常驻的 `TransactionAbortContext`（`xact.c:1222-1228`），错误报告用 8KB 常驻的 ErrorContext（`mcxt.c:392-397`）——清理路径从不在可能已耗尽的上下文里分配（支柱 2）。catcache 引用：`SearchCatCacheInternal` 命中时 `refcount++` 的同时 `ResourceOwnerRememberCatCacheRef` 登记（`catcache.c:1496-1498`）；中止时 ResourceOwner 按登记逐一 Release，assert 构建下漏登记会 warning 尖叫（支柱 3）。三支柱合起来：释放的正确性来自结构，而非任何一处代码的自觉。
