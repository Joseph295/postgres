## 本章导读

前面的章节里，我们反复说"解析器查系统目录把表名变成 OID"、"执行器拿到一个 `Relation` 描述符"。这一章就来拆开这两个黑盒：

4. **内存管理 MemoryContext**：PG 用 C 写了 40 年而内存泄漏极少，靠的不是纪律，而是一套"区域分配器 + 错误恢复一键重置"的工程设计——这是读一切 PG 代码的前置知识，也是本章的压轴戏。

## 11.1 系统目录：表的定义也是表

### 11.1.1 四张核心目录表的关系

| 目录表 | 存什么 | 关键列 | 类比 |
|---|---|---|---|
| `pg_class` | 一切"有存储或像表的东西"：表、索引、视图、序列、物化视图 | `oid`、`relname`、`relnamespace`、`relfilenode`、`relkind` | Hive Metastore 的 TBLS |
| `pg_attribute` | 每个表的每一列一行 | `attrelid`（指向 pg_class.oid）、`attname`、`atttypid`、`attnum` | COLUMNS_V2 |
| `pg_type` | 每种数据类型一行（含每个表隐含的行类型） | `oid`、`typname`、`typlen`、`typinput/typoutput` | 无直接对应——PG 类型系统本身可扩展 |
| `pg_proc` | 每个函数/操作符实现一行 | `oid`、`proname`、`prorettype`、`proargtypes` | UDF 注册表 |

关系链：`pg_class` 的一行说"有张表叫 t"，`pg_attribute` 里以 `attrelid = t 的 oid` 的若干行说"它有哪些列"，每列的 `atttypid` 指向 `pg_type`，而 `pg_type` 里类型的输入输出函数 `typinput/typoutput` 又指向 `pg_proc`。**整个数据库的元数据就是一张以 OID 为外键的关系图**，而 OID（Object Identifier，一个 32 位无符号整数）就是这张图里的"指针"。

`src/include/access/transam.h:197` 定义了一条重要分界线：

```c
#define FirstNormalObjectId		16384
```

小于 16384 的 OID 留给 initdb 灌入的内建对象（手工或脚本预分配，保证所有安装实例一致），用户创建的对象从 16384 起分配。这就是为什么你在任何 PG 实例上 `pg_class` 里 `pg_class` 自己的 OID 永远是 1259。

### 11.1.2 自举问题：genbki.pl、.dat 文件与 BKI

鸡生蛋问题：要执行 `INSERT INTO pg_class ...` 得先能读懂 pg_class 的列定义，而列定义存在 pg_attribute 里，读 pg_attribute 又要先读懂 pg_attribute 自己……PG 的解法分三层：

**第一层：C 头文件就是表定义。** 每张目录表在 `src/include/catalog/` 下有一个头文件，用 `CATALOG` 宏同时生成 C 结构体和表定义元数据：

```c
/* src/include/catalog/pg_class.h:34 */
CATALOG(pg_class,1259,RelationRelationId) BKI_BOOTSTRAP BKI_ROWTYPE_OID(83,RelationRelation_Rowtype_Id) BKI_SCHEMA_MACRO
```

逐段解读：`pg_class` 是表名，`1259` 是硬编码的 OID，`RelationRelationId` 是生成的 C 宏名；`BKI_BOOTSTRAP` 标记它是**自举表**——pg_class（1259）、pg_attribute（1249，`pg_attribute.h:39`）、pg_proc（1255，`pg_proc.h:32`）、pg_type（1247，`pg_type.h:38`）四张表享此待遇：它们的结构直接编译进二进制（C 结构体 `FormData_pg_class` 等），初始化时不需要"先查目录"就能按结构读写。

**第二层：.dat 文件是初始数据。** `src/include/catalog/pg_proc.dat`、`pg_type.dat` 等 Perl 数据文件记录了全部内建函数、类型的初始行（pg_proc.dat 有几千个条目）。改内核加一个内建函数，就是在 .dat 里加一段。

**第三层：genbki.pl 编译出 BKI 脚本。** `src/backend/catalog/genbki.pl`（文件头注释见其第 4-12 行）在**构建期**读取所有 pg_*.h + pg_*.dat，生成 `postgres.bki`——一种极简的"目录汇编语言"（`create`/`insert`/`open` 等原语）。`initdb` 时，postgres 以特殊的 bootstrap 模式启动，由 `src/backend/bootstrap/bootstrap.c` 解释执行 BKI 脚本，用最原始的堆插入接口把初始目录行写进去。此时没有 SQL、没有优化器、甚至没有完整的缓存失效机制（后文 11.2 会看到 bootstrap 模式禁用负缓存正是因为这个）。

> **对照 MySQL**：8.0 的数据字典初始化用的是硬编码的 SQL + InnoDB 内部接口；PG 的方案更"自产自销"——连初始数据都是从与 C 代码同源的声明式文件生成的，保证头文件里的结构体和磁盘上的行永远一致。

### 小结

- 目录表是普通堆表，元数据关系图以 OID 为指针，16384 是内建/用户对象的分界；
- 自举靠三板斧：四张 `BKI_BOOTSTRAP` 表结构编译进二进制、.dat 文件提供初始行、genbki.pl 生成的 BKI 脚本在 initdb 时由 bootstrap 模式解释执行。

## 11.2 syscache：按键缓存目录行

每解析一条 SQL 都要查十几次目录（表名→OID、每列类型、每个操作符……）。如果每次都走"打开堆表 + 索引扫描"，解析会比执行还慢。所以 PG 在每个 backend 的本地内存里做了 **syscache**：按键哈希缓存单条目录元组。

分两层实现：

- `src/backend/utils/cache/syscache.c`：定义"有哪些 cache"。每个 cache 对应一张目录表的一个唯一索引，例如 RELOID（按 OID 查 pg_class）、RELNAMENSP（按 (relname, relnamespace) 查 pg_class）。cache 清单如今由目录头文件里的声明自动生成（`syscache.c:82` 引入生成的 `catalog/syscache_info.h`）。入口是 `SearchSysCache1/2/3/4()`（`syscache.c:221` 起），按键个数封装；
- `src/backend/utils/cache/catcache.c`：通用的"目录元组哈希缓存"机制本身。

核心查找路径 `SearchCatCacheInternal()`（`catcache.c:1425`，节选）：

```c
	/* catcache.c:1458 附近 */
	hashValue = CatalogCacheComputeHashValue(cache, nkeys, v1, v2, v3, v4);
	hashIndex = HASH_INDEX(hashValue, cache->cc_nbuckets);

	bucket = &cache->cc_bucket[hashIndex];
	dlist_foreach(iter, bucket)
	{
		ct = dlist_container(CatCTup, cache_elem, iter.cur);

		if (ct->dead)
			continue;			/* 跳过已失效但还被引用的死条目 */
		if (ct->hash_value != hashValue)
			continue;			/* 先比哈希值，快速跳过 */
		if (!CatalogCacheCompareTuple(cache, nkeys, ct->keys, arguments))
			continue;			/* 再精确比较键值 */

		/* 命中：移到桶链表头（近似 LRU，热条目靠前） */
		dlist_move_head(bucket, &ct->cache_elem);

		if (!ct->negative)
		{
			ResourceOwnerEnlarge(CurrentResourceOwner);
			ct->refcount++;		/* 引用计数 + ResourceOwner 登记，防误释放 */
			ResourceOwnerRememberCatCacheRef(CurrentResourceOwner, &ct->tuple);
			return &ct->tuple;	/* 正条目：返回缓存的元组 */
		}
		else
			return NULL;		/* 负条目：确定"不存在"，直接返回 NULL */
	}

	return SearchCatCacheMiss(cache, nkeys, hashValue, hashIndex, v1, v2, v3, v4);
```

逐行要点：

- 桶内链表按"最近命中移到头部"排序，是零成本的近似 LRU；
- 命中的正条目要 `refcount++` 并向 ResourceOwner 登记——这样即使后续代码抛错，事务中止清理时也能自动归还引用（与 11.5 的内存管理是同一套"错误恢复不靠人"哲学）；
- **负条目（negative entry）**：`ct->negative` 为真表示"这个键在目录里**不存在**"。未命中时由 `SearchCatCacheMiss()`（`catcache.c:1533`）真正扫目录表，如果扫不到，就制造一个只有键、其余全空的假元组插入缓存（`catcache.c:1632-1654`）。

为什么负缓存重要？想想 `search_path`：你执行 `SELECT * FROM t`，PG 要按 search_path 顺序逐个 schema 试探 `(t, schema_oid)` 是否存在——大多数试探的答案是"不存在"。没有负缓存，每条 SQL 都会对不存在的组合反复做索引扫描。负缓存把"不存在"这个事实也缓存住，代价是失效机制必须同样覆盖它（否则 CREATE TABLE 之后负条目会让新表"隐身"——所以 11.4 的失效消息按哈希值失效，正负条目一视同仁）。`catcache.c:1628` 的注释还解释了 bootstrap 模式不建负条目：那时失效机制尚未运转，负条目一旦过期无人清除。

### 小结

- syscache = 每 backend 本地的、按唯一索引键组织的目录行哈希缓存；syscache.c 管"有哪些缓存"，catcache.c 管通用机制；
- 命中路径极短：哈希→桶链表→比键；引用计数由 ResourceOwner 兜底；
- 负缓存把"查无此物"也缓存，专治 search_path 这类高频 miss 场景，其正确性依赖失效机制。

## 11.3 relcache：表的"全息描述符"

syscache 缓存的是**一行**目录元组，但执行器打开一张表需要的是**聚合视图**：列描述、索引列表、触发器、分区信息、物理文件号……把这些每次都从十几张目录表现拼出来太贵，于是有了 relcache（`src/backend/utils/cache/relcache.c`）：以表 OID 为键，缓存 `RelationData` 结构。你在内核代码里到处见到的 `Relation rel` 参数就是指向它的指针。

### 11.3.1 RelationData 逐要点

结构定义在 `src/include/utils/rel.h:55`，字段可分为六组（节选自源码，注释为原文）：

```c
typedef struct RelationData
{
	RelFileLocator rd_locator;	/* relation physical identifier */
	SMgrRelation rd_smgr;		/* cached file handle, or NULL */
	int			rd_refcnt;		/* reference count */
	ProcNumber	rd_backend;		/* owning backend's proc number, if temp rel */
	bool		rd_islocaltemp; /* rel is a temp rel of this session */
	bool		rd_isnailed;	/* rel is nailed in cache */
	bool		rd_isvalid;		/* relcache entry is valid */
	...
	SubTransactionId rd_createSubid;	/* rel was created in current xact */
	...
	Form_pg_class rd_rel;		/* RELATION tuple */
	TupleDesc	rd_att;			/* tuple descriptor */
	Oid			rd_id;			/* relation's object id */
	LockInfoData rd_lockInfo;	/* lock mgr's info for locking relation */
	RuleLock   *rd_rules;		/* rewrite rules */
	TriggerDesc *trigdesc;		/* Trigger info, or NULL if rel has none */
	...
	PartitionKey rd_partkey;	/* partition key, or NULL */
	PartitionDesc rd_partdesc;	/* partition descriptor, or NULL */
```

- **物理定位**：`rd_locator`（表空间/数据库/文件号三元组，决定磁盘路径）+ `rd_smgr`（缓存的存储管理器文件句柄）。注意逻辑身份 `rd_id`（OID）与物理身份 `rd_locator.relNumber`（relfilenode）是两回事：TRUNCATE、VACUUM FULL 会换物理文件而 OID 不变；
- **生命周期管理**：`rd_refcnt` 引用计数（谁 open 谁 close）；`rd_isvalid` 表示条目是否仍然新鲜——**失效消息到达时并不删除条目，只是把它标脏**，下次使用时再重建；`rd_isnailed` 标记"钉死"的条目：pg_class、pg_attribute 等自举表的 relcache 条目不能依赖"查目录重建"（会无限递归），启动早期由 `formrdesc()`（`relcache.c:1889`）用编译进二进制的结构硬造出来，永不驱逐;
- **事务感知**：`rd_createSubid` 等四个 SubTransactionId 字段记录"这张表/它的物理文件是不是本事务新建的"——`RelationNeedsWAL()` 等优化（本事务新建的表 COPY 可以跳 WAL）全靠它们精确维护，rel.h:69 起有一大段注释专讲其微妙之处；
- **逻辑描述**：`rd_rel`（pg_class 行的拷贝）、`rd_att`（TupleDesc，列描述数组，堆元组的编解码都要它）；
- **行为附件**：`rd_rules`（视图/规则）、`trigdesc`（触发器）、`rd_rsdesc`（行级安全策略）、外键列表、索引列表；
- **分区**：`rd_partkey` / `rd_partdesc` 及其私有内存上下文——注意模式：**大的可变子结构各带一个自己的 MemoryContext**（`rd_partkeycxt`、`rd_pdcxt`……），重建条目时整块释放，不必逐字段 pfree（11.5 的思想渗透到每个角落）。

### 11.3.2 RelationIdGetRelation：拿描述符的唯一正门

`relcache.c:2093`：

```c
Relation
RelationIdGetRelation(Oid relationId)
{
	Relation	rd;

	/* 先查本地哈希表 */
	RelationIdCacheLookup(relationId, rd);

	if (RelationIsValid(rd))
	{
		/* 已被本事务 DROP 的条目返回 NULL */
		if (rd->rd_droppedSubid != InvalidSubTransactionId)
			return NULL;

		RelationIncrementReferenceCount(rd);
		/* 条目还在但被失效消息标脏了？当场重建 */
		if (!rd->rd_isvalid)
			RelationRebuildRelation(rd);
		return rd;
	}

	/* 缓存里没有：从目录构建一个新条目并插入缓存 */
	rd = RelationBuildDesc(relationId, true);
	if (RelationIsValid(rd))
		RelationIncrementReferenceCount(rd);
	return rd;
}
```

三个分支道尽了 relcache 的运转方式：**命中即用**（加引用计数）、**脏了懒重建**（失效只标记，使用时才重建——避免失效风暴时做无用功）、**没有就构建**。构建函数 `RelationBuildDesc()`（`relcache.c:1059`）就是"拼装流水线"：用 syscache 取 pg_class 行 → 建 TupleDesc（逐列查 pg_attribute）→ 按 relkind 装配规则、触发器、分区信息 → 初始化物理定位与锁信息 → 挂进哈希表。它本身大量调用 syscache，这就是两层缓存的层叠关系。

上层代码一般不直接调它，而是走 `relation_open(oid, lockmode)`：**先拿锁，再拿描述符**——为什么必须这个顺序，11.4.4 揭晓。

### 11.3.3 relcache init file：启动加速

新 backend 启动时，连"查 pg_class"都需要 pg_class 自己的 relcache 条目和一批系统索引的条目——又是自举。除了 `formrdesc()` 硬造，PG 还有个工程优化：把构建好的系统表 relcache 条目序列化到数据目录下的 **`pg_internal.init`** 文件（文件名定义在 `src/include/utils/relcache.h:25`）。

- `load_relcache_init_file()`（`relcache.c:6202`）：启动时尝试直接反序列化，跳过昂贵的目录扫描；
- `write_relcache_init_file()`（`relcache.c:6622`）：条目齐全后写出（先写临时文件再 rename，经典原子写法，见 6647 附近对 `%s.%d` 临时名的处理）；
- 对系统表做过 DDL（如 REINDEX 系统索引）时必须让所有实例的 init file 作废——所以 11.4 的 `AtEOXact_Inval()` 里有 `RelationCacheInitFilePreInvalidate/PostInvalidate` 的前后夹击（先删文件再广播消息，防止别的 backend 在窗口期用旧文件），这是失效机制里最阴险的正确性细节之一。

### 小结

- relcache 以 OID 为键缓存 `RelationData`——表的全息描述符，物理定位/生命周期/事务感知/逻辑描述/附件/分区六组字段；
- `RelationIdGetRelation()` 三分支：命中、脏则懒重建、缺则 `RelationBuildDesc()` 现拼（拼的过程消费 syscache）；
- 自举条目靠 `formrdesc()` 硬造 + nailed 钉死；启动加速靠 `pg_internal.init`，其失效需要与消息广播精密编排。

## 11.4 缓存失效：多进程下的一致性

现在到了真正的难题。缓存是**每 backend 私有**的，而目录是共享的。backend A `ALTER TABLE t ADD COLUMN` 提交后，backend B 缓存里的 t 还是旧结构——如果 B 拿着旧 TupleDesc 去解读新格式的元组，就是数据损坏。PG 的解法是三件套：**失效消息 + 共享内存环形队列 + 锁**。

### 11.4.1 发送端：inval.c 攒消息，提交时广播

`src/backend/utils/cache/inval.c` 是发送端。DDL 修改目录元组时，堆更新路径会调 `CacheInvalidateHeapTuple()`（`inval.c:1568`），它把失效消息（"catcache X 中哈希值为 H 的条目作废"或"relcache 中 OID=R 的条目作废"）**先攒在事务本地链表里**，而不是立即广播——因为事务可能回滚，回滚了就什么都不用发。

两个时机处理这些消息：

- **命令边界**（`CommandEndInvalidationMessages()`，`inval.c:1406`）：本事务内的下一条命令必须看见自己刚做的 DDL，所以每次 CommandCounterIncrement 时把当前命令的消息**在本进程内**执行一遍（只刷自己的缓存）；
- **事务提交**（`AtEOXact_Inval(true)`，`inval.c:1196`）：

```c
	if (isCommit)
	{
		/* 若涉及系统表变更，先删掉 init file（见 11.3.3） */
		if (transInvalInfo->ii.RelcacheInitFileInval)
			RelationCacheInitFilePreInvalidate();

		/* 把之前各命令攒下的消息与当前命令的合并 */
		AppendInvalidationMessages(&transInvalInfo->PriorCmdInvalidMsgs,
								   &transInvalInfo->ii.CurrentCmdInvalidMsgs);

		/* ★ 向所有 backend 广播 */
		ProcessInvalidationMessagesMulti(&transInvalInfo->PriorCmdInvalidMsgs,
										 SendSharedInvalidMessages);

		if (transInvalInfo->ii.RelcacheInitFileInval)
			RelationCacheInitFilePostInvalidate();
	}
	else
	{
		/* 回滚：只需在本进程内重放消息，把自己被本事务污染的缓存刷掉 */
		ProcessInvalidationMessages(&transInvalInfo->PriorCmdInvalidMsgs,
									LocalExecuteInvalidationMessage);
	}
```

注意对称美：**提交 = 广播给别人**（别人该忘掉旧版本了），**回滚 = 自己重放**（自己在 CommandEnd 时已经按新目录刷过缓存，现在新目录作废了，得把自己的缓存也刷回去）。此外，提交时这批消息还会由 `xactGetCommittedInvalidationMessages()` 附在 commit WAL 记录里——备库 redo 时据此让 hot standby 上的 backend 缓存失效。

### 11.4.2 传输层：sinvaladt.c 的共享内存环形队列

`SendSharedInvalidMessages()`（`src/backend/storage/ipc/sinval.c:47`）落到 `src/backend/storage/ipc/sinvaladt.c`——一个固定大小的共享内存环形缓冲：

```c
/* sinvaladt.c:130 */
#define MAXNUMMESSAGES 4096      /* 环形缓冲容量，2 的幂 */
```

写入端 `SIInsertDataEntries()`（`sinvaladt.c:372`，节选）：

```c
	while (n > 0)
	{
		int  nthistime = Min(n, WRITE_QUANTUM);   /* 分批写，别长时间持锁 */
		...
		LWLockAcquire(SInvalWriteLock, LW_EXCLUSIVE);

		for (;;)
		{
			numMsgs = segP->maxMsgNum - segP->minMsgNum;
			if (numMsgs + nthistime > MAXNUMMESSAGES ||
				numMsgs >= segP->nextThreshold)
				SICleanupQueue(true, nthistime);   /* 满了：清理，必要时 reset 落后者 */
			else
				break;
		}

		/* 写入环形缓冲：下标 = 消息序号 mod 容量 */
		max = segP->maxMsgNum;
		while (nthistime-- > 0)
		{
			segP->buffer[max % MAXNUMMESSAGES] = *data++;
			max++;
		}
```

这是标准的"单调递增序号 + 取模"环形队列（Kafka 位点既视感）：每个 backend 在共享数组里有自己的 `nextMsgNum` 消费位点。关键设计是**慢消费者问题**的处理——队列满时不能无限等：`SICleanupQueue()`（`sinvaladt.c:579`）把落后太多的 backend 直接标记 `resetState = true`（"你的位点作废了"），并给太落后的进程发 `PROCSIG_CATCHUP_INTERRUPT` 信号催它消费（`sinvaladt.c:676`）。被 reset 的 backend 读消息时会收到 -1，只能执行核弹级的 `InvalidateSystemCaches()`（`inval.c:916`）——**扔掉全部缓存重来**。宁可某个空闲连接偶尔全量重建缓存，也不让 DDL 被慢消费者阻塞。

读取端 `SIGetDataEntries()`（`sinvaladt.c:475`）先做无锁快速检查（`stateP->hasMessages` 标志位），没消息直接返回——保证"没有 DDL 发生时，检查失效几乎零成本"，这是它能被高频调用的前提。

### 11.4.3 接收端：什么时候处理失效？

消息躺在共享队列里，其他 backend **不会被打断去处理**（收到 catchup 信号除外）。真正的消费入口是 `AcceptInvalidationMessages()`（`inval.c:930`）：

```c
void
AcceptInvalidationMessages(void)
{
	ReceiveSharedInvalidMessages(LocalExecuteInvalidationMessage,
								 InvalidateSystemCaches);
	...
}
```

第一个回调逐条应用消息（标脏对应 catcache/relcache 条目），第二个回调处理被 reset 的情形（全量作废）。它在哪些点被调用？

1. **事务开始**：`AtStart_Cache()`（`src/backend/access/transam/xact.c:1197`）就一行——`AcceptInvalidationMessages()`。保证每个事务开局看到的目录不旧于事务开始时刻；
2. **拿表锁时**：`LockRelationOid()`（`src/backend/storage/lmgr/lmgr.c:107`，调用点在 136 行）——下一节细讲，这是正确性的支柱；
3. **收到 catchup 信号后**：`HandleCatchupInterrupt()`（`sinval.c:154`）在信号处理器里只置标志，真正的处理挪到下一个 `CHECK_FOR_INTERRUPTS()` 安全点由 `ProcessCatchupInterrupt()`（`sinval.c:173`）完成——信号处理器里不能碰缓存这种复杂数据结构（见下一篇文档 12.3 对 CFI 机制的展开）。

注意这个模型的含义：**失效处理是"拉"模式，且只发生在明确定义的安全点**。一个正在执行长查询、既不拿新锁也不开新事务的 backend，可以长时间不消费消息——这不出错吗？

### 11.4.4 失效与锁的配合：DDL 为什么必须拿锁

只靠消息队列有个致命时序窗口：

```
backend B: 查缓存得到表 t 的旧描述符 ──────────────► 用旧描述符读 t
                              ▲
backend A:        ALTER TABLE t 提交，广播失效消息
```

B 在"查缓存"和"使用"之间没有任何机会消费消息。消息队列只保证**最终**知道，不保证**及时**。补上这个洞的是锁，`lmgr.c:107` 的 `LockRelationOid()`：

```c
void
LockRelationOid(Oid relid, LOCKMODE lockmode)
{
	LOCKTAG		tag;
	LOCALLOCK  *locallock;
	LockAcquireResult res;

	SetLocktagRelationOid(&tag, relid);

	res = LockAcquire(&tag, lockmode, false, false);

	/*
	 * Now that we have the lock, check for invalidation messages, so that we
	 * will update or flush any stale relcache entry before we try to use it.
	 * ...（lmgr.c:130 起的原注释）
	 */
	if (res != LOCKACQUIRE_ALREADY_CLEAR)
	{
		AcceptInvalidationMessages();
		MarkLockClear(locallock);
	}
}
```

配合双方的义务，形成一个滴水不漏的协议：

- **DDL 方义务**：改表结构必须先拿 `AccessExclusiveLock`（与一切冲突）。这意味着它必须**等所有正在使用 t 的 backend 释放锁**才能动手；提交释放锁之前，失效消息已广播完毕（`AtEOXact_Inval` 在锁释放之前执行）；
- **DML 方义务**：使用表之前必须 `relation_open(oid, 某种锁)` → `LockRelationOid()` → **拿到锁后立刻 `AcceptInvalidationMessages()`**。

于是只有两种时序：DML 先拿到锁——DDL 等着，DML 用旧结构没问题（DDL 还没发生）；DDL 先拿到锁——DML 等着，等 DDL 提交释放锁后 DML 才获得锁，而**获锁后的第一件事就是消费失效消息**，旧缓存条目被标脏，`RelationIdGetRelation()` 走"懒重建"分支拿到新结构。时序窗口被锁焊死了。

代码里还有个精致的优化：`LOCKACQUIRE_ALREADY_CLEAR`——如果本进程已持有同级别锁且此前已消费过消息（`MarkLockClear`），说明从那时起没人可能改过这张表（改需要排他锁，而我一直持锁），可以跳过 `AcceptInvalidationMessages()`。锁的持有期成了"缓存有效性的租约"。

> **这回答了指南第 12 章的悬念**："为什么 PG 的 DDL 是事务性的？"因为目录修改走 MVCC（可回滚）、失效消息攒到提交才广播（回滚就地销毁）、并发安全由锁兜底。MySQL 8.0 之前 DDL 不能回滚，正是因为老数据字典（.frm 文件）不在事务机制之内。

### 11.4.5 plancache 的失效联动

缓存的计划树引用了表结构与函数——它们同样会过期。`src/backend/utils/cache/plancache.c` 不自己发明机制，而是**挂接**到上面这套：初始化时（`plancache.c:150-157`）注册回调：

```c
	CacheRegisterRelcacheCallback(PlanCacheRelCallback, (Datum) 0);
	CacheRegisterSyscacheCallback(PROCOID, PlanCacheObjectCallback, (Datum) 0);
	CacheRegisterSyscacheCallback(TYPEOID, PlanCacheObjectCallback, (Datum) 0);
	...
```

当 relcache 收到某表的失效消息时，`PlanCacheRelCallback()`（`plancache.c:2126`）遍历本 backend 的所有缓存计划，把**依赖该表**的计划标记为 invalid；下次执行 prepared statement 时发现计划失效，就从保存的解析树重新规划。函数（PROCOID）、类型（TYPEOID）的 syscache 失效同理。这就是"ALTER TABLE 后 prepared statement 不会用旧计划崩掉"的全部秘密——失效是一棵传播树：目录元组变更 → catcache/relcache 标脏 → 注册的回调把 plancache 等下游一并标脏。

### 小结

- 发送端事务内攒消息：命令边界本地重放（自己看见自己），提交时广播 + 附到 commit WAL（备库联动），回滚本地反向清理;
- 传输层是 4096 槽的共享内存环形队列，每 backend 一个消费位点；慢消费者被 reset（全量刷缓存）+ catchup 信号催赶，DDL 永不被拖死；
- 接收端拉模式，安全点：事务开始、**获锁后**、catchup 信号的 CFI 处理点；
- 锁把"广播了但没消费"的窗口焊死：DDL 拿排他锁等旧使用者退场，新使用者获锁后先消费消息再用缓存；
- plancache 通过注册回调挂在 relcache/syscache 失效上，形成传播树。

## 11.5 内存管理：MemoryContext 与 AllocSet

### 11.5.1 为什么不是裸 malloc

数据库 backend 的 C 代码有两大死敌：**泄漏**（长连接进程，漏一点积少成多）和**错误路径**（任何一步都可能 `elog(ERROR)` 长跳转走人，谁来 free 已分配的内存？）。PG 的答案是教科书级的**区域分配器（region/arena allocator）**：

- 所有分配都发生在某个 **MemoryContext** 里；`palloc()` 隐式使用全局变量 `CurrentMemoryContext`；
- 上下文构成一棵树，**删除/重置父节点会级联处理全部子孙**；
- 于是"释放"的主要形式不是 pfree 单个对象，而是**整棵子树一键蒸发**。

机制框架在 `src/backend/utils/mmgr/mcxt.c`（上下文树、palloc/pfree 分发），默认实现 AllocSet 在 `aset.c`（另有 generation.c、slab.c、bump.c 等特化实现，接口相同）。设计文档 `src/backend/utils/mmgr/README` 值得通读。

### 11.5.2 上下文树的骨干层级

`mcxt.c:167-171` 定义了几个全局上下文指针：

```c
MemoryContext TopMemoryContext = NULL;
MemoryContext ErrorContext = NULL;
MemoryContext CacheMemoryContext = NULL;
MemoryContext MessageContext = NULL;
```

典型 backend 的树（括号内为创建位置）：

```
TopMemoryContext                    进程根，与进程共存亡（mcxt.c:369）
├── ErrorContext                    错误处理专用，预留 8KB 保命钱——OOM 时报错也要有内存可用（mcxt.c:380 起注释）
├── CacheMemoryContext              relcache/syscache/plancache 的家（catcache.c:726 CreateCacheMemoryContext）
│                                   ——缓存长命，所以住长命上下文，靠 11.4 的失效机制而非生命周期管理
├── MessageContext                  当前客户端命令的解析树/查询树（postgres.c:4531 创建，每条命令开头重置）
├── TopTransactionContext           事务级数据（xact.c:1238 创建），提交/中止时删除
│   └── CurTransactionContext       当前(子)事务的数据（xact.c:1301）
│        └── ExecutorState          ★ per-query：一次执行的所有状态（execUtils.c:100）
│             └── ExprContext       ★ per-tuple：表达式求值的临时内存（execUtils.c:265）
└── TransactionAbortContext         中止处理专用的救生艇（xact.c:1225）
```

两个星号层级是性能与正确性的关键惯用法：执行器处理**每一行**时在 per-tuple 上下文里随便 palloc（detoast 大字段、函数临时结果……），处理完这行调一次 `MemoryContextReset()`（`mcxt.c:406`）整体清空——**内层循环完全不需要 pfree**，既不会漏也不用逐个释放的开销。Reset 对 AllocSet 来说只是把块链表还给 malloc/保留 keeper 块，是 O(块数) 而非 O(分配次数)。

`MemoryContextSwitchTo()`（`src/include/utils/palloc.h:138`，就是两行的 inline：换全局变量，返回旧值）是切换当前上下文的唯一手段，PG 代码里"switch 进去分配、switch 回来"的配对无处不在——看懂它，执行器代码的可读性瞬间提升。

### 11.5.3 AllocSet：freelist 分级 + 块指数增长

一个 AllocSet 上下文向 OS（malloc）批发**块（block）**，向调用者零售**chunk**。两条核心策略：

**策略一：小 chunk 按 2 的幂分级，空闲链表复用。** `aset.c:83-85`：

```c
#define ALLOC_MINBITS		3	/* smallest chunk size is 8 bytes */
#define ALLOCSET_NUM_FREELISTS	11
#define ALLOC_CHUNK_LIMIT	(1 << (ALLOCSET_NUM_FREELISTS-1+ALLOC_MINBITS))
```

即 8、16、32、…、8192 字节共 11 级 freelist；超过 8KB 的请求单独整块 malloc（`AllocSetAllocLarge()`，`aset.c:735`）。请求尺寸到级别的换算 `AllocSetFreeIndex()`（`aset.c:277`）用 CPU 的前导零指令（`pg_leftmost_one_pos32`）算 ceil(log2)，无循环。分配主路径 `AllocSetAlloc()`（`aset.c:1012`，节选）：

```c
	if (size > set->allocChunkLimit)
		return AllocSetAllocLarge(context, size, flags);   /* 大请求：独立块 */

	fidx = AllocSetFreeIndex(size);        /* 尺寸 → freelist 级别 */
	chunk = set->freelist[fidx];
	if (chunk != NULL)
	{
		/* 命中空闲链表：弹出即用，快到只有几条指令 */
		set->freelist[fidx] = link->next;
		...
		return MemoryChunkGetPointer(chunk);
	}
	...
	/* freelist 空：尝试从当前活跃块尾部切一片；不够则要新块 */
	return AllocSetAllocFromNewBlock(context, size, flags, fidx);
```

pfree 掉的 chunk 不还给 OS，而是挂回对应级别的 freelist，同尺寸的下次分配 O(1) 复用。向上取整到 2 的幂平均浪费约 25% 空间——用空间换"无碎片合并逻辑、无相邻块管理"的极简与快速。

**策略二：块尺寸指数增长。** `AllocSetAllocFromNewBlock()`（`aset.c:861`）要新块前，先把旧活跃块的尾巴切碎挂到 freelist（一字不浪费，`aset.c:887` 起的循环），然后：

```c
	/* aset.c:929 附近 */
	blksize = set->nextBlockSize;
	set->nextBlockSize <<= 1;              /* 下一块翻倍 */
	if (set->nextBlockSize > set->maxBlockSize)
		set->nextBlockSize = set->maxBlockSize;   /* 封顶（默认 8MB） */
```

标准参数 `ALLOCSET_DEFAULT_SIZES` 下：首块 8KB，之后 16KB、32KB…封顶 8MB。小上下文（绝大多数）只占一个 keeper 块，大上下文 malloc 次数是 O(log n)。这跟你熟悉的动态数组扩容/Arrow 内存池是同一招。上下文创建本身也被优化到极致：`AllocSetContextCreateInternal()`（`aset.c:347`）维护一个**整个上下文的 freelist**——被删除的标准尺寸上下文头连同 keeper 块整体回收复用，因为 PG 每查询要建/毁大量上下文。

### 11.5.4 palloc/pfree 的 chunk header 定位技巧

`pfree(p)` 只有一个裸指针，怎么知道 p 属于哪个上下文、该用哪个实现的 free？看 `mcxt.c:1619`：

```c
void
pfree(void *pointer)
{
	...
	MCXT_METHOD(pointer, free_p) (pointer);
	...
}
```

`MCXT_METHOD` 的原理：**每个 chunk 的用户数据前面紧贴一个 8 字节的 `MemoryChunk` 头**，头里用 3 个 bit 编码了所属分配器类型（MCTX_ASET_ID / MCTX_GENERATION_ID / …），其余 bit 编码"chunk 到所在 block 起点的偏移"或外部块标记。于是：

- `p - 8` 读出头 → 3 bit 查全局方法表 → 分发到 `AllocSetFree()`（`aset.c:1107`）等具体实现——**这就是 C 语言手搓的虚函数表，且对象头只占 3 个 bit**；
- 由偏移量找到 block，block 头里存着 `aset` 指针 → 找到上下文。`GetMemoryChunkContext()`（`mcxt.c:759`）走的就是这条链。

因此 palloc 出来的指针**绝不能用 free() 释放**，反之亦然；也不能对指针做偏移后 pfree。`palloc()` 本体（`mcxt.c:1390`）快到只有：取 `CurrentMemoryContext` → 清 isReset 标志 → 一次虚调用 `context->methods->alloc`。

### 11.5.5 错误恢复：sigsetjmp + 上下文重置 = 不可能泄漏

这是整套设计的点睛之笔，也是 PG 工程哲学最浓缩的地方。C 没有异常和 RAII，PG 用 `elog(ERROR)` → `siglongjmp` 模拟异常。问题是：长跳转飞过的那几十层栈帧里 palloc 的内存、拿着的锁、打开的文件怎么办？

答案在 `PostgresMain()` 的顶层恢复块（`postgres.c:4573`，节选）：

```c
	if (sigsetjmp(local_sigjmp_buf, 1) != 0)
	{
		/* 任何深度的 elog(ERROR) 都会跳到这里 */
		error_context_stack = NULL;
		HOLD_INTERRUPTS();                  /* 清理期间屏蔽新中断 */
		disable_all_timeouts(false);
		QueryCancelPending = false;
		...
		EmitErrorReport();                  /* 向客户端/日志报告错误（用 ErrorContext 的预留内存） */
		debug_query_string = NULL;

		AbortCurrentTransaction();          /* ★ 一切清理的总开关 */
		...
		MemoryContextSwitchTo(MessageContext);   /* 回到安全的上下文 */
		FlushErrorState();                  /* 清空 ErrorContext */
```

`AbortCurrentTransaction()`（`xact.c:3504`）→ `AbortTransaction()`（`xact.c:2855`）的清理序列（节选自源码顺序）：

```c
	HOLD_INTERRUPTS();
	AtAbort_Memory();            /* 先切到 TransactionAbortContext——救生艇 */
	AtAbort_ResourceOwner();
	LWLockReleaseAll();          /* 释放所有轻量锁（必须最早，后续清理还要拿锁） */
	pgstat_report_wait_end();
	UnlockBuffers();
	XLogResetInsertion();
	LockErrorCleanup();
	...                          /* 其后：写 abort 记录、ResourceOwner 释放
	                                buffer pin/催还 catcache 引用/关文件、释放常规锁 */
```

随后的 `CleanupTransaction()`（`xact.c:3062`）里 `AtCleanup_Memory()` 把 **TopTransactionContext 整棵删除**——事务里不管哪层代码 palloc 过什么，无论错误发生在哪一行，全部随树蒸发。三个支柱缺一不可：

1. **内存按生命周期归池**：事务内分配都在 TopTransactionContext 子树 → 删树即回收，不需要知道"谁分配了什么"；
2. **救生艇上下文**：`AtAbort_Memory()`（`xact.c:1926`）先切进 `TransactionAbortContext`（`xact.c:1225` 创建，32KB 预留）——**中止处理本身需要内存**，而当前事务上下文可能正是因 OOM 而崩的，绝不能再在里面分配；同理 `ErrorContext` 为错误报告预留了保命内存；
3. **非内存资源全部登记在 ResourceOwner**：buffer pin、catcache/relcache 引用（11.2 里看到的 `ResourceOwnerRememberCatCacheRef`）、文件句柄、DSM 段……中止时按登记逐一释放，并在 assert 构建下对"忘了释放"的资源报 warning——泄漏在开发期就会尖叫。

> 对照大数据世界：这就是 Spark task 失败时整个 TaskMemoryManager 回收、Arrow 的 per-batch allocator 的思路——**把"正确释放"从每处代码的义务，变成结构的必然**。PG 在 1990 年代就把它做成了 C 代码的全局纪律。

### 11.5.6 与手工 malloc 管理和 RAII 的对比讨论

| 维度 | 手工 malloc/free | C++ RAII / Rust 所有权 | PG MemoryContext |
|---|---|---|---|
| 释放的责任方 | 每个调用点（人肉配对） | 编译器按作用域插入析构 | 上下文生命周期的所有者（少数几处框架代码） |
| 错误路径安全 | 极难：每个 return/goto 都要补 free | 栈展开自动析构，天然安全 | longjmp + 删上下文树，天然安全 |
| 单次分配开销 | malloc：全局锁/bins 查找 | 同 malloc（或自定义 allocator） | freelist 命中约十条指令，无锁（进程私有） |
| 批量释放 | O(n) 逐个 free | O(n) 逐个析构 | O(blocks) 整树 reset/delete |
| 生命周期表达 | 无（靠注释） | 绑定作用域/所有者，编译期检查 | 绑定上下文树，**运行期结构**，可跨函数、可动态重挂 |
| 典型风险 | 泄漏、double-free、悬垂指针 | 循环引用（shared_ptr）、析构顺序 | 分配到错误的上下文（活得太短→悬垂；太长→"逻辑泄漏"） |

几点值得展开的讨论：

- **PG 方案对 RAII 的优势**在于生命周期与**语义阶段**对齐而非与词法作用域对齐：一个 TupleDesc 从解析活到执行结束，横跨几十个函数，RAII 需要把所有权层层 move 出去，而 PG 只需"分配在 per-query 上下文"一句话。批量蒸发也远快于逐对象析构——这对"每行都分配"的数据库负载是数量级差异；
- **代价**是失去了编译期检查：把本该 per-tuple 的东西分配进 CacheMemoryContext，就成了不断增长的"逻辑泄漏"（PG 自己诊断这类问题靠 `MemoryContextStats()` 打印整棵树的占用，以及 `debug_discard_caches`（见 `inval.c:942` 起注释）这类压测选项）；反过来把长命对象放短命上下文则得到悬垂指针。所以内核代码评审最常问的问题就是"这块内存活在哪个 context？"；
- **它不是 GC**：无追踪、无停顿、确定性释放；也**不是万能**——共享内存里的对象（锁表、缓冲池）不归 MemoryContext 管，跨进程的动态内存用 DSA（见下一篇文档）另起一套；
- MySQL/InnoDB 的 `mem_heap_t` 是同思想的近亲，但没有做成全内核统一的树形层级 + 错误恢复协议；PostgreSQL 把它升格为**所有代码共同遵守的资源模型**（内存归 context、其他资源归 ResourceOwner），这才是"防泄漏"的完整答案。

### 小结

- MemoryContext = 树形区域分配器：palloc 进当前上下文，释放主要靠整树 reset/delete；骨干层级 TopMemoryContext / CacheMemoryContext / MessageContext / TopTransactionContext / ExecutorState / ExprContext 各对应一种生命周期；
- AllocSet：11 级 2 的幂 freelist（8B–8KB）+ 超限独立块 + 块尺寸 8KB 起指数翻倍封顶 8MB；chunk 头 8 字节内嵌 3 bit 分配器 ID 实现 pfree 的"无上下文参数"分发；
- 错误恢复：sigsetjmp 顶层恢复块 → AbortCurrentTransaction →（救生艇上下文里）释放锁与 ResourceOwner 登记的资源 → 删除事务上下文树。内存安全来自结构而非纪律；
- 与 RAII 相比：换掉编译期检查，换来阶段对齐的生命周期表达与批量释放性能。

## 自测问题

**1. initdb 之前世界上还没有任何目录表，PG 怎么创建出 pg_class 自己？**

四张自举表（pg_class/pg_attribute/pg_proc/pg_type）在头文件里用 `CATALOG(...) BKI_BOOTSTRAP` 声明（如 `pg_class.h:34`），其 C 结构体编译进二进制；genbki.pl 在构建期把 pg_*.h/.dat 生成 postgres.bki 脚本；initdb 以 bootstrap 模式启动 postgres 解释执行 BKI，用原始堆接口直接写入初始行——此时按编译期结构解读元组，不需要查目录。relcache 侧配套 `formrdesc()`（relcache.c:1889）硬造这几张表的描述符并钉死（nailed）。

**2. 负缓存条目解决什么问题？它为什么在 bootstrap 模式下被禁用？**

它缓存"某键在目录中不存在"这一事实（catcache.c:1632 起用只含键的假元组表示），避免 search_path 探测等高频"查无此物"场景反复扫描目录索引。bootstrap 模式禁用（catcache.c:1628 注释）是因为当时失效机制未运转：若先缓存"不存在"、之后该对象被创建，无人能清除这条过期负条目。

**3. backend A 对表 t 做 ALTER TABLE 并提交，backend B 是在哪个精确时刻丢弃 t 的旧 relcache 条目的？**

A 提交时 `AtEOXact_Inval(true)`（inval.c:1196）把消息经 `SIInsertDataEntries()` 写入共享环形队列（sinvaladt.c:372）。B 不会立即处理；它在下一个安全点调用 `AcceptInvalidationMessages()`（inval.c:930）——事务开始（xact.c:1197）、或对任意表**获锁之后**（lmgr.c:136）、或收到 catchup 信号后的 CHECK_FOR_INTERRUPTS。处理时也只是把条目标 `rd_isvalid = false`；真正重建发生在 B 下次 `RelationIdGetRelation()`（relcache.c:2093）用到 t 时的懒重建分支。

**4. 为什么"DDL 必须拿 AccessExclusiveLock"是失效机制正确性的一部分，而不仅是并发礼貌？**

因为失效是拉模式，存在"B 已拿到旧描述符但尚未消费消息"的窗口。锁协议封死它：B 使用 t 前必须持有 t 的锁；A 的排他锁要等 B 释放才能拿到，故 A 改结构时无人在用旧描述符；B 之后再获锁时，`LockRelationOid()` 保证**获锁后先 `AcceptInvalidationMessages()`** 再使用缓存，必然看到新结构。`LOCKACQUIRE_ALREADY_CLEAR` 优化进一步说明：持锁期即缓存有效期租约。

**5. 一次查询执行中 elog(ERROR) 从执行器深处抛出，说明内存为什么不泄漏、以及为什么中止处理自己不会因 OOM 卡死。**

longjmp 到 postgres.c:4573 的恢复块后调 `AbortCurrentTransaction()`：非内存资源按 ResourceOwner 登记逐一释放，随后 CleanupTransaction 删除 TopTransactionContext 整棵子树——查询期间的一切 palloc（含 ExecutorState/ExprContext）都在这棵树里，随树蒸发，无需知道谁分配的。中止处理不会 OOM 卡死是因为 `AtAbort_Memory()`（xact.c:1926）先切换到预留内存的 `TransactionAbortContext`（xact.c:1225 创建），错误报告则使用带保命内存的 ErrorContext——清理路径从不在"可能已经耗尽"的上下文里分配。
