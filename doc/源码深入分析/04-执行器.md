## 本章导读

执行器（executor）是 PostgreSQL 中"真正干活"的模块：优化器产出的 `PlannedStmt`
（一棵 Plan 树）在这里被实例化成**执行状态树**（PlanState 树），再按 Volcano
迭代器模型一行一行"拉"出结果。本章自顶向下：Volcano 理论与向量化/编译执行的
对比 → Portal 层与游标 → 三阶段生命周期（`ExecInitNode` 大 switch、
`ExecutePlan` 主循环、`ExecProcNodeFirst` 函数指针技巧）→ TupleTableSlot 与
惰性 deform → 五个典型算子（SeqScan/NestLoop/HashJoin/Agg/Sort）→ 表达式求值
虚拟机（JIT 的地基）→ EvalPlanQual 并发重检。

## 前置知识

- **Plan 树**：优化器输出的物理计划（`src/include/nodes/plannodes.h`），**只读**。
- **元组**：堆表物理行是 `HeapTuple`（带 MVCC 头）；执行器内部用 `TupleTableSlot` 传递行。
- **MemoryContext**（指南第 13 章）：执行器内存挂在 per-query context 下，查询结束整体释放。
- **快照**（指南第 9 章）：MVCC 可见性依据，执行器拿 `estate->es_snapshot` 问存储层"这行可见吗"。
- 类比：Plan 树 ≈ Spark 物理计划，PlanState 树 ≈ RDD compute 链；MySQL 8.0 的
  iterator executor（`sql/iterators/`）与 PG 同为 Volcano，但 PG 早了二十多年。

## 1. Volcano 迭代器模型：理论与 PG 的对应

### 1.1 Graefe 的模型

Goetz Graefe 在 1994 年的论文 *Volcano — An Extensible and Parallel Query
Evaluation System* 中提出：每个算子实现统一的三个接口——`open()` 初始化、
`next()` 返回下一行（NULL 表示耗尽）、`close()` 释放资源。父算子调子算子的
`next()` **拉取**（pull）一行，整棵树由根节点的 `next()` 驱动，数据从叶子
（扫描节点）向根流动。PG 的对应关系精确到令人发指：

| Volcano 接口 | PG 函数 | 位置 |
|---|---|---|
| `open()`  | `ExecutorStart()` → `ExecInitNode()` 递归 | `execMain.c:124`、`execProcnode.c:142` |
| `next()`  | `ExecutorRun()` → `ExecProcNode()` 递归 | `execMain.c:308`、`executor.h:322` |
| `close()` | `ExecutorEnd()` → `ExecEndNode()` 递归 | `execMain.c:477`、`execProcnode.c:543` |

### 1.2 pull 模型的优缺点

**优点**：
- **组合性**：任意算子任意嵌套，新增算子只需实现三个接口。`src/backend/executor/`
  下约 40 个 `nodeXxx.c` 每个独立可读，正是得益于此。
- **流水线**：一行数据从扫描到输出全程不落地（Sort/Hash 等 pipeline breaker 除外），
  内存占用与数据量无关；`LIMIT 10` 天然只拉 10 行就停。
- **控制流简单**：调用栈即数据流，gdb 一个 `bt` 就能看清整条管道。

**缺点**：
- **每行一次间接调用**：`node->ExecProcNode(node)` 无法内联，分支预测压力大，
  一行穿过 5 层算子就是 5 次间接调用；
- **缓存不友好**：同一算子的代码/数据在 CPU cache 里反复换入换出。

### 1.3 与向量化 / 编译执行的对比

现代分析型引擎有两条路线克服上述缺点：
- **向量化**：`next()` 一次返回一批（如 1024 行）列式数据，把间接调用开销摊薄
  千倍，内层循环可用 SIMD。代表：MonetDB/X100、ClickHouse、DuckDB。
- **编译执行**：把整棵算子树编译成一个紧凑循环，消灭所有间接调用。代表：HyPer，
  以及 **Spark whole-stage codegen**——把相邻 narrow 算子折叠成一个 Java 类的
  `processNext()`，是 Neumann（VLDB 2011）思路的工程化。

PG 的选择是**渐进改良**：表达式层做了字节码编译 + JIT（第 6 节）；算子分发层做
运行期特化（SeqScan 按有无 qual/投影预选四个变体，第 5.1 节）；批量化在存储层
逐步渗透，但**算子间接口仍是一行一趟**。取舍的逻辑：PG 以 OLTP 为主，点查场景
向量化收益有限，而 Volcano 的组合性与可维护性是社区二十多年迭代的根基。

### 小结

Volcano 模型 = 统一的 open/next/close 接口 + pull 驱动，PG 用
`ExecInitNode / ExecProcNode / ExecEndNode` 三个递归函数精确实现；每行一次
间接调用的开销由表达式 JIT 与运行期特化缓解——与 Spark whole-stage codegen
是同一问题的两种答案。

## 2. Portal 层：执行器的"外壳"

### 2.1 Portal 是什么

Portal（门户）是**一次查询执行的顶层句柄**，即 SQL 游标在 PG 内部的统一实现
——`DECLARE CURSOR` 创建命名 Portal，普通 `SELECT` 也创建名为 `""` 的匿名
Portal。扩展协议里 Bind 消息产生的就是 Portal，Execute 消息可带 `max_rows`
分批取结果——JDBC `setFetchSize()` 的底层机制。

Portal 的状态机定义在 `src/include/utils/portal.h:105`：

```
PORTAL_NEW → PORTAL_DEFINED → PORTAL_READY → PORTAL_ACTIVE → PORTAL_DONE
                                     ↑______________|            （或 PORTAL_FAILED）
```

对应的生命周期函数：`CreatePortal()`（`src/backend/utils/mmgr/portalmem.c:177`）
→ `PortalDefineQuery()`（`portalmem.c:284`）→ `PortalStart()`（调 ExecutorStart，
`src/backend/tcop/pquery.c:430`）→ `PortalRun()`（调 ExecutorRun，`pquery.c:681`）
/ 游标 `PortalRunFetch()`（`pquery.c:1374`）→ `PortalDrop()`（调 ExecutorEnd，
`portalmem.c:469`）。

注意 `PORTAL_ACTIVE → PORTAL_READY` 的回边：一次 `Execute(max_rows=100)` 取完
100 行后 Portal 回到 READY，等待下一次 Execute——**游标的可暂停性就藏在这条回边里**。

### 2.2 PortalRun 走读

`src/backend/tcop/pquery.c:681`：

```c
    MarkPortalActive(portal);                   /* 712 行：READY → ACTIVE，防止重入 */
    ...
    MemoryContextSwitchTo(PortalContext);       /* 741 行：切到 portal 私有内存上下文 */
    switch (portal->strategy)                   /* 743 行 */
    {
        case PORTAL_ONE_SELECT:
        case PORTAL_ONE_RETURNING: ...
            if (portal->strategy != PORTAL_ONE_SELECT && !portal->holdStore)
                FillPortalStore(portal, isTopLevel);          /* 755-756 行 */
            nprocessed = PortalRunSelect(portal, true, count, dest);  /* 761 行 */
```

逐段解说：`MarkPortalActive` 防止游标未取完时被重入；`PortalContext` 是 Portal
私有内存上下文，跨多次 FETCH 存活——游标能"暂停"的物质基础（PlanState 树整棵
活在其中）；`portal->strategy` 中 `PORTAL_ONE_SELECT` 直接流式执行，而
`PORTAL_ONE_RETURNING`（如 `UPDATE ... RETURNING`）必须先把语句**跑完**、结果
暂存进 tuplestore 再取——写操作不能"执行一半挂起"；`PortalRunSelect`（函数体
`pquery.c:860`）换算方向数量后调 `ExecutorRun()`；末尾（775 行）置回
`PORTAL_READY` 等下一次 Execute/FETCH。

### 小结

Portal = 游标 = 一次执行的可暂停句柄，持有私有内存上下文（PlanState 树的
宿主）、状态机和取行进度；`PortalRun → PortalRunSelect → ExecutorRun` 是简单
协议与扩展协议的汇合点。理解 Portal 后，JDBC fetchSize、`WITH HOLD` 游标、
`UPDATE RETURNING` 先跑完再返回等行为都有了统一解释。

## 3. 三阶段生命周期

### 3.1 ExecutorStart：搭台

`ExecutorStart`（`src/backend/executor/execMain.c:124`）本体只有几行：

```c
    if (ExecutorStart_hook)
        (*ExecutorStart_hook) (queryDesc, eflags);
    else
        standard_ExecutorStart(queryDesc, eflags);
```

四个阶段入口（Start/Run/Finish/End）都是这个"钩子优先"形状（钩子声明在
`execMain.c:70-73`）——`pg_stat_statements`、`auto_explain` 就是替换这些钩子来
拦截每次执行的。**这是 PG 可扩展性的标准模式，读任何模块先找 hook。**

`standard_ExecutorStart()`（`execMain.c:143`）：只读事务检查（170-172 行）→
`CreateExecutorState()` 建 EState 与 per-query 内存上下文 `es_query_cxt`
（177-180 行），填入快照、参数、CommandId → 末尾调 `InitPlan()`，后者对 Plan
树根节点调 `ExecInitNode()`。

### 3.2 ExecInitNode：镜像出 PlanState 树

`src/backend/executor/execProcnode.c:142` 是一个覆盖全部算子的大 switch：

```c
PlanState *
ExecInitNode(Plan *node, EState *estate, int eflags)
{
    if (node == NULL)
        return NULL;         /* 叶子节点的孩子为空，递归终止 */
    check_stack_depth();     /* 计划树可能极深，防爆栈 */
    switch (nodeTag(node))
    {
        case T_SeqScan:
            result = (PlanState *) ExecInitSeqScan((SeqScan *) node,
                                                   estate, eflags);
            break;
        ...                  /* 共约 40 个 case，一类一段 */
    }
    ExecSetExecProcNode(result, result->ExecProcNode);   /* 391 行 */
    ...
    if (estate->es_instrument)                           /* 415 行 */
        result->instrument = InstrAllocNode(estate->es_instrument, ...);
    return result;
}
```

switch 的分段结构（`execProcnode.c:161-389`）就是 PG 算子的全景图：
- **控制节点**（166-204 行）：`Result`、`ModifyTable`（INSERT/UPDATE/DELETE 的执行
  节点！写操作也是算子）、`Append`/`MergeAppend`（分区表拼接）、`RecursiveUnion` 等；
- **扫描节点**（209-292 行）：`SeqScan`、`IndexScan`、`IndexOnlyScan`、
  `BitmapHeapScan`、`SubqueryScan`、`FunctionScan`、`CteScan`、
  `ForeignScan`（FDW）、`CustomScan`（扩展自定义算子）等；
- **连接节点**（297-310 行）：`NestLoop`、`MergeJoin`、`HashJoin`——教科书三大连接齐全；
- **物化类节点**（315-383 行）：`Material`、`Sort`、`Memoize`（参数化内表结果缓存）、
  `Agg`、`WindowAgg`、`Gather`/`GatherMerge`（并行汇聚）、`Hash`（HashJoin 建表侧）、
  `LockRows`（FOR UPDATE）、`Limit` 等。

每个 `ExecInitXxx` 内部再递归调 `ExecInitNode(outerPlan(node))` /
`innerPlan(node)` 初始化孩子，于是 **PlanState 树与 Plan 树一一镜像**：每个
PlanState 的 `ps.plan` 指回对应 Plan 节点（只读的形状与参数），运行时可变状态
（扫描位置、哈希表、排序缓冲……）全在 PlanState 里。"只读计划 / 可变状态"
分离是 prepared statement 复用计划的前提，与 Spark "Physical Plan 不可变、
Task 各带 state" 同构。

### 3.3 ExecProcNodeFirst：函数指针替换的技巧

Plan 节点没有虚表，PG 用 C 手工实现"虚函数"：每个 PlanState 有一个函数指针
`ExecProcNode`。精妙之处在 `execProcnode.c:429`：

```c
void
ExecSetExecProcNode(PlanState *node, ExecProcNodeMtd function)
{
    node->ExecProcNodeReal = function;      /* 真正的算子函数存到 Real */
    node->ExecProcNode = ExecProcNodeFirst; /* 对外先挂一个包装器 */
}

static TupleTableSlot *
ExecProcNodeFirst(PlanState *node)          /* execProcnode.c:448 */
{
    check_stack_depth();                    /* 只在第一次调用时查栈深 */
    if (node->instrument)
        node->ExecProcNode = ExecProcNodeInstr;      /* EXPLAIN ANALYZE 路径 */
    else
        node->ExecProcNode = node->ExecProcNodeReal; /* 普通路径：卸掉包装 */
    return node->ExecProcNode(node);
}
```

首次 `ExecProcNode(node)` 进的是包装器：做一次栈深检查（457 行，之后的百万次
调用不再做），然后**把自己从调用链上摘除**（464-467 行）——需要计时（EXPLAIN
ANALYZE 时 `ExecInitNode` 末尾 415-417 行分配了 `instrument`）就换成
`ExecProcNodeInstr`（真调用前后打点 `InstrStartNode/InstrStopNode`），否则换成
裸函数，**零常驻开销**。这就是"instrument 注入"：不在热路径写
`if (need_timing)`，而是**首调时把分支固化成函数指针**；任何 per-node 包装都走
这个入口（`execProcnode.c:424-427` 注释鼓励运行中换指针，包装器自动重装）。

`ExecProcNode` 本身是内联函数（`src/include/executor/executor.h:322`）：

```c
    if (node->chgParam != NULL) /* 参数变了？ */
        ExecReScan(node);       /* 让 ReScan 先重置本子树 */
    return node->ExecProcNode(node);
```

`chgParam` 检查是参数化执行（NestLoop 内表、子查询）的枢纽，第 5.2 节详述。

### 3.4 ExecutorRun → ExecutePlan：主循环逐行走读

`ExecutorRun`（`execMain.c:308`）过钩子进 `standard_ExecutorRun`
（`execMain.c:318`）：切入 `es_query_cxt`（341 行）→ 判断是否发行给客户端
（358-359 行，`SELECT` 或带 `RETURNING`）→ 启动 DestReceiver（362 行，
`printtup` 即"把行编码成 wire protocol"的 receiver）→ 调 `ExecutePlan`（377 行）。

`ExecutePlan`（`execMain.c:1722`）是整个执行器的心跳：

```c
    estate->es_direction = direction;                 /* 1743 行 */
    ...                                               /* 1752-1760 行：并行模式判定 */
    for (;;)
    {
        ResetPerTupleExprContext(estate);             /* 1768 行 */
        slot = ExecProcNode(planstate);               /* 1773 行 */
        if (TupIsNull(slot))                          /* 1779 行 */
            break;
        if (estate->es_junkFilter != NULL)            /* 1790 行 */
            slot = ExecFilterJunk(estate->es_junkFilter, slot);
        if (sendTuples)
        {
            if (!dest->receiveSlot(slot, dest))       /* 1804 行 */
                break;
        }
        if (operation == CMD_SELECT)
            (estate->es_processed)++;                 /* 1814 行 */
        current_tuple_count++;
        if (numberTuples && numberTuples == current_tuple_count)
            break;                                    /* 1822 行 */
    }
```

逐行解说：
- **1752-1755 行**：并行计划只允许"一口气跑完"，游标式分批取行
  （`numberTuples != 0`）或已部分执行的 Portal 强制退回串行——"为什么我的
  游标不走并行计划"的答案。
- **1768 行**：重置 per-tuple 内存上下文，上一行求值的临时内存一次性归零。
  **PG 不做每行 malloc/free，靠的是这个行级内存池。**
- **1773 行**：唯一一次 `ExecProcNode`——从根拉一行，根再拉孩子，递归直达
  扫描节点。整棵树的执行压缩在这一行代码里。
- **1790 行**：junk filter 剔除"垃圾列"（不在 SELECT 列表里的 ORDER BY 键、
  UPDATE 需要的 ctid 等）。
- **1804 行**：`receiveSlot` 返回 false 即客户端断开，提前收工；**1822 行**：
  取够 `numberTuples`（游标 FETCH 份额）就 break——Portal 挂起点。

### 3.5 ExecutorFinish 与 ExecutorEnd：收台

`ExecutorFinish`（`execMain.c:417`）跑 AFTER 触发器等收尾（单独拆出是为了让
EXPLAIN ANALYZE 计入这段耗时）；`ExecutorEnd`（`execMain.c:477`）→
`standard_ExecutorEnd`（486 行）：`ExecEndPlan` 递归调 `ExecEndNode`
（`execProcnode.c:543`）关闭每个节点 → 注销快照（518-519 行）→
`FreeExecutorState`（530 行）**整体释放 per-query 内存上下文**——PG 没有执行器
内存泄漏顽疾的原因：不靠人肉配对 free，靠上下文整树销毁。

### 小结

三阶段 = open/next/close 在查询级别的放大：`ExecInitNode` 大 switch 镜像出
PlanState 树（只读计划/可变状态分离）；`ExecutePlan` 每转一圈从根拉一行；
`ExecProcNodeFirst` 用"首调自摘除"实现零开销 instrument 注入；四个 hook 是
监控类扩展的挂载点。

## 4. TupleTableSlot：行的抽象容器

### 4.1 为什么需要这层抽象

算子之间传递的不是裸 `HeapTuple`，而是 `TupleTableSlot`。原因有三：
1. **来源异构**：一行可能来自堆页面（还在共享缓冲区）、落盘文件（无 MVCC 头的
   紧凑格式）、或表达式现算（只有 Datum 数组，没有物理元组）。
2. **Table AM 解耦**（PG 12 起）：执行器与可插拔存储引擎的边界
   `table_scan_getnextslot()`（`src/include/access/tableam.h`）用 slot 作交换
   货币，物理格式由引擎自带的 `table_slot_callbacks()` 决定。**没有 slot 抽象
   就没有可插拔存储引擎。**
3. **省拷贝**：slot 可只持有指向缓冲区页面的指针（连带 buffer pin），需要长期
   保存时才 `ExecMaterializeSlot()` 物化。

四种 slot 实现以回调表（`TupleTableSlotOps`）形式定义在
`src/backend/executor/execTuples.c`：

| 类型 | 定义处 | 存的是什么 | 典型场景 |
|---|---|---|---|
| virtual | `execTuples.c:1266` | 仅 Datum/isnull 数组，无物理元组 | 投影、表达式输出、算子间传值 |
| heap | `execTuples.c:1287` | 独立 palloc 的 HeapTuple | 触发器、需要完整元组头的场合 |
| minimal | `execTuples.c:1305` | MinimalTuple（去掉 MVCC 头的瘦身格式） | 排序/哈希落盘、tuplestore |
| buffer heap | `execTuples.c:1323` | 指向共享缓冲区页面内的 HeapTuple + pin | 顺序/索引扫描直读堆页 |

`buffer heap slot` 最能体现设计意图：扫描返回的行**根本没离开缓冲池**，slot 里
只是一个指针加一个 pin 计数；WHERE 过滤淘汰掉的行零拷贝零分配。

### 4.2 slot_getattr 与惰性 deform

堆元组是**变长紧凑编码**：变长列（text 等）带长度头、可能 TOAST，NULL 列不占
字节。要取第 n 列，必须从第一列开始逐个跳——这个"解包"过程叫 **deform**，
是 OLAP 查询的头号 CPU 热点之一。PG 的对策是**惰性 + 增量 + 有缓存**。

入口是内联函数 `src/include/executor/tuptable.h:417`：

```c
static inline Datum
slot_getattr(TupleTableSlot *slot, int attnum, bool *isnull)
{
    if (attnum > slot->tts_nvalid)          /* 已解包的列数不够？ */
        slot_getsomeattrs(slot, attnum);    /* 补解到第 attnum 列 */
    *isnull = slot->tts_isnull[attnum - 1];
    return slot->tts_values[attnum - 1];    /* 命中缓存，直接取数组 */
}
```

`tts_nvalid` 是该 slot **已解包到第几列**的水位线；不够时经
`slot->tts_ops->getsomeattrs` 虚调用（`tuptable.h:376` →
`execTuples.c:2177`）落到核心函数
`slot_deform_heap_tuple`（`execTuples.c:1017`）：

```c
static pg_always_inline void
slot_deform_heap_tuple(TupleTableSlot *slot, HeapTuple tuple, uint32 *offp,
                       int reqnatts, bool support_cstring)
{
    ...
    if (HeapTupleHasNulls(tuple))                     /* 1053 行 */
    {
        firstNullAttr = first_null_attr(bp, natts);   /* 1065 行 */
        populate_isnull_array(bp, natts, isnull);
    }
    else
        firstNullAttr = natts;    /* 无 NULL：全部列可线性解包 */
```

机制拆解（结合 `execTuples.c:996-1014` 头注释）：
- **增量**：`*offp`（slot 里的 `tts_off`）记住上次解包停在数据区的字节偏移，
  本次续解——头注释原话 "incremental version of heap_deform_tuple"。
- **惰性**：`reqnatts` 只解到被引用的最大列号。`SELECT c2 FROM t`（100 列表）
  只解前 2 列——**常用列放表定义前面确实更快**，PG DBA 的经典调优项。
- **NULL 位图**：有 NULL 的元组先扫位图（1053-1084 行），NULL 之后偏移不可静态
  计算，退化为逐列跳变长头；本版本的 `CompactAttribute`/`firstNonCachedOffsetAttr`
  （1035、1051 行）预计算"前 k 列偏移固定"分界点，分界前按缓存偏移直取。
- **谁触发**：表达式虚拟机在步骤流开头插入 `EEOP_*_FETCHSOME` 指令一次解包到位
  （`execExprInterp.c:644-648`），后续取列全是 O(1) 数组访问。

### 小结

TupleTableSlot 用回调表把"行"抽象成四种实现，既是算子间通用货币，也是执行器
与 Table AM 的隔离层。deform 是取列的真实成本，PG 用 `tts_nvalid` 水位线 +
`tts_off` 断点续解 + FETCHSOME 预解包，把它做成惰性、增量、只付一次的操作。

### 5.3 nodeHashjoin.c：hybrid hash join 状态机

文件头 `nodeHashjoin.c:182-189` 定义了状态机的全部状态：

`HJ_FILL_INNER_TUPLES`（右/全外吐未匹配内行）、`HJ_NEED_NEW_BATCH`（切批次）。

主体 `ExecHashJoinImpl`（`nodeHashjoin.c:225`，`pg_always_inline`，由
`ExecHashJoin`/`ExecParallelHashJoin` 在 802/818 行以 `parallel` 常量实例化成
两个特化版本——又一次编译期分支消除）是 `for(;;) + switch(hj_JoinState)` 状态机
（259-269 行）。不写嵌套循环是因为 **Volcano 要求每次调用只返回一行**：函数
必须能在任意点"返回并下次恢复"，把控制流显式存进 `hj_JoinState` 是 C 语言
实现"可恢复协程"的标准做法。

**HJ_BUILD_HASHTABLE**（271 行起）：调 `MultiExecProcNode(hashNode)`
（`execProcnode.c:488`——Hash 节点不产行、一次性返回整张哈希表，是 Volcano 接口
的少数例外）读入整个内表，按 `work_mem × hash_mem_multiplier` 决定批次数。
**hybrid** 的含义：batch 0 常驻内存直接建表，批号 >0 的内表行写入临时文件
`innerBatchFile[batchno]`。

**HJ_NEED_NEW_OUTER**（435 行起）：外表耗尽时，右/全外连接转
`HJ_FILL_INNER_TUPLES` 吐未匹配内行，否则转 `HJ_NEED_NEW_BATCH`（448-486 行）；
未耗尽则算出外行的 (bucket, batch) 并查 skew 桶（497-500 行），**外行不属于
当前批次 → 以 MinimalTuple 写入 `outerBatchFile[batchno]` 暂存**（507-528 行
`ExecHashJoinSaveTuple`）并继续拉下一外行——hybrid 的外表分区一侧。

**HJ_SCAN_BUCKET**（536 行起）：沿桶链比较哈希值与连接条件，匹配则投影并返回
一行（**这里 return，下次凭 hj_JoinState 恢复现场**）；桶扫尽转
`HJ_FILL_OUTER_TUPLE`（左外补 NULL）或回 `HJ_NEED_NEW_OUTER`。

**HJ_NEED_NEW_BATCH**（770 行起）→ `ExecHashJoinNewBatch`
（`nodeHashjoin.c:1279`）：释放上一批哈希表，从 `innerBatchFile[curbatch]`
重建，再回放 `outerBatchFile[curbatch]` 暂存的外行。类比 Spark shuffle hash
join："批次"即分区，只是分区文件在本机临时目录而非跨网络 shuffle。

**skew 优化**（防倾斜）：`ExecHashBuildSkewHash`（`nodeHash.c:2429`）建表时按
外表连接列的 **MCV 统计**为高频值单独建 skew 桶，预算约 2% 哈希内存
（`nodeHash.c:545`）。命中 skew 桶的外行（`nodeHashjoin.c:499`）**无论属于哪
个批次都在 batch 0 就地完成连接**，避免最热的键被反复写盘回放；第一批结束后
skew 桶退役（`nodeHashjoin.c:1304-1309`）。与 Spark AQE skew split 目标一致：
PG 靠统计前置规避，Spark 靠运行时切分补救。

### 5.4 nodeAgg.c：哈希聚合、分组聚合与落盘

入口 `ExecAgg`（`nodeAgg.c:2247`）按策略分发（2257-2270 行）：

**内存超限落盘**（PG 13 起，此前哈希聚合会吃内存直到 OOM）：每插入新分组后调

### 5.5 nodeSort.c → tuplesort.c：外排序多路归并

`ExecSort`（`nodeSort.c:50`）是"物化型算子"的极简样板：首次被拉取时（75 行
`!node->sort_Done`）进入建库阶段——`tuplesort_begin_heap`（114 行）建排序器，
148-154 行 `for(;;)` 把子节点**全部**拉完喂给 `tuplesort_puttupleslot`，160 行

`tuplesort_gettupleslot`（206 行）取下一行。单列时走 datum sort 快径（105 行）；
`ORDER BY ... LIMIT k` 时 `tuplesort_set_bound`（124 行）退化为**堆维护的
top-k**，无需全量排序。

真正的排序引擎在 `src/backend/utils/sort/tuplesort.c`，五状态状态机
（`tuplesort.c:155-160`）：`TSS_INITIAL`（内存装得下，攒着）→
`TSS_BUILDRUNS`（内存爆了，边排边写初始顺串 run 到磁带）→ 终态三选一：
`TSS_SORTEDINMEM`（纯内存排序完成）/ `TSS_SORTEDONTAPE`（归并彻底完成）/
`TSS_FINALMERGE`（最后一轮归并在线进行，边归并边吐行）。

关键路径：内存超限触发 `inittapes`（`tuplesort.c:1762`）转 `TSS_BUILDRUNS`，

**教科书式多路归并**：K 个磁带各出一行进堆，反复弹最小者；runs ≤ 磁带数时停在
`TSS_FINALMERGE` **在线归并**，上层每拉一行才弹一行，省掉最后一次全量写盘。
`trace_sort = on` 可在日志观察全程；EXPLAIN ANALYZE 的
`Sort Method: external merge  Disk: ...` 即对应外排路径。对照大数据：这是
MapReduce/Spark sort-spill-merge 的单机版。

### 小结

五个算子勾勒出执行器全部套路：SeqScan——三件套 + Table AM 边界 + 运行期特化；
NestLoop——参数化与 `chgParam`/`ExecReScan` 重扫协议；HashJoin——可恢复状态机 +
hybrid 分批落盘 + skew 规避；Agg——哈希/分组两形态与 spill-refill；
Sort/tuplesort——物化算子与外排序状态机。所有落盘算子共享同一套
BufFile/LogicalTape 临时文件基础设施。

## 6. 表达式求值虚拟机：JIT 的地基

### 6.1 从表达式树到线性字节码

PG 10 之前，`WHERE a > 10 AND b < 5` 按表达式**树**递归解释执行，每节点一次
函数调用加大量重复分支。PG 10 重写为**先编译、后执行**：初始化期把树展平成
一维**步骤数组**（`ExprEvalStep`，含 opcode 和操作数），执行期用小型虚拟机线性扫过。

编译入口 `ExecInitExpr`（`src/backend/executor/execExpr.c:143`）：

```c
ExprState *
ExecInitExpr(Expr *node, PlanState *parent)
{
    ...
    state = makeNode(ExprState);
    ...
    ExecCreateExprSetupSteps(state, (Node *) node);   /* 159 行：插入 FETCHSOME */
    ExecInitExprRec(node, state, &state->resvalue, &state->resnull);  /* 162 行 */
    scratch.opcode = EEOP_DONE_RETURN;                /* 165 行：追加终止指令 */
    ExprEvalPushStep(state, &scratch);
    ExecReadyExpr(state);                             /* 168 行：定稿 */
    return state;
}
```

159 行分析表达式引用了哪些列，生成 `EEOP_SCAN_FETCHSOME` 等步骤——即 4.2 节
"一次性预解包到最大列号"的来源。162 行 `ExecInitExprRec`（`execExpr.c:919`）
后序遍历表达式树：子表达式的步骤先入数组、结果写入约定的 `resvalue/resnull`
槽位，父步骤再消费——树的数据依赖被编译成数组的顺序依赖；AND/OR 短路编译成
`EEOP_JUMP_IF_*` 跳转指令。

定稿函数 `ExecReadyExpr`（`execExpr.c:902`）只有四行，却是 JIT 的总开关：
`if (jit_compile_expr(state)) return; ExecReadyInterpretedExpr(state);`
（904-907 行）。`jit_compile_expr` 在代价超过 `jit_above_cost` 时把**同一份
步骤数组**翻译成 LLVM IR 编成机器码，替换 `state->evalfunc`。**JIT 不是另起
炉灶，而是同一套字节码的第二个后端**——"表达式虚拟机是 JIT 地基"的确切含义。

### 6.2 ExecInterpExpr：computed goto 分发

解释器在 `src/backend/executor/execExprInterp.c`。分发机制由一组宏切换
（`execExprInterp.c:104-131`）：

```c
#if defined(EEO_USE_COMPUTED_GOTO)
#define EEO_CASE(name)      CASE_##name:
#define EEO_DISPATCH()      goto *((void *) op->opcode)
#else
#define EEO_SWITCH()        starteval: switch ((ExprEvalOp) op->opcode)
#define EEO_DISPATCH()      goto starteval
#endif

#define EEO_NEXT()   do { op++; EEO_DISPATCH(); } while (0)
```

**普通 switch**（可移植兜底）：每条指令执行完跳回 `starteval` 再 switch，所有
指令共享一个间接跳转点，分支预测器只能学到混杂分布。**computed goto**
（GCC/Clang 扩展：`&&label` 取标签地址 + `goto *ptr`）：`ExecInterpExpr`
（`execExprInterp.c:470`）内定义 `static const void *const dispatch_table[]`
（484 行），按 opcode 枚举顺序存放每个 `CASE_EEOP_*` 标签地址（485-605 行约百
条指令，608 行 StaticAssert 保证与枚举同步）；每条指令结尾的 `EEO_NEXT()` 展开
为 `op++; goto *op->opcode`——**每条指令自带独立的间接跳转**，预测器能分别学习
"指令 A 之后通常是 B"。CPython、LuaJIT 同款技巧。

更进一步的**直接线索化**（direct threading）：`ExecReadyInterpretedExpr`
（`execExprInterp.c:252`）在 `execExprInterp.c:446-453` 遍历步骤数组，执行
`op->opcode = EEO_OPCODE(op->opcode)`——把每个步骤的 `opcode` 字段**就地改写
成目标标签的机器地址**，执行期 `goto *op->opcode` 连查表都省了。（首次求值还
有个小机关：`state->evalfunc` 先指向 `ExecInterpExprStillValid`（276 行）校验
一次再换成真正入口，与 `ExecProcNodeFirst` 的"首调自摘除"如出一辙。）

执行体（630 行起）的一个例子：`EEOP_INNER_FETCHSOME`（644 行）调
`slot_getsomeattrs(innerslot, op->d.fetch.last_var)`——第 4 节惰性 deform 的
触发点。指令集里还有大量**特化指令**：`EEOP_FUNCEXPR_STRICT_1/_2`（1/2 参严格
函数免循环）、`EEOP_QUAL`（WHERE 专用短路）、`EEOP_AGG_PLAIN_TRANS_BYVAL`
（聚合转移快径）、`EEOP_HASHED_SCALARARRAYOP`（大 IN 列表转哈希查找）——
虚拟机的性能来自"为高频模式铸造专用指令"。

### 小结

表达式执行 = 编译期（树→线性步骤数组）+ 运行期（computed-goto/direct-threaded
虚拟机）。`ExecReadyExpr` 是解释与 JIT 的分岔口，JIT 只是同一字节码的 LLVM
后端；FETCHSOME 指令把 slot 惰性 deform 与表达式求值缝合起来。

## 7. EvalPlanQual：READ COMMITTED 下的并发更新重检

### 7.1 问题

事务 A `UPDATE ... WHERE balance > 100` 按快照看到某行 balance=150；提交前
事务 B 已把它改成 50 并提交。READ COMMITTED 下 A 不能报错重试（那是 REPEATABLE
READ 的做法），也不能改旧版本（会丢失 B 的更新）。PG 的答案：**等 B 提交 →
找到最新版本 → 只对这一行重新检查 WHERE**——新版本不再满足则跳过，仍满足则
基于新版本更新。这就是 EvalPlanQual（EPQ），实现在 `execMain.c` 尾部，设计
文档见 `src/backend/executor/README` 的 EvalPlanQual 一节。

### 7.2 机制走读

入口 `EvalPlanQual`（`src/backend/executor/execMain.c:2715`）：

```c
TupleTableSlot *
EvalPlanQual(EPQState *epqstate, Relation relation,
             Index rti, TupleTableSlot *inputslot)
{
    ...
    EvalPlanQualBegin(epqstate);            /* 2726 行：搭建/复位 EPQ 子执行器 */
    testslot = EvalPlanQualSlot(epqstate, relation, rti);   /* 2732 行 */
    if (testslot != inputslot)
        ExecCopySlot(testslot, inputslot);  /* 放入该行的最新版本 */

    epqstate->relsubs_done[rti - 1] = false;    /* 2741 行：只有这张表"有料" */
    epqstate->relsubs_blocked[rti - 1] = false;

    slot = EvalPlanQualNext(epqstate);      /* 2747 行：跑一遍迷你计划 */
    if (!TupIsNull(slot))
        ExecMaterializeSlot(slot);          /* 2757 行：物化，脱离子执行器 */
    ...
    return slot;                            /* 非空 = 新版本仍满足条件 */
}
```

要点：
- 调用方是 `ModifyTable`/`LockRows`：`heap_update` 返回 `TM_Updated` 后，先用
  `table_tuple_lock(...FIND_LAST_VERSION)` **沿 ctid 版本链追到最新版本并加锁**
  （期间阻塞等对方事务出结果），再交给 `EvalPlanQual` 重检（2704-2709 行注释）。
- EPQ 用**同一棵 Plan 子树**另建迷你执行器（独立 EState），但目标表的扫描节点
  被"掉包"：不真扫表，而是从 `relsubs_slot[rti-1]` 取出那**一行**测试元组
  （`relsubs_done/blocked` 数组控制每表吐 0 行还是 1 行）。5.1 节的
  `ExecSeqScanEPQ` 变体（`nodeSeqscan.c:206`）就是为这条路径准备的。
- 整个 WHERE（乃至 join 条件）对"新版本行 + 其余表原配行"重算一遍：非空 →
  基于新版本继续 UPDATE；NULL → 静默跳过该行。
- `EvalPlanQualInit`（`execMain.c:2784`）在 ModifyTable 初始化时准备状态，
  子执行器**惰性构建**（2795-2800 行注释）——无冲突时零开销。

**语义代价**：重检后的 UPDATE 混合了两个时间点的视图（目标行是最新版本，join
的其他行仍是旧快照），学术上不可串行化——READ COMMITTED 与 REPEATABLE READ/SSI
的本质分界。对照 MySQL：InnoDB 的 UPDATE 用**当前读**直接按最新已提交数据重新
定位，PG 是"快照读定位 + 冲突行单独重检"，殊途同归但边界行为不同（PG 不会
看到快照之后才插入的新行）。

### 小结

EPQ = "锁到最新版本 + 用掉包了扫描节点的迷你执行器对单行重跑计划"，让 READ
COMMITTED 的写语句不丢失并发更新、也无需整语句重试，代价是轻微的时间点混合。
`relsubs_*` 数组是阅读 `nodeModifyTable.c` 前的必备知识。

## 自测问题

**1. Plan 树和 PlanState 树为什么必须是两棵树？各自存什么？**

简答：Plan 树是只读计划（形状、代价、表达式原型），可被 prepared statement
多次复用、被并行 worker 共享；PlanState 树（`ExecInitNode` 镜像构建）持有单次
执行的可变状态（扫描位置、哈希表、排序缓冲、ExprState）。混在一起则计划无法
复用，也无法并发执行同一计划。

**2. EXPLAIN ANALYZE 的每节点计时怎么注入？为什么普通执行几乎零开销？**

简答：`ExecInitNode` 末尾把 `ExecProcNode` 指针统一挂成包装器
`ExecProcNodeFirst`（`execProcnode.c:429-470`）；首次调用时它按
`node->instrument` 把指针换成 `ExecProcNodeInstr`（前后打点）或裸的
`ExecProcNodeReal`。分支只发生一次，之后热路径没有任何 `if`。

**3. `SELECT c2 FROM t`（t 有 100 列），PG 会解包整行吗？**

简答：不会。表达式编译期生成 `EEOP_SCAN_FETCHSOME`，只解包到被引用的最大列号
（这里是 2）；执行时 `slot_getsomeattrs`（`tuptable.h:376`）查水位线
`tts_nvalid`，不足才由 `slot_deform_heap_tuple`（`execTuples.c:1017`）从上次
偏移 `tts_off` 增量续解。所以常用列放表定义前面确实省 CPU。

**4. hybrid hash join 的"hybrid"体现在哪里？skew 优化解决什么？**

简答：batch 0 在内存建表并即时连接（in-memory 的一半），批号 >0 的内、外表行
写临时文件、之后逐批重建回放（`ExecHashJoinNewBatch`，`nodeHashjoin.c:1279`，
Grace 的一半）——内存够则纯内存，不够平滑转外存，故称 hybrid。skew 优化
（`nodeHash.c:2429`）用外表 MCV 统计把最高频键的内表行留在内存 skew 桶
（约 2% 哈希内存），热键外行永远在 batch 0 就地完成，避免高频键被成批写盘
回放造成倾斜放大。

**5. READ COMMITTED 下，UPDATE 扫到的行已被并发事务改掉并提交，接下来发生什么？**

简答：`heap_update` 返回 `TM_Updated`；执行器用
`table_tuple_lock(...FIND_LAST_VERSION)` 沿版本链等锁并取最新版本；
`EvalPlanQual`（`execMain.c:2715`）把该行放进 EPQ 槽，用扫描节点被替换成
"只吐这一行"的迷你执行器重跑计划子树：非空则基于新版本更新，NULL 则跳过。
整语句不报错不重启——这是与 REPEATABLE READ（报 serialization failure）的
关键行为差异。
