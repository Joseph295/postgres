## 1. Volcano 迭代器模型：理论与 PG 的对应

### 1.1 Graefe 的模型

Goetz Graefe 在 1994 年的论文 *Volcano — An Extensible and Parallel Query
Evaluation System* 中提出：每个算子实现统一的三个接口——`open()` 初始化、
`next()` 返回下一行（NULL 表示耗尽）、`close()` 释放资源。父算子调子算子的
`next()` **拉取**（pull）一行，整棵树由根节点的 `next()` 驱动，数据从叶子
（扫描节点）向根流动。PG 的对应关系精确到令人发指：

| Volcano 接口 | PG 函数 | 位置 |
|---|---|---|
| `open()`  | `ExecutorStart()` → `ExecInitNode()` 递归 | `execMain.c:124`、`execProcnode.c:142` |
| `next()`  | `ExecutorRun()` → `ExecProcNode()` 递归 | `execMain.c:308`、`executor.h:322` |
| `close()` | `ExecutorEnd()` → `ExecEndNode()` 递归 | `execMain.c:477`、`execProcnode.c:543` |

### 1.2 三问：为什么是 pull 而不是 push

**① 解决什么问题**：查询计划是任意嵌套的算子组合，需要一个让"任意算子挂在任意
算子之上"都能工作的通用驱动协议。pull 模型把协议压缩为一个函数签名
`TupleTableSlot *ExecProcNode(PlanState *)`，新增算子只需实现它——
`src/backend/executor/` 下约 40 个 `nodeXxx.c` 每个独立可读，正是得益于此。

**② 为何此方案而非替代（pull vs push 的调用栈与 cache 对比）**：

- *控制流归属*。pull 模型中**消费者持有控制流**：`Limit` 取够 10 行就不再调
  `ExecProcNode`，提前终止零成本（见第 8.3 节）。push 模型中生产者驱动，
  终止必须靠回传"别再发了"的带外信号（Spark 的 codegen 里就是
  `shouldStop()` 检查散布在生成代码中）。
- *调用栈形状*。pull：一行数据从叶到根，穿过 N 层算子就是 N 次
  `node->ExecProcNode(node)` 间接调用 + N 次函数返回，栈帧反复建立/销毁；
  gdb 里一个 `bt` 就能看到整条管道——**调用栈即数据流**，可调试性极佳。
  push：算子逻辑被"翻转"内联进生产者的循环里，一层循环跑到底，没有逐行的
  调用开销，但栈上看不到管道形状，出错时难以定位到"第几个算子"。
- *cache 局部性*。pull 的劣势：每行都要把 N 个算子的代码轮流拉进 I-cache，
  算子状态轮流拉进 D-cache，行与行之间同一算子的代码/数据被反复驱逐。push +
  批处理（向量化）或 push + 编译（codegen）都是在修复这一点：让一个算子
  连续处理一批行（摊薄 I-cache miss），或把多个算子熔成一段直线代码
  （彻底消灭间接调用）。
- PG 的判断：它以 OLTP 为主，点查场景每查询只流过几行到几千行，间接调用
  开销占比很小；而 pull 的组合性、可暂停性（游标！见第 2 节）和可调试性
  是社区二十多年迭代的根基。真正热的部分——**表达式求值**（每行要算多次）
  ——已经用字节码 + JIT 重写（第 6 节），算子分发层用运行期特化缓解（第 5.1 节）。

**③ 正确性不变式**：`ExecProcNode` 的契约是——每次调用返回"下一行"或 NULL
（耗尽）；返回 NULL 之后**不得**再被调用（见 `execMain.c:364-375` 注释：
heapam 在 EOF 后再调会重新开始扫描）；`ExecReScan` 之后必须从头重新产出
（rescan 契约，第 5.2 节详述）。所有算子状态机的设计都围绕"能在任意一次
return 之后凭状态恢复现场"这条约束展开。

### 1.3 与向量化 / 编译执行的对比

现代分析型引擎有两条路线克服 pull 的每行开销：

- **向量化**：`next()` 一次返回一批（如 1024 行）列式数据，把间接调用开销摊薄
  千倍，内层循环可用 SIMD。代表：MonetDB/X100、ClickHouse、DuckDB。
- **编译执行**：把整棵算子树编译成一个紧凑循环，消灭所有间接调用。代表：HyPer，
  以及 **Spark whole-stage codegen**——把相邻 narrow 算子折叠成一个 Java 类的
  `processNext()`，是 Neumann（VLDB 2011）思路的工程化。

PG 的选择是**渐进改良**：表达式层做了字节码编译 + JIT（第 6 节）；算子分发层做
运行期特化（SeqScan 按有无 qual/投影预选四个变体，第 5.1 节）；批量化在存储层
逐步渗透，但**算子间接口仍是一行一趟**。

### 小结

Volcano 模型 = 统一的 open/next/close 接口 + pull 驱动，PG 用
`ExecInitNode / ExecProcNode / ExecEndNode` 三个递归函数精确实现。pull 换来
组合性、天然提前终止与可调试性，付出每行 N 次间接调用与 cache 局部性；PG 用
表达式 JIT 与运行期特化在不动接口的前提下收回大部分代价——与 Spark
whole-stage codegen 是同一问题的两种答案。

## 2. Portal 层：执行器的"外壳"

### 2.1 Portal 是什么

Portal（门户）是**一次查询执行的顶层句柄**，即 SQL 游标在 PG 内部的统一实现
——`DECLARE CURSOR` 创建命名 Portal，普通 `SELECT` 也创建名为 `""` 的匿名
Portal。扩展协议里 Bind 消息产生的就是 Portal，Execute 消息可带 `max_rows`
分批取结果——JDBC `setFetchSize()` 的底层机制。

Portal 的状态机定义在 `src/include/utils/portal.h:105`（`PORTAL_NEW` 起）：

```
PORTAL_NEW → PORTAL_DEFINED → PORTAL_READY → PORTAL_ACTIVE → PORTAL_DONE
                                     ↑______________|            （或 PORTAL_FAILED）
```

对应的生命周期函数：`CreatePortal()`（`src/backend/utils/mmgr/portalmem.c:177`）
→ `PortalDefineQuery()`（`portalmem.c:284`）→ `PortalStart()`（调 ExecutorStart，
`src/backend/tcop/pquery.c:430`，末尾 `pquery.c:606` 置 `PORTAL_READY`）→
`PortalRun()`（调 ExecutorRun，`pquery.c:681`）/ 游标 `PortalRunFetch()`
（`pquery.c:1374`）→ `PortalDrop()`（调 ExecutorEnd，`portalmem.c:469`）。

注意 `PORTAL_ACTIVE → PORTAL_READY` 的回边：一次 `Execute(max_rows=100)` 取完
100 行后 Portal 回到 READY，等待下一次 Execute——**游标的可暂停性就藏在这条回边里**。
这也正是 pull 模型的独门收益：暂停 = "不再调 `ExecProcNode`"，PlanState 树原地
冻结在 Portal 的私有内存上下文里；push 模型要做到这一点得把整个执行栈做成协程。

### 2.2 PortalRun 走读

`src/backend/tcop/pquery.c:681`，关键路径（行号已核实）：

- `pquery.c:712`：`MarkPortalActive(portal)`——READY → ACTIVE，防止游标未取完时被重入；
- `pquery.c:743`：`switch (portal->strategy)`；
- `pquery.c:756`：`PORTAL_ONE_RETURNING` 等策略先 `FillPortalStore(portal, isTopLevel)`
  ——`UPDATE ... RETURNING` 必须把语句**跑完**、结果暂存进 tuplestore 再取，
  因为写操作不能"执行一半挂起"；
- `pquery.c:761`：`nprocessed = PortalRunSelect(portal, true, count, dest)`
  （函数体 `pquery.c:860`）换算方向与数量后调 `ExecutorRun()`；
- `pquery.c:775`：取完本份额置回 `portal->status = PORTAL_READY`；全部取尽则
  `pquery.c:788` `MarkPortalDone(portal)`。

### 小结

Portal = 游标 = 一次执行的可暂停句柄，持有私有内存上下文（PlanState 树的
宿主）、状态机和取行进度；`PortalRun → PortalRunSelect → ExecutorRun` 是简单
协议与扩展协议的汇合点。理解 Portal 后，JDBC fetchSize、`WITH HOLD` 游标、
`UPDATE RETURNING` 先跑完再返回等行为都有了统一解释。

## 3. 三阶段生命周期

### 3.1 ExecutorStart：搭台

`ExecutorStart`（`src/backend/executor/execMain.c:124`）本体只有几行：

```c
	if (ExecutorStart_hook)
		(*ExecutorStart_hook) (queryDesc, eflags);
	else
		standard_ExecutorStart(queryDesc, eflags);
```

四个阶段入口（Start/Run/Finish/End）都是这个"钩子优先"形状（钩子变量声明在
`execMain.c:70-73`）——`pg_stat_statements`、`auto_explain` 就是替换这些钩子来
拦截每次执行的。**这是 PG 可扩展性的标准模式，读任何模块先找 hook。**

`standard_ExecutorStart()`（`execMain.c:143`）：只读事务/并行模式写检查
（170-172 行）→ `CreateExecutorState()` 建 EState 与 per-query 内存上下文
`es_query_cxt` 并切入（177-180 行）→ 填入参数、CommandId（186-236 行）→
末尾调 `InitPlan()`（`execMain.c:847`），后者对 Plan 树根节点调 `ExecInitNode()`。

### 3.2 ExecInitNode：镜像出 PlanState 树

`src/backend/executor/execProcnode.c:142` 是一个覆盖全部算子的大 switch：

```c
PlanState *
ExecInitNode(Plan *node, EState *estate, int eflags)
{
    if (node == NULL)
        return NULL;         /* 叶子节点的孩子为空，递归终止 */
    check_stack_depth();     /* 计划树可能极深，防爆栈 */
    switch (nodeTag(node))
    {
        case T_SeqScan:
            result = (PlanState *) ExecInitSeqScan((SeqScan *) node,
                                                   estate, eflags);
            break;
        ...                  /* 共约 40 个 case，一类一段 */
    }
    ExecSetExecProcNode(result, result->ExecProcNode);
    ...
    if (estate->es_instrument)
        result->instrument = InstrAllocNode(estate->es_instrument, ...);
    return result;
}
```

switch 的分段结构就是 PG 算子的全景图：

- **控制节点**：`Result`、`ModifyTable`（INSERT/UPDATE/DELETE 的执行节点！
  写操作也是算子）、`Append`/`MergeAppend`（分区表拼接）、`RecursiveUnion` 等；
- **扫描节点**：`SeqScan`、`IndexScan`、`IndexOnlyScan`、`BitmapHeapScan`、
  `SubqueryScan`、`FunctionScan`、`CteScan`、`ForeignScan`（FDW）、
  `CustomScan`（扩展自定义算子）等；
- **连接节点**：`NestLoop`、`MergeJoin`、`HashJoin`——教科书三大连接齐全；
- **物化类节点**：`Material`、`Sort`、`Memoize`（参数化内表结果缓存）、`Agg`、
  `WindowAgg`、`Gather`/`GatherMerge`（并行汇聚）、`Hash`（HashJoin 建表侧）、
  `LockRows`（FOR UPDATE）、`Limit` 等。

每个 `ExecInitXxx` 内部再递归调 `ExecInitNode(outerPlan(node))` /
`innerPlan(node)` 初始化孩子，于是 **PlanState 树与 Plan 树一一镜像**：每个
PlanState 的 `ps.plan` 指回对应 Plan 节点（只读的形状与参数），运行时可变状态
（扫描位置、哈希表、排序缓冲……）全在 PlanState 里。

**三问**：①解决什么问题——prepared statement 要求同一份计划被执行多次、并行
查询要求同一份计划被多个 worker 同时执行，计划必须只读；②为何此方案——若把
运行状态写进 Plan 节点，复用与共享都不可能，"镜像树"以一次 O(节点数) 的
初始化开销换来彻底的读写分离；③不变式——Plan 树在执行期间**绝不被修改**
（这也是 EXPLAIN 能在执行后仍打印原计划的原因）。与 Spark "Physical Plan
不可变、Task 各带 state" 同构。

### 3.3 ExecProcNodeFirst：函数指针替换的技巧

Plan 节点没有虚表，PG 用 C 手工实现"虚函数"：每个 PlanState 有一个函数指针
`ExecProcNode`。精妙之处在 `execProcnode.c:430`：

```c
void
ExecSetExecProcNode(PlanState *node, ExecProcNodeMtd function)
{
	/*
	 * Add a wrapper around the ExecProcNode callback that checks stack depth
	 * during the first execution and maybe adds an instrumentation wrapper.
	 * When the callback is changed after execution has already begun that
	 * means we'll superfluously execute ExecProcNodeFirst, but that seems ok.
	 */
	node->ExecProcNodeReal = function;
	node->ExecProcNode = ExecProcNodeFirst;
}
```

首次 `ExecProcNode(node)` 进的是包装器 `ExecProcNodeFirst`（`execProcnode.c:448`）：
做一次栈深检查（457 行，之后的百万次调用不再做），然后**把自己从调用链上摘除**
（464-467 行）——需要计时（EXPLAIN ANALYZE 时 `ExecInitNode` 末尾分配了
`instrument`）就换成 `ExecProcNodeInstr`（真调用前后打点
`InstrStartNode/InstrStopNode`），否则换成裸函数，**零常驻开销**。这就是
"instrument 注入"：不在热路径写 `if (need_timing)`，而是**首调时把分支固化成
函数指针**；任何 per-node 包装都走这个入口（`execProcnode.c:424-427` 注释鼓励
运行中换指针，包装器自动重装）。

`ExecProcNode` 本身是内联函数（`src/include/executor/executor.h:322`）：

```c
static inline TupleTableSlot *
ExecProcNode(PlanState *node)
{
	if (node->chgParam != NULL) /* something changed? */
		ExecReScan(node);		/* let ReScan handle this */

	return node->ExecProcNode(node);
}
```

`chgParam` 检查是参数化执行（NestLoop 内表、子查询）的枢纽，第 5.2 节详述。

### 3.4 ExecutorRun → ExecutePlan：主循环全函数走读

`ExecutorRun`（`execMain.c:308`）过钩子进 `standard_ExecutorRun`
（`execMain.c:318`）：切入 `es_query_cxt`（341 行）→ 判断是否发行给客户端
（358-359 行：`sendTuples = (operation == CMD_SELECT || hasReturning)`）→
启动 DestReceiver（361-362 行 `dest->rStartup(...)`，`printtup` 即"把行编码成
wire protocol"的 receiver）→ 非 NoMovement 方向时调 `ExecutePlan`（376-382 行）
→ 结束后累加 `es_total_processed`（388 行）、`rShutdown`（393-394 行）。

`ExecutePlan`（`execMain.c:1722`）是整个执行器的心跳。**全函数**（逐字摘自
1722-1835 行，仅省略函数头注释）：

```c
static void
ExecutePlan(QueryDesc *queryDesc,
			CmdType operation,
			bool sendTuples,
			uint64 numberTuples,
			ScanDirection direction,
			DestReceiver *dest)
{
	EState	   *estate = queryDesc->estate;
	PlanState  *planstate = queryDesc->planstate;
	bool		use_parallel_mode;
	TupleTableSlot *slot;
	uint64		current_tuple_count;

	/*
	 * initialize local variables
	 */
	current_tuple_count = 0;

	/*
	 * Set the direction.
	 */
	estate->es_direction = direction;

	/*
	 * Set up parallel mode if appropriate.
	 *
	 * Parallel mode only supports complete execution of a plan.  If we've
	 * already partially executed it, or if the caller asks us to exit early,
	 * we must force the plan to run without parallelism.
	 */
	if (queryDesc->already_executed || numberTuples != 0)
		use_parallel_mode = false;
	else
		use_parallel_mode = queryDesc->plannedstmt->parallelModeNeeded;
	queryDesc->already_executed = true;

	estate->es_use_parallel_mode = use_parallel_mode;
	if (use_parallel_mode)
		EnterParallelMode();

	/*
	 * Loop until we've processed the proper number of tuples from the plan.
	 */
	for (;;)
	{
		/* Reset the per-output-tuple exprcontext */
		ResetPerTupleExprContext(estate);

		/*
		 * Execute the plan and obtain a tuple
		 */
		slot = ExecProcNode(planstate);

		/*
		 * if the tuple is null, then we assume there is nothing more to
		 * process so we just end the loop...
		 */
		if (TupIsNull(slot))
			break;

		/*
		 * If we have a junk filter, then project a new tuple with the junk
		 * removed.
		 *
		 * Store this new "clean" tuple in the junkfilter's resultSlot.
		 * (Formerly, we stored it back over the "dirty" tuple, which is WRONG
		 * because that tuple slot has the wrong descriptor.)
		 */
		if (estate->es_junkFilter != NULL)
			slot = ExecFilterJunk(estate->es_junkFilter, slot);

		/*
		 * If we are supposed to send the tuple somewhere, do so. (In
		 * practice, this is probably always the case at this point.)
		 */
		if (sendTuples)
		{
			/*
			 * If we are not able to send the tuple, we assume the destination
			 * has closed and no more tuples can be sent. If that's the case,
			 * end the loop.
			 */
			if (!dest->receiveSlot(slot, dest))
				break;
		}

		/*
		 * Count tuples processed, if this is a SELECT.  (For other operation
		 * types, the ModifyTable plan node must count the appropriate
		 * events.)
		 */
		if (operation == CMD_SELECT)
			(estate->es_processed)++;

		/*
		 * check our tuple count.. if we've processed the proper number then
		 * quit, else loop again and process more tuples.  Zero numberTuples
		 * means no limit.
		 */
		current_tuple_count++;
		if (numberTuples && numberTuples == current_tuple_count)
			break;
	}

	/*
	 * If we know we won't need to back up, we can release resources at this
	 * point.
	 */
	if (!(estate->es_top_eflags & EXEC_FLAG_BACKWARD))
		ExecShutdownNode(planstate);

	if (use_parallel_mode)
		ExitParallelMode();
}
```

逐段解说（行号对应上文源码）：

- **1743 行** `es_direction`：扫描方向存进 EState，叶子节点（如 `SeqNext`）
  执行时读它——`FETCH BACKWARD` 就是把这里改成反向。
- **1752-1756 行** 并行判定：并行计划只允许"一口气跑完"。两种情况强制退回串行：
  该 Portal 已部分执行过（`already_executed`），或调用方要求分批取行
  （`numberTuples != 0`，即游标 FETCH / Execute(max_rows)）。这是"为什么我的
  游标不走并行计划"的答案：并行 worker 的启停无法随 Portal 挂起/恢复反复进行。
- **1759-1760 行**：进入并行模式（禁止期间做写操作等不安全动作）。
- **1768 行** `ResetPerTupleExprContext`：重置 per-tuple 内存上下文，上一行
  求值产生的所有临时内存（函数结果、detoast 副本……）一次性归零。
  **PG 不做每行 malloc/free 配对，靠的是这个行级内存池**——表达式代码可以
  随手 palloc 而不写 free，泄漏被上下文重置兜底。
- **1773 行** `slot = ExecProcNode(planstate)`：唯一一次拉取——从根拉一行，
  根再拉孩子，递归直达扫描节点。整棵树的执行压缩在这一行代码里。
- **1779-1780 行**：根返回空 slot 即全树耗尽，出主循环。
- **1790-1791 行** junk filter：剔除"垃圾列"——不在 SELECT 列表里的 ORDER BY
  键、`FOR UPDATE` 需要的 ctid 等。这些列必须流经计划树（否则上层没法用），
  但不能发给客户端。注释特意记录了一个历史 bug：早年把干净行写回原 slot，
  因描述符不匹配而出错。
- **1797-1806 行**：`receiveSlot` 把行交给 DestReceiver（编码发给客户端 / 存进
  tuplestore / 喂给 INSERT INTO ... SELECT 的上层）。返回 false 表示对端
  关闭，提前收工。
- **1813-1814 行**：只有 SELECT 在这里计数；INSERT/UPDATE/DELETE 的
  `es_processed` 由 ModifyTable 节点自己维护（它才知道真正写了几行）。
- **1821-1823 行**：取够 `numberTuples`（游标 FETCH 份额）就 break——
  **Portal 挂起点**，下次 FETCH 从这里继续。
- **1830-1831 行**：确定不会反向回退时，提前 `ExecShutdownNode`（
  `execProcnode.c:753`）释放可以早放的资源——最重要的是并行 worker（让
  EXPLAIN ANALYZE 能收齐 worker 的统计）与 Gather 的共享内存。
- **1833-1834 行**：退出并行模式。

### 3.5 ExecutorFinish 与 ExecutorEnd：收台

`ExecutorFinish`（`execMain.c:417`）跑 AFTER 触发器等收尾（单独拆出是为了让
EXPLAIN ANALYZE 计入这段耗时）；`ExecutorEnd`（`execMain.c:477`）→
`standard_ExecutorEnd`（486 行）：`ExecEndPlan`（515 行）递归调 `ExecEndNode`
（`execProcnode.c:543`）关闭每个节点 → 注销快照（518-519 行
`UnregisterSnapshot`）→ `FreeExecutorState`（530 行）**整体释放 per-query
内存上下文**——PG 没有执行器内存泄漏顽疾的原因：不靠人肉配对 free，靠上下文
整树销毁。错误路径（elog(ERROR) 长跳转）下这套 free 根本不会被执行，那谁来
清理？答案是 ResourceOwner + 内存上下文的事务级销毁，见第 8.2 节。

### 小结

三阶段 = open/next/close 在查询级别的放大：`ExecInitNode` 大 switch 镜像出
PlanState 树（只读计划/可变状态分离）；`ExecutePlan` 每转一圈从根拉一行，
循环体内藏着并行判定、行级内存池、junk 过滤、游标挂起点四个机制；
`ExecProcNodeFirst` 用"首调自摘除"实现零开销 instrument 注入；四个 hook 是
监控类扩展的挂载点。

## 4. TupleTableSlot：行的抽象容器

### 4.1 三问：为什么需要这层抽象

算子之间传递的不是裸 `HeapTuple`，而是 `TupleTableSlot`。

**① 解决什么问题**：一行数据可能来自堆页面（还在共享缓冲区里、带 MVCC 头）、
排序落盘文件（`MinimalTuple`，无 MVCC 头的瘦身格式）、或表达式现算（只有
Datum 数组，根本没有物理元组）。若算子接口直接用 `HeapTuple`，每个来源都得
先物化成堆元组格式——投影算子每行都要 `heap_form_tuple` 一次，纯粹浪费。

**② 反事实：没有 slot 抽象，Table AM 会卡在哪**。PG 12 引入可插拔存储引擎
（commit `8586bf7ed8`，2019-03，"tableam: introduce table AM infrastructure"）。
执行器与存储引擎的边界 `table_scan_getnextslot()`（`src/include/access/tableam.h`）
用 slot 作交换货币，物理格式由引擎自带的 `table_slot_callbacks()` 决定
（`nodeSeqscan.c:256` 就是执行器向引擎"要 slot 类型"的现场）。假如接口固定为
`HeapTuple`：列存引擎（如 zedstore 原型）每返回一行都必须把列数据拼装成
带 23 字节 MVCC 头的堆行——不仅拷贝，还强迫所有引擎复刻 heap 的可见性头布局。
**slot 的回调表（`TupleTableSlotOps`）正是为了让"行的物理表示"成为引擎的
私事**。这层抽象是 2018 年底两个铺路 commit 完成的：`1a0586de36`
（"Introduce notion of different types of slots"）与 `4da597edf1`
（"Make TupleTableSlots extensible, finish split of existing slot type"）——
先把单一 slot 拆成四种类型化实现，几个月后 Table AM 才可能落地。读提交史
能看清依赖顺序：**先抽象行，再抽象表**。

**③ 省拷贝不变式**：slot 可以只持有指向缓冲区页面的指针（连带 buffer pin，
保证页面不被驱逐——这就是不变式：slot 存活期间其指向的内存必须有效），需要
长期保存时才 `ExecMaterializeSlot()` 物化成自有拷贝。

### 4.2 四种 slot 实现

回调表定义在 `src/backend/executor/execTuples.c`：

| 类型 | 定义处 | 存的是什么 | 典型场景 |
|---|---|---|---|
| virtual | `execTuples.c:1266` | 仅 Datum/isnull 数组，无物理元组 | 投影输出、算子间传值 |
| heap | `execTuples.c:1287` | 独立 palloc 的 HeapTuple | 触发器、需要完整元组头的场合 |
| minimal | `execTuples.c:1305` | MinimalTuple（去 MVCC 头瘦身格式） | 排序/哈希落盘、tuplestore |
| buffer heap | `execTuples.c:1323` | 指向共享缓冲区页面的 HeapTuple + pin | 顺序/索引扫描直读堆页 |

`buffer heap slot` 最能体现设计意图：扫描返回的行**根本没离开缓冲池**，slot 里
只是一个指针加一个 pin 计数；WHERE 淘汰掉的行零拷贝零分配。

### 4.3 取列的调用链：slot_getattr → getsomeattrs → deform

堆元组是**变长紧凑编码**：变长列（text 等）带长度头、可能 TOAST，NULL 列不占
字节。要取第 n 列，必须从第一列开始逐个跳——这个"解包"过程叫 **deform**，
是宽表查询的头号 CPU 热点之一。PG 的对策是**惰性 + 增量 + 有偏移缓存**。

入口是内联函数 `src/include/executor/tuptable.h:417`：

```c
	if (attnum > slot->tts_nvalid)          /* 已解包的列数不够？ */
		slot_getsomeattrs(slot, attnum);    /* 补解到第 attnum 列 */
	*isnull = slot->tts_isnull[attnum - 1];
	return slot->tts_values[attnum - 1];    /* 命中缓存，直接取数组 */
```

`tts_nvalid` 是该 slot **已解包到第几列**的水位线；不够时经
`slot_getsomeattrs`（`tuptable.h:376` → `execTuples.c:2177` 的
`slot_getsomeattrs_int`）虚调用 `slot->tts_ops->getsomeattrs`（heap slot 落到
`tts_heap_getsomeattrs`，`execTuples.c:346`；buffer heap slot 落到
`tts_buffer_heap_getsomeattrs`，`execTuples.c:751`），二者都汇入核心函数
`slot_deform_heap_tuple`。

### 4.4 slot_deform_heap_tuple 全函数走读

函数在 `execTuples.c:1016-1264`。头注释（996-1014 行）自述其身份：
"essentially an incremental version of heap_deform_tuple"。签名：

```c
static pg_always_inline void
slot_deform_heap_tuple(TupleTableSlot *slot, HeapTuple tuple, uint32 *offp,
					   int reqnatts, bool support_cstring)
```

`pg_always_inline` + `support_cstring` 常量参数是编译期特化：MinimalTuple
里可能出现 cstring 列而 HeapTuple 不会，调用点以常量传入，编译器把不需要的
分支整体删除（1204-1209 行注释解释了这一点）——与第 5.1/5.3 节的手法同源。

**第一段：三个分界点**（1034-1051 行）。函数把 0..reqnatts 的列空间切成
若干区段，每个区段用不同强度的快径：

- `firstNonGuaranteedAttr`（1049 行）：`Min(reqnatts, slot->tts_first_nonguaranteed)`
  ——受 NOT NULL 约束保证**必然存在且非空**的前缀列数（`ExecInitScanTupleSlot`
  传入 `TTS_FLAG_OBEYS_NOT_NULL_CONSTRAINTS` 时启用，见 `nodeSeqscan.c:257`）。
  这段列连元组头里的 natts 都不用读。
- `firstNonCacheOffsetAttr`（1051 行）：来自 `tupleDesc->firstNonCachedOffsetAttr`
  ——**预计算的偏移缓存分界点**。推理：第 k 列的字节偏移是常量，当且仅当它
  前面全部列都是定长且（到此为止）无 NULL。表定义确定后这个分界点就固定了，
  于是由 `TupleDescFinalize()` 一次性算好每列的 `attcacheoff`（1035 行的
  Assert 检查它已被调用）。这是 2026-03 commit `c456e39113`（"Optimize tuple
  deformation"）的核心改动：以前 attcacheoff 是 deform 时边走边填的，现在
  彻底移出热路径。
- `firstNullAttr`：本元组第一个 NULL 列。

**第二段：NULL 位图处理**（1053-1104 行）。有 NULL 的元组（1053 行
`HeapTupleHasNulls`）：`first_null_attr(bp, natts)` 找到第一个 NULL（1065 行），
`populate_isnull_array` 一次性按位图填好 isnull 数组（1071 行）——同样来自
`c456e39113`，把"逐列查位图"改成"一次批量展开"。无 NULL 的元组：
`firstNullAttr = natts`，全部列可线性解包；且若 `reqnatts` 不超过保证前缀，
连 `HeapTupleHeaderGetNatts(tup)` 都省了（1094-1100 行）。

**第三段：接上次的断点**（1106-1155 行）。`attnum = slot->tts_nvalid` 从水位线
继续；1144-1155 行取回 `off = *offp`（slot 里的 `tts_off`）——**上次解包停在
数据区的哪个字节**，本次续解，这就是"增量"。保证前缀内的列走 1120-1143 行的
极简循环：全部定长、byval、偏移直取（`off = cattr->attcacheoff`），没有
NULL 检查也没有对齐计算。

**第四段：三个递进的解包循环**（1157-1247 行）：

```c
	/* We can use attcacheoff up until the first NULL */
	firstNonCacheOffsetAttr = Min(firstNonCacheOffsetAttr, firstNullAttr);
```

1. **缓存偏移段**（1165-1188 行）：`off = cattr->attcacheoff` 直接定位，
   `fetch_att_noerr` 取值——每列成本 ≈ 一次数组读 + 一次内存读；
2. **变长无 NULL 段**（1195-1217 行）：不能再用缓存偏移（前面出现过变长列），
   `align_fetch_then_add` 对齐 off → 取值 → 把 off 推进过本列——经典的
   "逐列跳"，但仍不用查 NULL；
3. **NULL 混杂段**（1224-1247 行）：与上一循环唯一的区别是先查
   `isnull[attnum]`，NULL 列置 0 跳过。

三个循环的内容几乎相同，为什么写三遍？——**把最热路径上的每一个不必要分支
都移到循环外**。deform 是 per-row × per-column 级别的热点，一个多余的
`if (isnull)` 在千万行 × 几十列下就是数亿次分支。

**第五段：收尾**（1249-1263 行）：若元组实际列数少于 reqnatts（ALTER TABLE
ADD COLUMN 后的旧行），`slot_getmissingattrs` 补默认值；最后 `*offp = off`
存回断点，供下次增量续解。

### 4.5 量化推演：没有惰性 deform，宽表查询多付多少

设表 100 列、平均每列解包成本 c（一次对齐计算 + 长度判断 + 内存读，量级几 ns），
查询 `SELECT c2 FROM t` 扫 1000 万行：

- 惰性方案：表达式编译期算出"最大被引列 = 2"（`EEOP_SCAN_FETCHSOME` 只解到
  第 2 列，见第 6 节），总成本 ≈ 10^7 × 2c；
- 全量方案：每行解 100 列，总成本 ≈ 10^7 × 100c——**多付 50 倍的 deform CPU**，
  对 c ≈ 2-3ns 就是每查询多烧约 2-3 秒纯 CPU。

推论 1：`SELECT c99 FROM t` 无法从惰性中获益（必须解到 99），所以
**常用列放表定义前面确实更快**——PG DBA 的经典调优项在这里有了源码级依据。
推论 2：若 c1..c98 全是定长非 NULL 列，c99 的偏移是缓存的，第四段循环 1 直取
——"前面的列都定长"与"常用列靠前"同样有效。

### 小结

TupleTableSlot 用回调表把"行"抽象成四种实现，既是算子间通用货币，也是执行器
与 Table AM 的隔离层（没有它就没有可插拔存储引擎）。deform 是取列的真实成本，
PG 用 `tts_nvalid` 水位线 + `tts_off` 断点续解 + FETCHSOME 预解包 + 预计算
偏移缓存（`c456e39113`），把它做成惰性、增量、只付一次、热循环无冗余分支的操作。

### 5.2 nodeNestloop.c 与参数化内表：rescan 机制

`ExecNestLoop`（`src/backend/executor/nodeNestloop.c:61`）是教科书双重循环，
但 PG 的价值在**参数化内表**（106-152 行，逐字摘录关键段）：

```c
		if (node->nl_NeedNewOuter)
		{
			ENL1_printf("getting new outer tuple");
			outerTupleSlot = ExecProcNode(outerPlan);
			...
			foreach(lc, nl->nestParams)
			{
				NestLoopParam *nlp = (NestLoopParam *) lfirst(lc);
				int			paramno = nlp->paramno;
				ParamExecData *prm;

				prm = &(econtext->ecxt_param_exec_vals[paramno]);
				...
				prm->value = slot_getattr(outerTupleSlot,
										  nlp->paramval->varattno,
										  &(prm->isnull));
				/* Flag parameter value as changed */
				innerPlan->chgParam = bms_add_member(innerPlan->chgParam,
													 paramno);
			}
			...
			ExecReScan(innerPlan);
		}
		...
		innerTupleSlot = ExecProcNode(innerPlan);      /* 160 行 */
```

129-146 行把外表当前行的连接列值（用第 4 节的 `slot_getattr` 取）塞进
`PARAM_EXEC` 参数槽，并把参数号记入内计划的 `chgParam` 位图；152 行
`ExecReScan(innerPlan)` **重置内子树**从头再产出。

**三问**：

① **没有参数化内表，NestLoop+Index 计划形态就不存在**。对比两个计划：

```
-- 有参数化：内表按外表当前行的值下降 B 树，每外行 O(log N)
Nested Loop
  ->  Seq Scan on orders o
  ->  Index Scan using pk_customers on customers c
        Index Cond: (id = o.customer_id)        ← 引用外表列的参数!

-- 无参数化：内表只能全量产出，退化为笛卡尔积过滤，每外行 O(N)
Nested Loop
  Join Filter: (c.id = o.customer_id)
  ->  Seq Scan on orders o
  ->  Seq Scan on customers c
```

`Index Cond: (id = o.customer_id)` 里的 `o.customer_id` 就是 PARAM_EXEC 槽：
没有"外行的值传给内子树"的通道，索引条件无从谈起，OLTP 最重要的连接方式
（index lookup join，MySQL 的默认连接）直接消失。

② **为何用 chgParam 位图 + 延迟重扫，而不是直接调用内节点的 reset 函数**？
因为内子树可能很深（IndexScan 上面还压着 Materialize、Sort、Memoize……），
"参数变了"的影响面只有 planner 知道。`ExecReScan` 递归下推时，每个节点检查
`chgParam` 与自己的依赖交集：Materialize/Sort 发现参数没变到自己头上，
**直接重放缓存**而不重跑孩子；planner 还可能在内侧插 `Memoize`
（commit `b6002a796d`，2021-04，原名 Result Cache，`83f4fcc655` 改名）按
参数值缓存内表结果，重复外键直接命中。3.3 节 `ExecProcNode` 内联里的
`if (node->chgParam != NULL) ExecReScan(node)` 是配套机制：重扫被**推迟到
下一次真正拉取时**才做，避免"参数设了但这个分支根本没被执行"的浪费。

③ **rescan 语义契约（不变式）**：`ExecReScan(node)` 之后，该子树必须表现得
如同刚 `ExecInitNode` 完——从第一行重新产出完整结果集（在新参数下）。缓存类
节点满足契约的方式是"重放"，扫描类节点是"重新 beginscan"，HashJoin 则有
精细的分档（`ExecReScanHashJoin`，`nodeHashjoin.c:1651`）：单批次且内子树参数
未变（1680-1681 行 `nbatch == 1 && innerPlan->chgParam == NULL`）时**保留整张
哈希表**只重扫外表（右连接时顺带重置匹配标志，1690-1691 行）；否则摧毁重建。
这是"契约允许任何不改变可见结果的捷径"的典型示例。

### 5.3 nodeHashjoin.c：hybrid hash join 状态机（全走读）

#### 5.3.1 为什么是状态机

主体 `ExecHashJoinImpl`（`nodeHashjoin.c:225`）不写教科书式嵌套循环，因为
**Volcano 要求每次调用只返回一行**：函数必须能在任意匹配点 `return`，下次
被调时恢复现场。把控制流显式存进 `node->hj_JoinState`，是 C 语言实现
"可恢复协程"的标准做法。函数被 `pg_always_inline` 标注，由 `ExecHashJoin`
（802 行）/`ExecParallelHashJoin`（818 行）以 `parallel` 常量实例化成两个
特化版本——编译器把 `if (parallel)` 分支整体消除，又一次编译期特化。

状态集（`nodeHashjoin.c:182-189`，逐字）：

`nodeHashjoin.c:1279`。四步：

1. **收尾上一批**（1291-1314 行）：关闭并置空上一批的外表文件；若刚结束的是
   batch 0，**退役 skew 优化**（1309-1313 行：`skewEnabled = false` 等）——
   skew 桶只服务于第一批。
2. **跳过可跳的批次**（1316-1360 行）。头注释给出三条不可跳规则（逐字）：
   *规则 1*——左/全外连接必须处理只有外行的批次（补 NULL），右/右反/全外
   必须处理只有内行的批次；*规则 2*——`nbatch != nbatch_original` 时必须处理
   只有内行的批次：重建哈希表时这些行可能再被推向更晚批次；*规则 3*——
   `nbatch != nbatch_outstart` 时必须处理只有外行的批次，理由对称。两边文件都
   空、或只缺不受规则保护的一边 → 直接 `BufFileClose` 释放并跳过（1352-1358 行）。
3. **重建哈希表**（1362-1399 行）：`ExecHashTableReset` 清桶，回放
   `innerBatchFile[curbatch]`——`BufFileSeek` 倒带（1376 行），循环
   `ExecHashJoinGetSavedTuple` → `ExecHashTableInsert`（1381-1391 行）。注释
   （1386-1389 行）强调：**插入过程中 nbatch 仍可能增长**，部分行会被继续推向
   更晚批次——这正是规则 2 存在的原因。装完即关闭删除内表批文件。
4. **倒带外表批文件**（1404-1410 行），供 `ExecHashJoinOuterGetTuple` 顺序读。

类比 Spark shuffle hash join："批次"即分区，只是分区文件在本机临时目录而非
跨网络 shuffle；PG 串行版还多一个"边跑边把行往后扔"的自由度（Spark/并行 PG
都必须先分好区才能各自独立处理）。

#### 5.3.4 ExecHashJoinOuterGetTuple 与 skew 全路径

`nodeHashjoin.c:1110`。batch 0 时从真正的外子计划拉行（首行可能是
BUILD_HASHTABLE 试探时存下的 `hj_FirstOuterTupleSlot`，1124-1126 行），对每行
求哈希（1143-1145 行，`hj_OuterHash` 是编译好的哈希表达式——commit
`adf97c1562` 把哈希计算也搬进了表达式虚拟机）；**键为 NULL 的行在此截流**
（1155-1161 行）：需要补 NULL（左/全外）就存进 `hj_NullOuterTupleStore`，
否则直接丢弃，然后继续拉下一行——它们从不进入哈希/分批管线。batch > 0 时
从 `outerBatchFile[curbatch]` 读暂存行（1170-1187 行），哈希值随行存储无需重算。

**skew 优化**（防倾斜）：`ExecHashBuildSkewHash`（`nodeHash.c:2429`）建表时按
**外表连接列的 MCV 统计**为最高频值单独建 skew 桶，预算约 2% 哈希内存
（`SKEW_HASH_MEM_PERCENT`，`hashjoin.h:132`；扣减在 `nodeHash.c:544-545`）。
命中 skew 桶的外行无论属于哪个批次都在 batch 0 就地完成连接
（`nodeHashjoin.c:507-508` 的豁免条件），避免最热的键被反复写盘回放；第一批
结束后 skew 桶退役（5.3.3 第 1 步）。与 Spark AQE skew split 目标一致：
PG 靠统计**前置规避**，Spark 靠运行时**切分补救**。

#### 5.3.5 不变式："同 batch 必相遇"及其证明

**命题**：任意一对满足连接条件的 (外行 o, 内行 i)，在算法执行中必然在同一个
批次的内存哈希表中相遇恰好一次。

证明分三步：

1. *同哈希、同批号*。连接条件成立 ⇒ 连接键相等 ⇒ 哈希值相等（同一哈希函数族，
   两侧对应参数用同一 operator family 的哈希）。批号是哈希值的纯函数
   （`ExecHashGetBucketAndBatch`，`nodeHash.c:1994-1998`：
   `*batchno = pg_rotate_right32(hashvalue, log2_nbuckets) & (nbatch - 1)`），
   所以**在任一时刻的 nbatch 下**，o 与 i 算出的批号相同。
2. *批号随 nbatch 翻倍单调不减*。nbatch 只按 2 的幂翻倍（文件头 35-36 行注释），
   翻倍相当于给批号掩码在高位加一位：新批号 = 旧批号 或 旧批号 + 旧 nbatch，
   恒 ≥ 旧批号，且 新批号 mod 旧 nbatch = 旧批号。因此一个行的批号在执行过程中
   只会"向后跳"，绝不会跳回已处理过的批次——这正是代码里两处
   `Assert(batchno > curbatch)`（`nodeHashjoin.c:519`、`nodeHash.c:1172`）
   守护的性质。
3. *处理顺序覆盖*。批次严格按 0,1,2,... 递增处理（`ExecHashJoinNewBatch`
   的 `curbatch++`）。设批次数最终稳定为 N。内行 i：要么最终留在其批号 b_i 的
   哈希表里，要么被逐级推入 `innerBatchFile[b_i]` 并在处理 b_i 时装载（规则 2
   保证 nbatch 中途增长时 b_i 批文件不会被跳过）。外行 o：在处理批 b < b_o 时
   被写入 `outerBatchFile[b_o]`（Assert 保证只向后写），处理 b_o 时被回放
   （规则 3 保证不被跳过）。于是 o 与 i 都恰好在批 b_o = b_i 出现于"外行回放 ×
   内存哈希表"这对组合中，桶扫描保证相遇。skew 行是显式豁免：命中 skew 桶的
   外行与 MCV 内行都被固定在 batch 0 的 skew 桶中，配对关系整体搬迁，不破坏
   命题。∎

这个不变式就是 hybrid hash join 正确性的全部：**分批是按哈希值的确定性划分，
且划分只会细化、处理只会向前**。

#### 5.3.6 hybrid vs Grace：差在哪

Grace hash join（80 年代 Grace 机器）：先把两侧**全部**按哈希分区写盘，再逐
分区装载连接——哪怕内存装得下也要写盘一轮。hybrid（Zeller & Gray 1990，文件头
15-24 行注释引用）：batch 0 常驻内存，边建表边连接，**内存够就完全不碰磁盘，
不够时只有溢出部分走 Grace 路径**——两个算法的线性组合，按内存水位自动滑动。
PG 再叠加两层弹性：运行期批次翻倍（估错时自救，见第 8.1 节）与 skew 桶
（热键豁免写盘）。

### 5.4 nodeAgg.c：分组聚合主线与哈希聚合落盘

#### 5.4.1 入口分发

`ExecAgg`（`nodeAgg.c:2247`，逐字）：


- **AGG_SORTED**（分组聚合）：输入已按 GROUP BY 有序（下面通常压着 Sort 或

- **AGG_HASHED**：无序输入，哈希表按分组键持有每组**转移状态**（如 sum 的

- **AGG_PLAIN**：无 GROUP BY，全表一组。**AGG_MIXED**：GROUPING SETS 同时需要

#### 5.4.2 agg_retrieve_direct 全函数走读

`nodeAgg.c:2283-2623`。这是 AGG_PLAIN/AGG_SORTED（以及 GROUPING SETS 多阶段）
的主线。函数结构是一个大 `while (!aggstate->agg_done)`（2323 行），每圈产出
一"组"。按执行顺序拆解：

**(a) 组边界的内存复位**（2337-2358 行）：`ReScanExprContext(econtext)` 清
per-output-tuple 上下文，再按 `numReset` 清相应 grouping set 的
`aggcontexts[i]`——**上一组的转移状态（pass-by-ref 的 sum 累计值等）在这里
整体释放**，不靠逐个 pfree。注释强调用 `ReScanExprContext` 而非
`ResetExprContext` 是为了触发聚合函数注册的 shutdown 回调（清理非内存资源）。

**(b) 输入耗尽后的收尾分派**（2364-2394 行）：`input_done && 所有 set 已投影`
时三选一——还有下一阶段（GROUPING SETS 的多轮排序）则
`initialize_phase(aggstate, current_phase + 1)` 继续；AGG_MIXED 则切回
phase 0、置 `table_filled = true`，**当场改道**
`return agg_retrieve_hash_table(aggstate)`（2387 行）——排序分组吐完了，
轮到哈希表；否则 `agg_done = true` 收工。

**(c) GROUPING SETS 子组边界判断**（2396-2437 行）：拿上一行与当前行比较
"下一个 grouping set 的分组列"（`ExecQualAndReset(phase->eqfunctions[...])`，
2430 行），不等则说明更粗粒度的小计（如 ROLLUP 的 (a) 层）也到了边界，
`projected_set += 1` 再投影一轮——**同一行输入可以触发多层小计输出**。
（只有 GROUP BY 的简单查询走 else 分支：`projected_set = 0`。）

**(d) 取新组首行**（2451-2505 行）：`grp_firstTuple` 是跨调用暂存的"下一组
第一行"（上一圈越界读到的那行）。为空则 `fetch_input_tuple` 真拉一行；拉不到
说明输入彻底为空——GROUPING SETS 还要为空集投影（2465-2496 行：size 0 的
grouping set 即 `GROUP BY ()`，空输入也要出一行，如 `count(*) = 0`），普通
分组则直接结束（2497-2503 行：AGG_PLAIN 空输入也要出一行，故不 return；
带 GROUP BY 则零组零行）。

**(e) 组内累加内循环**（2510-2586 行）——真正的热路径：

```c
			initialize_aggregates(aggstate, pergroups, numReset);   /* 2510 行 */
			...
				for (;;)
				{
					...                                    /* AGG_MIXED: 2536-2540 行
					                                          phase 1 时同时喂哈希表 */
					advance_aggregates(aggstate);          /* 2543 行 */
					ResetExprContext(tmpcontext);          /* 2546 行 */
```

`advance_aggregates` 执行的是编译好的转移表达式（`EEOP_AGG_PLAIN_TRANS_*`
指令，见第 6 节）：对每个聚合调用转移函数
`transvalue = transfn(transvalue, input)`。组边界检测（2576-2585 行）用编译好
的等值比较表达式比较"组首行 vs 当前行"的全部分组列；不等则把当前行拷出来存进
`grp_firstTuple`（2582 行）——它属于下一组，本圈到此为止。

**(f) 投影产出**（2597-2618 行）：`finalize_aggregates` 调 final 函数（如
avg 的 sum/count 相除），`project_aggregates` 过 HAVING（`aggstate->ss.ps.qual`）
并投影；HAVING 不过则 continue 下一组而非返回 NULL（2612-2616 行注释——
返回 NULL 会被上层误读为"聚合结束"，这是 Volcano 接口"NULL 即 EOF"约定的
一个微妙推论）。

**三问**：①流式分组解决"分组数无界但内存有限"——每组 O(1) 内存的代价是要求
输入有序；②为何不像哈希聚合那样统一哈希？当分组基数巨大（哈希表装不下）或
上游本来有序（索引扫描）时，排序分组严格更优，planner 按代价二选一——执行器
提供两条实现，选择权在优化器；③不变式——`grp_firstTuple`/`input_done`/
`projected_set` 三个字段完整编码了"读到哪、投影到哪"，任何一次 return 后都能
恢复；组内存清理以 grouping set 为粒度批量进行。

#### 5.4.3 哈希聚合：填表 → 检查水位 → 溢出 → 回灌（完整闭环）

**填表** `agg_fill_hash_table`（`nodeAgg.c:2629`）：循环

**查表** `lookup_hash_entries`（`nodeAgg.c:2184`）对每个 hashed grouping set：

`hash_agg_enter_spill_mode`（1910 行）：置 `hash_spill_mode = true`、重编译

`LogicalTapeSet`（1923 行，与 tuplesort 共用的"多个逻辑文件复用一个物理临时
文件"抽象）并为每个 grouping set 初始化分区好的 spill（1927-1935 行）。

内存中的组吐完 → `agg_refill_hash_table`（`nodeAgg.c:2683`）从批栈弹出一个
spill 分区重建哈希表：按该分区**实测基数**重算内存/组数上限

`hashagg_recompile_expressions`（2742 行，spill 行读回来是 MinimalTuple，

次收益）→ 逐行 `hashagg_batch_read`（2758 行）重放：能进表就累加（2771-2777
行），**还是装不下就再次 spill，且分区位数递增**（2780-2794 行：
`batch->used_bits` 传入 `hashagg_spill_init`，递归细分）→ 批处理完切回
phase 0、重置迭代器（2806-2822 行），上层继续吐这一批的组。

**不变式：内存记账何时保证不 OOM**。论证链：(1) 触发条件在**每新增一组**后
检查，超限立即停止建新组——内存增长的主力（新组键 + 新转移状态）被闸住；
(2) spill 模式下唯一还会增长的是已有组的转移状态变大（如 `array_agg`
攒数组）——1858-1863 行头注释坦率承认 "this check is imperfect"：PG 保证的是
"组数 × 平均组开销"不失控，**不承诺**对病态转移状态（单组吃掉全部内存）也
兜底；(3) 递归回灌每层分区位数增加、每批至少含一个组（"at least one group"
护条保证前进性），所以递归深度有限、必然终止。对 sum/count/avg 这类 O(1)
转移状态的常规聚合，这套机制给出硬上限——这就是 PG 13（commit `1f39bce021`，
"Disk-based Hash Aggregation"，2020-03）修掉"哈希聚合估错基数就 OOM"这一
历史顽疾的方式。EXPLAIN ANALYZE 中 `Batches: N  Disk Usage: ...` 即此路径。

### 5.5 nodeSort.c → tuplesort.c：外排序状态机（主线全走读）

#### 5.5.1 nodeSort：物化型算子的极简样板

`ExecSort`（`nodeSort.c:50`）首次被拉取时（75 行 `if (!node->sort_Done)`）
进入建库阶段：`tuplesort_begin_heap`（114 行；单列时 105-106 行走
`tuplesort_begin_datum` 快径，datum sort 免元组头开销）→ `ORDER BY ... LIMIT k`
时 `tuplesort_set_bound`（124 行 → `tuplesort.c:735`）声明只要前 k 名 →
`for (;;)` 把子节点**全部**拉完喂给 `tuplesort_puttupleslot`（147-154 行）→
`tuplesort_performsort`（160 行）→ 置 `sort_Done`（170 行）。此后每次调用只是
`tuplesort_gettupleslot`（206 行）取下一行。pipeline breaker 的标准形状：
**首次 next 吃掉全部输入，之后 next 只是回放**。

#### 5.5.2 tuplesort 状态机

真正的排序引擎在 `src/backend/utils/sort/tuplesort.c`，状态枚举
（`tuplesort.c:155-160`，逐字）：

```c
	TSS_INITIAL,				/* Loading tuples; still within memory limit */
	TSS_BOUNDED,				/* Loading tuples into bounded-size heap */
	TSS_BUILDRUNS,				/* Loading tuples; writing to tape */
	TSS_SORTEDINMEM,			/* Sort completed entirely in memory */
	TSS_SORTEDONTAPE,			/* Sort completed, final run is on tape */
	TSS_FINALMERGE,				/* Performing final merge on-the-fly */
```

转移图：

```
            attsort/内存足够            performsort
TSS_INITIAL ──────────────────────────────────────→ TSS_SORTEDINMEM
    │  bounded 且行数超 2×bound                        （纯内存快排）
    ├──────────────→ TSS_BOUNDED ──performsort──→ TSS_SORTEDINMEM（堆转数组）
    │  内存爆
    └──inittapes──→ TSS_BUILDRUNS ──performsort→ mergeruns ──┬→ TSS_FINALMERGE
                    （边排边写 run）                          └→ TSS_SORTEDONTAPE
```

**喂入 `tuplesort_puttuple_common`**（`tuplesort.c:1066`）。入口先处理
abbreviated key（首键缩略键：把首个排序键压缩成一个可快速比较的 Datum，比较
命中即免全键比较；1088-1106 行还有 `consider_abort_common` 的自适应放弃——
缩略后区分度太差就撤销）。然后按状态分派：

- `TSS_INITIAL`（1110 行起）：存入 `memtuples` 数组（满了 `grow_memtuples`
  翻倍，1119-1123 行）。两个转移检查：**转 bounded**（1138-1148 行）——设了
  bound 且 `memtupcount > bound * 2`，`make_bounded_heap` 把数组转成 k 大小的
  堆，进入 top-k 模式；**转外排**（1154-1168 行）——`LACKMEM` 内存超限，
  `inittapes(state, true)` + `dumptuples(state, false)`。
- `TSS_BOUNDED`（1171-1193 行）：新行与堆顶比较（方向反转的堆），比堆顶还差
  直接丢弃（1181-1186 行，**大多数输入行只花一次比较**），否则替换堆顶——
  `ORDER BY x LIMIT 10` 在千万行上只维护 10 元素的堆，内存 O(k)。
- `TSS_BUILDRUNS`（1195-1206 行）：追加进数组，`dumptuples(state, false)`
  在超限时把当前内存整体排好写成一个 run。

**`inittapes`**（`tuplesort.c:1762`）：`tuplesort_merge_order(allowedMem)` 算
归并路数（内存越大路数越多），建 `LogicalTapeSet`（1784-1787 行——多条逻辑
磁带复用一个物理临时文件，避免几百个真实文件），置 `TSS_BUILDRUNS`，
`selectnewtape` 选首条输出磁带（1845 行起：前 maxTapes 个 run 各占新磁带，
之后轮转追加）。

**`dumptuples`**（`tuplesort.c:2204`）：未超限且非最终调用则直接返回
（2213-2215 行）；否则把内存中全部行**快排**（2254 行
`tuplesort_sort_memtuples`）后顺序 `WRITETUP` 写到当前磁带（2261-2267 行），
`markrunend` 封口（2287 行），并整体重置元组内存上下文（2278 行，注释解释：
避免 size class 碎片）。**每个 run 都是一段已排序数据**——这是 2017 年
commit `8b304b8b72`（"Remove replacement selection sort"）之后的简化世界：
老算法用堆做"替换选择"以生成约 2 倍长的 run，但堆的随机访问对 CPU cache
极不友好，实测不如"攒满内存→快排→整段写盘"，于是教科书经典被删除。

**`tuplesort_performsort`**（`tuplesort.c:1260`）按状态收官（逐字节选）：

```c
		case TSS_INITIAL:
			if (SERIAL(state))
			{
				/* Sort in memory and we're done */
				tuplesort_sort_memtuples(state);
				state->status = TSS_SORTEDINMEM;
			}
			...
		case TSS_BOUNDED:
			sort_bounded_heap(state);
			...
		case TSS_BUILDRUNS:
			dumptuples(state, true);
			mergeruns(state);
			...
```

（并行 worker/leader 分支：worker 把唯一 run 落带交给 leader，leader
`leader_takeover_tapes` 接管全部 worker 磁带后归并，1282-1301 行。）

**`mergeruns`**（`tuplesort.c:1914`）——平衡 k 路归并：

- 先永久禁用缩略键（1921-1935 行：落盘行没存缩略键，读回后不值得重算）；
- 释放大 memtuples 数组、启用 **slab 分配器**（1941-1967 行：归并阶段每条
  输入磁带恰好占一个元组槽位，内存需求是确定的，slab 免碎片免记账）；
- 重建小 memtuples 作为 k 元素**败者堆**（1977-1980 行），剩余内存全部划给
  磁带读缓冲（1987-1988 行）；
- 主循环（1993-2076 行）：每轮把上一轮的输出磁带变成输入磁带（2013-2016 行），
  `mergeonerun`（2097 行）从每条输入磁带取一个 run 归并成一个更长的 run 写到
  输出磁带；**在线归并短路**（2051-2061 行，逐字条件）：

```c
			if ((state->base.sortopt & TUPLESORT_RANDOMACCESS) == 0
				&& state->nInputRuns <= state->nInputTapes
				&& !WORKER(state))
			{
				...
				beginmerge(state);
				state->status = TSS_FINALMERGE;
				return;
			}
```

  ——不需要随机访问、且剩余 run 数一轮能并完时，**停在 TSS_FINALMERGE 不做
  最后一次全量写盘**，把归并推迟到取行时在线进行；
- 否则并到只剩单 run 单带，`LogicalTapeFreeze` 后置 `TSS_SORTEDONTAPE`
  （2081-2086 行）。

这套"简单平衡 k 路归并"来自 2021-10 commit `65014000b3`（"Replace polyphase
merge algorithm with a simple balanced k-way merge"）：Knuth 的多相归并
（polyphase）靠斐波那契分布的 run 摆放省磁带倒带，那是**磁带机时代**的优化；
在磁盘/SSD 上 LogicalTape 可以 O(1) "倒带"，多相的复杂度只剩负担。两条
tuplesort 历史（8b304b8b72、65014000b3）合起来是同一主题：**当硬件前提变化，
教科书算法的常数假设失效，简单方案反而更快**。

**取行 `tuplesort_gettuple_common`**（`tuplesort.c:1367`）三个终态三条路：
`TSS_SORTEDINMEM`（1377 行）数组下标 `current++`（支持回退/mark-restore）；
`TSS_SORTEDONTAPE`（1421 行）从结果磁带顺序 READTUP；`TSS_FINALMERGE`
（1536-1592 行）——堆顶弹出即最小行，从该行的来源磁带 `mergereadnext` 补一行
`tuplesort_heap_replace_top`（1572-1589 行），磁带空则 `heap_delete_top` 并
关闭磁带（1578-1586 行）；`lastReturnedTuple` 的 slab 槽位下次调用时回收
（1545-1549 行）。**上层每拉一行才归并推进一步**——pipeline breaker 的后半段
其实是流式的。

`trace_sort = on` 可在日志观察全程；EXPLAIN ANALYZE 的
`Sort Method: external merge  Disk: ...` 即外排路径，`top-N heapsort` 即
TSS_BOUNDED 路径。对照大数据：这是 MapReduce/Spark sort-spill-merge 的单机版，
run ≈ spill 文件，mergeruns ≈ reduce 侧归并。

### 小结

五个算子勾勒出执行器全部套路：SeqScan——三件套 + Table AM 边界 + 运行期特化
（fb9f955025）；NestLoop——参数化与 `chgParam`/`ExecReScan` 重扫契约，索引
连接的存在前提；HashJoin——可恢复状态机 + hybrid 分批 + "同 batch 必相遇"
不变式 + skew/null-key 豁免通道；Agg——排序/哈希两形态与 spill-refill 闭环、
内存记账；Sort/tuplesort——六状态机与两次"硬件淘汰教科书算法"的历史。所有
落盘算子共享同一套 BufFile/LogicalTape 临时文件基础设施（清理见第 8.2 节）。

## 6. 表达式求值虚拟机：JIT 的地基

### 6.1 从表达式树到线性字节码

PG 10 之前，`WHERE a > 10 AND b < 5` 按表达式**树**递归解释执行，每节点一次
函数调用加大量重复分支。2017-03 的 commit `b8d7f053c5`（"Faster expression
evaluation and targetlist projection"）重写为**先编译、后执行**：初始化期把树
展平成一维**步骤数组**（`ExprEvalStep`，含 opcode 和操作数），执行期用小型
虚拟机线性扫过。这一重构的连锁反应贯穿此后十年：次年 `bf6c614a2f` 把分组
等值比较也搬进虚拟机、`432bb9e04d` 加上 LLVM JIT、2024 年 `adf97c1562` 把
哈希计算搬进来（5.3.4 节用到）——**一套字节码，多个后端与越来越多的客户**。

编译入口 `ExecInitExpr`（`src/backend/executor/execExpr.c:143`，逐字节选）：

```c
	/* Insert setup steps as needed */
	ExecCreateExprSetupSteps(state, (Node *) node);

	/* Compile the expression proper */
	ExecInitExprRec(node, state, &state->resvalue, &state->resnull);

	/* Finally, append a DONE step */
	scratch.opcode = EEOP_DONE_RETURN;
	ExprEvalPushStep(state, &scratch);

	ExecReadyExpr(state);
```

`ExecCreateExprSetupSteps`（`execExpr.c:2876`）分析表达式引用了哪些列，生成
`EEOP_*_FETCHSOME` 步骤——即第 4 节"一次性预解包到最大列号"的来源。
`ExecInitExprRec`（`execExpr.c:919`）后序遍历表达式树：子表达式的步骤先入
数组、结果写入约定的 `resvalue/resnull` 槽位，父步骤再消费——**树的数据依赖
被编译成数组的顺序依赖**；AND/OR 短路编译成 `EEOP_JUMP_IF_*` 跳转指令。

定稿函数 `ExecReadyExpr`（`execExpr.c:902`）只有四行，却是 JIT 的总开关：

```c
static void
ExecReadyExpr(ExprState *state)
{
	if (jit_compile_expr(state))
		return;

	ExecReadyInterpretedExpr(state);
}
```

`jit_compile_expr` 在代价超过 `jit_above_cost` 时把**同一份步骤数组**翻译成
LLVM IR 编成机器码，替换 `state->evalfunc`。**JIT 不是另起炉灶，而是同一套
字节码的第二个后端**（`src/backend/jit/llvm/llvmjit_expr.c` 就是逐 opcode
翻译）——"表达式虚拟机是 JIT 地基"的确切含义。对照 Spark：PG 只编译表达式层，
Spark whole-stage codegen 连算子一起编译；PG 的边界选在"每行执行次数最多、
且不含 I/O 的纯计算段"，编译收益密度最高。

### 6.2 分发机制：computed goto 与分支预测原理

解释器在 `src/backend/executor/execExprInterp.c`。分发由一组宏切换
（`execExprInterp.c:104-131`，逐字节选）：

```c
#if defined(EEO_USE_COMPUTED_GOTO)
...
#define EEO_SWITCH()
#define EEO_CASE(name)		CASE_##name:
#define EEO_DISPATCH()		goto *((void *) op->opcode)
#define EEO_OPCODE(opcode)	((intptr_t) dispatch_table[opcode])

#else							/* !EEO_USE_COMPUTED_GOTO */

#define EEO_SWITCH()		starteval: switch ((ExprEvalOp) op->opcode)
#define EEO_CASE(name)		case name:
#define EEO_DISPATCH()		goto starteval
#define EEO_OPCODE(opcode)	(opcode)

#endif							/* EEO_USE_COMPUTED_GOTO */

#define EEO_NEXT() \
	do { \
		op++; \
		EEO_DISPATCH(); \
	} while (0)
```

**为什么 computed goto 比 switch 快——分支预测视角**。普通 switch 版：每条
指令执行完 `goto starteval` 跳回**唯一的**间接跳转点（switch 的跳转表），CPU
的间接分支预测器（BTB）只能为这一个跳转地址维护历史——而它的目标在上百个
case 间随表达式内容跳变，预测命中率接近随机，每次误预测清空流水线（十几个
周期）。computed goto 版：`EEO_NEXT()` 展开为 `op++; goto *op->opcode`——
**每条指令的结尾各有一个独立的间接跳转指令**，预测器可以分别学习"指令 A 之后
通常跟 B"的二元组分布（真实表达式里指令序列高度重复：FETCHSOME→VAR→FUNCEXPR
→QUAL...），命中率大幅上升。CPython（3.x 的 computed goto 主循环）、LuaJIT
解释器同款技巧。GCC/Clang 扩展 `&&label` 取标签地址 + `goto *ptr` 是实现前提，
configure 检测在 commit `bc18126a6b`。

`ExecInterpExpr`（`execExprInterp.c:470`）内定义
`static const void *const dispatch_table[]`（484-606 行）按 opcode 枚举顺序
存放每个 `CASE_EEOP_*` 标签地址（约百条指令），608 行
`StaticAssertDecl(lengthof(dispatch_table) == EEOP_LAST + 1, ...)` 保证与枚举
同步。

更进一步的**直接线索化**（direct threading）：`ExecReadyInterpretedExpr`
（`execExprInterp.c:252`）在 446-453 行遍历步骤数组，执行
`op->opcode = EEO_OPCODE(op->opcode)`——把每个步骤的 opcode 字段**就地改写成
目标标签的机器地址**，执行期 `goto *op->opcode` 连查表都省了。同函数还有两个
机关：276 行 `state->evalfunc = ExecInterpExprStillValid`——首次求值先校验
slot 与表结构仍匹配再换成真正入口（与 `ExecProcNodeFirst` 的"首调自摘除"
如出一辙）；293-438 行 **fast-path**——步骤数极短的高频模式（如
`FETCHSOME + VAR + DONE`）直接绕过整个解释器循环，换成 `ExecJustInnerVar`
等专用小函数：**"启动解释器"本身对三步表达式来说都是可测的开销**。

### 6.3 代表性操作码逐个解说（10 个）

以下行号均为 `execExprInterp.c` 中 `EEO_CASE` 所在行。

**(1) `EEOP_SCAN_FETCHSOME`（662 行；INNER/OUTER 变体 644/653 行）**：

```c
		EEO_CASE(EEOP_INNER_FETCHSOME)
		{
			CheckOpSlotCompatibility(op, innerslot);

			slot_getsomeattrs(innerslot, op->d.fetch.last_var);

			EEO_NEXT();
		}
```

第 4 节惰性 deform 的触发点：编译期算好本表达式引用的最大列号
`last_var`，一次解包到位，后续取列全是 O(1) 数组访问。编译器甚至会在
已知 slot 必为 virtual 时**不生成**此指令（commit `36d22dd95b`）。

**(2) `EEOP_SCAN_VAR`（719 行；INNER/OUTER 689/706 行）**：取第 attnum 列——
`*op->resvalue = scanslot->tts_values[attnum]`，配一个
`Assert(attnum < scanslot->tts_nvalid)` 佐证 FETCHSOME 已把水位线抬够。
两条指令合起来是"解包与取用分离"：解包按最大列号做一次，取用零成本。

**(3) `EEOP_CONST`（907 行）**：把编译期存好的常量写进结果槽——
`*op->resnull = op->d.constval.isnull; *op->resvalue = op->d.constval.value;`。
虚拟机里最便宜的指令，也是 fast-path `ExecJustConst` 的全部内容。

**(4) `EEOP_FUNCEXPR_STRICT_2`（996 行；通用 STRICT 944 行、单参 972 行）**：

```c
		EEO_CASE(EEOP_FUNCEXPR_STRICT_2)
		{
			FunctionCallInfo fcinfo = op->d.func.fcinfo_data;
			NullableDatum *args = fcinfo->args;

			Assert(op->d.func.nargs == 2);

			/* strict function, so check for NULL args */
			if (args[0].isnull || args[1].isnull)
				*op->resnull = true;
			else
			{
				Datum		d;

				fcinfo->isnull = false;
				d = op->d.func.fn_addr(fcinfo);
				*op->resvalue = d;
				*op->resnull = fcinfo->isnull;
			}

			EEO_NEXT();
		}
```

严格函数（多数内置运算符）任一参数为 NULL 则结果 NULL、跳过调用。为 1/2 参
铸造专用指令是因为通用版要跑 `for (argno...)` 循环——930 行上方的注释直白：
函数调用"back operators!"，是最热路径，值得为免循环多开两个 opcode。中间变量
`d` 的存在也有注释解释：防止编译器认为 `*op->resvalue` 可能被被调函数改写而
产生多余的寄存器溢出。**虚拟机性能来自为高频模式铸造专用指令**，此处是最典型
标本。

**(5) `EEOP_BOOL_AND_STEP`（1057 行；FIRST 1045 行、LAST 1073 行）**：AND 的
逐子句求值。子句为 false 立即 `EEO_JUMP(op->d.boolexpr.jumpdone)` 短路；NULL
则记入 `anynull`，LAST 步骤按三值逻辑决定 false/NULL/true。**树形 AND 被编译
成带跳转的直线代码**——短路不再是递归提前返回，而是一条 goto。

**(6) `EEOP_QUAL`（1182 行）**：WHERE 专用的简化 AND——子句为 NULL 或 false
统一改写成 false 并跳到末尾（WHERE 语义里 NULL 即不取，无需三值精确性）。
`ExecQual` 的整个求值就是一串 `..._STEP + EEOP_QUAL` 步骤。

**(7) `EEOP_JUMP_IF_NOT_TRUE`（1228 行）**：`CASE WHEN` 等控制流的基石——
结果非 true 则跳转，否则顺序执行。加上无条件 `EEOP_JUMP`（1204 行）与
`EEOP_JUMP_IF_NULL`（1210 行），树上的一切条件分支都被编译成这三种跳转。

**(8) `EEOP_PARAM_EXEC`（1321 行）**：读 PARAM_EXEC 参数槽（out-of-line 到
`ExecEvalParamExec`）——第 5.2 节 NestLoop 塞进 `ecxt_param_exec_vals[paramno]`
的值在这里被内表的索引条件表达式读走；子计划（InitPlan）的结果也走这里。
参数化执行在表达式层的落点。

**(9) `EEOP_HASHED_SCALARARRAYOP`（1769 行）**：`x IN (大列表)` 的哈希版——
普通 `EEOP_SCALARARRAYOP`（1761 行）对数组线性扫描 O(n)，列表大时编译器改用
此指令，首次执行把数组建成 simplehash 哈希表，之后每行 O(1) 探测。同一语义、
按数据规模选指令的又一例。

**(10) `EEOP_AGG_PLAIN_TRANS_BYVAL`（2167 行）**：聚合转移快径——按
`(setoff, transno)` 定位 pergroup，直接调 `ExecAggPlainTransByVal` 执行
`transvalue = transfn(transvalue, input)`，免去 byref 版本的内存上下文切换与
拷贝判断。第 5.4 节 `advance_aggregates` 每行执行的就是一串这类指令；strict
转移函数的 NULL 输入检查也有专用指令（`EEOP_AGG_STRICT_INPUT_CHECK_ARGS`，
2052 行）。聚合内核 = 表达式虚拟机的特权客户。

### 小结

表达式执行 = 编译期（树 → 线性步骤数组，短路变跳转、取列变 FETCHSOME+VAR）+
运行期（computed-goto / direct-threaded 虚拟机，为高频模式铸造专用指令，
超短表达式再加 fast-path）。`ExecReadyExpr` 是解释与 JIT 的分岔口，JIT 只是
同一字节码的 LLVM 后端。

## 7. EvalPlanQual：READ COMMITTED 下的并发更新重检

### 7.1 问题

事务 A `UPDATE ... WHERE balance > 100` 按快照看到某行 balance=150；提交前
事务 B 已把它改成 50 并提交。READ COMMITTED 下 A 不能报错重试（那是 REPEATABLE
READ 的做法），也不能改旧版本（会丢失 B 的更新）。PG 的答案：**等 B 提交 →
沿版本链找到最新版本并加锁 → 只对这一行重新检查 WHERE**——新版本不再满足则
跳过，仍满足则基于新版本更新。这就是 EvalPlanQual（EPQ），实现在 `execMain.c`
尾部，设计文档见 `src/backend/executor/README` 的 EvalPlanQual 一节。

### 7.2 机制走读

入口 `EvalPlanQual`（`src/backend/executor/execMain.c:2715`，逐字节选）：

```c
	EvalPlanQualBegin(epqstate);

	testslot = EvalPlanQualSlot(epqstate, relation, rti);
	if (testslot != inputslot)
		ExecCopySlot(testslot, inputslot);

	epqstate->relsubs_done[rti - 1] = false;
	epqstate->relsubs_blocked[rti - 1] = false;

	slot = EvalPlanQualNext(epqstate);

	if (!TupIsNull(slot))
		ExecMaterializeSlot(slot);

	ExecClearTuple(testslot);
	epqstate->relsubs_blocked[rti - 1] = true;

	return slot;
```

要点：

- 调用方是 `ModifyTable`/`LockRows`：`heap_update` 返回 `TM_Updated` 后，先用
  `table_tuple_lock(...TUPLE_LOCK_FLAG_FIND_LAST_VERSION)` 沿 ctid 版本链追到
  最新版本并加锁（期间阻塞等对方事务出结果），再交给 `EvalPlanQual` 重检
  （函数头 2704-2709 行注释言明此前提）。
- EPQ 用**同一棵 Plan 子树**另建迷你执行器（独立 EState），但目标表的扫描节点
  被"掉包"：不真扫表，而是从 `relsubs_slot[rti-1]` 取出那**一行**测试元组
  （`relsubs_done/blocked` 数组控制每表吐 0 行还是 1 行）。5.1 节的
  `ExecSeqScanEPQ` 变体（`nodeSeqscan.c:206`）就是为这条路径准备的。
- 整个 WHERE（乃至 join 条件）对"新版本行 + 其余表原配行"重算一遍：非空 →
  基于新版本继续 UPDATE；NULL → 静默跳过该行。
- `EvalPlanQualInit`（`execMain.c:2784`）在 ModifyTable 初始化时只分配
  `relsubs_slot` 数组（2803 行），子执行器**惰性构建**（2795-2800 行注释）
  ——无并发冲突时零开销。

**语义代价**：重检后的 UPDATE 混合了两个时间点的视图（目标行是最新版本，join
的其他行仍是旧快照），学术上不可串行化——READ COMMITTED 与 REPEATABLE READ/SSI
的本质分界。对照 MySQL：InnoDB 的 UPDATE 用**当前读**直接按最新已提交数据重新
定位，PG 是"快照读定位 + 冲突行单独重检"，殊途同归但边界行为不同（PG 不会
看到快照之后才插入的新行）。

### 小结

EPQ = "锁到最新版本 + 用掉包了扫描节点的迷你执行器对单行重跑计划"，让 READ
COMMITTED 的写语句不丢失并发更新、也无需整语句重试，代价是轻微的时间点混合。
`relsubs_*` 数组是阅读 `nodeModifyTable.c` 前的必备知识。

## 8. what-if 推演

### 8.1 如果 hash join 构建侧估行差 100 倍——批次爆炸全过程

设 planner 估内表 10 万行、实际 1000 万行，`work_mem` 按估算只给了单批次预算。
执行时间线：

1. **建表期超限**：`MultiExecHash` 逐行 `ExecHashTableInsert`
   （`nodeHash.c:1774`），每次插入后检查 `spaceUsed > spaceAllowed`，超限调
   `ExecHashIncreaseNumBatches`（`nodeHash.c:1055`）。
2. **第一次翻倍**：先看能否用"增大内存"抵掉（`ExecHashIncreaseBatchSize`，
   1073 行：批文件缓冲的隐性内存已超过哈希表本身时，翻倍 spaceAllowed 更划算
   ——这是 2025 年加的止损阀，防止"批次越多、BufFile 缓冲吃的内存反而越多"
   的恶性循环）。不行则 `nbatch = oldnbatch * 2`（1076 行），分配/扩展批文件
   数组（1084-1102 行），然后**全表重分区**（1136-1192 行）：逐 chunk 扫现存
   哈希表，`ExecHashGetBucketAndBatch` 重算每行批号——按 5.3.5 的单调性，
   每行要么留在当前批（拷入新 chunk），要么写入更晚的批文件
   （`Assert(batchno > curbatch)`，1172 行）。理想哈希下恰好一半行被移走。
3. **反复翻倍**：估差 100 倍 ≈ 需要 7 次翻倍（2^7 = 128 分之一才装得下），
   每次翻倍都是一趟全表重分区 + 写盘。总写盘量约为内表大小 × O(log(误差))。
4. **止损开关**：若某次翻倍后发现"移走了全部或零行"（1207-1209 行
   `nfreed == 0 || nfreed == ninmemory`），说明是**同一个键的海量重复**
   （哈希无法再细分同值行），置 `growEnabled = false`——放弃控制内存，
   "gut it out and hope the server has enough RAM"（1204-1205 行注释原话）。
   这是 PG 少数明示"可能吃爆内存"的地方；PG 13+ 的哈希聚合选择了 spill
   兜底，而 hash join 的同值堆积在算法上无解（除非换 sort-merge）。
5. **外表侧连带**：外表扫描开始后 nbatch 若再涨，已写的外表批文件里会有
   "现在看来属于更晚批次"的行——`ExecHashJoinNewBatch` 的规则 3
   （`nodeHashjoin.c:1348-1350`）强制重扫这些文件，把行再往后搬。
6. **可观察后果**：EXPLAIN ANALYZE 显示 `Batches: 128 (originally 1)`、
   临时文件读写激增（`log_temp_files` 可见）、每行内表数据平均被写盘/读盘
   多次。对照 Spark：等价于 shuffle 分区数估小后的 AQE 重分区，但 PG 是
   **就地增量翻倍**而非重跑 stage。

根因治理永远在统计侧（ANALYZE、扩展统计），执行器的翻倍机制只是安全网——
这是"优化器估算 → 执行器自适应"分工的典型样本。

### 8.2 如果执行中途出错——ResourceOwner 资源清理串讲

`ExecutorEnd` 的清理代码在 `elog(ERROR)` 长跳转下根本不会执行。谁来关闭
HashJoin 的 8 个批文件、释放 buffer pin、干掉排序临时文件？

**分层答案**：

1. **内存**：全部执行器内存挂在 `es_query_cxt` 下，而它挂在事务级上下文下。
   事务 abort 时 `AbortTransaction` 整树删除内存上下文——palloc 过的东西
   无需任何登记。
2. **内核资源（pin/锁/文件/DSM）**：光释放内存不够——buffer pin 计数、打开的
   fd、共享内存段是内核/共享状态，删内存上下文不会归还它们。这就是
   **ResourceOwner**（`src/backend/utils/resowner/resowner.c`，设计文档同目录
   README）的职责：一个**资源登记簿**。每次 `ReadBuffer` 取 pin、`BufFileCreateTemp`
   开临时文件、`dsm_attach` 挂共享内存，都会在 `CurrentResourceOwner` 里登记
   （如 `bufmgr.c` 中随处可见的 `ResourceOwnerEnlarge(CurrentResourceOwner)`）；
   正常路径用完即注销，**登记簿正常情况下应当清空**。
3. **登记簿的树形结构**：`ResourceOwnerCreate(parent, name)`
   （`resowner.c:428`）构成层级——TopTransaction → Portal → 子事务……
   Portal 有自己的 owner，游标存活期间其 pin/文件都记在名下。
4. **出错路径**：`AbortTransaction` → `ResourceOwnerRelease`
   （`resowner.c:665`）按三个阶段递归释放整棵 owner 树
   （`resowner.h:54-56`）：`RESOURCE_RELEASE_BEFORE_LOCKS`（先还 buffer pin、
   关文件——这些不依赖锁）→ `RESOURCE_RELEASE_LOCKS`（释放锁，子事务提交时
   锁要"过继"给父级而非释放，所以单列一档）→ `RESOURCE_RELEASE_AFTER_LOCKS`
   （目录缓存引用等收尾）。在非 abort 的正常提交路径上，这套释放还兼任
   **泄漏检测器**：登记簿非空会打出 "buffer pin leak" / "temporary file leak"
   警告——泄漏在开发期就会大声暴露。
5. **执行器视角**：HashJoin 的 BufFile、tuplesort 的 LogicalTapeSet 底下的
   物理临时文件都在创建时登记到当前 owner；`elog(ERROR)` 后不管执行器死在
   状态机哪一步，abort 清理都能按登记簿逐项归还。**执行器代码里几乎看不到
   try/finally，正是因为清理被外置成了"登记 + 事务级统一释放"**。

对照一般系统编程：这是 RAII 的进程级替代品——C 没有析构函数，PG 用
"资源登记簿 + longjmp 后统一清算"取得同等安全性，且天然支持子事务的
部分回滚（只清算子 owner 子树）。

### 8.3 如果查询带 LIMIT——提前终止如何沿树传播

LIMIT 的"停"其实有三条独立通路，从被动到主动：

1. **pull 模型的天然停止**：`Limit` 节点（`ExecLimit`，`nodeLimit.c:40`）取够
   count 行后，对后续 `ExecProcNode` 调用直接返回 NULL，**不再拉孩子**。
   孩子不被拉就不产行——扫描停在半路，无需任何通知机制。`SELECT * FROM huge
   LIMIT 10` 只读几个页面，靠的就是这条。
2. **主动下推 bound**：光"不拉"对 pipeline breaker 没用——Sort 在第一次被拉时
   就要吃掉全部输入。所以 `recompute_limits`（`nodeLimit.c:353`）算出
   `offset + count` 后调 `ExecSetTupleBound`（423 行 → `execProcnode.c:829`）
   **沿树下推**："我最多只要 N 行"。下推规则（829-935 行逐类处理）：
   - `SortState`/`IncrementalSortState`：置 `bounded = true, bound = N`——
     下次 `ExecSort` 建库时走 `tuplesort_set_bound`，全量排序退化为 top-N
     堆（5.5.2 的 TSS_BOUNDED 路径），内存从 O(输入) 降到 O(N)；
   - `MergeAppendState`/`AppendState`：广播给每个孩子（每路最多也只需 N 行）；
   - 投影型 `Result`、无 qual 的 `SubqueryScan`：透传给孩子——**有 qual 就
     停止下推**，因为过滤会丢行，孩子的 N 行过滤后可能不够；
   - `GatherState`：传给并行 worker。
   这就是 EXPLAIN 里 `Sort Method: top-N heapsort` 的来历。
3. **终止后的资源提前释放**：`ExecutePlan` 主循环退出后（无论因 LIMIT、客户端
   断开还是取满 numberTuples），1830-1831 行对整树 `ExecShutdownNode`
   （`execProcnode.c:753`）——最关键的是让并行 worker 立即退出并回收
   instrumentation，否则它们会傻跑到自然结束。

三条通路的分工：通路 1 是 Volcano 免费赠品（push 模型要自己实现的部分）；
通路 2 是对 pipeline breaker 的补丁，本质是**把上层的需求信息静态注入下层**
（注意它绕过了逐行接口，是 Volcano 之外的带外信号）；通路 3 处理"树外"资源。
对照 Spark：`df.limit(10)` 会被优化成 `CollectLimitExec` + 每分区
`take(10)`，同样是"需求上限下推"，只是发生在计划期而非执行期。

### 小结

三个推演展示执行器的三类韧性：**自适应**（批次翻倍 + growEnabled 止损，对抗
估算错误）、**可恢复**（ResourceOwner 登记簿 + 上下文整树销毁，对抗任意点
出错）、**可短路**（pull 天然停止 + bound 下推 + 提前 shutdown，对抗
"不需要全部结果"）。

## 自测问题（附详解）

**1. Plan 树和 PlanState 树为什么必须是两棵树？各自存什么？**

详解：Plan 树是只读计划（形状、代价、表达式原型），可被 prepared statement
多次复用、被并行 worker 共享；PlanState 树（`ExecInitNode`，
`execProcnode.c:142` 镜像构建）持有单次执行的可变状态（扫描位置、哈希表、
排序缓冲、ExprState）。混在一起则计划无法复用、无法并发执行同一计划，
EXPLAIN 也无法在执行后打印原计划。不变式：执行期间 Plan 树绝不被修改。

**2. EXPLAIN ANALYZE 的每节点计时怎么注入？为什么普通执行几乎零开销？**

详解：`ExecInitNode` 末尾统一把 `ExecProcNode` 指针挂成包装器
`ExecProcNodeFirst`（`ExecSetExecProcNode`，`execProcnode.c:430`）；首次调用
时它做一次栈深检查，然后按 `node->instrument` 把指针换成 `ExecProcNodeInstr`
（前后打点）或裸的 `ExecProcNodeReal`（`execProcnode.c:448-470`）。分支只
发生一次，之后热路径没有任何 `if`——"把运行期分支固化成函数指针"是执行器
反复出现的手法（SeqScan 变体、ExecHashJoinImpl 特化同理）。

**3. `SELECT c2 FROM t`（t 有 100 列），PG 会解包整行吗？把常用列放前面为什么快？**

详解：不会。表达式编译期 `ExecCreateExprSetupSteps` 生成
`EEOP_SCAN_FETCHSOME`，只解包到被引用的最大列号（这里是 2）；执行时
`slot_getsomeattrs`（`tuptable.h:376`）查水位线 `tts_nvalid`，不足才由
`slot_deform_heap_tuple`（`execTuples.c:1016`）从 `tts_off` 断点增量续解。
deform 成本正比于"最大被引列号"，所以常用列靠前直接省 CPU；若前缀全是定长
非 NULL 列，还能命中预计算的 `attcacheoff` 快径（commit `c456e39113`）。

**4. hybrid hash join 的 "hybrid" 体现在哪里？它与 Grace hash join 差在哪？**

详解：Grace 先把两侧全部分区写盘再逐分区连接，内存再大也要写盘一轮。hybrid
的 batch 0 常驻内存、边建表边连接（in-memory 的一半），只有批号 > 0 的行写
临时文件、之后逐批重建回放（`ExecHashJoinNewBatch`，`nodeHashjoin.c:1279`，
Grace 的一半）——内存够则完全不碰磁盘，不够时平滑滑向 Grace。PG 还叠加运行期
批次翻倍与 skew 桶（MCV 热键固定在 batch 0，预算 2% 哈希内存，
`nodeHash.c:2429`、`hashjoin.h:132`）。

**5. 证明 hash join 的"同 batch 必相遇"：为什么运行中批次翻倍不会漏掉匹配？**

详解：批号是哈希值的纯函数（`ExecHashGetBucketAndBatch`，
`nodeHash.c:1986`），键相等 ⇒ 哈希相等 ⇒ 同一 nbatch 下批号相等。nbatch 只
翻倍，翻倍下新批号 = 旧批号或旧批号 + 旧 nbatch ⇒ 批号单调不减（代码哨兵：
`Assert(batchno > curbatch)`）。批次按升序处理，加上 `ExecHashJoinNewBatch`
的规则 2/3（nbatch 变过就不能跳过单侧非空的批次，`nodeHashjoin.c:1316-1350`
注释），保证每行最终恰好出现在其终值批号的处理窗口里——外行回放与内行哈希表
在同一批相遇。skew 行被显式豁免且两侧同时豁免，不破坏配对。

**6. 哈希聚合内存超限后，为什么已有组不受影响、新组才写盘？这如何保证不 OOM、
何时不保证？**

详解：`hash_agg_enter_spill_mode`（`nodeAgg.c:1910`）置 `hash_spill_mode`
后，`lookup_hash_entries`（2184 行）以 `p_isnew = NULL` 调哈希表——只查不建：
已有组继续正常累加（精度无损、不写盘），会建新组的行整行写入按哈希分区的
spill 文件；吐完内存组后 `agg_refill_hash_table`（2683 行）逐分区读回重建，
装不下再以更多分区位递归 spill。组数增长被闸住 + 每批至少一组保证前进性 ⇒
常规聚合（O(1) 转移状态）有硬内存上限；但单组转移状态无界增长（如
array_agg 巨组）时记账检查"imperfect"（1858-1863 行注释），不做承诺。

**7. NestLoop 的内表参数变了，为什么不立即重扫，而是记入 chgParam 位图？**

详解：`ExecReScan` 被推迟到下一次 `ExecProcNode`（`executor.h:322` 内联里的
`if (node->chgParam != NULL) ExecReScan(node)`）才触发，避免"参数设了但该
分支未被执行"的浪费；且重扫递归下推时每个节点用 chgParam 与自身依赖求交，
Material/Sort/Memoize 发现参数与己无关可直接重放缓存。rescan 契约只要求
"表现得像刚初始化"，允许一切不改变可见结果的捷径——`ExecReScanHashJoin`
（`nodeHashjoin.c:1651`）在单批次且内子树参数未变时整张哈希表原地复用。

**8. `ORDER BY x LIMIT 10` 为什么不会做全量排序？bound 是怎么传到 Sort 里的？**

详解：`recompute_limits`（`nodeLimit.c:353`）调 `ExecSetTupleBound`
（`execProcnode.c:829`）沿树下推"最多只要 N 行"：对 SortState 置
`bounded/bound`，`ExecSort` 建库时 `tuplesort_set_bound`（`tuplesort.c:735`）
使 tuplesort 在行数超 2×bound 时转 `TSS_BOUNDED`——维护 N 元素反向堆，多数
输入行一次比较即丢弃，内存 O(N)。下推遇到会丢行的节点（带 qual 的
SubqueryScan 等）即停止，保证语义安全。EXPLAIN 显示 `top-N heapsort`。

**9. 执行到一半 `elog(ERROR)`，HashJoin 的批文件和扫描持有的 buffer pin 谁清理？**

详解：内存由事务 abort 时整树删除内存上下文兜底；pin/文件/DSM 这类内核资源
靠 ResourceOwner 登记簿——获取时登记在 `CurrentResourceOwner`
（`resowner.c:428` 建层级，Portal/事务各有 owner），`AbortTransaction` →
`ResourceOwnerRelease`（`resowner.c:665`）分 BEFORE_LOCKS/LOCKS/AFTER_LOCKS
三阶段递归释放。正常提交路径上同一机制兼任泄漏检测（登记簿非空则告警）。
所以执行器代码里没有 try/finally。

**10. READ COMMITTED 下，UPDATE 扫到的行已被并发事务改掉并提交，接下来发生什么？**

详解：`heap_update` 返回 `TM_Updated`；执行器用
`table_tuple_lock(...FIND_LAST_VERSION)` 沿版本链等锁并取最新版本；
`EvalPlanQual`（`execMain.c:2715`）把该行放进 `relsubs_slot[rti-1]`，用扫描
节点被替换成"只吐这一行"（`ExecSeqScanEPQ`）的迷你执行器重跑计划子树：
非空则基于新版本更新，NULL 则跳过。整语句不报错不重启——与 REPEATABLE READ
（报 serialization failure）的关键行为差异；代价是目标行与 join 其余行来自
两个时间点，不可串行化。
