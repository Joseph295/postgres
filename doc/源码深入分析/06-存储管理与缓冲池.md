# 06 · 存储管理与缓冲池：页、BufferDesc 与磁盘之间

> 对应《PostgreSQL 源码学习指南》第 7 章。所有 `文件:行号` 均已在本地源码树（20devel，master 分支）核实，代码摘录与源码逐字一致（省略处以 `...` 或注释标明）。特别注意：本分支的缓冲区状态字已从历史版本的 32 位扩展为 **64 位**，并且**缓冲区内容锁已并入该状态字**（commit dac328c8a6 + fcb9c977aa，2026-01），与网上绝大多数旧资料（包括 PG 17/18 的介绍文章）都不同，本文以当前代码为准。

## 本章导读

存储层自底向上三级：**md.c**（把"表"落成一组 1GB 段文件）→ **smgr**（存储管理器抽象接口）→ **bufmgr**（共享缓冲池：页的缓存、淘汰与写出）。本章的主线是回答四个问题：

1. **一个 8KB 页里有什么？**——页头、行指针、元组区的布局，以及行指针这层间接性为什么值 4 个字节（7.1）；
2. **一次页访问在共享内存里经过什么？**——BufferDesc、64 位原子状态字、128 分区映射哈希表，以及命中路径如何被打磨到"一次分区共享锁 + 一次 CAS"（7.2–7.3）；
3. **缓冲池满了怎么办？**——clock-sweep 淘汰、环形缓冲策略、受害者获取与并发 I/O 协议（7.4–7.6），以及本分支的异步 I/O（AIO）与 read_stream（7.7）；
4. **脏页何时、由谁、凭什么写回磁盘？**——FlushBuffer 里的 WAL-before-data 检查、bgwriter 与 checkpointer 的分工（7.8）。

之后是本章的"所以然"部分：为什么是 clock-sweep 而不是 LRU/ARC（7.11.1）、为什么需要环形缓冲（7.11.2）、为什么把一切塞进一个原子变量（7.11.3）、以及 PG 的双缓冲问题与 O_DIRECT 之路（7.11.4）；再给出不变式清单（7.12）、设计史（7.13）、三个 what-if 推演（7.14）和 10 道自测题。

与 InnoDB buffer pool 的对照贯穿全章：InnoDB 用"LRU 链表 + midpoint 插入 + free/flush 链表 + 多 instance 分片"，PG 用"共享哈希表 + clock-sweep + 环形缓冲 + 单一原子状态字"。两者面对同样的问题（扫描抗性、锁争抢、脏页回写节奏），给出了风格迥异的答案。

## 前置知识

- **pin 与内容锁**：pin（引用计数）保证缓冲区不被重用于别的页；内容锁（shared / share-exclusive / exclusive 三级，见 `src/backend/storage/buffer/README` 开头）保护页内数据的读写。纪律是：先 pin 后锁；pin 可以长持（顺序扫描持着 pin 逐行读完一页）；锁必须短持。README 的规则 2 还允许"判定元组可见后放掉内容锁、仅持 pin 继续引用元组数据"——这条规则稍后会解释 ZeroAndLockBuffer 里一个不起眼的注释。
- **steal / no-force**：PG 允许未提交事务的脏页写盘（steal），提交时不强制刷数据页（no-force）——正确性完全由 WAL 兜底。这是读本章全部代码的前提：任何人在任何时刻把任何脏页写盘都必须先保证覆盖该页的 WAL 已落盘，除此之外没有任何时序约束。
- **TID**：`(块号, 行号)` 二元组，索引条目指向堆元组的方式。块号 → 段文件内偏移的换算在 md.c。
- **本分支读路径已 AIO 化**：`ReadBuffer()` 内部走的是 `StartReadBuffer()`（发起）+ `WaitReadBuffers()`（等待）两段式接口，单页同步读只是它的退化用法。`io_method` GUC 默认为 `worker`（aio.h:42），即由 I/O worker 进程代为执行读。

## 7.1 页布局：8KB 里的秩序

### 7.1.1 PageHeaderData 逐字段（bufpage.h:184）

`src/include/storage/bufpage.h:184`：

```c
typedef struct PageHeaderData
{
    PageXLogRecPtr pd_lsn;      /* 最后一次修改本页的 WAL 记录的结束 LSN */
    uint16      pd_checksum;    /* 页校验和（cluster 级开关，写盘时计算） */
    uint16      pd_flags;       /* 标志位，见下 */
    LocationIndex pd_lower;     /* 空闲空间起点 = 行指针数组的末尾 */
    LocationIndex pd_upper;     /* 空闲空间终点 = 元组数据区的起点 */
    LocationIndex pd_special;   /* 特殊区起点（堆页 = 页尾；索引页存放兄弟指针等） */
    uint16      pd_pagesize_version;  /* 页大小 + 布局版本号合一编码 */
    TransactionId pd_prune_xid; /* 页内最老的“可能可清理”XID 提示，0 表示无 */
    ItemIdData  pd_linp[FLEXIBLE_ARRAY_MEMBER];   /* 行指针数组，向后生长 */
} PageHeaderData;
```

- **pd_lsn** 是整个 WAL-before-data 机制的锚点（05 章已详述）：刷此页前必须 `XLogFlush(pd_lsn)`（本章 7.8.2 将看到执行现场）；恢复重放时若 `pd_lsn ≥ 记录LSN` 则跳过（幂等）。
- **pd_lower / pd_upper**：页内空间从两头向中间生长——行指针数组紧跟页头向后长（`pd_lower` 后移），元组数据从页尾（special 区之前）向前长（`pd_upper` 前移）。`pd_lower == pd_upper` 即页满。这与 InnoDB 页内"user records 堆 + page directory 槽"的思路同源，但 PG 的结构更简单：没有页内记录链表，元组物理位置无序，全靠行指针数组定序。
- **pd_flags**（bufpage.h:213–215）：`PD_HAS_FREE_LINES 0x0001`（有可复用的空行指针，加速插入时找槽）、`PD_PAGE_FULL 0x0002`（上次 UPDATE 没挤下，提示下次先 prune）、`PD_ALL_VISIBLE 0x0004`（页上全部元组对所有事务可见，与 VM 位联动，见 7.10.2）。
- **pd_prune_xid**：05 章看到 `PageSetPrunable(page, xid)` 写的就是它——"本页有个 XID=N 的事务动过元组，等 N 落到全局可见性水位之下，来清理吧"。让路过的查询能廉价判断"值不值得对这页做 HOT prune"。

### 7.1.2 ItemId：行指针这层间接性的意义（itemid.h:25）

```c
/* src/include/storage/itemid.h:25 */
typedef struct ItemIdData
{
    unsigned    lp_off:15,      /* 元组在页内的字节偏移 */
                lp_flags:2,     /* LP_UNUSED / LP_NORMAL / LP_REDIRECT / LP_DEAD */
                lp_len:15;      /* 元组字节长度 */
} ItemIdData;
```

4 字节一个，TID 的"行号"部分就是它在数组里的下标（1 起）。这层间接性是 PG 存储设计里回报最高的 4 个字节：

1. **页内搬移自由**：prune/整理时可以把元组在页内挪动压实（改 `lp_off`），所有指向该行的索引 TID 不变；
2. **HOT 的载体**：`LP_REDIRECT`（itemid.h:40）让链头行指针直接转发到存活版本，元组本体回收后索引仍可达；
3. **两阶段删除**：`LP_DEAD` 表示"元组已回收但索引可能还指着我"，VACUUM 删完索引项后才把它归还为 `LP_UNUSED`。

InnoDB 没有对应物——聚簇索引记录移动时靠 B-tree 自身重定位，二级索引存主键值而非物理位置。PG 的索引存物理 TID（回表少一次 B-tree 下降，且胖主键无副作用），代价就是必须用行指针层解耦"逻辑行号"与"物理偏移"。

### 小结

页 = 页头（LSN 打头）+ 向后长的行指针数组 + 向前长的元组区 + special 区。行指针这层间接性换来了页内自由整理与 HOT，是 TID 寻址体系的基石。

## 7.2 缓冲池结构：BufferDesc、64 位状态字与共享哈希表

缓冲池三大件：`BufferBlocks`（`shared_buffers` 个 8KB 页帧的大数组）、`BufferDescriptors`（等长的描述符数组，buf_init.c）、`SharedBufHash`（tag → 描述符下标的哈希表，buf_table.c:34）。三者在启动时一次性从共享内存划出。每缓冲区的 I/O 条件变量单独放在 `BufferIOCVArray`（buf_internals.h:414）——注释说它本可以塞进 BufferDesc 而不破坏 64 字节限制，但目前没这么做（buf_internals.h:322–324）。

### 7.2.1 BufferDesc 逐字段（buf_internals.h:326）

```c
/* src/include/storage/buf_internals.h:326 */
typedef struct BufferDesc
{
    BufferTag   tag;            /* 我装的是哪个磁盘块，改它必须持缓冲区头锁 */
    int         buf_id;         /* 我在数组里的下标，初始化后永不变 */
    pg_atomic_uint64 state;     /* ★ 状态字：flags+refcount+usagecount+内容锁 */
    int         wait_backend_pgprocno;  /* 等待 pin 数降到 1 的进程（VACUUM 用），至多一个 */
    PgAioWaitRef io_wref;       /* 异步 I/O 进行中时的等待句柄 */
    proclist_head lock_waiters; /* 等内容锁的进程链表，由头锁保护 */
} BufferDesc;
```

`BufferTag`（buf_internals.h:161）= `(spcOid, dbOid, relNumber, forkNum, blockNum)` 五元组——注意注释（buf_internals.h:152–156）强调 tag 必须**自足**：刷页的 backend 可能根本"看不见"这张表（创建它的事务比自己晚开始），不能依赖 pg_class 才能定位文件。

结构体被 `BufferDescPadded`（buf_internals.h:383）填充到 64 字节——正好一条缓存行，杜绝相邻描述符间的 false sharing（注释 buf_internals.h:361–380 明说这是实测有效的优化，且 32 位平台不做填充，因为高并发负载几乎都在 64 位机器上）。InnoDB 的 `buf_block_t` 同样讲究对齐，但体积大得多（内嵌 mutex、rw-lock、LRU/flush 链表节点等指针）；PG 把一切压进 8 字节 state，是两者最醒目的对照。

### 7.2.2 state：64 位原子变量的完整位布局（buf_internals.h:49）

三个位段宽度都不是随手写的，源码里有静态断言互相锁死：

- `BUF_REFCOUNT_BITS + BUF_USAGECOUNT_BITS + BUF_FLAG_BITS + BUF_LOCK_BITS <= 64`（buf_internals.h:54）；
- `MAX_BACKENDS_BITS <= BUF_REFCOUNT_BITS`（buf_internals.h:130）——refcount 必须装得下"每个 backend 各 pin 一次"的极端情形；
- `MAX_BACKENDS_BITS <= BUF_LOCK_BITS - 2`（buf_internals.h:132）——共享锁计数同理，再留 2 位给 SE/EX；
- `BM_MAX_USAGE_COUNT < 2^BUF_USAGECOUNT_BITS`（buf_internals.h:146）。

procnumber.h:29 的注释直说：**MAX_BACKENDS_BITS 取 18 就是因为缓冲区状态字里只给 refcount 留了这么多空间**——连接数上限（2^18-1 = 262143）是被这个字反向决定的。

关键增量常量：`BUF_REFCOUNT_ONE = 1`（refcount 在最低位段，加 1 就是普通整数加法）；`BUF_USAGECOUNT_ONE = 1<<18`（buf_internals.h:67）；`BM_LOCK_VAL_SHARED = 1<<34`。**所有位段的"加减"都可以用同一条 64 位原子加/CAS 完成，且互不进位污染**（只要遵守各自的上限约束）——这是整个设计的支点，后文 PinBuffer、StrategyGetBuffer、BufferLockAttempt 会反复兑现它。

### 7.2.3 头锁：BM_LOCKED 也只是 state 里的一个位

"缓冲区头自旋锁"没有独立的 slock_t——`BM_LOCKED`（bit22）就是锁本身：

- `LockBufHdr()`（bufmgr.c:7567）：CAS 置位，带自旋退避；
- `UnlockBufHdr()`（buf_internals.h:459）：`pg_atomic_fetch_sub_u64(&desc->state, BM_LOCKED)`——一条原子减；
- `UnlockBufHdrExt()`（buf_internals.h:476）：解锁的同时**捎带**置位/清位/改 refcount，一条 CAS 完成"改状态 + 放锁"：

```c
/* buf_internals.h:481，节选 */
for (;;)
{
    uint64      buf_state = old_buf_state;

    Assert(buf_state & BM_LOCKED);
    buf_state &= ~unset_bits;
    buf_state |= set_bits;
    buf_state &= ~BM_LOCKED;

    if (refcount_change != 0)
        buf_state += BUF_REFCOUNT_ONE * refcount_change;

    if (pg_atomic_compare_exchange_u64(&desc->state, &old_buf_state, buf_state))
        return old_buf_state;
}
```

头锁的使用纪律（buf_internals.h:277–289 注释）：持锁期间别人不能建立新 pin、不能改标志位；但**释放已有 pin 允许在别人持头锁时进行**（原子减不经过锁）。头锁只保护多字段一致读改写（改 tag、检查后置 I/O 位等），且只允许持有几条指令。

### 7.2.4 内容锁：从独立 LWLock 到 state 内位段（本分支新变化）

历史上每个 BufferDesc 内嵌一个 LWLock 作为内容锁。本分支（commit fcb9c977aa，2026-01）改为**专用实现，直接编码在 state 的高 20 位**。buf_internals.h:303–310 的注释给出动机：专用实现使一些原本困难的事情成为可能——"在排他加锁前无竞态地检查 AIO 是否在进行"、"解锁 + 解 pin 用一条原子操作完成"。加锁的核心是 `BufferLockAttempt()`（bufmgr.c:6119）：

```c
/* bufmgr.c:6136 起，节选 */
if (mode == BUFFER_LOCK_EXCLUSIVE)
{
    lock_free = (old_state & BM_LOCK_MASK) == 0;
    if (lock_free)
        desired_state += BM_LOCK_VAL_EXCLUSIVE;
}
else if (mode == BUFFER_LOCK_SHARE_EXCLUSIVE)
{
    lock_free = (old_state & (BM_LOCK_VAL_EXCLUSIVE | BM_LOCK_VAL_SHARE_EXCLUSIVE)) == 0;
    if (lock_free)
        desired_state += BM_LOCK_VAL_SHARE_EXCLUSIVE;
}
else
{
    lock_free = (old_state & BM_LOCK_VAL_EXCLUSIVE) == 0;
    if (lock_free)
        desired_state += BM_LOCK_VAL_SHARED;
}
```

三种模式的相容矩阵直接由位运算表达：共享锁只排斥 EX 位；share-exclusive 排斥 EX 和 SE（但与共享锁相容——这正是"写盘时允许读者继续读"的锁级，README 规则 6）；exclusive 排斥一切（`BM_LOCK_MASK` 覆盖共享计数 + SE + EX）。无竞争加锁 = 一条 CAS，与 pin 落在**同一条缓存行**上。加锁失败则经 `BufferLockQueueSelf()`（bufmgr.c:6186）把自己挂进 `lock_waiters` 链表（复用 PGPROC 的 lwWaiting 字段）睡眠，唤醒协议靠 `BM_LOCK_HAS_WAITERS`/`BM_LOCK_WAKE_IN_PROGRESS` 两个标志位，思路与 lwlock.c 相同。

share-exclusive 是三级锁里最年轻的一级：它取代了老代码"写盘时持共享锁"的约定，使 hint bit 更新（README 规则 4）与页写出互斥、但都不阻塞普通读者——写出的页映像因此保证自洽，校验和不会算废。

### 7.2.5 共享哈希表与 128 分区锁（buf_table.c）

"磁盘块 → 缓冲区"的映射由 `SharedBufHash`（buf_table.c:34，dynahash 实现）承担，接口极薄：`BufTableHashCode()`（buf_table.c:84）、`BufTableLookup()`（buf_table.c:96，查不到返回 -1）、`BufTableInsert()`（buf_table.c:124）、`BufTableDelete()`（buf_table.c:154）。

并发控制是**分区锁**：哈希空间按 tag 哈希值低位切成 `NUM_BUFFER_PARTITIONS = 128`（lwlock.h:83）个分区，各配一把 LWLock（`BufMappingPartitionLock()`，buf_internals.h:254）。规则（README"Buffer Manager's Internal Locking"节）：

- 查表持分区**共享**锁，且**必须在放锁前 pin 住查到的缓冲区**——否则放锁瞬间它就可能被淘汰（不变式 7，见 7.12）；
- 改映射持分区**排他**锁；需要同时锁两个分区时按分区号升序拿，防死锁；
- 唯一需要排他分区锁的常见操作是"读入一个不在池里的页"——反正它接下来要做磁盘 I/O，慢就慢了。

对比 InnoDB：buffer pool 用 `buf_pool->mutex` + 块级 mutex/rw-lock 的多级锁体系，靠**多 buffer pool instance** 分片摊薄争抢，页哈希另有自己的分区 rw-lock。PG 的哲学是"把每页的并发热点收敛到一个原子字"，分片只用于映射表；InnoDB 的哲学是"复制多份整套结构摊薄热点"。

### 小结

BufferDesc 被刻意瘦身到一条缓存行；state 把 pin / 热度 / 12 个标志 / 头锁 / 三级内容锁六类信息编进 64 位，配合三条静态断言互相锁死宽度；页定位靠 128 路分区锁的共享哈希表。接下来看这套数据结构在读路径上如何运转。

## 7.3 读页路径：ReadBufferExtended → ReadBuffer_common → BufferAlloc

### 7.3.1 入口分发（bufmgr.c:879 / 926 / 1276）

`ReadBuffer()`（bufmgr.c:879）是 `ReadBufferExtended(rel, MAIN_FORKNUM, blockNum, RBM_NORMAL, NULL)` 的简写；`ReadBufferExtended()`（bufmgr.c:926）多出 fork 选择、读取模式和访问策略参数，转调 `ReadBuffer_common()`（bufmgr.c:1276）。后者全文不长，逐段看（bufmgr.c:1276–1368）：

```c
    if (unlikely(mode == RBM_ZERO_AND_CLEANUP_LOCK ||
                 mode == RBM_ZERO_AND_LOCK))

        ...
```

③ 里的 `ZeroAndLockBuffer()`（bufmgr.c:1137）值得停一下：它不读盘，但也**不能直接 memset**——必须先走 `StartSharedBufferIO(bufHdr, true, true, NULL)`（bufmgr.c:1173）抢到 `BM_IO_IN_PROGRESS`，否则可能清零一个别人正 pin 着读的页。注释（bufmgr.c:1164–1170）点名了原因：拿排他内容锁都不够，因为 README 规则 2 允许读者**放了内容锁仅持 pin** 继续读元组。清零后先拿排他内容锁再 `TerminateBufferIO(bufHdr, false, BM_VALID, true, false)` 置 VALID（bufmgr.c:1196–1202）——保证任何人看到 VALID 页时，调用者已锁住它准备初始化，别人不会看见一个全零的"合法页"。

### 7.3.2 PinBufferForBlock → BufferAlloc：命中与未命中的分岔（bufmgr.c:1223 / 2197）

`StartReadBuffersImpl()`（bufmgr.c:1371）对每个目标块调用 `PinBufferForBlock()`（bufmgr.c:1223）：临时表走 `LocalBufferAlloc()`（localbuf.c，无锁版本，本文从略），共享表走 `BufferAlloc()`（bufmgr.c:2197）。BufferAlloc 是全章枢纽，逐段走读：

**第一段：查表（命中快路径）**（bufmgr.c:2215–2255）：

```c
    /* create a tag so we can lookup the buffer */

    /* determine its hash code and partition lock ID */
    newHash = BufTableHashCode(&newTag);
    newPartitionLock = BufMappingPartitionLock(newHash);

    /* see if the block is in the buffer pool already */
    LWLockAcquire(newPartitionLock, LW_SHARED);
    existing_buf_id = BufTableLookup(&newTag, newHash);
    if (existing_buf_id >= 0)
    {
        BufferDesc *buf;
        bool        valid;
        ...
        buf = GetBufferDescriptor(existing_buf_id);

        valid = PinBuffer(buf, strategy, false);

        /* Can release the mapping lock as soon as we've pinned it */
        LWLockRelease(newPartitionLock);

        *foundPtr = true;

        if (!valid)
        {
            /*
             * We can only get here if (a) someone else is still reading in
             * the page, (b) a previous read attempt failed, or (c) someone
             * called StartReadBuffers() but not yet WaitReadBuffers().
             */
            *foundPtr = false;
        }

        return buf;
    }
```

命中成本 = 一次分区共享锁 + 一次哈希查找 + PinBuffer 的一次 CAS。注意 `PinBuffer` 在放分区锁**之前**调用（不变式 7）；`valid=false` 的三种情形注释写得很全——(a)(b) 是 7.5 的 I/O 协议要处理的，(c) 是 AIO 两段式接口引入的新常态。

**第二段：未命中，先去弄一个可复用的缓冲区**（bufmgr.c:2257–2269）：

```c
    LWLockRelease(newPartitionLock);

    /*
     * Acquire a victim buffer. Somebody else might try to do the same, we
     * don't hold any conflicting locks. If so we'll have to undo our work
     * later.
     */
    victim_buffer = GetVictimBuffer(strategy, io_context);
    victim_buf_hdr = GetBufferDescriptor(victim_buffer - 1);
```

关键设计：**放掉分区锁再去找受害者**。找受害者可能要刷脏页（磁盘写 + XLogFlush），绝不能压在分区锁下做。代价是引入竞态——两个 backend 可能同时为同一页各自弄了一个受害者，靠下一段的 Insert 冲突检测收敛。

**第三段：插入映射，处理"撞车"**（bufmgr.c:2276–2317）：

```c
    LWLockAcquire(newPartitionLock, LW_EXCLUSIVE);
    existing_buf_id = BufTableInsert(&newTag, newHash, victim_buf_hdr->buf_id);
    if (existing_buf_id >= 0)
    {
        ...
        /* Got a collision. Someone has already done what we were about to do. */
        UnpinBuffer(victim_buf_hdr);
        /* remaining code should match code at top of routine */
        existing_buf_hdr = GetBufferDescriptor(existing_buf_id);
        valid = PinBuffer(existing_buf_hdr, strategy, false);
        LWLockRelease(newPartitionLock);

```

撞车的输家只是白拿了一个空闲缓冲区（放掉 pin 即归还），赢家的页照常可用——没有任何一方需要重试或回滚 I/O，因为**到此为止还没有发生任何 I/O**。

**第四段：给受害者换身份**（bufmgr.c:2322–2350）：

```c
    victim_buf_state = LockBufHdr(victim_buf_hdr);

    /* some sanity checks while we hold the buffer header lock */

        set_bits |= BM_PERMANENT;

```

注意此刻**只设了 `BM_TAG_VALID`，没设 `BM_VALID`**：映射已建立（别人能查到这个缓冲区并 pin 它），但内容还是上一任的垃圾。真正的磁盘读发生在 BufferAlloc 返回之后（7.6 的 AsyncReadBuffers），读完由 `TerminateBufferIO` 置 `BM_VALID`。`BM_PERMANENT` 的注释（bufmgr.c:2330–2335）解释了 unlogged 表的例外：它们的主 fork 只在 shutdown checkpoint 时才需要写盘，但 **init fork 必须按永久表对待**——崩溃恢复要靠它重置主 fork。

另外留意 `set_bits` 里捎带的 `BUF_USAGECOUNT_ONE`：新页出生自带 1 点热度，等价于旧代码里"分配即被使用一次"。

### 7.3.3 PinBuffer 全函数走读：单条 CAS 里的三件事（bufmgr.c:3295）

pin 是缓冲池最高频操作，值得逐行读完（bufmgr.c:3295–3386）：

```c
static bool
PinBuffer(BufferDesc *buf, BufferAccessStrategy strategy,
          bool skip_if_not_valid)
{
    Buffer      b = BufferDescriptorGetBuffer(buf);
    bool        result;
    PrivateRefCountEntry *ref;
    ...
    ref = GetPrivateRefCountEntry(b, true);

    if (ref == NULL)                          /* 本进程第一次 pin 这个缓冲区 */
    {
        uint64      buf_state;
        uint64      old_buf_state;

        old_buf_state = pg_atomic_read_u64(&buf->state);
        for (;;)
        {
            if (unlikely(skip_if_not_valid && !(old_buf_state & BM_VALID)))
                return false;

            /*
             * We're not allowed to increase the refcount while the buffer
             * header spinlock is held. Wait for the lock to be released.
             */
            if (unlikely(old_buf_state & BM_LOCKED))
            {
                old_buf_state = WaitBufHdrUnlocked(buf);
                /* perform checks at the top of the loop again */
                continue;
            }

            buf_state = old_buf_state;

            /* increase refcount */
            buf_state += BUF_REFCOUNT_ONE;

            if (strategy == NULL)
            {
                /* Default case: increase usagecount unless already max. */
                if (BUF_STATE_GET_USAGECOUNT(buf_state) < BM_MAX_USAGE_COUNT)
                    buf_state += BUF_USAGECOUNT_ONE;
            }
            else
            {
                /*
                 * Ring buffers shouldn't evict others from pool.  Thus we
                 * don't make usagecount more than 1.
                 */
                if (BUF_STATE_GET_USAGECOUNT(buf_state) == 0)
                    buf_state += BUF_USAGECOUNT_ONE;
            }

            if (pg_atomic_compare_exchange_u64(&buf->state, &old_buf_state,
                                               buf_state))
            {
                result = (buf_state & BM_VALID) != 0;
                TrackNewBufferPin(b);
                break;
            }
        }
    }
    else                                      /* 本进程已 pin 过：纯本地操作 */
    {
        result = (pg_atomic_read_u64(&buf->state) & BM_VALID) != 0;

        Assert(ref->data.refcount > 0);
        ref->data.refcount++;
        ResourceOwnerRememberBuffer(CurrentResourceOwner, b);
    }

    return result;
}
```

把 CAS 循环体里的位运算拆开看，一次成功的 CAS 同时完成了**四件事**：

1. **确认头锁未被持有**：`old_buf_state & BM_LOCKED` 为假才继续。这不是"先检查再操作"的竞态写法——如果 CAS 执行时别人恰好拿了头锁，state 值必然已变（BM_LOCKED 位从 0 变 1），CAS 失败重来。头锁的互斥性被 CAS 的比较语义免费继承；
2. **refcount + 1**：`+ BUF_REFCOUNT_ONE` 即 `+1`，最低位段，普通加法。溢出不可能：refcount ≤ 每 backend 的 pin 数总和 < 2^18（静态断言保证）；
3. **usagecount 条件 +1**：增量是 `1<<18`，同一条加法指令捎带。普通访问封顶 5；环形策略封顶 1（这一行是 7.9.2 扫描抗性的半壁江山）；
4. **读取 BM_VALID**：返回值顺便告诉调用者页内容是否有效，不需要第二次原子读。

`skip_if_not_valid` 参数服务于 `ReadRecentBuffer()` 一类"按缓冲区号投机 re-pin"的调用者。

**私有 refcount 表**：`GetPrivateRefCountEntry()`（bufmgr.c:507，慢路径 419）维护每 backend 的 `Buffer → 本地 pin 计数` 映射（8 槽数组 + 溢出哈希表）。同进程重复 pin 同页只在本地 `refcount++`，**完全不碰共享内存**——嵌套循环连接反复 pin 同一内表页几乎免费。共享 state 里的 refcount 语义因此是"**有多少个 backend 正 pin 着我**（每 backend 计 1）"而非 pin 的总次数。

**UnpinBuffer**（bufmgr.c:3479 → UnpinBufferNoOwner 3488）是镜像：本地计数减到 0 时才 `pg_atomic_fetch_sub_u64(&buf->state, BUF_REFCOUNT_ONE)`（bufmgr.c:3520）——注意这里连 CAS 都不用，一条无条件原子减（buf_internals.h:287–289 说明了为什么允许在别人持头锁时做原子减）。若旧值带 `BM_PIN_COUNT_WAITER`，调 `WakePinCountWaiter()`（bufmgr.c:3443）唤醒等待 cleanup lock 的 VACUUM。

### 小结

命中路径 = 分区共享锁 + 哈希查找 + 一次 CAS（首 pin）或纯本地自增（重复 pin）。未命中路径把"找受害者"放在无锁窗口里做，靠 BufTableInsert 的冲突返回值吸收竞态；`BM_TAG_VALID` 与 `BM_VALID` 的分离让"占坑"与"填内容"解耦——这个空隙由 7.5 的 I/O 协议看守。

## 7.4 淘汰：clock-sweep 全走读与环形缓冲

### 7.4.1 时钟指针：ClockSweepTick（freelist.c:109）

本分支已**彻底移除 freelist**（commit 2c78940527，2025-09："Remove freelist, always use clock-sweep"）：`BufferStrategyControl`（freelist.c:32）里只剩时钟指针、统计计数和 bgwriter 通知位，连"空闲链表头"字段都没有了。启动初期的全空池也走 clock-sweep——反正全是 usagecount=0 的未 pin 缓冲区，第一圈就能拿到。

```c
/* freelist.c:109 */
static inline uint32
ClockSweepTick(void)
{
    uint32      victim;

    /*
     * Atomically move hand ahead one buffer - if there's several processes
     * doing this, this can lead to buffers being returned slightly out of
     * apparent order.
     */
    victim =
        pg_atomic_fetch_add_u32(&StrategyControl->nextVictimBuffer, 1);

    if (victim >= NBuffers)
    {
        uint32      originalVictim = victim;

        /* always wrap what we look up in BufferDescriptors */
        victim = victim % NBuffers;
        ...
        if (victim == 0)
        {
            ...
            while (!success)
            {
                ...
                SpinLockAcquire(&StrategyControl->buffer_strategy_lock);
                wrapped = expected % NBuffers;
                success = pg_atomic_compare_exchange_u32(&StrategyControl->nextVictimBuffer,
                                                         &expected, wrapped);
                if (success)
                    StrategyControl->completePasses++;
                SpinLockRelease(&StrategyControl->buffer_strategy_lock);
            }

```

指针是**只增计数器**，取模得下标：推进 = 一条原子加，多个 backend 同时找受害者时各自领走相邻下标，天然并行。只有"恰好绕满一圈的那个人"才短暂拿 `buffer_strategy_lock` 把计数器折回并 `completePasses++`——这把自旋锁存在的唯一意义是让 `StrategySyncStart()`（freelist.c:331）能把 (指针位置, 圈数) 作为一致快照读走，供 bgwriter 估算扫描速度。

### 7.4.2 StrategyGetBuffer 全函数走读（freelist.c:184）

```c
BufferDesc *
StrategyGetBuffer(BufferAccessStrategy strategy, uint64 *buf_state, bool *from_ring)
{
    BufferDesc *buf;
    int         bgwprocno;
    int         trycounter;

    *from_ring = false;

    /* ① 环形策略优先在自己的环里转 */
    if (strategy != NULL)
    {
        buf = GetBufferFromRing(strategy, buf_state);
        if (buf != NULL)
        {
            *from_ring = true;
            return buf;
        }
    }

    /* ② 顺手叫醒 bgwriter（无锁读 bgwprocno，见 freelist.c:207 注释） */
    bgwprocno = INT_ACCESS_ONCE(StrategyControl->bgwprocno);
    if (bgwprocno != -1)
    {
        StrategyControl->bgwprocno = -1;
        SetLatch(&GetPGProcByNumber(bgwprocno)->procLatch);
    }

    /* ③ 分配计数 +1，供 bgwriter 反馈控制用 */
    pg_atomic_fetch_add_u32(&StrategyControl->numBufferAllocs, 1);

    /* ④ 时钟扫描主循环 */
    trycounter = NBuffers;
    for (;;)
    {
        uint64      old_buf_state;
        uint64      local_buf_state;

        buf = GetBufferDescriptor(ClockSweepTick());

        old_buf_state = pg_atomic_read_u64(&buf->state);
        for (;;)                              /* 对这一格做 CAS */
        {
            local_buf_state = old_buf_state;

            if (BUF_STATE_GET_REFCOUNT(local_buf_state) != 0)
            {
                if (--trycounter == 0)
                    elog(ERROR, "no unpinned buffers available");
                break;                        /* 被 pin：看下一格 */
            }

            /* See equivalent code in PinBuffer() */
            if (unlikely(local_buf_state & BM_LOCKED))
            {
                old_buf_state = WaitBufHdrUnlocked(buf);
                continue;
            }

            if (BUF_STATE_GET_USAGECOUNT(local_buf_state) != 0)
            {
                local_buf_state -= BUF_USAGECOUNT_ONE;

                if (pg_atomic_compare_exchange_u64(&buf->state, &old_buf_state,
                                                   local_buf_state))
                {
                    trycounter = NBuffers;    /* 扣热度成功：重置计数，看下一格 */
                    break;
                }
            }
            else
            {
                /* pin the buffer if the CAS succeeds */
                local_buf_state += BUF_REFCOUNT_ONE;

                if (pg_atomic_compare_exchange_u64(&buf->state, &old_buf_state,
                                                   local_buf_state))
                {
                    /* Found a usable buffer */
                    if (strategy != NULL)
                        AddBufferToRing(strategy, buf);
                    *buf_state = local_buf_state;

                    TrackNewBufferPin(BufferDescriptorGetBuffer(buf));

                    return buf;               /* 受害者到手（已 pin） */
                }
            }
        }
    }
}
```

要点：

- **每一格的判定/降温/占有都是纯 CAS**，不碰头锁（commit 5e89985928，2025-10："Don't lock buffer header in StrategyGetBuffer"）。淘汰路径全程只有两类共享写：时钟指针的原子加、目标缓冲区 state 的 CAS；
- `trycounter` 防死循环：连续 NBuffers 格都因"被 pin"跳过（期间没做出任何降温）才报 `no unpinned buffers available`（freelist.c:274）——运维偶见的这个报错源头就在这里，含义是"全池都被 pin 住了"（通常是 shared_buffers 太小 + 大量游标/长事务持 pin）。只要还有任何一次降温成功，计数就重置——降温单调递减保证总会有格降到 0；
- 拿到手的受害者**已经带上一个 pin**：从"选中"到"换 tag"之间的窗口里，别人可以看到它但无法淘汰它。

### 7.4.3 GetVictimBuffer：受害者的清洗流水线（bufmgr.c:2548）

StrategyGetBuffer 只保证"选中时没被 pin"，受害者可能是脏的、可能随时被别人 pin 回去。`GetVictimBuffer()`（bufmgr.c:2548–2686）负责把它洗成"干净、无 tag、仅自己 pin"的可复用状态：


三个细节都有血泪史：

- ① 的**条件锁**：注释（bufmgr.c:2589–2601）记载了真实死锁案例——两个 backend 同时分裂 btree 页，B 从 StrategyGetBuffer 拿到的受害者恰是 A 刚锁住的页，若无条件等锁，A 稍后又等 B，死锁。原则：**淘汰路径上没有任何值得等的东西，拿不到就换**；
- ② 是 BULKREAD 环的"脏页逃逸"（详见 7.4.4）；`XLogNeedsFlush` 判断"复用这页是否要先刷 WAL"——顺序扫描不该背这个开销；
- ③ `InvalidateVictimBuffer()`（bufmgr.c:2471）在分区排他锁 + 头锁下复查"refcount==1 且不脏"（bufmgr.c:2503：期间别人可能 pin 过又弄脏了它），通过才抹掉 tag、清标志、`BufTableDelete`。任何一步发现被抢，整条流水线 `goto again` 重来——**乐观到底，冲突即重试**。

### 7.4.4 环形缓冲策略（freelist.c:426）

大扫描如果走普通 clock-sweep，会把整个缓冲池刷成自己碰过一次的冷页。PG 的解法：给这类操作发一个小小的私有"环"，让它在环内自我循环复用。`GetAccessStrategy()`（freelist.c:426）按用途定环尺寸：

| 类型 | 环大小 | 使用者 |
|---|---|---|
| `BAS_NORMAL` | 无环（返回 NULL） | 普通访问 |
| `BAS_BULKREAD` | 基础 256KB + `io_combine_limit × effective_io_concurrency` 个块的 AIO 余量（freelist.c:459–484；默认 16×16=256 块即 2MB），受 pin 上限约束 | 大表顺序扫描（表 > NBuffers/4 时启用，heapam.c:398/410） |
| `BAS_BULKWRITE` | 16MB（freelist.c:488） | COPY IN、CREATE TABLE AS |
| `BAS_VACUUM` | 2MB（freelist.c:491；实际由 `vacuum_buffer_usage_limit` GUC 经 `GetAccessStrategyWithSize()` freelist.c:511 控制） | VACUUM/ANALYZE |

所有环统一封顶 `NBuffers/8`（freelist.c:526）。环的实现极简（freelist.c:74）：一个定长 Buffer 数组 + current 游标，`GetBufferFromRing()`（freelist.c:623）取下一槽：

```c
/* freelist.c:655–666 注释+判定，节选 */
        /*
         * If the buffer is pinned we cannot use it under any circumstances.
         *
         * If usage_count is 0 or 1 then the buffer is fair game (we expect 1,
         * since our own previous usage of the ring element would have left it
         * there, but it might've been decremented by clock-sweep since then).
         * A higher usage_count indicates someone else has touched the buffer,
         * so we shouldn't re-use it.
         */
        if (BUF_STATE_GET_REFCOUNT(local_buf_state) != 0
            || BUF_STATE_GET_USAGECOUNT(local_buf_state) > 1)
            break;
```

usagecount > 1 说明**别人**也在用这页（自己 pin 只会把它抬到 1，见 PinBuffer 的 strategy 分支）——环主动放弃它，改走正常 clock-sweep 另取新缓冲区补进环。这条判定同时是"环页归还全局池"的通道：热起来的环页自然留在池里。

`StrategyRejectBuffer()`（freelist.c:752）只对 BULKREAD 生效：把需要刷 WAL 的脏环页**从环里除名**（置 InvalidBuffer，freelist.c:767——注释说这是为了防止"环里全是脏页"时无限循环），交回正常淘汰路径。而 VACUUM/COPY 的环不踢脏页，宁可自己刷 WAL——README 明说这是刻意的："写流量由制造者自己消化，不祸害别人"；README 还记载了历史教训：8.3 之前 VACUUM 用"送回 freelist"的方式，等效于 1 个缓冲区的环，造成过量 WAL flush。

256KB 的依据（README"Buffer Ring Replacement Strategy"节）：小到能装进 L2 cache（OS cache → shared buffers 的拷贝更快），又大到足以覆盖扫描并发 pin 的页数，还给同步顺序扫描（synchronized seqscan）留了"缓存尾迹"。本分支因 AIO 预读在途页不可复用，环在 256KB 基础上按 I/O 并发度扩容（freelist.c:469–481 注释）。

### 小结

淘汰 = 原子加推进时钟 + 每格一次 CAS 判定/降温/占有 + GetVictimBuffer 的乐观清洗流水线，全程无全局锁、无等待（拿不到就换）。扫描抗性外包给三种尺寸的环。InnoDB 把"防污染"内置进 LRU（midpoint 插入 + `innodb_old_blocks_time`），所有访问都受其规则约束；PG 把它外置成按工作负载类型发放的私有环，普通 OLTP 路径完全无感，且 BULKREAD 环额外实现了"不为顺序扫描刷 WAL"。

## 7.5 并发 I/O 协议：WaitIO / StartSharedBufferIO / TerminateBufferIO

BufferAlloc 留下的空隙——"映射已建立、内容还无效"——由 `BM_IO_IN_PROGRESS` 位看守。这套协议同时服务读入（判据 `BM_VALID`）与写出（判据 `!BM_DIRTY`），是缓冲池里最精巧的一段状态机。

### 7.5.1 StartSharedBufferIO：抢 I/O 权（bufmgr.c:7290）

```c
StartBufferIOResult
StartSharedBufferIO(BufferDesc *buf, bool forInput, bool wait, PgAioWaitRef *io_wref)
{
    uint64      buf_state;

    ResourceOwnerEnlarge(CurrentResourceOwner);

    for (;;)
    {
        buf_state = LockBufHdr(buf);

        if (!(buf_state & BM_IO_IN_PROGRESS))
            break;                            /* 没人在做 I/O，跳出去自己做 */

        /* Join the existing IO */
        if (io_wref != NULL && pgaio_wref_valid(&buf->io_wref))
        {
            *io_wref = buf->io_wref;          /* ① 拿走异步等待句柄，回头再等 */
            UnlockBufHdr(buf);
            return BUFFER_IO_IN_PROGRESS;
        }
        else if (!wait)
        {
            UnlockBufHdr(buf);                /* ② 不等，走人 */
            return BUFFER_IO_IN_PROGRESS;
        }
        else
        {
            ...
            UnlockBufHdr(buf);
            ...
            pgaio_submit_staged();            /* 先把自己攒的异步 I/O 发出去，防死锁 */
            WaitIO(buf);                      /* ③ 睡等它完成，然后回循环头重查 */
        }
    }

    /* Once we get here, there is definitely no I/O active on this buffer */

    /* Check if someone else already did the I/O */
    if (forInput ? (buf_state & BM_VALID) : !(buf_state & BM_DIRTY))
    {
        UnlockBufHdr(buf);
        return BUFFER_IO_ALREADY_DONE;        /* 别人已替我干完 */
    }

    /*
     * No IO in progress and not already done; we will start IO. It's possible
     * that the IO was in progress but we're not done, because the IO errored
     * out. We'll do the IO ourselves.
     */
    UnlockBufHdrExt(buf, buf_state,
                    BM_IO_IN_PROGRESS, 0,
                    0);

    ResourceOwnerRememberBufferIO(CurrentResourceOwner,
                                  BufferDescriptorGetBuffer(buf));

    return BUFFER_IO_READY_FOR_IO;
}
```

三种返回值对应三种命运：`READY_FOR_IO`（I/O 权归我）、`ALREADY_DONE`（白嫖成功）、`IN_PROGRESS`（别人在干，我按 ①/② 的方式改天再看或异步等）。注意最后那段注释：**上一轮 I/O 失败的情形也落到"自己干"分支**——错误可以被任何后来者接力恢复。`BM_IO_IN_PROGRESS` 的置位始终在头锁保护下进行（"检查旧值 + 置位"必须原子）。

### 7.5.2 WaitIO：睡在每缓冲区一个的条件变量上（bufmgr.c:7188）

```c
static void
WaitIO(BufferDesc *buf)
{
    ConditionVariable *cv = BufferDescriptorGetIOCV(buf);
    ...
    ConditionVariablePrepareToSleep(cv);
    for (;;)
    {
        uint64      buf_state;
        PgAioWaitRef iow;
        ...
        buf_state = LockBufHdr(buf);
        /*
         * Copy the wait reference while holding the spinlock. This protects
         * against a concurrent TerminateBufferIO() in another backend from
         * clearing the wref while it's being read.
         */
        iow = buf->io_wref;
        UnlockBufHdr(buf);

        /* no IO in progress, we don't need to wait */
        if (!(buf_state & BM_IO_IN_PROGRESS))
            break;

        /* 异步 I/O：直接等 AIO 子系统 */
        if (pgaio_wref_valid(&iow))
        {
            pgaio_wref_wait(&iow);
            ...
            ConditionVariablePrepareToSleep(cv);
            continue;
        }

        /* wait on BufferDesc->cv, e.g. for concurrent synchronous IO */
        ConditionVariableSleep(cv, WAIT_EVENT_BUFFER_IO);
    }
    ConditionVariableCancelSleep();
}
```

PG 14 之前这里是每缓冲区一把 `io_in_progress` LWLock（发起者持排他锁、等待者请共享锁），commit d87251048a（2021-03）换成条件变量——LWLock 方案在发起者先于等待者退出时有唤醒竞态，且条件变量语义（"等某个状态改变"）本来就更贴切。本分支再叠加 AIO：若 `io_wref` 有效，说明 I/O 由 AIO 子系统执行（可能在 io worker 进程里），直接 `pgaio_wref_wait()` 等它。

### 7.5.3 TerminateBufferIO：交卷与广播（bufmgr.c:7407）

```c
void
TerminateBufferIO(BufferDesc *buf, bool clear_dirty, uint64 set_flag_bits,
                  bool forget_owner, bool release_aio)
{
    uint64      buf_state;
    uint64      unset_flag_bits = 0;
    int         refcount_change = 0;

    buf_state = LockBufHdr(buf);

    Assert(buf_state & BM_IO_IN_PROGRESS);
    unset_flag_bits |= BM_IO_IN_PROGRESS;

    /* Clear earlier errors, if this IO failed, it'll be marked again */
    unset_flag_bits |= BM_IO_ERROR;

    if (clear_dirty)
        unset_flag_bits |= BM_DIRTY | BM_CHECKPOINT_NEEDED;

    if (release_aio)
    {
        /* release ownership by the AIO subsystem */
        Assert(BUF_STATE_GET_REFCOUNT(buf_state) > 0);
        refcount_change = -1;
        pgaio_wref_clear(&buf->io_wref);
    }

    buf_state = UnlockBufHdrExt(buf, buf_state,
                                set_flag_bits, unset_flag_bits,
                                refcount_change);

    if (forget_owner)
        ResourceOwnerForgetBufferIO(CurrentResourceOwner,
                                    BufferDescriptorGetBuffer(buf));

    ConditionVariableBroadcast(BufferDescriptorGetIOCV(buf));
    ...
}
```

一条 `UnlockBufHdrExt` 同时完成：清 IO 位、清旧错误、（写成功时）清脏位和 `BM_CHECKPOINT_NEEDED`、（读成功时经 set_flag_bits）置 `BM_VALID`、（AIO 完成时）释放 AIO 子系统持有的那个 pin——然后广播条件变量唤醒所有等待者。失败路径由 AIO 完成回调或 `AbortBufferIO()`（bufmgr.c:7469，资源属主清理时调用）以 `set_flag_bits = BM_IO_ERROR` 走同一函数：**IO 位照清、VALID 不置**，被唤醒的等待者回到 StartSharedBufferIO 循环头，发现"无 I/O 进行中且仍非 VALID"，自己置位接班重试。

### 7.5.4 完整时序：两个 backend 同时 miss 同一页

```
   Backend A                      共享状态                        Backend B
──────────────────────────────────────────────────────────────────────────────
① BufTableLookup → 未命中
② GetVictimBuffer → 得 buf 97
③ 排他分区锁,BufTableInsert(tag→97)
④ 换 tag, 置 TAG_VALID            state: TAG_VALID,ref=1

                                                             返回 valid=false（无 BM_VALID）

                                                             B 也走"要做 I/O"的路径
⑦ StartBufferIO(forInput=true):
   头锁下查 IO 位=0 → 置
   BM_IO_IN_PROGRESS              state: +IO_IN_PROGRESS
   → READY_FOR_IO
                                                          ⑧ StartBufferIO:
                                                             头锁下见 IO 位=1
                                                             → 同步路径: WaitIO 睡在
                                                               buf 97 的条件变量上
                                                               (AIO 路径: 拿走 io_wref
                                                                异步等, 见 7.6)
⑨ smgrstartreadv/preadv 读盘
   （磁盘 I/O, 毫秒级, 全程无锁）
⑩ TerminateBufferIO:
   清 IO 位, 置 BM_VALID          state: -IO, +VALID
   ConditionVariableBroadcast ────────────────────────────▶ ⑪ 被唤醒, 回循环头:
                                                             IO 位=0 → break;

                                                             → ALREADY_DONE
⑫ 用页                                                    ⑬ 用页（磁盘只被读了一次）

失败分支：⑩' A 的读失败 → TerminateBufferIO(BM_IO_ERROR)，VALID 未置
          ⑪' B 醒来: IO 位=0 但仍无 VALID → 自己置 IO 位 → READY_FOR_IO → B 重读磁盘
          （A 的报错不影响 B 接班；BM_IO_ERROR 只用于 AbortBufferIO 的重复失败告警）
```

若 B 比 A 的 ③ 更早走到"插入映射"，则 B 的 `BufTableInsert` 返回 A 已插的 buf_id（或反之），走 7.3.2 第三段的撞车路径——两条竞态窗口（映射建立前/后）分别由 Insert 冲突与 IO 位收敛，磁盘读永远恰好一次（成功时）。

### 小结

`BM_IO_IN_PROGRESS` + 每缓冲区条件变量 + "读者判据 VALID / 写者判据 !DIRTY"构成一套"谁抢到谁干、其余人等结果、失败者可接班"的协议。它的美在于**没有专门的错误恢复路径**——失败只是"干完了但判据仍不满足"，循环结构天然重试。

## 7.6 AIO 与 read_stream：本分支的异步读路径

PG 18 起（commit da7226993f 等，2025-03）核心 AIO 基础设施进入源码树，`src/backend/storage/aio/` 目录结构见 aio.c:10–27 的导览注释：`method_*.c`（三种执行方式）、`aio_target.c`/`aio_io.c`/`aio_callback.c`（目标/操作/回调）、`read_stream.c`（预读辅助层）。

### 7.6.1 io_method 三档（aio.c:63）

```c
const struct config_enum_entry io_method_options[] = {
    {"sync", IOMETHOD_SYNC, false},
    {"worker", IOMETHOD_WORKER, false},
#ifdef IOMETHOD_IO_URING_ENABLED
    {"io_uring", IOMETHOD_IO_URING, false},
#endif
    {NULL, 0, false}
};
```

- `sync`：兼容模式，一切照旧同步执行（StartReadBuffersImpl 里专门保留了这条"去风险"路径，bufmgr.c:1531–1537 注释直言它最终会被移除）；
- `worker`（默认，aio.h:42）：backend 把 I/O 请求投进共享队列，专职 io worker 进程执行 `preadv` 后按完成回调收尾——**别的进程可以替你完成你的 I/O**，这解释了 TerminateBufferIO 里 `release_aio` 分支为什么要处理"完成者不是发起者"的 pin 转移；
- `io_uring`（Linux）：每 backend 一个 uring 实例，内核态异步，无进程间弹跳。

### 7.6.2 两段式读与 AsyncReadBuffers（bufmgr.c:1938）

`StartReadBuffers()`（bufmgr.c:1618）pin 住一段连续块，把其中需要 I/O 的前缀交给 `AsyncReadBuffers()`（bufmgr.c:1938）：

- 对首块 `StartBufferIO(..., wait=true, &operation->io_wref)`——若别人已在读，**不同步等**，而是把对方的 `io_wref` 记进 operation、标记 `operation->foreign_io = true` 返回（bufmgr.c:2090–2095）。注释（bufmgr.c:2036–2049）解释了为什么这对性能关键：索引扫描的预读窗口内可能多次请求同一块、多 backend 并发扫同一表——同步等待会压垮 I/O 深度；
- 对后续块 `StartBufferIO(..., wait=false, NULL)` 尽量合并（bufmgr.c:2114–2127），一次 `smgrstartreadv()` 向量化提交，上限 `io_combine_limit`（默认 128KB/BLCKSZ = 16 块，bufmgr.h:176）；遇到"中间有块已被别人读入"就切断，剩余已 pin 的缓冲区**转发**给下一次调用（buffers 数组里非 InvalidBuffer 的槽位，bufmgr.c:1409–1444）；
- 完成回调 `buffer_readv_complete()`（bufmgr.c:8725）在 I/O 落地的进程里执行校验和检查并逐块 `TerminateBufferIO`。

`WaitReadBuffers()`（bufmgr.c:1759）是一个"等待 + 补作业"循环：等 `io_wref` 完成 → 处理结果 → 若还有块没完成（部分读、foreign_io 失败、sync 模式从未发起）就再调 `AsyncReadBuffers` 直到 `nblocks_done == nblocks`。foreign_io 的失败处理（bufmgr.c:1868–1872 注释）再次体现 7.5 的接力哲学：别人的 I/O 失败了，`nblocks_done` 不前进，下一轮自己来。

### 7.6.3 read_stream：把"逐块 ReadBuffer"变成流（read_stream.c）

`read_stream.c`（commit b5a9b18cd0 引入，2024-04）是消费侧的抽象：调用者提供"下一个块号"回调，流负责在幕后攒块、合并成最多 `io_combine_limit` 的大读、按自适应距离预发 I/O，消费者只管 `read_stream_next_buffer()`。文件头注释（read_stream.c:20–28）描述了距离控制算法：全命中时距离收缩到 1（预读无益）；出现 miss 就放大合并距离与预读距离，尝试受益于 I/O 合并与并发。顺序扫描、ANALYZE、VACUUM、pg_prewarm 等已改造为 read_stream 用户。

与 InnoDB 对照：InnoDB 的 linear read-ahead（`innodb_read_ahead_threshold`）以 extent 为粒度、启发式触发，长期依赖自带的 AIO（Linux native AIO / 模拟 AIO）；PG 此前三十年靠 OS 预读 + `posix_fadvise`，直到 18 才有了自己的 AIO 执行层。方向一致，PG 起步晚但 io_uring 起点更现代。

### 小结

本分支读路径 = pin（BufferAlloc）→ 抢 I/O 权（StartBufferIO）→ 向量化异步提交（AsyncReadBuffers/smgrstartreadv）→ 完成回调置 VALID → WaitReadBuffers 兜底补作业。`io_wref` 与 `foreign_io` 把 7.5 的协议从"同进程同步"推广到"跨进程异步"，而协议本身的状态机一字未改——这是 `BM_IO_IN_PROGRESS` 设计三十年后仍然成立的最好证明。

## 7.7 弄脏一页：MarkBufferDirty（bufmgr.c:3170）

写路径（05 章）在临界区内调用的 `MarkBufferDirty()` 同样是 CAS 循环：

```c
/* bufmgr.c:3193 起，节选 */
    /*
     * NB: We have to wait for the buffer header spinlock to be not held, as
     * TerminateBufferIO() relies on the spinlock.
     */
    old_buf_state = pg_atomic_read_u64(&bufHdr->state);
    for (;;)
    {
        if (old_buf_state & BM_LOCKED)
            old_buf_state = WaitBufHdrUnlocked(bufHdr);

        buf_state = old_buf_state;

        Assert(BUF_STATE_GET_REFCOUNT(buf_state) > 0);
        buf_state |= BM_DIRTY;

```

前置断言要求持排他内容锁（bufmgr.c:3188）。为什么置一个位也要绕开头锁？注释点名 `TerminateBufferIO()`：写出方在头锁下执行"清 DIRTY"，若这里绕过锁语义盲目置位，可能与"写出后清脏"交错出"刚写完的页丢了 DIRTY 标记但其实又被改了"或反之的状态。`MarkBufferDirtyHint()`（bufmgr.c:5844）是它的弱化变体：hint bit 类修改不改逻辑内容、可容忍丢失，允许在 share-exclusive 锁下进行，且只在页此前不脏时才可能触发 WAL（wal_log_hints/校验和场景，防 torn page）。

## 7.8 脏页写出：FlushBuffer、bgwriter 与 checkpointer

### 7.8.1 FlushBuffer 全函数（bufmgr.c:4526）

任何人（backend 淘汰、bgwriter、checkpointer）把脏页写盘都走这里：

```c
static void
FlushBuffer(BufferDesc *buf, SMgrRelation reln, IOObject io_object,
            IOContext io_context)
{
    XLogRecPtr  recptr;
    ...
    Assert(BufferLockHeldByMeInMode(buf, BUFFER_LOCK_EXCLUSIVE) ||
           BufferLockHeldByMeInMode(buf, BUFFER_LOCK_SHARE_EXCLUSIVE));

    /* ① 抢写出权；别人抢先刷完就直接回家 */
    if (StartSharedBufferIO(buf, false, true, NULL) == BUFFER_IO_ALREADY_DONE)
        return;
    ...
    /*
     * As we hold at least a share-exclusive lock on the buffer, the LSN
     * cannot change during the flush (and thus can't be torn).
     */
    recptr = BufferGetLSN(buf);

    /* ② ★ WAL-before-data 的执行点 */
    if (pg_atomic_read_u64(&buf->state) & BM_PERMANENT)
        XLogFlush(recptr);

    /* ③ 写盘 */
    bufBlock = BufHdrGetBlock(buf);
    PageSetChecksum((Page) bufBlock, buf->tag.blockNum);
    ...
    smgrwrite(reln,
              BufTagGetForkNum(&buf->tag),
              buf->tag.blockNum,
              bufBlock,
              false);
    ...
    /* ④ 清脏、放 IO 位、广播 */
    TerminateBufferIO(buf, true, 0, true, false);
    ...
}
```

这就是 05 章反复预告的闭环：`heap_insert` 把 LSN 盖进页头 → `FlushBuffer` 刷页前 `XLogFlush(页头LSN)`。两行代码相隔两个文件，共同构成 ARIES 第一原则（WAL-before-data）的完整实现。细节：

- **unlogged 表跳过 ②**：注释（bufmgr.c:4572–4582）讲了一个隐蔽灾难——unlogged 索引页带着 `XLogGetFakeLSN` 造的假 LSN，若假 LSN 恰好超过当前 WAL 插入点，`XLogFlush` 会失败并引发系统级后果，所以非 `BM_PERMANENT` 一律不 flush WAL；
- **锁级是 share-exclusive**（① 前的断言 + bufmgr.c:4562 注释）：不阻塞读者，但排斥修改者与 hint bit 写者——落盘的页映像自洽，LSN 不会被撕裂，校验和不会算废。InnoDB 的对应物是 doublewrite buffer + 页 latch 下拷贝；PG 靠锁 + FPI（full page image）解决 torn page，不需要 doublewrite；
- ① 复用 7.5 协议（forInput=false，判据 `!BM_DIRTY`）：两个进程同时想刷同一页，一个干活一个 `ALREADY_DONE`；
- 淘汰路径上的调用者随后还会 `ScheduleBufferTagForWriteback()`（bufmgr.c:2637）——攒一批写出的页向内核发 `sync_file_range` 类提示（`backend_flush_after`），避免 OS 把回写憋成雪崩。

### 7.8.2 checkpointer：BufferSync 的排序与限速（bufmgr.c:3575）

**checkpointer**（checkpointer.c:206 CheckpointerMain）为**恢复时长**负责：周期性（`checkpoint_timeout` 或 WAL 量触发）执行 `CreateCheckPoint()`（xlog.c:7400）→ `BufferSync()`（bufmgr.c:3575）。走读要点：

1. **快照打标**（bufmgr.c:3615–3648）：线性扫全部描述符，给"当刻已脏"的页在头锁下置 `BM_CHECKPOINT_NEEDED` 并把 `(表空间, 文件号, fork, 块号, buf_id)` 记入 `CkptBufferIds` 数组。注释（bufmgr.c:3604–3609）说明该标志的语义：checkpoint 只负责这批页；期间新弄脏的页没有标志，留给下轮。任何人（backend/bgwriter）先把带标志的页刷了，`TerminateBufferIO(clear_dirty=true)` 会连标志一起清（7.5.3），checkpointer 到时发现标志已消失就跳过——**写出工作天然可以被别人分担**；
2. **排序 + 表空间交错**（bufmgr.c:3657–3664 注释，sort_checkpoint_bufferids）：按 (tsId, relNumber, fork, blockNum) 排序把随机写变顺序写；再用 binaryheap 在表空间间轮转（每个表空间按进度取最落后者），避免一次只压一块盘；
3. **匀速摊开**：每写一页调 `CheckpointWriteDelay()`（bufmgr.c:3820），按 `checkpoint_completion_target` 把写出摊满整个 checkpoint 周期；
4. 结束后 fsync 所有涉及的文件（经 md.c 注册的 sync 请求）、写 checkpoint WAL 记录、更新 pg_control。此后崩溃恢复只需从这个 REDO 点开始重放。

对应 InnoDB 的 fuzzy checkpoint：InnoDB 按 flush list（脏页按最老修改 LSN 有序）从尾部推进 checkpoint LSN，天然增量；PG 的 checkpoint 是"批处理"风格——一次圈定、排序、限速写完。InnoDB 用 `innodb_io_capacity` 控制节奏，PG 用完成目标百分比控制。

### 7.8.3 bgwriter：追着时钟指针预洗（bufmgr.c:3854）

**bgwriter**（bgwriter.c:89 BackgroundWriterMain）为**前台延迟**负责。`BgBufferSync()`（bufmgr.c:3854）沿 clock-sweep 指针**前方**扫描（`StrategySyncStart()` 读指针位置与圈数，但不推进指针），把"即将被时钟扫到、又脏又冷（无 pin、无热度）"的页提前洗干净，这样 backend 淘汰时大概率拿到干净页，省掉前台同步 `FlushBuffer + XLogFlush` 的延迟。

刷多少是一套反馈控制（bufmgr.c:3872–3874 的状态变量）：以 `numBufferAllocs` 的移动平均（`smoothed_alloc`，16 个采样周期平滑）预测近期分配需求，乘 `bgwriter_lru_multiplier`（默认 2.0）得目标清洁缓冲区数，减去扫描前方已有的可复用缓冲区估计（由 `smoothed_density` 密度估计换算），封顶 `bgwriter_lru_maxpages`（默认 100 页/轮）。此外有个下限：按 `scan_whole_pool_milliseconds = 120000` 保证 2 分钟内至少绕池一圈（bufmgr.c:3877）。

角色对照：bgwriter ≈ InnoDB page cleaner 的"LRU flush"半边（为腾空闲页服务），checkpointer ≈ "flush list flush"半边（为推进 checkpoint LSN 服务）。InnoDB 把两者合并在 page cleaner 线程组里自适应调度；PG 在 9.2（commit 806a2aee37，2011-11）把它们拆成两个进程——拆分理由见 7.13。

第三个相关角色 walwriter 刷的是 WAL 缓冲不是数据页；第四个写出者是 backend 自己（淘汰时，7.4.3），`pg_stat_io` 里 `io_context=normal` 的 backend write 数量高，就是前两者没跟上的信号。

### 小结

一切脏页写出都过 `FlushBuffer`，其中 `XLogFlush(BufferGetLSN(buf))` 一行落实 WAL-before-data；`BM_CHECKPOINT_NEEDED` 让 checkpoint 的工作可被任何写出者分担。checkpointer 面向 RTO（圈定、排序、交错、限速），bgwriter 面向延迟（反馈控制的预洗），backend 兜底。

## 7.9 smgr 抽象层与 md.c：段文件与 fork

### 7.9.1 smgr：一张函数指针表（smgr.c:88）

`src/backend/storage/smgr/smgr.c:88` 定义 `f_smgr` 接口：`smgr_open/close/create/unlink/extend/readv/writev/startreadv/writeback/nblocks/truncate/immedsync...`，目前表里只有一个实现 md.c（smgr.c:128 的 `smgrsw[]`）。这层间接性是 Berkeley 时代"多存储介质后端"的遗产，如今的作用：bufmgr 及以上代码完全不知道"文件"的存在，只对 `(RelFileLocator, fork, blockNum)` 编址——也为外部表 AM / 未来非文件系统后端留了插槽。

### 7.9.2 md.c：1GB 段文件（md.c:45–76 注释）

一个 fork 的数据 = 一串段文件：`基础文件名`、`.1`、`.2`……每段最多 `RELSEG_SIZE` 块（默认 131072 块 × 8KB = 1GB）。md.c:45 起的大段注释规定了段链的形态不变式：**零个或多个满段 + 恰好一个未满段 + 可选的若干 0 长度不活跃段**。0 长段是 `mdtruncate()` 刻意留下的——直接 unlink 的话，别的 backend/checkpointer 手里还开着的 fd 会变成"指向已删除文件"，表再涨回去时写进旧文件的数据会凭空消失（md.c:57–68）。

切段的原始动机是老文件系统的 2GB 上限（md.c:49 注释），今天更多是工具友好。块号换算（`mdextend`，md.c:487）：

```c
/* md.c:516 */
    v = _mdfd_getseg(reln, forknum, blocknum, skipFsync, EXTENSION_CREATE);

    seekpos = (pgoff_t) BLCKSZ * (blocknum % ((BlockNumber) RELSEG_SIZE));
```

读写入口 `mdreadv`（md.c:858）/`mdwritev`（md.c:1070）用 `preadv/pwritev` 向量化 I/O（配合 `io_combine_limit` 把相邻块合并成一次系统调用）；异步读入口是 `mdstartreadv`。fsync 被"外包"：md.c 把待 fsync 的段登记给 checkpointer（`register_dirty_segment`，md.c:149），checkpoint 时统一执行——平时 write 只进 OS page cache。

对比 InnoDB：独立表空间 `.ibd` 单文件自管页分配（extent/segment/XDES 元数据在文件内，受 redo 保护）；PG 把空间管理完全交给文件系统，文件内没有任何分配元数据，页号即偏移——结构极简，代价是依赖文件系统的行为（也因此有了 7.14.3 的 what-if）。

### 7.9.3 fork：一张表的四个"分身"（relpath.h:59）

```c
MAIN_FORKNUM = 0,        /* 主数据：16384、16384.1、... */
FSM_FORKNUM,             /* 空闲空间地图：16384_fsm */
VISIBILITYMAP_FORKNUM,   /* 可见性地图：16384_vm */
INIT_FORKNUM,            /* unlogged 表的“出厂镜像”：16384_init */
```

fork 是"同一逻辑表的多个物理文件流"，共享 RelFileLocator、各自独立编址块号。BufferTag 里含 forkNum，所以 FSM/VM 页与数据页在同一个缓冲池里被统一缓存管理。InnoDB 没有对应概念——位图类辅助数据内嵌在表空间固定页里；PG 的 fork 让辅助数据独立增长、独立截断。

## 7.10 FSM 与 VM：两张地图

### 7.10.1 FSM：定位空闲空间的三层树（freespace.c）

**用途**：`heap_insert` 找页（05 章）不能线性扫全表问"谁有空位"，FSM 提供 O(树高) 的查找。

**编码**（freespace.c:64）：每个堆页的空闲空间压缩成 1 字节类别值：`FSM_CATEGORIES = 256` 档、每档 `FSM_CAT_STEP = BLCKSZ/256 = 32` 字节。粗粒度足够——找页只需"至少这么大"。

**结构**：两级嵌套的树。页内：每个 FSM 页是一棵完全二叉树，叶子是若干堆页的类别值，内部节点存两孩子的 max；页间：FSM 页组成 `FSM_TREE_DEPTH = 3` 层的树（freespace.c:75，8KB 页约 4000 叶槽，4000³ 远超 2³² 块上限）。上层 FSM 页的叶槽存下层 FSM 页的根值。

**查找** `fsm_search()`（freespace.c:688）：从根页出发页内二叉下降，得下层页号，递归到叶层返回堆块号。页内下降从"上次成功位置的下一个"开始，让并发插入自然散开，避免全员挤同一个"最空的页"。

**更新** `RecordPageWithFreeSpace()`（freespace.c:194）：改叶槽后向上冒泡修正父 max。关键权衡：**FSM 不写 WAL**——它只是提示（hint），崩溃后略有失真无碍正确性（insert 拿到"其实没那么空"的页会失败重查并顺手纠正），VACUUM 也会重建。对比 InnoDB：XDES/inode 页是精确的、受 redo 保护的元数据；PG 的 FSM 是允许说谎的缓存——又一次"简单结构 + 事后修正"的设计口味。

### 7.10.2 VM：每堆页 2 个 bit（visibilitymap.c）

每个堆页在 VM fork 里占 `BITS_PER_HEAPBLOCK = 2` 位（visibilitymapdefs.h:17）：**all-visible**（页上所有元组对所有事务可见）与 **all-frozen**（所有元组已冻结；置位前提是已 all-visible）。一个 8KB VM 页覆盖约 32K 堆页 ≈ 256MB 堆（visibilitymap.c:125 `HEAPBLOCKS_PER_PAGE`），所以 VM 极小、常驻缓存。核心 API：`visibilitymap_pin()`（visibilitymap.c:209）、`visibilitymap_set()`（visibilitymap.c:260，仅 VACUUM 调用，需持堆页锁并记 WAL）、`visibilitymap_clear()`（visibilitymap.c:152，任何使页不再全可见的 DML 在临界区内调用）、`visibilitymap_get_status()`（visibilitymap.c:324）。

**两大用途**：

1. **index-only scan**：索引扫描拿到 TID 后查 VM，目标堆页 all-visible 就不用回表做可见性判断——PG 的索引不存版本信息（对比 InnoDB 二级索引可借 PAGE_MAX_TRX_ID + 回聚簇索引 undo 判断），VM 是 PG 实现覆盖扫描的先决条件。`EXPLAIN` 里 `Heap Fetches` 高即 VM 位没设上；
2. **VACUUM 跳页**：all-visible 页必无死元组，常规 VACUUM 直接跳过；all-frozen 页连 aggressive VACUUM 也能跳过——9.6 引入 all-frozen 位之前，每次反回卷 VACUUM 都要全表扫描，是大表运维的噩梦。

**一致性纪律**（visibilitymap.c 头部注释）：VM 位是"设了必须为真，没设可以为假"的单向保守提示。**置位**必须持堆页锁 + WAL 记录 + 给 VM 页盖 LSN；**清位**总是安全方向，与触发它的堆修改共用同一临界区和 WAL 记录。页头 `PD_ALL_VISIBLE` 与 VM 位同步维护——前者让 DML 不查 VM 就知道"要不要清位"。错误置位的后果：index-only scan 返回错误数据、relfrozenxid 被错误推进后清 clog → 冻结数据"复活"或消失。

### 小结

FSM 是"允许失真、不记 WAL"的 256 档空闲空间摘要树；VM 是"保守为真、置位记 WAL"的每页 2 bit 可见性摘要。两者都是 fork 形式的独立小文件，与数据页共用缓冲池。

## 7.11 所以然：四个"为什么"

### 7.11.1 为什么是 clock-sweep，而不是 LRU / LRU-K / ARC？

这不是纸面推演——PG **真的用过 LRU，也真的用过 ARC**。2005 年的 commit 5d5087363d（Tom Lane）写得明明白白：拆掉全局 BufMgrLock 时，"必须放弃全局 freelist（它是全局争抢点），**这把 ARC 和 2Q 连同朴素 LRU 一起击毙了**，改用 clock sweep"。三条论证线：

**（a）锁争用**。严格 LRU 的每次访问都要把页移到链表头 = 每次访问写两个全局指针 = 一把全局链表锁，且是**读放大成写**（逻辑上的只读页访问也要写共享结构）。PG 8.0 的 ARC 正是死在这里。clock-sweep 把"访问记录"打散到每个缓冲区自己的 state 里（pin 时 usagecount+1，与本来就要做的 refcount+1 同一条 CAS，边际成本为零），只有**淘汰**才需要集中动作，而淘汰路径又被做成了"原子加领号 + 各扫各的"。热点从"每次访问一把全局锁"变成"每次淘汰一次原子加"——频率差了几个数量级。InnoDB 保留了 LRU 链表，代价是 `buf_pool->mutex` 长期是顶级热点，不得不靠多 instance 分片 + "midpoint 插入 + 访问间隔阈值才晋升"来减少链表移动次数——本质上也是在往"近似 LRU"退让。

**（b）实现复杂度与崩溃面**。ARC/LRU-K 需要维护多条链表/历史队列（ARC 还要 ghost list），并发正确性验证困难；PG 的 ARC 时代还背过专利风险（IBM 持有 ARC 专利，8.0.2 紧急换成 2Q 变体）。clock-sweep 的全部状态 = 一个 uint32 指针 + 每缓冲区 4 bit，没有任何链表不变式需要维护，`InvalidateBuffer` 之类的路径不需要"从链表里摘除"的配套逻辑。

**（c）扫描抗性**。朴素 LRU 毫无扫描抗性；LRU-K/ARC/2Q 用访问历史自适应解决；PG 的选择是**不让淘汰算法背这个包袱**——clock-sweep 本身只有 5 级近似，扫描抗性外包给环形策略（7.4.4），按"我知道我自己是大扫描"的先验显式声明。自适应算法要付出常态开销去"学习"负载，PG 的方案是让知道答案的人直接把答案写在调用参数里（`GetAccessStrategy(BAS_BULKREAD)`）。

**usagecount 上限为什么是 5**？buf_internals.h:136–143 的注释给出权衡两端：上限越大越接近真实 LRU（保留更长访问历史），但受害者搜索最坏要绕 `BM_MAX_USAGE_COUNT + 1` 圈才能把一个热页降到 0——上限 5 意味着最坏 6 圈。上限取 NBuffers 就"精确 LRU"了，但淘汰延迟不可接受；取 0 则退化为"随机淘汰未 pin 页"。5 是精度/淘汰开销的经验折衷，另有 4 bit 位段的硬上限 15（静态断言 buf_internals.h:146 看守）。

### 7.11.2 为什么需要 ring buffer：一次 20GB 顺序扫描的血洗推演

设 `shared_buffers = 8GB`（NBuffers = 1,048,576），其中住着一个命中率 99% 的 2GB OLTP 热工作集（约 262,144 页，usagecount 普遍在 4–5）。现在来一条 20GB 大表（2,621,440 页）的顺序扫描，磁盘吞吐 500MB/s（64,000 页/秒），**假设没有环形策略**：

- 扫描每秒要求 64,000 个受害者缓冲区，每个新读入的页 usagecount=1。时钟指针的推进速度 ≥ 64,000 格/秒 ×（平均每拿到一个受害者要看的格数）。启动阶段池里全是热页，指针要先把它们逐圈降温：**第一圈**（100 万格）把全池 usagecount 各扣 1，此间只有原本 ≤0 的页可被淘汰；扫描以 16 秒/圈的速度连续碾压，**4–5 圈（约一分多钟）内热工作集的 usagecount 被从 5 磨到 0**——期间 OLTP 的 re-pin 能给活跃页回血，但 OLTP 的访问频率是"每秒热页访问数"，扫描的降温频率是"每秒 6.4 万格 × 圈数"，只要扫描吞吐大于热页回血速度，磨损单调累积；
- 20GB > 8GB，扫描页自身在池内也活不过一圈——**每一页都是"读进来、用一次、被后续扫描页挤出去"**，2,621,440 次磁盘读换来 0 次复用；
- 终态：池里装满了扫描尾部 8GB 的"一次性"页，2GB 热工作集全灭。扫描结束后 OLTP 命中率从 99% 跌到接近 0，需要 262,144 次随机读（按 200 IOPS 的机械盘要 20 分钟，按 NVMe 也有一段肉眼可见的延迟毛刺）才能重建。这正是 InnoDB 世界里经典的"一条 mysqldump 打挂线上库"事故的机理。

**有环之后**：表大小 2,621,440 页 > NBuffers/4 = 262,144，heapam（heapam.c:398/410）自动申请 `BAS_BULKREAD` 环。扫描的全部缓冲区消耗被限制在环内（基础 256KB = 32 个缓冲区，本分支加 AIO 余量后典型 2MB ≈ 288 个），对 8GB 池的污染 < 0.03%；环页 usagecount 封顶 1（PinBuffer 的 strategy 分支），出环后一圈内即可被回收。热工作集毫发无损。

**256KB 为什么够**？环的功能不是"缓存"而是"流水线缓冲"：只要 ≥ 扫描在任意时刻并发 pin 住的页数（保证复用槽位时自己的 pin 已放掉），再留出 L2 友好性与同步扫描尾迹即可（README 与 freelist.c:446–458 注释）。扫描不复用旧页，环再大也不提升命中率——大环只会平白多占池子。反过来，环也有代价：一旦扫描顺手改页（如扫描中设 hint bit 之外的修改、bulk UPDATE），环页复用要刷 WAL，README 直说此时"环策略实际退化为普通策略"；BULKREAD 靠 `StrategyRejectBuffer` 把脏页踢出环来免除这笔账，VACUUM/COPY 则自担。

### 7.11.3 为什么把一切塞进一个 64 位原子变量？

数一数"旧世界"（PG 9.5 及以前）一次页访问的原子操作账单。当时 BufferDesc = `slock_t` 自旋锁 + 普通字段 refcount/usage_count/flags + 独立 LWLock 内容锁：

| 操作 | 旧世界（≤9.5） | 本分支 |
|---|---|---|
| pin | 拿头自旋锁（1 原子）+ 改两字段 + 放锁（1 原子） | 1 次 CAS |
| 加内容锁 | LWLock：自己的自旋锁/原子对，**另一条缓存行** | 1 次 CAS，**同一条缓存行** |
| 放内容锁 | 同上 | 1 次 CAS（可与唤醒合并） |
| unpin | 再拿头锁 + 改 + 放（2 原子） | 1 次原子减 |

旧世界一次"pin + 锁 + 放锁 + unpin"至少 6 次原子操作、散布在 2 条缓存行上；每次原子操作都要把缓存行以独占态拉到本核。热页（btree 根、visibility map 页）被几十个核同时访问时，这两条缓存行在核间乒乓，自旋锁排队进一步放大——PG 9.5 时代 `s_lock` 高居 profile 榜首是社区痛点。commit 48354581a4（2016，PG 9.6）第一步：state 原子化，pin/unpin 免锁；本分支两步走完剩下的路：dac328c8a6 把 state 扩到 64 位腾出空间，fcb9c977aa 把内容锁搬进来。收益不只是省指令：

- **缓存行合并**：pin、锁、标志现在是同一 8 字节字，一次独占获取服务全套操作；
- **组合操作成为可能**：buf_internals.h:308–310 注释点名"解锁 + 解 pin 一条原子操作完成"（`UnlockReleaseBuffer` 是超高频调用，见 commit f39cb8c011）；
- **无竞态的跨字段判定**：如"在排他加锁前检查 AIO 是否在进行"——锁位和 IO 标志在同一个原子字里，一次读就是一致快照，不需要锁桥接两个字段。

代价也如实记录在案：refcount 从 32 位缩到 18 位（反向钉死 MAX_BACKENDS）；usagecount 上限受 4 bit 约束；所有位段运算必须小心不进位污染（`UnlockBufHdrExt` 的注释特意说明 usagecount 不能用这套"加减法"处理，因为它要封顶）。InnoDB 同期的演化方向类似（`buf_page_t` 的 io_fix/state 原子化、rw-lock 的 lock word 化），但没有走到"单字合一"这一步。

### 7.11.4 双缓冲问题：PG 为什么长期不用 O_DIRECT？

PG 走 buffered I/O：数据页同时可能存在于 shared_buffers 和 OS page cache——**同一份数据最多占两份内存**，这就是双缓冲（double buffering）问题。InnoDB 的标准姿势是 `O_DIRECT`（`innodb_flush_method=O_DIRECT`）绕过 OS cache。PG 三十年不跟进，理由是诚实的（aio/README.md"Why Direct / unbuffered IO"一节反向罗列了全部账目）：

- **O_DIRECT 没有 AIO 就慢得没法用**：绕过 OS cache 意味着也绕过了 OS 的预读与写合并。PG 传统上没有自己的预读（顺序扫描全靠内核 readahead）、没有自己的写调度（脏页 write 后由内核 writeback 线程慢慢刷）。直接换 O_DIRECT，每次 miss 都是裸盘同步延迟，顺序扫描性能崩塌；
- **OS cache 是免费的二级缓存**：shared_buffers 设小些（经典建议 25% 内存）时，OS cache 兜住溢出的工作集，"双缓冲"同时也是"两级缓存"。对多实例共享主机的场景（README 明说），这比每实例独占大 buffer pool 更有弹性；
- **可移植性**：O_DIRECT 的语义在不同 OS/文件系统上千奇百怪（对齐要求、并发写行为），而 PG 的可移植性纪律远严于 MySQL。

代价同样列在 README 里：buffered write 受制于"内核用 CPU 把页从 page cache 拷进拷出"，现代 NVMe 上这成了吞吐瓶颈；fdatasync 的延迟内核藏不住；且内核 writeback 的节奏 PG 控制不了（`*_flush_after` 参数就是在跟这个搏斗）。所以社区的路线图是：**先建 AIO（PG 18 的 aio 子系统，io_uring 后端），补齐自主预读（read_stream）与写调度，然后 DIO 才有意义**——目前 `debug_io_direct` 已可实验性开启 data/wal 的直接 I/O，正式化取决于 AIO 覆盖面的推进。这是一个"先造血、再断奶"的多版本工程，也是理解 PG 18–20 大量 I/O 相关 commit 的总纲。

## 7.12 不变式清单

每条给出"是什么"与"违反后果"。它们几乎都能在源码的 Assert 或注释里找到对应物。

1. **refcount > 0 的缓冲区不可淘汰、不可换 tag**（README"Pins"节；InvalidateVictimBuffer 的复查 bufmgr.c:2503）。违反：读者手里的页帧被替换成另一个磁盘块的内容，查询返回别的表的字节——最恶性的静默数据错乱。整个 GetVictimBuffer 的 goto-again 结构都是为守住它。
2. **BM_IO_IN_PROGRESS 互斥**：任一时刻每缓冲区至多一个进行中 I/O，置位必须在头锁下"检查旧值 + 置位"（StartSharedBufferIO，bufmgr.c:7298–7356）。违反：两个进程同时向同一页帧 readv（读入内容交错撕裂），或读入与写出并发（把半新半旧的页写上盘）。
3. **脏页落盘前，覆盖它的 WAL 必须先落盘**（FlushBuffer 的 `XLogFlush(recptr)`，bufmgr.c:4584）。违反：崩溃后磁盘上存在"WAL 里没有"的未来页——redo 从检查点重放时无法解释该页（pd_lsn 超前导致跳过重放），与其它页的状态失配，唯一性/外键/索引一致性全面失守。steal 策略的全部安全性系于此行。
4. **BM_VALID 只能由成功完成的读置位；TAG_VALID 而非 VALID 的页任何人不得消费内容**（BufferAlloc 注释 bufmgr.c:2345–2347；PinBuffer 返回值语义）。违反：消费到上一任页的残留字节。
5. **修改元组需排他内容锁；写出页需至少 share-exclusive**（README 规则 3/6；FlushBuffer 入口断言 bufmgr.c:4534）。违反前者：读者看到半更新的元组头（xmin/xmax 撕裂）；违反后者：落盘映像内部不一致，校验和错误或 DIO 场景下的设备层问题。
6. **MarkBufferDirty 必须在 WAL 记录插入的同一临界区内、持排他锁时进行**（05 章 + bufmgr.c:3188 断言；历史 commit a8b8f4db23 确立）。违反：页改了但 DIRTY 没置——页永远不被写出，崩溃后修改静默丢失（WAL 有记录尚可恢复；若 WAL 也没有，彻底丢失）。
7. **持映射分区锁查到缓冲区后，必须先 pin 再放分区锁**（README；BufferAlloc bufmgr.c:2237–2240）。违反：放锁到 pin 之间的窗口里缓冲区被淘汰重用，退化为第 1 条的违反。
8. **usagecount ≤ 5、refcount < 2^18、共享锁计数 < 2^18**（三条静态断言 buf_internals.h:54/130/132/146）。违反（若无断言）：位段进位污染相邻字段——比如 refcount 溢出会"凭空"给 usagecount 加一，state 整字损坏，后果不可枚举。
9. **VM 位单向保守：置位须持堆页锁 + WAL；PD_ALL_VISIBLE 与 VM 位同步**（7.10.2）。违反：index-only scan 返回已删除的行；relfrozenxid 错误推进 + clog 截断后数据不可恢复地错乱。
10. **FSM 允许失真但必须可自愈**：使用者拿到"虚报"的页必须能失败重查并回写真实值。违反自愈路径的话，插入会在同一个满页上永久打转。

## 7.14 What-if 推演

### 7.14.1 如果 fsync 返回 EIO 后重试——fsyncgate 与 PANIC 决策

**天真设想**：`fsync()` 返回 EIO（比如存储短暂抖动），打个 WARNING，下个 checkpoint 重试，成功了就继续——2018 年之前的 PG 正是这么干的。

**为什么是灾难**：关键在 Linux 内核的语义（fsyncgate 事件的核心发现）——fsync 失败时，内核**把那些没写出去的脏页标记为干净**（error 已经"报告"给了调用者，内核认为职责已尽；有的内核版本甚至直接丢弃页）。于是下一次 fsync 检查不到任何待写脏页，**返回成功**。推演一次事故：checkpoint N 期间某段文件 fsync 得到 EIO → PG 记 WARNING，checkpoint 失败重来 → 存储恢复，checkpoint N+1 的 fsync"成功" → PG 认为所有脏页安全落盘，推进 REDO 点、回收旧 WAL → 但那批页的数据**既不在磁盘上（内核丢了）、也不在 WAL 里（已回收）、也不在 shared_buffers（早被 TerminateBufferIO 清了脏标记并可能已淘汰）**——三个副本同时消失，且没有任何报错。用户在几周后发现表里有一段 8KB 的旧数据，无从追溯。

**PANIC 为什么是对的**（commit 9ccdd7f66e）：EIO 时刻，数据在世上仅存的可靠副本是 WAL。立即 PANIC → 崩溃恢复从上一个 checkpoint 的 REDO 点重放 → 所有"可能没写上盘"的修改被 WAL 重建重写。这里的洞见是：**fsync 失败不是一个可重试的操作失败，而是一个"进程内存中'哪些页是干净的'这一知识已经不可信"的状态污染**——唯一安全的动作是抛弃全部易失状态从日志重建。`data_sync_retry` GUC（fd.c:163）保留给确知"内核会保留脏页"的平台。同期社区还讨论过更釜底抽薪的方案（自己管理 I/O、不依赖内核 writeback）——这正是 AIO/DIO 路线（7.11.4）的动机之一。

### 7.14.2 如果 checkpoint 进行到一半崩溃了？

BufferSync 已写出一半脏页、fsync 做了几个文件，此时断电。一致性靠什么？

- checkpoint 在**全部**写出与 fsync 完成之前，不写 checkpoint WAL 记录、不更新 pg_control（7.8.2 步骤 4 的次序）。崩溃后恢复起点仍是**上一个**完成的 checkpoint 的 REDO 点——本轮写出的"一半"只是让一些页比 WAL 重放所需的更"新"了；
- "更新"无害：重放到某条 WAL 记录时，若页的 pd_lsn ≥ 记录 LSN，跳过（幂等规则，05 章）。已被本轮 checkpoint 刷出去的页会被自然跳过，没刷的页被重放补齐——两类页殊途同归到一致状态；
- torn page（8KB 页写到一半断电，前 4KB 新后 4KB 旧，pd_lsn 恰在"新"半边，幂等判断被骗）由 **full_page_writes** 兜底：每页在每个 checkpoint 周期内第一次被改时，WAL 里记完整页镜像，重放时整页覆写，不依赖页上任何现存内容。InnoDB 解决同一问题用的是 doublewrite buffer（先把页写到 doublewrite 区再写原位，崩溃后从 doublewrite 区修复半页）——FPI 付出 WAL 膨胀，doublewrite 付出双倍页写，是同一枚硬币的两面；
- 残留的 `BM_CHECKPOINT_NEEDED`（bufmgr.c:3611–3613 注释）：反正这些页下轮 checkpoint 也必须写，标志不清也对。

结论：checkpoint 是"要么完整生效（写了 checkpoint 记录），要么完全没发生（恢复用上一个）"的原子事件，中途崩溃唯一的代价是恢复时间退回上一周期。

### 7.14.3 如果表文件被外部截断/删除了？

PG 运行中，有人在文件系统层 `truncate -s 40MB 16384.1` 或直接 `rm`。会发生什么？

- **读路径**：缓冲池里已有的页照常命中——PG 根本不会发现文件变了（这也是"千万别动数据目录"的第一原因：内存态与磁盘态静默分叉）。未缓存页的读走 `mdreadv` → preadv 返回 0 字节（EOF）→ 报错 `could not read blocks ... read only 0 of 8192 bytes`（md.c:858 起的短读检查）。查询失败但库还活着；`zero_damaged_pages`/`RBM_ZERO_ON_ERROR` 可以把这种页当全零读入，用于救灾性导出；
- **写路径更阴险**：缓冲池里那些块号超出新文件尾的**脏页**照样会被 FlushBuffer → `mdwritev` 写出——POSIX 允许 seek 越过 EOF 写入，文件被**静默重新撑大**，中间隔出一段全零的空洞（sparse）。于是表"部分复活"：被截掉的区间里，有脏页的块变回新数据，没有的块成为全零页（PG 读到全零页按新页处理，通常不报错）——数据就这样不声不响地缺了一块。校验和（`data_checksums`）也帮不上：全零页被视为合法的新页；
- **`rm` 整个文件**：PG 进程手里还开着 fd（md.c 的描述符池），POSIX 语义下 inode 未真正释放——读写照旧成功，磁盘空间也不还；直到 PG 重启/`smgrclose`，文件才真正消失，下次访问报 `could not open file`。这就是"rm 了大表磁盘没变大"的经典疑问的答案，也是 md.c 保留 0 长不活跃段（7.9.2）防御的同类问题的另一面；
- **正规对照**：PG 自己的截断（VACUUM 尾部截断、TRUNCATE）走 `smgrtruncate`：先 `DropRelationBuffers` 把超界页从缓冲池驱逐（含脏页，直接丢弃不写出）、记 WAL、再动文件——三步纪律恰好补上外部截断缺失的每一环。

结论：md.c 层假设"文件系统状态只被 PG 自己改变"，此假设之外没有防御。监控上，`pg_stat_database.checksum_failures`、amcheck、以及备份校验是发现这类外部损伤仅有的手段。

## 自测问题

**1. 把 refcount、usagecount、flags、内容锁编进同一个 64 位原子变量，相比"自旋锁保护的普通字段 + 独立 LWLock"，具体赢在哪些操作上？各位段的宽度是被什么约束钉死的？**

简答：pin/unpin 从"取锁-改-放锁"（≥2 次原子操作）变成 1 次 CAS / 1 次原子减（bufmgr.c:3333–3352、3520）；内容锁加/放锁各 1 次 CAS 且与 pin 同缓存行（BufferLockAttempt，bufmgr.c:6119）；"解锁 + 解 pin"可合并为一条原子操作（buf_internals.h:308–310 注释）；clock-sweep 的判定/降温/占有单条 CAS（freelist.c:288–303）；头锁 BM_LOCKED 只是 state 的 bit22，检查它与业务修改共用同一条 CAS。宽度约束：refcount 与共享锁计数都必须 ≥ MAX_BACKENDS_BITS=18（buf_internals.h:130/132 静态断言），反过来 MAX_BACKENDS 也被钉死为 2^18-1（procnumber.h:29 注释）；usagecount 4 位须容纳 BM_MAX_USAGE_COUNT=5；四段总和 ≤64（buf_internals.h:54）。

**2. 两个 backend 同时读取一个不在缓冲池的页，源码在哪两个点上收敛竞态、保证磁盘只读一次？失败了怎么办？**

简答：竞态窗口一（映射未建立）：双方各自 GetVictimBuffer 后到分区排他锁下 `BufTableInsert`（bufmgr.c:2277），输家得到赢家的 buf_id，归还自己的受害者改 pin 赢家的缓冲区。竞态窗口二（映射已建、内容无效）：后到者 PinBuffer 返回 valid=false，进入 StartBufferIO 看到 `BM_IO_IN_PROGRESS` 已被先到者置位（bufmgr.c:7300），于是睡在该缓冲区条件变量（WaitIO，bufmgr.c:7188）或带走 io_wref 异步等。完成方 TerminateBufferIO 置 BM_VALID 并广播。失败时（置 BM_IO_ERROR、VALID 未置），被唤醒者回到循环头发现"无 IO 且仍无效"，自己置 IO 位接班重读（bufmgr.c:7349–7356 注释）——协议没有专门的错误路径，重试是循环结构的自然行为。

**3. clock-sweep 为什么把 usagecount 封顶在 5？调大或调小分别有什么后果？环形策略下为什么又封顶 1？**

简答：usagecount 是 LRU 的近似，上限越大历史越长越接近 LRU，但受害者搜索最坏要绕 `BM_MAX_USAGE_COUNT+1` 圈（buf_internals.h:136–143 注释）。调大：淘汰延迟与扫描开销上升，且 4 位段硬上限 15；调小到 0：退化为随机淘汰未 pin 页。环形策略下 PinBuffer 只把 usagecount 从 0 抬到 1（bufmgr.c:3341–3349）：既防止环内反复 pin 把页"捂热"赖在池里，又保持非零热度使其不会在仍被环引用时就被别人的时钟收走；GetBufferFromRing 又以"usagecount>1"为信号把被外人看上的环页让渡给全局池（freelist.c:664–666）。

**4. 一次 20GB 表的顺序扫描会把 8GB 的缓冲池冲掉吗？相应机制与 InnoDB 的 midpoint 插入有何异同？**

简答：不会。表大小超过 NBuffers/4 时 heapam 自动申请 `BAS_BULKREAD` 环（heapam.c:398/410；尺寸 = 256KB 基础 + AIO 余量，freelist.c:459–484），扫描在环内自我复用，污染上限即环大小；环页热度封顶 1，出环即速朽。同：都基于"单次访问的页不值得留"的假设。异：InnoDB 是全局 LRU 链内的隐式策略（新页进 old 区、`innodb_old_blocks_time` 内二次访问不晋升），学习成本摊在所有访问上；PG 是按操作类型显式发放的私有环，OLTP 路径零开销，且 BULKREAD 环经 StrategyRejectBuffer（freelist.c:752）把需刷 WAL 的脏页踢回正常路径，实现"顺序扫描不为别人刷 WAL"；VACUUM/COPY 的环反其道而行——脏页留环内自己刷，"写流量制造者自负"。

**5. 从 UPDATE 弄脏一页到该页字节落盘，最多可能有哪四个角色把它写出去？为什么无论谁写都不会破坏一致性？**

简答：① checkpointer——BufferSync（bufmgr.c:3575）圈定打标、排序、限速写出；② bgwriter——BgBufferSync（bufmgr.c:3854）在时钟指针前方预洗冷脏页；③ 任意 backend——GetVictimBuffer 淘汰到脏页时前台 FlushBuffer（bufmgr.c:2634）；④ 环形策略进程复用自己环内脏页时。四者殊途同归都进 FlushBuffer（bufmgr.c:4526），那里统一执行 `XLogFlush(BufferGetLSN(buf))`（bufmgr.c:4584）——WAL-before-data 由函数收口而非调用方自觉；并发写出由 StartSharedBufferIO(forInput=false) 的 IO 位互斥（一个干活其余 ALREADY_DONE）；写出期间的并发修改由 share-exclusive 内容锁排斥。

**6. BM_TAG_VALID 和 BM_VALID 为什么必须是两个位？合成一个会出什么问题？**

简答：两者分离使"占坑"（映射可查、缓冲区可 pin）与"内容可用"解耦。BufferAlloc 在拿分区排他锁的短窗口内只建映射置 TAG_VALID（bufmgr.c:2336），慢速磁盘读完全在锁外进行，读完由 TerminateBufferIO 置 VALID。若合成一个位：要么必须持分区锁做完 I/O 才能插入映射（分区锁被磁盘延迟拖垮，等价于退回 8.0 的全局锁灾难），要么先插入就等于宣布内容有效（消费垃圾字节）。分离的代价是多出"TAG_VALID 而非 VALID"的中间态，由 7.5 的 I/O 协议看守（PinBuffer 返回 valid、StartBufferIO 仲裁谁来填内容）。

**7. FlushBuffer 为什么只要求 share-exclusive 而不是排他内容锁？依据 README 的哪几条规则推理？**

简答：写出需要的只是"页映像在拷贝期间不变"，不需要排斥读者。README 规则 1/2 下读者持共享锁或仅持 pin，都不修改页；规则 3 的元组修改要排他锁，被 SE 排斥；规则 4 的 hint bit 更新自本分支起要求 share-exclusive，同样被 SE 排斥（SE 与 SE 互斥）——于是 SE 锁下页字节稳定（bufmgr.c:4561–4563 注释：LSN 不会撕裂），校验和可安全计算，而并发查询完全不被写出阻塞。若要求排他锁，checkpointer 的每次写出都会打断该页上的所有读者，热页写出将造成周期性延迟尖峰。

**8. `elog(ERROR, "no unpinned buffers available")` 在什么条件下触发？为什么"扣了一次热度"就要重置 trycounter？**

简答：触发条件（freelist.c:263–275）：时钟连续看了 NBuffers 格，每格都因 refcount≠0 被跳过，**期间没有做出任何一次状态修改**——说明全池都被 pin（常见于 shared_buffers 过小、大量并发游标/长事务持 pin、或 pin 泄漏 bug）。重置逻辑：只要成功扣掉一次 usagecount，说明池里存在未 pin 的页且其热度在单调下降，最多再绕若干圈必有页降到 0 成为受害者——"有降温发生"就证明"终将有收获"，此时报错为时过早；反之 NBuffers 次连续无修改则构成"当前无法取得任何受害者"的完备证据，报错优于无限空转。

**9. 为什么 fsync 失败后 PostgreSQL 选择 PANIC 而不是重试？如果内核在 fsync 失败后保留脏页，答案会变吗？**

简答：Linux 在 fsync 报 EIO 后把未落盘页标记为干净（甚至丢弃），下次 fsync 会"成功"却什么都没写——重试路径会让 PG 误信数据已安全，推进 REDO 点并回收 WAL，形成"磁盘、WAL、内存三处皆无"的静默数据丢失（7.14.1 推演）。PANIC 的本质是承认"进程对'哪些页干净'的知识已被污染"，唯一可靠副本是 WAL，于是崩溃恢复从上个 checkpoint 重放重建一切。若内核确实保留脏页（某些文件系统/OS），重试是安全的——这正是 `data_sync_retry` GUC（fd.c:163，data_sync_elevel fd.c:3967）存在的意义：它把这个平台知识交给 DBA 声明。commit 9ccdd7f66e。

**10. 崩溃恢复时，checkpoint 已写出一半的脏页、full_page_writes、pd_lsn 三者如何配合保证一致？**

简答：checkpoint 完成记录未写入前，恢复起点固定为上一个 checkpoint 的 REDO 点，本轮已写出的页只是"超前"；重放时 pd_lsn ≥ 记录 LSN 的页被幂等跳过，落后的页被逐条重放补齐——超前与落后殊途同归。而"pd_lsn 判断本身可信"依赖页未被撕裂写：断电可能留下半新半旧的 8KB 页，其 pd_lsn 恰在新半边会骗过幂等判断——full_page_writes 为此在每页每 checkpoint 周期首次修改时于 WAL 记录完整页镜像，重放时整页覆写、不读取页上任何现存内容，撕裂页被无条件修复。三者闭环：FPI 保证页级原子性，pd_lsn 保证重放幂等，checkpoint 记录的"最后写入"保证起点正确。（InnoDB 的等价三件套：doublewrite / 页 LSN / fuzzy checkpoint LSN。）
