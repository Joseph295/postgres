# 06 · 存储管理与缓冲池：页、BufferDesc 与磁盘之间

> 对应《PostgreSQL 源码学习指南》第 7 章。所有 `文件:行号` 均已在本地源码树（20devel，master 分支）核实。注意：本分支的缓冲区状态字已从历史版本的 32 位扩展为 **64 位**（内容锁并入其中），与网上多数旧资料不同，本文以当前代码为准。

## 本章导读

存储层自底向上三级：**md.c**（把"表"落成一组 1GB 段文件）→ **smgr**（存储管理器抽象接口）→ **bufmgr**（共享缓冲池，页的缓存、淘汰与写出）。本章从 8KB 页的内部布局讲起，逐字段解析 BufferDesc——特别是把 refcount/usagecount/flags/内容锁压进**一个 64 位原子变量**的精巧设计；然后走读读页路径（命中/未命中/并发 I/O 协议）、clock-sweep 淘汰、环形缓冲策略、脏页写出的 WAL-before-data 检查；最后补上两个"辅助 fork"：FSM 与 VM。

与 InnoDB buffer pool 的对照贯穿全章：InnoDB 用"LRU 链表 + midpoint 插入 + free/flush 链表 + 多 instance 分片"，PG 用"共享哈希表 + clock-sweep + 环形缓冲 + 单一原子状态字"。两者面对同样的问题（扫描抗性、锁争抢、脏页回写节奏），给出了风格迥异的答案。

## 前置知识

- **pin 与内容锁**：pin（引用计数）保证缓冲区不被重用于别的页；内容锁（shared/share-exclusive/exclusive）保护页内数据的读写。规则详见 `src/backend/storage/buffer/README`：先 pin 后锁、pin 可长持、锁必须短持。
- **steal / no-force**：PG 允许未提交事务的脏页写盘（steal），提交时不强制刷数据页（no-force）——正确性完全由 WAL 兜底，这是读 06 章全部代码的前提。
- **TID**：`(块号, 行号)` 二元组，索引条目指向堆的方式。块号 → 段文件内偏移的换算在 md.c。

## 7.1 页布局：8KB 里的秩序

### 7.1.1 PageHeaderData 逐字段（bufpage.h:184）

`src/include/storage/bufpage.h:184`：

```c
typedef struct PageHeaderData
{
    PageXLogRecPtr pd_lsn;      /* 最后一次修改本页的 WAL 记录的结束 LSN */
    uint16      pd_checksum;    /* 页校验和（cluster 级开关，写盘时计算） */
    uint16      pd_flags;       /* 标志位，见下 */
    LocationIndex pd_lower;     /* 空闲空间起点 = 行指针数组的末尾 */
    LocationIndex pd_upper;     /* 空闲空间终点 = 元组数据区的起点 */
    LocationIndex pd_special;   /* 特殊区起点（堆页 = 页尾；索引页存放兄弟指针等） */
    uint16      pd_pagesize_version;  /* 页大小 + 布局版本号合一编码 */
    TransactionId pd_prune_xid; /* 页内最老的“可能可清理”XID 提示，0 表示无 */
    ItemIdData  pd_linp[FLEXIBLE_ARRAY_MEMBER];   /* 行指针数组，向后生长 */
} PageHeaderData;
```

- **pd_lsn** 是整个 WAL-before-data 机制的锚点（05 章已详述）：刷此页前必须 `XLogFlush(pd_lsn)`；恢复重放时若 `pd_lsn ≥ 记录LSN` 则跳过（幂等）。
- **pd_lower / pd_upper**：页内空间从两头向中间生长——行指针数组紧跟页头向后长（`pd_lower` 后移），元组数据从页尾（special 区之前）向前长（`pd_upper` 前移）。`pd_lower == pd_upper` 即页满。这与 InnoDB 页内"user records 堆 + page directory 槽"的思路同源，但 PG 的结构更简单：没有页内记录链表，元组物理位置无序，全靠行指针数组定序。
- **pd_flags**（bufpage.h:213–218）：`PD_HAS_FREE_LINES 0x0001`（有可复用的空行指针，加速插入时找槽）、`PD_PAGE_FULL 0x0002`（上次 UPDATE 没挤下，提示下次先 prune）、`PD_ALL_VISIBLE 0x0004`（页上全部元组对所有事务可见，与 VM 位联动）。
- **pd_prune_xid**：05 章看到 `PageSetPrunable(page, xid)` 写的就是它——"本页有个 XID=N 的事务动过元组，等 N 落到全局可见性水位之下，来清理吧"。让路过的查询能廉价判断"值不值得对这页做 HOT prune"。

### 7.1.2 ItemId：行指针这层间接性的意义（itemid.h:25）

```c
/* src/include/storage/itemid.h:25 */
typedef struct ItemIdData
{
    unsigned    lp_off:15,      /* 元组在页内的字节偏移 */
                lp_flags:2,     /* LP_UNUSED / LP_NORMAL / LP_REDIRECT / LP_DEAD */
                lp_len:15;      /* 元组字节长度 */
} ItemIdData;
```

4 字节一个，TID 的"行号"部分就是它在数组里的下标（1 起）。这层间接性是 PG 存储设计里回报最高的 4 个字节：

1. **页内搬移自由**：prune/整理时可以把元组在页内挪动压实（改 `lp_off`），所有指向该行的索引 TID 不变；
2. **HOT 的载体**：`LP_REDIRECT`（itemid.h:40）让链头行指针直接转发到存活版本，元组本体回收后索引仍可达；
3. **两阶段删除**：`LP_DEAD` 表示"元组已回收但索引可能还指着我"，VACUUM 删完索引项后才把它归还为 `LP_UNUSED`。

InnoDB 没有对应物——聚簇索引记录移动时靠 B-tree 自身重定位，二级索引存主键值而非物理位置。PG 的索引存物理 TID（查询少一次回表定位，但列胖主键无副作用），代价就是必须用行指针层解耦"逻辑行号"与"物理偏移"。

### 小结

页 = 页头（LSN 打头）+ 向后长的行指针数组 + 向前长的元组区 + special 区。行指针这层间接性换来了页内自由整理与 HOT，是 TID 寻址体系的基石。

## 7.2 缓冲池结构：BufferDesc 与共享哈希表

缓冲池三大件：`BufferBlocks`（`shared_buffers` 个 8KB 页帧的大数组）、`BufferDescriptors`（等长的描述符数组）、`SharedBufHash`（tag → 描述符下标的哈希表）。三者在启动时一次性从共享内存划出。

### 7.2.1 BufferDesc 逐字段（buf_internals.h:326）

```c
/* src/include/storage/buf_internals.h:326 */
typedef struct BufferDesc
{
    BufferTag   tag;            /* 我装的是哪个磁盘块：(表文件定位符, fork号, 块号)
                                   修改需持缓冲区头锁 */
    int         buf_id;         /* 我在数组里的下标，初始化后永不变 */
    pg_atomic_uint64 state;     /* ★ 状态字：flags+refcount+usagecount+内容锁，
                                   只能用原子操作修改 */
    int         wait_backend_pgprocno;  /* 等待 pin 数降到 1 的进程（VACUUM 用），
                                           至多一个 */
    PgAioWaitRef io_wref;       /* 异步 I/O 进行中时的等待句柄 */
    proclist_head lock_waiters; /* 等内容锁的进程链表，由头锁保护 */
} BufferDesc;
```

结构体被 `BufferDescPadded`（buf_internals.h:383）填充到 64 字节——正好一条缓存行，杜绝相邻描述符间的 false sharing（注释 buf_internals.h:361 明说这是实测有效的优化）。InnoDB 的 `buf_block_t` 同样讲究对齐，但体积大得多（内嵌 mutex、rw-lock、LRU/flush 链表节点等指针）；PG 把一切压进 8 字节 state，是两者最醒目的对照。

### 7.2.2 state：一个 64 位原子变量的位域设计（buf_internals.h:49）

```c
/* buf_internals.h:49 */
#define BUF_REFCOUNT_BITS 18        /* [0..17]   pin 计数 */
#define BUF_USAGECOUNT_BITS 4       /* [18..21]  clock-sweep 热度 */
#define BUF_FLAG_BITS 12            /* [22..33]  状态标志 */
#define BUF_LOCK_BITS (18+2)        /* [34..53]  内容锁：18位共享持有数+SE位+EX位 */
```

低到高依次是：

| 位段 | 宽度 | 内容 |
|---|---|---|
| refcount | 18 | 当前 pin 住本缓冲区的总次数（18 位恰好容纳 `MAX_BACKENDS`，procnumber.h:38） |
| usagecount | 4 | 0–5 的热度计数（`BM_MAX_USAGE_COUNT=5`，buf_internals.h:144） |
| flags | 12 | `BM_LOCKED`(头锁) `BM_DIRTY` `BM_VALID` `BM_TAG_VALID` `BM_IO_IN_PROGRESS` `BM_IO_ERROR` `BM_PIN_COUNT_WAITER` `BM_CHECKPOINT_NEEDED` `BM_PERMANENT` `BM_LOCK_HAS_WAITERS` `BM_LOCK_WAKE_IN_PROGRESS`（buf_internals.h:106–127） |
| 内容锁 | 20 | 18 位共享锁持有者计数 + 1 位 share-exclusive + 1 位 exclusive（buf_internals.h:79–86） |

为什么值得如此设计？读三处代码就懂了：

**（a）PinBuffer 的无锁 CAS 循环（bufmgr.c:3295）**：

```c
/* bufmgr.c:3312 */
old_buf_state = pg_atomic_read_u64(&buf->state);
for (;;)
{
    if (unlikely(old_buf_state & BM_LOCKED))          /* 头锁被人持有？ */
    {
        old_buf_state = WaitBufHdrUnlocked(buf);      /* 自旋等它释放 */
        continue;
    }
    buf_state = old_buf_state;
    buf_state += BUF_REFCOUNT_ONE;                    /* refcount+1：普通加法！ */
    if (strategy == NULL)
    {
        if (BUF_STATE_GET_USAGECOUNT(buf_state) < BM_MAX_USAGE_COUNT)
            buf_state += BUF_USAGECOUNT_ONE;          /* usagecount+1：同一次 CAS 捎带 */
    }
    else
    {   /* 环形策略：热度封顶 1，防止大扫描把页“捂热” */
        if (BUF_STATE_GET_USAGECOUNT(buf_state) == 0)
            buf_state += BUF_USAGECOUNT_ONE;
    }
    if (pg_atomic_compare_exchange_u64(&buf->state, &old_buf_state, buf_state))
        break;                                        /* 一条 CAS 完成 pin+加热 */
}
```

pin 是缓冲池最高频操作（每次页访问至少一次）。因为 refcount 在最低位段，`+1` 就是普通整数加法不会进位污染别的位段；usagecount 增量是 `1<<18`，同样一次加法。**pin + 热度更新 + "头锁未被持有"的检查，全部压缩进一条 CAS**——这就是位域编排的全部意义。历史上（PG 9.5 之前）这里是"拿自旋锁、改三个字段、放锁"，高并发下自旋锁缓存行乒乓是著名瓶颈；9.6 把 state 原子化，本分支更进一步把 LWLock 实现的内容锁也吸收进来（buf_internals.h:303 注释：专用实现使"解锁+解 pin 一条原子指令完成"、"无竞争加锁一条 CAS 完成"成为可能）。

**（b）头锁本身也是 state 里的一个位**：`BM_LOCKED` 就是"缓冲区头自旋锁"。`LockBufHdr()` 用 CAS 置位，`UnlockBufHdr()`（buf_internals.h:462）原子减去该位。需要多字段一致读改写（如改 tag）时才用它，且只允许持有几条指令。

**（c）StrategyGetBuffer 淘汰扫描同样纯 CAS**（见 7.4 节）：判断"refcount==0 且 usagecount==0"、递减热度、pin 住受害者，都直接在 state 上完成，淘汰路径不摸任何全局锁（除了推进时钟指针的原子加）。

对比 InnoDB：buffer pool 用 `buf_pool->mutex` + 块级 mutex + 块级 rw-lock 的多级锁体系，靠**多 buffer pool instance** 分片摊薄争抢；PG 则把并发热点收敛到每缓冲区一个原子字上，分片的对象是映射哈希表的分区锁（下节）。

### 7.2.3 共享哈希表 buf_table.c

"磁盘块 → 缓冲区"的映射由 `SharedBufHash`（`src/backend/storage/buffer/buf_table.c:34`，dynahash 实现）承担，接口极薄：

- `BufTableHashCode()`（buf_table.c:84）：对 `BufferTag`（(RelFileLocator, forkNum, blockNum)，buf_internals.h:150 起）求哈希；
- `BufTableLookup()`（buf_table.c:96）：查 tag → buf_id，查不到返回 -1；
- `BufTableInsert()`（buf_table.c:124）/ `BufTableDelete()`（buf_table.c:154）：换页时改映射。

并发控制沿用 buffer/README 描述的**分区锁**方案：哈希空间按 tag 哈希值低位切成 `NUM_BUFFER_PARTITIONS`（128）个分区，各配一把 LWLock（`BufMappingPartitionLock()`，buf_internals.h:253）。查表持分区共享锁（**必须在放锁前 pin 住查到的缓冲区**，否则它可能瞬间被淘汰）；改映射持分区排他锁，需要同时锁两个分区时按分区号升序拿以防死锁。

### 小结

BufferDesc 被刻意瘦身到一条缓存行；state 把 pin/热度/标志/头锁/内容锁五类信息编进 64 位，让最高频操作都变成单条原子指令；页定位靠 128 路分区锁的共享哈希表。PG 的哲学是"收敛热点到原子变量"，InnoDB 的哲学是"复制多份摊薄热点"。

## 7.3 读页路径：ReadBufferExtended → ReadBuffer_common

### 7.3.1 入口分发（bufmgr.c:879 / 926 / 1276）

`ReadBuffer()`（bufmgr.c:879）是 `ReadBufferExtended(rel, MAIN_FORKNUM, blockNum, RBM_NORMAL, NULL)` 的简写；`ReadBufferExtended()`（**bufmgr.c:926**）多出 fork 选择、读取模式（`RBM_ZERO_AND_LOCK` 等）和访问策略参数，直接转调 `ReadBuffer_common()`（bufmgr.c:1276）：

```c
/* bufmgr.c:1276 */
static pg_always_inline Buffer
ReadBuffer_common(Relation rel, SMgrRelation smgr, char smgr_persistence,
                  ForkNumber forkNum, BlockNumber blockNum,
                  ReadBufferMode mode, BufferAccessStrategy strategy)
{
    ...
    if (rel && RELATION_IS_OTHER_TEMP(rel))
        ereport(ERROR, ...);                   /* 拒绝读别人会话的临时表 */
    if (unlikely(blockNum == P_NEW))
        return ExtendBufferedRel(...);         /* P_NEW=扩展文件（兼容老路径） */
    if (unlikely(mode == RBM_ZERO_AND_CLEANUP_LOCK || mode == RBM_ZERO_AND_LOCK))
    {
        buffer = PinBufferForBlock(...);       /* 不读盘，直接分配缓冲区 */
        ZeroAndLockBuffer(buffer, mode, found);/* 清零并锁住，调用者要重写整页 */
        return buffer;
    }
    flags = READ_BUFFERS_SYNCHRONOUSLY;        /* 单页同步读：不必走异步派发 */
    ...
    if (StartReadBuffer(&operation, &buffer, blockNum, flags))  /* bufmgr.c:1637 */

    return buffer;

```

本分支的读路径已被异步 I/O（AIO）改造为两段式：`StartReadBuffer`（可以只发起）+ `WaitReadBuffers`（等待完成）。单页同步读只是这套机制的退化用法。下面拆开命中与未命中两条线。

### 7.3.2 命中路径：PinBufferForBlock → PinBuffer

`StartReadBuffersImpl()`（bufmgr.c:1371）对每个目标块调用 `PinBufferForBlock()`（bufmgr.c:1223）→ 共享缓冲区走 `BufferAlloc()`（bufmgr.c:2197）：

```c
/* BufferAlloc 开头（bufmgr.c:2197 起，示意） */
newHash = BufTableHashCode(&newTag);
newPartitionLock = BufMappingPartitionLock(newHash);
LWLockAcquire(newPartitionLock, LW_SHARED);       /* 分区共享锁 */
existing_buf_id = BufTableLookup(&newTag, newHash);
if (existing_buf_id >= 0)
{
    buf = GetBufferDescriptor(existing_buf_id);
    valid = PinBuffer(buf, strategy, false);      /* 放分区锁之前先 pin！ */
    LWLockRelease(newPartitionLock);
    if (valid) { *foundPtr = true; return buf; }  /* 命中：一次哈希+一次 CAS，结束 */
    ...                                           /* 页在但无效：别人读到一半，见 7.3.4 */
}
```

命中成本 = 一次分区共享锁 + 一次哈希查找 + PinBuffer 的一次 CAS（7.2.2 已逐行读过）。此外每个 backend 还有**私有 pin 计数表**（`GetPrivateRefCountEntry`，bufmgr.c:3305）：同进程重复 pin 同页时只在本地 `refcount++`，完全不碰共享内存——嵌套扫描反复 pin 同一页几乎免费。

### 7.3.3 未命中：淘汰 + 换 tag + 发起 I/O

`BufTableLookup` 落空后（bufmgr.c:2244 起）：放分区锁 → `GetVictimBuffer(strategy, io_context)`（bufmgr.c:2548）要一个可复用的缓冲区（内部走 clock-sweep，victim 若是脏页先 `FlushBuffer`，见 7.4/7.5）→ 重新拿**排他**分区锁，`BufTableInsert` 新 tag（若 victim 原来有 tag，需按分区号顺序锁两个分区并 `BufTableDelete` 旧映射）→ 设置 `BM_TAG_VALID`（注意 **BM_VALID 还没设**：映射已建立但内容是垃圾）→ 放锁，然后才通过 smgr 发起真正的磁盘读，读完在 `TerminateBufferIO` 里置 `BM_VALID`。

### 7.3.4 并发 I/O 协议：StartBufferIO / WaitIO / TerminateBufferIO

两个 backend 同时未命中同一页怎么办？靠 `BM_IO_IN_PROGRESS` 位当"I/O 互斥锁"。`StartSharedBufferIO()`（bufmgr.c:7290）：

```c
/* bufmgr.c:7297 */
for (;;)
{
    buf_state = LockBufHdr(buf);
    if (!(buf_state & BM_IO_IN_PROGRESS))
        break;                                /* 没人在做 I/O，跳出去自己做 */
    /* 有人在做：三种态度 */
    if (io_wref != NULL && pgaio_wref_valid(&buf->io_wref))
    {   *io_wref = buf->io_wref;              /* ① 拿走异步等待句柄，回头再等 */
        UnlockBufHdr(buf); return BUFFER_IO_IN_PROGRESS; }
    else if (!wait)
    {   UnlockBufHdr(buf); return BUFFER_IO_IN_PROGRESS; }   /* ② 不等，走人 */
    else
    {   UnlockBufHdr(buf);
        pgaio_submit_staged();                /* 先把自己攒的异步 I/O 发出去，防死锁 */
        WaitIO(buf);                          /* ③ 睡在条件变量上等它完成 */
    }
}
/* 走到这里：没有进行中的 I/O。也许别人已经替我干完了？ */
if (forInput ? (buf_state & BM_VALID) : !(buf_state & BM_DIRTY))
{   UnlockBufHdr(buf); return BUFFER_IO_ALREADY_DONE; }      /* 白嫖成功 */
UnlockBufHdrExt(buf, buf_state, BM_IO_IN_PROGRESS, 0, 0);    /* 置位：I/O 归我 */
return BUFFER_IO_READY_FOR_IO;
```

- 读者判据是 `BM_VALID`（内容有效即不用读），写者判据是 `!BM_DIRTY`（已不脏即不用写）——同一套协议同时服务读入与写出；
- `WaitIO()`（bufmgr.c:7188）睡在**每缓冲区一个的条件变量**上（PG 14 前是每缓冲区一把 io_in_progress LWLock）；异步 I/O 场景则通过 `io_wref` 等待 AIO 子系统；
- I/O 完成方调用 `TerminateBufferIO()`（bufmgr.c:7407）：清 `BM_IO_IN_PROGRESS`，按结果置 `BM_VALID`（读成功）或清 `BM_DIRTY`（写成功）或置 `BM_IO_ERROR`，并广播条件变量唤醒所有等待者。失败时被唤醒的等待者会发现"仍非 VALID"，回到循环头自己重试 I/O——**错误可以被接力恢复**。

### 小结

命中路径被打磨到"分区共享锁 + 一次 CAS"；未命中先无 I/O 地占好映射（TAG_VALID），再在锁外做慢速磁盘读；`BM_IO_IN_PROGRESS` + 条件变量构成"谁抢到谁干、其余人等结果、失败者可接班"的并发 I/O 协议。

## 7.4 淘汰：clock-sweep 与环形缓冲

### 7.4.1 StrategyGetBuffer 逐行走读（freelist.c:184）

`src/backend/storage/buffer/freelist.c:184`。先看时钟指针：

```c
/* freelist.c:109 */
static inline uint32
ClockSweepTick(void)
{
    victim = pg_atomic_fetch_add_u32(&StrategyControl->nextVictimBuffer, 1);
    if (victim >= NBuffers)
    {
        victim = victim % NBuffers;       /* 指针是只增计数器，取模得下标 */
        if (victim == 0)
            ...  /* 绕满一圈者持 spinlock 把计数器减去 NBuffers、completePasses++
                    （bgwriter 靠 completePasses 估算扫描速度） */
    }
    return victim;
}
```

时钟指针的推进是**一条原子加**——多个 backend 同时找受害者时各自领走相邻的下标，天然并行。主体：

```c
/* freelist.c:184 */
StrategyGetBuffer(BufferAccessStrategy strategy, uint64 *buf_state, bool *from_ring)
{
    if (strategy != NULL)
    {
        buf = GetBufferFromRing(strategy, buf_state);   /* 环形策略优先复用自己的环 */
        if (buf != NULL) { *from_ring = true; return buf; }
    }
    bgwprocno = INT_ACCESS_ONCE(StrategyControl->bgwprocno);
    if (bgwprocno != -1)
        SetLatch(&GetPGProcByNumber(bgwprocno)->procLatch); /* 顺手叫醒 bgwriter */
    pg_atomic_fetch_add_u32(&StrategyControl->numBufferAllocs, 1); /* 供 bgwriter 估算需求 */

    trycounter = NBuffers;
    for (;;)                                           /* ---- 时钟扫描 ---- */
    {
        buf = GetBufferDescriptor(ClockSweepTick());   /* 指针前进一格 */
        old_buf_state = pg_atomic_read_u64(&buf->state);
        for (;;)                                       /* ---- 对这一格做 CAS ---- */
        {
            local_buf_state = old_buf_state;
            if (BUF_STATE_GET_REFCOUNT(local_buf_state) != 0)
            {                                          /* 被 pin：跳过 */
                if (--trycounter == 0)
                    elog(ERROR, "no unpinned buffers available");
                break;                                 /* 转出去看下一格 */
            }
            if (unlikely(local_buf_state & BM_LOCKED))
            {   old_buf_state = WaitBufHdrUnlocked(buf); continue; }
            if (BUF_STATE_GET_USAGECOUNT(local_buf_state) != 0)
            {
                local_buf_state -= BUF_USAGECOUNT_ONE; /* 有热度：扣一点 */
                if (pg_atomic_compare_exchange_u64(&buf->state, &old_buf_state,
                                                   local_buf_state))
                {   trycounter = NBuffers; break; }    /* 扣成功，重置计数看下一格 */
            }
            else
            {
                local_buf_state += BUF_REFCOUNT_ONE;   /* 冷透了：直接 pin 为己有 */
                if (pg_atomic_compare_exchange_u64(&buf->state, &old_buf_state,
                                                   local_buf_state))
                {
                    if (strategy != NULL)
                        AddBufferToRing(strategy, buf);
                    *buf_state = local_buf_state;
                    return buf;                        /* 受害者到手 */
                }
            }
        }
    }
}
```

要点：

- **近似 LRU 而非 LRU**：每次 pin 给 usagecount 加热（封顶 5），时钟扫过降温，降到 0 且无 pin 者出局。选它的理由（buffer/README + buf_internals.h:137 注释）：LRU 要维护全局链表 = 每次访问都写全局结构；clock-sweep 的"访问记录"分散在各缓冲区自己的 state 里，扫描才集中读。封顶 5 是精度与"最坏扫 6 圈"开销的折衷；
- `trycounter` 防死循环：全部缓冲区都被 pin 时（NBuffers 次都没做出任何状态修改）报错而非空转；每成功扣一次热度就重置计数——只要还有降温发生，总会有页降到 0；
- `elog(ERROR, "no unpinned buffers available")`（freelist.c:274）是运维时偶见的报错，源头就在这一行；
- 对照 InnoDB：LRU 链表 + `innodb_old_blocks_pct` midpoint 插入 + old/new 子链表——用"先进老区、二次访问才晋升"对抗扫描污染；PG 对抗扫描污染的武器不在 clock-sweep 里，而是下面的环形策略。

### 7.4.2 环形缓冲策略：GetAccessStrategy（freelist.c:426）

大扫描如果走普通 clock-sweep，会把整个缓冲池刷成自己碰过一次的冷页（InnoDB 世界的"LRU 污染"）。PG 的解法：给这类操作发一个小小的私有"环"，让它在环内自我循环复用。`GetAccessStrategy()`（freelist.c:426）按用途给环定尺寸：

| 类型 | 环大小 | 使用者 |
|---|---|---|
| `BAS_NORMAL` | 无环（返回 NULL，走正常 clock-sweep） | 普通访问 |
| `BAS_BULKREAD` | 基础 256KB，再加 `io_combine_limit × effective_io_concurrency` 个块的异步 I/O 余量（freelist.c:442–486） | 大表顺序扫描（表大小 > shared_buffers/4 时启用） |
| `BAS_BULKWRITE` | 16MB（freelist.c:487） | COPY IN、CREATE TABLE AS |
| `BAS_VACUUM` | 2MB（freelist.c:490，实际由 `vacuum_buffer_usage_limit` GUC 控制，`GetAccessStrategyWithSize()` freelist.c:511） | VACUUM/ANALYZE |

配套细节：

- 环内页被 pin 时 usagecount 封顶 1（PinBuffer 里的 strategy 分支，bufmgr.c:3341）：既防止环页被自己"捂热"赖在池里，又保持非零热度不被别人的 clock-sweep 抢走；
- **BULKREAD 的脏页逃逸**：环页若被弄脏（LSN 更新），复用它就得先刷 WAL——顺序扫描不该背这个开销，于是 `StrategyRejectBuffer()`（freelist.c:752，仅 `BAS_BULKREAD` 生效）把它踢出环、另找普通受害者；而 VACUUM/COPY 的环不踢脏页，宁可自己刷 WAL 换取"写流量由制造者自己消化，不祸害别人"（README 的设计说明）；
- 256KB 的另一个用意（README）：小到能装进 L2 cache，页从 OS cache 拷入 shared buffers 更快，还给同步顺序扫描（synchronized seqscan）留了"缓存尾迹"。

### 小结

淘汰 = 原子加推进时钟 + 每格一次 CAS 判定/降温/占有，全程无全局锁；扫描抗性外包给三种尺寸的环形策略。InnoDB 把"防污染"内置进 LRU（midpoint），PG 把它外置成按工作负载类型发放的私有环——后者还额外实现了"大扫描的写回成本自负"。

## 7.5 脏页写出：FlushBuffer、bgwriter 与 checkpointer

### 7.5.1 FlushBuffer 与 WAL-before-data 检查（bufmgr.c:4526）

任何人（backend 淘汰时、bgwriter、checkpointer）把脏页写盘都走 `FlushBuffer()`：

```c
/* bufmgr.c:4526 */
static void
FlushBuffer(BufferDesc *buf, SMgrRelation reln, IOObject io_object, IOContext io_context)
{
    ...
    if (StartSharedBufferIO(buf, false, true, NULL) == BUFFER_IO_ALREADY_DONE)
        return;                       /* 有人抢先刷完了 */
    ...
    recptr = BufferGetLSN(buf);       /* 读页头 pd_lsn（持共享级以上内容锁，读不撕裂） */
    if (pg_atomic_read_u64(&buf->state) & BM_PERMANENT)
        XLogFlush(recptr);            /* ★ WAL-before-data 的执行点：
                                         覆盖本页的 WAL 必须先落盘 */
    bufBlock = BufHdrGetBlock(buf);
    PageSetChecksum((Page) bufBlock, buf->tag.blockNum);
    smgrwrite(reln, BufTagGetForkNum(&buf->tag), buf->tag.blockNum, bufBlock, false);
    ...
    TerminateBufferIO(buf, true, 0, true, false);   /* 清 BM_DIRTY、放 IO 位 */
}
```

这就是 05 章反复预告的闭环：`heap_insert` 把 LSN 盖进页头 → `FlushBuffer` 刷页前 `XLogFlush(页头LSN)`。两行代码相隔两个文件，共同构成 ARIES 第一原则的完整实现。细节：

- unlogged 表（无 `BM_PERMANENT`）跳过 XLogFlush——它们不写 WAL，页头可能带着索引 AM 造的"假 LSN"，拿去 flush 会出灾难（注释 bufmgr.c:4572 起详述）；
- 写出时只需**share-exclusive 级**内容锁（buffer/README 规则 6）：不阻塞读者，但阻止修改者——保证落盘的页映像自洽（校验和不会算废）。InnoDB 刷页同样要求页 latch 下拷贝或 double-write 缓冲聚合；
- `StartSharedBufferIO(forInput=false)` 复用 7.3.4 的 I/O 互斥协议：两个进程同时想刷同一页，一个干活一个直接返回。

### 7.5.2 bgwriter 与 checkpointer 的分工

两个后台进程都刷脏页，但目标完全不同：

**checkpointer**（`src/backend/postmaster/checkpointer.c:206` CheckpointerMain）：为**恢复时长**负责。周期性（`checkpoint_timeout` 或 WAL 量触发）执行 `CreateCheckPoint()`（xlog.c:7400）→ `BufferSync()`（bufmgr.c:3575）：给当刻所有脏页打上 `BM_CHECKPOINT_NEEDED`，然后按**表空间交错 + 块号排序**的顺序全部写出（排序换顺序 I/O，交错防单盘过载），中间用 `CheckpointWriteDelay` 按 `checkpoint_completion_target` 匀速摊开；结束后 fsync 所有文件、写 checkpoint WAL 记录、更新 pg_control。此后崩溃恢复只需从这个 REDO 点开始重放。对应 InnoDB 的 fuzzy checkpoint + `innodb_io_capacity` 节流。

**bgwriter**（`src/backend/postmaster/bgwriter.c:89` BackgroundWriterMain）：为**前台延迟**负责。它不关心恢复，只做一件事——`BgBufferSync()`（bufmgr.c:3854）沿着 clock-sweep 指针**前方**扫描（用 `StrategySyncStart()` 读指针位置和 completePasses，但不推进指针），把"即将被时钟扫到、又脏又冷（无 pin、无热度）"的页提前洗干净。这样 backend 淘汰时拿到的大概率是干净页，省掉前台同步 `FlushBuffer + XLogFlush` 的延迟。刷多少由反馈控制：根据 `numBufferAllocs`（最近分配速率）预测需求，乘 `bgwriter_lru_multiplier`，封顶 `bgwriter_lru_maxpages`。角色上它相当于 InnoDB 的 page cleaner 里"LRU flush"的那一半（为腾空闲页服务），而 checkpointer 相当于"flush list flush"的那一半（为推进 checkpoint LSN 服务）。

第三个相关角色 walwriter 刷的是 WAL 缓冲不是数据页，此处不展开。

### 小结

一切脏页写出都过 `FlushBuffer`，其中 `XLogFlush(BufferGetLSN(buf))` 一行落实 WAL-before-data。checkpointer 面向 RTO（有序、全量、限速），bgwriter 面向前台延迟（追着时钟指针预洗冷脏页），backend 自己刷页是两者都没跟上时的兜底（`pg_stat_io` 里 `io_context=normal` 的 write 数量高即为信号）。

## 7.6 smgr 抽象层与 md.c：段文件与 fork

### 7.6.1 smgr：一张函数指针表（smgr.c:88）

`src/backend/storage/smgr/smgr.c:88` 定义 `f_smgr` 接口：`smgr_open/close/create/unlink/extend/readv/writev/writeback/nblocks/truncate/immedsync...`，目前表里只有一个实现 `md.c`（smgr.c:133 的 `smgrsw[]` 数组）。这层看似冗余的间接性是历史遗产（Berkeley 时代有过多种存储介质后端），如今作用是：bufmgr 及以上代码完全不知道"文件"的存在，只对 `(RelFileLocator, fork, blockNum)` 编址——也为将来非文件系统后端留了插槽。

### 7.6.2 md.c：1GB 段文件（md.c:487 等）

一个 fork 的数据 = 一串段文件：`基础文件名`、`.1`、`.2`……每段最多 `RELSEG_SIZE` 块（默认 131072 块 × 8KB = **1GB**，md.c:51–76 注释）。切段的原因是可移植性（老文件系统 2GB 上限）与工具友好（cp/rsync 处理超大单文件的尴尬）。块号换算（`mdextend`，md.c:487）：

```c
/* md.c:516 */
v = _mdfd_getseg(reln, forknum, blocknum, skipFsync, EXTENSION_CREATE);
                                    /* 段号 = blocknum / RELSEG_SIZE，逐段打开/创建 */
seekpos = (pgoff_t) BLCKSZ * (blocknum % ((BlockNumber) RELSEG_SIZE));
                                    /* 段内偏移 = 块号取余 × 8KB */
```

读写入口 `mdreadv`（md.c:858）/`mdwritev`（md.c:1070）用 `preadv/pwritev` 做向量化 I/O（配合 `io_combine_limit` 把相邻块合并成一次系统调用）。fsync 则被"外包"：md.c 把待 fsync 的文件登记给 checkpointer（`register_dirty_segment`），checkpoint 时统一执行——平时 write 只进 OS page cache。对比 InnoDB：独立表空间 `.ibd` 单文件自管页分配（extent/segment 元数据在文件内），PG 则把空间管理完全交给文件系统，文件内没有任何分配元数据，页号即偏移——结构极简，代价是依赖文件系统的行为。

### 7.6.3 fork：一张表的四个"分身"（relpath.h:59）

`src/include/common/relpath.h:59`：

```c
MAIN_FORKNUM = 0,        /* 主数据：16384、16384.1、... */
FSM_FORKNUM,             /* 空闲空间地图：16384_fsm */
VISIBILITYMAP_FORKNUM,   /* 可见性地图：16384_vm */
INIT_FORKNUM,            /* unlogged 表的“出厂镜像”：16384_init，
                            崩溃恢复时用它重置主 fork */
```

fork 是"同一逻辑表的多个物理文件流"，共享 RelFileLocator、各自独立编址块号。BufferTag 里含 forkNum（7.2.3），所以 FSM/VM 页与数据页在同一个缓冲池里被统一缓存管理。InnoDB 没有对应概念——它把位图（如 change buffer bitmap）内嵌在表空间的固定页里；PG 的 fork 方案让辅助数据可以独立增长、独立截断。

## 7.7 FSM 与 VM：两张地图

### 7.7.1 FSM：定位空闲空间的三层树（freespace.c）

**用途**：`heap_insert` 找页（05 章 ③ 步）不能线性扫全表问"谁有空位"，FSM 提供 O(树高) 的查找。

**编码**（freespace.c:37–65 注释）：每个堆页的空闲空间压缩成 **1 字节**类别值：`FSM_CATEGORIES=256` 档，每档 `FSM_CAT_STEP = BLCKSZ/256 = 32` 字节。粗粒度足够——找页只需"至少这么大"，不需精确值。

**结构**：两级嵌套的树。**页内**：每个 FSM 页是一棵完全二叉树，叶子是若干堆页的类别值，内部节点存两孩子的 max，根 = 本页所辖范围的最大空闲档；**页间**：FSM 页再组成一棵 `FSM_TREE_DEPTH = 3` 层的树（freespace.c:75，8KB 页可存约 4000 个叶槽，4000³ 远超 2³² 块的上限，3 层足矣）。上层 FSM 页的叶槽存的是下层 FSM 页的根值。

**查找** `fsm_search()`（freespace.c:688）：从根 FSM 页出发，`fsm_search_avail()` 在页内二叉树里下降找一个 ≥ 目标档位的叶槽 → 得到下层页号 → 递归到叶层，返回堆块号。页内下降从"上次成功位置的下一个"开始（存在 `fsm_local` 提示里），让并发插入自然散开到不同页，避免全员挤同一个"最空的页"。

**更新** `RecordPageWithFreeSpace()`（freespace.c:194）：改叶槽后向上冒泡修正父节点的 max（只在值真的变化时）。关键权衡：**FSM 不写 WAL**（`fsm_readbuf` 的页用 `RBM_ZERO_ON_ERROR`）——它只是提示（hint），崩溃后略有失真无碍正确性（insert 拿到"其实没那么空"的页会失败重查并顺手纠正 FSM），VACUUM 也会重建它。对比 InnoDB：空闲管理靠表空间内的 XDES（extent descriptor）页 + inode 页记录 extent/页的分配状态，是精确的、受 redo 保护的元数据；PG 的 FSM 是允许说谎的缓存——又一次体现"简单结构 + 事后修正"的设计口味。

### 7.7.2 VM：每堆页 2 个 bit（visibilitymap.c）

`src/backend/access/heap/visibilitymap.c`，每个堆页在 VM fork 里占 `BITS_PER_HEAPBLOCK = 2` 位（`src/include/access/visibilitymapdefs.h:17`）：

- **all-visible**：页上所有元组对所有事务可见（没有未提交/未过老快照的版本）；
- **all-frozen**：页上所有元组已冻结（xmin 被替换为永久可见标记），设置它的前提是已 all-visible。

一个 8KB VM 页覆盖约 32K 个堆页 ≈ 256MB 堆（visibilitymap.c:125 `HEAPBLOCKS_PER_PAGE`），所以 VM 极小，常驻缓存。核心 API：`visibilitymap_pin()`（visibilitymap.c:209，锁堆页前先 pin 对应 VM 页，05 章多次见到）、`visibilitymap_set()`（visibilitymap.c:260，仅 VACUUM 调用，需持堆页锁并记 WAL）、`visibilitymap_clear()`（visibilitymap.c:152，任何使页不再全可见的 DML 在临界区内调用）、`visibilitymap_get_status()`（visibilitymap.c:324）。

**两大用途**：

1. **index-only scan**：索引扫描拿到 TID 后查 VM，目标堆页 all-visible 就**不用回表**做可见性判断——PG 的索引不存版本信息（对比 InnoDB 二级索引可借助 PAGE_MAX_TRX_ID + 回聚簇索引 undo 判断），VM 是 PG 实现覆盖索引扫描的先决条件。`EXPLAIN` 里 `Heap Fetches` 高就是 VM 位没设上（表更新频繁或 VACUUM 不及时）；
2. **VACUUM 跳页**：all-visible 页必无死元组，常规 VACUUM 直接跳过；all-frozen 页连反回卷（aggressive）VACUUM 也能跳过——没有 all-frozen 位之前（PG 9.6 前），每次反回卷 VACUUM 都要全表扫描，这是当年大表运维的噩梦。

**一致性纪律**（visibilitymap.c:34–56 注释）：VM 位是"设了必须为真，没设可以为假"的单向保守提示；错误地置位会导致 index-only scan 返回错数据、relfrozenxid 被错误推进——所以**置位**必须持堆页锁 + WAL 记录 + 给 VM 页盖 LSN（确保"使页全可见的那条 WAL"先于 VM 页落盘）；**清位**则总是安全方向，与触发它的堆修改共用同一临界区和 WAL 记录（05 章 heap_insert/update/delete 里的 `visibilitymap_clear` 调用都在 START/END_CRIT_SECTION 之间，且 VM 缓冲区被注册进同一条 WAL 记录）。页头的 `PD_ALL_VISIBLE` 与 VM 位必须同步维护——前者让 DML 不查 VM 就知道"要不要清位"。

### 小结

FSM 是"允许失真、不记 WAL"的 256 档空闲空间摘要树，为 insert 提供 O(log) 找页；VM 是"保守为真、置位记 WAL"的每页 2 bit 可见性摘要，支撑 index-only scan 与 VACUUM 跳页。两者都是 fork 形式的独立小文件，与数据页共用同一个缓冲池。

## 自测问题

**1. 把 refcount、usagecount、flags 编进同一个原子变量，相比"自旋锁保护的三个普通字段"，具体赢在哪些操作上？**

简答：pin/unpin（每次页访问必经）从"取锁-改-放锁"三步变成一条 CAS（refcount 在最低位段，加 1 不进位；usagecount 增量 `1<<18` 同一条指令捎带，见 bufmgr.c:3333–3351）；clock-sweep 的判定+降温+占有同样单条 CAS（freelist.c:288、300）；头锁 BM_LOCKED 本身也是 state 的一个位，等锁与读状态合并为同一原子读。收益本质：消除自旋锁缓存行在多核间的乒乓，本分支更把内容锁也并入，使"解锁+解 pin"能一次原子操作完成（buf_internals.h:303 注释）。

**2. 两个 backend 同时读取一个不在缓冲池的页，源码如何保证磁盘只被读一次？失败了怎么办？**

简答：先到者经 BufferAlloc 建好映射（BM_TAG_VALID，无 BM_VALID），在 StartSharedBufferIO（bufmgr.c:7290）中置 BM_IO_IN_PROGRESS；后到者哈希命中同一缓冲区、pin 后发现无效，进入 StartBufferIO 看到 IO 位已置，便睡在该缓冲区的条件变量上（WaitIO，bufmgr.c:7188）或改用异步等待句柄。完成方 TerminateBufferIO 置 BM_VALID 并广播唤醒。若 I/O 失败（置 BM_IO_ERROR、不置 VALID），被唤醒者循环回来发现"无 IO 进行中且仍无效"，就自己置 IO 位接班重读。

**3. clock-sweep 为什么把 usagecount 封顶在 5？调大调小分别有什么后果？**

简答：usagecount 是 LRU 的近似——值越大越接近真实 LRU（保留更长访问历史），但受害者搜索最坏要绕 `BM_MAX_USAGE_COUNT+1` 圈才能把热页降到 0（buf_internals.h:137 注释）。调大：淘汰延迟和扫描开销上升，且 4 位位段最大只能到 15；调小到 0 则退化为"随机淘汰未 pin 页"。5 是精度/开销的经验折衷。另注意环形策略下热度封顶 1（bufmgr.c:3347），刻意让批量访问的页保持"易淘汰"。

**4. 一次 8GB 表的顺序扫描会把 128GB 的缓冲池冲掉吗？相应机制与 InnoDB 的 midpoint 插入有何异同？**

简答：不会。表超过 shared_buffers/4 时执行器申请 `BAS_BULKREAD` 环（基础 256KB + AIO 余量，freelist.c:442），扫描页在环内循环复用，最多污染环那么大的空间；环页 usagecount 封顶 1，出环后很快被正常时钟回收。同：都基于"单次访问的页不值得留"的假设。异：InnoDB 是全局 LRU 链内的策略（新页进 5/8 处 old 区，二次访问才晋升），所有访问都受其规则约束；PG 是按操作类型显式发放的私有环，普通 OLTP 访问完全不受影响，且 BULKREAD 环通过 StrategyRejectBuffer 把脏页踢回正常路径，避免扫描背上刷 WAL 的成本。

**5. 从 `UPDATE` 弄脏一页到该页字节最终落盘，最多可能有哪四个角色把它写出去？各自的触发逻辑和对应的 WAL 约束是什么？**

简答：① checkpointer——BufferSync（bufmgr.c:3575）按 checkpoint 周期给脏页打 BM_CHECKPOINT_NEEDED 后排序限速写出；② bgwriter——BgBufferSync（bufmgr.c:3854）在时钟指针前方预写"冷而脏"的页；③ 任意 backend——GetVictimBuffer 淘汰到脏页时前台同步写（bufmgr.c:2634 调 FlushBuffer）；④ 使用环形策略的进程（VACUUM/COPY）复用自己环内脏页时写出。四者殊途同归都进 FlushBuffer（bufmgr.c:4526），那里统一执行 `XLogFlush(BufferGetLSN(buf))`——无论谁写、何时写，覆盖该页的 WAL 一定先于页落盘，这就是 steal 策略下数据不坏的全部保证。
