## 1. 工具链安装

系统自带：Apple clang 21、bison 2.3、flex 2.6.4。缺 meson / ninja / pkg-config / icu4c，用 Homebrew 补齐：

```bash
brew install meson ninja pkg-config icu4c
```

注意：icu4c 是 keg-only（不在默认搜索路径），配置时需要设置：

```bash
export PKG_CONFIG_PATH="/opt/homebrew/opt/icu4c@78/lib/pkgconfig"
```

## 2. 配置与编译（Debug 构建）

```bash
cd ~/postgres
PKG_CONFIG_PATH=/opt/homebrew/opt/icu4c@78/lib/pkgconfig \
    meson setup build --prefix=$HOME/pgdev -Dbuildtype=debug -Dcassert=true
ninja -C build            # 1912 个目标，Apple Silicon 上数分钟
ninja -C build install    # 安装到 ~/pgdev
```

- `-Dbuildtype=debug`：-O0 + 调试符号，断点调试必需；
- `-Dcassert=true`：打开内核断言（学习期必开，生产禁用）；
- 配置结果确认：icu 78.3 / readline / openssl / lz4 / zstd 均已启用；
- `build/` 目录已加入 `.git/info/exclude`（编译产物不入库）。

## 4. 验收实验一：lldb 在 heap_insert 断住一条 INSERT ✅

### 4.1 遇到的问题：attach 被拒

对运行中的 backend 执行 `lldb -p <PID>` 报 `attach failed: no such process`。
**根因**：macOS Developer mode 未开启（`DevToolsSecurity -status` → disabled），
attach 需要的 task_for_pid 权限被拒，错误信息有误导性。

**待办**：执行 `sudo DevToolsSecurity -enable` 后，两窗口 attach 调试流即可用。

### 4.2 绕过方案：单用户模式 + lldb 启动（launch 不需要 Developer mode）

```bash
echo "INSERT INTO t VALUES (43, 'world');" > /tmp/insert.sql
~/pgdev/bin/pg_ctl -D ~/pgdata stop        # 单用户模式需独占数据目录
lldb -b \
  -o "br set -n heap_insert" \
  -o "process launch -i /tmp/insert.sql" \
  -o "bt 8" \
  -o "p relation->rd_rel->relname" \
  -o "p tup->t_len" \
  -o "continue" \
  -- ~/pgdev/bin/postgres --single -D ~/pgdata postgres
~/pgdev/bin/pg_ctl -D ~/pgdata -l ~/pgdev/server.log start
```

### 4.3 实验结果（实录）

断点命中于 `heapam.c:2008`（heap_insert 的第一条可执行语句 `GetCurrentTransactionId()`），
调用栈与《学习指南》6.4 节全景图逐帧吻合：

```
frame #0: heap_insert(relation=..., tup=..., cid=0, ...)   heapam.c:2008
frame #1: heapam_tuple_insert                              heapam_handler.c:161
frame #2: table_tuple_insert        ← Table AM 接口层       tableam.h:1461
frame #3: ExecInsert                                       nodeModifyTable.c:1276
frame #4: ExecModifyTable                                  nodeModifyTable.c:4970
frame #5: ExecProcNodeFirst                                execProcnode.c:469
frame #6: ExecProcNode                                     executor.h:327
frame #7: ExecutePlan(operation=CMD_INSERT, ...)           execMain.c:1773
```

断点现场取证：

```
p relation->rd_rel->relname → (NameData) (data = "t")   ← 正是测试表
p tup->t_len                → (uint32) 34               ← 23 字节头 + 数据
```

两个知识点：
- 断点停在 2008 行而非 2005 行：2005 是函数签名，首条可执行语句是取 XID；
  PG 刻意在拿页锁**之前**取 XID，因为 XID 分配可能走复杂路径，不能持锁做；
- 单用户模式（`postgres --single`）没有 postmaster、没有并发，
  是调试启动流程与崩溃恢复的唯一手段，内核开发者常用武器。

## 5. 验收实验二：pageinspect 直击 MVCC 版本链 ✅

```sql
-- UPDATE 前（表中已有两行）
SELECT lp, t_xmin, t_xmax, t_ctid FROM heap_page_items(get_raw_page('t', 0));
 lp | t_xmin | t_xmax | t_ctid
  1 |  696   |   0    | (0,1)      ← id=42 'hello'
  2 |  697   |   0    | (0,2)      ← id=43 'world'（lldb 实验插入的）

UPDATE t SET val = 'hello-v2' WHERE id = 42;

-- UPDATE 后
 lp | t_xmin | t_xmax | t_ctid
  1 |  696   |  698   | (0,3)      ← 旧版本：xmax=698 标记"被删"，ctid 指向新版本
  2 |  697   |   0    | (0,2)
  3 |  698   |   0    | (0,3)      ← 新版本：xmin=698，与旧版本同页 → HOT 更新
```

亲眼验证了《学习指南》第 0.2 / 6.2 节的核心论断：
**UPDATE = 插入新版本 + 旧版本打 xmax 标记，版本链靠 ctid 串联**。
